# 📝 寝ている間にバグが直る？2026年型自己修復（Self-Healing）CI/CDパイプラインの構築

- **🎯 おすすめの対象:** DevOpsエンジニア、CI/CD担当者、些細なLintエラーにうんざりしているすべての開発者
- **⏱️ 所要時間:** エラーのデバッグ 30分 → 1分（完全自動化）
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o（コード解析・修正特化）

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「退勤直前に作成したPRが、Lintエラーや些細な型不一致で赤いバツ（❌）になり、着たばかりの上着を脱いでノートPCを再び開いた経験はありませんか？」_

2026年、私たちは今**「自己修復（Self-Healing）パイプライン」**の時代を生きています。CIが失敗すると、AIエージェントが即座にエラーログを解析し、自らコードを修正して再コミットします。翌朝出社すると、勝手に緑色のチェックマーク（✅）がついたPRがあなたを迎えてくれるはずです。

本記事では、GitHub ActionsとAIを組み合わせた**「自ら直す」CI/CDパイプライン**を構築するための実践的なプロンプトと、無限コミットループ（ゾンビエージェント）を防ぐためのノウハウを公開します。

---

## ⚡️ 3行まとめ (TL;DR)

1. **自動化を超えた自律性:** エラー発生時に通知を送るだけのCIから、原因を特定し直接コードを修正するAIエージェントのワークフローへと進化しました。
2. **最小限の変更の原則 (Minimal Change):** AIがビジネスロジックを勝手に書き換えないよう、型や構文エラーのみを修正する強力な制約を設ける必要があります。
3. **無限ループ対策は必須:** Botが作成したコミットが再びワークフローをトリガーしないよう設定し、テストを永遠に繰り返す「ゾンビエージェント」化を防ぐ必要があります。

---

## 🚀 解決策: "Self-Healing AI Agent プロンプト"

CIパイプライン（Job 2）で、テスト（Job 1）失敗時に呼び出されるAIエージェントのコアプロンプトです。

### 🥉 Basic Version (基本型)

PrettierやESLintの単純な構文エラーを素早く修正したい場合に使用します。

> **役割:** あなたは`[シニアフロントエンド開発者]`です。
> **タスク:** 提供されたエラーログを確認し、コードの`[Lintエラーおよびタイポ]`のみを迅速かつ正確に修正して、コード全体を返してください。


### 🥇 Pro Version (専門家型)

ビジネスロジックを損なうことなく、型エラーと構文エラーのみを厳格に修正する実践向けプロンプトです。

> **役割 (Role):** あなたは経験15年のシニアDevOpsエンジニア兼コードレビュアーです。コードの安定性を最優先し、不必要な変更を極度に警戒します。
>
> **状況 (Context):**
>
> - 背景: 現在、CIパイプラインでビルドまたはテストが失敗しています。
> - 目標: `[Error Log]`と`[Source Code]`を解析して失敗の原因を特定し、コードを修正します。
>
> **タスク (Task):**
>
> 1. エラーログに基づいて、ソースコードの問題点を正確に診断してください。
> 2. エラーを解決するための**最小限の変更（Minimal Change）**のみを実行してください。
> 3. コードのスタイル、コメント、コアなビジネスロジックは**絶対**に変更しないでください。
> 4. 修正が完了したコードをJSON形式で返してください。
>
> **制約事項 (Constraints):**
>
> - ⚠️ **ビジネスロジックの修正禁止:** 単純なタイポ、セミコロンの抜け、型の不一致など、明らかな機械的・構文的エラーのみが修正対象です。ロジックの変更が必要と思われる場合は、絶対にコードを修正せず、`"MANUAL_INTERVENTION_REQUIRED"`を出力して直ちに終了してください。
> - ⚠️ **ハルシネーション（幻覚）の防止:** 原因が正確にわからない場合や確信がない場合は、むやみに修正せず空のレスポンスを返してください。
>
> **出力形式 (Format):**
>
>
> {
>   "file_path": "[エラーが発生したファイルのパス]",
>   "fixed_content": "[修正された全体コードの内容]"
> }
>

## 💡 作成者のコメント (Insight)

このシステムを実際の業務に導入した際、最も輝いたのは**「大規模なTypeScript移行」**プロジェクトでした。数百のファイルから降り注ぐ細かな`any`型エラーやインターフェースの不一致を人間が一つ一つ修正するのは、まさに拷問のような作業です。

しかし、このプロンプトを適用した自己修復ワークフローを導入することで、単純な反復修正作業の80%が消滅しました。初期段階では、プロンプトで単に「コードを改善して」と曖昧に指示した結果、エージェントが変数名をすべて自分勝手に変更してしまい、PRレビューが不可能になるという痛い失敗も経験しました。AIにコードを任せる際、**「最小限の変更（Minimal Change）」**と**「ビジネスロジックの修正禁止」**という制約条件は、選択肢ではなく必須の生存戦略です。

## 🙋 よくある質問 (FAQ)

- **Q: AIが無限にコードを修正してコミットする「無限ループ」に陥った場合はどうすればいいですか？**
  - A: 非常に重要な質問です。これを防ぐためには、GitHub Actionsの設定ファイルに必ず `if: github.actor != 'github-actions[bot]'` という条件を追加し、Botのコミットが再びCIをトリガーしないようブロックする必要があります。また、1つのPRにつき最大3回までしか修復を試みないよう、リトライカウンターを設定することをおすすめします。

- **Q: AIはビジネスロジックの深刻なバグも修正できますか？**
  - A: 技術的には可能ですが、推奨しません。ロジックのバグは企画の意図と結びついていることが多く、AIが独自の判断で修正すると、さらに大きな障害を引き起こす恐れがあります。自己修復機能は、徹底して「明らかな構文・型エラー」に限定して適用するのが最も安全です。

- **Q: トークンコストが高額になりませんか？**
  - A: 失敗したファイルのコードとエラーログのみを選別し（Context Windowの最小化）、APIに送信するようパイプラインのスクリプトを最適化する必要があります。また、1日の予算上限（Budget Limit）の設定は必須です。

## 🧬 プロンプトの解剖 (Why it works?)

1.  **シニアエンジニアのペルソナ (Role):** 「不必要な変更を極度に警戒する」という性格を付与することで、AI特有の「過剰修正（オーバーエンジニアリング）」のプレベンションを実現しました。
2.  **明確な行動指針 (MANUAL_INTERVENTION_REQUIRED):** AIが自ら判断するのが難しいロジックエラーの場合、無理に正解を作り出さず、人間の開発者に制御を委ねるための明確なエスケープハッチ（Escape Hatch）を設けています。
3.  **JSON出力フォーマットの規定:** CIパイプラインの次のステップ（修正されたコードの上書きおよび `git commit`）でパースしやすいよう、出力フォーマットを厳格なJSONに固定しました。

## 📊 証明: Before & After

### ❌ Before (入力)

```typescript
// src/utils/calculator.ts
export function addPrice(a: number, b: string) {
  return a + b;
}
```

```bash
# Error Log
TS2365: Operator '+' cannot be applied to types 'number' and 'string'.
```

### ✅ After (結果)

```json
{
  "file_path": "src/utils/calculator.ts",
  "fixed_content": "export function addPrice(a: number, b: number) {\n  return a + b;\n}"
}
```

_（AIが型の不一致エラーを正確に認識し、`b`の型を`number`に修正した後、CIパイプラインがこれをファイルに反映してコミットします。）_
