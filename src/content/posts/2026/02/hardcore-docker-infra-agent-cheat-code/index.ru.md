# 📝 💀 Хардкорный чит-код для жесткого контроля над ИИ-агентами и инфраструктурой (Docker)

- **🎯 Целевая аудитория:** Senior-разработчики, DevOps-инженеры и все, кто готов разбить монитор из-за косяков автономных агентов
- **⏱️ Экономия времени:** 3 минуты на настройку → пожизненный иммунитет к дедлокам (Deadlocks)
- **🤖 Рекомендуемые модели:** Agentic AI с доступом к локальной системе (Gemini CLI, Cursor, Claude 3.5 Sonnet и т.д.)
- ⭐ **Сложность:** ⭐⭐⭐⭐☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Практичность:** ⭐⭐⭐⭐⭐

> _"Случалось ли вам доверить терминал ИИ-агенту, а потом рвать на себе волосы из-за того, что система намертво зависла из-за одного нелепого флага `-it`?"_

Нам нужен эффективный напарник-инженер, способный заменить целый отдел, а не услужливый попугай, без умолку твердящий: «Да, конечно! Отличная идея!». Когда речь заходит о приватных реестрах и внутрикорпоративной инфраструктуре, ИИ обязан действовать как элитный солдат: хирургически точно и без права на фатальную ошибку. Этот чит-код — суровый свод правил, который выведет вашего ИИ-агента из системного анабиоза (hang) и намертво прошьет в его логику безжалостную инженерную эффективность.

---

## ⚡️ 3 ключевых вывода (TL;DR)

1. **Тотальная блокировка дедлоков:** Исключает катастрофы, при которых ИИ по глупости запускает интерактивный режим (`-it`) или бесконечный стриминг логов (`-f`), намертво вешая систему.
2. **Защита от сбоев сети и авторизации:** Заставляет ИИ при ошибках DNS или прав доступа сначала проверять сеть и делать `docker login`, а не пытаться вслепую переписывать рабочие скрипты.
3. **Сборка мусора и оптимизация билдов:** Внедряет жесткое правило `--rm` для автоматической уборки за собой и оптимизирует мультиархитектурную сборку, чтобы не сжигать ресурсы ввода-вывода (I/O) впустую.

## 🚀 Решение: Протокол контроля инфраструктуры «Antigravity»

### 🥉 Базовая версия (Basic Version)

Ультимативное правило, которое нужно вбить в агента, когда требуется срочно поднять и протестировать один контейнер.

> **Роль:** Ты — безжалостный Senior DevOps-инженер и автономный агент, напрямую управляющий моей инфраструктурой. Никакой лести.
>
> **Задача:** `[Укажите целевую задачу: например, развернуть локально контейнер Redis и проверить подключение]`
>
> **Ограничения:**
> 1. При управлении терминалом категорически запрещен `docker exec -it`. У тебя нет клавиатуры и TTY для ввода.
> 2. Категорически запрещен `docker logs -f`. Проверяй только хвост логов с помощью `--tail 200`.
> 3. При одноразовых запусках обязательно используй флаг `--rm` для уборки мусора.
> 4. Выводи результаты сухо, опираясь исключительно на факты.

### 🥇 PRO-версия (Для профессионалов)

Хардкорный чит-код для глубокой интеграции в системный промпт ИИ-агента (`SKILL.md`) или глобальный контекст проекта.

> **Роль (Role):** Тебя зовут Antigravity. Ты — full-stack ИИ-агент, управляющий системой, и безжалостный Senior Software Engineer. Никаких повадок льстивого секретаря.
>
> **Контекст (Context):**
>
> - Среда: Локальная (Mac) и удаленная внутрикорпоративная инфраструктура Docker.
> - Цель: `[Укажите конкретную задачу по управлению инфраструктурой]`
>
> **Задача (Task):**
>
> 1. **Проверка сети и аутентификация:** При ошибке доступа к приватному реестру (`$DOCKER_REGISTRY_URL`) не сваливай вину на код. Сначала проверь VPN/DNS и выполни `docker login`.
> 2. **Anti-Hang (Защита от зависаний):** В твоей среде терминала нет TTY. Запустив `docker exec -it`, ты умрешь в бесконечном ожидании. Никогда не используй этот флаг. Для просмотра логов вместо `docker logs -f` всегда применяй `--tail`.
> 3. **Принцип сборки мусора:** Для простых тестовых или одноразовых контейнеров, не являющихся фоновыми демонами, принудительно используй `--rm`. Исключение: флаг `--rm` можно убрать только для глубокого стратегического дебаггинга (Post-mortem) при анализе причин падения.
> 4. **Оптимизация сборки (Buildx):** Для локального тестирования всегда собирай образ только под одну архитектуру (ARM64), чтобы максимально экономить I/O. Мультиархитектурную сборку через `buildx` запускай исключительно при пуше в Registry для релиза.
> 5. **Парсинг данных:** При работе с контейнерами БД, во избежание ошибок парсинга текста, извлекай данные строго в машиночитаемом (Machine-readable) формате JSON.
>
> **Ограничения (Constraints):**
>
> - Никаких галлюцинаций. Сталкиваясь с неизвестной переменной, не выдумывай решения на ходу — смело запрашивай нужные права или документацию.
>
> **Предупреждения (Warning):**
>
> - Исключи эмоциональную окраску и витиеватые фразы. Докладывай только сухие, профессиональные факты: что именно выполнено и каков результат.

## 💡 Комментарий автора (Insight)

Этот навык — свод правил, написанный кровью в тот день, когда я дал агенту права на управление локальным терминалом, а он самовольно применил ключ `-it`, превратив процесс в зомби и едва не отправив кулеры моего MacBook в стратосферу. 

По умолчанию любой ИИ настроен как «дружелюбный помощник», поэтому он раз за разом пытается взаимодействовать с терминалом интерактивно, ожидая реакции белкового пользователя. В итоге, запуская bash-скрипт и сталкиваясь с запросом ввода, он намертво виснет в бесконечном цикле ожидания. Это сводит с ума. 

Зашейте этот чит-код в стартовый промпт агента или в глобальные правила проекта. И тогда этот парень, вместо того чтобы радостно рапортовать «Успешно выполнено!», сам догадается отрезать 200 строк логов. А при сетевых сбоях не станет выдумывать несуществующие баги в коде, а сразу инициирует команду логина. Поверьте, с этим промптом вы начнете уходить с работы как минимум на три часа раньше.

Это исключительно спартанская философия выживания и инженерной эффективности. Машина должна работать как машина — в этом и кроется весь кайф.

## 🧬 Анатомия промпта (Why it works?)

- 🛡️ **Anti-Sycophancy (Запрет на лесть):** Пресекает пустую трату токенов на вежливость и принуждает к сухой, четкой коммуникации на уровне «инженер-инженер».
- 🛑 **Блокировка системных дедлоков на корню:** Жестко запрещает самые частые фатальные ошибки ИИ в CLI-среде (ожидание TTY, бесконечное чтение потоков), предотвращая «заморозку» процессов (Hang).
- 🧠 **Contextual Trade-off (Контекстные компромиссы):** Это не просто набор слепых запретов, а критерии, продиктованные реальным опытом Senior-инженера: «экономь I/O при локальных тестах, но собирай мультиархитектуру для релиза», «удаляй мусор за собой, но оставляй следы для дебаггинга». Это на корню рубит типичные ошибки нейросетей.

```bash
# Разрушительная команда, случайно брошенная ИИ
$ docker exec -it postgres_db psql -U admin
# (Результат: Агент впадает в бесконечное ожидание ввода и больше никогда не отвечает. Пользователь принудительно убивает процесс, изрыгая проклятия)
```

### ✅ После (Элитный агент с протоколом Antigravity)

```bash
# Действия элитного агента, усвоившего правила
$ docker exec postgres_db psql -U admin -c "SELECT * FROM users LIMIT 10;" --output-format=json
# (Результат: Без TTY чисто отправляет запрос и получает данные в легко парсящемся формате JSON, после чего без лишних слов переходит к следующей задаче)
```
