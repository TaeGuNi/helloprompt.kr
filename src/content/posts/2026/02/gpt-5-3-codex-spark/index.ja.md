## 🙋 よくある質問 (FAQ)

- **Q: 既存のGitHub CopilotやClaude 4.5 Sonnetと比較して、実際のところどうですか？**
  - A: ゼロベースから複雑なアルゴリズムを構築するような「純粋な論理推論」の領域においては、現時点でもClaude 4.5 Sonnetに軍配が上がるケースがあります。しかし、IDEとシームレスに連携し、リアルタイムでコードを生成しながら自己完結的にエラーを修正していく**「実務におけるアジリティ（俊敏性）」**という点では、平均レイテンシ45msを叩き出すSparkの右に出るものはありません。

- **Q: 社外秘のソースコードが外部のAIサーバーに流出しないか、セキュリティ面が心配です。**
  - A: エンタープライズ（Enterprise）プランを導入することで、社内のローカルネットワーク、あるいはVPC（プライベートクラウド）内でのみデータ処理が完結するよう厳密に設定可能です。高い機密性が求められる金融機関や、ネットワーク分離が必須の環境でも安心して実戦投入できます。

- **Q: 200万トークンという巨大なコンテキストを扱うと、プロジェクトの規模に比例して動作が重くなりませんか？**
  - A: その心配は無用です。Sparkに搭載されている「Neural Sparkアーキテクチャ」は、ファイル間の複雑な依存関係グラフをバックグラウンドで事前キャッシュし、変更が加わった部分のみを差分（Delta）計算する仕組みを採用しています。そのため、巨大なモノレポ（Monorepo）環境であっても、パフォーマンスの低下を感じることはほぼありません。

1. **超広角コンテキスト活用の強制:** AIに単なるコードの書き換えを頼むのではなく、`[副作用の予測]`というタスクを明示的に与えることで、Sparkに眠る「200万トークンの全体スキャン能力」を強制的に100%アクティブ化させています。

3. **ハルシネーションの確実な防止 (Constraints & Warning):** AIが独断で不要なnpmパッケージをインストールし、プロジェクト全体のビルドを崩壊させるという、LLM特有の致命的な悪癖をプロンプトの段階で完全に封じ込めています。

```javascript
// 既存のレガシーコード (O(n^2)の計算量による深刻なボトルネックが発生)
function findDuplicateUsers(users) {
  let duplicates = [];
  for (let i = 0; i < users.length; i++) {
    for (let j = i + 1; j < users.length; j++) {
      if (users[i].email === users[j].email) {
        if (!duplicates.includes(users[i])) {
          duplicates.push(users[i]);
        }
      }
    }
  }
  return duplicates;
}
```

### ✅ After (結果)

```javascript
// GPT-5.3 Codex Spark リファクタリング結果 (O(n)の計算量を達成)
const findDuplicateUsers = (users) => {
  const emailCounts = new Map();

  for (const user of users) {
    emailCounts.set(user.email, (emailCounts.get(user.email) || 0) + 1);
  }

  const duplicates = [];
  const addedEmails = new Set();

  for (const user of users) {
    if (emailCounts.get(user.email) > 1 && !addedEmails.has(user.email)) {
      duplicates.push(user);
      addedEmails.add(user.email);
    }
  }

  return duplicates;
};

// ⚠️ Spark Warning:
// このリファクタリングにより、配列の順序が元と異なる可能性があります。
// UserList.tsxの42行目で、レンダリング順序に影響を受けないか必ず確認してください。
```
