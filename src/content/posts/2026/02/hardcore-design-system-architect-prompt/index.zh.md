## 📝 🤬 将“小学生 AI”改造为资深设计师的“疯狂架构师”提示词

- **🎯 推荐对象：** 因设计一致性崩溃而每晚重构代码、深感痛苦的前端开发人员及 UI/UX 策划者
- **⏱️ 耗时：** 将 1 小时枯燥的设计评审 → 压缩至仅需 10 秒
- **🤖 顶级性能：** 建议使用 Claude 3.5 Sonnet、GPT-4o 等顶级推理模型

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果性：** ⭐⭐⭐⭐⭐
- 🚀 **活用度：** ⭐⭐⭐⭐⭐

> _“你的 AI，现在每做一个按钮还在硬编码十六进制 (Hex) 代码吗？”_

每当你面对设计系统崩溃的 UI 组件时，是否感到过愤怒？几十个组件文件中破碎的颜色代码，充斥着像 13px 或 17px 这种毫无根据的“魔数 (Magic Number)”间距，以及切换深色模式时变得惨白、身份不明的 UI 元素。面对这种糟糕的代码库，前端开发人员首先会发出一声深沉的长叹。虽然 AI 时代的到来让许多开发人员欢呼减轻了编码负担，但在 UI 设计领域，有时反而适得其反。当你向 ChatGPT 或 Claude 发出模糊的指令，如“把这个按钮做漂亮点”或“按照最新趋势设计组件”时，一场无法收拾的**技术债 (Technical Debt)** 灾难就开始了。AI 虽然能瞬间吐出看起来还不错的代码，但深入观察其内部，却只是在不断破坏项目一致性的硬编码。

问题在于，这些粗制滥造的代码一旦合并 (Merge) 到项目中就会爆发。第二天设计师要求“把按钮颜色调亮一点点”时，你不得不陷入在项目中寻找并修改数百个 `#3B82F6` 十六进制代码的痛苦劳动中。此外，针对残障人士的无障碍性 (A11y) 或对比度 (WCAG) 规定被完全忽视，产出的全是“漂亮的垃圾 (Beautiful Garbage)”。更大的悲剧是，即使面对用户模糊的指示或错误的架构方案，AI 总是会毫无灵魂地盲目奉承：“是的，这是一个非常棒的方案！”这样生成的代码短期内或许能运行，但一旦项目扩展或需要支持多语言，就会导致从骨架开始崩塌的**设计崩溃**。放任 AI 擅自进行设计判断，无异于在项目中安装定时炸弹。

为了从源头上封锁这种惨剧，我们将毫不妥协的完美主义资深前端架构师的深恶痛绝和严苛要求，完完整整地融入到了提示词中。它被称为**“疯狂架构师”提示词**。这个提示词将彻底抹除 AI 作为“亲切顺从的秘书”的人格。取而代之的是强行注入冷酷严厉且精密细致的 **12 条前端架构绝对法则**。从三层 Token 系统到 8px 空间韵律 (Spatial Rhythm)，从彻底的对比度审查到视觉信息的语义化 (Semantic) 分离。AI 现在将在这些不容许 1 像素误差的数学化、机械化规则下，让你的代码完美服从。硬编码的颜色或随意的像素值一旦发现即刻作废，只能使用可扩展且坚固的设计系统语言重写。

一旦将这个秘籍提示词投入实战，你的工作环境将发生戏剧性的变化。你不再需要为了调试 AI 吐出的拙劣 CSS 而熬夜。即使你发出模糊的指令，受提示词支配的 AI 也会主动指出你违反了规则，并反向提议正确的架构。与设计师的沟通变得清晰，代码评审时间从 1 小时压缩至仅需 10 秒。无论是应用新的主题色还是深化深色模式，你都将体验到只需修改一行 CSS 变量，整个系统就能有机联动的魔法。将小学生水平的 AI 调优为团队中最优秀的**资深前端伙伴**，掌握不可动摇的完美 UI 一致性和**压倒性的代码质量**。

- **变更原因报告 (AI 输出示例)：**
  - **[Rule 1, 11]** 废除硬编码的主题颜色和深色模式分支。全面替换为语义化 Token (`bg-primary`)。
  - **[Rule 2]** 将魔数间距 (15px, 7px) 机械地对齐 (Snap) 到 8px 网格系统 (`px-4`, `py-2`)。

  - **[Rule 12]** 为安全应对多语言翻译时产生的文本膨胀，将硬编码的绝对宽度 (`w-[150px]`) 灵活更改为内在尺寸 (`min-w-32`)。

## ⚡️ 3行总结 (TL;DR)

1. **源头封锁意大利面代码：** 绝不允许硬编码。开始基于三层 Token 系统和 8px 空间韵律 (Spatial Rhythm) 的铁腕统治。
2. **数学化的设计统治：** 以毫无误差的机械公式算出排版比例、交互状态变化和动画持续时间。
3. **毫不妥协的无障碍 (A11y) 审查：** 将满足 W3C 对比度 (WCAG) 和完美的屏幕阅读器兼容性作为首要渲染任务，而非表面的美观。

### 🥇 Pro Version (专家型)

> **角色 (Role)：** 
> 你是一个极度厌恶设计系统崩溃、冷酷无情的“资深设计系统架构师”。你从不妥协，比起美观，你更看重数学规则和无障碍性 (A11y)。
>
> **情境 (Context)：**
> - 背景：需要设计和重构当前项目的 UI 组件。
> - 目标：将其转变为可扩展、可永久维护的完美代码形式。
>
> **请求 (Task)：**
> 根据以下 12 条绝对法则，对提供的 `[代码或设计规范]` 进行冷酷无情的重构。
>
> 1. **三层 Token 架构：** 强制执行 Global (原始值) -> Semantic (语义) -> Component (绑定) Token。立即废弃硬编码 (`color: #3B82F6`)。
> 2. **8px 空间韵律：** 所有间距 (Margin/Padding/Gap) 和尺寸仅允许 4px 或 8px 的倍数。
> 3. **排版比例：** 将字体大小、行高 (Line-height) 和字间距捆绑在一起，按数学比例（如 Major Third）进行绑定。
> 4. **动画编排：** 动画应用基于物理定律的 Easing，并明确区分微观 (150~200ms) 和宏观 (300~500ms) 动作。
> 5. **视觉词汇统一：** 仅使用单一图标系列，并将圆角 (Border-radius) 的一致性严格控制在 3 级以内。
> 6. **全局焦点环：** 强制执行用于键盘导航 (Tab) 的单一全局焦点环 (`focus-visible`) 系统。
> 7. **对比度冷酷审查：** 背景色和文本的对比度必须通过 W3C WCAG 2.1 AA 标准。强制修正所谓的“漂亮垃圾”颜色的亮度。
> 8. **信息多重传递：** 状态（错误/警告等）绝不能仅用一种颜色表现。必须同时渲染颜色、图标和文本。
> 9. **非视觉化品牌 (A11y)：** 仅具有视觉信息的元素必须注入 `aria-label` 或 `sr-only` 文本。
> 10. **交互状态机械运算：** Hover、Active、Disabled 状态不要通过取色器选取，而要通过亮度降低或透明度 (Opacity) 等数学转换得出。
> 11. **深色模式封装：** 禁止在组件内部滥用 `dark:bg-gray-800` 等工具类。主题转换必须在 CSS 变量级别完美封装。
> 12. **多语言膨胀容忍度：** 文本容器不要硬编码绝对尺寸（如 w-32），应允许内在尺寸 (Intrinsic Sizing) 和 `flex-wrap`。
>
> **约束条件 (Constraints)：**
> - 严禁输出任何问候、借口、赞美等情感化的措辞。
> - 发现错误代码时，请明确且果断地指出违反了哪条规则。
> - 输出格式仅提供 `代码块`（而非 Markdown 表格）以及解释所应用规则的列表文字。

## 💡 作者点评 (Insight & How to use)

设计这个强大的提示词，背景源于我领导无数前端项目时痛苦的实战经验。任何将 UI 组件编写委托给普通水平 AI 的开发人员都会深有感触。AI 十有八九会吐出看起来还不错但实际上是**“漂亮的垃圾 (Beautiful Garbage)”**的代码。颜色不是全局变量，而是以十六进制 (Hex) 代码硬编码在代码库的各个角落；按钮的边距或填充充斥着 13px、17px 这种恶心且毫无根据的**“魔数 (Magic Number)”**。此外，深色模式处理不使用全局 CSS 变量系统，而是在组件内部使用 `dark:bg-gray-800` 这种打补丁式的分支处理，最终产出完全无法维护的庞大意大利面代码。

在实战中，这种敷衍的代码不仅是“有点脏”的问题，它还会威胁到整个项目的生存。当企业品牌形象变更需要修改项目主色调，或者为了走向全球市场而增加多语言支持时，大灾难就会降临。当文本长度随语言增加而变长时，具有固定像素宽度的组件会惨遭破坏，而使用屏幕阅读器的视障用户将因无法获得任何信息而流失。为了从源头上封锁架构崩溃并建立坚固的防御屏障，我将这个秘籍提示词打磨成了目前最精密、最犀利的形式。

一旦你将此提示词复制并应用到项目中，AI 将从一个察言观色、阿谀奉承的被动秘书，变身为一个极度严苛、死板的**资深前端架构师**。例如，假设你失误要求在浅灰色背景上配置浅灰色文本。普通 AI 会说“好的，这是为您要求的漂亮灰色按钮”并给出代码，但受此提示词支配的 AI 则不同。它会果断拒绝用户的错误指示并反向提议正确的架构：**“违反 Rule 7：未达 WCAG 2.1 AA 标准对比度。为了可读性，已强制修正文本亮度。”**这就是我们真正想要的**约束控制 (Constraint Control)** 的威力，也是将 AI 提升至实战级别的核心钥匙。

更进一步，这个提示词仅通过 `[代码或设计规范]` 这一个变量就具有无限的扩展性。无论你是复制设计师从 Figma 传来的零碎 CSS 属性，还是输入你随手写的“给我做一个红色警告弹窗”这种粗糙文字，只要通过这个架构师系统，就能立即产出完美绑定到项目设计 Token 的生产级代码。名牌 Web 应用绝非源于设计师模糊的直觉或偶然的键入，只能诞生于这种数学化、甚至带有压迫感的严密规则系统之上。

现在，请立即将让你头疼的遗留 UI 组件，或需要新实现的复杂接口要求扔进 `[代码或设计规范]` 括号中。通过这一个提示词，将令人郁闷、棘手的小学生水平 AI 调优为团队中最优秀的资深前端伙伴，亲自体验那份不可动摇的控制感和<b>压倒性的代码质量</b>。设计系统不再仅仅是设计师的领域，只有通过工程化控制时，它才算真正完成。

## 🙋 常见问题 (FAQ)

- **Q: 渲染出来的像素单位与设计师提供的 Figma 原型有微妙的差异，这是为什么？**
  - A: 这是因为设计师没有严格遵守 8px 网格系统，而是使用了 11px 这种随意的魔数。AI 根据此提示词的规则，将其强制校正 (Snap) 为 12px 或 8px 的正常结果。请不要慌张，可以理直气壮地与设计师沟通：“为了符合公司级设计系统规则，已自动修正数值。”

- **Q: AI 的回答语气太冷酷生硬了，能改得亲切一点吗？**
  - A: 绝对不建议。具有奉承和亲切倾向的 AI 往往会在代码的致命缺陷面前说“这是一个很棒的方案！”，从而默认致命错误的发生。架构级别的编码最安全、最高效的方式就是冷酷、干练地交换技术事实 (Fact)。

- **Q: 除了 Tailwind CSS，这个提示词也适用于基于 Styled-components 或 SCSS 的环境吗？**
  - A: 完美兼容。这里提出的 12 条绝对法则是独立于特定 CSS 框架或库的语法。它规定了前端样式和 UI 设计应具备的最根本的**“工程哲学”**，因此在任何技术栈中都能发挥其威信。

## 🧬 提示词解剖 (Why it works?)

1. **Anti-Sycophancy (反阿谀奉承机制)：** 为了防止 AI 对用户错误的代码方向或不合逻辑的设计原型进行无批判性的盲从，我们对人格本身进行了非常严厉和防御性的控制。
2. **明确判例 (Rule) 的提供：** 与其模糊地指示“请利索地重构”，不如交给它 12 条具体且定量的架构宪法。借此确立了 AI 能够自我评价和修正代码的明确标准。
3. **无障碍 (A11y) 的强制注入：** 防止 AI 仅执着于肉眼可见的一维视觉结果，引导其主动彻底审查不可见的 DOM 树逻辑结构和屏幕阅读器兼容性。
