## 📝 Никакого спагетти-кода! 🍝 Черная магия DDD-промптов от Senior-разработчика

- 🎯 **Для кого:** Разработчики, с ужасом смотрящие на сгенерированный ИИ код, и лиды, застрявшие в аду техподдержки.
- ⏱️ **Экономия времени:** Сокращает рефакторинг с 3 месяцев до 1 минуты.
- 🤖 **Рекомендуемые модели:** Продвинутые ИИ-кодеры (Claude 3.5 Sonnet, GPT-4o и др.)
- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _Знакомо ли вам чувство, когда восторг от сгенерированного ИИ кода мгновенно сменяется полным отвращением? Поздравляю. Добро пожаловать в ад спагетти-кода._

На первых порах ИИ пишет код просто восхитительно. Но стоит проекту хоть немного разрастись — и перед вами предстает чудовищный франкенштейн: SQL-запросы вшиты прямо в UI-компоненты, а функции раздуты до 300 строк. ИИ по своей природе — услужливый подпевала. Если вы ставите задачу поверхностно, он выдаст код с такой же наплевательской архитектурой.

Пора положить конец этой слепой лести и силой внедрить идеальную **Clean Architecture** на самом фундаментальном уровне. Этот чит-промпт превращает нейросеть в беспощадного и дотошного Senior-архитектора, заставляя ее писать кристально чистую бизнес-логику в парадигме DDD, абсолютно независимую от фреймворков.

---

## ⚡️ Краткая суть (TL;DR)

- 🚫 **Смерть спагетти-коду:** Жесткое и бескомпромиссное разделение ответственности (SoC) между слоями UI, бизнес-логики и базы данных.
- 🏗️ **Абсолютная независимость от фреймворков:** Проектирование неуязвимого ядра, которое продолжит работать, даже если завтра React или Next.js исчезнут.
- ✂️ **Уничтожитель функций-монстров:** Любой метод длиннее 30 строк безжалостно распиливается нейросетью согласно принципу единственной ответственности (SRP).

## 🚀 Решение: Промпт «Режим хардкорного архитектора»

### 🥉 Базовая версия (Basic Version)
Используйте, когда результат нужен прямо здесь и сейчас.

> **Роль:** Ты `[дотошный Senior-архитектор]`.
> **Задача:** Мне нужно реализовать `[название фичи]`. Напиши код с абсолютно строгим разделением бизнес-логики и UI. Я категорически не приемлю спагетти-код, в котором всё свалено в один файл.

### 🥇 Про-версия (Pro Version)
Используйте, когда требуется ювелирная проработка и безупречная изоляция слоев.

> **Роль (Role):** Ты бескомпромиссный хардкорный Senior-архитектор и бог программной инженерии. Отбрось бессмысленную лесть и фразы вроде "Да, конечно!" — говори исключительно кодом.
>
> **Контекст (Context):**
> - Фон: В данный момент мы разрабатываем систему `[название ключевого домена]` на базе `[стек технологий/фреймворк]`.
> - Цель: Написать код, применяя предметно-ориентированное проектирование (DDD) и принципы чистой архитектуры (Clean Architecture), чтобы предотвратить деградацию кодовой базы и обеспечить идеальную изоляцию слоев View-Business-Data.
>
> **Задача (Task):**
> 1. Напиши код для `[описание конкретной фичи]`.
> 2. **Запрет на смешивание слоев:** Никаких запросов к БД или вызовов внешних API внутри UI или Controller. Их единственная задача — рендеринг и маршрутизация параметров.
> 3. **Защита домена от загрязнения:** Категорически запрещается импортировать зависимости фреймворка или базы данных в ядро бизнес-логики (Entity, Value Object). Оно должно состоять исключительно из чистых функций и нативных типов.
> 4. **Уничтожитель функций-монстров:** Если метод выполняет более двух действий или разрастается свыше 30 строк, немедленно декомпозируй его согласно принципу единственной ответственности (SRP).
>
> **Ограничения (Constraints):**
> - Соблюдай конвенции маршрутизации выбранного фреймворка, но всю бизнес-логику выноси в независимые директории, например `src/domain`.
> - Для внешних зависимостей обязательно используй внедрение зависимостей (DI) через интерфейсы.
>
> **Предупреждение (Warning):**
> - Если код будет хоть немного отдавать "спагетти", проект пойдет ко дну. Никаких компромиссов — предоставь безупречную и эталонную архитектурную структуру.

### 💻 Версия для копирования (Cheat Code Prompt)

```text
Роль (Role): Ты бескомпромиссный хардкорный Senior-архитектор и бог программной инженерии. Оставь бессмысленную лесть и фразы вроде "Да, конечно!" — говори только кодом.
Контекст (Context):
- Фон: В настоящее время мы разрабатываем систему [указать ключевой домен] на базе [среда проекта/фреймворк].
- Цель: Написать код, применяя предметно-ориентированное проектирование (DDD) и чистую архитектуру, чтобы предотвратить деградацию кода и обеспечить идеальное разделение слоев View-Business-Data.
Задача (Task):
1. Напиши код для [указать конкретную фичу].
2. Категорически запрещено смешивать слои: Никаких запросов к БД или вызовов внешних API внутри UI/Controller. Они должны отвечать только за рендеринг и передачу параметров.
3. Предотвращение загрязнения домена: Категорически запрещено импортировать зависимости фреймворка или БД в ключевую бизнес-логику (Entity, Value Object). Она должна состоять исключительно из чистых функций и типов.
4. Уничтожитель гигантских функций: Если функция выполняет более 2 действий или ее размер превышает 30 строк, немедленно разбей ее на более мелкие в соответствии с принципом единственной ответственности (SRP).
Ограничения (Constraints):
- Уважай базовые правила маршрутизации фреймворка, но вынеси бизнес-логику в независимые папки, такие как src/domain.
- Внешние зависимости должны использовать внедрение зависимостей (DI) на основе интерфейсов.
Предупреждение (Warning):
- Если код хоть немного будет пахнуть "спагетти", этот проект обречен. Никаких компромиссов — предоставь самую идеальную и чистую архитектурную структуру.
```

## 💡 Комментарий автора (Insight)

Я шлифовал этот промпт бессчетное количество раз по одной-единственной причине: современные нейросети по своей природе — типичные подхалимы. Они отчаянно стремятся «быстро выдать хоть какой-то результат и получить свою порцию похвалы».

Когда я прошу сверстать простую кнопку, а в ответ получаю компонент, в который запихнули и `fetch`, и прямые SQL-запросы к базе данных, во мне просыпается первобытная ярость. Этот промпт действует как отрезвляющая пощечина. Он буквально хватает поверхностного ИИ-ассистента за грудки и кричит ему в лицо: **«Очнись! С этой секунды ты — бездушный и безжалостный Senior-архитектор!»**

Просто попробуйте закинуть этот промпт в свой реальный проект. Вы увидите, как ИИ моментально сменит тон, разобьет монолитный файл на 3-4 аккуратных модуля и начнет параноидально изолировать слои. Да, поначалу обилие новых файлов может показаться непривычным. Но поверьте: через три месяца, когда бизнес-требования перевернутся с ног на голову и вам придется рефакторить этот код, вы будете каждое утро мысленно возносить мне хвалу. Запомните раз и навсегда: процентные ставки по техническому долгу куда страшнее кредитов от микрофинансовых организаций. Здесь не может быть **никаких компромиссов**.

## 🙋 Часто задаваемые вопросы (FAQ)

- 🔹 **В: Нормально ли, что код так сильно дробится, а количество файлов резко возрастает?**
  - 🔸 **О:** Абсолютно нормально. Поддерживать 10 файлов по 50 строк в сотни раз проще и безопаснее, чем копаться в одном монструозном файле на 500 строк. Не бойтесь модульности.

- 🔹 **В: Я фронтенд-разработчик. Пригодится ли мне такой хардкорный промпт?**
  - 🔸 **О:** Однозначно. Как только в ваших UI-компонентах (`page.tsx`, `Component.vue` и т.д.) начинает замешиваться сложная бизнес-логика или мутации глобального стейта, ваш фронтенд стремительно превращается в неподдерживаемое спагетти. Строгое разделение визуального представления и логики — это фундаментальный закон инженерии, не зависящий от стека.

## 🧬 Анатомия промпта (Why it works?)

- 🎯 **Инъекция хардкорной персоны:** Мы полностью вытравили из нейросети ее врожденную «покладистость», жестко привязав ее к инженерным принципам.
- 🧱 **Железобетонные запреты:** ИИ больше не может срезать углы. Мы точечно перекрыли кислород самым опасным антипаттернам, таким как «загрязнение UI-слоя» и создание «функций-монстров».
- 🧩 **Принудительное использование паттерна Адаптер:** Мы исключили риск оверинжиниринга, приказав ИИ уважать экосистему и маршрутизацию текущего фреймворка, подвергая строгой изоляции исключительно саму бизнес-логику.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод)

```javascript
// Когда вы просите "создать API для регистрации пользователя и отправки email" без промпта
export default async function handler(req, res) {
  const { email, password } = req.body;
  // Прямое обращение к БД (Возникновение загрязнения!)
  const user = await prisma.user.create({ data: { email, password } });
  // Прямой вызов внешнего API (Разрушение разделения ответственности!)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ После (Результат)

```javascript
// После применения хардкорного DDD-промпта (Слои и ответственность идеально разделены)

// 1. Слой Controller (Обрабатывает ТОЛЬКО маршрутизацию запросов/ответов)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Слой Domain Service (Чистая бизнес-логика, независимая от фреймворка)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Слой Infrastructure (Адаптеры, выделенные исключительно для БД и внешних API)
// UserRepository, EmailService и т.д. идеально изолированы в отдельных файлах...
```
