# 🦀 WebAssembly(Wasm)ポーティング：Webパフォーマンスの限界を突破しよう

- **🎯 おすすめの対象:** フロントエンドのパフォーマンス最適化が急務な開発者、重いバックエンドロジック(C++/Rust/Go)をブラウザに移行したいエンジニア
- **⏱️ 所要時間:** 何日もかかっていたバインディングの試行錯誤 → わずか5分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コード変換において圧倒的)、GPT-4o

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **実用性:** ⭐⭐⭐☆☆

> _「JSの最適化で何日も徹夜してやっと10%速くなった演算が、Wasmにポーティングしたら一瞬で10倍速くなりました。でも、その『ポーティング』自体はどうやったら簡単にできるのでしょうか？」_

ブラウザはもはや単なるドキュメントビューアではありません。画像処理、複雑な暗号化、大規模な数値計算など、JavaScript(JS)のV8エンジンだけでは限界にぶつかる瞬間が必ず訪れます。WebAssembly(Wasm)は完璧な解決策ですが、既存のC++やRustコードをWasmに変換し、JSとメモリを共有してバインディング(Binding)するプロセスには、途方もない試行錯誤が伴います。

このプロンプトは、単なるコードの翻訳にとどまらず、メモリリークの防止からJSブリッジコードの作成までを一度に行ってくれる、完璧な「Wasmポーティングアシスタント」の役割を果たします。

---

## ⚡️ 3行まとめ (TL;DR)

1. **言語の壁を破壊:** C++、Rust、Goなどの既存のレガシーコードを、Webで実行可能なWasmモジュールへ即座に変換します。
2. **バインディングコードの自動化:** JSとWasm間のデータ型マッピングや、メモリ共有(Shared Memory)のためのコードを自動で作成します。
3. **型安全性の保証:** フロントエンドですぐに利用できるように、完璧なTypeScriptインターフェース(`.d.ts`)を併せて提供します。

## 💡 筆者コメント (Insight)

WebAssemblyへのポーティングを試みる際、開発者が最も挫折しやすいのが**「JSの配列をWasmにどう渡し、どう受け取るのか？」**という部分です。単純な数値や文字列の受け渡しは簡単ですが、大容量の画像バッファやオーディオデータ(Float32Array)を扱う場合、メモリを直接制御する必要があるからです。

このプロンプトの核心は、`タスクの2番（メモリアーキテクチャ）`にあります。AIに単にコードを書いてもらうだけでは、メモリを丸ごとコピーして渡すような、非常に非効率なコードを出力することがよくあります。しかし、上記のプロンプトのように「コピーコストの最小化」「共有メモリの活用」を明示することで、AIが `wasm-bindgen` の強力な機能（ポインタを介したメモリへの直接アクセスなど）を活用した「現場で本当に使える実践的なコード」を作成してくれます。WebpackやVite環境でWasmをロードする際に発生しがちな非同期タイミングの問題も、Reactのサンプルコードを通じて一挙に解決できます。

## 🙋 よくある質問 (FAQ)

- **Q: 既存のコードが膨大すぎて、一度にコピーするのが難しいのですが、どうすればいいですか？**
  - A: プロジェクト全体を一度に入力するのではなく、コアロジックが含まれる「主要な関数単位」に分割してプロンプトを実行することをお勧めします。依存関係の強いモジュール構造の場合は、`[状況 (Context)]` に「このモジュールはこのようなアーキテクチャを持っている」と事前に説明を加えることで、AIが文脈を理解し、より正確なバインディングを生成してくれます。

- **Q: 変換されたRustコードはどのようにビルドするのですか？**
  - A: `wasm-pack` という公式ツールチェーンを使えば非常に簡単です。プロンプトの出力結果の最後に、通常AIがビルドコマンド（例: `wasm-pack build --target web`）を一緒に教えてくれるので、それをそのままコピーしてターミナルで実行するだけで、すぐに使えるpkgフォルダが生成されます。

- **Q: C++コードをEmscriptenでポーティングするのと、Rust + wasm-bindgenの違いは何ですか？**
  - A: EmscriptenはC/C++プロジェクトを丸ごとWebに移植するのには強力ですが、ビルド後のファイルサイズが大きくなりがちで、JSとの相互作用がやや無骨です。一方、Rust + `wasm-bindgen` はバンドルサイズが非常に小さく、TypeScriptエコシステムとの互換性が圧倒的に優れているため、最近のフロントエンド界隈では事実上の標準（デファクトスタンダード）として定着しています。そのため、このプロンプトでもRustへの変換を強く誘導するように設計されています。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **エンドツーエンド(E2E)ソリューション志向:** 単にコードをターゲット言語に翻訳するだけでなく、フロントエンドフレームワーク(React等)でそれをどうロードしてレンダリングするかまで要求することで、「実際に動くプロダクション用の成果物」を作り出します。
2. **明示的なパフォーマンス制約 (Constraints):** Wasm導入の主な目的は「パフォーマンス」です。「コピーの最小化」「シリアライズの回避」といったキーワードを通じて、AIがパフォーマンス低下を引き起こすような「安易な回避策」を選べないように強制しています。
3. **型安全性 (Type Safety):** JSの世界とWasmの世界の間の曖昧な型の境界線を、`.d.ts` の自動生成を通じて明確に規定し、開発段階でのランタイムエラーを完全に防ぎます。

## 📊 証明：Before & After

単に「これをWasmに変えて」とリクエストした時と、Proバージョンのプロンプトを使用した時では、生成されるコードの質に雲泥の差があります。

### ❌ Before (単純なリクエスト時のAIの回答)

```rust
// データをシリアライズして丸ごとコピーする、非効率的なアプローチ
#[wasm_bindgen]
pub fn apply_filter(data: Vec<u8>) -> Vec<u8> {
    let mut result = data.clone(); // 莫大なメモリコピーが発生
    // ... フィルタリングロジック ...
    result
}
```

### ✅ After (Proプロンプト使用時のAIの回答)

```rust
// メモリポインタを活用してコピーなしで直接アクセス (Zero-copy)
#[wasm_bindgen]
pub fn apply_filter_in_place(data: &mut [u8], width: usize, height: usize) {
    // JSのUint8Arrayが指すWasmのリニアメモリに直接アクセスして操作
    for i in (0..data.len()).step_by(4) {
        // Red, Green, Blueチャンネルに直接フィルタを適用
        let r = data[i];
        let g = data[i + 1];
        let b = data[i + 2];

        data[i] = (r as f32 * 0.393 + g as f32 * 0.769 + b as f32 * 0.189) as u8;
        // ... (中略) メモリ割り当て/解放のオーバーヘッドを完全に排除
    }
}
```
