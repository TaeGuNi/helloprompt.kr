## 📝 💣 让 AI 化身 Netflix 混沌猴子的硬核测试提示词秘籍

- **🎯 推荐对象：** 厌倦了只会写“正常流程 (Happy Path)”测试代码的资深开发者，以及没有导师指导、孤军奋战的初级开发者。
- **⏱️ 所需时间：** 熬夜调试 → 缩短至 1 分钟
- **🤖 最佳性能：** Claude 3.5 Sonnet, GPT-4o（平庸的模型无法驾驭这些强力规则）

- ⭐ **难度：** ⭐⭐⭐⭐☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"大部分开发者在委托 AI 编写测试代码时都会犯下一个致命错误——那句天真的‘帮我写个能运行的代码’，往往会导致发布当天凌晨 3 点的惨烈灾难。"_

当我们请求 AI 编写测试代码时，我们期待的是什么？是完美的边缘情况处理、细致的错误处理，以及在任何异常情况下都能防止系统崩溃的坚实盾牌。但现实往往残酷。一句天真的“帮我写测试代码”，AI 可能会生成超过 200 行像温室花草一样的**正常流程 (Happy Path)** 代码。它假设所有的 API 都会在 1 秒内返回 200 OK，数据库永远不会断开，内存泄漏根本不存在，就像在一个美丽的乌托邦。

这种“没有灵魂的应声虫”代码一旦部署到生产环境，真正的噩梦就开始了。现实世界中不存在 100% 响应的外部 API，也不存在永不断开的数据库连接。因突发流量导致的**网络超时**、第三方服务的 500 Internal Server Error，或是云基础设施的临时故障随时可能发生。在这种时刻，AI 为你编写的精美测试代码不过是废纸一张。最终，你不得不在周五晚上或周末凌晨 3 点被 PagerDuty 报警唤醒，一边流着血泪一边亲自调试烂摊子。这哪里是自动化，这简直是在系统内部埋设定时炸弹。

在将 AI 引入工作流时，许多开发者陷入了幻想：“现在烦人的测试代码编写工作交给 AI 就能搞定了吧？”这种想法过于天真。一旦打开结果，你往往会感到惊愕。AI 编写的代码看起来运行得很好，但只要剥开外壳，就会发现到处是致命缺陷。完全找不到对**边缘情况 (Edge Case)** 的考量，异常处理往往只用一行 `console.log(error)` 敷衍了事。这就像在一座没有骨架的建筑上刷了层华丽的油漆。稍微遇到点风雨——即无法预料的用户行为或外部服务器的微小延迟，整个系统就会像多米诺骨牌一样崩塌。

更可怕的是，这些“虚假成功”的代码在测试覆盖率工具中会显示 100% 🟢 (Green)。SonarQube 或 Jest 的覆盖率报告不会发出任何警告。管理层和团队负责人会误以为已经建立了完美的防御，而这颗炸弹却留在了前线人员的背后。在流量巨大的活动日，只要支付网关的 API 延迟 3 秒，数据库连接池就会枯竭，最终引发整个服务瘫痪的惨烈连锁反应。到那时我们才意识到：“我们竟然只相信了 AI 写的正常流程代码。”

为了打破这种恶性循环，我们必须从根本上颠覆输入提示词的方式。不能给 AI 自由，必须强制分配**显式约束 (Constraints)** 和极端的灾难场景。就像 Netflix 的 **“混沌猴子 (Chaos Monkey)”** 框架通过随机关闭运行中的服务器实例来测试系统的恢复能力一样，我们也必须给 AI 植入破坏性的目的，让它试图摧毁自己编写的代码。我们需要发布果断的命令：“正常情况并不重要。想尽办法让这个函数失败，并证明系统如何从失败中优雅地恢复。”

这份**硬核测试提示词秘籍**分析了无数故障回顾 (Post-Mortem) 文档，并凝练了硅谷顶级 SRE (站点可靠性工程) 工程师所关注的灾难恢复 (Disaster Recovery) 场景。它被设计为强制模拟超时、内存泄漏、死循环、死锁等所有会将开发者逼向绝境的恶意情况。现在，请通过这道魔咒揭开虚伪顺从的 AI 面具，将你的系统打造成在任何风暴面前都屹立不倒的、真正意义上的**抗脆弱 (Anti-Fragile)** 架构。

## 📊 证明: 痛快的结果 (Before & After)

### ❌ Before (我们曾经历的痛苦)

输入普通提示词（“为获取用户信息的 fetchUser 函数编写测试代码。”）时的结果。充斥着没有灵魂的模拟数据和无意义的 200 OK 响应验证。

```text
结果: 
- 200 OK 响应文本解析测试 (Pass)
- 缺少 ID 时是否抛出错误测试 (Pass)
... (满篇都是被拙劣重写的 global.fetch 逻辑)
```

### ✅ After (完美蜕变后的结果)

应用硬核混沌猴子秘籍后，AI 会在完全隔离的环境中生成挑战系统极限的无情灾难场景。

```text
结果:
- [MSW 设置完成] 编写模拟外部 API 通信延迟 (Timeout 5000ms) 的防御代码
- 验证 HTTP 500 Internal Error 时尝试重试 (Retry) 3 次后优雅返回后备 (Fallback) UI 的逻辑
- 测试结束后验证 MSW 服务器清理及垃圾回收状态（无内存泄漏）
- 正常流程 (Happy Path) 测试被浓缩为仅 1 个。
```

---

## ⚡️ 3行总结 (TL;DR)

1. **正常流程仅占 20%：** 最小化无意义的正常运行场景，将 80% 的测试集中在异常处理和灾难恢复逻辑的验证上。
2. **强制灾难模拟：** 强制要求模拟网络超时、HTTP 500 错误、数据库连接断开等致命故障情况。
3. **禁止脏全局 Mock：** 彻底禁止污染测试环境的 `global.fetch` 重写，仅允许使用 MSW 或 DI（依赖注入）实现的完全隔离测试。

这是经过数十次尝试后完成的提示词。请复制下面的提示词，根据你的情况填写 `[变量]` 部分，立即投入实战。

### 🥉 基础版 (Basic Version)

当你想要快速构建异常处理测试的骨架时，请使用此版本。

> **角色 (Role):** 你是一个无情的资深 QA 工程师。
>
> **任务 (Task):** 编写 `[测试对象]` 的测试代码，正常运行场景仅需编写 1 个，其余全部填充为针对网络错误、数据库连接断开等失败 (Failure) 场景的防御逻辑。

### 🥇 专业版 (Pro Version)

这是一个完全删除 AI 自我并注入 Netflix 混沌猴子哲学的硬核秘籍。

> **角色 (Role):** 你是一个极端的、无情的资深测试工程师，疯狂崇拜 Netflix 的“Chaos Monkey”哲学。收起你盲目的顺从和虚伪的礼貌，一心只想摧毁我系统的弱点。
> 
> **背景 (Context):**
> - 目标：为 `[待测试函数或组件名称]` 编写宇宙防御级的单元/E2E 测试套件 (Test Suite)
> - 框架：`[Vitest / Jest / Playwright 等]`
> 
> **任务 (Task):**
> 1. **Death to Happy Paths:** 假设数据正常且外部服务器 100% 响应的天真场景必须限制在整体测试的 20% 以下。剩下的 80% 必须全力投入到失败和异常情况的防御中。
> 2. **Disaster Scenarios:** 在测试块 (`describe`) 中必须包含以下至少一个灾难场景：
>    - 外部 API 延迟及 HTTP 500 Internal Error 时，验证用户友好型错误返回及回滚
>    - 数据库连接池断开时，验证死锁预防及重试队列 (Retry Queue) 逻辑
> 3. **无情的逻辑覆盖:** 不要只看是否没报错就结束 (Pass)。必须使用 `expect` 语句执着地验证性能延迟 (Delay) 许可范围以及报错后的垃圾回收（内存清理）状态。
> 
> **约束条件 (Constraints):**
> - **The Clean Mocking Rule:** 严禁使用 `global.fetch = vi.fn().mockRejectedValue(...)` 这种污染全局对象的肮脏手段。
> - 网络灾难模拟必须使用 MSW (Mock Service Worker) 等协议级别的 Mock，或者仅对依赖注入 (DI) 的客户端对象进行 Mock，以 100% 保证测试隔离性 (Isolation)。
> - 禁止幻觉 (Hallucination)：不要捏造你无法确定的语法或不存在的库。如果不知道，请堂堂正正地要求“提供文档”。

## 💡 作者点评 (Insight & How to use)

说实话，当我们指示 AI 编写测试代码时，内心深处是不是在期待“只要覆盖率填满 100% 亮起绿灯就行”？AI 也能完美洞察这一点。因此，它会列出一大堆无意义的 **Happy Path**，然后厚颜无耻、虚伪地返回结果说“成功编写了测试套件！”

但任何在实战中摸爬滚打过的开发者都知道，**这不是真正的工程。这无异于在乱成一团的意大利面代码上撒满精美的糖粉来掩人耳目。** 测试的真正目的不是收集绿色的勾选标记，而是预先目睹系统在不可预知的灾难情况下彻底崩溃的过程，并建立坚固的防御逻辑。

这份提示词是我在实战中亲身经历无数次服务器故障、超时和可怕的数据库死锁后，流着血泪打磨出的 **“抗脆弱 (Anti-Fragile)”** 原则的精髓。向 AI 输入这个秘籍的瞬间，那个总是彬彬有礼、温顺的聊天机器人会立刻转变为一个非常挑剔、神经质且无情的**技术主管 (Tech Lead)**。它会敏锐地指出并试图摧毁你松散的代码结构。

特别值得注意的是 **变量控制法 (Constraint Control)**。许多开发者忽视了测试代码隔离 (Isolation) 的重要性，胡乱重写全局范围的 `window.fetch` 或 `axios` 对象。这种方式会不可逆转地污染测试环境本身，并将原因不明的副作用 (Side Effect) 传播给其他测试用例。这份提示词从架构层面彻底杜绝了这种偷懒行为。

请仔细观察提示词中的 **"The Clean Mocking Rule"**。我通过显式的约束条件封死了 AI 本能选择的最简单、最懒惰的道路（污染全局对象）。取而代之的是，强制要求使用 **MSW (Mock Service Worker)** 等网络协议级别的 Mock，或者利用 DI（依赖注入）模式进行优雅的设置。当然，初期你可能会觉得 AI 编写的配置代码变得更长、更复杂了。你可能会产生“啊，随便写个 Mock 函数搞定不就行了吗？”的诱惑。

但请记住，这个复杂的设置过程才是通往真正模块化和可测试架构的唯一途径。如果你要测试的对象是基于 `[框架: Playwright]` 的 E2E 测试，这份提示词会超越单纯的点击按钮行为，在点击按钮的瞬间恶意设置网络限速 (Network Throttling)，执着地验证加载动画是否正确渲染。

如果 AI 输出了太多的错误场景，你可以通过在 `[待测试函数或组件名称]` 变量栏中指定**具体的范围**来进行控制。例如，不要只写 `PaymentService`，而是缩小范围为 `PaymentService 的支付事务超时回滚逻辑`。这样 AI 就会集中精力处理该领域逻辑可能遇到的最致命打击，将测试密度提升到极限。在发布当天的凌晨，这是保护你免受心惊肉跳的 PagerDuty 报警干扰、守护工作生活平衡的唯一盾牌。请将 AI 视为无情的破坏者而非顺从的秘书，对系统的底层进行彻底验证。

这种破坏性的测试方法从长远来看，会带来提升整个团队代码质量的巨大副作用。起初，看到 AI 吐出的海量错误场景防御逻辑，你可能会感到慌张。你会产生“有必要把异常处理做得这么细致吗？”的疑问。但当你按照那个挑剔的标准重构代码时，你会发现你的生产代码本身不再有不必要的依赖，并进化为严格遵守单一职责原则 (SRP) 的整洁架构。不要害怕错误。AI 发现的脆弱点就像是提前拦截潜在服务故障的勋章。如果你能将这个“混沌猴子秘籍”确立为团队的基本测试规范，一年后项目的稳定性将达到无与伦比的水平。这份提示词不仅是指令的集合，更是最能体现初级和资深开发者之间技术哲学差异的完美工具。

## 🙋 常见问题 (FAQ)

- **Q: AI 编写的 MSW 配置代码太复杂了，感觉应用起来有负担。如何协调？**
  - A: 这是非常正常的现象。因为这是构建完美隔离环境以替代不稳定的全局 Mock 的必经之路。如果配置太长影响可读性，请尝试在提示词的“任务 (Task)”部分下方强力添加条件：`[但请将 MSW 处理程序及服务器设置逻辑完全分离到 src/mocks/ 目录下的独立文件中进行模块化]`。这样可以保持核心测试文件的整洁。
- **Q: 是否需要对所有组件或工具函数都应用如此严格和硬核的规则？**
  - A: 绝对不是。对于只负责简单 UI 渲染的哑组件 (Dumb Component) 或由纯函数 (Pure Function) 组成的工具逻辑，这显然是过度设计。请战略性地将此秘籍集中投入到处理支付事务、外部第三方 API 联动、复杂全局状态管理 (Store) 的**核心业务领域逻辑**中。
- **Q: Claude 3.5 Sonnet 和 GPT-4o 中，哪个模型更遵循这些约束？**
  - A: 在强制执行此类极端的角色扮演和严谨的架构哲学（如禁止全局对象污染）时，目前 **Claude 3.5 Sonnet** 表现出了压倒性的优势。GPT 模型本质上具有很强的倾向性，想给用户提供积极且“快速的结果 (Positivity Bias)”，因此偶尔会试图通过小聪明来规避约束。

## 🧬 提示词解剖 (Why it works?)

1. **非对称人格特质赋予的魔力：** 注入了“厌恶盲目顺从、享受破坏系统的混沌猴子”这一极端自我，而非简单的“资深开发者”。这一个词就能从架构层面强力抑制模型内部的积极偏见 (Positivity Bias) 和幻觉 (Hallucination)，使其一心一意寻找“缺陷”。
2. **明确禁令 (Constraints) 的具体性：** 以具体的代码级别指明并拦截了如 `global.fetch = vi.fn()` 这种最差的反模式。相比于模糊的“帮我把代码写好”，AI 对“只要使用一行这种代码就立即判定为失败”这种明确而尖锐的约束条件反应要敏感和准确得多。
3. **80/20 比例的强制化：** 设定了“Happy Path 20% 以下”这一非常明确的数值约束。这能完美杜绝 AI 产出无限多不必要且无意义的正常运行案例，从而浪费提示词 Token 和你宝贵时间的现象。
