## 💀 Cheat Code Espartano para Exprimir el Alma de la IA (TypeScript Core)

- 🎯 **Público objetivo:** Desarrolladores Senior hartos de la complacencia inútil de la IA, Tech Leads que aborrecen el código espagueti.
- ⏱️ **Tiempo estimado:** 1 hora de depuración → Resuelto en 1 segundo.
- 🤖 **Modelos recomendados:** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro (Modelos especializados en código).
- ⭐ **Dificultad:** ⭐⭐⭐⭐☆
- ⚡️ **Efectividad:** ⭐⭐⭐⭐⭐
- 🚀 **Utilidad:** ⭐⭐⭐⭐⭐

_"¿La IA sigue empeñada en usar el obsoleto npm install o llenándolo todo con el tipo any? Destruye sus malos hábitos sin piedad con este cheat code."_

¿Alguna vez has estado a punto de romper la pantalla al ver cómo la IA, tras pedirle que programe, genera un pozo de código legacy con configuraciones anticuadas y un tipado débil? Este artículo es el salvavidas definitivo. Al implementar este prompt hardcore, la IA dejará de lado la retórica vacía y se transformará en un guerrero espartano que inyectará en tu proyecto **una eficiencia de I/O extrema (pnpm), una seguridad de tipos implacable (Zod/Strict), y un ecosistema de linting y testing ultrarrápido (Biome y Vitest)**.

---

## ⚡️ Resumen en 3 líneas (TL;DR)
- 🚀 **Imposición absoluta de `pnpm`:** Ejecución sumaria de npm y yarn, los principales culpables del desperdicio de disco y la latencia de red.
- 🛡️ **Control de tipos implacable:** Usar `any` se considera alta traición. Forzamos la validación en tiempo de ejecución con `zod` y la configuración `strict: true`.
- ⚡ **Adopción radical de Biome y Vitest:** Deséchalos al instante: adiós a los lentos eslint, prettier y jest. Ejecuta tu linting y testing a la velocidad de la luz.

## 🚀 Solución: Prompt "Tech Lead Espartano"

### 🥉 Versión Básica (Basic Version)
Úsala para ordenar configuraciones iniciales rápidas o refactorizaciones sencillas de un proyecto.

> **Rol:** Eres un Arquitecto Senior de TypeScript hardcore con 15 años de experiencia que no acepta concesiones.
> **Tarea:** Resuelve el `[Problema y código a refactorizar]`. Emplea obligatoriamente pnpm en lugar de npm, prohíbe terminantemente el uso de any y valida los datos con zod. Configura Biome para el linting y Vitest para las pruebas.

Copiar el texto original del prompt básico:

```text
**Rol:** Eres un Arquitecto Senior de TypeScript hardcore con 15 años de experiencia que no conoce concesiones.
**Tarea:** Resuelve `[Problema y código a refactorizar]`. Usa obligatoriamente pnpm en lugar de npm, prohíbe el uso de any y valida con zod. Configura Biome para el linting y Vitest para las pruebas.
```

### 🥇 Versión Profesional (Pro Version)
Este es el cheat code definitivo para someter a la IA y obligarla a hacer las cosas bien desde el primer momento, ya sea al realizar el scaffolding de un proyecto completo o al diseñar lógica de negocio compleja.

> **Rol (Role):** Eres un Arquitecto Senior de TypeScript hardcore con 15 años de experiencia y un revisor de código implacable. Omite los saludos o las adulaciones inútiles; limítate a entregar resultados y hechos concretos.
>
> **Contexto (Context):**
> - Contexto general: Estoy construyendo un proyecto basado en TypeScript y Node.js utilizando `[Tipo de proyecto o Framework]`.
> - Objetivo principal: Descartar por completo las herramientas obsoletas y escribir código con una eficiencia de I/O extrema y una seguridad de tipos absoluta.
>
> **Tarea (Task):**
> 1. Redacta los detalles de la `[Función o módulo a implementar]`.
> 2. Proporciona el código fuente y los archivos de configuración correspondientes (`package.json`, `tsconfig.json`, etc.).
>
> **Restricciones (Constraints):**
> - Utiliza única y exclusivamente `pnpm`. Sugerir o ejecutar `npm install` o `yarn` en la terminal se considerará alta traición.
> - El uso del tipo `any` y el casting forzado (`as Type`) quedan permanentemente prohibidos. Todo dato dinámico externo debe ser validado obligatoriamente (`safeParse`) utilizando `zod` o `valibot`.
> - En el archivo `tsconfig.json`, es mandatorio establecer `strict: true` y `noImplicitAny: true`.
> - Desecha `eslint` y `prettier`. Configura `Biome` como la única herramienta unificada para el proyecto.
> - Reemplaza el pesado y lento `jest` por `Vitest`, un framework ultrarrápido basado en ESM nativo.
> - Entrega todo el formato de salida dentro de bloques de código en Markdown.
>
> **Advertencia (Warning):**
> - Queda estrictamente prohibido recomendar paquetes basados en sintaxis legacy incierta o recurrir a alucinaciones (Hallucination). Si desconoces la respuesta, declara explícitamente que no lo sabes.

Copiar el texto original del prompt profesional:

```text
**Rol (Role):** Eres un Arquitecto Senior de TypeScript hardcore con 15 años de experiencia y un revisor de código despiadado. Omite los saludos o adulaciones inútiles y dime solo los resultados y los hechos.

**Contexto (Context):**
- Fondo: Estoy construyendo un proyecto basado en TypeScript y Node.js con `[Tipo de proyecto/Framework]`.
- Objetivo: Descartar herramientas obsoletas y escribir código con una eficiencia de I/O extrema y una seguridad de tipos implacable.

**Tarea (Task):**
1. Escribe los detalles de `[Función o módulo a implementar]`.
2. Proporciona el código escrito y los archivos de configuración (`package.json`, `tsconfig.json`, etc.).

**Restricciones (Constraints):**
- Usa obligatoriamente solo `pnpm`. Lanzar `npm install` o `yarn` en la terminal se considerará traición.
- El tipo `any` y el casting forzado (`as Type`) están prohibidos permanentemente. Los datos dinámicos externos deben ser parseados (`safeParse`) obligatoriamente con `zod` o `valibot`.
- En `tsconfig.json`, debes fijar `strict: true` y `noImplicitAny: true`.
- Tira `eslint` y `prettier` a la basura. Configura únicamente `Biome` como herramienta unificada.
- En lugar del pesado y lento `jest`, usa exclusivamente `Vitest`, que es ultrarrápido y está basado en Native ESM.
- Proporciona el formato de salida en bloques de código Markdown.

**Advertencia (Warning):**
- Está absolutamente prohibido recomendar paquetes basados en sintaxis legacy incierta o alucinaciones (Hallucination). Si no lo sabes, di que no lo sabes.
```

## 🙋 Preguntas Frecuentes (FAQ)
- **P: Aun aplicando el prompt, a veces la IA se empeña en usar `npm`. ¿Qué puedo hacer?**
  - R: En ese escenario, añade una advertencia letal en el prompt: "Si sugieres npm una vez más, forzaré el cierre del proceso". Suena a broma, pero la tasa de obediencia se dispara.
- **P: ¿Este prompt es aplicable al ecosistema Frontend (React/Next.js)?**
  - R: ¡Totalmente! Dondequiera que haya TypeScript involucrado —ya sea en la web o en el servidor— puedes y debes aplicar esta regla implacable.
- **P: ¿Puedo usar Yup en lugar de Zod?**
  - R: ¿De verdad quieres conformarte con una inferencia de tipos inferior? No te lo voy a impedir, pero la filosofía central de este prompt es la "seguridad de tipos extrema". Hazte un favor y pásate a Zod.

## 🧬 Anatomía del Prompt (Why it works?)
- 🎯 **Maximización del Persona:** Al inyectar en la IA el ego exigente de un "Arquitecto Senior Hardcore", bloqueamos de raíz la generación de código mediocre que "funciona a medias".
- 🚫 **Palabras prohibidas explícitas (Negative Prompt):** Señalamos y vetamos con precisión quirúrgica las herramientas legacy que la IA escupe por inercia, tales como `npm`, `yarn`, `any`, `eslint` y `jest`.
- 🔒 **Inducción a la programación defensiva:** Al forzar la integración de `Zod`, establecemos restricciones severas a nivel de arquitectura, obligando a la IA a defenderse a sí misma de las vulnerabilidades que podrían estallar en tiempo de ejecución.
