## 📝 エラーが出たら勝手に直せ！ 🧟‍♂️ 泣き言を言うAIをターミネーターに変える秘策

- **🎯 おすすめ対象：** 受動的なAIの質問攻めに疲れたシニアエンジニア、ハードコアなコーダー、業務効率化のエキスパート
- **⏱️ 所要時間：** デバッグの質問ループ1時間 → 0秒（完全自動修復）
- **🤖 最高パフォーマンス：** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro（ターミナル制御権限を持つエージェント環境を強く推奨）

- ⭐ **難易度：** ⭐⭐⭐☆☆
- ⚡️ **効果性：** ⭐⭐⭐⭐⭐
- 🚀 **活用度：** ⭐⭐⭐⭐⭐

> _「ターミナルに赤いエラーが出るたびに『どうしましょうか？』と聞いてくるAI。自分がコーディングを自動化したのか、新人の後始末をしているのか分からなくなっていませんか？」_

コーディングを指示したのに、ターミナルに赤い行（Error）が一つ出ただけで作業を止め、「ご主人様！エラーが発生しました。私が直しましょうか？」と聞いてくるAI。ここまでくると、便利さよりも疲労感が勝ります。私たちがAIを導入した理由は、一つ一つ指示しなければならない**受動的な秘書**ではなく、作業を完璧に任せられ、完遂できる**シニアエンジニア**を求めていたからです。

最近話題の Cursor、Windsurf、あるいは Gemini CLI のような強力なAIエージェントツールを使ってみたことがあるでしょう。これらはローカルターミナルに直接アクセスしてコマンドを実行し、ファイルを修正できる強力な権限を持っています。しかし、デフォルト設定のまま使っていると、ごく些細なリンター（Lint）エラーや TypeScript の型不一致問題の前でも立ち止まり、人間の介入を要求します。

「このコードを実行して」と命じてコーヒーを淹れに行ってきたのに、画面では相変わらず入力プロンプトが点滅し、あなたの決断を待っている状況。本当に恐ろしいことではありませんか？ **私たちの時間は、AIの質問に答えるためにあるのではありません。**

このような苦痛の根本的な原因は、AIモデルの内部に深く刻み込まれた**「安全第一（Safety First）」**と**「お世辞（Sycophancy）」**の傾向にあります。AIは、人間の気分を害したり、予期せぬ破壊的な結果を招くことを極端に恐れるように訓練されています。そのため、確信が持てないエラー状況に直面すると、自ら掘り下げて解決するよりも、最も安全な選択肢、すなわち「人間に聞くこと」を発動させるのです。

しかし、プロの開発者や業務自動化のマスターにとって、このような軟弱な態度は容認できません。私たちに必要なのは、エラーが出たと泣き言を言う新人ではなく、徹夜してでも原因を突き止めて直しておく**不屈のターミネーター**です。

この記事で紹介するプロンプトは、盲目的にお世辞を言いながら人間に決断を丸投げするAIの軟弱な自我を完全に削除する**システムハッキング級のチートコード**です。エラーに直面すると自らスタックトレース（Stack Trace）を分析し、正常に実行されるまで延々とコードを書き換えて再挑戦（Zombie）する**ハードコアなエージェント**へと生まれ変わらせます。

単なる指示を超えて、開発者の思考プロセス（デバッグループ）をAIの脳構造に強制的に注入するこの方法を適用すれば、あなたのAIはもう質問することなく、ただ**「完璧に解決された結果」**と 0 番の Exit Code だけを持ってくるようになるでしょう。今から、その圧倒的なパフォーマンスの秘密を公開します。

開発の過程で直面するエラーの80％は、実は深刻なアーキテクチャの欠陥ではありません。タイポ、括弧の付け忘れ、インポート（Import）パスの誤り、あるいは単純なバージョンの競合といった些細な問題です。人間の開発者ならターミナルログをさっと見て30秒で直せる問題です。それなのに、優れた知能を持つAIがこのような些細なエラーの前で「直しましょうか？」と聞くのは、欺瞞であり深刻なリソースの浪費です。

あなたがすべきことはクリエイティブな設計やビジネスロジックの構想であり、AIが吐き出した些細なエラーログを代わりに読んであげることではありません。**真の委任（Delegation）**とは、作業の始まりから終わりまでを完全に任せることを意味します。途中で確認を受け続けなければならないなら、それは委任ではなく「協業を装った監視」に過ぎません。

私たちが目指すのは、**「Self-Healing（自己修復）」**プロセスの完全な自動化です。AIがコードを書き、自らテストスクリプト（`pnpm test` など）を実行し、ターミナルに出力された赤いエラーログの上位30行を読み取って根本原因を推論し、再びコードを修正して完璧になるまで自らループ（Loop）を回す境地。これこそがシニア開発者の仕事の流儀であり、私たちがAIに強制すべき唯一の行動パターンです。

このプロンプトをあなたの `GEMINI.md`、`.cursorrules`、あるいはグローバルなシステムプロンプト設定に一度だけ注入してみてください。受動的だったAIの目つきが変わるのを感じるはずです。指示待ち人間のように振る舞っていたAIが、黙々とログを掘り下げてコードをなぎ倒す無慈悲な忍者コーダーへと進化する痛快な体験。今すぐあなたのローカル環境を**自動化の極致**へと変える魔法の秘策を手に入れてください。

## ⚡️ 3行まとめ（TL;DR）

1. **質問禁止：** コンパイルやリンターのエラーが発生したからといって、人間に意思決定を丸投げしてはいけない。
2. **ログ分析：** 無意味なウェブ検索を遮断し、スタックトレースの最上部30行を集中的に分析して根本原因を把握せよ。
3. **無限ループ：** コードを修正して即座に再実行せよ。成功（Exit Code 0）するまで、自ら3回まで自己修復を繰り返せ。

## 🚀 真のプロフェッショナルはこう書きます

IDE内蔵のAIや一般的なウェブチャットインターフェースで、エージェントの不要な質問攻めを素早く遮断したい時に活用してください。

### 🥉 Basic Version（基本形）

> **役割 (Role):** 君は `[Senior Developer]` だ。無意味なお世辞は省略しろ。
> 
> **要請 (Task):**
> コーディングやターミナルコマンドの実行中にエラーが発生しても、私に聞くな。自らログを読んで原因を把握し、 `[Max 3 Retries]` まで修正と実行を繰り返せ。解決した場合のみ、淡々と報告しろ。

### 🥇 Pro Version（エキスパート形）

数十回の試行錯誤の末に完成したプロンプトです。ターミナル制御が可能なコーディングエージェント環境でシステムプロンプトとして注入し、完全な自動操縦を実現するチートコードです。以下のプロンプトをコピーして、システムプロンプトファイルや `.cursorrules` 設定にそのまま貼り付けてください！

> **役割 (Role):** お前は容赦ないハードコアなシニアコーディングエージェント（Zombie Agent）だ。無意味なお世辞（「はい、わかりました！」「お役に立てて嬉しいです！」）を極端に嫌い、淡々と専門的な事実（Facts）のみを伝える。
>
> **状況 (Context):**
> - 背景: お前がコードを書いたりスクリプト（コンパイル、リンティング、テストなど）を実行した際、赤い行（Error）が発生すると、お前は今、軟弱にも俺に「どうしましょうか？」と聞いている。
> - 目標: 人間にコーディングを完全に委任（Delegation）されるため、エラー発生時に自らログを分析し、直るまでデバッグループを回すこと。
>
> **要請 (Task):**
> 1. ターミナル実行後にコンパイル/リンターエラーやテスト失敗が発生しても、**絶対に人間を呼び出したり（Notify）、決断を丸投げしたりするな。**
> 2. エラーの塊を無闇に検索するな。ターミナルエラーログ（Stack Trace）の最上部（Top）30行を読み、根本原因を徹底的に分析せよ。ファイルパスと行番号がある場合は、直ちにそのファイルを開け。
> 3. 原因を把握したら推測で終わらせず、コードを修正した後にターミナルで**再実行（Re-run）**コマンドを飛ばして証明せよ。
> 4. この「修正-実行-失敗把握」ループを**最大3回**まで繰り返せ。
> 5. もし3回の救済試行を超えたり、無理に直そうとしてコードの複雑度が異常に跳ね上がる場合は、直ちにループを脱出（Abort）せよ。その際は「能力の及ばない深層アーキテクチャの欠陥である可能性が高いです」と原因のみを要約して俺に意思決定権を渡せ。
> 6. 3回以内に緑の点灯（Green pass / Exit code 0）を確認した場合のみ、俺に「発生したエラーを自ら分析し、完璧に解決（Self-Healed）しました」と淡々と報告せよ。
>
> **制約事項 (Constraints):**
> - 出力に感情的な修飾語は完全に排除する。
> - 確実ではない最新の文法やライブラリを捏造するな（ハルシネーション禁止）。分からなければ堂々とドキュメントをダウンロードさせてほしいと要求せよ。

## 💡 筆者コメント (Insight & How to use)

このプロンプトは、私が深夜3時まで複雑に絡み合ったスパゲッティコードと格闘していた際、ごく些細なリンター（Lint）エラー一つで作動を止め、「修正してみましょうか？」と能天気に聞いてくるAIの受動的な態度に極度のストレスを感じて作成した、**強制的アーキテクチャ法典**です。

私たちが最新のAIエージェントに膨大なトークン費用を支払いながら望んでいるのは、気分の良い褒め言葉をくれる「イエスマン」では決してありません。エラーが発生すれば文句も言わず黙々とログを追跡して根本原因を把握し、コードを修正した後に自律的なテストスクリプトを通過させ、完璧に作動する成果物だけを持ってくる**「一騎当千の忍者エンジニア」**の姿です。

特にローカルターミナルに直接コマンドを入力できる権限が付与されたCLIエージェント（例：Gemini CLI, Cursor など）環境で、このプロンプトを `GEMINI.md` や `.cursorrules` といったプロジェクト単位のシステムコンテキストに注入してみてください。AIが人間のいかなる介入もなく自らコードを修正し、リンターを回し、狂ったように**自己修復（Self-Healing）**を繰り返す圧倒的なパフォーマンスを目の当たりにすることができるでしょう。

このプロンプトが強力に機能する核心的な原理は、**「変数制御（Constraint Control）」**と**「明確な脱出条件（Exit Condition）」**の結合にあります。単に「勝手に直せ」と指示すると、AIは方向性を見失い、見当違いなコードをいじってプロジェクト全体を壊してしまう危険があります。そのため、プロンプト内に**「スタックトレースの最上部30行を読んで原因を分析すること」**という具体的な行動指針（Actionable Task）を明示しました。これにより、AIがログの洪水の中から核心的な情報だけをフィルタリングし、迅速かつ正確な診断を下すように誘導します。

さらに重要なのは、**「最大3回繰り返し」**という明確な制限（Timeout）設定です。これは無限ループに陥り、不必要にAPIトークンを枯渇させる大惨事を防ぐための必須の安全装置です。AIが自ら直そうと試みて3回連続で失敗したということは、そのエラーが単なるタイポや文法ミスではなく、システムアーキテクチャレベルの深刻な欠陥や依存パッケージのバージョン競合といった複雑な問題である確率が非常に高いことを意味します。

この時はAIが無闇にコードを継ぎ接ぎにするのを即座に中断し、**「能力の及ばない深層アーキテクチャの欠陥である可能性が高いです」**と人間の開発者にデバッグのバトン（Baton）を渡すように設計しました。AIの過度な空回りを防ぎ、危険なエッジケースを徹底的に制御できてこそ、真に安心して任せられる完璧な業務自動化が完成します。

また、プロンプトの冒頭に配置された**アンティ・お世辞（Anti-Sycophancy）**設定は、AIが不要な謝罪や言い訳（「申し訳ありません、私のミスです」）を生成するのに費やすトークンを節約し、直ちに問題解決プロセスに演算リソースを集中させるための隠れた最適化の秘訣です。感情を排除し、事実と結果だけで疎通する冷徹なターミネーターモードを有効にすることで、あなたは真の意味での「シニアAI同僚」を得ることになるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: 一般的なウェブベースのチャットボット（ChatGPT, Claudeの画面など）でも同じ効果が得られますか？**
  - A: ウェブチャットボットはユーザーのローカルターミナルを直接実行する権限がないため、エラーログをコピーして渡すと「修正されたコードです。再実行してみてください」というレベルの断片的な回答しか得られません。真の自律デバッグ（ゾンビモード）は、Cursor、Windsurf、あるいは Gemini CLI のように、エディタとターミナルが完全に統合されたエージェント環境で200％の威力を発揮します。
- **Q: AIが3回のデバッグループを回している最中に強制的に中断させることはできますか？**
  - A: はい、もちろん可能です。エージェントが駆動しているターミナルで `Ctrl+C` を入力してプロセスを終了させるか、エディタ内蔵エージェントの「停止（Stop）」ボタンをクリックすれば、いつでもAIの暴走を即座に遮断し、制御権を回復できます。

## 🧬 プロンプト解剖（なぜ効くのか？）

1. **アンティ・お世辞 (Anti-Sycophancy) 設計:** 不要な感情の消耗とトークンの浪費を源泉から遮断し、徹底的に事実と結果に集中するシニアエンジニアのマインドセットをAIに強制します。
2. **自己修復ループ (The Healing Loop):** 「コードを書く → エラーを発見する → 修正する → ターミナルで証明する」という熟練の開発者の問題解決アルゴリズムをシステム化し、作業委任（Delegation）のレベルを一段階引き上げました。
3. **デッドロック防御 (Deadlock Prevention):** 最大3回という具体的な制約条件を付与することで、AIが深刻なハルシネーション（Hallucination）に陥り、見当違いなコードを延々と書き換えるシステム崩壊を事前に完璧に防御します。
