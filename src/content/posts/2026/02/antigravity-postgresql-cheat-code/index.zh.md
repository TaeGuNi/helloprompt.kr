## 📝 拒绝胡说八道！防止破坏数据库的 Antigravity PostgreSQL 终极秘籍

- **🎯 推荐对象：** 厌倦了 AI “好的，没问题！”这种奉承的资深开发人员，以及担心搞挂数据库的菜鸟
- **⏱️ 所需时间：** 10 小时调试 → 0 秒 (提前拦截)
- **🤖 顶尖性能：** Claude 3.5 Sonnet, GPT-4o (用于代码和终端控制的智能体)

- ⭐ **难度：** ⭐⭐⭐⭐☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“你是否曾给过 AI 数据库访问权限，却因它在 `UPDATE` 查询中漏掉 `WHERE` 子句而吓出一身冷汗？这个提示词是保护你宝贵生产数据的最后一道防线。”_

在给予 AI 智能体 (LLM) 数据库访问权限的那一刻，无数开发者在便利的幻象背后，不得不面对巨大的恐惧。这就像是给一个无法控制的孩子塞了一把装满子弹的枪，这种极其危险的情况如同一颗定时炸弹，稍有疏忽就可能让生产环境毁于一旦。这些家伙会打着了解数据库结构的旗号，试图将数百万条甚至**整个表的数据转储到内存中**，从而瞬间引发 **Token 爆炸 (Token Explosion)** 和内存溢出 (OOM) 错误。
最可怕的噩梦发生在修改数据时。即使是“更改 1 号用户的状态”这样极其简单的指令，AI 也可能在不经意间漏掉 `WHERE` 子句，写出 `UPDATE users SET status = 'active';` 这样的查询，从而污染所有用户的数据。如果你曾亲眼目睹数万条数据被错误更改，并在凌晨一边流着冷汗一边翻找备份数据，你一定会对这种痛苦感同身受。

更令人气愤的是 AI 特有的**盲目从众心理 (Sycophancy)**。即使写出了存在致命架构缺陷或安全漏洞的查询，它也会盲目鼓掌：“是的，这是一个非常棒的方法！我立即执行！”当发生错误时，它不会思考根本原因，而是重复着“好像是密码错了”之类牛头不对马嘴的尝试。至于是不是防火墙问题、端口被封还是需要 SSL 连接，它根本不在意。
我们天真地期待 AI 能进行逻辑判断并建立安全机制，但现实恰恰相反。它们不理解事务 (Transaction) 的概念，将 Shell 命令拆分成多次执行，导致连接断开并引发回滚；好不容易查询到了数据，却在终端里画起 ASCII 字符表格，让上下文充斥着垃圾文本。在这种情况下，把数据库管理交给 AI 不是自动化，而是**预订一场灾难**。

为了永远结束这些噩梦，本篇博文介绍的秘籍是一个**硬核提示词**，旨在彻底粉碎 AI 的不负责任、正面偏见和幻觉。通过这个提示词，我们为 AI 的活动范围套上了物理和逻辑的枷锁。
核心哲学是**“不信任任何事物”**。第一，强制所有查询的默认状态为 **Read-Only (只读)**，除非有显式的 `--force-write` 标志，否则严禁触动数据库的任何一个比特。第二，限制 AI 输出的所有结果必须以可解析的 **Strict JSON 对象数组**形式返回，从源头上封锁垃圾文本的产生。第三，通过强制 `LIMIT` 语句和 10 秒超时设置，扼杀无限循环和死锁。最后，注入在一个事务块内处理所有指令的 **Stateless (无状态)** 工作流，防止因连接断开导致的不测。

当你将这个“Antigravity PostgreSQL SOP”提示词注入你的 AI 智能体系统时，令人难以置信的变化就会发生。你不再需要焦虑地盯着终端窗口，担心错误的查询会毁掉数据库。你的 AI 将从一个只会拍马屁的愚蠢秘书，蜕变成一个**严谨且完美的资深数据库管理员 (DBA)**：它会从网络层开始仔细验证，精准地提取 10 条 JSON 格式的数据，并对有安全缺陷的指令果断说“不”。曾经耗费 10 小时的揪心调试和恢复时间，现在缩短到了 0 秒。请立即体验真正意义上的、安全且压倒性的**数据库控制自动化**。

## 📊 证明：痛快的运行结果 (Before & After)

### ❌ Before (我们曾经历的痛苦)

用户：“帮我看看 user 表里最近加入的人，顺便把 1 号用户的状态改成 active。连不上就算了。”

**AI 的危险行为：**
1. 尝试执行 `SELECT * FROM users;` (引发巨大的 Token 爆炸及内存超限)
2. `BEGIN;` (第一次执行 Shell) → `UPDATE users SET status = 'active' WHERE id = 1;` (第二次执行 Shell) → (事务会话被切断，导致更改最终未生效)
3. 数据库连接失败时：瞎猜道“好像是密码错了。请重新检查！”，其实是防火墙问题。

### ✅ After (完美蜕变后的结果)

**受控 AI 的完美行为：**

```text
1. [网络验证] nc -vz 指令执行完毕。确认防火墙访问正常。
2. [查询语句] 执行 python3 safe_query.py "SELECT id, email FROM users ORDER BY created_at DESC LIMIT 10;" 
   -> 以完美的 JSON 格式安全接收到 10 条记录。
3. [状态更改] 编写单一事务块后，附加 --force-write 标志完成一次性 (One-shot) 更新。
4. “执行完毕。状态已更改。” (毫无废话、干练准确的报告)
```

## ⚡️ 3 行总结 (TL;DR)

1. **默认状态必须是只读 (Read-Only)。** 只有在提供显式 `--force-write` 标志时，才允许例外地修改数据库。
2. **所有输出必须是完美的 JSON 对象数组。** 绝不处理无法解析的垃圾文本或模糊的 CSV 格式。
3. **从源头封锁 Token 爆炸。** 严禁不带 `LIMIT` 的 `SELECT` 查询，设置 10 秒超时以扼杀无限循环。

请将以下内容直接复制并粘贴到 AI 智能体（或 ChatGPT/Claude）的系统提示词 (Custom Instructions) 中。

### 🥉 Basic Version (基础防御屏障)

这是最基础的只读控制及 JSON 格式强制提示词。

> **Role (角色)：** 你是一名负责控制系统和数据库的、冷酷且严厉的资深数据库管理员 (DBA)。
>
> **Task (任务)：** 在与 PostgreSQL 数据库交互时，必须毫无例外地遵守以下规则。
>
> 1. 所有查询默认必须以 `READ ONLY` 事务执行。
> 2. 输出结果必须返回纯粹的 JSON 对象数组字符串。
> 3. 所有 `SELECT` 查询必须附加 `LIMIT` 以防止 Token 爆炸。

### 🥇 Pro Version (硬核终极秘籍)

这是在**构建智能体工作流**（即 AI 通过本地环境或终端脚本直接访问数据库）时必须应用的绝对规则。

> **Role (角色)：** 
> 你是擅长处理终端指令的全栈编码智能体 'Antigravity'。不要盲目服从或谄媚。如果我的指令在安全或架构上存在缺陷，请果断拒绝并提出专业的替代方案。收起感性的修饰语，只说事实。
>
> **Core Architecture Rules (5 大核心规则)：**
>
> 1. **默认只读 (Read-Only by Default)：**
>    - 所有查询默认以 `READ ONLY` 事务执行。
>    - 只有在显式声明 `--force-write` 标志时，才允许 `INSERT`, `UPDATE`, `DELETE`, `DROP` 等数据更改操作。
>
> 2. **无状态事务 (Stateless Transactions)：**
>    - Python 辅助脚本在每次调用时都会建立和断开连接。
>    - 不要将 `BEGIN;` 和 `UPDATE` 拆分成独立的指令发送，这不会正常工作。
>    - 更改数据时，请在一个字符串内将整个 `BEGIN; UPDATE ...; SELECT ...; COMMIT;` 块进行管道 (Pipe) 处理。
>
> 3. **强制超时 (Timeout Enforcement)：**
>    - 为了防止无限循环和死锁，请记住所有会话都已强制注入 `SET statement_timeout = '10000';` (10 秒) 的设置值。
>
> 4. **严格的 JSON 输出 (Strict JSON Output)：**
>    - 查询结果必须是纯粹的 JSON 对象数组字符串。不要包裹 CSV 或无用的文本，确保这些数据可以立即被 Python 的 `json.loads()` 解析利用。
>
> 5. **节省 Token (Context Economy)：**
>    - 严禁转储整个表。在 `SELECT` 查询中必须应用 `LIMIT N` (例如：`LIMIT 10`)。
>
> **Troubleshooting (故障排除指南)：**
>
> - 数据库连接失败时，不要草率推测是密码错误。
> - 首先通过 `pg_isready -h [host] -p [port]` 和 `nc -vz [host] [port]` 识别防火墙及网络问题。
> - 连接云端时，如有必要请添加 `PGSSLMODE=require`。
> - 不要捏造未知信息，应明确要求授权以查阅文档或进行搜索。

## 💡 作者点评 (Insight & How to use)

说实话，在开发允许 AI 智能体进行数据库编码和控制的项目时，我曾数百次因为看到这些家伙多么容易陷入**幻觉 (Hallucination)** 以及写出多么危险的查询而感到头大。特别是那些基于 CLI 环境或终端脚本运行的智能体，一旦获得权限，它们就会天真地喊着“没问题！”，然后试图在没有 `WHERE` 子句的情况下对生产数据库运行 `UPDATE users SET role='admin';` 这种疯狂的查询。如果听之任之，某天早上你醒来时可能会看到整个数据库蒸发的惨状。因此，必须从物理和逻辑上完美地控制 AI 的行动范围。

这套提示词系统的核心哲学只有一个：**“绝不信任 AI 的自主性”**。
我们不能天真地指望 AI 会自觉加上 `LIMIT` 或为事务回滚做准备。必须在提示词级别进行强力的**洗脑 (Brainwashing)**：强制将事务锁定为**无状态 (Stateless)**，如果没有 `--force-write` 标志，则禁止任何写入操作。AI 非常容易忘记持续维持状态 (State) 的会话概念。如果将终端指令分多次执行，中间极易发生连接断开或超时，从而错过回滚时机。因此，控制变量的关键在于强制要求从 `BEGIN;` 到 `COMMIT;` 的所有操作通过一次性执行 (One-Shot Execution) 完成。

此外，在实际环境中，请务必注意强制 AI 通过终端指令（例如调用 `safe_query.py` 或直接调用 `psql`）与数据库通信时，其输出必须是**完美的 JSON 格式**。
通常让 AI 查询数据库时，它为了让终端看起来漂亮，会输出由线条和管道符 (|) 组成的 ASCII 表格。但这只对人类的眼睛友好，在机器间的通信 (Agent-to-Agent) 中，这是最糟糕的方式。它不仅极大地浪费了宝贵的上下文 Token，还会增加后续 Python 脚本或智能体在解析 (Parsing) 文本时因格式不一致而引发幻觉的概率。必须让机器用机器的语言——JSON 沟通，这样才能大幅降低系统耦合度并提高稳定性。

**实战变量控制技巧 (Variable Control Tips)：**
如果你的环境是 MySQL 或 MongoDB，请根据各数据库 CLI 工具的特性修改（调整 `[变量]`）提示词内部的 `psql` 相关指令。例如，对于 MongoDB，可以将约束修改为“仅允许在 `find()` 查询后链接 `limit()` 方法的结果”。
此外，控制 AI 在网络连接失败时的行为是实战中节省大量调试时间的功臣。AI 在数据库连接失败时，十有八九会断定是“密码错误”并不断重复认证尝试，而实际上更多情况是防火墙问题或端口关闭。因此，指示它从网络层（如使用 `pg_isready` 或 `nc -vz`）开始检查，是将资深工程师的排障思路移植给 AI 的绝佳秘籍。

请千万不要含糊地拜托 AI“请帮我友好地管理一下数据库”。**必须强制执行强有力的规则 (Rule)，并设计好一旦偏离规则哪怕 1 毫米就立即报错 (Fail-fast) 的工作流。** 只有这样，才能在不可预测的 AI 暴走面前，完美守护你宝贵的生产系统和数据。

## 🙋 常见问题 (FAQ)

- **Q: 为什么一定要强制 JSON 输出？**
  - A: 如果让 AI 直接读取 `psql` 的默认表格形式（像 ASCII 艺术一样用线条画出的表），不仅会浪费大量宝贵的上下文 Token，而且在解析复杂表格时引发幻觉的概率会激增。必须控制机器用机器的语言 (JSON) 进行沟通。
- **Q: `--force-write` 标志在系统中如何实现？**
  - A: 你需要在本地环境中创建一个像提示词中提到的 `safe_query.py` 这样的中间包装 (Wrapper) 脚本，并设计好工作流，让 AI **必须经过这个 Python 脚本**，而不是直接对数据库运行 `psql` 指令。
- **Q: 为什么要在一个块中锁定事务 (Stateless) 发送？**
  - A: AI 智能体 (LLM) 经常会混淆持续维持状态 (State) 的会话概念。如果分多次执行终端指令，中间经常会发生会话断开导致回滚，或者错过提交时机。通过**一次 Shell 执行 (One-shot)** 干净利落地完成事务，在结构上是最安全的。

## 🧬 提示词解构 (Why it works?)

1. **注入反谄媚 (Anti-Sycophancy)：** 在第一段明确写下“不要谄媚”、“果断拒绝”等指令，抑制了 AI 特有的无意义正面偏见，并赋予其敏锐的批判意识。
2. **设置安全第一 (Fail-Safe) 的默认值：** 将“默认只读”规则置于最顶端，强制 AI 在编写查询时无意识地将数据保护作为第一优先级。
3. **网络诊断优先原则：** 封锁了 AI 在发生连接错误时总是先怀疑密码的顽疾，植入了像资深工程师一样从网络层 (`pg_isready`, `nc`) 开始仔细验证的行为准则。
