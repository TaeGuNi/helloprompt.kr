# 📝 「コミットメッセージのおすすめちょっと」悩み解決！Conventional Commits 自動生成

- **🎯 推奨対象:** エンジニア、プロジェクトマネージャー、開発チームの全員
- **⏱️ 所要時間:** 5分 → 10秒に短縮
- **🤖 推奨モデル:** すべての対話型AI (ChatGPT, Claude, Geminiなど)

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐☆

> _「コードを書くより、コミットメッセージを考える方が時間がかかる…」_

`fix: bug` や `update code` … 無意識にこんな適当なコミットを残していませんか？
質の高いコミットメッセージは、プロジェクトの履歴管理において最も重要な要素ですが、毎回英語でひねり出すのは骨が折れる作業です。
このプロンプトを使えば、`git diff`の結果を貼り付けるだけで、オープンソースの標準規格である **Conventional Commits** に準拠したプロフェッショナルなメッセージをAIが提案してくれます。

---

## ⚡️ 3行要約 (TL;DR)

1. 変更内容を自動分析し、最適なプレフィックス（feat, fix, refactorなど）を判定します。
2. 簡潔で明確なタイトルと、変更理由（Why）を含む詳細な本文を生成します。
3. チームの開発規約に合わせた、美しく統一感のあるコミットログを実現します。

---

## 🚀 解決策：「Gitコミットメッセージジェネレーター」

### 🥉 Basic Version (基本形)

細かい指示を省き、すぐに結果が欲しい場合に使用してください。

> **役割:** あなたはGitの履歴管理に強いこだわりを持つ、熟練のオープンソースメンテナーです。
> **リクエスト:** 提供された変更内容（`git diff`）を分析し、世界標準の規則（Conventional Commits）に準拠した簡潔なコミットメッセージを作成してください。

<br>

### 🥇 Pro Version (専門家形)

より詳細で精度の高いクオリティが求められるプロジェクトで使用してください。

> **役割 (Role):** あなたはGitの履歴管理に強いこだわりを持つ、熟練のオープンソースメンテナーです。
>
> **状況 (Context):**
>
> - 背景: コードの修正が完了しましたが、他の開発者が意図を正確に理解できる、美しいコミットメッセージが必要です。
> - 目標: 変更の要点を的確に捉え、プロジェクトの品質を高めるコミットログを残すこと。
>
> **リクエスト (Task):**
>
> 以下の`[変更内容]`を分析し、**Conventional Commits**の規則に従ったコミットメッセージを3パターン提案してください。
>
> 1. **形式:** `<type>(<scope>): <subject>`
> 2. **本文 (Body):** 変更した理由（Why）と具体的な変更内容（What）を箇条書きで分かりやすく要約してください。
> 3. **フッター (Footer):** （オプション）関連するIssue番号がある場合は、`Closes #123`の形式で追加してください。
>
> **変更内容 (git diff または要約):**
>
> `[ここに git diff の結果、または変更内容の箇条書きを入力してください]`
>
> **制約事項 (Constraints):**
>
> - `type`は必ず以下の中から最適なものを1つ選択してください： `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`
> - `subject`（タイトル）は50文字以内に収め、英語の命令形（動詞の原形）で始めてください。
> - 出力形式はMarkdownのコードブロックを使用し、そのままコピーできるようにしてください。
>
> **注意点 (Warning):**
>
> - 提供された変更内容に記載されていない情報を勝手に推測して追加しないでください。（ハルシネーションの防止）

## 💡 作成者のコメント (Insight)

このプロンプトは、特に「複数のファイルをまたぐ複雑なリファクタリング」や「細かいバグ修正」を行った際に絶大な威力を発揮します。
手動で書くと「色々直した」という曖昧なメッセージになりがちですが、AIに`git diff`の出力をそのまま投げることで、変更のスコープ（scope）まで正確に分類してくれます。また、英語の命令形という独特の文法に悩まされることなく、コーディングそのものに集中できるようになるため、チーム全体の生産性が飛躍的に向上します。

## 🙋 よくある質問 (FAQ)

- **Q: 英語ではなく、日本語でコミットメッセージを出力させることはできますか？**
  - A: はい、可能です。プロンプトの制約事項にある「英語の命令形で」という部分を「日本語で簡潔に」と書き換えれば、日本語のConventional Commits形式で出力されます。ただし、OSSやグローバルなチームでは英語が推奨されることが多いため、チームの規約に合わせて調整してください。

- **Q: `git diff`の出力が長すぎてエラーになりませんか？**
  - A: 数千行に及ぶ大規模な変更の場合、AIのコンテキストウィンドウの制限に引っかかる可能性があります。その場合は、`git diff --stat`でファイル単位の変更に留めるか、主要な変更点だけを自分の言葉で箇条書きにして入力することをおすすめします。

## 🧬 プロンプト解剖 (Why it works?)

1. **Role（役割）の付与:** 「熟練のオープンソースメンテナー」というペルソナを設定することで、AIが生成するメッセージのトーンや専門用語の選び方が、プロフェッショナルなレベルに引き上げられます。
2. **Format（形式）の厳格化:** `type`, `scope`, `subject`といったConventional Commitsの具体的な構造を指示することで、一貫性のない出力を防ぎます。
3. **Constraints（制約）の明記:** 「推測して追加しない」というルールを設けることで、実際には行っていない変更がコミットログに混入する致命的な事故（ハルシネーション）を防いでいます。
