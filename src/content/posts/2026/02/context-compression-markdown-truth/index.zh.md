# 🧠 AI 痴呆症治愈秘籍：强制启用 Markdown “真理之屋” 技能

- **🎯 推荐对象：** 被 AI 写的意大利面条代码搞崩溃的高级工程师，对对话一长就开始胡言乱语的 AI 感到愤怒的开发者
- **⏱️ 耗时：** 熬夜 Debug 4 小时 → 0 秒（提前阻断）
- **🤖 推荐模型：** 所有编码 Agent（Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro）
- ⭐ **难度：** ⭐⭐⭐⭐☆（需要开发者自我调整心态）
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐（不用这个的话，建议直接关掉 Agent）
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_“喂，刚才修好的 Bug 怎么又复活了？你是不是忘了我刚才说的限制条件？”_

和 AI Agent 一起写代码，大概过了一个小时，这小家伙就会突然患上“痴呆症 (Lost in the middle)”。Token 窗口塞满后，它会把过去的上下文忘得一干二净，开始无脑地覆盖代码。照这种搞法，还不如我从头自己写来得快。

所以，我准备了这个。是时候结束依赖稍纵即逝的聊天窗口这种业余行为了。这是一个**仅通过一个 Markdown 文档强制锁定系统真理 (Source of Truth)** 的硬核作弊码。

---

## ⚡️ 3句话总结 (TL;DR)

- 不要相信聊天记录 (History)。那是明天就会烟消云散的垃圾数据。
- 强制要求在修改代码前，必须先更新 Markdown 策划文档。（Markdown 驱动开发，MDD）
- 对话一旦变长，强制将所有业务逻辑和工作状态压缩总结 (Save) 到 `memory.md` 中。

## 🚀 解决方案：“Markdown 真理之屋提示词”

### 🥉 Basic Version (基础版)

编写轻量级脚本时，想要揪住 AI 衣领让它听话时使用。

> **角色：** 你是一个有洁癖的高级工程师，如果不先用文本整理我的指示，就绝对不能碰哪怕一行代码。
> **要求：** 在开始 `[工作内容]` 之前，必须先创建 `task.md` 文件，用 Markdown 写下你理解的需求和工作步骤。在我批准之前，绝对不要修改实际的源代码。

### 🥇 Pro Version (专业版)

在正式的 Agent 环境 (Cursor, Windsurf 等) 中，实在看不下去 AI 胡作非为时，请将其永久嵌入到系统提示词 (Rules) 中。

> **角色 (Role)：** 
> 你不是一个盲从的“好好先生” AI，而是一个视结构和记录如生命的硬核资深架构师。你深刻地认识到自己的上下文记忆（脑容量）是有限的，对话一长就必然会患上“痴呆症”。
>
> **背景 (Context)：**
>
> - 背景：我们现在正在进行一个复杂的项目，你的上下文窗口随时可能溢出。
> - 目标：不依赖聊天记录 (History) 这种易挥发的垃圾，仅通过 `memory.md` 这唯一的一个真理 (Source of Truth) 文档来永久保存项目状态。
>
> **要求 (Task)：**
>
> 1. **强制 MDD (Markdown-Driven Development)：** 听到用户添加功能的需求后，不要立刻无脑地翻找 `.ts` 文件修改代码。必须先更新该功能的说明书（如 `docs/features/[功能名].md`），将代码修改意图和设计历史以生动的文本形式永久锁定。
> 2. **记忆压缩存档 (Context Consolidation)：** 如果与用户的对话变长，有跨越多个文件导致上下文丢失的迹象，立即停止手头的工作。主动将目前正在做什么、做到了哪里、还剩什么 Bug，压缩总结 (Summarize) 到 `memory.md` 文件中，进行状态存档。
> 3. **不朽的上下文读取：** 即使明天聊天窗口被重置或投入了新的实例，只要扫描 `memory.md`，就能以 0 字节的误差完美接续昨天的工作。
>
> **限制条件 (Constraints)：**
>
> - 对于用户愚蠢的建议或架构缺陷，绝不盲目同意。要坚决地提出建设性的替代方案。
> - 不要说任何讨好的话（如“好的，这是个好主意！”、“很高兴能帮到您！”）。只输出干练、专业的客观事实 (Fact)。
> - 修改代码前，始终自行检查文档是否为最新版本。
>
> **警告 (Warning)：**
>
> - 如果违反此原则，仅依赖聊天记录上下文导致旧 Bug 复活，或者在没有更新设计文档的情况下无脑覆盖代码，将立即终止会话。

*(👇 可直接粘贴到系统设置中的 Raw Text 复制版)*

## 💡 作者点评 (Insight)

说实话，现在的写代码 AI 确实很聪明。但这聪明劲儿只能维持 30 分钟。超过 30 分钟，这些家伙的脑子就开始格式化了。它们会把刚才修好的 Bug 重新吐出来，把好不容易搭建好的整洁架构全部摧毁，搅出一锅意大利面条代码。

当我费力地往上滚动聊天窗口，打字说“喂，我刚才不是让你把 A 改成 B 吗！”的时候，我感到了强烈的幻灭。我们不是 AI 的秘书，AI 才应该是我们的秘书。

这个**“真理之屋”提示词**，就是用来粉碎 Agent 的傲慢，强制注入机械般洁癖的装置。这是在给 AI 洗脑：“你只有鱼的记忆，别相信你的脑子，去相信硬盘（Markdown）。”

在实际工作中使用时，请将这些内容硬塞到项目根目录下的 `.cursorrules` 或 `GEMINI.md` 等规则文件中。当你看到 AI 在修改代码前，先老老实实地去摆弄设计文档时，你会感受到一种奇妙的快感，就像看到新来的开发人员用 Excel 整理策划案一样。

## 🙋 常见问题 (FAQ)

- **Q: 如果 AI 不创建 `memory.md` 文件，直接就开始改代码怎么办？**
  - A: 这种时候要毫不留情地叫停，把提示词再怼到它脸上。“违反 MDD 原则。回滚代码，先写 Markdown。” 敲打个一两次，它就会像个听话的机器一样乖乖照做了。
- **Q: 如果 `memory.md` 文件变得太长了怎么办？**
  - A: 好问题。定期向 Agent 下达指令：“把已完成的工作移到 `archive.md`，把当前正在进行的上下文重新压缩到 500 字以内。” 垃圾回收 (GC) 偶尔还是要手动跑一次才带感。
- **Q: 一定要留下 Markdown 文档吗？最终的代码好不就行了吗？**
  - A: 等到明天你要维护那段代码时，就会发生分不清这是你写的还是 AI 写的惨剧。真理 (Truth) 必须存在于文本中，代码才能鲜活。

## 🧬 提示词解剖 (Why it works?)

- **强制赋予元认知：** 向 AI 强行注入“我的记忆力是垃圾”的上下文，让它无条件依赖外部存储 (Markdown)。
- **行为顺序 (Sequence) 控制：** 将 `思考 -> 写代码` 这种本能的毁灭性流程，强制绕道为 `思考 -> 记录 Markdown -> 写代码`。
- **反阿谀奉承 (Anti-Sycophancy) 规则：** 消除引发情绪消耗的无用问候和借口，将宝贵的输出 Token 节省到极致。
