## 🚀 解決策："Everything Claude Code" プロンプティング

### 🥉 Basic Version（基本型：迅速なセットアップと初期設計）

ターミナルからたった2分で、優秀なシニアエージェントを雇用しましょう。

> **1. プラグインのインストール（Claude Codeの場合）:**
> `/plugin marketplace add affaan-m/everything-claude-code`
> `/plugin install everything-claude-code@everything-claude-code`
>
> **2. 設計（Plan）プロンプトの実行:**
> `/plan "Next.js 14のApp Routerを使用して、レスポンシブ対応のブログ記事一覧ページを作成して。スタイリングにはTailwind CSSとshadcn/uiを使用する想定で進めてください。"`

<br>

### 🥇 Pro Version（専門家型：TDDとカスタムルールの強制）

プロジェクトに特化した、シニア開発者レベルの厳格なワークフローを強制したい場合に使用します。

> **役割 (Role):** あなたは厳格なTDD（テスト駆動開発）の原則を遵守する`[シニアTypeScript開発者]`です。
>
> **状況 (Context):**
>
> - 現在、私たちは`[everything-claude-code]`のルールに従って開発を進めています。
> - 目標: `[ユーザー認証機能]`を実装する必要があります。
>
> **要請 (Task):**
>
> 1. ターミナルで`/tdd`コマンドを有効化してください。
> 2. 実際のプロダクションコードを記述する前に、必ずエッジケースを含むテストコードを先に作成してください。
> 3. テストカバレッジが80%を超えない限り、次のステップへ進まないでください。
> 4. `[追加の要件]`がある場合は、それを反映させた上でコードを実装してください。
>
> **制約事項 (Constraints):**
>
> - すべての変数と関数には、明示的な型ヒント（Type Hint）を追加すること。
> - `any`型の使用は一切禁止します。
>
> **注意事項 (Warning):**
>
> - 既存のアーキテクチャやコーディング規約に違反するコードは作成しないでください。確信が持てない場合は、コードを書く前に必ず私に質問してください。（ハルシネーションの防止）

## 💡 筆者のインサイト (Insight)

この設定を実際の業務に適用してみて最も鳥肌が立ったのは、**「AIが怠惰なプロンプティングを拒否する」**という点でした。

以前であれば、「ログインページを作って」と適当に指示すれば、エラーハンドリングも考慮されていない粗悪なコードが出力されがちでした。しかし、この設定を導入した後は違います。`/plan`コマンドを使用すると、AIが自ら_「どの認証方式（OAuth, JWT）を採用しますか？エラー状態のUIはどのように処理すべきですか？」_と、逆に私を問い詰めてくるのです。まるで、**要件定義書がなければ一行のコードすら書かない、非常に優秀で厳しいシニア開発者**をメンターに持ったような感覚です。

特に感銘を受けたのは**`/evolve`**コマンドの存在です。プロジェクトが進行するにつれて、AIがチーム独自のコーディング規約（例：「インターフェース名の先頭に『I』を付けない」など）を自ら学習し、ルールとして定着させていきます。スパゲティコードの量産に疲弊している方や、シニアエンジニアのコードレビューを渇望している個人開発者にとって、このワークフローは選択肢ではなく「必須」と言えるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: これはClaude Code CLI環境でしか使えませんか？**
  - A: いいえ、そんなことはありません！Cursorエディタの`.cursorrules`ファイルに適用したり、OpenCode環境で使用したりと、完全に互換性があります。

- **Q: 社内の機密プロジェクトや業務で使用しても安全ですか？**
  - A: ルール（Rule）ファイル自体はオープンソースであり、ローカル環境に保存されるため安全です。ただし、LLMモデルへコードを送信する行為そのものについては、社内のセキュリティポリシー（Enterpriseプランの利用規定など）を必ず事前に確認してください。

- **Q: 初心者にとって、この設定は少し厳しすぎませんか？**
  - A: 確かに、最初はAIが何度も質問を投げかけてきたり、テストコードの作成を強制してきたりするため、煩わしく感じるかもしれません。しかし、このプロセスを初期段階で経ておくことこそが、後々膨大な「技術的負債（Technical Debt）」の返済に追われて徹夜する事態を防ぐ、唯一にして最大の防衛策なのです。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **強制されたワークフロー (Forced Workflow):** 単なる指示出しではなく、`/plan`や`/tdd`などのスラッシュコマンドを通じて、AIに特定のプロセス（要件分析 → テスト作成 → 実装）を強制的に踏ませます。
2. **ペルソナの分離 (Persona Separation):** コードを書く自我（Developer）、レビューする自我（Reviewer）、企画する自我（Planner）を明確に分離することで、AI自身にコードの客観的なクロスチェック（交差検証）を行わせます。
3. **継続的な学習 (Instincts):** 対話のコンテキストが揮発してしまう従来のチャットボットとは異なり、AIがプロジェクトを通じてユーザーの修正パターンを把握し、それを`.rules`ファイルとして内在化・記憶します。
