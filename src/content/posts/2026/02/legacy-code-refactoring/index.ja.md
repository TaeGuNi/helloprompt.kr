## 🧹 レガシーコード掃除機：リファクタリング＆テスト自動化

- **🎯 おすすめの対象:** ジュニアエンジニア、レガシープロジェクトの保守担当者、コードレビュアー
- **⏱️ 所要時間:** コードの分析・把握に要する1時間を → 5分以内に短縮
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o（複雑なロジックや長いコンテキストの把握にはClaudeを強く推奨）

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「一体このコード、誰が書いたんだ？（…あ、1年前の自分だ。）」_

複雑に絡み合った `if-else` のネスト、意図が全く読めない変数名、そしてドキュメント化されていない例外処理の数々。触れた瞬間にシステムが崩壊しそうな**レガシーコード（Legacy Code）**を前に、途方に暮れた経験はありませんか？

もう恐れる必要はありません。AIをあなた専属の**シニア・ペアプログラマー（Pair Programmer）**として雇い入れ、スパゲティコードをエレガントにリファクタリングし、強固な防壁となるテストコードまでを一気に構築する、完璧なプロンプトエンジニアリングの手法をご紹介します。

## ⚡️ 3行まとめ (TL;DR)

1. **まずは説明を求める:** むやみに修正させるのではなく、コードの実行フローと意図をAIに先行して把握させます。
2. **クリーンコードの原則を適用:** SOLID原則と可読性を基準に、命名規則や構造を安全にリファクタリングします。
3. **テストコードで防壁を構築:** リファクタリング前後で機能の同一性を保証するため、エッジケースを含む単体テスト（Unit Test）を自動生成させます。

## 🚀 解決策："Senior Dev Simulator"（シニアエンジニア・シミュレーター）

### 🥇 Pro Version (プロフェッショナル向け)

単に「リファクタリングして」と指示するだけでは、既存の機能が壊れるリスクが高まります。このプロンプトは、AIに論理的なステップを踏んで思考させる**Chain-of-Thought（思考の連鎖）**手法を適用しています。

> **役割 (Role):** あなたはGoogleやMetaなどのメガベンチャー出身の`[10年目のシニアバックエンドエンジニア]`です。クリーンコード（Clean Code）哲学の信奉者であり、TDD（テスト駆動開発）を徹底的に遵守します。
>
> **状況 (Context):**
> 私は今、`[PythonやJavaなどのプログラミング言語]`で書かれた非常に乱雑なレガシーコードを引き継ぎました。可読性が最悪で、保守がほぼ不可能な状態です。
>
> **指示 (Task):**
> 以下のステップを順を追って深く実行してください。
>
> **Step 1. コードの分析と診断:** このコードがビジネス的にどのような役割を果たしているのかを1文で要約し、致命的な問題点（Code Smell）を3つ具体的に指摘してください。
> **Step 2. 安全なリファクタリング:** 元のビジネスロジックと副作用（Side Effect）を100%維持しつつ、単一責任の原則（SRP）に従った関数の分離、明確な変数名/メソッド名の変更を通じて可読性を最大化してください。複雑なビジネスロジックには、必ずDocstringとコメントを追加してください。
> **Step 3. 防御的検証（テストコード）:** あなたが作成したリファクタリング後のコードが、元のコードと全く同じように正常動作することを完全に保証する単体テスト（Unit Test）コードを作成してください。成功ケースだけでなく、発生し得る例外状況（Edge Case）を最低2つ以上含める必要があります。
>
> **入力コード:**
>
>
> `[ここに分析するレガシーコードを貼り付けてください]`
>

## 💡 筆者のコメント (Insight)

このプロンプトの真の価値は、単に綺麗なコードを生成することではなく、**「安全網（テストコード）」を先に構築してくれる点**にあります。実際に私自身、5年前に作成された古い決済モジュール（Java）を引き継いだ際に、このプロンプトを実務へ適用してみました。

最も驚かされたのは、私自身が把握しきれていなかった隠れたエッジケース（例：決済金額が0円の場合の分岐処理など）を、AIがテストコードの形で真っ先に指摘してくれたことです。

**🔥 実務最適化のヒント:**
数百行に及ぶクラス全体を一度にコピー＆ペーストすると、AIが「ハルシネーション（幻覚）」を起こしたり、文脈を見失う可能性があります。最も複雑に見える**メソッド単位（100行以内）に分割して入力**してください。そうすることで、分析の深さとテストコードの正確性が圧倒的に向上します。

## 🙋 よくある質問 (FAQ)

- **Q: コーディングのロジックを扱う上で、どのAIモデルが最も適していますか？**
  - A: 2026年現在、コーディングおよびリファクタリングの分野では、**Claude 3.5 Sonnet**と**GPT-4o**が二大巨頭です。その中でも、複雑なコンテキストの維持と人間にとって分かりやすいクリーンコードの作成においては、Claude 3.5 Sonnetの方がやや優れたパフォーマンスを発揮します。

- **Q: 社内のコードをそのままAIに渡してもセキュリティ上の問題はありませんか？**
  - A: **絶対に避けてください。** APIキー、社内ネットワークのIP、実際の顧客データ、コアとなるセキュリティアルゴリズムなどの機密情報は、必ず`[マスキング処理]`を行うか、仮想のダミーデータ（Dummy Data）に置換してからプロンプトを実行する必要があります。または、社内に構築されたEnterprise AI環境でのみ使用してください。

- **Q: リファクタリング後にエラーが発生した場合はどう対処すればよいですか？**
  - A: AIが見落とした依存関係（Dependency）が存在する確率が高いです。エラーログ全体と元のコードの関連コンテキストを一緒にコピーし、_「この部分でこのようなエラーが発生した。既存のロジックの`[特定の条件]`をあなたが見落としているようだ。再度修正してほしい」_ と具体的にフィードバックしてください。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **シニアペルソナの付与 (Role):** 「10年目のシニア、クリーンコードの信奉者」という厳格なペルソナを付与することで、単に動くだけのコードではなく、保守性に優れた高品質なコードを出力するよう基準を引き上げました。
2. **Chain-of-Thought (思考の連鎖):** いきなり「直して」と命令するのではなく、**[分析 → 修正 → 検証]**という論理的な3段階のステップを強制しました。これが、AIにコードの本来の意図を損なわせることなく、構造だけを改善するよう誘導するコアとなる仕掛けです。
3. **エッジケースの強制 (Constraints):** テストコードの作成時にエッジケースを明示的に要求することで、実務ですぐに通用する堅牢なコード品質を保証します。

```python
def calc_trx(a, t, f=False):
    # a: amount, t: type
    if t == 1:
        if f: return a * 1.1 + 500
        else: return a * 1.1
    elif t == 2:
        return a - (a * 0.05)
    else: return 0 # エラー
```

### ✅ After (結果: シニアが作成したクリーンコード＆テスト)

```python
from enum import Enum
import unittest

class TransactionType(Enum):
    DEPOSIT = 1
    WITHDRAWAL = 2

def calculate_transaction_amount(amount: float, transaction_type: TransactionType, apply_fee: bool = False) -> float:
    """
    取引の種類（入金/出金）に応じて、税金および手数料が適用された最終金額を計算します。
    """
    if amount < 0:
        raise ValueError("取引金額は0より大きくなければなりません。")

    if transaction_type == TransactionType.DEPOSIT:
        taxed_amount = amount * 1.1
        return taxed_amount + 500 if apply_fee else taxed_amount

    elif transaction_type == TransactionType.WITHDRAWAL:
        return amount * 0.95

    return 0.0

# ----------------- Unit Tests ----------------- #
class TestCalculateTransaction(unittest.TestCase):
    def test_deposit_with_fee(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.DEPOSIT, True), 1600.0)

    def test_withdrawal_amount(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.WITHDRAWAL), 950.0)

    def test_negative_amount_raises_error(self):
        with self.assertRaises(ValueError):
            calculate_transaction_amount(-500, TransactionType.DEPOSIT)
```
