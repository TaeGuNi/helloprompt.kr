- **🎯 Кому будет полезно:** Junior-разработчикам, Backend-инженерам, поддерживающим legacy-код

> _"Говорите, Gemini 3 Pro плохо пишет код? Это только потому, что вы не дали ИИ 'время на раздумья'."_

В феврале 2026 года Google представила **Gemini 3 Pro**, и его главной фишкой стал режим **'Deep Think'**. Он позволяет использовать возможности 'рассуждения' (Reasoning), которые мы видели в модели O2 от OpenAI, но на гораздо более высокой скорости. Однако, если вы просто скомандуете "сделай рефакторинг этого кода", то, скорее всего, получите нелепый результат и впустую потратите вычислительные ресурсы. Сегодня мы разберем промпт, который позволит даже Junior-разработчику проектировать архитектуру на уровне Senior-инженера.

## ⚡️ Краткое содержание (TL;DR)

1. Режим Deep Think в Gemini 3 Pro не раскроет свой потенциал, если не задать ему четкое направление.
2. Прежде чем просить написать код, необходимо принудительно запустить этапы "Аудита кода" (Code Audit) и "Разработки стратегии рефакторинга".
3. Этот подход демонстрирует колоссальную эффективность при безопасной и систематичной миграции legacy-кода со сложными зависимостями.

## 🚀 Решение: Протокол "Архитектор" (The 'Architect' Protocol)

### 🥉 Базовая версия (Basic Version)

Используйте, когда нужно быстро выявить проблемы с безопасностью и производительностью кода.

> **Роль:** Ты `[Senior Backend-разработчик с 20-летним опытом]`.
> **Задача:** Проанализируй представленный ниже `[legacy-код]` и выдели ровно 3 проблемы, связанные с безопасностью и производительностью.


### 🥇 Профессиональная версия (Pro Version)

Используйте, когда нужно выйти за рамки простых правок и получить архитектурный рефакторинг с извлечением крайних случаев (Edge Cases) для тестирования.

> **Роль (Role):** Ты ведущий архитектор программного обеспечения с 20-летним опытом и амбассадор Clean Code. Ты эксперт, для которого поддерживаемость, масштабируемость и производительность важнее простой реализации функционала.
>
> **Контекст (Context):**
>
> - Фон: Необходимо перенести запутанный спагетти-образный `[legacy-код на Python]` в архитектуру, удобную для поддержки.
> - Цель: Преобразовать код так, чтобы он строго соответствовал принципу единственной ответственности (SRP) и имел низкую связанность (low coupling).
>
> **Задача (Task):**
> 
> Проанализируй предоставленный пользователем legacy-код и выполни рефакторинг, строго следуя этим 4 шагам:
>
> 1. **Аудит кода (Code Audit):** Выяви как минимум 3 проблемы текущего кода (Антипаттерны, Уязвимости безопасности, Узкие места производительности).
> 2. **Стратегия глубокого размышления (Deep Thinking Strategy):** Разработай конкретную стратегию рефакторинга. (Например: какие паттерны проектирования применить и почему они подходят).
> 3. **Рефакторинг (Refactoring):** Напиши улучшенный код, снабдив ключевую логику комментариями с обоснованием изменений.
> 4. **Генерация тестов (Test Case Generation):** Напиши 3 тест-кейса для пограничных ситуаций (Edge Cases), чтобы доказать безопасность кода после рефакторинга.
>
> **Ограничения (Constraints):**
>
> - Язык и версия: `[Python 3.12+]` (Обязательное использование Type Hinting).
> - Руководство по стилю: Строгое соблюдение PEP 8.
> - Внешние библиотеки: Использовать только стандартную библиотеку (Минимизация внешних зависимостей).
> - Формат вывода: Оформи ответ в чистом Markdown так, чтобы логику понял даже Junior-разработчик.
>
> **Предупреждения (Warning):**
>
> - Если результаты анализа неясны или неоднозначны, не придумывай код искусственно, а четко напиши "Невозможно определить". (Предотвращение галлюцинаций).
>
> **Входные данные (Input Data):**
> `[Вставьте сюда код для рефакторинга]`

Когда видишь, как ИИ сам выносит логику отправки писем в отдельный сервис и заботится об откате транзакций, возникает уверенность, что переработок станет как минимум вдвое меньше.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Можно ли использовать этот промпт в GPT-5?**
  - A: Да, он отлично работает и в GPT-5. Однако в ситуациях "сложного рефакторинга", когда нужно анализировать массивную кодовую базу и глубоко понимать зависимости между файлами, режим Deep Think у Gemini 3 Pro справляется с поиском крайних случаев (edge cases) гораздо лучше. С другой стороны, если нужно поправить простую утилитарную функцию, GPT-5 может сделать это быстрее.

- **Q: Как адаптировать его под другие языки программирования (Java, TypeScript и др.)?**
  - A: Просто измените раздел `Язык и версия` и `Руководство по стилю` в блоке "Ограничения (Constraints)". Например, на `[TypeScript 5.0+, соблюдение правил ESLint Airbnb]`, и промпт сработает идеально.

- **Q: ИИ постоянно выдает "Невозможно определить" в результатах анализа. Что делать?**
  - A: Это означает, что переданному коду не хватает контекста. Если вы добавите в "Входные данные" схему базы данных или интерфейсы связанных классов, ИИ сможет нарисовать гораздо более точный архитектурный проект.

## 🧬 Анатомия промпта (Why it works?)

1. **Запуск Chain-of-Thought:** Четкое разделение на 4 этапа (`Code Audit -> Strategy -> Refactoring -> Test`) заставляет ИИ размышлять логически и последовательно.
2. **Предотвращение галлюцинаций (Hallucination):** Блок `Warning` устанавливает жесткие ограничения, запрещая генерировать правдоподобный, но ложный код в ситуациях неопределенности.
3. **Детализация роли:** Присвоение узконаправленной персоны ("амбассадор Clean Code" и "ведущий архитектор") вместо абстрактного "разработчика" резко повышает стандарты качества (Quality Standard) итогового результата.

### ❌ До (Входные данные)

```python

    def handle_user(self, data, action):
        if action == "create":
            # Логика подключения к БД...
            # Логика отправки email...
            # Логика сохранения логов...
            pass
        elif action == "delete":
            # Удаление связанных данных...
            pass
```

### ✅ После (Результат)

Код идеально разделен с использованием `Service Layer` и `Repository Pattern`, строго соблюдая принцип единственной ответственности (SRP).

```python
# UserCreateService.py
from dataclasses import dataclass
from typing import Protocol

class UserRepository(Protocol):
    def save(self, user: 'User') -> bool: ...

class EmailSender(Protocol):
    def send_welcome_email(self, email: str) -> bool: ...

@dataclass
class UserDTO:
    email: str
    name: str

class UserCreateService:
    """
    [Refactoring Insight]

    Также применен принцип инверсии зависимостей (DIP), что упрощает модульное тестирование.
    """
    def __init__(self, user_repo: UserRepository, email_sender: EmailSender):
        self.user_repo = user_repo
        self.email_sender = email_sender

    def execute(self, user_dto: UserDTO) -> None:
        # Здесь находится логика транзакций и отката при сбоях.
```
