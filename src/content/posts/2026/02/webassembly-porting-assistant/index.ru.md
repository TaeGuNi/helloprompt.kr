## 🦀 Портирование в WebAssembly (Wasm): Пробиваем потолок веб-производительности

- **🎯 Рекомендуется для:** Фронтенд-разработчиков, нуждающихся в радикальной оптимизации производительности, и инженеров, переносящих тяжелую бэкенд-логику (C++/Rust/Go) в браузер.
- **⏱️ Сокращение времени:** От нескольких дней мучительной настройки биндингов → до 5 минут.
- **🤖 Оптимальные модели:** Claude 3.5 Sonnet (идеален для конвертации кода), GPT-4o.

- ⭐ **Сложность:** ⭐⭐⭐⭐☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Универсальность:** ⭐⭐⭐☆☆

> _"Вы можете неделями микрооптимизировать JavaScript ради жалких 10% прироста, но перенос тяжелых вычислений на Wasm ускоряет их в десятки раз. Вопрос лишь в том, как сделать этот переход безболезненным?"_

Современный браузер давно перестал быть просто средством для чтения документов. Обработка видео, сложная криптография, массивные математические вычисления — неизбежно наступает момент, когда возможностей движка V8 для JavaScript становится недостаточно. WebAssembly (Wasm) кажется идеальным решением, но ручной процесс преобразования существующего кода на C++ или Rust, настройка разделяемой памяти (Shared Memory) и написание мостов для JS обычно оборачивается настоящим кошмаром.

Этот промпт выходит далеко за рамки простого перевода синтаксиса. Он действует как ваш личный «ассистент по портированию в Wasm», который с первой попытки генерирует безопасный от утечек памяти код и все необходимые связующие JS-интерфейсы.

2. **Автоматизация биндингов:** ИИ самостоятельно пишет код для маппинга типов данных и настройки разделяемой памяти (Shared Memory) между средами JS и Wasm.
3. **Гарантия типобезопасности:** Получайте готовые интерфейсы TypeScript (`.d.ts`) для мгновенного и надежного использования модулей на фронтенде без риска ошибок в рантайме.

## 💡 Авторский комментарий (Insight)

Главный барьер, о который разбиваются разработчики при портировании логики в WebAssembly, кроется в одном вопросе: **"Как эффективно передать JS-массив в Wasm и как вернуть его обратно?"**. Перекинуть простое число или строку — тривиальная задача. Но когда дело доходит до буферов изображений, видео или потоковых аудиоданных (`Float32Array`, `Uint8Array`), вам необходимо напрямую управлять памятью, минуя сборщик мусора V8.

Ключевая ценность этого промпта заложена в **пункте 2 (Архитектура памяти)**. Если вы просто попросите ИИ "перевести код в Wasm", он в 90% случаев выдаст ленивое, крайне неэффективное решение, копирующее весь объем памяти при каждом вызове. Но когда мы жестко требуем **«минимизации затрат на копирование»** и **«использования разделяемой памяти»**, нейросеть генерирует код истинно продакшен-уровня. Она начинает задействовать мощнейшие возможности `wasm-bindgen`, такие как прямой доступ к памяти по сырым указателям. 

Кроме того, требование предоставить пример интеграции с React мгновенно избавляет вас от головной боли с асинхронным таймингом загрузки Wasm-файлов в современных сборщиках вроде Webpack или Vite.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Мой исходный код слишком огромный, чтобы скормить его ИИ за один раз. Что делать?**
  - **Ответ:** Не пытайтесь загрузить весь легаси-проект целиком. Разбейте логику на атомарные функции и прогоняйте промпт для каждой из них отдельно. Если модуль имеет сложную систему внутренних зависимостей, сначала опишите его архитектуру в секции `[Context]`, чтобы ИИ уловил общую картину и сгенерировал правильные биндинги интерфейсов.

- **Вопрос: Как скомпилировать полученный от ИИ код на Rust?**
  - **Ответ:** Проще всего использовать официальный тулчейн `wasm-pack`. ИИ обычно сам указывает команду сборки (например, `wasm-pack build --target web`) в конце своего ответа. Просто скопируйте её и выполните в терминале — на выходе вы получите папку `pkg`, полностью готовую к импорту в ваш JS-проект.

- **Вопрос: В чем принципиальная разница между портированием C++ через Emscripten и связкой Rust + wasm-bindgen?**
  - **Ответ:** Emscripten отлично подходит для "грубого" переноса огромных движков на C/C++ в веб, но он генерирует очень тяжелые файлы, а взаимодействие с JS выглядит неуклюже. Rust в связке с `wasm-bindgen`, напротив, создает хирургически точные, минимальные по размеру бандлы и обладает идеальной совместимостью с экосистемой TypeScript. Именно поэтому он стал де-факто стандартом во фронтенд-сообществе.

## 🧬 Анатомия промпта (Why it works?)

1. **Фокус на комплексном E2E-решении:** Промпт не останавливается на банальном переводе синтаксиса. Он сразу требует интеграции с фронтенд-фреймворком и примеров реального использования, выдавая разработчику «готовый к внедрению продукт».
2. **Строгие рамки производительности (Constraints):** Главная цель использования Wasm — это бескомпромиссная скорость. Жесткие формулировки вроде «минимизация копирования» и «отказ от сериализации» отсекают ИИ возможность использовать простые, но фатально медленные обходные пути.
3. **Абсолютная типобезопасность (Type Safety):** Автоматическая генерация файлов `.d.ts` стирает хрупкую границу типов между мирами JS и Wasm, превентивно исключая ошибки `Runtime Errors` еще на этапе написания кода в IDE.

## 📊 Доказательство: До и После (Before & After)

Разница в качестве архитектуры между инфантильной просьбой «переведи это в Wasm» и использованием нашего Pro-промпта поистине колоссальна.

### ❌ До (Ответ ИИ на простой запрос)

```rust
// Неэффективный метод, сериализующий и копирующий данные целиком
#[wasm_bindgen]
pub fn apply_filter(data: Vec<u8>) -> Vec<u8> {
    let mut result = data.clone(); // Огромные затраты на копирование памяти
    // ... логика фильтрации ...
    result
}
```

### ✅ После (Ответ ИИ при использовании Pro-промпта)

```rust
// Прямой доступ по указателю без лишнего копирования (Zero-copy)
#[wasm_bindgen]
pub fn apply_filter_in_place(data: &mut [u8], width: usize, height: usize) {
    // Прямой доступ и манипуляции с линейной памятью Wasm, 
    // на которую указывает Uint8Array из JS
    for i in (0..data.len()).step_by(4) {
        // Применение фильтра напрямую к каналам Red, Green, Blue
        let r = data[i];
        let g = data[i + 1];
        let b = data[i + 2];

        data[i] = (r as f32 * 0.393 + g as f32 * 0.769 + b as f32 * 0.189) as u8;
        // ... (пропущено) Накладные расходы на аллокацию/освобождение памяти полностью устранены
    }
}
```
