> _"Are you still holding your breath with every click in the AWS console, terrified of accidentally wiping out a server?"_

Infrastructure should be treated like cattle, not pets. When a server goes down, you shouldn't stay up all night fixing it; you should ruthlessly terminate it and spin up an exact replica in a second. **Infrastructure as Code (IaC)** makes this possible. However, memorizing the complex syntax of HCL (HashiCorp Configuration Language) and the endless provider options every time is no easy task.

This prompt takes a vague request like "Spin up an EC2 instance and connect it to a DB on AWS" and instantly converts it into **perfect, production-ready Terraform code** that incorporates strict security policies and modular design.

---

## ⚡️ TL;DR

1. Instantly generates **code that strictly adheres to standard HCL syntax**, eliminating the need to scour documentation.
2. Enforces **least privilege security settings (SG, IAM)** to prevent fatal mistakes like allowing `0.0.0.0/0`.
3. Eliminates hardcoding by splitting the configuration into `variables.tf` and `outputs.tf`, designing **highly reusable modules**.

## 🚀 The Solution: "Infrastructure Master Builder"

### 🥉 Basic Version

Use this for a quick setup when you urgently need a single test resource.

> **Role:** You are a Senior DevOps Engineer.
> **Task:** Write Terraform code to create the AWS `[Service Name]` resource.
> **Constraints:** Use the latest AWS Provider version and utilize variables instead of hardcoding values.

\

### 🥇 Pro Version

Use this when building actual production environments where security and high availability (HA) are mandatory.

> **Role:** You are a Cloud Architect and Terraform Expert with 10 years of experience, specializing in cloud security and High Availability (HA) design.
>
> **Context:**
>
> - Background: I am currently designing AWS infrastructure for `[Project Description, e.g., a global e-commerce backend handling 10,000 requests per second]`.
> - Objective: To obtain production-grade Infrastructure as Code (IaC) that implements strict security, high availability, and thorough modularization.
>
> **Task:**
>
> 1. **Architecture Configuration:** Write the main resource definitions (`main.tf`) required to build the requested infrastructure.
> 2. **Structuring & Modularization:** Strictly avoid hardcoding. Clearly separate configurations into `variables.tf` and `outputs.tf`.
> 3. **Strict Security Rules:** Apply the principle of least privilege to all Security Groups (absolutely NO unconditional `0.0.0.0/0` allowance).
> 4. **Permission Management:** If necessary, write IAM Role and Policy code with minimum required permissions.
>
> **Constraints:**
>
> - Use the latest HashiCorp Configuration Language (HCL) syntax and AWS Provider v5.0 or higher.
> - Include the `[Project Name]` prefix in all resource names and tags.
> - Strictly adhere to the indentation and style guide standards of `terraform fmt`.
>
> **Warning:**
>
> - Do NOT use deprecated options or insecure default values that are unacceptable in a production environment. (Prevent hallucinations and security breaches).

## 💡 Writer's Insight

The most terrifying part of entirely entrusting Terraform code to AI is **Security**. Code generated by generic prompts frequently opens all firewall ports (`0.0.0.0/0`) for convenience or grants unnecessarily broad permissions to IAM Roles.

The core of the Pro version prompt above is that it **forces the AI to adopt the 'principle of least privilege' and 'modularization'.** In practice, I highly recommend using this prompt to establish a robust foundation, and then running the output through static security analysis tools like `tfsec` or `checkov` in your CI/CD pipeline for secondary verification. You will experience the magic of reducing days of initial infrastructure setup down to just 3 minutes.

## 🙋 FAQ

- **Q: Can I use this for GCP or Azure instead of AWS?**
  - A: Absolutely. Simply change the Provider to 'Google Cloud' or 'AzureRM' in the `[Context]` and `[Constraints]` sections, and modify the requested service names to match the respective cloud vendor's terminology. It will work flawlessly.

- **Q: The AI keeps generating code with outdated syntax (e.g., v0.11), causing errors.**
  - A: This is a common issue when using older AI models (like GPT-3.5). Although the prompt explicitly requests the 'latest Terraform syntax', I highly recommend using coding-specialized models like **Claude 3.5 Sonnet** or the latest **GPT-4o** for infrastructure code generation. They consistently output much more stable and modern HCL code.

## 🧬 Anatomy of the Prompt (Why it works)

1. **Role Assignment:** By applying the persona of a '10-year Cloud Architect', we guide the AI to go beyond merely spinning up resources and instead generate enterprise-grade architectural code that considers security and availability.
2. **Constraints:** Enforcing the latest HCL syntax and `terraform fmt` compliance guarantees a clean, standardized output that can be immediately executed (`terraform init && terraform apply`) after copying.
3. **Security & Exception Control:** Explicitly stating the 'principle of least privilege' and 'no insecure defaults' preemptively blocks fatal firewall (SG) and permission (IAM) abuse incidents in production environments.

## 📊 Proof: Before & After

There is a stark contrast in the quality of HCL code generated by the AI when simply asking "make this" versus using a structured prompt.

### ❌ Before (Simple Input)

```text
Create a VPC in AWS with 2 public subnets and 2 private subnets.
```

**Result (Hardcoded & Vulnerable):**

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
# Hardcoded without variables, missing Tags, unscalable single-file structure
```

### ✅ After (Using the Pro Prompt)

**Result (Modularized & Production Standard):**

```hcl
# main.tf
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.project_name}-vpc"
  cidr = var.vpc_cidr

  azs             = var.availability_zones
  private_subnets = var.private_subnets
  public_subnets  = var.public_subnets

  enable_nat_gateway     = true
  single_nat_gateway     = false # NAT per AZ for High Availability
  one_nat_gateway_per_az = true

  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "Terraform"
  }
}
```

_(Note: `variables.tf` and `outputs.tf` are also generated perfectly separated alongside this.)_
