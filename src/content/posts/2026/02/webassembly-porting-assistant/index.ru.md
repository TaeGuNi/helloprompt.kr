# 🦀 Портирование в WebAssembly (Wasm): Пробиваем потолок веб-производительности

- **🎯 Рекомендуется для:** Фронтенд-разработчиков, нуждающихся в срочной оптимизации производительности, и инженеров, планирующих перенос тяжелой бэкенд-логики (C++/Rust/Go) в браузер.
- **⏱️ Затраченное время:** От дней мучительной настройки биндингов → сокращается до 5 минут.
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet (превосходно справляется с конвертацией кода), GPT-4o

> _"Вы можете сутками оптимизировать JS, чтобы получить 10% прироста, но перенос тяжелых вычислений на Wasm ускоряет их в 10 раз. Однако как сделать само 'портирование' безболезненным?"_

Современный браузер — это уже давно не просто программа для чтения документов. Обработка видео, сложная криптография, массивные математические вычисления — неизбежно наступает момент, когда возможностей движка V8 для JavaScript становится недостаточно. WebAssembly (Wasm) кажется идеальным решением, но процесс преобразования существующего кода на C++ или Rust, настройка разделяемой памяти и написание биндингов для JS обычно оборачивается настоящим кошмаром.

Этот промпт выходит за рамки простого перевода кода. Он выступает в роли полноценного «ассистента по портированию в Wasm», который с первого раза генерирует безопасный от утечек памяти код и необходимые JS-мосты.

## 💡 Авторский комментарий (Insight)

Главный барьер, о который спотыкаются разработчики при портировании в WebAssembly — это вопрос: **"Как передать JS-массив в Wasm и как получить его обратно?"**. Передать простое число или строку легко, но когда дело доходит до буферов изображений или аудиоданных (Float32Array), необходимо напрямую управлять памятью.

Ключевая ценность этого промпта кроется в `пункте 2 (Архитектура памяти)`. Если просто попросить ИИ написать код, он часто выдает неэффективное решение, копирующее память целиком. Но когда мы явно требуем «минимизации затрат на копирование» и «использования разделяемой памяти», ИИ генерирует код истинно продакшен-уровня, используя мощные возможности `wasm-bindgen` (например, прямой доступ к памяти по указателям). Кроме того, пример интеграции с React сразу решает проблемы с асинхронным таймингом загрузки Wasm в средах Webpack или Vite.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Мой исходный код слишком огромный, чтобы вставить его за один раз. Что делать?**
  - Ответ: Не пытайтесь скопировать весь проект целиком. Рекомендуется разбивать логику на ключевые функции и запускать промпт для каждой из них. Если модуль имеет сложную систему зависимостей, сначала опишите его архитектуру в секции `[Context]`, чтобы ИИ понял общую картину и сгенерировал правильные биндинги.

- **Вопрос: Как скомпилировать полученный код на Rust?**
  - Ответ: Проще всего использовать официальный тулчейн `wasm-pack`. ИИ обычно указывает команду сборки (например, `wasm-pack build --target web`) в конце своего ответа. Просто скопируйте её и выполните в терминале — на выходе вы получите папку `pkg`, полностью готовую к использованию.

- **Вопрос: В чем разница между портированием C++ через Emscripten и связкой Rust + wasm-bindgen?**
  - Ответ: Emscripten отлично подходит для полного переноса больших C/C++ проектов в веб, но он генерирует тяжелые файлы, а взаимодействие с JS выглядит довольно громоздко. В то же время Rust + `wasm-bindgen` создает минимальные по размеру бандлы и обладает превосходной совместимостью с экосистемой TypeScript. Именно поэтому он стал де-факто стандартом во фронтенд-сообществе, и наш промпт нацелен именно на переход к Rust.

## 🧬 Анатомия промпта (Why it works?)

1. **Ориентация на комплексное E2E-решение:** Промпт не останавливается на банальном переводе кода. Он требует интеграции с фронтенд-фреймворком (React) и примеров рендеринга, выдавая «готовый к продакшену продукт».
2. **Строгие ограничения по производительности (Constraints):** Главная цель внедрения Wasm — скорость. Ключевые слова вроде «минимизация копирования» и «отказ от сериализации» лишают ИИ возможности использовать простые, но медленные обходные пути.
3. **Безопасность типов (Type Safety):** Автоматическая генерация файлов `.d.ts` стирает размытую границу типов между мирами JS и Wasm, полностью исключая ошибки времени выполнения (Runtime Errors) на этапе разработки.

```rust
    // ... логика фильтрации ...

```

### ✅ После (Ответ ИИ при использовании Pro-промпта)

```rust
// Прямой доступ по указателю без лишнего копирования (Zero-copy)
#[wasm_bindgen]
pub fn apply_filter_in_place(data: &mut [u8], width: usize, height: usize) {
    // Прямой доступ и манипуляции с линейной памятью Wasm, 
    // на которую указывает Uint8Array из JS
    for i in (0..data.len()).step_by(4) {
        // Применение фильтра напрямую к каналам Red, Green, Blue
        let r = data[i];
        let g = data[i + 1];
        let b = data[i + 2];

        data[i] = (r as f32 * 0.393 + g as f32 * 0.769 + b as f32 * 0.189) as u8;
        // ... (пропущено) Накладные расходы на аллокацию/освобождение памяти полностью устранены
    }
}
```
