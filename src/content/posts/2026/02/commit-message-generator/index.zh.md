## 📝 “提交信息推荐一下”烦恼终结！自动生成 Conventional Commits

- **🎯 推荐对象：** 初级开发人员、开源贡献者、刚开始协作的团队
- **⏱️ 节省时间：** 从 5 分钟缩短至 10 秒
- **🤖 最佳性能：** ChatGPT (GPT-4o), Claude 3.5 Sonnet, Gemini 2.5 Pro

- ⭐ **难度：** ⭐☆☆☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“写代码用了 3 小时，纠结提交信息用了 30 分钟……你是否也有过只想打个 'Update code' 就赶紧下班的时候？”_

`fix: bug`, `update code`, `最终版_真的_最后一次修改`……今天你是不是又在 GitHub 上留下了这些毫无意义的提交日志，然后匆忙打卡下班了？
还是说，对着 `git commit -m "..."` 命令后闪烁的光标，因为不知道该写什么而盯着显示器发呆了 5 分钟？
写代码可能需要 3 小时，但每当要在终端编写提交信息（Commit Message）时，大脑就会瞬间断片。这种痛苦无论是前端还是后端开发人员，几乎每天都在经历。
尤其是对于需要用英文编写的全球化项目，再加上必须遵守像 **Conventional Commits** 这样严格的规范时，提交代码不再只是简单的版本管理，而变成了一场**棘手的写作考试**。“这个变更怎么用简短的英文概括？”、“虽然是新功能，但属于 feat 吗？还是因为修改了现有代码算 refactor？”、“Issue 编号该贴在哪里才不会报错？”
在经历了一番纠结后，我们往往会因为查字典查得精疲力竭，最后随手留下一个最简单、最没诚意的消息。

然而，这种敷衍的提交信息**迟早会像雪球一样越滚越大，最终无情地坑到未来的你和你的同事。**
想象一下，几个月后生产环境出现了致命 Bug，需要紧急回滚（Rollback）代码。面对几十条只写着 `fix: bug` 的提交日志，想要追踪问题是从哪次提交开始的，简直比大海捞针还痛苦。
更大的问题会体现在**团队协作过程**中。当同事评审你的 PR（Pull Request）时，如果无法理解你修改代码的意图或选择这种方式的原因，他们宝贵的工作时间就会浪费在解读代码上。
特别是在开源项目或具有系统化代码评审文化的科技公司中，敷衍的提交信息往往被直接等同于**开发人员能力和职业素养的缺失**，从而导致绩效评估偏低。在如今 CI/CD 流水线自动化的时代，提交信息直接关系到发行说明（Release Notes），不符合规则的信息甚至可能导致自动部署流程中断。
只写好代码的时代已经彻底结束了。<b>完美的提交信息是让你的代码脱颖而出的、最强大且最优雅的包装。</b>

那么，难道每次都要翻阅英文词典和规范指南文档去强行挤出句子吗？绝对不是。
从现在起，你完全没有必要为了写个提交信息而反复折腾翻译机。只需将你辛辛苦苦编写的代码**变更事项（`git diff`）**结果，或者一段简单的中文工作总结扔给提示词即可。
AI 将严丝合缝地遵循开源生态系统中的绝对全球标准——**Conventional Commits 规范**，即刻为你生成可以直接复制并粘贴到终端的**完美英文提交信息**。
这不仅仅是简单的单词翻译。AI 会仔细分析代码行增加或删除的上下文，精准地捕捉到你未能明确指出的工作根本意图（Why）和解决方式（What），从而完成一份令人惊叹的提交日志。

一旦将此提示词引入实战，你每天的下班时间至少能缩短 10 分钟。
你可以省下消耗在编写提交信息这种非生产性工作上的认知精力，全身心地投入到核心业务逻辑实现和架构设计思考中。
最重要的是，你将完美蜕变为一名专业的工程师，赢得共享 GitHub 仓库的同事们的赞美：**“这个人的提交日志真是既整洁又清晰，真是一个非常适合合作的资深级开发人员！”**
现在就去终端输入 `git diff`，通过下面准备好的提示词，亲自体验这种装点编码最后阶段的魔幻自动化吧！

3. 连同逻辑严密的意图（Why）和具体内容（What）在内的正文（Body），都会以完美的英文为你写好。

这是经过数十次尝试后完成的提示词。请复制下面的提示词，根据你的情况填好 `[变量]` 部分，立即投入实战。

### 🥉 Basic Version (基础版)

当你只需要快速生成一行标题时，请轻便地使用。

> **角色：** 你是一名追求完美 Git 历史记录的资深开发人员。
>
> **请求：** 请查看我提供的变更内容，并给出 3 个符合 Conventional Commits 标准的英文提交信息标题（格式为 `type(scope): subject`）。

### 🥇 Pro Version (专业版)

当你需要从标题、正文到脚注都完美结构化的高质量提交信息时，请使用此版本。复制下面的提示词并直接粘贴到 AI 窗口即可。

> **角色 (Role):**
> 你是一名管理着众多开源项目、对完美的 Git 历史记录有着近乎执念的严厉维护者，也是一名资深工程经理。
>
> **场景 (Context):**
>
> - 背景：我刚刚完成了代码编写，现在需要提交代码。
> - 目标：编写一份便于同事评审、且对日后追踪历史记录完美的提交信息。
>
> **请求 (Task):**
> 分析下方的 `[变更事项]`，并根据 **Conventional Commits v1.0.0** 规则提供 3 个完美的提交信息草案选项。
>
> 1. **格式：** `<type>(<scope>): <subject>` (标题 1 行)
> 2. **正文 (Body)：** 使用项目符号 (`-`) 明确总结为什么要改 (Why) 以及如何解决的 (What)。
> 3. **脚注 (Footer)：** (可选) 如果有相关的 Issue 编号，请以 `Closes #Issue编号` 的形式编写。
>
> **约束条件 (Constraints):**
>
> - Type 必须且只能使用以下之一：`feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`
> - Subject (标题) 不得超过 50 个字符，要简洁，且必须以 **英文命令式动词（原形）** 开头（例如：`Add`, `Fix`, `Update`）。结尾不要加句号 (`.`)。
> - 正文 (Body) 与标题之间空一行，并使用英文编写。
> - 请用中文简短地说明每个选项的区别或侧重点（例如：侧重修复 Bug，侧重 UI 改进）。
>
> **输入 (Input):**
>
> - 变更事项：`[在这里输入 git diff 结果，或者直接用中文输入你修改的内容]`

## 💡 作者点评 (Insight & How to use)

这个提示词最压倒性的优势在于，通过一次简单的复制粘贴，完美消除了每天困扰开发人员的两大障碍：**“英文写作的心理压力”**和**“严苛复杂的格式规则”**。

在提交代码时，除了简单地说“我把这个按钮颜色改成了蓝色”之外，尝试在终端环境中执行 `git diff` 命令，然后将屏幕上显示的红绿交错的复杂代码变更直接拖拽复制，粘贴到提示词的 `[变更事项]` 区域。令人惊讶的是，AI 会自动推断并分析你动了哪些文件、修改了哪些变量名，以及这些变更在整个系统中的上下文。它能自发完成高质量的提交信息，精准捕捉到开发人员未能用言语表达出的工作意图和逻辑变化。

<b>💡 100% 贴合实战的变量控制法 (Constraint Control):</b>
这个提示词之所以与普通的聊天机器人对话有着云泥之别，是因为它通过**约束条件（Constraints）**极端地控制了 AI 的幻觉（Hallucination）和自由度。Conventional Commits 的规则比想象中要严苛得多。例如，消息必须以**动词原形（英文命令式）**开头，标题结尾加句号 (`.`) 会导致 Lint 报错。此外，如果总长度超过 50 个字符，消息在 GitHub 界面上会被截断。
这个提示词经过精心设计，预先屏蔽了资深工程师在代码评审中可能会指出的所有潜在错误因素。如果你的团队要求必须包含特殊前缀或 Jira 票据编号，只需在约束条件部分增加一行条件：`- 请务必在标题最前面加上 '[ABC-123]' 格式的 Jira 票据编号`。AI 将在保持原有严苛规则的同时，完美合并并输出你团队专属的自定义规范。

<b>🚀 极速生产力的编辑器联动技巧：</b>
进一步提升这个提示词价值的秘诀其实另有所在。每次提交都要打开浏览器访问 ChatGPT 或 Claude 网站，终究还是会觉得麻烦。请将此提示词的完整内容作为模板，预设在最近在开发人员中大受欢迎的 **Cursor IDE**、**GitHub Copilot Chat** 或基于终端的 AI CLI 工具的 **System Prompt（系统提示词）** 设置中。
完成所有编码后，只需对编辑器内置的 AI 下达一个简短的指令：**“为刚才修改的内容写个提交信息”**。编辑器会自动读取当前的 `git diff` 状态，并根据这个完美设定的提示词规则在 1 秒内生成提交信息。你只需点击 `Accept` 按钮或复制提交即可。仅仅通过缩短这一步操作流程，你一天的开发效率和专注度就会呈现出超乎想象的飞跃式增长。

## 🙋 常见问题 (FAQ)

- **Q: 公司内部有专门的提交规则（例如：必须包含 Jira 票据编号），该如何应用？**
  - A: 只需在提示词的 **约束条件 (Constraints)** 部分增加一行：`- 请务必在标题最前面加上 '[ABC-123]' 格式的 Jira 票据编号`，即可获得完全遵守团队特有规范的结果。

- **Q: 标题想用英文，但提交正文 (Body) 想用中文写，可以吗？**
  - A: 请将提示词中的 `正文 (Body) 与标题之间空一行，并使用英文编写。` 修改为 `正文请使用中文清晰地编写。` 这样就能在保持全球标准外壳的同时，实现与团队成员沟通效率的最大化。

## 🧬 提示词解剖 (Why it works?)

1. **强制执行标准规范 (Conventional Commits v1.0.0)：** 与其模糊地要求 AI “写个好的消息”，不如直接指示全球通用的明确标准规范，从而强力固定输出格式。
2. **明确的约束条件 (Constraints)：** 通过在提示词层面预先拦截“以命令式开头”、“50字限制”、“禁止句号”等资深开发人员会纠结的严苛规则，将后续的修改成本降至为零。
3. **赋予选择权 (3个选项)：** 设计上避免让 AI 强加唯一的答案。通过同时提供 3 个性质不同的选项，用户可以根据自己工作意图的核心，挑选最匹配的最佳消息。
