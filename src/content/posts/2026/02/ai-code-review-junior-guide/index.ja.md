# 📝 シニアエンジニアのようにコードレビューする：ジュニア開発者のためのAIプロンプト

- **🎯 おすすめの対象:** コードレビューに不安を感じるジュニア開発者（1〜3年目）、メンターなしで働くソロ開発者
- **⏱️ 所要時間:** コード分析 30分 → 1分に短縮
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet（コード分析において最高峰）、GPT-4o

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「自分のコードに、致命的なバグが潜んでいたらどうしよう…？」プルリクエスト（PR）の作成ボタンを押す前に、冷や汗を流したことはありませんか？_

ジュニア開発者にとって、最も緊張する瞬間の一つが**コードレビュー（Code Review）**です。先輩エンジニアから「この変数名はどういう意図ですか？」「ここにメモリリークの可能性が見えますが、仕様ですか？」と指摘されると、頭が真っ白になってしまうことも多いでしょう。逆に、自分が同僚のコードをレビューする立場になった時も、「自分が指摘してもいいのだろうか？」と遠慮してしまい、結局「LGTM (Looks Good To Me)」だけを残して逃げてしまいがちです。

しかし、**AI**をあなたの「専属メンター」にすれば、状況は一変します。単にコードを書いてもらうのではなく、**「私のコードを、10年目のシニアエンジニアの鋭い視点で事前にレビューして」**と依頼するのです。2026年現在、最も効果的な**「AIシニアコードレビュアープロンプト」**を活用し、自分のコードの品質を守り、技術面接レベルの質の高いフィードバックを得る方法をご紹介します。

---

## ⚡️ 3行要約 (TL;DR)

1. **事前レビューで自信を獲得：** PRを出す前に、まずAIに厳しくチェックしてもらい、欠陥を修正しておくことで、基礎のしっかりした開発者として評価されましょう。
2. **多角的な分析を依頼：** バグ、セキュリティ脆弱性、可読性、パフォーマンスの4つの観点から、コードを徹底的に分解・分析するよう指示します。
3. **モバイルフレンドリーなフィードバック：** 可読性を高めるため、AIの回答形式を「表（Table）」ではなく「構造化されたリスト（List）」に強制し、スマートフォンからでも快適にレビュー内容を確認できるようにします。

## 🚀 解決策：「10年目の厳しいシニアレビュアーを召喚」

### 🥉 Basic Version (基本型)

素早く全体的なコードの品質だけをチェックしたい場合に使用してください。

> **役割：** あなたは10年目のシニアバックエンドエンジニアです。
> **タスク：** 以下のコードに論理的なエラーや可読性の問題がないか確認し、どのように修正すべきか3つのポイントで指摘してください。
> **コード：** `[ここにコードを貼り付け]`

<br>

### 🥇 Pro Version (プロ型)

PRを提出する直前や、難解なレガシーコードを分析する際など、より詳細な品質検証が必要な場合に使用してください。

> **役割 (Role)：**
> あなたは10年以上の経験を持つ、厳しくも親切なシニアソフトウェアエンジニア（Senior Software Engineer）です。特にクリーンコード（Clean Code）、セキュリティ脆弱性分析、システムパフォーマンス最適化の分野におけるトップクラスの専門家です。
> 
> **状況 (Context)：**
> 
> - 背景：私はジュニア開発者であり、チームに提出するPR（Pull Request）を準備しています。
> - 目標：自分のコードに潜む致命的な欠陥を見つけ出し、シニアエンジニアの視点からコードをよりエレガントにリファクタリングする方法を学びたいです。
> 
> **タスク (Task)：**
> 
> 以下の `[Input Code]` を徹底的にコードレビュー（Code Review）してください。次の4つの観点から潜在的な問題点を分析し、具体的な改善案を提示してください。
> 
> 1. バグと論理エラー (Bugs & Logic)：ランタイムエラーやエッジケースの処理漏れ
> 2. セキュリティ脆弱性 (Security)：SQLインジェクション、XSS、権限奪取などのセキュリティリスク
> 3. 可読性と保守性 (Readability)：命名規則、単一責任の原則（SRP）の違反の有無
> 4. パフォーマンス最適化 (Performance)：不要な演算、N+1問題、メモリリーク
> 
> **制約事項 (Constraints)：**
> 
> - モバイルでの可読性を考慮し、出力形式はMarkdownの表（Table）ではなく、**構造化された箇条書きリスト（Bullet List）**形式で作成すること。
> - 単なる批判ではなく、「建設的なフィードバック（Constructive Feedback）」を提供すること。
> - 「なぜ（Reasoning）」そのように修正すべきなのか、ジュニア開発者が理解できるように親切かつ詳細に原理を説明すること。
> - もしコードが完璧であれば、称賛の言葉とともに「LGTM 🚀」と出力すること。
> 
> **入力データ (Input Code)：**
> 
> ```python
> [ここにあなたのコードを貼り付けてください]
> ```

## 💡 筆者のインサイト (Insight)

このプロンプトを実務に導入した結果、コードレビューの準備にかかる時間が従来と比べて**40%以上短縮**されました。特に、先輩エンジニアが忙しかったり、そもそもメンターがいない環境で働くジュニア開発者にとって、このプロンプトは「仮想のシニアメンター」としての役割を完璧に果たしてくれます。

最大のメリットは、単にコードを修正してもらうだけでなく、**「なぜ（Why）修正すべきなのか」**を学習できる点にあります。レビューを通じて得た「根拠（Reasoning）」をもとに、実際のチームメンバーに対して自分のコードの設計意図を論理的に説明できるようになり、チーム内での信頼度も大きく向上しました。ただし、AIが提案するアーキテクチャやライブラリの導入は、チームの既存の規約（コンベンション）と衝突する可能性があるため、盲信せずにあくまで「参考用」として活用してください。

## 🙋 よくある質問 (FAQ)

- **Q: AIがチームで使っていない最新の構文や外部ライブラリを頻繁に提案してきます。**
  - A: プロンプトの「状況 (Context)」セクションに制約条件を追加してください。例：`私たちのチームはPython 3.8の構文のみを使用しており、外部ライブラリ（pandasなど）の導入は禁止されています。標準ライブラリのみを活用して改善案を出してください。`

- **Q: 会社のコードをそのままコピーして入力しても大丈夫ですか？セキュリティの問題が心配です。**
  - A: 機密情報（APIキー、DBのパスワード、独自のビジネスロジックなど）は絶対にそのまま入力しないでください。変数名を一般化したり、コアとなるロジックだけを抽象化したスニペット単位に切り出して質問することをお勧めします。エンタープライズ環境であれば、入力データが学習に使用されない法人向けプラン（Team / Enterprise）のAI環境で利用することを強く推奨します。

## 🧬 プロンプトの解剖 (Why it works?)

1. **ペルソナの具体化 (Role):** 単なる「エンジニア」ではなく、「10年目」「厳しくも親切な」「クリーンコードの専門家」という詳細な設定を与えることで、フィードバックの深さとトーンを完全にコントロールしています。
2. **多次元的な分析の強制 (Task):** バグ、セキュリティ、可読性、パフォーマンスという4つの明確な評価基準を提示することで、AIが単純な構文エラーだけを指摘して終わるのを防ぎます。
3. **モバイル最適化 (Constraints):** コードブロックを含む表（Table）は、スマートフォンの画面では横スクロールが発生し、非常に読みにくくなります。これをリスト形式に強制することで、いつでもどこでもレビュー結果を快適にスキャンできるようUXを配慮しました。

## 📊 証明：Before & After

### ❌ Before (入力)

```python
def get_user_data(user_id):
    db = connect_to_db()
    query = "SELECT * FROM users WHERE id = " + str(user_id)
    result = db.execute(query)
    if result:
        return result[0]
    else:
        return None
```

### ✅ After (結果)

```markdown
ジュニアさん、コード作成お疲れ様でした！全体のロジックの流れはしっかり把握できていますが、実務環境において致命的になり得るいくつかのポイントを発見したため、レビューを残します。

- **🚨 セキュリティ (Security): SQLインジェクションの脆弱性**
  - **問題点:** 文字列の結合（`+ str(user_id)`）によってSQLクエリを生成しています。悪意のあるユーザーが `user_id` の値として `1 OR 1=1` などを入力した場合、データベース全体が不正にアクセスされる危険性があります。
  - **改善案:** パラメータ化クエリ（Parameterized Query）を使用する必要があります。
  - **修正コード:** `query = "SELECT * FROM users WHERE id = %s"; db.execute(query, (user_id,))`

- **⚙️ バグと論理エラー (Bugs & Logic): リソースリーク (Resource Leak)**
  - **問題点:** データベースのコネクションを開いたまま（`connect_to_db()`）、閉じる処理が行われていません。このコードが繰り返し実行されると、コネクションプールが枯渇し、サーバーがダウンする可能性があります。
  - **改善案:** `finally` ブロック、またはコンテキストマネージャー（`with` 文）を使用して、必ずDB接続を解放する必要があります。

- **📖 可読性 (Readability): 冗長な条件分岐**
  - **問題点:** `if result:` の構文は正しく書けていますが、もう少しPythonらしい（Pythonic）簡潔なコードに改善できます。
  - **改善案:** `return result[0] if result else None` のように一行にまとめるか、フレームワークが提供する `.fetchone()` などのメソッドの活用を検討してみてください。
```
