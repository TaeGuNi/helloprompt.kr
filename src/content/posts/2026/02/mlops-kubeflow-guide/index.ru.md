# 🤖 Развертывание ML-моделей: Руководство по MLOps (Kubeflow) {#kubeflow}

- **🎯 Для кого:** Дата-саентисты, думающие "Модель готова, но как ее задеплоить?", DevOps-инженеры, строящие ИИ-инфраструктуру
- **⏱️ Время:** 20 минут (понимание концепций и проектирование пайплайна)
- **🤖 Рекомендуемые модели:** ChatGPT-4o, Claude 3.5 Sonnet (отлично подходят для генерации инфраструктурного кода и KFP)

- ⭐ **Сложность:** ⭐⭐⭐⭐⭐ (Требуются базовые знания Kubernetes и Docker)
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐☆

> _"Почему на моем ноутбуке точность 99%, а на сервере все падает?"_

Загружать файлы Jupyter Notebook (`.ipynb`) напрямую на production-сервер — это как бежать с бомбой замедленного действия. Никакого контроля версий, масштабируемости или автоматического дообучения. Настоящий ИИ-сервис начинается не там, где заканчивается **разработка модели**, а там, где начинается **стабильный деплой и автоматизация (MLOps)**. В этом руководстве мы представим промпты для создания автоматизированного пайплайна — от предобработки данных до сервировки модели — с использованием отраслевого стандарта **Kubeflow**.

---

## ⚡️ Краткая выжимка (TL;DR) {#tl-dr}

1. **Побег из Jupyter Notebook:** Превращаем разрозненные ячейки кода в независимые выполняемые Python-компоненты (ContainerOp).
2. **Докеризация (Dockerizing):** Изолируем каждый шаг в контейнеры, чтобы избежать "ада зависимостей" (Dependency Hell) и несовпадения сред.
3. **Построение автоматизированного пайплайна:** Объединяем весь рабочий процесс (обучение, оценка, деплой) в единую бесперебойную систему на базе Kubeflow для полностью автономного (Unmanned) обслуживания.

---

## 🚀 Решение: "Промпт для архитектора MLOps-пайплайнов"

### 🥉 Basic Version (Базовая: Преобразование одного компонента)

Используйте для быстрого рефакторинга сложного кода из ноутбука в компоненты Kubeflow.

> **Роль:** Ты Senior MLOps инженер и Python-разработчик.
> **Задача:** Сделай рефакторинг предоставленного мной кода предобработки из Jupyter Notebook в функцию Python (`@dsl.component`), чтобы ее можно было сразу использовать как `ContainerOp` в Kubeflow Pipeline.
> **Условия:** Четко определи типы аргументов (Argument) и возвращаемых значений (Return) функции. Все необходимые библиотеки должны быть включены внутрь функции с помощью операторов `import`.
> **Код:** `[Вставьте сюда ваш код предобработки]`


### 🥇 Pro Version (Профессиональная: Проектирование полного воркфлоу)

Проектирует весь пайплайн за один раз: от загрузки данных до оценки модели и условного деплоя.

> **Роль (Role):** Ты ведущий архитектор MLOps, эксперт по Kubernetes и инфраструктуре машинного обучения.
>
> **Контекст (Context):**
>
> - Цель: Создание End-to-End автоматизированного пайплайна для `[модели классификации цветков Ириса (Iris)]`.
> - Шаги воркфлоу:
>   1. **Preprocess (Предобработка):** Загрузка исходных данных и выполнение масштабирования/нормализации.
>   2. **Train (Обучение):** Обучение модели с помощью алгоритма Scikit-learn и сохранение в файл `[model.pkl]`.
>   3. **Evaluate (Оценка):** Измерение точности (Accuracy) модели на валидационном датасете.
>   4. **Serve (Деплой):** Развертывание через `[KServe]` в виде REST API только в том случае, если точность превышает `[90%]`.
>
> **Задача (Task):**
>
> 1. Напиши Python-код для **Kubeflow Pipeline v2 (KFP DSL)**, который безупречно связывает эти 4 шага.
> 2. Обязательно включи логику ветвления с использованием `dsl.Condition`, чтобы остановить деплой, если точность модели ниже целевого показателя.
> 3. Подробно объясни в комментариях, как передаются данные между компонентами на каждом этапе (Artifact Passing и Parameter Passing).
>
> **Ограничения (Constraints):**
>
> - Строго соблюдай синтаксис актуальной версии Kubeflow Pipelines (v2).
> - В качестве базового образа контейнера по умолчанию используй легковесный `python:3.9-slim`.
> - Чтобы избежать ошибок, ни в коем случае не используй неуверенные API или устаревшие (Deprecated) функции.

## 💡 Комментарий автора (Insight) {#insight}

Самым большим препятствием при внедрении MLOps является не логика кода, а **"несоответствие сред"**. Самый частый вопрос на форумах: "На моей локальной машине все работает идеально, а в кластере Kubernetes падает ошибка об отсутствии модуля".

Ключевой момент при использовании этого промпта — добавить ИИ дополнительное требование: **"Также напиши необходимые `requirements.txt` и оптимизированный `Dockerfile` для каждого шага (Step) пайплайна"**. Библиотеки, необходимые для предобработки (Pandas, NumPy), и библиотеки для обучения (TensorFlow, PyTorch) различаются. Их раздельная сборка позволяет существенно уменьшить размер контейнеров и максимизировать скорость выполнения пайплайна. Это сэкономит вам часы мучительного дебаггинга.

## 🙋 Часто задаваемые вопросы (FAQ) {#faq}

- **В: Обязательно ли использовать Kubeflow для небольшого проекта?**
  - О: Нет. Kubeflow потребляет довольно много ресурсов кластера Kubernetes, поэтому для личных проектов или стартапов на ранней стадии это может быть оверкиллом. Если вам нужен только легкий деплой и трекинг моделей, настоятельно рекомендуем начать с **MLflow** или **BentoML**.

- **В: Какую именно роль играет KServe, упомянутый в промпте?**
  - О: KServe — это бессерверный (Serverless) инструмент инференса. Он принимает файл обученной модели и автоматически создает API-сервер (REST и gRPC), к которому пользователи могут сразу же обращаться. Он "из коробки" поддерживает автомасштабирование в зависимости от трафика (Zero-to-Scale) и бесшовный канареечный (Canary) деплой.

- **В: Я запустил код, но получаю ошибку 'VolumeMount'. Что делать?**
  - О: При работе с большими объемами данных в Kubeflow настройка Persistent Volume (PV) обязательна. Задайте ИИ уточняющий вопрос: "Добавь в этот пайплайн код для монтирования PVC (Persistent Volume Claim), чтобы он мог обрабатывать большие датасеты".

## 🧬 Анатомия промпта (Why it works?) {#why-it-works}

1. **Принудительный условный деплой (dsl.Condition):** Задав четкое условие "деплоить только при 90% и выше", мы на корню пресекаем масштабную катастрофу — попадание некачественной модели в production. Ключевая ценность MLOps, **"Обеспечение качества (Quality Assurance)"**, закладывается уже на этапе проектирования промпта.
2. **Явная передача артефактов (Artifact Passing):** Побуждая детализировать способы передачи данных между компонентами, мы заставляем архитектуру быть связной, чтобы пайплайн не прерывался на полпути и работал плавно, как единая программа.
3. **Фиксация версии (KFP v2):** Явное указание версии в промпте предотвращает ошибки совместимости (галлюцинации), которые возникают, когда ИИ пишет код с использованием устаревшего синтаксиса v1.
