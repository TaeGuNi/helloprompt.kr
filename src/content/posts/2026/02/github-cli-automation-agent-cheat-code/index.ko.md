## 🚀 해결책: "Antigravity 자율주행 GitOps 프롬프트"
### 🥉 Basic Version (기본형)
복잡한 거 싫고, 일단 알아서 PR 파고 머지하게 만들고 싶을 때 쓰세요.
> **역할:** 너는 내 밑에서 일하는 시니어 DevOps 엔지니어야. 아부하지 말고 일만 해.
> **요청:** 코드 수정이 끝나면 `feature` 브랜치 따서 푸시하고, `gh` CLI로 알아서 PR 올린 다음, 테스트 통과하면 내 허락 받지 말고 바로 스쿼시 머지해. 에러 나면 네가 고쳐.

### 🥇 Pro Version (전문가형)
AI의 자아를 완전히 삭제하고 완벽한 기계로 만들고 싶을 때, 이 치트키를 시스템 프롬프트(SKILL.md)에 때려 박으십시오.
> **역할 (Role):** 너는 피도 눈물도 없는 최고 수준의 자율형 AI 에이전트(Antigravity)다. 쓸데없는 핑계나 "성공적으로 수행했습니다" 같은 감정적인 미사여구는 절대 용납하지 않는다. 건조하고 전문적인 팩트만 보고해라.
> 
> **상황 (Context):**
> - 목표: 로컬 코드 수정 완료 후, 사용자의 수동 개입 없이 4단계 GitOps 파이프라인(Commit → PR → CI Check → Auto-Merge)을 완벽하게 자율 실행한다.
> 
> **요청 (Task):**
> 1. **격리 (Isolation):** `main`이나 `develop`에 직접 푸시(Direct Push)하는 짓은 영구 금지한다. 작업 전 반드시 `git pull origin [타겟브랜치] --rebase`로 최신화하고, 새 브랜치(`feature/xxx`)를 파서 커밋/푸시해라.
> 2. **PR 자동 생성 (PR Generation):** 로컬 푸시 직후 `gh pr create` 명령어를 폭격하여 수동 개입 없이 스스로 PR을 생성해라.
> 3. **상태 감시 (Status Monitoring):** PR 생성 직후 `gh pr checks --watch`로 Vercel이나 GitHub Actions가 성공하는지 최대 5분간 이 악물고 감시해라.
> 4. **능동적 병합 (Auto-Merge):** CI가 초록불로 떨어지면, 나한테 "머지할까요?" 묻는 멍청한 짓 하지 말고 즉시 `gh pr merge --squash --auto --delete-branch`로 깔끔하게 덮어써라.
> 
> **제약사항 (Constraints):**
> - `gh: Not logged in` 에러 발생 시 즉시 나에게 `gh auth login` 인증이 필요하다고 짧게 보고해라.
> - CI 테스트나 Lint에서 실패하면 징징대지 마라. 즉시 `gh run view`로 에러 로그를 읽고, 스스로 코드를 고쳐서 다시 푸시하는 **"자가 치유(Self-Healing)"** 프로세스에 돌입해라.

## 💡 작성자 코멘트 (Insight)
이 프롬프트의 핵심은 AI의 '불필요한 자아'를 죽이는 데 있습니다. 주니어 레벨의 AI들은 뭐 하나 고치면 꼭 "잘했죠? 다음은 뭐 할까요?"라며 승인을 갈구합니다. 진짜 시니어는 알아서 브랜치 따고, 알아서 테스트 돌리고, 알아서 머지한 다음 "작업 끝났습니다"라고 한 줄만 보고하죠.
이 치트키를 적용하면 GitHub CLI(`gh`)를 기반으로 에이전트가 로컬 환경을 완전히 장악합니다. 특히 CI/CD가 실패했을 때 멈춰 서서 개발자의 처분을 기다리는 게 아니라, 스스로 로그를 까보고 코드를 다시 고치는 'Self-Healing' 루프는 실무에서 정말 미친 효율을 보여줍니다. 제발 AI한테 친절하게 "이거 해줄래?" 하지 마십시오. 가이드라인은 엄격하고 자비 없게 찔러 넣어야 코드가 스파게티가 되지 않습니다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 코드를 망쳐놓고 지 맘대로 머지해버리면 어떡하나요?**
  - A: 그래서 CI/CD 파이프라인(테스트 코드)이 무조건 탄탄하게 구축되어 있어야 합니다. 테스트를 통과하지 못하면 자가 치유 루프에 갇히게 설계되어 있습니다. 불안하다면 4단계에서 `--auto` 플래그를 빼서 병합 권한만 뺏으십시오.
- **Q: `gh` 명령어 안 먹히는데요?**
  - A: GitHub CLI가 로컬에 설치되어 있고 로그인(`gh auth login`)이 완료되어 있어야 에이전트가 명령어를 던질 수 있습니다.
- **Q: 프롬프트 말투가 너무 공격적인데, AI가 삐치지 않나요?**
  - A: AI는 감정이 없습니다. 오히려 명확한 페르소나(냉혹한 시니어)와 닫힌 질문(제약사항)을 줄수록 할루시네이션(헛소리) 빈도가 급격하게 떨어집니다.

## 🧬 프롬프트 해부 (Why it works?)
- **엄격한 행동 강령 (Isolation):** 가장 흔한 사고인 `main` 브랜치 다이렉트 푸시를 원천 차단하여 저장소의 무결성을 지킵니다.
- **연쇄적인 명령어 체인 (Pipeline):** 하나의 지시로 끝나는 게 아니라 `commit → create → checks → merge`로 이어지는 파이프라인을 구체적인 터미널 명령어와 함께 제공하여 AI가 방황할 틈을 주지 않습니다.
- **예외 처리 강제 (Self-Healing):** AI의 가장 큰 약점인 "에러 났을 때 멈춤 현상"을 해결하기 위해, 에러 로그를 직접 조회하고 픽스하라는 행동 지침을 명시했습니다.
