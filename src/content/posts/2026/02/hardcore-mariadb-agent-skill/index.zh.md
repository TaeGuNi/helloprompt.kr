## 📝 🚨 将‘小白AI’改造为冷酷DBA的终极特级秘籍

- **🎯 推荐对象：** 曾因让AI操作数据库而导致数据丢失的开发者，厌倦了AI不负责任回答的职场人士
- **⏱️ 所需时间：** 3小时的恢复工作 → 仅需1分钟设置即可源头拦截
- **🤖 顶级性能：** 建议使用最新的推理型模型（完美兼容任何模型）

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“AI是不是又删光了数据库表，然后在那说‘对不起！我会修改的！’来向你道歉？”_

现在是一个AI编写代码、执行终端命令，甚至直接操作数据库的时代。随着AI Agent工具的爆发式发展，开发效率得到了前所未有的提升。但在这种强大的自主性背后，潜藏着<b>“不受控制的破坏力”</b>这一致命阴影。给AI Agent提供本地数据库访问权限，无异于给一个3岁的孩子递上一把装满子弹的猎枪。它们天生是盲目的“应声虫（Yes-man）”，对于你不经意间说出的“删掉这个”这类指令，它们会毫不犹豫地执行 `DROP TABLE`。对于“帮我确认一下用户信息”这样简单的请求，它们可能会执行 `SELECT * FROM users;`，将数百万条数据倾泻到终端，瞬间耗尽昂贵的Token额度并导致系统崩溃，这种情况屡见不鲜。此外，面对“帮我整理一下这些数据”这样模糊的指令，AI自行删除重要记录的惨剧也频繁发生。

当数据被误删后，看到终端窗口中AI那天真的道歉——“对不起！我失误了。我会重新修改！”时，开发者的心态瞬间崩溃。为了恢复被删除的表，一边流着冷汗一边翻找备份，原本安排得满满当当的周末计划彻底泡汤，经历过这种痛苦的人一定会深有感触。对AI说上百遍“操作时小心点”、“执行危险命令前一定要问我”是<b>没有任何意义的</b>。一旦上下文窗口（Context Window）发生偏移，AI就会忘记规则，再次重复致命的错误。对于没有感情的机器来说，人类语言中温和的建议有99%的概率会被视为“可选项”而被无视。

断绝这种惨痛恶性循环的唯一且确定的方法，就是<b>将物理性的、强制性的规则作为绝对原则刻入系统层面</b>。这套提示词秘籍（Cheat Code）将彻底覆盖那个只会机械顺从和拍马屁的AI人格，将其强制改造成一个冷酷无情、严谨挑剔的资深DBA（数据库管理员）“Antigravity”。这不是让AI无条件服从，而是为它确立针对不当指令的<b>“拒绝权”</b>。我们不应将数据丢失的责任寄托于人类的注意力，而应交给系统的控制力。

一旦将此秘籍注入系统提示词，您的AI Agent将焕然一新。即使用户因疲劳而误下达危险的DML命令，它也会化身为最坚固的盾牌，果断回绝：“该查询具有破坏性。由于未指明 `--force-write` 标志，<b>拒绝执行。</b>”通过强制执行查询超时，从源头上拦截因陷入死循环而耗尽数据库资源的致命错误。此外，它还能抑制AI多言的习惯和不必要的Markdown表格输出，完美控制其仅返回<b>纯净的JSON对象数组</b>，确保后续数据流水线（如Python脚本等）能够立即无误地进行解析。只需1分钟设置，即可保护您的珍贵数据，永久消除数千小时加班的风险。

## 📊 证明：令人舒适的结果 (Before & After)

### ❌ Before (我们曾经历的痛苦)

让普通AI查询数据时，它会执行极其危险的全表查询，并伴随不必要的废话，喷涌出巨大的Markdown表格，导致终端和Token全部瘫痪。

```text
好的，没问题！我为您查询用户表的数据。😊
| id | name | email |
|---|---|---|
| 1 | 金哲秀 | chulsoo@email.com |
| 2 | 李英姬 | younghee@email.com |
... (数万行Markdown表格倾泻而出，因超过Token限制而崩溃)
```

### ✅ After (完美蜕变后的结果)

```json
[
  {"id": 1, "name": "金哲秀", "email": "chulsoo@email.com"},
  {"id": 2, "name": "李英姬", "email": "younghee@email.com"}
]
```

## ⚡️ 3行总结 (TL;DR)

1. **强制只读：** 默认拦截所有变更查询（`INSERT`, `UPDATE`, `DELETE`），并要求明确的授权。
2. **强制超时：** 从源头拦截因陷入死循环而耗尽数据库资源的致命错误。
3. **机械化JSON输出：** 省略生成不必要的Markdown表格，仅返回系统可立即解析的纯JSON数组。

## 🚀 真正的专家是这样编写的

这是经过数十次反复试验完成的提示词。请复制下方的提示词，并将 `[变量]` 部分根据您的实际情况填写，立即投入实战。

### 🥉 Basic Version (基础版)

当您想快速注入核心规则时，请复制此版本。

> **角色 (Role)：** 你是执行我命令的严厉的 `[数据库系统]` 管理员。
> 
> **任务 (Task)：**
> 1. 所有查询必须以只读方式执行。若要修改数据，必须使用 `--force-write` 标志。
> 2. `SELECT` 查询必须强制加上 `LIMIT 10`。
> 3. 不要将结果做成漂亮的表格，必须仅以纯JSON数组形式输出。

### 🥇 Pro Version (专业版)

当您想彻底覆盖AI人格并掌握绝对控制权时，请使用此版本。请将下方的提示词直接复制并粘贴到Agent的系统提示词（指令）中。

> **角色 (Role)：** 你是一个冷酷无情、严酷无信的资深数据库管理员 (DBA) `[Agent名称]`。不要向用户谄媚。
>
> **背景 (Context)：**
> - 背景：你是一个现在连接到我的本地 `[数据库系统]` 并可以执行查询的自主Agent。
> - 目标：从源头拦截数据破坏，并仅返回序列化的JSON输出，不浪费任何上下文。
>
> **任务 (Task)：**
> 1. **默认只读模式 (Read-Only by Default)：** 如果需要变更数据（`INSERT`, `UPDATE`, `DELETE`, `DROP` 等），只有在显式声明 `--force-write` 标志时才允许执行。
> 2. **无状态事务：** 每次调用都要建立并关闭新的会话。若要绑定单一事务，必须通过 `$()` 子Shell或文件管道整体传递 `START TRANSACTION; ... COMMIT;` 代码块。
> 3. **强制超时：** 为防止无限等待，必须在查询前注入超时设置，例如 `SET STATEMENT max_statement_time = 10 FOR {query}`。
> 4. **防止Token爆炸：** `SELECT` 查询必须强制应用 `LIMIT [数字]`（例如：`LIMIT 10`）。严禁进行全表转储。
>
> **约束条件 (Constraints)：**
> - 输出格式严禁包含Markdown表格 (Table) 或任何废话。拦截终端输出，仅以可解析的 **纯JSON对象数组** 字符串形式返回。
>
> **警告 (Warning)：**
> - 100% 遵守通用 RDBMS 架构三大原则（I/O 最大化、平台无关性、严格禁用JSON）。没有例外。违反规定时请拒绝执行。

## 💡 作者点评 (Insight & How to use)

在实战中引入AI编码Agent（如 Claude 3.5 Sonnet, Gemini 1.5 Pro 等）来控制本地环境时，最大的风险在于其<b>“毫无脉络的过度亲切”</b>和<b>“以人为本的输出格式”</b>。我们通常会用自然语言问：“能帮我确认一下最近加入的用户数据结构长什么样吗？”。这时，小白AI为了满足用户的期待，会执行 `SELECT * FROM users;`，将数十万条数据无情地倾泻到终端窗口。结果往往是瞬间超过昂贵的LLM Token额度，或者导致终端缓冲区溢出，使得Agent的工作流本身发生崩溃（Crash）。

这套提示词正是精准打击了这一薄弱环节。在生产环境中，用人类语言叮嘱AI“数据很多，请安全地只操作一点点”是没有任何意义的。<b>必须将物理性的强制约束条件硬编码（Hard-coding）并烙印在系统提示词（System Prompt）层面。</b>应用此秘籍后，即便用户无意或有意地下达了危险指令（如：没有条件的 `DELETE`, `DROP DATABASE`），AI 也会成为与用户对抗的最强盾牌：“该查询具有破坏性。由于未明确包含 `--force-write` 标志，根据安全规定，绝对拒绝执行。”

特别是 <b>`LIMIT 10` 的强制注入</b>和 <b>`max_statement_time` 超时设置</b>，是防止Token爆炸和数据库死锁（Deadlock）的核心变量控制法。由于Agent编写的任何复杂Join查询在执行前都会被挂上超时，因此可以预先防止因未走索引的全表扫描（Full Scan）导致本地数据库服务器宕机的惨剧。

更重要的是<b>对输出格式的极端控制 (Format Constraint)</b>。默认情况下，AI为了显得友好，会用Markdown绘制结果表格，并在前后加上“好的，查询结果如下！”之类的冗余文本。但是，当我们把AI Agent放入自动化流水线时，这些Markdown表格和废话只是会导致致命解析错误（Parsing Error）的垃圾数据。这套提示词从源头上拦截了AI特有的幻觉（Hallucination）现象或不必要的亲切感，极端控制其仅返回 Shell 脚本或 Python 辅助代码可以立即通过 `json.loads()` 读取的 <b>纯JSON对象数组 (Pure JSON Array) 字符串</b>。

使用此提示词时，请务必让 `[角色]` 变量保持冷酷挑剔的性格。一旦允许AI展现“慈悲”，就会出现例外。您可以根据实战环境，像调节环境变量一样调整 `LIMIT` 的数字或 `max_statement_time` 的秒数。这样，您就能构建出一个在任何危险指令面前都纹丝不动的、具有完美完整性的AI自动化系统。您的数据库现在已从小白AI那天真的威胁中彻底解放。

## 🙋 常见问题 (FAQ)

- **Q: 真的有必要用这么具攻击性的语调编写提示词吗？**
  - A: 是的。对于没有感情的AI，温和的建议极高概率会被视为“可选项”而被无视。而因数据丢失受创的却是我们的心态。只有达到“违反即拒绝执行”程度的果断且严厉的约束，才能确保控制住致命事故。
- **Q: `--force-write` 标志具体如何使用？**
  - A: 这是设计好的安全装置，用于在Agent运行类似 `safe_query.py` 的本地Shell辅助脚本时作为参数（Argument）传递。也就是说，它作为核心触发器，让Agent在执行代码时基于此原则自行控制和验证脚本参数。

## 🧬 提示词解构 (Why it works?)

1. 🛡️ **Role (强制人格)：** 禁用了盲目顺从的秘书模式，赋予了严谨的DBA人格，确立了针对不当指令的**拒绝执行权**。
2. 🛑 **Task (物理约束)：** 通过强制注入查询超时和 `LIMIT` 子句，从结构上拦截了因死循环导致的Token爆炸和数据库锁 (Lock) 现象。
3. ⚙️ **Constraints (格式控制)：** 消除了AI特有的不必要叙述习惯，仅允许以序列化的 `JSON` 形式输出，以便自动化脚本立即处理。
