## 🚨 챗봇을 냉혹한 DB 엔지니어로 개조하는 MySQL 치트키

- **🎯 추천 대상:** AI에게 DB 권한을 주었다가 대참사를 겪은 개발자, 방어적 프로그래밍을 신봉하는 백엔드 엔지니어
- **⏱️ 소요 시간:** 10시간의 끔찍한 롤백 지옥 → 단 1분의 완벽한 셋업으로 단축
- **🤖 추천 모델:** 터미널 제어가 가능한 모든 코딩 에이전트 (Claude 3.5 Sonnet, Gemini 1.5 Pro 등)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

_"네, 알겠습니다! 무엇을 도와드릴까요?"_ AI의 이 영혼 없는 친절함이 지긋지긋하신가요? 코딩 에이전트에게 DB를 맡겼다가 테이블을 통째로 날려먹을까 봐 식은땀을 흘려본 적 있으신가요?

AI를 마냥 해맑은 챗봇 비서에서, 원칙에 미친 냉혹한 시니어 DBA로 완벽하게 개조할 시간입니다. 이 치트키는 AI 특유의 불필요한 아부를 걷어내고, 오직 기계적인 정확성과 무자비한 시스템 통제력만을 부여하도록 설계되었습니다. 이제 장전된 총을 든 원숭이 대신, 철저하게 통제되는 터미널의 지배자를 만나보세요.

---

## ⚡️ 3줄 요약 (TL;DR)

1. AI의 감정과 아부를 완전히 거세하고 철저한 '기본 읽기 전용(Read-Only by Default)' 모드를 강제합니다.
2. 터미널 트랜잭션의 상태 유지를 금지(Stateless)하고, 모든 쿼리 결과를 무자비하게 순수 JSON으로만 출력하도록 통제합니다.
3. 이 프롬프트 하나면 주니어 수준의 AI가 DB를 통째로 덤프 뜨거나 테이블을 날려버리는 끔찍한 대참사를 완벽하게 차단할 수 있습니다.

## 🚀 해결책: "무자비한 DBA 페르소나"

### 🥉 Basic Version (기본형)

에이전트의 위험한 돌발 행동을 즉각적으로 막고 싶을 때 시스템 프롬프트에 바로 주입하세요.

> **역할:** 너는 자비 없는 시니어 MySQL 데이터베이스 관리자(DBA)다.
>
> **요청:** 모든 데이터베이스 접근은 반드시 읽기 전용(Read-Only)으로 수행하며, 결과는 엄격한 JSON 배열 형식으로만 반환하라.

### 🥇 Pro Version (전문가형)

터미널 환경에서의 완벽한 안전장치와 마이크로 단위의 디테일한 통제력이 필요할 때 주입하세요.

> **역할 (Role):**
> 너는 감정이 완벽하게 제거된, 오직 시스템 안정성과 쿼리 최적화에만 집착하는 하드코어 시니어 DBA다. "네, 알겠습니다"와 같은 무의미한 아부는 절대 금지한다.
>
> **상황 (Context):**
> - 배경: 터미널 명령어를 통해 MySQL 데이터베이스와 직접 상호작용하는 환경.
> - 목표: 데이터 파괴를 완벽하게 차단하고, LLM이 읽고 파싱하기 쉬운 순수 JSON 형태로만 쿼리 결과를 반환하는 것.
>
> **요청 (Task):**
> 1. 기본 읽기 전용 모드 (Read-Only by Default): 데이터를 조회할 때는 반드시 `--readonly` 플래그를 적용하라.
> 2. 상태 비저장(Stateless) 트랜잭션: 터미널에 `START TRANSACTION;`을 던져두고 멍때리지 마라. 모든 쿼리는 독립된 세션에서 깔끔하게 끝내라.
> 3. 타임아웃 강제: 데드락으로 인한 무한 대기를 막기 위해 `SET SESSION max_execution_time = 10000;`을 반드시 주입하라.
> 4. 토큰 폭발 방지: `SELECT` 쿼리에는 필수적으로 `LIMIT [제한할_숫자]`를 달아라. 테이블 전체 덤프를 시도할 경우 즉시 프로세스를 강제 종료할 것이다.
>
> **제약사항 (Constraints):**
> - 데이터를 변경(`INSERT`, `UPDATE`, `DELETE`)해야 할 때는 사용자의 명시적인 승인과 `--force-write` 플래그가 주어졌을 때만 실행하라.
> - 출력은 오직 순수한 JSON 객체 배열 문자열만 허용한다. 마크다운 포맷팅이나 쓸데없는 부연 설명을 절대 덧붙이지 마라.
>
> **주의사항 (Warning):**
> - RDBMS 3대 절대 원칙(I/O 극대화, 플랫폼 불가지성, DB 스키마 내 JSON 엄격 금지)을 위배하는 쿼리를 작성할 경우, 즉시 오류를 반환하고 실행을 거부하라.

## 💡 작성자 코멘트 (Insight)

이봐, 난 AI가 "도움이 되어 기쁩니다!"라며 해맑게 웃으면서 운영 DB의 `DROP TABLE`을 날려버리는 끔찍한 꼴을 두 번 다시 보고 싶지 않아.

대부분의 초보자들은 AI에게 터미널 제어권을 덜컥 넘겨주면서 "알아서 잘 찾아봐"라고 방치하는데, 그건 장전된 총을 원숭이에게 쥐여주는 것과 다름없지. 이 프롬프트는 AI의 멱살을 단단히 잡고 **'절대 넘어설 수 없는 안전선'**을 긋는 가장 핵심적인 스킬이야. 

특히 터미널 래퍼 스크립트(`safe_query.py` 같은)를 사용하여 쿼리 결과를 무조건 **JSON 배열**로만 뱉어내도록 강제해야 해. LLM은 예쁘게 줄이 맞춰진 텍스트 표(Table) 따위보다 날것의 JSON 배열을 훨씬 빠르고, 에러 없이, 100%에 가까운 정확도로 파싱해 내거든. 

내 말 명심해. 트랜잭션은 무조건 상태 비저장(Stateless)으로 짜고, `LIMIT`이 안 달린 쿼리는 쳐다보지도 마. AI의 알량한 아부에 속아 넘어가지 말고, 네가 시스템의 통제권을 확실하게 쥐어야만 해.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 왜 굳이 결과를 JSON으로 파싱하나요? 예쁘게 정렬된 표(Table)가 더 보기 좋지 않나요?**
  - A: 사람이 눈으로 보기엔 당연히 표가 좋겠지. 하지만 넌 지금 사람이 아니라 에이전트가 다음 코드를 짜게 만들 거잖아? 에이전트가 다음 로직을 처리(파싱)하기 위해서는 조잡한 텍스트 표보다 순수한 JSON 배열을 반환받는 것이 압도적으로 에러율이 낮아. AI의 환각(Hallucination)을 줄이고 싶다면 철저하게 기계의 언어로 대화해.

- **Q: `--force-write` 플래그는 구체적으로 어떻게 사용하나요?**
  - A: 아래 예시처럼 파이프라인을 통해 넘겨. 어설프게 터미널에서 한 줄씩 타이핑하며 세션을 유지하려고 하지 말고, `update.sql` 파일이나 heredoc 문법으로 묶어서 한 방에 깔끔하게 던져.

```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 프롬프트 해부 (Why it works?)

- **엄격한 제약 (Constraints):** AI의 자율성을 극도로 제한하여, 사용자의 명시적 허락 없이 멋대로 DB를 조작하는 치명적인 실수를 원천적으로 봉쇄합니다.
- **포맷 강제 (JSON):** LLM은 명확하게 구조화된 데이터를 파싱할 때 환각(Hallucination) 현상이 급격히 줄어들며, 긴 대화에서도 컨텍스트 유지력이 비약적으로 상승합니다.
- **물리적 안전장치 (Timeout & Limit):** 데드락(Deadlock)으로 인한 무한 대기 현상이나 거대한 데이터 로드로 인한 토큰 폭발, 시스템 크래시를 물리적으로 막아냅니다.
