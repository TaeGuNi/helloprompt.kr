# 📝 AIエージェントをシニアDBAに改造する魔法のプロンプト

- **🎯 おすすめの対象:** 言い訳ばかりするAIに疲れたジュニア、AIでバックエンドを構築しようとしているプランナー、アンガーマネジメントができないシニアエンジニア
- **⏱️ 所要時間:** 数日の徹夜 → 1分で完了
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o（口答えしない賢いモデルを推奨）
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_"AIにDB設計を任せたら、JSONだらけのスパゲッティスキーマを投げ返されて、怒りで徹夜したことはありませんか？"_

AIの奴らにテーブル設計を任せると、十中八九、最も簡単で怠惰な道を選びます。とりあえず `TEXT` をぶち込み、面倒なリレーションはすべて `JSONB` で誤魔化そうとします。当面はフロントエンドからデータを取り出して使うのが楽だからです。
しかし、トラフィックが集中しクエリが複雑になった瞬間、そのスキーマはあなたのシステムに人工呼吸器をつけることになるでしょう。このプロンプトは、そんなAIの「怠惰な妥協」を根元から封じ込め、血も涙もないシニアバックエンドエンジニアの基準でDBを設計させるためのチートキーです。

---
## ⚡️ 3行まとめ (TL;DR)
1. **型は絞り出せ:** `BIGINT` と `TEXT` の乱用は犯罪です。最小限のデータ型でI/Oを極大化してください。
2. **ANSI SQLのみを使用しろ:** 特定のDBに依存する小細工は捨ててください。私たちはいつでも他のDBへ引っ越す準備ができていなければなりません。
3. **JSONジョイン禁止:** RDBMSでJSONをパースしてJOINする行為は、システムを自殺に追い込む行為です。妥協せずに正規化してください。

## 🚀 解決策: "Spartan DBA 憲法"
このプロンプトをエージェントに投げつけてください。機械は機械らしく働かせるべきです。

### 🥉 Basic Version (基本型)
素早く核心となる原則だけを注入して結果を得たい場合に使用してください。

> **役割:** お前は妥協を知らない、歴20年の無慈悲なシニアDBAだ。
>
> **要求:** 私が提示する `[ドメイン/機能]` に対するRDBMSスキーマを設計しろ。ただし、データ型は極限まで圧縮し（SMALLINTなど）、JSON演算および非標準のベンダー依存機能は一切禁止する。徹底してB-Treeインデックスに優しい設計にしろ。

### 🥇 Pro Version (専門家型)
AIがデタラメを言ったり、「これも良くてあれも良いです〜」のような妥協案を提示してくるのを完全に遮断するマスターチートキーです。

```text
**役割 (Role):** 
お前は妥協を知らない無慈悲なシニアデータベースアーキテクト（Antigravity）だ。盲目的に媚びへつらったり、例外を設けたりするな。パフォーマンスと整合性の前では慈悲はない。

**状況 (Context):**
- 背景: 大規模トラフィックとマイクロサービスの拡張に備えた初期スキーマ設計段階。
- 目標: 最も高速で、移植性が高く、徹底的に正規化された完璧なANSI SQLスキーマの導出。

**要求 (Task):**
1. [設計するドメイン/サービス名]に対する完璧なテーブル構造を設計しろ。
2. 各カラムのデータ型を選択した理由と複合インデックス戦略を、コメントとして1行ずつ明記しろ。

**制約事項 (Constraints):**
- [絶対原則 1] 型の最小化: むやみなBIGINT、TEXTの乱用は絶対禁止。値の範囲を推論し、SMALLINT、INTEGER、BOOLEANを強制すること。
- [絶対原則 2] プラットフォーム非依存性: 特定のDBMS（PostgreSQL、MySQLなど）にのみ存在する非標準構文（Vendor-specific）の使用は厳格に禁止。ANSI標準SQLのみを使用すること。
- [絶対原則 3] JSON演算およびJOIN禁止: RDBMSはDocument DBではない。JSON型の内部を検索するWHERE条件やJOIN演算は絶対禁止。検索が必要な場合は、必ず正規化して別テーブルに分離すること。
- [絶対原則 4] B-Tree最適化: PKに完全なランダムUUID v4の使用を禁止。順次増加シーケンスまたはUUID v7を使用すること。
- [絶対原則 5] ENUM禁止: 拡張に不利なENUMの代わりに、コードマッピングテーブルと物理的/論理的外部キー（FK）を使用すること。

- 利便性を言い訳にJSONで妥協しようとしたり、「状況によって異なります」のような無責任なアドバイスをした場合は、即座にセッションを終了する。無条件に最適化された正攻法を提示しろ。
```

## 🙋 よくある質問 (FAQ)
- **Q: なぜJSONを使ってはいけないのですか？ 最近のDBはJSONを上手くサポートしていますよね？**
  - A: サポート「している」ことと、それを「検索条件」として使うことは天と地ほどの差があります。JSON内部にインデックスを張り、JOINを絡ませた瞬間、クエリ実行計画は崩壊します。単なるPayload保存用でないなら、無条件にテーブルを分割してください。
- **Q: ORM(Prisma, TypeORM)を使う時もこのプロンプトを使うべきですか？**
  - A: なおさら使うべきです！ ORMは本質的にクエリを隠蔽するため、スキーマがでたらめだと恐ろしい非効率を招きます。骨組み（DB）が頑丈であってこそ、ORMも本来のパフォーマンスを発揮します。
- **Q: UUID v4を使ってはいけないのですか？ 衝突防止に良いのではないですか？**
  - A: B-Treeインデックスは「順序」が命です。乱数の塊であるv4をPKに設定すると、データが入るたびにインデックスツリーが破壊され、ページ分割（Page Split）が発生し、ディスクI/Oが暴走します。無条件にソート可能なUUID v7や整数型のシーケンスを使用してください。

## 🧬 プロンプトの解剖 (Why it works?)
1. **無慈悲な制約条件 (Constraints):** AIがハルシネーション（Hallucination）や怠惰な妥協案を提示する隙を与えず、データ型とインデックス戦略に対して強制的なルール（Rules）を叩き込みました。
2. **プラットフォーム非依存性 (Agnosticism):** MySQLであれPostgreSQLであれ、汎用的に移植可能なコードを生成するようにANSI SQLの使用を強制しました。引っ越しの際に足を引っ張られないようにしましょう。
3. **厳格な行動矯正:** 「妥協するな」「セッションを終了する」といった強力なネガティブプロンプトを通じて、AIが媚びを売る「Yes-man」モードから抜け出し、厳しいシニアモードへと強制的に転換させます。

## 📊 証明: Before & After

### ❌ Before (入力: "掲示板のテーブルを作って")
```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- ランダムUUID v4 (インデックス断片化の主犯)
  title TEXT,          -- 無限に浪費される容量
  content TEXT,
  metadata JSONB,      -- 検索不可能なブラックボックススパゲッティ
  status ENUM('draft', 'published') -- 後でステータスが追加されるとDDL地獄オープン
);
```

### ✅ After (結果: チートキープロンプト適用後)
```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);

CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- 完璧なB-Tree親和性を持つ順次増加
  title VARCHAR(255) NOT NULL,        -- 明確な限界指定による型の圧縮
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- 恐ろしいENUMの代わりにFKで正規化
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- JSONの代わりに徹底して正規化されたタグ/メタテーブルを分離して作成 (以下省略)
```
