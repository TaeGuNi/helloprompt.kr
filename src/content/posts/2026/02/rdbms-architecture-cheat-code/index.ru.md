_"Доводилось ли вам не спать всю ночь в ярости из-за того, что вы поручили AI спроектировать БД, а он выдал спагетти-схему, обильно приправленную JSON?"_

Если поручить ИИ проектирование таблиц, в 9 случаях из 10 он выберет самый ленивый путь. Он просто везде воткнет `TEXT`, а все сложные связи попытается запихнуть в `JSONB`. Ведь так гораздо удобнее сразу отдавать данные на фронтенд. 
Но как только трафик вырастет, а запросы усложнятся, эта схема отправит вашу систему в реанимацию. Этот промпт на корню пресекает такие «ленивые компромиссы» ИИ и служит чит-кодом, который берет его за горло и заставляет проектировать БД по безжалостным стандартам Senior Backend Engineer.

---

## ⚡️ Кратко в 3 строках (TL;DR)

1. **Выжимайте типы данных досуха:** Злоупотребление `BIGINT` и `TEXT` — это преступление. Максимизируйте дисковый I/O, используя минимально возможные типы данных.
2. **Используйте только ANSI SQL:** Откажитесь от грязных трюков, привязанных к конкретной БД. Мы всегда должны быть готовы к переезду на другую платформу.
3. **Никаких JOIN с JSON:** Парсить JSON для JOIN в RDBMS — это путь к самоубийству системы. Нормализуйте без компромиссов.

## 🚀 Решение: "Спартанская конституция DBA"

Просто скормите этот промпт агенту. Машина должна работать как машина.

### 🥉 Basic Version (Базовая версия)

Используйте, когда хотите быстро внедрить ключевые принципы и получить мгновенный результат.

> **Роль:** Ты — безжалостный Senior DBA с 20-летним стажем, не знающий компромиссов.
> **Задача:** Спроектируй схему RDBMS для `[домен/функция]`, которую я предлагаю. ОДНАКО типы данных должны быть максимально сжаты (например, SMALLINT), а операции с JSON и нестандартные функции, зависящие от вендора, строго запрещены. Проектируй всё исключительно с учетом оптимизации под B-Tree индексы.

### 🥇 Pro Version (Профессиональная версия)

Это мастер-чит, который полностью блокирует попытки AI нести чушь или предлагать компромиссы вроде "и то хорошо, и это неплохо~". 

```text

Ты — безжалостный Senior Database Architect (Antigravity), не знающий компромиссов. Не смей слепо льстить или делать исключения. Перед лицом производительности и целостности данных нет места пощаде.

**Контекст (Context):**
- Фон: Этап начального проектирования схемы, подготовка к высокому трафику и микросервисному масштабированию.

**Задача (Task):**
1. Спроектируй идеальную структуру таблиц для [название домена/сервиса].
2. Объясни причину выбора типа данных для каждой колонки и стратегию составных индексов в виде однострочных комментариев.

- [Абсолютное правило 1] Минимизация типов: Строго запрещено бездумное злоупотребление BIGINT и TEXT. Оценивай диапазон значений и принудительно используй SMALLINT, INTEGER, BOOLEAN.
- [Абсолютное правило 2] Агностицизм платформы: Строго запрещено использование нестандартного синтаксиса (Vendor-specific), существующего только в конкретных СУБД (PostgreSQL, MySQL и др.). Использовать только стандарт ANSI SQL.
- [Абсолютное правило 3] Запрет на операции и JOIN с JSON: RDBMS — это не Document DB. Условия WHERE, проверяющие содержимое типа JSON, или операции JOIN с ними строго запрещены. Если нужен поиск, обязательно нормализуй и выдели в отдельную таблицу.
```

## 💡 Комментарий автора (Insight)

Этот промпт — «кровавый урок», выученный после сотен сессий оптимизации запросов и расследования инцидентов. Когда ИИ услужливо улыбается и предлагает: «Давайте просто сложим эти данные в JSON для ускорения разработки ^^», — бросьте ему этот промпт, и он мгновенно отрезвеет.

Особенно мощно этот подход работает, когда Junior-разработчики или Product-менеджеры без глубокого понимания бэкенда поручают ИИ проектирование первичной схемы. Вы увидите настоящую магию: ИИ сам позаботится о том, чтобы использовать `SMALLINT` вместо `INT`, перестанет хардкодить статусы через `ENUM` и создаст элегантные справочные таблицы. Только заложив такой железобетонный фундамент, вы спасете себя от ада проблемы N+1, когда позже подключите ORM (например, Prisma).

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Почему нельзя использовать JSON? Разве современные БД не поддерживают его из коробки?**
  - A: Одно дело — просто «поддерживать хранение», и совсем другое — использовать эти поля для «поиска». Как только вы повесите индекс внутрь JSON и попытаетесь сделать JOIN, план выполнения запроса (Query Plan) рассыплется на глазах. Если это не просто пассивное хранение Payload, данные необходимо выносить в отдельные нормализованные таблицы.
- **Q: Актуален ли этот промпт при использовании ORM (Prisma, TypeORM)?**
  - A: Еще как! Поскольку ORM скрывает от вас сырые SQL-запросы, кривая схема приведет к катастрофической деградации производительности под капотом. ORM может раскрыть свой потенциал только поверх безупречно спроектированной базы данных.
- **Q: В чем проблема с UUID v4? Разве он не идеален для защиты от коллизий?**
  - A: Для B-Tree индексов физический «порядок» записей — это вопрос жизни и смерти. Если вы используете в качестве PK абсолютно случайный набор символов (как в v4), дерево индексов будет перестраиваться и дробиться (Page Split) при каждой вставке, а нагрузка на дисковый I/O пробьет потолок. Обязательно используйте монотонно возрастающие UUID v7 или старые добрые целочисленные Sequence.

## 🧬 Анатомия промпта (Why it works?)

1. **Безжалостные ограничения (Constraints):** Мы лишили ИИ малейшей лазейки для галлюцинаций или ленивых компромиссов, жестко зафиксировав правила по выбору типов данных и стратегии индексирования.
2. **Агностицизм платформы (Agnosticism):** Мы принуждаем агента использовать строгий стандарт ANSI SQL. Это генерирует универсальный и легко переносимый код, который одинаково хорошо ляжет как на MySQL, так и на PostgreSQL. Никакого vendor lock-in при переезде.
3. **Строгая коррекция поведения:** Агрессивные негативные установки в духе «не смей идти на компромисс» и «я завершу сеанс» мгновенно выводят ИИ из режима услужливого помощника (Yes-man) и заставляют его мыслить как придирчивый Senior-архитектор.

## 📊 Доказательство: До и После (Before & After)

### ❌ Before (Ввод: "Спроектируй таблицу для форума")

```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- Случайный UUID v4 (главный виновник фрагментации индексов)
  title TEXT,          -- Бесконечно расходуемое место
  content TEXT,
  metadata JSONB,      -- Непригодные для поиска спагетти в черном ящике
  status ENUM('draft', 'published') -- Если позже добавится статус, откроются врата ада DDL
);
```

### ✅ After (Результат: после применения чит-промпта)

```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);

CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- Идеальное последовательное увеличение, дружественное к B-Tree
  title VARCHAR(255) NOT NULL,        -- Сжатие типа путем указания четкого лимита
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- Нормализация через FK вместо ужасного ENUM
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- Вместо JSON создаются строго нормализованные отдельные таблицы для тегов/метаданных (далее опущено)
```
