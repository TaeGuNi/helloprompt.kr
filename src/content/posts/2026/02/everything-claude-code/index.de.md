Trotz teurer Abogebühren hat man keinen Senior Engineer an seiner Seite, sondern einen **'Anfänger-Praktikanten'**, der nur Chaos anrichtet.

Sobald Sie dieses Setup in Ihrem Terminal anwenden, ändert sich Ihr Workflow um 180 Grad. Die AI schreibt nicht mehr blindlings Code. Ob beim Refactoring von komplexem Legacy-Code oder beim Entwurf einer neuen Architektur von Grund auf – mit diesem leistungsstarken Setup ist eine überragende Codequalität garantiert. ```text
User: > /plan Erstell mir eine Forenliste

## 🚀 So schreiben echte Experten

Dies ist der Workflow-Prompt, der nach dutzenden Versuchen perfektioniert wurde. Kopieren Sie den folgenden Prompt und füllen Sie nur die `[Variablen]` in den Klammern passend zu Ihrer Projektsituation aus.

### 🥉 Basic Version (Schnelle Installation & Basis-Planung)

Investieren Sie nur 2 Minuten im Terminal, um erstklassige Senior-Agenten in Ihr Projekt zu holen.

> **1. Plugin-Installation (Basis Claude Code):**
> `/plugin marketplace add affaan-m/everything-claude-code`
> `/plugin install everything-claude-code@everything-claude-code`
>
> **2. Planungs-Prompt ausführen:**
> `/plan "Erstelle eine responsive Blogpost-Listenansicht mit Next.js 14 App Router. Wir verwenden Tailwind CSS und shadcn/ui."`

### 🥇 Pro Version (Erzwingung von TDD & Custom Rules)

Nutzen Sie diesen Prompt, wenn Sie einen lückenlosen Senior-Entwickler-Workflow erzwingen wollen, der perfekt mit Ihrem Projekt synchronisiert ist.

> **Rolle (Role):** Du bist ein `[Senior TypeScript Entwickler]`, der sich strikt an die Prinzipien von TDD (Test-Driven Development) hält.
>
> **Kontext (Context):**
>
> - Wir folgen den Regeln von `[everything-claude-code]`.
> - Ziel: Die Implementierung der `[Benutzerauthentifizierungs-Funktion]` muss abgeschlossen werden.
>
> **Aufgabe (Task):**
>
> 1. Aktiviere den Befehl `/tdd` im Terminal.
> 2. Schreibe unbedingt zuerst Test-Code inklusive Exception-Cases, bevor du den eigentlichen Code schreibst.
> 3. Gehe niemals zum nächsten Schritt über, wenn die Testabdeckung nicht über 80% liegt.
> 4. Falls es `[zusätzliche Anforderungen]` gibt, spiegele diese perfekt im Code wider.
>
> **Einschränkungen (Constraints):**
>
> - Füge allen Variablen und Funktionen explizite Type Hints hinzu.
> - Die Verwendung des Typs `any` ist streng verboten.
>
> **Warnung (Warning):**
>
> - Schreibe keinen Code, der gegen die bestehende Architektur oder Coding-Conventions verstößt. Wenn du dir unsicher bist, stelle mir erst eine Frage, bevor du Code schreibst. (Vermeidung von Halluzinationen)

## 💡 Kommentar des Autors (Insights & Anwendung)

Nachdem ich dieses `everything-claude-code`-Setup in meinen realen Projekten eingeführt habe, war die beeindruckendste Veränderung, dass **"die AI begann, meine faulen Prompts bestimmt, aber höflich abzulehnen."**

Früher passierte es oft, dass bei unpräzisen Anweisungen wie _"Binde diese API an und zeig es auf dem Bildschirm an"_ Code zurückkam, bei dem das Handling von Edge-Cases oder Error-States völlig fehlte. Mit diesem Setup hat sich das Blatt gewendet. Sobald ich den Befehl `/plan` eingebe, schreibt die AI nicht sofort Code, sondern startet eine Offensive aus präzisen Rückfragen. _"Wie soll das Skeleton UI während des Ladens aussehen?", "Wie gestalten wir die Retry-Logik und das User-Feedback, wenn der API-Aufruf fehlschlägt?"_ Es fühlte sich an, als säße ein strenger Chefarchitekt neben mir, der die Tastatur nicht anrührt, bis die Anforderungen perfekt definiert sind.

Das Kernprinzip dieses Tools liegt in dem **'Multi-Agent-basierten, erzwungenen Workflow'**. Wenn der Benutzer seinen Stack und seine Situation in die `[Variablen]` eingibt, findet keine einfache Textergänzung statt, sondern es greifen nacheinander strikt getrennte Personas ein: **Planung (Planner) → Test (TDD) → Implementierung (Developer) → Review (Reviewer)**. Schon mit der Basic-Version lässt sich ein stabiles Grundgerüst bauen, aber durch die Pro-Version, die Rollen wie `[Senior TypeScript Entwickler]` erzwingt und konkrete Ziele wie `[Benutzerauthentifizierung]` vorgibt, vervielfacht sich die Wirkung. Indem man eine Testabdeckung von 80% als Einschränkung festlegt, kann man eine faszinierende Feedback-Schleife beobachten, in der die AI ihre Logik immer weiter verfeinert, um die selbst geschriebenen Tests zu bestehen.

Das absolute Highlight ist jedoch der Befehl **`/evolve`**. Bei normalen AI-Chats wird das Gedächtnis nach Ende einer Session zurückgesetzt, was oft dazu führt, dass man am nächsten Tag die gleichen Coding-Conventions erneut erklären muss. Mit `/evolve` jedoch aktualisiert und verinnerlicht die AI das während des Projekts gelernte implizite Wissen (z.B. _"Unser Team setzt kein 'I' vor Interface-Namen"_, _"Status-Management erfolgt ausschließlich mit Zustand"_) kontinuierlich in `.rules`- oder `.cursorrules`-Dateien. Das bedeutet: Je größer das Projekt wird, desto mehr entwickelt sich die AI durch **'Selbstlernen'** zu einem perfekt auf das Projekt optimierten Senior-Partner.

Der wichtigste Tipp für den Einsatz in der Praxis ist die Kontrolle der Einschränkungen (Constraint Control). Nutzen Sie das Feld **`[zusätzliche Anforderungen]`** im Pro-Prompt, um explizit auf spezifische Legacy-Strukturen oder Anti-Patterns hinzuweisen, die unbedingt vermieden werden müssen. Wenn Sie Details festlegen wie _"Rühre den bestehenden Redux-Code nicht an, sondern extrahiere es in einen unabhängigen Hook"_, wird dies mit dem strengen Regelprüfungssystem von `everything-claude-code` abgeglichen, wodurch die Fehlerquote gegen Null sinkt. Für Einzelentwickler ohne Code-Reviewer oder kleine Startups, die schnell ein MVP erstellen müssen, ist dieses systematische System mehr als nur ein Produktivitätstool – es ist die beste Waffe zum Überleben.

## 🙋 Häufig gestellte Fragen (FAQ)

- **Q: Funktioniert das nur in der Claude Code CLI-Umgebung?**
  - A: Nein! Sie können die Regeln direkt in die `.cursorrules`-Datei des Cursor-Editors exportieren, und es ist auch hervorragend mit anderen Umgebungen wie OpenCode kompatibel.

- **Q: Ist die Einstiegshürde oder das Setup für Junior-Entwickler nicht zu streng?**
  - A: Zu Beginn kann es etwas mühsam sein, da die AI ständig nach Edge-Cases fragt und Test-Code erzwingt. Aber nur wenn man dieses strenge Aufnahmeritual durchläuft, kann man verhindern, dass man später von unkontrollierbaren **technischen Schulden (Technical Debt)** erdrückt wird und Nächte durcharbeiten muss.

## 🧬 Analyse des Prompts (Warum es funktioniert?)

1. **Erzwungener Workflow (Forced Workflow):** Verhindert einseitige "Mach mal"-Anweisungen an der Quelle. Durch explizite Slash-Befehle wie `/plan` oder `/tdd` wird die AI dazu gezwungen, den sauberen Engineering-Prozess 'Analyse → Test-Code → Implementierung' zu durchlaufen.
2. **Kreuzvalidierung durch Multi-Personas (Persona Separation):** Trennt strikt zwischen dem planenden Ich (Planner), dem codierenden Ich (Developer) und dem prüfenden Ich (Reviewer). Dadurch entsteht eine perfekte Feedback-Schleife, in der die AI ihren eigenen Code kritisch hinterfragt und validiert.
3. **Permanentes Kontext-Lernen (Instincts):** Im Gegensatz zu normalen Chatbots, bei denen der Kontext nach der Session verpufft, erkennt dieses System die Code-Muster und Architektur-Absichten des Benutzers über das gesamte Projekt hinweg und prägt diese dauerhaft in `.rules`-Dateien ein, sodass der Agent ununterbrochen evolutioniert.
