## 🙋 常见问题 (FAQ)

- **Q: AI 真的能理解复杂业务逻辑的意图并进行审查吗？**
  - A: 很遗憾，AI 无法凭空洞悉你们公司的具体业务，也不知道这段代码背后的业务上下文。AI 的审查完全聚焦于**“技术缺陷、性能瓶颈和代码可读性”**。至于代码是否完美契合业务需求，这仍然是开发者必须亲自把关的核心领域。

- **Q: 在免费版的 AI 模型（如 ChatGPT 3.5）上也能跑通吗？**
  - A: 跑是能跑，但在提供具备架构深度的建议（Pro Tip）或捕捉隐蔽的性能问题时，免费模型的能力远逊于最新的推理型大模型（如 Gemini 3 Pro、GPT-5、Claude 3.5 Sonnet 及以上版本）。在代码审查这种关乎项目生死存亡的环节，直接使用最聪明的模型才是性价比最高的选择。

- **Q: 提示词生成的审查结果总是老生常谈，该如何破局？**
  - A: 喂给 AI 代码的同时，请务必在代码块中追加上下文注释。例如加上 `// 业务意图：此函数需承载每日 100 万次的并发流量`，将性能目标或业务约束（Context）一并告知 AI。你提供的上下文越丰满，AI 审查的质量就会呈指数级飙升。

## 📊 效果对比：Before & After

### ❌ Before (初级开发者的常见代码)

```javascript
// 增加用户积分的函数
function addP(u, p) {
  let user = db.findUser(u);
  if (user != null) {
    user.point = user.point + p;
    db.save(user);
    return true;
  } else {
    return false;
  }
}
```

### ✅ After (采纳 AI 技术主管审查建议后)

```javascript
/**
 * 处理用户积分累加
 * @param {string} userId - 目标用户 ID
 * @param {number} pointsToAdd - 需要累加的积分
 * @throws {UserNotFoundError} 当找不到指定用户时抛出异常
 */
async function addPointsToUser(userId, pointsToAdd) {
  if (pointsToAdd <= 0) {
    throw new InvalidArgumentError("累加积分必须大于 0。");
  }

  const user = await db.findUserById(userId);
  if (!user) {
    throw new UserNotFoundError(`无法找到该用户。ID: ${userId}`);
  }

  // 推荐使用原子更新 (Atomic Update) 以防止并发引发的竞争条件 (Race Condition)
  await db.updateUserPointsAtomic(userId, pointsToAdd);

  return true;
}
```

_💡 审查要点：彻底消灭了不知所云的变量命名（`u`, `p`），引入了防御性编程的异常抛出机制（Throw），修复了缺失的异步处理（async/await），并前瞻性地提出了规避并发冲突的原子性数据库更新方案。_
