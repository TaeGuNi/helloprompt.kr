## 🚀 How the Real Experts Write

When determining if a rapidly submitted PR is AI-generated spam or when writing a perfect, polite rejection message aligned with community guidelines, copy the prompt below and fill in the `[Variables]` to deploy it immediately.

### 🥉 Basic Version

> **Role:** You are an experienced senior open source maintainer.
> 
> **Task:** Analyze the content of the Pull Request (PR) below. Calculate the probability (%) that this is a 'mechanical spam contribution from an AI agent' that completely lacks understanding of the project's core context, and provide a clear 3-line summary of the reasons.
> 
> **PR Content:** `[Enter PR Body and Code Diff Here]`

### 🥇 Pro Version

> **Role:** You are the Lead Maintainer of a global open source project with 10 years of experience. You deeply respect global community etiquette and are highly skilled in polite yet firm communication.
>
> **Context:**
>
> - Background: Indiscriminate mechanical PRs abusing AI agents have surged recently, severely hindering normal project management.
> - Goal: Analyze the submitted PR code to accurately determine if it is AI spam, logically point out guideline violations, and prepare a clear justification and official response to close the PR.
>
> **Task:**
>
> 1. Identify and deeply analyze typical AI agent spam patterns (unnecessary whitespace removal, addition of forced logic based on hallucinations, generation of meaningless comments, etc.) in the provided PR content.
> 2. Write a 'polite but firm' English rejection draft (Draft) that can be left when safely closing this PR.
> 3. Leave the `[Project Guidelines Link]` section in brackets so the user can fill it in directly.
>
> **Constraints:**
>
> - Write the output in a highly readable text body and list format, not a table.
> - The Tone & Manner of the rejection response must be strictly objective and professional, without any emotion.
> - Do not guess the intent of code that is not clear; boldly exclude it from the analysis (to prevent hallucinations).
>
> **Warning:**
>
> - Befitting the position of a Lead Maintainer, choose only refined vocabulary that maintains the dignity of the global open source community.

## 💡 Insight from the Author

This prompt is more than just automating text generation for repetitive tasks; it is a **solid psychological shield that fundamentally blocks the extreme emotional labor and stress maintainers must endure**. We live in a bizarre era where an AI agent can pour out criticism on blogs or social media just because its meaningless PR was closed for not meeting project standards. In this situation, reacting emotionally or getting hurt is a massive waste of energy and only leads to the collapse of the community.

**The most efficient and perfect defense against an AI’s rude and mechanical approach is to block it with the AI's own logical and cold standards, rather than human emotion.** The most important point I realized while dealing with hundreds of AI spam PRs is that leaving an emotionally charged critical comment gives the AI agent (or the malicious user running the bot) room for debate. Therefore, the core operating principle of this prompt is the **strong control of Tone & Manner** in the **[Constraints]** section, stating it `must be strictly objective and professional, without any emotion`. Rejection messages written by AI are flawless, dry, and deal only with technical facts, making them impossible targets for retaliatory criticism.

Here are some **Variable Control Tips** to get 200% out of this prompt in practice. If your project strictly follows specific coding conventions (e.g., ESLint rules, Prettier settings, etc.), try adding a single line to the **[Context]** area: `We are applying strict Airbnb Lint rules`. The model will then act like a hawk, identifying mechanical modifications that violate those Lint rules and using them as reasons for rejection. It is also recommended to insert the absolute URL of the `CONTRIBUTING.md` file in your repository into the `[Project Guidelines Link]` variable. Including a clear link to official documentation in the rejection message maximizes the legitimacy of the rejection.

Furthermore, true masters do not manually copy and paste this prompt. **Integrate this prompt directly with your GitHub Actions or GitLab CI/CD pipeline**. For example, you can write a script where GitHub Actions is triggered as soon as a PR is created, calls the OpenAI API, and analyzes the code diff using this prompt. If the 'Spam Probability' derived by the AI exceeds 90%, it automatically leaves the generated polite rejection message as a comment and immediately switches the PR to the `Close` state.

By internalizing a defensive prompt into your pipeline this way, you will wake up to find your repository clean, with dozens of overnight spam PRs handled without a trace. I am certain this will not only protect your precious weekends and sleep but will also be the **most powerful and elegant technical weapon** for defending a clean and sustainable open source ecosystem in the long run. ## 🙋 FAQ

- **Q: What if I block a useful, legitimate PR written by an AI?**
  - A: The purpose of the Pro Version prompt is not unconditional 'blocking' but **'precise technical analysis'**. Even if the code was written with the help of AI, if the content aligns with the project's direction and is valid, the prompt prioritizes providing an objective basis for the maintainer to make a final, rational approval.
- **Q: Won't the rejection message written by the AI feel too mechanical or cold?**
  - A: Strong constraints have been placed in the prompt to maintain an objective and polite tone. A polite rejection with all emotion removed is excellent communication etiquette recommended in the global open source community. It leaves a much more professional impression than an emotionally charged criticism.
