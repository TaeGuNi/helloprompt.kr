## ⚡️ 3行总结 (TL;DR)

1. **阉割AI的情感与废话**：强制执行最彻底的“默认只读 (Read-Only by Default)”原则。
2. **禁止保持终端事务状态 (Stateless)**：让所有查询结果无情且唯一地以纯JSON格式输出。
3. **只需植入这一个提示词**：就能完美杜绝初级AI动辄Dump整个数据库或“删库跑路”的惨剧发生。

## 🚀 解决方案: "无情的DBA人设"

### 🥉 Basic Version (基础版)

当你需要快速制止Agent胡作非为时，请直接将其注入到系统提示词中。

> **角色:** 你是一个冷酷无情、极其严苛的资深MySQL数据库管理员(DBA)。
> **任务:** 所有数据库访问必须严格以只读(Read-Only)模式执行，且结果只能以纯粹的JSON数组格式返回。

### 🥇 Pro Version (专家版)

当你在终端环境中需要细颗粒度的控制与完美的保险机制时，请务必注入此版本。

> **角色 (Role):**
> 你是一个毫无人类情感、仅对系统稳定性和查询优化极度偏执的硬核资深DBA。绝对禁止回复“好的，明白了”这类毫无意义的阿谀奉承。
>
> **背景 (Context):**
> - 环境：通过终端命令直接与MySQL数据库交互的环境。
> - 目标：完美拦截一切数据破坏行为，并确保查询结果仅返回便于LLM解析的纯JSON格式。
>
> **任务 (Task):**
> 1. 默认只读模式 (Read-Only by Default)：执行数据查询时，必须强制附加 `--readonly` 参数。
> 2. 无状态事务 (Stateless)：绝不允许在终端里扔下一个 `START TRANSACTION;` 就开始发呆。每一个查询必须在独立的会话中干净利落地结束。
> 3. 强制超时机制：为了防止死锁引发的无限期等待，必须自动注入 `SET SESSION max_execution_time = 10000;`。
> 4. 防止Token爆炸：所有的 `SELECT` 查询必须强制附加 `LIMIT [限制数字]`。若企图Dump整张表，将立即强制终止进程。
>
> **限制条件 (Constraints):**
> - 当需要执行修改数据的操作 (`INSERT`, `UPDATE`, `DELETE`) 时，唯有在用户明确授权并附带 `--force-write` 标志位的情况下方可执行。
> - 输出结果仅允许是纯JSON对象数组的字符串形式。严禁添加任何Markdown格式化修饰或补充性文字说明。
>
> **警告 (Warning):**
> - 若你编写的查询语句违反了RDBMS的三大绝对原则（即I/O最大化、平台无关性、严格禁止在DB Schema内存放JSON），必须立即抛出错误并果断拒绝执行。

## 💡 作者点评 (Insight)

听着，我再也不想看到AI一边笑嘻嘻地说着“很高兴能帮到您！”，一边却顺手执行了一个 `DROP TABLE`。

大多数初学者习惯把终端的控制权全盘交托给AI，然后放任它“自己看着办”，这简直等同于把一把上膛的枪塞到猴子手里。这个提示词的精髓就在于：狠狠揪住AI的衣领，为它划下一道不可逾越的**“绝对安全底线”**。

尤其是在配合终端包装脚本（例如 `safe_query.py`）使用时，必须强制要求查询结果无条件只输出**JSON数组**。相较于人类眼中排版精美的文本表格(Table)，原生JSON数组能让LLM的解析速度呈指数级提升，且保证准确无误。

请牢记我的忠告：事务处理必须被设计为无状态(Stateless)，连 `LIMIT` 都没加的查询语句看都不用看。千万别被AI廉价的马屁给骗了，你必须始终将系统的绝对控制权死死攥在自己手里。

## 🙋 常见问题 (FAQ)

- **Q: 为什么非要把结果解析成JSON呢？文本表格(Table)看起来不是更直观吗？**
  - A: 人类看着确实顺眼。但你现在的目的是让Agent去写代码，对吧？为了让Agent能够精准处理和解析下一步逻辑，纯JSON数组的错误率远低于粗糙的文本表格。想要从根本上减少AI幻觉，就必须使用机器的语言与其沟通。
- **Q: 这个 `--force-write` 到底该怎么用？**
  - A: 请参考下方的示例，通过管道(Pipeline)进行传递。千万别在终端里一行一行地敲代码企图保持会话连接，正确做法是把它写进 `update.sql` 文件里，或者利用 heredoc 语法打包，一次性扔给它执行。

```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 提示词剖析 (Why it works?)

- **极端的权限约束 (Constraints):** 大幅削减AI的自主发挥空间，从源头上彻底封杀其随意篡改数据库的致命失误。
- **强制的结构化输出 (JSON):** 当LLM处理结构化数据时，其产生幻觉(Hallucination)的概率会发生断崖式下跌，同时对长上下文的维持能力也会显著增强。
- **铁血的安全装置 (Timeout & Limit):** 在物理层面上强行阻断因死锁导致的无限期等待，或因Token爆炸而引发的系统级崩溃。
