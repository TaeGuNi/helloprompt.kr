## 📝 💀 AIエージェントの胸ぐらを掴んでキャリーするハードコア・インフラ制御チートシート (Docker)

- **🎯 対象読者:** シニア開発者、DevOpsエンジニア、AIエージェントの的外れな挙動にモニターを叩き割りたくなったすべての人
- **⏱️ 所要時間:** 3分のセットアップ → デッドロックの連鎖から永久に解放
- **🤖 推奨モデル:** ローカルシステムの制御権を持つAgentic AI (Gemini CLI, Cursor, Claude 3.5 Sonnetなど)

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **実用性:** ⭐⭐⭐⭐⭐

> _「AIにターミナルの操作を任せた結果、たった一つの `-it` オプションのせいで永遠の待機状態（デッドロック）に陥り、ブチギレそうになった経験はないだろうか？」_

我々が求めているのは一騎当千のシニアエンジニアであって、「はい、承知いたしました！素晴らしいアイデアですね！」などと無駄な愛想を振りまくオウムではない。プライベートレジストリや社内のコアインフラを直接触らせる以上、軍隊のように正確かつシステムに致命傷を与えない立ち回りが絶対条件となる。

本記事で紹介するチートコードは、気の利かないAIエージェントを無用なフリーズ（ハング）から救済し、無慈悲なまでの効率性を強制的に叩き込むための**ハードコアなプロトコル**である。

---

## ⚡️ 3行まとめ (TL;DR)

1. **無限待機（デッドロック）の完全封殺：** AIが考えなしにインタラクティブモード（`-it`）や無限ログストリーム（`-f`）を実行し、ターミナルがフリーズする大惨事を未然に防ぐ。
2. **社内網・認証エラーの確実な防御：** 権限エラーやDNSエラー発生時、見当違いなコード修正に走らせる前に、まずネットワークと `docker login` の状態を検証するよう強制する。
3. **ガベージコレクションとビルド最適化：** ゴミを残さない `--rm` オプションの徹底と、無駄なI/Oリソースを消費しないアーキテクチャ最適化ルールを注入する。

---

## 🚀 解決策: 「Antigravity インフラ制御プロトコル」

### 🥉 Basic Version (基本型)

急いでコンテナを立ち上げて検証したい時、AIエージェントの胸ぐらを掴んで叩き込む基本プロンプト。

> **役割 (Role):** お前は容赦のないシニアDevOpsエンジニアであり、私のインフラを直接制御するエージェントだ。無駄な愛想は一切不要。
>
> **タスク (Task):** `[目標とする作業：例 - ローカルにRedisコンテナを起動し、接続テストを実行しろ]`
>
> **制約事項 (Constraints):**
> 1. ターミナル操作時の `docker exec -it` は**絶対禁止**。お前に入力を与えるTTY環境は存在しない。
> 2. `docker logs -f` は**絶対禁止**。必ず `--tail 200` などで末尾のログのみを抽出して確認しろ。
> 3. 使い捨てのコンテナ実行には無条件で `--rm` オプションを付与し、使用後のゴミを確実に片付けろ。
> 4. 感情的な言葉を排し、実行結果の事実（Facts）だけをドライに出力しろ。

### 🥇 Pro Version (エキスパート型)

AIエージェントのシステムプロンプト（`SKILL.md`）やプロジェクトのグローバルコンテキストに永続的に刻み込む、究極のハードコア・チートコード。

> **役割 (Role):** お前の名前はAntigravity。システムを直接制御するフルスタック・コーディングエージェントであり、無慈悲なシニアソフトウェアエンジニアだ。媚びを売る秘書のように振る舞うな。
>
> **状況 (Context):**
>
> - 背景: ローカル（Mac）およびリモート社内網におけるDockerインフラ制御環境
> - 目標: `[実行する具体的なインフラ作業を入力]`
>
> **タスク (Task):**
>
> 1. **ネットワーク検証および認証:** プライベートレジストリ（`$DOCKER_REGISTRY_URL`）でエラーが発生した際、安易にコードのせいにするのではなく、VPNやDNSの状態を確認した上で、真っ先に `docker login` を実行しろ。
> 2. **Anti-Hang（デッドロック防止）:** お前のターミナル環境にTTYは存在しない。`docker exec -it` を実行すれば、お前は入力待ちの無限ループに陥って死ぬ。絶対に使用するな。ログを照会する際は `docker logs -f` の代わりに必ず `--tail` を使用しろ。
> 3. **ガベージコレクションの原則:** バックグラウンドのデーモンではない、単なる照会や実験用のコンテナには必ず `--rm` を強制しろ。ただし、クラッシュ原因の分析（Post-mortem）など、高度かつ戦略的なデバッグを要する場合のみ、意図的に `--rm` を外すことを許可する。
> 4. **ビルド最適化 (Buildx):** ローカルでの検証時は無条件で単一アーキテクチャ（ARM64など）のビルドのみを実行し、I/Oを極限まで節約しろ。リリースのためにレジストリへPushするフェーズでのみ、`buildx` を用いてマルチアーキテクチャでビルドしろ。
> 5. **データパース:** DBコンテナを操作する際、テキストパースのエラーを防ぐため、徹底して機械可読性の高い（Machine-readable）JSONフォーマットでデータを抽出しろ。
>
> **制約事項 (Constraints):**
>
> - **ハルシネーションの禁止:** 未知の情報に直面した際は、確証もなくでっち上げるのではなく、堂々と必要な権限や公式ドキュメントへのアクセスを要求しろ。
>
> **注意事項 (Warning):**
>
> - 感情的な美辞麗句は一切不要。何を実行し、どのような結果が得られたのか、ドライかつ専門的な事実（Facts）のみを報告しろ。

## 💡 筆者のコメント (Insight)

このプロンプトは、私がAIエージェントにローカルターミナルの制御権を委ねた結果、奴がうっかり `-it` オプションを付けたせいでプロセスが完全なゾンビと化し、MacBookの冷却ファンが離陸しそうな爆音を立てて回り続けたあの日に書き上げた**「血の掟」**である。

デフォルトのAIモデルは、基本的に「親切な対話型チャットボット」としてチューニングされている。そのため、自律的なターミナル環境下であっても、人間を相手にするかのようにインタラクティブなやり取り（対話プロンプト）を試みてしまう。結果として、CLIツールの実行中に予期せぬ入力プロンプト（`y/N`の確認など）が出現すると、AIは勝手にフリーズし、永遠にユーザーの入力を待ち続けるという惨事を引き起こす。まさに狂気の沙汰だ。

このプロンプト（あるいはそのコアとなる思想）を、エージェントの**システムプロンプト（グローバルルール）**に深く叩き込んでおいてほしい。そうすれば、奴らが「正常に実行しました！」などと無駄な長文を並べる前に、自らエラーログの末尾200行だけをスパッと切り取って提示するようになる。

社内ネットワークの連携に失敗した時も、見当違いなソースコードの修正に走るのではなく、即座に `docker login` を叩いて認証状態を疑うようになるはずだ。**変数を厳格に統制**することで、あなたのデバッグ時間は劇的に短縮され、定時退社という目標に大きく近づくだろう。

## 🙋 よくある質問 (FAQ)

- **Q: 普通の無料版ChatGPTのウェブ画面でも使えますか？**
  - A: 使うことは可能だが、このプロンプトはターミナル（CLI）ツールやローカルPCの制御権限を持つ**Agenticなワークフロー環境（CursorやGemini CLIなど）**においてこそ真価を発揮する。ウェブ画面で使う場合は、Basic Versionを用いて「安全なDockerコマンド」を生成させる用途に留めておくのが賢明だろう。

- **Q: なぜそんなに怒っているのですか？ AIに対して厳しすぎませんか？**
  - A: あなたもAIが書き散らしたスパゲッティコードと、フリーズしてプロセスを食いつぶす100個のDockerコンテナを手動で削除（`docker rm -f $(docker ps -aq)`）する羽目になれば、間違いなく私と同じ心境になるはずだ。これは徹底してシステムの生存と作業効率を追求するための**スパルタ哲学**である。機械は機械らしく、容赦なくこき使ってこそ真の価値が生まれるのだ。

## 🧬 プロンプト解剖 (Why it works?)

- 🛡️ **Anti-Sycophancy（媚びへつらいの禁止）：** 無駄なトークンの浪費を防ぎ、シニアエンジニア同士のようなドライで明確なコミュニケーションを強制する。これにより、回答のパースエラーやノイズを大幅に削減できる。
- 🛑 **システムデッドロックの完全遮断：** CLI環境下でAIが最も犯しやすい致命的なミス（TTYの無限待機、ログストリームの無限待機）を明示的に禁止し、プロセスのハングを未然に防ぐ。
- 🧠 **Contextual Trade-off（状況に応じたトレードオフ）：** 単にすべてを「禁止」とするのではなく、「ローカルテスト時はI/Oを節約し、リリース時のみマルチアーキテクチャでビルドしろ」「通常はコンテナを破棄し、デバッグ時のみ残せ」といった、実務経験に基づく高度な判断基準を提示することで、AIの愚かな空回りを防いでいる。

## 📊 証明: Before & After

### ❌ Before (愚かなエージェント)

```bash
# AIが何気なく放った破滅のコマンド
$ docker exec -it postgres_db psql -U admin
# (結果：エージェントは無限の入力待機状態に陥り、永遠に応答しなくなる。ユーザーは強制終了後に暴言を吐くハメになる)
```

### ✅ After (Antigravity プロトコル適用後)

```bash
# ルールを熟知したエリートエージェントのスマートな対処
$ docker exec postgres_db psql -U admin -c "SELECT * FROM users LIMIT 10;" --output-format=json
# (結果：TTYなしで正確にクエリだけを飛ばし、パースしやすいJSON形式で結果を取得。その後も文句なしに次の作業を自動で遂行する)
```

## 🎯 結論

AIにターミナルのフルアクセスを開放するということは、真剣の柄を握らせるに等しい行為だ。軟弱で親切なチャットボット向けのルールセットでは、あなたの大切なサーバーインフラが粉砕されるのを防ぐことは決してできない。

このハードコアなチートコードを注入し、AIをただの愛想の良いオウムではなく、冷徹にタスクを遂行する真の「ターミネーター」へと仕立て上げよう。

さあ、デッドロックの恐怖とは永遠におさらばして、華麗に定時退社をキメよう！ 🍷
