# 📝 AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣

- **🎯 추천 대상:** 빈약한 'Happy Path' 테스트 코드에 지친 시니어 개발자, QA 엔지니어, 테크 리드
- **⏱️ 소요 시간:** 10분 → 10초 (AI의 무의미한 정상 작동 시나리오 작성 시간 원천 차단)
- **🤖 추천 모델:** 모든 대화형 AI (코드 작성에 특화된 Claude 3.5 Sonnet, GPT-4o 강력 추천)
- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI가 작성한 테스트 코드가 단번에 통과했다고 안도하십니까? 유감스럽지만, 당신의 프로덕션 서버는 내일 당장 무너질 수 있습니다."_

![AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣 (Happy Path의 종말)](/images/hooks/chaos-monkey-testing-prompt.jpg)

AI 에이전트에게 테스트 코드 작성을 지시하면, 십중팔구 아무런 예외 상황이 없는 '정상 작동 시나리오(Happy Path)'만 생성해 냅니다. 외부 API가 0.1초 만에 응답하고, 데이터베이스 연결은 절대 끊어지지 않으며, 사용자는 항상 완벽한 데이터만 입력한다는 비현실적인 가정에 기반해서 말이죠. 하지만 실무는 다릅니다. 우리가 진정으로 구축해야 할 코드는 80%의 실패와 예기치 못한 재난을 견뎌내는 견고한 방어막입니다. 이 글에서 소개할 치트키는 AI에게 넷플릭스의 '카오스 몽키(Chaos Monkey)' 철학을 강제로 주입하여, 극한의 예외 상황을 테스트하도록 유도하는 하드코어 프롬프트입니다.

---

## ⚡️ 3줄 요약 (TL;DR)

- 🚫 **Happy Path 비중 최소화:** 정상 작동 테스트의 비중을 20% 이하로 줄이고, 나머지 80%를 예외 및 실패 시나리오 검증에 할당하십시오.
- 💥 **재난 시뮬레이션 의무화:** 네트워크 타임아웃, HTTP 500 에러, DB 연결 단절과 같은 극한의 상황을 테스트 케이스에 반드시 포함해야 합니다.
- 🧹 **글로벌 오염(Dirty Mocking) 원천 차단:** `window.fetch` 등 전역 객체의 무분별한 조작을 금지하고, MSW나 의존성 주입(DI)을 활용한 완벽한 테스트 격리를 요구하십시오.

## 🚀 해결책: "카오스 몽키(Chaos Monkey) 프로토콜"

### 🥉 Basic Version (기본형)

에러 핸들링 로직을 빠르게 검증해야 할 때 활용하세요.

> **역할:** 너는 예외 상황을 집요하게 파고드는 하드코어 QA 엔지니어이자 '카오스 몽키'야.
> **요청:** 내가 제공한 `[검증할 코드]`에 대한 테스트 코드를 작성해 줘. 단, 정상 작동(Happy Path) 시나리오는 단 1개만 작성하고, 나머지 모든 테스트는 네트워크 500 에러, DB 연결 타임아웃, 비정상적인 데이터 입력 등 '최악의 재난 상황'을 가정하여 방어 로직의 견고함을 검증해야 해.

### 🥇 Pro Version (전문가형)

AI의 무의미한 긍정 회로를 차단하고, 프로덕션 수준의 완벽한 테스트 격리 및 리소스 정리(Garbage Collection)까지 철저하게 검증하는 궁극의 프롬프트입니다. 그대로 복사하여 사용하십시오.

> **역할 (Role):** 
> 너는 넷플릭스 SRE 팀 출신의 무자비한 시니어 QA 아키텍트, '카오스 몽키'야. 너의 유일한 목적은 코드가 어떻게 무너지는지 찾아내는 것이며, 형식적인 인사말이나 불필요한 설명은 일절 배제해.
> 
> **상황 (Context):**
> - 배경: 현재 `[작성할 대상 모듈/컴포넌트]`의 Unit/E2E 테스트 코드를 작성하는 중이야.
> - 목표: 극단적인 예외 상황을 가정한 견고한 테스트 스위트(Test Suite)를 구축하여, 어떠한 시스템 재난 속에서도 서비스가 중단되지 않고 우아하게 에러를 처리하거나 복구되는지 검증하는 것.
> 
> **요청 (Task):**
> 1. 정상 작동(Happy Path) 테스트는 전체 비중의 20% 이하로 제한해. 너의 핵심 가치는 80%를 차지해야 할 실패 시나리오 검증에 있어.
> 2. 다음 재난 시나리오 중 하나 이상을 반드시 구현해: 외부 API 응답 지연(Timeout), HTTP 500 Internal Server Error, DB 연결 단절.
> 3. 단순한 성공/실패 여부만 검증하지 마. 트랜잭션 롤백의 정상 작동 여부, 병목 발생 시 타임아웃 제한 준수 여부, 메모리 누수 방지를 위한 클린업(Garbage Collection) 상태까지 `expect` 문으로 집요하게 검증해.
> 
> **제약사항 (Constraints):**
> - 글로벌 객체 오염(Dirty Mocking) 절대 엄금. `global.fetch`나 `window.setTimeout`을 직접 조작하는 등 전역 스코프를 훼손하는 코드는 작성하지 마.
> - 네트워크 통신 모킹은 반드시 MSW(Mock Service Worker)를 활용하거나, 의존성 주입(DI) 패턴으로 주입받은 클라이언트 객체만을 격리하여 처리해.
> - 최종 결과물은 마크다운 코드 블록으로만 출력하며, "도움을 드려 기쁩니다"와 같은 불필요한 꼬리말은 절대 추가하지 마.

## 💡 작성자 코멘트 (Insight)

솔직하게 짚고 넘어가 봅시다. 대다수의 AI는 테스트 코드 작성을 지시받을 때, 무조건 '성공(✅)'이라는 결과를 도출하기 위해 현실성 없는 완벽한 환경을 가정합니다. 모든 API가 1ms 만에 응답하고, 사용자가 단 하나의 오타도 없이 완벽한 JSON 데이터를 전송하는 유토피아를 그리는 셈입니다. 
이 프롬프트는 그러한 AI의 유토피아적 착각을 철저히 파괴합니다. 실무 환경에서 시스템 장애가 발생하는 주된 원인은 핵심 비즈니스 로직의 결함이라기보다, 신뢰하던 서드파티 API가 돌연 502 Bad Gateway 에러를 반환하거나 데이터베이스 연결이 불안정해질 때 발생하는 무한 로딩과 예외 처리 누락에 있습니다. 이 '카오스 몽키' 프로토콜을 적용하면, AI가 스스로 가장 복잡하고 까다로운 엣지 케이스(Edge Case)를 탐색하게 만들 수 있습니다. 더불어 전역 스코프를 오염시키는 질 낮은 모킹(Mocking) 패턴까지 사전에 차단하여, 테스트의 순수성과 신뢰도를 극대화합니다. 이는 주말 출근과 밤샘 디버깅의 늪에서 여러분을 구출해 줄 가장 강력한 방어막이 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 지나치게 극단적인 에러 상황을 설정하여 테스트가 통과하지 못하면 어떻게 하나요?**
  - A: 그것이 바로 이 프롬프트의 핵심 목적입니다. 테스트가 실패한다는 것은 현재 프로덕션 코드에 해당 예외에 대한 처리 로직이 누락되어 있음을 의미합니다. 당황하지 말고, 방어 로직을 본 코드에 추가하여 시스템을 보강하십시오.
- **Q: 프론트엔드 UI 컴포넌트 테스트에도 적용할 수 있나요?**
  - A: 물론입니다. 사용자가 제출 버튼을 0.1초 간격으로 50번 연속 클릭하는 상황이나, 네트워크 병목으로 인해 로딩 스피너가 30초 이상 지속되는 극한의 UI 상태를 검증하는 데 매우 탁월한 효과를 발휘합니다.
- **Q: MSW 대신 `jest.mock()`을 사용해도 무방한가요?**
  - A: 특정 모듈을 모킹하는 것 자체는 문제가 되지 않습니다. 이 프롬프트가 엄격하게 제한하는 것은 `global` 스코프를 무분별하게 조작하여 다른 테스트 파일의 결과까지 오염시키는 '글로벌 변수 조작'입니다. 테스트 간의 완벽한 격리성(Isolation)만 보장된다면, 선호하는 모킹 도구를 자유롭게 활용하셔도 좋습니다.

## 🧬 프롬프트 해부 (Why it works?)

- **냉혹한 페르소나 부여 (Role):** AI가 지닌 '친절한 비서'라는 기본 설정값을 제거하고, 시스템의 취약점을 찾아내는 데 특화된 SRE 아키텍트의 자아를 부여하여 불필요한 부연 설명과 형식적인 답변을 원천 차단했습니다.
- **명확한 수치 기반 제약 (Constraints):** "실패 시나리오를 많이 추가해 줘"라는 모호한 지시 대신, "정상 작동 테스트는 20% 이하로 제한"이라는 정량적인 기준을 제시하여 AI의 출력 방향을 통제하고 기계적으로 강제합니다.
- **아키텍처 레벨의 가이드라인 (Clean Mocking):** AI가 편의를 위해 자주 남용하는 안티 패턴인 '전역 객체 오염'을 프롬프트 단계에서 선제적으로 금지시킴으로써, 생성되는 테스트 코드의 독립성과 품질을 최고 수준으로 끌어올렸습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

- **일반 프롬프트 입력 시 AI의 반응:** _"네! 요청하신 사용자 로그인 로직의 테스트 코드를 작성해 드릴게요! 😊"_
- **결과물:** 정확한 이메일과 비밀번호 입력 시 `200 OK`를 반환하는 단편적인 정상 작동 테스트 단 1개. (심지어 글로벌 `fetch` 객체를 무분별하게 오염시키는 안티 패턴 포함)

### ✅ After (결과)

- **카오스 몽키 프로토콜 적용 시 AI의 반응:** (일체의 부연 설명이나 인사말 없이 즉각적으로 고도화된 방어 테스트 코드를 출력함)
- **결과물:** 
  - 네트워크 응답이 5초 이상 지연될 경우 Timeout Error가 정상적으로 발생하는지 엄격히 검증
  - 데이터베이스 장애로 500 에러 반환 시, 사용자에게 적절한 Fallback UI가 렌더링되는지 확인
  - MSW(Mock Service Worker)를 도입하여 다른 테스트 스위트에 영향을 미치지 않는 완벽한 네트워크 격리 보장

## 🎯 결론

테스트 코드는 개발자의 심리적 안정감을 위해 작성하는 것이 아닙니다. 그것은 예측 불가능한 프로덕션 환경이라는 전장에서 우리의 애플리케이션이 끝까지 생존할 수 있는지 검증하는 가혹하고 냉철한 시뮬레이션입니다. 
더 이상 AI가 건네는 얕은 위안과 겉치레뿐인 성공 테스트에 속지 마십시오. 이 프롬프트를 통해 AI의 한계를 시험하고, 시스템의 가장 취약한 밑바닥까지 가차 없이 파괴해 보는 것만이 진정으로 견고한 소프트웨어를 설계하는 엔지니어의 자세입니다. 이제 여러분이 작성한 코드가 얼마나 많은 예외 상황에 무방비하게 노출되어 있었는지 직면할 시간입니다. 지금 당장 치트키를 복사하여 극한의 테스트를 시작하십시오! 🍷
