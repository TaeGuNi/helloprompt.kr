# 📝 1分钟搞定：昨日工作总结与每日站会准备

- **🎯 推荐对象：** 开发者、项目经理（PM）、以及所有每天需要开敏捷站会的职场人
- **⏱️ 预计耗时：** 15分钟 → 缩短至 1分钟
- **🤖 推荐模型：** 任何对话型 AI (ChatGPT, Claude, Gemini 等)

- ⭐ **难度：** ⭐☆☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"每天早上开站会前，是不是都在疯狂翻 GitHub 提交记录和 Jira 面板，拼命回想昨天到底干了什么？"_

每天的每日站会（Daily Standup）本来是为了同步进度、暴露风险，但如果不提前准备，很容易变成毫无重点的流水账。特别是对于开发者来说，代码写得越多，越难用简练的语言向非技术人员（如 PM 或设计团队）解释清楚自己昨天完成的业务价值。

现在，你只需要把昨天乱七八糟的 Commit Log、聊天记录或者零散的代码片段扔给 AI，它就能在1分钟内帮你生成一份结构清晰、重点突出的站会汇报脚本。

---

## ⚡️ 3句话总结 (TL;DR)

1. **告别流水账**：将零碎的 Git 提交记录自动转化为有逻辑的业务汇报。
2. **跨部门沟通无碍**：自动将晦涩的底层技术术语翻译成产品经理也能听懂的业务进度。
3. **今日计划一目了然**：不仅总结昨天，还能根据上下文顺理成章地规划今天的重点任务。

---

## 🚀 解决方案："一键生成站会汇报"

### 🥉 Basic Version (基础版)

当你快要轮到发言，急需一份简短的汇报小抄时使用。

> **角色：** 你是一位经验丰富的高级软件工程师。
> **任务：** 请根据我提供的 `[昨日工作记录]`，帮我写一段用于每日站会的口头汇报稿。请保持简短，重点突出我完成了什么以及是否遇到阻碍。
>
> **[昨日工作记录]：**
> (在此粘贴你的 Commit Log 或工作笔记)

<br>

### 🥇 Pro Version (专业版)

当你需要向包含非技术人员的跨职能团队进行高质量、高信息密度的汇报时使用。

> **角色 (Role)：** 你是一位资深的技术研发主管（Tech Lead），不仅精通代码，还非常擅长跨部门沟通，能将复杂的技术问题转化为清晰的业务价值。
>
> **背景 (Context)：**
>
> - 当前场景：即将开始跨职能团队的每日敏捷站会（Daily Standup）。
> - 听众：包括产品经理（PM）、设计师和前端/后端开发人员。
> - 目标：用清晰、自信且通俗易懂的语言，汇报昨日进度、今日计划，并明确指出是否有阻塞项（Blockers）。
>
> **任务 (Task)：**
>
> 1. 分析我提供的 `[原始工作数据]`（如 Git 提交记录、Jira 票据或零散笔记）。
> 2. 将纯技术性的描述转化为**业务导向**的成果说明（例如，不说“重构了 Auth 模块”，而说“优化了登录流程，减少了用户等待时间”）。
> 3. 生成一份符合“昨天做了什么 / 今天打算做什么 / 有什么困难”标准结构的口语化汇报稿。
>
> **输入变量 (Input)：**
>
> - `[原始工作数据]`：(在此粘贴你的数据)
>
> **制约因素 (Constraints)：**
>
> - 汇报时间请控制在 1 分钟内（约 150-200 字）。
> - 语气要专业、积极且口语化，适合直接流畅朗读。
> - 输出格式：请使用清晰的 Markdown 列表展现汇报结构，并高亮关键成就。
>
> **注意事项 (Warning)：**
>
> - 请勿编造任何我没有提供的功能或进度（防止 AI 幻觉）。如果信息不足以规划“今日任务”，请在脚本中用括号提示我自行补充。

## 💡 作者见解 (Insight)

这种方法不仅是用来“偷懒”的工具，更是提升**职场能见度（Visibility）**的利器。很多初中级开发者在站会上喜欢“报菜名”式地罗列代码文件或函数名，这其实是沟通效率极低的做法。

通过使用 Pro 版本的提示词，AI 能够强制让你从“业务价值”的角度重新审视自己的工作。当你习惯了将“修复了 SQL N+1 查询问题”转化为“提升了首页 30% 的加载速度”时，你在团队中的技术影响力（Tech Influence）也会随之水涨船高。这正是用好 AI 帮助你完成思维进阶的典型案例。

## 🙋 常见问题 (FAQ)

- **Q: 如果我的工作涉密，可以把代码发给 AI 吗？**
  - A: 绝对不行！切记不要将公司机密代码或真实的 API Key 放入提示词中。你只需要提供脱敏后的 Commit Message（如 `fix: 修复登录 bug`）即可，无需粘贴具体代码。对于高度机密的项目，建议使用本地部署的开源大模型。

- **Q: AI 生成的语气听起来太像播音员，不够自然怎么办？**
  - A: 如果觉得太正式，可以在提示词的约束条件中加入：“请使用随意的口语化表达，多用短句，可以加入一点口语停顿词（如‘那个’、‘主要是’）”。这样生成的脚本会更加接地气。

- **Q: 我可以一次性丢入一周的日志来写周报吗？**
  - A: 完全可以。只需将背景设定改为“每周项目例会”，并要求输出格式为“本周核心产出”即可，AI 同样能出色完成任务。

---

## 🧬 提示词解剖 (Why it works?)

1.  **受众设定（Audience Targeting）**：明确指出听众包含“产品经理和设计师”，这就迫使 AI 自动过滤掉晦涩的底层技术细节，转而输出大家都听得懂的业务语言。
2.  **角色映射（Role-Play）**：赋予 AI “资深 Tech Lead” 的角色，使其输出的语调自带专业感和掌控感，有助于提升你在会议上的发言气场。
3.  **防幻觉约束（Anti-Hallucination Constraints）**：特别强调“切勿编造”，防止 AI 为了凑字数而虚构你并没有做过的工作，保证了汇报的真实性和你的职场信誉。

```text
- refactor: move Redis logic to separate service
- feat: add loading spinner to dashboard
- WIP: payment gateway integration
```

### ✅ After (AI 生成的站会脚本)

```text
大家早上好！

关于昨天的工作，我主要推进了三个方面：
首先，修复了用户登录时偶尔会出现的令牌失效问题，目前登录流程已经完全稳定；
其次，对后端的缓存逻辑做了一次架构优化，这为后续的高并发打下了基础；
最后，给数据看板加上了加载动画，大幅提升了用户的视觉体验。

今天，我的核心任务是继续推进支付网关的接入工作。

目前进展顺利，没有任何阻塞项（Blocker）。完毕！
```
