## 📝 原因不明のバグは「AI探偵」に丸投げしよう

- **🎯 おすすめの対象:** ジュニア開発者、フロントエンド / バックエンドエンジニア、終わらないデバッグに絶望している方
- **⏱️ 所要時間:** 2時間（頭を抱える時間） → 3分に短縮
- **🤖 最適なモデル:** Claude 3.5 Sonnet, GPT-4o（コード推論に特化したモデルを推奨）

- ⭐ **難易度:** ⭐☆☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「スタックトレースを見ても、一体どこから手をつければいいのか全く見当がつかない…」_

意味不明なエラーメッセージに直面し、ログをいくら辿ってもバグの尻尾すら掴めない。開発者であれば誰もが一度は経験する、あの絶望的な時間を劇的に短縮する「AIデバッグ探偵」プロンプトをご紹介します。エラーログと周辺コードを丸投げするだけで、AIが論理的に根本原因を推論し、副作用のない的確な解決策を瞬時に提示してくれます。もう、終わりの見えないデバッグに何時間も溶かす必要はありません。

---

## ⚡️ 3行まとめ (TL;DR)

1. 難解なエラーログやスタックトレースをAIにそのまま貼り付けるだけで分析完了。
2. 問題が疑われるコードスニペットを添えることで、推論の精度が劇的に向上。
3. 表面的なコード修正にとどまらず、「なぜバグが発生したのか」という根本原因まで徹底解剖。

## 🚀 解決策: "デバッグ探偵プロンプト"

### 🥉 Basic Version (基本型)

手っ取り早く解決の糸口を掴みたい時に活用できる軽量版です。

> **役割 (Role):** あなたはシニアエンジニアです。
>
> **タスク (Task):** 以下のエラーログを確認し、バグの原因と解決策を提示してください。
>
> **エラーログ (Error Log):**
> `[ここにエラーログを貼り付けてください]`
>
> **関連コード (Related Code):**
> `[疑わしい周辺コードを貼り付けてください]`

### 🥇 Pro Version (専門家型)

複雑なバグの根本原因を正確に突き止め、副作用のない安全な修正コードを得たい実務向けの完全版です。

> **役割 (Role):** あなたはトラブルシューティングとパフォーマンス最適化に長けた、10年以上の経験を持つシニアソフトウェアエンジニア（AI探偵）です。
>
> **状況 (Context):**
>
> - 背景: アプリケーションの実行中に原因不明のエラーが発生し、開発が完全にストップしています。
> - 目標: エラーの根本原因（Root Cause）を正確に特定し、副作用（Side-effect）のない最適な修正コードを導き出すこと。
>
> **要求 (Task):**
> 提供されたエラーログとコードを分析し、以下の項目を順に説明してください。
>
> 1. **バグの根本原因:** なぜこのエラーが発生したのかを論理的に説明してください。
> 2. **修正コード:** 問題を解決するための具体的なコードを提示してください。
> 3. **予防策:** 今後同様のバグを防ぐための実践的なアドバイスを1〜2行で追加してください。
>
> **入力データ (Input):**
>
> - エラーログ / スタックトレース: `[ここにスタックトレース全体を貼り付けてください]`
> - 関連コード: `[問題が発生している周辺のコードを貼り付けてください]`
> - 使用技術: `[例: React, Node.js, Python など]`
>
> **制約事項 (Constraints):**
>
> - 視認性の高いマークダウン形式で出力してください。
> - 修正コードには、変更点がひと目でわかるよう明確なインラインコメントを記載してください。
> - **情報が不足しており原因の特定が困難な場合は、決して推測で答えず、「さらに確認が必要なファイルやログ」を私に要求してください。**

## 💡 筆者コメント (Insight)

このプロンプトの真の価値は、単に「コピペで動くコード」を得ることではありません。AIを通して**「優秀なシニアエンジニアのデバッグ思考プロセス」そのものを学べる**点にあります。

特に Pro Version に組み込んだ「**情報が不足している場合は、推測で答えずに追加情報を要求せよ**」という制約事項（Constraints）は極めて重要です。この一文があることで、AIが勝手な解釈で尤もらしい嘘（ハルシネーション）を吐き出すのを未然に防ぎます。まるでペアプログラミングをしているかのように、対話を通じてバグを正確に追い詰めることができるのです。

画面を埋め尽くすような真っ赤なスタックトレースを前に途方に暮れた時は、一人で抱え込まず、まずはこのAI探偵に一次分析を任せてみてください。バグ修正にかかる精神的な負担が格段に軽くなるのを実感できるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: エラーログが長すぎてプロンプトに入りきりません。どうすればいいですか？**
  - A: スタックトレースの「先頭の数行（Exceptionの種類とメッセージ）」と「自分のプロジェクトのファイルパスが含まれている行」を中心に抜粋して貼り付けてください。ReactやNode.jsなどのフレームワークやライブラリ内部の深いログ（`node_modules` 以下のもの）は省略しても推論にはほとんど影響しません。

- **Q: どのAIモデルを使うのが最も効果的ですか？**
  - A: コードの文脈理解や論理的推論においては、現時点で **Claude 3.5 Sonnet**（またはその後継モデル）が最も優れたパフォーマンスを発揮します。次点として OpenAI の **GPT-4o** も非常に優秀ですので、お好みの環境でご活用ください。

## 🧬 プロンプトの解剖学 (Why it works?)

1.  **AI探偵ペルソナの付与 (Role):** 「10年以上の経験を持つシニアエンジニア」という具体的な役割を与えることで、単なるタイポや文法エラーの指摘にとどまらず、アーキテクチャやベストプラクティスに基づいた深いレベルの洞察を引き出します。
2.  **根本原因の言語化 (Task):** 修正コードだけを出力させるのではなく、「なぜこの問題が起きたのか」を論理的に説明させる設計にしています。これにより、同種のバグの再発を防ぐとともに、開発者自身のデバッギングスキルの向上に直結します。
3.  **ハルシネーションの遮断 (Constraints):** 「わからない時は推測するな」という強力な制約を設けることで、見当違いな解決策を試して泥沼にハマる時間を物理的に排除しています。

## 📊 証明: Before & After

### ❌ Before (開発者を悩ませるエラーログ)

```text
TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (UserList.jsx:15:23)
    ...
```

### ✅ After (AI探偵による鮮やかな解決策)

````markdown
### 🔍 バグの根本原因

`UserList.jsx` の15行目で、`users` 配列に対して `.map()` メソッドを呼び出していますが、現在 `users` が `undefined`（未定義）の状態でレンダリングが実行されてしまっているためです。APIからの非同期データ取得が完了する前にコンポーネントが描画された可能性が高いと考えられます。

### 🛠️ 修正コード

```jsx
// 修正前: {users.map(user => ...)}
// 修正後: オプショナルチェーニング (?.) を使用するか、早期リターンで対処します。

function UserList({ users }) {
  // users が undefined や null の場合は何も表示しないか、ローディングUIを返す
  if (!users) return <p>Loading...</p>;

  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

### 🛡️ 予防策

配列を Props で受け取る際は、コンポーネント側でデフォルト値として空の配列 `[]` を設定する（例: `function UserList({ users = [] })`）か、TypeScriptを導入してコンパイル段階で型安全性を担保することをお勧めします。
````
