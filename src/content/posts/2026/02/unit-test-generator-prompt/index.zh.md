> _“明天就要发布了，光是写核心业务逻辑都忙不过来，哪有时间写单元测试啊？”_

作为开发人员，谁都经历过这种背脊发凉的时刻。周五下午，满怀信心地将新功能部署到生产环境，结果在意想不到的地方爆发了严重 Bug。打开代码准备热修复（Hotfix），却发现业务逻辑过于庞大，甚至难以确定问题出在哪里。_“要是当时认真写了测试代码就好了……”_ 这样的后悔已经太迟。我们深知测试代码的重要性，诸如整洁架构（Clean Architecture）、TDD、100% 覆盖率等词汇总是能让开发人员心潮澎湃。

但现实是残酷的。在明天就要发布、且必须处理堆积如山的策划需求的情况下，很少有开发人员能从容地投入比编写业务逻辑多出 2-3 倍的时间去写测试代码。尤其是处理具有大量外部 API 依赖的遗留代码（Legacy Code），或者测试与数据库紧密耦合的逻辑时，仅仅是设置数十行的 **Mocking（模拟）** 代码和样板代码（Boilerplate）就足以让人精疲力竭。

最终，在疲于思考各种棘手的边界情况（Edge Case）后，我们往往会做出危险的妥协：“这次发布先相信 QA 团队的人工测试，就这样上吧。” 这样积累下来的 **技术债（Technical Debt）** 最终会变成沉重的枷锁，导致代码无法重构，像回旋镖一样伤及自身。每次点击修改按钮，都会担心其他功能被破坏，这种焦虑感由此开始。这种没有安全网的高空走钢丝杂耍必须停止。

现在，请放下所有痛苦的过程和乏味的思考，只需将你辛勤编写的核心逻辑进行简单的复制粘贴。从开发人员容易忽略的微小异常情况（Edge Case）到严密的空指针（Null Pointer）防线，**AI 将瞬间为你编写出完美运行的零缺陷单元测试（Unit Test）代码**。

只需 5 分钟。从生成复杂的 Mocking 对象到应用 GIVEN-WHEN-THEN 模式的整洁测试场景，请雇佣一位完美主义倾向的高级 QA 工程师作为你的结对编程伙伴。AI 会用**机械般的缜密逻辑填补人类直觉容易遗漏的盲区**。带着 100% 覆盖率带来的心理安全感，享受自信按下“Merge”按钮的快感。

### ❌ Before (我们曾经历的痛苦)

这是逻辑编写的瞬间，由于传统的手动方式导致代码粗糙，或者面对空白画布感到痛苦。这是一个没有测试代码、孤零零留下的危险逻辑。

```javascript
// 简单的输入代码示例
function calculateDiscount(price, discountRate) {
  if (price < 0 || discountRate < 0) {
    throw new Error("无效的值。");
  }
  if (discountRate > 100) {
    return 0; // 超过 100% 的折扣即为免费
  }
  return price - price * (discountRate / 100);
}
```

### ✅ After (完美蜕变的结果)

!["“5分钟完成测试代码编写”的高效提示词"](/images/hooks/unit-test-generator-prompt.jpg)

```javascript
describe("calculateDiscount 函数测试", () => {
  // 1. 正常运行测试 (Happy Path)
  it("如果给定正常的价格和折扣率，应返回折扣后的价格", () => {
    // GIVEN
    const price = 10000;
    const discountRate = 20;
    // WHEN
    const result = calculateDiscount(price, discountRate);
    // THEN
    expect(result).toBe(8000);
  });

  // 2. 边界值测试 (Edge Case)
  it("如果折扣率超过 100%，应返回 0 元", () => {
    // GIVEN & WHEN
    const result = calculateDiscount(10000, 110);
    // THEN
    expect(result).toBe(0);
  });

  // 3. 异常处理测试 (Failure Case)
  it("如果价格为负数，应抛出错误", () => {
    // GIVEN, WHEN & THEN
    expect(() => calculateDiscount(-5000, 10)).toThrow(
      "无效的值。"
    );
  });
});
```

## ⚡️ 3句总结 (TL;DR)

1. **从 Happy Path 到 Edge Case：** 自动导出测试方案，不仅验证正常运行流程，还能严密检查人类难以预测的异常情况。
2. **适配主流框架：** 准确反映 Jest, PyTest, JUnit 等当前项目中采用的测试框架语法，生成即插即用的代码。
3. **自动化处理 Mocking：** AI 自行分析复杂的外部 API 调用或数据库依赖，完美配置相应的 Mock 数据和 Stub。

## 💡 作者点评 (Insight & How to use)

这个提示词不仅仅是“代写测试代码”的自动补全，它是一个强大的 **质量保证（QA）合作伙伴**。尤其是提示词中明确的第一项指令——**“导出测试用例列表”**，这才是该提示词的核心价值和差异化所在。在 AI 盲目编写代码之前，你可以先通过 Markdown 列表形式查看它如何分析和解释该业务逻辑。通过这种方式，你可以预先防御那些连开发人员自己都未曾考虑到的致命边界情况（Edge Case：如极端的 Null 输入、意料之外的类型不匹配、并发问题等），弥补逻辑上的漏洞。

在实际工作中，即使使用相同的测试框架（如 Jest, JUnit, PyTest），根据不同的开发团队或公司惯例，首选的 **Mocking 方式**或代码风格也难免会有细微差别。有些团队喜欢 `jest.spyOn`，而有些团队则坚持通过依赖注入（DI）创建伪对象（Fake Object）。如果 AI 生成的代码不符合你当前项目的严格标准，请尝试在提示词底部的约束条件部分添加一个 **`[我们团队的 Mocking 示例代码]`** 片段。由于最新的 AI 模型具有非常出色的 Few-Shot 学习能力，它们会立即学习你团队独特的风格和语法，输出可以直接投入实战的完美代码。

此外，测试代码不是写完就了事的，它是需要随着核心逻辑的更改而持续维护的“活文档”。这个提示词被设计为强制使用 **GIVEN-WHEN-THEN** 这一行业标准模式。这不仅仅是为了让代码看起来漂亮，更是一个强大的沟通工具，帮助几个月后重新打开代码的未来的你，或者是接手这段代码的其他队友，能在 1 秒钟内直观地把握复杂逻辑的意图和流程。

在编写测试代码时，必然会遇到数据库连接或与第三方 API 通信的情况。此时，初学者常会犯下编写调用实际数据库或 API 的集成测试（Integration Test）的错误。但这个提示词包含了 **“外部依赖完美 Mocking 处理”** 的严格指令。AI 会聪明地感测到发生网络通信或 I/O 操作的区间，并自动用完全不需要实际网络连接的伪对象（Stub/Mock）取而代之。得益于此，你可以从源头上杜绝那些因依赖网络状态而时好时坏的不稳定测试（Flaky Test），构建起一个无论何时何地运行都能在 0.1 秒内保证一致成功的坚固单元测试环境。

最后，在提交整段代码之前，请再次回想 **单一职责原则（SRP）**。如果你将一个长达数百行的庞大“面条函数”直接塞进提示词，即使是再出色的 AI 也可能会丢失核心上下文或遗漏重要的边界情况验证。最佳的使用方法是，按功能拆分成细小的独立函数单元，分别请求编写测试代码。这将成为一个极佳的重构指标，让你直观地反馈出你现有的核心逻辑是否被设计得足够模块化且易于测试（Testable Code）。

## 🙋 常见问题 (FAQ)

- **Q：代码太长导致 AI 的回答中途断开怎么办？**
  - A：强烈建议按函数或类为单位进行拆分请求。输入整个庞大的文件会增加 AI 遗漏核心边界情况的概率。请尝试分离出严格遵守单一职责原则（SRP）的函数，逐一请求编写测试。

- **Q：代码中包含数据库连接，这种情况也能生成单元测试代码吗？**
  - A：是的，完全可以！这个提示词包含了关于“外部依赖 Mocking”的强力指令。AI 会智能地感测到数据库 Repository 或 ORM 调用区间，并自动将其替换为不需要实际数据库连接的伪对象（Mock）。
