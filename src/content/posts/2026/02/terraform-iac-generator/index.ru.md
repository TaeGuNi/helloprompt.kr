## 🏗️ Инфраструктура без кликов! Генератор кода Terraform

- **🎯 Кому подойдет:** Разработчикам, уставшим от консоли AWS, и бэкендерам, которым лень вручную управлять инфраструктурой.
- **⏱️ Экономия времени:** 1 час чтения документации → 3 минуты.
- **🤖 Рекомендуемая модель:** Claude 3.5 Sonnet (идеально для написания кода и проектирования архитектуры).

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _"Как долго вы еще будете держаться за консоль, рискуя удалить базу данных одним неверным кликом?"_

К серверам нужно относиться не как к домашним питомцам, а как к скоту (концепция Pet vs. Cattle). Если сервер падает, вы не должны реанимировать его всю ночь — его нужно безжалостно уничтожить и поднять точную копию за секунду. Именно эту суперсилу дает **IaC (Infrastructure as Code)**. Но давайте честно: вспоминать замороченный синтаксис HCL (HashiCorp Configuration Language) и бесконечные атрибуты провайдеров каждый раз, когда нужно развернуть базу — то еще удовольствие.

Этот промпт превращает ленивое "подними мне EC2 в AWS и прикрути базу" в **идеальный, модульный и безопасный код Terraform**, который не стыдно сразу катить в production.

---

## ⚡️ 3 главных преимущества (TL;DR)

1. Мгновенная генерация кода, **строго отвечающего стандартам HCL**, без многочасового курения документации.
2. Принудительное внедрение **настроек безопасности по принципу наименьших привилегий (SG, IAM)** для защиты от фатальных ошибок вроде открытого порта `0.0.0.0/0`.
3. Создание **переиспользуемых модулей** за счет жесткого разделения на `variables.tf` и `outputs.tf` без единой строчки хардкода.

## 🚀 Решение: "Генератор идеальной инфраструктуры"

### 🥉 Базовая версия (Basic Version)

Используйте, когда нужно быстро поднять один ресурс для тестирования.

> **Роль (Role):** Ты Senior DevOps-инженер.
>
> **Задача (Task):** Напиши код Terraform для создания ресурса AWS `[Название сервиса, например: S3 bucket]`.
>
> **Ограничения (Constraints):** Используй последнюю версию AWS Provider. Избегай хардкода, все параметры выноси в переменные (variables).

### 🥇 Продвинутая версия (Pro Version)

Используйте для построения реальной production-среды, где безопасность и высокая доступность (HA) критически важны.

> **Роль (Role):** Ты облачный архитектор с 10-летним опытом и эксперт по Terraform, специализирующийся на безопасности и высокой доступности (HA).
>
> **Контекст (Context):**
>
> - Ситуация: Я проектирую инфраструктуру в AWS для `[Описание проекта, например: глобального e-commerce бэкенда на 10 000 RPS]`.
> - Цель: Получить production-ready код IaC (Infrastructure as Code), отвечающий строгим требованиям ИБ, отказоустойчивости и модульности.
>
> **Задача (Task):**
>
> 1. **Архитектура:** Напиши конфигурацию основных ресурсов (`main.tf`) для реализации требуемой инфраструктуры.
> 2. **Структура и модульность:** Никакого хардкода. Жестко разнеси логику по файлам `variables.tf` и `outputs.tf`.
> 3. **Безопасность:** Для Security Group применяй принцип наименьших привилегий (Least Privilege). Использование `0.0.0.0/0` строжайше запрещено, если это не обосновано архитектурой.
> 4. **Управление доступом:** Сгенерируй IAM Role и Policy, выдающие исключительно минимально необходимый набор прав.
>
> **Ограничения (Constraints):**
>
> - Используй актуальный синтаксис HashiCorp Configuration Language (HCL) и AWS Provider версии v5.0+.
> - Имена всех ресурсов и теги (Tags) должны содержать префикс `[Название проекта]`.
> - Код должен быть идеально отформатирован по стандарту `terraform fmt`.
>
> **Предупреждение (Warning):**
>
> - Категорически запрещено использовать устаревшие (Deprecated) аргументы или небезопасные дефолтные значения. Не выдумывай несуществующие параметры (защита от галлюцинаций).

## 💡 Комментарий автора (Insight)

Самая большая боль при слепом доверии написания Terraform-кода нейросетям — это **провалы в безопасности (Security)**. Код из-под обычного промпта обожает ради "удобства" открывать все порты настежь (`0.0.0.0/0`) или раздавать IAM-ролям права уровня AdminAccess.

Суть продвинутой (Pro) версии в том, что она **аппаратно заставляет ИИ соблюдать принцип наименьших привилегий и маниакальную модульность**. На практике я рекомендую использовать этот промпт для генерации железобетонного каркаса, а затем обязательно прогонять результат через сканеры безопасности вроде `tfsec` или `checkov` в вашем CI/CD. Вы ощутите настоящую магию, когда рутинная настройка инфраструктуры, на которую раньше уходили дни, сожмется до трех минут.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Будет ли этот промпт работать с GCP или Azure, а не только с AWS?**
  - О: Абсолютно. Просто замените облачного провайдера на 'Google Cloud' или 'AzureRM' в блоках `[Контекст]` и `[Ограничения]`, и укажите целевые сервисы нужного вендора. Логика промпта универсальна.

- **В: ИИ генерирует код со старым синтаксисом (v0.11), из-за чего сыплются ошибки. Что делать?**
  - О: Это классическая болезнь устаревших моделей вроде GPT-3.5. Несмотря на явное указание "использовать новейший синтаксис", для IaC-задач я настоятельно рекомендую использовать **Claude 3.5 Sonnet** (он гениален в коде) или свежую **GPT-4o**. Они выдают на порядок более стабильный и современный HCL-код.

## 🧬 Разбор промпта (Why it works?)

1. **Инъекция экспертизы (Role):** Персона "архитектора с 10-летним опытом" заставляет ИИ перестать мыслить отдельными ресурсами и начать проектировать Enterprise-архитектуру с прицелом на безопасность и отказоустойчивость.
2. **Жесткие рамки (Constraints):** Ультиматум на использование свежего HCL и стандарта `terraform fmt` выдает на выходе чистый, валидный код, готовый к `terraform init && terraform apply` без лишних правок.
3. **Security by Design:** Прямой запрет на дырявые дефолтные настройки и фокус на "наименьших привилегиях" отсекают 90% детских болезней с открытыми Security Groups и перекачанными IAM-ролями.

## 📊 Доказательство: До и После (Before & After)

Разница между "сделай мне по-братски" и профессиональным инженерным промптом — колоссальна.

### ❌ До (Обычный запрос)

```text
Создай VPC в AWS, сделай 2 публичных сабнета и 2 приватных.
```

**Результат (Хардкод и монолит):**

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
# Параметры захардкожены без переменных, отсутствуют теги (Tags), монолитный файл без возможности расширения.
```

### ✅ После (Результат Pro-промпта)

**Результат (Модульность и production-стандарты):**

```hcl
# main.tf
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.project_name}-vpc"
  cidr = var.vpc_cidr

  azs             = var.availability_zones
  private_subnets = var.private_subnets
  public_subnets  = var.public_subnets

  enable_nat_gateway     = true
  single_nat_gateway     = false # NAT для каждой AZ ради высокой доступности
  one_nat_gateway_per_az = true

  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "Terraform"
  }
}
```

_(※ Бонусом ИИ также генерирует идеально структурированные `variables.tf` и `outputs.tf`.)_
