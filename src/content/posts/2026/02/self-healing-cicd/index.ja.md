## 📝 寝ている間にバグが直る？2026年型自己修復（Self-Healing）CI/CDパイプラインの構築

- **🎯 推奨ターゲット:** DevOpsエンジニア、CI/CD担当者、些細なLintエラーにうんざりしているすべての開発者
- **⏱️ 所要時間:** エラーのデバッグ 30分 → 1分に短縮（完全自動化）
- **🤖 推奨モデル:** Claude 3.5 Sonnet, GPT-4o（コード解析・修正に特化）

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「退勤直前に作成したPRが、Lintエラーや些細な型不一致で赤いバツ（❌）になり、着たばかりの上着を脱いでノートPCを再び開いた経験はありませんか？」_

2026年の現在、私たちは**「自己修復（Self-Healing）パイプライン」**の時代を生きています。CIが失敗すると、AIエージェントが即座にエラーログを解析し、自律的にコードを修正して再コミットを実行します。翌朝出社したあなたを迎えるのは、いつの間にか緑色のチェックマーク（✅）が点灯しているPRです。

本記事では、GitHub ActionsとAIを組み合わせた**「自ら直す」CI/CDパイプライン**を構築するための実践的なプロンプトと、無限コミットループ（通称：ゾンビエージェント）を防ぐための核心的なノウハウを公開します。

---

## ⚡️ 3行まとめ (TL;DR)

1. **自動化を超えた自律性:** エラーを通知するだけの従来のCIから、原因を特定し直接コードを修正するAIエージェント主導のワークフローへと進化しました。
2. **最小限の変更の原則 (Minimal Change):** AIがビジネスロジックを勝手に書き換える事故を防ぐため、型や構文エラーのみに限定する強力な制約が不可欠です。
3. **無限ループ対策は必須:** Botによるコミットが再びワークフローをトリガーしないよう設定し、テストを永遠に繰り返す「ゾンビエージェント」化を未然に防ぎます。

## 🚀 解決策: "Self-Healing AI Agent プロンプト"

CIパイプライン（Job 2）において、テスト（Job 1）失敗時に呼び出されるAIエージェントのコアプロンプトです。

### 🥉 Basic Version (基本型)

PrettierやESLintが指摘する単純な構文エラーを、素早く一括修正したい場合に使用します。

> **役割:** あなたは`[シニアフロントエンド開発者]`です。
> **タスク:** 提供されたエラーログを確認し、コードの`[Lintエラーおよびタイポ]`のみを迅速かつ正確に修正して、コード全体を返してください。

### 🥇 Pro Version (専門家型)

ビジネスロジックの完全性を担保しつつ、型エラーと構文エラーのみを厳格に修正する実務向けプロンプトです。

> **役割 (Role):** あなたは経験15年のシニアDevOpsエンジニア兼コードレビュアーです。コードの安定性を最優先し、不必要な変更を極度に警戒します。
>
> **状況 (Context):**
>
> - 背景: 現在、CIパイプラインでビルドまたはテストが失敗しています。
> - 目標: `[エラーログ]`と`[ソースコード]`を解析して失敗の原因を特定し、コードを修正します。
>
> **タスク (Task):**
>
> 1. エラーログに基づいて、ソースコードの問題点を正確に診断してください。
> 2. エラーを解決するための**最小限の変更（Minimal Change）**のみを実行してください。
> 3. コードのスタイル、コメント、コアなビジネスロジックは**絶対**に変更しないでください。
> 4. 修正が完了したコードをJSON形式で返してください。
>
> **制約事項 (Constraints):**
>
> - ⚠️ **ビジネスロジックの修正禁止:** 単純なタイポ、セミコロンの抜け、型の不一致など、明らかな機械的・構文的エラーのみが修正対象です。ロジックの変更が必要と思われる場合は、絶対にコードを修正せず、`"MANUAL_INTERVENTION_REQUIRED"`を出力して直ちに終了してください。
> - ⚠️ **ハルシネーション（幻覚）の防止:** 原因が正確にわからない場合や確信がない場合は、むやみに修正せず空のレスポンスを返してください。
>
> **出力形式 (Format):**
>
> {
>   "file_path": "[エラーが発生したファイルのパス]",
>   "fixed_content": "[修正された全体コードの内容]"
> }

## 💡 作成者のコメント (Insight)

このシステムを実際の業務に導入した際、最もその威力を発揮したのは**「大規模なTypeScript移行」**プロジェクトでした。数百のファイルから降り注ぐ細かな`any`型エラーやインターフェースの不一致を人間が一つ一つ修正するのは、まさに拷問のような苦行です。

しかし、このプロンプトを適用した自己修復ワークフローを導入したことで、単純な反復修正作業の80%が消滅しました。初期段階では、プロンプトで単に「コードを改善して」と曖昧に指示した結果、エージェントが変数名をすべて自分勝手に変更してしまい、PRレビューが不可能になるという痛い失敗も経験しました。AIにコードを任せる際、**「最小限の変更（Minimal Change）」**と**「ビジネスロジックの修正禁止」**という制約条件は、単なる選択肢ではなく必須の生存戦略です。

## 🙋 よくある質問 (FAQ)

- **Q: AIが無限にコードを修正してコミットする「無限ループ」に陥った場合はどうすればいいですか？**
  - A: 非常に重要なポイントです。これを防ぐためには、GitHub Actionsの設定ファイルに必ず `if: github.actor != 'github-actions[bot]'` という条件を追加し、Botのコミットが再びCIをトリガーしないようブロックする必要があります。また、1つのPRにつき最大3回までしか修復を試みないよう、リトライカウンターを設定することをおすすめします。

- **Q: AIはビジネスロジックの深刻なバグも修正できますか？**
  - A: 技術的には可能ですが、強く非推奨とします。ロジックのバグは企画の意図と密接に結びついていることが多く、AIが独自の判断で修正すると、さらに大きな障害を引き起こすリスクがあります。自己修復機能は、徹底して「明らかな構文・型エラー」に限定して適用するのが最も安全な運用方法です。

- **Q: トークンコストが高額になりませんか？**
  - A: 失敗したファイルのコードとエラーログのみを選別し（Context Windowの最小化）、APIに送信するようパイプラインのスクリプトを最適化する必要があります。また、予期せぬ請求を防ぐため、1日の予算上限（Budget Limit）の設定は必須です。

## 🧬 プロンプトの解剖 (Why it works?)

1. **シニアエンジニアのペルソナ (Role):** 「不必要な変更を極度に警戒する」という保守的な性格を付与することで、AI特有の「過剰修正（オーバーエンジニアリング）」を効果的に抑制しました。
2. **明確なエスケープハッチ (MANUAL_INTERVENTION_REQUIRED):** AIが自ら判断するのが難しいロジックエラーの場合、無理に正解を作り出さず、人間の開発者に制御を委ねるための明確なフェイルセーフを設けています。
3. **JSON出力フォーマットの規定:** CIパイプラインの次のステップ（修正されたコードの上書きおよび `git commit`）で自動パースしやすいよう、出力フォーマットを厳格なJSONに固定しました。

## 📊 証明: Before & After

### ❌ Before (入力)

```typescript
// src/utils/calculator.ts
export function addPrice(a: number, b: string) {
  return a + b;
}
```

```bash
# Error Log
TS2365: Operator '+' cannot be applied to types 'number' and 'string'.
```

### ✅ After (結果)

```json
{
  "file_path": "src/utils/calculator.ts",
  "fixed_content": "export function addPrice(a: number, b: number) {\n  return a + b;\n}"
}
```

_（AIが型の不一致エラーを正確に認識し、`b`の型を`number`に修正した後、CIパイプラインがこれをファイルに反映してコミットします。）_
