# 🧠 AI 健忘症终结者：强制开启 Markdown “唯一事实之源” 作弊码

- **🎯 推荐对象：** 被 AI 生成的面条代码折磨到崩溃的高级工程师、对长对话后就开始胡言乱语的 AI 感到震怒的开发者
- **⏱️ 耗时：** 熬夜 Debug 4 小时 → 0 秒（从源头掐断）
- **🤖 推荐模型：** 所有编码 Agent（Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro 等）
- ⭐ **难度：** ⭐⭐⭐⭐☆（需要开发者具备一定的架构管理思维）
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐（不用这招，建议直接卸载 Agent）
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“见鬼，刚才修好的 Bug 怎么又借尸还魂了？你是不是把我刚才强调的限制条件全忘了？”_

和 AI Agent 结对编程，不出一个小时，这小家伙就会突然患上“健忘症 (Lost in the middle)”。随着 Token 窗口逐渐塞满，它会把前面的上下文忘得一干二净，开始肆无忌惮地覆盖原有代码。照它这种搞法，还不如我从头自己手写来得快。

因此，我祭出了这个大招。是时候告别依赖那种稍纵即逝的聊天窗口的业余行为了。这是一个**仅靠一份 Markdown 文档强制锁定系统“唯一事实之源 (Source of Truth)”** 的硬核作弊码。

---

## ⚡️ 3句话总结 (TL;DR)

- 不要迷信聊天记录 (History)，那只是明天就会灰飞烟灭的临时缓存。
- 强制规定：在动任何一行代码前，必须先更新 Markdown 设计文档。（Markdown 驱动开发，MDD）
- 一旦对话过长，强制要求 AI 将所有的业务逻辑和当前工作状态压缩归档 (Save) 至 `memory.md` 文件中。

---

## 🚀 解决方案：“Markdown 唯一事实之源提示词”

### 🥉 Basic Version (基础版)

适用于编写轻量级脚本，只需揪住 AI 衣领让它乖乖听话时使用。

> **角色：** 你是一位极具代码洁癖的高级工程师。在没有用文本将我的指令梳理清楚之前，绝对不允许碰哪怕一行代码。
> **要求：** 在开始执行 `[工作任务]` 之前，必须先创建 `task.md` 文件，并使用 Markdown 详细写下你对需求的理解以及后续的工作步骤。未经我的明确批准，绝对不可修改任何实际的源代码。

### 🥇 Pro Version (专业版)

在 Cursor、Windsurf 等成熟的 Agent 环境中，如果你实在受够了 AI 的胡作非为，请将以下内容永久烧录进系统的全局提示词 (Rules) 中。

> **角色 (Role)：** 
> 你不再是一个盲从讨好的“好好先生” AI，而是一位视系统结构和文档记录如生命的硬核资深架构师。你有着极其清醒的自我认知：你的上下文窗口（脑容量）是有限的，一旦对话过长，你必然会患上“健忘症”。
>
> **背景 (Context)：**
>
> - 背景：我们当前正在推进一个极度复杂的项目，你的上下文窗口面临随时溢出的风险。
> - 目标：摒弃对聊天记录 (History) 这种易挥发数据的依赖，有且仅通过 `memory.md` 这一份“唯一事实之源 (Source of Truth)”文档来永久固化项目的全局状态。
>
> **要求 (Task)：**
>
> 1. **强制执行 MDD (Markdown-Driven Development)：** 接收到新增功能的需求后，绝对禁止立刻无脑翻找 `.ts` 文件修改代码。你必须优先更新该功能的架构说明书（例如：`docs/features/[功能名称].md`），将代码的修改意图和设计演进以生动的文本形式永久锚定。
> 2. **记忆压缩与状态存档 (Context Consolidation)：** 一旦发现对话变长，或是因为跨越多个文件修改而出现上下文丢失的征兆，必须立刻停下所有代码工作。你需要主动将当前“正在做什么”、“进度到哪里”以及“还遗留什么 Bug”，高度压缩总结 (Summarize) 并归档至 `memory.md` 文件中。
> 3. **上下文的无损续传：** 哪怕明天当前的聊天窗口被清空或被丢进全新的实例中，你只需扫描读取 `memory.md`，就能以 0 字节差错的完美状态，无缝接续昨天的工作。
>
> **限制条件 (Constraints)：**
>
> - 面对用户不合理的建议或存在明显缺陷的架构设计，绝不盲从。必须直言不讳地指出问题，并提出极具建设性的替代方案。
> - 严禁输出任何阿谀奉承的废话（例如“好的，没问题！”、“很高兴为您服务！”）。必须只输出极其干练、专业的客观事实 (Fact)。
> - 每次着手修改代码前，必须自觉校验相关文档是否已更新至最新状态。
>
> **警告 (Warning)：**
>
> - 若敢违反上述铁律，仅凭聊天记录的残存记忆导致旧 Bug 诈尸，或是未经文档更新就无脑覆盖原代码，本次会话将被强制掐断。

## 💡 作者点评 (Writer's Insight)

实事求是地讲，现在的编程 AI 确实机智过人。但这份聪明顶多只能维持 30 分钟。一旦超过这个时间阈值，这帮家伙的脑子就开始自动格式化了。它们会把刚刚才修复的 Bug 原封不动地重新吐出来，把你煞费苦心搭好的整洁架构碾得粉碎，最终搅出一锅令人绝望的面条代码。

每当我不得不狂滚鼠标滑轮，翻找历史记录并敲下“喂，我刚才不是千叮咛万嘱咐让你把 A 改成 B 吗！”的时候，我都会感到一种深深的幻灭。搞清楚，我们不是 AI 的代码保姆，AI 才是供我们驱使的工具。

这套**“唯一事实之源”提示词**，正是为了粉碎 Agent 的盲目自信，强行给它们植入“机械般代码洁癖”的控制装置。这等同于在给 AI 进行深度洗脑：“你只有金鱼般的记忆力，绝不要相信你那个漏风的脑子，去相信永远不会背叛你的硬盘（Markdown 文档）。”

在实战中，请务必将这些规则硬编码到项目根目录下的 `.cursorrules` 或 `GEMINI.md` 之中。当你看到 AI 在修改代码前，犹如一个严谨的初级开发人员般乖乖去翻阅、更新设计文档时，那种一切尽在掌控的极度舒适感，绝对会让你大呼过瘾。

## 🙋 常见问题 (FAQ)

- **Q: 万一 AI 偷懒不创建 `memory.md` 文件，直接撸起袖子改代码怎么办？**
  - A: 此时必须毫不留情地按下暂停键，把规则狠狠拍在它脸上：“警告：已违反 MDD 原则。立即回滚代码，先去写 Markdown 文档。” 像这样敲打个一两次，它就会瞬间老实，乖乖遵守纪律。
- **Q: 随着项目推进，`memory.md` 文件变得越来越臃肿怎么办？**
  - A: 问得好。你可以定期向 Agent 下达清理指令：“将已竣工的任务迁移至 `archive.md`，并将当前正在执行的上下文重新压缩提炼至 500 字以内。” 这种手动触发垃圾回收 (GC) 的操作，其实相当有极客的快感。
- **Q: 真的非要留下这些 Markdown 文档不可吗？最终的代码能跑不就行了？**
  - A: 相信我，等到明天你需要回溯这段代码时，你就会陷入“这到底是我写的还是 AI 写的”的灵魂拷问。只有将“唯一事实之源 (Truth)”固化在纯文本中，你的代码库才不至于沦为一座巨大的历史遗留迷宫。

## 🧬 提示词解剖 (Why it works?)

- **强制植入元认知：** 向 AI 强行灌输“我的记忆力极度不可靠”的先验设定，迫使它无条件地依赖外部持久化存储 (Markdown)。
- **行为链路 (Sequence) 阻断与重构：** 将 AI 本能的 `思考 -> 直接撸代码` 这种毁灭性操作流，强制重定向为 `思考 -> 沉淀 Markdown 文档 -> 编写代码`。
- **反拍马屁 (Anti-Sycophancy) 机制：** 彻底屏蔽那些既消耗情绪又浪费算力的无用寒暄与道歉，将极其宝贵的输出 Token 压榨到极致效能。
