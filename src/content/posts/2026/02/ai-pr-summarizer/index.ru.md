## 💡 Комментарий автора (Insight)

Этот промпт спасает не только при проверке чужого кода, но и когда вы пишете описание к собственному PR, пытаясь вспомнить: "А что я тут вообще нагородил?". Просто выполните команду `git diff main...HEAD | pbcopy` (на macOS) в терминале и "скормите" результат LLM.

Даже если в вашей компании нет дорогих Enterprise-инструментов вроде GitHub Copilot, этот метод дает потрясающее "предварительное ревью" (First-pass review). Senior-разработчик сэкономит часы на проверке PR от джуниоров, а для Junior-специалиста ИИ выступит строгим ментором, который найдет недочеты в логике еще до отправки кода на ревью.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если `git diff` слишком длинный, и ИИ выдает ошибку ограничения токенов?**
  - A: Современные модели (Claude 3.5 Sonnet, Gemini 2.5 Pro) обладают огромными контекстными окнами (Context Window) и легко "переваривают" десятки тысяч строк кода. Если лимит все же достигнут, запросите анализ только по списку измененных файлов через `git diff --stat` или выгрузите `diff` исключительно для критичных файлов с доменной логикой.

- **Q: Безопасно ли отправлять закрытый код компании в сторонние ИИ-сервисы?**
  - A: **Требуется крайняя осторожность.** Исходный код — главный актив бизнеса. При использовании публичных веб-версий ChatGPT или Claude обязательно активируйте настройку Data Opt-out (запрет на использование ваших данных для обучения). Самый безопасный путь — использовать корпоративные тарифы (Enterprise) с гарантией приватности или развернуть локальную LLM (например, Ollama) в закрытом контуре компании.

## 🧬 Анатомия промпта (Why it works?)

1.  **Задание роли (Role):** Детализированная персона — "Senior-инженер с 10-летним стажем" и "дотошный ревьюер, внимательный к безопасности" — заставляет ИИ отказаться от поверхностного саммари в пользу глубокого критического аудита кода.
2.  **Структурированный вывод (Task):** Жесткие рамки из 4 пунктов (сводка, изменения, риски, фокус для ревью) отсекают "воду" и не дают нейросети генерировать бесполезные пространные объяснения.
3.  **Constraints & Warning (Ограничения):** Прямая директива "не выдумывать" работает как жесткий предохранитель от галлюцинаций (Hallucination), гарантируя, что ревью опирается исключительно на факты из кода.

### ❌ До (Ввод: сырой git diff)

```diff
@@ -15,7 +15,11 @@ export class PaymentService {

-    if (order.status === 'PAID') return;
+    if (order.status === 'PAID') {
+      throw new Error('Already paid');
+    }
+    await this.pg.pay(amount);
+    await this.db.updateOrderStatus(orderId, 'PAID');
+    // TODO: Send email
   }
 }
```

### ✅ После (Результат: ИИ предварительное ревью)

```text
1. Краткая сводка (Executive Summary):
В логику обработки платежей (PaymentService) добавлена обработка исключений для предотвращения двойной оплаты и функция обновления статуса платежа.

2. Ключевые изменения (Key Changes):
- Изменена защитная логика: теперь вместо тихого возврата (return) для заказов со статусом 'PAID' явно выбрасывается ошибка (Error).
- Добавлена логика внешней интеграции запроса на оплату через платежный шлюз (pg.pay).
- Внедрено обновление статуса заказа в базе данных на 'PAID'.

3. Оценка рисков (Risk Assessment):
- Отсутствие обработки ошибок: Не видно обработки исключений (Try-Catch и т.д.) на случай сбоя `pg.pay(amount)`. Необходим механизм отката (rollback), если платеж не пройдет до обновления статуса.
- Проблема конкурентности (Race Condition): Если два запроса на оплату поступят почти одновременно, между `findOrder` и `updateOrderStatus` может возникнуть состояние гонки, что создает риск двойного списания. Требуется использование транзакций или блокировок (Lock).

4. Руководство для ревьюера (Reviewer Guide):
- `PaymentService.ts`: Уделите первоочередное внимание ревью обеспечения транзакционности и логики обработки ошибок.
```
