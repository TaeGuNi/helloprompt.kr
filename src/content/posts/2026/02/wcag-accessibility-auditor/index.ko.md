## ♿️ 내 코드는 모두에게 친절할까? 웹 접근성(WCAG) 자동 감사 봇

- **🎯 추천 대상:** 공공기관 및 대기업 프로젝트 참여 개발자, 포트폴리오의 완성도를 높이려는 취업 준비생, 시니어 프론트엔드 엔지니어
- **⏱️ 소요 시간:** 접근성 가이드라인 수동 검토 2시간 → AI 자동 진단 1분
- **🤖 최상위 성능:** Claude 3.5 Sonnet (코드 분석 능력 최상), GPT-4o

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐☆

> _"아무리 화려한 애니메이션을 구현했더라도, 누군가 로그인 버튼조차 누를 수 없다면 그 코드는 철저히 실패한 것입니다."_

혹시 어젯밤에도 <b>웹 접근성(Web Accessibility)</b> 이슈 때문에 퇴근을 미루고 코드를 뒤적이지 않으셨나요?

"QA 팀에서 스크린 리더로 읽었을 때 버튼 이름이 안 나온다고 반려했어요."
"모달 창을 열었는데 키보드 포커스가 뒤쪽 배경으로 넘어가 버려요."
"디자인팀이 준 색상 코드의 명도 대비가 WCAG 기준을 통과하지 못합니다."

우리는 화려한 애니메이션과 복잡한 상태 관리를 다루는 데는 익숙하지만, 정작 <b>모두를 위한 코드</b>를 작성하는 데는 큰 어려움을 겪습니다. 마감일은 코앞으로 다가왔고, 방대하고 복잡한 WCAG(Web Content Accessibility Guidelines) 2.2 가이드라인 문서를 처음부터 끝까지 정독할 시간은 없습니다. 결국 급한 대로 `<div>` 태그에 `onClick` 이벤트를 달고, 에러를 피하기 위해 의미 없는 `aria-label`과 `tabindex="0"`을 무분별하게 남발하는 이른바 <b>'WAI-ARIA 떡칠'</b>을 시도하게 됩니다.

하지만 이런 땜질식 처방은 상황을 더욱 악화시킵니다. 잘못 부여된 ARIA 속성은 스크린 리더 사용자에게 도움은커녕 심각한 정보 왜곡을 초래합니다. 키보드 탐색 환경에서는 초점(Focus)이 우주로 날아가 버려, 사용자는 사이트에 갇히거나 로그인 버튼조차 누를 수 없는 절망적인 경험을 하게 됩니다. 아무리 뛰어난 프론트엔드 아키텍처를 설계했더라도, <b>누군가에게는 진입조차 불가능한 장벽</b>이 되어버린다면 그 코드는 철저히 실패한 것입니다. 더군다나 최근에는 웹 접근성 준수가 단순한 '배려'를 넘어, 전 세계적으로 <b>강력한 법적 의무이자 서비스의 필수 스펙</b>으로 자리 잡으면서 접근성 결함은 곧바로 비즈니스의 치명적인 리스크로 직결되고 있습니다.

우리는 보통 화면에 보이는 요소에만 집착합니다. 하지만 보이지 않는 영역, 즉 <b>접근성 트리(Accessibility Tree)</b>가 어떻게 그려지는지 상상하며 코딩하는 것은 매우 고통스러운 작업입니다. 특히 드롭다운(Dropdown), 아코디언(Accordion), 탭(Tab) 메뉴와 같은 복잡한 대화형 컴포넌트를 만들 때면, 상태 변화에 따라 `aria-expanded`, `aria-hidden`, `aria-controls` 같은 속성을 자바스크립트로 일일이 토글해 주어야 합니다. 이 과정에서 단 하나의 속성만 누락되어도 스크린 리더 사용자는 현재 화면의 상태를 전혀 인지할 수 없게 됩니다. "내가 짠 코드가 과연 스크린 리더에서 어떻게 들릴까?"라는 막연한 두려움 속에서, 개발자는 끝없는 테스트와 수정의 늪에 빠지게 됩니다.

이제 이 지긋지긋한 <b>'접근성 디버깅 지옥'</b>에서 벗어날 때가 되었습니다.

여러분이 작성한 컴포넌트 코드를 완벽하게 이해하고, WCAG 2.2 기준에 맞춰 1분 만에 깐깐하게 점검해 주는 <b>AI 시니어 프론트엔드 엔지니어</b>를 고용해 보세요. 이 프롬프트는 단순한 코드 검사기가 아닙니다. 무의미한 `<div>` 덩어리를 의미 있는 HTML5 <span style="color:var(--color-cyber-cyan)">시멘틱(Semantic) 태그</span>로 전면 재구조화하고, 가장 까다롭다는 모달의 <b>Focus Trap(초점 가두기) 논리</b>를 완벽하게 구현하며, 복잡한 폼(Form) 요소의 레이블 매핑을 빈틈없이 연결해 줍니다. 

이제 이 프롬프트를 통해 막연한 두려움을 완벽한 확신으로 바꾸세요. 코드를 복사해서 붙여넣기만 하면, AI는 단순히 에러를 고쳐주는 것을 넘어 <b>직접 VoiceOver나 NVDA로 테스트해 볼 수 있는 구체적인 시나리오</b>까지 친절하게 제공합니다. 여러분은 그저 스크린 리더를 켜고, AI가 안내하는 대로 `Tab` 키와 `Space` 바를 누르며 완벽하게 개선된 접근성을 두 눈과 두 귀로 직접 확인하기만 하면 됩니다. 당신의 코드가 누군가의 삶을 얼마나 편리하게 만들어주는지, 그 경이로운 변화를 지금 바로 경험해 보세요. 단 1분의 투자로, 길고 지루했던 가이드라인 암기 없이 완벽하고 따뜻한 코드를 완성할 수 있습니다.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 엉성하게 `<div>`로만 빚어낸 최악의 버튼 코드입니다. 키보드 포커스가 잡히지 않고, 스크린 리더는 단순히 "전송"이라는 텍스트만 읽거나 아예 무시해 버립니다. 엔터 키를 눌러도 작동하지 않는, 오직 마우스 사용자만을 위한 이기적인 코드입니다.

```html
> <div class="submit-btn" onclick="sendData()">
>   <img src="send-icon.png" />
>   <span>전송</span>
> </div>
```

### ✅ After (완벽하게 변신한 결과)

```html
<button type="button" class="submit-btn" onClick="sendData()">
  <img src="send-icon.png" alt="" aria-hidden="true" />
  <span>전송</span>
</button>
```

키보드의 `Tab` 키로 정확하게 포커스를 맞출 수 있고, `Enter`나 `Space` 키로 이벤트를 발생시킬 수 있는 <b>완벽한 네이티브 마크업</b>으로 재탄생했습니다. 불필요한 이미지 텍스트는 `aria-hidden="true"`로 숨겨 스크린 리더의 정보 공해를 완벽하게 차단했습니다.

## ⚡️ 3줄 요약 (TL;DR)

1. **시멘틱 마크업 전환:** 무분별하게 남발된 `<div>` 태그를 의미 있는 HTML5 시멘틱 태그로 자동 변환해 줍니다.
2. **WCAG 2.2 완벽 준수:** 명도 대비, 키보드 초점(Focus) 이동 논리, WAI-ARIA 속성 오남용 등 까다로운 접근성 기준을 깐깐하게 검증합니다.
3. **실전 테스트 가이드 제공:** 스크린 리더(VoiceOver, NVDA)와 키보드(Tab 키)만으로 직접 코드를 검증해 볼 수 있는 구체적인 액션 플랜을 제시합니다.

>
> **코드 (Code):**
> 
> `[여기에 검수할 HTML 코드를 붙여넣으세요]`

### 🥇 Pro Version (전문가형)

복잡한 상태를 가진 모달(Modal), 드롭다운(Dropdown), 탭(Tab) 인터랙션 등 높은 접근성 구현 난이도를 요구하는 UI 컴포넌트를 검수할 때 활용하세요.

> **역할 (Role):** 너는 매우 엄격한 WCAG 2.2 인증 심사위원이자 10년 차 시니어 프론트엔드 엔지니어 구실을 할 거야.
>
> **상황 (Context):**
>
> - 대상: 나는 현재 `[검수할 UI 컴포넌트 이름, 예: 회원가입 폼 모달]` 컴포넌트 개발을 완료했어.
> - 목표: 마우스 조작이 불가능한 키보드 전용 사용자나 시각장애인(스크린 리더 의존 사용자)도 아무런 정보의 장벽 없이 해당 UI를 원활하게 조작하고 인지할 수 있어야 해.
>
> **요청 (Task):**
>
> 1. **WCAG 2.2 심층 진단:** 내가 제공한 코드에서 WCAG 2.2 Level AA 기준에 위배되는 모든 결함(명도 대비, 초점 이동 논리, 폼 레이블 누락, 대체 텍스트 부재 등)을 샅샅이 찾아내 줘.
> 2. **시멘틱 리팩토링:** 의미 없이 남발된 `<div>`나 `<span>` 태그를 걷어내고, 접근성 트리에 올바르게 매핑되는 HTML5 시멘틱 태그를 활용하여 코드를 전면 재작성해 줘.
> 3. **WAI-ARIA 최적화:** '가장 훌륭한 ARIA는 ARIA를 사용하지 않는 것'이라는 대원칙(First Rule of ARIA)을 엄수해 줘. 기본 HTML 태그만으로 해결이 불가능한 필수적인 상황에서만 제한적으로 `aria-*` 속성과 `role`을 부여해.
> 4. **초점(Focus) 트랩 로직:** 검수 대상이 모달이나 팝업인 경우, 활성화된 상태에서 내부 키보드 초점이 바깥으로 빠져나가지 않도록(Focus Trap) 논리적인 초점 이동 흐름을 설계해 줘.
> 5. **실전 검증 가이드:** 내가 직접 키보드(Tab/Shift+Tab, Space/Enter)와 스크린 리더를 켜고 이 코드를 테스트해 볼 수 있는 구체적인 행동 시나리오를 딱 3줄로 요약해서 알려줘.
>
> **제약사항 (Constraints):**
>
> - 화면상에서는 시각적으로 숨겨져야 하지만, 스크린 리더는 반드시 읽어주어야 하는 정보(`sr-only` 클래스 등 활용)가 누락된 지점이 없는지 꼼꼼히 체크하고 코드에 반영해 줘.
>
> **코드 (Code):**
>
> `[여기에 검수받을 컴포넌트 코드를 붙여넣으세요]`

## 💡 작성자 코멘트 (Insight & How to use)

웹 접근성(Accessibility) 개선을 시작한 주니어 개발자들이 가장 흔히 저지르는 치명적인 실수가 바로 무분별한 <b>'WAI-ARIA 떡칠'</b>입니다. "무조건 접근성을 챙겨야 한다"는 압박감에 시달린 나머지, 화면에 보이는 모든 `<div>`와 `<span>` 태그에 `role="button"`이나 `aria-label`을 억지로 쑤셔 넣는 우를 범하곤 합니다. 하지만 개발자의 의도와 다르게 잘못 부여된 ARIA 속성은 스크린 리더 사용자에게 도움은커녕, 현재 어떤 요소를 탐색하고 있는지 전혀 알 수 없게 만드는 <b>최악의 정보 왜곡과 혼란</b>을 초래합니다.

이 프롬프트가 압도적으로 강력한 이유는 바로 AI에게 <b>"First Rule of ARIA(가능한 기본 HTML 태그를 최우선으로 사용할 것)"</b>라는 대원칙을 <b>가장 강력한 제약 조건(Constraints)</b>으로 걸어두었기 때문입니다. AI는 코드를 분석할 때 무작정 ARIA 속성을 추가하는 것이 아니라, 먼저 해당 요소를 `<button>`, `<nav>`, `<dialog>`와 같은 기본 네이티브 HTML5 시멘틱 태그로 교체할 수 있는지 철저하게 따져봅니다. 네이티브 태그를 사용하면 브라우저가 알아서 키보드 이벤트와 스크린 리더 포커싱을 기본적으로 처리해 주기 때문에, 자바스크립트 코드는 훨씬 가벼워지고 접근성 버그 발생 확률은 제로에 가깝게 수렴합니다.

실무에서 상태 관리가 복잡한 커스텀 셀렉트 박스(Select Box)나 모달(Modal) 컴포넌트를 구현한 뒤, 이 <b>A11y 오디터 프롬프트</b>를 통과시켜 보세요. 프롬프트 내의 `[검수할 UI 컴포넌트 이름]` 변수에 <b>'결제 정보 입력 폼'</b>이나 <b>'무한 스크롤이 적용된 상품 리스트'</b>처럼 구체적인 컨텍스트를 입력하는 것이 핵심입니다. 상황을 구체적으로 명시할수록, AI는 해당 UI 패턴에 특화된 접근성 가이드라인을 끌어와 훨씬 예리하고 정확한 코드 리뷰를 제공합니다. 개발자 본인도 미처 인지하지 못했던 미세한 키보드 포커스 유실 버그나 폼 레이블 누락 요소를 기가 막히게 잡아냅니다.

또한, 제약사항 섹션에 명시된 <b>"초점(Focus) 트랩 로직"</b> 지시어는 이 프롬프트의 백미라고 할 수 있습니다. 모달 창이 열렸을 때 사용자가 `Tab` 키를 누르면, 초점이 모달 밖의 흐려진 배경(Dim) 요소나 본문으로 빠져나가지 않고 모달 내부에서만 논리적으로 순환해야 합니다. 이는 키보드 의존 사용자에게 매우 중요한 경험이지만, 막상 자바스크립트로 직접 구현하려면 골치가 아픈 작업입니다. AI는 이 프롬프트를 통해 완벽하게 작동하는 포커스 트랩 방어 로직까지 함께 설계해 주어, 개발자의 수고를 극적으로 덜어줍니다.

특히 <b>Claude 3.5 Sonnet</b> 모델에 이 프롬프트를 입력해 볼 것을 강력히 권장합니다. 마치 옆자리에 산전수전 다 겪은 10년 차 깐깐한 프론트엔드 접근성 전문가가 앉아, 내 코드를 한 줄 한 줄 매의 눈으로 리뷰해 주는 듯한 <b>압도적인 퀄리티의 피드백</b>을 경험할 수 있습니다. 피드백 하단에 제시되는 <b>'실전 검증 가이드'</b>를 따라, 직접 Mac의 VoiceOver나 Windows의 NVDA를 켜고 `Tab` 키를 눌러보세요. 시각장애인 사용자가 여러분의 웹사이트를 어떻게 '듣고' 경험하는지 깨닫게 되는 순간, 여러분의 프론트엔드 개발 시야는 한 차원 더 높은 곳으로 도약할 것입니다. 

프롬프트를 실무에 적용할 때 한 가지 팁을 더 드리자면, 코드 베이스가 너무 큰 경우 페이지 단위로 통째로 검수를 맡기기보다는 <b>독립적인 개별 컴포넌트 단위(예: 하나의 네비게이션 바, 단일 폼 요소)로 쪼개서 요청</b>하는 것이 훨씬 유리합니다. AI 모델이 하나의 컴포넌트에 온전히 집중할 때, 명도 대비나 복잡한 `aria-describedby` 연결 관계 등 미세한 결함을 놓치지 않고 100% 완벽하게 잡아낼 수 있습니다. 이 프롬프트는 단순한 에러 수정기가 아니라, 여러분의 코드를 '모두를 위한 따뜻한 인터페이스'로 승격시켜 주는 가장 확실하고 빠른 마법의 지름길이 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코드 없이 화면 디자인(UI) 캡처 이미지로도 접근성 감사를 받을 수 있나요?**
  - A: 네, 충분히 가능합니다. 멀티모달 기능이 뛰어난 GPT-4o나 Claude 3.5 Sonnet에 UI 컴포넌트 캡처 이미지를 첨부한 뒤, "이 디자인의 명도 대비(Color Contrast)가 WCAG Level AA 기준을 충족하는지 검증해 줘"라고 추가 요청을 해보세요. 눈으로는 쉽게 구분하기 힘든 미세한 디자인 결함까지 정확하게 짚어냅니다.

- **Q: 저희 팀은 Tailwind CSS를 사용 중인데, 수정되는 코드에 클래스명도 알아서 반영해 주나요?**
  - A: 프롬프트의 상황(Context) 영역에 "우리 프로젝트는 Tailwind CSS를 사용 중이니, 스크린 리더 전용 텍스트를 숨길 때 반드시 `sr-only` 유틸리티 클래스를 적용해 줘"라고 한 줄만 명시해 보세요. 현재 사용 중인 기술 스택에 완벽하게 들어맞는 맞춤형 코드를 반환해 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **명확하고 엄격한 심사 기준 주입:** AI에게 단순히 "접근성을 좋게 고쳐줘"라고 모호하게 지시하는 대신, "WCAG 2.2 Level AA 인증 심사원"이라는 구체적인 국제 표준 규격과 페르소나를 부여함으로써 AI 특유의 할루시네이션(거짓 정보 생성)을 완벽하게 통제합니다.
2. **First Rule of ARIA 강제화:** AI가 접근성을 개선한답시고 무분별하게 ARIA 속성을 남발해 버리는 부작용을 원천 차단합니다. 대신 HTML5의 기본 스펙을 100% 활용한, 가장 근본적이고 건강한 시멘틱 마크업 구조를 유도해 냅니다.
3. **액셔너블(Actionable)한 테스트 시나리오 도출:** 단순히 수정된 정답 코드를 던져주는 것에서 끝나지 않습니다. 프롬프트 마지막에 개발자가 직접 키보드를 두드려보며 접근성 개선 효과를 체감하고 검증할 수 있는 행동 기반의 테스트 시나리오를 요구함으로써 실무 효용성을 극대화했습니다.
