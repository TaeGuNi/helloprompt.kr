>
> **Контекст (Context):**
>
> - Ситуация: Мне нужно развернуть веб-приложение в Docker-контейнере.

> **Запрос (Task):**

> 2. Создай подробный `docker-compose.yml`, связывающий приложение, базы данных и любые смежные сервисы.

> **Стек технологий (Stack):**

## 💡 Комментарий автора (Insight)

Написание грамотного `Dockerfile` с нуля часто превращается в изматывающий цикл проб и ошибок. Особенно когда дело доходит до многоэтапной сборки (Multi-stage build) — критического фактора для стабильного и легковесного production-окружения. **Этот промпт освобождает вас от зубрёжки докеровского синтаксиса.** Он заставляет нейросеть сходу применять индустриальные DevOps-практики: внедрять **базовую безопасность** (запуск от non-root пользователя) и **глубокую оптимизацию** (использование `alpine` или `slim` образов). Я сам постоянно использую этот шаблон при разворачивании новых микросервисов — он стабильно экономит минимум 40 минут рутины и выдает конфигурацию, которую с гордостью можно отдавать на самое дотошное код-ревью.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Подойдет ли этот промпт для сложной микросервисной архитектуры?**
  - A: Абсолютно. Нейросети превосходно справляются с оркестрацией `docker-compose.yml` для сложных экосистем. Достаточно перечислить все компоненты в блоке «Стек технологий» (например: _«Backend на Python, Frontend на React, брокер RabbitMQ и БД Postgres»_), и ИИ элегантно свяжет их в единую изолированную сеть.

- **Q: Что делать, если вес итогового образа всё равно слишком большой?**
  - A: Заставьте ИИ провести аудит сгенерированного `Dockerfile` с помощью уточняющего запроса: _«Как экстремально сжать размер этого образа? Дай пошаговый план»_. Модель предложит очистку кэша пакетных менеджеров (через `apt-get clean` или `pip cache purge`), удаление временных файлов сборки или миграцию на сверхлегкие базовые образы.

## 🧬 Анатомия промпта (Why it works?)

1. **Задание роли (Role):** Персона «Senior DevOps-инженер» принуждает нейросеть использовать продвинутые архитектурные паттерны (Multi-stage build, строгий контроль прав), напрочь отсекая примитивный и дырявый код из туториалов для новичков.

3. **Контекст (Context):** Четкое осознание конечной цели («унифицировать среды») дает модели установку генерировать максимально отказоустойчивую, предсказуемую и готовую к горизонтальному масштабированию конфигурацию.

## 📊 Доказательство: Before & After

### ❌ Before (Обычный запрос)

```text
Напиши dockerfile для python django и postgres
```

_(Типичный итог: неповоротливый образ на базе тяжелой Ubuntu весом под гигабайт, запуск процессов с `root`-правами и полное игнорирование кэширования слоев)._

### ✅ After (Результат Pro-промпта)

```dockerfile
# 1. Этап сборки (Build Stage)
FROM python:3.11-slim as builder
WORKDIR /app
COPY requirements.txt .
# Устанавливаем зависимости без кэша для экономии места
RUN pip install --user --no-cache-dir -r requirements.txt

# 2. Боевой этап (Production Stage)
FROM python:3.11-slim
WORKDIR /app

# Копируем только установленные пакеты, игнорируя исходники сборщика
COPY --from=builder /root/.local /root/.local
COPY . .

# Настройка путей для выполнения бинарников
ENV PATH=/root/.local/bin:$PATH

# Безопасность: создание и переход на пользователя без root-прав
RUN useradd -m appuser
USER appuser

# Запуск Production-сервера Gunicorn
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

_Разница налицо: нейросеть выбрала легковесный образ `slim`, элегантно разбила процесс на два независимых этапа для жесткой экономии дискового пространства и превентивно обеспечила безопасность, создав изолированного пользователя `appuser`._
