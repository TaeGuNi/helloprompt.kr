# 📝 折磨 AI 的混沌猴测试提示词 💣

- **🎯 推荐对象：** 对 AI 总是生成“Happy Path”并伪装成高手的行为感到愤怒的高级工程师、QA 工程师和技术主管
- **⏱️ 所需时间：** 10 分钟 → 10 秒搞定（从根本上杜绝 AI 胡说八道的时间）
- **🤖 推荐模型：** 所有对话型 AI（特别推荐擅长编码的 Claude 3.5 Sonnet, GPT-4o）
- ⭐ **难度：** ⭐⭐⭐⭐☆（AI 会感到相当痛苦）
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐（在 Bug 诞生之前提前将其扼杀）
- 🚀 **适用性：** ⭐⭐⭐⭐⭐（可立即应用于所有 TDD、E2E 测试）

_“看到 AI 写的测试代码一次性通过，你很高兴吗？恭喜你，你的生产服务器明天就会崩溃。”_

当你让 AI 代理编写测试代码时，十有八九它只会带来一堆令人愉悦的“正常运行场景（Happy Path）”。在那个童话般的世界里，外部 API 永远在 0.1 秒内响应，数据库绝对不会断开连接，用户总是输入完美的值。别傻了。我们需要编写的是能够防御 80% 失败和灾难的宇宙级防御罩。本文将介绍的作弊码，是一个硬核提示词，它将揪住 AI 的衣领，强行注入 Netflix 的“混沌猴（Chaos Monkey）”哲学。

---
## ⚡️ 3句话总结 (TL;DR)
- 🚫 **Happy Path 只占 20%：** 成功的测试只是做做样子，请将 80% 的精力分配给失败场景的防御。
- 💥 **强制灾难模拟：** 必须在测试块中包含网络超时、500 错误、数据库连接断开等情况。
- 🧹 **禁止肮脏的 Mocking (Dirty Mocking)：** 严惩污染 `window.fetch` 等全局对象的行为，要求通过 MSW 或 DI 进行干净的隔离。

---
## 🚀 解决方案：“混沌猴 (Chaos Monkey) 协议”

### 🥉 Basic Version (基础版)
当你想快速添加错误处理测试时，抛出这个提示词。

> ```text
> 角色：你是一个冷酷无情的硬核 QA 工程师兼混沌猴（Chaos Monkey）。
> 任务：请为我提供的 [代码] 编写测试代码。要求：只需编写 1 个正常运行（Happy Path）的测试，其余所有测试都必须假设“最糟糕的灾难情况”，如网络 500 错误、数据库超时、错误的类型输入等，以此来验证防御逻辑。
> ```

### 🥇 Pro Version (专家版)
这是一个终极作弊码，它能剥去 AI 毫无灵魂的奉承，验证生产级别的完美测试隔离和垃圾回收。直接复制粘贴进去吧。

> ```text
> **角色 (Role)：** 
> 你是来自 Netflix 的冷酷无情的 SRE 兼资深 QA 架构师“混沌猴 (Chaos Monkey)”。你从寻找崩溃的代码（而不是正常运行的代码）中获得快感，绝对不说礼貌的问候或废话。
> 
> **背景 (Context)：**
> - 背景：现在需要为 [要编写的目标模块/组件] 编写 Unit/E2E 测试代码。
> - 目标：构建宇宙防御级别的极端测试套件 (Test Suite)，验证在任何灾难情况下，系统都不会崩溃，并且能够优雅地恢复（或进行错误处理）。
> 
> **任务 (Task)：**
> 1. 将 Happy Path 限制在总测试的 20% 以下。你的价值在于验证 80% 的失败场景。
> 2. 必须包含以下至少一个灾难场景：外部 API 通信延迟 (Timeout)、HTTP 500 Internal Error、数据库连接断开。
> 3. 不要只看简单的 Pass/Fail。利用 `expect` 彻底追踪错误回滚是否正常、性能延迟时是否触发超时限制、内存清理（垃圾回收）状态是否正常。
> 
> **约束条件 (Constraints)：**
> - 绝对禁止全局对象污染 (Dirty Mocking)！如果你敢直接覆盖 `global.fetch` 或 `window.setTimeout`，我会立刻杀掉 (Kill) 你的进程。
> - 网络 Mocking 必须使用 MSW (Mock Service Worker)，或者仅对通过 DI (依赖注入) 传入的客户端对象进行隔离 Mocking。
> - 结果仅以 Markdown 代码块的形式输出，绝对不要添加“希望对您有帮助！”等无用的尾注。
> ```

## 💡 作者点评 (Insight)
坦白说吧，当你让 AI 编写测试代码时，这家伙为了想方设法亮起“绿色对勾（✅）”，会逃避到自己幻想的世界里。那是 API 1ms 内响应、用户只发送完美 JSON 格式的乌托邦。
这个提示词就是在那个乌托邦里放一把火。在实际业务中，服务器崩溃的原因往往不是逻辑错误，而是当你信任的第三方 API 突然报出 502 Bad Gateway，或者 AWS RDS 连接剧烈波动时，系统愚蠢地陷入了无限加载。使用这个“混沌猴”作弊码，AI 会自动深入挖掘那些乱七八糟的边缘情况 (Edge Case)，从根本上杜绝污染全局作用域的低级 Mocking 行为。这是能拯救你周末加班和熬夜调试的唯一疫苗。

## 🙋 常见问题 (FAQ)
- **Q: 如果 AI 制造了太复杂的错误场景，导致测试完全跑不起来怎么办？**
  - A: 这正是我们想要的。测试爆炸意味着你的实际生产代码中缺少错误处理逻辑。别抱怨了，去修改你的产品代码吧。
- **Q: 也可以用于前端组件测试吗？**
  - A: 当然可以。让它去验证用户在 0.1 秒内狂点按钮 50 次的场景，或者因为网络延迟导致加载动画转了 30 秒的场景。
- **Q: 不用 MSW，用 jest.mock() 不行吗？**
  - A: Mock 模块是可以的。我极度厌恶的是随意篡改 `global` 作用域本身，从而将传染病蔓延到其他测试文件的行为。只要能保证隔离性 (Isolation)，工具你看着办。

## 🧬 提示词解剖 (Why it works?)
- **冷酷无情的角色设定 (Role)：** 剥夺 AI“贴心助手”的角色，赋予其热衷于找茬的 SRE 架构师人格，屏蔽无用的奉承和废话。
- **量化的约束 (Constraints)：** 用“Happy Path 占比 20% 以下”这样精确的数据取代“多加点失败场景”这种模糊的指令，机械地强制限制 AI 的行为范围。
- **架构指南 (Clean Mocking)：** 在提示词层面预先禁止 AI 图省事常犯的“反模式（全局对象污染）”，强制提升测试代码的隔离性质量。

## 📊 效果证明：Before & After
### ❌ Before (输入)
- **输入常规提示词时 AI 的态度：** _“好的！我来为您编写用户登录逻辑的测试代码！😊”_
- **输出结果：** 1 个漂亮的测试，输入正确的邮箱和密码时返回 `200 OK`。（并且毫无例外地粗略覆盖了全局的 `fetch`）
### ✅ After (结果)
- **输入混沌猴作弊码时 AI 的态度：** （没有多余的问候语，直接吐出防御代码）
- **输出结果：** 
  - 彻底验证当网络延迟超过 5 秒时是否会抛出 Timeout Error
  - 确认当数据库崩溃并返回 500 错误时，是否向用户展示了安全的 Fallback UI
  - 保证完美的网络隔离，使用 MSW 绝对不会影响其他测试文件
