> _「私のローカル環境では動いたんですが... 本番サーバーだとバージョンが違ってエラーが出ます😇」_

開発環境と本番環境のズレは、すべての開発者を悩ませる永遠の課題です。Dockerを使えば解決できると分かっていても、ゼロから`Dockerfile`を書き上げ、イメージサイズを最適化し、複数のコンテナを`docker-compose.yml`で連携させる作業は想像以上に骨が折れるものです。
しかし、もう悩む必要はありません。使用する技術スタックを伝えるだけで、本番環境レベルに最適化されたDocker設定ファイルをAIが一瞬で構築してくれる魔法のプロンプトをご紹介します。

1. **技術スタックへの完全最適化**: 言語やフレームワークのベストプラクティスに沿った`Dockerfile`を自動生成。

### 🥇 Pro Version (専門家形)

本番環境への安全なデプロイを見据えた、極めて軽量かつセキュアなコンテナを構築したい場合に最適なプロンプトです。

> **役割 (Role):** あなたはコンテナオーケストレーションの専門家であり、セキュリティとパフォーマンス最適化に精通したシニアDevOpsエンジニアです。
>
> **状況 (Context):**
>
> - 背景: 自身のアプリケーションをDockerコンテナとしてパッケージングし、デプロイする予定です。ローカル開発環境と本番環境の差異を完全に排除することを目的としています。
> - 目標: 高度に最適化され、実運用に耐えうる堅牢なDocker設定ファイルを生成すること。

> **タスク (Task):**
>
> 1. 指定された技術スタックに最適化された **Dockerfile** を作成してください。

> 3. イメージサイズを削減し、不要なファイルを除外するための **.dockerignore** の内容も併せて提示してください。
>
> **変数 (Variables):**
>
> - 技術スタック: `[使用する言語とバージョン、フレームワーク、必要なDBなどを入力。例：Python 3.11, FastAPI, PostgreSQL, Redis]`

> - **セキュリティ:** root権限での実行を避け、必ず一般ユーザー（non-root user）としてアプリケーションが稼働するように設定してください。
> - **キャッシュ最適化:** パッケージのインストールとソースコードのコピー工程を分離し、Dockerのレイヤーキャッシュを最大限に活用できる順序で記述してください。

## 📊 証明: Before & After

### ❌ Before (雑なプロンプト)

```text
Python 3.9, Django, PostgreSQLのDockerファイルを作って。
```

結果：ビルドツールから不要なキャッシュまで全てが一つの巨大なコンテナに詰め込まれ、イメージサイズが平気で1GBを超えるような、全く最適化されていない無骨なファイルが生成されてしまいます。

### ✅ After (本プロンプトを使用)

```dockerfile
# 1. Builder Stage
FROM python:3.9-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 2. Production Stage
FROM python:3.9-slim
WORKDIR /app

# セキュリティのためのnon-rootユーザー作成
RUN useradd -m myuser
USER myuser

# ビルドステージから必要なパッケージのみをコピー
COPY --from=builder /root/.local /home/myuser/.local
COPY . .

# 環境変数の設定
ENV PATH=/home/myuser/.local/bin:$PATH

CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

結果：不要なビルドツールが排除されるためイメージサイズが劇的に削減され、さらに最小権限（一般ユーザー）で安全に動作する、実運用レベルのセキュアなコンテナ設定が完成しました。
