# 📝 스파게티 코드는 이제 그만! 내 프로젝트에 딱 맞는 아키텍처 추천

- **🎯 추천 대상:** 주니어 개발자, 사이드 프로젝트 진행자, 1~3년 차 풀스택 개발자
- **⏱️ 소요 시간:** 아키텍처 고민 3일 → 5분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet (아키텍처 설계 및 코드 구조화에 압도적으로 유리), GPT-4o

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐☆

> _"처음엔 간단한 토이 프로젝트였는데... 기능 몇 개 추가하니까 코드가 뒤죽박죽 섞여서 어디서부터 손대야 할지 막막한가요?"_

모든 거대한 레거시 시스템의 시작도 작은 파일 하나였습니다. 처음부터 완벽한 구조를 잡을 필요는 없지만, 기준 없는 개발은 금방 '기술 부채'의 늪으로 우리를 끌어내립니다. 내 프로젝트의 성격, 팀의 규모, 향후 확장성까지 고려한 맞춤형 설계 도면을 10년 차 수석 소프트웨어 아키텍트 AI에게 의뢰해 보세요.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **상황 맞춤형 추천:** 무조건 Clean Architecture가 정답은 아닙니다. 내 프로젝트 규모에 딱 맞는 패턴을 찾아줍니다.
2. **명확한 디렉토리 구조:** `src/` 폴더 하위를 어떻게 나눠야 할지 한눈에 파악할 수 있는 트리(Tree) 구조를 그려줍니다.
3. **트레이드오프 분석:** 해당 아키텍처를 도입했을 때의 장점과 치명적인 단점(Over-engineering 등)을 함께 경고합니다.

## 🚀 해결책: "수석 아키텍트 모드"

### 🥉 Basic Version (기본형)

빠르게 대략적인 구조만 잡고 싶을 때 사용하세요.

> **역할:** 너는 10년 차 수석 소프트웨어 아키텍트야.
> **요청:** 내가 만들려는 앱은 `[React 기반의 간단한 투두리스트]`야. 이 프로젝트에 가장 적합한 폴더 구조와 아키텍처 패턴을 하나 추천하고 이유를 설명해 줘.

<br>

### 🥇 Pro Version (전문가형)

프로젝트 초기 세팅 단계에서 팀원들과 구조를 논의하기 위한 완벽한 초안이 필요할 때 사용하세요.

> **역할 (Role):** 너는 글로벌 IT 기업에서 대규모 트래픽 시스템 설계를 담당하는 수석 소프트웨어 아키텍트야.
>
> **상황 (Context):**
>
> - 배경: 새로운 프로젝트를 바닥부터 세팅해야 해. 초기엔 작게 시작하지만 추후 기능이 크게 확장될 예정이야.
> - 목표: 유지보수가 용이하고, 새로운 개발자가 합류했을 때 코드를 파악하기 쉬운 명확한 기준(아키텍처)을 수립하는 것.
>
> **요청 (Task):**
>
> 1. 아래 `[프로젝트 세부 정보]`를 분석하여 **가장 적합한 아키텍처 패턴 2가지(1순위, 2순위)**를 추천해 줘.
> 2. 1순위로 추천한 아키텍처를 기반으로, 실제 프로젝트에 바로 적용할 수 있는 **폴더 구조(Directory Structure)를 트리 형태의 코드 블록**으로 작성해 줘.
> 3. 트리 구조 내의 핵심 폴더와 계층(Layer)이 정확히 어떤 역할을 하는지 1~2줄로 명확히 설명해 줘.
>
> **프로젝트 세부 정보 (Variables):**
>
> - 서비스 형태: `[사용자 간 중고 거래 모바일 앱]`
> - 기술 스택: `[React Native, Node.js (Express), PostgreSQL]`
> - 주요 기능: `[실시간 채팅, 위치 기반 검색, 결제 시스템 연동]`
> - 팀 규모: `[프론트엔드 1명, 백엔드 1명]`
>
> **제약사항 (Constraints):**
>
> - 절대 오버 엔지니어링(Over-engineering)을 하지 마. 2명으로 구성된 팀이 감당할 수 있는 현실적인 규모로 제안해.
> - 왜 이 아키텍처를 1순위로 선택했는지, 그리고 도입 시 예상되는 가장 큰 단점(Trade-off)은 무엇인지 반드시 논리적인 근거를 포함해.
> - 마크다운 문법을 사용하여 가독성 좋게 출력해.

## 💡 작성자 코멘트 (Insight)

이 프롬프트의 핵심은 **"제약사항(Constraints)"**에 있습니다. AI에게 단순히 "아키텍처 추천해 줘"라고 하면, 십중팔구 MSA(Microservices Architecture)나 복잡한 헥사고날(Hexagonal) 아키텍처 등 주니어 1~2명이 감당할 수 없는 거창한 구조를 던져줍니다.

따라서 반드시 현재의 **팀 규모**와 **오버 엔지니어링 방지**를 명시해야 합니다. 저는 실무에서 새로운 토이 프로젝트나 사내 어드민 페이지를 만들 때 이 프롬프트를 가장 먼저 실행합니다. AI가 그려준 폴더 트리를 그대로 복사해서 `mkdir` 명령어로 세팅하면, 초기 구조 잡는 데 걸리는 시간을 반나절 이상 아낄 수 있습니다. 만약 프론트엔드 중심이라면 "상태 관리(State Management) 전략도 포함해 줘"라고 한 줄 추가해 보세요. 훨씬 실용적인 답변이 나옵니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 추천한 구조를 무조건 따라야 하나요?**
  - A: 절대 아닙니다. AI의 답변은 '훌륭한 초안(Draft)'일 뿐입니다. 추천받은 구조 중 내 팀의 역량이나 기존 사내 컨벤션과 맞지 않는 부분은 과감하게 쳐내고 수정해서 사용하는 것이 올바른 방법입니다.

- **Q: 백엔드나 모바일 말고, 프론트엔드(React/Vue) 전용 구조도 잘 짜주나요?**
  - A: 네, 아주 잘 짭니다. `[프로젝트 세부 정보]`에 "Next.js 14 App Router 사용", "FSD(Feature-Sliced Design) 아키텍처 적용 여부 검토해 줘" 등 프론트엔드 특화 키워드를 넣으면, 컴포넌트 단위의 세밀한 구조까지 완벽하게 분리해 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **상황과 제약의 완벽한 조화:** `[팀 규모]`라는 현실적인 변수와 `오버 엔지니어링 금지`라는 강력한 제약을 걸어, AI가 비현실적인 '이론상 완벽한 설계'를 내놓는 환각(Hallucination)을 원천 차단했습니다.
2. **트리 구조(Tree Structure) 강제:** 단순히 말로 설명하는 것을 넘어, 시각적인 폴더 트리를 그리도록 포맷(Format)을 명확히 지정하여 즉시 프로젝트에 복사/붙여넣기 할 수 있게 만들었습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

```text
나 쇼핑몰 앱 만들 건데, 폴더 구조 어떻게 짜는 게 좋아?
```

### ✅ After (결과)

**(실제 Claude 3.5 Sonnet 결과 요약)**

1순위 추천으로 **FSD (Feature-Sliced Design) 변형 구조**를 제안하며, 2인 개발 팀에 맞춘 현실적인 폴더 트리를 생성해 줍니다.

```text
src/
├── app/                  # 앱 진입점 및 전역 설정 (라우팅, 전역 스타일)
├── processes/            # 다중 페이지에 걸친 복잡한 비즈니스 로직 (예: 결제 프로세스)
├── pages/                # 라우팅 라우트 기반 페이지 컴포넌트
├── features/             # 재사용 가능한 주요 기능 덩어리 (예: Chatting, ProductList)
├── entities/             # 비즈니스 도메인 데이터 (User, Product 타입 및 API 훅)
└── shared/               # 앱 전체에서 쓰이는 공통 UI, 유틸 함수 (Button, formatDate)
```

_AI의 조언: "2인 팀이므로 `processes/` 계층은 초기에는 생략하고 단순화하는 것을 권장합니다. Trade-off로 초반 학습 곡선이 있지만, 컴포넌트 간 결합도를 낮춰 추후 채팅 기능이 고도화될 때 유지보수가 압도적으로 편해집니다."_
