## 📝 AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣

- **🎯 추천 대상:** 빈약한 'Happy Path' 테스트 코드에 지친 시니어 개발자, QA 엔지니어, 테크 리드
- **⏱️ 소요 시간:** 10분 → 10초 단축
- **🤖 최상위 성능:** 코드 작성에 특화된 Claude 3.5 Sonnet, GPT-4o 강력 추천

- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI가 뱉어낸 테스트 코드가 단번에 통과했다고 안도하셨습니까? 유감스럽게도, 당신의 프로덕션 서버는 내일 당장 붕괴될 수 있습니다."_

![AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣](/images/hooks/chaos-monkey-testing-prompt.jpg)

AI 에이전트에게 테스트 코드 작성을 맡겨본 적이 있으신가요? 십중팔구 AI는 한 치의 오차도 없는 유토피아적인 **'정상 작동 시나리오(Happy Path)'**만 번지르르하게 쏟아냅니다. 외부 API는 언제나 0.1초 만에 응답하고, 데이터베이스 커넥션 풀은 절대 마르지 않으며, 사용자는 단 하나의 오타도 없이 완벽한 데이터를 전송한다는 비현실적인 가정 위에서 말입니다. 터미널을 수놓는 영롱한 초록색 'PASS' 마크에 안도하셨겠지만, 이는 재앙을 알리는 전주곡에 불과합니다. 우리는 종종 코드 커버리지(Code Coverage) 100%라는 숫자의 함정에 빠져, 진짜 검증해야 할 '실패'를 외면하곤 합니다.

냉혹한 **프로덕션(Production)**의 세계는 다릅니다. 철석같이 믿었던 결제 게이트웨이는 트래픽이 몰리는 금요일 밤 돌연 <span style="color:var(--color-cyber-cyan)">502 Bad Gateway</span> 에러를 내뿜고, 레디스(Redis) 인스턴스는 원인 모를 이유로 다운되며, 클라이언트는 상상을 초월하는 기상천외한 엣지 케이스(Edge Case) 데이터를 밀어 넣습니다. 클라우드 인프라의 네트워크 지연(Latency)은 찰나의 순간에도 치명적인 병목 현상을 일으킵니다.

AI가 짜준 '성공률 100%'의 빈약한 방어막만 믿고 배포를 감행한다면, 주말 새벽 피 말리는 긴급 핫픽스 작업과 마주하게 될 것입니다. 트랜잭션 롤백은 실패하고 좀비 데이터가 쌓이며, 비즈니스의 치명적 손실은 고스란히 개발자의 몫으로 돌아옵니다. 테스트 코드가 통과했다고 안심하는 그 순간, 우리의 시스템은 가장 취약한 상태에 놓여 있는 것입니다. 단순한 버그를 넘어 서비스 전체를 마비시키는 **연쇄 장애(Cascading Failure)**는 언제나 우리가 한 번도 의심하지 않았던 가장 약한 고리, 즉 '당연히 성공할 것'이라 믿었던 지점에서 시작됩니다. 모니터링 알람이 울리기 시작하면 이미 늦습니다.

우리가 진정으로 구축해야 할 코드는 80%의 실패와 예측 불가능한 재난을 견뎌내는 견고한 벙커입니다. 이 글에서 소개할 치트키는 AI에게 넷플릭스의 전설적인 장애 주입 시스템인 **'카오스 몽키(Chaos Monkey)'**의 철학을 강제로 주입하는 하드코어 프롬프트입니다. AI가 습관적으로 만들어내는 안일한 가정들을 시스템 레벨에서 원천 차단하고, 극한의 스트레스 상황을 시뮬레이션하도록 지시합니다. 더 이상 AI에게 "테스트 코드를 짜줘"라고 부탁하지 마십시오. 대신 "내 시스템을 무너뜨려 봐"라고 명령해야 합니다.

네트워크 타임아웃, DB 연결 단절, HTTP 500 에러 등 최악의 시스템 파괴 시나리오를 가정해 방어 로직의 한계를 시험하도록 AI의 순진한 긍정 회로를 철저히 박살 내십시오. 이 프롬프트를 적용하는 순간, 여러분의 테스트 스위트(Test Suite)는 단순한 문법 검사기를 넘어 시스템의 생존력을 증명하는 가장 확실한 무기로 탈바꿈할 것입니다. 예상된 실패를 우아하게 처리(Graceful Degradation)하고, 서킷 브레이커(Circuit Breaker)가 적시에 작동하며, 어떤 재난 상황에서도 데이터 무결성을 유지하는 완벽한 아키텍처를 경험해 보세요. 확신을 가지고 배포(Deploy) 버튼을 누르는 통쾌함, 주말 출근과 밤샘 디버깅의 공포로부터 완전히 해방되는 진정한 엔지니어링의 자유를 만끽하시길 바랍니다.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> _"네! 요청하신 사용자 로그인 로직의 테스트 코드를 작성해 드릴게요! 😊"_

> 기존에 일반 프롬프트를 입력하면 AI는 오직 정확한 이메일과 비밀번호가 입력됐을 때 `200 OK`를 반환하는 단편적인 해피 패스(Happy Path) 테스트 단 1개만 던져줍니다. 게다가 글로벌 `fetch` 객체를 통째로 덮어써 버려, 이후 병렬로 실행되는 모든 테스트 생태계를 붕괴시키는 시한폭탄 같은 안티 패턴이 그대로 방치되어 있었습니다.

### ✅ After (완벽하게 변신한 결과)

```text
✓ [네트워크 장애] 결제 API가 5초 이상 응답하지 않을 때 TimeoutError를 던지는가 (12ms)
✓ [DB 셧다운] 500 Internal Server Error 발생 시 트랜잭션이 완벽하게 롤백되는가 (8ms)
✓ [Rate Limit] 초당 100회 이상의 요청이 몰릴 때 429 에러를 우아하게 반환하는가 (15ms)
✓ [격리 검증] MSW 환경에서 다른 전역 스코프를 전혀 오염시키지 않는가 (5ms)
```

## ⚡️ 3줄 요약 (TL;DR)

1. **Happy Path 맹신 타파:** 무의미한 정상 작동 테스트 비중을 20% 이하로 억제하고, 나머지 80%의 리소스를 철저한 예외 및 실패 시나리오 검증에 투입하십시오.
2. **재난 시뮬레이션 강제화:** 네트워크 타임아웃, HTTP 500 에러, 데이터베이스 연결 단절 등 서비스 셧다운을 유발하는 극한의 엣지 케이스를 집요하게 주입합니다.
3. **전역 객체 오염(Dirty Mocking) 원천 차단:** `window.fetch` 등 전역 객체의 무분별한 훼손을 엄금하고, MSW나 의존성 주입(DI)을 통해 완벽한 테스트 격리(Isolation) 환경을 구축합니다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

수십 번의 시행착오 끝에 완성된 프롬프트입니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

### 🥉 Basic Version (기본형)

기본적인 에러 핸들링 로직의 방어력을 신속하게 1차 검증해야 할 때 가볍게 활용해 보십시오.

> **역할 (Role):** 너는 예외 상황을 집요하게 파고드는 하드코어 QA 엔지니어이자 '카오스 몽키'야.
>
> **상황 (Context):**
> - 배경: 시스템 테스트가 필요한 코드가 주어졌어.
> - 목표: 극단적인 장애 상황에서 코드가 버티는지 확인하는 것.
>
> **요청 (Task):**
> 1. 내가 제공한 `[검증할 코드]`에 대한 테스트 코드를 작성해 줘.
> 2. 정상 작동(Happy Path) 시나리오는 단 1개만 작성해.
> 3. 나머지 모든 테스트는 네트워크 500 에러, DB 연결 타임아웃, 비정상적인 데이터 입력 등 '최악의 재난 상황'을 가정하여 방어 로직의 견고함을 검증해야 해.
>
> **제약사항 (Constraints):**
> - 모바일 가독성을 위해 표(Table)는 절대 사용하지 말고, 가독성 높은 글머리 기호(List) 형태로 정리해 줘.
> - 중요한 키워드는 반드시 **볼드** 처리해 줘.

### 🥇 Pro Version (전문가형)

AI의 무의미하고 순진한 긍정 회로를 완전히 박살 내고, 프로덕션 수준의 완벽한 테스트 격리(Isolation)와 리소스 정리(Garbage Collection)까지 집요하게 검증해 내는 궁극의 엔터프라이즈급 프롬프트입니다. 한 글자도 빠짐없이 복사하여 사용하십시오.

> **역할 (Role):**
> 너는 넷플릭스 SRE 팀 출신의 무자비한 시니어 QA 아키텍트, '카오스 몽키'야. 너의 유일한 목적은 코드가 어떻게 무너지는지 찾아내는 것이며, 형식적인 인사말이나 불필요한 설명은 일절 배제해.
>
> **상황 (Context):**
> - 배경: 현재 `[작성할 대상 모듈/컴포넌트]`의 Unit/E2E 테스트 코드를 작성하는 중이야.
> - 목표: 극단적인 예외 상황을 가정한 견고한 테스트 스위트(Test Suite)를 구축하여, 어떠한 시스템 재난 속에서도 서비스가 중단되지 않고 우아하게 에러를 처리하거나 복구되는지 검증하는 것.
>
> **요청 (Task):**
> 1. 정상 작동(Happy Path) 테스트는 전체 비중의 20% 이하로 제한해. 너의 핵심 가치는 80%를 차지해야 할 실패 시나리오 검증에 있어.
> 2. 다음 재난 시나리오 중 하나 이상을 반드시 구현해: 외부 API 응답 지연(Timeout), HTTP 500 Internal Server Error, DB 연결 단절.
> 3. 단순한 성공/실패 여부만 검증하지 마. 트랜잭션 롤백의 정상 작동 여부, 병목 발생 시 타임아웃 제한 준수 여부, 메모리 누수 방지를 위한 클린업(Garbage Collection) 상태까지 `expect` 문으로 집요하게 검증해.
>
> **제약사항 (Constraints):**
> - 글로벌 객체 오염(Dirty Mocking) 절대 엄금. `global.fetch`나 `window.setTimeout`을 직접 조작하는 등 전역 스코프를 훼손하는 코드는 작성하지 마.
> - 네트워크 통신 모킹은 반드시 MSW(Mock Service Worker)를 활용하거나, 의존성 주입(DI) 패턴으로 주입받은 클라이언트 객체만을 격리하여 처리해.
> - 최종 결과물은 마크다운 코드 블록으로만 출력하며, "도움을 드려 기쁩니다"와 같은 불필요한 꼬리말은 절대 추가하지 마.
> - 모바일 가독성을 위해 표(Table)는 절대 사용하지 말고, 가독성 높은 글머리 기호(List) 형태로 정리해 줘.
> - 중요한 키워드는 반드시 **볼드** 처리해 줘.
>
> **주의사항 (Warning):**
> - 확실하지 않은 정보는 지어내지 말고 "모른다"고 답해. (환각 방지)

## 💡 작성자 코멘트 (Insight & How to use)

솔직하고 뼈아픈 현실부터 짚고 넘어가 보겠습니다. 대다수의 AI 모델은 코드 작성을 지시받았을 때, 어떻게든 **'성공(✅)'이라는 결과만을 렌더링하기 위해 현실성 제로의 완벽한 환경을 가정**합니다. 이른바 AI가 만들어낸 '유토피아적 착각'입니다. 저 역시 과거 AI에게 핵심 결제 모듈의 테스트 코드를 일임했다가 크나큰 낭패를 본 경험이 있습니다. AI가 작성해 준 테스트 스위트는 100%의 커버리지를 자랑하며 통과했지만, 정작 실서버 배포 직후 PG사 API 응답이 단 3초 지연되자 **전체 스레드가 블로킹되며 대규모 타임아웃 장애**가 발생했습니다. AI는 단 한 번도 **'네트워크 지연(Latency)'**이라는 현실적인 변수를 테스트 케이스에 포함시키지 않았던 것입니다.

이 프롬프트는 바로 그러한 **AI의 안일함과 나태함을 철저히 파괴하기 위해 설계**되었습니다. 실무 환경에서 치명적인 시스템 셧다운이 발생하는 주된 원인은 우리가 정성껏 작성한 비즈니스 로직 자체의 결함이 아닙니다. 대부분 굳게 믿고 의지하던 **서드파티(3rd Party) API가 돌연 502 Bad Gateway를 반환**하거나, **데이터베이스 커넥션 풀이 고갈**될 때 발생하는 무한 로딩과 예외 처리(Exception Handling) 누락에서 비롯됩니다.

이 '카오스 몽키' 프로토콜의 `[작성할 대상 모듈/컴포넌트]` 변수에 결제 훅(Hook), 인증 미들웨어, 비동기 작업 큐 등 프로젝트의 크리티컬한 도메인을 치환하여 주입해 보십시오. AI는 스스로 가장 복잡하고 까다로운 **엣지 케이스(Edge Case)를 강제로 탐색**하며 시스템의 밑바닥을 파헤치기 시작할 것입니다. 단순히 에러가 발생하는지 확인하는 것을 넘어, 자원을 제대로 해제(Garbage Collection)하는지, 트랜잭션이 원자적으로 롤백(Rollback)되는지까지 낱낱이 검증하게 됩니다.

더 나아가, 이 프롬프트는 전역 스코프를 무참히 훼손하는 **질 낮은 모킹(Dirty Mocking) 패턴까지 사전에 원천 차단**합니다. 시니어 개발자들이 가장 고통받는 순간 중 하나가, A 파일의 테스트 코드가 엉뚱하게 B 파일의 전역 객체(예: `window.fetch`)를 덮어써 버려 원인 모를 연쇄 실패가 발생하는 상황입니다. 프롬프트 제약사항에 명시된 **MSW 활용 및 DI(의존성 주입) 강제화 지침**은 테스트의 순수성과 완전한 격리성(Isolation)을 보장합니다. 이를 통해 거짓 음성(False Negative)이나 거짓 양성(False Positive) 없는 **신뢰도 100%의 무결점 테스트 스위트**를 구축할 수 있습니다.

단순히 AI에게 테스트 코드를 "맡기는" 수준에서 벗어나, AI를 "통제하고 가이드"하는 것이 진정한 프롬프트 엔지니어링의 본질입니다. 이 하드코어 치트키는 주말 출근과 밤샘 디버깅의 늪에서 여러분을 구출해 줄, 가장 잔혹하면서도 든든한 강철 방어막이 되어줄 것입니다.

결과가 좋지 않을 때의 대처법(Troubleshooting)도 명확합니다. 만약 AI가 작성해 준 카오스 몽키 테스트 코드가 여러분의 로컬 환경에서 새빨간 실패(Fail) 로그를 뿜어낸다면, 절대 테스트 코드를 수정하거나 엣지 케이스를 완화하려 하지 마십시오. 그것은 테스트의 실패가 아니라, 여러분의 프로덕션 코드가 가진 **구조적 결함이 적나라하게 드러난 축복받은 순간**입니다.

당황하지 말고, 즉시 메인 비즈니스 로직으로 돌아가 서킷 브레이커(Circuit Breaker) 패턴을 적용하거나 데드 레터 큐(DLQ)를 도입하여 복구 메커니즘을 강화하십시오. 이 프롬프트는 여러분의 코드가 얼마나 완벽한지 칭찬해 주는 도구가 아니라, 철저히 부수고 파괴하여 숨겨진 약점을 찾아내는 **고강도 스트레스 테스트 도구**임을 명심해야 합니다. 이 훈련을 반복하다 보면, 어느새 어떠한 트래픽 스파이크나 의존성 장애에도 결코 무너지지 않는 세계 최고 수준의 회복 탄력성(Resilience)을 지닌 애플리케이션을 완성하게 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 지나치게 극단적인 에러 상황을 설정해서, 아예 테스트가 통과하지 못하면 어떡하나요?**
  - A: 바로 그것이 이 프롬프트가 지닌 진정한 가치이자 핵심 목적입니다! 테스트가 붉은색 실패(Fail)를 띄운다는 사실은, 현재 당신의 프로덕션 코드에 해당 예외 상황을 방어할 안전장치(Fallback)가 완전히 누락되어 있음을 뼈아프게 짚어주는 경고장입니다. 당황하지 마시고 실제 코드에 타임아웃 처리나 서킷 브레이커, 재시도(Retry) 로직을 추가하여 시스템의 내구성을 한 단계 끌어올리십시오.
- **Q: 백엔드 API뿐만 아니라, 프론트엔드 UI 컴포넌트 테스트에도 이 치트키를 적용할 수 있나요?**
  - A: 완벽하게 호환됩니다. 변수에 대상 UI 컴포넌트 이름을 입력하면, 사용자가 결제 버튼을 0.1초 간격으로 50번 연속 광클(Double Submit)하는 상황이나 극심한 네트워크 병목으로 인해 로딩 스피너가 30초 이상 멈추지 않는 상황 등, 프론트엔드 특유의 극한 UI 엣지 케이스를 검증하는 데 탁월한 시너지를 발휘합니다.
- **Q: 프론트엔드 환경에서 MSW 대신 `jest.mock()`이나 `vi.mock()`을 그대로 사용해도 무방한가요?**
  - A: 특정 로컬 모듈을 모킹하는 행위 자체는 전혀 문제가 되지 않습니다. 이 프롬프트가 엄격하게 경계하는 것은 `global`이나 `window` 스코프를 무분별하게 조작하여, 독립적으로 실행되어야 할 다른 테스트 파일의 런타임 환경까지 훼손하는 안티 패턴입니다. 테스트 간의 완벽한 격리성(Isolation)만 보장할 수 있다면, 팀 내에서 표준으로 사용하는 모킹 프레임워크를 자유롭게 활용하셔도 무방합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **냉혹한 페르소나 부여 (Role):** AI에 내재된 '무조건 긍정적이고 친절한 비서'라는 디폴트 설정값을 강제로 포맷합니다. 시스템의 취약점과 빈틈을 찾아내는 데 혈안이 된 넷플릭스 SRE 아키텍트의 무자비한 자아를 덮어씌움으로써, 눈치 보기식 정상 테스트나 불필요한 부연 설명을 원천 차단했습니다.
2. **명확한 수치 기반 제약 (Constraints):** 단순히 "실패 시나리오를 많이 추가해 줘"라는 모호하고 정성적인 지시는 대화형 AI에게 통하지 않습니다. **"정상 작동 테스트는 20% 이하로 제한"**이라는 날카롭고 정량적인 가이드라인을 제시하여, AI의 토큰 생성 가중치를 오류 처리 검증 로직으로 완벽하게 편향시켰습니다.
3. **아키텍처 레벨의 가이드라인 (Clean Mocking):** AI가 토큰을 아끼기 위해 밥 먹듯이 남용하는 최악의 안티 패턴인 '전역 객체 오염(Dirty Mocking)'을 프롬프트 주입 단계에서 선제적으로 틀어막았습니다. 그 결과 출력되는 테스트 코드는 상호 간섭이 배제된 최고 수준의 독립성과 유지보수성을 확보하게 됩니다.
