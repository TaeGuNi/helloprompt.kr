## 🌪️ 数据管道 (ETL)：Airflow DAG 编写自动化

- **🎯 推荐对象：** 凌晨被 Crontab 报错警报惊醒的数据工程师、苦于梳理“面条代码”般批处理任务的后端开发者
- **⏱️ 预计时间：** 2 小时 → 缩短至 1 分钟
- **🤖 推荐模型：** ChatGPT-4o, Claude 3.5 Sonnet（在处理复杂 Python 逻辑与依赖管理方面表现绝佳）

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐⭐

> _“任务 A 完成后触发 B，若 B 失败则回滚至 C。哦对了，周末和节假日跳过执行。”——面对如此错综复杂的工作流，您难道还在靠 Shell 脚本和 Crontab 死死硬撑吗？_

构建稳定高效的 ETL 数据管道，堪称数据工程领域的“皇冠明珠”，这本质上是一场严密的逻辑博弈。Apache Airflow 作为业内顶级的编排工具，通过编排 DAG（有向无环图）代码，让原本如同黑盒般的复杂流程变得可视化且高度可控。然而，每次立项都要手动拼接 `Operator`、反复编写重试（Retry）与容错逻辑，无疑是在无谓地消耗工程师的宝贵生命。

现在，游戏规则变了。您只需用人类语言向 AI 描述 **“流程图（Flowchart）”** 与 **“触发条件（Conditions）”**，原本繁琐冗长的管道编排代码，只需短短 1 分钟即可由 AI 完美代工。

---

## ⚡️ 核心 3 步总结 (TL;DR)

1. **提取 (Extract)：** 明确界定数据源（数据库或 API）及其精确的访问与提取策略。
2. **转换 (Transform)：** 循序渐进地拆解业务逻辑（如数据过滤、多表连接、格式清洗）。
3. **加载 (Load)：** 严谨定义数据的最终归宿（如 S3、BigQuery）与具体的落盘方式。

## 🚀 解决方案："DAG 生成器提示词"

### 🥉 Basic Version (基础版)

适用于自动化基础调度或轻量级单一任务。

> **角色 (Role)：** 你是一个精通 Python 的数据工程师 AI 助手。
> 
> **任务 (Task)：** 请编写一段 Airflow DAG 代码，设定每天早上 6 点（KST）自动获取 MySQL 数据库的数据备份（Dump）并上传至 S3 存储桶。请强制使用 `BashOperator`，并配置在任务失败时自动重试 2 次。

### 🥇 Pro Version (专业版)

专为构建涵盖复杂依赖控制、分支处理及 Slack 故障告警等达到生产环境 (Production-Ready) 级别的数据管道而生。

> **角色 (Role)：** 你是一位拥有 10 年架构经验的资深数据工程师，极度擅长应对高并发与海量数据的流转。
>
> **背景 (Context)：**
>
> - 背景：我正在构建一条核心 ETL 管道，需要在每天午夜从多个异构数据源拉取数据，并将其无缝加载至企业数据仓库中。
> - 目标：编写一段极致健壮且具备高容错性的 Airflow DAG 代码，确保在任何节点发生故障时都能第一时间触发响应机制。
>
> **任务 (Task)：**
>
> 请严格按照以下业务逻辑流编写 Airflow DAG 管道代码。请将括号 `[ ]` 中的变量部分保留注释作为指引，以便我在代码生成后能快速填入实际业务值。
>
> 1. **[Task A]**: 从外部 API 接口拉取 `[数据主题（例如：实时汇率信息）]`。（强制使用 `PythonOperator`）
> 2. **[Task B]**:（仅当 Task A 成功时触发）将拉取的数据进行标准化清洗后，以 `[文件格式（例如：Parquet）]` 格式转储至 `[中间存储（例如：AWS S3）]`。（若预估数据吞吐量庞大，请直接分配 `KubernetesPodOperator`）
> 3. **[Task C]**:（仅当 Task B 成功时触发）将位于 `[中间存储]` 的清洗后数据，安全加载入 `[数据仓库（例如：Amazon Redshift）]`。
> 4. **[Task D]**:（当 Task A、B、C 中任意一环遭遇失败时触发）立即通过 `[告警渠道（例如：Slack Webhook）]` 广播致命错误日志及发生崩溃的任务名称。（强制使用 `on_failure_callback` 机制）
>
> **约束条件 (Constraints)：**
>
> - **重试机制 (Retry)：** 必须在 `default_args` 中全局声明：若任务失败，则每隔 `[重试间隔（分钟）]` 重试执行 `[最大重试次数]` 次。
> - **绝对幂等性 (Idempotency)：** 无论该 DAG 被回溯重跑多少次，都必须保证最终数据状态的一致性。请务必深度结合 `execution_date` 或 `logical_date` 参数来编写核心逻辑以捍卫这一原则。
> - **代码规约 (Code Style)：** 严守 PEP-8 规范，并在脚本最底部，以最直观的方式（`>>`, `<<` 运算符）清晰勾勒出所有任务节点的拓扑依赖关系。
>
> **注意事项 (Warning)：**
>
> - 严禁混入任何已被弃用的 Airflow 1.x 老旧语法。所有代码必须 100% 契合最新的 Airflow 2.x 最佳实践标准。

## 💡 作者笔记 (Insight)

在构思和设计 Airflow DAG 时，不可撼动的最高法则是 **“幂等性 (Idempotency)”**。试想一下，当您在今天不得不手动重跑昨天崩溃的批处理任务时，如果代码导致了数据重复插入或是破坏了原有的逻辑链，那绝对是一场灾难。在向 AI 委派编码任务时，只需在提示词中精准刺入一句 **“请务必利用 `execution_date`（或 `data_interval_start`）参数来捍卫幂等性”**，就能让一段原本平庸的入门级脚本，瞬间蜕变为散发着高级工程师智慧光芒的、极其健壮的工业级管道代码。

不仅如此，针对每个特定的 Task 精准指派最契合的 Operator（例如区分计算密集型的 `KubernetesPodOperator` 与轻量调度的 `BashOperator`），能够极大地减少基础设施资源的无谓开销。**精准的上下文约束，往往决定了 AI 输出代码的生产力上限。**

## 🙋 常见问题 (FAQ)

- **Q: 我们公司内部封装了海量的自定义 Operator（Custom Operators），AI 也能驾驭这些私有组件吗？**
  - A: 完全没问题。您只需要在提示词的 **“背景 (Context)”** 区块中，附带几行说明，简明扼要地描述该自定义 Operator 的核心机制以及它所接收的参数字典（Kwargs），AI 便能如同贵司的老员工一般，无缝将其编织进 DAG 中。

- **Q: 老实说，我连如何在本地把 Airflow 跑起来都一头雾水。**
  - A: 面对复杂的依赖，在本地环境采用 `docker-compose` 容器化部署绝对是最干净利落的解法。您可以直接甩给 AI 一个新指令：_“请为我一键生成一份专供本地沙盒测试 Airflow 2.x 的高标准 docker-compose.yaml 配置文件。”_

- **Q: 既然 AI 这么强，我能让它用 Airflow 写一套处理实时流数据（Streaming Data）的架构吗？**
  - A: 请踩下刹车。Airflow 的架构基因决定了它是一个专为 **批处理 (Batch Processing)** 而生的宏观调度器。如果您追求的是毫秒级或 1 分钟以内的超低延迟实时流计算，正确的架构选型是去向 AI 请教 Apache Kafka 或是 Flink 的最佳实践，强行用 Airflow 只会适得其反。

## 🧬 提示词解剖 (Why it works?)

1. **具象化拓扑依赖 (Dependency Formulation)：** 通过用口语化的场景描述（如“A 跑通了才跑 B，有谁挂了就喊 D”），精准且平滑地引导 AI 在底层代码中完美映射出 `A >> B >> C` 与 `[A, B, C] >> D` 这样严丝合缝的逻辑运算符。
2. **生产级边界压制 (Constraints Engineering)：** 毫不妥协地植入了自动重试 (Retry)、实时告警 (Alert) 以及绝对幂等性 (Idempotency) 等工业级实战中攸关生死的防线，将 AI 生成代码的鲁棒性（Robustness）直接拉满。
3. **强制版本锚定 (Version Locking)：** 鉴于开源生态中 API 语法的剧烈迭代，在提示词末尾强硬锁定 “Airflow 2.x 最佳实践”，直接从基因层面抹杀了 AI 吐出被弃用（Deprecated）遗留代码的致命隐患。
