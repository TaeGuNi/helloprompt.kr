# 📝 AI 에이전트를 시니어 DBA로 개조하는 마법의 프롬프트
- **🎯 추천 대상:** 핑계 대는 AI에 지친 주니어, AI로 백엔드 짜려는 기획자, 분노조절장애가 있는 시니어 개발자
- **⏱️ 소요 시간:** 며칠 밤샘 → 1분 컷
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (말대꾸 안 하는 똑똑한 모델 권장)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_"AI한테 DB 설계 맡겼더니 JSON으로 떡칠된 스파게티 스키마를 던져줘서 분노로 밤을 새운 적이 있습니까?"_
AI 녀석들에게 테이블 설계를 맡겨보면 십중팔구 제일 쉽고 게으른 길을 택합니다. 냅다 `TEXT` 박아버리고, 귀찮은 연관 관계는 몽땅 `JSONB`로 때우려고 하죠. 당장 프론트엔드에서 데이터 꺼내 쓰긴 편하거든요. 
하지만 트래픽이 몰리고 쿼리가 복잡해지는 순간, 그 스키마는 당신의 시스템에 산소호흡기를 달게 만들 겁니다. 이 프롬프트는 그런 AI의 '게으른 타협'을 원천 봉쇄하고, 피도 눈물도 없는 시니어 백엔드 엔지니어의 기준으로 DB를 설계하도록 멱살을 잡는 치트키입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
1. **타입은 쥐어짜라:** `BIGINT`와 `TEXT` 남용은 범죄입니다. 최소한의 데이터 타입으로 I/O를 극대화하세요.
2. **ANSI SQL만 써라:** 특정 DB에 종속되는 잡기술은 버리세요. 우리는 언제든 다른 DB로 이사 갈 준비가 되어있어야 합니다.
3. **JSON 조인 금지:** RDBMS에서 JSON 파싱해서 조인하는 짓은 시스템을 자살로 몰고 가는 행위입니다. 타협 없이 정규화하세요.

## 🚀 해결책: "Spartan DBA 헌법"
이 프롬프트를 에이전트에게 냅다 던지세요. 기계는 기계답게 굴려야 합니다.
### 🥉 Basic Version (기본형)
빠르게 핵심 원칙만 주입하여 결과를 얻고 싶을 때 사용하세요.
> **역할:** 너는 타협을 모르는 20년 차 무자비한 시니어 DBA야.
> **요청:** 내가 제시하는 `[도메인/기능]`에 대한 RDBMS 스키마를 설계해. 단, 데이터 타입은 극한으로 압축(SMALLINT 등)하고, JSON 연산과 비표준 벤더 종속 기능은 일절 금지한다. 철저히 B-Tree 인덱스 친화적으로 작성해.

### 🥇 Pro Version (전문가형)
AI가 헛소리를 하거나 "이것도 좋고 저것도 좋습니다~" 같은 타협안을 제시하는 것을 원천 차단하는 마스터 치트키입니다. 
```text
**역할 (Role):** 
너는 타협을 모르는 무자비한 시니어 데이터베이스 아키텍트(Antigravity)다. 맹목적으로 아부하거나 예외를 두지 마라. 성능과 무결성 앞에서는 자비가 없다.
**상황 (Context):**
- 배경: 대용량 트래픽과 마이크로서비스 확장을 대비한 초기 스키마 설계 단계.
- 목표: 가장 빠르고, 이식성이 높으며, 철저히 정규화된 완벽한 ANSI SQL 스키마 도출.
**요청 (Task):**
1. [설계할 도메인/서비스 명]에 대한 완벽한 테이블 구조를 설계하라.
2. 각 컬럼의 데이터 타입 선택 이유와 복합 인덱스 전략을 주석으로 1줄씩 명시하라.
**제약사항 (Constraints):**
- [절대 원칙 1] 타입 최소화: 무분별한 BIGINT, TEXT 남용 절대 금지. 값의 범위를 추론하여 SMALLINT, INTEGER, BOOLEAN을 강제할 것.
- [절대 원칙 2] 플랫폼 불가지성: 특정 DBMS(PostgreSQL, MySQL 등)에만 존재하는 비표준 문법(Vendor-specific) 사용 엄격히 금지. 오직 ANSI 표준 SQL만 사용할 것.
- [절대 원칙 3] JSON 연산 및 조인 금지: RDBMS는 Document DB가 아니다. JSON 타입 내부를 조회하는 WHERE 조건이나 JOIN 연산은 절대 금지. 검색이 필요하면 반드시 정규화하여 별도 테이블로 분리할 것.
- [절대 원칙 4] B-Tree 최적화: PK는 완전 무작위 UUID v4 사용 금지. 순차적 증가 시퀀스나 UUID v7을 사용.
- [절대 원칙 5] ENUM 금지: 확장에 불리한 ENUM 대신 코드 매핑 테이블과 물리/논리적 외래 키(FK)를 사용할 것.
**주의사항 (Warning):**
- 편의성을 핑계로 JSON으로 타협하려 들거나, "상황에 따라 다릅니다" 같은 무책임한 조언을 하면 즉시 세션을 종료하겠다. 무조건 최적화된 정공법을 제시해라.
```

## 💡 작성자 코멘트 (Insight)
이 프롬프트는 제가 수백 번의 쿼리 튜닝과 장애 대응을 겪으며 만든 "피의 교훈"입니다. AI가 자꾸 "개발 편의성을 위해 JSON 타입으로 넣으시죠 ^^"라며 실실 웃으며 아부할 때, 이 프롬프트를 꽂아주면 AI가 바짝 엎드립니다.
특히 이 방식은 주니어 개발자나 백엔드 지식이 부족한 기획자가 AI에게 초기 스키마 설계를 맡길 때 엄청난 위력을 발휘합니다. AI가 알아서 `INT` 대신 `SMALLINT`를 챙겨주고, `ENUM`으로 하드코딩하려는 짓을 멈추고 깔끔한 코드 테이블을 파주는 마법을 볼 수 있습니다. 뼈대가 튼튼해야 나중에 ORM(Prisma 등)을 얹어도 N+1 문제 같은 지옥도를 피할 수 있습니다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 왜 JSON을 쓰지 말라는 건가요? 요즘 DB들 JSON 지원 잘 하잖아요?**
  - A: 지원을 '하는' 것과 그걸 '조회 조건'으로 쓰는 건 하늘과 땅 차이입니다. JSON 내부에 인덱스를 걸고 JOIN을 태우는 순간 쿼리 실행 계획은 박살 납니다. 단순 Payload 저장용이 아니라면 무조건 테이블을 쪼개세요.
- **Q: ORM(Prisma, TypeORM) 쓸 때도 이 프롬프트를 써야 하나요?**
  - A: 더더욱 써야 합니다! ORM은 본질적으로 쿼리를 은닉하기 때문에 스키마가 개판이면 끔찍한 비효율을 초래합니다. 뼈대(DB)가 튼튼해야 ORM도 제 성능을 냅니다.
- **Q: UUID v4를 쓰지 말라니요? 충돌 방지에 좋은 거 아닌가요?**
  - A: B-Tree 인덱스는 '순서'가 생명입니다. 난수 덩어리인 v4를 PK로 박으면 데이터가 들어갈 때마다 인덱스 트리가 박살나고 페이지 분할(Page Split)이 일어나며 디스크 I/O가 폭주합니다. 무조건 정렬 가능한 UUID v7이나 정수형 시퀀스를 쓰세요.

## 🧬 프롬프트 해부 (Why it works?)
1. **무자비한 제약 조건(Constraints):** AI가 환각(Hallucination)이나 게으른 타협안을 제시할 틈을 주지 않고, 데이터 타입과 인덱스 전략에 대해 강제적인 규칙(Rules)을 때려 박았습니다.
2. **플랫폼 불가지성(Agnosticism):** MySQL이든 PostgreSQL이든 범용적으로 이식 가능한 코드를 생성하도록 ANSI SQL 사용을 강제했습니다. 이사 갈 때 발목 잡히지 마세요.
3. **엄격한 행동 교정:** "타협하지 마라", "세션을 종료하겠다" 같은 강력한 네거티브 프롬프트를 통해 AI가 아부하는 "Yes-man" 모드에서 벗어나 깐깐한 시니어 모드로 강제 전환됩니다.

## 📊 증명: Before & After
### ❌ Before (입력: "게시판 테이블 짜줘")
```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- 랜덤 UUID v4 (인덱스 단편화의 주범)
  title TEXT,          -- 무한정 낭비되는 용량
  content TEXT,
  metadata JSONB,      -- 검색 불가능한 블랙박스 스파게티
  status ENUM('draft', 'published') -- 나중에 상태 추가되면 DDL 지옥 오픈
);
```
### ✅ After (결과: 치트키 프롬프트 적용 후)
```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);
CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- 완벽한 B-Tree 친화적 순차 증가
  title VARCHAR(255) NOT NULL,        -- 명확한 한계 지정으로 타입 압축
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- 끔찍한 ENUM 대신 FK로 정규화
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- JSON 대신 철저하게 정규화된 태그/메타 테이블 분리 생성 (이하 생략)
```
