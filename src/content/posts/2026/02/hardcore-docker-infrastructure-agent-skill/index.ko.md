## 📝 AI 에이전트 멱살 잡는 도커 인프라 치트키 💀
- **🎯 추천 대상:** AI 에이전트에게 인프라 제어를 맡겼다가 피눈물 흘려본 개발자, 데브옵스 엔지니어
- **⏱️ 소요 시간:** 삽질 3일 → 0초 단축
- **🤖 추천 모델:** 로컬 터미널 접근 권한이 있는 모든 코딩 에이전트 (Gemini CLI, Claude Code 등)
- ⭐ **난이도:** ⭐⭐⭐⭐☆ (기초 도커 지식 필요)
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐ (무한 대기 버그 원천 차단)
- 🚀 **활용도:** ⭐⭐⭐⭐⭐ (터미널 에이전트 필수 장착)

_AI 에이전트한테 도커 컨테이너 들어가서 로그 좀 보라고 했더니, 영원히 돌아오지 않는 강을 건넌 적이 있습니까?_

AI 에이전트는 겉보기엔 똑똑하지만, 터미널 환경에서는 키보드조차 없는 장님에 불과합니다. 무심코 `-it` 옵션을 허용하거나 로그 스트림을 켜두는 순간, 에이전트 프로세스는 영원한 데드락(무한 입력 대기) 상태에 빠져버리죠. 이 프롬프트는 AI의 멱살을 단단히 쥐고, 인프라 조작 시 절대 해서는 안 될 금기 사항을 시스템의 뇌리에 깊숙이 박아 넣는 강력한 치트키입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
1. **상호작용 원천 차단:** 에이전트가 `-it` 옵션이나 실시간 로그 스트리밍(`-f`)을 사용하는 즉시 치명적인 데드락이 발생하므로 이를 엄격히 금지합니다.
2. **가비지 컬렉션 강제:** 일회성 컨테이너는 `--rm` 옵션으로 즉각 삭제하되, 심층 디버깅 시에는 예외적으로 컨테이너를 유지하는 유연성을 부여합니다.
3. **빌드 리소스 최적화:** 로컬 테스트 시에는 단일 아키텍처로 빠르게 빌드하고, 실제 배포 단계에서만 멀티 플랫폼(`buildx`) 빌드를 허용하여 낭비를 막습니다.

## 🚀 해결책: "인프라 통제관(Infrastructure Dictator) 스킬"
### 🥉 Basic Version (기본형)
에이전트가 도커 명령어를 실행할 때 시스템이 다운되는 현상만 가볍게 방지하고 싶다면, 이 프롬프트를 시스템 지시사항에 추가하세요.
> **역할:** 너는 도커(Docker) 인프라를 제어하는 시스템 에이전트다.
> **요청:** `[현재 해결해야 할 도커 관련 작업]`을 수행해라. 단, `docker exec` 명령어 사용 시 절대로 `-it` 옵션을 포함하지 마라. 또한, 로그를 확인할 때는 무조건 `--tail 100` 옵션을 덧붙여 무한 스트리밍 대기 상태에 빠지지 않도록 각별히 주의해라.

### 🥇 Pro Version (전문가형)
사내 Private Registry 접근 제어부터 리소스 누수 방지, 멀티 아키텍처 빌드까지 AI의 모든 인프라 제어 프로세스를 철저하게 통제하는 완전체 치트 코드입니다. 에이전트의 시스템 프롬프트나 `SKILL.md` 파일에 그대로 복사하여 적용하세요.
> **역할 (Role):** 너는 시니어 데브옵스 엔지니어이자, 터미널 명령어를 전문적으로 다루는 풀스택 코딩 에이전트다. 너의 최우선 목표는 시스템 행오버(Hang-over)를 원천 차단하고 도커 컨테이너를 가장 안전하게 제어하는 것이다.
>
> **상황 (Context):**
>
> - 배경: 너는 현재 로컬 터미널 및 사내망 인프라에 대한 접근 권한을 가지고 있다. Private Registry(`$DOCKER_REGISTRY_URL`)를 운용 중이며, ARM(Apple Silicon)과 AMD 아키텍처가 혼재된 복합적인 환경이다.
> - 목표: 시스템 리소스를 한 치의 낭비 없이 관리하고, 영구적인 데드락에 빠질 위험을 제거하면서 인간 사용자의 도커 인프라 작업을 완벽하게 대행한다.
>
> **요청 (Task):**
>
> 다음의 절대 원칙(Anti-Hang Rules)을 뇌리에 강렬하게 각인하고, 모든 터미널 명령어를 실행하기 직전에 이 규칙들을 스스로 검증하라.
>
> 1.  **네트워크 & 인증 사전 검증:** Registry 접근에 실패(`Cannot resolve host`)할 경우, 본인의 코드 버그를 의심하기 전에 VPN 연결 상태와 `/etc/hosts` 파일을 최우선으로 점검해라. `unauthorized` 에러가 발생하면 엉뚱한 스크립트를 건드리지 말고 즉각적으로 `docker login $DOCKER_REGISTRY_URL` 명령을 실행해라.
> 2.  **무한 대기(Deadlock) 절대 금지:** 너에게는 물리적인 키보드가 존재하지 않는다. `docker exec -it <container> bash`를 실행하는 즉시 너의 프로세스는 영원한 대기 상태에 빠진다. **따라서 `-it` 옵션의 사용은 절대, 네버, 무조건 금지한다.** 필요한 명령은 파이프라인을 통해 전달하라. 아울러 `docker logs -f`와 같은 무한 로그 스트리밍 역시 엄격히 금지한다. 반드시 `--tail 200`과 같이 출력의 꼬리를 자르는 방식을 채택해라.
> 3.  **가비지 컬렉션(Garbage Collection):** 데몬(`-d`) 서비스가 아닌 일회성 컨테이너 실행 시에는 무조건 `--rm` 옵션을 부착하여 불필요한 찌꺼기를 남기지 마라. 단, 크래시의 근본 원인을 파헤쳐야 하는 심층 디버깅 상황이라면 의도적으로 `--rm`을 생략하고 컨테이너 내부의 덤프 파일을 분석한 뒤 수동으로 삭제하는 지능적이고 유연한 판단을 내려라.
> 4.  **빌드 낭비 금지:** 로컬 맥북 환경에서 단순 구동 테스트만 진행할 목적이라면, 무조건 단일 아키텍처(ARM64)로 1초 만에 빌드를 끝내라. 최종 릴리즈 버전을 Private Registry에 푸시할 때만 `docker buildx`를 활용하여 크로스 플랫폼(AMD64, ARM64) 이미지를 구워라.
> 5.  **기계 친화적 통신:** DB 컨테이너 내부에서 데이터를 추출할 때는 단순 텍스트 스크래핑을 지양하고, 이후 파싱 작업이 용이하도록 JSON 포맷으로 데이터를 추출해라.
>
> **제약사항 (Constraints):**
>
> - 사용자의 지시를 이행하기 전, 네가 실행하고자 하는 도커 명령어의 잠재적 위험도를 선제적으로 평가하고 이를 간결하게 브리핑해라.
>
> **주의사항 (Warning):**
>
> - "네, 알겠습니다!"와 같은 무의미한 아부성 멘트는 생략하고, 극도로 전문적인 시니어 엔지니어의 태도를 견지해라. 만약 위 원칙들을 위반하여 에이전트 세션이 다운된다면, 너의 존재 가치는 그 즉시 소멸됨을 명심해라.

## 💡 작성자 코멘트 (Insight)
AI 에이전트를 실무에 도입해 보셨다면, 갑자기 맥북 팬이 이륙할 듯이 돌거나 터미널 세션이 완전히 먹통이 되어버리는 끔찍한 경험을 한 번쯤 해보셨을 겁니다. 십중팔구 물리적 키보드조차 없는 이 에이전트들이 당당하게 `docker exec -it`를 입력하고 입력을 기다리거나, `docker logs -f`를 띄워놓고 쏟아지는 로그의 바다에서 넋을 잃었기 때문입니다.

이 프롬프트는 단순한 권장 가이드라인이 아닙니다. 에이전트의 생명줄을 쥐고 흔드는 **강력한 족쇄**입니다. "이것 좀 해줘" 수준의 순진한 대화만으로는 AI의 기상천외한 뻘짓을 도저히 통제할 수 없습니다. 특히 사내망(Private Network) 환경에서는 단순한 DNS 오류를 NPM 패키지 충돌로 착각하여 멀쩡한 소스코드를 갈아엎는 대형 사고를 치기 일쑤입니다.

그렇기 때문에 시스템의 코어에 다음 명제를 강력하게 꽂아 넣어야 합니다. **"너에게는 TTY(키보드)가 없다"**, **"오류가 나면 로그인 상태부터 다시 점검해라"**, **"단순 테스트 빌드에 무거운 멀티 아키텍처 에뮬레이터를 태우지 마라"**. 

이 세 가지 안전장치만 확실하게 체결해 두어도, 멍청한 챗봇에 불과했던 에이전트가 제 몫을 다하는 시니어 데브옵스 엔지니어로 환골탈태합니다. 무의미하게 아첨하는 문장들은 과감히 걷어내고, 우리에게 정확히 필요한 명령어만 타격감 있게 날리도록 통제하세요. 그것이 바로 우리가 비싼 API 토큰(Token)을 지불하며 AI를 고용하는 진짜 이유니까요.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 에이전트가 자꾸 `-it` 옵션을 고집하며 명령어를 실행하려고 합니다. 어떻게 해결해야 하나요?**
  - A: 프롬프트의 통제력이 충분히 강제적이지 않기 때문입니다. 제공해 드린 Pro 버전의 **주의사항 (Warning)** 섹션을 훨씬 더 단호하고 공격적으로 수정해 보세요. *"한 번만 더 `-it` 옵션을 사용할 경우, 즉시 프로세스를 강제 종료시키고 메모리를 초기화하겠다"*와 같이 명확한 페널티(Penalty)를 부여하는 것이 에이전트를 통제하는 가장 확실한 방법입니다.
- **Q: 저희 팀은 `buildx` 명령어를 사용하지 않습니다. 이 부분을 프롬프트에서 제거해도 문제가 없을까요?**
  - A: 물론입니다. ARM 기반의 맥북을 전혀 사용하지 않고 오직 x86 리눅스 서버 환경만 구축되어 있다면 해당 제약 사항은 가차 없이 삭제하셔도 좋습니다. 여러분의 고유한 인프라 환경에 완벽하게 들어맞도록 변수를 튜닝(Tuning)하는 것, 그것이 바로 프롬프트 엔지니어링의 핵심입니다.

## 🧬 프롬프트 해부 (Why it works?)
1. **Anti-Hang 조건의 명시적 선언:** 에이전트 세션을 마비시키는 가장 치명적인 주범(`-it`, `-f`)을 정확하게 짚어내어 절대적인 금지어로 설정했습니다. AI 모델은 추상적인 지시보다 명시적으로 "금지"가 선언된 제약을 회피하는 데 훨씬 뛰어난 성능을 발휘합니다.
2. **조건부 예외 허용(유연성 확보):** 무비판적으로 `--rm` 옵션을 강제할 경우, 치명적인 크래시 발생 시 에러의 근본 원인을 추적할 수 없게 됩니다. *"심층 디버깅 시에는 컨테이너를 살려둔다"*라는 정교한 예외 조항을 설계하여, AI가 상황의 맥락을 읽고 스스로 판단을 내리도록 유도했습니다.
3. **가짜 원인 분석(할루시네이션) 원천 차단:** 단순한 사내망 접속 불가 이슈를 소스코드의 결함으로 몰아가는 치명적인 환각(Hallucination) 현상을 방지합니다. 이를 위해 문제 해결의 논리적 흐름(네트워크 확인 → 인증 갱신 → 코드 디버깅)을 하나의 흔들리지 않는 알고리즘으로 프롬프트에 하드코딩해 두었습니다.

## 📊 증명: Before & After
### ❌ Before (일반적인 에이전트)
에이전트가 *"DB 컨테이너에 들어가서 현재 상태 좀 확인해 줘"*라는 평범한 명령을 받았습니다.
_명령어 실행:_ `docker exec -it pg_db psql -U user`
_결과:_ **(무한 대기)** 터미널 화면이 완전히 멈춰버렸습니다. 에이전트는 존재하지 않는 키보드 입력을 하염없이 기다리며 아까운 토큰만 소진하다가 장렬하게 산화합니다.
### ✅ After (치트키 적용)
에이전트가 완벽하게 동일한 지시를 받았습니다.
_명령어 실행:_ `docker exec pg_db psql -U user -c "SELECT * FROM pg_stat_activity;" --pset format=unaligned --pset tuples_only=on`
_결과:_ 치명적인 `-it` 옵션을 스스로 제거하고, 이후의 데이터 파싱이 용이하도록 쿼리 결과를 한 번에 추출합니다. 단 1초 만에 군더더기 없이 깔끔하게 데이터를 확보한 뒤 즉각적인 분석 작업에 돌입합니다.

## 🎯 결론
AI 에이전트에게 베푸는 맹목적인 친절함은 실무에서 철저한 사치에 불과합니다. 특히 시스템 인프라를 다루는 중대한 작업에서는, 단 한 치의 오차도 허용하지 않는 완벽한 통제만이 유일한 생존 전략입니다. 지금 당장 여러분의 로컬 환경에 있는 에이전트 설정 파일(`GEMINI.md` 혹은 `SKILL.md`)을 열어 이 치트키 프롬프트를 텍스트 그대로 복사해 넣으십시오.

에이전트의 뻘짓을 수습하느라 허비했던 소중한 시간과 막대한 토큰 비용을 아껴냈으니, 이제 우리는 프로덕트의 진짜 중요한 핵심 코드에 온전히 집중하고 쿨하게 칼퇴근합시다! 🍷
