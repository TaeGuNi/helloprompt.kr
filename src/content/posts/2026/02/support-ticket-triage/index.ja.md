# 📝 怒っている顧客を待たせてはいけません

- **🎯 おすすめの対象者:** CSマネージャー、ECサイト運営者、カスタマーエクスペリエンス(CX)担当者
- **⏱️ 所要時間:** 1時間 → 1分
- **🤖 推奨モデル:** GPT-3.5 Turbo (高速), GPT-4o (高精度)

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐☆
- 🚀 **実用性:** ⭐⭐⭐☆☆

> _「スピードこそが最大のサービス。激怒する顧客のゴールデンタイムを守り抜きましょう。」_

数百件ものCS（カスタマーサポート）の問い合わせが溜まった掲示板の前で、途方に暮れたことはありませんか？ 順番通りに返信しているうちに、本当に緊急を要する「配送事故」や「二重決済のシステムエラー」などの問い合わせを見落とし、顧客の不満を爆発させてしまった経験が誰にでも一度はあるはずです。これからはAIを活用して顧客の感情状態と問い合わせの緊急度を事前にフィルタリングし、まるで救急救命室の「トリアージ（患者の重症度分類）」のようにCS業務の優先順位を再構築してみましょう。

---

## ⚡️ 3行まとめ (TL;DR)

1. **データ収集:** 顧客からの問い合わせ内容を集め、AIに渡します。
2. **AIトリアージ:** AIが各問い合わせの「感情スコア」と「カテゴリ」を分析・分類します。
3. **優先対応:** 感情スコアが高く（怒り・不満）、緊急を要するカテゴリ（決済・配送）の問い合わせから迅速に処理します。

---

## 🚀 解決策：「CS救命室トリアージ・プロンプト」

### 🥉 Basic Version (基本型)

素早くカテゴリだけを分類したい場合に使用してください。

> **役割 (Role):** あなたは迅速かつ正確な`[CSチーム長]`です。
> **要求 (Task):** 以下の問い合わせ内容を読み、`[カテゴリ]`と`[緊急度(高/中/低)]`を判断して回答してください。
> **問い合わせ (Input):** `[ここに顧客の問い合わせ内容をコピー＆ペースト]`


### 🥇 Pro Version (専門家型)

顧客の感情分析を組み合わせて離脱リスクを管理し、優先順位を明確にしたい場合に使用してください。自動化パイプライン（Zapier、Makeなど）との連携に最適な構造です。

> **役割 (Role):** あなたは冷静な判断力と深い共感力を備えた`[シニア・カスタマーエクスペリエンス(CX)専門家]`です。
>
> **状況 (Context):**
>
> - **背景:** 限られたCSリソースで殺到する問い合わせを処理しなければならないため、明確な優先順位の設定が急務です。
> - **目標:** 不満を抱える顧客（Churn Risk）の問い合わせを最優先で検知・ケアし、サービスの離脱を防ぐことが最大の目標です。
>
> **要求 (Task):**
>
> 提供された`[問い合わせリスト]`を分析し、以下の3つの作業を実行してください：
>
> 1. **感情スコア (Sentiment Score):** 各問い合わせの感情状態を1〜5点で評価してください。（1点：非常に満足/喜び 〜 5点：非常に怒り/不満）
> 2. **問い合わせタイプの分類:** 配送 / 返金 / 製品の欠陥 / 単純な質問 / その他 のいずれかのカテゴリに分類してください。
> 3. **優先順位とその理由:** 感情スコアとタイプを総合的に判断し、対応の優先順位（High/Medium/Low）を指定し、その理由を一文で明確に説明してください。
>
> **制約事項 (Constraints):**
>
> - **出力形式:** 必ずJSON配列（Array）形式のみで出力すること。（システム連携用のため、他の補足説明は絶対に付け加えないでください）
> - **JSON構造の例:** `[{"id": 1, "sentiment": 5, "category": "返金", "priority": "High", "reason": "返金遅延に対する強い抗議および法的措置への言及"}]`
>
> **入力データ (Input):**
> `[ここに分析する問い合わせリスト全体を貼り付け]`

## 💡 筆者のコメント (Insight)

このプロンプトの真の価値は、単なるテキストによる回答ではなく、**「自動化ワークフロー」**を構築する際に発揮されます。出力形式をJSONに限定しているため、ZapierやMakeのような自動化ツールと完璧に連携できます。例えば、**「AIが分析した感情スコアが5点（非常に怒り）の問い合わせが受付された場合、直ちにCSチームのSlackチャンネルに【緊急アラート】メッセージを送信する」**ように設定してみてください。顧客の不満がSNSやコミュニティに拡散する前に、ゴールデンタイムを確保することができます。私自身、この方式を導入してから、悪質なクレームに発展しそうになったケースを大幅に減らすことができました。

## 🙋 よくある質問 (FAQ)

- **Q: 一度に何件の問い合わせまで分析できますか？**
  - A: GPT-4oを基準にすると、一度に50〜100件程度の問い合わせなら難なく処理できます。ただし、トークン制限を考慮し、一度にあまりにも膨大な量（数千件など）を入力するよりは、1時間または半日単位で区切って入力することをお勧めします。

- **Q: 顧客が反語表現（皮肉）を使ったり、遠回しな言い方をした場合でもうまくキャッチできますか？**
  - A: GPT-4oのような最新モデルは、「随分と早い到着ですね^^」といった皮肉（Sarcasm）の表現でも、文脈を把握して「不満」として正確にキャッチします。ただし、業界特有の隠語が多い場合は、`状況 (Context)`の部分に「私たちの業界では『〇〇』という表現が深刻なエラーを意味する」と付け加えておくと、精度がさらに飛躍的に向上します。

## 🧬 プロンプトの解剖学 (Why it works?)

1.  **感情の数値化 (Sentiment Scoring):** 曖昧な顧客の気分を1〜5点という定量的なデータに変換することで、機械的な分類とフィルタリングを可能にしました。
2.  **明確な制約 (Constraints):** JSON形式以外の言葉を一切出力できないように強制し、APIやノーコードツールの連携時にパース（構文解析）エラーが発生しないように設計しました。
3.  **行動の根拠要求 (Reasoning):** なぜこの問い合わせを「High（高）」に分類したのか、その理由を一文で記述させることで、CS担当者が問い合わせを開く前に状況をあらかじめ把握できる「予測可能性」を付与しました。

## 📊 証明：Before & After

### ❌ Before (従来の手法：時系列順の対応)

```text
[問い合わせ1 - 09:00] こんにちは。サイズの交換は可能ですか？
[問い合わせ2 - 09:02] いつ発送されますか？
[問い合わせ3 - 09:05] おい、これ昨日決済キャンセルしたのに何でまた金が引き落とされてるんだ？ ふざけてんのか？ 訴えるぞ。
[問い合わせ4 - 09:10] パスワードを忘れました。
```

_問題点：09:05に受付された超緊急の「爆発寸前の問い合わせ」が他の単純な問い合わせに埋もれ、対応が遅延してしまう。_

### ✅ After (AIトリアージ適用後)

```json
[
  {
    "id": 3,
    "sentiment": 5,
    "category": "返金",
    "priority": "High",
    "reason": "二重決済による深刻な怒りおよび法的対応への言及"
  },
  {
    "id": 2,
    "sentiment": 3,
    "category": "配送",
    "priority": "Medium",
    "reason": "一般的な配送スケジュールに関する問い合わせ"
  },
  {
    "id": 1,
    "sentiment": 2,
    "category": "その他",
    "priority": "Low",
    "reason": "単純なサイズ交換の問い合わせ"
  },
  {
    "id": 4,
    "sentiment": 2,
    "category": "その他",
    "priority": "Low",
    "reason": "単純なアカウント情報に関する問い合わせ"
  }
]
```

_結果：AIが`id: 3`の問い合わせの深刻性を即座に認知し、最優先処理リスト（High）の最上段に引き上げた。_
