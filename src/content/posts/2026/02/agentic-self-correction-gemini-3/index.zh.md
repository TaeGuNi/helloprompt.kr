# 📝 为什么你的 AI Agent 总是失败：Gemini 3 Pro 的“自我修正循环”提示词模式

- **🎯 推荐对象：** 开发 AI Agent 的后端工程师、提示词工程师、产品经理
- **⏱️ 所需时间：** 代码调试 15分钟 → 缩短至 1分钟
- **🤖 推荐模型：** Gemini 3 Pro, GPT-4o, Claude 3.5 Sonnet (具备强大推理能力的模型)

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐☆

> *"看似完美的 AI Agent 是否总是在重复同样的错误，白白燃烧你的 API Token？现在，是时候停止仅仅指示它“编写代码”，而是要求它“编写、批判、修正”了。"*

如果你曾尝试使用 Gemini 3 Pro 或 GPT-4 等高性能模型构建 Agent，你一定有过这样的体验：它们生成代码的能力确实卓越，但在自行修复 Bug 方面却往往糟糕透顶。一旦尝试失败，它们就会不断吐出相同的错误代码，陷入所谓的“死亡螺旋 (Death Spiral)”，白白浪费宝贵的 Token 预算。

这个问题并非源于模型的智力不足，而是因为缺失了关键的**“自我修正循环 (Self-Correction Loop)”**。大多数开发者在对待 Agent 时，采用的是“发射后不管 (Fire and Forget)”的模式。然而，在一个真正的 Agentic Workflow（智能体工作流）中，模型在将结果呈现给用户之前，必须经历一个自我批判的“编辑 (Editor)”阶段。

今天，我将分享一个在实际生产环境中，将 Agent 错误率大幅降低 60% 以上的**自我修正提示词模式 (Self-Correction Prompt Pattern)**。

---

## ⚡️ 3行核心摘要 (TL;DR)

1. AI Agent 陷入无限错误循环并非模型的极限，而是缺乏有效的验证环节。
2. 同时赋予 AI “创作者 (Creator)”和“批评家 (Critic)”的双重角色，迫使其自行寻找逻辑漏洞。
3. 将起草、自我批判、最终修正这三个步骤融入单一提示词中，可以极大地减少幻觉 (Hallucination)。

---

## 🚀 解决方案：“批评家-执行者 (Critic-Actor) 模式”

该提示词的核心在于明确区分“起草 (Drafting)”和“批判 (Critique)”这两个阶段。

### 🥉 Basic Version (基础版)

当你只需要快速获取自我修正后的结果时使用。

> **角色：** 你是一名 `[资深后端工程师]`。
> **任务：** 请编写一个 `[合并两个已排序列表的函数]`。在编写代码后，请自行找出潜在的 Bug 或低效之处进行批判，并基于这些批判提供修改后的最终代码。

\

### 🥇 Pro Version (专业版)

当你需要严谨的代码质量并防御各种边缘情况 (Edge Case) 时使用。

> **角色 (Role)：** 你是一名拥有 10 年经验的资深 Python 后端工程师，也是一名严谨的代码审查员 (Code Reviewer)。你始终将编写整洁、高效且无 Bug 的代码作为首要原则。
>
> **背景 (Context)：**
>
> - 现状：在当前的生产环境中，我们需要极其可靠的代码来处理海量数据。
> - 目标：编写一个完美的函数，能够自行发现代码中的逻辑漏洞并加以完善。
>
> **任务 (Task)：**
>
> 1. **起草 (Draft)：** 编写一个 `[合并两个已排序列表的 Python 函数]`。
> 2. **批判 (Critique)：** 对你刚刚起草的代码进行自我审查。重点排查以下几个方面：
>    - 边缘情况（例如：输入空列表、包含负数等）
>    - 性能瓶颈（时间/空间复杂度）
>    - 潜在的运行时错误 (Runtime Error)
> 3. **修正 (Refine)：** 基于在批判阶段发现的问题，彻底重写并完善你的代码。
>
> **约束条件 (Constraints)：**
>
> - 输出格式必须严格遵循下方提供的 Markdown 结构。
> - 在“3. 最终完美代码”部分，绝对不要添加任何对代码的解释说明。只提供代码块。
> - 最终完美代码中，必须包含针对批判阶段识别出的边缘情况的异常处理逻辑。
>
> **输出格式 (Format)：**
>
> ## 1. 初始草稿 (Initial Draft)
>
> \`\`\`python
> (草稿代码)
> \`\`\`
>
> ## 2. 自我批判 (Self-Critique)
>
> - **批判 1：** (发现的问题及原因)
> - **批判 2：** (发现的问题及原因)
>
> ## 3. 最终完美代码 (Final Polished Code)
>
> \`\`\`python
> (最终修正后的代码)
> \`\`\`

## 💡 作者见解 (Insight)

这种提示词模式不仅适用于代码生成，在编写 SQL 查询、撰写商业邮件文案等任何需要逻辑验证的场景中，都能发挥出强大的威力。“自我批判 (Self-Critique)”阶段就像一道坚固的防波堤，有效阻挡了 AI 产生毫无意义的幻觉 (Hallucination)。

特别是在使用像 **Gemini 3 Pro** 这样拥有庞大上下文窗口的模型时，如果在批判阶段指示它直接引用现有代码库中的特定代码行进行审查，可以将代码的准确率和上下文契合度推向极致。

但需要注意的是：由于这个过程包含了生成“错误的草稿”和“批判意见”等额外文本，相比常规提示词，它会消耗大约两倍的 Token（即成本）。因此，我强烈建议将此模式应用于**“准确度远大于成本 (Accuracy > Cost)”的复杂逻辑处理**中，而不是简单的文本转换任务。根据我的实战经验，在此过程中节省下来的调试时间，其价值远远超过了多支出的 Token 成本。

## 🙋 常见问题 (FAQ)

- **Q: 我有点担心 Token 的消耗量。有办法降低成本吗？**
  - A: 在处理内部逻辑时，可以构建多智能体路由 (Multi-Agent Routing) 架构。比如，先使用成本较低的模型（如 Gemini 3 Flash）撰写草稿，然后只将“批判与修正”的审查工作交给 Gemini 3 Pro。这样不仅能保持极高的输出质量，还能大幅削减成本。

- **Q: 如果 AI 在批判阶段也没能发现问题怎么办？**
  - A: 在提示词的 Task 部分注入具体的检查清单 (Checklist)。例如，明确指出“是否存在内存泄漏可能”、“检查异步处理中的死锁”、“防御 SQL 注入漏洞”等。当你为 AI 指明审查的聚焦点时，其批判的敏锐度将得到质的飞跃。

- **Q: 这个模式可以用于代码以外的产品策划或博客写作吗？**
  - A: 当然可以。将其应用为“撰写初稿 -> 批判逻辑矛盾与目标受众不符之处 -> 最终文稿修正”的结构，即使没有人类的干预，你也能获得如同专业编辑润色过的顶级文案。

## 🧬 提示词解剖 (Why it works?)

1.  **角色与背景分离 (Role & Context)：** 强制赋予 AI“开发者”和“审查员”两种对立的人设，激活了它寻找自身代码缺陷的元认知 (Meta-cognition) 能力。
2.  **引导思维链 (Chain-of-Thought)：** 不要求它立刻给出完美答案，而是通过 Markdown 结构将“起草 -> 批判 -> 修正”的思考过程显性化，有效防止了逻辑跳跃和胡言乱语。
3.  **约束条件控制 (Constraints)：** 强制要求在最终输出中剔除所有多余解释，这一设计是为了方便后续在自动化流水线 (CI/CD 等) 中直接解析 (Parsing) 并测试代码。

## 📊 效果验证：Before & After

**测试条件：** “编写一个合并两个已排序列表的函数”

### ❌ Before (输入)

当使用标准提示词进行指示时，模型仅仅生成了先 `list1 + list2` 然后调用 `sorted()` 的低效代码。

```python
def merge_lists(list1, list2):
    # 导致 O(N log N) 的时间复杂度。在处理海量数据时会引发严重的性能下降。
    return sorted(list1 + list2)
```

### ✅ After (结果)

应用自我修正提示词后，模型在自我批判阶段主动指出：“对已经排序的列表使用 `sorted()` 是极其低效的，为了达到 O(N) 的复杂度，必须使用双指针 (Two-pointer) 方法”。随后，它给出了完美优化的代码。原本需要 15 分钟的代码审查和重构时间，被瞬间压缩到了 1 秒。

```python
def merge_lists(list1, list2):
    # 以 O(N) 的时间复杂度完成优化，完美防御空列表等边缘情况
    merged = []
    i, j = 0, 0

    while i < len(list1) and j < len(list2):
        if list1[i] < list2[j]:
            merged.append(list1[i])
            i += 1
        else:
            merged.append(list2[j])
            j += 1

    # 合并剩余的元素
    merged.extend(list1[i:])
    merged.extend(list2[j:])

    return merged
```
