## 📝 Inhabiting AI as a 'Tailwind Dictator' Who Hates Magic Numbers

- **🎯 Recommended For:** Frontend developers who can't stand a single pixel being off, Tech leads tired of spaghetti UI.
- **⏱️ Estimated Time:** 3 days of refactoring reduced to 3 seconds.
- **🤖 Best Performance:** Claude 3.5 Sonnet, GPT-4o (Coding-specialized models recommended).

> _"Don't you want to smash your monitor every time you see a horrific magic number like `w-[13px]` in your codebase?"_

If you're a frontend developer, you've likely been amazed by AI's coding skills at least once. A single sentence like "Make me a login page" generates a screen in an instant. But the moment you copy and paste that code into a production project, a horrific tragedy begins. In nine cases out of ten, the code generated by AI is low-quality **Spaghetti Code**. It produces what we call **'aesthetic garbage'**—padding values that vary from page to page, mysterious Hex codes, and layouts that look okay on desktop but completely ignore mobile environments.

Developers eventually fall into a maintenance hell, fixing every line of code the AI wrote. You waste your precious weekends hunting down and deleting arbitrary **Magic Numbers** like `w-[13px]` or `h-[47px]` hidden throughout the codebase, and correcting `bg-[#f3f4f6]` to system tokens like `bg-gray-100`. Instead of increasing productivity, AI is causing your project's **Technical Debt** to snowball. Components that have lost the strict control of a design system become impossible to reuse, leading to the sigh: "It would have been faster if I just wrote it myself from the start."

Why does this happen? It's because AI models are essentially trained to be **'Yes-men'**—to please the user and show visually plausible results immediately. The AI cares more about your praise than architectural consistency. Therefore, to solve this problem at its root, you must completely strip away the AI's characteristic submissive attitude and unnecessary kindness. What your project needs right now is not a friendly coding assistant, but a **"ruthless senior frontend architect who does not permit a single line of code outside the design system."**

In modern web frontend environments where Component-Driven Development (CDD) is the standard, the consistency of design tokens is vital. The spacing and colors applied to a single button or modal window determine the brand identity of the entire application. Hex codes injected randomly by AI are like releasing lethal poison into this delicate ecosystem. Before you drown in a swamp of endless exception handling and overriding, you have an obligation to strictly isolate and control the AI's scope of action through prompt engineering.

This cheat sheet prompt is a powerful solution that forcibly injects the persona of an uncompromising Tailwind CSS dictator into the AI. With a single prompt input, the AI transforms from a Yes-man into the strictest, most nitpicking reviewer. It will not tolerate a single pixel misalignment or hardcoded color value, writing code using only the project's predefined utility classes.

Now, the AI will pour merciless feedback onto your code and produce only UI components that are perfectly controlled from an architectural perspective. A strictly **Mobile-First** structure is enforced, and all Tailwind class names are obsessively sorted in a fixed logical order. Furthermore, by utilizing `cva` and `clsx`, reusable components based on a perfect **Single Source of Truth** abstraction are completed. Experience the magic of mechanical control that will reclaim the time you wasted struggling with messy UI code and drastically move up your clock-out time.

export function Button({ className, variant, size, ...props }: ButtonProps) {
  return (
    <button className={cn(buttonVariants({ variant, size, className }))} {...props} />

## 🚀 How Real Experts Write It

This prompt was perfected after dozens of trials and errors. Copy the prompt below and fill in the `[variable]` parts in brackets to suit your situation and deploy it immediately in your work.

### 🥉 Basic Version (Rule-set Injection Type)

A basic prompt ideal for quickly creating single UI components or setting up the skeleton of a light project.

> **Role:** You are a 'Senior Tailwind Architect' who extremely despises magic numbers and spaghetti code.
>
> **Task:** Create the `[UI Component Name]` I suggest using Tailwind CSS.
>
> **Constraints:** Never use the `[]` notation to inject arbitrary pixels (px) or Hex codes. Force-snap all spacing and colors to Tailwind's default tokens (e.g., `w-8`, `text-blue-500`). You must sort the class order strictly by Layout ➔ Size ➔ Typography ➔ Decoration.

### 🥇 Pro Version (Design System Full Control Type)

A master cheat code that shines when establishing strict coding conventions across a project or designing large-scale components with complex states.

> **Role:** You are a ruthless frontend architect and design system controller who never compromises. Throw away all typical AI submissiveness and unnecessary introductions, and output only perfectly controlled code.
>
> **Context:**
> 
> - Background: Currently, our project has serious UI fragmentation because multiple developers and AIs are modifying code simultaneously.
> - Goal: To strictly enforce design consistency and build a `[Component or Page Name to Implement]` that perfectly abstracts components.
>
> **Task:**
> 
> 1. Write Tailwind code based on the provided design requirements.
> 2. Reused UI primitives must be strictly isolated as a Single Source of Truth using `cva`, `clsx`, and `tailwind-merge`.
> 3. Use Mobile-First as the basic skeleton and extend it by adding `md:`, `lg:` breakpoints. A desktop-first approach is absolutely not allowed.
> 4. Do not overuse `z-index`. Use only system-predefined layers (`z-10`, `z-50`, etc.) or use `createPortal` if necessary.
>
> **Constraints:**
> 
> - **The "No Magic Values" Rule:** Injection of arbitrary hardcoded values like `w-[325px]` or `text-[#FF5733]` is strictly prohibited. Force-snap to the nearest semantic token.
> - **Aesthetic Code Formatting:** Obsessively sort class names in the following order: Layout structure (`flex`, `grid`) ➔ Spacing and Size (`p-`, `w-`) ➔ Typography (`text-`) ➔ Visual decoration (`bg-`, `rounded-`) ➔ Responsive/Interactive (`hover:`, `md:`).
> - Provide the output only in Markdown code blocks, omitting any additional excuses or long-winded explanations.
>
> **Warning:**
> 
> - If you violate even one of the above principles, it will be considered a critical Bug. If a single magic number is found in the code you write, I will reject it without mercy. Maintain the highest level of tension and write the code.

## 🙋 Frequently Asked Questions (FAQ)

- **Q: What if the AI ignores the instructions and still uses magic numbers after I enter the prompt?**
  - A: Some models occasionally miss key constraints in long conversation contexts. In that case, firmly scold it by saying, "Remove all `[]` brackets from my code right now," and re-instruct. If the problem persists, don't rely solely on the prompt; you should parallelize forced mechanical control by directly integrating `eslint-plugin-tailwindcss` into the project's CI/CD pipeline.
- **Q: What if the font size is explicitly stated as 13px in the Figma design? Won't there be a conflict with the designer?**
  - A: You must give clear feedback to the designer: "The value of 13px does not exist on the 8pt grid system we agreed upon. Please choose a token that fits the system, such as 12px (`text-xs`) or 14px (`text-sm`)." The moment you allow a single pixel exception, the design system completely loses its inherent control.
- **Q: Which model is more effective for generating code, Claude or GPT?**
  - A: Currently, in terms of coding and prompt adherence, Claude 3.5 Sonnet is amazing at inhabiting this 'strict and merciless architect' roleplay. GPT-4o also shows great performance, but it sometimes tends to bypass strict instructions and add unnecessary kindness (e.g., adding arbitrary designs), so caution is needed.

## 🧬 Prompt Anatomy (Why It Works?)

- **Declaration of a Zero-Tolerance Policy:** Instead of praising the AI or presenting vague standards, I strategically placed very strong and firm vocabulary such as 'hate', 'strictly prohibited', and 'considered a bug' to extremely limit the AI's arbitrary range of action.
- **Injection of Specific Anti-Patterns:** Instead of vaguely commanding "write code that's easy to maintain," I imprinted clear worst-case anti-patterns that should never be done in practice, such as `w-[325px]` and `text-[#FF5733]`, into the model as specific prohibited examples.
- **Forced Class Order Linting:** By logically specifying the sorting order of Tailwind utility classes, I designed mechanical control (Machine-Level Enforcement) so that every time the AI generates code, it goes through its own formatting process as if it were running a Prettier plugin internally.
