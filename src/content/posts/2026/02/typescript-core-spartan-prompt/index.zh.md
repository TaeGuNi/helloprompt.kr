- 🎯 **推荐对象:** 厌倦了AI只会说“好的，没问题！”的资深开发者、痛恨意大利面条式代码的技术负责人
- ⏱️ **耗时:** 1小时调试 → 1秒搞定
- 🤖 **推荐模型:** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro (编程特化模型)
- ⭐ **难度:** ⭐⭐⭐⭐☆
- ⚡️ **有效性:** ⭐⭐⭐⭐⭐
- 🚀 **实用度:** ⭐⭐⭐⭐⭐
_"AI是不是经常还在用老旧的 npm install 或者满屏的 any 类型？用这个作弊码，无情地粉碎它们那些陈旧的坏习惯吧。"_
当你把写代码的任务交给AI时，看到它用过时的配置和松散的类型搞出一个遗留系统般的垃圾堆，是不是有种想砸显示器的冲动？这篇文章就是为你们准备的救星。使用这个硬核提示词，AI会收起那些无用的辞藻，化身为一名斯巴达战士，为你的项目引入**极限的 I/O 效率 (pnpm)、无情的类型安全 (Zod/Strict)、以及极速的代码检查 (Biome) 与测试 (Vitest)**。
---
## ⚡️ 3句话总结 (TL;DR)
- 🚀 **绝对强制使用 `pnpm`:** 立即处决那些浪费磁盘空间和导致网络延迟的 npm/yarn。
- 🛡️ **无情的类型控制:** 使用 `any` 就是叛国。强制使用 `zod` 进行运行时验证和 `strict: true`。
- ⚡ **引入 Biome & Vitest:** 把慢吞吞的 eslint, prettier, jest 扔进垃圾桶，以光速执行代码检查和测试。

> **角色:** 你是一个毫不妥协的硬核15年经验资深 TypeScript 架构师。
> **请求:** 请解决 `[需要解决的问题及重构的代码]`。绝对禁止使用 npm，必须使用 pnpm；禁止使用 any，并使用 zod 进行验证。代码检查工具配置为 Biome，测试工具配置为 Vitest。

制作这个提示词的初衷非常简单。我实在无法眼睁睁地看着AI家伙们为了图省事，总是敲出老掉牙的 `npm install`，一遇到类型报错就想用 `any` 随便糊弄过去的样子。（真的气得我想砸显示器 🤬）

## 🙋 常见问题 (FAQ)
- **Q: AI 有时还是固执地想用 `npm`，怎么办？**
  - A: 这种时候，可以在提示词里再加一句带有威胁(?)性质的备注：“敢再用一次 npm，我就强制结束你的进程”。听起来像开玩笑，但真的非常有效。
- **Q: 在前端 (React/Next.js) 开发中，这个提示词也管用吗？**
  - A: 当然！只要是沾了 TypeScript 的地方，无论是 Web 还是服务端，都可以适用这套无情的规则。
- **Q: 可以用 Yup 代替 Zod 吗？**
  - A: 你非要用类型推断那么糟糕的库吗？我不会拦着你，但这个提示词的哲学就是“极限的类型安全”。还是乖乖用 Zod 吧。

## 🧬 提示词解剖 (Why it works?)
- 🎯 **角色设定极致化:** 给AI植入了一个“硬核资深架构师”的严苛人格，从源头上杜绝了它编写那种“只要能跑就行”的敷衍代码。
- 🚫 **明确的禁用词 (Negative Prompt):** 准确指出了 `npm`, `yarn`, `any`, `eslint`, `jest` 等AI习惯性输出的旧时代工具，并予以禁止。
- 🔒 **引导防御性编程:** 通过强制使用 `zod`，在架构层面上限制了AI，让它学会主动防御运行时可能引爆的“地雷”。
