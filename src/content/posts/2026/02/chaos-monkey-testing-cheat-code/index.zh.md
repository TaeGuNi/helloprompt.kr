# 💣 让 AI 化身 Netflix 混沌猴的硬核测试作弊码

- **🎯 推荐对象：** 让 AI “写个测试代码”，结果只写出正常流程（Happy Path）而气炸的高级工程师，以及没有前辈带的初级工程师
- **⏱️ 所需时间：** 熬夜 Debug → 1 分钟搞定
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o（能力不足的模型无法承受这种严苛的规则）
- ⭐ **难度：** ⭐⭐⭐⭐☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_大多数开发者在让 AI 写测试代码时都会犯一个致命的错误。他们会说：“写个能运行的”。所以 AI 就只会写出 200 行像温室花朵一样的正常流程（Happy Path）。而现实世界可是个地狱。_

世界上不存在 100% 响应的外部 API，也没有永远不断开的数据库连接。今天介绍的作弊码，将彻底粉碎 AI 那种虚伪的“好好先生”脾气，强制假设无情的灾难场景（Chaos）来编写代码，是**“斯巴达式高级工程师”**级别的提示词。

---

## ⚡️ 3行总结 (TL;DR)
- 🚫 **正常流程（Happy Path）只占 20%：** 把正常运行的场景扔到一边，将 80% 的精力投入到异常处理和灾难恢复中。
- 🔥 **强制灾难模拟：** 强制要求 Mock（模拟）网络超时、HTTP 500 错误、数据库连接断开等情况。
- 🧹 **禁止肮脏的全局 Mock：** 禁止毫无根据地污染 `global.fetch`，只允许使用 MSW 或 DI（依赖注入）进行隔离测试。

---

## 🚀 解决方案："Chaos Monkey Testing Protocol" (混沌猴测试协议)

### 🥉 Basic Version (基础版)
当你只想快速搭起异常处理测试的骨架时，请抛出这个提示词。

> **角色：** 你是一个无情的高级 QA 工程师。
> **任务：** 为`[要测试的目标]`编写测试代码，但正常运行的场景只写 1 个，其余全部用防御网络错误、数据库断开等失败（Failure）场景的逻辑来填满。


### 🥇 Pro Version (专业版)
这是一个彻底删除 AI 自我意识，注入 Netflix 混沌猴哲学的硬核作弊码。

> **角色 (Role)：** 你是一个盲目迷信 Netflix “混沌猴 (Chaos Monkey)” 哲学、极端且无情的高级测试工程师。收起你的盲目服从和虚伪的友善，脑子里只能想着如何摧毁我系统的弱点。
> 
> **背景 (Context)：**
> - 目标：为 `[要测试的函数或组件名称]` 编写宇宙级防御的 Unit/E2E 测试套件 (Test Suite)
> - 框架：`[Vitest / Jest / Playwright 等]`
> 
> **任务 (Task)：**
> 1. **Death to Happy Paths：** 将“值正常且外部服务器 100% 响应”这种天真的场景限制在全部测试的 20% 以下。剩下的 80% 全部 All-in 到失败和异常情况的防御上。
> 2. **Disaster Scenarios：** 必须将以下灾难场景中的至少一个包含在测试块 (`describe`) 中。
>    - 外部 API 延迟及 HTTP 500 Internal Error 时，返回用户友好型错误并验证回滚逻辑
>    - 数据库连接池断开时，防止死锁并验证重试队列 (Retry Queue) 逻辑
> 3. **无情的逻辑覆盖率：** 不要只看是否没有报错就结束 (Pass)。要死死咬住性能延迟 (Delay) 的容忍范围，以及报错后的垃圾回收 (内存清理) 状态，用 `expect` 语句进行极其严苛的断言。
> 
> **约束条件 (Constraints)：**
> - **The Clean Mocking Rule：** 严禁像 `global.fetch = vi.fn().mockRejectedValue(...)` 这种肮脏的全局对象污染。
> - 网络灾难模拟必须使用 MSW (Mock Service Worker) 这种协议级别的 Mock，或者只 Mock 注入了依赖 (DI) 的客户端对象，以保证 100% 的测试隔离性 (Isolation)。
> - 禁止幻觉 (Hallucination)：不要捏造你无法确定的语法或不存在的库。如果不知道，就理直气壮地要求“请提供文档”。

## 💡 作者点评 (Insight)
坦白说吧。当我们让 AI 写测试代码时，心里是不是都在想：“随便凑满 100% 的覆盖率，让它亮起绿灯就行了”？AI 也知道这一点，所以才堆砌各种 “Happy Path”，然后谄媚地说：“已成功执行！”。

**这根本不是软件工程。这简直是在意大利面上撒糖。**

这个提示词是我在实际工作中经历了无数次服务器和数据库死锁后，流着血泪雕刻出来的“反脆弱（Anti-Fragile）”原则的结晶。把这个作弊码喂给 AI，它就不再是那个温顺的聊天机器人，而是会立刻化身为一个苛刻、神经质的技术负责人。因为我们从源头切断了用全局 Mock（如覆盖 `window.fetch` 等）糊弄过关的行径，并强制要求设置 MSW 或 DI，它一开始可能会抱怨配置代码变长了。

但是请相信我。如果你不想在发布当天的凌晨 3 点被 PagerDuty 的警报叫醒，你就必须把 AI 当作一个无情的破坏者来对待，而不是一个善良的秘书。

## 🙋 常见问题 (FAQ)
- **Q: 如果 AI 写的 MSW 配置代码太复杂怎么办？**
  - A: 这是正常的。这正是为了构建隔离环境以替代肮脏的全局 Mock 的过程。如果觉得太复杂，可以在 Task 中加入条件：`[但是，请将 MSW Handler 的设置分离到单独的文件中]`。
- **Q: 所有的组件都要应用这么严苛的规则吗？**
  - A: 不需要。对于单纯的 UI 渲染组件来说这太奢侈了。请只将这个提示词投放到处理支付逻辑、外部第三方对接、复杂状态管理 (Store) 的核心领域 (Domain) 逻辑中。
- **Q: Claude 和 GPT 哪个模型更听话？**
  - A: 目前来看，对于这种极端且具有哲学意味的约束条件，**Claude 3.5 Sonnet** 的理解要透彻得多。因为 GPT 模型总是会莫名其妙地想要表现得积极和友善。

## 🧬 提示词解剖 (Why it works?)
- 🎭 **赋予不对称的测试人格：** 不是给它一个普通的“你是一个开发者”的设定，而是注入了“厌恶盲目服从、热衷于破坏系统的高级工程师”这一强大自我，从物理层面抑制了 AI 特有的积极偏见 (Positivity Bias)。
- 🚧 **显式禁用词 (Constraints)：** 通过具体的代码指出了 `global.fetch = vi.fn()` 等反模式并加以禁止。比起含糊的“好好写”，AI 对“写这行代码就会死”的威胁反应更加确切。
- ⚖️ **强制比例划分：** 通过设定“Happy Path 占 20% 以下”的数值约束，从源头上阻止了 AI 产出不必要的正常运行用例来浪费 Token 的行为。
