# 📝 알 수 없는 에러 로그, 10초 만에 원인 분석하기

- **🎯 추천 대상:** 주니어 개발자, 시스템 엔지니어, 1~3년 차 백엔드/프론트엔드 개발자
- **⏱️ 소요 시간:** 1시간 → 3분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet (코드 분석에 강력함), GPT-4o

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"수천 줄의 빨간색 Stack Trace가 화면을 뒤덮었을 때의 아찔함... Stack Overflow를 헤매기 전, AI 시니어 개발자에게 먼저 로그를 던져보세요."_

서버 로그나 프론트엔드 빌드 에러 메시지는 불친절하기 짝이 없습니다. 하지만 그 수많은 텍스트 더미 속에는 반드시 버그를 잡을 '단서'가 숨어 있습니다. AI는 수백만 건의 오픈소스 이슈와 로그 패턴을 학습했기 때문에, 인간이 눈으로 찾는 것보다 훨씬 빠르고 정확하게 '진짜 문제(Root Cause)'를 짚어냅니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. 수천 줄의 복잡한 에러 로그를 직접 읽으며 시간 낭비하지 마세요.
2. AI에게 '시니어 엔지니어' 역할을 부여하고 로그를 통째로 던지면, 10초 만에 핵심 원인을 요약해 줍니다.
3. 단순한 원인 파악을 넘어, 내 코드에 바로 적용할 수 있는 구체적인 해결 코드(Snippet)까지 제안받을 수 있습니다.

## 🚀 해결책: "버그 헌터 (Bug Hunter)"

### 🥉 Basic Version (기본형)

빠르게 원인만 짚고 넘어가고 싶을 때 사용하세요.

> **역할:** 너는 `[10년 차 시니어 백엔드 개발자]`야.
> **요청:** 아래 에러 로그의 핵심 원인이 무엇인지 한 줄로 요약하고, 해결 방법을 알려줘.
> **에러 로그:** `[복사한 에러 로그 붙여넣기]`

<br>

### 🥇 Pro Version (전문가형)

단순한 원인 파악을 넘어, 원리 이해와 구체적인 해결 코드까지 필요할 때 사용하세요.

> **역할 (Role):** 너는 트러블슈팅에 능통한 `[10년 차 시니어 시스템 엔지니어 및 개발자]`야.
>
> **상황 (Context):**
>
> - 배경: 로컬(또는 운영) 환경에서 애플리케이션을 실행(또는 빌드)하던 중 `[치명적인 에러]`가 발생했어.
> - 목표: 수많은 스택 트레이스(Stack Trace) 속에서 진짜 원인(Root Cause)을 찾고 버그를 픽스하는 것.
>
> **요청 (Task):**
>
> 1. 제공된 에러 로그를 분석하여, 가장 핵심적인 원인(Root Cause)을 개발자가 이해하기 쉽게 **한 문장**으로 요약해 줘.
> 2. 이 에러가 왜 발생했는지, 기술적 배경이나 작동 원리를 간략히 설명해 줘.
> 3. 이 문제를 당장 해결할 수 있는 구체적인 Action Item(코드 수정, 설정 파일 변경, 패키지 설치 등)을 3단계 이내로 제시해 줘. 코드가 필요하다면 코드 스니펫도 작성해 줘.
>
> **제약사항 (Constraints):**
>
> - 두루뭉술한 조언은 피하고, 로그에 나타난 파일명이나 변수명을 콕 집어서 구체적으로 설명해.
> - 원인을 확신할 수 없다면, 원인 파악을 위해 내가 추가로 확인해야 할 부분(디버깅 로그 추가, DB 연결 확인 등)을 역으로 질문해 줘. (환각 방지)
>
> **에러 로그 (Error Log):**
> `[여기에 복사한 에러 로그 전체를 붙여넣으세요]`

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 제가 실무에서 `NullPointerException`이나 알 수 없는 Webpack 빌드 에러를 만났을 때 매일같이 사용하는 치트키입니다.
단순히 "이 에러 뭐야?"라고 묻는 것보다, "시니어 개발자"라는 역할을 부여하고 "해결책을 3단계로 제시해"라고 포맷을 지정하면 AI의 답변 퀄리티가 비약적으로 상승합니다. 특히 Claude 3.5 Sonnet 모델과 함께 사용했을 때, 복잡한 컨텍스트를 파악하고 정확한 코드 수정 위치를 짚어내는 능력이 탁월했습니다. 로그가 너무 길어서 복사가 힘들다면, 에러의 시작 지점과 끝 지점 100줄 정도만 잘라서 넣어도 충분히 문맥을 파악해 냅니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 회사 내부 보안 로그나 API 키가 포함되어 있으면 어떡하나요?**
  - A: **(매우 중요)** 절대 그대로 붙여넣으면 안 됩니다! 프롬프트를 입력하기 전에 메모장에서 API 키, 비밀번호, 민감한 고객 정보, 내부 서버 IP 등은 반드시 `***` 또는 `[마스킹]`으로 치환한 후 사용하세요.

- **Q: AI가 엉뚱한 해결책을 줍니다. 어떻게 해야 하나요?**
  - A: 에러 로그만으로는 정보가 부족할 때 발생합니다. 에러 로그와 함께, 에러가 발생한 주변부 소스 코드(메서드 전체 등)를 함께 첨부해 보세요. 정확도가 훨씬 높아집니다.

- **Q: 에러 로그 텍스트가 너무 길어서 한 번에 안 올라가요.**
  - A: 보통 Stack Trace의 가장 윗부분(최초 발생 원인) 또는 `Caused by:` 로 시작하는 부분이 핵심입니다. 중간의 반복되는 프레임워크 내부 로그는 생략하고 핵심 부분만 발췌해서 넣어주세요.

## 🧬 프롬프트 해부 (Why it works?)

1. **시니어 엔지니어 페르소나 (Role):** AI에게 전문가의 지식을 활성화시켜, 초보적인 답변 대신 깊이 있는 기술적 분석을 이끌어냅니다.
2. **역질문 유도 (Constraints):** "확신할 수 없다면 추가 확인 사항을 물어봐"라는 제약을 통해, 정보 부족 시 AI가 아무 말이나 지어내는 환각(Hallucination) 현상을 원천 차단했습니다.
3. **단계별 지시 (Task):** 원인 분석 -> 원리 설명 -> 해결 코드 제안으로 이어지는 Chain-of-Thought 방식을 적용해 완벽한 디버깅 흐름을 강제했습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

```text
Exception in thread "main" java.lang.NullPointerException
    at com.example.service.UserService.getUserFullName(UserService.java:23)
    at com.example.controller.UserController.doGet(UserController.java:45)
    ... (알 수 없는 100줄의 스택 트레이스)
```

### ✅ After (결과)

```text
1. 💡 핵심 원인 요약:
`UserService.java` 파일의 23번째 줄에서 참조한 사용자 객체(User)가 null 상태인데, 그 객체 내부의 `getFullName()` 메서드를 호출하려고 시도하여 발생한 `NullPointerException`입니다.

2. ⚙️ 기술적 배경:
데이터베이스에서 해당 ID의 사용자를 찾지 못했거나 객체 초기화가 누락된 상태에서, 메모리에 존재하지 않는 객체의 속성이나 메서드에 접근하려고 할 때 자바 JVM이 메모리 참조 오류를 발생시킵니다.

3. 🛠️ 해결 방법 (Action Item):
- 1단계: DB에서 데이터를 가져오는 `findById` 로직의 반환값 확인
- 2단계: `UserService.java` 23번 줄 앞에 null 방어 로직(Null Check) 또는 Optional 적용

// 제안하는 수정 코드 (UserService.java)
User user = userRepository.findById(userId);
if (user == null) {
    throw new CustomException("해당 사용자를 찾을 수 없습니다.");
}
String fullName = user.getFullName();
```
