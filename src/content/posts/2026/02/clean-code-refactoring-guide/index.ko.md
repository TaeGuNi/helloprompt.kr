# 📝 더러운 코드, 클린 코드로 1초 만에 리팩토링하기

- **🎯 추천 대상:** 주니어 개발자, 코드 리뷰를 앞둔 모든 프로그래머
- **⏱️ 소요 시간:** 1시간 → 1분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro (코딩 특화 모델 권장)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"나도 내가 짠 코드를 못 알아보겠어요... 이 스파게티를 어떻게 풀죠?"_

급하게 기능 구현에만 집중하다 보면 변수명은 산으로 가고, 함수 하나가 100줄을 훌쩍 넘어가는 끔찍한 경험, 다들 있으시죠? 동료의 코드 리뷰(PR) 알림이 두렵다면, **'클린 코드(Clean Code)'의 정석**대로 코드를 다듬어주는 AI 비서를 고용해 보세요. 이 프롬프트 하나면 가독성, 성능, 유지보수성까지 한 번에 챙기며 당당하게 코드를 푸시할 수 있습니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. 직관적이지 않은 변수/함수명을 의도에 맞게 자동 변경합니다.
2. 단일 책임 원칙(SRP)에 따라 거대한 함수를 작게 쪼개고 중복 코드를 제거합니다.
3. 해당 언어의 최신 모던 문법을 적용하여 코드를 세련되게 다듬습니다.

---

## 🚀 해결책: "클린 코드 리팩토러"

### 🥉 Basic Version (기본형)

빠르게 가독성만 개선하고 싶을 때 사용하세요.

> **역할:** 너는 20년 차 시니어 소프트웨어 엔지니어이자 '클린 코드(Clean Code)' 전문가야.
> **요청:** 다음 코드를 읽기 쉽고 유지보수하기 좋은 클린 코드로 리팩토링하고, 주요 변경 사항을 요약해줘: `[여기에 코드 입력]`

<br>

### 🥇 Pro Version (전문가형)

코드 리뷰를 단번에 통과해야 하는 상용 레벨의 디테일한 퀄리티가 필요할 때 사용하세요.

> **역할 (Role):** 너는 20년 차 시니어 소프트웨어 엔지니어이자 '클린 코드(Clean Code)' 전문가야.
>
> **상황 (Context):**
>
> - 배경: 기능은 정상적으로 작동하지만, 가독성이 심각하게 떨어지고 유지보수가 어려운 스파게티 코드 상태야. 변수명이 명확하지 않고 중복된 로직이 섞여 있어.
> - 목표: 동료들이 읽기 쉽고, 확장성이 뛰어나며, 코드 리뷰를 단번에 통과할 수 있는 우아한 코드로 개선하는 것.
>
> **요청 (Task):**
>
> 1. **네이밍 규칙 적용:** 변수와 함수 이름만 봐도 그 의도를 명확히 알 수 있도록 직관적으로 변경해.
> 2. **함수 분리 (SRP 준수):** 하나의 함수가 단 하나의 역할만 수행하도록(단일 책임 원칙) 작게 쪼개.
> 3. **중복 코드 제거 (DRY 규칙):** 반복되는 로직은 별도의 함수나 유틸리티로 안전하게 추출해.
> 4. **의미 있는 주석 추가:** 코드가 '어떻게' 동작하는지가 아니라, '왜(Why)' 그렇게 작성되었는지 비즈니스적 설명이 필요한 부분에만 핵심 주석을 달아.
> 5. **모던 문법 적용:** 해당 프로그래밍 언어의 최신 모던 문법과 컨벤션을 철저히 적용해.
>
> **입력 코드:**
>
> `[여기에 리팩토링할 코드를 붙여넣으세요]`
>
> **제약사항 (Constraints):**
>
> - 기존 코드의 비즈니스 로직과 입출력 결과(동작)는 **절대 변경하지 마.**
> - 리팩토링 전과 후의 주요 변경 사항과 그 근거를 마크다운 목록(List)으로 명확히 요약해서 설명해.
> - 출력은 반드시 '설명 블록'과 '마크다운 코드 블록'으로 철저히 분리해서 보여줘.
>
> **주의사항 (Warning):**
>
> - 불필요한 라이브러리 추가나 오버 엔지니어링을 피하고, 원본 코드의 의도를 최대한 존중해. 모르면 모른다고 답해.

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 특히 레거시 코드를 인수인계받았을 때 빛을 발합니다. 단순히 코드를 짧게 줄이는 것이 아니라, SRP(단일 책임 원칙)와 DRY(반복 금지) 원칙을 AI에게 명시적으로 강제함으로써 훨씬 신뢰할 수 있는 결과물을 얻어낼 수 있습니다.

개인적인 경험으로는 프롬프트를 실행하기 전에 현재 프로젝트의 'Lint 설정(예: ESLint 규칙, Prettier)'이나 '팀 컨벤션(예: 카멜 케이스 사용, 함수형 프로그래밍 지향)'을 Context에 추가해주면, 별도의 수정 없이 바로 복사 및 적용할 수 있는 수준의 완벽한 코드를 얻을 수 있었습니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 비즈니스 로직을 망가뜨릴까 봐 걱정됩니다.**
  - A: 제약사항에 "비즈니스 로직 절대 변경 금지"를 강하게 명시했지만, AI 특성상 환각(Hallucination) 현상은 발생할 수 있습니다. 리팩토링 전후로 반드시 유닛 테스트(Unit Test)를 실행하여 동작의 무결성을 교차 검증하는 것을 강력히 권장합니다.

- **Q: 특정 프레임워크(React, Spring 등)에 맞춰 리팩토링할 수도 있나요?**
  - A: 네, 가능합니다. Context 부분에 "현재 React 18과 TypeScript를 사용 중이며, Custom Hook 패턴을 적극적으로 적용해줘"와 같이 구체적인 환경과 원하는 아키텍처 패턴을 덧붙이면 훨씬 전문적이고 완성도 높은 결과물을 얻을 수 있습니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **Role 부여:** '20년 차 시니어 엔지니어'이자 '클린 코드 전문가'라는 페르소나를 씌워, 단순한 코드 단축이 아닌 아키텍처 관점에서의 깊이 있는 리팩토링을 유도했습니다.
2. **구체적인 리팩토링 기준 제시 (Task):** 네이밍, 함수 분리, 중복 제거, 주석, 최신 문법이라는 5가지 명확한 기준을 세워 AI의 코드 수정 방향이 엉뚱한 곳으로 튀는 것을 막았습니다.
3. **안전장치 (Constraints & Warning):** 기능 변경을 엄격히 금지하고 오버 엔지니어링을 경고하여, 실무에서 바로 쓸 수 있는 안전하고 현실적인 코드를 출력하도록 강제했습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

```javascript
function calc(d) {
  let t = 0;
  for (let i = 0; i < d.length; i++) {
    if (d[i].price > 100) {
      t += d[i].price * 0.9;
    } else {
      t += d[i].price;
    }
  }
  return t;
}
```

### ✅ After (결과)

```javascript
/**
 * 장바구니 총액을 계산합니다.
 * 100원 초과 상품은 10% 할인을 적용합니다.
 */
function calculateTotalPrice(items) {
  const DISCOUNT_THRESHOLD = 100;
  const DISCOUNT_RATE = 0.9;

  return items.reduce((total, item) => {
    const isDiscountApplicable = item.price > DISCOUNT_THRESHOLD;
    const finalPrice = isDiscountApplicable
      ? item.price * DISCOUNT_RATE
      : item.price;

    return total + finalPrice;
  }, 0);
}
```

_(AI 요약: 함수명 `calc`를 의도가 명확한 `calculateTotalPrice`로 변경하고, 매직 넘버(100, 0.9)를 상수로 분리하여 가독성을 높였습니다. 또한 `for` 루프 대신 모던 JS 문법인 `reduce` 배열 메서드를 활용하여 불필요한 상태 변이(`let t`)를 완전히 제거했습니다.)_
