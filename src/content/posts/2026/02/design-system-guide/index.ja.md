# 🎨 デザインシステム構築：デザイナーと開発者が争わない方法

- **🎯 おすすめの対象:** 「ボタンの色がデザイン案と違います」とフィードバックを受ける開発者、「開発者が私のデザインを台無しにした」と絶望しているデザイナー
- **⏱️ 所要時間:** 10分 → 1分に短縮（初期トークンの設計およびコード自動変換）
- **🤖 おすすめのモデル:** ChatGPT-4o, Claude 3.5 Sonnet（アーキテクチャ設計およびコード生成）

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「Primary Colorは \`#0055FF\` じゃなかったですか？なぜここには \`#0050FF\` が反映されているんですか？」_

コードベースのあちこちに散らばっているハードコーディングされた色や余白の値は、メンテナンスを地獄にする元凶です。今すぐ**デザイントークン（Design Token）**を導入しましょう。色、フォント、余白、シャドウなど、すべての視覚的要素を意味のある変数として定義し、チーム全体で共有するための最も確実な約束です。トークンベースのシステムが構築されれば、デザイナーがFigmaで値を修正した瞬間に、開発者のコードも自動的に同期されるという魔法を体験できます。

---

## ⚡️ 3行まとめ (TL;DR)

1. **デザインのトークン化:** \`#0055FF\` のような絶対的なHex値の代わりに、意味を込めた \`primary-500\` という共通言語を使用します。
2. **Figma Variablesの適用:** FigmaのVariablesおよびStyles機能を活用して、視覚的属性を体系的なトークンとして定義します。
3. **コードの同期:** 抽出されたトークンデータを基に、\`tailwind.config.ts\` またはグローバルCSS変数へと自動変換し、プロジェクトのコードに即座に適用します。

---

## 🚀 解決策: "Design Token Generator Prompt"

### 🥉 Basic Version (基本型)

初期のカラーパレットを構成し、変数名をどのようにつけるべきか迷ったときに活用するプロンプトです。

> **リクエスト (Task):**
> 私は今、新しいプロダクトのデザインシステムにおいてカラーパレットを企画しています。
> \`[Primary(青)]\`、\`[Secondary(グレー)]\`、\`[Error(赤)]\` の各色を100から900までの9段階に分けたいです。
> これらの色をフロントエンドのコードに直接適用できるように、直感的でセマンティック（Semantic）な名前（例: \`text-primary\`、\`bg-surface-default\`）と一緒に、各段階のHexコードを表にまとめて提案してください。

<br>

### 🥇 Pro Version (専門家型)

Figmaから抽出したトークンのJSONデータを、実際のフロントエンド環境に合わせて完璧なコードに変換する際に使用します。

> **役割 (Role):** あなたは10年目のシニアデザインオプス（Design Ops）エンジニアであり、フロントエンドアーキテクトです。
>
> **状況 (Context):**
>
> - 背景: デザイナーがFigmaでデザイントークンを定義し、JSON形式で抽出しました。
> - 目標: このJSONデータを、フロントエンドプロジェクトに即時適用可能なコードへと完璧に変換する必要があります。
>
> **入力データ:**
>
> \`[ここにFigmaから抽出したトークンのJSONデータを貼り付けてください]\`
>
> **リクエスト (Task):**
>
> 1. 入力データを分析し、最新の**Tailwind CSS設定ファイル（\`tailwind.config.ts\`）**の形式に変換してください。
> 2. 特定のCSSフレームワークに依存しないように、**CSS Variables（\`:root { --color-blue-500: ... }\`）**のコードも一緒に作成してください。
> 3. システムのテーマに合わせて変更できるように、ダークモード（\`@media (prefers-color-scheme: dark)\` または \`.dark\` クラス）への対応戦略とサンプルコードを追加してください。
>
> **制約事項 (Constraints):**
>
> - 出力形式はマークダウンのコードブロック（\`ts\`, \`css\`）のみで提供してください。
> - すべての変数名はケバブケース（kebab-case）を厳格に維持してください。
>
> **注意事項 (Warning):**
>
> - 提供されたJSONの構造以外に、任意の色値や段階を捏造しないでください。（ハルシネーションの防止）

## 💡 筆者コメント (Insight)

デザインシステムの構築は、巨大なマスタープランを立てて一度に完成させるものではありません。最初からすべてのコンポーネントやタイポグラフィを完璧にトークン化しようとすると、高い確率で疲弊して諦めてしまいます。
**「最も多く、繰り返し使われる要素（例: Primary Buttonの背景色、本文のテキストカラー）」**から一つずつ変数に置き換えていく**漸進的導入（Incremental Adoption）**の戦略を強くお勧めします。
小さなトークン一つが、デザイナーと開発者の間の無駄なコミュニケーションコストを劇的に減らす最初のボタンとなるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: Figmaでトークンを抽出するにはどのプラグインを使うべきですか？**
  - A: 実務では**「Tokens Studio for Figma」**が事実上の標準（De facto）として使われています。これを使ってJSONを抽出し、GitHubリポジトリと直接同期（Sync）することができます。最近ではFigmaネイティブのVariables機能も強力になっており、REST APIを通じた連携も多く使われる傾向にあります。

- **Q: 2〜3人規模の小さなスタートアップでもデザインシステムは必要ですか？**
  - A: チームメンバーが2人以上であるか、プロジェクトのメンテナンス期間が3ヶ月を超えるなら、無条件に導入すべきです。「後で時間があるときに整理しよう」という技術的負債は、最終的にプロジェクトのスピードを著しく低下させる爆弾となって返ってきます。未来の自分（Future Self）も、結局は他人なのです。

## 🧬 プロンプトの解剖 (Why it works?)

1. **セマンティック・ネーミングの強制:** 「明るい青色」の代わりに「主要なボタンの背景色（\`bg-primary-default\`）」という意味ベースのネーミングを誘導します。こうすることで、後からブランドカラーが青から紫に全面修正されたとしても、Hex値を一つ変更するだけでプロジェクト全体に即時反映されます。
2. **反復的なフォーマット変換の委任:** JSONフォーマットをTailwindの設定オブジェクトやCSS変数の構文として一つ一つタイピングして移すことは、人間にとって非常に苦痛です。AIはこうしたパターンベースの単純なコード変換や構造化作業において圧倒的な正確さとスピードを誇り、ヒューマンエラー（Human Error）を根絶します。

## 📊 証明: Before & After

### ❌ Before (入力)

デザインが変更されるたびに、100を超えるファイルからHexコードを一つ一つ検索して置き換えなければなりません。😱

```css
.button-primary {
  background-color: #3b82f6;
}

.text-highlight {
  color: #3b82f6;
}
```

### ✅ After (結果)

変数の値をたった一つ修正するだけで、プロダクト全体のUIが完璧にアップデートされます。🚀

```css
:root {
  --primary-500: #3b82f6;
  --bg-primary-default: var(--primary-500);
}

.button-primary {
  background-color: var(--bg-primary-default);
}
```

## 🎯 結論

デザインと開発の間の通訳者は、感情的な「人」ではなく、明確なルールを持った**「システム」**であるべきです。
これ以上、ピクセルや色の値で争うのはやめましょう。トークンという共通の言語で対話することをお勧めします。

**「開発者さん、これは \`gray-200\` ではなく \`surface-subtle\` です。トークンの確認をお願いします。」** 🍷
