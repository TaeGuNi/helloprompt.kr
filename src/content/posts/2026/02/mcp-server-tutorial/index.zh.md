作为开发者或数据分析师，想必大家都经历过这种痛苦的日常：打开 ChatGPT 或 Claude 的空白对话框，手动拖动并复制粘贴公司数据库模式 (`schema.sql`) 或长达数千行的 API 规范说明书。“根据这个表结构写个查询语句”、“这是我们公司的 API 文档，该怎么调用？”——每当你开启一个新的对话框，都不得不手动喂给 AI 这一大堆文本。

一旦上下文窗口出现偏差或触及 Token 限制，AI 就会引用错误的表，或者编造不存在的列。更糟糕的是，每当你将绝不能外泄的<b>敏感公司安全代码</b>或<b>客户数据</b>复制到浏览器窗口时，总会感到背脊发凉。在这个过程中，我们并没有专注于编码，而是沦为了根据 AI 的口味修饰提示词、拼凑上下文的体力劳动者。

这种毫无意义的苦力活还要重复多久？假设你一天新开 10 次提示词窗口，仅在查找、复制和粘贴这些准备工作上，每天至少会有 30 分钟化为乌有。一周就是 2.5 小时，一个月下来，整整一天的时间都浪费在复制粘贴文本上。这不仅是时间的浪费，更会粉碎开发者的创造力和专注力。

更致命的问题是<span style="color:var(--color-cyber-cyan)">上下文碎片化 (Context Fragmentation)</span>。昨天粘贴的模式在今天上午更新了，但 AI 仍在根据昨天的旧模式编写代码。结果，你不得不花费数十分钟去修复因代码错误导致的 Bug，最后愤怒地心想“还不如我自己写呢”。别人都在用 AI 提前下班，而我们却因为要迎合和训练 AI 反而不得不加班，这真是一个讽刺的现状。

然而在 2026 年的今天，一个能瞬间结束所有痛苦的“规则改变者”出现了。这就是由 Anthropic 主导的 <b>Model Context Protocol (MCP)</b>。随着这一巨大的技术飞跃成为 AI 生态系统的标准，再也没有必要手动给 AI 喂数据了。因为你可以让 AI <b>直接接入你的本地数据，实现实时的自动读取</b>。

简单类比，MCP 就像是 AI 模型专属的<b>通用 USB 端口</b>。只要在你的电脑上启动一次服务端并连接好端口，Claude Desktop 或 Ollama 等客户端就能直接访问你的本地文件系统、内部数据库，甚至是内网环境下的公司 API，并实时流式传输最新信息。

更令人惊喜的是，构建这样一个强大的自定义服务端仅需 5 分钟。利用今天介绍的提示词，开发者无需亲自动手编写复杂的样板代码，只需向 AI 发出指令，即可瞬间<b>生成 (Scaffolding)</b> 出完美契合特定业务环境、基于 TypeScript 的 MCP 服务端代码。你无需在数十个依赖项和初始设置中迷失方向，一次即可获得完美运行的代码。

现在，请想象一下应用此提示词后的惊人变化。你只需打开一个空白对话框，自然地随口一问：<i>“今天新注册的用户有多少？”</i>或者<i>“昨天发生的支付错误日志的核心原因是什么？”</i>

Claude 会在后台安静地直接向你的本地数据库或日志文件发送查询，并基于分毫不差的最新准确数据给出完美的回答。无需担心数据泄露，在你自己的电脑里，一个最舒适、安全且强大的专属 AI 基础设施就此落成。你将不再仅仅是一个编码员 (Coder)，而是迈出了成为指挥专属强大 AI 系统的<b>架构师 (Architect)</b> 的第一步。现在，我们就公开这个“作弊码”。

## 💡 作者点评 (Insight & How to use)

这个提示词的真正价值不仅在于简单地请求 AI 编写代码，更在于它<b>强制将 MCP 生态系统要求的严格协议规范注入到 AI 中</b>。

我第一次尝试创建自定义 MCP 服务端时的惨痛失败构成了这个提示词的骨架。当我只是简单地指示“请帮我做一个能读取公司数据库并告知结果的 MCP 服务端”时，AI 生成的代码看起来像模像样，但实际上却满是根本无法运行的<b>幽灵 SDK 方法</b>。它完全忽视了 MCP 通信的核心生命周期——`ListTools` 和 `CallTool` 请求处理，反而吐出了一堆随处可见的普通 Express.js Web 服务端代码。结果，为了修复和调试那堆烂代码，我白白浪费了半天时间。

在亲历了这次惨痛失败后，我领悟到了一个真理：<span style="color:var(--color-cyber-cyan)">“在起草阶段，绝不能将架构的主导权交给 AI”。</span>

因此，我在提示词的指令中明确规定必须实现官方 `@modelcontextprotocol/sdk` 的必备接口 `ListToolsRequestSchema` 和 `CallToolRequestSchema`。此外，为了防止工具输入参数出现偏差，我加入了强制使用运行时类型验证库 `zod` 的安全装置。结果，仅需执行一次提示词，就能诞生出完全没有语法错误或类型错误、可在终端立即运行的<b>完美单文件 (`index.ts`) 服务端代码</b>。

特别是在实际工作中，这种自定义 MCP 服务端真正的闪光时刻在于<b>“极强的隐私保护 (Privacy)”</b>与<b>“动态上下文 (Dynamic Context)”</b>的结合。在以往的方式中，为了将困在公司内网深处的敏感客户日志数据或私有 API 规范与 AI 结合，不得不看安全团队的脸色，逐一进行脱敏处理。

但如果你启动了一个完全在本地环境运行的自定义 MCP 服务端，情况就大不一样了。你处理的敏感数据绝不会传输到外部云服务器，而仅在<b>本地物理内存中安全处理</b>。像 Claude Desktop 这样的客户端只是向本地主机 (localhost) 发送查询 (Query) 并安全地接收结果值。

实战应用时的核心技巧是<b>变量控制法</b>。在输入提示词中的 `[目的]` 变量时，不要写得太模糊，而应明确工具的具体形式。例如，与其写 `[查询特定数据库今日加入者数量的功能]`，不如明确控制为 `[在 PostgreSQL 数据库的 users 表中，以 date 列为基准，执行 SELECT 查询今日注册用户数量的工具]`。你越清晰地定义工具接收的参数和应返回的数据格式，AI 编写出的 `zod` 模式就越精细。

我实际上利用这个提示词构建并使用了一个与公司 Jira 任务追踪器及内部 Slack 通知实时联动的强大 MCP 服务端。早上一上班，我只需输入 <i>“汇总一下目前分配给我且进度低于 50% 的本迭代任务”</i>，AI 就会实时调用 Jira API 获取最新数据并构建上下文，即刻生成完美的简报。

大家也可以从简单的“只读 (Read-only)”工具开始，比如读取特定文件夹的文本文件，这样既轻便又安全。当你熟悉了服务端的骨架和数据往来的运行原理后，可以逐渐尝试直接向数据库发送查询、向公司通讯工具发送自动通知等，体验自由指挥本地基础设施的真正生产力新世界。

## 🙋 常见问题 (FAQ)

- **Q: 完全没有编码知识也能使用这个提示词吗？**
  - A: 是的，当然可以！只要在 `[目的]` 栏中用自然语言写下“公司百科文档搜索功能”，AI 就会代你编写结构完美的代码。你只需复制生成的代码，粘贴到执行环境中并运行脚本即可。

- **Q: 生成的服务端代码如何运行并连接到 LLM 客户端？**
  - A: 将代码保存为 `index.ts` 文件后，在 Claude Desktop 的配置文件 (`claude_desktop_config.json`) 中的 `mcpServers` 项下添加该脚本的执行命令（例如：`npx tsx index.ts`）和绝对路径，即可立即联动并显示在工具标签页中。

- **Q: 用这个提示词制作的服务端只能在 Claude 中运行吗？**
  - A: 不是的。Model Context Protocol 是一个不依赖于特定模型的通用<b>开放标准 (Open Standard)</b>。因此，在 Ollama、Cursor、Zed 等所有支持 MCP 规范的客户端环境中，无需修改即可完美兼容。

## 🧬 提示词解剖 (Why it works?)

1. **约束条件 (Constraints) 的强力控制：** 强制要求使用 `@modelcontextprotocol/sdk` 和 `zod` 库，并严格说明了必备模式 (`ListToolsRequestSchema`, `CallToolRequestSchema`)。这从源头上杜绝了 AI 编造不符合规范的虚假代码的幻觉 (Hallucination) 现象。
2. **单文件及纯代码输出控制：** 指示使用单文件 (`index.ts`) 结构，而非碎片化的复杂架构，并命令完全去除不必要的额外说明。结果是，从复制代码到测试的时间 (Time-to-Value) 大幅缩短。
3. **强制运行时异常处理：** 为了确保在本地环境执行工具期间即使发生意外错误，整个 MCP 服务端进程也不会崩溃 (Crash)，预先彻底规定了 `try/catch` 防御逻辑。
