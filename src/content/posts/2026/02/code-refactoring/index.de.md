> _„Das mache ich später richtig“, hast du dir damals gesagt. Ein Jahr ist vergangen. Heute hast du Panik, diese Datei überhaupt nur zu öffnen._

Spaghetti-Code ist der absolute Endgegner für jeden produktiven Entwickler. Sobald die Lesbarkeit auf der Strecke bleibt, wird das Debugging zur reinsten Qual und jedes neue Feature gleicht einem Spaziergang durch ein Minenfeld. Dieser Prompt verwandelt die KI in deinen persönlichen Robert C. Martin (den Autor von _Clean Code_) – einen unerbittlichen Mentor, der deinen Code schonungslos analysiert, strukturiert und auf absoluten Hochglanz poliert.

---

## ⚡️ 3-Zeilen-Zusammenfassung (TL;DR)

1. **Strukturierung:** Zerlegt überlange Funktionen und eliminiert gnadenlos redundanten Code (Extract Method).
2. **Namensgebung:** Ersetzt kryptische Variablen durch intuitive, selbsterklärende Bezeichnungen.
3. **Modernisierung:** Aktualisiert veraltete Syntax auf den neuesten Sprachstandard und optimiert gleichzeitig die Performance.

## 🚀 Lösung: Die "Clean Code" Waschmaschine

### 🥉 Basic Version (Basisversion)

Nutze diese kompakte Variante, wenn die Zeit drängt und du ein blitzschnelles Refactoring benötigst.

> **Rolle:** Du bist ein `[Clean Code Experte und Software-Architekt]`.
>
> **Aufgabe:** Unterziehe diesen unsauberen Code einem Refactoring. Mach ihn elegant, optimal lesbar und extrem wartungsfreundlich, ohne seine ursprüngliche Funktion zu verändern.

### 🥇 Pro Version (Expertenversion)

Verwende diesen hochdetaillierten Prompt für tiefgreifende Architektur-Upgrades und Code-Reviews auf Senior-Niveau.

> **Rolle (Role):** Du bist ein erfahrener `[Clean Code Evangelist und Chief Software Architect]`.
>
> **Kontext (Context):**
>
> - Hintergrund: Mein aktueller Code funktioniert zwar, ist jedoch extrem unübersichtlich, kaum testbar und ein Albtraum in der Wartung (Technical Debt).
> - Ziel: Ein vollständiges Refactoring nach den strengen Prinzipien von Clean Code und SOLID, um Skalierbarkeit und Lesbarkeit zu maximieren.
>
> **Aufgabe (Task):**
>
> 1. **`[Namensgebung]`**: Benenne Variablen und Funktionen aussagekräftig und intuitiv um (z. B. von `a` zu `userAge`).
> 2. **`[Struktur]`**: Zerlege monolithische Funktionen in kleine, testbare Einheiten (Single Responsibility Principle) und lösche jegliche redundante Logik.
> 3. **`[Modernisierung]`**: Wende die aktuellsten Best Practices und die modernste Syntax der jeweiligen Programmiersprache an (z. B. ES6+, Python 3.10+).
> 4. **`[Analyse]`**: Liefere einen strukturierten Vorher-Nachher-Vergleich und begründe jede deiner Architektur-Entscheidungen prägnant.
>
> **Einschränkungen (Constraints):**
>
> - Die Kernfunktionalität und das äußere Verhalten des Codes dürfen sich unter keinen Umständen verändern.
> - Formatiere dein Ergebnis als übersichtliche Markdown-Struktur mit entsprechenden Code-Blöcken.
> - Füge Kommentare ausschließlich dort hinzu, wo das "Warum" erklärt werden muss. Der Code selbst soll das "Wie" dokumentieren (Selbsterklärender Code).
>
> **Zielcode:**
>
> `[Füge hier deinen Spaghetti-Code ein]`

## 💡 Autorenkommentar (Insight)

Dieser Prompt ist ein absoluter Lebensretter – ganz besonders dann, wenn du Legacy-Code von Kollegen (oder von deinem eigenen Ich aus der Vergangenheit) erbst. Anstatt stundenlang frustriert zu versuchen, die verstrickte Logik händisch zu entwirren, liefert dir die KI innerhalb von Sekunden einen extrem soliden Ausgangspunkt.

**Pro-Tipp:** Übernimm das Ergebnis niemals blind! Die KI ist zwar phänomenal darin, "Early Returns" einzubauen und Variablen klug umzubenennen, aber bei hochkomplexer Geschäftslogik solltest du den refactorten Code immer durch deine Unit-Tests jagen. Falls der Code vorher überhaupt keine Tests hatte, gib der KI einfach diesen Folge-Prompt: _"Schreibe mir zuerst Unit-Tests für diesen unsauberen Code, bevor wir ihn refactoren."_ Genau das ist echtes Test-Driven Refactoring in der Praxis!

## 🙋 Häufig gestellte Fragen (FAQ)

- **Q: Berücksichtigt die KI auch unsere unternehmensinternen Coding-Richtlinien?**
  - A: Absolut! Du kannst den Prompt völlig flexibel erweitern, z. B. mit: _"Beachte, dass wir in unserem Projekt strikt funktional programmieren und Mutationen vermeiden."_ Je spezifischer du den Kontext vorgibst, desto passgenauer wird das Ergebnis.

- **Q: Kann es passieren, dass die KI beim Refactoring versehentlich Bugs einbaut?**
  - A: Ja, das Risiko besteht. Obwohl wir in den Einschränkungen unmissverständlich klarstellen, dass sich das Verhalten nicht ändern darf, kann es bei komplexen Nebenläufigkeiten (Concurrency) oder versteckten Abhängigkeiten zu unerwarteten Fehlern kommen. Verlasse dich beim Refactoring daher immer auf automatisierte Tests!

- **Q: Funktioniert dieser Prompt für jede beliebige Programmiersprache?**
  - A: Ja. Die fundamentalen Konzepte von Clean Code sind universell. Ganz gleich, ob JavaScript, Python, Go oder Rust – die KI passt die spezifischen Syntax-Vorschläge perfekt an die von dir bereitgestellte Sprache an.

## 🧬 Prompt-Anatomie (Why it works?)

1. **Rollen-Empathie (Role):** Durch die Zuweisung der hochkarätigen Rolle als "Chief Software Architect" zwingen wir das Modell, auf dem allerhöchsten Qualitätsniveau zu antworten, anstatt nur oberflächliche Syntax-Kosmetik zu betreiben.
2. **Klares Framework (Task):** Die gezielte Unterteilung in Namensgebung, Struktur und Modernisierung gibt der KI einen exakten Leitfaden an die Hand, _wie_ sie das Refactoring angehen soll. Das reduziert Halluzinationen drastisch.
3. **Erklärungszwang (Analyse):** Indem wir handfeste Begründungen für jede Änderung einfordern, können wir als Entwickler exakt nachvollziehen, _warum_ eine bestimmte Architektur gewählt wurde – und lernen so direkt aus dem Prozess.

## 📊 Beweis: Before & After

### ❌ Before (Eingabe)

```python
def calc(x, y):
    if x > 19:
        print("Erwachsener")
        p = y * 0.9
        return p
    else:
        print("Schüler")
        return y
```

### ✅ After (Ergebnis)

```python
def calculate_discounted_price(age: int, base_price: float) -> float:
    ADULT_AGE_THRESHOLD = 20
    DISCOUNT_RATE = 0.9

    if age >= ADULT_AGE_THRESHOLD:
        print("Kunde ist volljährig.")
        return base_price * DISCOUNT_RATE

    print("Kunde ist Schüler.")
    return base_price
```

**Warum ist das besser?**

- `calc` wurde zu `calculate_discounted_price` transformiert (die Absicht der Funktion ist sofort glasklar).
- Magische Zahlen (`19`, `0.9`) wurden in aussagekräftige Konstanten (`ADULT_AGE_THRESHOLD`, `DISCOUNT_RATE`) umgewandelt.
- Strikte Typisierung (`int`, `float`) wurde hinzugefügt (moderne Python-Standards).
- Das völlig überflüssige `else` wurde durch das elegante "Early Return"-Pattern eliminiert, was die Verschachtelungstiefe massiv reduziert.
