## 🙋 常见问题 (FAQ)

- **Q: 与老牌的 Flask 或 Django 相比，FastAPI 的性能优势真的有那么夸张吗？**
  - A: 绝对是降维打击。在现代 Web 服务的真实场景中，往往充斥着海量 I/O 密集型任务（例如高频数据库查询、第三方 API 疯狂调用），此时**异步 (Asynchronous)** 处理能力就成了生死攸关的刚需。基于 Starlette 与 Uvicorn 引擎的 FastAPI，在应对高并发洪峰时，展现出了无情碾压传统同步框架的超高吞吐量 (Throughput)。

- **Q: 配合 FastAPI 通常使用哪种 ORM 框架？它能完美支持异步数据库操作吗？**
  - A: 业界的绝对标杆是 `SQLAlchemy`，它自 2.0 版本起便已实现了对异步 (`asyncio`) 的原生完美支持。你只需在提示词中向 AI 明确下达指令：“**请使用 SQLAlchemy 2.0 异步会话**”，便能顺理成章地构建出一个从头到尾纯正的非阻塞异步后端系统。

- **Q: 为什么在提示词中必须死死咬住“Pydantic V2”不放？**
  - A: 因为 Pydantic V2 的底层核心已由 Rust 语言全面重写，数据校验的狂飙速度比 V1 版本最高提升了整整 **50 倍**！然而，各大 AI 模型（尤其是受限于知识库截止日期的模型）通常仍残留着输出 V1 语法的“肌肉记忆”。在提示词中利用强约束逼迫其输出 V2 语法，是你避免在生产环境中踩坑致命报错的最短捷径。

## 🧬 提示词解剖 (Why it works?)

1. **明确界定的架构分离边界：** 如果你只是轻描淡写地丢出一句“帮我写个 FastAPI 服务”，AI 往往会敷衍地抛给你一个所有业务逻辑全坨在 `main.py` 里的入门级“玩具”。我们通过强制其清晰划分 `api`、`core` 和 `schemas` 等独立模块，瞬间将该架构在生产环境中的可维护性直接拉满。
2. **严苛的版本锁死 (Pydantic V2 约束)：** 众所周知，Python 生态中最让人抓狂的莫过于依赖库的向下兼容与版本迁移灾难。在这份提示词中，我们直接动用约束条件 (Constraints)，在代码生成的源头阶段就将这种隐患彻底扼杀在摇篮里。
3. **着眼生产部署的端到端 (End-to-End) 视野：** 我们刻意将生成云原生 `Dockerfile` 文件的诉求也一并塞入核心指令中。这直接打破了“代码只能在本地跑通”的狭隘局限，一气呵成地为你铺平了直通云端服务器自动化流水线的大道。
