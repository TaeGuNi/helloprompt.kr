# 📝 Какова временная сложность (Big O) моего кода?

- **🎯 Рекомендуется для:** Junior-разработчиков, студентов CS, соискателей на технические интервью
- **⏱️ Требуемое время:** 30 минут → 1 минута
- **🤖 Рекомендуемая модель:** ChatGPT (GPT-4), Claude 3.5 Sonnet, Gemini 2.5 Pro

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _"Два вложенных цикла... Это $O(N)$ или $O(N^2)$? И не упадет ли сервер, если данных станет в тысячу раз больше?"_

Понимание и расчет временной сложности (Big-O) — это не просто академический навык для сдачи экзаменов или прохождения технических собеседований. Это фундамент для написания масштабируемого кода, способного выдерживать реальный highload-трафик. Однако, когда в коде появляются рекурсии, сложные условия ветвления или скрытые циклы внутри библиотечных функций, ручной расчет сложности может стать настоящей головной болью.

Что если доверить эту задачу ИИ? Нейросети способны не только мгновенно проанализировать ваш код в нотации Big-O, но и, подобно опытному Senior-инженеру, указать на узкие места и предложить элегантные пути оптимизации.

---

## ⚡️ 3 кратких вывода (TL;DR)

1. Мгновенно рассчитывает как временную, так и пространственную сложность (Big-O) любого фрагмента кода.

3. Предлагает рефакторинг с понижением асимптотической сложности (например, с $O(N^2)$ до $O(N \log N)$).

## 💡 Комментарий автора (Insight)

Этот промпт — настоящая находка для тех, кто изучает алгоритмы (например, решает задачи на LeetCode) или проводит код-ревью. ИИ отлично справляется с выявлением скрытых операций. Например, новички часто забывают, что вызов метода `indexOf()` или `includes()` внутри массива — это $O(N)$ операция. Если поместить ее внутрь цикла `for`, ваш алгоритм незаметно превращается в $O(N^2)$.

**Совет из практики:** Если ИИ предлагает оптимизацию за счет использования `Set` или `Map`, внимательно изучите пространственную сложность (Space Complexity). Очень часто мы ускоряем программу (экономим время), жертвуя оперативной памятью (компромисс Time-Space Tradeoff). Обязательно обдумайте, что критичнее для вашей конкретной задачи — скорость или потребление памяти.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Может ли ИИ ошибиться в расчете Big-O?**
  - A: Редко, но может, особенно если код содержит очень сложные, нестандартные рекурсивные вызовы без явного базового случая. Для 95% типичных алгоритмических задач и бизнес-логики анализ будет абсолютно точным.

- **Q: Поймет ли ИИ код на редких языках программирования?**
  - A: Да. Современные LLM отлично знают алгоритмы независимо от языка (от Python и C++ до Rust и Go). Главное — логика самого алгоритма.

- **Q: Можно ли использовать этот промпт для целого проекта?**
  - A: Нет, этот промпт предназначен для анализа конкретных функций или классов (от 10 до 100 строк). Анализ архитектуры целого проекта требует совершенно другого подхода и других промптов.

## 📊 Доказательство: Before & After

### ❌ Before (Исходный код — поиск дубликата)

```python
def find_duplicate(nums):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] == nums[j]:
                return nums[i]
    return None
```

### ✅ After (Результат анализа ИИ)

**Анализ исходного кода:**

- **Временная сложность:** $O(N^2)$. В коде используются два вложенных цикла `for`. Для каждого из $N$ элементов мы перебираем оставшиеся $N-1$ элементов. При массиве из 100,000 элементов это потребует около 5 миллиардов операций — код будет выполняться недопустимо долго.
- **Пространственная сложность:** $O(1)$. Алгоритм не использует дополнительных структур данных, выделение памяти постоянно.

**Оптимизированный подход (Использование Hash Set):**
Мы можем радикально сократить время выполнения, пожертвовав небольшим количеством памяти. Использование `set` (хеш-множества) позволяет проверять наличие элемента за константное время $O(1)$.

```python
def find_duplicate_optimized(nums):
    seen = set() # Выделяем память под множество
    for num in nums:
        # Поиск в set занимает O(1) времени
        if num in seen:
            return num
        seen.add(num)
    return None
```

**Результат оптимизации:**

- **Временная сложность:** $O(N)$. Массив обходится ровно один раз. Огромный прирост производительности на больших данных!
- **Пространственная сложность:** $O(N)$. Нам потребуется выделить память для хранения до $N$ элементов в множестве `seen`. Это классический компромисс между памятью и скоростью.
