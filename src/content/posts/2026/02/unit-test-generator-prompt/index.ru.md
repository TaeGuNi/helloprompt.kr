# 📝 Написание юнит-тестов: промпт, который справляется за 5 минут

- **🎯 Рекомендуется для:** Разработчиков, QA-инженеров, тимлидов
- **⏱️ Требуемое время:** 5 минут (вместо часов рутины)
- **🤖 Рекомендуемая модель:** Любая современная LLM (GPT-4o, Claude 3.5 Sonnet, Gemini 2.5 Pro)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _«Опять дедлайн горит, фича готова, а тесты писать еще на полдня?»_

Все мы знаем, насколько важны юнит-тесты, но давайте признаемся честно: их написание часто отнимает больше времени, чем реализация самой бизнес-логики.
Продумывание пограничных случаев (edge cases), настройка моков (mocking), подготовка фикстур... Звучит как рутина, не так ли?
Теперь вам достаточно просто скормить свой код ИИ, и он за считанные секунды сгенерирует надежные тесты, покрывающие абсолютно все сценарии.

---

## ⚡️ Главное в 3 пунктах (TL;DR)

1. **Глубокий анализ:** ИИ учитывает не только успешные сценарии (happy path), но и неочевидные пограничные случаи.
2. **Любой фреймворк:** Безупречная поддержка синтаксиса популярных библиотек тестирования (Jest, PyTest, JUnit и др.).
3. **Избавление от рутины:** Мгновенная генерация mock-объектов и фикстур без лишней головной боли.

---

## 🚀 Решение: Промпт «Генератор юнит-тестов»

### 🥉 Basic Version (Базовая версия)

Идеально подходит, когда нужен быстрый результат без долгих настроек.

> **Роль:** Ты дотошный QA-автоматизатор и Senior-разработчик.
> **Запрос:** Напиши юнит-тесты для следующего кода. Постарайся обеспечить 100% покрытие. `[Вставьте ваш код]`

### 🥇 Pro Version (Профессиональная версия)

Используйте этот вариант для сложных функций, которые требуют глубокого анализа и мокирования внешних зависимостей.

> **Роль (Role):** Ты дотошный QA-инженер и эксперт по автоматизации тестирования с многолетним опытом.
>
> **Контекст (Context):**
>
> - Исходные данные: У меня есть код, написанный на `[Укажите язык программирования, например: TypeScript]`.
> - Цель: Мне нужны надежные и исчерпывающие юнит-тесты для повышения отказоустойчивости продакшен-кода.
>
> **Задача (Task):**
>
> 1. Внимательно проанализируй предоставленный код и составь список тестовых сценариев (включая успешные пути, обработку ошибок и пограничные значения).
> 2. Напиши готовый к запуску код тестов, используя фреймворк `[Укажите библиотеку тестирования, например: Jest]`.
> 3. Если в коде присутствуют внешние зависимости (API, базы данных), реализуй для них корректные моки (Mocking).
> 4. Добавь исчерпывающие комментарии к каждому тестовому блоку с объяснением того, что именно проверяется.
>
> **Код:**
>
>
> [Вставьте ваш исходный код сюда]
>
>
> **Ограничения (Constraints):**
>
> - Обеспечь максимально возможное покрытие кода (Code Coverage).
> - Предоставь результат в виде единого блока кода, который можно сразу скопировать и запустить.
>
> **Предупреждение (Warning):**
>
> - Описания тестов должны быть исключительно на русском языке. Убедись, что моки не переусложняют тесты (избегай овер-инжиниринга).

## 💡 Комментарий автора (Insight)

Этот промпт — настоящее спасение при работе с легаси-кодом или запутанной бизнес-логикой. Главный секрет кроется в первом пункте задачи: мы просим ИИ **сначала составить список сценариев**, и лишь затем переходить к написанию кода. Такой подход заставляет модель применять логику Chain-of-Thought (цепочка рассуждений), что кардинально снижает риск упустить неочевидные баги или пограничные значения. Особенно ярко это проявляется при необходимости мокировать вызовы к внешним API — ИИ часто генерирует заглушки для таких краевых случаев, о которых вы сами могли даже не подумать!

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Справится ли бесплатная версия ChatGPT (GPT-3.5/GPT-4o-mini) с этой задачей?**
  - A: Базовую логику она покроет без проблем. Однако для сложных тестов с обилием моков и запутанными пограничными случаями я настоятельно рекомендую использовать продвинутые модели, такие как GPT-4o или Claude 3.5 Sonnet.

- **Q: Что делать, если сгенерированный тест падает с ошибкой?**
  - A: Просто скопируйте текст ошибки из консоли и отправьте его ИИ с комментарием: «Тест упал с этой ошибкой. Исправь код теста или укажи, где закралась ошибка в моей бизнес-логике». Зачастую ИИ обнаруживает реальный баг именно в вашем исходном коде!

## 🧬 Анатомия промпта (Why it works?)

1. **Двухэтапная генерация (Chain-of-Thought):** Требование сначала выписать все сценарии заставляет ИИ глубже проанализировать код, прежде чем выдавать готовые функции тестирования.
2. **Управление зависимостями:** Явное указание на необходимость мокирования спасает от написания хрупких (flaky) тестов, которые могут неожиданно сломаться из-за внешних факторов.

## 📊 Доказательство: Before & After

### ❌ Before (Ваш исходный код)

```javascript
function divide(a, b) {
  if (b === 0) throw new Error("Нельзя делить на 0.");
  return a / b;
}
```

### ✅ After (Результат ИИ)

```javascript
describe("Тестирование функции divide", () => {
  // 1. Тест нормальной работы (Happy path)
  test("Деление 10 на 2 должно возвращать 5", () => {
    expect(divide(10, 2)).toBe(5);
  });

  // 2. Тест обработки исключений (Error handling)
  test("Деление на 0 должно выбрасывать ошибку", () => {
    expect(() => divide(10, 0)).toThrow("Нельзя делить на 0.");
  });

  // 3. Тест пограничных значений (Edge case)
  test("Деление отрицательного числа на положительное дает отрицательный результат", () => {
    expect(divide(-10, 2)).toBe(-5);
  });
});
```
