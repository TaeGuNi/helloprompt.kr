# 📝 List? Map? Set? 결정 장애를 위한 자료구조 처방전

- **🎯 추천 대상:** 주니어 개발자, 코딩 테스트 준비생, 성능 최적화가 필요한 실무자
- **⏱️ 소요 시간:** 10분 → 1분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (논리적 추론이 뛰어난 모델 권장)

> _"맨날 쓰던 ArrayList만 쓰다가 '데이터가 100만 개쯤 되니까 API가 너무 느려요'라는 사용자 불만을 받고 식은땀 흘린 적 없으신가요?"_

자료구조 선택은 프로그램의 성능을 좌우하는 가장 중요한 첫 단추입니다. 하지만 실제 개발을 하다 보면 시간 압박에 쫓겨 늘 쓰던 익숙한 자료구조(주로 List나 Array)만 관성적으로 사용하게 되는 경우가 많습니다.

이 프롬프트는 여러분이 처한 구체적인 비즈니스 상황과 데이터 패턴을 입력하면, 시간 복잡도(Big-O)와 공간 효율성을 논리적으로 따져 **가장 완벽한 자료구조와 실제 언어별 구현체**를 족집게처럼 골라주는 든든한 시니어 멘토 역할을 합니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. 데이터의 접근, 삽입, 삭제 패턴을 분석하여 병목을 없애는 최적의 자료구조를 추천받습니다.
2. 각 연산에 대한 시간 복잡도(Big-O)를 명확히 비교하여 선택의 타당성을 검증합니다.
3. Java, Python, TypeScript 등 사용 중인 언어에서 바로 쓸 수 있는 구체적인 클래스명까지 제공합니다.

## 💡 작성자 코멘트 (Insight)

실제 현업에서 코드 리뷰를 하다 보면, `Set`이나 `Map`을 써야 할 곳에 `List`를 써서 `O(N^2)`의 참사를 내는 경우를 정말 자주 봅니다. (예: 리스트 두 개를 중첩 for문으로 돌면서 교집합 찾기)

이 프롬프트는 단순히 이론적인 '정답'을 찾는 것을 넘어, 실무에서 어떤 언어의 어떤 내장 클래스를 써야 하는지 디테일하게 짚어준다는 점에서 큰 가치가 있습니다. 예를 들어 Java에서 단순히 'Map을 쓰세요'가 아니라, 입력 순서가 보장되어야 한다면 `LinkedHashMap`, 키 값에 따른 정렬이 필요하다면 `TreeMap`을 제안하도록 유도하여 설계의 빈틈을 막아줍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코딩 테스트 문제 지문을 그대로 넣어도 되나요?**
  - A: 네, 매우 강력한 코테 멘토가 됩니다! `요구사항` 부분에 문제의 시간 제한(예: 1초)과 최대 데이터 크기 N(예: 100,000)을 입력하면, AI가 "N이 10만이므로 `O(N^2)` 알고리즘은 시간 초과가 납니다. 따라서 `O(N log N)` 이하인 Priority Queue를 사용해야 합니다"와 같이 수학적인 추론을 해줍니다.

- **Q: 메모리 절약 vs 처리 속도, 보통 무엇을 우선해야 할까요?**
  - A: 현대의 웹/앱 백엔드 환경에서는 특수한 임베디드 환경이 아닌 이상 **'메모리를 조금 더 쓰더라도 처리 속도를 비약적으로 높이는(Space-Time Tradeoff)'** 해시 테이블(Hash Table) 기반의 자료구조가 유리한 경우가 압도적으로 많습니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **상황과 목표의 명확화 (Context):** AI에게 단순히 '자료구조를 골라'라고 하는 것과, '속도와 메모리의 트레이드오프를 고려해 병목을 없애라'고 지시하는 것은 출력의 깊이가 다릅니다.
2. **평가 기준 강제 (Task & Constraints):** Big-O 표기법이라는 명확한 컴퓨터 과학적 척도를 사용하도록 강제하여, 주관적인 추천이 아닌 수학적/논리적 근거를 바탕으로 답변을 생성하게 만듭니다. 실무 관점(캐시 지역성 등)을 추가한 것도 답변의 퀄리티를 높이는 핵심입니다.

## 📊 증명: Before & After

### ❌ Before (단순한 질문)

```text
회원 ID 100만 개를 저장할 건데, 배열 쓸까? 리스트 쓸까? 검색이 좀 빠르면 좋겠어. 자바로 짤 거야.
```

### ✅ After (프롬프트 적용 결과)

```text
(AI의 분석 결과 요약)

배열이나 일반 리스트(ArrayList)의 사용은 추천하지 않습니다! 100만 개의 데이터에서 특정 ID를 검색할 때 최악의 경우 100만 번의 탐색(O(N))이 발생하여 성능 병목이 심각해집니다.

추천 자료구조: Hash Set (해시 집합)
- 특성: 중복을 허용하지 않고, 순서를 보장하지 않는 데이터의 집합.
- 검색/삽입/삭제 시간 복잡도: 평균 O(1)
- 추천 이유: 해시 함수를 통해 데이터가 저장된 메모리 위치를 즉시 계산하므로, 데이터가 100만 개든 1,000만 개든 단 한 번의 연산만으로 존재 여부를 매우 빠르게 확인할 수 있습니다.

추천 클래스 (Java):
- 기본 추천: `java.util.HashSet` (순서 상관없는 압도적 속도)
- 동시성 보장이 필요한 경우: `java.util.concurrent.ConcurrentHashMap.KeySetView` 또는 `Collections.newSetFromMap(new ConcurrentHashMap<>())`
```
