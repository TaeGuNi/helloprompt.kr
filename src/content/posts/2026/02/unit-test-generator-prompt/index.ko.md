# 📝 테스트 코드 작성, 5분 만에 끝내는 프롬프트

- **🎯 추천 대상:** 주니어 개발자, 프론트엔드/백엔드 엔지니어, QA
- **⏱️ 소요 시간:** 30분 → 5분 단축
- **🤖 추천 모델:** 코딩 특화 모델 (Claude 3.5 Sonnet, GPT-4o 등)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐☆

> _"당장 내일이 배포인데, 메인 비즈니스 로직 짜기도 벅찬 상황에서 테스트 코드는 도대체 언제 작성하나요?"_

![\"테스트 코드 작성, 5분 만에 끝내는 프롬프트\"](/images/hooks/unit-test-generator-prompt.jpg)

테스트 코드의 중요성은 누구나 뼈저리게 공감합니다. 하지만 막상 작성하려고 하면 메인 비즈니스 로직을 구현하는 것보다 훨씬 더 많은 시간과 에너지를 빼앗기곤 하죠. 까다로운 엣지 케이스(Edge Case)를 끊임없이 고민하고, 복잡하게 얽힌 수많은 외부 API 의존성을 일일이 모킹(Mocking)하다 보면 결국 "일단 이번 배포는 이대로 가시죠"라며 현실과 타협하는 자신을 발견하게 됩니다.

이제 메인 로직만 복사해서 붙여넣으세요. 지루한 초기 보일러플레이트 세팅부터 개발자가 놓치기 쉬운 예외 케이스 검증까지, AI가 완벽하게 작동하는 무결점 유닛 테스트 코드를 단숨에 작성해 드립니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **Happy Path부터 Edge Case까지:** 정상적인 동작 흐름은 물론, 예측하기 어려운 예외 상황까지 빈틈없이 검증하는 테스트 시나리오를 자동으로 도출합니다.
2. **원하는 프레임워크 맞춤형:** Jest, PyTest, JUnit 등 현재 프로젝트에 도입된 테스트 프레임워크 문법을 정확히 반영해 즉시 실행 가능한 코드를 생성합니다.
3. **까다로운 Mocking 자동화:** 번거로운 외부 API 호출이나 데이터베이스 의존성을 AI가 스스로 파악하고, 상황에 맞는 Mock 데이터와 Stub을 완벽하게 구성합니다.

## 🚀 해결책: "무한 테스트 코드 생성기"

### 🥉 Basic Version (기본형)

빠르게 기본적인 테스트 뼈대만 잡고 싶을 때 활용해 보세요.

> **역할:** 너는 완벽주의 성향을 가진 시니어 QA 엔지니어이자 테스트 자동화 전문가야.
> **요청:** 다음 코드를 분석해서 예외 케이스를 포함한 유닛 테스트(Unit Test) 코드를 작성해 줘.

### 🥇 Pro Version (전문가형)

복잡한 비즈니스 로직 검증, 까다로운 의존성 모킹(Mocking), 그리고 완벽한 테스트 커버리지가 필요할 때 활용해 보세요.

> **역할 (Role):** 너는 한 치의 오차도 허용하지 않는 10년 차 시니어 QA 엔지니어이자 테스트 자동화 전문가야.
>
> **상황 (Context):**
>
> - 배경: 나는 현재 `[사용 언어 및 프레임워크 (예: TypeScript, Jest)]` 환경에서 프로젝트를 개발하고 있어.
> - 목표: 새로 구현한 비즈니스 로직의 신뢰성을 완벽하게 보장하고, 테스트 커버리지(Test Coverage) 100%를 달성하기 위해 아주 촘촘한 유닛 테스트(Unit Test)를 작성해야 해.
>
> **요청 (Task):**
>
> 1. 제공된 코드를 심층 분석하여 **테스트 케이스 목록**을 가장 먼저 마크다운 리스트 형식으로 정리해 줘. (Happy Path, 예외 및 실패 케이스, 경계값 분석을 반드시 포함할 것)
> 2. `[사용할 테스트 라이브러리 (예: Jest)]`를 기반으로, 복사해서 즉시 실행할 수 있는 무결점 테스트 코드를 작성해 줘.
> 3. 코드 내에 외부 의존성(API 호출, DB 연결, 날짜 및 시간 함수 등)이 존재한다면, 완벽하게 Mocking 처리를 해 줘.
> 4. 작성되는 모든 `it` 또는 `test` 블록 내부에는 GIVEN-WHEN-THEN 패턴을 적용하여 명확한 주석을 달아 줘.
>
> **코드:**
>
>
> `[여기에 테스트할 코드를 붙여넣으세요]`
>
>
> **제약사항 (Constraints):**
>
> - 출력 결과는 반드시 마크다운 코드 블록 안에 담아 줘.
> - 테스트 설명(Description)은 누구나 이해할 수 있는 명확한 한국어 문장으로 작성해 줘. (예: "0으로 나누기를 시도하면 에러를 발생시켜야 한다")
> - 코드는 중간에 생략하지 말고(No placeholder), 복사 후 즉시 실행 가능한 완성된 형태로 제공해 줘.
>
> **주의사항 (Warning):**
>
> - 존재하지 않는 라이브러리나 메서드를 임의로 지어내지 마. (할루시네이션 절대 금지)

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 단순히 '테스트 코드를 대신 짜주는' 수준을 아득히 뛰어넘습니다. 특히 **Task 1의 '테스트 케이스 목록 도출'** 단계야말로 이 프롬프트가 지닌 진정한 가치라고 할 수 있죠. AI가 해당 코드를 어떻게 분석하고 해석했는지 먼저 눈으로 확인함으로써, 개발자가 미처 고려하지 못했던 치명적인 엣지 케이스(예: Null 입력, 타입 불일치 등)를 사전에 완벽하게 방어할 수 있습니다.

실무 환경에서는 동일한 테스트 프레임워크를 사용하더라도 팀마다 선호하는 모킹(Mocking) 방식이나 코드 컨벤션이 미묘하게 다를 수 있습니다. 만약 AI가 생성한 코드가 현재 프로젝트의 기준과 맞지 않는다면, 프롬프트 하단에 `[우리 팀의 Mocking 예시 코드]` 스니펫을 하나만 추가해 보세요. AI가 팀의 고유한 스타일을 즉각적으로 학습해 훨씬 더 정교하고 실무에 바로 투입할 수 있는 다듬어진 코드를 출력할 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코드가 너무 길어서 AI의 답변이 중간에 자꾸 끊기면 어떻게 해야 하나요?**
  - A: 함수나 클래스 단위로 잘게 쪼개어 요청하는 방식을 강력히 권장합니다. 거대한 파일을 통째로 입력하면 AI가 핵심적인 엣지 케이스를 놓칠 확률이 높아집니다. 단일 책임 원칙(SRP)이 철저하게 지켜진 함수를 하나씩 분리해 테스트 작성을 요청해 보세요.

- **Q: 데이터베이스 연결 코드가 포함되어 있는데, 이런 경우에도 유닛 테스트 코드 생성이 가능한가요?**
  - A: 네, 완벽하게 가능합니다! 이 프롬프트에는 '외부 의존성 모킹(Mocking)'에 대한 강력한 지시가 포함되어 있습니다. AI가 DB Repository나 ORM 호출 구간을 스마트하게 감지하고, 이를 실제 DB 연결이 필요 없는 가짜 객체(Mock)로 완벽히 대체하는 코드를 알아서 작성해 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **페르소나(Role) 부여:** '완벽주의 시니어 QA 엔지니어'라는 구체적인 역할을 부여함으로써, 단순히 구색만 갖춘 코드가 아니라 경계값(Boundary)까지 집요하게 파고들어 검증하는 고품질 테스트 코드를 작성하도록 유도했습니다.
2. **GIVEN-WHEN-THEN 패턴 강제:** 테스트 코드의 가독성을 극대화하는 업계 표준 패턴을 주석으로 강제화했습니다. 이를 통해 코드를 처음 보는 다른 팀원들도 테스트의 의도와 흐름을 직관적으로 파악할 수 있도록 구조를 탄탄하게 설계했습니다.
3. **모킹(Mocking) 자동화 지시:** 개발자가 유닛 테스트를 작성할 때 가장 큰 병목 구간이 되는 '의존성 주입 및 Mocking' 작업을 AI에게 명시적으로 위임하여, 테스트 작성에 소요되는 시간을 드라마틱하게 단축시켰습니다.

## 📊 증명: Before & After

### ❌ Before (입력 코드)

```javascript
// 단순한 입력 코드 예시
function calculateDiscount(price, discountRate) {
  if (price < 0 || discountRate < 0) {
    throw new Error("유효하지 않은 값입니다.");
  }
  if (discountRate > 100) {
    return 0; // 100% 이상 할인은 무료
  }
  return price - price * (discountRate / 100);
}
```

### ✅ After (결과)

```javascript
describe("calculateDiscount 함수 테스트", () => {
  // 1. 정상 동작 테스트 (Happy Path)
  it("정상적인 가격과 할인율이 주어지면 할인이 적용된 가격을 반환해야 한다", () => {
    // GIVEN
    const price = 10000;
    const discountRate = 20;
    // WHEN
    const result = calculateDiscount(price, discountRate);
    // THEN
    expect(result).toBe(8000);
  });

  // 2. 경계값 테스트 (Edge Case)
  it("할인율이 100%를 초과하면 0원을 반환해야 한다", () => {
    // GIVEN & WHEN
    const result = calculateDiscount(10000, 110);
    // THEN
    expect(result).toBe(0);
  });

  // 3. 예외 처리 테스트 (Failure Case)
  it("가격이 음수일 경우 에러를 던져야 한다", () => {
    // GIVEN, WHEN & THEN
    expect(() => calculateDiscount(-5000, 10)).toThrow(
      "유효하지 않은 값입니다.",
    );
  });
});
```
