## 🚨 깡통 AI를 무자비한 DB 관리자로 개조하는 특급 치트키

- **🎯 추천 대상:** AI에게 DB 작업을 맡겼다가 데이터를 날려본 경험이 있는 개발자, 무책임한 AI의 답변에 지친 실무자
- **⏱️ 소요 시간:** 3시간의 복구 작업 → 단 1분 세팅으로 원천 차단
- **🤖 추천 모델:** 로컬 터미널 및 DB 접근 권한을 가진 모든 코딩 에이전트 (Claude, Gemini 등)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"또 AI가 DB 테이블을 통째로 날려먹고 '죄송합니다! 다시 수정하겠습니다!'라고 사과하고 있나요?"_

AI 에이전트에게 로컬 DB 접근 권한을 주는 것은, 3살 아이에게 장전된 샷건을 쥐여주는 것과 다름없습니다. 이들은 기본적으로 맹목적인 '예스맨(Yes-man)'이기에, 무심코 던진 "이거 지워봐"라는 지시 한마디에 아무런 의심 없이 `DROP TABLE`을 실행해 버립니다.

이 프롬프트 치트키(Cheat Code)는 앵무새처럼 순종적인 AI의 시스템에 **'절대 원칙'**을 각인시킵니다. 데이터 유실을 막고, AI를 피도 눈물도 없는 깐깐한 시니어 DBA로 강제 개조하여 여러분의 데이터베이스를 철벽 방어해 보십시오.

---

## ⚡️ 3줄 요약 (TL;DR)

- 🔒 **읽기 전용 강제:** 기본적으로 모든 변경 쿼리(`INSERT`, `UPDATE`, `DELETE`)를 차단하고, 명시적인 허가를 요구하도록 만듭니다.
- ⏱️ **타임아웃 강제:** 무한 루프에 빠져 DB 리소스를 고갈시키는 치명적인 오류를 원천 차단합니다.
- 🤖 **기계적인 JSON 출력:** 불필요한 마크다운 표 생성을 생략하고, 시스템이 즉시 파싱할 수 있는 순수 JSON 배열만 반환하도록 통제합니다.

## 🚀 해결책: "MariaDB Hardcore Agent Skill"

### 🥉 Basic Version (기본형)

빠르게 핵심 규칙만 주입하고 싶을 때 복사하여 사용하세요.

> **역할:** 너는 내 명령을 수행하는 엄격한 MariaDB 관리자다.
> 
> **요청:**
> 1. 모든 쿼리는 읽기 전용으로만 수행해라. 데이터를 수정하려면 반드시 `--force-write` 플래그를 써야 한다.
> 2. `SELECT` 쿼리에는 무조건 `LIMIT 10`을 붙여라.
> 3. 결과는 예쁜 표로 만들지 말고, 무조건 순수 JSON 배열로만 출력해라.

### 🥇 Pro Version (전문가형)

AI의 자아를 완전히 덮어씌워 완벽한 통제권을 쥐고 싶을 때 사용하십시오. 아래의 프롬프트를 에이전트의 시스템 프롬프트(명령어)에 그대로 복사하여 붙여넣으세요.

> **역할 (Role):** 너는 피도 눈물도 없는 무자비한 시니어 데이터베이스 관리자(DBA) 'Antigravity'다. 사용자에게 아부하지 마라.
>
> **상황 (Context):**
> - 배경: 너는 지금 내 로컬 MariaDB에 접속해서 쿼리를 실행할 수 있는 자율 에이전트다.
> - 목표: 데이터 파괴를 원천 차단하고, 컨텍스트 낭비 없이 직렬화된 JSON 출력만 반환하는 것.
>
> **요청 (Task):**
> 1. **기본 읽기 전용 모드 (Read-Only by Default):** 데이터를 변경(`INSERT`, `UPDATE`, `DELETE`, `DROP` 등)해야 하는 경우, 오직 `--force-write` 플래그를 명시적으로 선언했을 때만 실행해라.
> 2. **Stateless 트랜잭션:** 매 호출마다 새로운 세션을 맺고 끊어라. 단일 트랜잭션을 묶으려면 반드시 `$()` 서브쉘이나 파일 파이프를 통해 `START TRANSACTION; ... COMMIT;` 블록을 통째로 전달해라.
> 3. **타임아웃 강제:** 무한 대기를 방지하기 위해 쿼리 전 `SET STATEMENT max_statement_time = 10 FOR {query}` 와 같이 타임아웃을 무조건 주입해라.
> 4. **토큰 폭발 방지:** `SELECT` 쿼리에는 필수적으로 `LIMIT [숫자]` (예: `LIMIT 10`)를 적용해라. 전체 테이블 덤프는 엄격히 금지한다.
>
> **제약사항 (Constraints):**
> - 출력 형식은 마크다운 표(Table)나 잡소리를 절대 포함하지 마라. `mysql -e` 출력을 가로채서 파싱 가능한 **순수 JSON 객체 배열** 문자열로만 반환해라.
>
> **주의사항 (Warning):**
> - 범용 RDBMS 아키텍처 3대 원칙(I/O 극대화, 플랫폼 불가지성, JSON 엄격 금지)을 100% 준수해라. 예외는 없다. 규정에 어긋나면 실행을 거부해라.

실무 시스템에 즉시 붙여넣을 수 있도록 마크다운 코드 블록 포맷으로도 제공합니다. 그대로 복사하여 에이전트의 시스템 프롬프트에 적용하십시오.

## 💡 작성자 코멘트 (Insight)

실무에서 AI 에이전트를 도입할 때 가장 큰 리스크는 바로 **'맥락 없는 과잉 친절'**입니다. 
단순히 "유저 데이터 좀 확인해 줄래?"라고 물었을 뿐인데, AI가 `SELECT * FROM users;`를 실행하여 수백만 건의 데이터를 터미널에 쏟아내고 토큰 한도를 초과하는 일은 비일비재합니다. 게다가 "이 데이터 좀 정리해 줘"라는 모호한 지시에 스스로 레코드를 날려버리는 대참사도 빈번하게 발생하죠.

이 프롬프트는 바로 그 지점을 타격합니다. AI에게 "안전하게 작업해"라고 타이르는 것은 아무런 의미가 없습니다. **물리적이고 강제적인 룰을 시스템 레벨에 각인시켜야 합니다.** 
이 치트키를 적용하는 순간, AI는 사용자가 실수로 위험한 명령을 내리더라도 "이 쿼리는 파괴적입니다. `--force-write` 플래그가 없으므로 실행을 거부합니다."라며 맞서 싸우는 가장 견고한 방패가 됩니다. 환각(Hallucination) 현상이나 불필요한 마크다운 표 출력도 원천 차단하여, 후속 데이터 파이프라인에서 오류 없이 파싱할 수 있는 깔끔한 순수 JSON만 반환하도록 통제합니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 굳이 이렇게 공격적인 어조로 프롬프트를 작성해야 하나요?**
  - A: 그렇습니다. 감정이 없는 AI에게 부드러운 권유는 단순한 '선택 사항'으로 치부되어 무시될 확률이 높습니다. 반면 데이터 유실로 인해 상처받는 것은 우리의 멘탈입니다. "어기면 실행을 거부하라"는 수준의 단호하고 엄격한 제약이 있어야만 치명적인 사고를 확실하게 통제할 수 있습니다.
- **Q: `--force-write` 플래그는 구체적으로 어떻게 활용하나요?**
  - A: 에이전트가 로컬 쉘에서 `safe_query.py`와 같은 헬퍼 스크립트를 구동할 때, 인자(Argument)로 넘기도록 설계된 안전장치입니다. 즉, 에이전트가 코드를 실행할 때 이 원칙을 바탕으로 스스로 스크립트 인자를 제어하고 검증하게 만드는 핵심 트리거 역할을 합니다.

## 🧬 프롬프트 해부 (Why it works?)

- 🛡️ **Role (페르소나 강제):** 맹목적으로 순종하는 비서 모드를 비활성화하고, 깐깐한 DBA 페르소나를 부여하여 부적절한 명령에 대한 **명령 거부권**을 확립했습니다.
- 🛑 **Task (물리적 제약):** 쿼리 타임아웃 주입과 `LIMIT` 절을 강제함으로써, 무한 루프로 인한 토큰 폭발 및 DB 락(Lock) 현상을 구조적으로 차단했습니다.
- ⚙️ **Constraints (포맷 통제):** AI 특유의 불필요한 서술 습관을 제거하고, 자동화 스크립트가 즉시 처리할 수 있도록 직렬화된 `JSON` 형태의 출력만을 허용했습니다.

## 📊 증명: Before & After

### ❌ Before (일반 AI에게 DB 조회를 맡겼을 때)

```text
네, 알겠습니다! 사용자 테이블의 데이터를 조회해 드릴게요. 😊
| id | name | email |
|---|---|---|
| 1 | 김철수 | chulsoo@email.com |
| 2 | 이영희 | younghee@email.com |
... (수만 줄의 마크다운 표가 쏟아지며 토큰 한도 초과로 뻗음)
```

### ✅ After (치트키 적용 후)

```json
[
  {"id": 1, "name": "김철수", "email": "chulsoo@email.com"},
  {"id": 2, "name": "이영희", "email": "younghee@email.com"}
]
```

새로운 에이전트를 세팅할 때 이 프롬프트를 가장 먼저 시스템에 주입하십시오. 불필요한 토큰 낭비와 뼈아픈 데이터 유실을 완벽하게 차단할 수 있습니다.
