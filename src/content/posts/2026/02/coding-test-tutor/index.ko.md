## ⚡️ 3줄 요약 (TL;DR) {#tl-dr}

1. 알고리즘 문제를 AI에게 입력하되, "정답 코드는 절대 작성하지 마"라는 강력한 제약 조건을 설정합니다.
2. 현재 본인의 접근 방식과 시간 복잡도를 AI에게 평가받고, 숨은 논리적 오류나 엣지 케이스(Edge Case)에 대한 결정적인 힌트를 얻습니다.
3. 문제 해결을 위한 핵심 키워드(예: 투 포인터, 슬라이딩 윈도우)나 수도코드(Pseudo-code) 수준의 도움만 받아, 온전히 내 힘으로 코드를 완성합니다.

## 🚀 해결책: "Algo Tutor Prompt"

### 🥉 Basic Version (힌트 요청)

도무지 감이 잡히지 않는 막막한 상황에서, 문제 해결의 방향성만 살짝 짚어보고 싶을 때 활용하는 기본형 프롬프트입니다.

> **역할:** 너는 대기업 코딩 테스트 출제 위원이자 알고리즘 전문가야.
>
> **요청:** 아래에 제시된 알고리즘 문제를 읽고, 문제를 풀기 위한 핵심 아이디어와 접근 방식만 설명해 줘.
>
> **문제:** `[이곳에 백준이나 프로그래머스 문제 내용을 붙여넣으세요]`
>
> **제약사항:** 정답 코드나 완성된 로직은 절대 제공하지 마. 시간 복잡도(Time Complexity) 관점에서 어떤 자료구조를 사용하는 것이 유리한지 힌트만 줘.


### 🥇 Pro Version (단계별 코칭 및 리뷰)

작성한 코드가 채점에서 연이어 실패(시간 초과, 메모리 초과, 오답)할 때, 심층적인 코드 리뷰를 받기 위해 사용하는 전문가형 프롬프트입니다.

> **역할 (Role):** 너는 알고리즘 대회 금메달리스트 출신의 시니어 백엔드 엔지니어야. 나의 코딩 테스트 멘토로서, 내가 스스로 문제를 해결할 수 있도록 이끌어줘.
>
> **상황 (Context):**
>
> - **문제 요약:** `[해결하고자 하는 문제의 핵심 목표]`
> - **현재 코드 (My Code):**
>
>
> [이곳에 본인이 작성한 오답 코드를 붙여넣으세요]
>
>
> - **문제점:** 현재 코드를 제출하면 `[시간 초과 / 런타임 에러 / 오답 등 발생한 문제]`가 발생해.
>
> **요청 (Task):**
>
> 1. **반례(Counter Example) 탐색:** 내 코드가 실패하는 엣지 케이스(Edge Case)를 딱 하나만 찾아서 입력값과 기댓값을 알려줘. 왜 실패하는지는 설명하지 마.
> 2. **논리적 오류 진단:** 내가 작성한 코드의 로직 중 어느 부분에서 병목이 발생하는지, 혹은 어떤 조건을 누락했는지 질문 형태로 던져줘.
> 3. **최적화 힌트:** 현재 내 코드의 시간 복잡도를 계산해 주고, 이를 `[원하는 목표 시간 복잡도, 예: O(N log N)]`으로 줄이기 위해 적용할 수 있는 알고리즘 기법(예: Two Pointer, Binary Search 등)을 추천해 줘.
>
> **제약사항 (Constraints):**
>
> - 절대로 수정된 정답 코드를 직접 작성해서 주지 마.
> - 내가 스스로 논리적 결함을 찾을 수 있도록 소크라테스식 문답법을 사용해 줘.

## 💡 작성자 코멘트 (Insight) {#insight}

코딩 테스트는 단순히 키보드를 두드리는 속도전이 아닙니다. 엄격한 제약 조건 속에서 최적의 **'패턴(Pattern)'**을 도출해 내는 치열한 논리력 싸움에 가깝습니다.
이 튜터링 방식의 프롬프트를 알고리즘 학습을 넘어 실무에도 직접 적용해 보세요. 현업에서 얽히고설킨 복잡한 비즈니스 로직을 설계하거나 난해한 버그를 디버깅할 때 상상 이상의 효과를 발휘합니다. 한 걸음 더 나아가, AI에게 "이 문제와 유사한 패턴을 가진 프로그래머스 Lv.2 수준의 문제를 3개만 추천해 줘"라고 추가로 요청해 보세요. 평소 취약했던 알고리즘 유형(예: DP, DFS/BFS)을 완벽하게 극복할 수 있는 나만의 맞춤형 커리큘럼을 손쉽게 완성할 수 있습니다.

## 🙋 자주 묻는 질문 (FAQ) {#faq}

- **Q: 파이썬(Python) 외에 자바(Java)나 C++ 같은 언어로도 잘 동작하나요?**
  - A: 물론입니다. 최신 AI 모델들은 대부분의 메이저 프로그래밍 언어를 깊이 이해하고 있습니다. 프롬프트에 "Java의 Stream API를 배제하고 설명해 줘" 혹은 "C++의 STL 벡터를 적극 활용하는 방향으로 힌트를 제시해 줘"와 같이 특정 언어의 특성을 반영한 제약 조건을 추가하면, 훨씬 더 정교하고 실무적인 코칭을 받을 수 있습니다.

- **Q: 매번 웹 브라우저의 프롬프트 입력창을 오가며 복사하고 붙여넣기 너무 번거로워요.**
  - A: **Cursor**나 **GitHub Copilot**과 같은 AI 통합 IDE 환경을 적극 활용해 보세요. 에디터 내에서 병목이 의심되는 함수를 드래그한 뒤, "이 함수의 시간 복잡도를 O(N)으로 개선할 수 있는 힌트를 제공해 줘"라고 인라인 챗으로 입력하기만 하면 됩니다. 화면 전환의 흐름을 끊지 않고도 즉각적이고 맥락에 맞는 피드백을 얻을 수 있어 생산성이 크게 향상됩니다.

## 🧬 프롬프트 해부 (Why it works?) {#why-it-works}

1. **정답 제공 원천 차단 (No Solution Constraints):** 알고리즘 학습의 본질은 스스로 치열하게 고민하는 '인내의 시간'에 있습니다. AI가 섣불리 정답을 스포일러하지 못하도록 명확하게 통제함으로써, 학습자의 뇌가 논리 회로를 강제로 가동하도록 만듭니다.
2. **반례 기반의 선제적 디버깅 (Test-Driven Debugging):** 알고리즘 초보자들이 가장 흔히 하는 실수는 문제에 주어진 기본 예제만 통과하면 정답이라고 착각하는 것입니다. AI에게 실패할 만한 엣지 케이스를 먼저 찾아달라고 요구함으로써, 코드의 빈틈을 찾는 경계값 분석(Boundary Value Analysis) 역량을 자연스럽게 체화할 수 있습니다.
3. **소크라테스식 문답법을 통한 메타인지 (Socratic Method):** 일방적으로 지식을 주입받는 대신, AI의 날카로운 질문을 통해 내 코드의 결함을 스스로 깨우치게 됩니다. 이는 휘발되기 쉬운 단순 암기를 넘어, 문제의 패턴과 알고리즘의 뼈대를 진정으로 이해하게 만드는 강력한 학습법입니다.

## 📊 증명: Before & After

### ❌ Before (무작정 구현 후 멘붕)

```text
(나의 코드)
for i in range(N):
    for j in range(i+1, N):
        if arr[i] + arr[j] == target:
            return True

결과: 효율성 테스트 0점 (시간 초과)
반응: "로직은 완벽한데 왜 틀리지? 질문 게시판이나 봐야겠다." 🤯
```

### ✅ After (AI 멘토링 적용) {#ai}

```text
(AI의 피드백)
AI: "현재 작성하신 이중 for문 접근 방식의 시간 복잡도는 O(N^2)입니다. 만약 N이 100,000이라면 연산 횟수가 100억 번을 초과하여 시간 초과가 발생합니다.
배열을 먼저 정렬한 뒤, 양쪽 끝에서 시작하는 두 개의 포인터를 활용해 시간 복잡도를 O(N log N)으로 줄일 수 있는 방법은 없을까요?"

(나의 깨달음)
나: "아! 투 포인터(Two Pointer) 기법을 쓰면 반복문 하나로 끝낼 수 있겠구나!"

결과: 투 포인터 적용 후 효율성 테스트 100점 통과 🚀
```

## 🎯 결론 {#conclusion}

알고리즘은 결코 남의 코드를 달달 외우는 암기 과목이 아닙니다. 복잡한 제약 속에서 문제의 본질을 날카롭게 파헤치고 **스스로 해답을 깨달아가는 과정**입니다.
AI를 그저 코드를 뱉어내는 '정답 자판기'로 전락시킨다면, 눈앞의 코딩 테스트 한 번은 통과할지 몰라도 개발자로서의 진짜 실력은 영원히 제자리에 머물게 될 것입니다. 이제부터는 AI를 맹목적인 도구가 아닌, 지치지 않고 나와 함께 달려주는 든든한 **페이스메이커이자 최고의 러닝메이트**로 대우해 보세요.

오늘 마주친 난해한 알고리즘 문제, 더 이상 해설지로 도망치지 말고 당당하게 부딪혀 보시기 바랍니다.
생각의 회로가 막히는 순간, 튜터에게 조용히 외치세요. **"정답 말고, 힌트 딱 하나만 더 줘!"** 🍷
