## 📝 静态菜单的终结：AI Agent 生成式 UI 设计提示词

- **🎯 推荐对象：** 前端开发人员、UX/UI 设计师、AI 产品经理
- **⏱️ 耗时：** 2 小时 → 缩短至 5 分钟
- **🤖 最佳性能：** Claude 3.5 Sonnet, GPT-4o (建议使用最新推理模型)

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐☆

> _“您是否仍然希望用户翻遍汉堡菜单来手动查找所需功能？”_

作为前端开发人员，数十年来我们一直被困在完全确定性（Deterministic）的世界里。为了将策划者要求的无数功能塞进屏幕的某个角落，我们不得不制作深不见底的汉堡菜单，设计复杂的巨型下拉菜单。如果用户想要的功能隐藏在菜单的第三层（Depth），我们只能祈祷用户的心智模型能奇迹般地与我们的信息架构（IA）相匹配，并不断地对页面进行硬编码。每当增加新需求时，侧边栏就会变得臃肿，路由结构像乱麻一样缠绕在一起，你是否也曾陷入过这种痛苦的维护泥潭？

这种静态方式的局限性非常明显。用户现在不再想为了获取所需信息而去学习网站的结构。在点击无数仪表盘屏幕和设置过滤器的过程中产生的认知负荷（Cognitive Load），最终会导致流失率增加。更糟糕的是，随着 AI 时代的到来，用户的期望值已经发生了变化。在这个只需输入“请以表格形式显示上个月的市场营销费用趋势”就能自动生成页面的时代，如果你发现自己还在为了制作在移动端不崩溃的表格而与媒体查询（Media Query）作斗争，雕琢数百行 CSS，强烈的自我怀疑感会油然而生。预先制作出涵盖用户所有不同语境的 UI，这从一开始就是一项不可能完成的任务。

然而，大语言模型（LLM）的爆发式发展在前端生态系统中催生了一个革命性的救星——**“生成式 UI (Generative UI)”**。现在，界面不再是开发人员在部署时确定的死像素集合，而是 AI 实时掌握用户的当前意图和对话语境，并即时组装成那一刻最完美形式的动态组件的**活的界面 (Living Interface)**。如果用户想以图表形式查看数据，LLM 会自行判断并指示渲染图表组件；如果用户想要输入数据，则渲染表单（Form）组件。

因此，前端工程师的角色已经进化到了一个全新的维度。我们不再是“将给定的 Figma 页面漂亮地搬到浏览器上的简单实现者”，而是“构建能够根据情况自行生成页面的智能系统，并设计安全网（Component Registry）和约束条件以防止 AI 渲染错误页面的 **UI 架构师 (Architect)**”。在本文中，我们将公开终极的 **“生成式 UI 编排提示词”**，它能控制 AI 在没有致命幻觉（Hallucination）的情况下，100% 仅利用我们允许并验证过的 UI 组件（图表、数据表格、按钮等），稳定地返回前端可立即渲染的完美 JSON。现在，请从复杂的路由和菜单设计的束缚中解脱出来，亲手构建一个与用户呼吸同步、不断进化的智能界面吧。

AI 理解语境，并组装预定义的 `SummaryCard`、`BarChart`、`ActionButton` 组件，立即返回完美的 UI 负载。前端接收此 JSON 后无需额外解析，即可立即渲染美观且可交互的仪表盘页面！

## ⚡️ 3 句总结 (TL;DR)

1. **静态 UI 的终结：** 用户费力探索汉堡菜单的时代已经结束。AI 会洞察用户的隐藏意图，即时渲染优化的交互式 UI。
2. **结构化输出控制：** 强制 LLM 返回可立即水合（Hydration）的完美 JSON 负载，防止其吐出破坏前端生态系统的无用长文本。
3. **利用组件注册表：** 为了防止致命的幻觉（Hallucination）错误，明确控制 AI 的界限，使其仅在预先完成类型指定的安全公司内部 UI 组件（图表、表单等）列表内组装乐高积木。

## 💡 作者点评 (洞察与使用技巧)

在将生成式 UI (Generative UI) 这一迷人的概念引入实际生产（Production）环境时，许多资深前端开发人员和 AI 工程师都会犯下一个共同的、也是最致命、最惨痛的错误。那就是由于过度自信 LLM 的能力，从而“根据用户请求自由编写合适的 UI 代码”，将页面生成的控制权完全且不负责任地交给 AI。

如果让模型在这样一张白纸上自生自灭，会发生什么样的惨剧呢？AI 会凭想象滥用并不存在的 Tailwind CSS 工具类，或者不负责任地吐出未闭合的 HTML 标签。甚至完全无视公司内部的设计系统（Design System）规范，应用自己那套古怪的内联样式（Inline Style）。最终，整个 React 或 Next.js 应用会因为解析错误而崩溃，出现惨不忍睹的白屏死机（White Screen of Death），用户体验坠入深渊。这不是创新，而是灾难。

我设计的这个编排器提示词所具有的真正的、压倒性的力量，就在于从系统层面强有力地强制执行 **“组件注册表 (Component Registry)”这一不可逾越的高耸控制围墙**。在生成式 UI 环境中，AI 绝不是一个可以随心所欲塑造乐高积木材质和形状、无中生有的设计师。相反，它必须严格作为一名**“最佳辅助操作员 (Assistant Operator)”**发挥作用，根据用户的实时语境，从我们根据公司设计系统指南预先精心开发并使用 TypeScript 进行安全类型定义（Typing）的规格化乐高积木中，挑选出最优雅、最高效的组合。

在实务中应用此提示词时，强烈建议与 Vercel 的 AI SDK（如 `ai/rsc`, `ai/core` 等）提供的流式 UI 功能相结合。当用户开启麦克风说“请按地区以饼图形式显示明年预计销售额”的一瞬间，后端 LLM 会立即基于此提示词喷涌出指示渲染 `<PieChart />` 组件的 JSON 流。前端以块（Chunk）为单位接收此流，并立即对页面进行水合（Hydration）。

结果是，您既能完美保证现有前端代码库 100% 的完整性和类型安全性，又能实现一个维度完全不同的动态界面，灵活且顺滑地应对用户的任何突发自然语言请求或非结构化数据探索需求。思考菜单结构的时间将完全消失，您可以全身心地投入到设计更好的组件中，真正实现“组件驱动开发 (CDD)”的乌托邦。不受控的 AI 会失控，但在精密设计的约束条件下的 AI 会将您的产品提升至艺术境界。

## 🙋 常见问题 (FAQ)

- **Q：前端具体如何渲染返回的 JSON 负载？**
  - A：以 React 环境为例进行说明。通过准确读取返回的 JSON 数据中的 `type` 字段值，并注入一个单独实现的**动态渲染工厂 (Dynamic Renderer Factory) 组件**，将其与代码库中预先导入（Import）的实际视图组件进行 1:1 映射（Mapping），即可非常优雅地解决。

- **Q：为了等待 LLM 生成响应，初始页面加载会不会延迟太久？**
  - A：为了解决这种延迟，绝对需要 **“流式水合 (Streaming Hydration)”** 技术。不要让用户在空白屏幕前等待巨大的 JSON 负载完全生成，而是每当完成单个组件单位的块（Chunk）解析时，就让 UI 渐进式地（Progressive）弹出。这样可以将用户的感知延迟（Perceived Latency）显著降低至接近零。

- **Q：公司内部业务数据在安全方面非常敏感，直接交给 LLM 安全吗？**
  - A：绝对不能将实际的业务原始数据（Raw Data）直接交给外部 LLM API。LLM 的角色仅限于解析用户的自然语言**“意图”**，并生成一份指示“调用哪个 API 端点，渲染何种形式的 UI 骨架”的**“指令书 (UI Schema)”**。实际敏感数据的获取（Fetch）和最终的组件绑定（Binding）必须在您的前端客户端或内部后端服务器端完全独立且安全地执行，必须彻底分离架构。

## 🧬 提示词剖析 (为什么有效？)

1. **基于注册表 (Registry) 的严格控制：** 以白名单 (Whitelist) 方式明确限制了 LLM 可以使用的可用组件列表。通过这种方式，从架构层面从源头上杜绝了模型随意编造任意 CSS 或不存在的标签从而破坏页面的幻觉 (Hallucination) 风险。
2. **100% 强制结构化输出 (Structured Output)：** 为了让前端应用无需复杂预处理即可立即解析 (Parsing)，强制执行了抽象语法树形式 (`type`, `props` 结构)。这在 AI 和前端之间建立了一个非常牢固且清晰的数据契约 (Contract)。
3. **智能的边缘情况 (Fallback) 机制：** 针对用户提示词模糊或缺失必要参数的情况，注入了行动指南。与其强行渲染错误数据导致信任度下降，不如通过返回 `TextInput` 表单来反问用户补充信息，以自然的**对话式 UX (Conversational UX)** 优雅地解决问题。
