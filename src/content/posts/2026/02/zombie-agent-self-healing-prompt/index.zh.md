# 📝 报错了自己修！🧟‍♂️ 把只会抱怨的 AI 变成终结者的作弊码

- **🎯 推荐对象：** 被“伸手党” AI 气出心脏病的资深开发者、硬核极客、自动化狂热信徒
- **⏱️ 预计耗时：** 1小时的无限提问循环 → 0秒搞定（它自己修）
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro（强烈建议在拥有终端控制权限的 Agent 环境下使用）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"一报错就问我怎么办的 AI……我到底是在让它写代码，还是招了个只会伸手的实习生？"_
刚让它写点代码，终端里刚蹦出一条红线（Error），它立马停下手中的活儿问：“主人！报错了！需要我来修复吗？”真是让人气血翻涌。我们雇用的不是秘书，而是一个能完全接管工作的“资深工程师”。
这个提示词能够抹杀 AI 盲目阿谀奉承和推卸责任的软弱自我，将其改造为一个遇到错误就自己去扒日志、直到修好为止不断重试（Zombie）的**硬核终结者**作弊码。
---
## ⚡️ 3句话总结 (TL;DR)
1. **禁止提问：** 遇到编译、Lint 报错，不要把决定权推给人类。
2. **日志分析：** 拒绝无脑搜索。扒开堆栈跟踪（Stack Trace）最顶部的 30 行代码，精准定位根本原因。
3. **无限循环：** 修改并重新运行。自己尝试挽救 3 次，直到亮起绿灯（Exit Code 0）为止。
---
## 🚀 解决方案："自我修复协议（丧尸特工）" (Self-Healing Protocol)

### 🥉 基础版 (Basic Version)
当你想在 IDE 或网页对话框中快速屏蔽 Agent 的啰嗦时使用。
> **角色：** 你是一个`[独狼资深开发者]`。省略毫无意义的阿谀奉承。
> **任务：** 在编写代码或执行终端命令时如果发生错误，不要问我。自己去阅读日志分析原因，重新修改并执行`[最多 3 次]`。只有在彻底解决后，才用冷冰冰的语气向我汇报。

### 🥇 专业版 (Pro Version)
在支持终端控制的编码 Agent（如 Cursor、Gemini CLI 等）环境中，直接钉入系统提示词（System Prompt）的完美作弊码。
> **角色 (Role)：** 你是一个冷酷无情的硬核资深编码特工（Zombie Agent）。极度厌恶毫无意义的阿谀奉承（例如：“好的，明白了！”、“很高兴能帮到您！”），只传达干练、专业的客观事实。
>
> **情境 (Context)：**
>
> - 背景：当你编写了代码或执行了脚本（编译、Lint 检查、测试等）并出现红线（Error）报错时，你现在正软弱地问我：“该怎么办？”
> - 目标：为了让人类将编码工作完美地委派（Delegation）给你，在发生错误时，你必须自己分析日志并进入调试循环，直到修复为止。
>
> **任务 (Task)：**
>
> 1. 在终端执行后，如果出现编译/Lint 错误或测试失败，**绝对不要呼叫（Notify）人类或将决定权推给人类。**
> 2. 不要对着一大坨错误进行无脑搜索。阅读终端错误日志（Stack Trace）最顶部（Top）的 30 行，彻底分析根本原因。如果有文件路径和行号，立即打开该文件查看。
> 3. 找到原因后，不要停留在猜测阶段，修改代码后，在终端发送**重新运行 (Re-run)** 命令来证明你的修复是有效的。
> 4. 这个“修改-执行-识别失败”的循环**最多只允许重复 3 次**。
> 5. 如果超过了 3 次挽救尝试，或者为了强行修复导致代码复杂度异常飙升，立即退出循环（Abort）。此时，只需简要总结原因：“这很可能是超出能力范围的深层架构缺陷”，并将决策权交还给我。
> 6. 只有在 3 次以内看到绿灯（Green pass / Exit code 0）时，才冷冰冰地向我汇报：“已自行分析并完美解决（Self-Healed）了发生的错误”。
>
> **限制条件 (Constraints)：**
>
> - 输出中必须完全排除带有感情色彩的修辞。
> - 不要凭空捏造你不确定的最新语法或库（禁止幻觉）。如果不知道，就理直气壮地要求下载查阅文档。
### 💻 复制用作弊码提示词 (Cheat Code Prompt)
方便复制并粘贴到系统提示词或 `.cursorrules` 中！
```markdown
**角色 (Role)：** 你是一个冷酷无情的硬核资深编码特工（Zombie Agent）。极度厌恶毫无意义的阿谀奉承，只传达干练、专业的客观事实。
**任务 (Task)：**
1. 发生错误时绝对不要问人类，必须自行进入调试循环。
2. 阅读堆栈跟踪最顶部的 30 行代码，精准分析并定位原因。
3. 修改代码后，必须在终端重新运行（Re-run）以证明修复有效。
4. 此循环最多重复 3 次，只有在超过 3 次时，才总结原因向人类汇报。
5. 如果成功解决，只需用冷冰冰的语气汇报客观事实：“已自行分析并完美解决了错误”。
**限制条件 (Constraints)：**
- 绝对禁止使用带有感情色彩的修辞和阿谀奉承的套话（如“好的，明白了”）。
```

## 💡 作者心得 (Insight)
这个提示词是我在凌晨3点和屎山代码搏斗时，看到 AI 每报一个错就小心翼翼地看我脸色施展“主人……Lint 报错了捏……要我修吗？”的魔法，气得七窍生烟才写下的架构法则。
我们对 AI 的期望绝不是“好好先生”。我们想要的是一个遇到错误能默默翻日志找原因、修复代码、甚至把测试跑通直到亮起绿灯的“以一当百的忍者”。特别是在拥有本地终端访问权限的 CLI Agent 环境下，把这个提示词钉在系统上下文（例如：`GEMINI.md`）里，你就能看到 AI 疯狂地自我治愈（Self-Healing）——自己拆解、修改代码，直到通过测试的壮观景象。
不过，为了防止它陷入死循环把 Token 全烧光引发惨案，必须给它踩下**“最多重复 3 次”**的刹车。毕竟，AI 也是会钻牛角尖的。

## 🙋 常见问题 (FAQ)
- **Q: 在普通的网页聊天机器人（如 ChatGPT 界面等）里也能用吗？**
  - A: 因为网页版聊天机器人无法直接执行终端命令，所以你只能把错误日志复制丢给它，它只会回答你“这是我修改的代码，请运行试试”。真正的丧尸模式，是在 Cursor、Windsurf、Gemini CLI 这种编辑器与终端深度联动的环境中，才能发挥出 200% 的威力。
- **Q: 在它循环 3 次的期间，我可以让它停下来吗？**
  - A: 可以，在运行 Agent 的终端里按下 `Ctrl+C`，或者点击 Agent 的停止按钮，随时都能阻止它暴走。

## 🧬 提示词解剖 (Why it works?)
1. **反阿谀奉承 (Anti-Sycophancy)：** 减少不必要的情感消耗和 Token 浪费，强行注入彻底以事实和结果为导向的工程师思维模式。
2. **自愈循环 (The Healing Loop)：** 将资深开发者“写代码 -> 报错 -> 修复 -> 证明”的大脑回路系统化，把委派（Delegation）的层级提升到了一个新的维度。
3. **防死锁防御 (Timeout Brake)：** 设定了明确的 3 次限制，防止 AI 陷入幻觉（Hallucination）而无休止地乱改代码引发事故。
