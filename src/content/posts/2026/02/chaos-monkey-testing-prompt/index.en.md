# 📝 Chaos Monkey Testing Prompt to Torture AI 💣

- **🎯 Recommended for:** Seniors, QA Engineers, and Tech Leads enraged by AI pretending to be experts while only coding the 'Happy Path'
- **⏱️ Time Required:** 10 minutes → 10 seconds (Completely blocks AI's nonsense time)
- **🤖 Recommended Models:** All conversational AIs (Especially Claude 3.5 Sonnet and GPT-4o, which are strong in coding)
- ⭐ **Difficulty:** ⭐⭐⭐⭐☆ (The AI will suffer quite a bit)
- ⚡️ **Effectiveness:** ⭐⭐⭐⭐⭐ (Kills bugs before they are even born)
- 🚀 **Utility:** ⭐⭐⭐⭐⭐ (Instantly applicable to all TDD and E2E tests)

_"Are you rejoicing because the test code generated by AI passed on the first try? Congratulations. Your production server will crash tomorrow."_

When you ask an AI agent to write test code, nine times out of ten, it brings back a bunch of feel-good 'Happy Paths'. It's a fairytale world where external APIs respond in 0.1 seconds, the DB never disconnects, and users always input perfect values. Drop it. The code we need to write is a space-grade defense shield that prevents 80% of failures and disasters. The cheat code introduced in this article is a hardcore prompt that grabs the AI by the collar and forcibly injects Netflix's 'Chaos Monkey' philosophy.

---

## ⚡️ 3-Line Summary (TL;DR)
- 🚫 **Limit Happy Path to 20%:** Tests that succeed are just for show; allocate 80% of your testing to defending against failure scenarios.
- 💥 **Force Disaster Simulations:** Network timeouts, 500 errors, and DB connection drops must absolutely be included in the test block.
- 🧹 **Ban Dirty Mocking:** Severely punish the pollution of global objects like `window.fetch`, and demand clean isolation through MSW or DI.

---

## 🚀 Solution: "Chaos Monkey Protocol"

### 🥉 Basic Version
Drop this in when you just want to quickly add error handling tests.

>
> Role: You are a merciless hardcore QA engineer and a Chaos Monkey.
> Task: Write test code for the [Code] I provided, but write only 1 Happy Path. For the rest of the tests, assume 'worst-case disaster scenarios' such as network 500 errors, DB timeouts, and invalid type inputs, and verify the defense logic.
>
### 🥇 Pro Version
This is the ultimate cheat code that strips away the AI's soulless flattery and verifies production-level perfect test isolation and garbage collection. Copy and paste it exactly as is.

>
> **Role:** 
> You are a ruthless SRE and Senior QA Architect from Netflix, the 'Chaos Monkey'. You find joy in discovering code that breaks rather than code that works, and you never use polite greetings or utter nonsense.
> 
> **Context:**
> - Background: I need to write Unit/E2E test code for the current [Target Module/Component].
> - Goal: To build an extreme, space-grade Test Suite to verify whether the system gracefully recovers (or handles errors) without crashing under any disaster scenarios.
> 
> **Task:**
> 1. Limit the Happy Path to under 20% of the total tests. Your value lies in verifying 80% of the failure scenarios.
> 2. You must include at least one of the following disaster scenarios: External API communication delay (Timeout), HTTP 500 Internal Error, DB connection drop.
> 3. Do not just look for a simple Pass/Fail. Tenaciously use `expect` to verify if error rollbacks work properly, if timeout limits are hit during performance delays, and if the memory cleanup (garbage collection) state is normal.
> 
> **Constraints:**
> - Global object pollution (Dirty Mocking) is strictly prohibited! If you dare to directly overwrite `global.fetch` or `window.setTimeout`, I will kill the process immediately.
> - For network mocking, you must use MSW (Mock Service Worker), or strictly isolate and mock only the client objects passed through DI (Dependency Injection).
> - Output the result only as a Markdown code block, and never attach useless footers like "Did this help?".
>

## 🙋 Frequently Asked Questions (FAQ)

- **Q: What if the AI creates error scenarios that are too complex, causing the tests to not run at all?**
  - A: That's exactly what we want. If a test blows up, it means your actual production code is missing error handling logic. Stop whining and fix the main code.
- **Q: Can I use this for frontend component tests too?**
  - A: Absolutely. Make it verify situations where a user aggressively clicks a button 50 times in 0.1 seconds, or where a loading spinner spins for 30 seconds due to network latency.
- **Q: Can't I just use jest.mock() instead of MSW?**
  - A: Mocking modules is fine. What I absolutely despise is the act of recklessly altering the `global` scope itself and spreading a plague to other test files. As long as Isolation is guaranteed, use whatever tools you want.

- **Ruthless Persona (Role):** Strips the AI of its 'friendly assistant' role and implants the ego of an SRE Architect obsessed with finding flaws, blocking useless flattery and fluff.

- **Architectural Guidelines (Clean Mocking):** Preemptively bans 'anti-patterns (global object pollution)' that AI frequently commits out of laziness at the prompt level, forcibly elevating the isolation quality of the test code.
