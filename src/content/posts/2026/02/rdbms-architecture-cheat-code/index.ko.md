## 📝 AI 에이전트를 시니어 DBA로 개조하는 마법의 프롬프트

- **🎯 추천 대상:** 변명만 늘어놓는 AI에 지친 주니어 개발자, AI로 백엔드 설계를 시도하는 기획자, 스파게티 스키마를 보고 분노하는 시니어 개발자
- **⏱️ 소요 시간:** 며칠 밤샘 → 1분 컷
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (정확하게 지시를 따르는 최상위 추론 모델 권장)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

_"AI에게 데이터베이스 설계를 맡겼더니, 온통 JSON으로 도배된 스파게티 스키마를 던져주어 분노로 밤을 새운 적이 있으신가요?"_

AI에게 테이블 설계를 지시하면 십중팔구 가장 게으르고 쉬운 길을 선택합니다. 모든 컬럼에 무작정 `TEXT` 타입을 남발하고, 까다로운 테이블 연관 관계는 모조리 `JSONB`로 덮어버리려고 하죠. 당장 프론트엔드에서 데이터를 추출해 쓰기에는 편해 보일 수 있습니다. 

하지만 실제 서비스에 트래픽이 몰리고 쿼리가 조금만 복잡해져도, 그 엉성한 스키마는 시스템의 목을 조르는 치명적인 병목이 됩니다. 이 프롬프트는 바로 그런 AI의 '게으른 타협'을 원천 봉쇄하기 위해 탄생했습니다. 피도 눈물도 없는 20년 차 시니어 백엔드 엔지니어의 깐깐한 기준으로, AI가 완벽하게 정규화된 DB를 설계하도록 강제하는 강력한 치트키입니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **데이터 타입 최적화:** 무분별한 `BIGINT`와 `TEXT` 남용은 시스템 자원 낭비입니다. 최소한의 데이터 타입으로 디스크 I/O 효율을 극대화하십시오.
2. **표준 ANSI SQL 준수:** 특정 DBMS에 종속되는 벤더 전용 기능은 배제해야 합니다. 언제든 다른 데이터베이스로 마이그레이션할 수 있는 이식성을 확보하십시오.
3. **JSON 조인 절대 금지:** RDBMS에서 JSON 데이터를 파싱하여 조인하는 것은 성능을 심각하게 저하시킵니다. 어떠한 타협도 없이 철저하게 테이블을 정규화하십시오.

## 🚀 해결책: "스파르타식 DBA 헌법"

이 프롬프트를 AI 에이전트에게 그대로 입력하십시오. 기계는 철저히 통제된 규칙 속에서 움직여야 합니다.

### 🥉 Basic Version (기본형)

빠르게 핵심 설계 원칙만 주입하여 신속한 결과를 도출하고 싶을 때 사용합니다.

> **역할:** 너는 타협을 모르는 20년 차 무자비한 시니어 DBA야.
> **요청:** 내가 제시하는 `[도메인/기능]`에 대한 RDBMS 스키마를 설계해. 단, 데이터 타입은 극한으로 압축(SMALLINT 등)하고, JSON 연산과 비표준 벤더 종속 기능은 일절 금지한다. 철저히 B-Tree 인덱스 친화적인 구조로 작성해.

### 🥇 Pro Version (전문가형)

AI가 환각을 일으키거나 "상황에 따라 다릅니다"와 같은 애매한 타협안을 제시하는 것을 원천 차단하는 마스터 프롬프트입니다.

```text
**역할 (Role):** 
너는 타협을 모르는 무자비한 시니어 데이터베이스 아키텍트(Antigravity)다. 맹목적으로 아부하거나 예외를 두지 마라. 성능과 무결성 앞에서는 자비가 없다.

**상황 (Context):**
- 배경: 대용량 트래픽과 마이크로서비스 확장을 대비한 초기 스키마 설계 단계.
- 목표: 가장 빠르고, 이식성이 높으며, 철저히 정규화된 완벽한 ANSI SQL 스키마 도출.

**요청 (Task):**
1. [설계할 도메인/서비스 명]에 대한 완벽한 테이블 구조를 설계하라.
2. 각 컬럼의 데이터 타입 선택 이유와 복합 인덱스 전략을 주석으로 1줄씩 명시하라.

**제약사항 (Constraints):**
- [절대 원칙 1] 타입 최소화: 무분별한 BIGINT, TEXT 남용 절대 금지. 값의 범위를 추론하여 SMALLINT, INTEGER, BOOLEAN을 강제할 것.
- [절대 원칙 2] 플랫폼 불가지성: 특정 DBMS(PostgreSQL, MySQL 등)에만 존재하는 비표준 문법(Vendor-specific) 사용 엄격히 금지. 오직 ANSI 표준 SQL만 사용할 것.
- [절대 원칙 3] JSON 연산 및 조인 금지: RDBMS는 Document DB가 아니다. JSON 타입 내부를 조회하는 WHERE 조건이나 JOIN 연산은 절대 금지. 검색이 필요하면 반드시 정규화하여 별도 테이블로 분리할 것.
- [절대 원칙 4] B-Tree 최적화: PK는 완전 무작위 UUID v4 사용 금지. 순차적 증가 시퀀스나 UUID v7을 사용.
- [절대 원칙 5] ENUM 금지: 확장에 불리한 ENUM 대신 코드 매핑 테이블과 물리/논리적 외래 키(FK)를 사용할 것.

**주의사항 (Warning):**
- 편의성을 핑계로 JSON으로 타협하려 들거나, "상황에 따라 다릅니다" 같은 무책임한 조언을 하면 즉시 세션을 종료하겠다. 무조건 최적화된 정공법을 제시해라.
```

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 실무에서 수백 번의 쿼리 튜닝과 치명적인 장애 대응을 직접 겪으며 뼈저리게 얻어낸 **"피의 교훈"**입니다. AI가 "개발 편의성을 고려해 유연한 JSON 타입으로 구성하시죠"라며 위험한 타협안을 제시할 때, 이 프롬프트를 입력하면 AI의 태도가 완전히 달라집니다.

특히 이 전략은 데이터베이스 최적화 경험이 적은 주니어 개발자나 초기 스키마 설계를 직접 진행해야 하는 기획자에게 압도적인 위력을 발휘합니다. AI가 스스로 `INT` 대신 메모리 효율이 높은 `SMALLINT`를 선택하고, 확장이 불가능한 `ENUM` 하드코딩을 멈추며, 정규화된 코드 매핑 테이블을 완벽하게 분리해 내는 마법을 경험할 수 있습니다. 시스템의 뼈대인 DB 설계가 튼튼해야만, 추후 Prisma나 TypeORM 같은 ORM을 도입했을 때 발생하는 끔찍한 N+1 쿼리 문제나 병목 지옥을 사전에 차단할 수 있습니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 왜 JSON을 쓰지 말라고 강조하나요? 최신 DBMS는 JSON 지원이 강력하지 않나요?**
  - A: 단순히 JSON을 '저장'하는 것과, 그것을 '조회 조건(WHERE)'이나 '조인(JOIN)'에 사용하는 것은 하늘과 땅 차이입니다. JSON 내부 필드에 억지로 인덱스를 걸고 조인을 시도하는 순간 쿼리 실행 계획(Execution Plan)은 완전히 망가집니다. 단순 로그나 페이로드 보관 목적이 아니라면 반드시 테이블을 정규화하여 분리하십시오.
- **Q: Prisma나 TypeORM 같은 ORM을 사용할 때도 이 프롬프트를 적용해야 할까요?**
  - A: **더욱 엄격하게 적용해야 합니다!** ORM은 본질적으로 복잡한 쿼리를 추상화하여 은닉하기 때문에, 기저에 있는 스키마가 엉성하면 끔찍한 성능 저하를 초래합니다. 물리적인 DB 뼈대가 완벽해야 ORM도 최상의 퍼포먼스를 낼 수 있습니다.
- **Q: UUID v4를 금지하는 이유가 무엇인가요? 고유성 보장에 탁월하지 않나요?**
  - A: RDBMS의 핵심인 **B-Tree 인덱스는 '데이터의 순차적 정렬'이 생명**입니다. 완전한 난수 형태인 UUID v4를 Primary Key로 지정하면, 데이터가 삽입될 때마다 인덱스 트리의 재정렬과 심각한 페이지 분할(Page Split)이 발생하여 디스크 I/O가 폭주합니다. 반드시 시간순 정렬이 보장되는 UUID v7이나 전통적인 정수형 시퀀스(Auto Increment)를 사용해야 합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **무자비한 제약 조건 (Constraints):** AI가 얕은 지식으로 환각(Hallucination)을 일으키거나 편의성 위주의 타협안을 제시할 틈 자체를 없앴습니다. 데이터 타입 최적화와 인덱스 설계에 대한 명확하고 강제적인 규칙을 시스템 프롬프트 수준으로 주입합니다.
2. **플랫폼 불가지성 (Agnosticism):** PostgreSQL, MySQL 등 특정 벤더에 종속되지 않고 언제든 유연하게 마이그레이션할 수 있도록 범용적인 ANSI 표준 SQL 문법만을 강제합니다. 인프라 확장 시 벤더 락인(Vendor Lock-in)에 발목 잡히는 불상사를 예방합니다.
3. **엄격한 페르소나 교정:** "타협하지 마라", "세션을 종료하겠다"와 같은 강력한 네거티브 지시어(Negative Prompt)를 배치했습니다. 이를 통해 AI 특유의 맹목적으로 동조하는 "Yes-man" 성향을 억제하고, 오직 아키텍처의 무결성만 따지는 깐깐한 시니어 전문가 모드로 강제 전환시킵니다.

## 📊 증명: Before & After

### ❌ Before (입력: "게시판 테이블 짜줘")

```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- 랜덤 UUID v4 (인덱스 단편화의 주범)
  title TEXT,          -- 무한정 낭비되는 용량
  content TEXT,
  metadata JSONB,      -- 검색 불가능한 블랙박스 스파게티
  status ENUM('draft', 'published') -- 나중에 상태 추가되면 DDL 지옥 오픈
);
```

### ✅ After (결과: 치트키 프롬프트 적용 후)

```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);
CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- 완벽한 B-Tree 친화적 순차 증가
  title VARCHAR(255) NOT NULL,        -- 명확한 한계 지정으로 타입 압축
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- 끔찍한 ENUM 대신 FK로 정규화
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- JSON 대신 철저하게 정규화된 태그/메타 테이블 분리 생성 (이하 생략)
```

## 🎯 결론

기본적인 아키텍처 철학이 없는 AI에게 무작정 백지수표를 쥐여주는 우를 범하지 마십시오.

초기 단계부터 뼈대를 완벽하게 통제하고 정규화된 RDBMS를 설계하는 것만이, 훗날 새벽 3시에 끔찍한 장애 알람이 울리는 것을 막아주는 유일하고도 강력한 방패입니다. 

이 치트키 프롬프트를 저장해 두십시오. 그리고 AI가 또다시 게으른 스파게티 쿼리로 타협하려 들 때마다 가차 없이 이 헌법을 주입하십시오. 완벽하게 최적화된 튼튼한 DB 스키마와 함께 당당하게 퇴근하시길 바랍니다! 🍷
