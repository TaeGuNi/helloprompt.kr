# 📝 AIで自分のコードの穴を見つけ出せ！(バグハンター)

- **🎯 対象者:** メンターなしで奮闘しているジュニア開発者、本番デプロイ前の自分のコードに不安を感じている実務者
- **⏱️ 所要時間:** デバッグ3時間 → 10分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コード分析の最強ツール), ChatGPT (GPT-4o)

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **有効性:** ⭐⭐⭐⭐⭐
- 🚀 **実用性:** ⭐⭐⭐⭐⭐

> _「自分のPCではちゃんと動くのに… 本番サーバーに上げたら落ちるんじゃないか？」_

一人でコーディングしていると、「自分が何を知らないのかすら分からない」という恐ろしい盲点に陥りがちです。なんとか動いてはいるものの、処理速度が遅かったり、セキュリティに脆弱性があったり、他人が読めないスパゲティコードになっているかもしれません。
これからは、24時間休むことのないGoogleのシニアエンジニア級のAIレビュアーにあなたのコードをチェックしてもらいましょう。潜在的なバグから非効率的なロジック、致命的なセキュリティの脆弱性まで、鋭く指摘してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. コードをむやみに投げるのではなく、AIに**「Googleのシニアエンジニア」**という明確なペルソナを与えましょう。
2. 単に「レビューして」ではなく、バグ、セキュリティ、可読性、パフォーマンスという**4つの評価基準**を明記しましょう。
3. AIの提案(リファクタリング)を盲信するのではなく、変更された理由をコメントで解説してもらい、自ら検証し学習しましょう。

## 🚀 解決策: "AIコードレビュアー"

> **役割:** あなたは10年目のシニアバックエンド/フロントエンド開発者です。
> **タスク:** 以下のコードから、致命的なバグや今すぐ改善すべきアンチパターン(Anti-pattern)を見つけてください。
>
> ```[言語名]
> [コードを貼り付け]
> ```

<br>

### 🥇 Pro Version (専門家型)

コードの品質をプロダクション(Production)レベルまで引き上げ、客観的なリファクタリングの根拠まで確保したい場合におすすめです。

> **役割 (Role):** あなたはGoogle出身の15年目のシニアソフトウェアエンジニアです。クリーンコード(Clean Code)とシステム最適化の専門家です。
>
> **状況 (Context):**
>
> - 言語: `[Python / JavaScript / Java など]`
> - フレームワーク: `[React / Spring / Django など]`
> - 目的: この関数は`[ユーザーログイン処理とセッション管理]`を担当しています。
>
> **コード (Input):**
>
> ```[言語名]
> [ここにレビューしてもらうコードを貼り付けます]
> ```
>
> **タスク (Task):**
> 次の4つの観点から私のコードを厳格にレビューし、改善された全体のコードを提供してください。
>
> 1. **バグ(Bug):** 例外処理の抜け漏れや、実行時に発生しうる潜在的なエラー (Edge cases)
> 2. **セキュリティ(Security):** SQLインジェクション、XSS、CSRFなどフレームワークの脆弱性の有無
> 3. **可読性(Readability):** 変数/関数名の命名規則、単一責任の原則(SRP)違反の有無
> 4. **パフォーマンス(Performance):** 不要なループ処理、時間計算量(Big-O)の観点からのボトルネック
>
> **制約事項 (Constraints):**
>
> - 改善されたコードを提示する際、**なぜ(Why)** そのようにリファクタリングしたのか、各変更点ごとにコメント(`//` または `#`)で詳細に説明してください。
> - 既存のビジネスロジックの意図は絶対に損なわないでください。
> - 外部ライブラリを追加せず、該当言語の組み込み機能のみで最適化してください。
>
> **注意事項 (Warning):**
>
> - 確信のない構文や非推奨(Deprecated)の関数を提案しないでください。分からない場合は「分からない」と答えてください。(ハルシネーション防止)

## 💡 筆者のインサイト (Insight)

このプロンプトは、私が新人時代に「自分のコードに自信が持てない時」に絶大な精神的安心感を与えてくれた奥の手です。
特に「セキュリティ(Security)」と「例外処理(Edge cases)」の項目は、人間の目では見逃しやすい部分をAIが機械的にキャッチしてくれるため、大事故を何度も防いでくれました。
AIが修正してくれたコードをそのままコピー＆ペーストしてはいけません。制約事項に「なぜ(Why)リファクタリングしたのかコメントで説明して」という条件を入れた理由は、AIのレビューを通じて皆さん自身がシニアの視点を学習するためです。

## 🙋 よくある質問 (FAQ)

- **Q: 会社のプロジェクトの全体コードを丸ごとコピーして渡してもいいですか？**
  - A: **絶対にダメです！** 会社のコードの外部流出は重大なセキュリティ違反です。API Key、DBのパスワード、会社の中核となるビジネスロジックは必ず`***`などでマスキング(ブラインド処理)した後、レビューが必要な最小限の関数やクラス単位に切り取って質問してください。

- **Q: AIは完璧な正解をくれますか？テストしなくても大丈夫でしょうか？**
  - A: AIもハルシネーション(もっともらしい嘘)を起こします。時には存在しないライブラリを使えと言ったり、構文を間違えたりすることもあります。AIが提案したコードは、必ずローカル環境でテストコードを回して動作を検証する必要があります。

## 🧬 プロンプトの解剖 (Why it works?)

1. **厳格なペルソナ付与 (Perspective):** 「Googleのシニアエンジニア」という最高レベルの基準を強制することで、適当に動くコードではなく「クリーンコード」の観点からのフィードバックを誘導しました。
2. **多面的な評価基準 (Specific Goals):** 単にレビューしてと言うと、AIは表面的なタイプミスしか指摘しません。バグ、セキュリティ、可読性、パフォーマンスという4つの明確な評価基準を与えることで、立体的な診断を下すように設計しました。
3. **学習誘導設計 (Explanation):** 正解のコードをもらうだけでは実力は伸びません。変更理由をコメントで要求することで、ユーザー自身にアーキテクチャを考えさせるチュータリング(Tutoring)効果を狙いました。

## 📊 証明: Before & After

### ❌ Before (入力: 初歩的なDB照会ロジック)

```python
def get_user_data(user_id):
    # ユーザーが入力したidをそのままクエリに連結 (危険！)
    sql = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(sql)
    return cursor.fetchone()
```

### ✅ After (結果: セキュリティと可読性が強化されたロジック)

```python
def get_user_data(user_id):
    """
    特定のユーザー情報を照会します。
    """
    # [Security] SQLインジェクション防止のため、Parameterized Query(バインド変数)を使用
    # [Readability] クエリ文を分離して可読性を向上
    query = "SELECT * FROM users WHERE id = %s"

    try:
        cursor.execute(query, (user_id,))
        result = cursor.fetchone()

        # [Bug] 結果がない場合の例外処理を追加
        if not result:
            return None

        return result

    except Exception as e:
        # [Bug] ランタイムエラーのロギング処理 (実際の環境ではloggerの使用を推奨)
        print(f"Database error occurred: {e}")
        return None
```

**レビューの要約:** ユーザーの入力をクエリに直接結合すると、致命的なSQLインジェクション攻撃にさらされます。バインド変数(`%s`)を使用してセキュリティをパッチし、DBの照会に失敗した際にサービスが停止しないよう`try-except`による例外処理を追加しました。
