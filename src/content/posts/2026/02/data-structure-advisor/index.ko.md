## 📝 List? Map? Set? 결정 장애를 위한 자료구조 처방전

- **🎯 추천 대상:** 주니어 백엔드 개발자, 코딩 테스트 준비생, 대용량 데이터 성능 최적화가 시급한 실무자
- **⏱️ 소요 시간:** 10분 → 1분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (논리적 추론 및 알고리즘 분석에 탁월한 최신 모델 권장)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐

> _"맨날 쓰던 ArrayList만 고집하다가, '데이터가 100만 건을 넘어가니 API 응답이 너무 느려요'라는 사용자 불만을 받고 등골이 서늘해진 적 없으신가요?"_

자료구조의 선택은 프로그램의 전체 성능을 좌우하는 가장 치명적이고 중요한 첫 단추입니다. 하지만 숨 막히는 일정과 데드라인에 쫓기다 보면, 깊은 고민 없이 손에 익은 자료구조(주로 `List`나 `Array`)만 관성적으로 가져다 쓰게 되는 경우가 허다합니다.

이 프롬프트는 여러분이 직면한 구체적인 비즈니스 요구사항과 데이터 접근 패턴을 입력하기만 하면, 시간 복잡도(Big-O)와 공간 효율성을 엄밀하게 계산하여 **병목을 원천 차단하는 가장 완벽한 자료구조와 사용 언어별 최적의 내장 구현체**를 족집게처럼 찾아주는 든든한 시니어 엔지니어가 되어줄 것입니다.

## ⚡️ 3줄 요약 (TL;DR)

1. 데이터의 접근, 삽입, 삭제 패턴을 정밀하게 분석하여 시스템 병목을 제거하는 최적의 자료구조를 추천받습니다.
2. 각 연산에 대한 시간 복잡도(Big-O)를 수학적으로 비교하여, 선택의 아키텍처적 타당성을 완벽하게 검증합니다.
3. Java, Python, TypeScript 등 현재 사용 중인 언어 환경에서 즉시 적용 가능한 구체적인 표준 라이브러리 클래스명까지 도출합니다.

### 🥇 Pro Version (전문가형)

> - 목표: 처리 속도(Time Complexity)와 메모리 사용량(Space Complexity) 간의 트레이드오프(Trade-off)를 엄밀히 계산하여, 병목이 전혀 없는 완벽한 자료구조를 선택하는 것.

> 3. **구현체 추천:** `[사용 언어]` 환경에서 즉시 가져다 쓸 수 있는 표준 라이브러리의 정확한 클래스명 제공

> **요구사항:**
>

> - 주요 연산: `[예: 초기에 대량의 데이터 삽입이 한 번 일어나고, 이후에는 특정 ID의 존재 여부를 묻는 단순 검색이 전체 연산의 99%를 차지함]`

## 💡 작성자 코멘트 (Insight)

실제 현업에서 수많은 코드 리뷰를 진행하다 보면, 당연히 `Set`이나 `Map`을 써야 할 로직에 무심코 `List`를 사용하여 `O(N^2)`이라는 재앙적인 성능 저하를 일으키는 경우를 셀 수 없이 목격합니다. (가장 흔한 예: 두 개의 거대한 리스트를 중첩 for문으로 돌리며 교집합을 찾는 끔찍한 안티 패턴)

이 프롬프트의 진정한 가치는 단순히 학술적인 '정답'을 찾는 수준을 넘어, **실무에서 특정 언어의 어떤 내장 클래스를 정확히 가져다 써야 하는지** 디테일하게 짚어준다는 점에 있습니다. 가령 Java 환경에서 무작정 'Map을 쓰라'고 뭉뚱그려 답하는 것이 아니라, 입력 순서 보장이 필수라면 `LinkedHashMap`을, 키 값에 기반한 정렬이 요구된다면 `TreeMap`을 선제적으로 제안하도록 유도하여 시스템 설계의 치명적인 빈틈을 완벽하게 틀어막아 줍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코딩 테스트 문제의 지문을 그대로 복사해서 넣어도 되나요?**
  - A: 물론입니다! 이 프롬프트는 매우 강력한 1:1 코딩 테스트 멘토로 돌변합니다. **요구사항** 항목에 문제의 제한 시간(예: 1초)과 최대 데이터 크기 N(예: 100,000)을 입력해 보세요. AI가 "N이 10만이므로 `O(N^2)` 알고리즘을 사용하면 무조건 시간 초과가 발생합니다. 따라서 시간 복잡도가 `O(N log N)` 이하인 Priority Queue를 도입해야 합니다"와 같이 뼈 때리는 수학적 추론을 제시해 줍니다.

- **Q: 메모리 절약과 처리 속도 향상 중 실무에서는 보통 어느 쪽을 우선시해야 할까요?**
  - A: 극도로 제한된 임베디드(Embedded) 환경이 아닌 현대의 웹/앱 백엔드 생태계에서는, **'메모리를 조금 더 희생하더라도 데이터 처리 속도를 비약적으로 끌어올리는(Space-Time Tradeoff)'** 해시 테이블(Hash Table) 기반의 자료구조가 압도적으로 유리한 경우가 대부분입니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **상황과 목표의 명확화 (Context):** 단순히 AI에게 '자료구조를 골라줘'라고 묻는 것과, '처리 속도와 메모리의 트레이드오프를 철저히 계산해 병목을 제거하라'고 지시하는 것은 출력되는 답변의 차원을 완전히 뒤바꿔 놓습니다.
2. **평가 기준의 강제성 부여 (Task & Constraints):** Big-O 표기법이라는 명확하고 엄밀한 컴퓨터 과학적 척도를 사용하도록 강제했습니다. 이로 인해 AI는 주관적이고 두루뭉술한 조언이 아닌 수학적, 논리적 근거가 탄탄한 해답을 내놓게 됩니다. 여기에 CPU 캐시 지역성 등 실무적 관점을 요구한 제약사항은 답변의 퀄리티를 시니어 레벨로 끌어올리는 핵심 트리거입니다.

## 📊 증명: Before & After

### ❌ Before (단순한 질문)

```text
회원 ID 100만 개를 저장할 건데, 배열 쓸까? 리스트 쓸까? 검색이 좀 빠르면 좋겠어. 자바로 짤 거야.
```

### ✅ After (프롬프트 적용 결과)

```text
(AI의 분석 결과 요약)

배열이나 일반 리스트(ArrayList)의 사용은 추천하지 않습니다! 100만 개의 데이터에서 특정 ID를 검색할 때 최악의 경우 100만 번의 탐색(O(N))이 발생하여 성능 병목이 심각해집니다.

추천 자료구조: Hash Set (해시 집합)
- 특성: 중복을 허용하지 않고, 순서를 보장하지 않는 데이터의 집합.
- 검색/삽입/삭제 시간 복잡도: 평균 O(1)
- 추천 이유: 해시 함수를 통해 데이터가 저장된 메모리 위치를 즉시 계산하므로, 데이터가 100만 개든 1,000만 개든 단 한 번의 연산만으로 존재 여부를 매우 빠르게 확인할 수 있습니다.

추천 클래스 (Java):
- 기본 추천: `java.util.HashSet` (순서 상관없는 압도적 속도)
- 동시성 보장이 필요한 경우: `java.util.concurrent.ConcurrentHashMap.KeySetView` 또는 `Collections.newSetFromMap(new ConcurrentHashMap<>())`
```
