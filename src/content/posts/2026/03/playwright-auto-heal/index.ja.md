# 📝 エラーで止まるE2Eテスト？Playwright Auto-healingプロンプトで1分で解決する 🛠️


- **🎯 おすすめの対象:** QAエンジニア、フロントエンド開発者、テスト自動化担当者
- **⏱️ 所要時間:** 2時間 → 1分に短縮
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_"昨日まで順調に動いていたE2Eテストが、今朝ボタンのCSSクラスが一つ変わっただけで全て失敗しました。またセレクタを探しに行かないといけませんか？"_
フロントエンド開発やQA業務をしていると、UIの変更によってE2E（End-to-End）テストが壊れるのは日常茶飯事です。特にPlaywrightやCypressのようなツールを使用する場合、DOM構造やクラス名が変わると、テストコードは容赦なくエラーを吐き出します。毎回壊れたテストの箇所へ行き、開発者ツールを開いて新しいセレクタをコピーしてくる作業は、とてつもない時間の無駄です。これからはAIにエラーログと変更されたHTML構造だけを渡し、勝手に「Auto-healing（自動復旧）」させるようにしてみましょう。
---
## ⚡️ 3行まとめ (TL;DR)
1. UI変更により壊れたPlaywrightのセレクタを、AIが自ら分析して復旧するプロンプトです。
2. 単にコードを直すだけでなく、変更に強い（Resilient）セレクタの作成戦略を提案してもらえます。
3. エラーログと最新のDOMスナップショットを提供するだけで、復旧されたコードを1分以内に取得できます。
---
## 🚀 解決策：「Playwright 自己治癒（Auto-healing）プロンプト」

### 🥉 Basic Version（基本型）
素早く壊れたコードとエラーメッセージだけを渡し、修正されたコードを受け取りたい時に使用してください。
> **役割:** あなたは `[シニアQAエンジニア]` です。
> **タスク:** 提供されたPlaywrightのエラーログと新しいHTMLを基に、`[壊れたテストコード]`を修正してください。

### 🥇 Pro Version（専門家型）
単なる修正を超えて、メンテナンスが容易で堅牢なコードにリファクタリングしたい時に使用してください。
> **役割 (Role):** あなたはテスト自動化専門の `[シニアSDET(Software Development Engineer in Test)]` です。Playwrightのベストプラクティス（Best Practices）を完全に理解しています。
> 
> **状況 (Context):**
> 
> - 背景: 最近のUIアップデートにより、既存のPlaywright E2Eテストが失敗しています。セレクタ（Selector）の変更が主な原因です。
> - 目標: 失敗したテストコードを復旧し、今後のUI変更で壊れにくい（Resilient）堅牢なセレクタ構造に改善すること。
> 
> **タスク (Task):**
> 
> 1. 以下に提供された `[エラーログ]` と `[変更されたHTML構造]` を分析し、エラーの根本原因を特定してください。
> 2. `[既存のテストコード]` を修正し、テストが再びパスするようにしてください。
> 3. 修正の際、CSSクラスやXPathに依存するのではなく、ユーザー向け属性（テキスト、ARIA属性、data-testidなど）を優先的に使用してください。
> 4. 修正した部分について、「なぜこのように修正したのか」の簡単な理由をコードのコメントとして追加してください。
> 
> **入力データ:**
> - エラーログ: `[エラーログ全体をコピー/ペースト]`
> - 変更されたHTML構造: `[開発者ツールからコピーした関連HTMLの断片]`
> - 既存のテストコード: `[失敗したPlaywrightのコード断片]`
> 
> **制約事項 (Constraints):**
> 
> - 出力はすぐにコピー/ペーストできる完全なコードブロック形式で提供すること。
> - Playwrightの組み込みLocators（例: `getByRole`、`getByText`、`getByTestId`）を最優先で使用すること。
> 
> **注意事項 (Warning):**
> 
> - もし提供されたHTMLの断片から適切な要素が見つからない場合は、無理に推測せず「HTML情報が不足しているため、正確なセレクタを見つけることができません。追加のDOM情報を提供してください」と回答してください。

## 💡 作成者コメント (Insight)
この方法は、現場で数十個のテストが同時に失敗した時に真価を発揮します。私は過去にセレクタが一つ変わると、30個のテストファイルを開いて一つ一つ `Ctrl + F` で探して直していました。しかし、このプロンプトをIDE内蔵のAIやClaudeに登録しておき、エラーが出たログとブラウザのElementsタブのコピースニペットだけを渡せば、面倒なセレクタ抽出作業を完全にアウトソーシングできます。
特に重要なのは、「ユーザー向け属性を使用しろ」という制約事項です。AIは基本的に最も簡単なCSSクラスをターゲットにしようとする傾向がありますが、この制約を設けておくと `getByRole('button', { name: 'Submit' })` のような素晴らしいコードを書いてくれます。皆さんの退社時間を最低でも1時間は早めてくれる武器です。まずは使ってみてください！

## 🙋 よくある質問 (FAQ)
- **Q: HTML構造が長すぎる場合はどうすればいいですか？**
  - A: ページ全体をコピーする必要はなく、エラーが出た要素（例：ボタンやフォーム）の親タグ程度までコピーして提供すれば十分です。
- **Q: Playwrightではなく、CypressやSeleniumでも使えますか？**
  - A: はい、プロンプト内の「Playwright」という単語を使用しているツールの名前に変更するだけで、同じように見事に動作します。
- **Q: `data-testid`がないプロジェクトでもうまく動作しますか？**
  - A: はい、AIが `getByRole` や `getByText` のようなアクセシビリティベースの代替案を賢く見つけ出します。

## 🧬 プロンプトの解剖 (Why it works?)
1. **SDETペルソナの付与:** 一般の開発者ではなくテスト専門エンジニアの視点を持たせることで、単に動くコードではなくPlaywrightのベストプラクティスに沿ったコードを生成するように誘導しました。
2. **User-Facing Locatorsの強制:** テストの不安定性（Flakiness）を根本的に減らすというPlaywrightの核心哲学をプロンプトレベルで強制し、クオリティを高めました。
3. **ハルシネーション（幻覚）防止の安全装置:** 情報が不足している時に無意味なコードを生成し、デバッグをさらに複雑にする現象を防ぐため、「分からない時は分からないと言え」という明示的な制約を設けました。

## 📊 証明: Before & After
### ❌ Before (入力)
```text
エラーログ:
Error: locator.click: Target closed
=========================== logs ===========================
waiting for locator('.btn-primary.submit-btn')
============================================================
変更されたHTML構造:
<div class="actions">
  <button class="button--blue" aria-label="Submit Order">Order Now</button>
</div>
既存のテストコード:
await page.locator('.btn-primary.submit-btn').click();
```
### ✅ After (結果)
```typescript
// 変更理由: CSSクラス（'.btn-primary.submit-btn'）が変更され、テストが失敗しました。
// 改善事項: 変更されやすいCSSクラスの代わりに、アクセシビリティ属性（ARIA label）とRoleを使用し、UIの変更に強くなるようにしました。
await page.getByRole('button', { name: 'Submit Order' }).click();
```
