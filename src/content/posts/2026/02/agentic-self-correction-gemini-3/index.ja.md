# 📝 AIエージェントが失敗する理由：Gemini 3 Proの「自己修正ループ」プロンプトパターン

- **🎯 おすすめの対象者:** AIエージェントを開発するバックエンドエンジニア、プロンプトエンジニア、プロダクトマネージャー
- **⏱️ 所要時間:** コードのデバッグ 15分 → 1分に短縮
- **🤖 おすすめのモデル:** Gemini 3 Pro, GPT-4o, Claude 3.5 Sonnet (推論能力に優れたモデル)

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐☆

> _「完璧に見えたAIエージェントが、同じエラーを繰り返してAPIトークンを浪費していませんか？今こそ、『コードを書け』ではなく『書いて、批判し、修正しろ』と指示する時です。」_

Gemini 3 ProやGPT-4oのような高性能モデルを使ってAIエージェントを構築した経験がある方なら、心当たりがあるはずです。コードを生成する能力は卓越していますが、自らバグを修正するのは絶望的に下手なケースが多々あります。一度実行して失敗すると、全く同じコードを繰り返し吐き出し、いわゆる「死の螺旋（Death Spiral）」に陥って、貴重なAPI予算を無駄にしてしまうのです。

この問題は、モデルの知能が不足しているからではありません。**「自己修正ループ（Self-Correction Loop）」**が欠落しているからです。多くの開発者はエージェントを「撃ちっ放し（Fire and Forget）」方式で扱います。しかし、真のエージェンティック・ワークフロー（Agentic Workflow）には、モデルが結果をユーザーに提示する前に自らを批判する「エディター（Editor）」のフェーズが不可欠です。

本日は、実際のプロダクション環境において、エージェントのエラー率を60%以上も劇的に低下させた**自己修正プロンプトパターン（Self-Correction Prompt Pattern）**を共有します。

## ⚡️ 3行まとめ (TL;DR)

1. AIエージェントの無限エラーループは、モデルの限界ではなく「検証フェーズ」の不在が原因です。
2. AIに「作成者（Creator）」と「批評家（Critic）」の役割を同時に与え、論理的なエラーを自ら発見させましょう。
3. ドラフト作成、自己批判、最終修正という3ステップのプロセスを1つのプロンプトに組み込むことで、ハルシネーション（幻覚）を劇的に減らすことができます。

## 🚀 解決策：「批評家・実行者（Critic-Actor）パターン」

ドラフト作成（Drafting）のステップと、批判（Critique）のステップを明確に分離することが、このプロンプトの核心です。

### 🥉 Basic Version (基本型)

自己修正の結果だけを素早く確認したい場合に使用してください。

> **役割:** あなたは`[シニアバックエンドエンジニア]`です。
> **タスク:** `[ソート済みの2つのリストをマージする関数]`を作成してください。コードを作成した後、自ら潜在的なバグや非効率な部分を探して批判し、その批判に基づいて修正した最終的なコードを提示してください。


### 🥇 Pro Version (専門家型)

細部までこだわったコード品質と、エッジケース（Edge Case）への防御が必要な場合に使用してください。

> **役割 (Role):** あなたは実務経験10年のシニアPythonバックエンドエンジニアであり、緻密なコードレビュアー（Code Reviewer）です。クリーンで効率的、かつバグのないコードを最優先事項としています。
>
> **状況 (Context):**
>
> - 背景: 現在、プロダクション環境において大規模なデータを処理するための、信頼性の高いコードを必要としています。
> - 目標: 自らコードの論理的な穴を見つけ出し、補完する完璧な関数を作成すること。
>
> **タスク (Task):**
>
> 1. **ドラフト (Draft):** `[ソート済みの2つのリストをマージするPython関数]`を作成してください。
> 2. **批判 (Critique):** 作成したドラフトコードを自らレビューしてください。以下の項目を重点的に見つけ出してください:
>    - エッジケース (例: 空のリストの入力、負の数の包含など)
>    - パフォーマンスのボトルネック (時間/空間計算量)
>    - 潜在的なランタイムエラー
> 3. **修正 (Refine):** 批判フェーズで発見した問題点に基づいて、コードを完璧に書き直してください。
>
> **制約事項 (Constraints):**
>
> - 出力形式は、以下に提供されたMarkdownの構造に厳密に従ってください。
> - 「3. 最終完成コード」のセクションでは、コードに対する追加の説明は絶対に行わないでください。コードブロックのみを提供してください。
> - 最終完成コードには、批判フェーズで特定されたエッジケースに対する例外処理のロジックが必ず含まれていなければなりません。
>
> **出力形式 (Format):**
>
> ## 1. 初期ドラフト (Initial Draft)
>
> \`\`\`python
> (ドラフトコード)
> \`\`\`
>
> ## 2. 自己批判 (Self-Critique)
>
> - **批判 1:** (発見した問題点と原因)
> - **批判 2:** (発見した問題点と原因)
>
> ## 3. 最終完成コード (Final Polished Code)
>
> \`\`\`python
> (最終修正されたコード)
> \`\`\`

## 💡 筆者コメント (Insight)

このプロンプトパターンは、単純なコード生成にとどまらず、SQLクエリの作成やビジネスメールのコピーライティングなど、論理的な検証が必要なあらゆる作業において強力な効果を発揮します。「自己批判（Self-Critique）」のステップは、AIの無意味なハルシネーション（幻覚）を防ぐ強固な防波堤の役割を果たします。

特に**Gemini 3 Pro**のような巨大なコンテキストウィンドウを持つモデルを使用する場合、批判フェーズにおいて既存のコードベースの特定の行を直接引用してレビューするように指示することで、コードの正確性とコンテキストの一致率を極限まで引き上げることができます。

ただし、注意点があります。「誤ったドラフト」と「批判」のプロセスを経てテキストを追加生成するため、従来のプロンプトと比較して約2倍のトークン（コスト）を消費します。したがって、単純なテキスト変換作業よりも、**「コストよりも正確性（Accuracy > Cost）」がはるかに重要な、複雑なロジック処理**にこのパターンを導入することを強く推奨します。実際に試してみると、このプロセスによって節約されるデバッグ時間は、トークンのコストを遥かに凌駕します。

## 🙋 よくある質問 (FAQ)

- **Q: トークンの消費量が心配です。コストを削減する方法はありますか？**
  - A: 内部ロジックを処理する際は、比較的安価なモデル（例：Gemini 3 Flashなど）を使用してドラフトを作成し、「批判と修正」フェーズの検閲役だけをGemini 3 Proに任せるマルチエージェント・ルーティング（Multi-Agent Routing）方式を構築すれば、品質を維持しながらコストを大幅に削減できます。

- **Q: 批判フェーズでもAIが問題点を見つけられない場合はどうすればよいですか？**
  - A: プロンプトのTask部分に具体的なチェックリストを注入してください。例えば、「メモリリークの可能性」、「非同期処理のデッドロック」、「SQLインジェクションの脆弱性」など、AIが検討すべきポイントを明示的に指定することで、批判の解像度が格段に鋭くなります。

- **Q: コーディング以外の企画やブログの執筆作業にも応用できますか？**
  - A: もちろんです。「ドラフト作成 -> 論理的矛盾および読者ペルソナの不一致の批判 -> 最終原稿の修正」という構造で応用すれば、人間の介入なしでも、専門家が推敲したような素晴らしいレベルの成果物を得ることができます。

## 🧬 プロンプトの解剖 (Why it works?)

1.  **Role & Contextの分離:** AIに「開発者」と「レビュアー」という相反する2つのペルソナを強制的に付与することで、自らコードの欠陥を見つけ出すメタ認知（Meta-cognition）能力を活性化させました。
2.  **段階的な推論 (Chain-of-Thought) の誘導:** すぐに完璧な正解を出力させるのではなく、「ドラフト -> 批判 -> 修正」という思考のプロセスをMarkdown構造で明示化し、論理の飛躍や無意味な出力を防ぎました。
3.  **Constraints (制約) の統制:** 最終的な出力では不要な説明をすべて排除するよう強制し、その後の自動化パイプライン（CI/CDなど）でコードをパース（Parsing）してすぐにテストしやすいように設計しました。

## 📊 証明: Before & After

**テスト条件:** 「ソート済みの2つのリストをマージする関数の作成」

### ❌ Before (入力)

標準的なプロンプトで指示した場合、単に `list1 + list2` を実行した後に `sorted()` を呼び出すという、非効率的なコードを生成しました。

```python
def merge_lists(list1, list2):
    # O(N log N)の時間計算量が発生。大規模データにおいて深刻なパフォーマンス低下の懸念。
    return sorted(list1 + list2)
```

### ✅ After (結果)

自己修正プロンプトを適用したところ、自己批判フェーズで「すでにソートされているリストに対して `sorted()` を使用するのは非効率であり、O(N) の計算量にするためにツーポインター（Two-pointer）アプローチを使うべきだ」と自ら指摘した上で、完璧に最適化されたコードを導き出しました。15分かかっていたコードレビューとリファクタリングの時間が、わずか1秒に短縮されました。

```python
def merge_lists(list1, list2):
    # O(N)の時間計算量で最適化し、空リストのエッジケースも完璧に防御
    merged = []
    i, j = 0, 0

    while i < len(list1) and j < len(list2):
        if list1[i] < list2[j]:
            merged.append(list1[i])
            i += 1
        else:
            merged.append(list2[j])
            j += 1

    # 残りの要素をマージ
    merged.extend(list1[i:])
    merged.extend(list2[j:])

    return merged
```
