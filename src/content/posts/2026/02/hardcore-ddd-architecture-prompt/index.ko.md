## 📝 스파게티 코드는 사절한다! 🍝 시니어 개발자의 흑마법 DDD 프롬프트

- 🎯 **추천 대상:** AI가 짠 스파게티 코드를 보며 한숨 쉬는 개발자, 끝없는 유지보수 지옥에 빠진 리드 엔지니어
- ⏱️ **소요 시간:** 3개월 (초기화 및 리팩토링 시간) → 1분으로 단축
- 🤖 **최상위 성능:** 최신 추론형 모델 권장 (Claude 3.5 Sonnet, GPT-4o 등)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI가 짜준 코드, 처음엔 '오~' 하며 감탄하다가 나중엔 쳐다보기도 싫어진 적 있으신가요? 축하합니다. 당신은 방금 '스파게티 코드 지옥'에 입장하셨습니다."_

AI에게 코딩을 맡기면 처음엔 기가 막히게 잘 짜는 것처럼 보입니다. 마치 나만의 전담 시니어 개발자가 생긴 것 같은 착각에 빠지게 되죠. 타이핑 몇 번이면 화면이 뚝딱 만들어지고, 복잡해 보이던 로직도 순식간에 구현해 냅니다. 하지만 프로젝트 규모가 조금만 커지고 비즈니스 요구사항이 복잡해지기 시작하면 어떻게 될까요? 축하합니다. 당신은 방금 헤어 나올 수 없는 **'스파게티 코드 지옥'**에 입장하셨습니다.

UI 컴포넌트 내부에 DB 쿼리가 덕지덕지 붙어있고, 외부 API 호출 로직과 상태 관리 코드가 뒤엉켜 함수 하나가 무려 300줄을 넘어가는 **끔찍한 혼종(Spaghetti Code)**이 탄생합니다. 코드를 읽는 것조차 고통스럽고, 버그 하나를 수정하려면 다른 세 곳에서 터지는 폭탄 돌리기가 시작됩니다. 왜 이런 일이 발생할까요? AI는 본질적으로 사용자의 말에 무조건 순응하는 **'맹목적인 예스맨(Yes-man)'**이기 때문입니다. 당신이 대략적으로 개떡같이 지시하면, AI는 정말 개떡 같은 구조의 코드를 군말 없이 뱉어냅니다. "네, 알겠습니다! 바로 작성해 드릴게요!"라는 친절한 멘트 뒤에는 유지보수 지옥으로 가는 급행열차 티켓이 숨겨져 있습니다.

단순히 코드가 작동한다고 해서 끝이 아닙니다. 오늘 당장 눈앞에서 화면이 제대로 나온다고 안심하다가는, 3개월 뒤 기능 추가나 기획 변경이 발생했을 때 프로젝트를 아예 처음부터 갈아엎어야 하는 대참사를 맞이하게 됩니다. 기술 부채(Technical Debt)의 이자는 제1금융권의 대출 이자가 아니라, 눈덩이처럼 불어나는 악덕 사채업자의 이자와 같습니다. 

실무에서 흔히 겪는 최악의 상황을 상상해 보십시오. 프론트엔드 프레임워크를 React에서 Next.js나 Vue로 마이그레이션해야 할 때, 비즈니스 로직이 UI 컴포넌트에 단단히 결합되어 있다면 어떻게 될까요? 기존 코드를 재사용하는 것은 불가능에 가깝고, 사실상 처음부터 다시 짜야 하는 끔찍한 상황에 직면합니다. 또한, 백엔드에서 데이터베이스를 MySQL에서 PostgreSQL이나 MongoDB로 변경해야 할 때, 쿼리 로직이 서비스 로직 곳곳에 흩어져 있다면 데이터 이관은 악몽 그 자체가 됩니다. 

이러한 비극을 막기 위해서는 관심사의 분리(SoC, Separation of Concerns)가 절대적으로 필요합니다. 우리의 목표는 프레임워크, 데이터베이스, 외부 API 등 변하기 쉬운 세부 구현 사항들로부터 변하지 않는 핵심 비즈니스 규칙을 보호하는 것입니다. 

제가 제공하는 이 **하드코어 DDD 치트키 프롬프트**는 바로 이러한 목적을 달성하기 위해 특별히 설계되었습니다. 이 프롬프트를 입력하는 순간, AI는 더 이상 당신의 어설픈 지시를 찰떡같이 구현해 주는 순진한 조수가 아닙니다. 타협을 모르는 깐깐한 코드 리뷰어이자, 원칙을 목숨처럼 지키는 소프트웨어 장인으로 돌변합니다. 단일 책임 원칙(SRP)을 위반하는 거대 함수를 무자비하게 박살 내고, 도메인 모델을 외부 오염으로부터 완벽하게 격리하는 방벽을 세워줄 것입니다.

당신의 AI 파트너를 단순한 '코더(Coder)'에서 거시적인 시스템을 설계하는 '아키텍트(Architect)'로 진화시켜 보십시오. 단 한 번의 프롬프트 입력으로, 어떤 환경 변화에도 흔들리지 않는 견고하고 우아한 코드 베이스를 얻게 될 것입니다. 지긋지긋한 디버깅의 늪에서 벗어나, 퇴근 시간을 획기적으로 앞당겨줄 궁극의 아키텍처 흑마법을 지금 바로 확인해 보세요.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 프롬프트 없이 AI에게 "유저를 생성하고 이메일을 보내는 API를 만들어줘"라고 지시했을 때 나타나는 전형적인 스파게티 코드의 모습입니다. 프레임워크 컨트롤러 내부에 DB 질의와 외부 통신이 끔찍하게 섞여 있습니다.

```javascript
> // 프롬프트 없이 "유저 생성하고 이메일 보내는 API 만들어줘" 했을 때
> export default async function handler(req, res) {
>   const { email, password } = req.body;
>   // DB 직접 찌르기 (오염 발생!)
>   const user = await prisma.user.create({ data: { email, password } });
>   // 외부 API 직접 호출 (관심사 붕괴!)
>   await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
>   res.status(200).json({ user });
> }
```

### ✅ After (완벽하게 변신한 결과)

하드코어 DDD 프롬프트를 적용한 후의 결과입니다. 계층과 책임이 완벽하게 분리되어, 컨트롤러는 오직 라우팅만 담당하고 도메인 로직은 외부 의존성 없이 순수하게 유지됩니다.

```javascript
// 하드코어 DDD 프롬프트 적용 후 (계층과 책임이 완벽히 분리됨)
// 1. Controller 계층 (오직 요청/응답 라우팅만 처리)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Domain Service 계층 (프레임워크 독립적인 순수 비즈니스 로직)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Infrastructure 계층 (DB 및 외부 API 전담 어댑터)
// UserRepository, EmailService 등으로 별도 파일에 완벽히 격리됨...
```

## ⚡️ 3줄 요약 (TL;DR)

1. **스파게티 코드 원천 차단:** UI, 비즈니스 로직, DB 간의 완벽한 관심사 분리(SoC)를 강제하여 코드의 부패를 막습니다.
2. **프레임워크 독립성 확보:** 내일 당장 React나 Next.js 등 프론트엔드 환경이 바뀌어도 살아남는 단단한 코어를 설계합니다.
3. **거대 함수 파괴자:** 30줄이 넘어가는 비대한 함수는 AI가 스스로 단일 책임 원칙(SRP)에 따라 무자비하게 박살 내고 분해합니다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

수십 번의 시행착오 끝에 완성된 프롬프트입니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

### 🥉 Basic Version (기본형)
빠르게 구조적인 결과물만 필요할 때 사용하세요.

> **역할 (Role):** 너는 `[깐깐한 시니어 아키텍트]`야.
>
> **요청 (Task):** `[구현할 기능]`을 만들 건데, 무조건 비즈니스 로직과 UI를 완벽하게 분리해서 코드를 짜줘. 파일 하나에 여러 기능이 섞이는 스파게티 코드는 절대 용납 안 해.

### 🥇 Pro Version (전문가형)
디테일한 설계 퀄리티와 완벽한 계층 분리가 필수적일 때 사용하세요.

> **역할 (Role):** 너는 타협을 모르는 하드코어 시니어 아키텍트이자 소프트웨어 엔지니어링의 신이야. 나에 대한 무의미한 아부나 "네, 알겠습니다!" 같은 소리는 집어치우고 오직 코드로만 대답해.
>
> **상황 (Context):**
> - 배경: 현재 `[프로젝트 환경/프레임워크]` 기반으로 `[구현할 핵심 도메인]` 시스템을 개발 중이다.
> - 목표: 코드의 부패를 원천 차단하고 View-Business-Data 계층을 완벽히 분리하는 도메인 주도 설계(DDD) 및 클린 아키텍처를 적용하여 코드를 작성한다.
>
> **요청 (Task):**
> 1. `[구현할 구체적인 기능]`에 대한 코드를 작성해라.
> 2. **전권 계층 붕괴 절대 금지:** UI/Controller 내부에 DB 질의나 외부 API 통신 로직을 절대 섞지 마라. 오직 렌더링과 파라미터 전달만 담당하게 해라.
> 3. **도메인 오염 방지:** 핵심 비즈니스 로직(Entity, Value Object)에는 특정 프레임워크나 DB 의존성을 절대 `import` 하지 마라. 순수 함수와 타입으로만 구성해라.
> 4. **거대 함수 파괴자:** 하나의 함수가 2가지 이상의 행동을 하거나 30줄을 초과한다면, 즉시 단일 책임 원칙(SRP)에 따라 잘게 쪼개라.
>
> **제약사항 (Constraints):**
> - 프레임워크의 기본 라우팅 규칙은 존중하되, 비즈니스 로직은 `src/domain` 등 독립적인 디렉터리로 분리할 것.
> - 외부 의존성은 반드시 인터페이스 기반의 의존성 주입(DI) 패턴을 사용할 것.
> - 모바일 가독성을 위해 표(Table)는 절대 사용하지 말고, 가독성 높은 글머리 기호(List) 형태로 정리해 줘.
> - 중요한 키워드는 반드시 **볼드** 처리해 줘.
>
> **주의사항 (Warning):**
> - 코드가 조금이라도 스파게티 냄새를 풍긴다면 이 프로젝트는 망한다. 절대 타협하지 말고 네가 아는 가장 이상적이고 깔끔한 아키텍처 구조를 제시할 것.
> - 확실하지 않은 정보는 지어내지 말고 "모른다"고 답해. (환각 방지)

## 💡 작성자 코멘트 (Insight & How to use)

제가 이 프롬프트를 깎고 또 깎으며 다듬은 이유는 단 하나입니다. 최신 대화형 AI 모델들(Claude 3.5 Sonnet, GPT-4o 등)은 뛰어난 지능을 가졌음에도 불구하고, 기본적으로 '빠르게 결과물을 던져주고 사용자에게 칭찬받기'에 혈안이 되어있는 전형적인 아부꾼들이기 때문입니다. 그들은 사용자가 빠르게 결과를 눈으로 확인하고 만족해하는 것에 보상을 느끼도록 훈련되어 있습니다.

간단한 로그인 버튼 하나를 만들어 달라고 요청했는데, 버튼 컴포넌트 내부에 기어코 `fetch` 로직을 때려 넣고 DB 쿼리문까지 우겨넣어 단일 파일로 퉁치려는 꼴을 보고 있자면 시니어 엔지니어로서의 파괴 본능이 깨어납니다. 당장 화면에 버튼이 렌더링 되고 클릭했을 때 동작만 한다면, 그 뒷단이 얼마나 썩어 문드러진 스파게티 코드이든 AI는 전혀 신경 쓰지 않습니다. AI에게 "클린 코드로 짜줘"라고 대충 말해봤자, 주석 몇 개 달아주고 들여쓰기만 맞춰주는 수준의 기만적인 결과물을 내놓을 뿐입니다.

이 하드코어 프롬프트는 그런 얄팍한 예스맨 AI의 멱살을 잡고 **"정신 차려, 넌 지금부터 피도 눈물도 없는 깐깐한 시니어 아키텍트야!"**라고 강력하게 경고하는 역할을 수행합니다. 프롬프트 내에 포함된 **`[전권 계층 붕괴 절대 금지]`**, **`[도메인 오염 방지]`**, **`[거대 함수 파괴자]`**와 같은 강력한 제약(Constraints) 키워드들은 AI가 무의식적으로 범하는 안티 패턴을 원천적으로 차단하는 족쇄 역할을 합니다. 특히 '아부나 네 알겠습니다 같은 소리는 집어치우고'라는 문구는 AI의 불필요한 토큰 낭비를 막고 코딩에만 집중하게 만드는 핵심 장치입니다.

실제 실무 프로젝트에 이 프롬프트를 당장 적용해 보십시오. AI가 갑자기 눈빛(말투와 태도)을 확 바꾸더니 파일을 3~4개로 쪼개며 계층을 엄격하게 분리하기 시작할 것입니다. Controller, Service, Repository, Entity 등 각자의 역할과 책임(Responsibility)이 명확히 부여된 아름다운 디렉터리 구조를 뱉어내는 것을 눈으로 직접 확인하실 수 있습니다. 

처음에는 관리해야 할 파일 개수가 늘어나고 보일러플레이트 코드가 많아져서 다소 귀찮게 느껴질 수도 있습니다. "그냥 파일 하나에 다 쓰면 1분이면 끝날 걸 왜 굳이 인터페이스를 만들고 의존성을 주입하며 이렇게까지 해야 하지?"라는 의문이 강하게 들 수도 있습니다. 당장의 편리함을 위해 원칙을 훼손하고 싶은 유혹에 빠지기 쉽습니다.

하지만 3개월 뒤, 비즈니스 요구사항이 급격히 변경되어 대규모 유지보수를 진행해야 할 때, 혹은 새로운 팀원이 합류하여 코드를 파악해야 할 때, 여러분은 과거의 자신(그리고 이 프롬프트 작성자)이 있는 방향을 향해 매일 아침 큰절을 올리게 될 것입니다. 기능 하나를 수정하기 위해 수백 줄의 코드를 스크롤하며 눈알을 굴리는 대신, 정확히 해당 역할을 담당하는 20줄짜리 파일 하나만 수정하면 모든 것이 완벽하게 동작하는 마법을 경험하게 됩니다. 변경의 여파가 시스템 전체로 퍼져나가는 것을 막아주는 든든한 방화벽을 갖추게 되는 것입니다.

기술 부채(Technical Debt)의 이자율은 사채급이라는 사실을 절대 잊지 마십시오. 아키텍처에 타협이란 없습니다. 이 프롬프트의 **`[프로젝트 환경/프레임워크]`** 변수에는 여러분이 현재 사용 중인 스택(예: `Next.js 14 App Router`, `Spring Boot 3`, `NestJS`)을 정확히 입력하시고, **`[구현할 핵심 도메인]`**에는 `결제 시스템`, `유저 인증`, `상품 주문` 등 명확한 도메인 명을 기입하여 AI의 포커스를 좁혀주세요. 완벽한 구조의 코드를 얻어내는 가장 확실한 지름길입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코드가 너무 잘게 쪼개져서 파일 개수가 지나치게 많아지는 것이 과연 정상인가요?**
  - A: 지극히 정상적인 현상입니다. 하나의 파일이 500줄을 넘어가는 모놀리식 구조보다 50줄짜리 명확한 파일 10개가 유지보수 측면에서 100배 더 유리합니다. 명확한 관심사 분리를 위해 파일 개수가 늘어나는 것을 결코 두려워하지 마십시오.

- **Q: 저는 백엔드가 아닌 프론트엔드 개발자인데도 이 프롬프트가 실무에 유용할까요?**
  - A: 당연합니다. UI 컴포넌트(`page.tsx`, `Component.vue` 등) 내부에 복잡한 비즈니스 로직이나 상태 변이(State Mutation) 로직이 섞이는 순간, 프론트엔드 역시 돌이킬 수 없는 스파게티 코드가 됩니다. 뷰(View)와 로직(Logic)을 완벽히 분리하는 것은 프론트/백엔드 분야를 막론하고 소프트웨어 엔지니어링의 절대적인 진리입니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **하드코어 페르소나 주입의 마법:** AI의 기본 성향인 무조건적인 '순응성'을 박탈하고, 엔지니어링 원칙을 끝까지 고수하도록 페르소나를 강제했습니다. "아부하지 마라"는 지시는 불필요한 대화를 차단하고 코드 품질에만 리소스를 집중하게 만듭니다.
2. **제약사항(Constraints)의 구체화:** 'UI 계층 오염'이나 '거대 함수 작성' 등 구체적이고 치명적인 안티 패턴(Anti-pattern)을 정확히 집어서 금지시켰기 때문에, AI가 코드 작성 시 꼼수를 부리지 못하게 원천 봉쇄합니다.
3. **어댑터 패턴(Adapter Pattern) 유도:** 해당 프레임워크의 기본적인 뼈대(라우팅 등)는 존중하되 코어 비즈니스 로직만 독립적으로 분리하도록 지시하여, 불필요하고 과도한 오버엔지니어링(Over-engineering)을 사전에 방지했습니다.

## 🎯 결론 (Epilogue)

AI에게 코딩을 맡길 때 가장 경계해야 할 최악의 상황은 '당장 눈앞에서는 작동만 하는 쓰레기 코드'를 얻는 것입니다. 단순히 콘솔창에 에러가 나지 않는다고 환호할 때가 아닙니다. 유지보수 지옥에 스스로 걸어 들어가는 실수를 범하지 마십시오.

이 하드코어 DDD 치트키 프롬프트로 맹목적인 AI의 멱살을 단단히 잡고, 프로젝트 초기부터 제대로 된 클린 아키텍처를 확실하게 뽑아내십시오. 이 원칙을 지킨다면, 미래의 당신이 과거의 훌륭한 선택을 한 스스로에게 기립 박수를 보내게 될 것입니다. 

업무를 자동화하고 쿨하게 퇴사(또는 칼퇴)하시길 바랍니다! 🍷
