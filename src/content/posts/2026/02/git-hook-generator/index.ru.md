# 📝 Автоматическая проверка перед коммитом! ИИ-генератор Git Hook (Git Hook Generator)

- **🎯 Рекомендуется для:** Frontend/Backend разработчиков, DevOps инженеров, Tech Lead'ов
- **⏱️ Затрачиваемое время:** 1 час → сокращается до 1 минуты
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o (модели, оптимизированные для написания кода)

- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _«Ой, забыл запустить линтер! 🚨» Навсегда попрощайтесь с красными логами ошибок в CI/CD пайплайнах благодаря автоматизации Git Hook._

Людям свойственно ошибаться, но системы не должны прощать ошибок. Запускать `Lint` или выполнять тесты вручную прямо перед коммитом (Commit) или пушем (Push) кода — это утомительно, и об этом легко забыть.

В такие моменты на помощь приходит **Git Hook**. Однако, если вы не знакомы с синтаксисом Shell-скриптов, написание кроссплатформенного, безупречно работающего хука с нуля может стать настоящим стрессом. Теперь достаточно просто описать ИИ конвенции вашей команды и рабочее окружение. Всего за секунду он создаст для вас надежный скрипт-страж.

## ⚡️ 3 главных преимущества (TL;DR)

1. **Вершина автоматизации:** Принудительно проверяет стиль коммит-сообщений и форматирование кода за секунду до коммита.
2. **Точный скриптинг от ИИ:** Больше не нужно ломать голову над сложными регулярными выражениями или кроссплатформенными shell-скриптами — ИИ сгенерирует их под ваше окружение.
3. **Экономия ресурсов CI:** 100% блокировка ошибок в локальной среде еще до их попадания в пайплайн экономит время и затраты на облачные вычисления.

### 🥉 Basic Version (Базовая версия)

Используйте, когда вам нужен простой скрипт для выполнения конкретного действия.

> **Роль:** Ты — `[DevOps инженер]`.
> **Задача:** Напиши shell-скрипт Git Hook и инструкцию по настройке с использованием `[Husky]`, чтобы `[запускать npm run lint перед выполнением git commit]`.

### 🥇 Pro Version (Продвинутая версия)

Используйте, когда требуется надежный скрипт с безупречной обработкой исключений для всей команды.

> **Роль (Role):** Ты — Senior `[DevOps инженер]` с 10-летним стажем и `[эксперт по автоматизации Git]`.
>
> **Контекст (Context):**
>
> - Ситуация: Члены команды не соблюдают конвенции кода перед коммитом, что приводит к частым сбоям сборок и трате ресурсов в CI пайплайне.
> - Окружение: `[Frontend-проект на базе Node.js, используется Husky]`
> - Цель: Создать надежные хуки `[pre-commit и commit-msg]`, которые на корню пресекают ошибки разработчиков.
>
> **Задача (Task):**
>
> 1. Хук `[pre-commit]`: Напиши shell-скрипт, который выполняет `[npm run lint]` и `[npm run format]` только для проиндексированных файлов (staged files: `git diff --staged`).
> 2. Хук `[commit-msg]`: Добавь скрипт с проверкой через регулярные выражения, строго контролирующий соблюдение формата `[Conventional Commits (feat, fix, docs и т.д.)]`.
> 3. **Сообщения для пользователя:** В случае провала тестов или проверок выведи в терминал понятное и дружелюбное сообщение об ошибке (с примером) и заблокируй коммит (exit 1).
> 4. **Руководство по внедрению:** Подготовь конфигурацию `[package.json]` и команды для начальной установки в виде Markdown блоков кода, чтобы команда могла быстро настроить окружение сразу после клонирования репозитория.
>
> **Ограничения (Constraints):**
>
> - Написанный shell-скрипт должен быть кроссплатформенным и безопасно работать на macOS, Linux и Windows (в среде Git Bash).
>
> **Предупреждения (Warning):**
>
> - Проект большой, поэтому проверка всех файлов займет слишком много времени. Обязательно оптимизируй производительность, чтобы проверялись **только измененные файлы (staged files)**. (Например, используй `lint-staged`).

## 💡 Комментарий автора (Insight)

Истинная ценность этого промпта заключается не только в **предотвращении ошибок**, но и в **психологическом спокойствии**, которое он дарит. Я всегда использую адаптированную версию этого промпта в личных проектах, чтобы настроить хук `pre-push`, который блокирует прямой пуш в главную ветку (`git push origin main`).

Облегчение, которое вы испытываете, когда сонные глаза пытаются сделать форсированный пуш в main посреди ночи, а сгенерированный ИИ скрипт блокирует его сообщением в терминале: **«🚨 Стоп! Изменения в ветку main можно вносить только через PR»**, — поймут только те, кто сам через это прошел. И не забудьте попросить интеграцию с `lint-staged` — это ускорит проверки более чем в 10 раз!

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Будет ли этот shell-скрипт работать у коллег на Windows?**
  - О: Да! Поскольку в промпте указано требование «кроссплатформенности», ИИ учтет это. Для Windows он сгенерирует код, совместимый с Git Bash. Однако в экосистеме Node.js самым надежным и безопасным решением с точки зрения совместимости будет попросить ИИ использовать связку `Husky` + `lint-staged` вместо написания чистого shell-скрипта.

- **В: Скрипт оказался слишком тяжелым, каждый коммит теперь занимает несколько минут.**
  - О: Скорее всего, вы запускаете `npm run lint` или `npm run test` для всей кодовой базы. Обязательно добавьте условие, указанное в секции «Предупреждения»: **«проверять только проиндексированные (измененные) файлы»**. ИИ перепишет скрипт в более легкий вариант, используя `git diff --cached --name-only`.

- **В: Хук мешает мне отменить уже сделанный крупный ошибочный коммит.**
  - О: Если вам экстренно нужно закоммитить или запушить код в обход хуков, добавьте флаг `--no-verify` (или `-n`) в конец команды. (Например: `git commit -m "fix: горячее исправление" --no-verify`). Но не злоупотребляйте этим!

## 🧬 Анатомия промпта (Why it works?)

1.  **Детализация контекста (Context):** Вместо абстрактного «напиши мне хук», мы явно указываем используемый стек (`Husky`, `Node.js`) и болевые точки (частые ошибки сборок, трата ресурсов). Это позволяет ИИ предложить точечные решения (например, использование `lint-staged`).
2.  **Ограничение для оптимизации (Warning):** Ошибка новичков — проверка всех файлов. Блокируя этот подход на уровне промпта, мы сразу получаем быстрый и легкий код, готовый к применению в реальной работе.
3.  **Кроссплатформенность (Constraints):** Учет многообразия операционных систем в команде гарантирует, что скрипт одинаково хорошо отработает на Windows, macOS и Linux.
