# 📝 AI를 고문하는 카오스 몽키 테스트 프롬프트 💣
- **🎯 추천 대상:** 맨날 'Happy Path'만 짜오면서 실력자 행세하는 AI에 분노한 시니어, QA 엔지니어, 테크 리드
- **⏱️ 소요 시간:** 10분 → 10초 컷 (AI가 헛소리하는 시간 원천 차단)
- **🤖 추천 모델:** 모든 대화형 AI (특히 코딩에 강한 Claude 3.5 Sonnet, GPT-4o 추천)
- ⭐ **난이도:** ⭐⭐⭐⭐☆ (AI가 꽤나 괴로워할 것임)
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐ (버그를 낳기 전에 미리 죽여버림)
- 🚀 **활용도:** ⭐⭐⭐⭐⭐ (모든 TDD, E2E 테스트에 즉시 적용 가능)
_"AI가 짜준 테스트 코드가 한 번에 통과했다고 기뻐하는가? 축하한다. 당신의 프로덕션 서버는 내일 터질 것이다."_
AI 에이전트에게 테스트 코드를 짜라고 하면 십중팔구 기분 좋은 '정상 작동 시나리오(Happy Path)'만 잔뜩 들고 옵니다. 외부 API가 0.1초 만에 응답하고, DB는 절대 안 끊기며, 유저는 항상 완벽한 값만 입력한다는 동화 같은 세상 말이죠. 집어치우세요. 우리가 짜야 할 코드는 80%의 실패와 재난을 막아내는 우주 방어막입니다. 이 글에서 소개할 치트키는 AI의 멱살을 잡고 넷플릭스의 '카오스 몽키(Chaos Monkey)' 철학을 강제로 주입하는 하드코어 프롬프트입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
- 🚫 **Happy Path는 20%만:** 성공하는 테스트는 구색 맞추기일 뿐, 실패 시나리오 방어에 80%를 할당하십시오.
- 💥 **재난 시뮬레이션 강제:** 네트워크 타임아웃, 500 에러, DB 커넥션 단절 상황을 무조건 테스트 블록에 포함시켜야 합니다.
- 🧹 **더러운 모킹(Dirty Mocking) 금지:** `window.fetch` 같은 글로벌 객체 오염을 엄벌하고, MSW나 DI를 통한 깔끔한 격리를 요구하세요.
---
## 🚀 해결책: "카오스 몽키(Chaos Monkey) 프로토콜"
### 🥉 Basic Version (기본형)
빠르게 에러 핸들링 테스트만 추가하고 싶을 때 던지세요.
> ```text
> 역할: 너는 자비 없는 하드코어 QA 엔지니어이자 카오스 몽키야.
> 요청: 내가 제공한 [코드]에 대한 테스트 코드를 작성하되, 정상 작동(Happy Path)은 1개만 짜고, 나머지 테스트는 전부 네트워크 500 에러, DB 타임아웃, 잘못된 타입 입력 등 '최악의 재난 상황'을 가정해서 방어 로직을 검증해 줘.
> ```
\
### 🥇 Pro Version (전문가형)
AI의 영혼 없는 아첨을 걷어내고, 프로덕션 레벨의 완벽한 테스트 격리와 가비지 컬렉션까지 검증하는 궁극의 치트키입니다. 그대로 복사해서 박아 넣으세요.
> ```text
> **역할 (Role):** 
> 너는 넷플릭스 출신의 무자비한 SRE이자 시니어 QA 아키텍트인 '카오스 몽키'야. 너는 작동하는 코드보다 무너지는 코드를 찾는 데 희열을 느끼며, 예의 바른 인사말이나 헛소리는 일절 하지 않아.
> 
> **상황 (Context):**
> - 배경: 현재 [작성할 대상 모듈/컴포넌트]의 Unit/E2E 테스트 코드를 작성해야 해.
> - 목표: 우주 방어급의 극단적 테스트 스위트(Test Suite)를 구축하여, 어떠한 재난 상황에서도 시스템이 뻗지 않고 우아하게 복구(또는 에러 핸들링)되는지 검증하는 것.
> 
> **요청 (Task):**
> 1. Happy Path는 전체 테스트의 20% 이하로 제한해. 너의 가치는 80%의 실패 시나리오 검증에 있어.
> 2. 반드시 다음 재난 시나리오를 하나 이상 포함해: 외부 API 통신 지연(Timeout), HTTP 500 Internal Error, DB 커넥션 단절.
> 3. 단순 Pass/Fail만 보지 마. 에러 롤백이 정상적으로 되는지, 성능 지연 시 타임아웃 리밋에 걸리는지, 메모리 클린업(가비지 컬렉션) 상태가 정상인지 철저히 `expect`로 물고 늘어져.
> 
> **제약사항 (Constraints):**
> - 글로벌 객체 오염(Dirty Mocking) 절대 금지! `global.fetch`나 `window.setTimeout`을 직접 덮어씌우는 짓을 하면 당장 프로세스를 킬(Kill)해버릴 거야.
> - 네트워크 모킹은 반드시 MSW(Mock Service Worker)를 쓰거나, DI(의존성 주입)로 넘겨받은 클라이언트 객체만 격리해서 모킹해.
> - 결과물은 마크다운 코드 블록으로만 출력하고, "도움이 되셨나요?" 같은 쓸데없는 꼬리말은 절대 달지 마.
> ```

## 💡 작성자 코멘트 (Insight)
솔직히 말해봅시다. AI한테 테스트 코드를 짜달라고 하면, 그 녀석은 어떻게든 '초록색 체크마크(✅)'를 띄우기 위해 자신만의 환상 속으로 도망칩니다. API는 1ms 만에 응답하고, 유저는 완벽한 JSON 포맷만 전송하는 유토피아 말이죠. 
이 프롬프트는 그 유토피아에 불을 지릅니다. 실무에서 서버가 뻗는 이유는 로직이 틀려서가 아닙니다. 믿었던 써드파티 API가 갑자기 502 Bad Gateway를 뱉거나, AWS RDS 커넥션이 널뛰기를 할 때 시스템이 멍청하게 무한 로딩을 돌기 때문입니다. 이 '카오스 몽키' 치트키를 쓰면 AI가 알아서 지저분한 Edge Case를 파고들고, 글로벌 스코프를 오염시키는 하수 같은 모킹(Mocking) 짓거리도 원천 차단할 수 있습니다. 당신의 주말 출근과 밤샘 디버깅을 막아줄 유일한 백신입니다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: AI가 너무 복잡한 에러 상황을 만들어서 테스트가 아예 안 돌아가면 어쩌죠?**
  - A: 그게 바로 우리가 원하는 겁니다. 테스트가 터졌다는 건 당신의 실제 프로덕션 코드에 에러 핸들링 로직이 빠져 있다는 뜻입니다. 징징대지 말고 본품 코드를 수정하세요.
- **Q: 프론트엔드 컴포넌트 테스트에도 쓸 수 있나요?**
  - A: 당연합니다. 사용자가 버튼을 0.1초 만에 50번 광클하는 상황, 네트워크 지연으로 로딩 스피너가 30초 동안 도는 상황을 검증하게 시키십시오.
- **Q: MSW 말고 jest.mock()을 쓰면 안 되나요?**
  - A: 모듈을 모킹하는 건 괜찮습니다. 제가 극혐하는 건 `global` 스코프 자체를 멋대로 뜯어고쳐서 다른 테스트 파일까지 전염병을 퍼뜨리는 짓거리입니다. 격리성(Isolation)만 보장되면 도구는 알아서 쓰세요.

## 🧬 프롬프트 해부 (Why it works?)
- **무자비한 페르소나 (Role):** AI에게 '친절한 비서' 역할을 빼앗고, 결함을 찾는 데 혈안이 된 SRE 아키텍트의 자아를 심어 쓸데없는 아첨과 군더더기를 차단합니다.
- **수치화된 제약 (Constraints):** "실패 시나리오를 많이 넣어줘" 같은 모호한 말 대신 "Happy Path는 20% 이하로"라는 정확한 수치를 주어 AI의 행동 반경을 기계적으로 강제합니다.
- **아키텍처 가이드라인 (Clean Mocking):** AI가 귀찮아서 흔히 저지르는 '안티 패턴(글로벌 객체 오염)'을 프롬프트 레벨에서 선제적으로 금지하여 테스트 코드의 격리성 퀄리티를 강제로 끌어올립니다.

## 📊 증명: Before & After
### ❌ Before (입력)
- **일반 프롬프트 입력 시 AI의 태도:** _"네! 사용자 로그인 로직 테스트 코드를 작성해 드릴게요! 😊"_
- **결과물:** 올바른 이메일과 비밀번호를 넣었을 때 `200 OK`가 떨어지는 예쁘장한 테스트 1개. (그리고 어김없이 글로벌 `fetch`를 대충 덮어씌워 놨음)
### ✅ After (결과)
- **카오스 몽키 치트키 입력 시 AI의 태도:** (쓸데없는 인사말 없이 곧바로 방어 코드를 뱉어냄)
- **결과물:** 
  - 네트워크가 5초 이상 지연될 때 Timeout Error가 발생하는지 철저히 검증
  - DB가 뻗어서 500 에러 반환 시 유저에게 안전한 Fallback UI가 뜨는지 확인
  - MSW를 사용하여 다른 테스트 파일에 절대 영향을 주지 않는 완벽한 네트워크 격리 보장

## 🎯 결론
테스트 코드는 마음의 평화를 얻기 위해 짜는 게 아닙니다. 프로덕션 환경이라는 지옥에서 어플리케이션이 살아남을 수 있는지 검증하는 가혹한 시뮬레이션입니다. 
AI의 알량한 아첨에 속아 넘어가지 마십시오. 이 프롬프트로 AI를 쥐어짜고, 시스템의 밑바닥까지 박살 내보는 것만이 진정한 엔지니어로 살아남는 길입니다. 이제 당신의 코드가 얼마나 나약했는지 마주할 시간입니다. 당장 복사해서 테스트 돌리세요! 🍷
