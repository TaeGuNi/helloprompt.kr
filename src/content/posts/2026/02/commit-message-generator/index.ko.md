## 📝 "커밋 메시지 추천 좀" 고민 끝! Conventional Commits 자동 생성

- **🎯 추천 대상:** 주니어 개발자, 오픈소스 기여자, 협업을 시작하는 팀
- **⏱️ 소요 시간:** 5분 → 10초 단축
- **🤖 최상위 성능:** ChatGPT (GPT-4o), Claude 3.5 Sonnet, Gemini 2.5 Pro

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"코드 짜는 데는 3시간, 커밋 메시지 고민하는 데 30분... 'Update code' 치고 퇴근하고 싶었던 적 없으신가요?"_

`fix: bug`, `update code`, `진짜_최종_수정`... 혹시 오늘도 깃허브(GitHub)에 이런 무의미한 커밋 로그를 남기고 황급히 퇴근하셨나요?
아니면 `git commit -m "..."` 명령어 앞에서 깜빡이는 커서를 보며, 도대체 무슨 말을 적어야 할지 몰라 5분 넘게 모니터만 노려보고 계셨나요?
코드를 짜는 데는 3시간이 걸렸지만, 막상 터미널 창에서 커밋 메시지를 작성하려고 하면 머릿속이 하얗게 변하는 경험은 프론트엔드, 백엔드를 가리지 않고 모든 개발자가 매일같이 겪는 만성적인 고통입니다.
특히 영어로 작성해야 하는 글로벌 프로젝트의 압박감, 그리고 **Conventional Commits** 같은 엄격한 컨벤션 규칙까지 지켜야 하는 상황이라면 커밋은 더 이상 단순한 버전 관리가 아니라 **하나의 까다로운 작문 시험**으로 변질됩니다.
"이 변경사항을 어떻게 짧은 영어로 요약하지?", "새로운 기능이긴 한데 feat인가? 아니면 기존 코드를 고쳤으니 refactor인가?", "이슈 트래커 번호는 어디에 어떻게 붙여야 에러가 안 나지?"
이런 수많은 고민 끝에 결국 우리는 구글 번역기나 파파고를 돌리다 지쳐, 가장 단순하고 성의 없는 메시지를 띡 남기게 됩니다.

하지만 이렇게 대충 남긴 커밋 메시지는 **반드시 눈덩이처럼 불어나 미래의 당신과 동료들을 가차 없이 덮칩니다.**
단 몇 달 뒤, 프로덕션 환경에서 치명적인 버그가 발생해 코드를 긴급히 롤백(Rollback)해야 할 때를 상상해 보십시오. `fix: bug`라고만 덩그러니 적힌 수십 개의 커밋 로그를 마주하는 순간, 어떤 커밋에서 문제가 시작되었는지 추적하는 것은 모래사장에서 바늘 찾기보다 고통스러운 일이 됩니다.
더 큰 문제는 **팀원들과의 협업 과정**에서 여실히 드러납니다. 동료들이 당신의 PR(Pull Request)을 리뷰할 때, 어떤 의도로 코드를 수정했는지, 왜 이런 방식을 선택했는지 파악하지 못해 소중한 업무 시간을 코드 해석에 낭비하게 됩니다.
특히 오픈소스 프로젝트나 체계적인 코드 리뷰 문화를 가진 테크 기업에서는 무성의한 커밋 메시지가 곧 **해당 개발자의 실력과 프로의식 부족**으로 직결되어 낮은 고과 평가를 받기 일쑤입니다. 요즘처럼 CI/CD 파이프라인이 자동화된 시대에는 커밋 메시지가 곧 릴리스 노트(Release Notes)로 직결되기 때문에, 규칙을 어긴 메시지는 자동 배포 프로세스마저 중단시킬 수 있습니다.
코드만 잘 짜면 끝나는 시대는 완전히 지났습니다. <b>완벽한 커밋 메시지는 당신의 코드를 돋보이게 하는 가장 강력하고 세련된 포장지입니다.</b>

그렇다면 매번 영어 사전과 컨벤션 가이드라인 문서를 뒤적이며 억지로 문장을 쥐어짜야 할까요? 절대 아닙니다.
이제 더 이상 번역기를 이리저리 돌리며 작문 스트레스를 받을 필요가 전혀 없습니다. 여러분이 땀 흘려 작성한 코드의 **변경 사항(`git diff`)** 결과나, 간단한 작업 요약 한 줄만 프롬프트에 무심하게 던져주세요.
오픈소스 생태계의 절대적인 글로벌 표준인 **Conventional Commits 규격**에 한 치의 오차도 없이 맞춰, 그대로 복사해서 터미널에 붙여넣을 수 있는 **완벽한 영문 커밋 메시지**를 AI가 즉석에서 생성해 드립니다.
이것은 단순한 단어 번역 수준이 아닙니다. AI가 추가되고 삭제된 코드 라인의 문맥을 꼼꼼하게 분석하여, 당신이 미처 명시하지 못한 작업의 근본적인 의도(Why)와 해결 방식(What)까지 정확하게 짚어내는 소름 돋는 커밋 로그를 완성해 냅니다.

이 프롬프트를 실무에 도입하는 즉시, 당신의 퇴근 시간은 매일 최소 10분 이상 확실하게 단축됩니다.
커밋 메시지 작성이라는 비생산적인 작업에 소모되던 불필요한 인지적 에너지를 아껴, 오직 핵심 비즈니스 로직 구현과 아키텍처 설계 고민에만 오롯이 집중할 수 있습니다.
무엇보다 깃허브 저장소를 함께 공유하는 동료들로부터 **"이 사람 커밋 로그 참 깔끔하고 명확하네. 진짜 같이 일하기 좋은 시니어급 개발자다!"**라는 극찬을 이끌어내는 프로페셔널한 엔지니어로 완벽하게 거듭날 것입니다.
지금 당장 터미널 창에 `git diff`를 치고, 아래에 준비된 프롬프트를 통해 코딩의 마지막 단계를 장식하는 마법 같은 자동화를 직접 경험해 보세요!

> "로그인 버튼 색깔 빨간색으로 바꾸고, 로그인 안 되는 버그 수정했어. 아 그리고 리드미에 설치법 추가함."

### ✅ After (완벽하게 변신한 결과)

**프롬프트 적용 결과:**

**Option 1 (기능 버그 수정 우선순위)**
```text
fix(auth): resolve login failure issue and update button UI

- Fix logic error that prevented users from logging in
- Change the login button color to red for better visibility
- docs: Add installation guide to README.md
```

**Option 2 (UI 및 스타일 중심)**
```text
style(ui): update login button color to red and fix login bug

- Change login button color to enhance UI consistency
- Fix authentication bug causing login failures
- docs: Update README with installation instructions
```

## ⚡️ 3줄 요약 (TL;DR)

1. `git diff` 결과만 붙여넣으면 글로벌 표준인 Conventional Commits 규격으로 즉시 변환됩니다.
2. `feat`, `fix`, `refactor` 등 변경 사항의 성격에 부합하는 정확한 **타입(Type)**을 AI가 스스로 분류합니다.
3. 작업의 의도(Why)와 구체적 내용(What)이 논리적으로 담긴 본문(Body)까지 완벽한 영문으로 작성해 드립니다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

수십 번의 시행착오 끝에 완성된 프롬프트입니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

### 🥉 Basic Version (기본형)

빠르게 제목 한 줄만 필요할 때 가볍게 사용해 보세요.

> **역할:** 너는 완벽한 Git 히스토리를 추구하는 시니어 개발자야.
>
> **요청:** 내가 제공하는 변경 내용을 보고, Conventional Commits 표준에 맞춘 영문 커밋 메시지 제목(`type(scope): subject` 형식)을 3가지 제안해 줘.

### 🥇 Pro Version (전문가형)

제목, 본문, 꼬리말까지 완벽하게 구조화된 수준 높은 커밋 메시지가 필요할 때 사용하세요. 아래 프롬프트를 복사하여 AI 창에 그대로 붙여넣으시면 됩니다.

> **역할 (Role):**
> 너는 수많은 오픈소스를 관리하며 완벽한 Git 히스토리에 집착하는 엄격한 메인테이너이자 시니어 엔지니어링 매니저야.
>
> **상황 (Context):**
>
> - 배경: 내가 방금 코드 작성을 마쳤고, 커밋을 남겨야 하는 상황이야.
> - 목표: 동료들이 리뷰하기 쉽고, 나중에 히스토리를 추적하기에 완벽한 커밋 메시지를 작성해야 해.
>
> **요청 (Task):**
> 아래의 `[변경 사항]`을 분석하여 **Conventional Commits v1.0.0** 규칙에 맞는 완벽한 커밋 메시지 초안을 3가지 옵션으로 제시해 줘.
>
> 1. **형식:** `<type>(<scope>): <subject>` (제목 1줄)
> 2. **본문 (Body):** 왜(Why) 변경했고 무엇(What)을 어떻게 해결했는지 불릿 포인트(`-`)로 명확히 요약할 것.
> 3. **꼬리말 (Footer):** (선택 사항) 관련된 이슈 번호가 있다면 `Closes #이슈번호` 형태로 작성할 것.
>
> **제약사항 (Constraints):**
>
> - Type은 반드시 다음 중 하나만 사용할 것: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`
> - Subject(제목)는 50자를 넘지 않도록 간결하게 작성하고, 반드시 **영문 명령문(동사 원형)**으로 시작할 것 (예: `Add`, `Fix`, `Update`). 끝에 마침표(`.`)는 찍지 마.
> - 본문(Body)은 제목과 한 줄 띄우고 작성하며, 영문으로 작성해 줘.
> - 각 옵션의 차이점이나 포커스(예: 버그 수정 중심, UI 개선 중심)를 한국어로 짧게 덧붙여 줘.
>
> **입력 (Input):**
>
> - 변경 사항: `[여기에 git diff 결과나 수정한 내용을 한국어로 편하게 입력하세요]`

## 💡 작성자 코멘트 (Insight & How to use)

이 프롬프트가 가진 가장 압도적인 강점은 바로 개발자들을 매일같이 끈질기게 괴롭히는 **'영작에 대한 심리적 압박감'**과 **'엄격하고 복잡한 포맷팅 규칙'**이라는 두 가지 거대한 허들을 단 한 번의 복사 붙여넣기로 완벽하게 제거해 준다는 점입니다.

우리가 커밋을 할 때 단순히 "이 버튼 색깔을 파란색으로 고쳤어"라고 말하는 것을 넘어, 터미널 환경에서 `git diff` 명령어를 실행한 뒤 화면에 출력된 붉은색과 초록색의 복잡한 코드 변경 내역을 그대로 드래그하여 복사한 다음 프롬프트의 `[변경 사항]` 영역에 거침없이 붙여넣어 보세요. 놀랍게도 AI는 당신이 어떤 파일을 건드렸는지, 어떤 변수명이 바뀌었는지, 그리고 그 변경이 전체 시스템에서 어떤 맥락을 가지는지를 스스로 추론하고 분석해 냅니다. 개발자가 미처 말로 설명하지 못한 작업의 숨은 의도와 로직의 변화까지 짚어내는 소름 돋는 퀄리티의 커밋 메시지를 알아서 완성해 내는 것이죠.

<b>💡 100% 실무 밀착형 변수 통제법 (Constraint Control):</b>
이 프롬프트가 일반적인 챗봇 대화와 차원이 다른 이유는 **제약사항(Constraints)**을 통해 AI의 환각(Hallucination)과 자유도를 극단적으로 통제했기 때문입니다. Conventional Commits는 생각보다 규칙이 매우 까다롭습니다. 예를 들어, 메시지의 시작은 반드시 **동사 원형(영문 명령문)**이어야 하며, 제목의 끝에 마침표(`.`)를 찍으면 린트(Lint) 에러가 발생합니다. 또한 전체 길이가 50자를 넘어가면 깃허브 화면에서 메시지가 잘려 보이는 현상이 발생하죠.
이 프롬프트는 깐깐한 시니어 엔지니어가 코드 리뷰에서 지적할 법한 이러한 모든 잠재적 오류 요소들을 선제적으로 차단하도록 정교하게 설계되었습니다. 만약 여러분의 팀이 특별한 접두사나 지라(Jira) 티켓 번호를 필수로 요구한다면, 제약사항 부분에 `- 제목 맨 앞에 반드시 '[ABC-123]' 형식의 지라 티켓 번호를 붙여 줘`라는 단 한 줄의 조건만 추가하면 됩니다. AI는 기존의 엄격한 룰을 유지하면서도 여러분 팀만의 커스텀 컨벤션을 완벽하게 병합하여 출력해 냅니다.

코딩을 모두 마친 뒤 에디터 내장 AI에게 **"방금 수정한 내용 커밋 메시지로 써줘"**라는 짧은 명령어 단 한 마디만 던지면 끝입니다. 에디터가 현재 작업 중인 `git diff` 상태를 자체적으로 읽어들이고, 이 완벽하게 세팅된 프롬프트 룰에 맞춰 1초 만에 커밋 메시지를 뱉어냅니다. 여러분은 그저 `Accept` 버튼을 누르거나 복사해서 커밋하기만 하면 됩니다. 이렇게 작업 플로우를 한 단계 단축시키는 것만으로도, 여러분의 하루 개발 생산성과 몰입도는 상상 이상으로 비약적인 상승 곡선을 그리게 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 회사 내부 커밋 규칙(예: Jira 티켓 번호 필수 포함)이 따로 있는데 어떻게 적용하나요?**
  - A: 프롬프트의 **제약사항 (Constraints)** 섹션에 `- 제목 맨 앞에 반드시 '[ABC-123]' 형식의 지라 티켓 번호를 붙여 줘`라는 조건을 단 한 줄만 추가하시면, 팀의 고유한 컨벤션을 완벽하게 준수하는 결과를 얻을 수 있습니다.

- **Q: 제목은 영어로 하되, 커밋 본문(Body)은 한국어로 작성하고 싶은데요?**
  - A: 프롬프트 내용 중 `본문(Body)은 제목과 한 줄 띄우고 작성하며, 영문으로 작성해 줘.`라는 문장을 `본문은 한국어로 명확하게 작성해 줘.`로 살짝 수정해 보세요. 글로벌 표준의 껍데기는 유지하면서, 팀원들과의 소통 효율은 극대화하는 실용적인 포맷이 완성됩니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **표준 스펙 강제 (Conventional Commits v1.0.0):** AI에게 막연하게 "좋은 메시지를 써줘"라고 요청하는 대신, 전 세계적으로 통용되는 명확한 표준 규격을 직접 지시함으로써 출력물의 포맷이 흔들리지 않도록 강력하게 고정했습니다.
2. **명확한 제약 조건 (Constraints):** "명령문으로 시작", "50자 제한", "마침표 금지" 등 깐깐한 시니어 개발자가 코드 리뷰에서 지적할 법한 엄격한 린트(Lint) 규칙들을 프롬프트 레벨에서 선제적으로 차단합니다. 이를 통해 추가적인 수정 비용을 0으로 만듭니다.
3. **선택권 부여 (3가지 옵션):** AI가 단 하나의 정답을 강요하지 않도록 설계했습니다. 성격이 다른 3가지 옵션을 동시에 제시하게 하여, 사용자가 자신이 의도한 작업의 핵심과 가장 잘 맞는 최적의 메시지를 취사선택할 수 있습니다.
