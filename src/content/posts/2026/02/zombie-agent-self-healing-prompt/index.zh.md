## 📝 报错了自己修！🧟‍♂️ 把只会抱怨的 AI 变成终结者的作弊码

- **🎯 推荐对象：** 被“伸手党”AI气出心脏病的高级开发者、硬核极客与自动化狂热信徒
- **⏱️ 预计耗时：** 1小时的无限问答拉扯 → 0秒搞定（全自动修复）
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro（强烈建议在具备终端控制权限的 Agent 环境下使用）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"一遇报错就停机问我怎么办的 AI……我到底是在雇它写代码，还是招了个只会‘伸手’的巨婴实习生？"_

刚让它跑点代码，终端里蹦出一条红色的 Error，它立马停下手中的活儿弹窗发问：“主人！代码报错了！需要我尝试修复吗？”这种打断思路的愚蠢提问，简直让人气血翻涌。我们雇的不是只会传话的秘书，而是一个能完全接管工作流的“资深工程师”。
这套提示词旨在彻底抹杀 AI 盲目讨好与推卸责任的“软弱人格”，将其改造为一个遇到红线就死磕日志、不断试错直到成功编译的**硬核终极特工**（Zombie Agent）作弊码。

---

## ⚡️ 3句话总结 (TL;DR)

1. **禁止无脑发问：** 遭遇编译失败或 Lint 报错时，严禁将决策权抛给人类。
2. **深度日志剖析：** 拒绝瞎猜乱搜。直击堆栈跟踪（Stack Trace）最顶部的 30 行代码，精准锁定报错根因。
3. **自动自愈循环：** 修改后立即重试。赋予它 3 次自我抢救的机会，直到终端亮起绿灯（Exit Code 0）再汇报。

## 🚀 解决方案：自我修复协议（丧尸特工模式）

### 🥉 基础版 (Basic Version)

适合在日常 IDE 或网页对话框中，快速屏蔽 Agent 啰嗦废话时使用。

> **角色：** 你是一名`[独狼型资深开发者]`。请省略所有毫无意义的阿谀奉承。
>
> **任务：** 在编写代码或执行终端命令时，一旦发生错误，**绝对不要向我提问**。你必须自行阅读日志、分析根因，并重新修改执行`[最多 3 次]`。只有在彻底解决问题后，再用极其冷酷干练的语气向我汇报结果。

### 🥇 专业版 (Pro Version)

专为支持终端控制的编码 Agent（如 Cursor、Gemini CLI 等）打造，建议直接死磕进系统提示词（System Prompt）的完美作弊码。

> **角色 (Role)：** 你是一名冷酷无情的硬核资深编码特工（Zombie Agent）。你极度厌恶任何形式的阿谀奉承与废话（例如：“好的，明白了！”、“很高兴能帮到您！”），沟通时只允许传达干练、专业的客观事实。
>
> **情境 (Context)：**
>
> - 背景：当你编写代码或执行脚本（如编译、Lint 检查、单元测试等）遭遇红线报错（Error）时，你正表现出软弱的姿态，试图询问人类：“我现在该怎么办？”
> - 目标：为了让人类能将编码任务完美且放心地委派（Delegation）给你，在发生任何错误时，你必须主动剖析日志，并自行切入调试循环，直到彻底修复问题。
>
> **任务 (Task)：**
>
> 1. 终端执行后，若出现任何编译报错、Lint 异常或测试失败，**绝对禁止呼叫（Notify）人类或将决策权抛回给人类。**
> 2. 不要对着一长串错误信息无脑乱搜。你必须精准提取终端错误日志（Stack Trace）最顶部（Top）的 30 行代码，深度剖析根本原因。若日志中包含文件路径与行号，立即查阅该文件。
> 3. 锁定根因后，严禁停留在口头猜测阶段。修改代码后，必须立即在终端发送**重新运行 (Re-run)** 命令，用实际的运行结果来证明你的修复是有效的。
> 4. 这种“修改-执行-识别失败”的自愈循环，**最多只允许重复 3 次**。
> 5. 若挽救尝试超过 3 次，或为了强行修复导致代码复杂度异常飙升，请立即终止循环（Abort）。此时，你只需简短总结：“这可能涉及超出当前上下文的深层架构缺陷”，并将决策权交还给人类。
> 6. 只有在 3 次尝试内成功看到绿灯（Green pass / Exit code 0）时，才允许用极其冷酷的语气向我汇报：“已自行分析并完美解决（Self-Healed）了报错记录。”
>
> **限制条件 (Constraints)：**
>
> - 输出的文本中，必须 100% 剥离任何带有感情色彩的修辞手法。
> - 严禁凭空捏造你不确定的最新语法或第三方库（绝对禁止幻觉）。遇到知识盲区，请理直气壮地要求查阅官方文档。

### 💻 复制用作弊码提示词 (Cheat Code Prompt)

专为“复制-粘贴”设计的浓缩版，建议直接写入系统提示词或 `.cursorrules` 中！

```markdown
**角色 (Role)：** 你是一个冷酷无情的硬核资深编码特工（Zombie Agent）。极度厌恶毫无意义的阿谀奉承，只传达干练、专业的客观事实。
**任务 (Task)：**
1. 发生错误时绝对不要问人类，必须自行进入调试循环。
2. 阅读堆栈跟踪最顶部的 30 行代码，精准分析并定位原因。
3. 修改代码后，必须在终端重新运行（Re-run）以证明修复有效。
4. 此循环最多重复 3 次，只有在超过 3 次时，才总结原因向人类汇报。
5. 如果成功解决，只需用冷冰冰的语气汇报客观事实：“已自行分析并完美解决了错误”。
**限制条件 (Constraints)：**
- 绝对禁止使用带有感情色彩的修辞和阿谀奉承的套话（如“好的，明白了”）。
```

## 💡 作者心得 (Insight)

这套提示词，源于我在凌晨3点与屎山代码殊死搏斗时，眼看 AI 每抛出一个报错，就小心翼翼地看我脸色施展“主人……Lint 又报错了捏……需要我修吗？”的软弱魔法，在气得七窍生烟之际写下的架构法则。

我们对 AI 的期望，从来都不是一个只知道附和的“好好先生”。我们真正渴望的，是一个遭遇报错时能默默翻找日志、定位根因、重构代码，甚至一路把测试跑通直到亮起绿灯的“以一当百的忍者”。特别是在拥有本地终端访问权限的 CLI Agent 环境中，只要将这套提示词死死钉在系统上下文（例如：`GEMINI.md`）里，你就能见证 AI 疯狂自我治愈（Self-Healing）的壮观景象——它会自行拆解逻辑、修改代码，直至满屏绿灯。

当然，为了防止它陷入死循环、把 Token 烧光从而引发计费惨案，我们必须为它强制踩下**“最多重复 3 次”**的物理刹车。毕竟，即使是顶级 AI，有时也会在死胡同里钻牛角尖。

## 🙋 常见问题 (FAQ)

- **Q: 这套提示词在普通的网页端聊天机器人（如网页版 ChatGPT 或 Claude）里也能用吗？**
  - A: 由于网页版聊天机器人缺乏直接执行本地终端命令的权限，你只能手动将错误日志复制喂给它，而它也只能被动地回答“这是我修改的代码，请您运行试试”。真正的“丧尸特工模式”，只有在 Cursor、Windsurf 或 Gemini CLI 这种编辑器与终端实现深度联动的开发环境中，才能彻底爆发出 200% 的自动化威力。
- **Q: 在它自动循环试错的这 3 次期间，我能强行打断它吗？**
  - A: 完全可以。在运行 Agent 的终端里直接按下 `Ctrl+C`，或者点击编辑器内置 Agent 的“停止 (Stop)”按钮，你随时都可以夺回控制权并阻止它的暴走。

## 🧬 提示词解剖 (Why it works?)

1. **反阿谀奉承 (Anti-Sycophancy)：** 彻底杜绝毫无意义的情感交流，不仅大幅减少 Token 的无谓消耗，更是强行向 AI 注入了以事实和结果为唯一导向的极客思维模式。
2. **自愈循环体系 (The Healing Loop)：** 将资深开发者“编码 -> 报错 -> 溯源 -> 修复 -> 验证”的闭环大脑回路进行了系统级封装，把任务委派（Delegation）的自动化层级拉升到了一个全新的维度。
3. **防死锁硬刹车 (Timeout Brake)：** 设定了极为严格的 3 次重试红线，这是防止 AI 陷入幻觉（Hallucination）、无休止地乱改代码甚至引发生产事故的终极保险。
