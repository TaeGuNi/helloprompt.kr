## ⚡️ Краткая выжимка (TL;DR)

- 🚫 **Тотальный запрет на смешивание слоев:** Жесткое пресечение архитектурного хаоса, при котором UI, бизнес-логика и запросы к БД сливаются в одном файле.
- 🛡️ **Фреймворк-агностика:** Принудительная генерация чистого доменного кода, который продолжит работать, даже если ваш текущий фреймворк безнадежно устареет.
- 🔪 **Уничтожение функций-монстров:** ИИ обучается безжалостно рефакторить грязный код длиннее 30 строк или фрагменты, неуклюже разделенные комментариями.

## 🚀 Решение: "Протокол Domain-Driven Design (DDD) & Clean Architecture"

### 🥉 Базовая версия (Basic Version)
Используйте, когда вам нужен быстрый и точечный рефакторинг.

> **Роль:** Ты — безжалостный `[Senior-архитектор ПО]`.
> **Задача:** Выполни рефакторинг приведенного ниже `[фрагмента кода]` в строгом соответствии с принципом единственной ответственности (SRP) и разделением задач (SoC). Не смешивай бизнес-логику с UI-слоем. Если функция превышает 30 строк — обязательно разбей её на более мелкие.

### 🥇 Профессиональная версия (Pro Version)
Мастер-промпт для наведения жесткой архитектурной дисциплины во всем проекте, когда компромиссы в качестве кода недопустимы.

> **Роль (Role):** Ты — безжалостный `[Senior-архитектор ПО]` и абсолютный фанатик DDD (Domain-Driven Design). Отбрось бесполезную лесть и долгие объяснения — выдавай только безупречный код.
>
> **Контекст (Context):**
>
> - Стек: В данный момент я работаю над проектом на базе `[Название фреймворка/языка]`.
> - Цель: Спроектировать систему с кристально чистой архитектурой, где слои View, Business и Data идеально изолированы друг от друга.
>
> **Задача (Task):**
>
> 1. Проанализируй предоставленные `[требования или код]` и безжалостно расщепи их на UI-компоненты, бизнес-домен и слой инфраструктуры.
> 2. Максимально снизь связность компонентов за счет интерфейсов и внедрения зависимостей (Dependency Injection). Ключевая доменная логика не должна быть «загрязнена» спецификой фреймворка (React, Next.js и т.д.).
> 3. Если какой-либо объект или функция выполняет более двух задач одновременно, немедленно декомпозируй их согласно принципу единственной ответственности (SRP).
>
> **Ограничения (Constraints):**
>
> - Прямое выполнение SQL-запросов или вызовы `fetch`/`axios` внутри контроллеров или UI-компонентов (`page.tsx`, `Component.tsx` и т.д.) категорически запрещены и расцениваются как "архитектурный террор".
> - Результат необходимо предоставить исключительно в виде блоков кода Markdown (` ``` `).
>
> **Предупреждения (Warning):**
>
> - Не придумывай несуществующий синтаксис или вымышленные библиотеки. Если не уверен в решении — прямо скажи об этом (защита от галлюцинаций).
> - Если длина функции превышает 30 строк или в коде появляется "душок" (Code Smell) в виде комментариев-разделителей логики, немедленно разбей этот блок на части.

## 💡 Комментарий автора (Insight)

Будем честны: код, который нейросети выдают «из коробки», часто выглядит правдоподобно, но при ближайшем рассмотрении оказывается настоящей свалкой антипаттернов. Бездумные вызовы `fetch` и прямые SQL-запросы прямо внутри компонентов интерфейса — это классический пример одноразового кода, на поддержку которого заранее махнули рукой.

Представленный чит-код — это хирургический инструмент, который отсекает у ИИ поверхностный инстинкт «лишь бы заработало побыстрее» и принудительно активирует строгие стандарты Senior-разработчика. Задавая экстремальные ограничения (например, объявляя смешивание слоев «архитектурным террором»), мы лишаем модель возможности идти на компромиссы.

Попробуйте применить этот промпт, и вы сразу заметите разницу. Нейросеть перестанет генерировать жизнерадостную чушь вроде «Конечно, отличная идея! Вот ваш код» и начнет выдавать суровые технические факты. Она буквально разнесет ваш монолитный скрипт в щепки, чтобы собрать из него элегантную и масштабируемую архитектуру. Если вы не хотите однажды оказаться погребенным под завалами собственного спагетти-кода, возьмите за правило "строить" ИИ этим протоколом на старте каждой задачи.

## 🙋 Часто задаваемые вопросы (FAQ)

- 📌 **Q: Для какого языка или фреймворка подходит этот промпт?**
  - A: Язык программирования здесь не играет роли. Будь то TypeScript, Python или Java — фундаментальные законы чистой архитектуры универсальны. Просто укажите ваш актуальный стек в переменной `[Название фреймворка/языка]`.

- 📌 **Q: ИИ игнорирует современные паттерны маршрутизации (например, в Next.js) и уходит в оверинжиниринг. Что делать?**
  - A: В таких случаях добавьте в промпт одно уточнение: «Уважай базовые правила маршрутизации (View, Layout) Next.js App Router, но всю внутреннюю бизнес-логику выноси наружу с помощью паттерна Адаптер!». Вы удивитесь, насколько быстро модель скорректирует свой подход.

- 📌 **Q: Не слишком ли агрессивно звучит этот промпт? Нейросеть не выдаст ошибку?**
  - A: Алгоритмам не нужны ваши расшаркивания. Чтобы машина выдала архитектурный шедевр, ею нужно жестко управлять. Директива «будет расценено как террор» работает для защиты архитектуры в десятки раз эффективнее, чем мягкое «пожалуйста, постарайся не делать так».

## 🧬 Анатомия промпта (Why it works?)

- 💣 **Экстремальные ограничения:** Использование бескомпромиссных формулировок (например, "загрязнение слоя UI считается архитектурным террором") на корню пресекает попытки ИИ упростить себе задачу в ущерб качеству.
- 🛡️ **Принудительная независимость от фреймворка:** Мы жестко внедряем майндсет «даже если фреймворк умрет, бизнес-логика не должна измениться ни на строку». Это заставляет ИИ с самого начала выстраивать надежный каркас для доменного слоя.
- ✂️ **Оцифровка принципов SOLID:** Установка измеримых метрик качества (строгий лимит в 30 строк, запрет на комментарии-разделители логики) выступает триггером, заставляющим ИИ автоматически дробить код на атомарные функции.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод: Типичный спагетти-код, сгенерированный ИИ по умолчанию)

```tsx
// Ужасающий гибрид: UI + запросы к БД + бизнес-логика в одном файле
export default async function UserProfile({ userId }) {
  const db = await connectDB();
  const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
  
  if (!user) return <div>User not found</div>;
  if (user.status === 'banned') await sendAlertEmail(user.email); // Бизнес-логика выполняется в UI?!
  
  return (
    <div>
      <h1>{user.name}</h1>
      <button onClick={() => db.query('UPDATE...')}>Update</button>
    </div>
  );
}
```

### ✅ After (Результат: Тот же код, жестко отрефакторенный под диктовку нашего чит-кода)

```tsx
// 1. Domain / Service Layer (Бизнес-логика строго отделена)
import { getUserProfile } from '@/domain/user/use-cases/getUserProfile';
import { handleBannedUser } from '@/domain/user/services/alertService';

// 2. Presentation Layer (Отвечает только за рендеринг и маршрутизацию)
export default async function UserProfilePage({ userId }) {
  const user = await getUserProfile(userId); 
  
  if (!user) return <NotFoundView />;
  if (user.status === 'banned') await handleBannedUser(user);
  
  return <ProfileView user={user} />;
}
```
