## 🎨 デザインシステム構築：デザイナーと開発者が争わない方法

- **🎯 おすすめの対象:** 「ボタンの色がデザイン案と違います」と指摘されがちな開発者、「私のデザインが台無しにされた」と絶望しているデザイナー
- **⏱️ 所要時間:** 10分 → 1分（初期トークンの設計およびコードへの自動変換）
- **🤖 おすすめのモデル:** ChatGPT-4o, Claude 3.5 Sonnet（アーキテクチャ設計およびコード生成）

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「プライマリーカラーは `#0055FF` のはずですが、なぜここでは `#0050FF` になっているんですか？」_

コードベースのあちこちに散らばるハードコーディングされたカラーや余白の値は、保守作業を地獄へと突き落とす元凶です。今すぐ**デザイントークン（Design Token）**を導入しましょう。カラー、フォント、余白、シャドウといったすべての視覚的要素を「意味のある変数」として定義することは、チーム全体で共有できる最も確実な約束となります。トークンベースのシステムさえ構築できれば、デザイナーがFigma上で値を修正した瞬間に、開発者のコードも自動的に同期されるという魔法のような体験が待っています。

---

## ⚡️ 3行まとめ (TL;DR)

1. **デザインのトークン化:** `#0055FF` のような絶対的なHex値の代わりに、意味を持たせた `primary-500` という共通言語を使用します。
2. **Figma Variablesの適用:** FigmaのVariables（変数）やStyles機能を活用し、視覚的属性を体系的なトークンとして定義します。
3. **コードの同期:** 抽出したトークンデータをもとに、`tailwind.config.ts` やグローバルCSS変数へと自動変換し、プロジェクトのコードへ即座に反映させます。

---

## 🚀 解決策: "Design Token Generator Prompt"

### 🥉 Basic Version (基本型)

初期のカラーパレットを構成する際や、変数名の規則に迷ったときに活用できるプロンプトです。

> **リクエスト (Task):**
> 私は今、新規プロダクトのデザインシステム向けにカラーパレットを企画しています。
> `[Primary（青）]`、`[Secondary（グレー）]`、`[Error（赤）]` の各カラーを100から900までの9段階スケールで分割したいと考えています。
> これらのカラーをフロントエンドのコードへ直接適用できるよう、直感的かつセマンティック（意味論的）な名称（例: `text-primary`、`bg-surface-default`）と共に、各スケールのHexコードを表形式で提案してください。

### 🥇 Pro Version (専門家型)

Figmaから抽出したトークンのJSONデータを、実際のフロントエンド環境に合わせて完璧なコードへと変換する際に使用します。

> **役割 (Role):** あなたは10年の経験を持つシニアデザインオプス（Design Ops）エンジニアであり、フロントエンドアーキテクトです。
>
> **状況 (Context):**
>
> - 背景: デザイナーがFigmaでデザイントークンを定義し、JSON形式で抽出しました。
> - 目標: このJSONデータを、フロントエンドプロジェクトに即時適用可能なコードへ完璧に変換する必要があります。
>
> **入力データ:**
>
> `[ここにFigmaから抽出したトークンのJSONデータを貼り付けてください]`
>
> **リクエスト (Task):**
>
> 1. 入力データを分析し、最新の**Tailwind CSS設定ファイル（`tailwind.config.ts`）**の形式へ変換してください。
> 2. 特定のCSSフレームワークに依存しないよう、**CSS Variables（`:root { --color-blue-500: ... }`）**のコードも併せて作成してください。
> 3. システムのテーマに合わせて切り替えられるよう、ダークモード（` @public/images/hooks/social-media-planner.jpg (prefers-color-scheme: dark)` または `.dark` クラス）への対応戦略とサンプルコードを追加してください。
>
> **制約事項 (Constraints):**
>
> - 出力形式はマークダウンのコードブロック（`ts`、`css`）のみで提供してください。
> - すべての変数名はケバブケース（kebab-case）を厳格に維持してください。
>
> **注意事項 (Warning):**
>
> - 提供されたJSONの構造以外に、任意のカラー値や段階を勝手に捏造しないでください。（ハルシネーションの防止）

## 💡 筆者コメント (Insight)

デザインシステムの構築は、壮大なマスタープランを立てて一挙に完成させるものではありません。初期段階からすべてのコンポーネントやタイポグラフィを完璧にトークン化しようとすると、高い確率で疲弊し、挫折してしまいます。
**「最も頻繁に、そして繰り返し使われる要素（例: プライマリーボタンの背景色、本文のテキストカラー）」**から順に一つずつ変数へ置き換えていく**漸進的導入（Incremental Adoption）**の戦略を強くお勧めします。
たった一つの小さなトークンが、デザイナーと開発者の間に生じる無駄なコミュニケーションコストを劇的に削減するための、最初の一歩となるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: Figmaでトークンを抽出するには、どのプラグインを使うべきですか？**
  - A: 実務においては**「Tokens Studio for Figma」**が事実上の標準（デファクトスタンダード）として広く利用されています。これを用いてJSONを抽出し、GitHubリポジトリと直接同期（Sync）させることが可能です。最近ではFigmaネイティブのVariables機能も非常に強力になっており、REST APIを通じた連携も増加傾向にあります。

- **Q: 2〜3人規模の小さなスタートアップでも、デザインシステムは必要ですか？**
  - A: チームメンバーが2人以上、あるいはプロジェクトの保守期間が3ヶ月を超えるのであれば、無条件に導入すべきです。「後で時間があるときに整理しよう」と後回しにした技術的負債は、最終的にプロジェクトの開発スピードを著しく低下させる時限爆弾となって跳ね返ってきます。未来の自分（Future Self）も、結局のところ「他人」なのです。

## 🧬 プロンプトの解剖 (Why it works?)

1. **セマンティック・ネーミングの強制:** 「明るい青色」の代わりに「主要なボタンの背景色（`bg-primary-default`）」といった意味ベースのネーミングを誘導します。これにより、後からブランドカラーが青から紫へと全面的に変更されたとしても、Hex値を1つ修正するだけでプロジェクト全体へ即座に反映させることができます。
2. **反復的なフォーマット変換の委任:** JSONフォーマットをTailwindの設定オブジェクトやCSS変数の構文に合わせて手作業でタイピングし直すのは、人間にとって非常に苦痛な作業です。AIは、こうしたパターンベースの単純なコード変換や構造化作業において圧倒的な正確さとスピードを誇り、ヒューマンエラー（Human Error）を根絶します。

## 📊 証明: Before & After

### ❌ Before (入力)

デザインが変更されるたびに、100を超えるファイルからHexコードを一つ一つ検索して置き換える地獄の作業が待っています。😱

```css
.button-primary {
  background-color: #3b82f6;
}

.text-highlight {
  color: #3b82f6;
}
```

### ✅ After (結果)

変数の値をたった1つ修正するだけで、プロダクト全体のUIが完璧にアップデートされます。🚀

```css
:root {
  --primary-500: #3b82f6;
  --bg-primary-default: var(--primary-500);
}

.button-primary {
  background-color: var(--bg-primary-default);
}
```

## 🎯 結論

デザインと開発の間を取り持つ通訳者は、感情を持った「人」ではなく、明確なルールに基づいた**「システム」**であるべきです。
ピクセルやカラーの値で不毛な争いをするのは、もう終わりにしましょう。これからは、トークンという共通言語で対話することを強くお勧めします。

**「開発者さん、ここは `gray-200` ではなく `surface-subtle` です。トークンの確認をお願いします。」** 🍷
