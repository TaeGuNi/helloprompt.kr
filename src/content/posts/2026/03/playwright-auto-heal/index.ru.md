# 📝 E2E-тесты падают из-за ошибок? Решите проблему за 1 минуту с помощью Auto-healing промпта для Playwright 🛠️


- **🎯 Рекомендуется для:** QA-инженеров, frontend-разработчиков, специалистов по автоматизации тестирования
- **⏱️ Затрачиваемое время:** 2 часа → сокращено до 1 минуты
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o
- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

_"E2E-тесты, которые вчера работали отлично, сегодня утром все упали только потому, что изменился один CSS-класс кнопки. Опять идти искать новые селекторы?"_
В frontend-разработке или QA поломка E2E (End-to-End) тестов из-за изменений UI — это повседневная рутина. Особенно при использовании таких инструментов, как Playwright или Cypress: стоит измениться структуре DOM или именам классов, как тестовый код безошибочно выдает ошибку. Постоянно находить упавший тест, открывать инструменты разработчика и копировать новый селектор — это колоссальная трата времени. Теперь давайте просто отдадим лог ошибки и измененную HTML-структуру ИИ, чтобы он сам выполнил «Auto-healing» (автоматическое восстановление).
---
## ⚡️ Краткое содержание (TL;DR)
1. Промпт, с помощью которого ИИ самостоятельно анализирует и восстанавливает селекторы Playwright, сломавшиеся из-за изменений UI.
2. Помимо простого исправления кода, вы получите стратегию написания устойчивых (Resilient) селекторов, невосприимчивых к изменениям.
3. Просто предоставьте лог ошибки и актуальный снимок DOM, и вы получите восстановленный код менее чем за 1 минуту.

## 🚀 Решение: "Промпт для самовосстановления (Auto-healing) Playwright"

### 🥉 Basic Version (Базовая версия)
Используйте, когда нужно быстро закинуть сломанный код и сообщение об ошибке, чтобы получить исправленный вариант.
> **Роль:** Ты `[Senior QA Engineer]`.
> **Задача:** На основе предоставленного лога ошибки Playwright и нового HTML исправь `[сломанный тестовый код]`.
### 🥇 Pro Version (Профессиональная версия)
Используйте, когда помимо простого исправления нужен рефакторинг в надежный код, который легко поддерживать.
> **Роль (Role):** Ты `[Senior SDET (Software Development Engineer in Test)]`, эксперт по автоматизации тестирования. Ты в совершенстве понимаешь лучшие практики (Best Practices) Playwright.
> 
> **Контекст (Context):**
> 
> - Фон: Из-за недавнего обновления UI существующие E2E-тесты Playwright начали падать. Основная причина — изменение селекторов (Selector).
> - Цель: Восстановить упавший тестовый код и улучшить его структуру селекторов, сделав её более устойчивой (Resilient) к будущим изменениям UI.
> 
> **Задача (Task):**
> 
> 1. Проанализируй предоставленный `[лог ошибки]` и `[измененную структуру HTML]`, чтобы выявить первопричину ошибки.
> 2. Исправь `[существующий тестовый код]`, чтобы тест снова проходил.
> 3. При исправлении отдавай приоритет атрибутам, ориентированным на пользователя (User-Facing атрибуты: текст, атрибуты ARIA, data-testid и т.д.), а не полагайся на CSS-классы или XPath.
> 4. Добавь в код краткие комментарии с объяснением, "почему это было исправлено именно так".
> 
> **Входные данные:**
> - Лог ошибки: `[Скопировать/вставить весь лог ошибки]`
> - Измененная структура HTML: `[Скопированный фрагмент HTML из инструментов разработчика]`
> - Существующий тестовый код: `[Фрагмент падающего кода Playwright]`
> 
> **Ограничения (Constraints):**
> 
> - Вывод должен быть предоставлен в виде полного блока кода, который можно сразу скопировать и вставить.
> - В первую очередь используй встроенные локаторы Playwright (например: `getByRole`, `getByText`, `getByTestId`).
> 
> **Предупреждение (Warning):**
> 
> - Если в предоставленном фрагменте HTML невозможно найти подходящий элемент, не пытайся угадывать. Просто ответь: "Из-за недостатка информации в HTML невозможно определить точный селектор. Пожалуйста, предоставьте дополнительную информацию о DOM."

## 💡 Комментарий автора (Insight)
Этот метод раскрывает весь свой потенциал, когда в рабочем процессе одновременно падает несколько десятков тестов. В прошлом при изменении одного селектора мне приходилось открывать 30 тестовых файлов и вручную искать их через `Ctrl + F`, чтобы исправить. Но если сохранить этот промпт во встроенном ИИ вашей IDE или в Claude, вам останется только закинуть лог ошибки и скопированную вкладку Elements из браузера, полностью делегировав нудную работу по извлечению селекторов. 
Особого внимания заслуживает ограничение "используй User-Facing атрибуты". По умолчанию ИИ склонен выбирать самый простой CSS-класс, но благодаря этому ограничению он пишет отличный код вроде `getByRole('button', { name: 'Submit' })`. Это оружие, которое приблизит конец вашего рабочего дня как минимум на час. Просто попробуйте!

## 🙋 Часто задаваемые вопросы (FAQ)
- **В: Что делать, если структура HTML слишком длинная?**
  - О: Нет необходимости копировать всю страницу целиком. Достаточно скопировать код до родительского тега элемента, вызвавшего ошибку (например, кнопки или формы).
- **В: Можно ли использовать этот промпт для Cypress или Selenium, а не только для Playwright?**
  - О: Да, просто замените слово «Playwright» в промпте на название используемого вами инструмента, и он сработает так же отлично.
- **В: Будет ли это работать в проектах без `data-testid`?**
  - О: Да, ИИ достаточно умен, чтобы найти альтернативы на основе доступности, такие как `getByRole` или `getByText`.

## 🧬 Разбор промпта (Why it works?)
1. **Задание персоны SDET:** Наделяя ИИ взглядом профессионального инженера по тестированию, а не обычного разработчика, мы побуждаем его генерировать не просто работающий код, а код, соответствующий Best Practice Playwright.
2. **Принудительное использование User-Facing локаторов:** На уровне промпта заложена ключевая философия Playwright, которая фундаментально снижает нестабильность (Flakiness) тестов, тем самым повышая их качество.
3. **Защита от галлюцинаций:** Чтобы предотвратить генерацию бессмысленного кода при нехватке информации (что только усложнило бы отладку), установлено явное ограничение: "если не знаешь — так и скажи".

```text
Лог ошибки:

  <button class="button--blue" aria-label="Submit Order">Order Now</button>
</div>
Существующий тестовый код:
await page.locator('.btn-primary.submit-btn').click();
```
### ✅ После (Результат)
```typescript
// Причина изменения: CSS-класс ('.btn-primary.submit-btn') был изменен, из-за чего тест упал.
// Улучшение: Вместо подверженного изменениям CSS-класса использованы атрибуты доступности (ARIA label) и Role, что делает селектор устойчивым к изменениям UI.
await page.getByRole('button', { name: 'Submit Order' }).click();
```
