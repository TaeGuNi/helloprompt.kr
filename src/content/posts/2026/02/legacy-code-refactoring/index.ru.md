## 🧹 Очистка легаси-кода: Рефакторинг и автоматизация тестов

- **🎯 Целевая аудитория:** Junior-разработчики, мейнтейнеры легаси-кода, код-ревьюеры
- **⏱️ Экономия времени:** 1 час на анализ → сокращено до 5 минут
- **🤖 Оптимальные модели:** Claude 3.5 Sonnet, GPT-4o (Claude настоятельно рекомендуется для сложной логики и длинного контекста)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Практическая польза:** ⭐⭐⭐⭐⭐

> _«Кто вообще написал этот ужасный код? ...А, стоп, это же я год назад»._

Глубоко вложенные конструкции `if-else`, переменные с загадочными названиями и полное отсутствие нормальной обработки исключений. Знакомо ли вам это парализующее чувство страха перед **легаси-кодом (Legacy Code)**, когда кажется, что малейшее изменение обрушит весь продакшен?

Больше не нужно бояться прикасаться к старому коду. Сейчас мы покажем, как превратить ИИ в вашего личного **Senior-разработчика для парного программирования (Pair Programmer)**. Вы освоите продвинутые техники промпт-инжиниринга, которые позволят элегантно распутать спагетти-код и моментально выстроить бронебойную защиту в виде автоматических тестов.

## ⚡️ Краткая выжимка (TL;DR)

1. **Сначала требуйте объяснений:** Не просите ИИ сразу переписывать код. Заставьте его сначала проанализировать поток выполнения и досконально понять изначальный замысел бизнес-логики.
2. **Внедряйте принципы Clean Code:** Проводите безопасный рефакторинг архитектуры и нейминга, строго опираясь на принципы SOLID и максимальную читаемость.
3. **Стройте защиту через тесты:** Автоматически генерируйте юнит-тесты (Unit Tests), включая неочевидные пограничные случаи (Edge Cases), чтобы гарантировать 100% идентичность работы кода до и после изменений.

## 🚀 Решение: "Senior Dev Simulator" (Симулятор Senior-разработчика)

### 🥇 Pro Version (Профессиональная версия)

Если просто приказать ИИ «отрефактори это», вы почти наверняка сломаете существующий функционал. В этом промпте применяется мощный паттерн **Chain-of-Thought (Цепочка рассуждений)**, заставляющий модель мыслить строгими логическими шагами.

> **Роль (Role):** Ты `[Senior Backend-разработчик с 10-летним опытом]`, работавший в ведущих BigTech-компаниях (Google, Meta). Ты строгий приверженец философии чистого кода (Clean Code) и неукоснительно соблюдаешь принципы TDD (Test-Driven Development).
>
> **Контекст (Context):**
> Мне только что передали на поддержку ужасный легаси-код, написанный на `[введите язык, например: Python/Java]`. У него отвратительная читаемость, и его практически невозможно безопасно поддерживать.
>
> **Задача (Task):**
> Выполни следующие шаги строго по порядку и с максимальной глубиной анализа:
>
> **Шаг 1. Анализ и диагностика кода:** Опиши ровно одним предложением, какую бизнес-задачу решает этот код, и выдели 3 критические проблемы (Code Smells).
> **Шаг 2. Безопасный рефакторинг:** Сохрани изначальную бизнес-логику и побочные эффекты (Side Effects) на 100%. Кардинально улучши читаемость за счет разделения функций согласно принципу единственной ответственности (SRP) и переименования переменных/методов. Для сложной логики обязательно добавь Docstring и комментарии.
> **Шаг 3. Защитная проверка (Тесты):** Напиши юнит-тесты (Unit Tests), которые на 100% гарантируют, что твой отрефакторенный код работает абсолютно так же, как оригинал. Тесты должны покрывать не только успешные сценарии, но и как минимум 2 пограничных случая (Edge Cases).
>

>
>
> [Вставьте сюда легаси-код для анализа]

## 💡 Комментарий автора (Insight)

Истинная мощь этого промпта кроется не просто в красивом форматировании, а в **предварительном выстраивании «сети безопасности» с помощью автоматических тестов**. Я лично применил этот подход в «боевых» условиях, когда мне в наследство достался монструозный модуль обработки платежей на Java пятилетней давности.

Что меня действительно поразило — ИИ самостоятельно выявил через тесты скрытые пограничные случаи, которые я изначально не заметил (например, неочевидное ветвление логики при нулевой сумме транзакции).

**🔥 Практический совет:**
Если вы скормите ИИ простыню из сотен строк кода за один раз, он с высокой долей вероятности словит «галлюцинацию» (Hallucination) или потеряет контекст. Дробите код и ставьте задачи **на уровне отдельных методов (не более 100 строк)**. Глубина анализа и надежность сгенерированных тестов в таком случае возрастают многократно.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Какая ИИ-модель лучше всего справляется со сложной логикой кода?**
  - О: По состоянию на 2026 год абсолютными лидерами в написании кода и рефакторинге остаются **Claude 3.5 Sonnet** и **GPT-4o**. При этом Claude 3.5 Sonnet демонстрирует заметное превосходство в удержании сложного контекста и генерации максимально чистого, человекочитаемого кода.

- **В: Безопасно ли передавать ИИ закрытый корпоративный код?**
  - О: **Категорически нет.** API-ключи, внутренние IP-адреса, реальные данные клиентов или проприетарные алгоритмы безопасности необходимо `[ЗАСЕКРЕТИТЬ]` (маскировать) или заменить на заглушки (Dummy Data) перед отправкой. В идеале — используйте исключительно Enterprise-версии ИИ с гарантией конфиденциальности данных.

- **В: Что делать, если отрефакторенный код выдает ошибки компиляции?**
  - О: Скорее всего, ИИ упустил внешнюю зависимость (Dependency). Скопируйте полный лог ошибки вместе с проблемным участком старого кода и дайте прямой фидбек: _«В этой строке падает ошибка. Похоже, ты упустил [определенное условие] из оригинальной логики. Пожалуйста, восстанови этот фрагмент»._

## 🧬 Разбор промпта (Why it works?)

1. **Сила конкретной роли (Role):** Задавая жесткие рамки «Senior с 10-летним опытом, фанатик Clean Code», мы искусственно завышаем стандарты: модель генерирует не просто «работающий» скрипт, а промышленный код с высочайшей степенью поддерживаемости.
2. **Жесткая структура Chain-of-Thought:** Вместо расплывчатого «сделай лучше», мы принуждаем ИИ пройти 3 логических этапа: **[Анализ → Рефакторинг → Проверка]**. Это фундаментальный механизм, не позволяющий нейросети сломать исходный замысел в погоне за красотой.
3. **Обязательные Edge Cases (Constraints):** Прямое требование написать тесты для неочевидных сценариев гарантирует предсказуемость кода, который можно без страха деплоить в продакшен.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод: Кошмарный легаси-код)

```python
def calc_trx(a, t, f=False):
    # a: amount, t: type
    if t == 1:
        if f: return a * 1.1 + 500
        else: return a * 1.1
    elif t == 2:
        return a - (a * 0.05)
    else: return 0 # ошибка
```

### ✅ После (Результат: Чистый код и тесты от Senior-разработчика)

```python
from enum import Enum
import unittest

class TransactionType(Enum):
    DEPOSIT = 1
    WITHDRAWAL = 2

def calculate_transaction_amount(amount: float, transaction_type: TransactionType, apply_fee: bool = False) -> float:
    """
    Рассчитывает итоговую сумму с учетом налогов и комиссий в зависимости от типа транзакции (внесение/снятие).
    """
    if amount < 0:
        raise ValueError("Сумма транзакции должна быть больше 0.")

    if transaction_type == TransactionType.DEPOSIT:
        taxed_amount = amount * 1.1
        return taxed_amount + 500 if apply_fee else taxed_amount

    elif transaction_type == TransactionType.WITHDRAWAL:
        return amount * 0.95

    return 0.0

# ----------------- Unit Tests ----------------- #
class TestCalculateTransaction(unittest.TestCase):
    def test_deposit_with_fee(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.DEPOSIT, True), 1600.0)

    def test_withdrawal_amount(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.WITHDRAWAL), 950.0)

    def test_negative_amount_raises_error(self):
        with self.assertRaises(ValueError):
            calculate_transaction_amount(-500, TransactionType.DEPOSIT)
```
