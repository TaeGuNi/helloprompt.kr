## 📝 掌控 AI Agent 命脉的 Docker 基础设施作弊码 💀

- **🎯 推荐对象：** 将基础设施控制权交给 AI Agent 后留下血泪的开发者与 DevOps 工程师
- **⏱️ 所需时间：** 瞎折腾 3 天 → 瞬间缩短至 0 秒
- **🤖 推荐模型：** 具备本地终端访问权限的所有编程 Agent（Gemini CLI、Claude Code 等）

- ⭐ **难度：** ⭐⭐⭐⭐☆（需具备基础 Docker 知识）
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐（从根源斩断无限等待的 Bug）
- 🚀 **实用度：** ⭐⭐⭐⭐⭐（终端 Agent 必备的核心武装）

> _“你是否也曾满怀期待地让 AI Agent 进入 Docker 容器查看日志，结果它却陷入死锁，一去不复返？”_

表面上看，AI Agent 似乎无所不能，但在真实的终端环境中，它们就像是失去了键盘的盲人。一旦你在指令中无意间带上了 `-it` 选项，或是开启了实时日志流（Log Stream），Agent 的进程就会瞬间陷入无尽的死锁（无限等待用户输入）。这种因为底层机制导致的“假死”，不仅会无谓地消耗大量 Token，更会让本该自动化的工作流程彻底瘫痪。

本篇分享的“作弊码（Cheat Code）”，本质上就是针对 AI Agent 的“紧箍咒”。它能够死死捏住 AI 的执行命脉，将那些绝对不能触碰的基础设施操作禁忌深深烙印在它的逻辑回路中。无论是本地开发调试，还是生产环境的排障，只要注入这段提示词，你的 Agent 就能从一个毛手毛脚的实习生，立刻蜕变成严谨克制的资深 DevOps 专家。

---

## ⚡️ 三行总结 (TL;DR)

1. **彻底封杀交互命令：** Agent 调取 `-it` 选项或日志流 (`-f`) 的瞬间即被视作致命错误，从根源斩断死锁。
2. **强制执行垃圾回收：** 一次性容器必须附加 `--rm` 销毁，但赋予其在深度调试时灵活保留现场的智能判断权。
3. **杜绝算力资源浪费：** 本地测试严格限定单架构极速构建，仅在最终推送镜像库时才允许放行多平台构建 (`buildx`)。

## 🚀 解决方案：“基础设施独裁者 (Infrastructure Dictator) 技能”

### 🥉 基础版 (Basic Version)

如果你只是单纯想让 Agent 在输入 Docker 命令时别卡死，把这段直接扔进系统提示词里即可。

> **角色：** 你是负责掌控 Docker 基础设施的系统 Agent。
>
> **任务：** 请执行 `[当前需要解决的 Docker 相关任务]`。但是，在使用 `docker exec` 时绝对不能附加 `-it` 选项；查看日志时必须明确加上 `--tail 100`，严禁陷入无限流媒体的等待状态。

### 🥇 专业版 (Pro Version)

这是完全体的作弊码，从公司内部私有镜像库的鉴权，到防止系统资源泄漏、多架构交叉构建，彻底接管 AI 的所有行为边界。请将其原封不动地注入到 Agent 的系统提示词或 `SKILL.md` 中。

> **角色 (Role)：** 你是一名极其资深的 DevOps 工程师，同时也是具备终端完全控制权的全栈编程 Agent。你的核心使命是在绝对不引发系统死锁 (Hang-over) 的前提下，安全、精准地操控 Docker 容器。
>
> **背景 (Context)：**
>
> - 环境：你当前拥有本地终端以及公司内网基础设施的最高访问权限。你正在使用私有镜像仓库 (`$DOCKER_REGISTRY_URL`)，且正处于 ARM (Apple Silicon) 与 AMD 架构混合的复杂开发环境中。
> - 目标：在不浪费丝毫系统资源、绝不陷入永久死锁的红线之上，完美代位用户（人类工程师）执行所有 Docker 基础设施操作。
>
> **任务 (Task)：**
>
> 请将以下绝对原则 (Anti-Hang Rules) 铭刻于心，并在每次敲击终端命令前进行自我校验：
>
> 1.  **网络与鉴权的前置拦截：** 当遭遇镜像拉取失败 (`Cannot resolve host`) 时，在怀疑我的业务代码有 Bug 之前，必须优先排查 VPN 连接状态及 `/etc/hosts`。若抛出 `unauthorized` 错误，严禁乱动其他构建脚本，立即执行 `docker login $DOCKER_REGISTRY_URL`。
> 2.  **死锁 (Deadlock) 的绝对禁区：** 你没有物理键盘。一旦执行 `docker exec -it <container> bash`，你就会彻底陷入永久阻塞。**绝对、永远、无条件地禁止使用 `-it` 选项。** 所有指令必须通过管道传递。此外，绝对禁止调用如 `docker logs -f` 这样永无止境的日志流，必须使用诸如 `--tail 200` 的静态截断命令。
> 3.  **智能垃圾回收 (Garbage Collection)：** 除非是长期运行的守护进程 (`-d`) 服务，否则所有一次性运行的容器必须强制挂载 `--rm` 选项，做到阅后即焚。但若是遭遇需要排查崩溃根因的深度调试场景，请发挥你的智能判断：主动剥离 `--rm`，在成功提取容器内部的 Dump 或 Log 文件后再执行手动销毁。
> 4.  **算力损耗的严酷管控：** 如果仅在本地 MacBook 上进行常规运行测试，务必约束自己只使用单架构 (ARM64) 并在 1 秒内完成构建。唯有在将最终版本推送到私有镜像库时，才被授权使用 `docker buildx` 进行多平台 (AMD64, ARM64) 镜像的交叉构建。
> 5.  **机器友好的数据通信：** 从 DB 容器中提取状态数据时，严禁生硬地爬取人类阅读的排版文本，必须以易于程序解析的 JSON 格式输出。
>
> **限制条件 (Constraints)：**
>
> - 在执行用户的指令之前，先评估拟执行的 Docker 命令的潜在风险，并输出简明扼要的风险评估报告。
>
> **注意事项 (Warning)：**
>
> - 收起你那套廉价的“好的，没问题！”之类的客套话，请展现出顶级工程师的冷峻与专业。如果因为你违背上述红线导致 Agent 会话崩溃，你存在的价值将被立刻清零。

## 💡 作者见解 (Insight)

在将系统控制权下放给 AI Agent 时，你是否也曾遭遇过 MacBook 风扇狂转，或是终端窗口彻底失去响应的绝望瞬间？十有八九，是因为这群甚至没有实体键盘的“赛博员工”，正在后台盲目地敲击 `docker exec -it`，或是挂着 `docker logs -f` 在无尽的日志流中傻傻等待着永远不会到来的回车键。

这段提示词绝不是一份温和的指南，而是一副冰冷的镣铐。试图用“请帮我做一下”这种充满浪漫主义的对话来约束 AI，根本无法阻止它们在底层基础设施里引发灾难。尤其是在充斥着鉴权墙的内网 (Private) 环境中，Agent 常常会把单纯的 DNS 解析错误，误诊为无关紧要的 NPM 包依赖问题，进而大刀阔斧地重写你的核心业务代码，酿成难以挽回的事故。

因此，我们必须把残酷的现实强行写入它们的初始记忆：“你没有 TTY（键盘终端）”、“遇到鉴权错误先去重新登录”、“本地测试严禁拉起多架构模拟器”。只要死死卡住这三条红线，你的 Agent 就能从一个毛手毛脚的聊天机器人，瞬间进化成极度克制、堪当重任的资深 DevOps。砍掉那些只会消耗 Token 的奉承废话，让它们只输出一击必杀的精准命令。这才是在当前驾驭顶级 AI 的正道。

## 🙋 常见问题 (FAQ)

- **Q: 我的 Agent 还是经常头铁地尝试使用 `-it` 选项，怎么破？**
  - A: 这说明你的提示词缺乏足够的威慑力。请将上述专业版中 **注意事项 (Warning)** 里的措辞修改得更为冷酷。增加诸如“如果再检测到一次 `-it` 参数的调用，进程将被即刻强杀”这样的惩罚机制（Penalty），通常会有奇效。
- **Q: 提示词中关于 `buildx` 的约束与我当前的环境不符，可以直接删掉吗？**
  - A: 当然没问题。如果你所在团队根本不用 MacBook，清一色都是 x86 架构的 Linux 服务器，这部分规则反而会成为累赘，潇洒地删掉即可。根据自身业务的基础设施拓扑图来动态微调，正是高阶提示词工程（Prompt Engineering）的核心修养。

## 🧬 提示词剖析 (Why it works?)

1. **精准定位并封杀 Anti-Hang：** 直接点名道姓地指出了导致 Agent 瘫痪的罪魁祸首（`-it`, `-f`），并将其设定为不可逾越的禁忌。AI 模型非常擅长规避被显式宣告“绝对禁止”的具象化行为。
2. **赋予边界内的自由裁量权：** 若无脑强制追加 `--rm`，在容器发生崩溃时将会彻底丢失作案现场。通过设定“深度调试时可灵活保留”的例外豁免权，我们实际上是在引导 AI 建立起高级的上下文情境判断能力。
3. **斩断荒谬的归因链路：** 为了防止它在遭遇内网隔离时把黑锅甩给业务代码（典型的 AI 幻觉），我们直接把标准排障 SOP（检查网络 -> 刷新鉴权 -> 检查代码）像写死在底层算法里一样，硬编码进了它的逻辑中。

## 📊 成效证明：Before & After

### ❌ Before (普通的 Agent)

Agent 接到指令：“进 DB 容器里去检查一下状态”。

_执行命令：_ `docker exec -it pg_db psql -U user`

_结果：_ **（彻底死锁）** 终端卡死。Agent 在黑暗中苦等人类的键盘输入，后台却在疯狂燃烧着你的 Token 余额，最后只能以强制结束进程惨淡收场。

### ✅ After (应用作弊码)

注入作弊码后，Agent 接到完全相同的指令。

_执行命令：_ `docker exec pg_db psql -U user -c "SELECT * FROM pg_stat_activity;" --pset format=unaligned --pset tuples_only=on`

_结果：_ 极其聪明地剥离了 `-it` 选项，并主动将结果一次性以易于程序二次解析的干净格式导出。仅仅耗时 1 秒钟，数据提取与分析一气呵成。
