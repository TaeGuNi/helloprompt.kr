## 🚀 解决方案："Everything Claude Code" 提示词

### 🥉 Basic Version (基础版)

当你需要快速完成基础架构设计时，请使用此版本，2 分钟内即可“雇佣”一位资深 Agent。

> **1. 安装插件 (以 Claude Code 为例):**
> `/plugin marketplace add affaan-m/everything-claude-code`
> `/plugin install everything-claude-code@everything-claude-code`
> 
> **2. 执行架构规划提示词:**
> `/plan "请使用 Next.js 14 App Router 为我搭建一个响应式的博客文章列表页。技术栈需包含 Tailwind CSS 和 shadcn/ui。"`

### 🥇 Pro Version (专业版)

当你希望在项目中强制执行顶尖开发者的严苛工作流（如 TDD 驱动或极致的代码洁癖）时，请祭出这套高阶版本。

> **角色 (Role)：** 你是一位极其严格、始终恪守 TDD（测试驱动开发）原则的 `[资深 TypeScript 开发者]`。
>
> **背景 (Context)：**
>
> - 我们目前正在全面遵循 `[everything-claude-code]` 的系统规则。
> - 核心目标：我们需要实现一个高可靠性的 `[用户身份验证模块]`。
>
> **任务 (Task)：**
>
> 1. 请在终端中激活 `/tdd` 指令。
> 2. 在编写任何业务逻辑代码之前，**必须**优先编写包含各种边界与异常情况的测试用例。
> 3. 若当前代码的测试覆盖率未达到 80%，坚决不允许进入下一开发阶段。
> 4. 若有 `[额外补充需求]`，请将其完美融合到代码架构中。
>
> **约束条件 (Constraints)：**
>
> - 所有的变量、函数与接口都必须添加明确的 Type Hint（类型注解）。
> - 全局**严禁**使用 `any` 类型。
>
> **注意事项 (Warning)：**
>
> - 绝对不要编写任何破坏现有系统架构或违背既定约定的代码。如果在实现过程中存在任何不确定性，必须先向我提问确认。（防止幻觉与越权操作）

## 💡 作者洞察 (Insight)

将这套配置真正落地到企业级业务后，最让我感到灵魂震撼的是：**“AI 竟然学会了向人类说‘不’，并且开始主动拒绝那些含糊其辞、企图‘偷懒’的提示词了。”**

过去，当我们随口丢出一句“帮我加个登录页面”时，AI 通常会像个急于邀功的新人，立刻吐出一大堆缺乏错误兜底、没有任何业务边界考虑、甚至逻辑漏洞百出的“面条代码” (Spaghetti Code)。你看着那一坨代码，不仅毫无美感，还会引发后续无穷无尽的 Bug 修复噩梦。但是，在彻底引入这套配置后，整个开发范式被彻底颠覆了。当你敲下 `/plan` 指令时，AI 会瞬间反客为主，开启严密审慎的“夺命连环问”模式：_“请问我们采用哪种认证机制（OAuth 2.0 还是 JWT）？如果 Token 过期，前端的刷新策略是什么？发生网络异常时的 UI 状态该如何兜底？”_ 这种感觉，就像你身边坐着一位拿着高薪、**“没有明确需求边界和架构设计就绝对拒绝敲击键盘”的严苛架构师**，它在用极其专业的方式，时刻为你项目的长期可维护性保驾护航。

这种**“约束即自由” (Freedom through Constraints)** 的工程思想，正是高阶提示词的灵魂所在。我们在 Pro 版本的提示词中，故意设置了极高的壁垒：**“严禁使用 `any` 类型”**、**“覆盖率未达 80% 坚决不进入下一步”**。这些看似繁琐的约束条件，实则是为了对抗大语言模型 (LLM) 固有的“幻觉 (Hallucination)”与“路径依赖”。当 AI 被强制置于 TDD 的轨道上时，它就必须先思考业务逻辑的边界与异常态，这就从根本上阻断了它“胡编乱造”的可能性。

而在整个系统中，最令人拍案叫绝、甚至感到一丝毛骨悚然的，当属神级的 **`/evolve`** 指令。传统的 AI 工具往往患有严重的“上下文失忆症”，你今天纠正了它的代码规范，明天它又会抛之脑后。但在这里，随着项目代码量的日益庞大，AI 会像一位敏锐的人类观察者，主动分析并总结你们团队特有的暗默知识与编码习惯（例如：“我们在定义接口时从不加 'I' 前缀”）。随后，它会自动将这些隐式习惯提炼、固化为 `.rules` 规则库中的永久法则。对于那些苦于代码风格朝令夕改、规范难以落地的技术团队，或是极度渴望拥有大师级代码审查体验的独立开发者而言，这套具备自我进化能力的工作流，绝对是降维打击般的存在。通过这种深度的约束与进化机制，我们不再是代码的搬运工，而是真正成为了掌控全局的系统指挥官。

## 🙋 常见问题 (FAQ)

- **Q: 这套神级配置只能局限在 Claude Code CLI 环境下使用吗？**
  - A: 绝非如此！它的架构设计极为灵活，你可以将其无缝“平替”并注入到 Cursor 编辑器的 `.cursorrules` 文件中，甚至在 OpenCode 或任何支持自定义系统级指令的 IDE 环境下，都能完美发挥其强大的架构师威力。

- **Q: 将其直接用于公司内部的核心商业保密项目，是否存在安全隐患？**
  - A: 该规则 (Rule) 文件本身完全开源，且仅持久化存储在你的本地开发环境中，因此配置框架本身是 100% 安全的。但务必切记：向 LLM 模型发送核心业务代码这一动作，必须严格遵守贵公司的安全合规底线（例如，务必确认你们使用的是不将数据用于模型训练的企业级 Enterprise 隐私套餐）。

- **Q: AI 老是开启“夺命连环问”，还被强制要求先写测试，对于追求快的新手而言是不是太繁琐、太拖效率了？**
  - A: 引入初期确实会经历短暂的“阵痛期”，你甚至会觉得被 AI 严厉地绑住了手脚。但请务必相信我，正是这种看似“繁琐”的前置校验与测试先行流程，才能真正让你彻底告别未来为了偿还堆积如山的技术债务 (Technical Debt) 而被迫连续熬夜重构的惨痛经历。慢，即是快。

## 🧬 提示词解剖 (Why it works?)

1. **强制工作流约束 (Forced Workflow) 的降维打击：** 彻底摒弃了过去那种毫无章法、随性而为的自由对话模式。通过强行植入 `/plan`, `/tdd` 等斜杠指令，强制引导 AI 必须严格遵循正规的软件工程生命周期（需求深度拆解 → 测试用例先行 → 核心业务代码实现），从源头上掐断了劣质代码的产生。
2. **多重人格博弈 (Persona Separation) 的交叉验证：** 巧妙地在系统内部将“编码者 (Developer)”、“审查者 (Reviewer)”和“规划者 (Planner)”的视域进行物理级剥离。这迫使 AI 在输出最终结果前，必须在内部进行激烈的逻辑互搏与交叉验证，从而以指数级提升了交付代码的健壮性与抗风险能力。
3. **经验沉淀与系统级自我进化 (Instincts)：** 强势打破了传统大模型工具“阅后即焚”的上下文失忆绝症。AI 在长期协助开发的过程中，能以极高的敏锐度捕捉使用者的重构意图与偏好，并将其转化为硬性规范沉淀至 `.rules` 规则库中，打造出真正意义上“越用越趁手、越用越懂你”的专属数字分身。
