# 📝 没有同事也没关系，AI 高级开发者的毒舌代码审查

- **🎯 推荐对象:** 独立开发者、初级程序员、外包从业者
- **⏱️ 所需时间:** 5 分钟 → 缩短至 1 分钟
- **🤖 推荐模型:** 所有大语言模型 (ChatGPT, Claude, Gemini 等)

- ⭐ **难度:** ⭐⭐☆☆☆
- ⚡️ **有效性:** ⭐⭐⭐⭐⭐
- 🚀 **实用性:** ⭐⭐⭐⭐☆

> _“这段代码是最佳实践吗？要是能有一位大神帮我 Review 一下就好了……”_

独自写代码，视野难免会变得狭窄。
如果抱着“能跑就行”的心态得过且过，技术水平是很难获得突破性增长的。
想象一下，如果一位来自世界级科技巨头（如 Google 或 Amazon）的高级工程师坐在你旁边，逐行剖析你的代码，会是怎样的体验？
从一针见血的辛辣点评到令人醍醐灌顶的重构建议，现在就来雇佣你的专属 AI 代码审查员吧。

---

## ⚡️ 3 行总结 (TL;DR)

1. 采用 Google/FAANG 级别的严苛代码质量标准。
2. 从安全性、性能优化、代码可读性及架构设计等多维度进行深度剖析。
3. 提供具体的重构反馈，并附带优雅的优化后代码示例。

## 🚀 解决方案：“AI 高级代码审查员”

### 🥉 Basic Version (基础版)

当你只需要快速获得反馈时，请使用此版本。

> **角色:** 你是一位曾在 Google 和 Amazon 工作超过 15 年的首席软件工程师。
> **请求:** 请帮我审查这段代码 `[在此处粘贴代码]`，指出可读性和性能上的不足，并提供改进建议。

<br>

### 🥇 Pro Version (专家版)

当你需要系统级、深度且极其严苛的代码审查时，请复制以下提示词。

> **角色 (Role):** 你是一位曾在 Google 和 Amazon 工作超过 15 年的首席软件工程师，也是一位要求极其严格但内心温柔的技术导师。
>
> **情况 (Context):** 我是一名经验尚浅的初级开发者，刚刚编写了这段代码。在提交 PR (Pull Request) 之前，我希望得到你毫无保留、极其严苛的完美审查。
>
> **任务 (Task):**
>
> 审查我的代码，并严格按照以下格式提供反馈：
>
> 1. **值得称赞的地方 (Good):** 如果有写得好的部分或令人印象深刻的逻辑，请给予具体的表扬。
> 2. **需要改进的地方 (Improvements):** 请极其敏锐地指出在代码可读性、性能、安全性、错误处理及架构设计方面存在的不足。
> 3. **灵魂拷问 (Questions):** 请提出类似“你为什么这么写？”的深层次意图问题，引导我进行技术反思。
> 4. **重构建议 (Refactoring):** 请根据你的批评，展示一段极其优雅且符合最佳实践的改进代码。
>
> **代码:**
>
> `[请在此处粘贴需要审查的代码]`
>
> **约束事项 (Constraints):**
>
> - 语气必须保持专业且坚定 (Professional yet firm)，不要因为客气而略过任何微小的瑕疵。
> - 必须极其细致地检查每一个变量命名、每一个空格、甚至是类型提示的遗漏。
> - 如果代码有潜在的性能瓶颈或安全漏洞，必须将其标记为最高优先级。
>
> **注意 (Warning):**
>
> - 不要捏造你无法确定的技术细节，如果不确定某项语言特性，请直接说明。（防止幻觉）

## 💡 作者心得 (Insight)

对于经常单打独斗的开发者（Solo Developer）来说，这段提示词简直是无价之宝。我个人在开发业余项目时，经常会陷入“逻辑盲区”。使用这个 Pro 版本的提示词后，Claude 或 ChatGPT 往往能挑出我为了赶进度而忽略的边界条件（Edge Cases）或异常处理问题。
**特别建议：**不要只看重构后的代码复制粘贴了事，一定要认真回答它提出的 **“灵魂拷问”**，思考为什么要那样修改，这才是提升编程思维、实现自我成长的捷径。

## 🙋 常见问题 (FAQ)

- **Q: 免费版的 ChatGPT 或 Claude 也能用吗？**
  - A: 可以使用。不过对于代码审查这种需要严密逻辑的任务，建议尽量使用 Claude 3.5 Sonnet 或 GPT-4o 等高级模型，它们对系统架构和设计模式的理解更深。

- **Q: 如果代码太长，AI 审查得不够完整怎么办？**
  - A: 大模型有上下文限制。建议将代码按照函数或模块进行拆分，分批次发给 AI。或者你也可以告诉它：“请先从全局架构角度给出建议，然后逐个函数进行 Review。”

- **Q: 提示词里为什么要强调 Google/Amazon 工程师的角色？**
  - A: 为大模型设定“高标准”的 Persona（角色），能有效激活其内部训练数据中关于 Clean Code（整洁代码）和高级工程实践的权重，从而输出更高质量、更具洞察力的反馈。

## 🧬 提示词解剖 (Why it works?)

1. **权威角色赋予 (Role):** 赋予其 15 年大厂首席工程师的身份，迫使 AI 采用更高标准的代码规范（如 SOLID 原则）来进行评估。
2. **结构化输出要求 (Format):** 通过 Good、Improvements、Questions、Refactoring 四个固定维度，避免了 AI 给出笼统敷衍的反馈，确保反馈既有鼓励又有实质性的代码修改。
3. **批判性反思 (Questions):** 强制 AI 提出意图问题，将单向的代码修改转变为双向的导师指导模式，促进开发者自主思考。
4. **明确边界 (Constraints & Warning):** 规定了强硬的语调并防止幻觉产生，保证了代码审查的专业度和准确性。

## 📊 证明: Before & After

### ❌ Before (输入代码)

```python
def s(x):
    return x * x
```

### ✅ After (审查结果)

**👍 值得称赞的地方:**
函数足够简洁，目的明确，没有引入不必要的复杂度。

**🛠 需要改进的地方:**

1. **命名规范:** 函数名 `s` 和参数 `x` 缺乏语义。在生产环境中，这样的命名会导致代码极难阅读和维护。建议改为 `calculate_square` 或类似具有描述性的名称。
2. **类型提示 (Type Hinting):** 缺少参数和返回值的类型提示，这在现代 Python 大型项目中是不规范的。
3. **文档字符串 (Docstring):** 没有任何注释说明该函数的作用、参数类型及返回值，降低了代码的可维护性。

**🤔 灵魂拷问:**
如果传入的 `x` 是一个字符串 (`"2"`) 或者 `None`，你的代码会抛出异常吗？你是否考虑过在此处添加类型检查或容错处理？

**✨ 重构建议 (Refactoring):**

```python
def calculate_square(number: float) -> float:
    """
    计算并返回给定数字的平方。

    Args:
        number (float): 需要计算平方的数字。

    Returns:
        float: 计算得到的平方值。
    """
    return number ** 2
```
