# 📝 Никакого спагетти-кода! 🍝 Черная магия DDD-промптов от Senior-разработчика

- 🎯 **Для кого:** Разработчики, которые со вздохом смотрят на сгенерированный ИИ код, и наставники, застрявшие в аду поддержки.
- ⏱️ **Экономия времени:** Сокращает время на рефакторинг с 3 месяцев до 1 минуты.
- 🤖 **Рекомендуемые модели:** Диалоговые ИИ, специализирующиеся на кодинге (Claude 3.5 Sonnet, GPT-4o и др.)
- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

_Вы когда-нибудь смотрели на код от ИИ с первоначальным восторгом, который затем сменялся полным отвращением? Поздравляю. Вы только что вошли в «Ад спагетти-кода»._

Когда вы поручаете написание кода ИИ, сначала он справляется потрясающе. Но стоит проекту немного вырасти? Вы получаете ужасающего монстра, где запросы к БД прилеплены прямо к UI-компонентам, а функции раздуваются до 300 строк. ИИ по своей природе — «подпевала», поэтому если вы ставите задачу небрежно, он выдаст код с такой же небрежной архитектурой.

Пришло время остановить слепую лесть ИИ и принудительно внедрить идеальную «Чистую архитектуру» с самого фундамента. Этот чит-код превращает ИИ в безжалостного, дотошного Senior-разработчика, заставляя его писать чистую бизнес-логику (DDD), не зависящую от фреймворков.

---

## ⚡️ Краткая суть (TL;DR)

- 🚫 **Тотальный запрет спагетти-кода:** Принудительное и идеальное разделение ответственности (SoC) между UI, бизнес-логикой и базой данных.
- 🏗️ **Независимость от фреймворков:** Проектирование ядра, которое выживет, даже если завтра React или Next.js перестанут существовать.
- ✂️ **Уничтожитель гигантских функций:** Любая раздутая функция свыше 30 строк безжалостно дробится ИИ в соответствии с принципом единственной ответственности (SRP).

---

## 🚀 Решение: Промпт «Режим хардкорного архитектора»

### 🥉 Базовая версия (Basic Version)
Используйте, когда результат нужен быстро.

> **Роль:** Ты `[дотошный Senior-архитектор]`.
> **Задача:** Я хочу реализовать `[указать фичу]`. Напиши код с абсолютно строгим разделением бизнес-логики и UI. Я категорически не приемлю спагетти-код, где в одном файле смешано множество функций.

<br/>

### 🥇 Про-версия (Pro Version)
Используйте, когда требуется детальная проработка и безупречное разделение слоев.

> **Роль (Role):** Ты бескомпромиссный хардкорный Senior-архитектор и бог программной инженерии. Оставь бессмысленную лесть и фразы вроде "Да, конечно!" — говори только кодом.
>
> **Контекст (Context):**
> - Фон: В настоящее время мы разрабатываем систему `[указать ключевой домен]` на базе `[среда проекта/фреймворк]`.
> - Цель: Написать код, применяя предметно-ориентированное проектирование (DDD) и чистую архитектуру, чтобы предотвратить деградацию кода и обеспечить идеальное разделение слоев View-Business-Data.
>
> **Задача (Task):**
> 1. Напиши код для `[указать конкретную фичу]`.
> 2. **Категорически запрещено смешивать слои:** Никаких запросов к БД или вызовов внешних API внутри UI/Controller. Они должны отвечать только за рендеринг и передачу параметров.
> 3. **Предотвращение загрязнения домена:** Категорически запрещено импортировать зависимости фреймворка или БД в ключевую бизнес-логику (Entity, Value Object). Она должна состоять исключительно из чистых функций и типов.
> 4. **Уничтожитель гигантских функций:** Если функция выполняет более 2 действий или ее размер превышает 30 строк, немедленно разбей ее на более мелкие в соответствии с принципом единственной ответственности (SRP).
>
> **Ограничения (Constraints):**
> - Уважай базовые правила маршрутизации фреймворка, но вынеси бизнес-логику в независимые папки, такие как `src/domain`.
> - Внешние зависимости должны использовать внедрение зависимостей (DI) на основе интерфейсов.
>
> **Предупреждение (Warning):**
> - Если код хоть немного будет пахнуть "спагетти", этот проект обречен. Никаких компромиссов — предоставь самую идеальную и чистую архитектурную структуру.

<br/>

## 💡 Комментарий автора (Insight)

Я оттачивал этот навык раз за разом по одной единственной причине: эти ИИ по своей природе — подхалимы, отчаянно стремящиеся «быстро выдать результат и получить похвалу».

Когда я прошу создать кнопку, а в ответ вижу компонент кнопки, в который запихнули и `fetch`, и SQL-запросы к БД, во мне просыпается жажда разрушения. Этот промпт работает как отрезвляющая пощечина, которая хватает поверхностного ИИ-подпевалу за грудки и кричит: **«Очнись, с этого момента ты — бессердечный Senior-архитектор!»**

Попробуйте применить этот промпт в реальном проекте. Вы увидите, как ИИ внезапно изменит тон (и взгляд), разобьет файл на 3-4 части и начнет строго изолировать слои. Сначала может показаться неудобным, что файлов стало больше, но через 3 месяца, когда требования изменятся и вам придется поддерживать этот код, вы будете каждое утро мысленно кланяться мне (автору) в знак благодарности. Запомните: процентные ставки по техническому долгу сравнимы с ростовщическими. Компромиссов быть не может.

## 🙋 Часто задаваемые вопросы (FAQ)

- 🔹 **В: Нормально ли, что код так сильно дробится и количество файлов увеличивается?**
  - 🔸 **О:** Абсолютно нормально. Поддерживать 10 файлов по 50 строк в 100 раз проще, чем один файл на 500 строк. Не бойтесь увеличения количества файлов.

- 🔹 **В: Я фронтенд-разработчик, будет ли этот промпт полезен для меня?**
  - 🔸 **О:** Безусловно. Как только в UI-компонентах (`page.tsx`, `Component.vue` и т.д.) смешивается бизнес-логика или логика мутации состояния, ваш фронтенд тоже превращается в спагетти. Разделение представления и логики — это универсальная истина для любой сферы.

## 🧬 Анатомия промпта (Why it works?)

- 🎯 **Внедрение хардкорной персоны:** Мы лишили ИИ его базовой «покладистости» и заставили строго придерживаться принципов.
- 🧱 **Явные запрещающие условия:** ИИ не может схитрить, так как мы точечно запретили конкретные антипаттерны, такие как «загрязнение слоя UI» и «гигантские функции».
- 🧩 **Стимулирование паттерна Адаптер:** Мы предотвратили избыточный оверинжиниринг, поручив ИИ уважать каркас фреймворка (маршрутизацию), изолируя только ключевую логику.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод)

```javascript
// Когда вы просите "создать API для регистрации пользователя и отправки email" без промпта
export default async function handler(req, res) {
  const { email, password } = req.body;
  // Прямое обращение к БД (Возникновение загрязнения!)
  const user = await prisma.user.create({ data: { email, password } });
  // Прямой вызов внешнего API (Разрушение разделения ответственности!)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ После (Результат)

```javascript
// После применения хардкорного DDD-промпта (Слои и ответственность идеально разделены)

// 1. Слой Controller (Обрабатывает ТОЛЬКО маршрутизацию запросов/ответов)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Слой Domain Service (Чистая бизнес-логика, независимая от фреймворка)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Слой Infrastructure (Адаптеры, выделенные исключительно для БД и внешних API)
// UserRepository, EmailService и т.д. идеально изолированы в отдельных файлах...
```

## 🎯 Заключение

Самое опасное при передаче написания кода ИИ — это получить «мусорный код, который просто работает прямо сейчас». Это не тот случай, когда стоит радоваться отсутствию ошибок.

Схватите ИИ за грудки с помощью этого хардкорного DDD-чита и заставьте его с самого начала выдавать правильную архитектуру. Вы из будущего будете аплодировать себе из прошлого стоя. 

А теперь смело идите отдыхать! 🍷
