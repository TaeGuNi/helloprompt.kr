> _“每天光是赶业务代码就快猝死了，谁还有闲功夫去补那些枯燥乏味的测试覆盖率？”_

所有人都心知肚明，**单元测试（Unit Test）**是保障系统稳定性的生命线。然而，在真实残酷的开发排期中，编写测试用例所耗费的心智与时间，往往远超实现核心业务逻辑本身。你不仅要绞尽脑汁去穷举各种刁钻的边界情况（Edge Cases），还要深陷错综复杂的外部依赖中，被无休止的模拟对象（Mocking）折磨得心力交瘁……光是回想起这些，是不是就已经感到一阵头皮发麻了？

如果在代码提交前，因为覆盖率不达标而被 CI/CD 管道无情拦截，紧随其后的便是无休止的加班与重构。但现在，游戏规则改变了。只需简单的复制粘贴，把那些繁琐、机械且极度消耗耐心的测试代码编写工作，统统外包给全天候待命的 AI 吧！它将成为你最忠实的 QA 助理，让你在几分钟内就拥有一张严密的安全网。

## ⚡️ 3 行总结 (TL;DR)

1. **滴水不漏的全面覆盖：** 不仅能顺畅跑通正常的快乐路径（Happy Path），更能像侦探一样严密排查并防范各种隐蔽的边界异常。
2. **开箱即用的完美兼容：** 无缝适配 Jest、PyTest、JUnit 等所有主流测试框架的独有语法与断言特性。
3. **直击痛点的智能 Mock：** 自动敏锐识别业务代码中的外部依赖，精准生成隔离逻辑与逼真的 Mock 数据。

## 🙋 常见问题 (FAQ)

- **Q：AI 生成的 Mock 代码跑起来总是报错“找不到模块路径”，我该怎么解决？**
  - A：当代码的外部依赖链条过于复杂时，AI 确实很难凭空“猜想”出你本地真实的项目目录树。最有效的破局点是：在 `[请在此处粘贴你需要测试的业务逻辑代码]` 的下方，顺手附带一两个你项目中已有的、正确的 Mock 示例片段，或者干脆直接告诉它核心依赖项的绝对路径。记住，只要给足精准的上下文，AI 产出代码的“一次通过率”将会迎来质的飞跃。

- **Q：这套提示词可以直接生搬硬套去写集成测试（Integration Test）或端到端测试（E2E Test）吗？**
  - A：绝对不能完全照搬。这套提示词是专门针对系统底层的“单元测试”逻辑进行深度调优的。如果你打算编写 E2E 测试（例如使用 Playwright 或 Cypress），你必须额外向 AI 投喂真实的 DOM 结构（HTML 代码快照）以及极度详尽的用户交互行为链路。面对这类高维度的复杂场景，强烈建议你将其拆解为更细颗粒度的原子级任务步骤，然后再分批次交给 AI 处理。

## 🧬 提示词解剖 (Why it works?)

1. **确立专家角色 (Role)：** 强势为 AI 注入“追求极致完美的 QA 工程师”这一硬核职业人设，彻底迫使它跳出“代码凑合能跑就行”的平庸思维定势，转而死磕测试逻辑的严密性与边界覆盖的极限。
2. **强制罗列清单 (Task)：** 明确要求 AI 在提笔生成代码前，必须先老老实实地输出一份“测试用例清单”。这是业界极为经典的思维链（Chain of Thought, CoT）调教技巧，它能强迫 AI 先在“脑海”中彻底理清所有复杂的逻辑分支，然后再动手编写精准的断言，从而大幅降低极端场景被遗漏的概率。
3. **明确 Mock 指令 (Constraints)：** 现代工程化开发中最令人抓狂的痛点，莫过于处理那些盘根错节的外部服务依赖。通过在约束条件中强制命令 AI 妥善处理 Mock，可以一剑封喉地斩断它试图去连接真实数据库或向外发起真实网络请求的荒谬行径，从而确保最终生成的测试用例是绝对纯粹、高效且相互独立隔离的。

## 📊 证明：Before & After

### ❌ Before (你枯燥的业务代码)

```javascript
// 一个简单的用户余额扣除逻辑
function deductBalance(user, amount) {
  if (amount <= 0) throw new Error("扣除金额必须大于 0");
  if (user.balance < amount) throw new Error("余额不足");
  user.balance -= amount;
  return user.balance;
}
```

### ✅ After (AI 秒速生成的测试代码)

```javascript
describe("deductBalance 函数测试", () => {
  let mockUser;

  beforeEach(() => {
    // 每次测试前初始化干净的用户状态
    mockUser = { id: 1, balance: 100 };
  });

  // 1. 成功场景测试 (Happy Path)
  test("正常扣除金额时，应返回正确的剩余余额并更新用户对象", () => {
    const newBalance = deductBalance(mockUser, 30);
    expect(newBalance).toBe(70);
    expect(mockUser.balance).toBe(70);
  });

  // 2. 异常场景测试 - 边界值分析
  test("扣除金额为 0 时，应抛出异常", () => {
    expect(() => deductBalance(mockUser, 0)).toThrow("扣除金额必须大于 0");
  });

  test("扣除金额为负数时，应抛出异常", () => {
    expect(() => deductBalance(mockUser, -10)).toThrow("扣除金额必须大于 0");
  });

  // 3. 异常场景测试 - 业务规则限制
  test("扣除金额大于用户可用余额时，应抛出余额不足异常", () => {
    expect(() => deductBalance(mockUser, 150)).toThrow("余额不足");
  });
});
```
