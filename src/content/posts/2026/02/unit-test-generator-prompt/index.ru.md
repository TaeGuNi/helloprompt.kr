## 📝 Написание тест-кейсов за 5 минут: Идеальный промпт

- **🎯 Рекомендуется:** Junior-разработчикам, Frontend/Backend инженерам, QA
- **⏱️ Затраты времени:** Сокращение с 30 до 5 минут
- **🤖 Лучшая производительность:** Модели, специализирующиеся на кодинге (Claude 3.5 Sonnet, GPT-4o и др.)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Практичность:** ⭐⭐⭐⭐☆

> _"Завтра релиз, а я едва успеваю написать основную бизнес-логику. Когда мне вообще писать тесты?"_

У каждого разработчика был тот самый леденящий душу момент. Пятница, вечер, вы с энтузиазмом выкатываете новую фичу в продакшен, и вдруг в самом неожиданном месте вылетает критический баг. Вы открываете код для хотфикса, но в дебрях сложной бизнес-логики даже не можете понять, где именно произошел сбой. _"Если бы я только написал тесты..."_ — но сожалеть уже поздно. Мы все прекрасно знаем важность тестирования. Такие термины, как чистая архитектура, TDD и 100% покрытие кода, всегда заставляют сердца разработчиков биться чаще.

Однако реальность сурова. Когда релиз уже завтра, а поток входящих требований от бизнеса не иссякает, мало кто может позволить себе тратить время на написание тестов, которое зачастую занимает в 2–3 раза больше времени, чем создание самой логики. Особенно когда приходится иметь дело с легаси-кодом, имеющим массу внешних API-зависимостей, или логикой, тесно связанной с базой данных — одна только настройка десятков строк <b>мок-объектов (Mocking)</b> и бойлерплейта может выжать из вас все соки.

В итоге, устав от бесконечных раздумий над сложными краевыми случаями (Edge Cases), вы идете на опасный компромисс: "На этот раз поверим мануальному тестированию QA-команды и выкатим так". Накопленный таким образом <b>технический долг (Technical Debt)</b> превращается в огромные кандалы, которые со временем делают рефакторинг невозможным. Каждое нажатие кнопки "Изменить" начинает сопровождаться страхом, что сломается что-то другое. Пора прекратить этот цирк с хождением по канату без страховки.

Оставьте в прошлом все мучительные процессы и скучные раздумья. Просто скопируйте и вставьте вашу основную логику, над которой вы так усердно трудились. От мельчайших исключений, которые легко упустить, до надежной защиты от Null Pointer — <b>ИИ мгновенно напишет для вас безупречный код юнит-тестов (Unit Test)</b>.

Всего 5 минут. От создания сложных мок-объектов до выстраивания чистых тестовых сценариев по паттерну GIVEN-WHEN-THEN — наймите себе в партнеры по парному программированию старшего QA-инженера-перфекциониста. Он <b>безупречно заполнит все слепые зоны</b> своей механической тщательностью там, где интуиция человека может подвести. Почувствуйте уверенность от 100% покрытия кода и испытайте истинное удовольствие, нажимая кнопку 'Merge'.

## 📊 Доказательство: Ошеломляющий результат (Before & After)

### ❌ Before (Страдания, к которым мы привыкли)

Момент написания логики, когда вы остаетесь один на один с пустым холстом или неопрятным ручным методом. Опасная логика, брошенная на произвол судьбы без тестов.

```javascript
// 단순한 입력 코드 예시
function calculateDiscount(price, discountRate) {
  if (price < 0 || discountRate < 0) {
    throw new Error("유효하지 않은 값입니다.");
  }
  if (discountRate > 100) {
    return 0; // 100% 이상 할인은 무료
  }
  return price - price * (discountRate / 100);
}
```

### ✅ After (Идеальное преображение)

!["Написание тест-кейсов за 5 минут: Идеальный промпт"](/images/hooks/unit-test-generator-prompt.jpg)

```javascript
describe("calculateDiscount 함수 테스트", () => {
  // 1. 정상 동작 테스트 (Happy Path)
  it("정상적인 가격과 할인율이 주어지면 할인이 적용된 가격을 반환해야 한다", () => {
    // GIVEN
    const price = 10000;
    const discountRate = 20;
    // WHEN
    const result = calculateDiscount(price, discountRate);
    // THEN
    expect(result).toBe(8000);
  });

  // 2. 경계값 테스트 (Edge Case)
  it("할인율이 100%를 초과하면 0원을 반환해야 한다", () => {
    // GIVEN & WHEN
    const result = calculateDiscount(10000, 110);
    // THEN
    expect(result).toBe(0);
  });

  // 3. 예외 처리 테스트 (Failure Case)
  it("가격이 음수일 경우 에러를 던져야 한다", () => {
    // GIVEN, WHEN & THEN
    expect(() => calculateDiscount(-5000, 10)).toThrow(
      "유효하지 않은 값입니다."
    );
  });
});
```

## ⚡️ Краткое резюме (TL;DR)

1. **От Happy Path до Edge Case:** Автоматически выстраивает сценарии тестирования, проверяя не только стандартный ход работы, но и исключительные ситуации, которые сложно предсказать человеческим умом.
2. **Под нужный фреймворк:** Генерирует сразу готовый к запуску код, точно следуя синтаксису Jest, PyTest, JUnit или любого другого фреймворка, используемого в вашем проекте.
3. **Автоматизация сложного Mocking:** ИИ самостоятельно определяет внешние вызовы API или зависимости от баз данных и идеально настраивает подходящие Mock-данные и стабы (Stubs).

> **Контекст (Context):**

>

>
> **Ограничения (Constraints):**
>
> - Результат вывода должен быть строго внутри блока кода Markdown.

## 💡 Комментарий автора (Инсайты и советы)

Этот промпт — не просто инструмент автодополнения, который "пишет код за вас", это мощный <b>партнер по обеспечению качества (QA)</b>. Самая большая ценность и отличие этого промпта заключается в первом пункте инструкции: <b>"составление списка тест-кейсов"</b>. Прежде чем ИИ начнет бездумно писать код, вы сможете увидеть, как он проанализировал и интерпретировал вашу бизнес-логику в виде списка. Это позволяет заранее защититься от критических краевых случаев (Edge Cases: передача Null, неожиданное несовпадение типов, проблемы конкурентности и т.д.), о которых мог не подумать даже сам разработчик.

В реальной рабочей среде, даже при использовании одного и того же фреймворка (например, Jest, JUnit, PyTest), <b>методы мокинга (Mocking)</b> и стиль кода могут немного отличаться в зависимости от стандартов команды. Некоторые команды предпочитают `jest.spyOn`, другие — создание поддельных объектов (Fake Objects) через внедрение зависимостей (DI). Если сгенерированный ИИ код не соответствует строгим стандартам вашего проекта, просто добавьте в раздел ограничений промпта сниппет <b>`[пример кода Mocking в нашей команде]`</b>. Современные модели ИИ отлично обучаются на примерах (Few-Shot Learning), поэтому они мгновенно усвоят ваш уникальный стиль и синтаксис.

Кроме того, тесты — это не то, что пишется один раз и забывается. Это живая документация, которая должна поддерживаться при каждом изменении основной логики. Этот промпт специально заставляет использовать паттерн <b>GIVEN-WHEN-THEN</b> в комментариях. Это делается не только для красоты. Это мощнейший инструмент коммуникации, помогающий вам в будущем или вашим коллегам за одну секунду интуитивно понять намерение и ход работы сложной логики.

При написании тестов неизбежно приходится сталкиваться с участками кода, взаимодействующими с базой данных или сторонними API. Новички часто совершают ошибку, пытаясь написать интеграционные тесты (Integration Test), вызывающие реальные БД или API. Однако этот промпт содержит строгую инструкцию по <b>"полному мокированию внешних зависимостей"</b>. ИИ умно распознает участки с сетевым взаимодействием или операциями ввода-вывода и заменяет их на стабы/моки (Stub/Mock), не требующие реального подключения. Благодаря этому вы исключаете нестабильные тесты (Flaky Tests), которые то проходят, то нет, и создаете надежную среду юнит-тестирования, гарантирующую успех за 0.1 секунды в любом месте и в любое время.

Наконец, прежде чем скармливать код целиком, вспомните о <b>принципе единственной ответственности (SRP)</b>. Если вы вставите в промпт огромную функцию-спагетти на сотни строк, даже самый продвинутый ИИ может потерять контекст или упустить важный краевой случай. Лучший способ использования — запрашивать тесты для небольших независимых функций. Это станет отличным индикатором рефакторинга: вы сами поймете, достаточно ли модульным и пригодным для тестирования (Testable Code) является ваш код.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если код слишком длинный и ответ ИИ обрывается посередине?**
  - О: Мы настоятельно рекомендуем разбивать запросы на уровне функций или классов. При вводе огромного файла целиком повышается вероятность того, что ИИ упустит ключевые краевые случаи. Попробуйте запрашивать тесты для каждой функции, строго следующей принципу SRP, по отдельности.

- **Q: Мой код содержит подключение к базе данных. Можно ли в этом случае сгенерировать юнит-тест?**
  - О: Да, абсолютно! Этот промпт содержит строгие указания по "мокингу внешних зависимостей". ИИ интеллектуально обнаружит вызовы DB Repository или ORM и заменит их на Mock-объекты, не требующие реального подключения к БД.

## 🧬 Анатомия промпта (Почему это работает?)

1. **Назначение роли (Persona):** Мы дали конкретную роль — "старший QA-инженер-перфекционист". Это побуждает ИИ выдавать не просто формальный код, а высококачественные тесты, дотошно проверяющие даже граничные значения (Boundary).
2. **Принудительный паттерн GIVEN-WHEN-THEN:** Мы закрепили в комментариях отраслевой стандарт, который максимизирует читаемость. Это создает прочную структуру, позволяющую любому члену команды мгновенно понять суть теста.
3. **Инструкция по автоматизации Mocking:** Мы явно делегировали ИИ самую трудоемкую часть написания юнит-тестов — "внедрение зависимостей и мокинг", что драматически сокращает время разработки.
