# 📝 Claude Code가 멍청해졌다? 예전 성능 100% 끌어내는 프롬프트

- **🎯 추천 대상:** 시니어/주니어 개발자, Claude를 코딩에 활용하는 모든 직장인
- **⏱️ 소요 시간:** 10분 → 1분 단축 (삽질 시간 방지)
- **🤖 추천 모델:** Claude 3.5 Sonnet, Claude 3.7 Sonnet 등 코딩 특화 모델

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"예전엔 개떡같이 말해도 찰떡같이 짜주던 Claude가, 요즘 들어 자꾸 엉뚱한 코드를 뱉어내서 답답하신가요?"_

최근 해커뉴스(Hacker News)와 레딧 등 개발자 커뮤니티를 중심으로 "Claude Code의 성능이 예전만 못하다"는 주장이 끊임없이 제기되고 있습니다. 복잡한 아키텍처를 줘도 문맥을 놓치거나, 지나친 안전성(Safety) 필터 때문에 창의적인 리팩토링을 거부하는 현상, 일명 'Catastrophic Forgetting' 혹은 'Dumbed Down' 논란입니다.

하지만 모델이 멍청해졌다고 우리도 멍청하게 질문할 수는 없습니다. 오늘은 AI의 제약 조건을 강제하고 Claude 특유의 깊은 추론(Deep Reasoning) 능력을 억지로 깨우는 **'디버깅 마스터 프롬프트'**를 소개합니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **성능 저하 체감:** 최근 Claude가 복잡한 코딩 문제에서 문맥을 놓치거나 기존 코드를 임의로 축약하는 현상이 빈번하게 보고되고 있습니다.
2. **해결책은 '제약'과 '추론 강제':** 모델이 대충 대답하지 못하도록 `<thinking>` 태그를 강제하고, 구조적인 제약 조건을 명확히 주어야 합니다.
3. **효과:** 아래 제공된 프롬프트를 사용하면, 예전의 날카로웠던 Claude의 코딩 실력을 다시 100% 체감할 수 있습니다.

## 🚀 해결책: "Claude 성능 100% 복구 프롬프트"

### 🥉 Basic Version (기본형)

단순한 버그 수정이나 가벼운 코드 작성 시, Claude가 대충 답변하는 것을 막고 싶을 때 사용하세요.

> **역할:** 너는 `[언어/프레임워크, 예: React]` 전문가 수준의 시니어 개발자야.
>
> **요청:** 아래 코드의 `[발생한 문제/에러 내용]`을 해결해줘. 코드를 바로 작성하지 말고, 문제의 원인을 3단계로 분석한 뒤에 해결책을 제시해.

<br>

### 🥇 Pro Version (전문가형)

복잡한 리팩토링, 아키텍처 설계, 혹은 Claude가 계속해서 맥락을 놓치고 엉뚱한 소리를 할 때 강력하게 멱살을 잡고 끌고 가는 프롬프트입니다.

> **역할 (Role):** 너는 20년 차 수석 소프트웨어 아키텍트이자 깐깐한 코드 리뷰어인 `[전문가 역할, 예: Principal TypeScript Engineer]`야.
>
> **상황 (Context):**
>
> - 배경: 현재 `[프로젝트 설명, 예: Next.js 기반의 결제 시스템]`을 개발 중인데, Claude 네가 자꾸 이전 문맥을 잊어버리거나 비효율적인 코드를 제안하고 있어.
> - 목표: `[최종 목표, 예: 결제 상태 동기화 로직의 Race Condition 해결]`을 완벽하고 안전하게 구현하는 것.
>
> **요청 (Task):**
>
> 1. 아래 제공된 기존 코드와 요구사항을 완벽하게 숙지할 것.
> 2. 코드를 작성하기 전에, 반드시 `<thinking>` 태그 안에서 네가 어떤 논리로 이 문제를 해결할 것인지, 시간 복잡도와 공간 복잡도는 어떻게 최적화할 것인지 5단계 이상으로 깊게 추론(Deep Reasoning)할 것.
> 3. 추론이 끝난 후, 실무에 즉시 적용 가능한 수준의 최적화된 코드를 `[출력 형식, 예: 전체 파일 코드블럭]` 형태로 제공할 것.
> 4. `[변수]`나 수정이 필요한 환경 설정값은 주석으로 명확히 표시해줄 것.
>
> **제약사항 (Constraints):**
>
> - 절대 기존에 잘 작동하던 로직을 임의로 삭제하거나 축약하지 마. (주석으로 `// 기존 코드 생략` 처리 절대 금지)
> - 불필요한 서론이나 사과("죄송합니다, 제가 실수했습니다" 등)는 생략하고 기술적인 분석과 완벽한 코드만 제시해.
>
> **주의사항 (Warning):**
>
> - 네가 작성한 코드가 실서버에 바로 배포된다고 가정하고 엄격하게 작성해. 보안 취약점이나 에러 처리 누락이 발견되면 안 돼. 확실하지 않은 라이브러리 API는 절대 사용하지 마.

## 💡 작성자 코멘트 (Insight)

최근 Anthropic이 모델을 업데이트하면서 연산 비용을 줄이기 위해 보이지 않는 곳에서 최적화를 진행했다는 소문이 파다합니다. 실제로 예전에는 "이거 고쳐줘" 한 마디면 찰떡같이 알아듣던 녀석이, 이제는 구체적인 가이드라인이 없으면 엉성한 뼈대 코드만 던져주곤 하죠.

이 프롬프트의 핵심은 바로 **`<thinking>` 태그를 통한 추론 강제**와 **"기존 코드 생략 금지" 제약**입니다. AI에게 코드 작성 전 구조화된 고민의 시간(Chain-of-Thought)을 강제로 부여하면, 성능 저하 논란이 무색해질 만큼 다시 예전의 그 천재적인 아키텍트로 돌아옵니다. 답답해하며 키보드를 샷건 치지 마시고, 강력한 프롬프트로 기강을 꽉 잡아보세요!

## 🙋 자주 묻는 질문 (FAQ)

- **Q: Claude 3.5 Sonnet과 최신 3.7 Sonnet 모두 적용 가능한가요?**
  - A: 네, 두 모델 모두에서 매우 효과적입니다. 특히 안전성(Safety) 필터가 강력하게 적용된 최신 모델일수록 명시적인 제약 조건이 필수적입니다.

- **Q: `<thinking>` 태그는 구체적으로 무슨 역할을 하나요?**
  - A: LLM은 내부적으로 토큰을 순차적으로 생성하면서 다음 답변의 방향성을 결정합니다. 생각하는 과정을 텍스트로 먼저 출력하게 만들면, AI가 스스로 논리적 오류를 검증하게 되어 최종 코드의 퀄리티가 비약적으로 상승합니다.

## 🧬 프롬프트 해부 (Why it works?)

1.  **`<thinking>` 태그 강제:** 눈에 보이지 않는 AI의 추론 과정을 명시적으로 강제하여, 복잡한 문제 해결 시 빈번하게 발생하던 논리적 비약을 차단했습니다.
2.  **생략 금지(No Truncation) 제약:** 개발자들이 가장 분노하는 _"나머지 코드는 여기에 작성하세요"_ 식의 성의 없는 답변을 원천 차단했습니다.
3.  **사과 금지 및 컨텍스트 리셋:** 잦은 사과로 인한 불필요한 토큰 낭비를 막고, 오직 기술적 해결책 도출에만 집중하도록 역할을 엄격하게 통제했습니다.

## 📊 증명: Before & After

### ❌ Before (단순한 지시)

```text
User: 결제 모듈에서 자꾸 동시성 이슈가 생겨. 이거 고쳐봐.

Claude: 죄송합니다! 동시성 이슈를 해결하기 위한 코드입니다.
(중요한 기존 비즈니스 로직을 다 날리고 껍데기만 있는 코드를 제시함)
// ... 나머지 로직 구현 ...
```

### ✅ After (Pro 프롬프트 적용)

```text
User: (Pro Version 프롬프트 입력)

Claude:
<thinking>
1. 현재 결제 모듈의 동시성 이슈는 DB 트랜잭션 격리 수준과 관련이 깊음.
2. 기존 로직을 분석해보니, A와 B 상태가 겹칠 때 Race Condition 발생 가능성 존재.
3. 이를 해결하기 위해 Optimistic Lock 패턴을 적용하겠음.
4. 기존 비즈니스 로직은 100% 보존하면서 Version Lock 관련 필드만 추가하는 방식으로 설계.
</thinking>

다음은 실무에 즉시 적용 가능한 Optimistic Lock 패턴이 적용된 전체 파일 코드입니다. 기존 비즈니스 로직은 모두 안전하게 보존되었습니다.
(완벽하게 수정된 생략 없는 전체 코드블럭)
```
