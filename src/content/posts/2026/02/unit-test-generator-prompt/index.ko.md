## 📝 테스트 코드 작성, 5분 만에 끝내는 프롬프트

- **🎯 추천 대상:** 주니어 개발자, 프론트엔드/백엔드 엔지니어, QA
- **⏱️ 소요 시간:** 30분 → 5분 단축
- **🤖 최상위 성능:** 코딩 특화 모델 (Claude 3.5 Sonnet, GPT-4o 등)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐☆

> _"당장 내일이 배포인데, 메인 비즈니스 로직 짜기도 벅찬 상황에서 테스트 코드는 도대체 언제 작성하나요?"_

개발자라면 누구나 한 번쯤 겪어본 등골 서늘한 순간이 있습니다. 금요일 오후, 야심 차게 작성한 신규 기능을 프로덕션에 배포했는데 예상치 못한 곳에서 크리티컬한 버그가 터집니다. 핫픽스를 위해 코드를 열어보지만, 방대한 비즈니스 로직 어디에서 문제가 발생했는지 파악조차 쉽지 않습니다. _"테스트 코드만 꼼꼼히 짜뒀더라면..."_ 하는 후회는 이미 늦었습니다. 우리 모두 테스트 코드의 중요성을 뼈저리게 알고 있습니다. 클린 아키텍처, TDD, 커버리지 100% 같은 단어들은 언제나 개발자들의 가슴을 뛰게 만듭니다.

하지만 현실은 냉혹합니다. 당장 내일이 릴리즈이고 쏟아지는 기획 요구사항을 쳐내기도 벅찬 상황에서, 메인 비즈니스 로직 작성보다 2~3배의 시간이 더 걸리는 테스트 코드 작성에 여유롭게 시간을 투자할 수 있는 개발자는 극히 드뭅니다. 특히나 방대한 외부 API 의존성을 가진 레거시 코드를 다루거나, 데이터베이스와 강하게 결합된 로직을 테스트하려면 수십 줄의 <b>모킹(Mocking)</b> 코드와 보일러플레이트를 세팅하느라 진이 빠지기 일쑤입니다.

결국 까다로운 엣지 케이스(Edge Case)를 끊임없이 고민하다 지쳐 "일단 이번 배포는 QA 팀의 매뉴얼 테스트를 믿고 이대로 가시죠"라며 위험한 타협을 하고 맙니다. 이렇게 쌓인 <b>기술 부채(Technical Debt)</b>는 결국 리팩토링조차 불가능하게 만드는 거대한 족쇄가 되어 부메랑처럼 돌아옵니다. 수정 버튼을 누를 때마다 다른 기능이 망가질까 두려워지는 불안감이 시작되는 것입니다. 안전망 없이 외줄 타기를 하는 서커스는 이제 멈춰야 합니다.

이제 그 모든 고통스러운 과정과 지루한 고민은 접어두고, 여러분이 땀 흘려 작성한 메인 로직만 가볍게 복사해서 붙여넣으세요. 개발자가 놓치기 쉬운 미세한 예외 케이스(Edge Case)부터 까다로운 널 포인터(Null Pointer) 방어선까지, <b>AI가 완벽하게 작동하는 무결점 유닛 테스트(Unit Test) 코드를 단숨에 작성</b>해 드립니다.

단 5분. 복잡한 모킹(Mocking) 객체 생성부터 GIVEN-WHEN-THEN 패턴이 적용된 깔끔한 테스트 시나리오 도출까지, 완벽주의 성향의 시니어 QA 엔지니어를 여러분의 페어 프로그래밍 파트너로 고용해 보세요. 사람의 직관이 놓치기 쉬운 <b>사각지대를 기계적인 꼼꼼함으로 완벽하게 메워줍니다</b>. 커버리지 100%의 심리적 안정감을 안고, 자신 있게 'Merge' 버튼을 누르는 통쾌한 경험을 선사합니다.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 엉성했던 기존의 수작업 방식, 혹은 답답했던 빈 캔버스의 고통을 야기하는 로직 작성 순간입니다. 테스트 코드 없이 덩그러니 남겨진 위험한 로직입니다.

```javascript
> // 단순한 입력 코드 예시
> function calculateDiscount(price, discountRate) {
>   if (price < 0 || discountRate < 0) {
>     throw new Error("유효하지 않은 값입니다.");
>   }
>   if (discountRate > 100) {
>     return 0; // 100% 이상 할인은 무료
>   }
>   return price - price * (discountRate / 100);
> }
```

### ✅ After (완벽하게 변신한 결과)

!["테스트 코드 작성, 5분 만에 끝내는 프롬프트"](/images/hooks/unit-test-generator-prompt.jpg)

```javascript
describe("calculateDiscount 함수 테스트", () => {
  // 1. 정상 동작 테스트 (Happy Path)
  it("정상적인 가격과 할인율이 주어지면 할인이 적용된 가격을 반환해야 한다", () => {
    // GIVEN
    const price = 10000;
    const discountRate = 20;
    // WHEN
    const result = calculateDiscount(price, discountRate);
    // THEN
    expect(result).toBe(8000);
  });

  // 2. 경계값 테스트 (Edge Case)
  it("할인율이 100%를 초과하면 0원을 반환해야 한다", () => {
    // GIVEN & WHEN
    const result = calculateDiscount(10000, 110);
    // THEN
    expect(result).toBe(0);
  });

  // 3. 예외 처리 테스트 (Failure Case)
  it("가격이 음수일 경우 에러를 던져야 한다", () => {
    // GIVEN, WHEN & THEN
    expect(() => calculateDiscount(-5000, 10)).toThrow(
      "유효하지 않은 값입니다."
    );
  });
});
```

## ⚡️ 3줄 요약 (TL;DR)

1. **Happy Path부터 Edge Case까지:** 정상적인 동작 흐름은 물론, 사람의 머리로는 예측하기 까다로운 예외 상황까지 빈틈없이 검증하는 테스트 시나리오를 자동으로 도출합니다.
2. **원하는 프레임워크 맞춤형:** Jest, PyTest, JUnit 등 현재 프로젝트에 도입된 테스트 프레임워크의 문법을 정확히 반영하여 즉시 실행 가능한 코드를 생성합니다.
3. **까다로운 Mocking 자동화:** 번거로운 외부 API 호출이나 데이터베이스 의존성을 AI가 스스로 파악해, 상황에 맞는 Mock 데이터와 Stub을 완벽하게 구성해 줍니다.

빠르게 기본적인 테스트 뼈대만 잡고 싶을 때 활용해 보세요.

>
> **요청:** 다음 코드를 분석해서 예외 케이스를 포함한 유닛 테스트(Unit Test) 코드를 작성해 줘.

### 🥇 Pro Version (전문가형)

복잡한 비즈니스 로직 검증, 까다로운 의존성 모킹(Mocking), 그리고 완벽한 테스트 커버리지가 필요할 때 이 프롬프트를 실무에 투입하세요.

> **역할 (Role):** 너는 한 치의 오차도 허용하지 않는 10년 차 시니어 QA 엔지니어이자 테스트 자동화 전문가야.
>
> **상황 (Context):**
>
> - 배경: 나는 현재 `[사용 언어 및 프레임워크 (예: TypeScript, Jest)]` 환경에서 프로젝트를 개발하고 있어.
> - 목표: 새로 구현한 비즈니스 로직의 신뢰성을 완벽하게 보장하고, 테스트 커버리지(Test Coverage) 100%를 달성하기 위해 아주 촘촘한 유닛 테스트(Unit Test)를 작성해야 해.
>
> **요청 (Task):**
>
> 1. 제공된 코드를 심층 분석하여 **테스트 케이스 목록**을 가장 먼저 마크다운 리스트 형식으로 정리해 줘. (Happy Path, 예외 및 실패 케이스, 경계값 분석을 반드시 포함할 것)
> 2. `[사용할 테스트 라이브러리 (예: Jest)]`를 기반으로, 복사해서 즉시 실행할 수 있는 무결점 테스트 코드를 작성해 줘.
> 3. 코드 내에 외부 의존성(API 호출, DB 연결, 날짜 및 시간 함수 등)이 존재한다면, 완벽하게 Mocking 처리를 해 줘.
> 4. 작성되는 모든 `it` 또는 `test` 블록 내부에는 GIVEN-WHEN-THEN 패턴을 적용하여 명확한 주석을 달아 줘.
>
> **코드:**
>
> `[여기에 테스트할 코드를 붙여넣으세요]`
>
> **제약사항 (Constraints):**
>
> - 출력 결과는 반드시 마크다운 코드 블록 안에 담아 줘.
> - 테스트 설명(Description)은 누구나 이해할 수 있는 명확한 한국어 문장으로 작성해 줘. (예: "0으로 나누기를 시도하면 에러를 발생시켜야 한다")
> - 코드는 중간에 생략하지 말고(No placeholder), 복사 후 즉시 실행 가능한 완성된 형태로 제공해 줘.
>
> **주의사항 (Warning):**
>
> - 존재하지 않는 라이브러리나 메서드를 임의로 지어내지 마. (할루시네이션 절대 금지)

## 💡 작성자 코멘트 (Insight & How to use)

이 프롬프트는 단순히 '테스트 코드를 대신 짜주는' 자동 완성 수준을 아득히 뛰어넘는, 강력한 <b>품질 보증(QA) 파트너</b>입니다. 특히 프롬프트 내에 명시된 첫 번째 지시사항인 <b>'테스트 케이스 목록 도출'</b> 단계야말로 이 프롬프트가 지닌 진정한 핵심 가치이자 차별점입니다. AI가 무작정 코드를 짜기 전에 해당 비즈니스 로직을 어떻게 분석하고 해석했는지, 마크다운 리스트 형태로 먼저 눈으로 확인할 수 있습니다. 이를 통해 개발자 본인조차 미처 고려하지 못했던 치명적인 엣지 케이스(Edge Case: 극단적인 Null 입력, 예상치 못한 타입 불일치, 동시성 문제 등)를 사전에 완벽하게 방어하고 논리적 허점을 메울 수 있습니다.

실무 환경에서는 동일한 테스트 프레임워크(예: Jest, JUnit, PyTest)를 사용하더라도, 각 개발 팀이나 사내 컨벤션에 따라 선호하는 <b>모킹(Mocking) 방식</b>이나 코드 스타일이 미묘하게 다를 수밖에 없습니다. 어떤 팀은 `jest.spyOn`을 선호하고, 어떤 팀은 의존성 주입(DI)을 통한 가짜 객체(Fake Object) 생성을 원칙으로 하기도 합니다. 만약 AI가 생성한 코드가 현재 여러분 프로젝트의 엄격한 기준과 맞지 않는다면, 프롬프트 하단 제약사항 부분에 <b>`[우리 팀의 Mocking 예시 코드]`</b> 스니펫을 단 하나만 추가해 보세요. 최신 AI 모델들은 Few-Shot 러닝 능력이 매우 뛰어나기 때문에, 여러분 팀의 고유한 스타일과 문법을 즉각적으로 학습하여 별도의 수정 없이 실무에 바로 투입할 수 있는 완벽한 코드를 출력할 것입니다.

또한, 테스트 코드는 한 번 작성하고 끝나는 것이 아니라 메인 로직이 변경될 때마다 지속적으로 유지보수되어야 하는 살아있는 문서입니다. 이 프롬프트는 <b>GIVEN-WHEN-THEN</b>이라는 업계 표준 패턴을 주석으로 강제하도록 설계되었습니다. 이는 단순히 코드를 예쁘게 보이게 하려는 목적이 아닙니다. 수개월 뒤 코드를 다시 열어볼 미래의 여러분 자신, 혹은 이 코드를 인계받을 다른 팀원들이 복잡한 로직의 의도와 흐름을 단 1초 만에 직관적으로 파악할 수 있도록 돕는 가장 강력한 커뮤니케이션 도구입니다.

테스트 코드를 작성하다 보면 필연적으로 데이터베이스 연결이나 외부 서드파티 API와의 통신 구간을 마주하게 됩니다. 이때 초보자들은 실제 DB나 API를 호출하는 통합 테스트(Integration Test)를 작성하는 실수를 범하곤 합니다. 하지만 이 프롬프트에는 <b>'외부 의존성 완벽 모킹(Mocking) 처리'</b>라는 엄격한 지시가 포함되어 있습니다. AI는 영리하게 네트워크 통신이 발생하는 구간이나 I/O 작업을 감지하고, 실제 네트워크 연결이 전혀 필요 없는 가짜 객체(Stub/Mock)로 이를 알아서 대체해 줍니다. 덕분에 네트워크 상태에 의존하여 어쩔 때는 성공하고 어쩔 때는 실패하는 불안정한 테스트(Flaky Test)를 원천 차단하고, 언제 어디서 실행하든 0.1초 만에 일관된 성공을 보장하는 견고한 유닛 테스트 환경을 구축할 수 있습니다.

마지막으로, 코드를 통째로 넘기기 전에 <b>단일 책임 원칙(SRP)</b>을 한 번 더 상기하시기 바랍니다. 만약 수백 줄이 넘어가는 거대한 스파게티 함수를 그대로 프롬프트에 밀어 넣는다면, 아무리 뛰어난 AI라도 핵심 문맥을 잃어버리거나 중요한 엣지 케이스 검증을 누락할 수 있습니다. 가장 좋은 활용법은, 기능별로 잘게 쪼개진 독립적인 함수 단위로 테스트 코드 작성을 요청하는 것입니다. 이는 자연스럽게 여러분의 기존 메인 로직이 충분히 모듈화되고 테스트하기 좋은 구조(Testable Code)로 설계되었는지 스스로 피드백을 받는 훌륭한 리팩토링의 지표가 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코드가 너무 길어서 AI의 답변이 중간에 자꾸 끊기면 어떻게 해야 하나요?**
  - A: 함수나 클래스 단위로 잘게 쪼개어 요청하는 방식을 강력히 권장합니다. 거대한 파일을 통째로 입력하면 AI가 핵심적인 엣지 케이스를 놓칠 확률이 높아집니다. 단일 책임 원칙(SRP)이 철저하게 지켜진 함수를 하나씩 분리해 테스트 작성을 요청해 보세요.

- **Q: 데이터베이스 연결 코드가 포함되어 있는데, 이런 경우에도 유닛 테스트 코드 생성이 가능한가요?**
  - A: 네, 완벽하게 가능합니다! 이 프롬프트에는 '외부 의존성 모킹(Mocking)'에 대한 강력한 지시가 포함되어 있습니다. AI가 DB Repository나 ORM 호출 구간을 스마트하게 감지하고, 이를 실제 데이터베이스 연결이 필요 없는 가짜 객체(Mock)로 알아서 대체해 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **페르소나(Role) 부여:** '완벽주의 성향의 시니어 QA 엔지니어'라는 구체적인 역할을 부여했습니다. 이를 통해 단순히 구색만 갖춘 코드가 아니라, 경계값(Boundary)까지 집요하게 파고들어 검증하는 고품질 테스트 코드를 유도해 냅니다.
2. **GIVEN-WHEN-THEN 패턴 강제:** 테스트 코드의 가독성을 극대화하는 업계 표준 패턴을 주석으로 강제했습니다. 코드를 처음 보는 다른 팀원들도 테스트의 의도와 흐름을 직관적으로 파악할 수 있도록 탄탄한 구조를 잡아줍니다.
3. **모킹(Mocking) 자동화 지시:** 유닛 테스트 작성 시 가장 큰 병목 구간인 '의존성 주입 및 모킹' 작업을 AI에게 명시적으로 위임하여, 테스트 작성에 소요되는 시간을 드라마틱하게 단축시켰습니다.
