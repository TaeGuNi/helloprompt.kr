## 🦀 WebAssembly(Wasm)ポーティング：Webパフォーマンスの限界を突破しよう

- **🎯 おすすめの対象:** フロントエンドのパフォーマンス最適化が急務の開発者、重厚なバックエンドロジック(C++/Rust/Go)をブラウザへ移植したいエンジニア
- **⏱️ 所要時間:** 何日も費やしていたバインディングの試行錯誤 → わずか5分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コード変換において圧倒的)、GPT-4o

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **実用性:** ⭐⭐⭐☆☆

> _「JSの最適化で徹夜を重ねてやっと10%高速化できた演算が、Wasmに移植した途端、一瞬で10倍速くなりました。でも、その『移植作業』自体をもっと簡単に終わらせる方法はないのでしょうか？」_

現代のブラウザは、もはや単なるドキュメントビューアではありません。画像処理や複雑な暗号化、大規模な数値計算など、JavaScriptのV8エンジンだけではどうしてもパフォーマンスの壁に直面する瞬間が訪れます。WebAssembly (Wasm) はその完璧な解決策となりますが、既存のC++やRustコードをWasmへ変換し、JSとメモリを共有して連携（バインディング）させるプロセスには、途方もない試行錯誤が伴うのが現実です。

本プロンプトは、単なるコードの直訳にとどまりません。メモリリークの確実な防止からJSブリッジコードの自動生成までをワンストップで完結させる、極めて優秀な「Wasmポーティングアシスタント」として機能します。

## 💡 筆者コメント (Insight)

WebAssemblyへのポーティングに挑む際、多くの開発者が最も深い挫折を味わうのが**「JSの配列をWasmへどう渡し、どう結果を受け取るのか？」**というデータブリッジの壁です。単純な数値や文字列の受け渡しは容易ですが、大容量の画像バッファやオーディオデータ (`Float32Array` など) を扱う場合、ポインタを通じてメモリを直接制御する高度な技術が要求されるからです。

このプロンプトの真髄は、`タスクの2番（メモリアーキテクチャ）`の指示にあります。AIに単なるコード変換を丸投げすると、メモリ領域を丸ごとコピーして渡すような、極めて非効率的なコードを平然と出力するケースが多々あります。しかし、上記のプロンプトのように「コピーコストの最小化」「共有メモリの活用」という制約を明示的に与えることで、AIは `wasm-bindgen` の強力な機能（ポインタを介したリニアメモリへの直接アクセスなど）をフル活用した「現場で即戦力となる本物のコード」を生成してくれます。さらに、WebpackやVite環境でWasmをロードする際につまずきがちな非同期処理のタイミング問題も、Reactのサンプルコードを通じて一挙に解決できる仕組みになっています。

## 🙋 よくある質問 (FAQ)

- **Q: 既存のコードベースが膨大すぎて一度にAIへ入力しきれません。どう進めればよいでしょうか？**
  - A: プロジェクト全体を一度に処理させるのではなく、コアロジックを構成する「主要な関数単位」に分割してプロンプトを実行することを強くお勧めします。依存関係が複雑に絡み合ったモジュール構造の場合は、`[状況 (Context)]` の項目に「このモジュールは〇〇のようなアーキテクチャを持っている」と前提知識を補足することで、AIが文脈を正確に把握し、より精度の高いバインディングを生成してくれます。

- **Q: 変換されたRustのコードは、具体的にどのようにビルドするのですか？**
  - A: 公式のツールチェーンである `wasm-pack` を使用すれば驚くほど簡単です。通常、プロンプトの出力結果の末尾に、AIが親切にビルドコマンド（例: `wasm-pack build --target web`）を提示してくれます。それをそのままコピーしてターミナルで実行するだけで、npmパッケージのようにすぐにインポート可能な `pkg` フォルダが自動生成されます。

- **Q: C++コードをEmscriptenで変換する手法と、Rust + wasm-bindgenの組み合わせにはどのような違いがありますか？**
  - A: Emscriptenは巨大なC/C++プロジェクトをそのままWebへ移植する際には非常に強力ですが、ビルド後のファイルサイズが肥大化しやすく、JSとの相互運用性がやや煩雑です。一方、Rustと `wasm-bindgen` の組み合わせは、最終的なバンドルサイズを極めて小さく抑えられる上、TypeScriptエコシステムとの親和性が圧倒的に高いため、近年のモダンフロントエンド開発においては事実上の標準 (デファクトスタンダード) となっています。そのため、本プロンプトでもRustへの変換を前提としたアプローチを採用しています。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **エンドツーエンド(E2E)ソリューションの追求:** 単にコードの構文をターゲット言語に翻訳させるだけでなく、フロントエンドフレームワーク (React等) でモジュールをどのようにロードし、レンダリングサイクルに組み込むかまでを厳格に要求することで、「開発現場ですぐに動くプロダクションレベルの成果物」を確実に引き出します。
2. **明示的なパフォーマンス制約 (Constraints):** Wasmを導入する最大の目的は「圧倒的なパフォーマンス」の獲得です。「ゼロコピー」や「シリアライズの回避」といった強力なキーワードを制約として組み込むことで、AIが安易でパフォーマンスを犠牲にするような「逃げのコード」を生成することを根本から封じ込めています。
3. **型安全性の徹底 (Type Safety):** JavaScriptの動的な世界とWebAssemblyの静的な世界の間に横たわる曖昧な境界線を、`.d.ts` ファイルの自動生成指示によって明確に規定し、開発フェーズでの予期せぬランタイムエラーを未然に防ぎます。

## 📊 証明：Before & After

単に「このコードをWasmに変換して」と雑にリクエストした場合と、緻密に計算されたProバージョンのプロンプトを使用した場合とでは、生成されるコードの品質に天と地ほどの差が生まれます。

### ❌ Before (単純なリクエスト時のAIの回答)

```rust
// データをシリアライズして丸ごとコピーする、非効率的なアプローチ
#[wasm_bindgen]
pub fn apply_filter(data: Vec<u8>) -> Vec<u8> {
    let mut result = data.clone(); // 莫大なメモリコピーが発生
    // ... フィルタリングロジック ...
    result
}
```

### ✅ After (Proプロンプト使用時のAIの回答)

```rust
// メモリポインタを活用してコピーなしで直接アクセス (Zero-copy)
#[wasm_bindgen]
pub fn apply_filter_in_place(data: &mut [u8], width: usize, height: usize) {
    // JSのUint8Arrayが指すWasmのリニアメモリに直接アクセスして操作
    for i in (0..data.len()).step_by(4) {
        // Red, Green, Blueチャンネルに直接フィルタを適用
        let r = data[i];
        let g = data[i + 1];
        let b = data[i + 2];

        data[i] = (r as f32 * 0.393 + g as f32 * 0.769 + b as f32 * 0.189) as u8;
        // ... (中略) メモリ割り当て/解放のオーバーヘッドを完全に排除
    }
}
```

_(これに加え、React環境で `memory.buffer` をどのように読み込み、効率的にレンダリングへ繋げるかを示した、完璧なカスタムフックのコードも併せて提供されます。)_
