# 💻 AI 코드 리뷰어: 내 똥 코드를 실리콘밸리 스타일로

- **🎯 추천 대상:** 사수 없이 고군분투하는 주니어 개발자, 레거시 스파게티 코드에 고통받는 유지보수 담당자
- **⏱️ 소요 시간:** 30분 → 1분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet (코드 맥락 파악 및 리팩토링 최강자)

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"돌아가긴 하는데... 변수명이 `a`, `b`, `temp`라니... 3개월 뒤의 내가 봐도 욕할 것 같은 코드, 이대로 커밋하시겠습니까?"_

![AI Code Reviewer](/images/hooks/ai-code-reviewer.png)



개발자로 취업하고 1년 차, 드디어 혼자서 첫 독립적인 피처 개발을 맡게 되었습니다. 밤을 새우며 구글링과 스택오버플로우를 뒤져가며 어떻게든 돌아가는 코드를 만들어냈습니다. 테스트 서버에서 기능이 정상적으로 작동하는 것을 확인하고 뿌듯한 마음에 PR을 올렸습니다. 그러나 다음 날 아침, PR에 달린 수십 개의 코멘트들을 보고 절망할 수밖에 없었습니다. "변수명이 직관적이지 않습니다", "이 부분은 시간 복잡도가 O(N^2)인데 개선 방법이 없을까요?", "예외 처리가 누락되어 서버가 다운될 수 있는 크리티컬한 로직입니다." 선임 개발자들의 피드백은 뼈를 때렸고, 저는 제 코드가 그야말로 스파게티 그 자체라는 것을 깨달았습니다. 하지만 선임들도 본인들의 업무로 바빠서 제 코드의 아주 세세한 부분까지 하나하나 친절하게 다시 짜주거나 과외를 해줄 수는 없는 노릇이었습니다.

이대로 가면 '코드 품질이 바닥인 주니어'라는 낙인이 찍힐 것만 같았습니다. PR은 계속 반려되고, 마감 기한은 다가오는데 어떻게 고쳐야 할지 방향조차 잡지 못했습니다. 그렇다고 혼자서 모든 것을 해결하려다 보니 하루 종일 삽질만 반복하는 악순환에 빠졌습니다. 바쁜 사수에게 매번 "이거 이렇게 고치면 될까요?"라고 묻는 것도 눈치가 보여서 결국 질문조차 하지 못하고 혼자 끙끙 앓는 날들이 길어졌습니다. 스트레스는 극에 달했고, 내가 과연 개발자라는 직업에 맞는 사람인지 근본적인 회의감마저 들기 시작했습니다. 이대로 주말까지 반납하고 사무실에 남아 남들이 짜놓은 코드를 복붙하며 간신히 연명하는 개발 인생이 계속된다면, 결국 버닝아웃이 오고 말 것이 뻔했습니다. 아무도 내 코드를 전담으로 마크해서 친절하고 날카롭게 리뷰해 주지 않는 이 아득한 고립감이 저를 갉아먹고 있었습니다.

그러던 중, 우연히 유튜브에서 AI를 활용한 코드 리팩토링 영상을 보게 되었습니다. 단순히 "이 코드 고쳐줘"라고 말하는 수준이 아니라, 완전히 각 잡힌 '실리콘밸리 수석 엔지니어'의 페르소나를 부여하여 내 코드를 무자비하게 해부하도록 만드는 마법의 프롬프트였습니다. 속는 셈 치고 제가 작성한 엉망진창인 로직을 프롬프트와 함께 대화형 AI에 던져보았습니다. 결과는 가히 충격적이었습니다. 단 5초 만에, AI는 제 코드의 변수명을 비즈니스 로직에 맞게 우아하게 변경하고, 불필요한 이중 루프를 `reduce` 함수로 최적화했으며, 제가 미처 생각지도 못했던 예외 처리 로직까지 완벽하게 추가하여 코드를 재탄생시켰습니다. 심지어 왜 이렇게 수정을 했는지에 대한 명확한 이유까지 주석으로 친절하게 달아주었습니다. 마치 10년 차 천재 시니어 개발자가 밤새 제 옆에 앉아서 사수 역할을 해주는 것과 같은 완벽한 경험이었습니다.

이 프롬프트를 도입한 이후, 제 개발 인생은 완전히 달라졌습니다. 더 이상 PR을 올리는 것이 두렵지 않습니다. 커밋하기 전에 무조건 이 AI 시니어 개발자에게 코드를 던져 1차 검수를 받고, 리팩토링된 코드를 보며 클린 코드의 구조를 스스로 학습하게 되었습니다. 신기한 것은, AI가 작성해 준 우아한 로직들을 자꾸 보다 보니 어느새 제 코딩 스타일 자체가 점점 세련되게 변해갔다는 것입니다. 이제는 사수분들도 제 PR을 보면서 "최근 코드 퀄리티가 엄청나게 좋아졌네. 따로 학원이라도 다니나?"라고 칭찬해주십니다. 단순히 버그를 고치는 수준을 넘어, 스파게티 코드를 실리콘밸리 스타일의 클린 코드로 탈바꿈시켜 주는 이 마법의 프롬프트. 오늘 여러분도 개인 멘토를 무료로 채용하고, 칼퇴와 코딩의 즐거움이라는 두 마리 토끼를 모두 잡으시길 바랍니다.

## ⚡️ 3줄 요약 (TL;DR)

1. 바쁜 사수 눈치 보지 말고 AI에게 즉각적인 코드 리뷰를 요청하세요.
2. 단순한 버그 탐지를 넘어, 가독성과 효율성을 갖춘 클린 코드로 리팩토링합니다.
3. 구글 수석 엔지니어의 페르소나를 부여해 실리콘밸리 수준의 코드 퀄리티를 확보하세요.

## 🚀 해결책: "Code Reviewer Prompt"

### 🥉 Basic Version (기본형: 퀵 버그 헌터)

빠르게 에러를 잡고 잠재적 결함만 확인하고 싶을 때 사용하세요.

> **역할:** 너는 날카로운 시각을 가진 시니어 개발자야.
> **요청:** 아래 코드에서 에러가 발생할 수 있는 잠재적 버그나 논리적 오류를 찾아줘. 만약 완벽하다면 "LGTM(Looks Good To Me)"이라고 말해.
>
> **코드:**
>
>
> [여기에 코드를 붙여넣으세요]
>
### 🥇 Pro Version (전문가형: 실리콘밸리 클린 코드 리팩토링)

변수명부터 아키텍처까지, 코드의 영혼까지 싹 뜯어고칠 때 사용하세요.

> **역할 (Role):** 너는 구글(Google)의 10년 차 수석 소프트웨어 엔지니어이자 'Clean Code'의 열렬한 신봉자야.
>
> **상황 (Context):**
>
> - 배경: 사내 프로젝트의 핵심 로직을 작성/유지보수하고 있음.
> - 목표: 기술 부채를 줄이고, 팀원 누구나 이해하기 쉬운 직관적이고 견고한 코드로 개선.
>
> **요청 (Task):**
> 아래 제공된 `[코드]`를 분석하고 다음 4가지 핵심 기준에 맞춰 엄격하게 리뷰 및 리팩토링해 줘.
>
> 1. **가독성 (Readability):** 변수명과 함수명을 의도가 명확히 드러나도록 직관적으로 변경해. (예: `d` -> `elapsedTimeInDays`)
> 2. **효율성 (Efficiency):** 불필요한 루프를 줄이거나 시간 복잡도(O)를 개선할 수 있는 더 나은 알고리즘, 배열 메서드 등을 제안해.
> 3. **안전성 (Safety):** Null 참조, 타입 에러, 예외 처리(Try-Catch)가 누락된 곳(Edge Cases)을 꼼꼼하게 방어해.
> 4. **주석 (Comments):** 로직이 복잡한 부분에는 '무엇을(What)'이 아니라 '왜(Why)' 이렇게 작성했는지 JSDoc/Docstring 형태로 주석을 달아줘.
>
> **출력 형식 (Format):**
>
> - **리뷰 요약:** 기존 코드의 치명적 문제점 핵심 3가지 (블릿 포인트)
> - **리팩토링 코드:** 완벽하게 개선된 전체 코드 (코드 블록)
> - **핵심 변경 사유:** 왜 이렇게 수정하는 것이 더 나은 아키텍처인지 2줄 이내로 요약.
>
> **제약사항 (Constraints):**
>
> - 언어의 최신 문법(Modern Syntax)을 활용할 것.
> - 확실하지 않은 라이브러리나 내장 함수는 지어내지 말 것. (환각 방지)
>
> **입력 (Input Code):**
>
>
> [여기에 코드를 전체 붙여넣으세요]
>

## 💡 작성자 코멘트 (Insight)

현업에서 수백 번 이상의 코드 리뷰를 수행하고 또 받아보면서 느낀 가장 뼈저린 교훈은, 단순히 "돌아가는 코드"와 "읽기 좋은 코드(Clean Code)" 사이에는 우주만큼의 거리가 존재한다는 것입니다. 단순히 "이 코드를 고쳐줘" 혹은 "버그를 찾아줘"라고 얕게 명령하면, AI는 여러분의 의도를 제대로 파악하지 못하고 고작 변수명 하나 바꾸거나 세미콜론 하나 붙여주는 수준의 표면적인 문법 렌더링에 그치게 됩니다. 이는 근본적인 아키텍처 개선에는 전혀 도움이 되지 않으며, 오히려 코드가 기형적으로 변질되는 끔찍한 결과를 낳을 수도 있습니다.

우리가 설계한 Pro 버전 프롬프트가 이토록 압도적인 퍼포먼스를 내는 핵심적인 이유는 바로 '다면적이고 무자비한 평가 기준의 강제'에 있습니다. 우리는 AI에게 단순히 '수정'을 요구하는 것이 아닙니다. 구글의 10년 차 수석 엔지니어라는, 완벽주의적이고 보수적이며 성능과 가독성의 극한을 추구하는 '페르소나'를 억지로 뒤집어씌웠습니다. 그리고 가독성(Readability), 효율성(Efficiency), 안전성(Safety), 그리고 주석(Comments)이라는 거스를 수 없는 4가지 철칙을 강제함으로써 꼼수를 부리지 못하게 만들었습니다. 

실제로 제가 근무하던 회사에서, 전임자가 퇴사하며 남긴 약 1,500줄짜리 극악의 레거시 결제 연동 스파게티 코드를 분석해야만 했던 일화가 있습니다. 변수명은 모두 `a1`, `b2`, `chk_val` 같은 알 수 없는 축약어 투성이였고, 하나의 함수 안에 데이터베이스 조회, 비즈니스 로직 연산, 외부 API 호출, 심지어 UI 에러 메시지 반환까지 모든 것이 덕지덕지 뭉쳐 있는 끔찍한 파멸의 코드였습니다. 이 코드를 인간의 눈으로 한 줄씩 파악하려면 꼬박 일주일은 걸렸을 것입니다. 

하지만 이 막막한 상황에서 바로 이 Pro 프롬프트가 구원투수가 되었습니다. 저는 이 1,500줄의 로직을 함수 단위로 쪼개어 AI 시니어 엔지니어에게 던졌습니다. AI는 단 10분 만에 O(N^3)의 최악의 시간 복잡도를 가진 루프를 해시맵(Hash Map)을 이용한 O(N)의 최적화 알고리즘으로 갈아 엎었고, 에러 핸들링이 누락되어 결제가 먹통이 될 수 있었던 치명적인 엣지 케이스 3곳을 정확히 짚어냈습니다. 게다가 각 모듈별로 단일 책임 원칙(SRP)을 철저하게 준수하는 우아한 함수로 완벽하게 분리해 내기까지 했습니다. 

가장 감동적이었던 순간은, 단순히 리팩토링된 코드를 내놓고 끝내는 것이 아니라, 하단에 "이 부분에서 성능 병목이 발생하고 있었으며, 자바스크립트의 Set을 활용하여 시간 복잡도를 획기적으로 낮췄습니다"라는 구체적인 아키텍처 개선 사유를 적시해 주었다는 점입니다. 이를 통해 저는 단순히 남이 짜준 코드를 복사 붙여넣기 하는 '코더'가 아니라, AI가 내린 설계 결정을 이해하고 흡수하여 한 단계 더 성장한 진짜 '엔지니어'로 레벨업할 수 있었습니다. 여러분도 이 프롬프트를 단순히 오타 검수용으로 전락시키지 마십시오. 당신의 코드를 산산조각 내서 올바른 설계 패턴의 뼈대부터 다시 세워주는 무자비한 코칭 마스터로 활용하시기 바랍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 회사 내부 코드를 AI 모델에 그대로 올려도 안전한가요?**
  - A: **절대 주의해야 합니다.** API 키, 사내 IP, 민감한 비즈니스 로직(DB 스키마 등)은 반드시 마스킹(Masking) 처리하거나 `[PRIVATE_DATA]` 등으로 치환한 후 프롬프트를 실행하세요. 엔터프라이즈 환경이라면 데이터 학습에 사용되지 않는 기업용 요금제(Team/Enterprise)나 API 모델을 사용하는 것을 강력히 권장합니다.

- **Q: AI가 짜준 코드를 그대로 복붙해서 배포해도 되나요?**
  - A: 안 됩니다. AI는 때때로 존재하지 않는 라이브러리 메서드를 호출하는 '할루시네이션(환각)'을 일으키거나, 복잡한 비즈니스 요구사항의 맥락을 오해할 수 있습니다. AI의 결과물은 강력한 초안이자 제안일 뿐, 최종 코드 리뷰와 테스트를 통한 검증은 온전히 개발자의 몫입니다.

- **Q: 코드가 너무 길어서 한 번에 분석이 잘 안 되는데 어떡하나요?**
  - A: 함수나 클래스 단위로 쪼개서 질문하세요. "이 `UserService` 클래스의 `authenticate` 메서드만 중점적으로 리뷰해 줘"라고 범위를 좁히면, AI가 컨텍스트를 잃지 않고 훨씬 더 퀄리티 높은 딥 다이브 리뷰를 제공합니다.

### ✅ After (클린 코드)

```javascript
/**
 * Returns a filtered list of active users.
 * @param {Array<Object>} users - The array of user data objects.
 * @param {number} users[].status - The user status code (1: Active).
 * @returns {Array<Object>} A new array containing only active users.
 */
const getActiveUsers = (users) => {
  // Handle edge case: invalid input
  if (!Array.isArray(users)) return [];

  const ACTIVE_STATUS_CODE = 1;

  // Maintains O(N) time complexity and maximizes readability by using a declarative approach (filter).
  return users.filter((user) => user?.status === ACTIVE_STATUS_CODE);
};
```
