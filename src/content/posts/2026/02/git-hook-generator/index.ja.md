人間は誰しもミスをするものですが、システムはミスを許容すべきではありません。コードをコミット（Commit）したりプッシュ（Push）する直前に、`Lint`チェックや`Test`コードを手動で実行するのは煩わしく、忘れがちです。

そんな時に救世主となるのが**Git Hook**です。しかし、シェルスクリプト（Shell Script）の構文に慣れていない場合、クロスプラットフォームで完璧に動作するHookを自作するのはかなりのストレスになります。これからは、チームの規約と環境をAIに説明するだけにしましょう。たった1秒で、隙のない門番のようなスクリプトを完成させてくれます。

> **状況 (Context):**
>
> - 背景: チームメンバーがコミット前にコード規約を守らないため、CIパイプラインで頻繁にビルドが失敗し、リソースの無駄遣いが発生しています。

> **タスク (Task):**
>
> 1. `[pre-commit]`フック: ステージングされたファイル（`git diff --staged`）に対してのみ`[npm run lint]`と`[npm run format]`を実行するシェルスクリプトを作成してください。
> 2. `[commit-msg]`フック: コミットメッセージが`[Conventional Commits (feat, fix, docs など)]`の形式を厳格に守っているかを正規表現でチェックするスクリプトを追加してください。
> 3. **案内メッセージ:** テストやチェックに失敗した場合、ターミナルに親切で明確なエラーメッセージ（例を含む）を出力し、コミットをブロック（exit 1）してください。
> 4. **適用ガイド:** チームメンバーがリポジトリをクローンした後、すぐに環境をセットアップできるように、`[package.json]`の設定と初期インストールコマンドをMarkdownのコードブロックで整理してください。
>
> **制約事項 (Constraints):**
>
> - 作成するシェルスクリプトは、macOS、Linux、Windows（Git Bash環境）のすべてで安全にクロスプラットフォームで動作する必要があります。
>
> **注意事項 (Warning):**
>
> - プロジェクトの規模が大きいため、全ファイルを検査すると速度が遅すぎます。必ず**「変更が発生したファイル（staged files）」**だけを抽出して検査するようにパフォーマンスを最適化してください。（例: `lint-staged`の活用など）

## 🙋 よくある質問 (FAQ)

- **Q: Windowsを使っているチームメンバーもいますが、シェルスクリプトは正常に動作しますか？**
  - A: プロンプトに「クロスプラットフォームの制約事項」を設けているため、AIがよしなに処理してくれます。Windowsでは主にGit Bash環境で実行されるように作成されます。Node.js環境であれば、純粋なシェルスクリプトよりも`Husky` + `lint-staged`の組み合わせを推奨してもらうのが、互換性の面で最も安全です。

- **Q: スクリプトが重すぎて、コミットのたびに数分かかってしまいます。**
  - A: `npm run lint`や`npm run test`をコードベース全体に対して実行しているためです。プロンプトの注意事項セクションにあるように、**「必ずステージングされたファイル（変更されたファイル）のみを検査してほしい」**と明記してください。AIが`git diff --cached --name-only`を活用した軽量なスクリプトを再生成してくれます。

- **Q: すでに壊れてしまった大規模なコミットを取り消したいのですが、Hookが邪魔をして進めません。**
  - A: 緊急でHookを無視してコミット/プッシュしたい場合は、コマンドの後ろに `--no-verify` (または `-n`) フラグを付けます。（例: `git commit -m "fix: ホットフィックス" --no-verify`）ただし、乱用は厳禁です！
