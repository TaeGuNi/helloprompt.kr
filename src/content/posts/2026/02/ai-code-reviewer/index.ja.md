>
> **状況 (Context):**
>
> - 背景: 私はジュニア開発者として、新しい機能の実装を行いました。

>
> **タスク (Task):**
> 私が提供するコードを徹底的にレビューし、以下の構成に従ってフィードバックを提供してください。
>
> 1. **👏 称賛する点 (Good):** 優れたロジックや工夫が見られる箇所があれば、具体的に褒めてください。
> 2. **🛠 改善する点 (Improvements):** 可読性、パフォーマンス、セキュリティ、エラーハンドリング（エッジケース）の観点から、妥協なく厳しく指摘してください。
> 3. **🤔 思考を促す質問 (Questions):** 「なぜこの設計にしたのか？」「このデータ構造を選んだ理由は？」など、私のエンジニアとしての思考力を鍛えるような鋭い質問を投げかけてください。
> 4. **✨ リファクタリング (Refactoring):** 指摘事項をすべて反映させた、プロダクションレベルの美しいコードスニペットを提示してください。

> **制約事項 (Constraints):**
>
> - 口調は丁寧ですが、プロフェッショナルとして一切の妥協を排した断固とした態度 (Professional yet firm) を保ってください。
> - 変数の命名規則、関数の分割単位、さらにはスペースの使い方一つに至るまで、マイクロレベルで几帳面にチェックしてください。
> - 明確な根拠のない思い込みや推測による指摘（ハルシネーション）は絶対に避けてください。

## 💡 筆者からのコメント (Insight)

このプロンプトの真の価値は、単に「綺麗なコードを出力してくれる」ことではなく、**「なぜそのように書くべきなのか」という思考プロセス（Why）を学べる点**にあります。
特に「思考を促す質問」のセクションは非常に効果的です。AIからの鋭い質問に答えるつもりで自己分析をすることで、自分では気づけなかった設計の甘さや、エッジケースの考慮漏れに気づくことができます。
実務で新しい言語やフレームワークを学ぶ際にも、このプロンプトを使って自分が書いたコードをレビューさせると、公式ドキュメントを読む何倍ものスピードでベストプラクティスを吸収できるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: 業務用のコードをそのまま入力してもセキュリティ上問題ないですか？**
  - A: 機密情報（APIキー、パスワード、個人情報など）や、企業のコアとなる独自のアルゴリズムは絶対に入力しないでください。レビューを受けたい該当の関数やロジックの部分だけを切り抜き、一般的な変数名に置き換えてから入力することを強くお勧めします。

- **Q: どのAIモデルを使うのが一番良いですか？**
  - A: コーディングの文脈では、Claude 3.5 Sonnet が現在最も人間らしい自然なリファクタリングと的確なレビューを提供してくれます。次点で GPT-4o や Gemini 2.5 Pro も優れています。無料版のモデルでも動作しますが、Pro版モデルの方がアーキテクチャ全体を見据えた高度な指摘が期待できます。

## 🧬 プロンプト解剖 (Why it works?)

1. **トップティアのペルソナ設定:** 「Google/Amazonの15年目シニア」という役割を与えることで、AIの出力基準が一般的なレベルから「スケーラビリティと保守性を重視するグローバル標準」に引き上げられます。
2. **構造化された出力形式:** 称賛 → 改善点 → 質問 → 模範解答 というレビューの黄金律を強制することで、単なるコードの書き換えではなく、「メンタリング」としての機能を持たせています。
3. **思考の強制（Questions）:** 答えを教えるだけでなく質問を投げかけさせることで、ユーザーのプログラミングスキルの根本的な向上を促します。

## 📊 証明: Before & After

### ❌ Before (入力)

```python
def s(x):
    return x * x
```

### ✅ After (結果)

**👏 称賛する点 (Good):**
関数が単一の責任を持ち、非常に簡潔に書かれている点は評価できます。

**🛠 改善する点 (Improvements):**

1. **ネーミングルール:** 関数名 `s` と引数 `x` は、この関数が何をするのかという文脈を一切提供していません。可読性を著しく損なっています。
2. **型ヒント (Type Hints):** Python 3.5以降の標準である型ヒントが欠落しています。入力値と戻り値の型を明示することで、バグを未然に防ぎ、IDEのサポートを受けやすくなります。
3. **ドキュメンテーション (Docstring):** 関数の目的や使い方を説明するDocstringが存在しません。

**🤔 思考を促す質問 (Questions):**
もし、引数 `x` に数字ではなく文字列や `None` が渡された場合、この関数はどのように振る舞うべきでしょうか？そのような予期せぬ入力（エッジケース）に対するエラーハンドリングは考慮されていますか？

**✨ リファクタリング (Refactoring):**

```python
def calculate_square(number: float) -> float:
    """
    与えられた数値の二乗を計算して返します。

    Args:
        number (float): 二乗する対象の数値。

    Returns:
        float: 計算された二乗値。
    """
    return number ** 2
```
