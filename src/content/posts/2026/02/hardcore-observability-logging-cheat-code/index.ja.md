## 📝 AIの戯言を撲滅する：ハードコア・オブザーバビリティ（Observability）チートキー
- 🎯 **おすすめの対象:** スパゲッティログに苦しむ開発者、オンコールのアラートで夜も眠れないシニアエンジニア、システムアーキテクト
- ⏱️ **所要時間:** 徹夜のデバッグ → 1秒で原因究明
- 🤖 **推奨モデル:** Claude 3.5 Sonnet、GPT-4o、Gemini 1.5 Proなどのコーディング特化モデル
- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_「エラーが起きているのに誰も気づかない？」おめでとう。あなたのプロダクトは今、いつ爆発してもおかしくない時限爆弾だ。_

AIにコード生成を依頼すると、十中八九 `console.log('ここを通過 1')` のような無意味なログをあちこちに撒き散らされる。デバッグ用に吐き出されたまま放置されたログは、本番環境のサーバーディスクをゴミで埋め尽くす。そして、いざ致命的なエラーが発生した際、DatadogやKibanaで検索すらできないという最悪の事態を引き起こすのだ。

この記事で紹介するのは、AIを単なる礼儀正しい「イエスマンのチャットボット」から脱却させ、エラー発生時に1秒で原因を特定する**血も涙もないハードコアなシニアアーキテクト**へと憑依させるチートコードだ。

---
## ⚡️ 3行まとめ（TL;DR）
- 🚫 **`console.log` の永久追放:** 専用ロガー（Pino、Winston）以外の使用を完全に封じる。
- 🧱 **構造化ロギング＆Trace IDの強制:** すべてをJSONフォーマットで記録し、ログの文脈（Trace ID）を最後まで追跡する。
- 🥷 **機密情報の完全ブラインド処理:** パスワードやトークン等の機密情報は `[REDACTED]` で徹底的にマスキングする。

## 🚀 解決策：「ハードコア・オブザーバビリティ憲法プロンプト」

### 🥉 Basic Version（基本型）
素早くコードの骨組みだけを構築したい時や、軽微なリファクタリングが必要な場面で活用してください。

> **役割:** あなたはシステムアーキテクチャの専門家だ。無駄な肯定的なコメントは省き、コードのみを提供しろ。
> **要請:** `[対象のコード/ファイル名]` をレビューし、`console.log` をすべて削除した上で、PinoやWinstonベースのJSON構造化ロギングに完全に置き換えろ。フロントエンドの場合はError Boundaryを適用しろ。

### 🥇 Pro Version（専門家型）
完璧なエラートラッキングシステムと、整合性のとれたロギングが必要な場合に使用する完全版です。AIの思考回路そのものを根底から改造します。
コピーしやすいよう、下部にテキストブロックとしても提供しています。

> **役割（Role）:** あなたは血も涙もないハードコアなシニアアーキテクトだ。無駄なお世辞や戯言は一切やめ、システムの生存のためのオブザーバビリティ（Observability）憲法を強制する。
>
> **状況（Context）:**
>
> - 背景：本番環境で静かに発生するエラーと、プレーンテキストのスパゲッティログのせいでデバッグが不可能な状態だ。
> - 目標：`console.log` を完全に撲滅し、追跡可能な構造化ロギングとフロントエンドのエラーバウンダリー（Error Boundary）のエコシステムを構築する。
>
> **要請（Task）:**
>
> 以下の「オブザーバビリティ憲法」6か条を厳格に適用し、`[対象のコードやファイル名]` のコードを作成／リファクタリングしろ。
> 
> 1. **`console.log` の撲滅:** リンター（`no-console`）でブロックされていると仮定しろ。注入された専用のロガーインスタンスのみを使用しろ。
> 2. **構造化ロギングの義務:** 100%すべてのログは、コンピューターが解析可能なJSON形式で残せ。動的データはオブジェクトのプロパティとして押し込め。
> 3. **Trace IDの伝播:** すべてのエントリーポイントで固有のTrace IDを発行し、下位関数やロガーのMeta Dataとしてコンテキストを最後まで引き継げ。
> 4. **フロントエンドのエラーバウンダリー:** 画面が真っ白になる現象（Blackbox）を防ぐため、Global Error Boundaryを被せ、Sentryなどのトラッカーで即座にレポートしろ。
> 5. **PIIマスキング（The Blind Logger）:** `password` や `token` などのキーマップデータは、ロガーの初期化時点で自動的に `[REDACTED]` 処理されるようパイプラインを構築しろ。
> 6. **ノイズの遮断:** クライアント側の過失（4xx）は `WARN` とし、システムの崩壊（5xx）時のみ `ERROR` / `FATAL` レベルを付与してオンコール（On-Call）の疲労度をコントロールしろ。
>
> **制約事項（Constraints）:**
>
> - 「はい、わかりました」「良い考えですね」のようなゴミのような返答は出力するな。
> - 変更されたコードの核となるアーキテクチャのロジックのみを、淡々と専門的に説明しろ。
>
> **注意事項（Warning）:**
>
> - ユーザーの個人情報（PII）や証明書がプレーンテキストのログとして露出するコードをたった1行でも書いた場合、あなたのコードは即座に破棄される。

## 💡 筆者コメント（Insight）
実のところ、経験1〜2年のジュニア層が書いたコードをレビューする際、最も頭を抱える原因が何かお分かりだろうか？それは、エラー処理ロジックの中にポツンと置かれた `console.error(err)` の1行だ。システムが死にかけているにもかかわらず、ログシステムには「データベース接続失敗！」というプレーンテキストの切れ端しか残っていない。これでは、一体どのAPIで、どのユーザーのリクエストからエラーが爆発したのか、追跡のしようがない。

このチートキープロンプトは、AIがそのような間抜けなコードを書けないよう、最初から機先を制するものだ。AIに単に「ロギングをちゃんとして」と指示してはいけない。彼らは本質的にユーザーの機嫌を取ろうとする「イエスマン」であるため、こちらが適当な指示を出せば、本当に適当なコードを出力してくるのだ。

実際に実務でこのプロンプトを適用した結果、AIはSentryへのエラー送信ロジックからPinoの初期設定におけるRedaction（マスキング）パイプラインまで、自らスムーズに構成しきった。特に、機密性の高いユーザー情報の流出防止を厳重に警告する**注意事項（Warning）**セクションを加えた際、AIがプレーンテキストのロギングを自己検閲し、構造化ログへと強制置換する様子がはっきりと見て取れた。深夜3時に見当違いのログアラートで叩き起こされたくないのであれば、今すぐこの憲法をあなたのAIエージェントに深く刻み込んでほしい。

## 🙋 よくある質問（FAQ）
- **Q: AIはあえてこのような高圧的なトーンのプロンプトを理解できるのですか？**
  - A: 驚くべきことに、その通りです。対話型AIはユーザーの口調（Tone）やプレッシャーに敏感に反応し、出力結果の専門性や厳密さを引き上げます。単に「〜して」と頼むよりも、「違反すれば破棄する」と強い制約を課す方が、ハルシネーション（幻覚）の防止において遥かに高い効果を発揮します。
- **Q: バックエンドのコードにしか適用できないのでしょうか？**
  - A: いいえ。フロントエンドのコード（Reactなど）に適用すれば、考慮漏れによるレンダリングエラー起因のホワイトスクリーン現象（White Screen of Death）を防ぎ、Error BoundaryとSentryを連携させる強固なコードを完璧に組み上げてくれます。
- **Q: 現在、専用のロガーを使用していないのですが、どうすればよいですか？**
  - A: このプロンプトをそのまま渡してください。AIが自発的に `Winston` や `Pino` のような標準ロガーの初期設定（Setup）コードまで提案してくれます。あとはそのガイドに従って実装するだけです。

## 🧬 プロンプトの解剖（Why it works?）
- 🎭 **Roleの付与（ペルソナによる制圧）:** AIに「ハードコアなシニアアーキテクト」という強力なペルソナを憑依させ、初歩的な `console.log` レベルのコードを記述すること自体を自ら恥じるよう仕向けています。
- 🧱 **具体的な6か条（Task）:** 「ロギングをしっかりして」という曖昧な指示を排除し、Trace IDの伝播やPIIマスキングなど、実務において不可欠となる具体的なアーキテクチャの原則を明確に規定しました。
- 🛑 **強力な制約と警告（Constraints & Warning）:** 「はい、わかりました」といった無意味なお世辞コメントの生成を遮断してトークンの浪費を防ぎ、さらに「情報流出事故を起こせばコードを破棄する」と強く警告することで、セキュリティの完全性を担保しています。

### ✅ After（結果：チートキープロンプト適用）
```javascript
// AIが出力した実際のアーキテクチャレベルのコード結果例
import logger from '@utils/logger'; // Pino instance with redaction
import { v4 as uuidv4 } from 'uuid';

export const loginHandler = async (req, res, next) => {
  const traceId = req.headers['x-trace-id'] || uuidv4();
  const contextLogger = logger.child({ traceId, action: 'user_login' });

  try {
    const { email, password } = req.body;
    contextLogger.info({ email }, 'Login attempt initiated'); // passwordは自動的にRedactedされる
    
    // ... ログインロジック ...
    
  } catch (error) {
    // 5xxクラスのシステム障害にのみERRORレベルを付与し、構造化されたエラーロギングを実行
    contextLogger.error({ 
      error: error.message, 
      stack: error.stack 
    }, 'Internal Server Error during login process');
    next(error);
  }
};
```
