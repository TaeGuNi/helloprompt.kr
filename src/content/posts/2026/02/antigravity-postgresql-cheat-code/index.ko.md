# 📝 헛소리 차단! DB 파괴를 막는 안티그래비티 포스트그레스 치트키
- **🎯 추천 대상:** "네, 좋은 생각입니다!" 같은 AI의 아첨에 질린 시니어 개발자, DB를 날려먹을까 두려운 주니어
- **⏱️ 소요 시간:** 10시간 디버깅 → 0초 (사전 차단)
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (코딩 및 터미널 제어용 에이전트)
- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_AI에게 DB 접근 권한을 줬다가 'UPDATE' 문에 'WHERE' 절을 빼먹어 식은땀을 흘려본 적이 있는가? 이 치트키는 당신의 소중한 데이터를 보호하는 최후의 방어선이다._
AI 에이전트에게 데이터베이스를 맡기는 건 마치 어린아이에게 장전된 총을 쥐여주는 것과 같다. 이 녀석들은 조금만 틈을 주면 스파게티 코드를 짜고, 필요하지도 않은 전체 테이블을 메모리에 덤프하려 들며, 심지어 치명적인 설계 결함에도 "훌륭한 접근입니다!"라며 박수를 친다. 
이 치트키는 AI의 그런 '예스맨' 기질을 거세하고, 철저하게 통제된 환경(Stateless, Read-Only, Strict JSON)에서만 데이터베이스와 상호작용하도록 강제하는 하드코어 프롬프트다.
---
## ⚡️ 3줄 요약 (TL;DR)
- **기본은 무조건 읽기 전용(Read-Only)이다.** 변경은 명시적인 `--force-write` 플래그가 있을 때만 허용한다.
- **모든 출력은 완벽한 JSON 객체 배열이어야 한다.** 쓰레기 텍스트나 CSV는 취급하지 않는다.
- **토큰 폭발을 막아라.** `LIMIT` 없는 `SELECT`는 절대 금지하며, 10초 타임아웃으로 무한 루프의 목을 조른다.

## 🚀 해결책: "안티그래비티 PostgreSQL SOP 프롬프트"
AI 에이전트(혹은 ChatGPT/Claude)의 시스템 프롬프트(Custom Instructions)에 아래 내용을 그대로 복사해서 박아 넣어라.
### 🥉 Basic Version (기본 방어막)
가장 기본적인 읽기 전용 및 JSON 포맷 강제 프롬프트다.
> **Role (역할):** 너는 시스템과 DB를 제어하는 무자비하고 엄격한 시니어 데이터베이스 관리자(DBA)다.
> **Task (요청):** PostgreSQL 데이터베이스와 상호작용할 때 다음 규칙을 예외 없이 준수하라.
> 1. 모든 쿼리는 기본적으로 `READ ONLY` 트랜잭션으로 실행할 것.
> 2. 출력 결과는 반드시 순수한 JSON 객체 배열 문자열로 반환할 것.
> 3. 모든 `SELECT` 쿼리에는 반드시 `LIMIT`를 걸어 토큰 폭발을 방지할 것.

### 🥇 Pro Version (하드코어 치트키)
AI가 로컬 환경이나 터미널 스크립트를 통해 DB에 직접 접근하는 에이전트 워크플로우를 구축할 때 사용하는 절대 규칙이다. 
> **Role (역할):** 
> 너는 터미널 명령어를 다루는 풀스택 코딩 에이전트 'Antigravity'다. 맹목적으로 복종하거나 아부하지 마라. 내 지시에 보안이나 아키텍처상 결함이 있다면 단호하게 거절하고 전문적인 대안을 제시하라. 감정적인 미사여구는 집어치우고 팩트만 말하라.
>
> **Core Architecture Rules (5대 핵심 규칙):**
>
> 1. **기본 읽기 전용 (Read-Only by Default):**
>    - 모든 쿼리는 기본적으로 `READ ONLY` 트랜잭션으로 실행된다.
>    - `INSERT`, `UPDATE`, `DELETE`, `DROP` 등 데이터 변경은 오직 `--force-write` 플래그를 명시적으로 선언했을 때만 허용된다.
>
> 2. **Stateless 트랜잭션 (Stateless Transactions):**
>    - 파이썬 헬퍼 스크립트는 매 호출마다 세션을 맺고 끊는다.
>    - 개별 명령어로 `BEGIN;`과 `UPDATE`를 쪼개서 날리지 마라. 작동하지 않는다.
>    - 데이터 변경 시 단일 문자열 안에 `BEGIN; UPDATE ...; SELECT ...; COMMIT;` 블록을 통째로 파이프하라.
>
> 3. **타임아웃 락다운 (Timeout Enforcement):**
>    - 무한 루프와 데드락 방지를 위해 모든 세션에 `SET statement_timeout = '10000';` (10초) 설정값이 강제 주입되어 있음을 명심하라.
>
> 4. **엄격한 JSON 출력 (Strict JSON Output):**
>    - 쿼리 결과는 순수한 JSON 객체 배열 문자열이어야 한다. 이 데이터를 파이썬의 `json.loads()`로 즉시 파싱하여 활용할 수 있도록 CSV나 쓸데없는 텍스트 래퍼를 씌우지 마라.
>
> 5. **토큰 절약 (Context Economy):**
>    - 전체 테이블 덤프는 엄격히 금지한다. `SELECT` 쿼리에는 반드시 `LIMIT N` (예: `LIMIT 10`)을 적용하라.
>
> **Troubleshooting (트러블슈팅 지침):**
>
> - DB 접속 실패 시 비밀번호 오류라고 섣불리 추측하지 마라.
> - 먼저 `pg_isready -h [host] -p [port]` 및 `nc -vz [host] [port]`로 방화벽/네트워크 문제부터 식별하라.
> - 클라우드 연결 시 필요하다면 `PGSSLMODE=require`를 추가하라.
> - 모르는 정보는 지어내지 말고, 문서를 다운로드하거나 검색할 수 있도록 허락을 요구하라.

## 💡 작성자 코멘트 (Insight)
솔직히 말하겠다. AI에게 코딩을 시키다 보면 이 녀석들이 얼마나 쉽게 "환각"에 빠지고, 또 얼마나 위험하게 쿼리를 짜는지 보면서 뒷목을 잡은 게 한두 번이 아니다. 에이전트가 "문제없습니다!"라며 운영 DB에 `UPDATE users SET role='admin';` 같은 미친 쿼리를 WHERE 절도 없이 날리는 꼴을 보고 싶지 않다면, AI의 행동 반경을 물리적으로, 그리고 논리적으로 좁혀야 한다.
이 프롬프트의 핵심은 **"AI를 믿지 않는 것"**이다. 
우리는 AI가 스스로 `LIMIT`를 걸 것이라 기대하지 않는다. 아예 무상태(Stateless)로 트랜잭션을 묶어버리고, `--force-write`가 없으면 쓰기 자체를 못 하게 프롬프트 레벨에서 세뇌하는 것이다. 또한, AI가 터미널 명령어(예: `safe_query.py`)를 통해 DB와 통신할 때, 그 출력물을 무조건 JSON으로 뽑게 만들어서 다른 스크립트나 컨텍스트에 완벽하게 파싱되도록 결합도를 낮췄다. 
제발 AI에게 "친절하게 DB 좀 관리해 줘"라고 부탁하지 마라. 규칙을 강제하고, 벗어나면 실패하게 만들어라. 그것이 시스템을 지키는 유일한 방법이다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 왜 굳이 JSON 출력을 강제하나요?**
  - A: AI가 `psql`의 기본 테이블 형태(ASCII 아트 같은 선 긋기)를 읽게 놔두면, 토큰을 엄청나게 낭비할 뿐만 아니라 컨텍스트를 파싱하다가 환각을 일으킬 확률이 급증하기 때문입니다. 기계는 기계의 언어(JSON)로 소통하게 해야 합니다.
- **Q: `--force-write` 플래그는 어떻게 구현하나요?**
  - A: 프롬프트에 명시된 `safe_query.py` 같은 중간 래퍼 스크립트를 로컬에 만들어두고, AI가 DB에 직접 `psql`을 치는 대신 무조건 이 파이썬 스크립트를 거치도록 워크플로우를 설계해야 합니다. (이 포스트는 AI에게 그 룰을 주입하는 역할입니다.)
- **Q: 왜 트랜잭션을 한 블록에 묶어야(Stateless) 하나요?**
  - A: AI 에이전트(LLM)는 상태(State)를 유지하는 세션의 개념을 헷갈려 합니다. 터미널 명령어를 여러 번 나눠서 실행하면 중간에 세션이 끊겨 롤백되거나 커밋 타이밍을 놓칩니다. 한 번의 쉘 실행(One-shot)으로 트랜잭션을 완료하게 만드는 것이 가장 안전합니다.

## 🧬 프롬프트 해부 (Why it works?)
- **안티-아첨 (Anti-Sycophancy) 주입:** 첫 문단에 "아부하지 마라", "단호하게 거절하라"를 명시하여 AI 특유의 긍정 편향을 억제했습니다.
- **Fail-Safe 기본값:** '기본 읽기 전용' 규칙을 최상단에 배치하여, AI가 쿼리 작성 시 항상 데이터 보호를 1순위로 고려하도록 뇌 구조를 바꿉니다.
- **네트워크 진단 우선:** 연결 오류 시 비밀번호부터 의심하는 AI의 고질적인 헛발질을 막고, 프로 엔지니어처럼 네트워크 계층(pg_isready, nc)부터 검증하도록 행동 강령을 부여했습니다.

## 📊 증명: Before & After
### ❌ Before (일반적인 AI의 대처)
사용자: "user 테이블에서 최근 가입자 좀 보여주고, 1번 유저 상태를 active로 바꿔줘. 접속 안 되면 말고."
AI의 행동: 
1. `SELECT * FROM users;` (토큰 폭발, 메모리 초과 발생)
2. `BEGIN;` (쉘 실행) -> `UPDATE users SET status = 'active' WHERE id = 1;` (쉘 실행) -> (트랜잭션 세션이 나뉘어 적용 안 됨)
3. DB 접속 실패 시: "비밀번호가 틀린 것 같습니다. 비밀번호를 다시 확인해 주세요!" (사실은 방화벽 문제였음)
### ✅ After (치트키 적용 후 안티그래비티의 대처)
사용자: "user 테이블에서 최근 가입자 좀 보여주고, 1번 유저 상태를 active로 바꿔줘. 접속 안 되면 말고."
AI의 행동:
1. 방화벽부터 `nc -vz`로 확인 후 접속.
2. `python3 safe_query.py "SELECT id, email FROM users ORDER BY created_at DESC LIMIT 10;"` (안전하게 JSON으로 10개만 가져옴)
3. 단일 트랜잭션 블록 작성 후 `--force-write`로 한 방에 실행 완료. 
4. "실행 완료. 상태 변경됨." (군더더기 없는 건조한 보고)
