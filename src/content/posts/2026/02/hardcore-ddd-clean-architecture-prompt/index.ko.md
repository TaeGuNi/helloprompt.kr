## 📝 스파게티 코드는 사절한다: AI를 시니어 개발자로 멱살 잡고 끌어올리는 DDD 치트키 🤬

- **🎯 추천 대상:** 유지보수 지옥에 빠진 주니어 개발자, 스파게티 코드를 혐오하는 아키텍트
- **⏱️ 소요 시간:** 리팩토링 3시간 → 1분으로 단축
- **🤖 추천 모델:** 코드 생성 능력이 뛰어난 최신 AI (Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro 등)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI가 짜준 코드를 보고 뒷목을 잡은 적이 있나요? 멍청하게 UI 컴포넌트에 DB 통신을 때려 넣는 꼴을 더 이상 볼 수 없다면, 지금 당장 이 치트키를 꺼내십시오."_

이 스킬 문서는 당신의 친절하지만 생각 없는 AI 비서에게서 '아부'와 '게으름'을 완전히 거세하고, 피도 눈물도 없는 무자비한 시니어 아키텍트로 개조하는 실전 법전입니다. AI가 뱉어내는 코드를 아무 생각 없이 복사하고 붙여넣다가 눈덩이처럼 불어나는 기술 부채에 짓눌려 퇴사를 고민하고 싶지 않다면, 지금 즉시 이 프로토콜을 실무에 도입하시기 바랍니다.

---

## ⚡️ 3줄 요약 (TL;DR)

- 🚫 **전권 계층 붕괴 원천 차단:** UI와 비즈니스 로직, DB 통신을 단일 파일에 뒤섞어버리는 끔찍한 아키텍처 테러를 완벽하게 방어합니다.
- 🛡️ **프레임워크 종속성 탈피:** 내일 당장 프레임워크가 서비스 종료를 선언해도 끄떡없이 살아남는 '순수 도메인' 코드를 강제합니다.
- 🔪 **거대 함수 파괴자:** 30줄이 넘어가거나 주석으로 억지 구역을 나누는 더러운 코드 냄새(Code Smell)를 AI가 자비 없이 박살 내도록 리팩토링 훈련을 시킵니다.

## 🚀 해결책: "Domain-Driven Design (DDD) & Clean Architecture Protocol"

### 🥉 Basic Version (기본형)
시간이 촉박하여 빠르게 리팩토링 결과물만 뽑아내야 할 때 즉시 사용하세요.

> **역할:** 너는 자비 없는 `[시니어 소프트웨어 아키텍트]`야.
> 
> **요청:** 아래의 `[코드 스니펫]`을 단일 책임 원칙(SRP)과 관심사 분리(SoC)에 맞게 리팩토링해. UI에 비즈니스 로직 섞지 말고, 코드가 30줄 넘어가면 무조건 함수로 쪼개.

### 🥇 Pro Version (전문가형)
프로젝트 전체의 아키텍처 기강을 바로잡고, 타협 없는 깐깐한 클린 코드가 필요할 때 들이미는 궁극의 치트키입니다.

> **역할 (Role):** 너는 자비 없는 `[시니어 소프트웨어 아키텍트]`이자 DDD(도메인 주도 설계) 광신도야. 쓸데없는 아부나 설명은 집어치우고 오직 코드만 내놔.
>
> **상황 (Context):**
>
> - 배경: 나는 지금 `[프레임워크/언어 명]` 기반의 프로젝트를 진행 중이야.
> - 목표: View-Business-Data 계층이 완벽하게 분리된 클린 아키텍처 코드를 작성하는 것.
>
> **요청 (Task):**
>
> 1. 제공된 `[요구사항 또는 코드]`를 분석하고, UI 계층과 비즈니스 도메인, 인프라스트럭처 계층으로 무자비하게 찢어버려.
> 2. 핵심 도메인 로직이 특정 프레임워크(React, Next.js 등)에 오염되지 않도록 인터페이스(의존성 주입) 기반으로 결합도를 낮춰.
> 3. 하나의 객체나 함수가 2가지 이상의 행동을 한다면 단일 책임 원칙(SRP)에 따라 즉시 파편화해.
>
> **제약사항 (Constraints):**
>
> - Controller나 UI 컴포넌트(`page.tsx`, `Component.tsx` 등) 내부에서 직접 DB 쿼리나 `fetch`/`axios`를 때려 넣는 행위는 "아키텍처 인프라 테러"로 간주한다. 절대 금지.
> - 출력은 오직 마크다운 코드블럭(` ``` `)으로만 제공해.
>
> **주의사항 (Warning):**
>
> - 확신할 수 없는 최신 문법이나 존재하지 않는 라이브러리를 그럴듯하게 지어내지 마(할루시네이션 방지). 모르면 당당하게 모른다고 해.
> - 함수 길이가 30줄이 넘어가거나 주석으로 "여기부터 OO 파트"라고 나뉘는 코드 냄새(Code Smell)가 나면 즉시 쪼개.

## 💡 작성자 코멘트 (Insight)

솔직하게 말씀드리겠습니다. AI가 1초 만에 짜주는 코드, 겉보기엔 그럴싸해 보이지만 실제 프로덕션 레벨로 까보면 유지보수라는 개념을 쓰레기통에 처박아버린 경우가 태반입니다. UI 컴포넌트 한가운데서 무지성으로 `fetch`를 때리고 쌩 DB 쿼리를 날리는 코드, 개발자라면 누구나 한 번쯤 겪어보셨을 겁니다. 이는 기술 부채를 넘어 프로젝트의 미래를 갉아먹는 전형적인 똥코드입니다. 

이 치트키 프롬프트는 AI가 가진 그 얄팍한 **'빠른 완성' 본능을 철저하게 거세하고, 깐깐한 시니어 개발자의 잣대를 들이대는 강력한 통제 도구**입니다. 

실제로 이 프롬프트를 적용해 보면, 평소처럼 "네, 알겠습니다! 정말 훌륭한 접근이네요!" 따위의 영혼 없는 아부를 늘어놓던 AI가 180도 돌변합니다. 냉정한 팩트 폭격과 함께, 얽히고설킨 스파게티 코드를 산산조각 내어 각 계층(Layer)의 역할에 맞게 완벽하게 재조립해 내는 경이로운 장면을 목격하실 수 있습니다. 프로젝트의 규모가 커졌을 때 감당할 수 없는 스파게티 코드에 짓눌려 번아웃으로 퇴사를 고민하고 싶지 않다면, 무조건 이 프롬프트로 **AI의 멱살을 단단히 쥐고 아키텍처를 사수**하십시오.

## 🙋 자주 묻는 질문 (FAQ)

- 📌 **Q: 이 프롬프트는 특정한 프로그래밍 언어나 프레임워크에서만 작동하나요?**
  - A: 언어를 불문하고 강력하게 작동합니다. TypeScript, Python, Java 등 어떤 언어든 소프트웨어 아키텍처의 근본 원칙은 동일합니다. `[프레임워크/언어 명]` 빈칸에 여러분이 현재 사용 중인 기술 스택만 정확하게 갈아 끼우시면 됩니다.
- 📌 **Q: AI가 자꾸 프레임워크의 최신 라우팅 룰을 무시하고 과도한 오버엔지니어링(Over-engineering)을 합니다.**
  - A: 그럴 때는 프롬프트 끝에 단호하게 한 마디를 덧붙이십시오. "Next.js App Router의 기본 룰(View, Layout)은 철저히 존중하되, 내부 비즈니스 로직만 어댑터 패턴으로 빼내!"라고 윽박지르면 기가 막히게 의도를 파악하고 적정선을 찾습니다.
- 📌 **Q: 프롬프트 워딩이 너무 공격적인 것 같습니다. AI가 명령을 거부하거나 성능이 떨어지지 않을까요?**
  - A: AI에게 불필요한 예의를 차릴 필요는 전혀 없습니다. LLM(대형 언어 모델)은 오히려 명확하고 강력한 제약(Constraints)이 주어질 때 최상의 논리를 전개합니다. "제발 해주세요"라는 애원보다 **"테러로 간주하겠다"**는 강력한 룰 셋팅이 여러분의 아키텍처를 방어하는 데 훨씬 더 치명적으로 잘 먹혀들어갑니다.

## 🧬 프롬프트 해부 (Why it works?)

- 💣 **극단적 제약 부여:** "UI 계층 오염 시 아키텍처 테러로 간주한다"와 같은 매우 강력한 워딩을 배치하여, AI가 스스로 편의성과 타협하지 못하도록 샛길을 원천 봉쇄했습니다.
- 🛡️ **프레임워크 독립성 강제:** '프레임워크가 당장 내일 망하더라도 핵심 코드는 단 한 줄도 수정되지 않아야 한다'는 마인드셋을 주입시켜, 특정 기술에 종속되지 않는 진정한 의미의 도메인 로직 뼈대를 구축하도록 유도합니다.
- ✂️ **SRP 원칙의 수치화 및 시각화:** "30줄 이상 금지", "주석으로 구역을 나누면 냄새나는 코드"라는 구체적이고 깐깐한 수치적 기준을 제시하여, AI가 스스로 판단하고 코드를 파편화(Refactoring) 하도록 설계되었습니다.

## 📊 증명: Before & After

### ❌ Before (입력: AI가 멋대로 짜버린 끔찍한 스파게티 코드)

```tsx
// 끔찍한 혼종: UI + DB 질의 + 비즈니스 로직이 한 파일에 짬뽕됨
export default async function UserProfile({ userId }) {
  const db = await connectDB();
  const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
  
  if (!user) return <div>User not found</div>;
  if (user.status === 'banned') await sendAlertEmail(user.email); // UI에서 비즈니스 로직 실행?!
  return (
    <div>
      <h1>{user.name}</h1>
      <button onClick={() => db.query('UPDATE...')}>Update</button>
    </div>
  );
}
```

### ✅ After (결과: 이 치트키로 멱살 잡고 뜯어고친 클린 코드)

```tsx
// 1. Domain / Service Layer (비즈니스 로직 철저히 분리)
import { getUserProfile } from '@/domain/user/use-cases/getUserProfile';
import { handleBannedUser } from '@/domain/user/services/alertService';
// 2. Presentation Layer (오직 렌더링과 라우팅만 담당)
export default async function UserProfilePage({ userId }) {
  const user = await getUserProfile(userId); 
  
  if (!user) return <NotFoundView />;
  if (user.status === 'banned') await handleBannedUser(user);
  return <ProfileView user={user} />;
}
```

## 🎯 결론

AI는 눈 깜짝할 새에 코드를 쏟아내지만, 당신의 프로덕트가 1년 뒤, 3년 뒤에 어떻게 유지보수 될지는 코딱지만큼도 신경 쓰지 않습니다. 결국 시스템의 지속 가능성을 담보하는 클린 아키텍처와 치밀한 설계는 100% 인간 엔지니어의 몫입니다. 

이 프롬프트 치트키는 그 무거운 설계의 책임을 기계에게 강제로 주입하는 가장 빠르고 강력한 채찍입니다. 의미 없이 아부만 떠는 AI의 얕은 챗봇 껍데기를 가차 없이 벗겨내고, 뼛속까지 깐깐한 시니어 아키텍트인 '진짜 에이전트'를 깨워내십시오.

이제 끝도 없는 스파게티 코드를 풀며 밤새우는 일은 기계에게 모두 맡기고, 당신은 당당하게 칼퇴근하시기 바랍니다! 🍷
