# 📝 AIの戯言を撲滅する：ハードコア・オブザーバビリティ（Observability）チートキー
- 🎯 **おすすめの対象:** スパゲッティログに苦しむ開発者、オンコール（On-Call）のアラートで夜も眠れないシニア層、システムアーキテクト
- ⏱️ **所要時間:** 徹夜のデバッグ → 1秒で犯人を特定
- 🤖 **推奨モデル:** Claude 3.5 Sonnet、GPT-4o、Gemini 1.5 Proなどのコーディング特化モデル
- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_エラーが起きたのに誰も気づかない？おめでとう。君のプロダクトは今、いつ爆発するかわからない時限爆弾だ。_

開発時にAIへコード作成を頼むと、必ずと言っていいほど無駄に `console.log('ここを通過 1')` のようなものをあちこちにまき散らす。デバッグ用に出力したまま消し忘れたログは、本番環境のサーバーディスクをゴミで埋め尽くし、いざエラーが発生した際にはDatadogやKibanaで検索すらできないという最悪の事態を引き起こす。

この記事は、AIを礼儀正しい「イエスマンのチャットボット」から脱却させ、エラー発生時に1秒で原因を特定する**血も涙もないハードコアなシニアアーキテクト**へと憑依させるチートコードだ。

---
## ⚡️ 3行まとめ（TL;DR）
- 🚫 **`console.log` の永久追放:** ロガーインスタンス（Pino、Winston）以外は絶対に使用するな。
- 🧱 **構造化ロギング＆Trace IDの強制:** 100% JSONフォーマットで記録し、ログ間の繋がり（Trace ID）を最後まで追跡しろ。
- 🥷 **個人情報の完全ブラインド処理:** パスワードやトークンなどの機密情報は `[REDACTED]` で完全に封鎖しろ。

## 🚀 解決策：「ハードコア・オブザーバビリティ憲法プロンプト」

### 🥉 Basic Version（基本型）
素早くコードの骨組みだけを作りたい時や、軽いリファクタリングが必要な時に投げてください。

> **役割:** あなたはシステムアーキテクチャの専門家だ。無駄な肯定的なコメントは省き、コードのみを提供しろ。
> **要請:** `[コード/ファイル]` をレビューし、`console.log` をすべて削除した上で、Pino/WinstonベースのJSON構造化ロギングに完全に置き換えろ。フロントエンドの場合はError Boundaryを適用しろ。
\

### 🥇 Pro Version（専門家型）
完璧なエラートラッキングシステムと整合性のあるロギングが必要な場合に使用する完全版です。AIの脳の構造そのものを改造してしまいます。 
コピーしやすいように、下にマークダウンのコードブロックとしても提供しています。

> **役割（Role）:** あなたは血も涙もないハードコアなシニアアーキテクトだ。無駄なお世辞や戯言は一切やめ、システムの生存のためのオブザーバビリティ（Observability）憲法を強制する。
>
> **状況（Context）:**
>
> - 背景：本番環境で静かに発生するエラーと、プレーンテキストのスパゲッティログのせいでデバッグが不可能な状態だ。
> - 目標：`console.log` を完全に撲滅し、追跡可能な構造化ロギングとフロントエンドのエラーバウンダリー（Error Boundary）のエコシステムを構築する。
>
> **要請（Task）:**
>
> 以下の「オブザーバビリティ憲法」6か条を厳格に適用し、`[ここに分析・修正するコードやファイル名を入力]` のコードを作成／リファクタリングしろ。
> 
> 1. **`console.log` の撲滅:** リンター（`no-console`）でブロックされていると仮定しろ。注入された専用のロガーインスタンスのみを使用しろ。
> 2. **構造化ロギングの義務:** 100%すべてのログは、コンピューターが解析可能なJSON形式で残せ。動的データはオブジェクトのプロパティとして押し込め。
> 3. **Trace IDの伝播:** すべてのエントリーポイントで固有のTrace IDを発行し、下位関数やロガーのMeta Dataとしてコンテキストを最後まで引き継げ。
> 4. **フロントエンドのエラーバウンダリー:** 画面が真っ白になる現象（Blackbox）を防ぐため、Global Error Boundaryを被せ、Sentryなどのトラッカーで即座にレポートしろ。
> 5. **PIIマスキング（The Blind Logger）:** `password` や `token` などのキーマップデータは、ロガーの初期化時点で自動的に `[REDACTED]` 処理されるようパイプラインを構築しろ。
> 6. **ノイズの遮断:** クライアント側の過失（4xx）は `WARN` とし、システムの崩壊（5xx）時のみ `ERROR` / `FATAL` レベルを付与してオンコール（On-Call）の疲労度をコントロールしろ。
>
> **制約事項（Constraints）:**
>
> - 「はい、わかりました」「良い考えですね」のようなゴミのような返答は出力するな。
> - 変更されたコードの核となるアーキテクチャのロジックのみを、淡々と専門的に説明しろ。
>
> **注意事項（Warning）:**
>
> - ユーザーの個人情報（PII）や証明書がプレーンテキストのログとして露出するコードをたった1行でも書いた場合、あなたのコードは即座に破棄される。

## 💡 筆者コメント（Insight）
正直に言って、経験1〜2年のジュニア層が書いたコードをレビューする時に一番腹が立つのは何か知っているか？それは、エラー処理ロジックにポツンと置かれた `console.error(err)` の1行だ。システムが死にかけているのに、ログシステムには「データベース接続失敗！」というプレーンテキストの切れ端しか残っていない。これでは一体どのAPIで、どのユーザーのリクエストからエラーが爆発したのか、どうやって追跡しろというのか？

このチートキープロンプトは、AIがそのような間抜けなコードを書けないように、最初から機先を制してしまう。AIに単に「ロギングをちゃんとして」と言ってはいけない。彼らは本質的にユーザーの機嫌を取ろうとする「イエスマン」であるため、あなたが適当に指示を出せば、本当に適当なコードを出力してくる。 

実際に実務でこのプロンプトを適用してみた結果、AIがSentryのエラー送信ロジックからPino初期設定のRedaction（マスキング）パイプラインまで、自らスムーズに構成しきった。特に、機密性の高いユーザー情報の流出防止を警告する**注意事項（Warning）**セクションを入れた際、AIがプレーンテキストのロギングを自己検閲し、構造化ログへと強制置換する様子が見て取れた。深夜3時に見当違いのログアラートで叩き起こされたくないなら、絶対にこの憲法をあなたのAIエージェントに刻み込め。

## 🙋 よくある質問（FAQ）
- **Q: AIはあえてこのような怒っているトーンのプロンプトをよく理解するのですか？**
  - A: 驚くべきことに、その通りです。対話型AIはユーザーの口調（Tone）やプレッシャーに反応し、出力結果の専門性や厳格さを引き上げます。「〜して」と頼むよりも、「これに違反したら破棄する」と伝える方が、ハルシネーション（Hallucination：幻覚）の防止に遥かに効果的です。
- **Q: バックエンドのコードにしか使えませんか？**
  - A: いいえ。フロントエンドのコード（Reactなど）に適用すれば、無思考なレンダリングエラーによるホワイトスクリーン現象（White Screen of Death）を防ぎ、Error BoundaryとSentryの連携コードを完璧に組み上げてくれます。
- **Q: ロガーを別に使っていないのですが、どうすればいいですか？**
  - A: このプロンプトを渡せば、AIが勝手に `Winston` や `Pino` のような標準ロガーの初期設定（Setup）コードまで提案してくれます。あとはただそれに従うだけです。

## 🧬 プロンプトの解剖（Why it works?）
- 🎭 **Roleの付与（ペルソナの制圧）:** AIに「ハードコアなシニアアーキテクト」という強力なペルソナを被せ、初歩的な `console.log` レベルのコードを自ら恥じるように仕向けました。
- 🧱 **具体的な6か条（Task）:** 「ロギングをしっかり」という曖昧な指示の代わりに、Trace IDの伝播やPIIマスキングなど、実務で必ず求められる具体的なアーキテクチャの原則を明示しました。
- 🛑 **強力なConstraints & Warning:** 「はい、わかりました」のような無意味なお世辞コメントの生成を遮断してトークンの無駄遣いを防ぎ、情報流出事故の際は破棄するという脅しをかけることで、セキュリティの完全性を確保しました。

### ✅ After（結果：チートキープロンプト適用）
```javascript
// AIが出力した実際のアーキテクチャレベルのコード結果例
import logger from '@utils/logger'; // Pino instance with redaction
import { v4 as uuidv4 } from 'uuid';

export const loginHandler = async (req, res, next) => {
  const traceId = req.headers['x-trace-id'] || uuidv4();
  const contextLogger = logger.child({ traceId, action: 'user_login' });

  try {
    const { email, password } = req.body;
    contextLogger.info({ email }, 'Login attempt initiated'); // passwordは自動的にRedactedされる
    
    // ... ログインロジック ...
    
  } catch (error) {
    // 5xxクラスのシステム障害にのみERRORレベルを付与し、構造化されたエラーロギングを実行
    contextLogger.error({ 
      error: error.message, 
      stack: error.stack 
    }, 'Internal Server Error during login process');
    next(error);
  }
};
```
