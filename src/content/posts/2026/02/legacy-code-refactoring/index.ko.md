## 📝 레거시 코드 청소기: 리팩토링 & 테스트 자동화

- **🎯 추천 대상:** 주니어 개발자, 레거시 프로젝트 유지보수 담당자, 코드 리뷰어
- **⏱️ 소요 시간:** 코드 분석 및 파악 1시간 → 5분 이내 단축
- **🤖 최상위 성능:** Claude 3.5 Sonnet, GPT-4o (복잡한 로직 및 긴 컨텍스트 파악에는 Claude를 강력히 추천)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"도대체 이 코드는 누가 짠 거야? (...아, 1년 전의 나였구나.)"_

복잡하게 얽힌 `if-else` 중첩 지옥, 도무지 의도를 파악할 수 없는 암호 같은 변수명, 그리고 어떤 부작용(Side Effect)을 낳을지 모르는 문서화되지 않은 예외 처리들. 건드리는 순간 전체 시스템이 주저앉을 것만 같은 **레거시 코드(Legacy Code)** 앞에서 밤을 지새운 적이 있으신가요? 코드를 열어보는 것 자체가 두려운 순간, 개발자라면 누구나 한 번쯤 겪어보았을 **절망적인 고통**입니다.

특히 배포 일정이 코앞으로 다가왔을 때, 원인을 알 수 없는 버그가 발생하여 수백 줄의 코드를 한 줄 한 줄 트레이싱(Tracing)해야 하는 상황은 끔찍하기 그지없습니다. 기능을 하나 추가하려고 해도, 기존 로직 어디서 폭탄이 터질지 몰라 코드를 수정하는 내내 **극도의 스트레스와 불안감**에 시달리게 됩니다. 테스트 코드조차 없는 환경에서는, 내가 수정한 단 한 줄의 코드가 결제 오류를 일으키거나 서비스 장애로 이어질 수 있다는 공포가 등 뒤를 서늘하게 만듭니다. 결국 우리는 리팩토링을 포기하고, 기존의 스파게티 코드 위에 또 다른 스파게티 가닥을 얹어 놓는 **기술 부채의 악순환**에 빠지게 됩니다. _"이 코드를 짠 사람은 도대체 무슨 생각이었을까?"_ 원망하며 Git History를 확인해보면, 부끄럽게도 과거의 나 자신이었던 적은 없으신가요?

코드 리뷰 시간은 어떨까요? 동료들에게 나의 지저분한 코드를 보여주기 부끄러워 변명하기 급급했던 경험이 있을 것입니다. 리뷰어들 역시 어디서부터 어떻게 피드백을 주어야 할지 막막해하며 _"일단 동작은 하니까 머지(Merge)합시다"_ 라는 타협을 하게 됩니다. 이는 결국 팀 전체의 생산성 저하와 프로덕트 품질 하락으로 직결됩니다. 주말에도 혹시나 장애 알림이 울릴까 노심초사하며 슬랙(Slack) 메시지에 신경을 곤두세우는 삶, 언제까지 계속하시겠습니까?

하지만 이제 더 이상 두려워하거나 좌절할 필요가 없습니다. 여러분의 모니터 안으로 구글, 메타 출신의 **<span style="color:var(--color-cyber-cyan)">시니어 페어 프로그래머(Pair Programmer)</span>**를 즉시 고용할 수 있는 시대가 왔기 때문입니다. 생성형 AI를 단순한 코드 검색기가 아닌, 철저하게 훈련된 **'코드 리뷰어이자 리팩토링 전문가'**로 활용하는 프롬프트 엔지니어링 기법을 소개합니다. 단순히 _"코드를 깔끔하게 고쳐줘"_ 라고 말하는 수준을 넘어섭니다. 이 기법은 코드를 논리적으로 분석하고, 클린 코드(Clean Code) 원칙에 따라 구조를 재설계하며, 가장 중요한 **방어적 단위 테스트(Unit Test)**까지 한 번에 완벽하게 구축해 냅니다.

이 가이드에서 제공하는 프롬프트를 실무에 적용하는 순간, 쳐다보기도 싫었던 레거시 코드는 **아름답고 견고한 모던 아키텍처**로 탈바꿈할 것입니다. 1시간 넘게 끙끙대던 코드 분석과 수정 작업이 단 5분 이내로 단축되는 마법을 경험해 보세요. 무엇보다 촘촘하게 작성된 테스트 코드가 제공하는 **강력한 심리적 안전망** 덕분에, 언제든 자신감 있게 코드를 배포하고 기분 좋게 칼퇴근할 수 있는 개발자로 거듭날 수 있습니다. 자, 이제 낡은 코드를 우아하게 청소할 시간입니다!

### ❌ Before (우리가 겪던 고통)

> 아무런 문서도 없고 의도를 알 수 없는 매직 넘버(Magic Number)와 축약된 변수명으로 가득 찬, 쳐다보기도 싫은 스파게티 코드입니다.

```python
> def calc_trx(a, t, f=False):
    # a: amount, t: type
>     if t == 1:
>         if f: return a * 1.1 + 500
>         else: return a * 1.1
>     elif t == 2:
>         return a - (a * 0.05)
>     else: return 0 # 에러
```

### ✅ After (완벽하게 변신한 결과)

```python
from enum import Enum
import unittest

class TransactionType(Enum):
    DEPOSIT = 1
    WITHDRAWAL = 2

def calculate_transaction_amount(amount: float, transaction_type: TransactionType, apply_fee: bool = False) -> float:
    """
    거래 유형(입금/출금)에 따라 세금 및 수수료가 적용된 최종 금액을 계산합니다.
    """
    if amount < 0:
        raise ValueError("거래 금액은 0보다 커야 합니다.")

    if transaction_type == TransactionType.DEPOSIT:
        taxed_amount = amount * 1.1
        return taxed_amount + 500 if apply_fee else taxed_amount

    elif transaction_type == TransactionType.WITHDRAWAL:
        return amount * 0.95

    return 0.0

# ----------------- Unit Tests ----------------- #
class TestCalculateTransaction(unittest.TestCase):
    def test_deposit_with_fee(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.DEPOSIT, True), 1600.0)

    def test_withdrawal_amount(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.WITHDRAWAL), 950.0)

    def test_negative_amount_raises_error(self):
        with self.assertRaises(ValueError):
            calculate_transaction_amount(-500, TransactionType.DEPOSIT)
```

## 🚀 진짜 전문가들은 이렇게 작성합니다

### 🥇 Pro Version (전문가형)

AI에게 단순히 _"이 코드를 리팩토링 해줘"_ 라고 지시하면, 기존의 핵심 비즈니스 로직이 훼손될 위험이 매우 큽니다. 아래의 프롬프트는 AI가 논리적인 단계를 밟아가며 코드를 분석하고 개선하도록 **Chain-of-Thought(생각의 사슬)** 기법을 정교하게 적용했습니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

> **역할 (Role):** 너는 구글, 메타 등 빅테크 출신의 `[10년 차 시니어 백엔드 개발자]`야. 클린 코드(Clean Code) 철학의 열렬한 신봉자이며, TDD(테스트 주도 개발) 원칙을 철저하게 준수하지.
>
> **상황 (Context):**
> 내가 지금 `[Python/Java 등 사용 중인 언어]`로 작성된 아주 지저분한 레거시 코드를 인수인계받았어. 가독성이 끔찍할 뿐만 아니라, 유지보수 자체가 불가능에 가까운 상태야.
>
> **요청 (Task):**
> 다음 단계를 차례대로 심도 있게 수행해 줘.
>
> **Step 1. 코드 분석 및 진단:** 이 코드가 비즈니스적으로 어떤 역할을 수행하는지 단 한 문장으로 명확히 요약하고, 치명적인 문제점(Code Smell) 3가지를 구체적으로 지적해.
> **Step 2. 안전한 리팩토링:** 원본의 비즈니스 로직과 부수 효과(Side Effect)를 100% 동일하게 유지하되, 단일 책임 원칙(SRP)에 따른 함수 분리와 명확한 변수명/메서드명 변경을 통해 가독성을 극대화해. 복잡한 비즈니스 로직이 포함된 곳에는 반드시 Docstring과 주석을 추가해.
> **Step 3. 방어적 검증 (테스트 코드):** 네가 리팩토링한 코드가 원본과 동일하게 정상 작동함을 완벽히 보장하는 단위 테스트(Unit Test) 코드를 작성해. 성공 케이스뿐만 아니라, 발생 가능한 예외 상황(Edge Case)을 최소 2개 이상 반드시 포함해야 해.
>
> **제약사항 (Constraints):**
> - 모바일 가독성을 위해 표(Table)는 절대 사용하지 말고, 가독성 높은 글머리 기호(List) 형태로 정리해 줘.
> - 중요한 키워드는 반드시 **볼드** 처리해 줘.
>
> **입력 코드:**
>
> `[여기에 분석할 레거시 코드를 붙여넣으세요]`

## 💡 작성자 코멘트 (Insight & How to use)

이 프롬프트의 진정한 가치는 단순히 보기 좋은 코드를 만들어 내는 데 있는 것이 아닙니다. 핵심은 코드를 수정하기 전에 **'테스트 코드라는 안전망'을 먼저 치도록 강제한다**는 점에 있습니다. 실무에서 우리가 레거시 코드를 함부로 건드리지 못하는 가장 큰 이유는, 수정한 코드가 기존 시스템에 어떤 나비효과를 불러올지 예측할 수 없기 때문입니다. 이 프롬프트는 AI에게 리팩토링 작업과 검증 작업을 동시에 지시함으로써, 개발자가 안심하고 코드를 교체할 수 있는 환경을 스스로 구축하게 만듭니다.

실제로 제가 5년 이상 방치된 낡은 결제 모듈(Java)을 인수인계받았을 때의 일입니다. 수많은 할인 조건과 쿠폰 적용 로직이 하나의 `calculateTotal()` 메서드 안에 500줄이 넘는 `if-else` 블록으로 복잡하게 얽혀 있었습니다. 이 프롬프트를 실무에 직접 적용해 보았을 때, 가장 감탄했던 부분은 제가 미처 파악하지 못하고 놓쳤던 **숨겨진 엣지 케이스(Edge Case)**들을 AI가 단위 테스트 코드로 먼저 날카롭게 짚어주었다는 사실입니다. 예를 들어, _'결제 금액이 0원일 때의 분기 처리 로직'_ 이나 _'유효기간이 지난 쿠폰이 중복 적용되었을 때의 예외 처리'_ 등을 AI가 스스로 테스트 케이스로 작성하고, 이를 통과하는 리팩토링 코드를 완벽하게 제시해 주었습니다. 덕분에 저는 단 한 번의 결제 오류도 내지 않고 악명 높았던 레거시 모듈을 성공적으로 현대화할 수 있었습니다.

**🔥 실무 최적화 및 변수 통제법 (Constraint Control):**
이 프롬프트를 사용할 때 가장 주의해야 할 점은 **컨텍스트의 크기 조절**입니다. 한 번에 수백 줄, 혹은 수천 줄에 달하는 클래스 전체를 복사해서 붙여 넣으면, 아무리 뛰어난 성능의 AI라도 맥락을 잃거나 환각(Hallucination)을 일으켜 비즈니스 의도와 전혀 엉뚱한 로직을 생성할 위험이 큽니다. 따라서 가장 복잡해 보이고 얽혀 있는 **메서드 단위(100줄 이내)로 잘게 쪼개서 질문**하는 것을 강력히 권장합니다. 이렇게 범위를 좁혀주면 AI의 분석 깊이와 생성되는 테스트 코드의 정확도가 압도적으로 수직 상승합니다.

또한, 팀마다 사용하는 **테스트 프레임워크와 네이밍 컨벤션**이 다를 수 있습니다. 이럴 때는 프롬프트의 `[요청 (Task)]` 섹션에 여러분의 팀 규칙을 구체적인 제약사항으로 과감하게 추가해 보세요.
예를 들어, _"테스트 코드는 반드시 JUnit5와 AssertJ를 사용하고, BDD 스타일(Given-When-Then)로 주석을 달아 작성해 줘."_ 또는 _"메서드명은 카멜 케이스(camelCase)를 엄격하게 적용하고, boolean을 반환하는 함수는 반드시 'is'나 'has'로 시작하도록 해."_ 와 같이 뾰족하고 세밀하게 통제할수록 AI는 여러분의 팀에 완벽하게 동화된 진짜 시니어 동료처럼 행동할 것입니다.

만약 AI가 리팩토링한 코드가 마음에 들지 않거나 특정 비즈니스 로직을 누락했다면, 곧바로 새 채팅창을 열지 마시고 **연속적인 피드백(Iterative Prompting)**을 시도하세요. _"네가 수정한 코드에서 `VIP 고객에 대한 5% 추가 할인` 로직이 누락된 것 같아. 이 부분을 다시 확인하고 원본의 기능이 100% 반영되도록 코드를 재수정해 줘."_ 라고 콕 집어 지적하면, AI는 즉시 자신의 실수를 인지하고 완벽하게 보완된 코드를 다시 제시할 것입니다. 인간 페어 프로그래머와 대화하듯 끊임없이 티키타카를 주고받으며 코드를 조율해 나가는 것, 이것이 바로 AI를 200% 활용하는 진짜 엔지니어들의 숨겨진 노하우입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코딩 로직을 다루는 데 어떤 AI 모델이 가장 적합한가요?**
  - A: 2026년 현재 코딩 및 리팩토링 분야에서는 **Claude 3.5 Sonnet**과 **GPT-4o**가 양대 산맥을 이루고 있습니다. 그중에서도 복잡한 컨텍스트를 오랫동안 유지하고 인간 친화적인 클린 코드를 작성하는 데 있어서는 Claude 3.5 Sonnet이 한층 더 우세한 퍼포먼스를 보여줍니다.

- **Q: 회사 내부 코드를 AI에게 그대로 넘겨도 보안상 문제가 없나요?**
  - A: **절대 안 됩니다.** API 키, 사내망 IP 주소, 실제 고객 데이터, 핵심 보안 알고리즘 등 민감한 정보는 반드시 `[MASKED]` 처리하거나 가상의 더미 데이터(Dummy Data)로 치환한 후에 프롬프트를 실행하셔야 합니다. 가장 안전한 방법은 사내에 구축된 Enterprise AI 환경 내에서만 사용하는 것입니다.

- **Q: 리팩토링 후 에러가 발생하면 어떻게 대처해야 하나요?**
  - A: AI가 놓친 의존성(Dependency) 문제가 있을 확률이 매우 높습니다. 에러 로그 전체와 원본 코드의 관련 컨텍스트를 함께 복사하여 _"이 부분에서 이런 에러가 발생했어. 기존 로직의 `[특정 조건]`을 네가 누락한 것 같아. 다시 수정해 줘"_ 라고 매우 구체적으로 피드백을 주시면 즉각 해결됩니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **시니어 페르소나 부여 (Role):** "10년 차 시니어, 클린 코드 신봉자"라는 엄격한 페르소나를 부여함으로써, 단순히 굴러가기만 하는 코드가 아니라 유지보수성이 뛰어난 전문가 수준의 코드를 출력하도록 AI의 기준치를 한껏 끌어올렸습니다.
2. **Chain-of-Thought (사고의 사슬):** 다짜고짜 _"고쳐"_ 라고 명령하는 대신, **[분석 → 수정 → 검증]**으로 이어지는 논리적 3단계를 강제했습니다. 이는 AI가 코드의 본래 의도를 훼손하지 않으면서 구조만 안전하게 개선하도록 유도하는 핵심 안전장치입니다.
3. **엣지 케이스 강제 (Constraints):** 테스트 코드 작성 시 엣지 케이스를 명시적으로 요구함으로써, 실무 운영 환경에 곧바로 배포해도 무방할 만큼 견고하고 수준 높은 코드 품질을 완벽하게 보장합니다.
