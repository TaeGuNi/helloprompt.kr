## 📝 복잡한 SQL 쿼리, 말로 하면 코드로 짜줍니다

- **🎯 추천 대상:** 데이터 분석가, 퍼포먼스 마케터, 주니어 백엔드 개발자
- **⏱️ 소요 시간:** 30분 → 1분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet (코드 생성 및 데이터베이스 구조 이해도 최상), GPT-4o

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"JOIN, GROUP BY, 서브쿼리... 스택오버플로우를 뒤지느라 통째로 날려버린 당신의 오전 시간, 이제 단 1분으로 압축해 드립니다."_

급하게 핵심 데이터를 추출해야 하는데, 가물가물한 SQL 문법 때문에 스택오버플로우만 하염없이 뒤적거린 경험이 있으신가요? 특히 여러 테이블을 복잡하게 엮는 JOIN 연산이나 까다로운 필터링 조건이 붙는 순간, 머릿속은 하얗게 변하기 마련입니다. 이제는 걱정하지 마세요. 본 프롬프트는 당신의 평범한 일상어(자연어) 요청을 완벽하게 최적화된 SQL 코드로 즉각 변환해 줍니다. 심지어 쿼리가 어떻게 설계되었는지 명쾌한 주석과 해설까지 덧붙여서 말이죠. 더 이상의 문법 스트레스는 없습니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **자연어를 SQL 쿼리로 완벽 변환:** "작년 가입자 중 VIP 고객 찾아줘"와 같은 일상적인 요청을 즉시 실행 가능한 정교한 SQL 코드로 번역합니다.
2. **복잡한 관계형 데이터 처리:** 까다로운 다중 JOIN 연산, 중첩 서브쿼리, 윈도우 함수까지 어떠한 문법적 오류 없이 자동으로 구성해 냅니다.
3. **DB 성능 최적화 코칭:** 단순히 작동하는 수준을 넘어, 데이터베이스의 부하를 최소화하는 인덱스 전략과 쿼리 리팩토링 팁까지 선제적으로 제안합니다.

## 🚀 해결책: "SQL Wizard"

### 🥉 Basic Version (기본형)

간단한 `SELECT` 문이나 단일 테이블 내에서의 조건부 데이터 추출이 필요할 때 가볍게 활용하세요.

> **역할 (Role):** 너는 시니어 데이터베이스 관리자(DBA)야.
> **요청 (Task):** `[테이블명]` 테이블에서 `[구체적인 데이터 추출 조건]`에 완벽히 부합하는 데이터를 뽑아주는 SQL 쿼리를 작성해 줘.


### 🥇 Pro Version (전문가형)

다수의 테이블을 병합(JOIN)하거나, 데이터베이스 부하를 고려한 쿼리 최적화 및 상세한 주석 처리가 필수적인 실무 환경에 최적화된 프롬프트입니다.

> **역할 (Role):** 너는 10년 차 시니어 데이터 엔지니어(Data Engineer)야.
>
> **상황 (Context):**
>
> - 데이터베이스(RDBMS) 종류: `[MySQL / PostgreSQL / BigQuery / Oracle 중 택 1]`
> - 테이블 스키마 정보:
>   - `[테이블명 1]`: `[컬럼 1, 컬럼 2, 컬럼 3...]`
>   - `[테이블명 2]`: `[컬럼 A, 컬럼 B, 컬럼 C...]`
> - 테이블 간 연관 관계(Relation): `[테이블명 1.특정 컬럼]` = `[테이블명 2.특정 컬럼]`
>
> **요청 (Task):**
>
> 1. `[원하는 데이터의 구체적인 조건]`을 완벽하게 추출해 내는 가장 효율적이고 정확한 SQL 쿼리를 작성해 줘.
> 2. 작성된 쿼리의 핵심 로직(JOIN 조건, WHERE 절 필터링 등)이 어떻게 작동하는지 쿼리 내 주석(`--`)으로 상세히 설명해 줘.
> 3. 만약 대용량 데이터 환경이라고 가정할 때, 이 쿼리의 실행 속도를 획기적으로 높일 수 있는 인덱스(Index) 전략이나 쿼리 리팩토링 방안을 하단에 간략히 제안해 줘.
>
> **제약사항 (Constraints):**
>
> - 코드 가독성을 극대화하기 위해 SQL 예약어(SELECT, FROM, WHERE 등)는 반드시 대문자로 표기할 것.
> - 데이터베이스 성능 저하를 방지하기 위해 불필요한 중첩 서브쿼리 사용을 지양하고, 가급적 명시적인 JOIN 문법을 우선적으로 사용할 것.
>
> **주의사항 (Warning):**
>
> - 내가 제공한 스키마 정보에 존재하지 않는 가상의 컬럼이나 테이블을 임의로 상상해서(환각) 쿼리에 포함하지 마. 구조가 확실하지 않은 부분이 있다면 코드를 생성하기 전에 먼저 나에게 질문해 줘.

## 💡 작성자 코멘트 (Insight)

본 프롬프트가 지닌 파괴력의 핵심은 바로 **'데이터베이스 종류(Dialect)'**와 **'스키마 구조'**를 AI에게 명확히 통제하는 데 있습니다. 예를 들어, MySQL과 PostgreSQL은 날짜를 다루는 함수(Date Functions)부터 미묘하게 다르기 때문에 이 기준을 명시하지 않으면 엉뚱한 문법 에러를 마주하게 됩니다.

여기서 실무 효율을 극대화하는 **강력한 꿀팁**을 하나 공개합니다. 수많은 테이블 스키마 정보를 일일이 프롬프트에 타이핑할 필요가 전혀 없습니다. 여러분이 사용하는 데이터베이스 클라이언트(DBeaver, DataGrip 등)에서 해당 테이블을 우클릭한 뒤, **`SHOW CREATE TABLE` (또는 DDL 스크립트 보기) 기능을 이용해 생성된 SQL 스키마 원문을 통째로 복사하여 프롬프트의 `상황 (Context)` 영역에 그대로 붙여넣으세요.** 이렇게 하면 AI가 테이블의 물리적 구조와 데이터 타입을 완벽하게 인지하여, 단 한 번의 수정도 필요 없는 100% 실행 가능한 정교한 쿼리를 토해냅니다.

다만, 아무리 AI가 작성한 쿼리라 할지라도 프로덕션(Production) 환경의 DB에 직접 쿼리를 날릴 때는 주의가 필요합니다. 쿼리 실행 전 반드시 `EXPLAIN` 명령어를 앞에 붙여 실행 계획(Execution Plan)을 점검하거나, 쿼리 끝에 `LIMIT 100`을 추가하여 대용량 테이블 스캔으로 인해 데이터베이스 서버가 마비되는 대참사를 선제적으로 예방하는 전문가다운 습관을 들이시기 바랍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 회사 내부 데이터베이스 스키마 정보를 ChatGPT에 그대로 입력해도 보안상 안전한가요?**
  - A: 개인 식별 정보(PII - 고객의 실제 이름, 전화번호, 이메일, 트랜잭션 데이터 등)와 같은 '실제 데이터 값'은 절대로 입력해서는 안 됩니다. 하지만 테이블의 뼈대인 **'테이블명'**이나 **'컬럼명(예: `user_id`, `created_at`)'** 자체는 단순한 메타데이터이므로, 사내 보안 규정을 위반하지 않는 선에서 충분히 활용 가능합니다. 만약 이조차도 불안하다면 테이블명을 `Table_A`로, 핵심 컬럼명을 `Column_1`과 같은 형태로 철저하게 마스킹(Masking) 처리한 뒤 프롬프트를 실행하는 것을 적극 권장합니다.

- **Q: 날짜와 시간을 다루는 쿼리인데, 막상 실행해 보면 결과 값이 묘하게 엇나갑니다.**
  - A: 데이터베이스마다 타임존(Timezone) 설정이 다르기 때문입니다. 이럴 때는 프롬프트의 요청(Task) 영역에 *"날짜 필터링 기준을 `YYYY-MM-DD` 문자열 포맷으로 정확히 캐스팅(Casting)해 줘"* 또는 *"서버 타임존을 KST(Asia/Seoul) 기준으로 변환해서 로직을 짜줘"*라는 디테일한 조건을 한 줄만 추가해 보세요. 마법처럼 문제가 즉시 해결됩니다.

- **Q: 쿼리를 그대로 복사해서 실행했는데 Syntax Error(문법 오류)가 발생합니다.**
  - A: 프롬프트 최상단의 `데이터베이스 종류(RDBMS)` 변수가 정확히 명시되었는지 다시 한번 점검해 보세요. Oracle의 분석 함수와 BigQuery의 윈도우 함수는 지원하는 문법과 구조가 완전히 다를 수 있습니다. 만약 종류를 명시했는데도 에러가 난다면, 출력된 에러 메시지 원문을 그대로 복사하여 AI에게 다시 던져주세요. AI가 스스로 원인을 분석하고 즉각적으로 디버깅된 코드를 다시 제안할 것입니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **역할(Role) 부여를 통한 전문성 극대화:** AI를 단순한 챗봇이 아닌, '실무 경험이 풍부한 시니어 데이터 엔지니어'로 세팅(Persona)함으로써 생성되는 쿼리의 품질과 최적화 수준을 실무 수준으로 대폭 끌어올렸습니다.
2. **맥락(Context) 통제를 통한 무결성 확보:** 데이터베이스 방언(SQL Dialect) 차이에서 발생하는 치명적인 문법 오류를 원천 차단했습니다. 또한 물리적인 스키마 구조를 미리 주입하여 완벽하게 실행 가능한 코드만을 생성하도록 강력한 제약(Constraints)을 걸었습니다.
3. **경고(Warning)를 통한 환각(Hallucination) 억제:** 존재하지 않는 컬럼이나 테이블을 멋대로 지어내서 쿼리를 짜버리는 AI 특유의 고질적인 환각 현상을 억제하기 위해, "모르면 질문하라"는 엄격한 통제 장치를 프롬프트에 심었습니다.

## 📊 증명: Before & After

### ❌ Before (단순한 질문)

"유저 테이블이랑 주문 테이블 합쳐서, 2024년에 가입한 유저 중에 총 주문 금액이 10만 원 이상인 사람 이름이랑 주문금액 보여줘."

_(결과: 어떤 DB를 쓰는지 알 수 없어 범용적인 표준 SQL로 뭉뚱그려 작성하거나, 존재하지도 않는 가상의 컬럼을 지어내어 실행 즉시 치명적인 에러를 뱉어냄)_

### ✅ After (Pro Version 결과)

```sql
-- 1. 유저(users) 테이블과 주문(orders) 테이블을 user_id 기준으로 JOIN 합니다.
-- 2. 가입일(created_at) 필터를 통해 2024년 가입자를 추출합니다.
-- 3. 유저별 총 주문 금액을 합산(SUM)하고 HAVING 절로 10만 원 이상 필터링합니다.

SELECT
    U.name AS user_name,
    SUM(O.amount) AS total_order_amount
FROM
    users U
INNER JOIN
    orders O ON U.user_id = O.user_id
WHERE
    U.created_at >= '2024-01-01 00:00:00'
    AND U.created_at < '2025-01-01 00:00:00'
GROUP BY
    U.user_id,
    U.name
HAVING
    SUM(O.amount) >= 100000
ORDER BY
    total_order_amount DESC;

/*
💡 [성능 최적화 팁 (Index Strategy)]
- WHERE 절의 검색 속도를 높이기 위해 `users` 테이블의 `created_at` 컬럼에 인덱스(Index)를 생성하는 것을 권장합니다.
- `orders` 테이블의 조인 키인 `user_id` 컬럼에도 인덱스가 걸려 있는지 확인해 보세요.
*/
```
