# 📝 Githubプロジェクトの顔、README.md自動生成ツール

- **🎯 おすすめの対象:** オープンソースのメンテナー、サイドプロジェクトの開発者、ポートフォリオを準備中の就活生
- **⏱️ 所要時間:** 1時間 → 3分に短縮
- **🤖 おすすめのモデル:** 全ての対話型AI (ChatGPT, Claude, Geminiなど)

- ⭐ **難易度:** ⭐☆☆☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「どんなに優れたコードを書いても、READMEがお粗末だと、誰もあなたのプロジェクトを開こうとはしません。」_

素晴らしいオープンソースやサイドプロジェクトを完成させたのに、`README.md`の作成を後回しにしていませんか？開発者にとって、ドキュメント作成はコーディングよりも高い壁に感じられることがよくあります。「インストール方法はどう書けばいい？」「機能の説明はどこまで詳しくするべき？」「ライセンスやバッジはどう付ければいい？」と悩んだ挙句、結局白紙のまま放置してしまうケースも少なくありません。

これからはコーディングにだけ集中してください。プロジェクトのファイルツリーと主要な技術スタックをAIに投げるだけで、グローバルスタンダードに準拠した、プロフェッショナルな紹介ページがあっという間に完成します。

## ⚡️ 3行まとめ (TL;DR)

1. **劇的な時間の節約:** 1時間以上かかっていたREADME.mdの草案作成を、わずか3分で終わらせることができます。
2. **構造化されたテンプレート:** バッジ、機能紹介、インストールガイド、ライセンスなど、オープンソースの標準フォーマットを自動的に適用します。
3. **視認性と信頼性の向上:** プロフェッショナルに作成されたドキュメントは、他の開発者のコントリビュート（Contribute）や利用を積極的に促します。

## 🚀 解決策: "AI README Architect"

### 🥉 Basic Version (基本型)

素早くプロジェクトの骨組みだけを作りたい時に使用してください。

> **役割:** あなたは`[シニアテクニカルライター]`です。
> **タスク:** 私のプロジェクトの`[ファイル構造]`と`[主要な技術スタック]`を基に、簡潔で分かりやすいMarkdown形式のREADME.mdの草案を作成してください。

\

### 🥇 Pro Version (専門家型)

オープンソースの標準に合わせた完璧なディテールと、視覚的な要素（バッジ、絵文字など）が必要な場合に使用してください。

> **役割 (Role):** あなたはグローバルなオープンソースプロジェクトを管理する`[シニアメンテナー兼テクニカルライター]`です。
>
> **状況 (Context):**
>
> - 背景: 新しいサイドプロジェクトをローンチしました。他の開発者が簡単に理解し、貢献（Contribute）できるような魅力的な紹介ドキュメントが必要です。
> - 目標: GitHubの標準フォーマットに準拠した高品質な`README.md`の完成。
>
> **タスク (Task):**
>
> 以下のプロジェクト情報を基に、`README.md`全体のコードを作成してください。
>
> **[プロジェクト情報]**
>
> - **プロジェクト名:** `[プロジェクト名]`
> - **一言紹介:** `[プロジェクトのコアバリューを説明する一文]`
> - **技術スタック:** `[例: React, TypeScript, TailwindCSS]`
> - **ファイル構造:**
>   ```text
>   [treeコマンドの実行結果を貼り付け]
>   ```
>
> **制約事項 (Constraints):**
>
> 1. **Badges:** 最上部にプロジェクトのステータス、ライセンス、使用技術スタックに合わせた動的なシールドバッジ(Shields.io)を配置すること。
> 2. **Features:** 提供されたファイル構造とスタックを分析し、主要な機能を3〜5つの箇条書き（絵文字を含む）で推測して作成すること。
> 3. **Getting Started:** `npm install`や`npm run dev`など、開発環境のセットアップと実行ガイドを段階的に明確に記述すること。
> 4. **License:** MITライセンスのセクションを最下部に追加すること。
> 5. **出力形式:** 全体の内容を1つのMarkdown（` ```markdown `）コードブロックとして出力すること。
>
> **注意事項 (Warning):**
>
> - ファイル構造に存在しない、不明な機能やパッケージを勝手に捏造しないでください。（ハルシネーションの防止）
> - 説明は簡潔で専門的なトーン（Tone & Manner）を維持してください。

## 💡 作成者のコメント (Insight)

このプロンプトの意義は、単にテキストを生成することにとどまらず、オープンソースエコシステムの暗黙のルール（バッジの配置、目次の構造、インストールガイドの自然な流れ）をAIに強制的に守らせる点にあります。

特に、`tree`コマンドを通じてファイル構造を提供するのが핵심（核心/キーポイント）です。AIはフォルダとファイル名（`auth.ts`、`payment.controller.ts`など）を見ただけで、このプロジェクトがどのような機能を持っているかを驚くほど正確に推測してくれます。

**🍯 Tips:** 生成されたドキュメントは、あくまで優れた「草案（ドラフト）」です。ここに実際に動作するUIのスクリーンショットやデモのGIFをご自身で追加すれば、完璧なポートフォリオが完成します。もしプロジェクトのロゴがなければ、DALL-EやMidjourneyを活用してみるのも良い方法です。

## 🙋 よくある質問 (FAQ)

- **Q: ファイルツリーが長すぎて、プロンプトの文字数制限に引っかかります。**
  - A: `node_modules`、`dist`、`.git`など、コアなビジネスロジックと関係のないビルドアーティファクトは除外してください。ターミナルで `tree -I 'node_modules|dist|.git'` コマンドを使用すれば、重要な構造だけをすっきりと抽出できます。

- **Q: 英語でREADMEを作成したいのですが。**
  - A: プロンプトの**タスク (Task)**の最後に_「すべての内容は、自然なビジネス英語で記述してください。」_という一文を追加するだけで、グローバル向けのREADMEが即座に生成されます。

- **Q: コントリビューター向けのガイド (CONTRIBUTING.md) も作れますか？**
  - A: もちろんです。作成されたREADMEをベースにした続く会話で、_「このプロジェクトのためのオープンソースコントリビューターガイド(CONTRIBUTING.md)と、PRテンプレート(pull_request_template.md)も作成して」_と追加でリクエストしてみてください。

## 🧬 プロンプトの解剖 (Why it works?)

1. **Roleの付与:** 「シニアメンテナー」というペルソナを与えることで、ジュニア開発者が見落としがちなバッジやライセンスなど、オープンソースに必須の要素をAIが自発的にカバーするように誘導しました。
2. **Contextの提供:** 「ファイル構造」という強力なコンテキストを提供することで、AIがよくある一般論ではなく、そのプロジェクトに正確に合致したカスタマイズされた説明を生成するようにしました。
3. **Constraints (制約):** 任意の機能を捏造しないように警告することで、AIのハルシネーション（幻覚）を遮断し、ドキュメントの信頼性を確保しました。
