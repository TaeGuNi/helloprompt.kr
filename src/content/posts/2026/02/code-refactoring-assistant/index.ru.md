## 🚀 Решение: Мастер рефакторинга чистого кода

### 🥉 Базовая версия (Basic)

Идеально подходит для быстрой «уборки» небольших и изолированных участков кода.

> **Роль:** Ты — Senior Software Engineer и непревзойденный эксперт по рефакторингу.
> **Задача:** Кардинально улучши читаемость и структуру следующего фрагмента, сделав код кристально понятным и современным: `[Вставьте сюда ваш код]`

### 🥇 Профессиональная версия (Pro)

Применяйте для рефакторинга сложной бизнес-логики, где критически важны архитектура, масштабируемость и соблюдение канонов Clean Code.

> **Роль (Role):** Ты — Staff Software Engineer и бескомпромиссный адепт чистого кода (Clean Code) и принципов SOLID. Ты славишься умением превращать самый жуткий спагетти-код в элегантные, модульные и легко тестируемые архитектурные решения.
>
> **Контекст (Context):**
>
> - **Фон:** Данный код написан на `[Укажите язык, например: TypeScript]`. Он выполняет свою задачу, но его архитектура оставляет желать лучшего: логика запутана, а поддержка, масштабирование и тестирование крайне затруднены.
> - **Цель:** Провести глубокий хирургический рефакторинг с полным сохранением исходного внешнего поведения (без изменения бизнес-логики).
>
> **Задача (Task):**
>
> 1. Проанализируй предоставленный код и выяви ключевые уязвимости (нарушения SOLID, «запахи кода», дублирование логики).
> 2. Полностью перепиши код, сделав его модульным, интуитивно понятным и самодокументируемым. Жестко избавься от лишней вложенности (callback hell и бесконечные if-else).
> 3. Добавь профессиональные JSDoc/Docstring комментарии для всех сложных алгоритмов и публичных интерфейсов.
> 4. Код для рефакторинга:
>
>
> `[Вставьте ваш спагетти-код сюда]`
>
>
> **Ограничения (Constraints):**
>
> - Выведи итоговый исправленный код строго в одном изолированном блоке.
> - Категорически запрещено использовать сторонние библиотеки, если они не были импортированы в исходнике или не запрошены явно.
> - Полностью сохрани все существующие внешние зависимости и переменные окружения.
>
> **Формат (Format):**
>
> - Сначала предоставь лаконичный маркированный список: **что именно** было изменено и, главное, **почему** (приведи веские архитектурные обоснования).
> - Затем выведи финальный результат в виде чистого Markdown-блока с кодом.

## 💡 Комментарий автора (Insight)

В суровых реалиях коммерческой разработки рефакторинг часто откладывают «до лучших времен» из-за горящих дедлайнов или банального страха сломать работающий функционал. Этот промпт выступает в роли вашего личного сурового Tech Lead, который проводит въедливое и бескомпромиссное Code Review. 

Главная суперсила версии **Pro** заключается в жестком требовании к ИИ аргументировать **причины** каждого внесенного изменения. Такой подход не просто спасает вашу текущую кодовую базу — он работает как персональный ментор, позволяя вам изучать лучшие мировые практики на живом примере вашего же кода. И помните золотое правило инженерии: всегда прогоняйте Unit-тесты после любых манипуляций ИИ с вашим продакшеном!

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Может ли ИИ случайно сломать работающую бизнес-логику в процессе рефакторинга?**
  - О: Да, нейросети иногда упускают неочевидные пограничные случаи (edge cases) или скрытые взаимосвязи. Именно поэтому критически важно фиксировать изменения в Git и прогонять весь набор тестов (Unit/E2E) после интеграции сгенерированного кода.

- **В: Какая языковая модель справляется с архитектурными задачами лучше всего?**
  - О: По состоянию на 2026 год, абсолютными лидерами в глубоком понимании комплексных архитектурных паттернов и удержании огромного контекста являются модели Claude 3.5 Sonnet и GPT-4o.

- **В: Распознает ли ИИ специфичные паттерны фреймворков (React, Vue, NestJS)?**
  - О: Безусловно! Однако нейросеть не умеет читать мысли. Чтобы получить эталонный результат, прямо укажите в блоке **Контекст** используемый стек и желаемую методологию (например, Feature-Sliced Design или Clean Architecture).

## 🧬 Разбор промпта (Why it works?)

1. **Роль Staff Software Engineer:** Мгновенно переводит ИИ в режим максимальной строгости, заставляя его применять высшие стандарты индустрии, а не ограничиваться ленивой косметической правкой.
2. **Фокус на SOLID и Clean Code:** Задает абсолютно прозрачный фреймворк для оценки: код переписывается ради модульности и уничтожения глубокой вложенности, а не просто так.
3. **Объяснение «Почему»:** Жесткое требование аргументировать каждое архитектурное решение защищает от слепого Copy-Paste и превращает промпт в мощный инструмент вашего профессионального роста.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Исходный спагетти-код)

```javascript
function processData(users, req) {
  let res = [];
  for (let i = 0; i < users.length; i++) {
    if (users[i].age > 18 && users[i].isActive == true) {
      if (req.type == "admin" || users[i].role == "vip") {
        let u = users[i];
        u.discount = 20;
        res.push(u);
      } else {
        let u = users[i];
        u.discount = 5;
        res.push(u);
      }
    }
  }
  return res;
}
```

### ✅ После (Результат рефакторинга)

```javascript
const DISCOUNT_RATES = {
  VIP_OR_ADMIN: 20,
  DEFAULT: 5,
};

const isEligibleForDiscount = (user) => user.age > 18 && user.isActive;
const hasPremiumAccess = (user, requestType) =>
  requestType === "admin" || user.role === "vip";

/**
 * Обрабатывает список пользователей и применяет к ним скидки
 * в зависимости от их статуса и типа запроса.
 *
 * @param {Array} users - Список пользователей для обработки.
 * @param {Object} request - Объект текущего запроса.
 * @returns {Array} Список пользователей с примененными скидками.
 */
function applyDiscountsToEligibleUsers(users, request) {
  return users.filter(isEligibleForDiscount).map((user) => ({
    ...user,
    discount: hasPremiumAccess(user, request.type)
      ? DISCOUNT_RATES.VIP_OR_ADMIN
      : DISCOUNT_RATES.DEFAULT,
  }));
}
```
