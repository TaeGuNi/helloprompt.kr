## 📝 내 코드가 스파게티? 리팩토링 전문가에게 맡겨보세요

- **🎯 추천 대상:** 주니어 개발자, 코드 리뷰를 앞둔 엔지니어, 기술 부채에 시달리는 팀
- **⏱️ 소요 시간:** 1시간 → 10분 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet (코드 리팩토링 최상위), ChatGPT (GPT-4o)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"기능만 겨우 돌아가는 '스파게티 코드', 내일의 당신과 동료를 괴롭히는 시한폭탄입니다. 이제 AI 시니어 개발자에게 코드의 품격을 맡겨보세요."_

당장의 마감일에 쫓겨 '일단 돌아가게만' 짜놓은 스파게티 코드. 지금 당장은 괜찮아 보여도, 이는 결국 내일의 나와 동료의 발목을 잡는 거대한 기술 부채(Technical Debt)로 돌아옵니다. 복잡하게 얽힌 로직, 직관적이지 않은 변수명, 그리고 겹겹이 쌓인 조건문들. 이제 이런 골칫거리 코드를 AI의 압도적인 능력을 빌려 우아하게 해결할 수 있습니다. 본 포스팅에서 소개하는 프롬프트를 활용하면, 마치 실리콘밸리의 시니어 엔지니어에게 깐깐한 코드 리뷰를 받는 것처럼 여러분의 코드를 완벽한 '클린 코드(Clean Code)'의 경지로 끌어올릴 수 있습니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **압도적인 가독성 확보:** 모호했던 변수명과 함수명을 직관적으로 재구성하여, 코드 자체만으로 진짜 의도를 명확하게 전달합니다.
2. **모던 문법 및 패턴 적용:** ES6+ 등 최신 프로그래밍 트렌드와 언어 스펙에 부합하는 세련되고 안전한 문법으로 리팩토링합니다.
3. **유지보수성 및 안정성 극대화:** 단일 책임 원칙(SRP)을 엄격히 준수하고 잠재적인 엣지 케이스를 방어하는 꼼꼼한 예외 처리를 추가합니다.

## 🚀 해결책: "Clean Code Mentor"

### 🥉 Basic Version (기본형)

빠르게 코드의 전반적인 가독성을 높이고 싶을 때 가볍게 활용하세요.

> **역할:** 너는 클린 코드(Clean Code) 전문가야.
> **요청:** 아래 제공된 코드를 가독성이 높고 유지보수하기 쉽게 리팩토링해 줘.
> **코드:**
> `[여기에 수정할 코드를 붙여넣으세요]`

### 🥇 Pro Version (전문가형)

단순한 수정을 넘어, 아키텍처 관점의 깊이 있는 코드 리뷰와 안정성을 보장하는 테스트 코드까지 필요할 때 강력히 추천합니다.

> **역할 (Role):** 너는 실리콘밸리 빅테크 기업의 10년 차 시니어 소프트웨어 엔지니어(Senior Software Engineer)야.
>
> **상황 (Context):**
>
> - 사용 언어: `[프로그래밍 언어, 예: TypeScript]`
> - 주요 목적: `[성능 최적화 / 가독성 향상 / 버그 수정 중 택 1]`
>
> **요청 (Task):**
>
> 1. 제공된 코드를 꼼꼼히 리뷰하고, 현재 코드의 구조적 문제점(Code Smell)을 3가지 이내로 날카롭게 지적해 줘.
> 2. 최신 언어 스펙과 디자인 패턴을 적용하여 완벽하게 리팩토링된 코드를 작성해 줘.
> 3. 변경된 핵심 로직에 대해 JSDoc(또는 해당 언어의 표준 문서화 형식) 스타일의 주석을 달아 줘.
> 4. (선택) 리팩토링된 코드를 안전하게 검증할 수 있는 단위 테스트(Unit Test) 코드를 2개 이상 작성해 줘.
>
> **제약사항 (Constraints):**
>
> - 변수명과 함수명은 반드시 그 역할이 명확히 드러나도록(Descriptive) 지어야 해.
> - 함수는 오직 하나의 일만 수행하도록(Single Responsibility Principle) 작게 분리해 줘.
> - 발생 가능한 엣지 케이스(Edge Case)를 고려하여 에러 처리를 철저하게 해 줘.
>
> **주의사항 (Warning):**
>
> - 리팩토링 과정에서 기존 비즈니스 로직의 결과값이 절대 변경되어서는 안 돼.

## 💡 작성자 코멘트 (Insight)

AI에게 리팩토링을 맡길 때 개발자들이 가장 흔히 저지르는 치명적인 실수는 바로 **'AI가 뱉어낸 코드를 생각 없이 그대로 복사해서 덮어쓰기'**입니다. 진정한 엔지니어링 실력을 키우려면 결과물이 아니라, AI가 코드를 왜 그렇게 수정했는지 그 **'이유(Code Smell)'**에 집요하게 파고들어야 합니다.

또한, AI가 제안한 코드를 프로덕션 환경에 무턱대고 적용하기 전에, 반드시 기존 테스트 코드를 돌려 회귀 버그(Regression Bug)가 발생하지 않는지 교차 검증하세요. 만약 프로젝트에 선행된 테스트 코드가 없다면? 이때는 프롬프트를 약간 비틀어 AI에게 **"이 코드를 리팩토링하기 전, 검증을 위한 단위 테스트 코드를 먼저 작성해 달라"**고 요구하는 것이 가장 안전하고 전문적인 리팩토링의 정석입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 회사 코드베이스의 특정 코딩 컨벤션(예: Airbnb, Google)을 엄격하게 따르게 할 수 있나요?**
  - A: 네, 완벽하게 통제할 수 있습니다. 프롬프트의 제약사항(Constraints) 섹션에 `"Airbnb JavaScript Style Guide를 엄격하게 준수해 줘"` 또는 `"우리 팀의 사내 Prettier/ESLint 규칙을 무조건 따른다고 가정해 줘"`와 같이 구체적인 컨텍스트를 주입하면, AI가 해당 스타일 가이드를 한 치의 오차도 없이 반영하여 결과물을 출력합니다.

- **Q: 수십 개의 파일이 복잡하게 얽힌 대형 프로젝트도 한 번에 리팩토링할 수 있나요?**
  - A: 현재 웹 기반 LLM 모델들의 컨텍스트 윈도우(Context Window) 한계를 고려할 때, 전체 프로젝트를 통째로 쏟아붓는 것은 권장하지 않습니다. 파일 단위, 혹은 핵심 비즈니스 로직을 담당하는 함수나 클래스 단위로 잘게 쪼개어 요청하는 것이 할루시네이션을 막고 정확도를 극대화하는 비결입니다. 만약 프로젝트 전반의 거시적인 아키텍처 개선이 필요하다면, Cursor나 GitHub Copilot Chat 등 로컬 코드베이스 전체를 인덱싱하는 **AI 전용 IDE**를 도입하는 것을 강력히 추천합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **Role(페르소나)의 정교한 부여:** AI에게 "10년 차 시니어 엔지니어"라는 구체적이고 권위 있는 페르소나를 입혀, 단순한 오타 수정이나 문법 교정 수준을 넘어 소프트웨어 아키텍처 관점에서의 예리하고 깐깐한 코드 리뷰를 유도했습니다.
2. **Constraints(제약사항)의 강력한 통제:** '단일 책임 원칙(SRP)', '명확한 변수명' 등 클린 코드의 절대 불변 원칙을 명시적으로 강제했습니다. 이를 통해 AI가 자의적으로 코드를 너무 과도하게 축약하거나, 오히려 인간이 읽기 힘든 방향으로 코드를 오염시키는 부작용을 원천적으로 차단했습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

```javascript
function a(n) {
  if (n < 2) return n;
  return a(n - 1) + a(n - 2);
}
```

### ✅ After (결과)

```javascript
/**
 * 피보나치 수열의 n번째 값을 계산합니다. (재귀 방식)
 *
 * @param {number} n - 계산할 순번 (0 이상의 정수)
 * @returns {number} n번째 피보나치 수
 * @throws {Error} 입력값이 음수일 경우 에러를 발생시킵니다.
 */
function getFibonacci(n) {
  if (n < 0) {
    throw new Error("입력값은 0 이상의 정수여야 합니다.");
  }

  if (n < 2) {
    return n;
  }

  return getFibonacci(n - 1) + getFibonacci(n - 2);
}
```
