## 💡 作者点评 (Insight & How to use)

在实战项目中引入 `everything-claude-code` 设置后，我经历的最令人震撼的变化是 **“AI 开始果断且礼貌地拒绝我那些偷懒的提示词”**。

以前，如果我随口指示“把这个 API 连上并展示在屏幕上”，往往会得到一份边缘情况或错误状态 (Error State) 管理一团糟的代码。但应用此设置后，情况完全不同了。在提示词中输入 `/plan` 命令的瞬间，AI 不会立即写代码，而是发起一轮尖锐的反问。“加载时显示的 Skeleton UI 怎么处理？”、“API 请求失败时的重试 (Retry) 逻辑和用户反馈如何设计？”感觉就像有一位严谨的首席架构师坐在身边，在完美的定义书出来之前绝对不会动键盘。

该工具的核心运作原理在于 **“基于多智能体 (Multi-Agent) 的强制工作流”**。当用户在 `[变量]` 中输入自己的技术栈和情况时，AI 进行的不再是简单的文本补全，而是 **策划 (Planner) → 测试 (TDD) → 实现 (Developer) → 评审 (Reviewer)** 这一层次分明的角色依次介入。仅靠 Basic 版本的提示词就能打好坚实的基础，而通过 Pro 版本的提示词强制执行 `[高级 TypeScript 开发工程师]` 的角色并赋予 `[用户认证功能]` 这样的具体目标，其威力将倍增。通过将 80% 的测试覆盖率设为约束条件，您可以欣赏到 AI 为了通过自己编写的测试代码而不断精雕细琢逻辑的神奇自我反馈闭环。

最让人拍案叫绝的绝对是 **`/evolve`** 命令的存在。普通的 AI 聊天在会话结束后记忆会重置，导致第二天还要重复解释同样的编码规范，非常疲惫。但使用 `/evolve` 后，AI 会将进行项目过程中识别到的团队隐性知识（例如：_“我们团队不在接口名前加 'I'”_，_“状态管理必须只使用 Zustand”_）持续更新到 `.rules` 文件或 `.cursorrules` 中并内化。也就是说，随着项目规模的扩大，AI 也会通过 **“自学习”** 进化为最适合该项目的定制化高级开发人员。

在实务中投入此提示词时，最重要的技巧是变量控制 (Constraint Control)。请在 Pro 提示词的 **`[额外需求]`** 栏中明确写下您项目特有的遗留结构或必须避免的反模式。例如，“不要改动现有的 Redux 代码，请将其抽离为独立的 Hook 编写”，这种详细的约束与 `everything-claude-code` 严谨的规则验证系统相结合，可以将 Bug 发生率降低至接近零。对于没有代码审查者的独立开发者，或是需要快速产出 MVP 的小型创业公司来说，这套系统不仅是生产力工具，更是生存的终极武器。

## 🙋 常见问题 (FAQ)

- **Q: 必须在 Claude Code CLI 环境下才能运行吗？**
  - A: 并非如此！您可以将规则直接移植到 Cursor 编辑器的 `.cursorrules` 文件中，或者在 OpenCode 等其他环境下也能获得优秀的兼容性。

- **Q: 初级开发人员使用时，准入门槛或设置会不会太严苛？**
  - A: 引入初期，由于 AI 会不停询问边缘情况并强制写测试代码，可能会觉得有些繁琐。但只有经过这种严格的“成人礼”，才能在日后防止被无法承受的 **技术债务 (Technical Debt)** 压垮，避免熬夜加班的惨剧。

## 🧬 提示词解析 (Why it works?)

1. **强制工作流 (Forced Workflow):** 从源头上切断了单方面的“帮我做”式指令。通过 `/plan`、`/tdd` 等显式的斜杠命令，强力引导 AI 遵循“需求分析 → 编写测试代码 → 实现正文代码”这一正规的工程流程。
2. **多重人格交叉验证 (Persona Separation):** 将策划人格 (Planner)、编码人格 (Developer) 和评审人格 (Reviewer) 彻底分离。通过这种方式，形成了一个完美的自我反馈闭环，使 AI 能够对自己编写的代码进行尖锐的批判和验证。
3. **永久性的上下文学习 (Instincts):** 与会话结束后上下文就会挥发的普通聊天机器人有着本质区别。它能在整个项目范围内识别用户的代码修改模式和架构意图，并将其持续刻印在 `.rules` 文件中，使智能体自身能够不断进化。
