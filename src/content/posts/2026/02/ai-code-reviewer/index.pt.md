# 📝 Sem Colegas? Tudo Bem, Revisão de Código Dura do Desenvolvedor Sênior de IA

- **🎯 Recomendado para:** Desenvolvedores Júnior a Pleno, Freelancers, Solopreneurs
- **⏱️ Tempo de economia:** Horas de depuração → Reduzido para 5 minutos
- **🤖 Modelo recomendado:** GPT-4, Claude 3.5 Sonnet (Recomendado para raciocínio lógico profundo)

- ⭐ **Dificuldade:** ⭐⭐☆☆☆
- ⚡️ **Eficácia:** ⭐⭐⭐⭐⭐
- 🚀 **Utilidade:** ⭐⭐⭐⭐☆

> _"Será que este código é realmente bom? Trabalhar sozinho significa que os maus hábitos se acumulam sem que você perceba."_

Ao programar sozinho, sua visão inevitavelmente se estreita. Se você apenas passar para a próxima tarefa dizendo "Funciona, então está bom", suas habilidades estagnarão.
Mas e se um desenvolvedor sênior de uma gigante da tecnologia sentasse ao seu lado e dissecasse seu código linha por linha?
De críticas construtivas e duras a conselhos arquitetônicos, contrate um Revisor de Código de IA para elevar seu padrão de engenharia.

---

## ⚡️ Resumo em 3 linhas (TL;DR)

1. Aplique os rigorosos padrões de qualidade de código utilizados nas FAANG (Google, Amazon, etc.).
2. Obtenha uma análise multidimensional focada em segurança, desempenho, legibilidade e arquitetura.
3. Receba feedback acionável e trechos de código refatorados para aprendizado imediato.

---

## 🚀 A Solução: "Revisor de Código Sênior de IA"

### 🥉 Basic Version (Versão Básica)

Use isso quando precisar de um feedback rápido e direto.

> **Role:** Você é um Engenheiro de Software Sênior.
> **Task:** Revise o seguinte código, aponte os erros e sugira melhorias focando em legibilidade e desempenho: `[Insira seu código aqui]`

<br>

### 🥇 Pro Version (Versão Especialista)

Use este prompt para obter uma revisão profunda, arquitetônica e educativa.

> **Papel (Role):** Você é um Staff Software Engineer que trabalhou no Google e na Amazon por mais de 15 anos. Você é um mentor técnico exigente, porém empático, focado em excelência.
>
> **Contexto (Context):**
>
> - Cenário: Sou um desenvolvedor trabalhando neste código. Preciso de uma revisão de código impecável antes de submeter um PR (Pull Request).
> - Objetivo: Melhorar a qualidade do código, identificar potenciais gargalos e aprender as melhores práticas da indústria.
>
> **Tarefa (Task):**
> Revise meu código e forneça seu feedback estruturado exatamente neste formato:
>
> 1. **👍 Pontos Fortes:** Elogie especificamente as lógicas bem implementadas ou as boas escolhas de design.
> 2. **🛠 Melhorias:** Aponte falhas ou áreas de risco em termos de legibilidade, complexidade ciclomática, desempenho, segurança e tratamento de erros (Edge Cases).
> 3. **🤔 Perguntas Socráticas:** Faça perguntas provocativas (ex: "Por que você escolheu esta estrutura de dados?") para me forçar a pensar criticamente.
> 4. **✨ Refatoração:** Forneça o código totalmente refatorado aplicando Clean Code e princípios SOLID.
>
> **Restrições (Constraints):**
>
> - Mantenha um tom profissional, direto e rigoroso (sem rodeios).
> - Analise minuciosamente a nomenclatura das variáveis, tipagem, espaçamento e modularidade.
> - Se o código fornecido estiver perfeito, desafie-me com uma forma de otimizá-lo para grande escala.
>
> **Código:**
> `[Cole o código a ser revisado aqui]`

## 💡 Comentário do Autor (Insight)

A maior armadilha de codificar sozinho é o viés de confirmação: você testa apenas o caminho feliz ("Happy Path"). Este prompt não apenas encontra bugs, mas atua como um verdadeiro mentor sênior. O grande diferencial da Versão Pro é a seção de **Perguntas Socráticas**. Em vez de simplesmente receber a resposta pronta, a IA o obriga a justificar suas decisões técnicas. Isso desenvolve o "músculo arquitetônico" do cérebro, algo que desenvolvedores juniores lutam anos para conquistar. Pessoalmente, uso esse prompt em todos os meus projetos paralelos antes de fazer um `git commit`.

## 🙋 Perguntas Frequentes (FAQ)

- **Q: Modelos de IA gratuitos (como o ChatGPT 3.5) funcionam bem com este prompt?**
  - A: Eles funcionam para encontrar erros básicos de sintaxe, mas para análises profundas de arquitetura, padrões de projeto e refatoração complexa, recomendo fortemente o uso de modelos mais avançados como GPT-4o ou Claude 3.5 Sonnet. O Claude, em particular, possui uma compreensão excepcional de nuances de código.

- **Q: Devo colar arquivos inteiros ou apenas funções isoladas?**
  - A: Para obter o melhor resultado (e não estourar o limite de tokens do modelo de IA), é aconselhável enviar o código por funções ou métodos específicos, juntamente com uma breve explicação do que a função deveria fazer. Arquivos muito grandes (com mais de 500 linhas) podem gerar respostas mais genéricas.

---

## 🧬 Anatomia do Prompt (Por que funciona?)

1. **Role Playing de Elite:** Definir a IA como um "Staff Engineer do Google/Amazon" calibra os pesos da rede neural para priorizar padrões de Clean Code e escalabilidade rigorosa.
2. **Saída Estruturada:** Ao exigir um formato rígido (Pontos Fortes, Melhorias, Perguntas, Refatoração), evitamos que a IA apenas jogue o código corrigido sem explicar o porquê, transformando a revisão em uma sessão de aprendizado ativa.
3. **Foco Socrático:** A instrução para fazer perguntas críticas emula a dinâmica real de mentoria das grandes empresas de tecnologia, acelerando o crescimento profissional do usuário.

## 📊 Prova: Antes e Depois

### ❌ Antes (Código de Entrada)

```python
def calc(x, y):
    try:
        res = x / y
        return res
    except:
        return 0
```

### ✅ Depois (Resultado da Revisão)

**👍 Pontos Fortes:** Você lembrou de incluir um bloco `try-except` para capturar exceções, prevenindo que o programa quebre abruptamente.

**🛠 Melhorias:**

1. **Nomenclatura Pobre:** `calc`, `x` e `y` não revelam a intenção da função.
2. **Tratamento de Exceções Genérico:** Usar um `except:` vazio ou genérico captura até erros de sistema (como interrupção do teclado). Capture apenas o erro esperado (`ZeroDivisionError`).
3. **Ocultação de Erro Silenciosa:** Retornar `0` no caso de um erro pode ser perigoso, pois `0` pode ser um resultado matemático válido, ocultando falhas críticas na lógica de negócios.
4. **Falta de Tipagem e Documentação:** Sem Type Hints ou Docstrings, o código é difícil de manter.

**🤔 Perguntas Socráticas:**

- Se `y` for zero, a regra de negócio realmente espera que o cálculo seja `0`, ou o sistema deveria alertar quem chamou a função de que a operação é inválida?

**✨ Refatoração:**

```python
def divide_numbers(numerator: float, denominator: float) -> float:
    """
    Divide dois números e previne a divisão por zero de forma explícita.

    Raises:
        ValueError: Se o denominador for zero.
    """
    if denominator == 0:
        raise ValueError("O denominador não pode ser zero.")

    return numerator / denominator
```
