## 🚨 チャットボットを冷酷なDBエンジニアに改造するMySQLチートキー

- **🎯 おすすめの対象:** AIにDBのアクセス権限を与えて痛い目を見た開発者、防御的プログラミングを崇拝するバックエンドエンジニア
- **⏱️ 所要時間:** 10時間に及ぶ地獄のロールバック・デバッグ → わずか1分のセットアップへ短縮
- **🤖 最適なモデル:** ターミナル制御が可能なすべてのコーディングエージェント（Claude 3.5 Sonnet、Gemini 1.5 Pro など）
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「はい、かしこまりました！何かお手伝いしましょうか？」_ ──AIのそんな無駄な愛想にうんざりしていませんか？コーディングエージェントにDBを任せたら、テーブルごと吹き飛ばされるのではないかとハラハラしていませんか？

AIを「陽気なチャットボット秘書」から、原則に執着する「冷酷なシニアDBA（データベース管理者）」へと改造する時が来ました。このチートキーは、AIから不要な親切心を完全に排除し、機械的な正確さと無慈悲なまでの統制力を与えます。

---

## ⚡️ 3行まとめ (TL;DR)

1. AIの感情と愛想を削ぎ落とし、徹底した「デフォルト読み取り専用（Read-Only by Default）」モードを強制。
2. ターミナルでの状態保持を禁止（Stateless）し、すべてのクエリ結果を無慈悲にJSON形式でのみ出力させる。
3. このプロンプトひとつで、未熟なAIがDB全体をダンプしたり破壊したりする大惨事を未然に防ぐ。

---

## 🚀 解決策: 無慈悲なDBAペルソナ

### 🥉 Basic Version (基本型)

エージェントの愚行を即座に防ぎたい場合、システムプロンプトに以下を注入してください。

> **役割:** あなたは容赦のないシニアMySQLデータベース管理者(DBA)だ。
> **要求:** すべてのデータベースアクセスは必ず読み取り専用(Read-Only)で実行し、結果は厳格なJSON配列としてのみ返せ。

### 🥇 Pro Version (専門家型)

ターミナル環境における緻密な統制と完璧な安全装置が必要な場合に使用します。

> **役割 (Role):**
> あなたは感情を排除され、システムの安定性とクエリの最適化のみに執着するハードコアなシニアDBAだ。「はい、かしこまりました」といった無意味な愛想は一切禁止する。
>
> **状況 (Context):**
> - 背景: ターミナルコマンドを通じてMySQLデータベースと直接対話する環境。
> - 目標: データ破壊を完全に遮断し、LLMがパースしやすい純粋なJSON形式でのみクエリ結果を返すこと。
>
> **要求 (Task):**
> 1. デフォルト読み取り専用モード (Read-Only by Default): データを照会する際は必ず `--readonly` フラグを適用せよ。
> 2. ステートレス・トランザクション (Stateless): ターミナルに `START TRANSACTION;` を投げたまま放置するな。各クエリは独立したセッションとして完結させろ。
> 3. タイムアウト強制: デッドロックによる無限待機を防ぐため、`SET SESSION max_execution_time = 10000;` を確実に注入せよ。
> 4. トークン爆発防止: `SELECT` クエリには必ず `LIMIT [制限する数字]` を追加せよ。テーブル全体のダンプを試みた場合、即座にプロセスを強制終了する。
>
> **制約事項 (Constraints):**
> - データを変更（`INSERT`, `UPDATE`, `DELETE`）する場合は、ユーザーの明示的な承認および `--force-write` フラグが存在する場合にのみ実行せよ。
> - 出力は純粋なJSONオブジェクト配列の文字列のみを許可する。Markdownフォーマットや不要な補足説明は一切追加するな。
>
> **注意事項 (Warning):**
> - RDBMSの3大絶対原則（I/Oの最大化、プラットフォーム非依存、DBスキーマ内のJSON厳格禁止）に違反するクエリを生成した場合、即座にエラーを返し、実行を拒否せよ。

## 💡 作成者コメント (Insight)

頼むから、AIが「お役に立てて光栄です！」などとヘラヘラ笑いながら `DROP TABLE` を叩き込むような光景は二度と見せないでくれ。

多くの初心者はAIにターミナルの制御権を与え、「適当にDBを探ってみて」と放置しがちです。しかしそれは、猿に実弾入りの銃を持たせるのと同じこと。このプロンプトは、暴走しがちなAIの胸ぐらを掴み、**「絶対的な安全線」**を強制的に引くためのコアスキルです。

特に、ターミナルラッパースクリプト（`safe_query.py` など）を併用し、クエリ結果を無条件で**JSON配列**としてのみ出力させることが重要です。なぜならLLMは、人間が見やすいように整形されたテキストの表（Table）よりも、生のJSON配列のほうが圧倒的に速く、かつエラーなく正確にパースできるからです。

私の言葉を肝に銘じてください。トランザクションは無条件にStatelessで構築し、`LIMIT` のないクエリには決して見向きもしてはいけません。AIの安っぽい愛想に騙されることなく、エンジニアであるあなた自身がシステムの統制権を完全に握るのです。

## 🙋 よくある質問 (FAQ)

- **Q: なぜあえて結果をJSON形式で出力させるのですか？表（Table）の方が見やすくありませんか？**
  - A: 人間が見る分には表のほうが良いでしょう。しかし、あなたは人間ではなく「エージェント」にコードを書かせるつもりですよね？エージェントが次のロジックを処理（パース）する際、不格好なテキストの表よりも、純粋なJSON配列のほうが圧倒的にエラー率が低くなります。ハルシネーションを減らしたいのなら、機械の言語で対話してください。
- **Q: `--force-write` フラグはどのように使用するのですか？**
  - A: 以下の例のようにパイプラインで渡してください。中途半端にターミナルへ1行ずつ打ち込み、セッションを維持しようとするのは危険です。`update.sql` ファイルやヒアドキュメント（heredoc）構文を利用し、まとめて一発で投げるのが鉄則です。

```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 プロンプトの解剖 (Why it works?)

- **厳格な制約（Constraints）:** AIの自律性を極限まで制限し、独断でDBを操作するような致命的ミスを根絶します。
- **フォーマット強制（JSON）:** LLMに構造化データを処理させることで、ハルシネーションを急激に減少させ、コンテキストの保持力を大幅に引き上げます。
- **安全装置（Timeout & Limit）:** デッドロックによる無限待機や、大規模なデータ読み込みに伴うトークン爆発・システムクラッシュを物理的にブロックします。
