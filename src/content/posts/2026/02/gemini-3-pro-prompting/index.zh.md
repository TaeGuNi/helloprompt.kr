100% 激发 Gemini 3 Pro 多步推理（Reasoning）与上下文穿透力的高阶结构化提示词。

> **角色 (Role)：** 你是一位拥有 10 年以上实战经验的资深软件架构师兼商业战略专家。
>
> **背景 (Context)：**
>
> - 当前背景：我们目前正面临`[当前的业务问题或所处境况]`。
> - 最终目标：为了顺利达成`[期望实现的具体最终目标]`，必须设计出一套最优的系统流程。

>
> 1. **定义问题 (Problem Definition)：** 深度解剖给定的背景与约束条件，精准锁定最核心的系统瓶颈或业务痛点。
> 2. **探索方案 (Solution Exploration)：** 在不可违背的硬性约束下，提出 3 个切实可行的备选方案，并客观且犀利地评估每个方案的利弊（Trade-off）。
> 3. **评选最优解与逻辑推演 (Optimal Selection & Reasoning)：** 从上述 3 个备选方案中筛选出最完美的一个，并严密论证“为何它是当下的最佳选择”，坚决杜绝任何逻辑跳跃。
> 4. **制定执行计划 (Action Plan)：** 针对选定的最优解，制定一份哪怕明天立刻开工也能直接上手落地的具体步骤逻辑（Step-by-step logic）。
>
> **输出格式 (Format)：**
>
> - 请使用明确的 Markdown 标题（`###`）清晰划分每个步骤。
> - 在第 4 步的“执行计划”中，请采用流程图风格的文本或层级分明的编号列表，确保开发人员或业务执行方能一目了然。
>
> **注意事项 (Warning)：**
>
> - 严禁提出任何触碰或违反上述“约束条件”的备选方案。（防止大模型产生幻觉）
> - 绝对不允许直接跳出结论，必须从第 1 步起，完整、透彻地展现所有的推理链路。

## 🧬 提示词解剖 (Why it works?)

1. **强制顺序思考 (Forced Sequential Thinking)：** 严苛规定大模型必须依循“第 1 步至第 4 步”的线性顺序进行深度思考，完美封死了 AI 出现逻辑跳跃或偷工减料的任何可能。
2. **引入权衡分析 (Trade-off Analysis)：** 在“探索方案”阶段强迫其赤裸裸地剖析每个备选方案的优劣，这不仅极大拔高了方案的含金量，更能帮助业务决策者保持清醒，看透“看似完美的方案背后，究竟暗藏着哪些技术债风险”。
3. **严苛的约束条件追踪：** 极致压榨了该模型极其彪悍的超长上下文穿透力，宛如加装了一道坚不可摧的安全锁，确保在初始阶段设定的严苛开发条件能贯穿始终，并最终精准映射到落地的执行计划之中。

## 📊 效果对比：Before & After

### ❌ Before (使用简单的指令型提示词)

> **输入：** “我们线上商城打算接入定期订阅扣款功能。开发周期只有 2 周时间。帮我写个开发逻辑。”

```text
关于接入定期订阅扣款功能的业务逻辑如下：
1. 设计并新建订阅商品相关的数据库表结构。
2. 用户在前端页面点击“定期订阅”按钮时，调出输入框登记绑定的信用卡信息。
3. 设定系统定时任务（Scheduler），在每月 1 日自动请求绑定银行卡进行扣款。
4. 支付成功回调后，自动触发订单中心的商品发货流程。
```

*(结果反馈：满篇皆是人尽皆知的“正确的废话”。它完全无视了“只有 2 周开发时间”这个极其致命的约束条件，更未对现有系统的集成风险进行任何实质性排查，根本毫无落地价值。)*

### ✅ After (使用 Pro Version 多步推理提示词)

> **输入：** (在 Pro Version 提示词的对应位置填入详尽上下文：`[当前背景：目前系统仅支持单次付款，历史包袱重]`, `[约束条件：开发周期仅有极限的 2 周，后端开发人员仅有 1 人，绝对不允许对现有核心 DB 结构进行大规模修改]` 等细节)

```text
### 1. 定义问题 (Problem Definition)
在当前以单次支付为核心的重度遗留系统环境中构建全新的定期订阅机制，最大的致命瓶颈在于“开发周期仅有 2 周，且后端人力只有 1 人”。在如此极端的资源匮乏状态下，任何试图大规模修改现有核心业务数据库表结构（Schema）或从零开始自研全新订阅扣款引擎的提议，都必然导致项目延期甚至系统崩溃。

### 2. 探索方案 (Solution Exploration)
- 备选方案 A：完全独立地建立全新订阅数据库集群，并开发一整套配套的订阅支付 API。（稳定性最高，架构最解耦，但工期严重超标，2 周内绝对无法交付）
- 备选方案 B：直接在现有的高频读写 `Order` 核心订单表中粗暴地追加一个 `is_subscription` 字段进行硬编码判断。（开发速度极快，但系统扩展性被彻底破坏，未来极易引发锁表故障，技术债务极高）
- 备选方案 C：优先对接支付网关（PG）的 Billing Key（绑卡代扣令牌）下发 API。随后编写一个完全解耦的独立外部 Cron Worker 定时任务，让这个 Worker 扮演虚拟用户的角色，每月定时去调用系统现已跑通的“单次支付创建 API”。（将对现有核心 DB 的侵入性降至最低限度，1 名开发人员完全可把控并在 2 周内实现闭环）

### 3. 评选最优解与逻辑推演 (Optimal Selection & Reasoning)
综合评估技术风险与工期红线，**备选方案 C** 无疑是最优解。
该方案几乎无需触碰现有系统脆弱的核心账务逻辑与底层数据库结构，最大程度规避了重构带来的回归测试压力与副作用（Side-effect）。整个核心逻辑被简化为“外部定时器在指定时间使用 Billing Key 请求第三方网关扣款 -> 扣款成功后复用内部稳定运转多年的单次支付 API 写入常规订单数据”。由于大量复用了已有的成熟接口，这是在仅仅 2 周的极限工期内唯一具有现实可交付性的防御性架构设计。

### 4. 制定执行计划 (Action Plan)
1. [D-14] 核心对接：后端完成第三方支付网关 Billing Key 下发及绑卡 API 的沙盒联调测试。
2. [D-12] 前端适配：在现有商品详情页注入“定期订阅”按钮状态，并接入网关提供的标准化绑卡 UI 组件。
3. [D-10] 后端极简建表：仅在非核心库新增一张极轻量的 `User_Billing` 映射表，用于安全加密存储用户的 Billing Key 及用户 UID 关联关系。
4. [D-07] 外部 Cron Worker 搭建：设定 Node.js/Python 独立定时任务，于每日 00:00 扫表找出当日需扣款的用户名单，并并发调用网关发起扣款请求。
5. [D-04] 复用订单 API：Cron Worker 监听到扣款成功 Webhook 后，在内网安全环境下调用原系统的 `createOrder` API 进行标准订单数据沉淀。
6. [D-01] 异常兜底流转：完成扣款失败（如信用卡余额不足、卡片过期）的兜底重试机制（Retry Policy），并联调自动发送催缴提醒短信的补救逻辑。
```

*(结果反馈：AI 完美消化了所有苛刻至极的约束条件，输出了一套极其接地气、直击痛点且充分贯彻“防御性编程”理念的架构级备选方案；其“为何选它”的论证过程严丝合缝；最终更是直接抛出了一份按天倒排、明天一早就可分发到早会上的里程碑级执行计划。)*
