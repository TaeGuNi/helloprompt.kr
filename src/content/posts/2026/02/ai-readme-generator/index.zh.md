- **🤖 推荐模型：** 所有对话型 AI（ChatGPT, Claude, Gemini 等）

> _"无论你的代码多么优雅，如果 README 写得很糟糕，没人会愿意打开你的项目。"_

你是否也曾完成了一个出色的开源项目或个人作品，却迟迟不愿动笔写 `README.md`？对开发者而言，写文档往往比写代码更让人头疼。“安装步骤该怎么写？”、“功能介绍要写多详细？”、“许可证和徽章（Badge）要怎么加？”——在这些纠结中，项目最终常常落得个空文档的下场，甚至直接被埋没。

现在，请把宝贵的精力集中在代码开发上。只需把项目的文件目录树和核心技术栈交给 AI，它瞬间就能为你生成一份符合全球开源生态标准、排版精美的项目主页。

## 🚀 解决方案："AI README Architect"

### 🥉 Basic Version (基础版)

适用于只想快速搭建项目文档骨架的场景。

> **角色 (Role)：** 你是一位 `[资深技术撰稿人]`。
> 
> **任务 (Task)：** 请根据我项目的 `[文件结构]` 和 `[主要技术栈]`，编写一份格式清晰、结构完整的 Markdown 版 README.md 草稿。

### 🥇 Pro Version (专业版)

适用于需要符合开源标准、追求完美细节及视觉元素（徽章、Emoji 等）的场景。

> **角色 (Role)：** 你是一位管理着全球顶尖开源项目的 `[资深维护者兼技术撰稿人]`。
>
> **背景 (Context)：**
>
> - 背景：我刚刚发布了一个新的业余项目，需要一份极具吸引力且通俗易懂的项目介绍文档，以便其他开发者能够轻松理解并参与贡献。
> - 目标：生成一份完全符合 GitHub 规范的高质量 `README.md`。
>
> **任务 (Task)：**
>
> 请根据以下项目信息，编写完整的 `README.md` 代码。
>
> **[项目信息]**
>
> - **项目名称：** `[项目名称]`
> - **一句话简介：** `[用一句话说明项目的核心价值]`
> - **技术栈：** `[例如：React, TypeScript, TailwindCSS]`
> - **文件结构：**
>
>   [在此粘贴 tree 命令的输出结果]
>
> **约束条件 (Constraints)：**
>
> 1. **Badges（徽章）：** 在文档顶部放置与项目状态、许可证以及所用技术栈相符的动态 Shield 徽章（Shields.io）。
> 2. **Features（功能特性）：** 深入分析所提供的文件结构和技术栈，推导出 3~5 个核心功能，并以带 Emoji 的无序列表形式展示。
> 3. **Getting Started（快速开始）：** 清晰且分步骤地编写开发环境的配置及运行指南，例如 `npm install`, `npm run dev` 等。
> 4. **License（许可证）：** 在文档底部明确添加 MIT 许可证部分。
> 5. **输出格式：** 请将所有生成的内容完整包裹在一个 Markdown 代码块（` ```markdown `）中输出，不要附加任何多余的解释。
>
> **注意事项 (Warning)：**
>
> - 绝对不要凭空捏造文件结构中不存在的功能或依赖包。（严禁幻觉）
> - 全文保持简洁、专业的语调（Tone & Manner）。

## 💡 作者见解 (Insight)

这个提示词（Prompt）的核心价值不仅在于“生成文本”，更在于它能强制 AI 遵守开源生态中的“潜规则”——例如徽章的摆放位置、目录层次的划分，以及安装指南的逻辑流向。

特别是通过 `tree` 命令直接提供**文件结构**，这是整个指令的精髓所在。AI 仅仅通过观察文件夹和文件名（如 `auth.ts`, `payment.controller.ts` 等），就能惊人地准确推断出你的项目实现了哪些具体业务逻辑，从而生成极具针对性的功能描述。

**🍯 小贴士：** AI 生成的文档是一份极佳的“初稿”。在此基础上，如果你能亲自补充上实际运行的 UI 截图或演示 GIF，那么一份无可挑剔的个人作品集就正式诞生了。如果你还没有项目 Logo，不妨尝试使用 DALL-E 3 或 Midjourney 快速生成一个，这会让你的 GitHub 主页看起来更加专业。

## 🙋 常见问题 (FAQ)

- **Q: 我的文件目录树太庞大了，超过了 AI 提示词的输入字数限制怎么办？**
  - A: 请在提取目录时排除 `node_modules`, `dist`, `.git` 等与核心业务逻辑无关的构建产物。在终端运行 `tree -I 'node_modules|dist|.git'` 命令，即可干净利落地提取出项目的核心脉络。

- **Q: 我想生成一份英文版本的 README 以面向全球开发者，该怎么做？**
  - A: 非常简单。只需在提示词的 **任务 (Task)** 最后追加一句：*“请将所有内容使用自然、地道的商务英语编写。”* 这样就能瞬间生成一份具备国际视野的英文版 README。

- **Q: 能够顺便生成贡献者指南（CONTRIBUTING.md）吗？**
  - A: 当然可以。在生成 README 之后的后续对话中，你可以追加指令：*“基于以上项目信息，请继续为这个项目编写一份开源贡献者指南（CONTRIBUTING.md）和 PR 模板（pull_request_template.md）。”*

## 🧬 提示词解剖 (Why it works?)

1. **赋予角色 (Role)：** 通过赋予 AI“资深维护者”的专家人设，我们成功引导它自动补全初级开发者极易遗漏的动态徽章、许可证声明等开源必备元素。
2. **提供背景 (Context) 的力量：** 详实的“文件结构”提供了强有力的底层逻辑支撑，迫使 AI 彻底放弃假大空的套话，生成完全契合当前代码库的定制化说明。
3. **约束条件 (Constraints) 的边界：** 明确警告 AI 不要捏造功能，有效阻断了大型语言模型的幻觉（Hallucination），从根本上保证了技术文档的真实性与可信度。
