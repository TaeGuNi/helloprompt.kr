## 📝 複雑なコードはもう卒業！AI複雑度分析器 (Code Complexity Analyzer)

- **🎯 おすすめ対象:** ジュニア開発者、レガシーシステムのメンテナンス担当者、厳しいコードレビュアー
- **⏱️ 所要時間:** 1時間 → 3分に短縮
- **🤖 推奨モデル:** GPT-4o, Claude 3.5 Sonnet (コード推論能力に優れたモデルを推奨)

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _"if文の中にfor文、その中にまたif文... 何とか動いてはいるけれど、触れた瞬間に爆発しそうな時限爆弾のようなレガシーコードを前に、絶望したことはありませんか？"_

開発をしていると、誰もが一度は息が詰まるような**「スパゲッティコード」**に遭遇します。
絶え間ない機能追加や仕様変更、そしてリリーススケジュールに追われて急いで詰め込まれたパッチ当てのようなバグ修正（Hotfix）が何度も繰り返されるうちに、コードは次第に肥大化し、収拾がつかないほど複雑に絡み合っていきます。
**循環的複雑度 (Cyclomatic Complexity)** は幾何級数的に跳ね上がり、一つの関数が数百行を軽々と超える恐ろしい怪物がお誕生するのです。

このようなレシーコードは、単にコードの物理的な可読性を低下させるだけではありません。
些細なボタンの色一つ、テキスト一つを修正しようとしても、システムの全く予期しない場所で**予期せぬサイドエフェクト (Side Effect)** が発生するかもしれず、開発者を極度の不安に陥れます。
「この条件文は一体どんなビジネスロジックのために存在しているのか？」「このグローバル変数はどこで状態が変容 (Mutation) されているのか？」
コードを読んで意図を把握することだけに貴重な一日の業務をすべて費やすことになり、本来価値のある開発よりも過去の残骸を分析することに多くのエネルギーを注ぐことになります。
明日が定期デプロイ日なのに、絡み合ったコードの沼でもがきながら夜を明かした経験、どこから手をつければいいか分からずただモニターをぼんやりと眺めて絶望した経験、現役の開発者なら誰もが骨身にしみて共感できるはずです。

特に**認知的複雑度 (Cognitive Complexity)** が異常に高いコードは、個人の苦しみを超えてチーム全体の生産性を著しく阻害する主犯となります。
新しく合流したチームメンバーは膨大なコードベースの流れを把握できず、オンボーディング期間がいつまでも長引き、10年目のシニア開発者でさえコードレビューで一体どこから指摘すべきか分からず途方に暮れます。
結局、「とにかくプロダクションで動いているから、とりあえず触らないでおこう」という暗黙的で妥協的なルールがチーム内に定着し、システムは次第に耐えられないほどの技術的負債 (Technical Debt) に押しつぶされ、徐々に崩壊の道を歩むことになります。
リファクタリングの第一歩であるテストコードを書こうとしても、モジュール間の依存関係が強すぎる (Tightly Coupled) ため、Mockingすら不可能な恐ろしいジレンマに直面することになるのです。

私たちはしばしば「今回のスプリントさえ終われば、来週こそはリファクタリングしよう」という空虚な自己満足とともに技術的負債を放置します。
しかし、あいにくその「来週」は決して来ず、コードは複利の魔法のようにさらに恐ろしく絡まっていくだけです。
**レガシーコード (Legacy Code)** を作り直す作業は、あたかも時速100kmで走る自動車のタイヤを走行中に付け替えるようなものです。
ライブシステムは絶対に止まってはならず、ビジネスのための新しい機能は絶えず追加されなければならず、当然新しいバグを作ってもいけないからです。
この極限の制約とプレッシャーの中で、人間の脳は限界を迎えます。`if` 条件文がわずか3〜4回深くネストされた瞬間、開発者の認知能力は急激に低下し、致命的な論理的ミスを犯すことになります。

しかし今、どこから手をつければいいか分からなかったその大きな恐怖と途方に暮れる思いを完璧に解消してくれる究極のツールがあなたの前にあります。
それが**「AIコード複雑度分析器 (Code Complexity Analyzer)」**プロンプトです。
もはや開発者の漠然とした直感や危うい感覚に頼った危険極まりないリファクタリングは必要ありません。
このプロンプトを通じてAIは、あたかも私の隣に座ってコードを細かく見てくれる厳格で実力のあるシニアペアプログラマーのように、客観的なソフトウェア工学の指標に基づいてコードの致命的な問題点を正確かつ鋭く診断してくれます。

さらに驚くべきことは、AIが単にコードを短くしてくれる一次元的なフォーマッター (Formatter) の役割をするのではないという点です。
AIは**デザインパターン (Design Pattern)** と**クリーンコード (Clean Code)** の原則に基づき、実務のプロダクション環境に即座に適用可能な具体的でエレガントなアーキテクチャ改善戦略をわずか1分で明確に提示します。
既存のビジネスロジックの整合性は100%安全に維持しながら、際限なく深まる `if-else` 地獄を**アーリーリターン (Early Return)** ですっきりと広げ、数百行の巨大な関数を**単一責任の原則 (SRP)** に合わせて複数の小さなモジュール型関数に完璧に分離してくれます。
このプロンプトがあれば、恐れるばかりだったレガシーコードも、あなたの優れたエンジニアリング能力を証明する素晴らしいキャンバスになるでしょう。同僚から称賛されるきれいな Pull Request を作成し、真の開発の楽しさを取り戻してください。

## 📊 証明：爽快な結果 (Before & After)

### ❌ Before (私たちが経験していた苦しみ)

めちゃくちゃに絡み合った手作業のコード、if文の中にfor文が際限なくネストされ、どこから手をつければいいか途方に暮れていたレガシーコードの典型です。ビジネスロジックを把握するだけで数十分かかります。

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ After (完璧に変身した結果)

![複雑なコードはもう卒業！AI複雑度分析器 (Code Complexity Analyzer)](/images/hooks/code-complexity-analyzer.jpg)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: 例外状況の早期リターン（ネストの削減）
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. 高階関数の活用: ループの単純化
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. ビジネスロジックの分離と単純化
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```

## 💡 執筆者コメント (Insight)

このプロンプトは、私が実務で他人が書いた——あるいは過去の私が無責任に書いた——レガシーコードに遭遇したとき、真っ先に取り出す究極の武器です。
開発者なら誰でも他人のコードを読むことを苦痛に感じます。特に変数名が不明確で条件文が乱舞するスパゲッティコードは、見るだけでため息が出て、どこから手を付ければいいのか見当もつきません。
しかし、AIは苛立ちません。感情の動揺なく、最も客観的で冷徹な視点でコードの構造的な欠陥を解剖し、精密射撃します。

このプロンプトが強力な理由は、単にコードの物理的な長さを短くする一次元的な作業に留まらない点にあります。
「現在この条件文は深度 (Depth) が4段階に達しているため、**Guard Clauses (早期リターン)** パターンを適用して上部に分離するのが構造的に妥当です」といったように、コードレビューで同僚を完璧に納得させられる合理的で論理的な名分を与えてくれます。
私たちがよく犯すミスの一つは、リファクタリングを「単に自分のコーディングスタイルに合わせて、自分の気に入るように直すこと」と勘違いすることです。
しかし、真の意味のリファクタリングは明確な理由と客観的なソフトウェア工学の原則に基づかなければならず、このプロンプトはその原則をAIの口を借りてあなたに強力に提供します。

プロンプト内の **`[変数]` コントロール法**は、このプロンプトの真の価値を爆発させる核心的なチートキーです。
例えば、`[TypeScript/NestJS]` という技術スタック変数はAIに膨大なコンテキスト (Context) を与えます。
単にJavaScriptレベルの文法的リファクタリングではなく、フレームワークが提供する依存性の注入 (Dependency Injection) をどのようにさらにエレガントに活用するか、カスタムデコレーター (Decorator) を生成して横断的関心事 (Cross-cutting Concerns) をどのように分離するかなど、**該当するフレームワークのエコシステムに最適化された最高級のデザインパターン**を引き出すことができます。
もし皆さんが `[Python/FastAPI]` を使用中だと変数を修正すれば、Pydanticモデルを活用したデータ検証ロジックの分離や、非同期 (Async) 処理のボトルネック最適化ポイントを指摘してくれるでしょう。
言語とフレームワークに合わせてプロンプトをアレンジした瞬間、汎用的なコーディングAIはあなたのプロジェクトに完璧に同化した**特級ドメインエキスパート**へと変貌します。

特にこのプロンプトは、ギットハブ (GitHub) に Pull Request (PR) を上げる直前、自分のコードを**セルフレビュー (Self-Review)** するときに真の威力を発揮します。
コードを書いた直後には「自分が書いたコードだから構造的に完璧だ」という確証バイアス (Confirmation Bias) に陥りやすいものです。
しかし、このプロンプトを通じて一度フィルタリングを行うだけで、結果は驚くほど変わります。思いもよらなかったエッジケース (Edge Case) に対する例外処理や、結合度 (Coupling) を下げることができる構造的な改善点を鋭く指摘してくれるからです。
このプロセスを経たコードは、同僚の開発者たちから「今回のコードアーキテクチャ、本当に芸術的ですね！」「可読性が良すぎてレビューすることがありません」という痺れるような賛辞を引き出すことができます。

また、**制約事項 (Constraints) の強力なコントロール**部分に格別注目する必要があります。
`元のコードの核心的なビジネスロジックはいかなる場合でも絶対に変更されてはいけません。` という文章は、AIの致命的なハルシネーション (Hallucination) を防いでくれる最も心強い絶対防衛線です。
LLMはしばしばコードを「きれいで短く」することに夢中になるあまり、例外状況を処理する重要な防衛ロジックを無断で削除したり、ビジネスルール自体を微妙に変えてしまう取り返しのつかないミスを犯します。
このような不祥事を根源的に遮断するために、動作の不変性をプロンプトレベルで厳格にコントロールすることで、安心してプロダクション (Production) 環境にデプロイできる信頼性の高いリファクタリングコードを手に入れることができるのです。
実務で明日からすぐに役立つこの強力な武器を皆さんの開発武器庫にしっかりと装備し、技術的負債との戦いに勝利してください。

## 🙋 よくある質問 (FAQ)

- **Q: ソースコードファイル全体を一度にすべて貼り付けて分析しても大丈夫でしょうか？**
  - A: できるだけおすすめしません。LLMのコンテキストウィンドウの限界により、指示事項への集中度が落ち、ハルシネーションが発生する可能性があります。数百行を超える膨大なファイルを丸ごと入れるよりも、50〜150行程度の「特定の関数」や「単一クラス」単位で区切って入力してみてください。AIが問題の核心をはるかに鋭く突き、驚くほど正確なリファクタリングの洞察を提示してくれるでしょう。

- **Q: AIが完璧に直してくれたコードをそのままプロダクション (Production) 環境にデプロイしても安全ですか？**
  - A: 絶対にそのままコピー＆ペーストしてはいけません！いかに優れたAIであっても、ビジネスロジックの微妙なエッジケースを見落としたり、意図せずロジックを変えてしまうハルシネーション (Hallucination) 現象を起こす可能性があります。AIが作成したコードは素晴らしい「下書き」に過ぎません。**必ず既存のユニットテスト (Unit Test) がすべて正常にパスするか確認**し、開発者が直接ロジックの整合性を細かく検証した後にマージ (Merge) する必要があります。

- **Q: 私が使用している特定のプログラミング言語やフレームワークの特性に合わせてコードを改善してくれますか？**
  - A: もちろんです。専門家型 (Pro) プロンプトの `[TypeScript/NestJS]` 変数の位置に 'Java/Spring Boot'、'Python/FastAPI'、'Go/Gin' など、現在実務で使用している技術スタックを正確に明記してみてください。AIが該当する言語とフレームワークのエコシステムで公式に推奨されている命名規則 (Naming Convention) と最適なデザインパターンを反映して、カスタマイズされた形でコードを整えてくれます。

1. **ソフトウェア工学指標中心の精密射撃:** AIに「コードを単にきれいに整理して」と曖昧に命令する代わりに、「循環的複雑度 (Cyclomatic Complexity)」という具体的な指標を明示しました。これにより、AIが単なるフォーマットを超えて、アーキテクチャの根本的な構造欠陥を解決することにコンピューティングパワーを集中させるように設計しました。
