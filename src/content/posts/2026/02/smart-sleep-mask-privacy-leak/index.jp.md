## 📝 スマート睡眠マスクが脳波をダダ漏れにしている件

- **🎯 おすすめ対象:** IoT開発者、セキュリティエンジニア、フロントエンドエンジニア

> _"あなたの脳波データ、寝ている間にBluetoothで近隣住民に丸見えになっているかもしれませんよ？"_

最高の睡眠を手に入れるため、テック業界のトレンドは手首のトラッカーから、より身体に密着した「スマート睡眠マスク」へとシフトしました。サーカディアンリズムをハックし、明晰夢をコントロールすると謳うこれらのデバイスは、2026年の今やすっかり日常の光景です。

しかし、セキュリティを第一に考えるエンジニアにとって、これはIoT（モノのインターネット）における**新たな悪夢の最前線**です。

市場で大ヒットしているウェアラブル端末「Neuro-Sleep（ニューロスリープ）」を調査したところ、驚愕の事実が判明しました。機密性の高いEEG（脳波）データをデバイス内に安全に保管したり、暗号化通信を使用したりする代わりに、一部のデバイスは**生の脳波データを標準のBluetooth Low Energy (BLE) プロトコルで垂れ流し**にしていたのです。多くの場合、PINコードの入力やハンドシェイクすら不要という、信じがたい無防備な状態でした。

---

## ⚡️ 3行まとめ (TL;DR)

1. **認証なしのBLE通信の恐怖:** 多くのスマート睡眠マスクがペアリングなしでGATTサーバーを公開しており、誰でも脳波データを傍受できる状態にある。
2. **生体データは変更不可能:** 脳波はストレス状態や感情まで読み取れる究極の個人情報であり、パスワードのように「流出したから変更する」ことができない。
3. **AIによるコード監査の必須化:** 開発の初期段階でAIを活用し、BLE接続の脆弱性を自動で検知・修正するフローを組み込むことが急務である。

## 💡 筆者コメント (Insight)

IoTデバイスの開発現場では、コンパニオンアプリとの「**接続のしやすさ**」や「**バッテリー寿命の最適化**」を優先するあまり、GATT（Generic Attribute Profile）の特性（Characteristics）を**誰でも読み取れる状態**に放置してしまうケースが後を絶ちません。

このプロンプトが圧倒的に強力な理由は、**「ファームウェア側の要件」にまでAIを踏み込ませる点**にあります。フロントエンド（Web Bluetooth APIなど）のコードを渡すだけでも、AIは「クライアント側が認証なしで通信できているということは、そもそもデバイス側のパーミッション設定が極めて甘い」と鋭く推論し、システム全体のアーキテクチャの改善を提案してくれます。生体データを扱うプロダクトにおいて、このAIプロンプトはコードレビューの第一関門として**絶対に欠かせないプロセス**です。

## 🙋 よくある質問 (FAQ)

- **Q: Web Bluetooth API以外のネイティブアプリのコード(Swift/Kotlin)でも監査可能ですか？**
  - A: はい、全く問題ありません。コンテキスト内の「Web Bluetooth API」という記述を「Core Bluetooth (iOS)」や「Android Bluetooth LE」に変更するだけで、各プラットフォームに特化した精度の高い脆弱性監査が可能です。

- **Q: デバイス側のソースコード（C/C++）がないと正確な監査はできませんか？**
  - A: クライアント側（アプリ側）の接続コードだけでも、どのような通信方式が採用されているか（暗号化の有無など）は十分に判別可能です。もちろん、ファームウェア側のコードを `[対象コード]` に直接入力すれば、よりピンポイントで精度の高い脆弱性特定が可能になります。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **具体的な脅威へのフォーカス:** 「ペアリングの欠如」「GATTサーバーの権限設定」と、AIに探すべき脆弱性の的を意図的に絞り込むことで、一般的なコードレビューよりも圧倒的に精度の高い指摘を引き出しています。
2. **コンテキストの重大性提示:** 「生体データ(EEG)を扱っている」という背景を明示することで、AIがデータ保護の重要度（Severity）を最高レベルに見積もり、一切妥協のない厳格な修正案を提示するよう設計されています。

## 📊 証明: Before & After

### ❌ Before (入力コード: 脆弱性あり)

```typescript
// Web Bluetooth APIを使用した仮想の攻撃ベクター（誰でも脳波データを読み取れる）
navigator.bluetooth
  .requestDevice({
    filters: [{ namePrefix: "DreamWeaver" }],
    optionalServices: ["0000ffe0-0000-1000-8000-00805f9b34fb"], // カスタムEEGサービス
  })
  .then((device) => device.gatt.connect())
  .then((server) =>
    server.getPrimaryService("0000ffe0-0000-1000-8000-00805f9b34fb"),
  )
  .then((service) =>
    service.getCharacteristic("0000ffe1-0000-1000-8000-00805f9b34fb"),
  )
  .then((characteristic) => characteristic.startNotifications())
  .then((characteristic) => {
    characteristic.addEventListener(
      "characteristicvaluechanged",
      handleBrainwaves,
    );
  });
```

### ✅ After (AIの出力: 修正案と指摘)

````markdown
**🚨 検出された脆弱性:**
現在の実装では、BLEのペアリング（Bonding）プロセスを要求せずに直接GATTサーバーに接続し、Notificationを開始しています。これは、近くにいる第三者がユーザーの脳波データ（EEG）を傍受できることを意味します。

**🛠️ 修正アプローチ:**
フロントエンド側でのハンドリング強化に加え、**デバイス（ファームウェア）側でCharacteristicのパーミッションを `Encrypted Read/Notify` に設定する**必要があります。以下はクライアント側でセキュアな接続を前提とした実装例です。

```typescript
// セキュアな実装例（暗号化通信を要求）
navigator.bluetooth
  .requestDevice({
    filters: [{ namePrefix: "DreamWeaver" }],
    optionalServices: ["0000ffe0-0000-1000-8000-00805f9b34fb"],
  })
  .then((device) => {
    // デバイス側がセキュリティモード1・レベル2（暗号化必須）以上を要求する場合、
    // OSレベルで自動的にペアリングのダイアログがトリガーされます。
    return device.gatt.connect();
  })
  .then((server) =>
    server.getPrimaryService("0000ffe0-0000-1000-8000-00805f9b34fb"),
  )
  .then((service) =>
    service.getCharacteristic("0000ffe1-0000-1000-8000-00805f9b34fb"),
  )
  .then((characteristic) => characteristic.startNotifications())
  .catch((error) => {
    console.error(
      "接続または認証に失敗しました。ペアリングを確認してください:",
      error,
    );
  });
```
````

_(※ファームウェアエンジニアへ: 該当のCharacteristic定義において、`SEC_PERMIT_READ` および `SEC_PERMIT_WRITE` を暗号化要求レベルに変更してください。)_

## 🎯 結論

生体データの漏洩は、**新たな次元のパスワード漏洩**を意味します。しかもパスワードとは異なり、あなたの脳波パターンは一生リセットすることができません。一度生体シグナルが侵害され、ユーザーIDと紐付けられてしまえば、それは永久に危険に晒され続けることになります。

開発者として、私たちは金融機関の認証情報と同等、あるいはそれ以上の厳格さで生体データを扱う責任があります。「Move fast and break things（素早く動いて破壊せよ）」というテック業界の合言葉は、人間の神経系に直接接続するデバイスには決して適用してはならないのです。

今日からAIを活用して強固なセキュリティコード監査を実施し、私たちの**「認知プライバシー（Cognitive Privacy）」**を共に守り抜きましょう！
