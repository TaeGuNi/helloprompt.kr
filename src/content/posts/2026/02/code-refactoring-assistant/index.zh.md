- **🎯 推荐对象：** 初中级开发者、负责维护遗留系统的工程师、急于上线导致代码混乱的程序员
- **⏱️ 预计时间：** 1 小时 → 缩短至 2 分钟
- **🤖 推荐模型：** Claude 3.5 Sonnet、GPT-4o、Gemini 2.5 Pro（擅长处理代码逻辑的模型）

- ⭐ **难度：** ⭐⭐☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"还在写‘只要能跑就行’的垃圾代码（Working Garbage）吗？让 AI 帮你写出符合 Clean Code 规范的艺术品吧！"_

在赶进度的巨大压力下，我们常常会写出结构混乱、难以维护的“意大利面代码”。虽然程序勉强能跑通，但对于下一个接手的同事（或者一个月后的你自己）来说，这绝对是一场噩梦。今天，我们将使用这套“重构专家”提示词，让 AI 帮你将杂乱无章的代码瞬间转化为优雅、易读且完全符合最佳实践的整洁代码（Clean Code）。

---

## ⚡️ 3句话总结 (TL;DR)

1. **彻底摆脱技术债：** 瞬间将晦涩难懂的面条代码重构为结构清晰的模块化代码。
2. **遵循最佳实践：** 自动应用 SOLID 原则与设计模式，全面提升代码的可读性与可维护性。
3. **附带详细解析：** AI 不仅提供重构后的代码，还会深入解释“为什么这样改”，助你快速提升编程思维与技术水平。

## 🚀 解决方案："Clean Code 重构专家"

### 🥉 Basic Version (基础版)

适用于代码量较小、仅需简单整理格式和命名规范的场景。

> **角色：** 你是一位拥有 10 年开发经验的高级软件工程师。
> **请求：** 请帮我重构以下代码，使其更加易读，并严格符合 Clean Code 规范。请提供重构后的完整代码，并简要说明你修改了哪些地方：
> `[在此处粘贴你的代码]`

### 🥇 Pro Version (专业版)

适用于逻辑复杂、需要深度重构、解耦或引入设计模式的核心业务代码。

> **角色 (Role)：** 你是一位顶级的软件架构师兼 Clean Code 专家。你对 SOLID 原则、设计模式以及代码的可维护性有着极致的追求。
>
> **背景 (Context)：**
>
> - 当前状况：这段代码虽然能够运行，但存在过多的嵌套（回调地狱 / 厄运金字塔），变量命名随意且不规范，同时多个功能被严重耦合在同一个函数中。
> - 目标：在绝对不改变原有业务逻辑（外部行为）的前提下，对其进行深度重构与优化。
>
> **任务 (Task)：**
>
> 1. 请深入分析原代码中存在的“坏味道（Code Smells）”。
> 2. 将臃肿复杂的函数拆分为单一职责的纯函数（Pure Functions）或小型模块。
> 3. 优化变量与函数的命名，使其具备高度的自我描述性（Self-documenting）。
> 4. 如果有更合适的现代语法或设计模式，请进行替换与升级。
> 5. 最终输出重构后的完整代码：`[你的目标编程语言，例如 TypeScript, Python 等]`
>
> **待重构代码 (Input)：**
>
>
> `[在此处粘贴需要重构的意大利面代码]`
>
>
> **约束条件 (Constraints)：**
>
> - 必须保持原有的输入与输出结果完全一致。
> - 重构后的代码请使用 Markdown 代码块包裹，并在核心逻辑处添加简短清晰的注释。
> - 请使用列表形式（不要使用表格）列出“重构前”与“重构后”的核心差异对比。
>
> **警告 (Warning)：**
>
> - 绝对不要改变核心业务规则，也不要擅自引入未经确认的第三方库，除非你明确说明并给出极具说服力的理由。

## 💡 作者点评 (Insight)

这套提示词的真正价值不仅在于“帮你把代码改好”，更在于**它就像是一位免费且随时待命的资深 Code Reviewer**。
很多时候，初中级开发者心里清楚自己的代码写得不够好，却苦于不知道该如何改进。通过仔细比对 AI 重构前后的代码，并认真阅读它给出的重构理由（坏味道分析），你能够快速学习到高级开发者的架构思维与编码习惯。强烈建议在每次提交 PR（Pull Request）之前，都先让这位“重构专家”帮你全面审查一遍，这将会大大减少 Code Review 阶段的修改意见与返工时间！

## 🙋 常见问题 (FAQ)

- **Q: AI 重构的代码可以直接复制粘贴到生产环境吗？**
  - A: **绝对不行。** 尽管 AI 的能力非常强大，但它仍可能会遗漏某些特定的边缘情况（Edge Cases）。请务必在拥有完善的单元测试（Unit Tests）覆盖的前提下进行重构，并由人工仔细 Review AI 提供的最终代码。

- **Q: 如果代码太长，导致 AI 报错或输出中断怎么办？**
  - A: 遇到超长代码时，建议按模块或函数进行拆分，分批次发送给 AI。或者你也可以在提示词中加入“请一步步指导我进行重构，先从核心逻辑开始”的指令，引导 AI 逐步完成重构工作。

---

## 🧬 提示词解剖 (Why it works?)

1. **明确架构师角色设定 (Role)：** 赋予 AI “顶级架构师”的身份，能够有效促使它调用训练数据中最高质量的代码规范与架构模式。
2. **强调不改变业务逻辑 (Context/Constraints)：** 这是代码重构的铁律。通过明确设定这条约束，能够防止 AI 擅自增删功能，从而确保重构过程的安全边界。
3. **强制要求差异对比 (Task)：** 要求列出修改前后的差异，这强迫 AI 进行深度的逻辑推演（Chain of Thought），不仅大幅提升了重构结果的准确性，还极大地增加了提示词的教学价值。

## 📊 案例对比：Before & After

### ❌ Before (输入代码 - 糟糕的嵌套与命名)

```javascript
function processUser(u) {
  if (u != null) {
    if (u.age > 18) {
      if (u.status === "active") {
        let d = new Date();
        u.lastLogin = d;
        saveData(u);
        return true;
      } else {
        return false;
      }
    } else {
      return false;
    }
  }
  return false;
}
```

### ✅ After (AI重构后 - 卫语句与清晰命名)

```javascript
/**
 * 验证用户是否处于活跃状态的成年人
 */
function isValidAdultUser(user) {
  return user?.age > 18 && user?.status === "active";
}

/**
 * 更新用户登录时间并保存状态
 */
function processUserLogin(user) {
  // 使用卫语句 (Guard Clauses) 提前返回，消除嵌套
  if (!user || !isValidAdultUser(user)) {
    return false;
  }

  user.lastLogin = new Date();
  saveData(user);

  return true;
}
```
