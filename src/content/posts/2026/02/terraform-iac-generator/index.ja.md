# 🏗️ インフラ構築、もうクリック作業はやめよう！Terraformコード生成ツール

- **🎯 おすすめの対象者:** AWSコンソールで迷子になっている開発者、インフラ管理が面倒なバックエンド開発者
- **⏱️ 所要時間:** 公式ドキュメントを漁る時間 1時間 → 3分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet（コーディングおよびアーキテクチャ設計に特化）

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐☆

> _"クリックミス1回でサーバーが吹き飛ぶあのヒヤリとした経験、いつまでコンソール画面に頼り続けるつもりですか？"_

インフラはペットではなく、家畜のように扱うべきです。サーバーに問題が発生した場合、徹夜して直すのではなく、容赦なく破棄して、全く同じサーバーを1秒で新しく立ち上げる必要があります。これを可能にするのが**IaC（Infrastructure as Code）**です。しかし、HCL（HashiCorp Configuration Language）の複雑な構文や数多くのProviderのオプションを毎回暗記して記述するのは容易ではありません。

このプロンプトは、「AWSにEC2を1つとDBを1つ接続して立ち上げて」という漠然とした要件を、即座に実務に投入可能な、**セキュリティ要件とモジュール化が適用された完璧なTerraformコード**に変換してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. 長々としたドキュメントを探す必要なく、**標準のHCL構文に準拠したコード**を即座に生成します。
2. `0.0.0.0/0`の許可などの致命的なミスを防ぐため、**最小権限のセキュリティ設定（SG、IAM）**を強制します。
3. ハードコーディングを排除し、`variables.tf`と`outputs.tf`に分離することで、**再利用性の高いモジュール**を設計します。

---

## 🚀 解決策: "インフラマスタービルダー"

### 🥉 Basic Version (基本型)

テスト用のリソースが急に1つだけ必要になった際に、手軽に使用してください。

> **役割:** あなたはシニアDevOpsエンジニアです。
> **タスク:** AWSの`[サービス名]`リソースを作成するためのTerraformコードを記述してください。
> **制約事項:** 最新のAWS Providerバージョンを使用し、ハードコーディングをせずに変数（variable）で処理してください。

\

### 🥇 Pro Version (専門家型)

セキュリティと高可用性が必須となる、実際の本番（プロダクション）環境を構築する際に使用してください。

> **役割 (Role):** あなたはクラウドセキュリティと高可用性（HA）設計に精通した、10年目のクラウドアーキテクト兼Terraformの専門家です。
>
> **状況 (Context):**
>
> - 背景: 私は現在、`[プロジェクトの説明、例: 秒間1万件のトラフィックを処理するグローバルeコマースのバックエンド]`構築のためのインフラをAWS上に設計しようとしています。
> - 目標: セキュリティ、高可用性、そして徹底したモジュール化が適用された、本番環境レベルのIaC（Infrastructure as Code）コードを取得すること。
>
> **要求 (Task):**
>
> 1. **アーキテクチャ構成:** 要求されたインフラを構築するためのメインリソース定義（`main.tf`）を作成してください。
> 2. **構造化およびモジュール化:** ハードコーディングを厳格に排除し、`variables.tf`と`outputs.tf`を明確に分離してください。
> 3. **セキュリティルールの強化:** セキュリティグループ（Security Group）には、必ず最小権限の原則（Least Privilege、`0.0.0.0/0`の無条件許可の禁止）を適用してください。
> 4. **権限管理:** 必要な場合、最小権限が付与されたIAM RoleとPolicyのコードも併せて記述してください。
>
> **制約事項 (Constraints):**
>
> - HashiCorp Configuration Language (HCL) の最新構文および AWS Provider v5.0 以上を基準に記述すること。
> - すべてのリソース名とタグ（Tag）には、`[プロジェクト名]`の接頭辞を含めること。
> - `terraform fmt`基準のインデントとスタイルガイドを完璧に遵守すること。
>
> **注意事項 (Warning):**
>
> - 実務で使用不可能な非推奨（Deprecated）オプションや、安全でないセキュリティのデフォルト値は絶対に使用しないでください。（ハルシネーションおよびセキュリティ事故の防止）

## 💡 筆者コメント (Insight)

Terraformのコード生成を完全にAIに任せる際、最も恐ろしいのは**「セキュリティ（Security）」**です。一般的なプロンプトで生成されたコードは、利便性のためにファイアウォールのポートをすべて開けてしまったり（`0.0.0.0/0`）、不必要に広い権限のIAM Roleを付与してしまうケースが非常に頻繁に発生します。

上記のPro版プロンプトの核心は、まさに**「最小権限の原則」と「モジュール化」をAIに強制する**点にあります。実務では、このプロンプトで堅牢な骨組みを先に作り、`tfsec`や`checkov`といった静的セキュリティ分析ツールをCI/CDパイプラインに組み込んで再度検証するアプローチを強く推奨します。初期インフラのセットアップに費やしていた数日間が、わずか3分に短縮される魔法を体験できるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: AWSだけでなく、GCPやAzure環境でも使えますか？**
  - A: はい、もちろんです。`[状況]`と`[制約事項]`の部分で、Providerを「Google Cloud」または「AzureRM」に変更し、要求するサービス名も各クラウドベンダーの名称に合わせて修正していただければ完璧に動作します。

- **Q: AIが古いバージョン（v0.11など）の構文でコードを出力し、エラーになります。**
  - A: 旧型のAIモデル（例：GPT-3.5）を使用する際によく発生する問題です。プロンプトに「Terraformの最新構文を使用」と明記していますが、インフラコードの生成には、可能な限りコーディングに特化した**Claude 3.5 Sonnet**や最新の**GPT-4o**の使用を推奨します。はるかに安定的で最新トレンドに合ったHCLコードを出力してくれます。

## 🧬 プロンプトの解剖 (Why it works?)

1. **Roleの付与:** 「10年目のクラウドアーキテクト」というペルソナを設定することで、単なるリソースの立ち上げを超えた、セキュリティと可用性を考慮したエンタープライズ級のアーキテクチャコードを引き出しています。
2. **Constraints（制約）:** HCLの最新構文と`terraform fmt`の遵守を強制することで、コピー後に即座に実行（`terraform init && terraform apply`）可能な、クリーンで標準化された成果物を保証します。
3. **セキュリティと例外の統制:** 「最小権限の原則」と「安全でないデフォルト値の使用禁止」を明記し、本番環境において致命的となり得るファイアウォール（SG）および権限（IAM）の乱用事故を根本から遮断しています。

## 📊 検証: Before & After

単に「作って」と指示した場合と、構造化されたプロンプトを使用した場合とでは、AIが出力するHCLコードの品質に明白な差が生まれます。

### ❌ Before (単純な入力)

```text
AWSにVPCを作って、パブリックサブネット2つとプライベートサブネット2つを作って。
```

**結果 (ハードコーディングおよびセキュリティの脆弱性):**

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
# 変数処理なしでハードコーディングされ、タグ（Tags）が欠落。拡張不可能な単一ファイル構成。
```

### ✅ After (Proプロンプト使用結果)

**結果 (モジュール化および実務標準の適用):**

```hcl
# main.tf
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.project_name}-vpc"
  cidr = var.vpc_cidr

  azs             = var.availability_zones
  private_subnets = var.private_subnets
  public_subnets  = var.public_subnets

  enable_nat_gateway     = true
  single_nat_gateway     = false # 高可用性のためにAZごとにNATを構成
  one_nat_gateway_per_az = true

  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "Terraform"
  }
}
```

_（※ これ以外にも、variables.tfやoutputs.tfが完璧に分離されて出力されます。）_
