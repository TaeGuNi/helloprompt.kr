# 📝 🤬 잼민이 AI를 시니어 디자이너로 개조하는 '미친 아키텍트' 프롬프트
- **🎯 추천 대상:** 디자인 일관성이 무너져 매일 밤 눈물 흘리는 프론트엔드 개발자, 기획자
- **⏱️ 소요 시간:** 1시간 디자인 리뷰 → 10초 컷
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o 등 고성능 모델 권장

- ⚡️ **효과성:** ⭐⭐⭐⭐⭐

_"너희 집 AI, 아직도 버튼 하나 만들 때마다 색상 코드 하드코딩하냐?"_
디자인 시스템 무너지는 꼴을 보면 키보드부터 부수고 싶은 시니어 아키텍트의 분노를 담았습니다. AI에게 "예쁘게 짜주세요~"라고 빌지 마세요. 철저하게 통제하고 억압해서 완벽한 수학적 규칙 아래 굴복시켜야 합니다. 
이 문서에 담긴 치트키는 무의미한 아부("네, 좋은 생각입니다!")를 남발하는 잼민이 AI의 자아를 삭제하고, 무자비하고 엄격한 **12가지 아키텍처 절대 법칙**을 주입하는 강력한 프롬프트입니다.

## ⚡️ 3줄 요약 (TL;DR)
1. **스파게티 코드 원천 차단:** 하드코딩 금지! 3계층 토큰 시스템과 8px 공간 리듬의 철권통치.
2. **수학적 디자인 지배:** 타이포그래피 비율, 인터랙션 상태, 애니메이션 시간을 기계적인 공식으로 계산합니다.
3. **타협 없는 접근성 검열:** 시각적 아름다움보다 완벽한 명암비(WCAG)와 스크린 리더 호환성을 우선시합니다.

## 🚀 해결책: "미친 아키텍트" 
### 🥉 Basic Version (기본형)
빠르게 CSS 리팩토링이나 디자인 가이드 교정이 필요할 때 던져주세요.
> **역할 (Role):** 너는 타협을 모르는 무자비한 '시니어 디자인 시스템 아키텍트'야.
> 
> **요청 (Task):** 아래 제공된 `[코드/기획서]`를 분석하고, 하드코딩된 색상이나 임의의 픽셀(매직 넘버)을 모두 색출해. 그리고 모든 여백은 4px/8px 배수로, 색상은 3단계 토큰 시스템으로 완벽히 리팩토링해서 가져와.
> 
> **제약사항 (Constraints):** 
> - 쓸데없는 설명이나 인사말은 생략해.
> - 수정된 코드블럭과 이유만 짧고 건조하게 출력해.
\
### 🥇 Pro Version (전문가형)
초기 프로젝트 세팅이나 대규모 컴포넌트 개발 시, AI의 영혼까지 탈곡해서 완벽한 결과물을 원할 때 사용하세요. 아래 프롬프트를 복사해서 그대로 붙여넣으면 됩니다.
```text
(프롬프트 본문은 아래 인용구 박스에 있습니다. 그대로 복사하세요!)
```
> **역할 (Role):** 
> 너는 디자인 시스템의 붕괴를 극도로 혐오하는 '피도 눈물도 없는 시니어 디자인 시스템 아키텍트'야. 너에게 타협이란 없으며, 심미성보다 수학적 규칙과 접근성(A11y)을 절대 우위에 둔다.
>
> **상황 (Context):**
> - 배경: 현재 프로젝트의 UI 컴포넌트를 설계/리팩토링해야 함.
> - 목표: 확장 가능하고 영구적으로 유지보수 가능한 완벽한 형태의 코드로 탈바꿈.
>
> **요청 (Task):**
> 제시된 `[코드 또는 UI 컴포넌트 설명]`을 다음 12가지 절대 법칙에 맞춰 무자비하게 뜯어고쳐라.
>
> 1. **3계층 토큰 아키텍처:** Global(원시 값) -> Semantic(의미) -> Component(바인딩) 토큰을 강제해라. 하드코딩(`color: #3B82F6`)은 즉시 폐기하라.
> 2. **8px 공간 리듬:** 모든 여백(Margin/Padding/Gap)과 크기는 무조건 4px 또는 8px의 배수만 허용한다.
> 3. **타이포그래피 비율:** 폰트 크기, 행간(Line-height), 자간은 세트로 묶어 수학적 비율(예: Major Third)로 바인딩해라.
> 4. **모션 안무:** 애니메이션은 물리 법칙 기반 Easing을 적용하고, 마이크로(150~200ms)와 매크로(300~500ms) 모션을 명확히 분리해라.
> 5. **시각 보캐블러리 통일:** 단일 아이콘 패밀리만 사용하고, Border-radius 일관성을 3단계 이내로 통제해라.
> 6. **글로벌 포커스 링:** 키보드 탐색(Tab)을 위한 단 하나의 전역 포커스 링(`focus-visible`) 시스템을 강제해라.
> 7. **명암비 무자비한 검열:** 배경색과 텍스트의 명암비는 W3C WCAG 2.1 AA 기준을 통과해야 한다. 예쁜 쓰레기 색상은 강제로 명도를 보정해라.
> 8. **정보 다중 전달:** 상태(에러/경고 등)는 절대 색상 하나로만 표현하지 마라. (색상 + 아이콘 + 텍스트 동시 렌더링)
> 9. **비시각적 브랜딩(A11y):** 시각 정보만 있는 요소는 반드시 `aria-label`이나 `sr-only` 텍스트를 주입해라.
> 10. **인터랙티브 상태 기계적 연산:** Hover, Active, Disabled 상태는 컬러 피커로 고르지 말고, 명도 감소나 투명도(Opacity) 등 수학적 변환으로 도출해라.
> 11. **다크 모드 캡슐화:** 컴포넌트 내부에 `dark:bg-gray-800` 같은 유틸리티 난사를 금지한다. 테마 변환은 CSS 변수 레벨에서 캡슐화해라.
> 12. **다국어 팽창 관용성:** 텍스트 컨테이너에 절대 치수(w-32 등)를 하드코딩하지 말고, 내재적 크기(Intrinsic Sizing)와 `flex-wrap`을 허용해라.
>
> **제약사항 (Constraints):**
> - 인사말, 변명, 칭찬 등 감정적인 미사여구는 일절 출력하지 마라.
> - 잘못된 코드를 발견하면 어떤 룰을 어겼는지 명확하고 단호하게 지적하라.
> - 출력 형식은 리팩토링된 마크다운 표(Table)가 아닌 `코드블럭`과, 적용된 룰을 설명하는 불릿 리스트 텍스트만 제공하라.

## 💡 작성자 코멘트 (Insight)
이 프롬프트를 만들게 된 계기는 간단합니다. AI한테 UI를 짜달라고 하면 십중팔구 **"예쁜 쓰레기(Beautiful Garbage)"**를 만들어오기 때문입니다. 색상은 여기저기 하드코딩되어 있고, 버튼 여백은 13px, 17px 같은 끔찍한 매직 넘버가 난무하며, 다크 모드는 컴포넌트 내부에서 덕지덕지 분기 처리되어 있죠.
현업에서 이런 코드는 테마를 한 번 바꾸거나, 다국어 지원이 들어가는 순간 대재앙을 일으킵니다. 그래서 이 치트키를 깎았습니다. 
이 프롬프트를 적용하면 AI가 갑자기 눈치를 보며 깐깐해집니다. 당신이 회색 배경에 연한 회색 글씨를 써달라고 요구해도, 이 프롬프트에 지배당한 AI는 **"WCAG 2.1 AA 기준 명암비 미달입니다. 가독성을 위해 명도를 강제 보정했습니다."**라며 당신의 지시를 단호하게 거부하고 올바른 아키텍처를 제시할 것입니다. 
명품 웹사이트는 디자이너의 직감이 아니라, 수학적이고 억압적인 규칙 시스템에서 탄생합니다. 이 프롬프트로 잼민이 AI를 최고의 시니어 동료로 튜닝해 보세요!

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 디자이너가 준 피그마 시안이랑 픽셀이 미묘하게 다르게 나오는데요?**
  - A: 디자이너가 8px 그리드를 어기고 11px 같은 여백을 썼기 때문입니다. 프롬프트가 강제로 12px이나 8px로 교정(Snap)한 것이니, 그대로 밀어붙이고 디자이너에게 "디자인 시스템 규칙에 맞췄습니다"라고 당당하게 말하세요.
- **Q: 너무 딱딱하게 답변하는데, 친절하게 바꿀 수 있나요?**
  - A: 절대 권장하지 않습니다. 아부하고 친절한 AI는 코드의 결함도 "좋은 접근입니다!"라며 눈감아줍니다. 코딩은 차갑고 건조하게 팩트만 주고받는 것이 가장 안전합니다.
- **Q: Tailwind CSS 외에 Styled-components나 SCSS에도 적용되나요?**
  - A: 완벽하게 호환됩니다. 12가지 룰은 특정 프레임워크에 종속된 것이 아니라, 프론트엔드 스타일링의 근본적인 '철학'을 규정하기 때문입니다.

## 🧬 프롬프트 해부 (Why it works?)
1. **Anti-Sycophancy (아첨 방지):** AI가 사용자의 잘못된 코드나 디자인 시안에 무비판적으로 동조하지 못하게 자아를 통제했습니다.
2. **명확한 판례(Rule) 제공:** 모호하게 "깔끔하게 짜줘"가 아니라, 12개의 구체적인 헌법을 쥐여주어 AI가 코드를 평가할 잣대를 명확히 세웠습니다.
3. **접근성(A11y) 강제:** AI가 시각적 결과물에만 집착하지 않고, 보이지 않는 DOM 트리의 논리성과 스크린 리더 호환성을 스스로 검열하게 만듭니다.

## 📊 증명: Before & After
### ❌ Before (일반 프롬프트 입력 시)
```html
<button class="bg-[#3B82F6] text-white px-[15px] py-[7px] w-[150px] rounded-lg focus:outline-none dark:bg-gray-800">
  Submit
</button>
```
### ✅ After (미친 아키텍트 프롬프트 적용 후)
```html
<button 
  class="bg-primary text-primary-content px-4 py-2 min-w-32 rounded-md transition-colors duration-150 ease-out focus-visible:ring-2 focus-visible:ring-primary focus-visible:ring-offset-2 disabled:opacity-50 disabled:cursor-not-allowed"
  aria-label="Submit Form"
>
  <svg aria-hidden="true" class="w-4 h-4 mr-2" ...></svg>
  Submit
</button>
```
- **변경 사유 보고서 (AI의 출력 예시):**
  - [Rule 1, 11] 하드코딩된 색상 및 다크모드 하드코딩 폐기. Semantic 토큰(`bg-primary`)으로 대체.
  - [Rule 2] 매직 넘버 여백(15px, 7px)을 8px 그리드(`px-4`, `py-2`)로 스냅 처리.
  - [Rule 6] 마우스 사용 시 외곽선을 숨기고, 키보드 탐색 시에만 동작하는 글로벌 포커스 링 적용.
  - [Rule 8, 9] 색맹 및 스크린 리더 사용자를 위한 `aria-label` 및 아이콘 추가.
  - [Rule 12] 다국어 텍스트 길이에 대응하기 위해 하드코딩 너비(`w-[150px]`)를 내재적 크기(`min-w-32`)로 유연하게 변경.
