## 🚀 Решение: Протокол "Архитектор" (The 'Architect' Protocol)

### 🥉 Базовая версия (Basic Version)

Идеально подходит для быстрого сканирования кода на уязвимости и узкие места в производительности.

> **Роль:** Ты `[Senior Backend-разработчик с 20-летним опытом]`.
> **Задача:** Проанализируй представленный ниже `[legacy-код]` и выдели ровно 3 проблемы, связанные с безопасностью и производительностью.

### 🥇 Профессиональная версия (Pro Version)

Применяйте, когда косметических правок недостаточно. Позволяет провести глубокий архитектурный рефакторинг и извлечь крайние случаи (Edge Cases) для надежного тестирования.

> **Роль (Role):** Ты ведущий архитектор программного обеспечения с 20-летним опытом и амбассадор Clean Code. Ты эксперт, для которого поддерживаемость, масштабируемость и производительность важнее простой реализации функционала.
>
> **Контекст (Context):**
>
> - Фон: Необходимо перенести запутанный спагетти-образный `[legacy-код на Python]` в архитектуру, удобную для поддержки.
> - Цель: Преобразовать код так, чтобы он строго соответствовал принципу единственной ответственности (SRP) и имел низкую связанность (low coupling).
>
> **Задача (Task):**
> 
> Проанализируй предоставленный пользователем legacy-код и выполни рефакторинг, строго следуя этим 4 шагам:
>
> 1. **Аудит кода (Code Audit):** Выяви как минимум 3 проблемы текущего кода (антипаттерны, уязвимости безопасности, узкие места производительности).
> 2. **Стратегия глубокого размышления (Deep Thinking Strategy):** Разработай конкретную стратегию рефакторинга (например, какие паттерны проектирования применить и почему они подходят).
> 3. **Рефакторинг (Refactoring):** Напиши улучшенный код, снабдив ключевую логику комментариями с обоснованием изменений.
> 4. **Генерация тестов (Test Case Generation):** Напиши 3 тест-кейса для пограничных ситуаций (Edge Cases), чтобы доказать безопасность кода после рефакторинга.
>
> **Ограничения (Constraints):**
>
> - Язык и версия: `[Python 3.12+]` (Обязательное использование Type Hinting).
> - Руководство по стилю: Строгое соблюдение PEP 8.
> - Внешние библиотеки: Использовать только стандартную библиотеку (минимизация внешних зависимостей).
> - Формат вывода: Оформи ответ в чистом Markdown так, чтобы логику понял даже Junior-разработчик.
>
> **Предупреждения (Warning):**
>
> - Если результаты анализа неясны или неоднозначны, не придумывай код искусственно, а четко напиши "Невозможно определить". (Предотвращение галлюцинаций).
>
> **Входные данные (Input Data):**
> `[Вставьте сюда код для рефакторинга]`

Когда своими глазами видишь, как нейросеть самостоятельно выносит логику отправки писем в отдельный сервис и заботится об откатах транзакций, понимаешь: бессонных ночей за дебагом станет как минимум вдвое меньше.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Будет ли этот промпт работать в GPT-5?**
  - A: Да, он отлично справляется и там. Однако в задачах сложного рефакторинга, где требуется анализировать массивную кодовую базу и выстраивать связи между сотнями файлов, режим Deep Think у Gemini 3 Pro находит неочевидные крайние случаи (edge cases) на порядок лучше. При этом, если нужно просто накидать утилитарную функцию, GPT-5 сделает это быстрее.

- **Q: Как адаптировать этот подход под условный Java или TypeScript?**
  - A: Элементарно. Просто измените пункты `Язык и версия` и `Руководство по стилю` в блоке ограничений (Constraints). Подставьте туда `[TypeScript 5.0+, соблюдение правил ESLint Airbnb]`, и промпт отработает безупречно.

- **Q: ИИ постоянно упирается в ответ «Невозможно определить». Что я делаю не так?**
  - A: Это верный признак того, что модели не хватает базового контекста. Добавьте во `[Входные данные]` схему базы данных или интерфейсы смежных классов, и ИИ выдаст вам детально проработанный архитектурный чертеж.

1. **Активация Chain-of-Thought:** Жесткое разделение на 4 шага (`Code Audit -> Strategy -> Refactoring -> Test`) заставляет ИИ мыслить системно и последовательно.

Код изящно декомпозирован с применением `Service Layer` и `Repository Pattern`, а принцип единственной ответственности (SRP) соблюден безукоризненно.

```python
# UserCreateService.py
from dataclasses import dataclass
from typing import Protocol

class UserRepository(Protocol):
    def save(self, user: 'User') -> bool: ...

class EmailSender(Protocol):
    def send_welcome_email(self, email: str) -> bool: ...

@dataclass

    email: str

    """

        self.email_sender = email_sender
```
