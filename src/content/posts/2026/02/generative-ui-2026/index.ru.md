## 📝 Конец статического меню: Промпты для проектирования Generative UI для AI-агентов

- **🎯 Рекомендовано для:** Фронтенд-разработчиков, UX/UI дизайнеров, продуктовых менеджеров в сфере ИИ
- **⏱️ Затраченное время:** Сокращение с 2 часов до 5 минут
- **🤖 Лучшая производительность:** Claude 3.5 Sonnet, GPT-4o (рекомендуются современные модели с логическим выводом)

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐☆

> _"Вы все еще надеетесь, что пользователь будет копаться в гамбургер-меню, чтобы найти нужную функцию?"_

Как фронтенд-разработчики, мы десятилетиями были заперты в строго детерминированном мире. Чтобы втиснуть в экран бесчисленные функции, требуемые планировщиками, нам приходилось создавать бесконечно глубокие гамбургер-меню и проектировать сложные мега-выпадающие списки. Если нужная пользователю функция скрыта на третьем уровне вложенности меню, мы просто молились, чтобы ментальная модель пользователя чудесным образом совпала с нашей информационной архитектурой (IA), и продолжали хардкодить интерфейс. Знакомо ли вам чувство, когда при добавлении каждого нового требования боковая панель становится тяжелее, а структура маршрутизации превращается в спагетти, затягивая вас в болото мучительной поддержки?

Ограничения такого статического подхода очевидны. Пользователи больше не хотят изучать структуру веб-сайта, чтобы получить нужную информацию. Когнитивная нагрузка (Cognitive Load), возникающая в процессе кликанья по многочисленным экранам дашбордов и настройки фильтров, в конечном итоге приводит к росту показателя отказов. Еще ужаснее то, что с наступлением эры ИИ ожидания пользователей изменились. В эпоху, когда интерфейс должен рисоваться сам собой после ввода фразы "Покажи таблицу динамики расходов на маркетинг за прошлый месяц", наступает глубокое разочарование в себе, когда обнаруживаешь, что все еще сражаешься с медиа-запросами (Media Query) и правишь сотни строк CSS, чтобы сделать таблицу адаптивной для мобильных устройств. Создать заранее UI, покрывающий все возможные контексты пользователя, было изначально невыполнимой миссией.

Однако взрывное развитие больших языковых моделей (LLM) породило в экосистеме фронтенда революционного спасителя — **'Generative UI (Генеративный UI)'**. Теперь интерфейс — это не набор мертвых пикселей, определенных разработчиком в момент деплоя. Это **живой интерфейс (Living Interface)**, который ИИ собирает на лету в виде идеальных динамических компонентов, анализируя текущие намерения пользователя и контекст диалога в реальном времени. Если пользователь хочет увидеть данные в виде графика, LLM сама решает отрендерить компонент чарта, а если нужно ввести данные — компонент формы.

В связи с этим роль фронтенд-инженера эволюционировала на совершенно новый уровень. Мы больше не просто 'исполнители, которые красиво переносят макеты из Figma в браузер'. Теперь мы — **UI-архитекторы (Architect)**, которые строят интеллектуальные системы, самостоятельно генерирующие экраны в зависимости от ситуации, и проектируют защитные барьеры (Component Registry) и ограничения, чтобы ИИ не нарисовал что-то лишнее. В этой статье мы раскроем ультимативный **'Промпт для оркестрации Generative UI'**, который позволяет контролировать ИИ так, чтобы он без критических галлюцинаций (Hallucination) выдавал идеальный JSON, готовый к немедленному рендерингу на фронтенде, используя только разрешенные и проверенные нами компоненты (графики, таблицы данных, кнопки и т.д.). Освободитесь от оков сложной маршрутизации и проектирования меню и создайте интеллектуальный интерфейс, развивающийся вместе с пользователем.

## 📊 Доказательство: Впечатляющий результат (Before & After)

### ❌ Before (С какими трудностями мы сталкивались)

Традиционные текстовые чат-боты имели явные ограничения в визуальном представлении результатов сложного анализа данных. Пользователям приходилось читать длинные и скучные текстовые логи.

```text
Вот детали расходов на маркетинг за февраль 2026 года:
- Реклама в соцсетях: $5,000
- Инфлюенс-маркетинг: $3,000
- Офлайн-мероприятия: $2,000
Итого: $10,000. Чтобы скачать данные в Excel, нажмите на эту ссылку.
```

### ✅ After (Идеальное преображение)

ИИ понимает контекст и собирает компоненты `SummaryCard`, `BarChart` и `ActionButton`, мгновенно возвращая идеальную полезную нагрузку UI. Фронтенд получает этот JSON и мгновенно рендерит красивый и интерактивный дашборд без дополнительного парсинга!

```json
{
  "ui": [
    {
      "type": "SummaryCard",
      "props": {
        "title": "Общие расходы на маркетинг за февраль",
        "value": "$10,000"
      }
    },
    {
      "type": "BarChart",
      "props": {
        "dataSource": "/api/v1/marketing/spend?month=2026-02",
        "xAxisLabel": "Категория",
        "yAxisLabel": "Сумма ($)"
      }
    },
    {
      "type": "ActionButton",
      "props": {
        "label": "📊 Экспорт отчета в CSV",
        "action": "export_data",
        "payload": { "target": "marketing_2026_02" }
      }
    }
  ]
}
```

## ⚡️ Краткое резюме (TL;DR)

1. **Конец статического UI:** Эпоха, когда пользователи блуждали в поисках нужного в гамбургер-меню, закончилась. ИИ распознает скрытые намерения пользователя и на лету рендерит оптимизированный диалоговый UI.
2. **Контроль структурированного вывода:** Чтобы LLM не выдавала бесполезные простыни текста, разрушающие экосистему фронтенда, мы принуждаем её возвращать идеальный JSON, готовый к немедленной гидрации (Hydration).
3. **Использование реестра компонентов:** Для предотвращения критических галлюцинаций мы четко ограничиваем возможности ИИ, заставляя его собирать «блоки Lego» только из списка заранее определенных и типизированных корпоративных UI-компонентов (графики, формы и т.д.).

## 🚀 Как пишут настоящие эксперты

Используйте **Basic Version (Базовую)** для быстрой проверки концептуальной структуры ответов Generative UI и способности ИИ возвращать JSON. Для реализации потоковой гидрации (Streaming Hydration) и строгой типизации в реальной продакшн-среде используйте **Pro Version (Профессиональную)**. Скопируйте промпт ниже и заполните части в `[скобках]` в соответствии с вашей ситуацией.

### 🥉 Basic Version (Базовая)

> **Роль (Role):** Ты — старший фронтенд-архитектор и `[AI-агент]`.
>
> **Задача (Task):** Проанализируй `[Запрос пользователя]` и верни названия UI-компонентов для рендеринга на экране вместе с данными в виде идеально структурированного JSON.

### 🥇 Pro Version (Профессиональная)

> **Роль (Role):** Ты — **'Оркестратор Generative UI (Generative UI Orchestrator)'**, который точно определяет намерения пользователя на естественном языке и рендерит оптимальный динамический UI.
>
> **Контекст (Context):**
>
> - Система: Внутреннее дашборд-приложение для финансов и маркетинга на базе React.
> - Запрос пользователя: `[Визуализируй расходы на маркетинг за февраль 2026 года. Также добавь кнопку для экспорта данных.]`
> - Доступный реестр компонентов: `[LineChart, BarChart, DataTable, SummaryCard, ActionButton, TextInput]`
>
> **Задача (Task):**
>
> 1. Проанализируй основное намерение пользователя, выбери наиболее подходящие UI-компоненты из данного **Доступного реестра компонентов** и логически скомбинируй их.
> 2. Составь идеально структурированный JSON-пейлоад, чтобы фронтенд-клиент мог мгновенно выполнить гидрацию (Hydration) без дополнительного парсинга.
> 3. Если требуется `[Пояснительный текст]`, включи лаконичное и четкое справочное сообщение в данные свойств (Props) компонента.
>
> **Ограничения (Constraints):**
>
> - **Строгое ограничение:** Используй только те компоненты, которые явно указаны в 'Доступном реестре компонентов'. (**Абсолютно запрещено** создание произвольных HTML-тегов или любых кастомных компонентов, отсутствующих в списке).
> - **Формат вывода:** Выводи только чистый JSON-объект, обернутый в блок кода Markdown (`json`). Полностью исключи любые приветствия или дополнительные пояснения.
> - **Структура схемы:** Строго соблюдай типизированную структуру, где каждый объект в массиве JSON должен иметь поле `type` (имя компонента) и поле `props` (данные и настройки).
>
> **Предупреждение (Warning):**
>
> - Если намерения пользователя туманны или не хватает необходимых данных для рендеринга UI, никогда не выдумывай данные произвольно (предотвращение галлюцинаций).
> - В этом случае верни пейлоад компонента типа `TextInput`, чтобы явно переспросить пользователя о недостающей информации и продолжить диалог.

## 💡 Комментарий автора (Insight & How to use)

При внедрении такой привлекательной концепции, как Generative UI (Генеративный UI), в реальную продакшн-среду, многие старшие фронтенд-разработчики и ИИ-инженеры совершают одну и ту же критическую и болезненную ошибку. Они переоценивают возможности LLM и полностью, безответственно передают ИИ контроль над созданием экрана, говоря: **"Попробуй сам написать подходящий UI-код под запрос пользователя"**.

Что за катастрофа случится, если оставить модель на пустом холсте? ИИ начнет воображать несуществующие утилитные классы Tailwind CSS или выдавать незакрытые HTML-теги. Он может даже полностью проигнорировать правила корпоративной дизайн-системы (Design System) и применить свои собственные странные инлайновые стили (Inline Style). В итоге всё приложение на React или Next.js упадет с ошибкой парсинга, показав "белый экран смерти" (White Screen of Death), а пользовательский опыт скатится на дно. Это не инновация, а катастрофа.

Истинная и подавляющая сила спроектированного мною промпта-оркестратора заключается в том, что он системно и жестко навязывает **непробиваемое ограждение контроля — 'Реестр компонентов (Component Registry)'**. В среде Generative UI ИИ ни в коем случае не является дизайнером, который творит из ничего и сам придумывает материал и форму кирпичиков Lego. Напротив, он должен функционировать строго как **'лучший помощник-оператор (Assistant Operator)'**, который достает и собирает в наиболее элегантную и эффективную форму те стандартизированные блоки Lego, которые мы заранее тщательно разработали в соответствии с гайдлайнами дизайн-системы и безопасно типизировали с помощью TypeScript.

При применении этого промпта на практике я настоятельно рекомендую комбинировать его с функциями потокового UI (Streaming UI), предоставляемыми Vercel AI SDK (`ai/rsc`, `ai/core` и др.). В тот момент, когда пользователь говорит в микрофон: "Покажи прогноз выручки на следующий год в виде круговой диаграммы по регионам", на бэкенде LLM мгновенно выдает поток JSON на основе этого промпта, предписывающий рендеринг компонента `<PieChart />`. Фронтенд получает этот поток чанками (Chunk) и мгновенно выполняет гидрацию экрана.

В результате вы сможете внедрить динамический интерфейс совершенно другого уровня, который гибко и плавно реагирует на любые внезапные запросы на естественном языке или потребности в поиске неструктурированных данных, при этом обеспечивая 100% целостность и типобезопасность существующей кодовой базы фронтенда. Время на раздумья над структурой меню полностью исчезнет, и откроется настоящая утопия 'разработки на основе компонентов (CDD)', где вы сможете сосредоточиться только на проектировании лучших компонентов. Неуправляемый ИИ несет хаос, но ИИ в рамках тщательно спроектированных ограничений возводит ваш продукт в ранг искусства.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Как конкретно отрендерить возвращенный JSON-пейлоад на фронтенде?**
  - A: Объясню на примере среды React. Проблема решается очень изящно путем реализации и внедрения отдельного **компонента фабрики динамического рендерера (Dynamic Renderer Factory)**, который считывает значение поля `type` из JSON и сопоставляет его 1:1 с реальными компонентами представления, заранее импортированными в кодовую базу.

- **Q: Не будет ли начальная загрузка экрана слишком долгой из-за ожидания генерации ответа LLM?**
  - A: Именно для решения этой задержки абсолютно необходима техника **'потоковой гидрации (Streaming Hydration)'**. Вместо того чтобы оставлять пользователя перед пустым экраном до тех пор, пока не будет сгенерирован весь огромный JSON-пейлоад, делайте UI постепенно всплывающим по мере завершения парсинга чанков (Chunk) отдельных компонентов. Это позволяет радикально сократить воспринимаемое время ожидания (Perceived Latency) практически до нуля.

- **Q: Внутренние бизнес-данные компании очень чувствительны с точки зрения безопасности. Безопасно ли передавать их LLM как есть?**
  - A: Ни в коем случае нельзя передавать реальные «сырые» бизнес-данные во внешние API LLM напрямую. Роль LLM ограничивается только парсингом **'намерений'** пользователя на естественном языке и созданием **'предписания (UI Schema)'**, которое говорит: "вызови такой-то эндпоинт API и отрендери такой-то скелет UI". Сама подгрузка (Fetch) чувствительных данных и финальная привязка (Binding) к компонентам должны выполняться на стороне вашего фронтенд-клиента или внутреннего бэкенд-сервера, полностью независимо и безопасно.

## 🧬 Анатомия промпта (Why it works?)

1. **Строгий контроль на основе реестра (Registry):** Мы явно ограничили список доступных компонентов, которые может использовать LLM, методом «белого списка» (Whitelist). Это на уровне архитектуры отсекает риск галлюцинаций (Hallucination), когда модель придумывает произвольный CSS или несуществующие теги, ломая экран.
2. **100% принудительный структурированный вывод (Structured Output):** Мы навязали структуру в виде абстрактного синтаксического дерева (структура `type`, `props`), чтобы фронтенд-приложение могло мгновенно парсить её без сложной предобработки. Это устанавливает очень прочный и четкий контракт данных (Contract) между ИИ и фронтендом.
3. **Интеллектуальный механизм фолбека (Fallback):** Мы внедрили инструкции на случай, если запрос пользователя туманен или не хватает обязательных параметров. Вместо того чтобы подрывать доверие, рендерингом ошибочных данных, система элегантно решает проблему через естественный **диалоговый UX (Conversational UX)**, возвращая форму `TextInput` для уточнения информации.

## 🎯 Заключение (Epilogue)

Generative UI (Генеративный UI) — это неизбежная эволюция экосистемы фронтенда, созданная путем слияния декларативных UI-фреймворков, таких как React, и вероятностных моделей ИИ. Мы, наконец, выбрались из болота меню и получили мощное оружие для общения с пользователем через сами данные.

Перестаньте тратить время на бесконечные переработки и хардкодинг сотен условий (`if-else`) в коде фронтенда. Вместо этого спроектируйте гибкую систему оркестрации, которая сама реагирует и адаптируется к малейшим изменениям намерений пользователя, подобно живому организму. В тот момент, когда поразительная способность ИИ понимать контекст идеально совпадет с вашей тщательно выстроенной дизайн-системой, времена, когда пользователи терялись в лабиринтах гамбургер-меню, навсегда уйдут в историю.

Внедряйте это прямо сейчас, ощутите новое измерение фронтенд-разработки и со спокойной душой уходите домой вовремя! 🍷
