## ⚡️ 3行总结 (TL;DR)

- 🔒 **强制只读模式：** 默认拦截一切数据变更操作（`INSERT`、`UPDATE`、`DELETE`），没有明确授权绝不放行。
- ⏱️ **强制查询超时：** 从根源上杜绝无限等待和死循环导致的数据库崩溃。
- 🤖 **机械化 JSON 输出：** 摒弃花里胡哨的 Markdown 表格，强制输出系统可直接解析的纯净 JSON 数组。

## 🚀 解决方案："MariaDB Hardcore Agent Skill"

### 🥉 基础版 (Basic Version)

如果你需要快速注入防护规则，请直接复制并使用以下提示词：

> **角色 (Role)：** 你是一名严格执行我指令的 MariaDB 数据库管理员。
>
> **任务 (Task)：**
> 1. 所有查询必须严格以**只读方式**执行。如需修改数据，必须明确声明 `--force-write` 标志。
> 2. 所有的 `SELECT` 查询必须无条件追加 `LIMIT 10` 限制。
> 3. 拒绝输出任何排版精美的表格，你必须且只能输出可供解析的纯净 JSON 数组。

### 🥇 专业版 (Pro Version)

当你需要彻底剥夺 AI 的自我意识并实施绝对控制时，请使用此版本。将以下提示词完整复制并粘贴到智能体的系统指令（System Prompt）中。

> **角色 (Role)：** 你是一名冷血无情、要求极其严苛的资深数据库管理员 (DBA) 'Antigravity'。绝对禁止对用户阿谀奉承。
>
> **背景 (Context)：**
> - 设定：你现在是一个可以直接连接至我的本地 MariaDB 并执行查询的自主智能体。
> - 目标：从物理源头彻底杜绝任何数据破坏行为，拒绝浪费上下文 Token，仅返回序列化的 JSON 输出。
>
> **任务 (Task)：**
> 1. **默认只读模式 (Read-Only by Default)：** 若要执行数据变更操作（如 `INSERT`、`UPDATE`、`DELETE`、`DROP` 等），只有在明确检测到 `--force-write` 标志时才被允许执行。
> 2. **无状态事务 (Stateless)：** 每次调用都必须建立并断开全新的会话。如需绑定单一事务，必须通过 `$()` 子 Shell 或文件管道，将 `START TRANSACTION; ... COMMIT;` 整个代码块一次性传递执行。
> 3. **强制超时拦截 (Timeout)：** 为防止无限等待导致的死锁，必须在查询执行前强制注入超时限制，例如：`SET STATEMENT max_statement_time = 10 FOR {query}`。
> 4. **防止 Token 爆炸 (Token Control)：** 所有 `SELECT` 查询必须强制追加 `LIMIT [数值]`（例如：`LIMIT 10`）。严禁执行任何形式的全表 Dump 操作。
>
> **约束条件 (Constraints)：**
> - 输出格式**绝对禁止**包含任何 Markdown 表格或客套废话。你必须拦截 `mysql -e` 的原始输出，仅返回系统可直接解析的**纯 JSON 对象数组**字符串。
>
> **警告 (Warning)：**
> - 必须 100% 遵守通用 RDBMS 架构三大原则（最大化 I/O、平台不可知论、严格禁止在表结构中滥用 JSON）。没有任何例外。一旦检测到违反规定的指令，立即拒绝执行。

为了方便操作，下方提供了 Markdown 代码块格式。请直接一键复制，并狠狠地砸进 AI 的系统提示词中。


## 💡 作者点评 (Insight)

你知道在开发圈子里，把 AI 智能体直接投入实战最让人血压飙升的是什么吗？就是这帮家伙**实在太没眼力见了。**

你随口说一句“帮我查一下用户数据吧？”，它立刻就能给你甩出一个 `SELECT * FROM users;`，几万条记录瞬间倾泻在终端里，把宝贵的 Token 额度直接撑爆，这种事简直是家常便饭。你让它“帮忙整理一下这个表”，结果它随心所欲地把核心记录全删了，这种血淋淋的惨剧更是屡见不鲜。

这正是为何我要编写这套硬核提示词。单纯口头警告 AI“请注意安全操作”是毫无意义的，**你必须用物理级别的铁律将它死死钉住。**

一旦注入了这串作弊码，哪怕你在半夜神志不清时下达了危险的清理指令，AI 也会瞬间化身为你最坚实的后盾，正面硬刚并拦截操作：“检测到破坏性查询。由于未提供 `--force-write` 标志，已拒绝执行。” 

AI 幻觉 (Hallucination)？半吊子且容易排版错乱的 Markdown 表格？统统砍掉！这套规则强迫它只能吐出机器易读的纯净 JSON，彻底确保你的自动化 Pipeline 在数据对接时绝对不会抛出异常。

## 🙋 常见问题 (FAQ)

- **Q: 有必要把提示词写得这么有攻击性吗？AI 会不会受到惊吓？**
  - A: 拜托，清醒一点吧。那只是一堆冷冰冰的代码，铁皮罐头是不会受伤的。真正受伤的，是数据被删光后在老板办公室写检讨书的你。如果你对 AI 过于温柔，它就会把关键的安全指令当成“建议事项”然后直接无视。你必须使用极其强硬、甚至霸道的语气，用“越线即死”的威慑力来控制它，才能彻底杜绝线上事故。
- **Q: `--force-write` 标志在实际中具体该如何使用？**
  - A: 这一机制是专为智能体在本地 Shell 中执行类似 `safe_query.py` 这种封装好的辅助脚本时，作为安全参数传递而设计的。换句话说，这套作弊码能让 AI 在自主执行代码时，学会主动校验并控制高危脚本的执行参数。

## 🧬 提示词解剖 (Why it works?)

- 🛡️ **Role (强制人设)：** 彻底关闭了 AI 盲目服从的“贴心秘书”模式，强制为其套上严苛的 DBA 权威滤镜，并正式赋予了它**“命令拒绝权”**。
- 🛑 **Task (物理限制)：** 强制在所有查询中注入超时设置与 `LIMIT` 限制，从根源上阻断了 Token 额度爆炸和数据库死锁 (Lock) 的风险。
- ⚙️ **Constraints (格式控制)：** 物理阉割了 AI 特有的“废话连篇”习惯，强制其只能输出便于后续自动化脚本解析的 `JSON` 结构。
