# AURA-CORE 문서 관리 프로토콜 (Documentation Protocol)

> **목표:** 문서의 파편화(Fragmentation)를 방지하고, 단일 진실 공급원(SSOT)을 유지한다.

## 1. 파일 보존 원칙 (File Conservation)
**"새로운 파일을 만들지 마십시오."**

프로젝트에는 오직 다음의 핵심 문서만 존재해야 합니다. 새로운 아이디어나 기획은 아래 파일 중 적절한 곳에 **통합(Merge)**해야 합니다.

1.  **`docs/ALGORITHM_OVERVIEW.md`**: (기획/로직) 서비스 흐름, 질문 리스트, 비즈니스 룰. **(가장 중요)**
2.  **`docs/DATA_MODEL.md`**: (데이터) DB 스키마, 타입 정의.
3.  **`docs/API_REFERENCE.md`**: (기술) API 명세, 인터페이스.
4.  **`docs/DOCS_PROTOCOL.md`**: (규칙) 현재 이 파일.

*   *금지 사항:* `IDEA.md`, `DRAFT_v2.md`, `MEETING_LOG.md` 등 임시 파일 생성 금지.

## 2. 트리 문법 표준 (Tree Syntax Standard)
문서의 가독성과 파싱(Parsing) 편의를 위해, 모든 로직 기술은 아래의 **단일 트리 구조**를 따릅니다.

### 2.1. 기본 기호
*   **`+` (Plus)**: 질문, 화면, 사용자에게 보여지는 요소. (반드시 굵게 `**` 표시)
*   **`-` (Minus)**: 답변, 선택지, 사용자의 행동.
*   **`*` (Asterisk)**: 부가 설명, 내부 로직, 참고 사항.

### 2.2. 작성 예시
```markdown
+ **"사용자에게 보여질 질문이나 화면 제목?"**
  - 답변 옵션 A
  - 답변 옵션 B
    + **"옵션 B 선택 시 나타나는 하위 질문?"**
      - 하위 답변 1
      - 하위 답변 2 (#F5E0D8)
  - 답변 옵션 C
    * (Note: 이 옵션은 특정 조건에서만 노출됨)
```

## 3. 유지보수 원칙 (Maintenance)
1.  **Docs as Code:** 문서는 코드와 같습니다. 기능이 변경되면 문서도 즉시 수정되어야 합니다.
2.  **No Dead Links:** 문서 내에서 더 이상 사용되지 않는 로직(Legacy)은 즉시 삭제합니다. (주석 처리 하지 말고 삭제)
3.  **Review Criteria:** PR 요청 시 문서 업데이트가 포함되지 않은 기능 변경은 거절(Reject)합니다.
