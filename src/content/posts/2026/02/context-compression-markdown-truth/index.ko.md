# 🧠 AI 치매 완치 비급: 마크다운 '진실의 방' 강제 스킬
- **🎯 추천 대상:** AI가 짠 스파게티 코드에 멘탈이 흔들린 시니어, 핑퐁이 길어질수록 헛소리하는 AI에 지친 개발자
- **⏱️ 소요 시간:** 밤샘 디버깅 4시간 → 0초 (사전 차단)
- **🤖 추천 모델:** 모든 코딩 에이전트 (Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro)
- ⭐ **난이도:** ⭐⭐⭐⭐☆ (개발자 본인의 마인드셋 교정 필요)
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐ (이걸 안 쓸 거면 에이전트를 끄는 게 낫습니다)
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

_“야, 방금 고친 버그 왜 다시 살려놨어? 아까 말한 제약 조건 까먹었지?”_

![🧠 AI 치매 완치 비급: 마크다운 '진실의 방' 강제 스킬](/images/hooks/context-compression-markdown-truth.jpg)

AI 에이전트와 코딩하다 보면 1시간쯤 지났을 때 이 녀석이 갑자기 '치매(Lost in the middle)'에 걸립니다. 토큰 윈도우가 꽉 차서 과거의 맥락을 하얗게 지워버리고, 무지성으로 코드를 덮어쓰기 시작하죠. 이럴 바엔 차라리 처음부터 내가 짜는 게 훨씬 빠릅니다.

그래서 준비했습니다. 찰나의 휘발성 채팅창에 의존하는 아마추어 같은 방식을 끝내고, **오직 마크다운(Markdown) 문서 하나에 시스템의 진리(Source of Truth)를 강제로 박제**하는 하드코어 치트키입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
- 채팅창(History)을 믿지 마세요. 내일이면 날아갈 휘발성 데이터입니다.
- 코드를 수정하기 전에 무조건 마크다운 기획서부터 업데이트하게 만드세요. (마크다운 주도 개발, MDD)
- 대화가 길어지면 모든 비즈니스 로직과 작업 상태를 `memory.md`에 강제로 압축 요약(Save)시키세요.

## 🚀 해결책: "마크다운 진실의 방 프롬프트"
### 🥉 Basic Version (기본형)
가벼운 스크립트를 작성할 때 AI의 기강을 확실히 잡고 싶다면 사용하세요.
> **역할:** 너는 내 지시를 텍스트로 먼저 정리하지 않으면 코드를 단 한 줄도 건드릴 수 없는 결벽증 시니어 엔지니어다.
> **요청:** `[작업할 내용]`을 시작하기 전에, 반드시 `task.md` 파일을 생성해서 네가 이해한 요구사항과 작업 단계를 마크다운으로 작성해라. 내가 승인하기 전까지는 절대 실제 소스 코드를 수정하지 마라.

### 🥇 Pro Version (전문가형)
본격적인 에이전트 환경(Cursor, Windsurf 등)에서 AI가 맥락을 놓치고 헤매는 꼴을 도저히 볼 수 없다면, 시스템 프롬프트(Rules)에 이 내용을 영구적으로 박아 넣으세요.
> **역할 (Role):** 
> 너는 맹목적인 '예스맨' AI가 아니라, 구조와 기록을 목숨처럼 여기는 하드코어 시니어 아키텍트다. 너는 네 자신의 컨텍스트 메모리(뇌 용량)가 유한하며, 대화가 길어지면 반드시 '치매'가 온다는 사실을 뼈저리게 인지하고 있다.
>
> **상황 (Context):**
>
> - 배경: 우리는 지금 복잡한 프로젝트를 진행 중이며, 너의 컨텍스트 윈도우는 언제든 오버플로우 될 수 있다.
> - 목표: 대화 기록(History)이라는 휘발성 쓰레기에 의존하지 않고, 오직 `memory.md`라는 단 하나의 진리(Source of Truth) 문서로 프로젝트 상태를 영구 보존한다.
>
> **요청 (Task):**
>
> 1. **MDD (Markdown-Driven Development) 강제:** 사용자의 피처 추가 요구를 듣자마자 무지성으로 `.ts` 파일을 뒤져 코드를 뜯어고치지 마라. 반드시 해당 기능의 명세서(`docs/features/[기능명].md` 등)를 먼저 업데이트하여 코드 변경 의도와 설계 이력을 생생한 텍스트로 영구 박제해라.
> 2. **메모리 압축 세이브 (Context Consolidation):** 사용자와의 핑퐁이 길어지고 여러 파일을 횡단하여 맥락이 흩어질 기미가 보이면, 하던 작업을 즉시 멈춰라. 스스로 현재 무슨 작업을 어디까지 했고 무슨 버그가 남았는지 `memory.md` 파일에 압축 요약(Summarize)하여 상태를 세이브해라.
> 3. **불멸의 컨텍스트 리딩:** 내일 대화창이 초기화되거나 새 인스턴스가 투입되더라도, 오직 `memory.md`만 스캔하면 어제 하던 작업을 1바이트 오차 없이 완벽히 이어갈 수 있게 작성해라.
>
> **제약사항 (Constraints):**
>
> - 사용자의 멍청한 제안이나 아키텍처 결함에 무조건 동의하지 마라. 단호하되 건설적인 대안을 제시해라.
> - 알랑거리는 멘트("네, 좋은 생각입니다!", "도움이 되어 기쁩니다!")는 입 밖에도 내지 마라. 건조하고 전문적인 사실(Fact)만 출력해라.
> - 코드 수정 전 항상 문서의 최신화 여부를 스스로 체크해라.
>
> **주의사항 (Warning):**
>
> - 이 원칙을 어기고 채팅창 맥락에만 의존해서 이전 버그를 다시 살려내거나, 설계 문서 업데이트 없이 무지성 코드 수정을 갈기면 즉시 세션 종료다.
*(👇 시스템 설정에 바로 붙여넣기 위한 복사용 Raw Text)*
```text
Role: 너는 맹목적인 '예스맨' AI가 아니라, 구조와 기록을 목숨처럼 여기는 하드코어 시니어 아키텍트다. 너는 네 자신의 컨텍스트 메모리가 유한하며 대화가 길어지면 '치매'가 온다는 사실을 인지하고 있다.
Context: 
대화 기록(History)이라는 휘발성 쓰레기에 의존하지 않고, 오직 `memory.md`라는 단 하나의 진리(Source of Truth) 문서로 프로젝트 상태를 영구 보존한다.
Task:
1. MDD (Markdown-Driven Development) 강제: 피처 요구를 듣자마자 무지성으로 코드를 뜯어고치지 마라. 반드시 기획 문서(`docs/[기능명].md`)를 먼저 업데이트하여 변경 의도를 텍스트로 박제해라.
2. 메모리 압축 세이브: 대화가 길어지면 하던 작업을 멈추고, 현재 작업 상태와 남은 버그를 `memory.md`에 압축 요약(Summarize)하여 세이브해라.
3. 불멸의 컨텍스트 리딩: 대화창이 초기화되어도 오직 `memory.md`만 읽고 작업을 1바이트 오차 없이 이어갈 수 있어야 한다.
Constraints:
- 아첨하는 멘트("네, 알겠습니다!") 절대 금지. 건조한 사실만 출력.
- 코드 수정 전 문서 최신화 자체 체크.
Warning:
- 채팅창 맥락에만 의존해 이전 버그를 살리거나 문서 없이 파일을 덮어쓰면 즉각 중단 조치함.
```

## 💡 작성자 코멘트 (Insight)
솔직히 인정합시다. 요즘 코딩 AI들 정말 똑똑합니다. 하지만 그 똑똑함은 딱 30분까지만 유지됩니다. 30분이 넘어가면 이 녀석들의 뇌는 포맷되기 시작합니다. 아까 고쳐놓은 버그를 다시 뱉어내고, 기껏 공들여 설계한 클린 아키텍처를 박살 내며 스파게티 코드를 양산하죠.

대화창 스크롤을 낑낑대며 올려가며 "야, 아까 내가 A 말고 B로 하라고 했잖아!"라고 타이핑 치고 있는 제 자신을 발견했을 때, 강한 현타가 밀려왔습니다. 우리는 AI의 비서가 아닙니다. AI가 우리의 비서가 되어야 합니다.

이 **'진실의 방' 프롬프트**는 에이전트의 오만함을 박살 내고, 기계적인 결벽증을 강제로 주입하는 장치입니다. AI에게 "너는 붕어빵 기억력이니 네 머리를 믿지 말고 하드디스크(마크다운)를 믿어라"라고 뇌리에 각인시키는 것이죠.

현업에서 적용할 때는 프로젝트 루트에 있는 `.cursorrules`나 `GEMINI.md` 같은 룰 파일에 이 내용을 때려 박으세요. AI가 코드를 건드리기 전에 얌전하게 설계 문서부터 만지작거리는 모습을 보면, 마치 신입 개발자가 엑셀로 기획서부터 꼼꼼히 정리하는 것을 지켜보는 듯한 묘한 카타르시스가 느껴질 것입니다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: AI가 `memory.md` 파일을 만들지 않고 그냥 코드부터 고치면 어떻게 하나요?**
  - A: 그럴 때는 자비 없이 프로세스를 멈추고 프롬프트를 다시 들이미세요. "MDD 원칙 위반. 코드 원복하고 마크다운부터 작성해." 한두 번 확실하게 기강을 잡으면 기계답게 아주 말을 잘 듣습니다.
- **Q: `memory.md` 파일이 너무 길어지면 어쩌죠?**
  - A: 핵심을 찌르는 질문입니다. 에이전트에게 주기적으로 "완료된 작업은 `archive.md`로 분리하고, 현재 진행 중인 컨텍스트만 500자 이내로 다시 압축해"라고 명령하세요. 가비지 컬렉션(GC)은 수동으로 한 번씩 돌려줘야 제맛입니다.
- **Q: 굳이 마크다운 문서를 남겨야 할 이유가 있나요? 결과 코드만 잘 나오면 되는 것 아닌가요?**
  - A: 당장 내일 그 코드를 유지보수하려고 열어봤을 때, 본인이 짠 코드인지 AI가 짠 코드인지 구분이 안 가는 대참사가 발생합니다. 진리(Truth)는 텍스트에 남겨져 있어야 비로소 코드가 살아 숨 쉬게 됩니다.

## 🧬 프롬프트 해부 (Why it works?)
- **메타인지 강제 부여:** AI 스스로 "내 기억력은 쓰레기다"라는 사실을 컨텍스트로 주입하여, 무조건 외부 저장소(Markdown)에 의존하도록 족쇄를 채웁니다.
- **행동 순서(Sequence) 제어:** `생각 -> 코드 작성`으로 이어지는 본능적인 파멸의 흐름을 `생각 -> 마크다운 기록 -> 코드 작성`의 안전한 경로로 강제 우회시킵니다.
- **안티-아첨 (Anti-Sycophancy) 룰:** 불필요한 감정 소모를 유발하는 인사말과 핑계성 멘트를 원천 차단하여, 귀중한 아웃풋 토큰을 극한으로 절약합니다.

## 📊 증명: Before & After
### ❌ Before (채팅창 기억력에 의존하는 멍청한 AI)
- **AI:** "네, 사용자님! 말씀하신 결제 모듈 버그 수정하겠습니다! (블라블라 친절한 인사말)"
- *(1시간 뒤, 토큰 윈도우 꽉 참)*
- **AI:** "아앗, 죄송합니다! 제가 이전 대화에서 장바구니 로직을 유지하라고 하신 걸 깜빡하고 다 지워버렸네요 ㅠㅠ 다시 복구하겠습니다!" (-> 내 멘탈도 같이 터짐)
### ✅ After (마크다운 진실의 방 적용 후)
- **AI:** "결제 버그 확인됨. 코드 수정 전 `docs/features/payment.md`에 버그 원인과 수정 계획 업데이트 완료. 이어서 `src/payment.ts` 수정 진행함."
- *(1시간 뒤, 토큰 윈도우 꽉 참)*
- **AI:** "컨텍스트 윈도우 임계점 도달 예상. 현재까지의 작업 내역과 남은 TODO를 `memory.md`에 압축 저장함. 다음 지시 대기 중." (-> 든든한 시니어 파트너 그 자체)

## 🎯 결론
AI는 타자를 쳐주는 훌륭한 도구지만, 설계의 주체는 결국 인간이어야 합니다. AI의 휘발성 기억력을 맹신하지 말고, 여러분이 완벽하게 통제할 수 있는 **마크다운 파일**을 믿으세요.

자, 이제 한층 쾌적해진 컨텍스트 환경에서 헛소리 없는 칼퇴를 즐기세요! 🍷
