## 소개

수십 년 동안 프론트엔드 개발은 결정론적 계약에 의해 지배되었습니다. 개발자는 화면을 설계하고, 사용자는 이를 탐색하는 방식이었습니다. 우리는 내비게이션 바, 햄버거 메뉴, 정적 대시보드를 구축하며 우리의 정보 구조가 사용자의 멘탈 모델과 일치하기를 희망했습니다. 하지만 인터페이스가 스스로 구축될 수 있다면 어떨까요?

제너레이티브 UI(Generative UI)는 사용자 인터페이스가 미리 렌더링된 정적 결과물이 아니라, 즉석에서 생성되는 동적 구성 요소가 되는 패러다임의 전환을 의미합니다. 대형 언어 모델(LLM)과 구조화된 컴포넌트 디자인 시스템에 기반한 이 접근 방식은, 애플리케이션이 대화나 작업의 현재 맥락에 맞춰 특별히 제작된 맞춤형 UI 요소를 렌더링할 수 있게 해줍니다. 이는 "화면을 만드는 것"에서 "화면을 만드는 시스템을 구축하는 것"으로의 이동입니다.

## 분석

제너레이티브 UI의 핵심 메커니즘은 AI 에이전트가 사용자의 요청을 해석하고, 단순한 텍스트 대신 UI 컴포넌트를 나타내는 구조화된 페이로드(주로 JSON)를 반환하는 오케스트레이션 계층을 포함합니다. 그러면 프론트엔드 클라이언트는 이 페이로드를 기능적인 React, Vue 또는 네이티브 위젯으로 하이드레이션(hydration)합니다.

### 정적 메뉴의 종말

전통적인 GUI는 기억력과 탐색 경로에 크게 의존합니다. 사용자는 설정이 어디에 있는지, 분석 데이터를 보기 위해 어떻게 들어가야 하는지 기억해야 합니다. 제너레이티브 UI는 이러한 상호작용 모델을 뒤집습니다. **사용자의 의도에 따라 적응하고 스스로 생성되는 인터페이스가 정적 메뉴를 대체하고 있습니다.**

은행 앱을 생각해 보십시오. `메뉴 -> 명세서 -> 2026년 2월 -> 내보내기`로 이동하는 대신, 사용자는 단순히 "2월 지출 내역 보여줘"라고 요청합니다. 시스템은 대화 스트림 내에 대화형 도구 설명(tooltip)과 내보내기 버튼이 포함된 상호작용 가능한 지출 그래프 컴포넌트를 바로 생성합니다. "메뉴"는 의도가 그것을 필요로 할 때까지 존재하지 않았습니다.

### 기술적 구현

이를 효과적으로 구현하기 위해 개발자는 세 가지 핵심 요소가 필요합니다.

1.  **의도 인식 (Intent Recognition):** 자연어를 특정 도메인 기능에 매핑할 수 있는 강력한 백엔드 에이전트 (Gemini나 GPT-4 같은 모델 사용).
2.  **컴포넌트 레지스트리 (Component Registry):** AI가 호출할 수 있는 "단순(Dumb)" UI 컴포넌트(차트, 폼, 카드)들의 깊이 있게 타입 지정된 라이브러리. AI는 HTML을 환각(hallucinate)해내는 것이 아니라, 사용 가능한 안전한 도구 목록에서 선택합니다.
3.  **스트리밍 하이드레이션 (Streaming Hydration):** 텍스트와 함께 UI 정의를 스트리밍하는 기능으로, 인터페이스가 점진적으로 "나타나게(pop)" 하여 속도감과 반응성을 유지할 수 있게 합니다.

## 결론 {#conclusion}

제너레이티브 UI는 단순히 화려한 유행이 아닙니다. 이것은 선언적 UI 프레임워크가 확률적 AI와 결합하여 도달한 논리적 귀결입니다. 정적 페이지를 제작하는 것에서 유연한 컴포넌트 시스템을 정의하는 것으로 초점을 옮김으로써, 우리는 사용자가 자신의 경험을 직접 형성할 수 있도록 힘을 실어줍니다.

앞으로 나아감에 따라 프론트엔드 엔지니어의 역할도 진화합니다. 우리는 더 이상 단순히 픽셀을 옮기는 사람이 아닙니다. 우리는 지능이 시각적으로 발현될 수 있도록 하는 제약 조건과 구성 요소(building blocks)를 설계하는 아키텍트입니다. UI의 미래는 그려지는 것이 아니라, 생성되는 것입니다.
