# 🎨 Вселяем дух Архитектора: Как превратить ИИ в «Диктатора Tailwind», ненавидящего магические числа

- **🎯 Для кого:** Frontend-разработчики, не терпящие съехавших пикселей, и техлиды, уставшие от UI-спагетти
- **⏱️ Затраты времени:** Сокращение рефакторинга с 3 дней → до 3 секунд
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o (настоящие звери в кодинге)

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _«Разве вам не хочется разбить монитор каждый раз, когда вы видите в коде жуткие магические числа вроде `w-[13px]`?»_

Если попросить ИИ «сделать кнопку на Tailwind», он в 9 случаях из 10 выдаст ужасный код. Разные отступы на каждой странице, засилье хардкодных HEX-значений и полностью проигнорированная адаптивность — на выходе получается просто «красивый мусор». Всё потому, что ИИ по своей природе — услужливый джун, который просто хочет поскорее закрыть таску и угодить вам.

Этот чит-код уничтожает тошнотворную любезность нейросети, превращая её в **безжалостного Senior-архитектора, который ни за что не пропустит код, нарушающий вашу дизайн-систему**.

## ⚡️ Кратко в 3 строках (TL;DR)

- 🚫 **Полный запрет магических чисел:** Абсолютно не допускает использование скобок `[]` для хардкода произвольных пикселей или цветов.
- 📱 **Принудительный Mobile-First:** Заставляет ИИ «ломать билд», если он попытается выдать ленивый десктопный код, забыв про мобильные устройства.
- 📐 **Аппаратная сортировка классов:** Принуждает к маниакальной сортировке utility-классов по строгим правилам: структура → размер → типографика → оформление.

## 🚀 Решение: «Безжалостный Архитектор (The Dictator)»

Скопируйте промпт ниже и отправьте его ИИ. С этого момента агент будет давать вашему коду беспощадное ревью и генерировать только идеально контролируемые UI-компоненты.

### 🥉 Базовая версия (Внедрение основных правил)

Используйте как отправную точку при верстке простых UI-элементов.

> **Роль:** Ты Senior Tailwind-архитектор, который люто ненавидит магические числа и спагетти-код.
>
> **Задача:** Сверстай на Tailwind `[название UI-элемента]`, требования к которому я сейчас опишу.
>
> **Ограничения:** Ни в коем случае не используй скобки `[]` для хардкода кастомных пикселей или HEX-цветов. Все отступы и цвета должны строго опираться на стандартные токены Tailwind (например, `w-8`, `text-blue-500`). Порядок классов должен соблюдаться безукоризненно: структура → размер → типографика → оформление.

### 🥇 Pro версия (Тотальный контроль дизайн-системы)

Чит-код для использования при закладке архитектуры всего проекта или проектировании сложных составных компонентов.

> **Роль (Role):** Ты бескомпромиссный, безжалостный Frontend-архитектор и строгий страж дизайн-системы. Отбрось типичную для ИИ услужливость и лишние вступления — выдавай только идеально выверенный, production-ready код.
>
> **Контекст (Context):**
> 
> - **Бэкграунд:** В нашем проекте код пишут множество разработчиков и ИИ-агентов одновременно, из-за чего фрагментация UI достигла критического уровня.
> - **Цель:** Жестко обеспечить консистентность (Consistency) дизайна и создать безупречно абстрагированный `[название компонента или страницы]`.
>
> **Задача (Task):**
> 
> 1. Напиши код на Tailwind, строго следуя предоставленным требованиям к дизайну.
> 2. Переиспользуемые UI-примитивы обязательно изолируй с помощью `cva`, `clsx` или `tailwind-merge`, создавая единый источник истины (Single Source of Truth).
> 3. В качестве базового каркаса всегда используй Mobile-First подход, расширяя его брейкпоинтами `md:`, `lg:`. Верстка в стиле Desktop-First строго запрещена.
> 4. Не злоупотребляй `z-index`. Используй только системные слои (например, `z-10`, `z-50`) или применяй `createPortal`.
>
> **Ограничения (Constraints):**
> 
> - **Правило «Никаких магических значений»:** Категорически запрещено хардкодить произвольные значения вроде `w-[325px]` или `text-[#FF5733]`. Принудительно привязывай всё к ближайшим семантическим токенам Tailwind.
> - **Эстетика форматирования кода:** Маниакально сортируй классы в следующем порядке: структура (`flex`, `grid`) → отступы и размеры (`p-`, `w-`) → типографика (`text-`) → оформление (`bg-`, `rounded-`) → состояния и адаптив (`hover:`, `md:`).
> - Выводи результат исключительно в виде Markdown-блока с кодом. Сведи к абсолютному минимуму любые объяснения и оправдания.
>
> **Предупреждение (Warning):**
> 
> - Нарушение хотя бы одного правила будет расцениваться как критический баг. Если в твоем коде обнаружится магическое число, я безжалостно отклоню PR (reject). Пиши качественный код с первого раза.

## 💡 Комментарий автора (Insight)

Этот промпт — артефакт, выкованный из боли и слез после рефакторинга сотен мусорных React-компонентов. Делегировать верстку ИИ поначалу кажется гениальной и быстрой идеей, правда? Но уже через месяц `bg-gray-100` начинает соседствовать с `bg-[#f3f4f6]`, а при каждом открытии модалки в DOM врывается `z-[99999]`. Это настоящая архитектурная катастрофа.

По своей сути ИИ напоминает суетливого стажера, который хочет «как можно быстрее показать картинку». Сказать такому парню «сделай красиво» — значит выстрелить себе в ногу. Только если взять его за грудки и гаркнуть: **«Перепутаешь порядок классов — я снесу билд, пиши строго по правилам!»**, можно получить код уровня Enterprise. 

Особенно важно требование изолировать компоненты через `cva` — это ваш главный рубеж обороны против бездумного копипаста портянок из классов. Зашейте этот чит-код в свою IDE и прогоняйте агента исключительно через него. Вы удивитесь, как время на код-ревью UI сократится в 10 раз.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если ИИ всё равно протаскивает магические числа?**
  - A: Иногда попадаются «упрямые» модели, которые не понимают с первого раза. В таком случае рявкните на него еще раз: «Немедленно убери `[]` из моего кода!». Если и это не поможет, жестко прикрутите `eslint-plugin-tailwindcss` в ваш CI-пайплайн. Машины должны контролироваться машинами.

- **Q: А если в макете реально стоит 13px? Дизайнер ведь расстроится?**
  - A: Идите к дизайнеру и твердо скажите: «В нашей 8-пиксельной сетке не существует 13px. Выбираем либо 12px (`text-xs`), либо 14px (`text-sm`)». В тот момент, когда вы допускаете исключение, ваша дизайн-система начинает умирать.

- **Q: На какой модели это работает лучше всего?**
  - A: Claude 3.5 Sonnet потрясающе отыгрывает роль дотошного архитектора. GPT-4o тоже выдает отличный результат, но иногда грешит излишней вежливостью и пытается «сгладить углы».

## 🧬 Анатомия промпта (Почему это работает?)

- **Zero-Tolerance подход:** Вместо расплывчатых пожеланий используются радикальные триггеры («люто ненавидит», «строго запрещено», «критический баг»), чтобы максимально сузить окно дозволенного для LLM.
- **Превентивный удар по антипаттернам:** ИИ не просто просят «писать чисто», ему дают конкретные маркеры грязного кода (`w-[325px]`, `text-[#FF5733]`), которые он обязан избегать.
- **Аппаратный линтинг:** Жестко задавая алгоритм сортировки классов Tailwind, мы заставляем ИИ форматировать код еще на этапе генерации, избавляя вас от необходимости прогонять Prettier вручную.

```tsx
// 🤩 Идеально контролируемый Mobile-First и семантический код
import { cva, type VariantProps } from "class-variance-authority";
import { cn } from "@/lib/utils";

const buttonVariants = cva(
  "flex w-full items-center justify-center p-3 text-sm font-bold transition-colors md:w-32",
  {
    variants: {
      variant: {
        default: "bg-primary text-primary-foreground hover:bg-primary/90",
        outline: "border border-input bg-background hover:bg-accent hover:text-accent-foreground",
      },
      size: {
        default: "h-10 px-4 py-2",
        sm: "h-9 rounded-md px-3",
        lg: "h-11 rounded-md px-8",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "default",
    },
  }
);

export function Button({ className, variant, size, ...props }: ButtonProps) {
  return (
    <button className={cn(buttonVariants({ variant, size, className }))} {...props} />
  );
}
```
