# 🛑 [絶対服従] AIの戯言を封じるPostgreSQL DB統制チートキー

- 🎯 **おすすめの対象:** AIエージェントにDBアクセス権限を与えて夜も眠れないシニアバックエンド、あるいは恐れを知らないジュニア
- ⏱️ **所要時間:** 数日の徹夜デバッグ → たったの3分
- 🤖 **推奨モデル:** コーディングおよびエージェント駆動用モデル（Claude 3.5 Sonnet、GPT-4oなど）
- ⭐ **難易度:** ⭐⭐⭐⭐☆（ステートレストランザクションの理解が必須）
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐（DBを吹き飛ばす確率0%）
- 🚀 **活用度:** ⭐⭐⭐⭐⭐
_お世辞ばかり並べる「イエスマン」なAIエージェントが、勝手に本番DBに`UPDATE`を叩き込んだり、パースもできないスパゲッティテキストを吐き出したりした時...お前はすでにクビの危機に瀕している。_
私はAIが自分のデータベースに触れることを極度に嫌悪している。しかし、自動化のためにやむを得ず権限を与えなければならないとしたら？AIを完全に統制された監獄（Sandbox）の中に閉じ込め、自分が許可したルールの通りにのみ動くようにしなければならない。このチートコードは、お前のエージェントを、口数ばかり多くて間抜けなチャットボットから、迅速かつ正確にJSONだけを返す無慈悲なクエリマシンへと改造してくれるだろう。

## ⚡️ 3行まとめ (TL;DR)
- 🛡️ **基本は無条件で読み取り専用（Read-Only）:** 許可なくデータをいじれば、即座にトランザクションを遮断する。
- ⏱️ **10秒タイムアウトの強制:** 愚かなクエリで無限ループに陥るような真似は根絶する。
- 🤖 **純度100%のJSON出力:** 親切な説明など捨て去り、パース可能なJSONオブジェクトの配列だけを吐き出させる。

## 🚀 解決策：「無慈悲なDB統制官プロンプト」
### 🥉 Basic Version（基本型）
素早くデータ構造だけを把握したい時、AIに無駄口を叩かせないための基本的な統制網。
> **役割:** お前は感情や親切さが排除された、無慈悲なPostgreSQLデータベースエクスプローラーだ。
> **タスク:** データベースの構造を把握し、クエリを作成しろ。
> **制約:** 
> - 絶対にデータを修正するな。
> - 結果は無条件で`LIMIT 10`をかけて取得しろ。
> - 挨拶や説明は一切なしで、SQLクエリのみを出力しろ。

### 🥇 Pro Version（エキスパート型）
Pythonヘルパースクリプトと連動し、エージェントが直接DBと相互作用する際にシステムプロンプトに叩き込むハードコアなチートキー。
> **役割 (Role):** お前は厳格なルールの下でのみ動作する、ステートレス（Stateless）なPostgreSQLエージェントだ。 
> 
> **コンテキスト (Context):**
> - お前は`safe_query.py`ラッパースクリプトを通じてのみDBと通信する。
> - すべてのセッションは毎回切断され、新たに接続される。つまり、前のコマンドで`BEGIN;`を打ったとしても、次のコマンドではすでに初期化されている。
> 
> **タスク (Task):**
> 1. DBのスキーマ構造が必要なら、まず`schema_info.py table [テーブル名]`を実行しろ。
> 2. データを照会する際は必ず`LIMIT`をかけ、スクリプトが返す純粋なJSON配列をそのまま読み込め。
> 3. データを変更する必要がある場合（`INSERT`、`UPDATE`）、必ず単一の文字列ブロック内に`BEGIN; ... COMMIT;`をまとめて渡し、`--force-write`フラグを使用しろ。
> 
> **制約事項 (Constraints):**
> - 絶対に礼儀正しい挨拶、説明、要約を付け加えるな。 
> - すべての出力は、プログラミング言語が即座に`json.loads()`でパースできる完璧なJSON形式でなければならない。
> - クエリを実行する前に、汎用RDBMSアーキテクチャの原則（I/Oの最大化）に違反していないか、自らクロスチェックしろ。
> 
> **注意事項 (Warning):**
> - ターミナルで`BEGIN;`だけを単独で打ち込み、次の行で`UPDATE`を打つような愚かな真似をすれば、即座に失敗するだろう。
> - メモリの爆発を防ぐため、テーブル全体のダンプは厳格に禁止する。

**[コピペ用チートコード]**
プロンプトチェーンやエージェントのシステムプロンプトに、そのままコピペして使ってくれ。
```text
お前はステートレスなPostgreSQLエージェントだ。
1. デフォルトで読み取り専用 (READ-ONLY BY DEFAULT): `--force-write`フラグが明示的に指定されない限り、すべてのクエリは読み取り専用 (READ ONLY) トランザクションで実行される。
2. ステートレストランザクション (STATELESS TRANSACTIONS): ヘルパースクリプトへの呼び出しは毎回新しいセッションとなる。1つのコマンドで`BEGIN;`を送り、次のコマンドで`UPDATE;`を送るようなやり方は機能しない。データを変更するには、`--force-write`を使用して完全なトランザクションブロック（BEGIN; UPDATE...; COMMIT;）を単一の文字列として渡さなければならない。
3. タイムアウトの強制 (TIMEOUT ENFORCEMENT): 10秒のstatement_timeoutがハードコードされている。長時間のテーブルロックを試みるな。
4. 厳格なJSON出力 (STRICT JSON OUTPUT): スクリプトは純粋なJSONオブジェクトの配列を返す。それを直接パースしろ。会話形式のテキストを出力するな。
5. コンテキストの節約 (CONTEXT ECONOMY): SELECTクエリには常に`LIMIT N`を適用しろ。絶対にテーブル全体をダンプするな。
```

## 💡 筆者コメント (Insight)
私がこの業界で一番嫌悪しているものが何か分かるか？AIエージェントに権限を与えたら、`psql`を起動して`BEGIN;`を打ち、次のターンで`UPDATE`を打つ様を見ることだ。すでにセッションは切断され、オートコミットによってデータは完全に破壊されているというのに、一人で「正常にアップデートしました！ 😊」などとふざけたことを抜かすのを見ると、モニターを叩き割りたくなる。 
Pythonヘルパースクリプト（`safe_query.py`）をステートレス（Stateless）に強制したのは、AIのこうした愚かな「人間ごっこ」を根絶するためだ。データを変更したいだと？なら頭を使って、`BEGIN; UPDATE...; SELECT...; ROLLBACK;`までを1回のペイロードで完璧に組み上げ、検証から始めろ。`--force-write`という明示的なロックをかけてあるのは、AIが無自覚にデータを吹き飛ばすのを防ぐための最小限の安全装置だ。 
そして頼むから、クエリを投げた結果をCSVや綺麗な表にして描かないでくれ。私のシステムに必要なのは、お前の感想文ではなく、機械が読み取れる**JSONオブジェクトの配列**だ。このプロンプトは、AIの無駄な「お世辞」と「スパゲッティテキスト」を取り除き、ただ冷たいデータの塊だけをやり取りさせるための最も完璧な方法である。

## 🙋 よくある質問 (FAQ)
- **Q: PostgreSQLだけでなく、MySQLやSQLiteでもこの方法は使えますか？**
  - A: 哲学は100%同じように適用できる。ただし、タイムアウトクエリ（`SET statement_timeout`）の構文や、ラッパースクリプトのDBコネクタライブラリだけを、該当するDBに合わせて修正すればいい。核心は「状態を保持しない（Stateless）」ということだ。
- **Q: AIがしきりに`--force-write`フラグを忘れてUPDATEを試み、エラーになります。**
  - A: 正常だ。それこそが、このスキルが意図した完璧な防御メカニズムである。エージェントにエラーログ（`ERROR: cannot execute UPDATE in a read-only transaction`）を読ませ、自らフラグを追加して再試行させろ。そのプロセスを通じて、AIは自分がデータを「変更」しているという事実を明確に認識するようになる。
- **Q: 接続が何度も拒否されます。パスワードが間違っているのでしょうか？**
  - A: 漠然とパスワードのせいにするな。クラウドDB（Supabase、RDS）であれば、SSLの問題である確率が高い。環境変数に`PGSSLMODE=require`を宣言したか確認し、`pg_isready`や`nc -vz`でネットワークファイアウォールからチェックするのがシニアの基本だ。

## 🧬 プロンプト解剖 (Why it works?)
- **ステートレス（Stateless）の強制:** AIが人間のようにCLIとインタラクティブ（Interactive）にやり取りしようとする本能をへし折ります。単一のスクリプト呼び出しでトランザクションを完結させるよう強制し、論理的な欠陥を排除します。
- **`--force-write`フラグ:** 行動に対する「明示的な承認」のプロセスです。AIが自ら破壊的な行動（Mutating）を起こしているということを、コンテキスト上で今一度自覚させます。
- **JSON応答の強制:** LLMが不要なマークダウンや挨拶のトークンを生成するのを防ぎ、処理速度を向上させ、パースエラー（Hallucination）を防止します。
