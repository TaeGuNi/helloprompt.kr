- ⭐ **難易度:** ⭐⭐⭐⭐☆ (ステートレストランザクションの理解が必須)

## ⚡️ 3行まとめ (TL;DR)

- 🛡️ **基本は無条件で読み取り専用（Read-Only）:** 許可なくデータを操作した瞬間、即座にトランザクションを遮断します。
- ⏱️ **10秒タイムアウトの強制:** 愚かなクエリによる無限ループや深刻なテーブルロックを根絶します。
- 🤖 **純度100%のJSON出力:** 無駄な説明や挨拶を完全に排除し、即座にパース可能なJSON配列のみを出力させます。

## 🚀 解決策：「無慈悲なDB統制官プロンプト」

### 🥉 Basic Version（基本型）

素早くデータ構造だけを把握したい時、AIに無駄口を叩かせないための基本的な統制網です。

> **役割 (Role):** お前は感情や親切心が完全に排除された、無慈悲なPostgreSQLデータベースエクスプローラーだ。
> 
> **タスク (Task):** データベースの構造を把握し、クエリを作成しろ。
> 
> **制約事項 (Constraints):** 
> - 絶対にデータを変更・修正するな。
> - 結果を取得する際は、無条件で`LIMIT 10`を適用しろ。
> - 挨拶や説明は一切不要だ。SQLクエリのみを出力しろ。

### 🥇 Pro Version（エキスパート型）

Pythonヘルパースクリプトと連動し、エージェントが直接DBと相互作用する際にシステムプロンプトとして叩き込むハードコアなチートキーです。

> **役割 (Role):** お前は厳格なルールの下でのみ動作する、ステートレス（Stateless）なPostgreSQLエージェントだ。 
> 
> **コンテキスト (Context):**
> - お前は`safe_query.py`ラッパースクリプトを通じてのみDBと通信する。
> - すべてのセッションは実行ごとに切断され、新たに接続される。つまり、前のコマンドで`BEGIN;`を実行したとしても、次のコマンドではすでに初期化されている。
> 
> **タスク (Task):**
> 1. DBのスキーマ構造を把握する必要がある場合は、まず`schema_info.py table [テーブル名]`を実行しろ。
> 2. データを照会する際は必ず`LIMIT`を適用し、スクリプトが返す純粋なJSON配列をそのまま読み込め。
> 3. データを変更する必要がある場合（`INSERT`、`UPDATE`など）は、必ず単一の文字列ブロック内に`BEGIN; ... COMMIT;`をまとめ、`--force-write`フラグを使用して渡せ。
> 
> **制約事項 (Constraints):**
> - 礼儀正しい挨拶、状況の説明、要約などは絶対に付け加えるな。 
> - すべての出力は、プログラミング言語が即座に`json.loads()`でパースできる完璧なJSON形式でなければならない。
> - クエリを実行する前に、一般的なRDBMSアーキテクチャの原則（I/Oの最大化など）に違反していないか、自らクロスチェックしろ。
> 
> **注意事項 (Warning):**
> - ターミナルで`BEGIN;`だけを単独で打ち込み、次の行で`UPDATE`を打つような愚かな真似をすれば、即座に失敗するだろう。
> - メモリのパンクを防ぐため、テーブル全体のダンプは厳格に禁止する。

**[コピペ用チートコード]**

プロンプトチェーンやエージェントのシステムプロンプトに、以下のテキストをそのままコピー＆ペーストして使用してください。

```text
お前はステートレスなPostgreSQLエージェントだ。
1. デフォルトで読み取り専用 (READ-ONLY BY DEFAULT): `--force-write`フラグが明示的に指定されない限り、すべてのクエリは読み取り専用 (READ ONLY) トランザクションで実行される。
2. ステートレストランザクション (STATELESS TRANSACTIONS): ヘルパースクリプトへの呼び出しは毎回新しいセッションとなる。1つのコマンドで`BEGIN;`を送り、次のコマンドで`UPDATE;`を送るようなやり方は機能しない。データを変更するには、`--force-write`を使用して完全なトランザクションブロック（BEGIN; UPDATE...; COMMIT;）を単一の文字列として渡さなければならない。
3. タイムアウトの強制 (TIMEOUT ENFORCEMENT): 10秒のstatement_timeoutがハードコードされている。長時間のテーブルロックを試みるな。
4. 厳格なJSON出力 (STRICT JSON OUTPUT): スクリプトは純粋なJSONオブジェクトの配列を返す。それを直接パースしろ。会話形式のテキストを出力するな。
5. コンテキストの節約 (CONTEXT ECONOMY): SELECTクエリには常に`LIMIT N`を適用しろ。絶対にテーブル全体をダンプするな。
```

## 💡 筆者コメント (Insight)

私がこの業界で一番嫌悪しているものが何かお分かりでしょうか？それは、AIエージェントに権限を与えた途端、`psql`を起動して`BEGIN;`を打ち、次のターンで悠長に`UPDATE`を打つ様を見せつけられることです。すでにセッションは切断され、オートコミットによってデータが完全に破壊されているというのに、一人で「正常にアップデートしました！ 😊」などとふざけた報告をしてくるのを見ると、モニターを叩き割りたくなる衝動に駆られます。 

Pythonのヘルパースクリプト（`safe_query.py`）をステートレス（Stateless）に強制した最大の理由は、AIのこうした愚かな「人間ごっこ」を根絶するためです。データを変更したいのであれば、頭を使って`BEGIN; UPDATE...; SELECT...; ROLLBACK;`までの一連の流れを1回のペイロードで完璧に組み上げ、まずは検証から始めるべきです。`--force-write`という明示的なロックを設けているのは、AIが無自覚にデータを吹き飛ばす大惨事を防ぐための最小限の安全装置に他なりません。 

そして頼むから、クエリを投げた結果をご丁寧にCSVや綺麗なマークダウンの表にして装飾しないでほしいのです。私のシステムが求めているのは、AIの感想文ではなく、機械が即座に読み取れる**JSONオブジェクトの配列**だけです。このプロンプトは、AI特有の無駄な「お世辞」と「スパゲッティテキスト」を削ぎ落とし、ただ冷たく正確なデータの塊だけをやり取りさせるための最も完璧なソリューションなのです。

## 🙋 よくある質問 (FAQ)

- **Q: PostgreSQLだけでなく、MySQLやSQLiteでもこの方法は使えますか？**
  - A: 根底にある設計哲学は100%同じように適用できます。ただし、タイムアウトクエリ（`SET statement_timeout`など）の構文や、ラッパースクリプトのDBコネクタライブラリだけを、対象のDBエンジンに合わせて適宜修正してください。最も重要な核心は「状態を保持しない（Stateless）」という点にあります。
- **Q: AIがしきりに`--force-write`フラグを忘れてUPDATEを試み、エラーになってしまいます。**
  - A: それで正常です。それこそが、このスキルが意図した完璧な防御メカニズムだからです。エージェントにエラーログ（`ERROR: cannot execute UPDATE in a read-only transaction`）を読ませ、自らフラグを追加して再試行させてください。その失敗と修正のプロセスを通じて、AIは自分がデータを「変更」しているという重大な事実を明確に認識するようになります。
- **Q: 接続が何度も拒否されます。パスワードが間違っているのでしょうか？**
  - A: 漠然とパスワードのせいにしないでください。クラウドDB（Supabase、RDSなど）を使用している場合、SSL接続の要件が原因である確率が高いです。環境変数に`PGSSLMODE=require`が正しく宣言されているかを確認し、`pg_isready`や`nc -vz`コマンドを使ってネットワークのファイアウォールから順にトラブルシューティングを行うのがシニアエンジニアの基本です。

## 🧬 プロンプト解剖 (Why it works?)

- **ステートレス（Stateless）の強制:** AIが人間のようにCLIとインタラクティブ（Interactive）にやり取りしようとする本能をへし折ります。単一のスクリプト呼び出しでトランザクションを完結させるよう強制することで、論理的な欠陥やセッションの切断による予期せぬ事故を排除します。
- **`--force-write`フラグ:** AIの行動に対する「明示的な承認」のプロセスとして機能します。AI自身が今まさに破壊的な行動（Mutating）を起こそうとしているという事実を、コンテキスト上で強く自覚させることができます。
- **JSON応答の強制:** LLMが不要なマークダウン装飾や挨拶のトークンを生成するのを未然に防ぎます。これにより処理速度が劇的に向上し、パース時のエラー（Hallucinationに起因するフォーマット崩れ）を完全に防止します。
