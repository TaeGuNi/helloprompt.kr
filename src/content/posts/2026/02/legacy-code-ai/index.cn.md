## 🚀 解决方案：“遗留代码拯救者”提示词

### 🥉 基础版 (Basic Version)

当你面对一段宛如天书、逻辑断层的函数，急需快速摸清其脉络时使用。

> **角色 (Role)：** 你是一位拥有 20 年实战经验的资深架构师。
> 
> **请求 (Task)：** 请用通俗易懂的语言，为我拆解并解释以下代码的核心业务逻辑，同时精准定位出其中最可能引爆的两个性能瓶颈或安全隐患。

### 🥇 进阶版 (Pro Version)

当你需要对核心系统模块动刀进行深度重构，且对线上稳定性有着极严苛的要求时使用。

> **角色 (Role)：** 你是一位顶尖的软件工程师兼代码重构专家，深谙各类设计模式与遗留代码的现代化改造（完美契合经典《重构》一书的核心理念）。
>
> **背景 (Context)：**
>
> - 当前状况：我刚刚接手了一段毫无注释的 `[填写具体的语言或框架，例如：Java/Spring Boot]` 遗留代码。该模块目前耦合度极高，处于牵一发而动全身的脆弱状态。
> - 最终目标：在**绝对不改变任何原有业务逻辑**的前提下，将其重构为符合现代 `[填写目标语言或框架，例如：Node.js/NestJS]` 规范、高内聚且低耦合的整洁代码，并为其补全严密的单元测试。
>
> **任务 (Task)：**
>
> 1. 深度剖析并还原这段代码的真实业务意图与核心执行链路。
> 2. 针对当前代码的所有极端边界情况（Edge Cases），编写高覆盖率的单元测试用例。
> 3. 提供重构后的完整代码。**要求**：严格遵守现代命名规范，抽取重复逻辑，并彻底抹除嵌套地狱（Callback Hell / Deep Nesting）。
> 4. `[填写需重点优化的特定函数名称]` 是当前的重灾区，请针对该函数提供详细的优化前后思路对比。
>
> **约束条件 (Constraints)：**
>
> - 所有的输出必须采用排版清晰的 Markdown 格式。
> - 测试用例代码必须基于 `[填写指定的测试框架，例如：Jest 或 JUnit]` 编写。
> - 必须确保重构前后的函数输入（参数）与输出（返回值）类型绝对一致。
>
> **警告 (Warning)：**
>
> - 如果你在解析过程中发现某段逻辑存在明显的业务缺陷，但无法 100% 确认其真实意图，请务必高亮指出。**严禁**依靠主观臆测去擅自修改任何隐性业务规则。

## 💡 作者见解 (Insight)

在处理遗留代码时，最致命的诱惑莫过于“为了重构而重构”——这种冲动往往是引发线上服务全线崩溃的罪魁祸首。因此，这套提示词的底层灵魂在于 **“测试先行” (Test-Driven Refactoring)** 策略。

先强制要求 AI 读懂代码并逆向推导出完整的**测试用例**，这一步动作死死锁定了原有的业务逻辑边界，相当于在你大刀阔斧修改代码之前，先在悬崖下方拉起了一张坚不可摧的安全网。

我曾亲身实践过这种极客流派：仅用了一个下午，就将一个充斥着恶心全局变量、逻辑盘根错节的 3000 行老旧 jQuery 面条代码，丝滑且零风险地迁移到了现代的 React Hooks 架构中。整个过程没有触发任何一起线上 P0 级事故。这种“先筑墙，后动土”的稳健做派，才是区分普通码农与**顶尖资深工程师**的真正分水岭。

## 🙋 常见问题 (FAQ)

- **Q: 如果遗留代码中夹杂着公司核心机密，可以直接扔给 AI 处理吗？**
  - A: **绝对不行。** 在调用任何公共云端 AI 服务前，必须进行严格的数据脱敏（例如手动替换掉真实的 API Key、敏感数据库表名以及核心业务算法参数）。最稳妥的方案是，向公司申请使用已部署的私有大模型，或是采购明确承诺“不使用用户数据进行模型训练”的企业版服务（如 ChatGPT Enterprise / Claude for Work）。

- **Q: AI 一键生成的重构代码，真的能做到“开箱即用”吗？**
  - A: 并非 100% 完美。即使是目前最强大的大模型，也偶尔会产生“幻觉”或自作主张地引入项目中并不存在的第三方依赖库。因此，你**必须**在本地环境中实际跑通 AI 提供的单元测试，并拉上同事进行极其严苛的代码审查（Code Review）后，再将其合并入主分支。请永远记住：AI 只是为你减轻负担的副驾驶，真正的方向盘**必须**死死握在你的手中。

## 🧬 提示词解剖 (Why it works?)

1. **绝对的约束力边界 (Constraints)：** 在提示词中严正警告“不改变原有业务逻辑”与“保持输入输出一致”，这就如同给 AI 带上了项圈，极其有效地防止了模型在重构时过度自我发挥，从而死守系统稳定性的底线。
2. **高阶角色注入 (Role Prompting)：** 直接赋予 AI“资深架构师”与“重构专家”的头衔，这一指令能瞬间激活大模型底层训练数据中关于《重构 (Refactoring)》、《整洁代码 (Clean Code)》等工程学圣经的高质量认知权重。
3. **思维链拆解 (Chain-of-Thought)：** 强制要求 AI 遵循“理解业务 -> 构建测试 -> 执行重构 -> 逻辑解释”的流水线步骤。这不仅通过逐步推导大幅压制了 AI 产生幻觉的概率，更为人类开发者提供了一份清晰的、可分步验证的重构审计报告。

## 📊 效果对比：Before & After

### ❌ Before (旧代码片段)

```javascript
function processData(d) {
  var res = [];
  for (var i = 0; i < d.length; i++) {
    if (d[i].status == 1 && d[i].t > new Date().getTime() - 86400000) {
      var x = d[i].v * 1.2; // tax
      res.push({ id: d[i].id, val: x });
    }
  }
  return res;
}
```

### ✅ After (AI 重构结果)

```javascript
/**
 * 筛选过去24小时内状态为激活(1)的数据，并计算含税价值(20%税率)。
 */
const TAX_RATE = 1.2;
const ONE_DAY_MS = 86400000;

export function processRecentActiveData(dataItems) {
  if (!Array.isArray(dataItems)) return [];

  const twentyFourHoursAgo = Date.now() - ONE_DAY_MS;

  return dataItems
    .filter((item) => item.status === 1 && item.t > twentyFourHoursAgo)
    .map((item) => ({
      id: item.id,
      val: Number((item.v * TAX_RATE).toFixed(2)),
    }));
}
```
