## 📝 Votre code ressemble à un plat de spaghettis ? Confiez-le à un expert en refactoring

- **🎯 Public cible :** Développeurs juniors, Ingénieurs logiciels, Tech Leads
- **⏱️ Temps gagné :** De plusieurs heures de débogage → à 2 minutes seulement
- **🤖 Modèles recommandés :** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro

- ⭐ **Difficulté :** ⭐⭐☆☆☆
- ⚡️ **Efficacité :** ⭐⭐⭐⭐⭐
- 🚀 **Utilité :** ⭐⭐⭐⭐⭐

> _"Votre code fonctionne, certes, mais avez-vous peur d'y toucher ? Il est grand temps d'arrêter de produire des « déchets fonctionnels » et d'embrasser l'art du Clean Code."_

Nous avons tous, un jour ou l'autre, rédigé du code à la va-vite pour respecter une échéance oppressante. Le véritable problème surgit quelques mois plus tard, lorsque ce même code se métamorphose en un véritable cauchemar de maintenance. Vous hésitez à modifier la moindre ligne par crainte de faire s'effondrer tout le système ? C'est ce que l'on appelle la dette technique. Heureusement, ce prompt est conçu pour métamorphoser n'importe quel script confus en un authentique chef-d'œuvre de lisibilité, en appliquant avec une précision chirurgicale les principes fondamentaux du Clean Code et de l'ingénierie logicielle.

---

## ⚡️ 3 points clés (TL;DR)

1. **Analyse instantanée :** Détection immédiate des "code smells" (mauvaises pratiques) dissimulés au cœur de votre logique métier.
2. **Refactoring ciblé :** Amélioration drastique de la lisibilité sans jamais altérer le comportement initial de votre application.
3. **Apprentissage continu :** Explication détaillée du _pourquoi_ de chaque modification, vous transformant en un meilleur développeur au quotidien.

## 🚀 La Solution : "L'Architecte du Clean Code"

### 🥉 Version Basique (Pour des corrections rapides)

Idéale pour un nettoyage express d'une petite fonction ou d'un composant isolé.

> **Rôle :** Tu es un développeur senior expert en Clean Code.
> **Tâche :** Refactore le code suivant pour le rendre plus lisible et maintenable, tout en conservant scrupuleusement la même fonctionnalité.
> **Code :** `[Insérez votre code ici]`

### 🥇 Version Pro (Pour une restructuration experte)

À privilégier pour des fichiers complexes ou des modules entiers qui exigent une refonte architecturale approfondie.

> **Rôle (Role) :** Tu es un Ingénieur Logiciel Principal (Principal Software Engineer) fort de plus de 10 ans d'expérience. Tu es un expert absolu des principes SOLID, DRY, KISS et du Clean Code.
>
> **Contexte (Context) :**
>
> - Projet : `[Description brève de l'application ou du module]`
> - Problème actuel : Le code fonctionne, mais il s'avère difficile à lire, mal structuré et complexe à tester.
> - Objectif : Obtenir un code élégant, modulaire, hautement lisible et prêt pour une mise en production.
>
> **Tâche (Task) :**
>
> 1. Analyse le code fourni et identifie l'ensemble des "code smells" ou anti-patterns.
> 2. Refactore le code en appliquant rigoureusement les meilleures pratiques du langage utilisé.
> 3. Ajoute des commentaires pertinents (uniquement là où la logique métier l'exige, en évitant les évidences).
> 4. Sépare les responsabilités (Single Responsibility Principle) si une fonction accomplit trop de tâches simultanément.
>
> **Contraintes (Constraints) :**
>
> - Le comportement final DOIT être strictement identique au code d'origine (aucune régression fonctionnelle n'est tolérée).
> - Utilise des noms de variables et de fonctions extrêmement explicites.
> - Fournis le résultat sous la forme de blocs de code Markdown parfaitement formatés.
>
> **Avertissement (Warning) :**
>
> - Ne modifie en aucun cas la logique métier. Si une section du code te semble ambiguë, signale-le explicitement au lieu d'inventer une solution de toutes pièces.
>
> **Format de sortie :**
>
> 1. Diagnostic : Liste à puces détaillant les problèmes identifiés.
> 2. Code Refactorisé : Le nouveau code, prêt à être intégré.
> 3. Explication des changements : Résumé concis des principes d'ingénierie appliqués.
>
> **Code à analyser :**
>
> `[Collez votre code spaghetti ici]`

## 💡 Note de l'auteur (Insight)

L'erreur la plus fréquente lors de l'utilisation de l'IA consiste à formuler une requête basique telle que : "Améliore ce code". Le résultat ? Un algorithme générique, souvent dénué de contexte, qui bafoue l'architecture globale de votre projet. En attribuant le rôle de _Principal Engineer_ et en exigeant le respect strict des principes SOLID, vous contraignez le modèle à justifier chacun de ses choix architecturaux.

J'utilise personnellement cette version Pro lors de mes revues de code (Code Reviews). Non seulement elle assainit la logique sous-jacente, mais elle m'enseigne régulièrement de nouvelles méthodes natives ou des design patterns optimisés que je n'aurais pas nécessairement envisagés. C'est l'équivalent d'avoir un mentor senior à vos côtés, disponible 24 heures sur 24 !

## 🙋 Foire Aux Questions (FAQ)

- **Q : L'IA risque-t-elle de modifier la logique métier de mon application à mon insu ?**
  - R : Ce risque existe toujours. C'est précisément la raison pour laquelle la contrainte exigeant un "comportement strictement identique" est indispensable. Je vous recommande vivement de vous assurer que vos tests unitaires (TDD) sont opérationnels avant de fusionner le refactoring proposé.

- **Q : Quel modèle d'IA est actuellement le plus performant pour le refactoring de code ?**
  - R : À l'heure actuelle, **Claude 3.5 Sonnet** excelle tout particulièrement dans la compréhension du contexte logiciel et génère un code d'une propreté clinique. **GPT-4o** et **Gemini 2.5 Pro** constituent également d'excellentes alternatives de haut niveau.

- **Q : Puis-je utiliser ce prompt pour n'importe quel langage de programmation ?**
  - R : Absolument ! L'IA adaptera de manière autonome les conventions (PEP 8 pour Python, CamelCase pour JavaScript/TypeScript, etc.) en se basant sur la syntaxe du code que vous lui soumettez.

## 🧬 Décryptage du Prompt (Why it works?)

1. **Personnalité de l'expert :** En invoquant un "Principal Software Engineer", nous calibrons l'IA pour qu'elle dépasse le simple objectif de faire fonctionner le code. Elle est forcée de l'optimiser pour une maintenabilité pérenne.
2. **Exigence de diagnostic :** Obliger l'IA à expliquer _ce qui posait problème_ avant de dévoiler la solution renforce considérablement son raisonnement interne (Chain-of-Thought), garantissant ainsi un refactoring beaucoup plus pertinent.
3. **Tolérance zéro pour les régressions :** La contrainte explicite relative à la préservation du comportement bride l'IA, l'empêchant d'ajouter des fonctionnalités superflues ou de supprimer des cas limites (edge cases) obscurs par inadvertance.

## 📊 Preuve à l'appui : Avant & Après

### ❌ Avant (Le plat de spaghettis)

```javascript
function calc(d, m) {
  let res = 0;
  if (d.length > 0) {
    for (let i = 0; i < d.length; i++) {
      if (d[i].t === "A") {
        res += d[i].v * m;
      } else {
        res += d[i].v;
      }
    }
  }
  return res;
}
```

### ✅ Après (Le Clean Code)

```javascript
/**
 * Calcule le coût total en appliquant un multiplicateur aux articles de type 'A'.
 */
function calculateTotalCost(items, multiplier) {
  if (!Array.isArray(items) || items.length === 0) {
    return 0;
  }

  return items.reduce((total, item) => {
    const itemValue = item.t === "A" ? item.v * multiplier : item.v;
    return total + itemValue;
  }, 0);
}
```
