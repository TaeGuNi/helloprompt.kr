## 🚀 How Real Experts Write Prompts

These are prompts perfected after dozens of trials and errors. Copy the prompts below and fill in the `[variables]` inside the brackets to match your situation for immediate use in the field.

### 🥉 Basic Version (For Cursor Beginners)

A general-purpose prompt suitable for implementing single component features or light bug fixes. Use this immediately in Cursor’s `Cmd+K` input window to generate results.

> **Role:** You are a senior frontend developer with 10 years of experience.
>
> **Task:** 
> 
> 1. Perfectly handle `[problem_to_solve_or_feature_to_add]` in the currently open `[target_file]`.
> 2. When writing code, you must strictly follow TypeScript's strict mode and our project's Tailwind CSS conventions.

### 🥇 Pro Version (For Windsurf / OpenClaw Architecture Refactoring)

A senior-level master prompt used when instructing large-scale architectural overhauls that shake the foundation of the system, utilizing editors that grasp the entire project context (such as Windsurf's Cascade).

> **Role:** You are our team's lead software architect and a top-tier DevOps Engineer.
>
> **Context:**
>
> - Background: The legacy codes under `[directory_path]` are currently extremely tightly coupled, reaching a limit where no further feature expansion or safe maintenance is possible.
> - Goal: The final goal is to completely refactor the architecture into one that is easy to unit test by fully introducing Dependency Injection (DI) patterns, and to update the CI/CD pipeline to a flawless state according to the changed structure.
>
> **Task:**
>
> 1. Closely analyze the core business logic of `[target_module]`, then extract abstracted interfaces to perfectly decouple dependencies between modules.
> 2. Write robust unit tests for each decoupled module, ensuring at least 80% code coverage.
> 3. Safely modify the test steps in `.github/workflows/ci.yml` to align with the newly reorganized directory structure and test environment.
>
> **Constraints:**
>
> - Under no circumstances should the public API endpoints currently used by existing clients be changed or deleted. You must guarantee 100% perfect backward compatibility.
> - All source code change history must strictly follow the `Conventional Commits` standard (`feat:`, `fix:`, `refactor:`, etc.) and be clearly summarized in commit messages.
>
> **Warning:**
>
> - Never use old libraries or packages that are already deprecated. If you are unsure about version compatibility, only apply code after checking the latest release in the official documentation using the web browsing feature. (Prevention of hallucinations and securing system stability)

## 💡 Author's Comments (Insight & How to use)

In fierce real-world environments, I utilize a **'3-Tier Hybrid System'** where I organically switch between these three editors based on different business purposes. The key is not to be buried in a single tool, but to maximize the strengths and limitations of the **Context Awareness** capabilities of each AI agent. When I need to validate a new project plan or require agile UI/UX-centered prototyping, I immediately open **Cursor**. However, when tracking mysterious critical bugs in a legacy backend system tangled over several years, or when performing a major refactor without system downtime, I switch to **Windsurf** without hesitation to secure a massive dependency safety net. Finally, right before heading home exhausted, I open the **OpenClaw** terminal and delegate the tedious DevOps closing tasks: "Update the Swagger API specs based on the business logic written today, and if all E2E tests pass, go ahead and open a GitHub PR."

**The Critical Trap of Cursor and Code Ownership:** Because AI writes code so perfectly and quickly, there is a critical danger of junior developers becoming **'Prompt Typers'** who simply rely on the Tab key and copy-pasting without deep consideration of software architecture or data structures. To prevent this, one must stop blindly accepting code and go through a rigorous review stage where the developer personally verifies the time complexity or potential memory leaks of the generated code. While AI can type code for you, you must remember that **the responsibility and ownership of the code always lie with the engineer** when a failure occurs in a production environment.

**Cautions for Windsurf and Progressive Context Strategy:** In exchange for running a powerful, deep-dive analysis engine, very large Monorepo projects may experience intermittent UI lag or response delays. Also, because the shortcut system and extension ecosystem differ slightly from the existing VS Code, there is a definite learning curve for initial adjustment. Especially when instructing it to analyze massive legacy code with thousands of files all at once, local resource (RAM, CPU) consumption can become extreme, or the AI model may hallucinate. Therefore, a **'Progressive Context Feeding Strategy'** is absolutely necessary, narrowing the scope of analysis starting from the core domain area of the project. Don't pour everything in at once; instruct by limiting the range.

**OpenClaw's Entry Barrier and Strict Sandbox Control:** For typical frontend developers accustomed to friendly GUI environments, the entry barrier is quite steep. One must endure a strange tension, anxiously watching the terminal output for fear the agent might autonomously decide to execute unexpected system destruction commands (e.g., reckless `rm -rf` or executing database schema initialization scripts). Therefore, you must allow this agent's permissions only in an **isolated Sandbox environment**, like a Docker container, that is completely segregated and safe to destroy, to prevent major system failures. Granting unlimited Root system access can quickly lead to an irreversible disaster.

**The Core of Constraint Control in Prompts:** The most important parts of the Pro Version prompt provided earlier are the **Constraints** and **Warning** blocks. Simply saying "Refactor the code" gives the AI volatile authority, leading to the production of unpredictable spaghetti code. You must strongly specify safeguards at the bottom of the prompt that could have critical business impacts, such as "Do not change existing public API endpoints" or "No use of deprecated packages." The essence of prompt engineering is to suppress excessive autonomy of the model and **strongly set boundaries** so it operates safely only within the developer's intended architecture blueprint.

In the history of software engineering, a perfect 'Silver Bullet' has never existed. Do not waste precious time searching for a single perfect master editor. In the upcoming true AI-native era, the core competency of a senior engineer will entirely depend on the **'Orchestration capability to select the AI agent that best fits the nature of the current domain and problem, and to direct them organically to derive the best results.'** Do not be a slave to the tools; be a master conductor who perfectly commands multiple AI agents.

## 🧬 Prompt Anatomy (Why it works?)

The technical reasons why the 'Pro Version' prompt introduced earlier yields accurate results without wavering, even in large-scale complex projects with dozens of modules, are as follows:
