## 🚀 Решение: "Генератор идеального Regex"

### 🥉 Basic Version (Базовая версия)

Используйте этот вариант, когда результат нужен прямо сейчас, без погружения в лишние детали.

> **Запрос:** Напиши Regex для проверки `[формат, например: номера телефонов США]`. Приведи пример кода на `[язык программирования]`.

### 🥇 Pro Version (Профессиональная версия)

Используйте этот промпт для сложных паттернов, когда требуется стопроцентная гарантия качества, подробная документация и готовые тесты для безопасной интеграции в продакшен.

> **Роль (Role):** Ты `[Senior Developer / QA Engineer]`, эксперт мирового уровня по регулярным выражениям (Regex).
>
> **Ситуация (Context):**
>
> - Контекст: Мне нужно регулярное выражение для использования в `[Python / JavaScript / Java]`.
> - Цель: Создать максимально надежный, оптимизированный и безопасный (с защитой от ReDoS) Regex для валидации данных.
>
> **Запрос (Task):**
>
> 1. Напиши регулярное выражение, которое строго удовлетворяет перечисленным ниже требованиям.
> 2. Предоставь подробное построчное объяснение (Интерпретацию) каждого символа и группы в Regex, чтобы я и моя команда могли без труда поддерживать этот код в будущем.
> 3. Напиши готовый код для юнит-тестирования этого Regex, включающий как минимум 3 совпадающих (Positive) и 3 несовпадающих (Negative) сценария.
>
> **Требования к паттерну (Constraints & Variables):**
>
> - Формат: `[номера телефонов США]`
> - Допустимые вариации: `[123-456-7890 или (123) 456-7890 или 1234567890]`
> - Опционально: `[Код города может отсутствовать]`
>
> **Внимание (Warning):**
>
> - Убедись, что Regex абсолютно не подвержен катастрофическому возврату (Catastrophic Backtracking). Если условие окажется слишком сложным, предложи разбить его на два отдельных этапа проверки.

## 💡 Комментарий автора (Insight)

Регулярные выражения — это классический пример задачи, с которой ИИ справляется в десятки раз эффективнее человека. Однако здесь кроется опасная ловушка: сгенерированный Regex может выглядеть вполне рабочим на первый взгляд, но при этом пропускать граничные случаи (edge cases) или быть уязвимым к атакам типа ReDoS (Regular expression Denial of Service).

Именно поэтому в **Pro Version** добавлен критически важный блок: жесткое требование тестовых сценариев (Positive/Negative cases) и обязательной защиты от катастрофического возврата. Мой личный опыт доказывает, что внедрение ИИ-генерации Regex в связке с автотестами экономит до 90% времени при парсинге сложных логов или валидации форм, полностью исключая пресловутый человеческий фактор при чтении запутанного синтаксиса.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если сгенерированный Regex работает некорректно на некоторых примерах?**
  - A: Просто отправьте ИИ строку, которая не прошла валидацию (или, наоборот, прошла ошибочно), со следующим комментарием: _"Этот ложный пример `[строка]` проходит проверку, хотя категорически не должен. Скорректируй Regex"_. Модель моментально проанализирует ошибку и обновит паттерн.

- **Q: Зависит ли синтаксис Regex от выбранного языка программирования?**
  - A: Да, существуют важные различия (например, PCRE, стандарты POSIX, специфические особенности движков JavaScript или Python). Поэтому в промпте мы всегда явно указываем целевой язык `[Python / JavaScript]`, чтобы ИИ подобрал правильное экранирование спецсимволов и поддерживаемые функции (например, Lookbehind).

- **Q: Можно ли попросить ИИ сделать обратное — расшифровать непонятный Regex из старого проекта?**
  - A: Абсолютно! Отправьте ИИ чужой нечитаемый Regex с запросом: _"Объясни мне этот Regex шаг за шагом и приведи 3 примера строк, которые под него подходят"_. Это буквально спасет вас при рефакторинге сложного легаси-кода.

## 📊 Доказательство: До и После

### ❌ До (Обычный запрос)

```text
Напиши мне regex для номеров телефонов США. Дефисы, скобки или ничего.
```

_(Результат обычно содержит сырой паттерн, в котором сложно разобраться, и в нем отсутствуют тесты для граничных условий)._

### ✅ После (Результат работы Pro-промпта)

```javascript
// Целевой язык: JavaScript

// 1. Регулярное выражение
const phoneRegex = /^(\+?1\s?)?(\(\d{3}\)|\d{3})[- \.]?\d{3}[- \.]?\d{4}$/;

// 2. Тестовые сценарии (Unit Tests)
const testCases = [
  { phone: "123-456-7890", expected: true, desc: "Стандартный с дефисами" },
  { phone: "(123) 456-7890", expected: true, desc: "Код в скобках" },
  { phone: "1234567890", expected: true, desc: "Только цифры" },
  { phone: "+1 (123) 456-7890", expected: true, desc: "С кодом страны" },
  { phone: "123-45-67890", expected: false, desc: "Неверный формат" },
  { phone: "123abc4567", expected: false, desc: "Содержит буквы" },
];

testCases.forEach(({ phone, expected, desc }) => {
  const result = phoneRegex.test(phone);
  console.log(`[${result === expected ? "PASS" : "FAIL"}] ${desc}: ${phone}`);
});
```

### 🔍 Подробное объяснение (Интерпретация)

- `^` — Начало строки.
- `(\+?1\s?)?` — Опциональный код страны (+1), за которым может следовать пробел.
- `(\(\d{3}\)|\d{3})` — 3 цифры регионального кода, которые могут быть в скобках `(123)` или без них `123`.
- `[- \.]?` — Опциональный разделитель (дефис, пробел или точка).
- `\d{3}` — 3 цифры префикса.
- `[- \.]?` — Опциональный разделитель.
- `\d{4}` — 4 последние цифры номера.
- `$` — Конец строки.
