## 📝 더러운 코드, 클린 코드로 1초 만에 리팩토링하기

- **🎯 추천 대상:** 주니어 개발자, 코드 리뷰를 앞둔 모든 프로그래머
- **⏱️ 소요 시간:** 1시간 → 1분 단축
- **🤖 최상위 성능:** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro (코딩 특화 모델 권장)

- ⭐ **난이도:** ⭐⭐☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"나도 내가 짠 코드를 못 알아보겠어요... 이 스파게티 코드를 도대체 어떻게 풀죠?"_

기능 구현에 쫓겨 급하게 코드를 짜다 보면, 어느새 변수명은 의미를 잃고 함수 하나가 100줄을 거뜬히 넘어가는 아찔한 경험, 개발자라면 누구나 한 번쯤 겪어보셨을 겁니다. 배포 일정은 턱밑까지 다가왔고, 머릿속은 오직 '일단 버그 없이 화면에 렌더링되게 만들자'는 생각뿐입니다. 그렇게 정신없이 키보드를 두드리다 보면, 화면은 정상적으로 동작하지만 코드 베이스는 처참한 몰골을 하고 있습니다. 동료가 남긴 **코드 리뷰(PR)** 코멘트를 확인하기가 두려우신가요? 알림이 울릴 때마다 혹시 내 코드를 보고 "이게 대체 무슨 로직인가요?"라는 뼈아픈 지적이 달렸을까 봐 심장이 덜컥 내려앉는 기분, 결코 당신 혼자만의 고민이 아닙니다.

"일단 돌아가니까 나중에 리팩토링해야지" 하고 넘어갔던 코드들은 결코 스스로 정화되지 않습니다. 시간이 지날수록 눈덩이처럼 불어나 거대한 **기술 부채(Technical Debt)**라는 괴물로 돌아옵니다. 변수명 `temp`, `data`, `flag1` 같은 의미 없는 이름들이 코드 전반에 난무하고, 하나의 함수 안에서 데이터 패칭부터 UI 렌더링, 에러 핸들링까지 모든 것을 처리하는 거대한 **스파게티 코드**는 곧 유지보수의 지옥문을 활짝 엽니다. 새로운 기능을 하나 추가하려고 하면 어디서부터 손대야 할지 막막해집니다. 기존 버그를 하나 고치면 전혀 상관없어 보이던 다른 컴포넌트에서 세 개의 새로운 버그가 터지는 끔찍한 연쇄 작용이 일어납니다. 무엇보다 당신을 갉아먹는 것은 "나조차도 일주일만 지나면 내가 짠 코드를 전혀 이해하지 못한다"는 슬픈 현실입니다. 매번 코드를 읽기 위해 머릿속으로 수많은 변수의 상태를 트래킹해야 하는 인지적 과부하는 개발자의 퇴근을 늦추고 번아웃을 앞당기는 가장 큰 주범입니다.

그렇다면 이제 **'클린 코드(Clean Code)'의 정석**을 완벽하게 이해하고 당신의 코드를 무자비하게, 그러나 가장 안전하게 다듬어주는 전담 AI 비서를 고용할 때입니다. 단순히 들여쓰기를 맞춰주거나 린트(Lint) 에러만 잡아주는 수준의 얕은 교정이 아닙니다. 객체지향과 함수형 프로그래밍의 핵심 철학인 **SOLID 원칙**, 그중에서도 특히 **단일 책임 원칙(SRP)**과 **DRY(Don't Repeat Yourself, 반복 금지 원칙)**를 완벽하게 이해하고 적용하는 프롬프트입니다. 이 AI 비서는 복잡하게 얽혀 있는 당신의 핵심 비즈니스 로직은 단 한 줄도 훼손하지 않은 채, 코드의 뼈대와 구조만을 가장 현대적이고 우아하게 분리해 냅니다. 기계적인 번역이 아니라, 당신이 처음 이 코드를 짤 때 머릿속에 품었던 '진짜 의도'를 파악하여 그에 걸맞은 가장 직관적인 이름을 변수와 함수에 부여합니다. 

이 프롬프트 하나면 **압도적인 가독성, 최적화된 성능, 그리고 미래지향적인 유지보수성**까지 단번에 확보할 수 있습니다. 어떤 깐깐한 시니어 리뷰어 앞에서도 당당하게 코드를 푸시(Push)할 수 있는 무적의 방패가 생기는 것입니다. 기존 같았으면 며칠 밤낮을 머리 싸매고 고민해야 했던 레거시 코드 파악과 대대적인 리팩토링 작업이, 이제는 복사하고 붙여넣는 단 1분의 시간으로 단축됩니다. 코드 리뷰 창에서 수많은 지적 대신 "코드 진짜 깔끔하고 우아하게 잘 짰네요. 바로 머지(Merge)하겠습니다"라는 찬사를 받는 짜릿한 쾌감을 상상해 보세요. 버그 발생에 대한 막연한 불안감을 완벽하게 떨쳐내고, 팀 내에서 '가장 믿을 수 있는 코드를 작성하는 개발자'로 인정받으며 당당하게 칼퇴근하는 놀라운 변화의 시작, 지금 당장 아래의 프롬프트로 직접 경험해 보시길 바랍니다.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 기능 구현에만 집중하여 변수명이 불명확하고, 할인 로직과 총액 합산 로직이 하나의 함수에 혼재된 전형적인 레거시 스파게티 코드입니다.

```javascript
> function calc(d) {
>   let t = 0;
>   for (let i = 0; i < d.length; i++) {
>     if (d[i].price > 100) {
>       t += d[i].price * 0.9;
>     } else {
>       t += d[i].price;
>     }
>   }
>   return t;
> }
```

### ✅ After (완벽하게 변신한 결과)



```javascript
/**
 * 장바구니 총액을 계산합니다.
 * 100원 초과 상품은 10% 할인을 적용합니다.
 */
function calculateTotalPrice(items) {
  const DISCOUNT_THRESHOLD = 100;
  const DISCOUNT_RATE = 0.9;

  return items.reduce((total, item) => {
    const isDiscountApplicable = item.price > DISCOUNT_THRESHOLD;
    const finalPrice = isDiscountApplicable
      ? item.price * DISCOUNT_RATE
      : item.price;

    return total + finalPrice;
  }, 0);
}
```

_(AI 요약: 함수명 `calc`를 의도가 명확한 `calculateTotalPrice`로 변경하고, 매직 넘버(100, 0.9)를 직관적인 상수로 분리하여 가독성을 극대화했습니다. 또한 기존의 `for` 루프 대신 모던 JS 문법인 `reduce` 배열 메서드를 활용하여 불필요한 상태 변이(`let t`)를 완전히 제거했습니다.)_

## ⚡️ 3줄 요약 (TL;DR)

1. 직관적이지 않은 변수명과 함수명을 **개발자의 본래 의도에 맞게 자동 변경**합니다.
2. **단일 책임 원칙(SRP)**에 따라 비대해진 거대 함수를 작게 쪼개고, 불필요한 중복 코드를 완벽히 제거합니다.
3. 해당 프로그래밍 언어의 **가장 최신 모던 문법과 컨벤션**을 적용하여 코드를 세련되게 다듬습니다.

코드 리뷰를 단번에 프리패스(Free-pass)해야 하는 **상용 제품(Production) 레벨의 디테일과 퀄리티**가 필요할 때 사용하세요.

> **역할 (Role):** 너는 20년 차 시니어 소프트웨어 엔지니어이자 '클린 코드(Clean Code)' 전문가야.
>
> **상황 (Context):**
>
> - 배경: 기능은 정상적으로 작동하지만, 가독성이 심각하게 떨어지고 유지보수가 불가능에 가까운 스파게티 코드 상태야. 변수명이 명확하지 않고, 중복된 로직이 곳곳에 섞여 있어.
> - 목표: 동료들이 읽기 쉽고, 추후 확장성이 뛰어나며, 까다로운 코드 리뷰도 단번에 통과할 수 있는 우아한 코드로 전면 개선하는 것.
>
> **요청 (Task):**
>
> 1. **네이밍 규칙 적용:** 변수와 함수 이름만 봐도 그 역할을 명확히 유추할 수 있도록 직관적으로 변경해 줘.
> 2. **함수 분리 (SRP 준수):** 하나의 함수가 단 하나의 역할만 수행하도록(단일 책임 원칙) 작게 분리해 줘.
> 3. **중복 코드 제거 (DRY 규칙):** 반복되는 로직은 별도의 유틸리티나 함수로 안전하게 추출해 줘.
> 4. **의미 있는 주석 추가:** 코드가 '어떻게' 동작하는지가 아니라, **'왜(Why)' 그렇게 작성되었는지** 비즈니스적 맥락 설명이 필요한 부분에만 핵심 주석을 달아 줘.
> 5. **모던 문법 적용:** 해당 프로그래밍 언어의 최신 모던 문법과 팀 컨벤션을 철저히 적용해 줘.
>
> **입력 코드:**
>
> `[여기에 리팩토링할 코드를 붙여넣으세요]`
>
> **제약사항 (Constraints):**
>
> - 기존 코드의 비즈니스 로직과 입출력 결과(동작)는 **절대 변경하지 마.**
> - 리팩토링 전과 후의 주요 변경 사항과 그 근거를 **마크다운 목록(List)**으로 명확히 요약해서 설명해 줘.
> - 출력은 반드시 '변경 사항 설명 블록'과 '마크다운 코드 블록'으로 철저히 분리해서 제공해 줘.
>
> **주의사항 (Warning):**
>
> - 불필요한 외부 라이브러리를 추가하거나 오버 엔지니어링을 철저히 피하고, 원본 코드의 초기 의도를 최대한 존중해 줘. 혹시라도 모르는 정보가 있다면 지어내지 말고 모른다고 답해.

## 💡 작성자 코멘트 (Insight & How to use)

이 프롬프트는 특히 이전 담당자가 남겨두고 떠난, **설계가 완전히 무너진 레거시 코드를 인수인계받았을 때** 그 어떤 도구보다 압도적인 위력을 발휘합니다. 대부분의 주니어 개발자들은 코드를 깨끗하게 만든다고 하면 단순히 변수명을 조금 바꾸거나 줄바꿈을 맞추고, 코드의 절대적인 길이를 짧게 줄이는 '요약 작업' 정도로 오해하곤 합니다. 하지만 진정한 클린 코드는 코드의 길이가 아니라 **의존성(Dependency)의 분리**와 **책임의 할당**에 있습니다. 이 프롬프트는 단순히 텍스트를 다듬는 것이 아니라, **SRP(단일 책임 원칙)와 DRY(반복 금지 원칙)**라는 객체지향의 절대적인 진리를 AI에게 명시적으로 강제합니다. 그 결과, 사람이 수십 분 동안 머리를 쥐어짜며 직접 분리해 낸 것보다 훨씬 더 신뢰할 수 있고 논리적으로 완벽하게 구조화된 결과물을 1초 만에 얻어낼 수 있습니다. 

저의 실제 실무 경험을 하나 덧붙이자면, 이 프롬프트를 있는 그대로 사용하는 것도 좋지만, **자신의 팀 환경에 맞게 변수(Context)를 적극적으로 통제하고 튜닝**할 때 진짜 마법이 일어납니다. 프롬프트를 AI에게 실행하기 직전에 현재 여러분의 프로젝트에서 강제하고 있는 **'Lint 설정' (예: Airbnb ESLint 규칙, Prettier 포맷팅)이나 '고유한 팀 컨벤션' (예: 모든 변수는 엄격한 카멜 케이스 엄수, 불변성을 보장하는 함수형 프로그래밍 지향, 특정 라이브러리 네이밍 룰)**을 `[상황 (Context)]` 영역에 텍스트로 추가해서 넣어보세요. 이렇게 구체적인 제약 조건을 부여하면, AI가 내놓은 코드를 에디터로 가져와서 팀 규칙에 맞게 다시 한번 린트(Lint) 에러를 수정해야 하는 번거로운 2차 가공 작업조차 전혀 필요 없어집니다. 생성된 즉시 IDE에 복사해서 붙여넣고 바로 PR(Pull Request)을 올려도 무방한 **완벽한 수준의 프로덕션 레디(Production-ready) 코드**를 가장 쉽고 빠르게 얻을 수 있는 비결입니다. 

또한 리팩토링 과정에서 AI가 기존 비즈니스 로직을 몰래 훼손하는 치명적인 **할루시네이션(Hallucination)**을 원천적으로 방지하기 위해, 우리는 프롬프트 하단에 `제약사항 (Constraints)`과 `주의사항 (Warning)`을 통해 "비즈니스 로직은 절대 건드리지 마"라는 철칙을 매우 강력하게 박아두었습니다. 이 문장이 있음으로써 AI는 코드의 동작 원리를 섣불리 재해석하는 위험한 모험을 피하고, 오직 '구조적 아름다움'을 달성하는 데에만 모든 연산력을 집중하게 됩니다. 만약 결과물이 예상보다 너무 잘게 쪼개져서 오히려 눈에 잘 들어오지 않는 '오버 엔지니어링' 현상이 발생했다면, AI에게 _"함수 분리를 조금 덜 엄격하게 하고, 파일 하나 안에서 응집도 있게 유지해 줘"_ 라고 가벼운 추가 프롬프트를 날려보세요. 순식간에 난이도가 조절된 맞춤형 코드를 반환할 것입니다. 개발자에게 리팩토링은 더 이상 밤을 새워야 하는 두려운 숙제가 아니라, 똑똑한 AI 동료와 함께하는 즐겁고 생산적인 대화의 시간으로 완전히 탈바꿈하게 될 것입니다. 

마지막으로 팁을 하나 더 드리자면, 거대한 스파게티 코드를 한 번에 프롬프트 창에 쏟아붓는 것보다, 논리적인 도메인 단위로 쪼개어 단계적으로 리팩토링을 요청하는 것이 훨씬 더 높은 품질의 결과를 보장합니다. 예를 들어 1,000줄짜리 파일 하나를 통째로 넣으면 AI가 문맥을 잃어버릴 수 있지만, '사용자 인증(Auth) 로직' 200줄과 '장바구니 결제(Payment) 로직' 300줄을 나누어서 각각 프롬프트에 입력하면 AI는 각 도메인의 핵심 역할에 완벽하게 몰입합니다. 그리고 리팩토링이 끝난 후에는 반드시 AI가 작성해 준 '변경 사항 요약 블록'을 꼼꼼히 읽어보세요. 그 요약본을 그대로 복사해서 깃허브(GitHub) PR 상세 설명 란에 붙여넣기만 해도, 동료 리뷰어들에게 당신이 얼마나 구조적인 고민을 깊게 했는지 완벽하게 어필할 수 있는 훌륭한 문서가 완성됩니다. 이처럼 AI 프롬프트를 단순한 코드 수정 도구가 아닌, 팀 내 협업과 커뮤니케이션의 퀄리티를 한 단계 끌어올리는 전략적 무기로 활용해 보시길 강력히 권장합니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 리팩토링하다가 기존의 비즈니스 로직을 망가뜨릴까 봐 걱정됩니다.**
  - A: 프롬프트 제약사항에 "비즈니스 로직 절대 변경 금지"를 매우 강하게 명시해 두었으나, 생성형 AI의 특성상 환각(Hallucination) 현상이 0%라고 장담할 수는 없습니다. 리팩토링 전후로 반드시 **유닛 테스트(Unit Test)**를 실행하여 동작의 무결성을 스스로 교차 검증하는 것을 강력히 권장합니다.

- **Q: 특정 프레임워크(React, Spring Boot 등) 환경에 맞춰서 리팩토링을 지시할 수도 있나요?**
  - A: 물론 가능합니다. 프롬프트의 상황(Context) 부분에 _"현재 React 18과 TypeScript를 사용 중이며, Custom Hook 패턴을 적극적으로 적용해서 분리해 줘"_ 와 같이 현재의 구체적인 개발 환경과 원하는 아키텍처 패턴을 덧붙여 보세요. 일반적인 코드보다 훨씬 전문적이고 프로젝트에 밀착된 완성도 높은 결과물을 얻을 수 있습니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **Role(페르소나) 부여의 마법:** 단순한 코딩 어시스턴트가 아닌 **'20년 차 시니어 엔지니어'이자 '클린 코드 전문가'**라는 강력한 페르소나를 씌웠습니다. 이를 통해 AI는 단순한 코드 단축을 넘어, 소프트웨어 아키텍처 관점에서의 깊이 있는 리팩토링을 유도하게 됩니다.
2. **구체적인 평가 기준 (Task):** 네이밍, 함수 분리, 중복 제거, 주석 작성, 최신 문법이라는 **5가지 명확한 평가 기준**을 세웠습니다. AI가 자의적으로 코드를 수정하여 엉뚱한 방향으로 결과물이 튀는 것을 원천 차단하는 핵심 설계입니다.
3. **제약사항(Constraints)의 강력한 통제력:** 기능 변경을 엄격히 금지하고 불필요한 오버 엔지니어링을 경고했습니다. 그 결과, 학술적으로만 멋진 코드가 아니라 **실무에 즉시 투입 가능한 안전하고 현실적인 코드**를 출력하도록 강제합니다.
