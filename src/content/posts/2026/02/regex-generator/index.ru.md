### 🥇 Pro Version (Профессиональная версия)

Используйте этот промпт для сложных паттернов, когда требуется гарантия качества, подробная документация и готовые тесты для интеграции в проект.

> **Роль (Role):** Ты `[Senior Developer / QA Engineer]`, эксперт мирового уровня по регулярным выражениям (Regex).
>
> **Ситуация (Context):**
>
> - Контекст: Мне нужно регулярное выражение для использования в `[Python / JavaScript / Java]`.
> - Цель: Создать надежный, оптимизированный и безопасный (защита от ReDoS) Regex для валидации данных.
>
> **Запрос (Task):**
>
> 1. Напиши регулярное выражение, строго удовлетворяющее требованиям ниже.
> 2. Предоставь построчное объяснение (Интерпретацию) каждого символа и группы в Regex, чтобы я (и моя команда) мог легко поддерживать код.
> 3. Напиши готовый код для тестирования этого Regex, включающий как минимум 3 совпадающих (Positive) и 3 несовпадающих (Negative) случая.
>
> **Требования к паттерну (Constraints & Variables):**
>
> - Формат: `[номера телефонов США]`
> - Допустимые вариации: `[123-456-7890 или (123) 456-7890 или 1234567890]`
> - Опционально: `[Код города может отсутствовать]`
>
> **Внимание (Warning):**
>
> - Убедись, что Regex не подвержен катастрофическому возврату (Catastrophic Backtracking). Если условие слишком сложное, предложи разбить его на два этапа проверки.

## 💡 Комментарий автора (Insight)

Регулярные выражения — классический пример задачи, которую ИИ решает в разы эффективнее человека. Однако здесь кроется ловушка: сгенерированный Regex может выглядеть рабочим с первого взгляда, но пропускать крайние случаи (edge cases) или быть уязвимым к атакам (ReDoS).

Именно поэтому в **Pro Version** добавлен критически важный пункт: требование тестовых сценариев (Positive/Negative cases) и защиты от катастрофического возврата. Мой опыт показывает, что внедрение ИИ-генерации Regex вместе с тестами экономит до 90% времени при парсинге логов или валидации форм, полностью исключая человеческий фактор при чтении синтаксиса.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если сгенерированный Regex работает неправильно на некоторых примерах?**
  - A: Просто отправьте ИИ строку, которая не прошла валидацию (или наоборот, прошла ошибочно), со словами: _"Этот ложный пример `[строка]` проходит проверку, хотя не должен. Скорректируй Regex"_. Модель моментально обновит паттерн.

- **Q: Зависит ли синтаксис Regex от языка программирования?**
  - A: Да, существуют различия (например, PCRE, стандарты POSIX, особенности движков JS или Python). Поэтому в промпте мы всегда явно указываем целевой язык `[Python / JavaScript]`, чтобы ИИ подобрал правильные экранирования и поддерживаемые функции (например, Lookbehind).

- **Q: Можно ли попросить ИИ сделать обратное — расшифровать непонятный Regex из старого проекта?**
  - A: Абсолютно! Отправьте ИИ чужой Regex с запросом: _"Объясни мне этот Regex шаг за шагом и приведи 3 примера строк, которые под него подходят"_. Это спасет вас при рефакторинге сложного легаси-кода.

## 📊 Доказательство: До и После

### ❌ До (Обычный запрос)

```text
Напиши мне regex для номеров телефонов США. Дефисы, скобки или ничего.
```

_(Результат обычно содержит сырой паттерн, в котором сложно разобраться, и в нем отсутствуют тесты для граничных условий)._


### ✅ После (Результат работы Pro-промпта)

```javascript
// Целевой язык: JavaScript

// 1. Регулярное выражение
const phoneRegex = /^(\+?1\s?)?(\(\d{3}\)|\d{3})[- \.]?\d{3}[- \.]?\d{4}$/;

// 2. Тестовые сценарии (Unit Tests)
const testCases = [
  { phone: "123-456-7890", expected: true, desc: "Стандартный с дефисами" },
  { phone: "(123) 456-7890", expected: true, desc: "Код в скобках" },
  { phone: "1234567890", expected: true, desc: "Только цифры" },
  { phone: "+1 (123) 456-7890", expected: true, desc: "С кодом страны" },
  { phone: "123-45-67890", expected: false, desc: "Неверный формат" },
  { phone: "123abc4567", expected: false, desc: "Содержит буквы" },
];

testCases.forEach(({ phone, expected, desc }) => {
  const result = phoneRegex.test(phone);
  console.log(`[${result === expected ? "PASS" : "FAIL"}] ${desc}: ${phone}`);
});
```

### 🔍 Подробное объяснение (Интерпретация)

- `^` — Начало строки.
- `(\+?1\s?)?` — Опциональный код страны (+1), за которым может следовать пробел.
- `(\(\d{3}\)|\d{3})` — 3 цифры кода зоны, которые могут быть в скобках `(123)` или без них `123`.
- `[- \.]?` — Опциональный разделитель (дефис, пробел или точка).
- `\d{3}` — 3 цифры префикса.
- `[- \.]?` — Опциональный разделитель.
- `\d{4}` — 4 последние цифры номера.
- `$` — Конец строки.
