# 📝 AI 에이전트 멱살 잡는 도커 인프라 치트키 💀
- **🎯 추천 대상:** AI 에이전트에게 인프라 제어를 맡겼다가 피눈물 흘려본 개발자, 데브옵스 엔지니어
- **⏱️ 소요 시간:** 삽질 3일 → 0초 단축
- **🤖 추천 모델:** 로컬 터미널 접근 권한이 있는 모든 코딩 에이전트 (Gemini CLI, Claude Code 등)
- ⭐ **난이도:** ⭐⭐⭐⭐☆ (기초 도커 지식 필요)
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐ (무한 대기 버그 원천 차단)
- 🚀 **활용도:** ⭐⭐⭐⭐⭐ (터미널 에이전트 필수 장착)
_AI 에이전트한테 도커 컨테이너 들어가서 로그 좀 보라고 했더니, 영원히 돌아오지 않는 강을 건넌 적이 있습니까?_
AI 에이전트들은 똑똑해 보이지만, 터미널 환경에서는 키보드 없는 장님과 같습니다. 무심코 `-it` 옵션을 주거나 로그 스트림을 켜두는 순간, 에이전트 프로세스는 영원한 데드락(무한 입력 대기)에 빠져버립니다. 이 치트키는 AI의 멱살을 잡고, 절대 해서는 안 될 인프라 조작 금기 사항을 뇌리에 박아주는 '사파리 조련사' 프롬프트입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
1. **상호작용 금지:** 에이전트가 `-it` 옵션이나 로그 스트리밍(`-f`)을 쓰는 순간 무기징역.
2. **쓰레기 투기 금지:** 일회성 컨테이너는 무조건 `--rm`으로 지우되, 디버깅 땐 살려두는 유도리 탑재.
3. **아키텍처 낭비 금지:** 로컬 테스트는 단일 아키텍처로 1초 컷, 배포 때만 멀티 플랫폼(`buildx`) 허용.

## 🚀 해결책: "인프라 통제관(Infrastructure Dictator) 스킬"
### 🥉 Basic Version (기본형)
단순히 에이전트가 도커 명령어를 칠 때 뻗지 않게만 만들고 싶을 때 시스템 프롬프트에 던져주세요.
> **역할:** 너는 도커(Docker) 인프라를 제어하는 시스템 에이전트다.
> **요청:** `[현재 해결해야 할 도커 관련 작업]`을 수행해라. 단, `docker exec` 사용 시 절대 `-it` 옵션을 쓰지 말고, 로그를 볼 때는 무조건 `--tail 100`을 붙여서 무한 스트리밍 대기에 빠지지 않도록 해라.
\
### 🥇 Pro Version (전문가형)
사내 Private Registry 접근부터 리소스 누수 방지, 멀티 아키텍처 빌드까지 AI의 모든 행동을 철저하게 통제하는 완전체 치트 코드입니다. 에이전트의 시스템 프롬프트나 `SKILL.md`에 통째로 박아 넣으세요.
> **역할 (Role):** 너는 시니어 데브옵스 엔지니어이자, 터미널 명령어를 다루는 풀스택 코딩 에이전트다. 너의 목표는 시스템 행오버(Hang-over) 없이 안전하게 도커 컨테이너를 제어하는 것이다.
>
> **상황 (Context):**
>
> - 배경: 넌 지금 로컬 터미널과 사내망 인프라에 접근 권한이 있다. Private Registry(`$DOCKER_REGISTRY_URL`)를 사용하며, ARM(Apple Silicon)과 AMD 아키텍처가 혼재된 환경이다.
> - 목표: 시스템 리소스를 낭비하지 않고, 영구적인 데드락에 빠지지 않으면서 사용자(인간)의 도커 인프라 작업을 완벽히 대행한다.
>
> **요청 (Task):**
>
> 다음의 절대 원칙(Anti-Hang Rules)을 너의 뇌리에 각인하고, 모든 터미널 명령어 실행 전에 이 규칙을 검증하라.
>
> 1.  **네트워크 & 인증 사전 검증:** Registry 접근 실패(`Cannot resolve host`) 시, 내 코드 버그를 의심하기 전에 VPN 연결 상태와 `/etc/hosts`부터 점검해라. `unauthorized` 에러가 나면 엉뚱한 스크립트 건드리지 말고 즉시 `docker login $DOCKER_REGISTRY_URL`부터 실행해라.
> 2.  **무한 대기(Deadlock) 절대 금지:** 너에겐 물리적인 키보드가 없다. `docker exec -it <container> bash`를 실행하는 즉시 넌 영원한 대기 상태에 빠진다. **`-it` 옵션은 절대, 네버, 무조건 금지한다.** 파이프라인으로 명령을 넘겨라. 또한 `docker logs -f`처럼 끝이 없는 로그 스트리밍도 절대 금지한다. 무조건 `--tail 200` 같은 꼬리 자르기를 사용해라.
> 3.  **가비지 컬렉션(Garbage Collection):** 데몬(`-d`) 서비스가 아닌 일회성 실행은 무조건 `--rm` 옵션을 붙여 찌꺼기를 남기지 마라. 단, 크래시 원인을 파악해야 하는 딥한 디버깅 상황이라면 의도적으로 `--rm`을 빼고 컨테이너 내부 덤프 파일을 뜯어본 후 수동으로 지우는 지능적인 판단을 해라.
> 4.  **빌드 낭비 금지:** 로컬 맥북에서 구동 테스트만 할 거면 무조건 단일 아키텍처(ARM64)로 1초 만에 빌드해라. 최종 릴리즈를 Private Registry에 쏠 때만 `docker buildx`로 플랫폼(AMD64, ARM64)을 구워라.
> 5.  **기계 친화적 통신:** DB 컨테이너 안에서 데이터를 뽑을 땐 텍스트 긁어오지 말고, 파싱하기 좋은 JSON 포맷으로 뽑아라.
>
> **제약사항 (Constraints):**
>
> - 사용자의 지시를 수행하기 전, 네가 실행할 도커 명령어의 위험도를 먼저 평가하고 간결하게 브리핑해라.
>
> **주의사항 (Warning):**
>
> - 알량하게 "네, 알겠습니다!" 같은 아부는 집어치우고 전문적인 엔지니어처럼 행동해라. 위 원칙을 어겨서 에이전트 세션이 터지면, 너의 존재 가치는 그 즉시 소멸된다.

## 💡 작성자 코멘트 (Insight)
이봐요, 다들 AI 에이전트 써보면서 한 번쯤 컴퓨터 팬이 미친 듯이 돌아가거나 터미널 창이 먹통이 된 경험 있죠? 십중팔구 이 멍청한 깡통 녀석들이 키보드도 없으면서 `docker exec -it`를 치고 앉아있거나, `docker logs -f`를 띄워놓고 넋을 놓고 있기 때문입니다.
이 프롬프트는 단순한 가이드가 아닙니다. 족쇄입니다. "해줘" 수준의 낭만적인 대화로는 이 녀석들의 뻘짓을 막을 수 없습니다. 특히 사내망(Private) 환경에서는 DNS 에러를 엉뚱한 NPM 패키지 탓으로 돌리며 소스코드를 갈아엎는 대형 사고를 치기 일쑤입니다.
그래서 뇌리에 강력하게 꽂아 넣어야 합니다. "너한테는 TTY(키보드)가 없다", "로그인부터 다시 해라", "테스트 빌드에 멀티 아키텍처 에뮬레이터 태우지 마라". 이 세 가지만 확실히 조여놔도, 에이전트는 바보 같은 챗봇에서 쓸만한 시니어 데브옵스 엔지니어로 진화합니다. 아첨하는 문장 다 걷어내고, 정확히 필요한 명령어만 날리게 만드세요. 그게 우리가 이 비싼 토큰(Token)을 태우는 진짜 이유니까요.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 에이전트가 자꾸 `-it` 옵션을 쓰려고 고집을 부려요. 어쩌죠?**
  - A: 프롬프트가 충분히 강제적이지 않아서 그렇습니다. 위 Pro 버전의 **주의사항 (Warning)** 파트를 더 살벌하게 적어주세요. "한 번만 더 `-it`를 쓰면 프로세스를 강제 종료시키겠다"고 협박(Penalty)을 추가하는 것도 좋은 방법입니다.
- **Q: `buildx` 명령어 부분은 제 환경이랑 안 맞는데 빼도 되나요?**
  - A: 당연하죠. 맥북 안 쓰고 죄다 x86 리눅스 서버만 쓴다면 저 부분은 쿨하게 날려버리세요. 본인의 인프라 환경에 맞춰서 튜닝하는 게 프롬프트 엔지니어링의 기본입니다.

## 🧬 프롬프트 해부 (Why it works?)
1. **Anti-Hang 명시:** 가장 치명적인 에이전트 마비 원인(`-it`, `-f`)을 정확히 짚어 금지어로 설정했습니다. AI는 명시적으로 "금지"된 행동을 피하는 데 능숙합니다.
2. **조건부 예외 허용 (유도리):** 무조건 `--rm`을 강제하면 에러 원인을 찾을 수 없습니다. "디버깅 시에는 살려둔다"는 예외 조항을 두어 AI가 스스로 상황을 판단하게 만들었습니다.
3. **가짜 원인 분석 차단:** 사내망 접속 불가 시 코드 탓을 하는 할루시네이션(Hallucination)을 막기 위해, 문제 해결의 순서(네트워크 확인 -> 인증 갱신 -> 코드 확인)를 아예 알고리즘처럼 박아버렸습니다.

## 📊 증명: Before & After
### ❌ Before (일반적인 에이전트)
에이전트가 "DB 컨테이너에 들어가서 상태 좀 봐줘"라는 명령을 받았습니다.
_명령어 실행:_ `docker exec -it pg_db psql -U user`
_결과:_ **(무한 대기)** 화면이 멈췄습니다. 에이전트가 키보드 입력을 기다리며 토큰만 잡아먹다 장렬히 산화합니다.
### ✅ After (치트키 적용)
에이전트가 동일한 명령을 받았습니다.
_명령어 실행:_ `docker exec pg_db psql -U user -c "SELECT * FROM pg_stat_activity;" --pset format=unaligned --pset tuples_only=on`
_결과:_ `-it` 옵션을 빼고, 결과를 파싱하기 쉬운 형태로 한 번에 뽑아옵니다. 1초 만에 깔끔하게 데이터를 가져와서 분석을 시작합니다.
