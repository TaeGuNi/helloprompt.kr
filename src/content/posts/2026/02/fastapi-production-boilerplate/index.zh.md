3. **API 文档全自动化：** 真正实现“代码即文档”，Swagger (OpenAPI) 接口文档实时同步，将前后端联调的沟通成本降至冰点。

## 🚀 解决方案："FastAPI 架构生成提示词"

### 🥉 Basic Version (基础版)

当你需要快速搭建并运行一个轻量级 API Mock 服务器时，请使用此版本。

> **角色：** 你是一名资深的 Python 后端专家。
> **请求：** 请使用 FastAPI 编写一段最基础的 `main.py` 代码。要求能够处理指向 `/items/{item_id}` 路径的 GET 请求，并支持接收可选的查询参数 `q`。请在代码注释中提供启动服务器所需的 `uvicorn` 执行命令。

### 🥇 Pro Version (专业版)

当你需要为真实的业务部署，从零构建可扩展的微服务架构时，请使用此专业版本。

> **角色 (Role)：** 你是一位主导高并发企业级服务设计的 Python 后端首席架构师。
>
> **背景 (Context)：**
>
> - 背景：我目前正准备构建一个微服务后端，专门用于提供全新的 AI 模型预测服务。
> - 目标：生成一套具有极高可维护性与扩展能力的 FastAPI 模板代码及标准目录结构。
>
> **任务 (Task)：**
>
> 1. **目录结构：** 首先，请输出一个层次分明的树状目录。必须明确分离路由层 (`app/api/v1/endpoints`)、配置层 (`app/core/config`)、数据模型层 (`app/schemas`) 以及核心业务逻辑层 (`app/services`)。
> 2. **依赖注入 (Dependency Injection)：** 请展示行业最佳实践，利用 FastAPI 的 `Depends` 机制来优雅地注入数据库会话 (`SessionLocal`)。
> 3. **Pydantic V2 语法规范：** 严格遵循 V2 最新版本标准，定义标准的请求体与响应体数据模型（例如 `UserCreate`、`UserResponse`）。
> 4. **容器化部署方案：** 编写一份专为云原生部署优化的 `Dockerfile`，并指定使用官方推荐的 `tiangolo/uvicorn-gunicorn-fastapi` 基础镜像。
>
> **约束条件 (Constraints)：**
>
> - 所有的函数定义与类声明必须 100% 覆盖 Python 类型提示 (Type Hint)。
> - 必须在所有核心业务逻辑代码中，添加详尽且清晰的 docstring 文档注释。
>
> **注意事项 (Warning)：**
>
> - 遇到超出知识储备的不确定信息，请坦诚回答“不知道”，绝不允许捏造任何虚假信息（严禁 AI 幻觉）。
> - 绝对禁止使用已被废弃的 Pydantic V1 语法（如 `@validator`），必须强制且严格地采用最新的 V2 语法（如 `@field_validator`）。

## 💡 作者洞察 (Insight)

全面引入 FastAPI 的核心价值，不仅在于它被誉为“极速的 Python Web 框架”，更在于它为 Python 生态带来了革命性的 **“类型安全 (Type Safety)”**。当你向 AI 传达代码生成需求时，只需附加一句“请严格使用 Type Hint”，所生成的代码质量便能瞬间脱胎换骨。尽管 Python 本质上是一门动态类型语言，但借助严谨的类型提示，你完全可以获得媲美静态编译型语言的工程稳定性。得益于这些类型注解，Pydantic 会在底层自动拦截并校验异常数据，同时“顺手”为你生成可交互的 Swagger API 文档。只要在项目启动初期（直接使用 Pro 版提示词）构建起这套坚如磐石的架构底座，后续你便能毫无后顾之忧地将所有精力倾注于核心业务逻辑的攻坚中。

## 🙋 常见问题 (FAQ)

- **Q: 如果与老牌的 Flask 或 Django 相比，FastAPI 的性能差异真的那么明显吗？**
  - A: 绝对明显。在现代 Web 服务的实际场景中，往往充斥着海量的 I/O 密集型任务（例如高频数据库查询、第三方 API 并发调用等），此时异步 (Asynchronous) 处理能力就成了决定生死的刚需。基于 Starlette 和 Uvicorn 引擎的 FastAPI，在处理此类高并发场景时，展现出了能够无情碾压传统同步框架的超高吞吐量 (Throughput)。

- **Q: 配合 FastAPI 通常会使用哪种 ORM 框架？它支持异步数据库操作吗？**
  - A: 业界绝对标杆 `SQLAlchemy`，自 2.0 版本起便已实现了对异步 (`asyncio`) 的原生完美支持。你只需要在发给 AI 的提示词中明确下达指令：“请使用 SQLAlchemy 2.0 异步会话”，就能顺理成章地构建出一个从头到尾纯正的非阻塞异步后端服务器。

- **Q: 为什么在提示词中一定要再三强调并特指 Pydantic V2？**
  - A: 因为 Pydantic V2 的底层核心引擎已经使用 Rust 语言进行了全面重写，其数据校验的执行速度比 V1 版本最高飙升了整整 50 倍！然而，由于各大 AI 模型（尤其是受限于知识截止日期设定的模型）通常仍保留着输出 V1 旧版语法的“肌肉记忆”，因此在提示词中通过强约束强制要求输出 V2 语法，是你在生产环境中避免遭遇致命运行时报错的最短捷径。

## 🧬 提示词解剖 (Why it works?)

1. **明确界定的架构分离：** 假如你仅仅是泛泛地抛出一句“帮我写个 FastAPI 代码”，AI 往往只会敷衍地甩给你一个把所有业务逻辑全坨在 `main.py` 里的入门级“玩具”。而我们通过强制其清晰划分 `api`、`core` 以及 `schemas` 等独立模块，瞬间将该架构在生产环境中的可维护性拉满。
2. **严苛的版本控制 (Pydantic V2 约束)：** 众所周知，Python 生态圈中最令人抓狂的痛点莫过于依赖库的向下兼容与版本迁移灾难。而在这份提示词中，我们直接利用约束条件 (Constraints)，在代码生成的源头阶段就将这种隐患彻底扼杀。
3. **着眼于生产部署的端到端 (End-to-End) 视野：** 我们刻意将生成云原生 `Dockerfile` 文件的诉求一并纳入了核心指令，这直接打破了代码仅仅只能在“本地机器上跑通”的狭隘局限，一气呵成地为你铺平了直通云端服务器自动化部署的流水线大道。
