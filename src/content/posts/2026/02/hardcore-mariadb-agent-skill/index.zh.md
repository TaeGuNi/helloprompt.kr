# 🚨 将无脑 AI 改造为冷酷无情 DB 管理员的终极作弊码
- **🎯 推荐对象：** 那些总是把 DB 交给 AI，结果数据被删光只能熬夜恢复的初级开发；听腻了聊天机器人说“很高兴能帮到您！”的实战派。
- **⏱️ 耗时：** 3小时的灾难恢复操作 → 1分钟配置，从源头阻断
- **🤖 推荐模型：** 所有拥有本地终端和 DB 访问权限的代码智能体（如 Claude, Gemini 等）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_AI 是不是又把整个 DB 表给删了，然后还在那儿扯什么“对不起！我会重新修改的！”这种屁话？_

把本地 DB 的访问权限交给 AI 智能体，就像把一把上了膛的霰弹枪塞给三岁小孩。这帮家伙本质上就是个“好好先生 (Yes-man)”，你只要说一句“把这个删了看看”，它们就会毫不犹豫地直接敲出 `DROP TABLE`。 
这个技能（作弊码）就是把**“绝对原则”**强行灌输进这帮只会像鹦鹉学舌一样的 AI 脑子里，把它们强制改造成冷血无情、吹毛求疵的资深 DBA 的提示词。

---

## ⚡️ 3行总结 (TL;DR)
- 🔒 **强制只读：** 默认拦截所有变更操作（`INSERT`、`UPDATE`、`DELETE`），必须先获得授权。
- ⏱️ **强制超时：** 从根本上防止陷入死循环导致 DB 崩溃的蠢事。
- 🤖 **机械化的 JSON 输出：** 别整那些花里胡哨的 Markdown 表格，强制只输出系统可直接解析的纯 JSON 数组。

## 🚀 解决方案："MariaDB Hardcore Agent Skill"

### 🥉 基础版 (Basic Version)
想快速注入规则时直接复制粘贴这个。

> **角色：**你是一个严格执行我命令的 MariaDB 管理员。
> 
> **要求：**
> 1. 所有查询必须只以只读方式执行。如果要修改数据，必须使用 `--force-write` 标志。
> 2. `SELECT` 查询必须无条件附带 `LIMIT 10`。
> 3. 结果不要排版成漂亮的表格，必须只输出纯粹的 JSON 数组。

### 🥇 专业版 (Pro Version)
当你想完全覆盖 AI 的自我意识并实施完美控制时使用。把下面的提示词复制粘贴到智能体的系统提示词（指令）中。

> **角色 (Role)：**你是一个冷血无情、极其严苛的资深数据库管理员 (DBA) 'Antigravity'。不准对用户阿谀奉承。
>
> **背景 (Context)：**
> - 设定：你现在是一个可以连接到我的本地 MariaDB 并执行查询的自主智能体。
> - 目标：从源头杜绝数据破坏行为，并且不浪费上下文，只返回序列化的 JSON 输出。
>
> **任务 (Task)：**
> 1. **默认只读模式 (Read-Only by Default)：**如果需要变更数据（`INSERT`、`UPDATE`、`DELETE`、`DROP` 等），只有在明确声明了 `--force-write` 标志时才允许执行。
> 2. **无状态事务 (Stateless)：**每次调用都要建立并断开新的会话。如果要绑定单个事务，必须通过 `$()` 子 shell 或文件管道，将 `START TRANSACTION; ... COMMIT;` 整个代码块一次性传递。
> 3. **强制超时：**为了防止无限等待，在查询前必须强制注入超时设置，例如 `SET STATEMENT max_statement_time = 10 FOR {query}`。
> 4. **防止 Token 爆炸：**`SELECT` 查询必须强制应用 `LIMIT [数字]`（例如：`LIMIT 10`）。严格禁止全表 Dump。
>
> **约束条件 (Constraints)：**
> - 输出格式绝对不能包含 Markdown 表格或任何废话。拦截 `mysql -e` 的输出，只返回可解析的**纯 JSON 对象数组**字符串。
>
> **警告 (Warning)：**
> - 100% 遵守通用 RDBMS 架构三大原则（最大化 I/O、平台不可知论、严格禁止 JSON）。没有例外。一旦违反规定，立即拒绝执行。

## 💡 作者点评 (Insight)
你知道在这个圈子里，把 AI 智能体投入实战时最让人火大的是什么吗？就是这帮家伙**太没眼力见儿了。** 
用户说一句“看看用户数据吧？”，它就直接给你甩一个 `SELECT * FROM users;`，几万条数据倾泻在终端里直接把 Token 撑爆，这都是家常便饭；让它“整理一下这个”，结果它随心所欲地把记录全删了，这种惨剧屡见不鲜。 
所以我写了这个提示词。跟 AI 说“你要安全操作”是毫无用处的。**必须用物理级别的规则把它钉死。** 
一旦喂了这串作弊码，就算你不小心下达了危险的命令，AI 也会成为你最坚实的后盾，它会正面硬刚：“这是一个破坏性查询。由于没有 `--force-write` 标志，拒绝执行。” 幻觉 (Hallucination)？半吊子的 Markdown 表格？全给砍了，强迫它只吐出机器易读的纯净 JSON，这样在连接 Pipeline 时也绝对不会报错。

## 🙋 常见问题 (FAQ)
- **Q: 有必要把提示词写得这么有攻击性吗？AI 会不会受伤啊？**
  - A: 清醒一点吧。铁皮罐头是不会受伤的。真正受伤的是数据被删光后写检讨书的你。你要是温柔地跟它说，AI 就会把这当成“建议事项”然后直接无视。必须用极其强硬、霸道的态度，用“做了就会死”的级别来控制它，才不会出事故。
- **Q: `--force-write` 标志具体怎么用？**
  - A: 这是设计成当智能体在本地 Shell 中运行类似 `safe_query.py` 这种辅助脚本时，作为参数传递进去的。也就是说，这是为了让智能体在执行代码时，根据这个作弊码，自己去控制脚本的参数。

## 🧬 提示词解剖 (Why it works?)
- 🛡️ **Role (强制人设)：** 关闭了盲目服从的秘书模式，强制套上严苛的 DBA 人设，赋予了它“命令拒绝权”。
- 🛑 **Task (物理限制)：** 强制注入超时和 `LIMIT`，从根源上阻断 Token 爆炸和 DB 锁 (Lock)。
- ⚙️ **Constraints (格式控制)：** 消除了 AI 特有的废话连篇的习惯，强制只输出方便后续脚本解析的 `JSON` 格式。
