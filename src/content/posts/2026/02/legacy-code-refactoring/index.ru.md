# 🧹 Очистка легаси-кода: Рефакторинг и автоматизация тестов

- **🎯 Рекомендуемая аудитория:** Junior-разработчики, специалисты по поддержке легаси-кода, ревьюеры кода
- **⏱️ Затрачиваемое время:** 1 час на анализ → сокращение до 5 минут
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o (Claude настоятельно рекомендуется для сложной логики и длинного контекста)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _"Кто вообще написал этот код? (...а, так это же был я год назад.)"_

Запутанные вложенные `if-else`, переменные с непонятными названиями и недокументированная обработка исключений. Знакомо ли вам чувство безысходности перед **легаси-кодом (Legacy Code)**, когда кажется, что система рухнет от малейшего прикосновения?

Больше не нужно бояться. Мы покажем, как нанять ИИ в качестве вашего личного **Senior-разработчика для парного программирования (Pair Programmer)**. Вы освоите идеальные техники промпт-инжиниринга, чтобы элегантно отрефакторить спагетти-код и мгновенно создать надежную защиту в виде автоматических тестов.

## ⚡️ Краткая выжимка (TL;DR)

1. **Сначала требуйте объяснений:** Не просите ИИ сразу менять код. Заставьте его сначала проанализировать поток выполнения и понять первоначальный замысел.
2. **Применяйте принципы Clean Code:** Безопасно рефакторьте структуру и нейминг, строго опираясь на принципы SOLID и читаемость.
3. **Стройте защиту с помощью тестов:** Автоматически генерируйте юнит-тесты (Unit Tests), включая пограничные случаи (Edge Cases), чтобы гарантировать идентичность работы кода до и после рефакторинга.

## 🚀 Решение: "Senior Dev Simulator" (Симулятор Senior-разработчика)

### 🥇 Pro Version (Профессиональная версия)

Если просто сказать "отрефактори это", высок риск сломать существующую функциональность. В этом промпте применяется метод **Chain-of-Thought (Цепочка рассуждений)**, заставляющий ИИ мыслить логическими шагами.

> **Роль (Role):** Ты `[Senior Backend-разработчик с 10-летним опытом]`, работавший в BigTech-компаниях (Google, Meta). Ты строгий приверженец философии чистого кода (Clean Code) и неукоснительно соблюдаешь принципы TDD (разработка через тестирование).
>
> **Контекст (Context):**
> Я только что получил в поддержку ужасный легаси-код, написанный на `[введите язык, например, Python/Java]`. У него отвратительная читаемость, и его практически невозможно поддерживать.
>
> **Задача (Task):**
> Выполни следующие шаги по порядку и максимально глубоко:
>
> **Шаг 1. Анализ и диагностика кода:** Опиши одним предложением, какую бизнес-задачу решает этот код, и укажи 3 критические проблемы (Code Smells).
> **Шаг 2. Безопасный рефакторинг:** Сохрани бизнес-логику и побочные эффекты (Side Effects) оригинала на 100%. Максимально улучши читаемость за счет разделения функций согласно принципу единственной ответственности (SRP) и переименования переменных/методов. Для сложной бизнес-логики обязательно добавь Docstring и комментарии.
> **Шаг 3. Защитная проверка (Тесты):** Напиши юнит-тесты (Unit Test), которые на 100% гарантируют, что твой отрефакторенный код работает абсолютно так же, как оригинал. Тесты должны покрывать не только успешные сценарии, но и как минимум 2 пограничных случая (Edge Cases).

## 💡 Комментарий автора (Insight)

Истинная ценность этого промпта заключается не просто в создании красивого кода, а в **предварительном выстраивании "сети безопасности" в виде тестов**. Я лично применил этот промпт на практике, когда мне передали на поддержку устаревший модуль платежей на Java, написанный 5 лет назад.

Что меня поразило больше всего — ИИ сам указал через тесты на скрытые пограничные случаи, которые я изначально упустил (например, ветвление логики, когда сумма платежа равна 0).

**🔥 Совет для применения на практике:**
Если вы скопируете класс целиком на сотни строк за один раз, ИИ может словить «галлюцинацию» (Hallucination) или потерять контекст. Разбивайте код и задавайте вопросы **на уровне методов (до 100 строк)**. Глубина анализа и точность тестов в таком случае возрастают многократно.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Какая ИИ-модель лучше всего подходит для работы с логикой кода?**
  - О: По состоянию на 2026 год лидерами в области кодинга и рефакторинга являются **Claude 3.5 Sonnet** и **GPT-4o**. При этом Claude 3.5 Sonnet демонстрирует небольшое преимущество в удержании сложного контекста и написании человекочитаемого чистого кода.

- **В: Безопасно ли передавать ИИ внутренний код компании?**
  - О: **Категорически нет.** API-ключи, внутренние IP-адреса, реальные данные клиентов или ключевые алгоритмы безопасности необходимо обязательно `[ЗАСЕКРЕТИТЬ]` (маскировать) или заменить на фиктивные данные (Dummy Data) перед отправкой. Либо используйте исключительно корпоративную среду Enterprise AI, настроенную в вашей компании.

- **В: Что делать, если после рефакторинга появляются ошибки?**
  - О: Скорее всего, ИИ упустил какую-то зависимость (Dependency). Скопируйте полный лог ошибки вместе с соответствующим контекстом оригинального кода и дайте четкий фидбек: _"В этой части возникла такая ошибка. Кажется, ты упустил [определенное условие] из оригинальной логики. Пожалуйста, исправь это."_

## 🧬 Разбор промпта (Why it works?)

1. **Назначение роли (Role):** Задавая строгую персону "Senior с 10-летним опытом, адепт чистого кода", мы поднимаем планку: ИИ генерирует не просто работающий код, а высококачественный код с отличной поддерживаемостью.
2. **Chain-of-Thought (Цепочка рассуждений):** Вместо слепой команды "исправь", мы принудительно задаем 3 логических шага: **[Анализ → Рефакторинг → Проверка]**. Это ключевой механизм, который заставляет ИИ улучшать структуру, не разрушая первоначальный замысел.
3. **Обязательные Edge Cases (Constraints):** Явное требование написать тесты для пограничных случаев гарантирует надежность кода, который можно без опасений применять в продакшене.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод: Кошмарный легаси-код)

```python
def calc_trx(a, t, f=False):
    # a: amount, t: type
    if t == 1:
        if f: return a * 1.1 + 500
        else: return a * 1.1
    elif t == 2:
        return a - (a * 0.05)
    else: return 0 # ошибка
```

### ✅ После (Результат: Чистый код и тесты от Senior-разработчика)

```python
from enum import Enum
import unittest

class TransactionType(Enum):
    DEPOSIT = 1
    WITHDRAWAL = 2

def calculate_transaction_amount(amount: float, transaction_type: TransactionType, apply_fee: bool = False) -> float:
    """
    Рассчитывает итоговую сумму с учетом налогов и комиссий в зависимости от типа транзакции (внесение/снятие).
    """
    if amount < 0:
        raise ValueError("Сумма транзакции должна быть больше 0.")

    if transaction_type == TransactionType.DEPOSIT:
        taxed_amount = amount * 1.1
        return taxed_amount + 500 if apply_fee else taxed_amount

    elif transaction_type == TransactionType.WITHDRAWAL:
        return amount * 0.95

    return 0.0

# ----------------- Unit Tests ----------------- #
class TestCalculateTransaction(unittest.TestCase):
    def test_deposit_with_fee(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.DEPOSIT, True), 1600.0)

    def test_withdrawal_amount(self):
        self.assertEqual(calculate_transaction_amount(1000, TransactionType.WITHDRAWAL), 950.0)

    def test_negative_amount_raises_error(self):
        with self.assertRaises(ValueError):
            calculate_transaction_amount(-500, TransactionType.DEPOSIT)
```
