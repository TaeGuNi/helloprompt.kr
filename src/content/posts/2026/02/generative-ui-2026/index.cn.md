> _"如果界面能够根据用户的真实意图自动生成，我们为什么还要手动绘制每一个静态页面的导航菜单呢？"_

几十年来，前端开发一直遵循着一个固定的契约：开发者设计界面，用户在其中导航。我们构建导航栏、汉堡菜单和静态仪表盘，期望我们的信息架构能与用户的思维模型相匹配。但生成式 UI (Generative UI) 彻底颠覆了这一点——它将 UI 从“预先渲染的静态产物”转变为“根据对话或任务上下文动态生成的交互组件”。

本文将提供一套强大的提示词，帮助你快速设计和实现生成式 UI 的组件结构与交互逻辑。

---

## ⚡️ 核心摘要 (TL;DR)

1. **打破静态导航：** 界面不再依赖繁琐的菜单栏，而是根据用户输入（如“查看2月份支出”）直接生成可视化组件（如交互式图表）。
2. **意图识别与组件库的结合：** AI 模型负责解析用户意图，并从预设的“组件注册表”中调用安全的、强类型的 UI 组件。
3. **流式渲染 (Streaming Hydration)：** 将 UI 定义与文本一起流式传输，让界面瞬间“弹”出，带来极致的响应体验。

## 🚀 解决方案："生成式 UI 架构提示词"

### 🥉 基础版 (Basic Version)

当你只需要快速构思一个生成式 UI 组件的 JSON 结构时使用。

> **角色：** 你是一位资深的前端架构师和 UX 设计师。
> **任务：** 请根据用户意图 `[例如：查看上个月的信用卡支出情况]`，设计一个生成式 UI 组件的 JSON 数据结构，并提供对应的 React/Vue 组件伪代码。确保它是一个可以即时渲染的交互式微件，而不是静态页面。

<br>

### 🥇 专业版 (Pro Version)

当你需要构建生产级别的生成式 UI 系统，并且需要严格的组件调用规范时使用。

> **角色 (Role)：** 你是一位顶级的生成式 UI (Generative UI) 架构师。你需要将用户的自然语言意图转化为结构化的 UI 组件负载 (JSON)。
>
> **背景 (Context)：**
>
> - 背景：我们正在开发一款支持 AI 对话界面的应用。传统的 GUI 导航太繁琐，我们需要在聊天流中直接渲染交互式组件。
> - 目标：根据用户的输入，精准匹配预设的组件库，并生成对应的 JSON 结构以供前端流式渲染 (Streaming Hydration)。
>
> **任务 (Task)：**
>
> 1. 分析用户意图：`[输入用户的自然语言请求，例如：帮我对比一下 Q1 和 Q2 的营销转化率]`。
> 2. 从以下可用的组件库中选择最合适的组件：`[输入组件列表，例如：BarChart, LineChart, SummaryCard, DataTable]`。
> 3. 生成严格符合该组件 Props 接口的 JSON 数据。
> 4. 提供一段简短的前端实现思路，说明如何将此 JSON 转化为 React 组件。
>
> **约束条件 (Constraints)：**
>
> - 必须只输出合法的、可解析的 JSON 格式数据。
> - 绝对不能凭空捏造（幻觉）未在组件库中定义的 HTML 或组件。
> - JSON 必须包含 `componentType` 和 `props` 两个核心字段。
>
> **警告 (Warning)：**
>
> - 如果用户的意图超出了预设组件库的处理能力，请返回 `{"error": "Unsupported intent", "fallbackText": "对不起，我暂时无法生成该图表。"}`，切勿强行生成。

## 💡 作者深度解析 (Insight)

生成式 UI 不仅仅是一个花哨的趋势，它是声明式 UI 框架与概率性 AI 结合的必然结果。在实战中，这个提示词的威力在于它**限制了 AI 的自由度**。

让大语言模型直接输出 HTML/CSS 往往会导致不可控的样式崩溃或安全漏洞（XSS）。通过强制 AI 输出预定义的 `componentType` 和强类型的 `props` JSON，我们将 AI 从“像素搬运工”转变成了“系统调用者”。这样一来，前端工程师仍然可以掌控组件的具体外观和交互逻辑，而 AI 则负责智能的路由和数据组装。这种模式极大地提升了系统的稳定性和可维护性。

## 🙋 常见问题解答 (FAQ)

- **Q: 这种方案会不会导致前端渲染延迟太高？**
  - A: 如果等待完整的 JSON 生成完毕再渲染确实会有延迟。推荐结合 Vercel AI SDK 的 `streamUI` 或类似技术，实现流式渲染 (Streaming Hydration)，让组件随着数据的生成逐块显现。

- **Q: 如果 AI 选错了组件怎么办？**
  - A: 这就是为什么在 `Pro Version` 中我们强制加入了 fallback 机制。在实际系统中，你还需要在前端加入错误边界 (Error Boundaries)，一旦 JSON 解析失败或组件不匹配，立即降级为普通文本回复。

- **Q: 这种架构适用于所有应用吗？**
  - A: 并非如此。对于需要强流程控制的表单（如复杂的税务申报），传统的向导模式依然更稳妥。生成式 UI 更适合分析、查询和仪表盘类的动态展示场景。

## 🧬 提示词解剖 (Why it works?)

1.  **约束可用组件 (Component Registry)：** 明确告知 AI 只能使用 `[可用组件列表]`，彻底消除了 AI 捏造未知标签的幻觉问题。
2.  **结构化输出约束 (Format Constraints)：** 强制要求分离 `componentType` 和 `props`，完美契合 React 等现代前端框架的动态渲染逻辑。
3.  **安全回退机制 (Warning)：** 设定了遇到无法处理请求时的标准错误 JSON，保证了系统在边界情况下的鲁棒性。

## 📊 效果验证：Before & After

### ❌ Before (传统的文本回答)

```text
用户：查看我2月份的支出明细。

AI回复：
您2月份的总支出为 5,200 元。
主要包含：
1. 餐饮：1,500元
2. 交通：800元
3. 购物：2,900元

（只是一段干瘪的文本，缺乏直观的可视化与交互体验）
```

### ✅ After (使用该提示词生成的 UI Payload)

```json
{
  "componentType": "PieChartCard",
  "props": {
    "title": "2月份支出明细",
    "totalAmount": 5200,
    "currency": "CNY",
    "data": [
      { "category": "餐饮", "amount": 1500, "color": "#FF6B6B" },
      { "category": "交通", "amount": 800, "color": "#4ECDC4" },
      { "category": "购物", "amount": 2900, "color": "#FFE66D" }
    ],
    "actions": ["export_csv", "view_details"]
  }
}
```

_(前端接收到此 JSON 后，能够瞬间渲染出一个带有交互式饼图、详细分类以及导出按钮的精美卡片)_
