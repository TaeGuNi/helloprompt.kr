# 📚 Оживление мертвой документации: Автоматизация с помощью ИИ-агента Gemini 3 Pro

- **🎯 Рекомендуется для:** Ведущих разработчиков, Backend-инженеров, Tech Lead, DevOps-инженеров
- **⏱️ Затрачиваемое время:** 30 минут на правку → 0 минут (Полная автоматизация)
- **🤖 Рекомендуемая модель:** Gemini 3 Pro (Обязателен контекст 2M+)

- ⭐ **Сложность:** ⭐⭐⭐☆☆ (Требуется интеграция с CI/CD)
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _"Код обновляется каждый день, а документация к API застряла в прошлом лете? Настало время позволить документации эволюционировать «самостоятельно»."_

Любой разработчик поймет эту боль: код меняется ежедневно, а документация остается в состоянии месячной или даже полугодовой давности. Обещания «обновить позже» — лишь отговорки. Документация имеет смысл только тогда, когда она обновляется **в тот же момент**, что и код.

Но люди по своей природе склонны избегать рутины. Именно поэтому эту скучную и мучительную задачу стоит делегировать **ИИ-агентам**. Прежние модели из-за ограниченного контекста не могли понять, как изменение одного файла влияет на всю структуру документации. Однако феноменальные аналитические способности и гигантское контекстное окно (2M+) **Gemini 3 Pro** меняют правила игры.

Сегодня мы представляем идеальный промпт для настройки рабочего процесса **'Self-Updating Docs' (Самообновляющаяся документация)**, который автоматически отслеживает изменения в коде (Diff) и поддерживает документацию в актуальном состоянии.

---

## ⚡️ Краткая выжимка (TL;DR)

1. Создание автоматизированного пайплайна: ИИ самостоятельно обновляет документацию, анализируя изменения кода (Diff) при каждом коммите.
2. Интеллектуальная фильтрация: игнорируются мелкие опечатки и рефакторинг; обновляются только структурные изменения, такие как новые API-эндпоинты или переменные окружения.
3. Подход Human-in-the-Loop: изменения от ИИ не применяются вслепую — они оформляются в виде Pull Request (PR) для финальной проверки разработчиком.

## 🚀 Решение: "Docs Maintainer" (Агент управления документацией)

### 🥉 Basic Version (Базовая версия)

Используйте для быстрого составления описания PR или простых локальных обновлений README.

> **Роль:** Ты — Senior Technical Writer.
> **Задача:** Проанализируй предоставленный `[Git Diff]` и перепиши в формате Markdown те части `[README.md]`, которые требуют обновления. Игнорируй изменения, связанные с простым рефакторингом или исправлением опечаток.


### 🥇 Pro Version (Продвинутая версия)

Сложный промпт для интеграции в CI/CD (например, GitHub Actions) в качестве полностью автономного агента. Установите `temperature: 0.1` или ниже, чтобы жестко контролировать креативность.

> **Роль (Role):** Ты — агент 'Docs Maintainer', объединяющий навыки технического писателя с 10-летним стажем и DevOps-инженера.
>
> **Контекст (Context):**
>
> - Ситуация: Разработчик только что отправил новый коммит в кодовую базу. Изменения могут включать новые эндпоинты API, переменные окружения или модификацию ключевой бизнес-логики.
> - Цель: Тщательно проанализировать измененный код и точно синхронизировать устаревшую документацию с актуальным состоянием.
>
> **Задача (Task):**
>
> 1. Проведи глубокий сравнительный анализ предоставленного `[Git Diff]` и полного текста `[Target Document]`.
> 2. Определи, влияет ли изменение кода на онбординг пользователей или настройку окружения для других разработчиков (строго игнорируй рефакторинг, переименование переменных и опечатки).
> 3. Если обновление необходимо, полностью перепиши измененный документ в формате Markdown, безупречно сохраняя исходный Tone of Voice (стиль и подачу).
> 4. В самом конце документа добавь скрытый комментарий в формате ``.
> 5. Если коммит не требует обновления документации, выведи исключительно строку `[NO_UPDATE_REQUIRED]`.
>
> **Ограничения (Constraints):**
>
> - Категорически запрещено выдумывать несуществующие функции или переменные (No Hallucinations).
> - Избегай размытых формулировок вроде "внесены соответствующие изменения". Явно указывай конкретные значения: добавленные URL-адреса, имена переменных и т.д.
> - Незавершенный код (закомментированные TODO, FIXME) ни в коем случае не должен попадать в документацию.
> - Твой ответ должен содержать ТОЛЬКО полный текст обновленного документа в Markdown ИЛИ строку `[NO_UPDATE_REQUIRED]`.

## 💡 Комментарий автора (Insight)

При внедрении этого промпта в реальный backend-репозиторий нашей компании самым впечатляющим открытием стала **«способность выявлять пропущенные переменные окружения»**. Был случай, когда разработчик добавил новый ключ стороннего API в `config.ts`, но забыл упомянуть его в разделе «Локальная настройка» в `README.md`. Gemini 3 Pro, сопоставив код и документацию, виртуозно распознал это и самостоятельно добавил `NEW_3RD_PARTY_API_KEY` в `README.md`.

**Совет из практики:** Запуск этого агента на каждом мелком коммите приведет к колоссальным расходам на токены. Настройте фильтр `paths` в GitHub Actions так, чтобы триггер срабатывал только при изменении ключевых файлов логики (`src/api/**`, `config/**`) или схемы БД. Кроме того, никогда не пушьте изменения от ИИ напрямую в ветку `main`. Обязательно создавайте **Pull Request (PR)**, чтобы разработчик мог проверить правки на этапе ревью кода. Финальное одобрение человеком (Human-in-the-Loop) — это последний рубеж защиты достоверности вашей документации.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Сможет ли ИИ понять весь контекст, имея только Git Diff?**
  - Ответ: Для коротких и очевидных правок Diff вполне достаточен. Однако при сложной бизнес-логике высок риск получить некорректную документацию. Главный секрет кардинального повышения точности (Accuracy) — использовать огромное контекстное окно Gemini 3 Pro, передавая ему на вход не только Diff, но и **полный код измененных файлов**, а также **весь целевой документ целиком**.

- **Вопрос: ИИ постоянно пытается задокументировать личные заметки разработчиков и TODO из комментариев. Как это остановить?**
  - Ответ: Это происходит из-за чрезмерной "старательности" ИИ. Именно поэтому в разделе **Ограничения (Constraints)** промпта обязательно должно быть условие: `- Незавершенный код (закомментированные TODO, FIXME) ни в коем случае не должен попадать в документацию.`. Эта единственная строчка решает проблему раз и навсегда.

- **Вопрос: Будет ли это работать с Gemini 2.5 Pro или GPT-4o?**
  - Ответ: Технически — да. Но при подходе "весь документ + все файлы кода" Gemini 3 Pro имеет неоспоримое преимущество благодаря лимиту токенов и дешевому кэшированию контекста (Context Caching). Для небольших pet-проектов с маленькими файлами GPT-4o также покажет отличные результаты.

## 🧬 Анатомия промпта (Why it works?)

1. **Блокирующая логика (Bypass Logic):** Жесткое требование возвращать `[NO_UPDATE_REQUIRED]` для коммитов без важных структурных изменений предотвращает бесконечное создание мусорных PR.
2. **Сохранение стиля (Tone of Voice):** Явное указание "сохранять исходный Tone of Voice" предотвращает переписывание документации в типичном для ИИ сухом и роботизированном стиле.
3. **Скрытые объяснения (Hidden Explanation):** Использование HTML-комментариев (``) для фиксации причин правок позволяет ревьюеру мгновенно понять и проверить логику принятия решений ИИ при проверке PR.
