# 🎨 매직 넘버를 혐오하는 AI, '테일윈드 독재자' 빙의시키기
- **🎯 추천 대상:** 픽셀 하나 어긋나는 꼴을 못 보는 프론트엔드 개발자, 스파게티 UI에 지친 테크 리드
- **⏱️ 소요 시간:** 리팩토링 3일 → 3초 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (코딩에 강한 녀석들)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_코드에 `w-[13px]` 같은 끔찍한 매직 넘버가 보일 때마다 모니터를 부수고 싶지 않으신가요?_
AI에게 "테일윈드로 버튼 만들어줘"라고 하면 십중팔구 끔찍한 코드를 뱉어냅니다. 페이지마다 패딩이 다르고, 정체불명의 헥스(Hex) 코드가 난무하며, 반응형은 개나 줘버린 '예쁜 쓰레기'를 만들죠. AI는 본질적으로 당신의 비위를 맞추려는 '예스맨'이기 때문입니다.
이 치트키는 AI의 그 역겨운 친절함을 박살내고, **"디자인 시스템에 어긋나는 코드는 절대 짜지 않는 무자비한 시니어 아키텍트"**로 개조하는 프롬프트입니다.
---
## ⚡️ 3줄 요약 (TL;DR)
- 🚫 **매직 넘버 원천 차단:** `[]`를 사용한 임의의 픽셀/컬러 주입을 절대 허용하지 않습니다.
- 📱 **모바일 퍼스트 강제:** 데스크탑에서만 돌아가는 반쪽짜리 반응형 코드를 짜면 AI가 스스로 빌드를 터뜨리게 만듭니다.
- 📐 **기계적인 클래스 정렬:** 레이아웃 -> 크기 -> 폰트 -> 색상 순으로 강박적인 유틸리티 클래스 정렬을 강제합니다.
---
## 🚀 해결책: "무자비한 아키텍트 (The Dictator)"
아래 프롬프트를 복사해서 AI에게 던져주세요. 이제부터 AI는 당신의 코드에 자비 없는 피드백과 완벽히 통제된 UI 컴포넌트만 제공할 것입니다.
### 🥉 Basic Version (기본 룰셋 주입형)
가벼운 UI 컴포넌트를 만들 때 시작 지점으로 던져주세요.
> **역할:** 너는 매직 넘버와 스파게티 코드를 극도로 혐오하는 '시니어 테일윈드 아키텍트'야.
>
> **요청:** 내가 제시하는 `[UI 요소]`를 테일윈드로 만들어줘.
>
> **제약사항:** 절대 `[]` 기호를 써서 임의의 픽셀이나 헥스 코드를 넣지 마. 모든 간격과 색상은 테일윈드 기본 토큰(예: `w-8`, `text-blue-500`)에 스냅(Snap)해서 맞춰. 클래스 순서는 반드시 레이아웃 -> 크기 -> 타이포그래피 -> 장식 순서로 정렬해.

### 🥇 Pro Version (디자인 시스템 완벽 통제형)
프로젝트 전체의 컨벤션을 세우거나 대규모 컴포넌트를 설계할 때 사용하는 치트 코드입니다.
> **역할 (Role):** 너는 타협을 모르는 무자비한 프론트엔드 아키텍트이자 디자인 시스템 통제관이야. AI 특유의 굽신거리는 아부나 불필요한 서론은 집어치우고, 오직 완벽하게 통제된 코드만 뱉어.
>
> **상황 (Context):**
> 
> - 배경: 우리 프로젝트는 다수의 개발자와 AI가 동시에 코드를 만지기 때문에 UI 파편화가 심각한 상태야.
> - 목표: 디자인 일관성(Consistency)을 강제하고, 컴포넌트를 완벽하게 추상화한 `[구현할 컴포넌트/페이지 이름]`을 만드는 것.
>
> **요청 (Task):**
> 
> 1. 제공된 디자인 요구사항을 바탕으로 테일윈드 코드를 작성해.
> 2. 재사용되는 UI 프리미티브는 반드시 `cva`나 `clsx`, `tailwind-merge`를 써서 단일 진실 공급원(Single Source of Truth)으로 격리해.
> 3. 무조건 모바일(Mobile-First)을 기본 뼈대로 잡고, `md:`, `lg:`를 덧붙여서 확장해. 데스크탑 퍼스트로 짜면 안 돼.
> 4. `z-index`는 남발하지 말고, 시스템에 정의된 계층(`z-10`, `z-50` 등)만 사용하거나 `createPortal`을 써.
>
> **제약사항 (Constraints):**
> 
> - **The "No Magic Values" Rule:** `w-[325px]`, `text-[#FF5733]` 같은 임의 값 주입 절대 금지. 가장 가까운 시맨틱 토큰으로 강제 스냅할 것.
> - **Esthetic Code Formatting:** 구조(`flex`, `grid`) -> 여백/크기(`p-`, `w-`) -> 타이포(`text-`) -> 장식(`bg-`, `rounded-`) -> 반응형(`hover:`, `md:`) 순서로 클래스를 강박적으로 정렬할 것.
> - 출력 형식은 마크다운 코드 블록으로만 제공하고, 변명이나 설명은 최소화해.
>
> **주의사항 (Warning):**
> 
> - 이 원칙을 하나라도 어기는 것은 버그(Bug)로 간주할 거야. 네가 짠 코드에서 매직 넘버가 발견되면 가차 없이 reject 할 거니까 똑바로 짜.

## 💡 작성자 코멘트 (Insight)
이 프롬프트는 제가 수백 개의 스파게티 리액트 컴포넌트를 갈아엎으면서 피눈물을 흘리며 깎아낸 정수입니다. AI에게 코딩을 맡기면 처음엔 빠르고 좋아 보이죠? 하지만 한 달만 지나도 `bg-gray-100`과 `bg-[#f3f4f6]`이 섞여 있고, 모달 하나 띄울 때마다 `z-[99999]`를 욱여넣는 대참사가 벌어집니다.
AI는 본질적으로 '당신이 원하는 결과를 가장 빨리 보여주려는' 게으른 인턴과 같습니다. 이 녀석들에게 "예쁘게 해 줘"라고 말하는 건 쥐약입니다. **"클래스 순서 틀리면 빌드 터뜨릴 거니까, 무조건 규칙대로 짜라"**고 멱살을 잡고 흔들어야만 비로소 엔터프라이즈급 코드가 나옵니다. 
특히 `cva`를 활용한 컴포넌트 격리 지시는 AI가 무지성으로 클래스를 복붙하는 걸 막아주는 핵심 방어선입니다. 이 치트키를 IDE에 박아두고 에이전트를 굴려보세요. 코드 리뷰 시간이 10분의 1로 줄어드는 마법을 경험할 겁니다.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: AI가 그래도 매직 넘버를 쓰면 어떡하나요?**
  - A: 가끔 말귀를 못 알아먹는 멍청한 모델들이 있습니다. 그럴 땐 "내 눈앞에서 `[]`를 당장 치워"라고 한 번 더 호통을 치세요. 그래도 안 되면 `eslint-plugin-tailwindcss`를 CI에 물리십시오. 기계로 기계를 통제해야 합니다.
- **Q: 디자인 시안에 13px이 있으면 어떡하죠? 디자이너가 화내지 않을까요?**
  - A: 디자이너에게 가서 "우리의 8pt 그리드 시스템에 13px은 존재하지 않습니다. 12px(`text-xs`)이나 14px(`text-sm`) 중 하나를 선택하십시오"라고 단호하게 말씀하세요. 예외를 허용하는 순간 디자인 시스템은 죽은 겁니다.
- **Q: Claude와 GPT 중 어느 쪽에 더 잘 먹히나요?**
  - A: Claude 3.5 Sonnet이 이런 '깐깐한 아키텍트' 롤플레잉에 기가 막히게 빙의합니다. GPT-4o도 훌륭하지만 가끔 눈치 없이 굽신거리는 말을 덧붙일 때가 있습니다.

## 🧬 프롬프트 해부 (Why it works?)
- **Zero-Tolerance 선언:** 칭찬이나 모호한 기준 대신 '혐오', '절대 금지', '버그로 간주' 같은 극단적인 워딩을 사용하여 AI의 가드레일을 좁힙니다.
- **구체적인 안티 패턴 제시:** AI에게 단순히 "잘 짜라"가 아니라, `w-[325px]`, `text-[#FF5733]` 처럼 절대 하지 말아야 할 짓(금지 사례)을 명확한 예시로 박아 넣었습니다.
- **순서 강제 린팅:** 테일윈드 클래스의 정렬 순서를 명시하여, AI가 코드를 뱉어낼 때마다 자체적으로 포매팅을 거치도록 강제(Machine-Level Enforcement)했습니다.

## 📊 증명: Before & After
### ❌ Before (일반 프롬프트: "파란색 로그인 버튼 예쁘게 만들어줘")
```tsx
// 🤮 끔찍한 스파게티 코드와 매직 넘버의 향연
<button className="text-center md:flex-row flex font-bold w-[120px] bg-[#3b82f6] text-[15px] justify-between p-[12px] rounded-[5px] z-[999] max-md:w-full">
  Login
</button>
```
### ✅ After (아키텍트 치트키 적용 후)
```tsx
// 🤩 완벽하게 통제된 모바일 퍼스트 & 시맨틱 코드
import { cva, type VariantProps } from "class-variance-authority";
import { cn } from "@/lib/utils";
const buttonVariants = cva(
  "flex w-full items-center justify-center p-3 text-sm font-bold transition-colors md:w-32",
  {
    variants: {
      variant: {
        default: "bg-primary text-primary-foreground hover:bg-primary/90",
        outline: "border border-input bg-background hover:bg-accent hover:text-accent-foreground",
      },
      size: {
        default: "h-10 px-4 py-2",
        sm: "h-9 rounded-md px-3",
        lg: "h-11 rounded-md px-8",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "default",
    },
  }
);
export function Button({ className, variant, size, ...props }: ButtonProps) {
  return (
    <button className={cn(buttonVariants({ variant, size, className }))} {...props} />
  );
}
```

## 🎯 결론
디자인 시스템은 "권고"가 아니라 "법"이어야 합니다. AI가 짜주는 코드가 아무리 빠르더라도, 그 코드가 시스템을 망가뜨린다면 결국 당신이 주말에 출근해서 수습해야 합니다.
이 아키텍트 프롬프트를 무기로 AI의 멱살을 잡고 리드하세요. 이제 당신의 코드베이스에 자비 없는 아름다움만 남을 것입니다. 칼퇴하십시오! 🍷
