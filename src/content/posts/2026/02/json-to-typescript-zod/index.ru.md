## 💡 Комментарий автора (Insight)

Я использую этот промпт практически каждый день, когда интегрирую новые эндпоинты API. Самая большая ценность здесь кроется не в TypeScript интерфейсах, а именно в **Zod схемах**.
Интерфейсы TS защищают вас только во время написания кода (compile time). Если бэкенд внезапно пришлет `string` вместо `number`, ваше приложение всё равно упадет в браузере пользователя. Создав схему Zod с помощью этого промпта и обернув ответ `fetch` в `schema.parse(data)`, вы возводите непробиваемую стену от невалидных данных. Секрет эффективности этого промпта заключается в требовании "разделять вложенные объекты" — это делает сгенерированный код модульным, чистым и легко переиспользуемым в других частях проекта.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Можно ли адаптировать промпт под Yup или Joi вместо Zod?**
  - A: Абсолютно! Просто замените все упоминания "Zod" на "Yup" или "Joi" в секции _Задача (Task)_. Логика ИИ отработает точно так же безупречно.

- **Q: Что делать, если в массиве JSON приходят разные типы данных (Mixed Array)?**
  - A: ИИ обычно справляется с этим, генерируя `z.union()` или объединения типов TypeScript (`string | number`). Однако, для идеального результата, старайтесь предоставлять ИИ максимально полный пример JSON, охватывающий все возможные варианты данных.

- **Q: Модель выдает слишком много текстовых объяснений вместе с кодом. Как это исправить?**
  - A: В Pro-версии уже есть блок _Внимание (Warning)_, который запрещает модели болтать лишнее. Если вы используете Basic-версию, просто допишите в конце: `Отвечай только кодом, без комментариев`.

## 🧬 Анатомия промпта (Why it works?)

1. **Архитектурный подход (Task 1-3):** Промпт не просит "напиши интерфейс", он просит "создай Zod схему и выведи интерфейс из неё (`z.infer`)". Это паттерн проектирования _Single Source of Truth_, который исключает рассинхронизацию между валидацией и типами.
2. **Модульность (Constraints):** Указание "выделить вложенные объекты" предотвращает появление гигантских, нечитаемых "спагетти-схем", которые ИИ часто выдает по умолчанию.
3. **Безопасность типов (Constraints):** Строгое правило "все поля обязательны, если нет null" спасает от ленивого поведения ИИ, когда он ставит знак вопроса `?` повсюду, сводя на нет всю пользу строгой типизации TypeScript.

## 📊 Доказательство: Before & After

### ❌ Before (Входной JSON от API)

```json
{
  "id": "USR-88291",
  "isActive": true,
  "profile": {
    "username": "alex_dev",
    "avatarUrl": null,
    "lastLogin": "2026-02-04T10:30:00Z"
  },
  "roles": ["admin", "editor"]
}
```

### ✅ After (Результат от ИИ)

```typescript
import { z } from "zod";

/** Схема профиля пользователя */
export const UserProfileSchema = z.object({
  username: z.string(),
  avatarUrl: z.string().nullable().optional(),
  lastLogin: z.string().datetime(),
});

/** Основная схема пользователя */
export const UserSchema = z.object({
  id: z.string(),
  isActive: z.boolean(),
  profile: UserProfileSchema,
  roles: z.array(z.string()),
});

/** Интерфейс профиля пользователя */
export type UserProfile = z.infer<typeof UserProfileSchema>;

/** Интерфейс пользователя (Single Source of Truth) */
export type User = z.infer<typeof UserSchema>;
```
