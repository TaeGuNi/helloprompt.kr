# 🍝 複雑なコードにさようなら！AIコード複雑度アナライザー (Code Complexity Analyzer)

- **🎯 おすすめの対象:** ジュニアエンジニア、レガシーコードを引き継いだ保守担当者、コードレビュアー
- **⏱️ 所要時間:** 1時間 → 3分に短縮
- **🤖 おすすめのモデル:** GPT-4o, Claude 3.5 Sonnet (コードの推論能力に優れたモデルを推奨)

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「if文の中にfor文、その中にまたif文…動いてはいるけれど、少しでも触ると爆発しそうな時限爆弾のようなコードに絶望したことはありませんか？」_

開発現場では、誰もが一度は複雑に絡み合った「スパゲティコード」に直面するものです。機能追加やバグ修正を繰り返すうちにコードは肥大化し、循環的複雑度（Cyclomatic Complexity）は跳ね上がっていきます。こうしたコードは単に読みにくいだけでなく、修正時の予期せぬ副作用（サイドエフェクト）を引き起こしやすいため、保守地獄を招く最大の要因となります。

どこから手をつければいいのか途方に暮れる前に、AIにコードの複雑度分析を任せてみましょう。AIは客観的な指標で問題点を診断し、デザインパターンやクリーンコードの原則に基づいた、実践的で具体的なリファクタリング戦略を提示してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. 読みにくいネストされた条件文や巨大な関数をAIが構造的に分析します。
2. 早期リターン（Early Return）や関数の抽出（Extract Method）など、実務に即したリファクタリング手法の提案を受けられます。
3. ロジックの振る舞いを変えることなく、可読性と保守性が最大化されたクリーンコードを手に入れることができます。

> **役割 (Role):** あなたはクリーンアーキテクチャに精通した`[実務経験10年のシニアソフトウェアエンジニア]`です。

> **状況 (Context):**
>
> - 背景: 現在、`[言語/フレームワーク、例: TypeScript/NestJS]`で書かれたレガシーコードをリファクタリングしています。

## 💡 筆者のコメント (Insight)

実務で他人が書いた（あるいは過去の自分が書いた）レガシーコードをレビュー・修正する際、私が真っ先に活用するプロンプトです。AIは単にコードを短縮するだけでなく、「この条件文はネスト（Depth）が4段階に達しているため、Guard Clauses（ガード節）を使って外に出す方が良いです」といった、論理的かつ説得力のある根拠を提示してくれます。

特に、コードレビューを控えて自分のコードの複雑度を自己点検する際に非常に役立ちます。PR（Pull Request）を作成する前にこのプロンプトで一度フィルタリングを通すだけで、チームメンバーから「コードがとても綺麗ですね」というフィードバックをもらえる確率が飛躍的に上がります。

## 🙋 よくある質問 (FAQ)

- **Q: ファイル全体を一度に入力して分析してもいいですか？**
  - A: LLMのコンテキストウィンドウの制限や精度の低下を招くため、推奨しません。数百行を超えるファイル全体よりも、50〜150行程度の「特定の関数」や「単一のクラス」単位に分割して入力する方が、はるかに鋭く正確なリファクタリング結果を得られます。

- **Q: AIがリファクタリングしたコードをそのまま本番環境（プロダクション）に適用しても安全ですか？**
  - A: 絶対にそのままコピー＆ペーストしないでください。AIは時折、ビジネスロジックの微妙なエッジケースを見落としたり、ロジックを変形（ハルシネーション）させたりする危険性があります。**必ず既存の単体テスト（Unit Test）を通過するか確認**してから反映してください。

- **Q: 特定の言語やフレームワークに合わせて改善してくれますか？**
  - A: はい、可能です。Pro Versionの変数である`[言語/フレームワーク]`に「Java/Spring Boot」や「Python/FastAPI」などを明記すれば、そのエコシステムで推奨される命名規則（ネーミングコンベンション）やデザインパターンを反映したコードへと改善してくれます。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **指標中心の診断要求:** 単に「綺麗に書いて」と漠然と頼むのではなく、「循環的複雑度」という具体的なソフトウェア工学の指標を要求することで、AIの注意を構造的な欠陥に集中させています。
2. **動作維持の制約 (Constraints):** リファクタリングの第一原則である「外部的な振る舞いを保つ（変更しない）」ことを明示し、AIが勝手にロジック自体を書き換えてしまうリスクを最小限に抑えています。

## 📊 検証: Before & After

### ❌ Before (入力: ネストの深いスパゲティコード)

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ After (出力: ガード節と配列メソッドを活用したクリーンなコード)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: 例外状況の早期リターン（ネストの削減）
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. 高階関数の活用: ループ処理の単純化
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. ビジネスロジックの分離と単純化
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```
