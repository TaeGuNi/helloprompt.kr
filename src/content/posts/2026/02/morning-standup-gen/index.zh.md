## 📝 1 分钟搞定：昨日工作总结与每日站会准备

- **🎯 推荐对象：** 开发者、项目经理（PM）及所有需要参与敏捷站会的职场人
- **⏱️ 预计耗时：** 15 分钟 → 缩短至 1 分钟
- **🤖 推荐模型：** 任何对话型 AI（ChatGPT、Claude、Gemini 等）

- ⭐ **难度：** ⭐☆☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“每天早上开站会前，是不是都在疯狂翻找 GitHub 提交记录和 Jira 面板，拼命回想昨天到底干了什么？”_

每日站会（Daily Standup）的核心意义在于同步进度与暴露风险。然而，若缺乏充分的会前准备，汇报极易沦为毫无重点的“流水账”。尤其是对于开发者而言，代码写得越多，往往越难用精炼的语言向 PM 或设计师等非技术职能人员讲清昨日产出的核心业务价值。

现在，你只需将昨天杂乱无章的 Commit Log、聊天记录或零散的代码片段直接丢给 AI，它就能在短短 1 分钟内为你自动生成一份结构清晰、直击重点的高质量站会发言稿。

---

## ⚡️ 3 句话总结 (TL;DR)

1. **告别流水账式汇报：** 将零散的 Git 提交记录一键转化为逻辑严密的业务汇报。
2. **跨部门沟通零障碍：** 自动将晦涩的技术术语“翻译”为产品经理秒懂的业务进展。
3. **今日计划一目了然：** 在总结昨日工作的同时，顺理成章地梳理出今日的核心任务。

---

## 🚀 解决方案：“一键生成站会汇报”

### 🥉 Basic Version (基础版)

马上就要轮到你发言，急需一份简明扼要的汇报“小抄”时，请使用此版本。

> **角色：** 你是一位经验丰富的高级软件工程师。
> 
> **任务：** 请根据我提供的 `[昨日工作记录]`，帮我撰写一段用于每日站会的口头汇报稿。请保持简明扼要，重点突出我完成了哪些工作，以及是否遇到了阻碍。
> 
> **[昨日工作记录]：**
> (在此粘贴你的 Commit Log 或工作笔记)

### 🥇 Pro Version (专业版)

当面对包含非技术人员的跨职能团队，需要进行高质量、高信息密度的汇报时，请务必使用此版本。

> **角色 (Role)：** 你是一位资深的技术研发主管（Tech Lead），不仅精通代码，更擅长跨部门沟通，能够将复杂的技术问题精准转化为清晰的业务价值。
> 
> **背景 (Context)：**
> 
> - 当前场景：即将参与跨职能团队的每日敏捷站会（Daily Standup）。
> - 听众画像：包含产品经理（PM）、设计师以及前端/后端开发人员。
> - 核心目标：使用清晰、自信且通俗易懂的语言，汇报昨日进展与今日计划，并明确指出是否存在阻塞项（Blockers）。
> 
> **任务 (Task)：**
> 
> 1. 深度分析我提供的 `[原始工作数据]`（如 Git 提交记录、Jira 票据或零碎笔记）。
> 2. 将纯技术视角的描述转化为**业务导向**的成果说明（例如：将“重构了 Auth 模块”转化为“优化了登录流程，有效减少了用户的等待时间”）。
> 3. 生成一份符合“昨日进展 / 今日计划 / 当前阻碍”标准结构的口语化汇报发言稿。
> 
> **输入变量 (Input)：**
> 
> - `[原始工作数据]`：(在此粘贴你的数据)
> 
> **制约因素 (Constraints)：**
> 
> - 篇幅限制：汇报时长需控制在 1 分钟以内（约 150-200 字）。
> - 语气基调：专业、积极且高度口语化，确保能够直接、流畅地朗读。
> - 输出格式：请使用清晰的 Markdown 列表呈现汇报结构，并高亮核心成就。
> 
> **注意事项 (Warning)：**
> 
> - 严禁编造任何我未提供的功能或进度（坚决杜绝 AI 幻觉）。若提供的信息不足以推演“今日计划”，请在脚本中通过括号占位符提示我自行补充。

## 💡 作者见解 (Insight)

这套提示词绝不仅仅是一个用来“偷懒”的工具，更是能大幅提升你**职场能见度（Visibility）**的社交利器。许多初中级开发者在站会上习惯于“报菜名”式地罗列代码文件或函数名称，这其实是一种沟通效率极低的做法。

通过运用 Pro 版本的提示词，AI 会强制你跳出底层的代码视角，从“业务价值”的高度重新审视自己的产出。当你逐渐习惯将“修复了 SQL N+1 查询问题”转化为“提升了首页 30% 的加载速度”时，你在团队内部的技术影响力（Tech Influence）自然也会水涨船高。这正是巧用 AI 赋能思维进阶的绝佳案例。

## 🙋 常见问题 (FAQ)

- **Q: 如果我的工作涉及商业机密，能把代码直接发给 AI 吗？**
  - A: 绝对不行！请务必牢记，切勿将公司的核心机密代码或真实的 API Key 填入提示词。你只需提供脱敏后的 Commit Message（例如 `fix: 修复登录鉴权 bug`）即可，完全不需要粘贴具体的代码实现。若项目涉密等级极高，强烈建议采用本地部署的开源大模型。
- **Q: AI 生成的文案听起来像“播音腔”，不够自然怎么办？**
  - A: 若觉得行文过于正式，可以在提示词的“制约因素”中追加一条指令：“请采用轻松的口语化表达，多用短句，并适当穿插一些口语化的停顿词（如‘那个’、‘主要是’）”。这样生成的发言稿会显得更加自然、接地气。
- **Q: 我能一次性丢入一周的工作日志，让它帮我写周报吗？**
  - A: 当然可以。你只需将“背景”设定调整为“每周项目例会”，并要求“输出格式”聚焦于“本周核心产出”，AI 同样能够出色地完成这项进阶任务。

## 🧬 提示词解剖 (Why it works?)

1. **受众设定（Audience Targeting）：** 明确界定听众包含“产品经理和设计师”，这会迫使 AI 自动过滤掉晦涩难懂的底层技术细节，转而输出跨部门都能听懂的业务语言。
2. **角色映射（Role-Play）：** 赋予 AI “资深 Tech Lead” 的角色定位，使其生成的语调自带专业感与掌控力，无形中提升了你在会议上的发言气场。
3. **防幻觉约束（Anti-Hallucination Constraints）：** 明确强调“严禁编造”，有效防止 AI 为了凑字数而虚构你并未参与的工作，从而捍卫了汇报的客观真实性与你的职场信誉。
