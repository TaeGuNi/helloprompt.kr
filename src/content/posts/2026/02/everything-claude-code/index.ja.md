## 🚀 解決策："Everything Claude Code" プロンプティング

### 🥉 Basic Version（基本型：迅速なセットアップと初期設計）

ターミナルからたった2分で、優秀なシニアエージェントをチームに迎え入れましょう。

> **1. プラグインのインストール（Claude Codeの場合）:**
> `/plugin marketplace add affaan-m/everything-claude-code`
> `/plugin install everything-claude-code@everything-claude-code`
>
> **2. 設計（Plan）プロンプトの実行:**
> `/plan "Next.js 14のApp Routerを使用して、レスポンシブ対応のブログ記事一覧ページを作成して。スタイリングにはTailwind CSSとshadcn/uiを使用する想定で進めてください。"`

### 🥇 Pro Version（専門家型：TDDとカスタムルールの強制）

プロジェクト固有の要件に合わせ、シニア開発者レベルの厳格なワークフローを強制したい場合に活用します。

> **役割 (Role):** あなたは厳格なTDD（テスト駆動開発）の原則を遵守する`[シニアTypeScript開発者]`です。
>
> **状況 (Context):**
>
> - 背景: 現在、私たちは`[everything-claude-code]`のルールに則って開発を進行しています。
> - 目標: `[ユーザー認証機能]`を実装する必要があります。
>
> **要請 (Task):**
>
> 1. ターミナルで`/tdd`コマンドを有効化してください。
> 2. プロダクションコードを記述する前に、必ずエッジケースを含むテストコードを先に作成してください。
> 3. テストカバレッジが80%を超えない限り、次の実装ステップへは進まないでください。
> 4. `[追加の要件]`がある場合は、それを完全に反映させた上でコードを実装してください。
>
> **制約事項 (Constraints):**
>
> - すべての変数と関数に、明示的な型注釈（Type Annotation）を付与すること。
> - `any`型の使用は**一切禁止**します。
>
> **注意事項 (Warning):**
>
> - 既存のアーキテクチャやコーディング規約に違反するコードは作成しないでください。確信が持てない仕様については、勝手に推測せず、コードを書く前に必ず私に質問してください。（ハルシネーションの防止）

## 💡 筆者のインサイト (Insight)

この設定を実際のプロジェクトに適用して最も驚かされたのは、**「AIが怠惰なプロンプティングを明確に拒否する」**という事実でした。

以前であれば、「ログインページを作って」と雑な指示を投げただけで、エラーハンドリングすら考慮されていない表面的なコードが出力されがちでした。しかし、この設定を導入すると世界が変わります。`/plan`コマンドを実行した瞬間、AIが自ら_「どの認証方式（OAuth, JWT）を採用しますか？エラー発生時のUIはどう処理すべきですか？」_と、逆に仕様の抜け漏れを鋭く問い詰めてくるのです。それはまるで、**明確な要件定義が揃うまでは一行のコードすら書かない、極めて優秀で厳格なシニア開発者**をメンターに迎えたかのような錯覚に陥る体験です。

特に感銘を受けたのは**`/evolve`**コマンドの強力さです。プロジェクトの進行に伴い、AIがチーム独自のコーディング規約（例：「インターフェース名の先頭に『I』を付けない」等）を自律的に学習し、それを絶対のルールとして定着させていきます。スパゲティコードの量産に疲弊している方や、シニアエンジニアの鋭いコードレビューを渇望している個人開発者にとって、このワークフローはもはや選択肢ではなく、導入**必須**のパラダイムシフトと言えるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: これはClaude Code CLI環境でしか使えませんか？**
  - A: いいえ、特定の環境に依存しません。Cursorエディタの`.cursorrules`ファイルに適用したり、OpenCode環境で活用したりと、他の主要な開発ツールとも完全に互換性があります。

- **Q: 社内の機密プロジェクトや実務で使用しても安全ですか？**
  - A: ルール（Rule）ファイル自体はオープンソースであり、ローカル環境で完結するため安全です。ただし、LLMモデルへコードを送信する行為そのものについては、所属組織のセキュリティポリシー（Enterpriseプランのデータ利用規定など）を必ず事前に確認してください。

- **Q: 初心者にとって、この設定は厳しすぎませんか？**
  - A: 確かに、導入直後はAIから仕様に関する逆質問が飛んできたり、テストコードの作成を強制されたりと、少しハードルが高く感じるかもしれません。しかし、この厳格なプロセスを初期段階で組み込むことこそが、後々膨大な「技術的負債（Technical Debt）」の返済に追われて徹夜する事態を防ぐ、唯一にして最強の防衛策となります。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **強制されたワークフロー (Forced Workflow):** 単純な指示出しを排除し、`/plan`や`/tdd`といったスラッシュコマンドを通じて、AIに「要件分析 → テスト作成 → 実装」という正しい開発プロセスを強制的に踏ませます。
2. **ペルソナの分離 (Persona Separation):** コードを書く自我（Developer）、レビューする自我（Reviewer）、企画する自我（Planner）を明確に分離することで、AI自身にコードの客観的なクロスチェック（交差検証）を促します。
3. **継続的な学習 (Instincts):** 会話のコンテキストが揮発してしまう従来のチャットボットとは異なり、AIが開発を通じたユーザーの修正パターンを学習し、それを`.rules`ファイルとしてプロジェクトのDNAに内在化させます。
