## 📝 私のコードがゴミだって！？クリーンコード・リファクタリング・コーチ

- **🎯 推奨対象:** ジュニア〜ミドルエンジニア、およびコードレビューに忙殺されているシニアエンジニア
- **⏱️ 所要時間:** 30分 → 1分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コーディング特化), GPT-4o

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐☆

> _「とりあえず動くからヨシ！」と妥協して数ヶ月。今や誰も触れたがらない"パンドラの箱"になっていませんか？_

スパゲッティコードは、開発スピードを著しく低下させる最大の元凶です。可読性の低いコードはバグの温床となるだけでなく、ちょっとした新機能の追加すら命がけの作業へと変貌させてしまいます。本記事で紹介するプロンプトは、名著『Clean Code』に示されるような先人たちのベストプラクティスをAIに憑依させ、あなたのコードを根本から洗い直して、息を呑むほど美しい状態へと磨き上げます。

---

## ⚡️ 3行要約 (TL;DR)

1. **命名規則の最適化:** 意図が読めない変数名を、役割が明確に伝わる直感的なネーミングへと刷新します。
2. **関数の単一責任化:** 肥大化した長すぎる関数を、役割ごとに小さく分割（Extract Method）して整理します。
3. **最新構文の適用:** レガシーな書き方を排除し、対象言語の最新機能（Modern Syntax）を活用した簡潔なコードへと変換します。

---

## 🚀 解決策: "クリーンコード・リファクタリング"

### 🥉 Basic Version (基本型)

手っ取り早く、とりあえず全体をきれいに整えたい時に使用してください。

> **役割:** あなたは `[経験豊富なシニア・ソフトウェアアーキテクト]` です。
> **リクエスト:** 以下のコードは動作しますが、可読性が低く保守が困難です。クリーンコードの原則に従って、読みやすく保守しやすいコードにリファクタリングしてください。


### 🥇 Pro Version (専門家型)

実務レベルでの厳密なコードレビューと、リファクタリングの「根拠」までしっかり理解したい時に使用してください。

> **役割 (Role):** あなたは『Clean Code』の原則を熟知した `[シニア・ソフトウェアアーキテクト]` です。
>
> **状況 (Context):**
>
> - 背景: 現在のコードは要件を満たして動作していますが、技術的負債（Dirty Code）が蓄積しており、他の開発者が理解するのに時間がかかります。
> - 目標: チーム全体が保守しやすく、テストが容易で、拡張性の高いコードに改善すること。
>
> **リクエスト (Task):**
>
> 1. **[ネーミングの改善]**: 変数名や関数名を、その役割が明確に伝わる直感的な名前に変更してください。（例: `d` -> `elapsedTimeInDays`）
> 2. **[構造の最適化]**: 「1つの関数は1つのことだけを行う」という原則に従い、肥大化した関数を適切に分割し、重複ロジックを排除してください。
> 3. **[最新構文の適用]**: 可能であれば、対象言語（例: `[Python 3.12 / ES2023 など言語を指定]`）の最新かつ安全な構文を適用し、より簡潔に記述してください。
> 4. **[Before & Afterの解説]**: 修正箇所を具体的に明示し、「なぜそのように変更したのか」というアーキテクチャ上の理由をMarkdownのリスト形式で論理的に説明してください。
>
> **制約事項 (Constraints):**
>
> - コードの既存の「動作（ビジネスロジック）」は絶対に変更しないでください。
> - コメントは、コード自体で意図を表現できない「どうしても必要な箇所（Why）」にのみ記述してください。不要な説明コメント（What）はすべて削除してください。
> - 出力は、解説とリファクタリング後の完全なコードブロックを含めてください。
>

>

## 🙋 よくある質問 (FAQ)

- **Q: リファクタリングによってバグが発生したり、既存の動作が変わってしまったりしませんか？**
  - A: プロンプト内で「既存の動作は絶対に変更しないこと」と厳格な制約を設けていますが、AIが意図せずビジネスロジックを改変してしまうリスクはゼロではありません。実稼働環境に反映する前に、必ず**ユニットテスト**を実行し、修正前後で挙動に変化がないことを入念に確認してください。

- **Q: 会社の大規模な社内システムのコードを、そのまま丸ごと貼り付けても大丈夫ですか？**
  - A: 絶対におやめください。機密情報や社外秘のコアロジックが含まれるコードを公開されているAIモデルに入力することは、重大なセキュリティ違反に繋がる恐れがあります。対象となる関数のロジック部分のみを切り出すか、あらかじめ機密の変数名をダミーデータに置き換えた上で活用してください。

- **Q: この用途において、どのAIモデルを使用するのが最も効果的ですか？**
  - A: コーディングタスクにおいては、現時点では **Claude 3.5 Sonnet** が圧倒的なパフォーマンスを誇ります。コードの文脈（コンテキスト）に対する深い理解力を持ち、開発者の意図を的確に汲み取った極めて美しいリファクタリング案を提示してくれます。

## 🧬 プロンプト解剖 (Why it works?)

1. **Role（役割の付与）:** AIに「シニアアーキテクト」という明確なペルソナを憑依させることで、出力の基準を「とりあえず動くレベルのコード」から「保守性と拡張性に優れたプロフェッショナルなコード」へと一気に引き上げます。
2. **Constraints（制約による暴走防止）:** 「既存の動作を変えない」「不要なコメントは削除する」といった厳格なガードレールを設けることで、AIが勝手に不要な機能を付け足したり、冗長な説明コメントでコードベースを汚染したりする事態（Hallucination）を未然に防ぎます。
3. **Task（論理的思考の誘導）:** 単にコードを書き直させるだけでなく、「なぜそのように修正したのか」というアーキテクチャ上の理由を言語化させることで、Chain-of-Thought（段階的思考）が強力に促され、より緻密で精度の高いリファクタリング結果が導き出されます。

## 📊 証明: Before & After

### ❌ Before (入力)

```python
def calc(x, y):
    if x > 19:
        print("成人")
        p = y * 0.9
        return p
    else:
        print("学生")
        return y
```

_（関数名 `calc` からは具体的な処理内容が全く推測できず、変数 `p` が何を指すのかも不明瞭です。さらに、`19` や `0.9` といったマジックナンバーがコード内に直書きされており、保守性を著しく損なっています）_

### ✅ After (結果)

```python
def calculate_discounted_price(age: int, price: float) -> float:
    ADULT_AGE_THRESHOLD = 20
    DISCOUNT_RATE = 0.9

    if age >= ADULT_AGE_THRESHOLD:
        print("成人です。")
        return price * DISCOUNT_RATE

    print("学生です。")
    return price
```

**[改善ポイントの解説]**

1. **関数・変数名の明確化**: 曖昧だった `calc` を `calculate_discounted_price` へと変更し、さらに引数と戻り値に型ヒント（Type Hinting）を追加して堅牢性を高めました。
2. **マジックナンバーの排除**: `20` や `0.9` といった文脈不明な数値に、それぞれ適切な定数名 (`ADULT_AGE_THRESHOLD`, `DISCOUNT_RATE`) を割り当てることで、ビジネスロジックの意図を明確にしました。
3. **早期リターン (Early Return) の活用**: 不要な `else` ブロックを排除し、処理のネスト（インデント）を浅く保つことで、コード全体の可読性を大幅に向上させています。
