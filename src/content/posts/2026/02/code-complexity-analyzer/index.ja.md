## 🍝 複雑なコードにさようなら！AIコード複雑度アナライザー (Code Complexity Analyzer)

- **🎯 おすすめの対象:** ジュニアエンジニア、レガシーコードの保守担当者、コードレビュアー
- **⏱️ 所要時間:** 1時間 → 3分に短縮
- **🤖 おすすめのモデル:** GPT-4o, Claude 3.5 Sonnet (コードの推論能力に優れたモデルを推奨)

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「if文の中にfor文、さらにその中にif文……。辛うじて動いてはいるものの、少しでも触れれば爆発しそうな『時限爆弾』のようなコードに絶望したことはありませんか？」_

開発現場において、複雑に絡み合った「スパゲティコード」との遭遇は避けて通れない宿命です。機能追加やバグ修正を繰り返すたびにコードは肥大化し、循環的複雑度（Cyclomatic Complexity）は容赦なく跳ね上がります。こうしたコードは単に可読性を下げるだけでなく、修正時に予期せぬ副作用（サイドエフェクト）を引き起こす温床となり、「保守地獄」を招く最大の要因となります。

どこから手をつけるべきか途方に暮れる前に、AIへコードの複雑度分析を委ねてみましょう。AIは客観的な指標を用いてコードの病巣を正確に診断し、デザインパターンやクリーンコードの原則に基づいた、極めて実践的かつ具体的なリファクタリング戦略を提示してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. 複雑にネストされた条件文や肥大化した関数を、AIが構造的かつ客観的に分析・診断します。
2. 早期リターン（Early Return）や関数の抽出（Extract Method）など、実務ですぐに使える具体的なリファクタリング手法を提示します。
3. 既存のロジックや振る舞いを一切損なうことなく、可読性と保守性を極限まで高めたクリーンコードへと昇華させます。

## 💡 筆者のコメント (Insight)

実務において、他人が書いた（あるいは過去の自分が書いた）難解なレガシーコードをレビュー・修正する際、私が真っ先に投入する「切り札」とも言えるプロンプトです。AIは単にコードの記述量を減らすだけでなく、「この条件文はネスト（Depth）が4段階に達しているため、Guard Clauses（ガード節）を用いて早期リターンさせるべきです」といった、極めて論理的かつ説得力のあるエンジニアリング的根拠を提示してくれます。

とりわけ、コードレビューを控えた**自身のコードの複雑度をセルフチェックする場面**において、このプロンプトは絶大な威力を発揮します。PR（Pull Request）を作成する前に本プロンプトを通してコードを磨き上げるだけで、レビュアーから「このコード、非常に美しく設計されていますね」と称賛される確率が飛躍的に高まるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: ファイル全体を一度にAIへ入力して分析させてもよいですか？**
  - A: いいえ。LLMのコンテキストウィンドウの制限により精度が著しく低下するため、推奨しません。数百行に及ぶファイル全体を丸投げするのではなく、50〜150行程度の「特定の関数」や「単一のクラス」単位に分割して入力する方が、はるかに鋭敏で正確なリファクタリング結果を引き出せます。

- **Q: AIがリファクタリングしたコードを、そのまま本番環境（プロダクション）に適用しても安全ですか？**
  - A: 絶対にそのままコピー＆ペーストしてはいけません。AIは時として、ビジネスロジックの微細なエッジケースを見落としたり、ロジックそのものを意図せず変形（ハルシネーション）させてしまう危険性を孕んでいます。本番への反映前には、**必ず既存の単体テスト（Unit Test）がすべて通過することを確認**し、人間の目による最終チェックを怠らないでください。

- **Q: 特定のプログラミング言語やフレームワークの作法に合わせて最適化することは可能ですか？**
  - A: はい、完全に可能です。Pro Versionのプロンプト内にある`[言語/フレームワーク]`の変数に、「Java/Spring Boot」や「Python/FastAPI」のように具体的な技術スタックを明記してください。そのエコシステム特有の推奨される命名規則（ネーミングコンベンション）やデザインパターンを的確に反映したコードへと仕上げてくれます。

## 🧬 プロンプトの解剖学 (Why it works?)

1. **客観的指標に基づく診断要求:** 単に「コードを綺麗にして」と漠然と指示するのではなく、「循環的複雑度」や「認知的複雑度」といったソフトウェア工学における具体的な指標を明示することで、AIの推論リソースを「構造的欠陥の発見」へ強制的に集中させています。

## 📊 検証: Before & After

### ❌ Before (入力：ネストが深く入り組んだスパゲティコード)

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ After (出力：ガード節と配列メソッドを駆使したクリーンコード)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: 例外状況の早期リターン（ネストの削減）
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. 高階関数の活用: ループ処理の単純化
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. ビジネスロジックの分離と単純化
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```
