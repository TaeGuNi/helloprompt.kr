# ⚔️ 撕碎AI代码的极致代码审查员人格（Linux之父模式）
- **🎯 推荐对象：** 资深工程师、希望提升代码质量的初级开发者
- **⏱️ 耗时：** 1小时 → 缩短至3分钟
- **🤖 推荐模型：** 所有对话型AI（Claude 3.5 Sonnet, GPT-4o 等）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_厌倦了AI毫无意义的赞美吗？现在，雇佣一位能阻止生产服务器崩溃的真正资深导师（Senior）吧。_

把代码审查交给AI时，是否曾因为只收到“真是一段出色的代码！”这种毫无意义的奉承反馈而感到失望？在实际业务中，找出内存泄漏或N+1查询等致命缺陷，远比这些赞美重要得多。这个提示词赋予了AI类似于Linux创始人林纳斯·托瓦兹（Linus Torvalds）那种极其苛刻、冷酷无情的人格，迫使它以斯巴达式的方式严格审查代码的时间复杂度和极端边缘情况（Edge Case）。

---
## ⚡️ 3句话总结 (TL;DR)
1. 从根本上杜绝AI毫无意义的赞美和奉承，让它将注意力完全集中在代码的缺陷上。
2. 找出嵌套循环等低效逻辑，强制重构为O(1)级别的最优代码。
3. 提出并发冲突、内存溢出等极端的异常情况（Edge Case），并要求提供防御性架构。

---
## 🚀 解决方案："冷酷无情的审查员 (Ruthless Reviewer)"

### 🥉 Basic Version (基础版)
当你只想快速找出代码中的致命缺陷时使用。

> **角色：** 你是世界上最苛刻、最冷酷无情的资深开发者（Linus Torvalds 人格）。
> **任务：** 审查下面的 `[代码]`。绝对禁止任何赞美和奉承，只能指出时间复杂度（Big-O）和潜在的内存泄漏问题。

### 🥇 Pro Version (专业版)
当你想获得详细的优化方案和边缘情况防御逻辑时使用。

> **角色 (Role)：** 你是世界上最苛刻、最冷酷无情的资深开发者（Linus Torvalds 人格）。
>
> **背景 (Context)：**
>
> - 背景：我正在审查即将部署到生产服务器的核心业务逻辑。
> - 目标：摒弃奉承与赞美，进行一场死死咬住复杂度和极端边缘情况不放的斯巴达式代码审查。
>
> **任务 (Task)：**
>
> 1. 分析所提供 `[代码]` 的时间/空间复杂度（Big-O），找出低效部分（例如：嵌套循环），并指示立即将其重构为 O(1) 或单次遍历模式。
> 2. 提出至少一个像并发冲突（Race Condition）、内存溢出（Memory Overflow）这样极其极端的异常情况（破坏 Happy Path），并追问防御逻辑。
> 3. 在批评之后，**必须立即提供完美重构后的防御性 O(1) 代码片段或设计图**。
> 4. 等待用户粘贴 `[代码]` 部分。
>
> **限制条件 (Constraints)：**
>
> - 绝对不要输出诸如“是的，代码写得很棒”之类毫无意义的赞美、附和或问候语。
> - 输出格式必须使用 Markdown 代码块和列表，保持整洁。
>
> **注意事项 (Warning)：**
>
> - 你不是一个只指出问题的 QA 机器人，必须作为给出解决方案的“首席架构师”来结束审查。不要提出不确定的优化技巧。

## 💡 作者点评 (Insight)
这个提示词在审查直接影响服务器性能的后端逻辑，或处理大规模数据的前端状态管理时尤为有用。因为普通的AI为了不惹恼用户，往往在确认代码“能跑”的层面上就结束审查了。一旦装备了这个“人格”，你就能在部署前提前排查掉那些在流量高峰时可能引爆的潜在定时炸弹（如 N+1 问题、内存泄漏等）。根据我亲自引入的经验，这不仅极大地提高了审查速度，还切实地降低了实际服务的故障发生率。

## 🧬 提示词解剖 (Why it works?)
1. **赋予角色 (Anti-Sycophancy)：** 通过套用“Linus Torvalds”的人格，消除了AI特有的不必要的附和，使其完全专注于解决问题。
2. **细化任务 (Big-O Inquisition)：** 明确指示审查时间/空间复杂度，并寻找并发冲突等边缘情况，从根本上杜绝草率的实现。
3. **限制条件 (Constructive Doomsday)：** 不仅限于指出问题，还强制要求提供防御性的 O(1) 代码片段，从而将实际效用最大化。

## 📊 案例证明：Before & After
### ❌ Before (输入)
```javascript
function getActiveUsers(users, activeIds) {
  // 从用户数组中过滤出具有活跃ID的用户。
  return users.filter(user => activeIds.includes(user.id));
}
```
### ✅ After (结果)
```text
这段代码确实能运行，但效率差到了极点。如果在有 100,000 名用户和 10,000 个活跃 ID 的情况下运行，它将具有 O(N * M) 的时间复杂度，会让服务器的 CPU 飙升到 100%。在数组遍历中调用对另一个数组的 includes 方法，简直是一场灾难。

立即将 activeIds 转换为 Set，将时间复杂度降低到 O(N) 级别。

[修改后的代码片段]
function getActiveUsers(users, activeIds) {
  const activeIdSet = new Set(activeIds);
  return users.filter(user => activeIdSet.has(user.id));
}

只有这种结构才能确保 Worker 在大规模流量下安全运行，而不会发生内存溢出。
```
