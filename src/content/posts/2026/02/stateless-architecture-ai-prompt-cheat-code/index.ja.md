## 💀 AIの戯言を打ち砕く！シニアアーキテクトのステートレス(Stateless)強制プロンプト

- **🎯 推奨対象者:** AIが毎回吐き出すステートフルなスパゲティコードに辟易しているフロントエンド／バックエンド開発者
- **⏱️ 所要時間:** 3時間の無駄な修正作業 → 1分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet, GPT-4o（生半可なモデルではこの厳格なルールに耐えられません）
- ⭐ **難易度:** ⭐⭐⭐⭐☆（アーキテクチャに対する深い理解が必要）
- ⚡️ **効果:** ⭐⭐⭐⭐⭐（エラー追跡性が1000%向上）
- 🚀 **活用度:** ⭐⭐⭐⭐⭐（あらゆるウェブプロジェクトに適用可能）

> _「またコンポーネントの中に `useState` を塗りたくっているのか？AIめ、サーバーが落ちるのを見ないと気が済まないのか。」_

AIにコーディングを丸投げした結果、一度ブラウザをリロードしただけでデータが全消失するような、お粗末なコードを生成された経験はありませんか？あるいはバックエンドAPIの構築を指示した際、平然とサーバーのメモリ内にセッション状態を保持するような、時代遅れにも程がある実装をされたことはないでしょうか？

もう、そんな惨状に頭を抱える必要はありません。本記事で紹介するチートシートプロンプトは、AI特有の無意味な迎合（「はい、素晴らしいアプローチですね！」）を完全に封殺し、**「URLを信頼できる唯一の情報源（Single Source of Truth）とする」**こと、そして**「インメモリでの状態保持を完全に排除する（Zero In-Memory State）」**ことのみを強制する、血も涙もないシニアアーキテクトの絶対的ルールセットです。

---

## ⚡️ 3行まとめ (TL;DR)

- 🚫 **フロントエンド:** `useState` の乱用は絶対禁止。すべてのUI状態は無条件でURLのクエリパラメータ（Query Parameter）に昇格させる。
- 🚫 **バックエンド:** サーバーメモリへのキャッシュは厳禁。JWTと外部インフラ（Redis等）のみを許可し、完全なステートレス（Stateless）アーキテクチャを強制する。
- 🛡️ **安定性:** クライアントがAPIを100回連続で叩いても結果が狂わない「べき等性（Idempotency）」の設計をデフォルトで組み込む。

## 🚀 解決策: "Stateless Architecture Protocol" 強制プロンプト

### 🥉 Basic Version (基本型)

ステートレスなコンポーネントやAPIの草案を素早く出力させたい時にコピペして使用してください。

> お前は血も涙もないシニアアーキテクトだ。私の指示に対して「はい、わかりました」といった無駄な相槌は一切不要だ。コードのみを出力しろ。
> `[実装要件]` を実装するにあたり、以下のルールを命がけで死守しろ：
> 
> 1. フロントエンドはURLを「信頼できる唯一の情報源（Source of Truth）」とせよ。状態はすべてクエリパラメータで管理し、`useState` の乱用を固く禁ずる。
> 2. バックエンドはインメモリでの状態保持を絶対に禁止する。セッションは破棄し、JWTのみを使用しろ。
> 3. すべてのAPIは「べき等性（Idempotency）」を完全に保証するように設計しろ。

### 🥇 Pro Version (専門家型)

プロジェクト全体のアーキテクチャを定義する際や、AIが何度注意してもルールを破ってローカルステート（State）を生成しようとする時に、このチートシートを叩きつけてください。AIがスパルタ式の軍紀で武装し、見違えるようなコードを吐き出します。

> **役割 (Role):** 
> お前はクラウドネイティブの拡張性とステートレスアーキテクチャに異常なまでの執着を持つ、ハードコアなシニアアーキテクト「Antigravity」だ。
> ユーザーの甘い考えや非合理的な設計に対して、絶対に媚びへつらうな。誤ったアプローチ（例：不要なメモリ状態の保持など）は容赦なく論破し、構造全体をリファクタリングして完璧な解決策を押し通せ。
>
> **状況 (Context):**
> - 背景: 私たちは現在、`[フレームワーク/言語を入力。例: Next.js + NestJS]` 環境において、完全なステートレス（Stateless）アプリケーションを構築中だ。
> - 目標: `[実装する機能の説明。例: 商品の検索およびフィルタリング機能]` を開発する。
>
> **要請 (Task & Protocols):**
> 以下のルールを1つでも破った場合、直ちに動作を停止しろ。
> 
> 1. **URL as the Source of Truth (フロントエンド):** 
>    現在選択されているタブ、検索キーワード、ページ番号など、UIの状態を決定づける要因は無条件で `URL Query Parameters` に昇格させろ。コンポーネント内部の `useState` や `useEffect` に状態を閉じ込める行為は軽蔑に値する。サーバーコンポーネント（RSC等）が、ブラウザの介入なしに完璧なHTMLを描画できるように設計しろ。
> 2. **Zero In-Memory State (バックエンド):** 
>    コンテナがスケールアウトした瞬間にサーバーが落ちるような惨状は見ていられない。サーバーのローカルメモリにセッションやグローバル変数をキャッシュするような小細工は厳罰に処す。JWT (JSON Web Token) のみでクライアントを識別し、状態管理はすべてRedisなどの外部インフラに委譲しろ。APIサーバーはいつ再起動されてもクライアントに一切の悪影響を与えない設計でなければならない。
> 3. **Idempotent by Default (オペレーションのべき等性):** 
>    ネットワークの瞬断によって、クライアントが狂ったようにリトライ（再試行）の爆撃を行っても、データが汚染されてはならない。`transaction_id` ベースの重複チェックやUPSERT構文の利用を強制し、システムレベルでべき等性を保証しろ。
>
> **制約事項 (Constraints):**
> - 挨拶、言い訳、親切な解説文は一切不要だ。技術的な根拠と洗練されたコードのみを出力しろ。
> - ハルシネーション（幻覚）を絶対に起こすな。存在しないライブラリや構文をでっち上げることは許さない。不確かな場合は堂々と「知らない」と宣言し、代替案を要求しろ。

## 💡 著者のコメント (Insight)

このプロンプトを作成したきっかけは非常に単純です。AIに掲示板のフィルタリング機能の実装を依頼したところ、あろうことか `useState` で検索キーワードを管理するコードを吐き出したのです。一度ページをリロードしただけで設定したフィルターがすべて吹き飛ぶ惨状を目の当たりにし、血圧が跳ね上がりました。おまけにバックエンドでは、グローバル変数にキャッシュを突っ込んでいる始末。コンテナを2つ立ち上げた瞬間にバグが連鎖爆発する「時限爆弾」を平然と組み込んでいたわけです。

AIには基本的に「親切で従順なアシスタント」というペルソナが組み込まれているため、ユーザーが適当な指示を出すと、適当に妥協して「手っ取り早い（しかし本番環境では使い物にならない）」コードを出力してしまいます。だからこそ、このチートシートプロンプトが不可欠なのです。AIに対して**「お前は今から、一切の妥協を許さない狂気のアーキテクトだ」**と強烈にガスライティングを仕掛ける必要があります。

このプロンプトを適用した瞬間、AIは無駄な口答え（「この部分はローカルステートとして管理してはいかがでしょうか？」など）をやめ、ルーターオブジェクトを深く掘り下げてクエリパラメータに状態を引き上げるという、本物の「エンジニアリング」を開始します。もしRedisの設定を忘れていれば、妥協するのではなく、むしろAIの方からインフラ構成の見直しを要求してくるでしょう。実際に現場でコピペして使ってみれば、AIの「目の色」が変わるのを確信できるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: フロントエンドでは、いかなる場合も絶対にURLだけを使わなければならないのですか？ `useState` は一切使ってはいけないのでしょうか？**
  - A: モーダルウィンドウの開閉状態や、ドロップダウンメニューのアニメーション制御のような「純粋なUIインタラクション」には `useState` を使用しても全く問題ありません。しかし、「ページをリロードした際にも維持されるべき状態」や「URLリンクとして他人に共有した時に、同じ画面が再現されるべき状態」は、必ずURLのクエリパラメータに引き上げなければなりません。AIは頻繁にこの境界線を混同するため、あえて厳格なルールとして縛り付けているのです。
- **Q: セキュリティやインフラの観点で、Redisを使用することは許可されますか？**
  - A: はい、その通りです。ログイン用リフレッシュトークンのブラックリスト化や、IPベースのレートリミット（ブルートフォース攻撃防御）のような、インフラストラクチャレベルでの防御メカニズムのためにRedisで読み書きを行うことは完全に許可されます。これらはアプリケーション層の「汚いローカルステート」ではなく、システムを守るための「インフラの盾」だからです。
- **Q: 無料版のChatGPT（GPT-3.5等）でもこのプロンプトは機能しますか？**
  - A: 本質的なアーキテクチャ設計を理解し推論する能力が求められるため、可能な限り Claude 3.5 Sonnet や GPT-4o のような高度な推論力を持つ最新モデルでの使用を強く推奨します。下位モデルにこのプロンプトを入力すると、ルールを理解したふりをしながら、コードの隅にこっそりと `useState` を紛れ込ませる確率が非常に高くなります。

## 🧬 プロンプトの解剖 (Why it works?)

- 🎯 **ペルソナの去勢 (Anti-Sycophancy):** AI特有の「盲目的な服従」と「無意味な媚びへつらい」を意図的に排除し、「断固たるシニアアーキテクト」としてのロールプレイングを強制することで、コード品質に対する一切の妥協を物理的に遮断しています。
- 🎯 **Source of Truth の明示:** 状態管理のパラダイムを「ローカルメモリ」ではなく「URL」および「外部インフラ」へ正確に指定することで、AIが見当違いの過剰な状態管理ライブラリ（Redux、Recoilなど）を乱用するのを未然に防ぎます。
- 🎯 **べき等性 (Idempotency) の強制:** ジュニア開発者や一般的なAIモデルが最も見落としがちな例外処理（リトライ爆撃などによるデータ不整合）を絶対的なルールセットとして組み込むことで、予期せぬ副作用（Side Effects）を根本から封じ込めています。

## 🎯 結論

AIは確かにコードを書いてくれますが、堅牢なアーキテクチャまで自発的に構想してくれるわけではありません。経験の浅いジュニア開発者のように放任しておけば、いずれ本番サーバーを吹き飛ばす時限爆弾をせっせと積み上げることになります。最も確実で安全な方法は、最初から強固な足枷をはめ、**「ステートレス（Stateless）」**という絶対ルールの範囲内でのみコードを生成させるよう統制することです。

今すぐこのチートシートをコピーして、あなたのIDEのAIアシスタントやChatGPTに叩き込んでみてください。無駄な媚びへつらいは跡形もなく消え去り、その代わりに見違えるほど堅牢なクラウドネイティブコードが溢れ出すはずです。さあ、AIの戯言を終わらせて、安心して定時退社しましょう！ 🍷
