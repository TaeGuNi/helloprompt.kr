## 📝 Schluss mit der Suche nach Commit-Messages! Automatische Erstellung von Conventional Commits

- **🎯 Empfohlen für:** Junior-Entwickler, Open-Source-Mitwirkende, Teams am Beginn der Zusammenarbeit
- **⏱️ Zeitersparnis:** Von 5 Minuten auf 10 Sekunden verkürzt
- **🤖 Beste Performance:** ChatGPT (GPT-4o), Claude 3.5 Sonnet, Gemini 2.5 Pro

- ⭐ **Schwierigkeitsgrad:** ⭐☆☆☆☆
- ⚡️ **Effektivität:** ⭐⭐⭐⭐⭐
- 🚀 **Anwendbarkeit:** ⭐⭐⭐⭐⭐

> _"Drei Stunden Code geschrieben, 30 Minuten über die Commit-Message nachgedacht... Wollten Sie nicht auch schon mal einfach 'Update code' tippen und Feierabend machen?"_

`fix: bug`, `update code`, `wirklich_letzte_änderung`... Haben Sie heute auch wieder ein bedeutungsloses Commit-Log auf GitHub hinterlassen und sind schnell in den Feierabend verschwunden?
Oder haben Sie vor dem blinkenden Cursor des `git commit -m "..."`-Befehls gesessen und länger als fünf Minuten den Monitor angestarrt, weil Sie nicht wussten, was Sie schreiben sollen?
Drei Stunden für den Code, aber wenn es darum geht, die Commit-Message im Terminal zu verfassen, ist der Kopf plötzlich leer – eine chronische Qual, die Entwickler im Frontend wie im Backend täglich erleben.
Besonders bei globalen Projekten, die englische Nachrichten erfordern, und unter dem Druck strenger Konventionen wie **Conventional Commits**, wird das Committen nicht mehr zur bloßen Versionsverwaltung, sondern zu einer **mühsamen Schreibprüfung**.
"Wie fasse ich diese Änderung kurz auf Englisch zusammen?", "Es ist ein neues Feature, also `feat`? Oder habe ich bestehenden Code korrigiert, also `refactor`?", "Wo und wie füge ich die Issue-Nummer ein, damit kein Fehler auftritt?"
Nach all diesem Grübeln geben wir oft erschöpft auf, nutzen Google Translate oder DeepL und hinterlassen eine lielose, einfache Nachricht.

Doch solche nachlässigen Commit-Messages werden **garantiert wie ein Schneeball anwachsen und Sie sowie Ihre Kollegen in der Zukunft gnadenlos überrollen.**
Stellen Sie sich vor, in ein paar Monaten tritt in der Produktionsumgebung ein kritischer Bug auf und Sie müssen den Code dringend zurückrollen (Rollback). In dem Moment, in dem Sie vor dutzenden Logs stehen, in denen nur `fix: bug` steht, wird die Suche nach dem Ursprung des Fehlers schmerzhafter als die Suche nach der Nadel im Heuhaufen.
Noch deutlicher wird das Problem **im Prozess der Zusammenarbeit**. Wenn Kollegen Ihren PR (Pull Request) reviewen, verschwenden sie wertvolle Arbeitszeit mit dem Interpretieren des Codes, weil sie nicht verstehen, mit welcher Absicht der Code geändert wurde oder warum dieser Weg gewählt wurde.
Besonders bei Open-Source-Projekten oder Tech-Unternehmen mit einer strukturierten Code-Review-Kultur wird eine lieblose Commit-Message oft direkt mit **mangelndem Talent und Professionalität des Entwicklers** gleichgesetzt, was häufig zu schlechteren Bewertungen führt. In Zeiten automatisierter CI/CD-Pipelines führen Commit-Messages direkt zu den Release Notes; eine fehlerhafte Nachricht kann sogar den automatischen Deployment-Prozess stoppen.
Die Zeiten, in denen es reichte, nur guten Code zu schreiben, sind vorbei. **Eine perfekte Commit-Message ist die stärkste und eleganteste Verpackung, die Ihren Code glänzen lässt.**

Müssen Sie also jedes Mal im Englisch-Wörterbuch und in den Konventions-Richtlinien blättern und mühsam Sätze herausquetschen? Absolut nicht.
Sie müssen sich nicht mehr mit Übersetzern herumschlagen und Schreibstress haben. Werfen Sie dem Prompt einfach die **Änderungen (`git diff`)** Ihres mühsam erarbeiteten Codes oder eine kurze einzeilige Zusammenfassung Ihrer Arbeit hin.
Die KI erstellt Ihnen sofort eine **perfekte englische Commit-Message**, die exakt dem **Conventional Commits Standard** – dem absoluten globalen Standard des Open-Source-Ökosystems – entspricht und die Sie direkt kopieren und in Ihr Terminal einfügen können.
Dies ist kein einfaches Übersetzungsniveau. Die KI analysiert den Kontext der hinzugefügten und gelöschten Codezeilen akribisch und erstellt ein beeindruckendes Commit-Log, das sogar die grundlegende Absicht (Why) und die Lösungsmethode (What) auf den Punkt bringt, die Sie vielleicht gar nicht explizit genannt haben.

Sobald Sie diesen Prompt in Ihre tägliche Arbeit integrieren, wird sich Ihre Feierabendzeit garantiert jeden Tag um mindestens 10 Minuten verkürzen.
Sparen Sie die unnötige kognitive Energie, die für die unproduktive Aufgabe des Verfassens von Commit-Messages verbraucht wurde, und konzentrieren Sie sich stattdessen ganz auf die Implementierung der Kern-Geschäftslogik und die Architekturplanung.
Vor allem werden Sie als professioneller Ingenieur wahrgenommen, der von Kollegen, die dasselbe GitHub-Repository nutzen, Lob erntet: **"Die Commit-Logs dieser Person sind so sauber und klar. Mit so einem Senior-Entwickler arbeitet man wirklich gerne zusammen!"**
Geben Sie jetzt sofort `git diff` in Ihr Terminal ein und erleben Sie die magische Automatisierung, die den letzten Schritt Ihrer Programmierung mit dem unten stehenden Prompt krönt!

fix(auth): resolve login failure issue and update button UI

- Change the login button color to red for better visibility
- docs: Add installation guide to README.md

**Option 2 (Fokus auf UI und Stil)**
```text
style(ui): update login button color to red and fix login bug

- Change login button color to enhance UI consistency
- Fix authentication bug causing login failures
- docs: Update README with installation instructions
```

## 🚀 So schreiben echte Profis

Dieser Prompt wurde nach dutzenden Versuchen perfektioniert. Kopieren Sie den unten stehenden Prompt und füllen Sie nur den Teil `[Variable]` passend zu Ihrer Situation aus, um ihn sofort in der Praxis einzusetzen.

### 🥉 Basic Version (Basis-Version)

Verwenden Sie diese Leichtgewicht-Version, wenn Sie schnell nur eine Titelzeile benötigen.

> **Rolle:** Du bist ein Senior-Entwickler, der eine perfekte Git-Historie anstrebt.
>
> **Anfrage:** Schau dir die von mir bereitgestellten Änderungen an und schlage mir 3 Titel für englische Commit-Messages vor, die dem Conventional Commits Standard entsprechen (Format: `type(scope): subject`).

### 🥇 Pro Version (Experten-Version)

Verwenden Sie diese Version, wenn Sie eine hochwertige, strukturierte Commit-Message mit Titel, Textkörper und Fußzeile benötigen. Kopieren Sie den folgenden Prompt und fügen Sie ihn einfach in das KI-Fenster ein.

> **Rolle (Role):**
> Du bist ein strenger Maintainer und Senior Engineering Manager, der zahlreiche Open-Source-Projekte verwaltet und auf eine perfekte Git-Historie fixiert ist.
>
> **Situation (Context):**
>
> - Hintergrund: Ich habe gerade den Code fertiggestellt und muss einen Commit hinterlassen.
> - Ziel: Ich muss eine Commit-Message verfassen, die für Kollegen leicht zu reviewen ist und später eine perfekte Nachverfolgung der Historie ermöglicht.
>
> **Aufgabe (Task):**
> Analysiere die unten stehenden `[Änderungen]` und schlage 3 Optionen für einen perfekten Commit-Message-Entwurf gemäß den **Conventional Commits v1.0.0** Regeln vor.
>
> 1. **Format:** `<type>(<scope>): <subject>` (1 Zeile Titel)
> 2. **Textkörper (Body):** Fasse klar in Bullet Points (`-`) zusammen, warum (Why) die Änderung vorgenommen wurde und was (What) wie gelöst wurde.
> 3. **Fußzeile (Footer):** (Optional) Falls es eine zugehörige Issue-Nummer gibt, schreibe sie im Format `Closes #Issue-Nummer`.
>
> **Einschränkungen (Constraints):**
>
> - Als Type darf ausschließlich einer der folgenden verwendet werden: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
> - Der Subject (Titel) sollte kurz sein (unter 50 Zeichen) und zwingend mit einem **englischen Imperativ (Verb im Infinitiv)** beginnen (z. B. `Add`, `Fix`, `Update`). Setze am Ende keinen Punkt (`.`).
> - Der Textkörper (Body) wird mit einer Leerzeile Abstand zum Titel verfasst und muss auf Englisch sein.
> - Füge zu jeder Option eine kurze Erklärung der Unterschiede oder des Fokus (z. B. Fokus auf Bugfix, Fokus auf UI-Verbesserung) auf Deutsch hinzu.
>
> **Eingabe (Input):**
>
> - Änderungen: `[Geben Sie hier das git diff-Ergebnis oder die vorgenommenen Änderungen bequem auf Deutsch ein]`

## 💡 Kommentar des Autors (Insight & How to use)

Die überragende Stärke dieses Prompts liegt darin, dass er mit einem einzigen Klick zwei riesige Hürden beseitigt, die Entwickler täglich quälen: den **'psychologischen Druck beim Schreiben auf Englisch'** und die **'strengen und komplexen Formatierungsregeln'**.

Wenn wir committen, sagen wir nicht nur einfach "Ich habe die Farbe dieses Buttons auf Blau geändert". Führen Sie stattdessen den Befehl `git diff` in Ihrem Terminal aus, markieren Sie die komplexen roten und grünen Codeänderungen, die auf dem Bildschirm erscheinen, und kopieren Sie sie direkt in den Bereich `[Änderungen]` des Prompts. Erstaunlicherweise wird die KI selbstständig schlussfolgern und analysieren, welche Dateien Sie angefasst haben, welche Variablennamen geändert wurden und welchen Kontext diese Änderung im gesamten System hat. Sie vervollständigt eine Commit-Message in einer Qualität, die sogar die verborgenen Absichten und Logikänderungen aufgreift, die der Entwickler vielleicht gar nicht in Worten erklärt hat.

<b>💡 Praxisnahe Steuerung durch Variablen (Constraint Control):</b>
Der Grund, warum dieser Prompt eine ganz andere Dimension als ein gewöhnlicher Chatbot-Dialog hat, ist die extreme Kontrolle über die Halluzinationen und die Freiheit der KI durch **Einschränkungen (Constraints)**. Conventional Commits haben überraschend komplizierte Regeln. Zum Beispiel muss die Nachricht zwingend mit einem **Verb im Infinitiv (englischer Imperativ)** beginnen, und ein Punkt (`.`) am Ende des Titels führt zu einem Lint-Fehler. Außerdem wird die Nachricht auf GitHub abgeschnitten, wenn die Gesamtlänge 50 Zeichen überschreitet.
Dieser Prompt wurde sorgfältig so konzipiert, dass all diese potenziellen Fehlerelemente, die ein strenger Senior-Entwickler im Code-Review bemängeln würde, präventiv blockiert werden. Wenn Ihr Team ein spezielles Präfix oder eine Jira-Ticketnummer erfordert, müssen Sie nur eine einzige Bedingung zu den Einschränkungen hinzufügen: `- Setze zwingend eine Jira-Ticketnummer im Format '[ABC-123]' an den Anfang des Titels`. Die KI wird die bestehenden strengen Regeln beibehalten und Ihre teaminternen Konventionen perfekt integrieren.

<b>🚀 Profi-Tipp für die Editor-Integration für maximale Produktivität:</b>
Es gibt ein echtes Geheimnis, um den Wert dieses Prompts um 200 % zu steigern. Es wird irgendwann lästig, für jeden Commit den Browser zu öffnen und ChatGPT oder Claude aufzusuchen. Speichern Sie den gesamten Inhalt dieses Prompts als Vorlage in den **System Prompt (System-Prompt)** Einstellungen von Tools wie **Cursor IDE**, **GitHub Copilot Chat** oder Terminal-basierten KI-CLI-Tools, die unter Entwicklern derzeit extrem beliebt sind.
Nachdem Sie mit dem Programmieren fertig sind, müssen Sie der Editor-internen KI nur noch einen kurzen Befehl geben: **"Schreib mir eine Commit-Message für die gerade vorgenommenen Änderungen."** Fertig. Der Editor liest den aktuellen `git diff`-Status selbstständig aus und spuckt in einer Sekunde eine Commit-Message aus, die perfekt auf diesen Prompt-Regeln basiert. Sie müssen nur noch auf `Accept` klicken oder sie kopieren. Allein durch diese Verkürzung des Workflows werden Ihre tägliche Entwicklungsproduktivität und Ihre Konzentrationsfähigkeit massiv ansteigen.

## 🙋 Häufig gestellte Fragen (FAQ)

- **Q: Wir haben eigene firmeninterne Commit-Regeln (z. B. Jira-Ticketnummer muss enthalten sein). Wie wende ich das an?**
  - A: Wenn Sie im Abschnitt **Einschränkungen (Constraints)** des Prompts eine Bedingung wie `- Setze zwingend eine Jira-Ticketnummer im Format '[ABC-123]' an den Anfang des Titels` hinzufügen, erhalten Sie Ergebnisse, die die spezifischen Konventionen Ihres Teams perfekt erfüllen.

- **Q: Ich möchte den Titel auf Englisch, aber den Textkörper (Body) der Commit-Message auf Deutsch verfassen. Geht das?**
  - A: Ändern Sie einfach den Satz im Prompt `Der Textkörper (Body) wird mit einer Leerzeile Abstand zum Titel verfasst und muss auf Englisch sein.` in `Verfasse den Textkörper klar und deutlich auf Deutsch.` So erhalten Sie ein praktisches Format, das die äußere Hülle des globalen Standards beibehält, aber die Kommunikationseffizienz mit Ihren Teammitgliedern maximiert.

## 🧬 Anatomie des Prompts (Why it works?)

1. **Erzwingung des Standard-Specs (Conventional Commits v1.0.0):** Anstatt die KI vage zu bitten, eine "gute Nachricht zu schreiben", wird durch die direkte Anweisung eines weltweit anerkannten Standards das Format der Ausgabe fest fixiert.
2. **Klare Einschränkungen (Constraints):** Strenge Lint-Regeln wie "Beginn mit Imperativ", "50-Zeichen-Limit" und "Kein Punkt", die sonst ein Senior-Entwickler bemängeln würde, werden bereits auf Prompt-Ebene abgefangen. Dadurch sinken die Kosten für nachträgliche Korrekturen auf Null.
3. **Wahlmöglichkeit (3 Optionen):** Der Prompt ist so konzipiert, dass die KI nicht eine einzige "richtige" Antwort erzwingt. Durch das gleichzeitige Vorschlagen von 3 verschiedenen Optionen kann der Benutzer die Nachricht auswählen, die am besten zum Kern seiner beabsichtigten Arbeit passt.
