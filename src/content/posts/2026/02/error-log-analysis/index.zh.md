## 💡 作者点评 (Insight & How to use)

这个提示词是我在实务中遇到烦人的 `NullPointerException`、完全看不出原因的 Webpack 构建错误，或者是运行时突然爆发的内存泄漏（Memory Leak）时，每次都会拿出来的**最强大、最可靠的调试作弊码**。过去看到几百行的堆栈跟踪会感到恐慌，现在反而觉得错误日志越长越详细，“AI 就越好分析”，从而感到安心。

仅仅问一句“这个错误是什么？怎么修？”与赋予 **“10年经验高级工程师”** 明确人格后再要求 “分3步明确给出解决方案” 这种强制输出格式，在结果的质量上简直是天壤之别。因为被赋予角色的 AI 不仅会翻译错误的表面现象，还会基于框架的内部运作原理和最佳实践（Best Practice）推导出**最优雅、最安全和最合适**的解决方案。

<b>🔥 变量控制及实战技巧 (Constraint Control)</b>

1. <b>`[角色]` 变量的精细调整：</b>
尝试将提示词中的 `[10年经验高级后端开发人员]` 部分修改为更贴合你当前环境的角色。例如，如果是前端 React 环境中发生的渲染错误，可以改为 `[精通 React 渲染优化及故障排除的10年经验前端高级开发人员]`。如果是 AWS 云基础设施部署中发生的错误，请赋予 `[AWS DevOps 及基础设施构建专家]` 角色。人格越具体，回答的解析度就会呈几何级数提高。

2. <b>日志太长导致截断时的应对法：</b>
企业级应用程序的错误日志有时长达数万行。如果一次性全部复制进去，可能会超过 AI 的上下文窗口（Context Window）限制，或者导致中间的重要信息被遗漏（Lost in the middle）。这时，请以<b>堆栈跟踪的最顶部（通常是最初发生点）</b>和位于最底部的 <b>`Caused by:` 语句</b>为中心，摘取核心的 100~200 行左右输入即可。即使果断省略掉无意义重复的 Spring 或 Next.js 内部框架调用日志，AI 也能天才般地把握全局语境。

3. <b>提供额外的上下文（周边源代码）：</b>
如果你想要最完美的调试结果，不要只给错误日志。如果在提示词下方附上日志中指出的报错文件（如 `UserService.java`）的<b>实际源代码内容（整个方法）</b>，奇迹就会发生。AI 会交叉验证错误日志和实际代码，从而返回 100% 完美契合你项目情况的准确修改代码。

4. <b>最佳 AI 模型选择指南：</b>
在分析复杂的系统错误或庞大的代码上下文时，我个人最强烈推荐 <b>Claude 3.5 Sonnet</b> 模型。Sonnet 模型阅读代码并追踪逻辑流的能力目前领先于市面上任何其他模型。GPT-4o 也是一个极佳的选择，但在捕捉框架内部运作中微妙的语义错误方面，Claude 特有的细致往往能大放异彩。

5. <b>防止幻觉的约束条件魔力：</b>
提示词下方注明的“如果不能 100% 确定原因，请反过来询问”这句话看似简单，却起到了核心的<b>安全护栏（Safety Guardrail）</b>作用。AI 模型天生有一种必须生成答案的强迫症，当信息不足时，存在编造看似合理的谎言（Hallucination）从而让开发人员陷入更大混乱的风险。通过赋予这一约束条件，AI 不会强行给出错误答案，而是会转变为一名合格的助手，给出调试方向，例如“能否让我看看数据库连接配置文件（application.yml）的内容？”。

## 🙋 常见问题 (FAQ)

- **Q: 如果日志中包含公司内部安全日志或 API 密钥怎么办？**
  - A: **（非常重要）** 绝对不能原封不动地复制粘贴日志！在运行提示词之前，请务必在记事本或编辑器中将 API 密钥、密码、敏感客户个人信息、内部服务器 IP 等使用 `***` 或 `[脱敏]` 处理安全替换后再使用。虽然有点麻烦，但安全性是不可逾越的第一优先级。

- **Q: AI 总是给出不符合语境的离谱方案，该怎么办？**
  - A: 这种情况通常发生在仅凭错误日志本身不足以判断原因的语境下。请尝试在错误日志下方添加提示，并附上发生问题周边的**实际源代码**（如整个方法）。你会发现 AI 的分析准确度会有惊人的提升。

1. **高级工程师人格 (Role):** 强制激活 AI 内在潜伏的专家级技术知识，引导出超越平庸、肤浅回答的深入且有洞察力的技术分析。
