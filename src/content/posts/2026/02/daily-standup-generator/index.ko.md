# 🗣️ "어... 어제 뭐 했더라?" 스탠드업 공포증 해결

- **🎯 추천 대상:** 애자일(Agile) 조직의 개발자, 기획자, 디자이너 등 모든 실무자
- **⏱️ 소요 시간:** 스탠드업 준비 15분 → 단 1분으로 단축
- **🤖 추천 모델:** 모든 대화형 AI (ChatGPT, Claude, Gemini 등)

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"매일 아침 10시, 데일리 스크럼 시간이 다가오면 줌(Zoom) 마이크를 켜기가 두려우신가요?"_

데일리 스탠드업(Daily Standup)은 팀원들에게 내 진행 상황을 짧고 굵게 공유하는 자리입니다. 하지만 막상 내 차례가 되면 어제 한 일은 기억나지 않고, 불필요한 변명이나 TMI(Too Much Information)만 늘어놓게 되는 경우가 많죠. 이 프롬프트는 당신의 두서없는 메모를 **'성과 중심의 프로페셔널한 스탠드업 스크립트'**로 1분 만에 변환해 줍니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **자동화된 성과 포장:** 고생한 과정(삽질)보다 해결한 결과(성과)에 초점을 맞춰 문장을 재구성합니다.
2. **완벽한 포맷 통일:** 팀원들이 가장 듣기 편안한 어제 한 일(Yesterday) / 오늘 할 일(Today) / 방해 요소(Blockers)의 표준 구조를 강제합니다.
3. **명확한 도움 요청:** 감정적인 호소가 아닌, 문제 해결을 위해 정확히 누구의 어떤 도움이 필요한지 객관적으로 명시해 줍니다.

---

## 🚀 해결책: "Standup Speaker"

### 🥉 Basic Version (기본형)

슬랙(Slack), 팀즈(Teams) 등 메신저에 텍스트로 가볍게 진행 상황을 툭 던져야 할 때 유용합니다.

> **역할:** 너는 능숙한 스크럼 마스터야.
>
> **요청:** 내가 쓴 메모를 바탕으로 슬랙에 공유할 깔끔한 데일리 스탠드업 메시지를 작성해 줘.
>
> **내 메모:**
>
> - 어제: 로그인 API 수정, DB 스키마 변경, 팀 회식
> - 오늘: 회원가입 페이지 연동, 디자인 팀 미팅
> - 이슈: AWS 서버 접속 불안정

<br>

### 🥇 Pro Version (전문가형)

화상 회의에서 직접 말로 발표해야 하거나, 리더급이 참석하는 회의에서 더 프로페셔널하고 주도적인 인상을 남기고 싶을 때 사용하세요.

> **역할 (Role):** 너는 실리콘밸리 애자일 팀의 커뮤니케이션 능력이 뛰어난 시니어 개발자야.
>
> **상황 (Context):**
>
> - 나는 어제 하루 종일 원인을 알 수 없는 에러와 씨름하며 바쁘게 보냈지만, 겉으로는 큰 진척이 없어 보일까 봐 걱정이야.
> - 변명이나 불평처럼 들리지 않고, 문제를 주도적으로 해결한 '전문가'처럼 보이고 싶어.
>
> **입력 데이터 (My Notes):**
>
> - 어제: 결제 모듈 버그 잡느라 4시간 씀. 원인은 외부 API 타임아웃이었음. 결국 재시도 로직 넣어서 임시 해결함.
> - 오늘: 어제 해결한 거 배포하고 모니터링 할 예정. 오후엔 기획팀이랑 요건 리뷰 미팅 있음.
> - 방해 요소: 지금 테스트 서버가 너무 느려서 QA가 자꾸 밀림. 인프라 팀의 확인이 필요함.
>
> **요청 (Task):**
>
> 1. 위 데이터를 바탕으로 1분 안에 읽을 수 있는 **구두 발표용 대본(Script)**을 작성해 줘.
> 2. "어제 ~하느라 고생했다"는 식의 감정적인 표현은 철저히 배제하고, "외부 API 타임아웃 문제를 파악하고 재시도 로직을 적용해 방어 코드를 구축했다"는 **성취(Achievement) 및 문제 해결 중심**으로 문장을 재구성해.
> 3. Blockers(방해 요소)는 정중하지만 단호하게 인프라 팀의 지원을 요청하는 뉘앙스로 작성해 줘.
>
> **제약사항 (Constraints):**
>
> - 출력 형식은 마크다운을 사용해 아래 구조를 엄격히 지켜줘.
>   - **[어제 한 일]**: (핵심 성과 위주, 2문장 이내)
>   - **[오늘 할 일]**: (구체적 계획, 2문장 이내)
>   - **[도움이 필요한 부분 (Blockers)]**: (행동 촉구형 요청)
> - 구어체(~했습니다, ~할 예정입니다)로 자연스럽게 읽히도록 작성해.

## 💡 작성자 코멘트 (Insight)

스탠드업 미팅은 '나 어제 이렇게 야근하고 고생했어요'를 증명하는 보고 자리가 아닙니다. **'팀의 목표 달성을 위해 내가 무엇을 뚫어냈고, 지금 어디서 병목(Bottleneck)이 발생하고 있는지'를 싱크(Sync)하는 자리**입니다.

많은 주니어들이 자신이 겪은 어려움(과정)을 장황하게 설명하는 실수를 범합니다. 이 프롬프트의 핵심은 AI를 필터로 사용하여 **나의 감정과 피로를 덜어내고, 오직 팩트와 성과, 그리고 실행 가능한 다음 단계(Action Item)만 남기는 것**입니다. 이렇게 정제된 스탠드업을 며칠만 반복해도, 동료와 리더는 당신을 '문제 해결에 집중하는 프로페셔널'로 인식하게 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 비개발 직군(마케터, 디자이너)도 이 프롬프트를 쓸 수 있나요?**
  - A: 물론입니다. Pro 프롬프트의 '역할(Role)' 부분만 "유능한 시니어 마케터" 혹은 "시니어 프로덕트 디자이너"로 변경하고, 본인의 업무 메모를 넣으면 직군에 맞는 완벽한 스크립트가 나옵니다.

- **Q: 스크립트가 내 평소 말투와 너무 다르게 나오면 어떡하죠?**
  - A: 프롬프트 제약사항(Constraints)에 `평소 내 말투: "~했고요, ~할 생각입니다" 스타일로 친근하게 써줘`라는 조건을 추가하면 훨씬 자연스러운 결과물을 얻을 수 있습니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **상황(Context)의 구체화:** "겉으로 진척이 없어 보일까 봐 걱정"이라는 심리적 제약을 AI에게 인지시킴으로써, AI가 문장을 방어적이지 않고 능동적인 톤으로 재구성하게 만들었습니다.
2. **감정 배제와 성과 치환:** 고생한 시간(4시간 삽질)을 해결 능력(재시도 로직 구축)으로 치환하도록 명시적으로 지시하여 보고의 질을 높였습니다.
3. **행동 촉구형(CTA) Blockers:** 단순히 "서버가 느려요"가 아니라, "인프라 팀의 확인이 필요합니다"라는 명확한 행동 방향성을 제시하게 만들어 회의의 효율성을 극대화합니다.

## 📊 증명: Before & After

### ❌ Before (입력 메모)

```text
어제 결제 오류 나서 하루 종일 고생했어요. 외부 API가 자꾸 끊기더라고요. 그래서 일단 급한 대로 재시도하는 코드 넣어놨습니다. 오늘은 이거 서버에 올리고 잘 돌아가나 봐야 되고요. 오후엔 기획팀 미팅 있습니다. 아, 그리고 지금 테스트 서버 너무 느려터져서 일하기 힘든데 인프라팀 확인 좀 해주세요.
```

### ✅ After (AI 변환 스크립트)

```text
**[어제 한 일]**
어제는 결제 모듈에서 발생한 간헐적 오류를 디버깅하는 데 집중했습니다. 외부 API 타임아웃이 원인임을 파악하여, 시스템 안정성을 위해 재시도 로직을 적용해 문제를 임시 해결했습니다.

**[오늘 할 일]**
오늘은 어제 수정한 결제 로직을 배포하고 모니터링을 진행할 예정입니다. 오후에는 기획팀과 신규 요건 리뷰 미팅이 잡혀 있습니다.

**[도움이 필요한 부분 (Blockers)]**
현재 테스트 서버의 응답 지연이 심해 QA 일정에 병목이 생기고 있습니다. 테스트 환경 정상화를 위해 인프라 팀에서 이슈를 확인해 주시면 큰 도움이 되겠습니다.
```
