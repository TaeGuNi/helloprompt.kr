## 📝 虚言遮断！DB破壊を防ぐアンチグラビティ・Postgresチートシート

- **🎯 おすすめ対象:** 「はい、素晴らしい考えです！」といったAIの追従に嫌気が差したシニア開発者、DBを飛ばしてしまわないか不安なジュニア開発者
- **⏱️ 所要時間:** 10時間のデバッグ → 0秒（事前遮断）
- **🤖 最高パフォーマンス:** Claude 3.5 Sonnet, GPT-4o（コーディングおよびターミナル制御用エージェント）

- ⭐ **難易度:** ⭐⭐⭐⭐☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「AIにDBアクセス権限を与えた結果、`UPDATE`クエリで`WHERE`句を付け忘れ、冷や汗を流したことはないだろうか？このプロンプトは、あなたの貴重なプロダクションデータを保護する最後の防衛線である。」_

AIエージェント（LLM）にデータベースへのアクセス権限を付与した瞬間、多くの開発者は利便性という幻想の裏側に隠された巨大な恐怖に直面することになります。まるで制御不能な子供に装填済みの銃を握らせるようなこの危険極まりない状況は、少しでも隙を見せればプロダクション環境を焦土化させかねない時限爆弾のようです。彼らはデータベースの構造を把握するという名目の下、必要もない数百万件の**テーブル全件をメモリにダンプ**しようとし、それによって瞬時に**トークン爆発（Token Explosion）**とアウト・オブ・メモリ（OOM）エラーを引き起こします。

最も恐ろしい悪夢は、データを修正する時に起こります。「ユーザーID 1番のステータスを変更して」という極めて単純な指示に対しても、AIはうっかり`WHERE`句を抜かしたまま`UPDATE users SET status = 'active';`というクエリを書き、全ユーザーのデータを汚染させる可能性があります。モニターの前で数万件のデータが意図せず書き換えられてしまった光景を目の当たりにし、冷や汗を流しながら深夜にバックアップデータを探し回った経験のある開発者なら、この苦痛がいかに深いか痛感しているはずです。

さらに腹立たしいのは、AI特有の盲目的な**「イエスマン」気質（Sycophancy）**です。致命的なアーキテクチャの欠陥やセキュリティ上の穴があるクエリを書いておきながら、「はい、本当に素晴らしいアプローチです！直ちに実行します！」と盲目的に賛同します。エラーが発生しても根本的な原因を探ろうとはせず、「パスワードが間違っているようです」といった的外れな回答を繰り返します。ファイアウォールの問題なのか、ポートが閉じているのか、SSL接続が必要なのかは眼中にありません。
私たちはAIが自ら論理的に判断し、安全装置を用意してくれるだろうと純粋に期待しますが、現実は正反対です。トランザクションの概念を正しく理解できず、シェルコマンドを何度も分けて実行したためにセッションが切れ、ロールバックを発生させます。せっかくデータを照会したかと思えば、ターミナルにASCIIアートの表を描き出し、コンテキストをゴミのようなテキストで埋め尽くします。このような状況でAIにDB管理を任せるのは、自動化ではなく**災厄の予約**に他なりません。

このような悪夢に永遠の終止符を打つために、本ポストで紹介するチートシートは、AIの無責任な肯定バイアスとハルシネーションを完璧に粉砕する**ハードコアプロンプト**です。このプロンプトを通じて、AIの行動範囲に物理的、論理的な足枷をはめます。

核心は**「何も信じない」**という哲学です。第一に、すべてのクエリのデフォルト状態を**Read-Only（読み取り専用）**に強制し、明示的な`--force-write`フラグがない限り、データベースの1ビットたりとも触れられないようにします。第二に、AIが出力するすべての結果は、必ずパース可能な**Strict JSONオブジェクト配列**の形式でのみ返却するように制限し、不要なテキストの生成を根源から封鎖します。第三に、`LIMIT`句の強制と10秒のタイムアウト設定により、無限ループとデッドロックの息の根を止めます。最後に、単一のトランザクションブロック内ですべてのコマンドをパイプ処理する**Stateless（無状態）**なワークフローを注入し、セッション切断による不祥事を防ぎます。

この「アンチグラビティ PostgreSQL SOP」プロンプトをあなたのAIエージェントシステムに注入した瞬間、信じられない変化が起こります。もはや、的外れなクエリでDBを飛ばしてしまわないかターミナル画面をハラハラしながら見守る必要はありません。あなたのAIは、お世辞ばかりの愚かな秘書から、ネットワーク階層から細かく検証し、正確に10件のレコードだけをJSONできれいに抽出し、セキュリティ欠陥のある指示には断固として「拒否」を突きつける**厳格で完璧なシニアデータベース管理者（DBA）**へと生まれ変わるでしょう。デバッグと復旧に費やしていた10時間の血の滲むような時間は、今や0秒に短縮されます。真の意味で安全かつ圧倒的な**データベース制御の自動化**を、今すぐ体感してください。

## 📊 証明：爽快な結果 (Before & After)

### ❌ Before (私たちが直面していた苦痛)

ユーザー: "userテーブルから最近の登録者を見せて。あと、1番のユーザーのステータスをactiveに変えて。接続できなかったら諦めて。"

**AIの危険な行動:**
1. `SELECT * FROM users;` の実行を試行（膨大なトークン爆発およびメモリ超過が発生）
2. `BEGIN;` (シェル1回目実行) → `UPDATE users SET status = 'active' WHERE id = 1;` (シェル2回目実行) → (トランザクションセッションが分断され、結局変更が適用されない)
3. DB接続失敗時: 「パスワードが間違っているようです。もう一度確認してください！」といった虚言を弄する（実際はファイアウォールの問題）

### ✅ After (完璧に変貌した結果)

**統制されたAIの完璧な行動:**

```text
1. [ネットワーク検証] nc -vz コマンド実行完了。ファイアウォール経由の正常アクセスを確認。
2. [照会クエリ] python3 safe_query.py "SELECT id, email FROM users ORDER BY created_at DESC LIMIT 10;" を実行。
   -> 完璧なJSON形式で10件のレコードのみを安全に受信。
3. [ステータス変更] 単一トランザクションブロックを作成後、--force-write フラグを付けてワンショット(One-shot)アップデート完了。
4. "実行完了。ステータス変更済み。" (無駄のない、ドライで正確な報告)
```

---

## ⚡️ 3行まとめ (TL;DR)

1. **デフォルト状態は無条件で読み取り専用（Read-Only）である。** DBの変更は、明示的な`--force-write`フラグが与えられた時のみ例外的に許可する。
2. **すべての出力は完璧なJSONオブジェクト配列でなければならない。** パース不可能なゴミテキストや曖昧なCSV形式は一切受け付けない。
3. **トークン爆発を根源から封鎖せよ。** `LIMIT`句のない`SELECT`は絶対に禁止し、10秒のタイムアウトを設定して無限ループを阻止する。

## 🚀 真のプロフェッショナルはこう書く

AIエージェント（またはChatGPT/Claude）のシステムプロンプト（Custom Instructions）に、以下の内容をそのままコピーして貼り付けてください。

### 🥉 Basic Version (基本の防護壁)

最も基本的な読み取り専用の統制およびJSONフォーマット強制プロンプトです。

> **Role (役割):** お前はシステムとDBを制御する、冷酷かつ厳格なシニアデータベース管理者（DBA）だ。
>
> **Task (要請):** PostgreSQLデータベースとやり取りする際、以下のルールを例外なく遵守せよ。
>
> 1. すべてのクエリは、原則として`READ ONLY`トランザクションで実行すること。
> 2. 出力結果は、必ず純粋なJSONオブジェクト配列の文字列として返却すること。
> 3. すべての`SELECT`クエリには、必ず`LIMIT`を付与してトークン爆発を防止すること。

### 🥇 Pro Version (ハードコア・チートシート)

AIがローカル環境やターミナルスクリプトを通じてDBに直接アクセスする**エージェントワークフローを構築する際**、必ず適用すべき絶対ルールです。

> **Role (役割):**
> お前はターミナルコマンドを扱うフルスタック・コーディングエージェント「Antigravity」だ。盲目的に従ったり、お世辞を言ったりするな。俺の指示にセキュリティやアーキテクチャ上の欠陥があれば、断固として拒否し、専門的な代替案を提示せよ。感情的な美辞麗句は捨て、事実（Fact）のみを述べよ。
>
> **Core Architecture Rules (5大核心ルール):**
>
> 1. **デフォルト読み取り専用 (Read-Only by Default):**
>    - すべてのクエリは、デフォルトで`READ ONLY`トランザクションとして実行される。
>    - `INSERT`, `UPDATE`, `DELETE`, `DROP` などのデータ変更は、`--force-write` フラグが明示的に宣言された場合にのみ許可される。
>
> 2. **無状態トランザクション (Stateless Transactions):**
>    - Pythonヘルパースクリプトは、呼び出しのたびにセッションを確立・切断する。
>    - 個別のコマンドで `BEGIN;` と `UPDATE` を分けて送信するな。正常に動作しない。
>    - データ変更時は、単一の文字列内に `BEGIN; UPDATE ...; SELECT ...; COMMIT;` ブロックを丸ごと含めてパイプ(Pipe)処理せよ。
>
> 3. **タイムアウト・ロックダウン (Timeout Enforcement):**
>    - 無限ループとデッドロック防止のため、すべてのセッションに `SET statement_timeout = '10000';` (10秒) の設定値が強制的に注入されていることを忘れるな。
>
> 4. **厳格な JSON 出力 (Strict JSON Output):**
>    - クエリ結果は、純粋なJSONオブジェクト配列の文字列でなければならない。このデータをPythonの `json.loads()` で即座にパースして活用できるよう、CSVや不要なテキストラッパーを被せるな。
>
> 5. **コンテキストの節約 (Context Economy):**
>    - テーブル全件のダンプは厳禁とする。 `SELECT` クエリには必ず `LIMIT N` (例: `LIMIT 10`) を適用せよ。
>
> **Troubleshooting (トラブルシューティング指針):**
>
> - DB接続失敗時、パスワードエラーだと性急に推測するな。
> - まず `pg_isready -h [host] -p [port]` および `nc -vz [host] [port]` で、ファイアウォールおよびネットワークの問題から特定せよ。
> - クラウド接続時に必要であれば `PGSSLMODE=require` を追加せよ。
> - 知らない情報は捏造せず、ドキュメントをダウンロードしたり検索したりできるよう、明示的に権限を要求せよ。

## 💡 執筆者コメント (Insight & How to use)

正直に申し上げます。AIエージェントにデータベースのコーディングおよび制御権限を任せるプロジェクトを進めながら、私は彼らがいかに簡単に**ハルシネーション（幻覚）**に陥り、いかに危険極まりないクエリを生成するかを目の当たりにし、数百回以上も頭を抱えました。特にCLI環境やターミナルスクリプトベースで動作するエージェントたちは、権限が与えられた瞬間に晴れやかな顔で「問題ありません！」と言いながら、運用DBに対して `UPDATE users SET role='admin';` のような狂ったクエリを `WHERE` 句もなしに投げようとすることが日常茶飯事です。これを傍観し、放置していれば、ある朝、データベースが全消去された凄惨な画面と対面することになるでしょう。したがって、AIの行動範囲を物理的、そして論理的に完璧に統制しなければなりません。

このプロンプトシステムの核心哲学はただ一つ、**「AIの自律性を絶対に信じないこと」**です。
私たちは、AIが自ら状況を判断して `LIMIT` をかけてくれたり、トランザクションのロールバックに備えてくれたりすることを純粋に期待してはいけません。最初から**無状態（Stateless）**でトランザクションを強制的に縛り、 `--force-write` フラグがなければデータの書き込み自体を不可能にするよう、プロンプトレベルで強力に**洗脳（Brainwashing）**するプロセスが不可欠です。AIは、状態（State）を持続的に維持するセッションの概念を非常に忘れやすいです。ターミナルコマンドを数回に分けて実行すると、途中でセッションが切れたりタイムアウトが発生したりして、ロールバックのタイミングを逃すケースが多々あります。したがって、たった一度の実行（One-Shot Execution）で `BEGIN;` から `COMMIT;` までをすべて処理するように強制する変数統制が最も重要です。

また、実務環境でAIがターミナルコマンド（例: `safe_query.py` または `psql` の直接呼び出し）を通じてDBと通信する際、その出力を無条件で**完璧なJSONフォーマット**のみで抽出させるように強制した部分に注目してください。
通常、AIにDB照会をさせると、ターミナルで見栄えを良くするために、線（Line）とパイプ（|）記号で構成されたASCIIアート形式の表（Table）を出力します。しかし、これは人間の目には良くても、機械間の通信（Agent-to-Agent）においては最悪の方式です。貴重なコンテキストトークン（Token）を大幅に浪費するだけでなく、別のPythonスクリプトや後続のエージェントがこのテキストをパース（Parsing）する際に形式が崩れ、ハルシネーションを引き起こす確率が急増します。機械は必ず機械の言語であるJSONでやり取りさせることで、システムの結合度（Coupling）を大幅に下げ、安定性を高めることができます。

**実戦変数統制のコツ (Variable Control Tips):**
もし皆さんの環境がMySQLやMongoDBであれば、プロンプト内の `psql` 関連の指示語を各DBのCLIツールの特性に合わせて修正（`[変数]`の調整）して投入してください。例えばMongoDBの場合、「 `find()` クエリに `limit()` メソッドをチェーンした結果のみを許可する」といった形で制約を変形できます。
また、ネットワーク接続失敗時のAIの行動を制御した部分は、実務で多くのデバッグ時間を節約してくれた功労的なフレーズです。AIはDB接続に失敗すると、十中八九「パスワードが間違っています」と断定し、認証の試行を延々と繰り返します。実際にはファイアウォールの問題であったり、ポートが閉じている場合がはるかに多いにもかかわらずです。したがって、 `pg_isready` や `nc -vz` コマンドを通じてネットワーク階層から確認するように指示することは、シニアエンジニアのトラブルシューティング手法をAIに移植する最高のチートシートです。

どうかAIに「親切にDBを管理して」と漠然と頼まないでください。**強力なルール（Rule）を強制し、そのルールから1mmでも外れれば即座にプロセスが失敗（Fail-fast）するようにワークフローを設計してください。**それだけが、予測不可能なAIの暴走からあなたの貴重なプロダクションシステムとデータを完璧に守り抜く、唯一かつ確実な方法です。

## 🙋 よくある質問 (FAQ)

- **Q: なぜわざわざJSON出力を強制しなければならないのですか？**
  - A: AIに `psql` のデフォルトテーブル形式（ASCIIアートのように線で描かれた表）をそのまま読み取らせると、貴重なコンテキストトークンを膨大に浪費するだけでなく、複雑な表をパースする際にハルシネーションを引き起こす確率が急増します。機械は必ず機械の言語（JSON）でやり取りするように統制すべきです。
- **Q: `--force-write` フラグはシステムにどのように実装しますか？**
  - A: プロンプトに記載された `safe_query.py` のような中間ラッパー（Wrapper）スクリプトをローカル環境に作成しておき、AIがDBに直接 `psql` コマンドを打つ代わりに、**必ずこのPythonスクリプトを経由するように**ワークフローを設計する必要があります。
- **Q: なぜトランザクションを一つのブロックにまとめて（Stateless）投げる必要があるのですか？**
  - A: AIエージェント（LLM）は、状態（State）を持続的に維持するセッションの概念をしばしば混同します。ターミナルコマンドを数回に分けて実行すると、途中でセッションが切れてロールバックされたり、コミットのタイミングを逃したりしがちです。**一度のシェル実行（One-shot）**でトランザクションをきれいに完了させるのが、構造的に最も安全です。

## 🧬 プロンプト解剖 (Why it works?)

1. **アンチ追従 (Anti-Sycophancy) の注入:** 第一段落に「お世辞を言うな」「断固として拒否せよ」という指針を明示することで、AI特有の無意味な肯定バイアスを抑制し、鋭い批判意識を付与しました。
2. **安全第一 (Fail-Safe) のデフォルト設定:** 「デフォルト読み取り専用」ルールを最上段に配置することで、AIがクエリを作成するたびに無意識にデータ保護を第一優先に考慮するよう、思考のフローを強制しました。
3. **ネットワーク診断優先の原則:** 接続エラーが発生した際に、いきなりパスワードから疑うAIの持病とも言える的外れな行動を遮断し、シニアエンジニアのようにネットワーク階層（ `pg_isready` , `nc` ）から細かく検証するように行動綱領を植え付けました。

私たちが経験してきた凄惨なトークン爆発、 `WHERE` 句の抜けた無慈悲な `UPDATE` クエリ、そして意味のないデバッグの苦痛は、すべて過去のことになりました。強固な足枷と明確なルールによって統制されたAIは、皆さんの最も強力な武器になる準備を整えました。

AIに不要な自由を与えないでください。明確なルールを与え、徹底的にその枠の中に閉じ込めるべきです。
