## ⚡️ 3 главных преимущества (TL;DR)

1. **Устранение «узких мест»:** Идеально реализуйте фронтенд-логику и UI, не дожидаясь бэкенда.
2. **Реалистичная тестовая среда:** Симуляция задержек сети (Delay) и случайных ошибок для проверки спиннеров загрузки и UI обработки ошибок.
3. **Скопируй и запусти:** Никаких сложных настроек. Получите код, который запускается одной командой `node server.js`.

### 🥇 Продвинутая версия (Pro Version)

Используйте для создания реалистичного окружения: со случайными задержками, тестированием ошибок и массивами фейковых данных.

> **Роль (Role):** Ты — `[Senior Backend Developer]` и `[Эксперт по настройке тестовых сред]`.
>
> **Контекст (Context):**
>
> - Ситуация: `[Разработка бэкенд API задерживается, из-за чего блокируется тестирование логики на фронтенде]`.
> - Цель: `[На основе спецификации API создать идеальный Mock-сервер, который работает максимально приближенно к реальности]`.
>
> **Задача (Task):**
> Опираясь на спецификацию API ниже, напиши полный и готовый к запуску код Mock-сервера.
>
> **Технологический стек:**
>
> - Язык: `[TypeScript / JavaScript]`
> - Фреймворк: `[Выбери Express.js / Hono / Next.js Route Handler и т.д.]`
>
> **Спецификация API:**
>
> 1. GET /users: Возвращает список пользователей (включая пагинацию, поля id, name, email)
> 2. GET /users/:id: Детальная информация о конкретном пользователе
> 3. POST /users: Регистрация нового пользователя (включи пример ошибки 400 при провале валидации)
> 4. DELETE /users/:id: Удаление пользователя (включи пример ошибки 403 при отсутствии прав)
>
> **Обязательные требования:**
>
> 1. **Фейковые данные:** Используй `faker.js` или сгенерируй 10-20 реалистичных захардкоженных записей.
> 2. **Симуляция задержки (Latency):** Добавь случайную задержку от 500ms до 1500ms, как в реальной сети (для тестирования лоадеров).
> 3. **Внедрение ошибок (Error Injection):** Настрой логику так, чтобы с вероятностью 5-10% возвращалась ошибка 500 Internal Server Error (для тестирования UI ошибок).
> 4. **Формат кода:** Сделай так, чтобы весь код находился в одном файле и запускался одной командой (`node server.js` или `ts-node server.ts`) на порту 3000.
>
> **Ограничения (Constraints):**
>
> - Верни результат исключительно в виде Markdown блока кода (` ``` `).
> - Если требуются дополнительные пакеты, укажи команду для их установки (`npm install ...`) в комментариях в самом верху кода.
>
> **Предупреждения (Warning):**
>
> - Обязательно включи настройку CORS (через пакет `cors` или заголовки), чтобы у фронтенд-разработчика не возникло проблем при кросс-доменных запросах.

## 💡 Комментарий автора (Insight)

Самый главный секрет этого промпта — обязательное требование **«Случайной задержки (Random Delay)»** и **«Случайных ошибок (Random Error)»**.
Локальный Mock-сервер отвечает за миллисекунды. Если не добавить искусственную задержку, вы просто не увидите, как рендерятся ваши Skeleton UI или спиннеры загрузки. А благодаря 5% шансу на 500-ю ошибку, вы сможете заранее и без боли протестировать всплывающие уведомления (Toasts) и Fallback-интерфейсы в крайних случаях (edge cases).

Почувствуйте тот кайф, когда бэкендер говорит: «API готово!», а вы невозмутимо отвечаете: «Отлично, интеграция и обработка ошибок уже протестированы». 😎

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: У меня есть файл Swagger JSON, подойдет ли он?**
  - О: Да! Просто скопируйте текстовое содержимое Swagger (OpenAPI) JSON в раздел «Спецификация API» в промпте. ИИ сгенерирует Mock-сервер с еще более точными типами и схемами ответов.

- **В: Сохраняются ли данные, добавленные или измененные через POST/PUT?**
  - О: По умолчанию данные хранятся в оперативной памяти (в массивах) и сбрасываются при перезапуске сервера. Если вам нужно персистентное хранение, добавьте в промпт фразу: *"Используй lowdb или json-server для сохранения данных в локальный файл (db.json)"*.

- **В: Я использую Next.js. Можно ли обойтись без Express?**
  - О: Конечно! В блоке «Технологический стек» просто укажите `Next.js API Routes (Route Handler для App Router)`. ИИ выдаст код, который можно сразу положить в ваш проект на Next.js.

## 🧬 Разбор промпта (Why it works?)

1. **Роль (Role) и Контекст (Context):** Мы наделяем ИИ персоной не просто кодера, а «Эксперта по тестовым средам». Это заставляет его превентивно решать проблемы, с которыми реально сталкивается фронтенд (CORS, нулевой пинг и т.д.).
2. **Реалистичные ограничения:** Вместо мусорных данных вроде 'test1' и 'test2' мы требуем использовать `faker.js` или адекватные данные. Это позволяет оценить визуальное качество UI на уровне продакшена.
3. **Симуляция Edge Cases:** Явное требование вероятности ошибок и случайной задержки сети заставляет нас тестировать не только «счастливый путь» (200 OK), но и принудительно проверять обработку сбоев.

### ✅ После (Результат: Mock-сервер, написанный ИИ)

```javascript
// npm install express cors @faker-js/faker
const express = require("express");
const cors = require("cors");
const { faker } = require("@faker-js/faker");

const app = express();
app.use(cors());
app.use(express.json());

// Автоматическая генерация 15 реалистичных фейковых записей
let users = Array.from({ length: 15 }).map(() => ({
  id: faker.string.uuid(),
  name: faker.person.fullName(),
  email: faker.internet.email(),
  createdAt: faker.date.recent(),
}));

// Для реальных условий: Middleware случайной задержки (от 500ms до 1500ms)
const randomDelay = (req, res, next) => {
  const delay = Math.floor(Math.random() * 1000) + 500;
  setTimeout(next, delay);
};

// Для реальных условий: Middleware случайных ошибок (5% шанс на ошибку 500)
const randomError = (req, res, next) => {
  if (Math.random() < 0.05) {
    return res.status(500).json({ error: "Internal Server Error Simulation" });
  }
  next();
};

app.use(randomDelay);
app.use(randomError);

app.get("/users", (req, res) => {
  res.json({ data: users, total: users.length });
});

app.listen(3000, () => {
  console.log("🚀 Mock Server is running on http://localhost:3000");
});
```
