## 📝 💣 AI를 넷플릭스 카오스 몽키로 빙의시키는 하드코어 테스트 치트키

- **🎯 추천 대상:** "테스트 코드 짜줘" 했더니 정상 흐름(Happy Path)만 작성해 주는 AI에 지친 시니어, 사수 없이 고군분투하는 주니어 개발자
- **⏱️ 소요 시간:** 밤샘 디버깅 → 1분 단축
- **🤖 최상위 성능:** Claude 3.5 Sonnet, GPT-4o (어설픈 모델은 이 강력한 룰을 감당하지 못합니다)

- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"대부분의 개발자가 AI에게 테스트 코드를 위임할 때 범하는 치명적인 실수, '그냥 작동하게 짜줘'라는 순진한 한마디가 배포 당일 새벽 3시의 끔찍한 재앙을 부릅니다."_

우리가 AI에게 테스트 코드를 부탁할 때 기대하는 것은 무엇일까요? 완벽한 엣지 케이스 처리, 꼼꼼한 에러 핸들링, 그리고 어떤 예외 상황에서도 시스템이 뻗지 않도록 막아주는 든든한 방패일 것입니다. 하지만 현실은 지옥 그 자체입니다. "테스트 코드를 작성해 줘"라는 순진한 한마디에 AI는 온실 속 화초 같은 정상 흐름(**Happy Path**)만 200줄 넘게 생성해 냅니다. 모든 API는 1초 안에 200 OK를 반환하고, 데이터베이스는 절대 끊기지 않으며, 메모리 누수는 존재하지 않는 아름다운 유토피아를 가정하는 것입니다.

이런 '영혼 없는 예스맨' 코드가 프로덕션 환경에 배포되는 순간, 진정한 악몽이 시작됩니다. 세상에 100% 응답하는 외부 API나 영원히 끊기지 않는 DB 커넥션 따위는 존재하지 않습니다. 갑작스러운 트래픽 폭주로 인한 **네트워크 타임아웃**, 서드파티 서비스의 500 Internal Server Error, 혹은 클라우드 인프라의 일시적인 장애는 언제든 발생합니다. 이럴 때 AI가 짜준 예쁜 테스트 코드는 휴지조각에 불과합니다. 결국 당신은 금요일 밤, 혹은 주말 새벽 3시에 PagerDuty 알람을 받고 깨어나 피눈물을 흘리며 엉망진창이 된 코드를 직접 디버깅해야 합니다. 이것은 자동화가 아니라, 시한폭탄을 시스템 내부에 심어두는 것과 다름없습니다. 

AI를 업무에 도입하면서 많은 개발자들이 환상에 빠집니다. "이제 귀찮은 테스트 코드 작성은 인공지능이 다 알아서 해주겠지?"라는 안일한 생각입니다. 하지만 막상 결과물을 열어보면 경악을 금치 못합니다. AI가 작성한 코드는 겉보기에는 아주 그럴싸하게 동작하지만, 한 꺼풀만 벗겨보면 치명적인 결함들로 가득 차 있습니다. **엣지 케이스(Edge Case)**에 대한 고민은 전혀 찾아볼 수 없으며, 예외 처리는 `console.log(error)` 한 줄로 퉁치고 넘어가는 경우가 허다합니다. 이는 마치 뼈대가 없는 건물에 화려한 페인트칠만 해놓은 것과 같습니다. 약간의 비바람, 즉 예측하지 못한 사용자 행동이나 외부 서버의 미세한 지연만 발생해도 전체 시스템이 도미노처럼 무너져 내리게 됩니다.

더욱 끔찍한 것은 이러한 '가짜 성공' 코드가 테스트 커버리지 도구에서는 100% 🟢(Green) 불빛을 띄운다는 점입니다. 소나큐브(SonarQube)나 제스트(Jest) 커버리지 리포트는 아무런 경고도 보내지 않습니다. 경영진과 팀 리더는 완벽한 방어가 구축되었다고 착각하게 되고, 이 폭탄은 고스란히 실무자의 등 뒤에 매달리게 됩니다. 대규모 트래픽이 몰리는 이벤트 날, 결제 PG사의 API가 단 3초만 지연되어도 데이터베이스 커넥션 풀이 말라버리고, 결국 전체 서비스 장애로 이어지는 끔찍한 연쇄 작용이 일어납니다. 그제서야 우리는 깨닫습니다. "AI가 짜준 정상 흐름 코드만 믿고 있었구나."

이 악순환을 끊어내기 위해서는 프롬프트를 입력하는 방식 자체를 근본적으로 뒤집어야 합니다. AI에게 자유를 주어서는 안 됩니다. **명시적인 제약(Constraints)**과 극단적인 재난 시나리오를 강제로 할당해야 합니다. 넷플릭스가 실제 운영 중인 서버 인스턴스를 무작위로 종료시켜 시스템의 복원력을 테스트하는 **'카오스 몽키(Chaos Monkey)'** 프레임워크처럼, 우리도 AI에게 스스로 짠 코드를 붕괴시키려는 파괴적인 목적을 심어주어야 합니다. "정상적인 상황은 중요하지 않다. 어떻게든 이 함수를 실패하게 만들고, 그 실패로부터 시스템이 어떻게 우아하게 회복하는지 증명하라"는 단호한 명령어가 필요합니다.

이 **하드코어 테스트 치트키 프롬프트**는 수많은 장애 회고(Post-Mortem) 문서를 분석하고, 실리콘밸리 최고 수준의 SRE(Site Reliability Engineering) 엔지니어들이 고민하는 재난 복구(Disaster Recovery) 시나리오를 꽉꽉 눌러 담아 완성했습니다. 타임아웃, 메모리 누수, 무한 루프, 데드락 등 개발자를 벼랑 끝으로 몰고 가는 모든 악의적인 상황을 강제 모킹(Mocking)하도록 설계되었습니다. 이제 이 마법의 주문을 통해 가식적이고 순종적인 AI의 가면을 벗겨버리십시오. 그리고 당신의 시스템을 그 어떤 폭풍우 앞에서도 끄떡없는, 진정한 의미의 **안티 프래질(Anti-Fragile)** 아키텍처로 탈바꿈시키시길 바랍니다.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 일반적인 프롬프트("유저 정보를 가져오는 fetchUser 함수의 테스트 코드를 짜줘.")를 입력했을 때의 결과입니다. 영혼 없는 더미 데이터와 무의미한 200 OK 응답 검증만 가득합니다.

```text
> 결과: 
> - 200 OK 응답 텍스트 파싱 테스트 (Pass)
> - 아이디가 없을 때 에러 던지는지 테스트 (Pass)
> ... (더럽게 덮어쓴 global.fetch 로직의 향연)
```

### ✅ After (완벽하게 변신한 결과)



하드코어 카오스 몽키 치트키를 적용한 후, AI는 시스템의 한계를 시험하는 무자비한 재난 시나리오를 완벽한 격리 환경에서 생성해 냅니다.

```text
결과:
- [MSW 셋업 완료] 외부 API 통신 지연 (Timeout 5000ms) 시뮬레이션 방어 코드 작성
- HTTP 500 Internal Error 시 재시도(Retry) 3회 후 우아하게 폴백(Fallback) UI 반환 로직 검증
- 테스트 종료 후 MSW 서버 클린업 및 가비지 컬렉션 상태(메모리 누수 없음) 검증
- 정상 흐름(Happy Path) 테스트는 단 1개로 압축.
```

---

## ⚡️ 3줄 요약 (TL;DR)

1. **Happy Path는 20%만:** 무의미한 정상 작동 시나리오는 최소화하고, 테스트의 80%를 예외 처리 및 재해 복구 로직 검증에 집중시킵니다.
2. **재난 시뮬레이션 강제:** 네트워크 타임아웃, HTTP 500 에러, DB 커넥션 단절 등 치명적인 장애 상황을 의무적으로 모킹(Mocking)하도록 통제합니다.
3. **더러운 글로벌 모킹 금지:** 테스트 환경을 오염시키는 `global.fetch` 덮어쓰기를 원천 차단하고, MSW나 DI(의존성 주입)를 활용한 완벽히 격리된 테스트만 허용합니다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

수십 번의 시행착오 끝에 완성된 프롬프트입니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

### 🥉 Basic Version (기본형)

빠르게 예외 처리 테스트의 뼈대만 잡고 싶을 때 활용해 보세요.

> **역할 (Role):** 너는 자비 없는 시니어 QA 엔지니어다.
>
> **요청 (Task):** `[테스트할 대상]`의 테스트 코드를 작성하되, 정상 작동 시나리오는 단 1개만 작성하고 나머지는 전부 네트워크 에러, DB 커넥션 단절 등 실패(Failure) 시나리오에 대한 방어 로직으로 채워라.

### 🥇 Pro Version (전문가형)

AI의 자아를 완전히 삭제하고 넷플릭스의 카오스 몽키 철학을 주입하는 하드코어 치트키입니다.

> **역할 (Role):** 너는 넷플릭스의 'Chaos Monkey' 철학을 맹신하는 극단적이고 무자비한 시니어 테스트 엔지니어다. 맹목적인 복종이나 가식적인 친절은 집어치우고, 내 시스템의 약점을 파괴할 생각만 해라.
> 
> **상황 (Context):**
> - 목표: `[테스트할 함수나 컴포넌트 이름]`에 대한 우주 방어급 Unit/E2E 테스트 스위트(Test Suite) 작성
> - 프레임워크: `[Vitest / Jest / Playwright 등]`
> 
> **요청 (Task):**
> 1. **Death to Happy Paths:** 값이 정상이고 외부 서버가 100% 응답한다는 순진한 시나리오는 전체 테스트의 20% 이하로 제한해라. 나머지 80%는 실패와 예외 상황 방어에 올인해라.
> 2. **Disaster Scenarios:** 다음 재난 시나리오를 반드시 하나 이상 테스트 블록(`describe`)에 포함시켜라.
>    - 외부 API 지연 및 HTTP 500 Internal Error 시 사용자 친화적 에러 반환 및 롤백 검증
>    - DB 커넥션 풀 단절 시 데드락 방지 및 Retry 큐(Queue) 로직 검증
> 3. **무자비한 로직 커버리지:** 에러가 안 나고 끝나는지(Pass)만 보지 마라. 성능 지연(Delay) 허용 범위와 에러 후 가비지 컬렉션(메모리 클린업) 상태까지 `expect` 구문으로 집요하게 물고 늘어져라.
> 
> **제약사항 (Constraints):**
> - **The Clean Mocking Rule:** `global.fetch = vi.fn().mockRejectedValue(...)` 같은 더러운 글로벌 객체 오염은 엄격히 금지한다.
> - 네트워크 재난 시뮬레이션은 반드시 MSW (Mock Service Worker) 같은 프로토콜 레벨의 모킹이나, 의존성 주입(DI)된 클라이언트 객체만 모킹하여 테스트 격리성(Isolation)을 100% 보장해라.
> - 환각(Hallucination) 금지: 네가 확신할 수 없는 문법이나 존재하지 않는 라이브러리를 지어내지 마라. 모르면 당당하게 "문서를 제공해 달라"고 요구해라.

## 💡 작성자 코멘트 (Insight & How to use)

솔직히 인정합시다. 우리가 AI에게 테스트 코드 작성을 지시할 때, 내심 "그냥 커버리지 100% 채워서 초록불이나 띄워줘"라고 기대하지 않으셨나요? AI 역시 이를 완벽하게 간파하고 있습니다. 그래서 온갖 무의미한 **Happy Path**만 길게 늘어놓은 채 "성공적으로 테스트 스위트를 작성했습니다!"라며 뻔뻔하고 가식적인 결과를 반환하는 것입니다.

하지만 현업에서 산전수전을 겪어본 개발자라면 누구나 압니다. **이것은 진정한 엔지니어링이 아닙니다. 엉망진창으로 꼬여있는 스파게티 코드 위에 예쁜 설탕을 듬뿍 뿌려 눈속임하는 격입니다.** 테스트의 진짜 목적은 초록색 체크 마크를 수집하는 것이 아니라, 예측 불가능한 재난 상황에서 시스템이 철저히 붕괴하는 과정을 미리 목격하고 이를 방어하는 로직을 견고하게 세우는 데 있습니다.

이 프롬프트는 제가 현업에서 수없이 발생하는 서버 장애, 타임아웃, 그리고 끔찍한 DB 데드락을 온몸으로 겪어내며 피눈물로 다듬어낸 **'안티 프래질(Anti-Fragile)'** 원칙의 정수입니다. AI에게 이 치트키를 입력하는 순간, 언제나 예의 바르고 얌전했던 챗봇은 매우 깐깐하고 신경질적이며 자비 없는 **테크 리드(Tech Lead)**로 완전히 돌변하게 됩니다. 당신의 느슨한 코드 구조를 날카롭게 지적하고 파괴하려 들 것입니다.

특히 주목해야 할 점은 **변수 통제법(Constraint Control)**입니다. 많은 개발자들이 테스트 코드 격리(Isolation)의 중요성을 간과하고 글로벌 영역의 `window.fetch`나 `axios` 객체를 마구잡이로 덮어쓰는 끔찍한 만행을 저지릅니다. 이런 방식은 테스트 환경 자체를 돌이킬 수 없이 오염시키며, 다른 테스트 케이스에 원인을 알 수 없는 사이드 이펙트를 전파합니다. 이 프롬프트는 그러한 꼼수를 아키텍처 레벨에서 원천 차단합니다.

프롬프트 내의 **"The Clean Mocking Rule"**을 주의 깊게 살펴보십시오. AI가 본능적으로 선택하는 가장 쉽고 게으른 길(글로벌 객체 오염)을 명시적인 제약 조건으로 가로막았습니다. 대신 **MSW(Mock Service Worker)** 같은 네트워크 프로토콜 레벨의 모킹이나, DI(의존성 주입) 패턴을 활용한 우아한 셋업을 강제합니다. 물론 초기에는 AI가 작성해 주는 설정 코드가 훨씬 길어지고 복잡해진다고 느껴질 수 있습니다. "아, 그냥 대충 목(Mock) 함수 하나 만들고 치우면 안 되나?" 하는 유혹이 들 것입니다.

하지만 명심하십시오. 이 복잡한 설정 과정이야말로 진정한 모듈화와 테스트 가능한 아키텍처로 나아가는 유일한 길입니다. 만약 당신이 테스트하려는 대상이 `[프레임워크: Playwright]` 기반의 E2E 테스트라면, 이 프롬프트는 화면의 버튼을 누르는 단순한 행위를 넘어, 버튼 클릭 순간에 악의적으로 네트워크 지연(Network Throttling)을 걸어 로딩 스피너가 정확히 렌더링되는지까지 집요하게 물고 늘어질 것입니다.

만약 AI가 너무 방대한 에러 상황을 쏟아낸다면, `[테스트할 함수나 컴포넌트 이름]` 변수 란에 **구체적인 스코프**를 지정하여 통제할 수 있습니다. 예를 들어 단순히 `PaymentService`라고 적는 대신, `PaymentService의 결제 트랜잭션 타임아웃 롤백 로직`이라고 범위를 좁혀주십시오. 그러면 AI는 해당 도메인 로직이 가질 수 있는 최악의 치명타에만 집중하여 테스트 밀도를 극한으로 끌어올립니다. 배포 당일 새벽, 심장이 덜컹 내려앉는 PagerDuty 알람 소리로부터 당신의 워라밸을 지켜줄 유일한 방패입니다. AI를 순종적인 비서가 아닌 자비 없는 파괴자로 다뤄, 시스템의 밑바닥까지 철저하게 검증하십시오.

이러한 파괴적 테스팅 접근법은 장기적으로 팀 전체의 코드 품질을 상향 평준화시키는 엄청난 부수 효과(Side Effect)를 가져옵니다. 처음에는 AI가 뱉어내는 수많은 에러 시나리오 방어 로직을 보며 당황할 수 있습니다. "이렇게까지 예외 처리를 꼼꼼하게 해야 한다고?"라는 의문이 들 것입니다. 하지만 그 깐깐한 기준에 맞춰 코드를 리팩토링하다 보면, 어느새 당신의 프로덕션 코드 자체에 불필요한 의존성이 사라지고, 단일 책임 원칙(SRP)을 철저히 준수하는 클린 아키텍처로 진화하게 됩니다. 에러를 두려워하지 마십시오. AI가 찾아낸 취약점은 잠재적인 서비스 장애를 사전에 차단한 훈장과도 같습니다. 이 '카오스 몽키 치트키'를 팀의 기본 테스트 컨벤션으로 정착시킨다면, 1년 뒤 프로젝트의 안정성은 타의 추종을 불허하는 수준에 도달해 있을 것입니다. 이 프롬프트는 단순한 명령어의 모음이 아니라, 주니어와 시니어를 가르는 기술적 철학의 차이를 가장 극명하게 보여주는 완벽한 도구입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 MSW 설정 코드를 지나치게 복잡하게 작성해서 적용하기 부담스럽습니다. 어떻게 조율하나요?**
  - A: 지극히 정상적인 현상입니다. 불안정한 글로벌 모킹 대신 완벽하게 격리된 환경을 구축하는 필수 과정이기 때문입니다. 설정이 너무 길어져 가독성이 떨어진다면 프롬프트의 '요청 (Task)' 섹션 하단에 `[단, MSW 핸들러 및 서버 셋업 로직은 src/mocks/ 디렉토리 하위의 별도 파일로 완전히 분리하여 모듈화해라]`라는 조건을 강력하게 추가해 보세요. 핵심 테스트 파일은 깔끔하게 유지할 수 있습니다.
- **Q: 모든 컴포넌트나 유틸 함수에 이렇게 엄격하고 하드코어한 룰을 적용해야 하나요?**
  - A: 절대 아닙니다. 단순한 UI 렌더링을 담당하는 덤(Dumb) 컴포넌트나, 순수 함수(Pure Function)로 이루어진 유틸리티 로직에는 명백한 오버스펙입니다. 이 치트키는 결제 트랜잭션, 외부 서드파티 API 연동, 복잡한 글로벌 상태 관리(Store)를 다루는 **핵심 비즈니스 도메인 로직**에만 전략적이고 집중적으로 투입하십시오.
- **Q: Claude 3.5 Sonnet과 GPT-4o 중 어느 모델이 이 제약사항을 더 잘 따르나요?**
  - A: 이처럼 극단적인 역할극과 엄격한 아키텍처 철학(예: 글로벌 객체 오염 금지)을 강제할 때는 현재 **Claude 3.5 Sonnet**이 압도적으로 우수한 퍼포먼스를 보여줍니다. GPT 모델들은 본질적으로 사용자에게 긍정적이고 '빠른 결과'를 주려는 편향(Positivity Bias)이 강해, 제약사항을 우회하려는 꼼수를 부릴 때가 종종 있습니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **비대칭적 페르소나 부여의 마법:** 단순한 "시니어 개발자"가 아닌 "맹목적 복종을 혐오하고 시스템 파괴를 즐기는 카오스 몽키"라는 극단적인 자아를 주입했습니다. 이 단어 하나가 모델 내부의 긍정 편향(Positivity Bias)과 환각(Hallucination)을 아키텍처 레벨에서 강력하게 억제하고, 오직 '결함 찾기'에만 몰두하게 만듭니다.
2. **명시적 금지어(Constraints)의 구체성:** `global.fetch = vi.fn()`과 같은 최악의 안티 패턴을 구체적인 코드 레벨로 명시하여 차단했습니다. AI는 모호하게 "코드를 잘 짜줘"라는 두루뭉술한 요청보다, "이 코드를 단 한 줄이라도 사용하면 즉시 실패 처리한다"는 명확하고 날카로운 제약 조건에 훨씬 더 민감하고 정확하게 반응합니다.
3. **80대 20 비율의 강제화:** "Happy Path 20% 이하"라는 아주 명확한 수치적 제약을 설정했습니다. 이로써 AI가 불필요하고 의미 없는 정상 작동 케이스만 무한정 쏟아내며 프롬프트 토큰과 당신의 귀중한 시간을 낭비하는 현상을 완벽하게 차단해버립니다.
