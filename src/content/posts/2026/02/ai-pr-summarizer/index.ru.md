Коллега открыл Pull Request (PR), в котором изменено 50 файлов и добавлено более 1000 строк кода? 😱
Чувство растерянности наступает еще до начала ревью. «Что вообще здесь изменили? Нет ли побочных эффектов?»
Как было бы здорово, если бы можно было мгновенно выудить ключевые изменения из этого моря кода и заранее узнать о потенциально опасных участках! Мы расскажем, как с помощью ИИ радикально сократить время на ревью PR и повысить его качество.

## ⚡️ 3 главных тезиса (TL;DR)

1. ИИ мгновенно анализирует объемный `git diff`, выделяя главную цель PR и ключевые изменения.
2. Нейросеть заранее обнаруживает сложный код и потенциальные побочные эффекты (Risk), предупреждая ревьюера.
3. Вы получаете рекомендации по файлам и логике, требующим особого внимания, что максимизирует эффективность ревью.

### 🥇 Pro Version (Про-версия)

Используйте, когда требуется высокая детализация, анализ побочных эффектов и точное определение моментов для ревью.

> **Роль (Role):** Ты — Senior Software Engineer с 10-летним опытом, а также скрупулезный ревьюер кода (Code Reviewer), крайне внимательный к архитектуре и безопасности.
>
> **Контекст (Context):**
>
> - Ситуация: Коллега-разработчик внес массивные изменения в код и открыл крупный Pull Request.
> - Цель: Помочь ревьюеру быстро понять ключевые изменения и потенциальные риски, не теряя общего контекста, чтобы повысить эффективность проверки.
>

>
> Проанализируй следующие данные `git diff` и представь результат по 4 пунктам:

> 4. **Руководство для ревьюера (Reviewer Guide):** Порекомендуй приоритетные файлы или логику, на которые ревьюеру следует обратить особое внимание.

## 💡 Комментарий автора (Insight)

Этот промпт невероятно полезен даже тогда, когда вы сами пишете описание к своему PR, чтобы проверить, "что же я на самом деле изменил". Выполните команду `git diff main...HEAD | pbcopy` (на macOS) в локальном терминале, скопируйте результат в буфер обмена и закиньте его в LLM.

Даже если у вас нет коммерческих инструментов вроде Github Copilot Enterprise, один только этот метод обеспечивает отличное "предварительное ревью" (First-pass review). Senior-разработчик сможет значительно сократить время на ревью PR от джуниоров, а для Junior-разработчика ИИ станет отличным наставником (Mentor), помогающим находить уязвимости в собственной логике перед отправкой PR.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Что делать, если текст Diff слишком длинный и ИИ отказывается его принимать?**
  - A: Последние модели, такие как Claude 3.5 Sonnet или Gemini 2.5 Pro, поддерживают огромные контекстные окна (Context Window) и легко справляются с десятками тысяч строк кода. Если вы все же столкнулись с ограничением, сначала запросите сводку только по списку файлов с помощью `git diff --stat`, или выделите для анализа только изменения в файлах с ключевой доменной логикой.

- **Q: Безопасно ли загружать закрытый внутренний код компании в ИИ?**
  - A: **Будьте крайне осторожны.** Корпоративный код — это ключевой актив компании. При использовании публичных версий ChatGPT или Claude обязательно убедитесь, что в настройках включена опция "Не использовать данные для обучения модели" (Data Opt-out). Самый безопасный путь — использовать корпоративные тарифные планы (Enterprise), гарантирующие отказ от обучения на ваших данных, или развернуть локальную LLM (например, Ollama) внутри инфраструктуры компании.

## 🧬 Анатомия промпта (Why it works?)

1.  **Задание роли (Role):** Назначение конкретной персоны — "Senior-инженера с 10-летним опытом" и "ревьюера, внимательного к безопасности" — побуждает ИИ не просто делать сводку, а проводить критический и глубокий анализ.
2.  **Структурированный вывод (Task):** Четкое определение 4 необходимых элементов (краткая сводка, ключевые изменения, риски, гайд для ревьюера) предотвращает генерацию ИИ неуместных или избыточных объяснений.
3.  **Constraints & Warning (Ограничения):** Прямой запрет "не выдумывать" жестко блокирует галлюцинации (Hallucination) и заставляет нейросеть предоставлять только достоверные результаты ревью.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод: сырой git diff)

```diff
@@ -15,7 +15,11 @@ export class PaymentService {
   async processPayment(orderId: string, amount: number) {
     const order = await this.db.findOrder(orderId);
-    if (order.status === 'PAID') return;
+    if (order.status === 'PAID') {
+      throw new Error('Already paid');
+    }
+    await this.pg.pay(amount);
+    await this.db.updateOrderStatus(orderId, 'PAID');
+    // TODO: Send email
   }
 }
```

### ✅ После (Результат: ИИ предварительное ревью)

```text
1. Краткая сводка (Executive Summary):
В логику обработки платежей (PaymentService) добавлена обработка исключений для предотвращения двойной оплаты и функция обновления статуса платежа.

2. Ключевые изменения (Key Changes):
- Изменена защитная логика: теперь вместо тихого возврата (return) для заказов со статусом 'PAID' явно выбрасывается ошибка (Error).
- Добавлена логика внешней интеграции запроса на оплату через платежный шлюз (pg.pay).
- Внедрено обновление статуса заказа в базе данных на 'PAID'.

3. Оценка рисков (Risk Assessment):
- Отсутствие обработки ошибок: Не видно обработки исключений (Try-Catch и т.д.) на случай сбоя `pg.pay(amount)`. Необходим механизм отката (rollback), если платеж не пройдет до обновления статуса.
- Проблема конкурентности (Race Condition): Если два запроса на оплату поступят почти одновременно, между `findOrder` и `updateOrderStatus` может возникнуть состояние гонки, что создает риск двойного списания. Требуется использование транзакций или блокировок (Lock).

4. Руководство для ревьюера (Reviewer Guide):
- `PaymentService.ts`: Уделите первоочередное внимание ревью обеспечения транзакционности и логики обработки ошибок.
```
