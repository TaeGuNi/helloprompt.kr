> _「APIのレスポンスが変わるたびに、深くネストされたTypeScriptの型定義を手作業で修正するのに疲弊していませんか？」_

まだ `interface User { ... }` のように、一つずつ型を手打ちしているのでしょうか？
深くネストされたオブジェクトや配列の配列、予測不能なオプショナル値など、手作業での型定義は予期せぬヒューマンエラーの温床となるだけでなく、エンジニアの貴重なリソースを容赦なく奪い去ります。
本記事では、APIレスポンスの生のJSONをそのままコピー＆ペーストするだけで、完璧なTypeScriptの型定義（Interface/Type）はもちろん、堅牢なランタイムバリデーションを実現するZodスキーマまで全自動で生成する画期的な手法をご紹介します。

## 💡 作成者コメント (Insight)

このプロンプトの真の価値は、単なる「TypeScriptの型生成」にとどまらず、**「Zodスキーマの同時生成とモジュール化（分離）」**までを一手に担ってくれる点にあります。

実際の開発現場では、API仕様書（Swaggerなど）の更新が追いついていなかったり、ドキュメントには記載されていない隠しフィールドが突然現れたりすることが日常茶飯事です。そのような予期せぬ事態でも、Postmanやブラウザのネットワークタブから取得した「実際の生のJSONレスポンス」をそのままこのプロンプトに投入するだけで、**100%実データに基づいた型安全なフロントエンド環境**をわずか数秒で構築できます。

特に、ネストされたオブジェクトをインライン化させず、自動的に独立したスキーマ（例: `ContactSchema` と `UserSchema`）として分離するようプロンプト内で強力に制御しているため、生成されるコードの**可読性と再利用性が劇的に向上**します。これにより、後から仕様変更があった際のメンテナンスコストも大幅に削減されるのです。

## 🙋 よくある質問 (FAQ)

- **Q: JSONデータ内に複雑な配列が含まれている場合、生成される型はどうなりますか？**
  - A: プロンプトの制約事項で「配列内のオブジェクトも分離する」よう明確に指示しているため、`Array<{...}>` のような読みにくいインライン定義にはなりません。例えば、配列の中身用に `ItemSchema` を独立して作成した上で、親スキーマでは `z.array(ItemSchema)` として参照するクリーンな構造が自動生成されます。

- **Q: プロジェクトでZodではなく、YupやValibotを採用しているのですが対応可能ですか？**
  - A: はい、容易に応用可能です。プロンプト内の `Zod` や `z.infer` といったキーワードを、お使いの `Yup` や `Valibot` の構文に置き換えるだけで、指定したバリデーションライブラリの仕様に合わせた正確なスキーマを生成してくれます。

- **Q: 数千行にも及ぶ巨大で複雑なJSONデータをそのまま投げても機能しますか？**
  - A: AIモデルのコンテキストウィンドウ（トークン制限）の範囲内であれば動作しますが、入力が長すぎると途中で出力が途切れるリスクがあります。その場合は、配列内の重複する要素を1〜2件に減らすなど、**「データの構造」自体は保ったまま全体の行数をダイエット**させてから投入すると、より高速かつ精度の高いコードが生成されます。

## 🧬 プロンプト解剖 (Why it works?)

1. **JSDocの強制付与によるDX向上:** AIに各フィールドの文脈を推論させ、適切なJSDocコメントを記述させることで、VS CodeなどのIDEにおける開発体験（マウスホバー時の詳細なツールチップ表示）が格段に向上します。
2. **分離設計の厳格な指示 (Separation of Concerns):** ネストされたオブジェクトのインライン化を禁止し、強制的にスキーマを分離させることで、将来的なAPIの仕様変更にも柔軟に対応できる、保守性の高いクリーンなコードが生成されます。
3. **信頼できる唯一の情報源 (Single Source of Truth):** 従来の「Interfaceを先に手書きする」アプローチではなく、「まずランタイム用のZodスキーマを定義し、そこから `z.infer` を用いて静的な型を推論する」という、現代のTypeScriptにおける強固なベストプラクティスをAIに強制しています。

## 📊 証明: Before & After

### ❌ Before (入力JSON)

```json
{
  "id": 1,
  "name": "Jay",
  "contact": {
    "email": "jay@test.com",
    "phone": null
  },
  "tags": ["developer", "admin"]
}
```

### ✅ After (出力結果)

```typescript
import { z } from "zod";

/**
 * 連絡先情報
 */
export const ContactSchema = z.object({
  /** メールアドレス */
  email: z.string().email(),
  /** 電話番号（未登録の場合はnull） */
  phone: z.string().nullable().optional(),
});
export type Contact = z.infer<typeof ContactSchema>;

/**
 * ユーザー情報
 */
export const UserSchema = z.object({
  /** ユーザー固有ID */
  id: z.number(),
  /** ユーザー名 */
  name: z.string(),
  /** ユーザーの連絡先情報 */
  contact: ContactSchema,
  /** 関連タグ一覧 */
  tags: z.array(z.string()),
});
export type User = z.infer<typeof UserSchema>;
```
