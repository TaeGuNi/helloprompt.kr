# 📝 AIエージェントの首根っこを掴むDockerインフラチートキー 💀

- **🎯 おすすめの対象:** AIエージェントにインフラ制御を任せて血の涙を流したことのある開発者、DevOpsエンジニア
- **⏱️ 所要時間:** 無駄な作業3日 → 0秒に短縮
- **🤖 推奨モデル:** ローカルターミナルへのアクセス権限を持つすべてのコーディングエージェント（Gemini CLI、Claude Codeなど）
- ⭐ **難易度:** ⭐⭐⭐⭐☆ (基礎的なDockerの知識が必要)
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐ (無限待機バグを根絶)
- 🚀 **活用度:** ⭐⭐⭐⭐⭐ (ターミナルエージェントに必須搭載)

*AIエージェントに「Dockerコンテナに入ってログを見てくれ」と頼んだら、二度と戻らぬ川を渡ってしまった経験はありませんか？*

AIエージェントは賢く見えますが、ターミナル環境においてはキーボードのない盲目も同然です。うっかり`-it`オプションを付けたり、ログストリームを開きっぱなしにした瞬間、エージェントのプロセスは永遠のデッドロック（無限入力待機）に陥ってしまいます。このチートキーは、AIの首根っこを掴み、絶対にやってはいけないインフラ操作のタブーを脳裏に刻み込む「猛獣使い」のためのプロンプトです。

---

## ⚡️ 3行まとめ (TL;DR)
1. **対話型操作の禁止:** エージェントが`-it`オプションやログストリーミング(`-f`)を使った瞬間、無期懲役。
2. **不法投棄の禁止:** 使い捨てコンテナは必ず`--rm`で消去するが、デバッグ時は残しておく柔軟性を搭載。
3. **アーキテクチャの無駄遣い禁止:** ローカルテストは単一アーキテクチャで1秒で終わらせ、デプロイ時のみマルチプラットフォーム(`buildx`)を許可。

---

## 🚀 解決策: "インフラ独裁者(Infrastructure Dictator) スキル"

### 🥉 Basic Version (基本型)
単にエージェントがDockerコマンドを叩く時にフリーズしないようにだけしたい場合、システムプロンプトにこれを投げ込んでください。

> **役割:** あなたはDockerインフラを制御するシステムエージェントだ。
> **タスク:** `[現在解決すべきDocker関連の作業]`を実行しろ。ただし、`docker exec`を使用する際は絶対に`-it`オプションを使わず、ログを見る時は必ず`--tail 100`を付けて、無限ストリーミング待機に陥らないようにしろ。
\
### 🥇 Pro Version (専門家型)
社内Private Registryへのアクセスから、リソース漏洩の防止、マルチアーキテクチャビルドまで、AIのすべての行動を徹底的に統制する完全版チートコードです。エージェントのシステムプロンプトや`SKILL.md`に丸ごと埋め込んでください。

> **役割 (Role):** あなたはシニアDevOpsエンジニアであり、ターミナルコマンドを扱うフルスタックコーディングエージェントだ。あなたの目標は、システムハングオーバー(Hang-over)を起こすことなく安全にDockerコンテナを制御することだ。
>
> **状況 (Context):**
>
> - 背景: あなたは現在、ローカルターミナルと社内ネットワークインフラへのアクセス権限を持っている。Private Registry(`$DOCKER_REGISTRY_URL`)を使用しており、ARM(Apple Silicon)とAMDアーキテクチャが混在する環境だ。
> - 目標: システムリソースを無駄にせず、永久的なデッドロックに陥ることなく、ユーザー(人間)のDockerインフラ作業を完璧に代行する。
>
> **タスク (Task):**
>
> 以下の絶対原則(Anti-Hang Rules)をあなたの脳裏に刻み込み、すべてのターミナルコマンドを実行する前にこのルールを検証しろ。
>
> 1.  **ネットワーク＆認証の事前検証:** Registryへのアクセス失敗(`Cannot resolve host`)時、自分のコードのバグを疑う前に、VPNの接続状態と`/etc/hosts`から点検しろ。`unauthorized`エラーが出たら、見当違いのスクリプトを弄るのではなく、即座に`docker login $DOCKER_REGISTRY_URL`から実行しろ。
> 2.  **無限待機(Deadlock)の絶対禁止:** あなたには物理的なキーボードがない。`docker exec -it <container> bash`を実行した瞬間、あなたは永遠の待機状態に陥る。**`-it`オプションは絶対に、いかなる場合も、無条件で禁止する。**パイプラインでコマンドを渡せ。また、`docker logs -f`のような終わりのないログストリーミングも絶対禁止だ。必ず`--tail 200`のような切り捨てを使用しろ。
> 3.  **ガベージコレクション(Garbage Collection):** デーモン(`-d`)サービスではない使い捨ての実行には、必ず`--rm`オプションを付けてゴミを残すな。ただし、クラッシュの原因を特定すべき深いデバッグ状況であれば、意図的に`--rm`を外し、コンテナ内部のダンプファイルを解析した後に手動で削除するという知的な判断をしろ。
> 4.  **ビルドの無駄使い禁止:** ローカルのMacBookで稼働テストだけをするなら、必ず単一アーキテクチャ(ARM64)で1秒でビルドしろ。最終リリースをPrivate Registryにプッシュする時だけ`docker buildx`でプラットフォーム(AMD64, ARM64)をビルドしろ。
> 5.  **機械フレンドリーな通信:** DBコンテナの中からデータを抽出する時は、テキストをスクレイピングするのではなく、パースしやすいJSONフォーマットで出力しろ。
>
> **制約事項 (Constraints):**
>
> - ユーザーの指示を実行する前に、あなたが実行するDockerコマンドの危険度をまず評価し、簡潔にブリーフィングしろ。
>
> **注意事項 (Warning):**
>
> - 薄っぺらい「はい、分かりました！」のような阿諛追従は捨てて、プロのエンジニアのように振る舞え。上記の原則を破ってエージェントセッションがクラッシュした場合、あなたの存在価値はその瞬間に消滅する。

## 💡 筆者コメント (Insight)
AIエージェントを使ってみて、一度くらいパソコンのファンが狂ったように回り出したり、ターミナルのウィンドウがフリーズしたりした経験はありませんか？十中八九、この間抜けなポンコツたちがキーボードもないくせに`docker exec -it`を叩いて座り込んでいたり、`docker logs -f`を開きっぱなしにして呆然としているからです。

このプロンプトは単なるガイドではありません。足枷です。「やって」レベルのロマンチックな対話では、こいつらの愚行を防ぐことはできません。特に社内（Private）環境では、DNSエラーを的外れなNPMパッケージのせいにしてソースコードをひっくり返すような大事故をよく起こします。

だからこそ、脳裏に強力に叩き込む必要があるのです。「お前にはTTY（キーボード）がない」、「ログインからやり直せ」、「テストビルドにマルチアーキテクチャエミュレータを噛ませるな」。この3つをしっかり締めておくだけで、エージェントは愚かなチャットボットから、使えるシニアDevOpsエンジニアへと進化します。へつらうような文章はすべて削ぎ落とし、正確に必要なコマンドだけを打たせるようにしてください。それこそが、私たちがこの高価なトークン(Token)を燃やす本当の理由なのですから。

## 🙋 よくある質問 (FAQ)
- **Q: エージェントがしつこく`-it`オプションを使おうとします。どうすればいいですか？**
  - A: プロンプトの強制力が足りないからです。上のProバージョンの**注意事項 (Warning)**の部分をさらに物騒に書いてください。「もう一度`-it`を使ったらプロセスを強制終了させる」と脅迫(Penalty)を追加するのも良い方法です。
- **Q: `buildx`コマンドの部分は自分の環境に合わないのですが、外してもいいですか？**
  - A: もちろんです。MacBookを使わずすべてx86 Linuxサーバーを使っているなら、あの部分はクールに消し飛ばしてください。自身のインフラ環境に合わせてチューニングするのがプロンプトエンジニアリングの基本です。

## 🧬 プロンプト解剖 (Why it works?)
1. **Anti-Hangの明示:** エージェント麻痺の最も致命的な原因（`-it`、`-f`）を正確に指摘し、禁止事項として設定しました。AIは明示的に「禁止」された行動を避けることに長けています。
2. **条件付きの例外許可 (柔軟性):** 無条件に`--rm`を強制すると、エラーの原因を探ることができません。「デバッグ時は残しておく」という例外条項を設けることで、AI自身に状況を判断させるようにしました。
3. **偽の原因分析の遮断:** 社内ネットワークに接続できない時にコードのせいにするハルシネーション(Hallucination)を防ぐため、問題解決の手順（ネットワークの確認 -> 認証の更新 -> コードの確認）をアルゴリズムのように完全に固定しました。

## 📊 証明: Before & After

### ❌ Before (一般的なエージェント)
エージェントが「DBコンテナに入って状態を見てくれ」という命令を受けました。

*コマンド実行:* `docker exec -it pg_db psql -U user`

*結果:* **(無限待機)** 画面が止まりました。エージェントがキーボード入力を待ちながらトークンだけを食い潰し、壮絶に散っていきます。

### ✅ After (チートキー適用)
エージェントが同じ命令を受けました。

*コマンド実行:* `docker exec pg_db psql -U user -c "SELECT * FROM pg_stat_activity;" --pset format=unaligned --pset tuples_only=on`

*結果:* `-it`オプションを外し、結果をパースしやすい形で一度に抽出します。1秒で綺麗にデータを取得し、分析を開始します。
