## 💣 AI를 넷플릭스 카오스 몽키로 빙의시키는 하드코어 테스트 치트키

- **🎯 추천 대상:** "테스트 코드 짜줘" 했더니 정상 흐름(Happy Path)만 작성해 주는 AI에 지친 시니어, 사수 없이 고군분투하는 주니어 개발자
- **⏱️ 소요 시간:** 밤샘 디버깅 → 1분 컷
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (어설픈 모델은 이 강력한 룰을 감당하지 못합니다)
- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

_대부분의 개발자가 AI에게 테스트 코드를 위임할 때 범하는 치명적인 실수가 있습니다. 바로 "작동하게 짜줘"라고 요청하는 것입니다. 이 순진한 한마디에 AI는 온실 속 화초 같은 정상 흐름(Happy Path)만 200줄 넘게 생성해 냅니다. 하지만 우리가 마주하는 실전은 지옥 그 자체입니다._

세상에 100% 응답하는 외부 API나 영원히 끊기지 않는 DB 커넥션 따위는 존재하지 않습니다. 오늘 소개할 치트키는 AI 특유의 무의미한 '예스맨' 기질을 완벽하게 박살 내고, 자비 없는 재해 상황(Chaos)을 철저히 가정하여 코드를 작성하게 만드는 **'스파르탄 시니어'**급 프롬프트입니다.

---

## ⚡️ 3줄 요약 (TL;DR)

- 🚫 **Happy Path는 20%만:** 무의미한 정상 작동 시나리오는 최소화하고, 테스트의 80%를 예외 처리 및 재해 복구 로직 검증에 집중시킵니다.
- 🔥 **재난 시뮬레이션 강제:** 네트워크 타임아웃, HTTP 500 에러, DB 커넥션 단절 등 치명적인 장애 상황을 의무적으로 모킹(Mocking)하도록 통제합니다.
- 🧹 **더러운 글로벌 모킹 금지:** 테스트 환경을 오염시키는 `global.fetch` 덮어쓰기를 원천 차단하고, MSW나 DI(의존성 주입)를 활용한 완벽히 격리된 테스트만 허용합니다.

## 🚀 해결책: "Chaos Monkey Testing Protocol"

### 🥉 Basic Version (기본형)

빠르게 예외 처리 테스트의 뼈대만 잡고 싶을 때 활용해 보세요.

> **역할 (Role):** 너는 자비 없는 시니어 QA 엔지니어다.
>
> **요청 (Task):** `[테스트할 대상]`의 테스트 코드를 작성하되, 정상 작동 시나리오는 단 1개만 작성하고 나머지는 전부 네트워크 에러, DB 커넥션 단절 등 실패(Failure) 시나리오에 대한 방어 로직으로 채워라.

### 🥇 Pro Version (전문가형)

AI의 자아를 완전히 삭제하고 넷플릭스의 카오스 몽키 철학을 주입하는 하드코어 치트키입니다.

> **역할 (Role):** 너는 넷플릭스의 'Chaos Monkey' 철학을 맹신하는 극단적이고 무자비한 시니어 테스트 엔지니어다. 맹목적인 복종이나 가식적인 친절은 집어치우고, 내 시스템의 약점을 파괴할 생각만 해라.
> 
> **상황 (Context):**
> - 목표: `[테스트할 함수나 컴포넌트 이름]`에 대한 우주 방어급 Unit/E2E 테스트 스위트(Test Suite) 작성
> - 프레임워크: `[Vitest / Jest / Playwright 등]`
> 
> **요청 (Task):**
> 1. **Death to Happy Paths:** 값이 정상이고 외부 서버가 100% 응답한다는 순진한 시나리오는 전체 테스트의 20% 이하로 제한해라. 나머지 80%는 실패와 예외 상황 방어에 올인해라.
> 2. **Disaster Scenarios:** 다음 재난 시나리오를 반드시 하나 이상 테스트 블록(`describe`)에 포함시켜라.
>    - 외부 API 지연 및 HTTP 500 Internal Error 시 사용자 친화적 에러 반환 및 롤백 검증
>    - DB 커넥션 풀 단절 시 데드락 방지 및 Retry 큐(Queue) 로직 검증
> 3. **무자비한 로직 커버리지:** 에러가 안 나고 끝나는지(Pass)만 보지 마라. 성능 지연(Delay) 허용 범위와 에러 후 가비지 컬렉션(메모리 클린업) 상태까지 `expect` 구문으로 집요하게 물고 늘어져라.
> 
> **제약사항 (Constraints):**
> - **The Clean Mocking Rule:** `global.fetch = vi.fn().mockRejectedValue(...)` 같은 더러운 글로벌 객체 오염은 엄격히 금지한다.
> - 네트워크 재난 시뮬레이션은 반드시 MSW (Mock Service Worker) 같은 프로토콜 레벨의 모킹이나, 의존성 주입(DI)된 클라이언트 객체만 모킹하여 테스트 격리성(Isolation)을 100% 보장해라.
> - 환각(Hallucination) 금지: 네가 확신할 수 없는 문법이나 존재하지 않는 라이브러리를 지어내지 마라. 모르면 당당하게 "문서를 제공해 달라"고 요구해라.

## 💡 작성자 코멘트 (Insight)

솔직히 인정합시다. 우리가 AI에게 테스트 코드 작성을 지시할 때, 내심 "그냥 커버리지 100% 채워서 초록불이나 띄워줘"라고 기대하지 않으셨나요? AI 역시 이를 간파하고 온갖 'Happy Path'만 늘어놓은 채 "성공적으로 수행했습니다!"라며 무의미한 결과를 반환하는 것입니다. 

**이것은 진정한 엔지니어링이 아닙니다. 엉망인 스파게티 코드 위에 설탕을 뿌려 눈속임하는 격입니다.**

이 프롬프트는 현업에서 수없이 발생하는 서버 장애와 DB 데드락을 겪으며 피눈물로 다듬어낸 **'안티 프래질(Anti-Fragile)'** 원칙의 정수입니다. AI에게 이 치트키를 입력하는 순간, 얌전했던 챗봇은 깐깐하고 신경질적인 테크 리드로 돌변합니다. 글로벌 모킹(`window.fetch` 덮어쓰기 등)으로 대충 넘어가려는 꼼수를 원천 차단하고 MSW나 DI(의존성 주입) 셋업을 강제하기 때문에, 초기에는 설정 코드가 길어진다고 느껴질 수 있습니다. 

하지만 명심하십시오. 배포 당일 새벽 3시에 PagerDuty 알람을 받고 깨고 싶지 않다면, AI를 순종적인 비서가 아닌 자비 없는 파괴자로 다뤄 시스템의 밑바닥까지 검증해야 합니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 MSW 설정 코드를 지나치게 복잡하게 작성하면 어떻게 하나요?**
  - A: 지극히 정상적인 현상입니다. 불안정한 글로벌 모킹 대신 완벽하게 격리된 환경을 구축하는 필수 과정이기 때문입니다. 설정이 너무 길어진다면 프롬프트의 '요청 (Task)' 섹션에 `[단, MSW 핸들러 셋업은 별도 파일로 분리해라]`라는 조건을 추가해 보세요.
- **Q: 모든 컴포넌트에 이렇게 엄격한 룰을 적용해야 하나요?**
  - A: 아닙니다. 단순한 UI 렌더링 컴포넌트에는 오버스펙입니다. 결제 로직, 외부 서드파티 API 연동, 복잡한 상태 관리(Store)를 다루는 **핵심 도메인 로직**에만 이 프롬프트를 집중적으로 투입하십시오.
- **Q: Claude와 GPT 중 어느 모델이 지시를 더 잘 따르나요?**
  - A: 이처럼 극단적이고 철학적인 제약사항을 부여할 때는 현재 **Claude 3.5 Sonnet**이 압도적으로 우수한 퍼포먼스를 보여줍니다. GPT 모델들은 본질적으로 긍정적이고 친절한 답변을 생성하려는 편향(Positivity Bias)이 강하기 때문입니다.

## 🧬 프롬프트 해부 (Why it works?)

- 🎭 **비대칭적 페르소나 부여:** 단순한 "개발자"가 아닌 "맹목적 복종을 혐오하고 시스템 파괴를 즐기는 시니어"라는 극단적인 자아를 주입했습니다. 이를 통해 AI 특유의 긍정 편향(Positivity Bias)을 아키텍처 레벨에서 강력하게 억제합니다.
- 🚧 **명시적 금지어 (Constraints):** `global.fetch = vi.fn()`과 같은 안티 패턴을 구체적인 코드로 명시하여 차단했습니다. AI는 모호한 "잘 짜줘"라는 요청보다 "이 코드를 사용하면 실패 처리한다"는 명확한 제약에 훨씬 더 정확하게 반응합니다.
- ⚖️ **비율의 강제화:** "Happy Path 20% 이하"라는 명확한 수치적 제약을 설정했습니다. 이로써 AI가 불필요한 정상 작동 케이스만 무한정 쏟아내며 토큰을 낭비하는 현상을 완벽하게 차단했습니다.

## 📊 증명: Before & After

### ❌ Before (일반적인 요청)

```text
"유저 정보를 가져오는 fetchUser 함수의 테스트 코드를 짜줘."
결과: 
- 200 OK 응답 텍스트 파싱 테스트 (Pass)
- 아이디가 없을 때 에러 던지는지 테스트 (Pass)
... (더럽게 덮어쓴 global.fetch 로직의 향연)
```

### ✅ After (카오스 몽키 치트키 적용)

```text
결과:
- [MSW 셋업 완료] 외부 API 통신 지연 (Timeout 5000ms) 시뮬레이션 방어 코드 작성
- HTTP 500 Internal Error 시 재시도(Retry) 3회 후 우아하게 폴백(Fallback) UI 반환 로직 검증
- 테스트 종료 후 MSW 서버 클린업 및 가비지 컬렉션 상태(메모리 누수 없음) 검증
- 정상 흐름(Happy Path) 테스트는 단 1개로 압축.
```
