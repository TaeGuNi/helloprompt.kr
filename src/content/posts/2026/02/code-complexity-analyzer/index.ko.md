# 🍝 복잡한 코드는 이제 그만! AI 복잡도 분석기 (Code Complexity Analyzer)

- **🎯 추천 대상:** 주니어 개발자, 레거시 코드를 물려받은 유지보수 담당자, 코드 리뷰어
- **⏱️ 소요 시간:** 1시간 → 3분 단축
- **🤖 추천 모델:** GPT-4o, Claude 3.5 Sonnet (코드 추론 능력이 뛰어난 모델 권장)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"if문 안에 for문, 그 안에 또 if문... 어떻게든 돌아가긴 하지만, 건드리는 순간 터질 것 같은 시한폭탄 레거시 코드를 마주하고 절망해 본 적 있으신가요?"_

![복잡한 코드는 이제 그만! AI 복잡도 분석기 (Code Complexity Analyzer)](/images/hooks/code-complexity-analyzer.jpg)

개발을 하다 보면 누구나 한 번쯤 숨이 턱 막히는 '스파게티 코드'를 마주하게 됩니다. 끊임없는 기능 추가와 땜질식 버그 수정이 반복되면서 코드는 괴물처럼 비대해지고, 순환 복잡도(Cyclomatic Complexity)는 기하급수적으로 치솟기 마련입니다. 이러한 코드는 단순히 가독성이 떨어지는 수준을 넘어섭니다. 작은 기능 하나를 수정하려 해도 어디서 어떤 예상치 못한 사이드 이펙트가 터질지 몰라 두려움에 떨게 만들며, 결국 개발자를 끔찍한 유지보수의 악몽 속으로 몰아넣습니다. 당장 내일 배포는 다가오는데 코드의 늪에서 허우적대며 밤을 지새운 경험, 다들 한 번쯤 있으실 겁니다.

이제 도저히 어디서부터 손대야 할지 막막한 그 두려움을 AI 코드 복잡도 분석기에게 맡겨보세요. 더 이상 감에 의존한 코드 수정은 필요 없습니다. AI가 순환 복잡도와 인지적 복잡도 등 객관적인 소프트웨어 공학 지표를 바탕으로 코드의 치명적인 문제점을 정확히 진단합니다. 나아가, 디자인 패턴과 클린 코드(Clean Code) 원칙에 입각하여 실무에 당장 적용할 수 있는 구체적이고 우아한 리팩토링 전략을 1분 만에 제시해 드립니다. 이 프롬프트와 함께라면 두려웠던 레거시 코드도 당신의 성장을 증명할 훌륭한 캔버스가 될 것입니다.

## ⚡️ 3줄 요약 (TL;DR)

1. 악명 높은 중첩 조건문과 비대해진 레거시 함수를 AI가 입체적이고 구조적으로 진단합니다.
2. 조기 리턴(Early Return), 함수 추출(Extract Method) 등 실무에 즉시 적용할 수 있는 검증된 리팩토링 기법을 제안받습니다.
3. 기존 비즈니스 로직은 안전하게 보존하면서, 가독성과 유지보수성이 극대화된 우아한 코드를 순식간에 얻을 수 있습니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 소스 코드 파일 전체를 한 번에 다 붙여넣고 분석해도 괜찮을까요?**
  - A: 가급적 추천하지 않습니다. LLM의 컨텍스트 윈도우 한계와 지시사항 집중도 저하 현상이 발생할 수 있기 때문입니다. 수백 줄이 넘어가는 방대한 파일 전체를 통째로 넣기보다는, 50~150줄 내외의 '특정 함수'나 '단일 클래스' 단위로 쪼개어 프롬프트에 입력해 보세요. AI가 문제의 핵심을 훨씬 더 날카롭게 파고들어 소름 돋게 정확한 리팩토링 인사이트를 제시해 줄 것입니다.

- **Q: AI가 완벽하게 고쳐준 코드를 그대로 프로덕션(Production) 환경에 배포해도 안전한가요?**
  - A: 절대 그대로 복사해서 붙여넣으시면 안 됩니다! 아무리 뛰어난 AI라도 비즈니스 로직의 미묘한 엣지 케이스를 놓치거나 의도치 않게 로직을 변형해 버리는 환각(Hallucination) 현상을 일으킬 수 있습니다. AI가 작성한 코드는 훌륭한 '초안'일 뿐입니다. **반드시 기존 유닛 테스트(Unit Test)가 모두 정상적으로 통과하는지 확인**하고, 개발자가 직접 로직의 무결성을 수동으로 꼼꼼히 검증한 뒤에 머지(Merge)해야 합니다.

- **Q: 제가 사용하는 특정 프로그래밍 언어나 프레임워크의 특성에 맞춰서 코드를 개선해 줄 수도 있나요?**
  - A: 물론입니다. 전문가형(Pro) 프롬프트의 `[TypeScript/NestJS]` 변수 위치에 'Java/Spring Boot', 'Python/FastAPI', 'Go/Gin' 등 현재 여러분이 실무에서 사용 중인 기술 스택을 정확히 명시해 보세요. AI가 해당 언어와 프레임워크 생태계에서 공식적으로 권장하는 네이밍 컨벤션(Naming Convention)과 최적의 디자인 패턴을 찰떡같이 반영하여 맞춤형으로 코드를 다듬어 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **소프트웨어 공학 지표 중심의 정밀 타격:** AI에게 "코드를 단순히 예쁘게 정리해 줘"라고 모호하게 명령하는 대신, '순환 복잡도(Cyclomatic Complexity)'라는 명확하고 구체적인 지표를 명시했습니다. 이를 통해 AI가 단순한 포맷팅을 넘어 아키텍처의 근본적인 구조 결함을 해결하는 데에만 온전히 컴퓨팅 파워를 집중하도록 설계했습니다.
2. **동작 유지 제약(Constraints)의 강력한 통제:** 리팩토링의 절대 원칙인 '외부 동작의 불변성'을 제약 조건의 최상단에 배치하여 엄격하게 통제했습니다. AI가 코드의 길이를 줄이려는 욕심에 핵심 비즈니스 로직을 마음대로 훼손하거나 엣지 케이스를 누락하는 치명적인 대참사를 원천적으로 차단했습니다.

## 📊 증명: Before & After

### ❌ Before (입력: 중첩이 심한 스파게티 코드)

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ After (결과: Guard Clauses 및 배열 메서드를 활용한 깔끔한 코드)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: 예외 상황 조기 리턴 (중첩 감소)
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. 고차 함수 활용: 반복문 단순화
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. 비즈니스 로직 분리 및 단순화
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```
