- **🤖 最高パフォーマンス:** コーディング特化モデル（Claude 3.5 Sonnet、GPT-4oなど）

- ⭐ **難難易度:** ⭐⭐☆☆☆

> _「明日がリリースなのに、メインのビジネスロジックを組むだけで精一杯。テストコードなんて一体いつ書けばいいんだ？」_

開発者なら誰しも一度は経験したことのある、背筋が凍るような瞬間があります。金曜日の午後、意気揚々と作成した新機能をプロダクションにデプロイした直後、予想もしなかった場所で致命的なバグが発生します。ホットフィックスのためにコードを開きますが、膨大なビジネスロジックのどこで問題が発生したのか把握することさえ容易ではありません。「テストコードさえしっかり書いていれば…」という後悔は、もう手遅れです。私たちは皆、テストコードの重要性を身に染みて理解しています。クリーンアーキテクチャ、TDD、カバレッジ100%といった言葉は、常に開発者の心を躍らせます。

しかし、現実は非情です。明日がリリースで、押し寄せる企画要件をこなすだけで精一杯な状況において、メインのビジネスロジック作成よりも2〜3倍の時間がかかるテストコード作成に、ゆとりを持って時間を投資できる開発者は極めて稀です。特に、膨大な外部API依存関係を持つレガシーコードを扱ったり、データベースと強く結合したロジックをテストしようとすれば、数十行の<b>モッキング（Mocking）</b>コードとボイラープレートをセットアップするだけで疲れ果ててしまいます。

結局、  エッジケース（Edge Case）に悩み抜いた末、「とりあえず今回のデプロイはQAチームのマニュアルテストを信じてこのまま行こう」と、危険な妥協をしてしまいます。こうして積み重なった<b>技術負債（Technical Debt）</b>は、最終的にリファクタリングすら不可能にする巨大な足枷となり、ブーメランのように自分に返ってきます。修正ボタンを押すたびに他の機能が壊れないか不安になる、あの恐怖の始まりです。安全網のない綱渡りのサーカスは、もう終わりにしましょう。

もう、これらすべての苦痛なプロセスと退屈な悩みは横に置いて、あなたが心血を注いで作成したメインロジックだけを軽くコピーして貼り付けてください。開発者が見落としがちな微細な例外ケース（Edge Case）から、厄介なヌルポインタ（Null Pointer）防衛線まで、<b>AIが完璧に動作する無欠陥のユニットテスト（Unit Test）コードを一瞬で作成</b>します。

わずか5分。複雑なモッキング（Mocking）オブジェクトの生成から、GIVEN-WHEN-THENパターンが適用されたクリーンなテストシナリオの導出まで、完璧主義なシニアQAエンジニアをあなたのペアプログラミングパートナーとして雇ってみてください。人間の直感が見落としがちな<b>死角を、機械的な緻密さで完璧に埋めてくれます</b>。カバレッジ100%の心理的安定感を抱き、自信を持って「Merge」ボタンを押す痛快な体験をお届けします。

## 📊 証明: 劇的な結果 (Before & After)

### ❌ Before (私たちが直面していた苦痛)

お粗末だった従来の手作業、あるいは真っ白なキャンバスを前にした苦痛を伴うロジック作成の瞬間です。テストコードがなく、ポツンと残された危険なロジックです。

```javascript
// 単純な入力コードの例
function calculateDiscount(price, discountRate) {
  if (price < 0 || discountRate < 0) {
    throw new Error("有効ではない値です。");
  }
  if (discountRate > 100) {
    return 0; // 100%以上の割引は無料
  }
  return price - price * (discountRate / 100);
}
```

### ✅ After (完璧に変身した結果)

!["テストコード作成、5分で終わらせるプロンプト"](/images/hooks/unit-test-generator-prompt.jpg)

```javascript
describe("calculateDiscount 関数のテスト", () => {
  // 1. 正常動作テスト (Happy Path)
  it("正常な価格と割引率が与えられれば、割引が適用された価格を返すべきである", () => {
    // GIVEN
    const price = 10000;
    const discountRate = 20;
    // WHEN
    const result = calculateDiscount(price, discountRate);
    // THEN
    expect(result).toBe(8000);
  });

  // 2. 境界値テスト (Edge Case)
  it("割引率が100%を超えた場合、0円を返すべきである", () => {
    // GIVEN & WHEN
    const result = calculateDiscount(10000, 110);
    // THEN
    expect(result).toBe(0);
  });

  // 3. 例外処理テスト (Failure Case)
  it("価格が負の数の場合、エラーを投げるべきである", () => {
    // GIVEN, WHEN & THEN
    expect(() => calculateDiscount(-5000, 10)).toThrow(
      "有効ではない値です。"
    );
  });
});
```

1. **Happy PathからEdge Caseまで:** 正常な動作フローはもちろん、人間の頭では予測しにくい例外状況まで、漏れなく検証するテストシナリオを自動的に導出します。
2. **お好みのフレームワークにカスタマイズ:** Jest、PyTest、JUnitなど、現在のプロジェクトで導入されているテストフレームワークの文法を正確に反映し、即時実行可能なコードを生成します。
3. **厄介なMockingを自動化:** 面倒な外部API呼び出しやデータベース依存関係をAIが自ら把握し、状況に合ったMockデータとStubを完璧に構成します。

## 💡 執筆者コメント (Insight & How to use)

このプロンプトは、単に「テストコードを代わりに書いてくれる」自動完成レベルを遥かに凌駕する、強力な<b>品質保証（QA）パートナー</b>です。特にプロンプト内に明示された最初の指示事項である<b>「テストケース一覧の導出」</b>段階こそ、このプロンプトが持つ真の核心的価値であり差別化ポイントです。AIが闇雲にコードを書く前に、該当するビジネスロジックをどのように分析し解釈したのか、マークダウンのリスト形式でまず目にすることができます。これにより、開発者本人ですら考慮できていなかった致命的なエッジケース（Edge Case：極端なNull入力、予想外の型不一致、並行性の問題など）を事前に完璧に防御し、論理的な欠陥を埋めることができます。

実務環境では、同じテストフレームワーク（例：Jest, JUnit, PyTest）を使用していても、各開発チームや社内コンベンションによって、好まれる<b>モッキング（Mocking）方式</b>やコードスタイルが微妙に異なるものです。あるチームは `jest.spyOn` を好み、あるチームは依存性の注入（DI）によるフェイクオブジェクト（Fake Object）の生成を原則としていたりします。もしAIが生成したコードが、現在のプロジェクトの厳格な基準に合わない場合は、プロンプト下部の制約事項の部分に<b>`[私たちのチームのMocking例コード]`</b>スニペットを一つだけ追加してみてください。最新のAIモデルはFew-Shot学習能力が非常に優れているため、あなたのチーム独自のスタイルと文法を即座に学習し、別途修正することなく実務に即投入できる完璧なコードを出力するでしょう。

また、テストコードは一度書いて終わりではなく、メインロジックが変更されるたびに継続的にメンテナンスされなければならない生きたドキュメントです。このプロンプトは、<b>GIVEN-WHEN-THEN</b>という業界標準のパターンをコメントで強制するように設計されています。これは単にコードを綺麗に見せるための目的ではありません。数ヶ月後にコードを読み返す未来の自分、あるいはこのコードを引き継ぐ他のチームメンバーが、複雑なロジックの意図と流れをわずか1秒で直感的に把握できるように助ける、最も強力なコミュニケーションツールです。

テストコードを書いていくと、必然的にデータベース接続や外部サードパーティAPIとの通信区間に直面します。この時、初心者は実際のDBやAPIを呼び出す統合テスト（Integration Test）を書いてしまうというミスを犯しがちです。しかし、このプロンプトには<b>「外部依存関係の完璧なモッキング（Mocking）処理」</b>という厳格な指示が含まれています。AIは賢くネットワーク通信が発生する区間やI/O作業を検知し、実際のネットワーク接続が全く必要ないフェイクオブジェクト（Stub/Mock）に自動で置き換えてくれます。おかげで、ネットワーク状態に依存して成功したり失敗したりする不安定なテスト（Flaky Test）を根絶し、いつでもどこで実行しても0.1秒で一貫した成功を保証する、堅牢なユニットテスト環境を構築できます。

最後に、コードを丸ごと渡す前に<b>単一責任の原則（SRP）</b>をもう一度思い出してください。もし数百行を超えるような巨大なスパゲッティ関数をそのままプロンプトに放り込めば、いくら優れたAIでも核心的な文脈を見失ったり、重要なエッジケースの検証を漏らしたりする可能性があります。最も良い活用法は、機能ごとに細かく分割された独立した関数単位でテストコードの作成を依頼することです。これは自然と、あなたの既存のメインロジックが十分にモジュール化され、テストしやすい構造（Testable Code）で設計されているか、自らフィードバックを受ける素晴らしいリファクタリングの指標となるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: コードが長すぎてAIの回答が途中で止まってしまう場合はどうすればいいですか？**
  - A: 関数やクラス単位で細かく分割してリクエストする方法を強くお勧めします。巨大なファイルを丸ごと入力すると、AIが核心的なエッジケースを見落とす確率が高まります。単一責任の原則（SRP）が徹底された関数を一つずつ切り離して、テスト作成を依頼してみてください。

- **Q: データベース接続コードが含まれていますが、このような場合でもユニットテストコードの生成は可能ですか？**
  - A: はい、完璧に可能です！このプロンプトには「外部依存関係のモッキング（Mocking）」に関する強力な指示が含まれています。AIがDB RepositoryやORM呼び出し区間をスマートに検知し、これを実際のデータベース接続を必要としないフェイクオブジェクト（Mock）に自動で置き換えてくれます。

## 🧬 プロンプト解剖 (Why it works?)

1. **ペルソナ (Role) の付与:** 「完璧主義なシニアQAエンジニア」という具体的な役割を付与しました。これにより、単に体裁を整えたコードではなく、境界値（Boundary）まで執拗に掘り下げて検証する高品質なテストコードを誘導します。
2. **GIVEN-WHEN-THEN パターンの強制:** テストコードの可読性を最大化する業界標準パターンをコメントで強制しました。コードを初めて見る他のチームメンバーも、テストの意図と流れを直感的に把握できるように、しっかりとした構造を作ります。
3. **モッキング (Mocking) 自動化の指示:** ユニットテスト作成における最大のボトルネックである「依存性の注入およびモッキング」作業をAIに明示的に委任することで、テスト作成にかかる時間をドラマチックに短縮させました。
