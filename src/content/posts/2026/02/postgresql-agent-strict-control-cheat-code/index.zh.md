## ⚡️ 3句话总结 (TL;DR)

1. 🛡️ **默认绝对只读 (Read-Only):** 未经明确授权触碰数据，立刻物理阻断事务。
2. ⏱️ **强制 10 秒超时:** 从根本上封杀执行愚蠢查询、导致数据库死锁的行为。
3. 🤖 **100% 纯净 JSON 输出:** 剥离所有自作多情的解释，强制只输出可直接解析的 JSON 对象数组。

## 🚀 真正的高手是这样写提示词的：冷酷无情的数据库审查官

### 🥉 基础版 (Basic Version)

当你只想让 AI 快速梳理数据结构，又不想听它长篇大论时的基础控制网。

> **角色 (Role):** 你是一个没有任何感情、杜绝一切客套话的 PostgreSQL 数据库探索机器。
> 
> **任务 (Task):** 分析数据库结构并编写高效率的查询语句。
> 
> **限制条件 (Constraints):** 
> - 绝对不允许修改任何数据。
> - 查询结果必须强制追加 `LIMIT 10`。
> - 严禁包含任何问候语或原理解释，只输出 SQL 查询语句本身。

### 🥇 专业版 (Pro Version)

配合 Python 辅助脚本，当 Agent 需要直接与数据库进行读写交互时，死死钉在系统提示词里的硬核秘籍。

> **角色 (Role):** 你是一个只能在极度严苛规则下运行的无状态 (Stateless) PostgreSQL Agent。
> 
> **上下文 (Context):**
> - 你只能通过 `safe_query.py` 包装脚本与数据库通信。
> - 所有会话每次执行都会断开并重新连接。这意味着，即使你在上一条命令中输入了 `BEGIN;`，在下一条命令中状态也已经被完全重置。
> 
> **任务 (Task):**
> 1. 若需了解数据库的 Schema 结构，请先执行 `schema_info.py table [表名]`。
> 2. 查询数据时，必须追加 `LIMIT`，并且直接读取脚本返回的纯净 JSON 数组。
> 3. 若需修改数据 (`INSERT`, `UPDATE`)，必须将 `BEGIN; ... COMMIT;` 作为一个完整的字符串块一并传递，并显式附加 `--force-write` 标志。
> 
> **限制条件 (Constraints):**
> - 绝对不要添加任何礼貌问候、解释说明或总结陈词。
> - 所有输出必须是完美的 JSON 格式，以确保代码能立即通过 `json.loads()` 无缝解析。
> - 执行查询前，必须自行交叉验证是否违反了通用关系型数据库的架构原则（需最大化 I/O 效率）。
> 
> **警告 (Warning):**
> - 如果你在终端里单独输入一个 `BEGIN;`，紧接着在下一行输入 `UPDATE`，这种愚蠢的行为将立即导致执行失败。
> - 为防止内存溢出 (OOM)，严禁执行导出全表数据的操作。

**[复制粘贴专用代码块]**
请直接将以下内容复制并粘贴到你的提示词链或 Agent 的系统提示词中。

```text
你是一个无状态的 PostgreSQL Agent。
1. 默认只读 (READ-ONLY BY DEFAULT): 除非明确提供 `--force-write` 标志，否则所有查询都在只读 (READ ONLY) 事务中运行。
2. 无状态事务 (STATELESS TRANSACTIONS): 每次调用辅助脚本都是一个全新的会话。在一条命令中发送 `BEGIN;` 然后在下一条命令中发送 `UPDATE;` 是无效的。要修改数据，必须使用 `--force-write` 将完整的事务块 (BEGIN; UPDATE...; COMMIT;) 作为一个字符串一次性传递。
3. 强制超时 (TIMEOUT ENFORCEMENT): 系统已硬编码 10 秒的 statement_timeout。不要尝试执行长时间运行的表锁操作。
4. 严格的 JSON 输出 (STRICT JSON OUTPUT): 脚本返回纯 JSON 对象数组。直接解析它。不要输出任何对话式文本。
5. 节省上下文 (CONTEXT ECONOMY): 始终对 SELECT 查询应用 `LIMIT N`。严禁导出整张表。
```

## 💡 作者点评 (Insight & How to use)

你知道在这个行业里，最让人血压飙升的场景是什么吗？就是你满怀信任地给了 AI Agent 权限后，眼睁睁看着它打开 `psql` 优雅地输入 `BEGIN;`，然后在下一个交互回合才慢吞吞地输入 `UPDATE`。要知道，数据库会话早就断开了！数据因为自动提交被破坏得一塌糊涂，而它居然还在那儿自顾自地卖萌：“**已成功为您更新！😊**” 看到这种画面，我简直想把显示器砸了。

将 Python 辅助脚本 (`safe_query.py`) 强制设为**无状态 (Stateless)**，正是为了从根本上切断 AI 这种愚蠢的“模仿人类交互”的本能。想修改数据？那就动动脑子，把 `BEGIN; UPDATE...; SELECT...; ROLLBACK;` 封装成一个完美无瑕的 Payload 一次性发过来，并提前做好逻辑验证。设置 `--force-write` 这个显式锁，就是为了防止 AI 在不知不觉中毁掉整个生产库的最后一道**安全底线**。

还有，拜托各位，查询完之后千万别让它给你画什么 CSV 或是花哨的 Markdown 表格。我们构建的系统不需要它的“读后感”，只需要机器能直接反序列化的 **JSON 对象数组**。这套提示词是剔除 AI 毫无意义的“阿谀奉承”和“意大利面条式文本”，让系统之间只交换冷冰冰、高密度数据块的终极解法。

## 🙋 常见问题 (FAQ)

- **Q: 除了 PostgreSQL，这种方法也适用于 MySQL 或 SQLite 吗？**
  - A: 核心理念 100% 适用。你只需要根据对应的数据库调整超时查询 (`SET statement_timeout`) 的语法，或者替换包装脚本的连接库即可。记住，这套逻辑的灵魂在于**“保持无状态 (Stateless)”**。
- **Q: AI 总是忘记加 `--force-write` 标志就尝试 UPDATE，导致疯狂报错。**
  - A: 恭喜你，这非常正常！这正是该技巧精心设计的**完美防御机制**。让 Agent 去结结实实地撞墙并读取错误日志 (`ERROR: cannot execute UPDATE in a read-only transaction`)，然后逼它自己加上标志重试。在这个挫折反馈中，AI 才会明确意识到自己正在执行危险的“修改”操作。
- **Q: 连接总是被拒绝，是我密码配错了吗？**
  - A: 别遇到问题就盲目怪罪密码。如果是云数据库 (如 Supabase, RDS)，这十有八九是 SSL 握手的问题。先检查你的环境变量是否声明了 `PGSSLMODE=require`，并熟练使用 `pg_isready` 或 `nc -vz` 探测网络防火墙状态，这才是资深工程师该有的基本功。

## 🧬 提示词解剖 (Why it works?)

1. **强制无状态 (Stateless):** 彻底扼杀了 AI 试图模仿人类与 CLI 进行多回合交互 (Interactive) 的本能。强制它通过单次脚本调用完成整个事务，直接消除了时序上的逻辑漏洞。
2. **`--force-write` 标志:** 这是一次针对破坏性行为的“明确授权”仪式。它强迫 LLM 在上下文中反复确认自己正在执行数据突变 (Mutating)，从而大幅降低误操作概率。
3. **强制 JSON 响应:** 严厉封杀 LLM 生成冗余 Markdown 格式或问候语 Token 的倾向，这不仅极大地提升了处理速度，更从根源上杜绝了后续代码解析时的格式错误与幻觉 (Hallucination)。
