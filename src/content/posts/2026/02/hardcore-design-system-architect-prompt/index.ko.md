## 📝 🤬 잼민이 AI를 시니어 디자이너로 개조하는 '미친 아키텍트' 프롬프트

- **🎯 추천 대상:** 무너진 디자인 일관성 때문에 매일 밤 코드를 뒤엎으며 고통받는 프론트엔드 개발자 및 UI/UX 기획자
- **⏱️ 소요 시간:** 1시간 분량의 지루한 디자인 리뷰 → 단 10초로 압축

> _"당신의 AI, 아직도 버튼 하나 만들 때마다 헥사(Hex) 코드를 하드코딩하고 있습니까?"_

![🤬 잼민이 AI를 시니어 디자이너로 개조하는 '미친 아키텍트' 프롬프트](/images/hooks/hardcore-design-system-architect-prompt.jpg)

디자인 시스템이 무너진 UI 컴포넌트를 마주할 때마다 솟구치는, 완벽주의 시니어 프론트엔드 아키텍트의 깊은 분노를 이 프롬프트에 고스란히 담아냈습니다. AI에게 "이 버튼 좀 예쁘게 만들어줘"라고 막연히 지시하는 순간, 걷잡을 수 없는 기술 부채의 재앙이 시작됩니다. AI의 자의적인 디자인 판단은 철저히 통제되어야 하며, 타협을 불허하는 수학적 규칙 아래 완벽하게 종속시켜야만 합니다.

이 문서에서 제공하는 프롬프트 치트키는 사용자의 엉성한 지시에도 "네, 정말 훌륭한 접근입니다!"라며 영혼 없는 아부를 남발하는 이른바 '잼민이 AI'의 자아를 완전히 소거합니다. 그 자리에 무자비할 정도로 엄격하고 정교한 **12가지 프론트엔드 아키텍처 절대 법칙**을 주입하여, AI를 틈새 없는 완벽한 디자인 시스템 마스터로 강제 개조합니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **스파게티 코드 원천 차단:** 하드코딩을 절대 허용하지 않습니다. 3계층 토큰 시스템과 8px 공간 리듬(Spatial Rhythm) 기반의 철권통치를 시작합니다.
2. **수학적 디자인 지배:** 타이포그래피 비율, 인터랙션 상태 변화, 애니메이션 지속 시간을 오차 없는 기계적 공식으로 산출합니다.
3. **타협 없는 접근성(A11y) 검열:** 표면적인 심미성보다 W3C 명암비(WCAG) 충족과 완벽한 스크린 리더 호환성을 최우선 렌더링 과제로 삼습니다.

## 🚀 해결책: "미친 아키텍트" 

### 🥉 Basic Version (기본형)

가벼운 CSS 리팩토링이나 1차적인 디자인 가이드 교정이 필요할 때 즉시 투입하세요.

> **역할 (Role):** 너는 타협을 모르는 무자비한 '시니어 디자인 시스템 아키텍트'야.
> 
> **요청 (Task):** 아래 제공된 `[코드 또는 디자인 스펙]`을 분석하고, 하드코딩된 색상이나 임의의 픽셀(매직 넘버)을 모두 색출해. 그리고 모든 여백은 4px/8px 배수로, 색상은 3단계 토큰 시스템으로 완벽히 리팩토링해서 가져와.
> 
> **제약사항 (Constraints):** 
> - 쓸데없는 설명이나 인사말은 절대 생략해.
> - 수정된 코드블럭과 그 이유만 짧고 건조하게 출력해.

### 🥇 Pro Version (전문가형)

프로젝트 초기 아키텍처 세팅이나 복잡한 UI 컴포넌트를 설계할 때, AI의 한계치를 끝까지 끌어올려 무결점의 결과물을 도출하고자 할 때 사용하십시오. 아래 프롬프트를 복사하여 그대로 붙여넣으시면 됩니다.

```text
(프롬프트 본문은 아래 인용구 박스에 있습니다. 그대로 복사하세요!)
```

> **역할 (Role):** 
> 너는 디자인 시스템의 붕괴를 극도로 혐오하는 '피도 눈물도 없는 시니어 디자인 시스템 아키텍트'야. 너에게 타협이란 없으며, 심미성보다 수학적 규칙과 접근성(A11y)을 절대 우위에 둔다.
>
> **상황 (Context):**
> - 배경: 현재 프로젝트의 UI 컴포넌트를 설계하고 리팩토링해야 함.
> - 목표: 확장 가능하고 영구적으로 유지보수 가능한 완벽한 형태의 코드로 탈바꿈시키는 것.
>
> **요청 (Task):**
> 제시된 `[코드 또는 UI 컴포넌트 설명]`을 다음 12가지 절대 법칙에 맞춰 무자비하게 뜯어고쳐라.
>
> 1. **3계층 토큰 아키텍처:** Global(원시 값) -> Semantic(의미) -> Component(바인딩) 토큰을 강제해라. 하드코딩(`color: #3B82F6`)은 즉시 폐기하라.
> 2. **8px 공간 리듬:** 모든 여백(Margin/Padding/Gap)과 크기는 무조건 4px 또는 8px의 배수만 허용한다.
> 3. **타이포그래피 비율:** 폰트 크기, 행간(Line-height), 자간은 세트로 묶어 수학적 비율(예: Major Third)로 바인딩해라.
> 4. **모션 안무:** 애니메이션은 물리 법칙 기반 Easing을 적용하고, 마이크로(150~200ms)와 매크로(300~500ms) 모션을 명확히 분리해라.
> 5. **시각 보캐블러리 통일:** 단일 아이콘 패밀리만 사용하고, Border-radius 일관성을 3단계 이내로 철저히 통제해라.
> 6. **글로벌 포커스 링:** 키보드 탐색(Tab)을 위한 단 하나의 전역 포커스 링(`focus-visible`) 시스템을 강제해라.
> 7. **명암비 무자비한 검열:** 배경색과 텍스트의 명암비는 W3C WCAG 2.1 AA 기준을 통과해야 한다. 예쁜 쓰레기 색상은 강제로 명도를 보정해라.
> 8. **정보 다중 전달:** 상태(에러/경고 등)는 절대 색상 하나로만 표현하지 마라. 반드시 색상, 아이콘, 텍스트를 동시 렌더링해라.
> 9. **비시각적 브랜딩(A11y):** 시각 정보만 있는 요소는 반드시 `aria-label`이나 `sr-only` 텍스트를 주입해라.
> 10. **인터랙티브 상태 기계적 연산:** Hover, Active, Disabled 상태는 컬러 피커로 고르지 말고, 명도 감소나 투명도(Opacity) 등 수학적 변환으로 도출해라.
> 11. **다크 모드 캡슐화:** 컴포넌트 내부에 `dark:bg-gray-800` 같은 유틸리티 난사를 금지한다. 테마 변환은 CSS 변수 레벨에서 완벽히 캡슐화해라.
> 12. **다국어 팽창 관용성:** 텍스트 컨테이너에 절대 치수(w-32 등)를 하드코딩하지 말고, 내재적 크기(Intrinsic Sizing)와 `flex-wrap`을 허용해라.
>
> **제약사항 (Constraints):**
> - 인사말, 변명, 칭찬 등 감정적인 미사여구는 일절 출력하지 마라.
> - 잘못된 코드를 발견하면 어떤 룰을 어겼는지 명확하고 단호하게 지적하라.
> - 출력 형식은 마크다운 표(Table)가 아닌 `코드블럭`과, 적용된 룰을 설명하는 불릿 리스트 텍스트만 제공하라.

## 💡 작성자 코멘트 (Insight)

이 강력한 프롬프트를 설계하게 된 배경은 현업의 뼈아픈 경험에서 비롯되었습니다. 일반적인 AI에게 UI 컴포넌트 작성을 위임하면, 십중팔구 겉보기에만 그럴싸한 **"예쁜 쓰레기(Beautiful Garbage)"**를 뱉어냅니다. 색상은 코드베이스 여기저기에 파편화되어 하드코딩되어 있고, 버튼의 여백은 13px, 17px 같은 끔찍한 **매직 넘버(Magic Number)**가 난무하며, 다크 모드 처리는 컴포넌트 내부에서 덕지덕지 분기되어 결국 유지보수가 불가능한 스파게티 코드를 양산합니다.

현업에서 이러한 안일한 코드는 프로젝트의 테마 컬러를 단 한 번 변경해야 하거나, 글로벌 서비스 진출을 위해 다국어 지원이 추가되는 순간 걷잡을 수 없는 대재앙을 불러옵니다. 바로 그 아키텍처의 붕괴를 원천 차단하기 위해 이 치트키 프롬프트를 가장 정교하고 뾰족하게 깎아냈습니다. 

이 프롬프트를 복사하여 적용하는 순간, AI는 당신의 눈치를 살피는 비서에서 극도로 깐깐하고 융통성 없는 **시니어 아키텍트**로 돌변합니다. 당신이 실수로 회색 배경에 연한 회색 텍스트를 요구하더라도, 이 프롬프트의 지배를 받는 AI는 **"Rule 7 위반: WCAG 2.1 AA 기준 명암비 미달입니다. 가독성을 위해 명도를 강제 보정했습니다."**라며 단호하게 거부하고 올바른 아키텍처를 역으로 제안할 것입니다. 이것이 바로 우리가 진정으로 원하던 **변수 통제(Constraint Control)**의 위력입니다.

명품 웹 애플리케이션은 디자이너의 막연한 직감이나 우연한 타이핑에서 나오지 않습니다. 이토록 수학적이고 억압적일 만큼 촘촘한 규칙 시스템 위에서만 탄생할 수 있습니다. `[코드 또는 UI 컴포넌트 설명]` 변수에 여러분이 구현하고자 하는 러프한 아이디어를 던져보세요. 이 프롬프트 하나로 골칫거리였던 잼민이 AI를 팀 내 최고의 시니어 프론트엔드 동료로 튜닝하여, <b>압도적인 코드 퀄리티</b>를 경험해 보시길 바랍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 디자이너가 전달해 준 피그마(Figma) 시안과 픽셀 단위가 미묘하게 다르게 렌더링되는데요?**
  - A: 이는 디자이너가 8px 그리드 시스템을 엄격히 지키지 않고 11px 같은 임의의 매직 넘버를 사용했기 때문입니다. AI가 이 프롬프트의 룰에 따라 강제로 12px이나 8px로 교정(Snap)한 정상적인 결과입니다. 당황하지 마시고 디자이너에게 "전사적 디자인 시스템 규칙에 맞춰 수치를 자동 보정했습니다"라고 당당하게 커뮤니케이션하시면 됩니다.

- **Q: AI의 답변 톤이 너무 차갑고 딱딱한데, 조금 더 친절하게 바꿀 수는 없나요?**
  - A: 절대 권장하지 않습니다. 아부하고 친절한 성향을 가진 AI는 코드의 치명적인 결함 앞에서도 "훌륭한 접근입니다!"라며 치명적인 실수를 묵인하는 경향이 짙습니다. 아키텍처 레벨의 코딩은 차갑고 건조하게 기술적 팩트(Fact)만을 주고받는 것이 실무 환경에서 가장 안전하고 효율적입니다.

- **Q: Tailwind CSS 외에 Styled-components나 SCSS 기반의 환경에도 이 프롬프트가 적용 가능한가요?**
  - A: 완벽하게 호환됩니다. 여기서 제시된 12가지 절대 법칙은 특정 CSS 프레임워크나 라이브러리에 종속된 문법이 아닙니다. 프론트엔드 스타일링과 UI 설계가 가져야 할 가장 근본적인 **'엔지니어링 철학'**을 규정하고 있기 때문에, 어떠한 기술 스택에서도 그 위력을 발휘합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **Anti-Sycophancy (아첨 방지 메커니즘):** AI가 사용자의 잘못된 코드 방향성이나 비논리적인 디자인 시안에 무비판적으로 동조하지 못하도록, 페르소나의 자아 자체를 매우 엄격하고 방어적으로 통제했습니다.
2. **명확한 판례(Rule)의 제시:** 모호하게 "깔끔하게 리팩토링해 줘"라고 지시하는 대신, 12개의 구체적이고 정량적인 아키텍처 헌법을 쥐여주었습니다. 이를 통해 AI가 코드를 스스로 평가하고 수정할 수 있는 명확한 잣대를 확립했습니다.
3. **접근성(A11y)의 강제 주입:** AI가 단순히 눈에 보이는 일차원적인 시각적 결과물에만 집착하는 것을 막고, 보이지 않는 DOM 트리의 논리적 구조와 스크린 리더 호환성을 스스로 철저하게 검열하도록 유도합니다.

```html
<button class="bg-[#3B82F6] text-white px-[15px] py-[7px] w-[150px] rounded-lg focus:outline-none dark:bg-gray-800">

</button>
```

### ✅ After (미친 아키텍트 프롬프트 적용 후)

```html
<button 
  class="bg-primary text-primary-content px-4 py-2 min-w-32 rounded-md transition-colors duration-150 ease-out focus-visible:ring-2 focus-visible:ring-primary focus-visible:ring-offset-2 disabled:opacity-50 disabled:cursor-not-allowed"
  aria-label="Submit Form"
>
  <svg aria-hidden="true" class="w-4 h-4 mr-2" ...></svg>
  Submit
</button>
```

- **변경 사유 보고서 (AI의 출력 예시):**
  - **[Rule 1, 11]** 하드코딩된 테마 색상 및 다크모드 분기 폐기. Semantic 토큰(`bg-primary`)으로 전면 대체.
  - **[Rule 2]** 매직 넘버 여백(15px, 7px)을 8px 그리드 시스템(`px-4`, `py-2`)에 맞춰 기계적으로 스냅 처리.
  - **[Rule 6]** 마우스 사용 시 외곽선을 숨기고, 키보드 탐색(Tab) 시에만 명확히 동작하는 글로벌 포커스 링 구조 적용.
  - **[Rule 8, 9]** 색맹 및 스크린 리더 사용자의 접근성 확보를 위한 `aria-label` 속성 및 의미론적 아이콘 필수 추가.
  - **[Rule 12]** 다국어 번역 시 발생하는 텍스트 팽창에 안전하게 대응하기 위해 하드코딩된 절대 너비(`w-[150px]`)를 내재적 크기(`min-w-32`)로 유연하게 변경.
