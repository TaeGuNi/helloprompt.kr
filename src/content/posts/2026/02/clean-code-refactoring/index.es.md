## 📝 ¡Mi Código es Basura! Entrenador de Refactorización de Código Limpio

- **🎯 Recomendado para:** Desarrolladores de todos los niveles, ingenieros de software, estudiantes de TI
- **⏱️ Tiempo requerido:** 5 minutos → Reducido a 1 minuto
- **🤖 Modelo recomendado:** Cualquier modelo de IA conversacional (ChatGPT, Claude, Gemini, etc.)

- ⭐ **Dificultad:** ⭐⭐☆☆☆
- ⚡️ **Efectividad:** ⭐⭐⭐⭐⭐
- 🚀 **Utilidad:** ⭐⭐⭐⭐☆

> _"Dije que lo arreglaría más tarde, pero ya pasó un año. Ahora me aterra siquiera abrir ese archivo."_

El código espagueti es el enemigo público número uno de cualquier equipo de desarrollo. Cuando la legibilidad es deficiente, rastrear _bugs_ se convierte en una auténtica pesadilla y añadir nuevas funcionalidades resulta casi imposible. Este _prompt_ invoca el espíritu de Robert C. Martin (autor de _Clean Code_), analizando, depurando y puliendo tu código hasta dejarlo impecable.

---

## ⚡️ Resumen en 3 líneas (TL;DR)

1. **Extracción de métodos:** Elimina el código duplicado y modulariza las funciones demasiado extensas.
2. **Nomenclatura semántica:** Reemplaza los nombres ambiguos de variables y funciones por términos descriptivos y evidentes.
3. **Sintaxis moderna:** Optimiza el rendimiento aplicando las características más avanzadas y recientes de tu lenguaje de programación.

---

## 🚀 La Solución: "Lavandería de Código Limpio"

### 🥉 Versión Básica (Basic Version)

Úsala cuando necesites un resultado rápido y directo sin demasiadas explicaciones.

> **Rol:** Eres un experto Evangelista de Clean Code y Arquitecto Jefe de Software.
>
> **Solicitud:** Refactoriza el siguiente código para que sea limpio, legible, eficiente y fácil de mantener, respetando estrictamente las mejores prácticas del lenguaje.

### 🥇 Versión Pro (Pro Version)

Úsala cuando necesites un análisis arquitectónico profundo y una refactorización de altísima calidad.

> **Rol (Role):** Eres un `[Evangelista de Clean Code y Arquitecto Jefe de Software]`.
>
> **Contexto (Context):**
>
> - Antecedentes: Mi código funciona, pero es un desastre y resulta extremadamente difícil de leer (_Dirty Code_).
> - Objetivo: Refactorizarlo para que cumpla con los más altos estándares de calidad de la industria, asegurando que sea escalable y fácil de mantener.
>
> **Tarea (Task):**
>
> 1. **Nomenclatura Intuitiva**: Cambia los nombres de variables y funciones para que revelen su intención de forma clara (ej., `a` -> `userAge`).
> 2. **Mejora de Estructura**: Divide las funciones que son demasiado largas basándote en el Principio de Responsabilidad Única (SRP) y consolida la lógica duplicada.
> 3. **Modernización**: Si es posible, aplica la sintaxis más reciente y eficiente del lenguaje (ES6+, Python 3.10+, Java 17+, etc.).
> 4. **Análisis Comparativo**: Proporciona una explicación detallada de _qué_ cambiaste y _por qué_, comparando el estado anterior con el nuevo.
>
> **Restricciones (Constraints):**
>
> - La lógica de negocio y la funcionalidad del código **jamás** deben alterarse.
> - Agrega comentarios únicamente donde sea absolutamente necesario para explicar decisiones arquitectónicas complejas. El código debe ser autodescriptivo.
> - La respuesta debe estructurarse de forma clara usando bloques de código Markdown.
>
> **Código Objetivo (Target Code):**
>
> `[Pega aquí el código que deseas refactorizar]`

## 💡 Comentarios del Autor (Insight)

Este _prompt_ es un salvavidas absoluto cuando heredas un proyecto _legacy_ o cuando tienes que enfrentarte a una revisión de código (_Code Review_) de forma rápida. La clave de su éxito radica en su enfoque estricto en el **Principio de Responsabilidad Única** y en la **eliminación de números mágicos**. A diferencia de pedirle a la IA un simple "mejora este código", al asignarle el rol de un Arquitecto Jefe y exigirle que justifique sus decisiones, no solo obtienes un código impoluto, sino que recibes una clase magistral de ingeniería de software. Es una herramienta indispensable para aquellos desarrolladores que buscan interiorizar y aplicar las mejores prácticas de programación en su día a día.

## 🙋 Preguntas Frecuentes (FAQ)

- **Q: ¿La IA puede romper mi código al refactorizarlo?**
  - A: Aunque hemos incluido una directriz estricta ("la funcionalidad jamás debe alterarse"), los modelos de IA pueden sufrir alucinaciones. Es absolutamente obligatorio contar con pruebas unitarias (_Unit Tests_) que cubran el código original para garantizar que la versión refactorizada pase las validaciones sin problemas.

- **Q: ¿Funciona bien con fragmentos de código muy largos?**
  - A: Los modelos de IA tienen un límite de atención y de ventana de contexto. Si tu archivo contiene miles de líneas, resulta mucho más efectivo y seguro refactorizar función por función o clase por clase, en lugar de procesar todo el documento de golpe.

- **Q: ¿Qué pasa si uso un lenguaje de programación antiguo o con reglas muy específicas?**
  - A: Simplemente necesitas especificar la versión exacta de tu lenguaje o las convenciones internas de tu equipo en la sección de "Restricciones" (ej., "Utiliza sintaxis compatible con Java 8 y sigue la guía de estilo estandarizada de Google").

## 🧬 Anatomía del Prompt (Why it works?)

1.  **Rol de Autoridad:** Al posicionar a la IA como un "Arquitecto Jefe", el modelo descarta automáticamente las soluciones mediocres y se concentra en aplicar patrones de diseño robustos y convenciones reconocidas por la industria.
2.  **Restricciones de Comentarios:** Neutraliza la tendencia de la IA a sobrecomentar cada línea evidente, obligándola a escribir código que se explique por sí mismo a través de una nomenclatura puramente semántica.
3.  **Explicación Requerida:** Forzar a la IA a justificar el "Antes y Después" garantiza que no alterará el código de forma arbitraria, sino que aplicará un razonamiento arquitectónico sólido a cada modificación.

## 📊 Prueba: Before & After

### ❌ Before (Entrada)

```python
def calc(x, y):
    if x > 19:
        print("Adulto")
        p = y * 0.9
        return p
    else:
        print("Estudiante")
        return y
```

_(El nombre de la función `calc` es ambiguo, la variable `p` carece de significado semántico y existe una lógica condicional anidada innecesaria)._

### ✅ After (Resultado)

```python
def calculate_discounted_price(age: int, base_price: float) -> float:
    """Calcula el precio final aplicando un descuento según la edad."""
    ADULT_AGE_THRESHOLD = 20
    ADULT_DISCOUNT_RATE = 0.9

    if age >= ADULT_AGE_THRESHOLD:
        print("Eres un adulto.")
        return base_price * ADULT_DISCOUNT_RATE

    print("Eres un estudiante.")
    return base_price
```

**[Puntos de Mejora]**

1. **Nomenclatura Semántica**: `calc` se transformó en `calculate_discounted_price`. Ahora la intención de la función es innegable.
2. **Eliminación de "Magic Numbers"**: Se extrajeron los valores `19` y `0.9` en constantes autodescriptivas como `ADULT_AGE_THRESHOLD`.
3. **Patrón _Early Return_**: Se eliminó la cláusula `else` redundante, reduciendo el nivel de anidación y mejorando drásticamente la legibilidad.
4. **Tipado Estático**: Se añadieron sugerencias de tipo (_Type Hints_) para `age` y `base_price`, aumentando la seguridad y predictibilidad del código.
