# 🎨 构建设计系统：设计师与开发者如何和平共处

- **🎯 推荐对象：** 经常被抱怨“按钮颜色和设计稿不一样”的开发者，以及经常受挫于“开发者毁了我的设计”的设计师
- **⏱️ 所需时间：** 10 分钟 → 缩短至 1 分钟（初始 Token 设计与代码自动转换）
- **🤖 推荐模型：** 所有对话型 AI (ChatGPT, Claude, Gemini 等)

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐⭐

> _"主色调 (Primary Color) 不是 `#0055FF` 吗？为什么这里变成了 `#0050FF`？"_

散落在代码库各个角落、硬编码的颜色和间距值，是让后期维护沦为地狱的罪魁祸首。请引入**设计令牌 (Design Token)**。这是将颜色、字体、间距、阴影等所有视觉元素定义为有意义的变量，并在团队间共享的最可靠的契约。一旦建立起基于 Token 的系统，当设计师在 Figma 中修改数值的瞬间，开发者的代码也会自动同步，你将体验到这种宛如魔法般的流畅协作。

---

## ⚡️ 核心三要素 (TL;DR)

1. **设计 Token 化：** 摒弃绝对的十六进制 (Hex) 颜色值（如 `#0055FF`），转而使用具备语义化的通用语言，例如 `primary-500`。
2. **应用 Figma Variables：** 充分利用 Figma 的 Variables（变量）和 Styles（样式）功能，将视觉属性系统化地定义为 Token。
3. **代码自动同步：** 基于提取出的 Token 数据，自动转换为 `tailwind.config.ts` 配置或全局 CSS 变量，无缝应用到项目代码中。

---

## 🚀 解决方案："Design Token Generator Prompt"

### 🥉 Basic Version (基础版)

当你刚开始构建初始调色板，对如何命名变量感到毫无头绪时，可以使用此提示词。

> **任务 (Task)：**
> 我正在为一个新产品规划设计系统的调色板。
> 我想将 `[Primary(蓝色)]`、`[Secondary(灰色)]` 和 `[Error(红色)]` 这三种颜色，分别从 100 到 900 划分为 9 个层级。
> 为了能将这些颜色直接应用到代码中，请为每个层级推荐直观且语义化 (Semantic) 的命名（例如：`text-primary`、`bg-surface-default`），并将每个层级对应的十六进制 (Hex) 代码整理成表格形式输出。

<br>

### 🥇 Pro Version (专业版)

当你需要将 Figma 中提取的 Token JSON 数据，完美转换为符合实际前端环境的代码时使用。

> **角色 (Role)：** 你是一位拥有 10 年经验的资深 Design Ops（设计运维）工程师兼前端架构师。
>
> **背景 (Context)：**
>
> - 背景：设计师已经在 Figma 中定义了设计 Token，并将其导出为 JSON 格式。
> - 目标：我需要将这份 JSON 数据完美转换为可立即应用于前端项目的代码。
>
> **输入数据：**
>
> `[请在此处粘贴从 Figma 导出的 Token JSON 数据]`
>
> **任务 (Task)：**
>
> 1. 分析输入数据，并将其转换为最新的 **Tailwind CSS 配置文件 (`tailwind.config.ts`)** 格式。
> 2. 为了不依赖特定的 CSS 框架，请同时生成 **CSS Variables (`:root { --color-blue-500: ... }`)** 代码。
> 3. 添加暗黑模式（`@media (prefers-color-scheme: dark)` 或 `.dark` 类）的适配策略和示例代码，确保系统能根据主题动态切换。
>
> **约束条件 (Constraints)：**
>
> - 输出格式仅限使用 Markdown 代码块（`ts`, `css`）。
> - 所有变量名必须严格遵循短横线命名法 (kebab-case)。
>
> **注意事项 (Warning)：**
>
> - 严禁在提供的 JSON 结构之外，凭空捏造任何颜色值或层级。（防止 AI 幻觉）

## 💡 作者点评 (Insight)

构建设计系统绝不是制定一个宏大的总体规划然后一蹴而就。如果你试图从一开始就将所有组件和排版完美地 Token 化，十有八九会因为过度消耗精力而半途而废。
我强烈建议采用**渐进式采用 (Incremental Adoption)** 策略：从**“最常用、最频繁重复出现的元素（例如：主按钮的背景色、正文文本颜色）”**开始，逐一将其替换为变量。
哪怕只是一个小小的 Token，也将成为大幅减少设计师与开发者之间不必要沟通成本的完美开端。

## 🙋 常见问题 (FAQ)

- **Q: 在 Figma 中提取 Token 应该使用什么插件？**
  - A: 在实际业务中，**'Tokens Studio for Figma'** 几乎是事实上的标准 (De facto)。通过它，你可以导出 JSON 并直接与 GitHub 仓库同步 (Sync)。最近，Figma 原生的 Variables 功能也变得非常强大，通过 REST API 进行集成的方案也越来越受欢迎。

- **Q: 我们是只有 2~3 个人的初创团队，也需要设计系统吗？**
  - A: 只要团队成员超过 2 人，或者项目的维护周期超过 3 个月，就必须引入设计系统。那些“以后有时间再整理”的技术债务，最终都会变成严重拖慢项目进度的定时炸弹。毕竟，未来的你 (Future Self) 对现在的你来说，也是个陌生人。

## 🧬 提示词剖析 (Why it works?)

1. **强制语义化命名：** 引导 AI 放弃使用“浅蓝色”这种描述，转而使用基于含义的命名法，如“主要按钮背景色 (`bg-primary-default`)”。这样一来，即使将来品牌色从蓝色全面更改为紫色，你也只需修改一个 Hex 值，即可瞬间应用到整个项目。
2. **委托重复性的格式转换：** 让开发人员手动将 JSON 格式逐字输入转换为 Tailwind 配置对象或 CSS 变量语法，是一件极其痛苦的事。AI 在这种基于模式的简单代码转换和结构化任务中，展现出了压倒性的准确度和速度，从根本上杜绝了人为错误 (Human Error)。

## 📊 验证：Before & After

### ❌ Before (输入)

每次设计变更时，都需要在 100 多个文件中逐一搜索并替换十六进制代码。😱

```css
.button-primary {
  background-color: #3b82f6;
}

.text-highlight {
  color: #3b82f6;
}
```

### ✅ After (结果)

只需修改一个变量值，整个产品的 UI 就能完美同步更新。🚀

```css
:root {
  --primary-500: #3b82f6;
  --bg-primary-default: var(--primary-500);
}

.button-primary {
  background-color: var(--bg-primary-default);
}
```
