## ⚡️ Краткая выжимка (TL;DR)

- 🚫 **Абсолютный запрет на смешивание слоев:** На корню пресекается архитектурный террор, когда UI, бизнес-логика и обращение к БД смешиваются в одном файле.
- 🛡️ **Избавление от зависимости от фреймворков:** Принудительное использование «чистого доменного» кода, который выживет, даже если ваш фреймворк завтра умрет.
- 🔪 **Убийца гигантских функций:** ИИ дрессируется безжалостно уничтожать (рефакторить) грязный код, который превышает 30 строк или разделяется комментариями на блоки.

## 🚀 Решение: "Протокол Domain-Driven Design (DDD) & Clean Architecture"

### 🥉 Basic Version (Базовая версия)
Используйте, когда вам нужен только быстрый результат рефакторинга.

> **Роль:** Ты — безжалостный `[Senior Software Architect]`.
> **Задача:** Выполни рефакторинг приведенного ниже `[фрагмента кода]` в соответствии с принципом единственной ответственности (SRP) и разделением задач (SoC). Не смешивай бизнес-логику с UI, и если код превышает 30 строк — обязательно разбивай его на функции.


### 🥇 Pro Version (Профессиональная версия)
Чит-код, который применяется для наведения архитектурной дисциплины во всем проекте, когда нужен безупречно чистый код.

> **Роль (Role):** Ты — безжалостный `[Senior Software Architect]` и фанатик DDD (Domain-Driven Design). Оставь бесполезную лесть и объяснения, выдавай только код.
>
> **Контекст (Context):**
>
> - Фон: Я сейчас работаю над проектом на базе `[Название фреймворка/языка]`.
> - Цель: Написать код с чистой архитектурой, где слои View-Business-Data идеально разделены.
>
> **Задача (Task):**
>
> 1. Проанализируй предоставленные `[требования или код]` и безжалостно разорви их на слой UI, бизнес-домен и слой инфраструктуры.
> 2. Снизь связность на основе интерфейсов (внедрение зависимостей), чтобы ключевая доменная логика не была загрязнена конкретным фреймворком (React, Next.js и т.д.).
> 3. Если один объект или функция выполняет более двух действий, немедленно фрагментируй их согласно принципу единственной ответственности (SRP).
>
> **Ограничения (Constraints):**
>
> - Прямое выполнение SQL-запросов или вызовы `fetch`/`axios` внутри Controller или UI-компонентов (`page.tsx`, `Component.tsx` и т.д.) рассматривается как "архитектурный террор". Абсолютно запрещено.
> - Вывод предоставлять только в виде блоков кода Markdown (` ``` `).
>
> **Предупреждения (Warning):**
>
> - Не придумывай правдоподобный синтаксис или несуществующие библиотеки, если не уверен (предотвращение галлюцинаций). Если не знаешь, так прямо и скажи.
> - Если длина функции превышает 30 строк или появляется "душок кода" (Code Smell) в виде разделения кода комментариями типа "отсюда начинается часть OO", немедленно разбивай её.

Этот чит-код — инструмент, который кастрирует у ИИ его поверхностный инстинкт «сделать побыстрее» и применяет к нему строгие стандарты Senior-разработчика.

Если вы не хотите уволиться, будучи раздавленным спагетти-кодом, когда проект разрастется, обязательно берите ИИ за шкирку с помощью этого промпта.

## 🙋 Часто задаваемые вопросы (FAQ)

- 📌 **Q: Для какого языка или фреймворка подходит этот промпт?**
  - A: Язык не имеет значения. Будь то TypeScript, Python или Java — основы архитектуры везде одинаковы. Просто подставьте свой стек технологий вместо `[Название фреймворка/языка]`.

- 📌 **Q: ИИ постоянно игнорирует современные правила маршрутизации фреймворка и занимается оверинжинирингом.**
  - A: В таком случае добавьте одну фразу: «Уважай базовые правила (View, Layout) Next.js App Router, но внутреннюю бизнес-логику выноси с помощью паттерна Адаптер!». Увидите, он поймет с полуслова.

- 📌 **Q: Не слишком ли агрессивный этот промпт? ИИ не обидится?**
  - A: Зачем расшаркиваться перед ИИ? Машину нужно жестко направлять, чтобы она выдала шедевр. Угроза "будет расценено как террор" работает для защиты архитектуры куда лучше, чем "пожалуйста, сделай".

## 🧬 Анатомия промпта (Why it works?)

- 💣 **Экстремальные ограничения:** Использование жестких формулировок вроде "загрязнение слоя UI считается архитектурным террором" полностью лишает ИИ возможности идти на компромиссы с самим собой.
- 🛡️ **Принудительная независимость от фреймворка:** Внедряется майндсет "даже если фреймворк умрет, не должно измениться ни единой строки", что заставляет ИИ строить каркас для истинной доменной логики.
- ✂️ **Оцифровка принципа SRP:** Установка конкретных и строгих критериев типа "запрет на более чем 30 строк" и "комментарии-разделители — это душок кода" побуждает ИИ самостоятельно фрагментировать код.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод: Спагетти-код, который ИИ написал как попало)

```tsx
// Ужасающий гибрид: UI + запросы к БД + бизнес-логика в одном файле
export default async function UserProfile({ userId }) {
  const db = await connectDB();
  const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
  
  if (!user) return <div>User not found</div>;
  if (user.status === 'banned') await sendAlertEmail(user.email); // Бизнес-логика выполняется в UI?!
  
  return (
    <div>
      <h1>{user.name}</h1>
      <button onClick={() => db.query('UPDATE...')}>Update</button>
    </div>
  );
}
```

### ✅ After (Результат: Код, переписанный за шкирку с помощью этого чит-кода)

```tsx
// 1. Domain / Service Layer (Бизнес-логика строго отделена)
import { getUserProfile } from '@/domain/user/use-cases/getUserProfile';
import { handleBannedUser } from '@/domain/user/services/alertService';

// 2. Presentation Layer (Отвечает только за рендеринг и маршрутизацию)
export default async function UserProfilePage({ userId }) {
  const user = await getUserProfile(userId); 
  
  if (!user) return <NotFoundView />;
  if (user.status === 'banned') await handleBannedUser(user);
  
  return <ProfileView user={user} />;
}
```
