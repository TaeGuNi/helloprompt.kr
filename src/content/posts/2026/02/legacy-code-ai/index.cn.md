> _"还在几万行的“祖传代码”里苦苦排查 Bug 吗？是时候让 AI 成为你的首席重构工程师了。"_

接手毫无文档、原作者早已离职的老旧项目，无疑是每位开发者的噩梦。沉重的技术债务不仅严重拖慢了新功能的迭代节奏，更让人身心俱疲。然而，得益于先进的 AI 大语言模型，我们如今终于能轻松化解这一难题。本指南将带你掌握如何利用 AI 安全、高效地剖析与重构遗留代码。

## ⚡️ 3条核心摘要 (TL;DR)

1. **精准解析代码：** 让 AI 快速理清错综复杂的业务逻辑与依赖关系，并自动生成清晰易读的文档。
2. **构筑安全网：** 在动手修改任何代码前，先由 AI 辅助补全单元测试，确保重构过程万无一失。
3. **现代化转型：** 将陈旧的语法与框架一键升级，完美契合现代工程标准的最佳实践。

## 🚀 解决方案：“遗留代码拯救者”提示词

### 🥉 基础版 (Basic Version)

适用于需要快速弄懂一段宛如天书的函数逻辑时。

> **角色：** 你是一位拥有 20 年经验的资深架构师。
> **请求：** 请用通俗易懂的语言解释以下代码的核心逻辑，并精准指出其中可能存在的两个最严重的性能或安全隐患。

### 🥇 进阶版 (Pro Version)

适用于需要对核心模块进行深度重构，且对安全性有极高要求时。

> **角色 (Role)：** 你是一位顶尖的软件工程师兼重构专家，精通各类设计模式与遗留代码的现代化改造（如《重构》一书中所述的理念）。
>
> **背景 (Context)：**
>
> - 当前状况：我接手了一段毫无注释的 `[语言/框架名称]` 遗留代码。目前该代码耦合度极高，维护举步维艰。
> - 最终目标：在绝对不改变原有业务逻辑的前提下，将其重构为符合现代 `[目标语言/框架名称]` 规范、高内聚低耦合的整洁代码，并为其补充完整的单元测试。
>
> **任务 (Task)：**
>
> 1. 首先，深度剖析并解释这段代码的业务意图与核心逻辑。
> 2. 编写针对当前代码各项边界情况（Edge Cases）的单元测试，确保测试覆盖率。
> 3. 提供重构后的代码。要求：严格遵守命名规范、提取重复逻辑、彻底消除嵌套地狱（Callback Hell / Deep Nesting）。
> 4. `[需重点优化的函数名称]` 部分是重灾区，请务必针对该部分提供详细的优化思路对比。
>
> **约束条件 (Constraints)：**
>
> - 请务必使用 Markdown 格式输出。
> - 测试代码必须基于 `[测试框架名称，如 Jest 或 JUnit]` 编写。
> - 必须保证重构前后函数的输入与输出类型绝对一致。
>
> **警告 (Warning)：**
>
> - 若发现某段逻辑存在明显的业务缺陷但无法确切把握其真实意图，请务必明确指出，严禁主观臆测并擅自修改业务规则。

## 💡 作者见解 (Insight)

重构遗留代码时最致命的陷阱莫过于“为了重构而重构”，这往往会导致线上业务全线崩溃。这套提示词的核心哲学正在于 **“测试先行” (Test-Driven Refactoring)**。
先让 AI 读懂代码并编写出测试用例，这一步便牢牢锁定了原有的业务逻辑，相当于为你的重构拉起了一张坚不可摧的安全网。我曾利用这种方法，在一个下午的时间内，将一个充斥着全局变量的 3000 行老旧 jQuery 脚本，安全平滑地迁移到了现代的 React Hooks 架构中，且未引发任何线上 P0 级事故。

## 🙋 常见问题 (FAQ)

- **Q: 如果代码中包含公司机密，可以直接发给 AI 吗？**
  - A: 绝对不行。在使用公共 AI API 时，请务必先进行数据脱敏（例如替换真实的 API Key、数据库表名以及特定的业务参数），或者选择使用公司内部部署的私有大模型及企业版服务（如 ChatGPT Enterprise 等明确承诺不会将用户数据用于模型训练的服务）。

- **Q: AI 生成的重构代码一定能直接运行吗？**
  - A: 并非 100% 完美。AI 偶尔会出现幻觉或引入未知的依赖项。请务必在本地环境中运行 AI 提供的单元测试，并经过严格的代码审查（Code Review）之后再进行代码合并。请记住，AI 只是你的得力副驾驶，方向盘必须始终掌握在你手中。

## 🧬 提示词解剖 (Why it works?)

1. **明确目标与约束 (Constraints)：** 严正强调“不改变原有业务逻辑”与“保持输入输出一致”，有效防止 AI 擅自发挥而破坏系统的整体稳定性。
2. **角色注入与专业对标：** 赋予其“架构师”与“重构专家”的身份，促使模型激活其训练数据中关于《重构》、《Clean Code》等经典工程著作的高质量专业认知。
3. **拆解任务 (Chain-of-Thought)：** 强制要求 AI 严格按照“理解 -> 编写测试 -> 重构 -> 解释”的步骤逐一执行，这不仅极大地降低了出错概率，更方便开发者按部就班地进行验证。

## 📊 效果对比：Before & After

### ❌ Before (旧代码片段)

```javascript
function processData(d) {
  var res = [];
  for (var i = 0; i < d.length; i++) {
    if (d[i].status == 1 && d[i].t > new Date().getTime() - 86400000) {
      var x = d[i].v * 1.2; // tax
      res.push({ id: d[i].id, val: x });
    }
  }
  return res;
}
```

### ✅ After (AI 重构结果)

```javascript
/**
 * 筛选过去24小时内状态为激活(1)的数据，并计算含税价值(20%税率)。
 */
const TAX_RATE = 1.2;
const ONE_DAY_MS = 86400000;

export function processRecentActiveData(dataItems) {
  if (!Array.isArray(dataItems)) return [];

  const twentyFourHoursAgo = Date.now() - ONE_DAY_MS;

  return dataItems
    .filter((item) => item.status === 1 && item.t > twentyFourHoursAgo)
    .map((item) => ({
      id: item.id,
      val: Number((item.v * TAX_RATE).toFixed(2)),
    }));
}
```
