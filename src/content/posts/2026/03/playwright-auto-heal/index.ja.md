# 📝 E2Eテストがエラーで止まる？PlaywrightのAuto-healingプロンプトを使えば1分で解決 🛠️

- **🎯 推奨ターゲット:** QAエンジニア、フロントエンドエンジニア、テスト自動化担当者
- **⏱️ 所要時間:** 2時間 → 1分に短縮
- **🤖 推奨AIモデル:** Claude 3.5 Sonnet, GPT-4o
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「昨日まで完璧に動いていたE2Eテストが、今朝ボタンのCSSクラスが一つ変わっただけで全滅しました。またDevToolsを開いてセレクタを探し直さないといけないのでしょうか？」_

フロントエンド開発やQA業務において、UIの微細な変更によってE2E（End-to-End）テストが壊れる（Flakyになる）のは日常茶飯事です。特にPlaywrightやCypressといったツールでは、DOM構造やクラス名が少し変わるだけで、テストコードは容赦なくエラーを吐き出します。

エラー箇所を特定し、ブラウザの開発者ツールを開いて新しいセレクタをコピーし、コードを修正する……この繰り返しは、膨大な時間の無駄です。これからは、AIに「エラーログ」と「変更されたHTML構造」を渡すだけで、自律的にコードを「Auto-healing（自動復旧）」させるアプローチを取り入れましょう。

---

## ⚡️ 3行まとめ (TL;DR)

1. UI変更によって壊れたPlaywrightのセレクタを、AIが自動分析して即座に復旧させるプロンプトです。
2. 単なるコード修正にとどまらず、将来の変更にも強い（Resilientな）セレクタ戦略まで提案してくれます。
3. エラーログと最新のDOM要素をコピペして渡すだけで、修正済みの完璧なコードが1分以内に手に入ります。

## 🚀 解決策：「Playwright Auto-healing プロンプト」

### 🥉 Basic Version (基本型)

スピード重視で、壊れたコードとエラーメッセージからすぐに修正版のコードを受け取りたい場合に使用します。

> **役割:** あなたは `[シニアQAエンジニア]` です。
> **タスク:** 提供されたPlaywrightのエラーログと新しいHTML構造を基に、以下の `[壊れたテストコード]` を修正してください。

### 🥇 Pro Version (専門家型)

単なる応急処置ではなく、メンテナンスが容易で堅牢なテストコードへとリファクタリングしたい場合に使用します。

> **役割 (Role):** あなたはテスト自動化を専門とする `[シニアSDET (Software Development Engineer in Test)]` です。Playwrightのベストプラクティスを熟知しています。
>
> **状況 (Context):**
>
> - 背景: 最近のUIアップデートにより、既存のPlaywrightのE2Eテストが失敗しています。主な原因はセレクタ（Selector）の変更です。
> - 目標: 失敗したテストコードを復旧させると同時に、今後のUI変更にも耐えうる（Resilientな）堅牢なセレクタ構造へと改善すること。
>
> **タスク (Task):**
>
> 1. 以下に提供された `[エラーログ]` と `[変更されたHTML構造]` を分析し、エラーの根本原因を特定してください。
> 2. `[既存のテストコード]` を修正し、テストが確実にパスするように書き直してください。
> 3. 修正の際、壊れやすいCSSクラスやXPathへの依存を避け、ユーザー向け属性（表示テキスト、ARIA属性、data-testidなど）を優先的に使用してください。
> 4. 修正箇所には、「なぜこのアプローチで修正したのか」という簡潔な理由をコードコメントとして付与してください。
>
> **入力データ:**
> - エラーログ: `[エラーログ全体をここにペースト]`
> - 変更されたHTML構造: `[DevToolsからコピーした該当要素のHTMLスニペットをここにペースト]`
> - 既存のテストコード: `[失敗しているPlaywrightのテストコードをここにペースト]`
>
> **制約事項 (Constraints):**
>
> - 出力は、そのままコピー＆ペーストして使える完全なコードブロック形式（TypeScript/JavaScript）で提供してください。
> - Playwrightが推奨する組み込みLocators（例: `getByRole`, `getByText`, `getByTestId`）の利用を最優先とすること。
>
> **注意事項 (Warning):**
>
> - もし提供されたHTMLスニペットの中に適切な要素が見当たらない場合は、勝手に推測してコードを生成しないでください。その場合は「HTML情報が不足しているため、堅牢なセレクタを特定できません。より広い範囲のDOM情報を提供してください」と正直に回答して、幻覚（Hallucination）を防いでください。

## 💡 作成者コメント (Insight)

このアプローチは、大規模なUIリニューアルなどで数十個のテストが一斉に落ちた瞬間に真価を発揮します。

過去の私は、コンポーネントの構造が一つ変わるたびに、関連する30個のテストファイルをエディタで開き、`Ctrl + F` でひたすらセレクタを置換するという不毛な作業に追われていました。しかし、このプロンプトをGitHub CopilotやClaudeにスニペットとして登録してからは劇的に変わりました。エラーログと、ブラウザのElementsタブからコピーした数行のHTMLを渡すだけで、あの面倒なセレクタ抽出と置換作業をAIに完全アウトソーシングできるのです。

ここで最も重要なポイントは、**「ユーザー向け属性（User-Facing Locators）を使用しろ」**という制約事項です。AIは放っておくと、一番手っ取り早いCSSクラス（例：`.btn-primary`）をターゲットにしがちです。しかし、この制約を明示することで、`getByRole('button', { name: 'Submit Order' })` のような、堅牢でアクセシビリティにも配慮した美しいコードを返してくれます。皆さんの退社時間を確実に1時間は早めてくれる強力な武器になります。騙されたと思って、次のテスト修正で試してみてください！

## 🙋 よくある質問 (FAQ)

- **Q: ページ全体のHTMLをコピーすると長すぎるのですが、どうすればいいですか？**
  - A: ページ全体のソースコードは不要です。エラーが発生した要素（例：クリックできなかったボタンやフォーム）とその親タグ（ラッパー要素）周辺の数行〜十数行をコピーして渡すだけで、AIは十分に文脈を理解できます。

- **Q: Playwright以外のテストフレームワーク（CypressやSeleniumなど）でも使えますか？**
  - A: もちろんです。プロンプト内の「Playwright」という単語をお使いのフレームワーク名（Cypress, Selenium, Puppeteerなど）に書き換えるだけで、各ツールのベストプラクティスに従ったコードを生成してくれます。

- **Q: 開発現場のルールで `data-testid` が付与されていないのですが、それでも機能しますか？**
  - A: はい、問題ありません。指定されたHTMLの中に `data-testid` が見つからない場合でも、AIは `getByRole` や `getByText` 、あるいはARIA属性などのアクセシビリティベースの堅牢な代替セレクタを賢く見つけ出して適用します。

## 🧬 プロンプトの解剖 (Why it works?)

1. **専門的なSDETペルソナの付与:** 一般的なプログラマーではなく、テスト自動化の専門家（SDET）というペルソナを持たせることで、「とりあえず動くコード」ではなく「Playwrightのベストプラクティスに準拠した高品質なコード」を生成するように方向付けています。
2. **User-Facing Locatorsの強制:** 「テストの不安定性（Flakiness）を根本から排除する」というPlaywrightの設計哲学をプロンプトの制約として強制することで、UI変更に極めて強いテストコードを生み出します。
3. **ハルシネーション（幻覚）防止の安全装置:** AIに十分なHTML情報が与えられていない場合に、適当なセレクタをでっち上げてデバッグをさらに混乱させることを防ぐため、「情報不足の時は素直に分からないと答えろ」というフェイルセーフを組み込んでいます。

## 📊 証明: Before & After

### ❌ Before (入力)

```text
エラーログ:
Error: locator.click: Target closed
=========================== logs ===========================
waiting for locator('.btn-primary.submit-btn')
============================================================
変更されたHTML構造:
<div class="actions">
  <button class="button--blue" aria-label="Submit Order">Order Now</button>
</div>
既存のテストコード:
await page.locator('.btn-primary.submit-btn').click();
```

### ✅ After (結果)

```typescript
// 変更理由: CSSクラス（'.btn-primary.submit-btn'）が変更され、テストが失敗しました。
// 改善事項: 変更されやすいCSSクラスの代わりに、アクセシビリティ属性（ARIA label）とRoleを使用し、UIの変更に強くなるようにしました。
await page.getByRole('button', { name: 'Submit Order' }).click();
```
