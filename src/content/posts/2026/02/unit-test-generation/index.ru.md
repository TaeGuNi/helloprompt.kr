# 📝 Слишком ленивы писать код тестов? ИИ сделает это за вас

- **🎯 Рекомендуется для:** Backend и Frontend разработчиков
- **⏱️ Требуемое время:** 5 минут → 30 секунд
- **🤖 Рекомендуемая модель:** Любая (ChatGPT, Claude, Gemini)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _"Писать код — весело, писать тесты — скучно. Но без тестов каждый деплой превращается в русскую рулетку."_

Модульное тестирование (Unit Testing) абсолютно необходимо для создания стабильных сервисов, но его написание часто отнимает много времени и сил. Особенно трудно предусмотреть все хитрые граничные случаи (Edge cases). Оставьте эту рутину искусственному интеллекту, а сами сосредоточьтесь на проектировании архитектуры и бизнес-логики.

---

## ⚡️ 3 кратких вывода (TL;DR)

1. ИИ способен мгновенно создавать надежные модульные тесты, покрывающие не только идеальные условия, но и скрытые ошибки.
2. Присвоение ИИ роли опытного QA-инженера заставляет его искать нестандартные сценарии и уязвимости.
3. Вы экономите время на рутине и можете отправлять код на ревью с уверенностью в его стабильности.

### 🥇 Профессиональная версия (Pro Version)

Используйте, когда требуется максимальное покрытие и готовый к продакшену код тестов.

> **Роль (Role):** Ты `[Ведущий QA-инженер]`.
>
> **Ситуация (Context):**
>
> - Я написал функцию/класс: `[Название]`.

>

> 3. Добавь комментарии к каждому тесту, объясняющие, что именно и почему он проверяет.
>
> **Код для тестирования:**
>
> ```
> [Вставьте ваш код сюда]
> ```
>
> **Ограничения (Constraints):**
>
> - Убедись, что все тесты полностью изолированы и независимы друг от друга.
> - Выведи только готовый код тестов в формате Markdown, без лишних вступлений и объяснений.

## 💡 Комментарий автора (Insight)

Написание тестов — это та задача, которую разработчики часто откладывают "на потом" из-за ее рутинности. Но с помощью ИИ вы можете превратить скучную обязанность в быстрый и эффективный процесс. Этот промпт особенно полезен для выявления граничных случаев (Edge cases), о которых замыленный глаз может даже не подумать при написании бизнес-логики. ИИ здесь выступает в роли "адвоката дьявола", целенаправленно пытаясь сломать ваш код, что делает итоговый продукт в разы стабильнее.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Можно ли использовать этот промпт для интеграционных или E2E (End-to-End) тестов?**
  - A: Да, но потребуется небольшая адаптация. Для интеграционных тестов обязательно опишите ИИ структуру вашей базы данных или API, чтобы он мог правильно смоделировать взаимодействие между компонентами системы.

- **Q: Что делать, если ИИ генерирует тесты, которые не проходят (падают)?**
  - A: Это отличный сигнал! Либо ИИ нашел реальный баг (недочет) в вашем коде, либо он неправильно понял контекст. Внимательно проверьте свою логику или добавьте больше поясняющих деталей в блок "Ситуация (Context)".

## 🧬 Анатомия промпта (Why it works?)

1. **Смена перспективы (Role):** Роль строгого QA-инженера заставляет ИИ мыслить не как разработчик (который просто хочет, чтобы код запустился), а как тестировщик (который ищет уязвимости).
2. **Фокус на уязвимостях (Task):** Прямое требование найти минимум 3 исключительные ситуации не позволяет ИИ ограничиться только базовыми "счастливыми путями".
3. **Требование к комментариям:** Заставляет ИИ аргументировать выбор проверок, что помогает вам понять логику теста.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод: простая функция деления)

```javascript
function divide(a, b) {
  if (b === 0) throw new Error("Нельзя делить на 0.");
  return a / b;
}
```

<br>

### ✅ After (Результат)

```javascript
// Пример с использованием Jest

describe("Тестирование функции divide", () => {
  // 1. Успешный сценарий (Happy path)
  test("Деление 10 на 2 должно возвращать 5", () => {
    expect(divide(10, 2)).toBe(5);
  });

  // 2. Граничный случай: отрицательные числа
  test("Деление с отрицательными числами должно давать корректный результат", () => {
    expect(divide(-10, 2)).toBe(-5);
    expect(divide(-10, -2)).toBe(5);
  });

  // 3. Исключительная ситуация: деление на ноль
  test("Деление на 0 должно выбрасывать ожидаемую ошибку", () => {
    expect(() => divide(10, 0)).toThrow("Нельзя делить на 0.");
  });

  // 4. Граничный случай: дробные результаты
  test("Деление, приводящее к дробному результату, должно обрабатываться точно", () => {
    expect(divide(5, 2)).toBe(2.5);
  });
});
```
