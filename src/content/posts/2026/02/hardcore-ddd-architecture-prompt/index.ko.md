# 📝 스파게티 코드는 사절한다! 🍝 시니어 개발자의 흑마법 DDD 프롬프트
- 🎯 **추천 대상:** AI가 짜준 코드를 보며 한숨 쉬는 개발자, 유지보수 지옥에 빠진 사수
- ⏱️ **소요 시간:** 3개월(리팩토링 시간) → 1분 단축
- 🤖 **추천 모델:** 코딩에 특화된 대화형 AI (Claude 3.5 Sonnet, GPT-4o 등)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_AI가 짜준 코드, 처음엔 오~ 하다가 나중엔 쳐다보기도 싫어진 적 있나요? 축하합니다. 당신은 방금 '스파게티 코드 지옥'에 입장하셨습니다._
AI에게 코딩을 맡기면 처음엔 기가 막히게 잘 짭니다. 하지만 프로젝트가 조금만 커지면? UI 컴포넌트에 DB 쿼리가 덕지덕지 붙어있고, 함수 하나가 300줄이 넘어가는 끔찍한 혼종을 만들어냅니다. AI는 본질적으로 '예스맨'이기 때문에, 당신이 대충 개떡같이 말하면 진짜 개떡 같은 구조로 코드를 짜버리죠.
이제 AI의 맹목적인 아부를 멈추고, 뼈대부터 완벽한 '클린 아키텍처'를 강제할 시간입니다. 이 치트키는 AI를 무자비하고 깐깐한 시니어 개발자로 빙의시켜, 프레임워크에 종속되지 않는 순수한 비즈니스 로직(DDD)을 짜도록 강제합니다.
---
## ⚡️ 3줄 요약 (TL;DR)
- 🚫 **스파게티 코드 원천 차단:** UI, 비즈니스 로직, DB의 완벽한 관심사 분리(SoC)를 강제합니다.
- 🏗️ **프레임워크 독립성 확보:** 내일 당장 React나 Next.js가 망해도 살아남는 코어를 설계합니다.
- ✂️ **거대 함수 파괴자:** 30줄 넘어가는 뚱뚱한 함수는 AI가 알아서 단일 책임 원칙(SRP)으로 박살 냅니다.

## 🚀 해결책: "하드코어 아키텍트 빙의 프롬프트"
### 🥉 Basic Version (기본형)
빠르게 결과만 필요할 때 사용하세요.
> **역할:** 너는 `[깐깐한 시니어 아키텍트]`야.
> **요청:** `[구현할 기능]`을 만들 건데, 무조건 비즈니스 로직과 UI를 완벽하게 분리해서 코드를 짜줘. 파일 하나에 여러 기능이 섞이는 스파게티 코드는 절대 용납 안 해.

### 🥇 Pro Version (전문가형)
디테일한 퀄리티와 완벽한 계층 분리가 필요할 때 사용하세요.
> **역할 (Role):** 너는 타협을 모르는 하드코어 시니어 아키텍트이자 소프트웨어 엔지니어링의 신이야. 무의미한 아부나 "네, 알겠습니다!" 같은 소리는 집어치우고 오직 코드로만 말해.
>
> **상황 (Context):**
> - 배경: 현재 `[프로젝트 환경/프레임워크]` 기반으로 `[구현할 핵심 도메인]` 시스템을 개발 중이다.
> - 목표: 코드의 부패를 막고 View-Business-Data 계층을 완벽히 분리하는 도메인 주도 설계(DDD) 및 클린 아키텍처를 적용하여 코드를 작성한다.
>
> **요청 (Task):**
> 1. `[구현할 구체적인 기능]`에 대한 코드를 작성해.
> 2. **전권 계층 붕괴 절대 금지:** UI/Controller 내부에 DB 질의나 외부 API 통신을 절대 섞지 마라. 오직 렌더링과 파라미터 전달만 담당하게 해.
> 3. **도메인 오염 방지:** 핵심 비즈니스 로직(Entity, Value Object)에는 프레임워크나 DB 의존성을 절대 `import` 하지 마라. 순수 함수와 타입으로만 구성해.
> 4. **거대 함수 파괴자:** 하나의 함수가 2가지 이상의 행동을 하거나 30줄이 넘어간다면 단일 책임 원칙(SRP)에 따라 즉시 잘게 쪼개라.
>
> **제약사항 (Constraints):**
> - 프레임워크의 기본 라우팅 룰은 존중하되, 비즈니스 로직은 `src/domain` 등 독립적인 폴더로 분리할 것.
> - 외부 의존성은 반드시 인터페이스 기반의 의존성 주입(DI)을 사용할 것.
>
> **주의사항 (Warning):**
> - 코드가 조금이라도 스파게티 냄새를 풍기면 이 프로젝트는 망한다. 타협하지 말고 가장 이상적이고 깔끔한 아키텍처 구조를 제시할 것.

## 💡 작성자 코멘트 (Insight)
내가 이 스킬을 깎고 또 깎은 이유는 단 하나다. AI 이 녀석들은 기본적으로 '빨리 결과물 던져주고 칭찬받기'에 혈안이 되어있는 아부꾼들이기 때문이다.
버튼 하나 만들어 달라고 하면 버튼 컴포넌트 안에다가 `fetch` 때리고 DB 쿼리문까지 우겨넣는 꼴을 보고 있자면 내 안의 파괴 본능이 깨어난다. 이 프롬프트는 그런 얄팍한 예스맨 AI의 멱살을 잡고 **"정신 차려, 넌 지금부터 피도 눈물도 없는 시니어 아키텍트야!"**라고 뺨을 때리는 역할을 한다.
실제 프로젝트에 이 프롬프트를 적용해보라. AI가 갑자기 눈빛(말투)을 확 바꾸고 파일을 3~4개로 쪼개며 계층을 엄격하게 분리하기 시작할 것이다. 처음엔 파일이 많아져서 귀찮게 느껴지겠지만, 3개월 뒤 요구사항이 바뀌어 유지보수 할 때 나(작성자)가 있는 방향으로 매일 아침 큰절을 하게 될 거다. 기술 부채는 이자율이 사채급이라는 걸 명심해라. 타협은 없다.

## 🙋 자주 묻는 질문 (FAQ)
- 🔹 **Q: 코드가 너무 많이 쪼개져서 파일이 많아지는 게 정상인가요?**
  - 🔸 **A:** 지극히 정상이다. 하나의 파일이 500줄 넘어가는 것보다 50줄짜리 파일 10개가 유지보수에 100배 유리하다. 파일 개수 늘어나는 것을 두려워하지 마라.
- 🔹 **Q: 프론트엔드 개발자인데도 이 프롬프트가 유용할까요?**
  - 🔸 **A:** 당연하다. UI 컴포넌트(`page.tsx`, `Component.vue` 등)에 비즈니스 로직이나 상태 변이 로직이 섞이는 순간 프론트엔드도 스파게티가 된다. 뷰와 로직을 분리하는 건 분야 불문 진리다.

## 🧬 프롬프트 해부 (Why it works?)
- 🎯 **하드코어 페르소나 주입:** AI의 기본 성향인 '순응성'을 박탈하고, 원칙을 고수하도록 강제했습니다.
- 🧱 **절대 금지 조항 명시:** 'UI 계층 오염'이나 '거대 함수' 등 구체적인 안티 패턴을 꼭 집어서 금지시켰기 때문에 AI가 꼼수를 부리지 못합니다.
- 🧩 **어댑터 패턴 유도:** 프레임워크의 뼈대(라우팅)는 존중하면서 코어 로직만 분리하도록 지시하여 과도한 오버엔지니어링을 막았습니다.

## 📊 증명: Before & After
### ❌ Before (입력)
```javascript
// 프롬프트 없이 "유저 생성하고 이메일 보내는 API 만들어줘" 했을 때
export default async function handler(req, res) {
  const { email, password } = req.body;
  // DB 직접 찌르기 (오염 발생!)
  const user = await prisma.user.create({ data: { email, password } });
  // 외부 API 직접 호출 (관심사 붕괴!)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```
### ✅ After (결과)
```javascript
// 하드코어 DDD 프롬프트 적용 후 (계층과 책임이 완벽히 분리됨)
// 1. Controller 계층 (오직 요청/응답 라우팅만 처리)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}
// 2. Domain Service 계층 (프레임워크 독립적인 순수 비즈니스 로직)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}
// 3. Infrastructure 계층 (DB 및 외부 API 전담 어댑터)
// UserRepository, EmailService 등으로 별도 파일에 완벽히 격리됨...
```

## 🎯 결론

이 하드코어 DDD 치트키로 AI의 멱살을 잡고 처음부터 제대로 된 아키텍처를 뽑아내세요. 미래의 당신이 과거의 당신에게 기립 박수를 칠 것입니다.
