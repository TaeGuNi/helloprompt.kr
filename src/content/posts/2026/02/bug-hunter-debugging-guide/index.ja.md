# 📝 原因不明のバグは「AI探偵」に丸投げしよう

- **🎯 おすすめの対象:** ジュニア開発者、フロントエンド/バックエンドエンジニア、終わらないデバッグに絶望している方
- **⏱️ 所要時間:** 2時間（頭を抱える時間） → 3分に短縮
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o (コード推論に強いモデル)

- ⭐ **難易度:** ⭐☆☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「スタックトレースを見ても、どこから手をつければいいのか全く見当がつかない…。」_

意味不明なエラーメッセージや、ログをいくら追っても原因が掴めないバグ。開発者なら誰もが一度は経験するあの絶望的な瞬間を救ってくれる「AIデバッグ探偵」プロンプトをご紹介します。エラーログと関連コードを投げるだけで、AIが論理的に根本原因を推論し、的確な解決策を提示してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. 難解なエラーログやスタックトレースをそのままAIに貼り付けるだけでOK。
2. 関連するコードスニペットを添えることで、推論の精度が劇的に向上します。
3. 単純なコードの修正にとどまらず、「なぜそのバグが発生したのか」という根本原因まで徹底的に解説してくれます。

---

## 🚀 解決策: "デバッグ探偵プロンプト"

### 🥉 Basic Version (基本型)

手っ取り早く解決への糸口を掴みたい時に活用してください。

> **役割:** あなたはシニアエンジニアです。
> **タスク:** 以下のエラーログを確認し、バグの原因と解決策を教えてください。
>
> **エラーログ:**
> `[ここにエラーログを貼り付ける]`
>
> **関連コード:**
> `[疑わしいコードを貼り付ける]`

### 🥇 Pro Version (専門家型)

複雑なバグの根本原因を正確に突き止め、副作用のない安全な修正コードを得たい場合に使用してください。

> **役割 (Role):** あなたはトラブルシューティングとパフォーマンス最適化に長けた、10年以上の経験を持つシニアソフトウェアエンジニア（AI探偵）です。
>
> **状況 (Context):**
>
> - 背景: アプリケーションの実行中に原因不明のエラーが発生し、開発が完全にストップしています。
> - 目標: エラーの根本原因（Root Cause）を正確に特定し、副作用（Side-effect）のない最適な修正コードを作成すること。
>
> **要求 (Task):**
> 提供されたエラーログとコードを分析し、以下の項目を順に説明してください。
>
> 1. **バグの根本原因:** なぜこのエラーが発生したのかを論理的に説明してください。
> 2. **修正コード:** 問題を解決するための具体的なコードを提示してください。
> 3. **予防策:** 今後同様のバグを防ぐためのアドバイスを1〜2行で追加してください。
>
> **入力データ (Input):**
>
> - エラーログ/スタックトレース: `[ここにスタックトレース全体を貼り付ける]`
> - 関連コード: `[問題が発生している周辺のコードを貼り付ける]`
> - 使用技術: `[例: React, Node.js, Pythonなど]`
>
> **制約事項 (Constraints):**
>
> - 出力形式は読みやすいマークダウンを使用してください。
> - 修正コードには、変更点がひと目でわかるようにコメントを記載してください。
> - 情報が不足していて原因を特定できない場合は、推測で答えず、「さらに確認すべきファイルやログ」を私に要求してください。

## 💡 筆者コメント (Insight)

このプロンプトの真の価値は、単に「動くコード」を得ることではなく、「シニアエンジニアの思考プロセス」そのものを学べる点にあります。
特にPro版の `情報が不足している場合は、推測で答えず追加情報を要求して` という制約は非常に重要です。これにより、AIが勝手な解釈（ハルシネーション）で誤ったコードを提示するのを未然に防ぎ、対話を通じて正確にバグを追い詰めることができます。長大なスタックトレースを前に途方に暮れた時は、一人で抱え込まずにまずはAI探偵に分析を任せてみましょう。精神的な負担が格段に軽くなるはずです。

## 🙋 よくある質問 (FAQ)

- **Q: エラーログが長すぎてプロンプトに入りきりません。どうすればいいですか？**
  - A: スタックトレースの「先頭の数行（Exceptionの種類とメッセージ）」と「自分のプロジェクトのファイルパスが含まれている行」を中心に抜粋して貼り付けてください。ライブラリ内部の深いログは省略しても問題ありません。

- **Q: どのAIモデルを使うのが最も効果的ですか？**
  - A: コードの文脈理解や論理的推論においては、現在 **Claude 3.5 Sonnet** が最も優れています。次点として OpenAI の **GPT-4o** をおすすめします。

## 🧬 プロンプトの解剖学 (Why it works?)

1.  **AI探偵ペルソナ (Role):** 「シニアエンジニア」という役割を与えることで、単なる文法エラーの指摘にとどまらず、アーキテクチャやベストプラクティスに基づいた深い洞察を引き出します。
2.  **根本原因の要求 (Task):** 単にコードを直すだけでなく「なぜ起きたか」を説明させることで、開発者自身の本質的なスキルアップに繋がります。
3.  **ハルシネーション防止 (Constraints):** 情報不足時に「追加情報を求める」よう指示することで、誤った解決策に振り回される無駄な時間を削ることができます。

## 📊 証明: Before & After

### ❌ Before (入力)

```text
TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (UserList.jsx:15:23)
    ...
```

### ✅ After (結果)

````markdown
### 🔍 バグの根本原因

`UserList.jsx` の 15行目で、`users` 配列に対して `.map()` メソッドを呼び出していますが、`users` が `undefined`（未定義）の状態でレンダリングが実行されているためです。非同期でデータを取得する前にコンポーネントが描画された可能性が高いです。

### 🛠️ 修正コード

```jsx
// 修正前: {users.map(user => ...)}
// 修正後: オプショナルチェーンング (?.) を使用するか、デフォルト値を設定します。

function UserList({ users }) {
  // users が undefined や null の場合は何も表示しないか、ローディングを出す
  if (!users) return <p>Loading...</p>;

  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```
````

### 🛡️ 予防策

配列を Props で受け取る際は、デフォルト値として空の配列 `[]` を設定する（例: `function UserList({ users = [] })`）か、TypeScriptを導入して型安全性を高めることをお勧めします。
