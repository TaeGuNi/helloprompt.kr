# 📝 利用AI寻找代码漏洞！（Bug猎手）

- **🎯 推荐对象：** 没有资深导师指导、独自奋斗的初级开发者；在代码部署前感到不安的实战开发者
- **⏱️ 预计耗时：** 调试 3小时 → 缩短至 10分钟
- **🤖 推荐模型：** Claude 3.5 Sonnet（代码分析最强者），ChatGPT (GPT-4o)

- ⭐ **难度：** ⭐⭐☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐⭐

> _"在我的电脑上运行得很好... 部署到生产环境会不会崩溃？"_

独自编写代码时，很容易陷入“不知道自己不知道什么”的可怕盲区。代码或许能运行，但可能速度缓慢、容易受到黑客攻击，或者是别人难以阅读的“面条代码”。
现在，让这位不知疲倦、拥有谷歌首席工程师水平的 AI 审查员 24 小时为您检查代码吧。从潜在的 Bug 到低效的逻辑，甚至是致命的安全漏洞，它都能为您敏锐地指出。

---

## ⚡️ 3句话总结 (TL;DR)

1. 不要盲目地把代码扔给 AI，请赋予 AI **谷歌首席工程师**的明确角色定位。
2. 不要仅仅说“请审查代码”，而是明确提出 Bug、安全性、可读性和性能这 **4个审查标准**。
3. 不要盲目相信 AI 的重构建议，要求它通过注释解释修改的原因，以便自我验证和学习。

### 🥉 Basic Version (基础版)

当您只需要快速扫描致命问题时使用。

> **角色：** 你是一名拥有 10 年经验的资深后端/前端开发者。
> **任务：** 请检查以下代码中是否存在致命的 Bug 或需要立即改进的反模式（Anti-pattern）。
>
>
> [粘贴代码]
>
### 🥇 Pro Version (专业版)

当您希望将代码质量提升至生产（Production）级别，并获得客观的重构依据时推荐使用。

> **角色 (Role)：** 你是一名曾就职于谷歌 (Google)、拥有 15 年经验的首席软件工程师。你是整洁代码 (Clean Code) 和系统优化的专家。
>
> **背景 (Context)：**
>
> - 语言：`[Python / JavaScript / Java 等]`
> - 框架：`[React / Spring / Django 等]`
> - 目的：这个函数负责`[用户登录处理和会话管理]`。
>
> **代码 (Input)：**
>
>
> [在此处粘贴需要审查的代码]
>
>
> **任务 (Task)：**
> 请从以下 4 个维度严格审查我的代码，并提供改进后的完整代码。
>
> 1. **Bug：** 遗漏的异常处理或运行时可能发生的潜在错误（Edge cases）。
> 2. **安全性 (Security)：** 是否存在 SQL 注入、XSS、CSRF 等框架级安全漏洞。
> 3. **可读性 (Readability)：** 变量/函数命名规范，是否违反单一职责原则 (SRP)。
> 4. **性能 (Performance)：** 不必要的循环，时间复杂度 (Big-O) 方面的性能瓶颈。
>
> **约束条件 (Constraints)：**
>
> - 提供改进后的代码时，必须在每处修改的地方通过注释（`//` 或 `#`）详细解释**为什么 (Why)** 这样重构。
> - 绝对不能破坏现有业务逻辑的意图。
> - 在不引入外部库的情况下，仅使用该语言的内置功能进行优化。
>
> **注意事项 (Warning)：**
>
> - 不要建议你不确定的语法或已弃用 (Deprecated) 的函数。如果你不知道，请直接回答不知道，防止产生幻觉 (Hallucination)。

## 🙋 常见问题 (FAQ)

- **Q: 可以把公司项目的全部代码直接复制给 AI 吗？**
  - A: **绝对不行！** 泄露公司代码是严重的安全违规行为。API Key、数据库密码以及公司的核心业务逻辑必须使用 `***` 等方式进行脱敏（Masking）处理，并且只截取需要审查的最小函数或类单元进行提问。

- **Q: AI 提供的代码是完美无缺的吗？可以不经过测试直接使用吗？**
  - A: AI 也会产生幻觉 (Hallucination)。有时它会建议使用不存在的库，或者出现语法错误。对于 AI 建议的代码，您必须在本地环境中运行测试代码以验证其是否能正常工作。

## 🧬 提示词解剖 (Why it works?)

1. **设定严格的角色定位 (Perspective)：** 强制设定“谷歌首席工程师”这一最高标准，促使 AI 提供基于“整洁代码”视角的反馈，而不是仅仅能运行的代码。
2. **多维度评估标准 (Specific Goals)：** 如果只是简单地要求“审查代码”，AI 通常只会挑出表面上的拼写错误。明确给出 Bug、安全性、可读性、性能这 4 个评估标准，旨在引导 AI 进行全方位的立体诊断。
3. **引导学习的设计 (Explanation)：** 仅仅获得正确答案无法提升编码实力。要求 AI 在注释中说明修改原因，能够起到辅导 (Tutoring) 的作用，促使用户自己思考架构。

## 📊 效果对比：Before & After

### ❌ Before (输入：初级的数据库查询逻辑)

```python
def get_user_data(user_id):
    # 直接将用户输入的 id 拼接到查询语句中（危险！）
    sql = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(sql)
    return cursor.fetchone()
```

### ✅ After (结果：增强了安全性和可读性的逻辑)

```python
def get_user_data(user_id):
    """
    查询特定用户信息。
    """
    # [Security] 为了防止 SQL 注入，使用参数化查询（绑定变量）
    # [Readability] 将查询语句分离，提高代码可读性
    query = "SELECT * FROM users WHERE id = %s"

    try:
        cursor.execute(query, (user_id,))
        result = cursor.fetchone()

        # [Bug] 添加查询结果为空时的异常处理
        if not result:
            return None

        return result

    except Exception as e:
        # [Bug] 处理运行时错误并记录日志（在实际环境中建议使用 logger）
        print(f"Database error occurred: {e}")
        return None
```

**审查总结：** 将用户输入直接拼接到查询语句中会使其暴露在致命的 SQL 注入攻击之下。通过使用绑定变量 (`%s`) 修复了安全漏洞，并添加了 `try-except` 异常处理，以防止在数据库查询失败时导致服务崩溃。
