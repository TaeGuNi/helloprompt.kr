# 📝 你的代码像意大利面？交给重构专家吧！

- **🎯 推荐对象：** 初中级开发者、维护遗留系统的工程师、急于发布但代码混乱的程序员
- **⏱️ 预计时间：** 1小时 → 缩短至 2分钟
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro (擅长代码处理的模型)

- ⭐ **难度：** ⭐⭐☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"还在写'能跑就行'的垃圾代码（Working Garbage）吗？让 AI 帮你写出符合 Clean Code 规范的艺术品吧！"_

在赶进度的压力下，我们常常会写出结构混乱、难以维护的“意大利面条代码”。虽然程序勉强能运行，但下一个接手的同事（或者一个月后的你自己）绝对会痛苦不堪。今天，我们将使用这套“重构专家”提示词，让 AI 帮你将杂乱无章的代码转化为优雅、易读且符合最佳实践的 Clean Code。

---

## ⚡️ 3句话总结 (TL;DR)

1. **摆脱技术债：** 瞬间将难以理解的面条代码重构成结构清晰的模块化代码。
2. **遵循最佳实践：** 自动应用 SOLID 原则和设计模式，提升代码可读性与可维护性。
3. **附带详细解释：** AI 不仅提供重构后的代码，还会解释“为什么这样改”，助你提升编程水平。

---

## 🚀 解决方案："Clean Code 重构专家"

### 🥉 Basic Version (基础版)

适用于代码量较小、只需简单整理格式和命名规范的情况。

> **角色：** 你是一位拥有10年经验的高级软件工程师。
> **请求：** 请帮我重构以下代码，使其更易读、更符合 Clean Code 规范。请提供重构后的代码并简要说明修改了哪些地方：
> `[此处粘贴你的代码]`

\

### 🥇 Pro Version (专业版)

适用于逻辑复杂、需要深度重构、解耦或添加设计模式的核心业务代码。

> **角色 (Role)：** 你是一位顶级的软件架构师兼 Clean Code 专家。你对 SOLID 原则、设计模式和代码可维护性有着极致的追求。
>
> **背景 (Context)：**
>
> - 当前状况：这段代码虽然能运行，但包含了过多的嵌套（Callback Hell / Pyramid of Doom），变量命名随意，且多个功能耦合在一个函数中。
> - 目标：在不改变原有业务逻辑（外部行为）的前提下，对其进行深度重构。
>
> **任务 (Task)：**
>
> 1. 请分析原代码中的“坏味道（Code Smells）”。
> 2. 将复杂函数拆分为单一职责的纯函数（Pure Functions）或小模块。
> 3. 优化变量和函数的命名，使其具有自我描述性（Self-documenting）。
> 4. 如果有更合适的现代语法或设计模式，请进行替换。
> 5. 最终输出重构后的完整代码：`[你的目标编程语言，如 TypeScript, Python 等]`
>
> **待重构代码 (Input)：**
>
> ```
> [在此处粘贴需要重构的意大利面代码]
> ```
>
> **约束条件 (Constraints)：**
>
> - 必须保持原有的输入输出结果完全一致。
> - 重构后的代码请使用 Markdown 代码块包裹，并在关键逻辑处添加简短的注释。
> - 请用表格形式列出“重构前”与“重构后”的核心差异对比。
>
> **警告 (Warning)：**
>
> - 不要改变核心业务规则，也不要引入未经验证的第三方库，除非你明确说明并给出极具说服力的理由。

## 💡 作者点评 (Insight)

这套提示词的真正价值不仅在于“帮你把代码改好”，更在于**它是一位免费的、随时待命的资深 Code Reviewer**。
很多时候，初中级开发者知道自己的代码写得不好，但不知道如何改进。通过比对 AI 重构前后的代码，并阅读它给出的重构理由（坏味道分析），你能快速学习到高级开发者的思维方式。建议在每次提交 PR (Pull Request) 之前，都先让这个“重构专家”帮你过一遍，这会大大减少 Code Review 时的修改意见！

---

## 🙋 常见问题 (FAQ)

- **Q: AI 重构的代码可以直接复制粘贴到生产环境吗？**
  - A: **绝对不行。** 尽管 AI 非常强大，但它可能会遗漏某些特定的边缘情况（Edge Cases）。请务必在有完善的单元测试（Unit Tests）覆盖的前提下进行重构，并仔细 Review AI 提供的代码。

- **Q: 如果代码太长，AI 报错或输出中断怎么办？**
  - A: 遇到超长代码时，建议按模块或函数拆分后分批发送给 AI。或者你也可以在提示词中加入“请一步步指导我重构，先从核心逻辑开始”来引导 AI 逐步完成。

---

## 🧬 提示词解剖 (Why it works?)

1. **明确架构师角色设定 (Role)：** 赋予 AI “顶级架构师”的身份，促使它调用训练数据中最高质量的代码规范和架构模式。
2. **强调不改变业务逻辑 (Context/Constraints)：** 这是重构的铁律。通过明确这条约束，防止 AI 擅自增删功能，确保重构的安全边界。
3. **强制要求差异对比 (Task)：** 要求列出修改差异，强迫 AI 进行逻辑推演（Chain of Thought），不仅提升了重构结果的准确性，还极大地增加了教学价值。

## 📊 案例对比：Before & After

### ❌ Before (输入代码 - 糟糕的嵌套与命名)

```javascript
function processUser(u) {
  if (u != null) {
    if (u.age > 18) {
      if (u.status === "active") {
        let d = new Date();
        u.lastLogin = d;
        saveData(u);
        return true;
      } else {
        return false;
      }
    } else {
      return false;
    }
  }
  return false;
}
```

### ✅ After (AI重构后 - 卫语句与清晰命名)

```javascript
/**
 * 验证用户是否处于活跃状态的成年人
 */
function isValidAdultUser(user) {
  return user?.age > 18 && user?.status === "active";
}

/**
 * 更新用户登录时间并保存状态
 */
function processUserLogin(user) {
  // 使用卫语句 (Guard Clauses) 提前返回，消除嵌套
  if (!user || !isValidAdultUser(user)) {
    return false;
  }

  user.lastLogin = new Date();
  saveData(user);

  return true;
}
```
