## 🚀 Решение: Промпт для генерации скриптов Git Hook

### 🥉 Базовая версия (Basic)

Идеально подходит, когда нужен простой скрипт для выполнения конкретной проверки.

> **Роль:** Ты — `[DevOps-инженер]`.
> **Задача:** Напиши shell-скрипт Git Hook и инструкцию по настройке с использованием `[Husky]`, чтобы `[запускать npm run lint перед выполнением git commit]`.

### 🥇 Продвинутая версия (Pro)

Используйте, когда требуется пуленепробиваемый скрипт с безупречной обработкой исключений для всей команды.

> **Роль (Role):** Ты — Senior `[DevOps-инженер]` с 10-летним опытом и `[эксперт по автоматизации Git]`.
>
> **Контекст (Context):**
>
> - Ситуация: Члены команды регулярно забывают проверять код перед коммитом, что приводит к частым падениям сборок и пустой трате минут в CI-пайплайне.
> - Окружение: `[Frontend-проект на Node.js, используется Husky]`
> - Цель: Создать строгие хуки `[pre-commit и commit-msg]`, которые рубят на корню любые человеческие ошибки.
>
> **Задача (Task):**
>
> 1. Хук `[pre-commit]`: Напиши shell-скрипт, который выполняет `[npm run lint]` и `[npm run format]` исключительно для проиндексированных файлов (staged files: `git diff --staged`).
> 2. Хук `[commit-msg]`: Добавь скрипт с регулярным выражением, строго контролирующий формат `[Conventional Commits (feat, fix, docs и т.д.)]`.
> 3. **Сообщения для пользователя:** Если проверки не пройдены, выведи в терминал понятное, но строгое сообщение об ошибке (с примером правильного варианта) и заблокируй коммит (exit 1).
> 4. **Инструкция по внедрению:** Подготовь фрагменты для `[package.json]` и команды для инициализации в виде блоков Markdown-кода, чтобы любой разработчик мог быстро всё настроить сразу после клонирования репозитория.
>
> **Ограничения (Constraints):**
>
> - Сгенерированный shell-скрипт должен быть полностью кроссплатформенным: без проблем работать на macOS, Linux и Windows (через Git Bash).
>
> **Предупреждения (Warning):**
>
> - Проект масштабный, поэтому проверка всех файлов займет вечность. Обязательно оптимизируй скрипт так, чтобы линтер проверял **только измененные файлы (staged files)**. (Например, используй `lint-staged`).

Истинная ценность этого промпта кроется не только в **предотвращении багов**, но и в абсолютном **психологическом спокойствии**, которое он дарит. В своих личных проектах я всегда использую слегка адаптированную версию этого промпта для создания хука `pre-push`, который наглухо блокирует прямые пуши в главную ветку (`git push origin main`).

То чувство облегчения, когда посреди ночи слипающимися глазами ты случайно пытаешься сделать форс-пуш в production, а заботливый скрипт бьет по рукам сообщением: **«🚨 Стоп! Изменения в ветку main — только через PR»**, поймут лишь те, кто хоть раз ронял прод. И да, обязательно просите ИИ добавить интеграцию с `lint-staged` — это ускоряет процесс проверок в десятки раз!

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Будет ли этот bash-скрипт работать у коллег на Windows?**
  - О: Абсолютно! Поскольку в промпте жестко задано требование «кроссплатформенности», ИИ это учтет и выдаст код, совместимый с Git Bash. Но если вы работаете в экосистеме Node.js, самым изящным и надежным решением будет попросить ИИ использовать готовую связку `Husky` + `lint-staged` вместо написания сырых shell-скриптов с нуля.

- **В: Хук получился невероятно медленным, теперь каждый коммит висит по несколько минут. Что делать?**
  - О: Скорее всего, ваш скрипт натравливает `npm run lint` или тесты на всю кодовую базу целиком. Обязательно используйте условие из блока «Предупреждения»: **«проверять только проиндексированные файлы»**. ИИ моментально перепишет код под использование `git diff --cached --name-only`, и коммиты снова станут летать.

- **В: Хук не дает мне откатить огромный ошибочный коммит, линтер ругается на старый код.**
  - О: Если нужно экстренно закоммитить код в обход всех проверок, просто добавьте флаг `--no-verify` (или `-n`). Например: `git commit -m "fix: горячее исправление" --no-verify`. Но помните: это инструмент спасения, а не ежедневная практика!

## 🧬 Анатомия промпта (Why it works?)

1. **Предельная детализация (Context):** Вместо ленивого «напиши мне хук», мы четко обозначаем стек (`Husky`, `Node.js`) и реальные боли команды (упавшие сборки, сожженные минуты CI). Это заставляет ИИ предлагать архитектурно верные решения вроде `lint-staged`.
2. **Защита от "тормозов" (Warning):** Главная ошибка новичков — линтинг всего проекта при каждом коммите. Блокируя этот подход прямо в промпте, мы на выходе получаем молниеносный скрипт, не раздражающий разработчиков.
3. **Учет зоопарка ОС (Constraints):** Требование кроссплатформенности гарантирует, что хук не сломается у разработчика с Windows, пока вся остальная команда сидит на macOS.
