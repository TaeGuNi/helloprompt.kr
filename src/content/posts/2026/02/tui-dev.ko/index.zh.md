## 💡 作者见解 (Insight)

这个提示词在处理 TUI 应用开发中最令人头疼的环节——**“布局设计与屏幕分割”** 时，能够展现出惊人的威力。在过去，为了在终端中分割屏幕并调整内边距 (Padding) 和外边距 (Margin)，我们往往需要反复修改代码并几十次地运行测试。但是，只要在提示词的 `[需求]` 中清晰地描述你想要的排版，AI 就能自动为你计算出如同 Flexbox 般完美的比例。

**🔥 实战技巧：** 当你使用 Go 的 Bubble Tea 或 Python 的 Textual 时，千万不要一开始就让 AI 把所有的业务逻辑和 UI 混合在一起写。你可以先让 AI 帮你搭建一个“UI 空壳”（填入占位假数据的布局），然后再逐步将各个视图区与真实数据连接起来。这种循序渐进的开发方式，能为你节省大量的调试时间，大幅降低开发压力。

## 🙋 常见问题 (FAQ)

- **Q: 我该如何选择编程语言和 TUI 框架？**
  - A: 如果你偏好快速编写脚本并且习惯了类似 Web CSS 的样式编写方式，推荐使用 Python 的 **Textual**；如果你需要稳定且易于分发的单文件二进制程序，Go 的 **Bubble Tea** 是绝佳选择；如果你追求极致的性能与类型安全，那么强烈推荐 Rust 的 **Ratatui**。

- **Q: 运行 AI 生成的代码后，屏幕上的线条或图标显示乱码怎么办？**
  - A: 这通常是因为你的终端模拟器不支持 Nerd Font，或者存在 Unicode 渲染问题。遇到这种情况，只需在提示词的约束条件中添加一句：_“考虑到终端环境兼容性，请仅使用安全的基础 ASCII 字符来构建 UI，不要使用特殊符号”_，问题就能迎刃而解。

## 🧬 提示词解剖 (Why it works?)

1. **强制指定框架：** 并没有使用“帮我做一个 TUI”这样模糊的指令，而是通过引入 `[技术栈]` 变量，迫使 AI 严格遵守特定框架的最新设计模式（例如：Elm 架构或基于状态的响应式 UI）。
2. **强调 UX 与状态管理：** 将 TUI 生态中最核心的“键盘导航”与“组件间状态 (State) 管理”作为具体任务下达，从而确保生成的代码不仅是表面好看，更是真正达到生产级别的稳定代码。
