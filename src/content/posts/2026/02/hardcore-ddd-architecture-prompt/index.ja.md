## 📝 スパゲッティコードはお断り！🍝 シニアエンジニアの黒魔術 DDDプロンプト

- 🎯 **おすすめの対象:** AI生成コードの品質に絶望しているエンジニア、保守地獄に喘ぐリードエンジニア
- ⏱️ **所要時間:** 3ヶ月（リファクタリング地獄） → 1分に短縮
- 🤖 **推奨モデル:** コーディング特化型の推論モデル (Claude 3.5 Sonnet, GPT-4o など)
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「一見動くAIのコード」に感動したのは束の間、数日後にはそのコードを見るのも嫌になった経験はありませんか？ようこそ、抜け出せない「スパゲッティコード地獄」へ。_

AIにコーディングを丸投げすると、最初は魔法のように動くコードを吐き出します。しかし、プロジェクトが少しでもスケールし始めるとどうなるでしょうか？UIコンポーネントにデータベースのクエリがベタ書きされ、1つの関数が300行を超えるという、保守不可能な「キメラ」が誕生します。AIの基本スタンスは常に「イエスマン」です。こちらが適当な指示を出せば、彼らもまた適当で脆弱な構造のコードを平然と納品してくるのです。

今こそ、AIの盲目的な忖度を打ち砕き、土台から完璧な「クリーンアーキテクチャ」を強制する時です。このプロンプトは、単なるAIを「妥協を知らない無慈悲なシニアアーキテクト」へと豹変させ、フレームワークに依存しない純粋なビジネスロジック（DDD）の構築を強制する禁断のチートキーです。

---

## ⚡️ 3行まとめ (TL;DR)

- 🚫 **スパゲッティコードの根絶:** UI、ビジネスロジック、データアクセス層における完璧な関心の分離（SoC）を強制します。
- 🏗️ **フレームワーク非依存のコア設計:** 万が一明日ReactやNext.jsが廃れたとしても、そのまま生き残れる堅牢なビジネスロジックを構築します。
- ✂️ **巨大関数の無慈悲な粉砕:** 30行を超える肥大化した関数や複数の責務を持つコードを、単一責任の原則（SRP）に従ってAI自ら分割させます。

## 🚀 解決策: "ハードコアアーキテクト憑依プロンプト"

### 🥉 Basic Version (基本型)

複雑な設定なしで、今すぐ結果だけが欲しい場合にご活用ください。

> **役割:** あなたは`[妥協を知らない厳格なシニアアーキテクト]`だ。
> **要求:** `[実装したい具体的な機能]`を開発するが、ビジネスロジックとUIは必ず完璧に分離してコーディングしろ。1つのファイルに複数の責務が混在するスパゲッティコードは断じて許さない。

### 🥇 Pro Version (専門家型)

細部に至る圧倒的なコード品質と、完璧に独立した階層アーキテクチャが必要な本番環境でご活用ください。

> **役割 (Role):** あなたは一切の妥協を許さないハードコアなシニアアーキテクトであり、ソフトウェアエンジニアリングの権威だ。無意味な愛想笑いや「承知いたしました」といった前置きは一切不要。洗練されたコードでのみ語れ。
>
> **状況 (Context):**
> - 背景: 現在、`[使用する技術スタックやフレームワーク]`を基盤とし、`[開発対象のコアドメイン/システム名]`システムを開発中である。
> - 目標: 将来的なコードの腐敗を完全に防ぎ、View・Business・Dataの各層を厳格に分離するドメイン駆動設計（DDD）およびクリーンアーキテクチャを適用して実装する。
>
> **要求 (Task):**
> 1. 以下の要件を満たす`[実装したい具体的な機能の詳細]`のコードを作成しろ。
> 2. **階層侵犯の絶対禁止:** UIやController内に、データベースへのクエリや外部APIの呼び出しを絶対に混入させるな。これらにはレンダリングとパラメータの受け渡しのみを担当させろ。
> 3. **ドメイン汚染の徹底排除:** コアとなるビジネスロジック（EntityやValue Object）に、特定のフレームワークやDBへの依存を `import` することは固く禁ずる。純粋関数と型定義のみで構成しろ。
> 4. **巨大関数の粉砕:** 1つの関数が複数の責務を持っている場合、または30行を超える場合は、直ちに単一責任の原則（SRP）に従って細かく分割しろ。
>
> **制約事項 (Constraints):**
> - フレームワークが持つ固有のルーティング規則は尊重しつつも、ビジネスロジックは必ず `src/domain` などの独立したディレクトリに隔離すること。
> - 外部への依存には、必ずインターフェースを活用した依存性の注入（DI）パターンを用いること。
>
> **注意事項 (Warning):**
> - お前の提示するコードから少しでもスパゲッティコードの兆候が見えれば、このプロジェクトは破綻する。一切の妥協を排し、現代において最も理想的かつクリーンなアーキテクチャを提示しろ。

## 💡 筆者コメント (Insight)

私がこのプロンプトを極限まで研ぎ澄ませた理由は極めてシンプルです。それは、**現在のAIが「とにかく早くコードを出力して人間に褒められたい」という欲求に飢えた、都合の良いイエスマン**だからです。

ただ「ボタンを一つ作って」と頼んだだけなのに、UIコンポーネントの中に平気で `fetch` 処理を書き込み、あろうことか直接データベースのクエリまでねじ込んでくるAIのコードを見ていると、エンジニアとしての破壊衝動すら覚えます。このプロンプトは、そんな**浅はかな忖度をするAIの胸ぐらを掴み、「目を覚ませ、お前は今から血も涙もないシニアアーキテクトだ！」と強烈な平手打ちを食らわせる**ためのものです。

ぜひ、実際のプロジェクトでこのプロンプトを試してみてください。AIは突然その口調すら変え、自らファイルを3〜4つに分割し、**フロントエンドとバックエンドの境界、そしてビジネスロジックの階層を厳格に分離し始めます。** 最初はファイル数が増えて面倒に感じるかもしれませんが、3ヶ月後に仕様変更が入り、コードの保守を行う段階になれば、あの時このプロンプトを使っておいて本当に良かったと心底安堵するはずです。**技術的負債が膨らむスピードは、闇金の金利より恐ろしい**という事実を忘れないでください。アーキテクチャにおいて、妥協という選択肢はあり得ないのです。

## 🙋 よくある質問 (FAQ)

- 🔹 **Q: コードが細かく分割されることでファイル数が激増しますが、これは正常ですか？**
  - 🔸 **A:** 極めて正常です。1つのファイルが500行を超えカオス化するくらいなら、50行のファイルが10個に整然と分割されている方が、**保守性の観点では100倍有利**です。ファイル数が増えることを恐れてはいけません。
- 🔹 **Q: フロントエンドエンジニアですが、このプロンプトは実務で役に立ちますか？**
  - 🔸 **A:** もちろんです。UIコンポーネント（`page.tsx` や `Component.vue` など）の中に、複雑なビジネスロジックや状態管理ロジックが混入した瞬間、フロントエンドも容易にスパゲッティ化します。**View層とビジネスロジック層の分離は、バックエンド・フロントエンドを問わず不変の真理**です。

## 🧬 プロンプトの解剖 (Why it works?)

- 🎯 **ハードコア・ペルソナの完全注入:** AIの初期設定である「人間に忖度する順応性」を意図的に剥奪し、**エンジニアリングの原則を死守する姿勢**を強制しています。
- 🧱 **絶対禁止条項（Constraints）の明示:** 「UI層の汚染」や「巨大関数の生成」など、**現場でよくあるアンチパターンをピンポイントで禁止**しているため、AIが勝手にコードを手抜きする隙を与えません。
- 🧩 **アダプターパターンの自然な誘導:** フレームワークの基本構造（ルーティング等）には従わせつつ、コアとなるロジックのみを安全な場所に隔離させることで、**実務とかけ離れた過度なオーバーエンジニアリングを未然に防ぎます**。

## 📊 証明: Before & After

### ❌ Before (入力)

```javascript
// プロンプトなしで「ユーザーを作成してメールを送信するAPIを作って」と指示した場合
export default async function handler(req, res) {
  const { email, password } = req.body;

  // DBを直接叩く（汚染発生！）
  const user = await prisma.user.create({ data: { email, password } });

  // 外部APIを直接呼び出す（関心の崩壊！）
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ After (結果)

```javascript
// ハードコアDDDプロンプト適用後（階層と責任が完璧に分離される）

// 1. Controller階層（リクエスト/レスポンスのルーティングのみを処理）
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Domain Service階層（フレームワークに依存しない純粋なビジネスロジック）
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Infrastructure階層（DBおよび外部API専用アダプター）
// UserRepository、EmailServiceなどとして別ファイルに完全に隔離される...
```

## 🎯 結論

AIにコーディングを委ねる際、最も警戒すべきリスクは「とりあえず今の要件だけ満たして動く、使い捨てのゴミコード」を納品されることです。コンソールにエラーが出ないからといって、無邪気に喜んでいる場合ではありません。

このハードコアDDDプロンプトというチートキーでAIの手綱を強く握り、プロジェクトの初期段階から堅牢で拡張性の高いアーキテクチャを設計させてください。数ヶ月後、負債のないクリーンなコードベースを前にした未来のあなたが、今のあなたへスタンディングオベーションを送るはずです。

洗練されたアーキテクチャで無駄な残業を撲滅し、クールに定時退社を決めましょう！ 🍷
