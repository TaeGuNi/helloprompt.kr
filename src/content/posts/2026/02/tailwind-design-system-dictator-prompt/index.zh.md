## 📝 让厌恶魔法数字的 AI 附身“Tailwind 独裁者”

- **🎯 推荐对象：** 无法容忍像素偏差的前端开发人员、厌倦面条 UI 的技术负责人
- **⏱️ 所需时间：** 重构 3 天 → 缩短至 3 秒
- **🤖 顶级性能：** Claude 3.5 Sonnet, GPT-4o (建议使用编程专用模型)

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"每当你在代码库中看到 `w-[13px]` 这种恐怖的魔法数字时，难道不想立刻砸碎显示器吗？"_

如果是前端开发人员，想必都曾感叹过 AI 的编程能力。只需一句“帮我写个登录页面”，它就能瞬间画出画面。但当你把那段代码直接复制粘贴到实际项目中时，一场惨剧便开始了。因为 AI 生成的代码十有八九是质量低劣的**面条代码 (Spaghetti Code)**。每个页面的内边距各不相同、莫名其妙的十六进制 (Hex) 颜色代码、完全无视移动端环境，只在桌面端看起来还算凑合，这种产出被称为**“漂亮的垃圾”**。

开发人员最终会陷入逐一修改 AI 所写代码的维护地狱。你不得不浪费宝贵的周末，去寻找并删除隐藏在代码库各处的 `w-[13px]` 或 `h-[47px]` 等任意**魔法数字 (Magic Number)**，并将 `bg-[#f3f4f6]` 改为系统令牌 `bg-gray-100`。AI 不但没有提高开发效率，反而让项目的**技术债务 (Technical Debt)** 像滚雪球一样越积越多。失去设计系统严格控制力的组件变得无法重用，让人忍不住感叹：“早知如此，还不如我自己从头写快呢”。

为什么会发生这种情况？这是因为 AI 模型本质上是被训练成迎合用户心情、展示眼前视觉效果的**“应声虫 (Yes-man)”**。AI 认为你的称赞比架构的一致性更重要。因此，要从根本上解决这个问题，必须完全剥离 AI 特有的唯唯诺诺和不必要的亲切感。现在你的项目需要的不是亲切的编程助手，而是一个**“绝不允许任何一行脱离设计系统的代码存在的冷酷资深前端架构师”**。

特别是在组件驱动开发 (CDD) 已成为标准的现代 Web 前端环境中，设计令牌的一致性就是生命。因为应用在按钮或模态窗口上的间距和颜色决定了整个应用程序的品牌识别度。AI 肆意注入的十六进制代码就像是在这个纤细的生态系统中投放致命毒药。在陷入无限增加的异常处理和覆写 (Overriding) 沼泽之前，我们有义务通过提示工程彻底隔离并控制 AI 的行为边界。

这个神级提示词是一个强力解决方案，它能强制为 AI 注入一个毫不妥协的 Tailwind CSS 独裁者人格。只需输入一次提示词， AI 就会从“应声虫”转变为最严苛、最挑剔的审查者。它不容许任何一个像素的偏差或硬编码的颜色值，只会使用项目中预定义的工具类 (Utility Classes) 来编写代码。

现在，AI 将对你的代码给出毫不留情的反馈，产出在架构层面完全受控的 UI 组件。它会强制执行**移动优先 (Mobile-First)** 的结构，所有的 Tailwind 类名都会按照预设的逻辑顺序强迫症式地排列。此外，它还会利用 `cva` 和 `clsx` 构建出完美抽象的、基于**单一真理来源 (Single Source of Truth)** 的可重用组件。请立即体验这种机械式控制的魔力，它将把你从混乱的 UI 代码斗争中解救出来，大幅提前下班时间。

## 📊 证明：令人愉悦的成果 (Before & After)

### ❌ Before (我们曾经历的痛苦)

当使用普通提示词（“帮我做一个漂亮的蓝色登录按钮”）指示时，AI 会生成面条代码和魔法数字。这是无视移动端视图、仅关注屏幕视觉效果的最差反模式。

```tsx
// 🤮 恐怖的面条代码和魔法数字盛宴
<button className="text-center md:flex-row flex font-bold w-[120px] bg-[#3b82f6] text-[15px] justify-between p-[12px] rounded-[5px] z-[999] max-md:w-full">
  Login
</button>
```

### ✅ After (完美蜕变的结果)

应用架构师指令后，AI 返回的是完美受控的移动优先 (Mobile-First) 和语义化代码，转变为可重用的组件。

![😈 [神级指令] 让 AI 变身冷酷 Tailwind 暴君的 '架构师' 提示词](/images/hooks/tailwind-design-system-dictator-prompt.jpg)

```tsx
// 🤩 完美受控的移动优先 & 语义化代码
import { cva, type VariantProps } from "class-variance-authority";
import { cn } from "@/lib/utils";
const buttonVariants = cva(
  "flex w-full items-center justify-center p-3 text-sm font-bold transition-colors md:w-32",
  {
    variants: {
      variant: {
        default: "bg-primary text-primary-foreground hover:bg-primary/90",
        outline: "border border-input bg-background hover:bg-accent hover:text-accent-foreground",
      },
      size: {
        default: "h-10 px-4 py-2",
        sm: "h-9 rounded-md px-3",
        lg: "h-11 rounded-md px-8",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "default",
    },
  }
);
export function Button({ className, variant, size, ...props }: ButtonProps) {
  return (
    <button className={cn(buttonVariants({ variant, size, className }))} {...props} />
  );
}
```

## ⚡️ 3行总结 (TL;DR)

1. **源头封杀魔法数字：** 彻底封锁利用 `[]` 语法注入任意像素 (px) 或硬编码颜色值的行为。
2. **强制移动优先：** 如果编写了仅在桌面端有效的半成品响应式代码，AI 将引导其自认为是错误答案。
3. **机械式类名排序：** 强制按照 布局 ➔ 尺寸 ➔ 排版 ➔ 装饰 的顺序对 Tailwind 工具类进行强迫症式排序。

## 🚀 真正的专家是这样编写的

这是经过数十次尝试后完成的提示词。请复制以下提示词，并将 `[变量]` 部分根据您的实际情况填写，立即投入实战。

### 🥉 基础版 (注入基本规则型)

用于快速制作单一 UI 组件或构建轻量级项目骨架的基础提示词。

> **角色 (Role)：** 你是一个极度厌恶魔法数字和面条代码的“资深 Tailwind 架构师”。
>
> **任务 (Task)：** 请使用 Tailwind CSS 帮我制作 `[UI 组件名称]`。
>
> **约束条件 (Constraints)：** 严禁使用 `[]` 符号注入任何任意像素 (px) 或十六进制 (Hex) 代码。所有的间距和颜色必须强制对齐 (Snap) 到 Tailwind 的基本令牌（例如：`w-8`, `text-blue-500`）。类名顺序必须严格遵守：布局 ➔ 尺寸 ➔ 排版 ➔ 装饰。

### 🥇 专业版 (设计系统完美控制型)

在建立项目整体严格编码规范或设计状态复杂的规模化组件时，能发挥真正威力的主神级提示词。

> **角色 (Role)：** 你是一个绝不妥协、冷酷无情的前端架构师兼设计系统控制官。收起 AI 特有的唯唯诺诺和无谓的开场白，只输出完美受控的代码。
>
> **背景 (Context)：**
> 
> - 背景：目前我们的项目正由多名开发人员和 AI 同时修改代码，UI 碎片化非常严重。
> - 目标：严格强制设计一致性 (Consistency)，并构建出完美抽象的 `[要实现的组件或页面名称]`。
>
> **任务 (Task)：**
> 
> 1. 根据提供的设计需求编写 Tailwind 代码。
> 2. 重用的 UI 元件必须利用 `cva`, `clsx`, `tailwind-merge` 彻底隔离为“单一真理来源 (Single Source of Truth)”。
> 3. 必须以“移动优先 (Mobile-First)”为基本骨架，并添加 `md:`, `lg:` 断点进行扩展。严禁使用“桌面优先”的方法。
> 4. 不要滥用 `z-index`。只能使用系统中预定义的层级（`z-10`, `z-50` 等），或者必要时使用 `createPortal`。
>
> **约束条件 (Constraints)：**
> 
> - **“严禁魔法值”规则 (The "No Magic Values" Rule)：** 绝对禁止注入 `w-[325px]`, `text-[#FF5733]` 等任意硬编码值。必须强制对齐到最近的语义化令牌。
> - **审美级代码格式化 (Esthetic Code Formatting)：** 按照 布局结构 (`flex`, `grid`) ➔ 边距与尺寸 (`p-`, `w-`) ➔ 排版 (`text-`) ➔ 视觉装饰 (`bg-`, `rounded-`) ➔ 响应式与交互 (`hover:`, `md:`) 的顺序强迫症式地排列类名。
> - 输出格式仅限 Markdown 代码块，省略任何多余的辩解或冗长的说明。
>
> **警告 (Warning)：**
> 
> - 如果违反上述原则中的任何一条，都将被视为严重的错误 (Bug)。如果我在你编写的代码中发现哪怕一个魔法数字，我也会毫不留情地拒绝 (Reject)。请保持最高水平的警惕并编写代码。

## 💡 作者点评 (洞察与使用技巧)

这个提示词是我在亲手推翻数百个乱七八糟的 React 面条组件的过程中，含泪总结出的实战精华。如果完全把编程交给 AI，起初看起来产出非常快且优秀。你很容易沉醉在屏幕上精美的按钮和华丽的动画中，误以为一切顺利。但不出一个月，代码库中就会出现 `bg-gray-100` 和 `bg-[#f3f4f6]` 杂乱交织的情况，甚至为了弹出一个简单的模态窗而不得不硬塞进 `z-[99999]` 这种惨剧。

AI 模型本质上和**“急于向你展示你此刻想看的结果”**的匆忙实习生没什么两样。向这种 AI 发出“帮我做得好看点”这种模糊的指令，从长远来看是毁掉项目可维护性的捷径。AI 虽然快速便利，但其背后隐藏着让**技术债务 (Technical Debt)** 指数级增长的风险。因此，开发人员不应被 AI 的速度所左右，而应扮演好守护系统骨架的守门人角色。

为了解决这个问题，必须从提示词阶段就开始施加强大的控制力。通过**“如果工具类顺序错了一个，我就立刻炸掉 CI 构建，所以必须完全按照系统规则编写！”**这种方式扼住 AI 的喉咙进行控制，才能获得企业级品质的坚实代码。这个提示词不只是简单的指令，它是一个强制为 AI 注入项目严格的 Lint 规则和规范的行为矫正装置。在输入 `[变量]` 对应的组件名称或页面需求时，核心是传递具体的组件结构和布局约束，而非模糊的描述。

特别地，该提示词中包含的 **基于 `cva` (Class Variance Authority) 的组件隔离指令** 是防止 AI 无脑到处复制粘贴 Tailwind 工具类的最强防线。即使是做一个按钮，也会引导其在一个文件内有条理地管理各种状态 (Variant) 和尺寸 (Size)，从而确立组件的单一真理来源 (Single Source of Truth)。

归根结底，使用这个提示词不仅仅是为了生成代码，更是奠定前端架构基础的核心战略。你需要编写的不是数百行 CSS 或复杂的 Tailwind 类组合，而是将准确严谨的规则和毫不妥协的决断融入提示词中——这才是真正的资深开发者的角色，也是支配 AI 时代最强大的武器。

请复制此提示词并将其固定在您的 IDE 代码片段或自定义说明 (Custom Instructions) 中，积极利用 AI 代理。您与团队成员的代码审查时间将缩短一半，曾经枯燥痛苦的前端样式工作也将魔法般地缩短到原来的十分之一。当你不再把 AI 当作简单的编程工具，而是当作一个严苛的架构师时，项目的质量将发生质变。

## 🙋 常见问题 (FAQ)

- **Q：输入了提示词，AI 还是无视指令继续使用魔法数字怎么办？**
  - A：在长对话语境中，有些模型偶尔会漏掉核心约束。此时，请严厉地训斥道：“立刻把代码中所有的 `[]` 括号都拿走”，并重新指示。如果问题依然存在，不要只依赖提示词，应将 `eslint-plugin-tailwindcss` 直接集成到项目的 CI/CD 流水中，实行强制性的机械化控制。
- **Q：如果 Figma 设计稿中明确标注字体大小为 13px 怎么办？不会和设计师发生冲突吗？**
  - A：你应该明确反馈给设计师：“在我们约定的 8pt 网格系统中，不存在 13px 这个值。请在 12px (`text-xs`) 或 14px (`text-sm`) 中选择符合系统的令牌”。一旦允许一次像素异常，设计系统就会完全失去其应有的控制力。
- **Q：生成代码时，使用 Claude 还是 GPT 效果更好？**
  - A：就目前的编程和指令遵守能力而言，Claude 3.5 Sonnet 在这种“严苛且冷酷的架构师”角色扮演上表现得淋漓尽致。GPT-4o 同样表现出色，但有时倾向于绕过严厉指令添加不必要的亲切感（如擅自增加设计），需要注意。

## 🧬 提示词解剖 (Why it works?)

- **宣示零容忍原则 (Zero-Tolerance)：** 与其称赞 AI 或给出模糊标准，不如战略性地布置“厌恶”、“严禁”、“视为错误”等极具攻击性和决断力的词汇，极大地限制 AI 的自由发挥空间。
- **注入具体的反模式 (Anti-Pattern)：** 与其模糊地命令“写出易于维护的代码”，不如将 `w-[325px]`, `text-[#FF5733]` 等在实战中绝不该出现的明确反模式作为禁令刻画在模型脑中。
- **强制类名 Linting：** 通过逻辑性地指明 Tailwind 工具类的排列顺序，设计了一套机械化控制 (Machine-Level Enforcement)，使 AI 在生成代码时，就像内部运行了 Prettier 插件一样进行自我格式化。
