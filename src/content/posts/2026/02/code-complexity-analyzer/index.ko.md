## 📝 복잡한 코드는 이제 그만! AI 복잡도 분석기 (Code Complexity Analyzer)

- **🎯 추천 대상:** 주니어 개발자, 레거시 시스템 유지보수 담당자, 깐깐한 코드 리뷰어
- **⏱️ 소요 시간:** 1시간 → 3분 단축
- **🤖 추천 모델:** GPT-4o, Claude 3.5 Sonnet (코드 추론 능력이 뛰어난 모델 권장)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"if문 안에 for문, 그 안에 다시 if문... 어떻게든 돌아가긴 하지만, 건드리는 순간 터질 것 같은 시한폭탄 레거시 코드를 마주하고 절망해 본 적 있으신가요?"_

개발을 하다 보면 누구나 한 번쯤 숨이 턱 막히는 **'스파게티 코드'**와 마주하게 됩니다.
끊임없는 기능 추가와 기획 변경, 그리고 릴리즈 일정에 쫓겨 급하게 욱여넣은 땜질식 버그 수정(Hotfix)이 수없이 반복되면서, 코드는 점차 비대해지고 걷잡을 수 없이 얽혀만 갑니다.
**순환 복잡도(Cyclomatic Complexity)**는 기하급수적으로 치솟고, 하나의 함수가 수백 줄을 거뜬히 넘어가는 끔찍한 괴물이 탄생하는 것이죠.

이러한 레거시 코드는 단순히 코드의 물리적인 가독성을 떨어뜨리는 데 그치지 않습니다.
사소한 버튼 색상 하나, 텍스트 하나를 수정하려 해도 시스템의 전혀 엉뚱한 곳에서 **예상치 못한 사이드 이펙트(Side Effect)**가 터질지 몰라 개발자를 극도의 불안감에 떨게 만듭니다.
"이 조건문은 도대체 어떤 비즈니스 로직 때문에 존재하는 거지?", "이 전역 변수는 어디서 상태가 변형(Mutation)되는 걸까?"
코드를 읽고 의도를 파악하는 데에만 소중한 하루 일과를 모두 허비하게 되며, 정작 가치 있는 개발보다 과거의 잔해를 분석하는 데 더 많은 에너지를 쏟게 됩니다.
당장 내일이 정기 배포일인데 얽히고설킨 코드의 늪에서 허우적대며 밤을 지새운 경험, 어디서부터 손대야 할지 몰라 그저 모니터만 멍하니 바라보며 절망하던 경험, 실무 개발자라면 다들 한 번쯤 뼛속 깊이 공감하실 겁니다.

특히 **인지적 복잡도(Cognitive Complexity)**가 비정상적으로 높은 코드는 한 개인의 고통을 넘어 팀 전체의 생산성을 심각하게 저해하는 주범이 됩니다.
새로 합류한 팀원은 방대한 코드베이스의 흐름을 파악하지 못해 온보딩 기간이 한없이 길어지고, 10년 차 시니어 개발자조차 코드 리뷰에서 도대체 어디부터 지적해야 할지 몰라 막막해집니다.
결국 "어쨌든 프로덕션에서 작동하니까 일단 건드리지 말자"라는 암묵적이고 타협적인 룰이 팀 내에 자리 잡게 되고, 시스템은 서서히 감당할 수 없는 기술 부채(Technical Debt)에 짓눌려 서서히 붕괴의 길을 걷게 됩니다.
리팩토링의 첫걸음인 테스트 코드를 작성하려 해보아도, 모듈 간의 의존성이 너무 강하게 결합되어 있어(Tightly Coupled) Mocking조차 불가능한 끔찍한 진퇴양난에 직면하게 되죠.

우리는 종종 "이번 스프린트만 끝나면, 다음 주에는 꼭 리팩토링해야지"라는 헛된 자기위안과 함께 기술 부채를 방치합니다.
하지만 야속하게도 그 '다음 주'는 결코 오지 않으며, 코드는 복리의 마법처럼 더욱 끔찍하게 꼬여만 갈 뿐입니다.
**레거시 코드(Legacy Code)**를 뜯어고치는 작업은 마치 시속 100km로 달리는 자동차의 바퀴를 주행 중에 갈아 끼우는 것과 같습니다.
라이브 시스템은 절대 멈추면 안 되고, 비즈니스를 위한 새로운 기능은 끊임없이 추가되어야 하며, 당연히 새로운 버그를 만들어내서도 안 되니까요.
이 극한의 제약과 압박감 속에서 인간의 뇌는 한계를 맞이합니다. `if` 조건문이 단 3~4번만 깊게 중첩되는 순간, 개발자의 인지 능력은 급격히 저하되고 치명적인 논리적 실수를 범하게 됩니다.

하지만 이제, 어디서부터 손대야 할지 막막했던 그 엄청난 두려움과 막막함을 완벽하게 해소해 줄 궁극의 도구가 여러분 앞에 있습니다.
바로 **'AI 코드 복잡도 분석기(Code Complexity Analyzer)'** 프롬프트입니다.
더 이상 개발자의 막연한 직감이나 아슬아슬한 감에 의존한 위험천만한 리팩토링은 필요하지 않습니다.
이 프롬프트를 통해 AI는 마치 내 옆에 앉아 코드를 꼼꼼히 살펴주는 깐깐하고 실력 있는 시니어 페어 프로그래머처럼, 객관적인 소프트웨어 공학 지표를 바탕으로 코드의 치명적인 문제점을 정확하고 예리하게 진단해 줍니다.

더욱 놀라운 것은 AI가 단순히 코드를 짧게 줄여주는 1차원적인 포맷터(Formatter) 역할을 하는 것이 아니라는 점입니다.
AI는 **디자인 패턴(Design Pattern)**과 **클린 코드(Clean Code)** 원칙에 입각하여, 실무 프로덕션 환경에 즉시 적용 가능한 구체적이고 우아한 아키텍처 개선 전략을 단 1분 만에 명확히 제시합니다.
기존 비즈니스 로직의 무결성은 100% 안전하게 유지하면서, 끝없이 파고드는 `if-else` 지옥을 **조기 리턴(Early Return)**으로 깔끔하게 펴주고, 수백 줄의 거대한 함수를 **단일 책임 원칙(SRP)**에 맞게 여러 개의 작은 모듈형 함수로 완벽하게 분리해 줍니다.
이 프롬프트와 함께라면 두렵기만 했던 레거시 코드도 당신의 뛰어난 엔지니어링 역량을 증명할 훌륭한 캔버스가 될 것입니다. 동료들에게 찬사를 받는 깔끔한 Pull Request를 작성하고 진정한 개발의 즐거움을 되찾아 보세요.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 엉망으로 얽힌 수작업 코드, if문 안에 for문이 끝없이 중첩되어 어디서부터 손대야 할지 막막했던 레거시 코드의 전형입니다. 비즈니스 로직을 파악하는 데만 수십 분이 걸립니다.

```javascript
> function processOrder(order) {
>   if (order != null) {
>     if (order.status === "PENDING") {
>       if (order.items && order.items.length > 0) {
>         let total = 0;
>         for (let i = 0; i < order.items.length; i++) {
>           if (order.items[i].price > 0) {
>             total += order.items[i].price * order.items[i].quantity;
>           }
>         }
>         if (total > 100) {
>           order.discount = total * 0.1;
>         } else {
>           order.discount = 0;
>         }
>         return total - order.discount;
>       } else {
>         return 0;
>       }
>     } else {
>       throw new Error("Order is not pending");
>     }
>   } else {
>     throw new Error("Invalid order");
>   }
> }
```

### ✅ After (완벽하게 변신한 결과)

![복잡한 코드는 이제 그만! AI 복잡도 분석기 (Code Complexity Analyzer)](/images/hooks/code-complexity-analyzer.jpg)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: 예외 상황 조기 리턴 (중첩 감소)
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. 고차 함수 활용: 반복문 단순화
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. 비즈니스 로직 분리 및 단순화
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```

## ⚡️ 3줄 요약 (TL;DR)

1. 악명 높은 중첩 조건문과 비대해진 레거시 함수를 AI가 입체적이고 구조적으로 진단합니다.
2. 조기 리턴(Early Return), 함수 추출(Extract Method) 등 실무에 즉각 적용 가능한 검증된 리팩토링 기법을 제안받을 수 있습니다.
3. 기존 비즈니스 로직의 무결성은 안전하게 보존하면서, 가독성과 유지보수성이 극대화된 우아한 코드를 순식간에 얻어냅니다.

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 제가 실무에서 타인이 작성한—혹은 과거의 제가 무책임하게 짜놓은—레거시 코드를 마주했을 때 가장 먼저 꺼내 드는 궁극의 무기입니다.
개발자라면 누구나 남의 코드를 읽는 것을 고통스러워합니다. 특히 변수명이 불명확하고 조건문이 난무하는 스파게티 코드는 보기만 해도 한숨이 나오며, 어디서부터 손을 대야 할지 엄두가 나지 않죠.
하지만 AI는 짜증을 내지 않습니다. 감정의 동요 없이, 가장 객관적이고 냉철한 시각으로 코드의 구조적 결함을 해부하고 정밀 타격합니다.

이 프롬프트가 강력한 이유는 단순히 코드의 물리적인 길이를 줄여주는 1차원적인 작업에 머무르지 않는다는 점입니다.
"현재 이 조건문은 뎁스(Depth)가 4단계에 달하므로, **Guard Clauses(조기 리턴)** 패턴을 적용하여 상단으로 분리하는 것이 구조적으로 타당합니다"와 같이, 코드 리뷰에서 동료들을 완벽하게 설득할 수 있는 합리적이고 논리적인 명분을 쥐여줍니다.
우리가 흔히 범하는 실수 중 하나는 리팩토링을 '그냥 내 코딩 스타일에 맞게, 내 맘에 들게 고치는 것'으로 착각하는 것입니다.
하지만 진정한 의미의 리팩토링은 명확한 이유와 객관적인 소프트웨어 공학 원칙에 기반해야 하며, 이 프롬프트는 그 원칙을 AI의 입을 빌려 당신에게 강력하게 제공합니다.

프롬프트 내의 **`[변수]` 통제법**은 이 프롬프트의 진정한 가치를 폭발시키는 핵심 치트키입니다.
예를 들어, `[TypeScript/NestJS]`라는 기술 스택 변수는 AI에게 엄청난 컨텍스트(Context)를 부여합니다.
단순히 자바스크립트 수준의 문법적 리팩토링이 아니라, 프레임워크가 제공하는 의존성 주입(Dependency Injection)을 어떻게 더 우아하게 활용할 것인지, 커스텀 데코레이터(Decorator)를 생성하여 횡단 관심사(Cross-cutting Concerns)를 어떻게 분리할 것인지 등, **해당 프레임워크 생태계에 최적화된 최고급 디자인 패턴**을 끌어낼 수 있습니다.
만약 여러분이 `[Python/FastAPI]`를 사용 중이라고 변수를 수정한다면, Pydantic 모델을 활용한 데이터 검증 로직 분리나, 비동기(Async) 처리의 병목 현상 최적화 포인트를 짚어줄 것입니다.
언어와 프레임워크에 맞춰 프롬프트를 기출 변형하는 순간, 범용적인 코딩 AI는 여러분의 프로젝트에 완벽히 동화된 **특급 도메인 전문가**로 변모하게 됩니다.

특히 이 프롬프트는 깃허브(GitHub)에 Pull Request(PR)를 올리기 직전, 본인의 코드를 **자체 검열(Self-Review)**할 때 진정한 위력을 발휘합니다.
코드를 작성한 직후에는 "내가 작성한 코드니까 구조적으로 완벽해"라는 확증 편향(Confirmation Bias)에 빠지기 쉽습니다.
하지만 이 프롬프트를 통해 단 한 번의 필터링만 거쳐도 결과는 놀라울 정도로 달라집니다. 미처 생각하지 못했던 엣지 케이스(Edge Case)에 대한 예외 처리나, 결합도(Coupling)를 낮출 수 있는 구조적 개선점을 날카롭게 짚어주기 때문입니다.
이 과정을 거친 코드는 동료 개발자들로부터 "이번 코드 아키텍처, 정말 예술이네요!", "가독성이 너무 좋아서 리뷰할 게 없습니다"라는 짜릿한 찬사를 이끌어낼 수 있습니다.

또한, **제약사항(Constraints)의 강력한 통제** 부분에 각별히 주목해야 합니다.
`원본 코드의 핵심 비즈니스 로직은 어떠한 경우에도 절대 변경되어서는 안 돼.`라는 문장은 AI의 치명적인 환각(Hallucination)을 막아주는 가장 든든한 절대 방어막입니다.
LLM은 종종 코드를 '예쁘고 짧게' 만드는 데 심취한 나머지, 예외 상황을 처리하는 중요한 방어 로직을 무단으로 삭제하거나, 비즈니스 규칙 자체를 미묘하게 변형해 버리는 돌이킬 수 없는 실수를 저지릅니다.
이러한 불상사를 원천적으로 차단하기 위해 동작의 불변성을 프롬프트 레벨에서 엄격하게 통제함으로써, 안심하고 프로덕션(Production) 환경에 배포할 수 있는 신뢰성 높은 리팩토링 코드를 얻어낼 수 있는 것입니다.
실무에서 당장 내일부터 바로 써먹을 수 있는 이 강력한 무기를 여러분의 개발 무기고에 단단히 장착하시고, 기술 부채와의 전쟁에서 승리하시길 바랍니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 소스 코드 파일 전체를 한 번에 다 붙여넣고 분석해도 괜찮을까요?**
  - A: 가급적 추천하지 않습니다. LLM의 컨텍스트 윈도우 한계로 인해 지시사항 집중도가 떨어지고 환각이 발생할 수 있습니다. 수백 줄이 넘어가는 방대한 파일을 통째로 넣기보다는, 50~150줄 내외의 '특정 함수'나 '단일 클래스' 단위로 쪼개어 입력해 보세요. AI가 문제의 핵심을 훨씬 날카롭게 파고들어 소름 돋게 정확한 리팩토링 인사이트를 제시해 줄 것입니다.

- **Q: AI가 완벽하게 고쳐준 코드를 그대로 프로덕션(Production) 환경에 배포해도 안전한가요?**
  - A: 절대 그대로 복사해서 붙여넣으시면 안 됩니다! 아무리 뛰어난 AI라도 비즈니스 로직의 미묘한 엣지 케이스를 놓치거나 의도치 않게 로직을 변형해 버리는 환각(Hallucination) 현상을 일으킬 수 있습니다. AI가 작성한 코드는 훌륭한 '초안'일 뿐입니다. **반드시 기존 유닛 테스트(Unit Test)가 모두 정상적으로 통과하는지 확인**하고, 개발자가 직접 로직의 무결성을 꼼꼼히 검증한 뒤에 머지(Merge)해야 합니다.

- **Q: 제가 사용하는 특정 프로그래밍 언어나 프레임워크의 특성에 맞춰서 코드를 개선해 줄 수도 있나요?**
  - A: 물론입니다. 전문가형(Pro) 프롬프트의 `[TypeScript/NestJS]` 변수 위치에 'Java/Spring Boot', 'Python/FastAPI', 'Go/Gin' 등 현재 실무에서 사용 중인 기술 스택을 정확히 명시해 보세요. AI가 해당 언어와 프레임워크 생태계에서 공식적으로 권장하는 네이밍 컨벤션(Naming Convention)과 최적의 디자인 패턴을 찰떡같이 반영하여 맞춤형으로 코드를 다듬어 줍니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **소프트웨어 공학 지표 중심의 정밀 타격:** AI에게 "코드를 단순히 예쁘게 정리해 줘"라고 모호하게 명령하는 대신, '순환 복잡도(Cyclomatic Complexity)'라는 구체적인 지표를 명시했습니다. 이를 통해 AI가 단순한 포맷팅을 넘어 아키텍처의 근본적인 구조 결함을 해결하는 데 온전히 컴퓨팅 파워를 집중하도록 설계했습니다.
2. **동작 유지 제약(Constraints)의 강력한 통제:** 리팩토링의 절대 원칙인 '외부 동작의 불변성'을 제약 조건의 최상단에 배치하여 엄격하게 제어했습니다. AI가 코드 길이를 줄이려는 욕심에 핵심 비즈니스 로직을 마음대로 무시하거나 엣지 케이스를 누락하는 치명적인 대참사를 원천적으로 차단합니다.
