## 💡 Комментарий автора (Insight)

Работа с сервером через терминал не прощает ошибок. Раньше я тратил часы на чтение мануалов (`man`) и отладку регулярных выражений для `awk` или скриптов для `cron`. Главная ценность этого промпта — **встроенный фокус на безопасность**.

Запрашивая у ИИ скрипт без строгих инструкций, вы часто получаете "грязный" код без проверок, который при пустой переменной может выполнить `rm -rf /` вместо `rm -rf /tmp/logs/`. Инструкции в Pro-версии заставляют нейросеть использовать `set -e` (остановка при ошибке) и `set -u` (ошибка при использовании необъявленных переменных), превращая ИИ из "опасного новичка" в осторожного Senior-инженера.

_Совет из практики:_ Всегда сначала запускайте сгенерированные скрипты в "сухом" режиме (например, заменяя опасные команды вроде `rm` на `echo rm`) или тестируйте их в изолированном Docker-контейнере.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Поймет ли ИИ, если я использую специфический дистрибутив Linux (например, Alpine или CentOS)?**
  - A: Да. Просто добавьте в `[Требования]` фразу: "Скрипт будет выполняться на CentOS 7, используй `yum`". ИИ автоматически адаптирует команды под пакетный менеджер и файловую структуру вашего дистрибутива.

- **Q: Можно ли попросить ИИ сделать скрипт интерактивным?**
  - A: Конечно! В блоке требований укажите: "Скрипт должен запрашивать у пользователя путь к папке и подтверждение (Y/N) перед выполнением".

- **Q: Как запустить сгенерированный скрипт?**
  - A: Сохраните код в файл (например, `script.sh`), сделайте его исполняемым командой `chmod +x script.sh`, а затем запустите: `./script.sh`.

## 🧬 Анатомия промпта (Почему это работает?)

1. **Строгие стандарты надежности:** Указание на использование `set -e`, `set -u` и `pipefail` сразу переводит контекст ИИ в режим написания "промышленного" (production-ready) кода.
2. **Защита от фатальных ошибок (Fail-safe):** Явный запрет на использование опасных команд без проверок защищает сервер от случайного удаления важных данных.
3. **Прозрачность выполнения:** Требование логирования (`echo`) позволяет вам в реальном времени видеть, что именно делает скрипт, и не паниковать, когда процесс занимает много времени.

## 📊 Доказательство: Before & After

### ❌ Before (Ваш запрос)

```text
Перемести все JPG файлы из текущей папки в папку images и добавь к имени файла сегодняшнюю дату.
```

### ✅ After (Результат ИИ)

```bash
#!/bin/bash
# Защита от непредвиденных ошибок
set -euo pipefail

TARGET_DIR="images"
DATE_PREFIX=$(date +%Y%m%d)

# 1. Проверка существования и создание целевой директории
if [[ ! -d "$TARGET_DIR" ]]; then
    echo "📁 Директория '$TARGET_DIR' не найдена. Создаю..."
    mkdir -p "$TARGET_DIR"
fi

# 2. Поиск и перемещение файлов
echo "🚀 Начинаю перемещение файлов..."
shopt -s nullglob # Избежать ошибки, если jpg-файлов нет
for file in *.jpg; do
    if [[ -f "$file" ]]; then
        new_name="${DATE_PREFIX}_${file}"
        mv "$file" "${TARGET_DIR}/${new_name}"
        echo "✅ Перемещено: $file -> ${TARGET_DIR}/${new_name}"
    fi
done

echo "🎉 Операция успешно завершена!"
```
