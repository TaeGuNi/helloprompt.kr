> _「AIが書いたコードでエラーが出たとき、未だにエラーメッセージをコピーしてAIに聞き直していませんか？」_

AIにPythonスクリプトを生成させて実行したら `SyntaxError` が発生。エラーをコピーしてAIに貼り付け、修正を依頼して初めてコードが動く。

しかし、ここで一つ質問です。**なぜ「人間」がその間で単純なコピー＆ペーストの仲介役をしているのでしょうか？**

2026年、最先端のエンジニアたちは単にAIへコード作成を依頼するだけではありません。彼らは**「自己修復システム（Self-Healing Systems）」**を構築しています。本記事では、「作成 → 実行 → 修正」へと続く退屈なデバッグループを自動化し、受動的なコード生成ツールを「自律的な問題解決エージェント」へと変貌させる方法をご紹介します。

## ⚡️ 3行要約 (TL;DR)

1. AIが一度で完璧なコードを書くという「One-Shot」の幻想から抜け出しましょう。
2. 実行エラー（`stderr`）をキャプチャし、AIにフィードバックとして返す「反省（Reflection）ループ」が鍵となります。
3. Pythonの `subprocess` モジュールを活用すれば、わずか50行のコードで自己修復エージェントを作成できます。

## 💻 Pythonでのループ自動化 (Implementation)

重量級のAIフレームワーク（LangChainなど）を使わなくても、Pythonの標準ライブラリである `subprocess` だけで、このループを見事にオーケストレーションできます。

```python
import subprocess
# OpenAIや他のLLMクライアントが設定されていることを前提とします。
# from openai import OpenAI
# client = OpenAI()

def generate_fix(prompt):
    # LLMを呼び出して修正されたコードを取得する関数（GPT-4、Claude、Llama 3など）
    # response = client.chat.completions.create(...)
    # return response.choices[0].message.content
    pass

def run_and_heal(script_path, max_retries=3):
    for attempt in range(max_retries):
        # 1. コード実行: エラーを安全にキャプチャするため、別プロセスで実行
        process = subprocess.run(
            ['python', script_path],
            capture_output=True, text=True
        )

        # 成功: returncode 0はエラーがないことを意味する
        if process.returncode == 0:
            print(f"✅ 成功!（試行回数: {attempt + 1}）")
            return process.stdout

        # 失敗: stderrからエラーメッセージを抽出
        error_msg = process.stderr
        print(f"❌ 失敗（試行回数: {attempt + 1}）: {error_msg.strip().splitlines()[-1]}")

        # 2. 自己修復 (Agentic Step)
        with open(script_path, 'r') as f:
            broken_code = f.read()

        healing_prompt = f"""
        このエラーを基にPythonコードを修正してください。

        [元のコード]
        {broken_code}

        [エラーメッセージ]
        {error_msg}

        説明やマークダウンの装飾は省き、「修正されたコード」のみを返してください。
        """

        fixed_code = generate_fix(healing_prompt)

        # 修正されたコードで元のスクリプトを上書き
        with open(script_path, 'w') as f:
            f.write(fixed_code)

        print(f"🩹 コードの修復が完了しました。再実行します...")

    print("💀 最大再試行回数を超過したため、修復に失敗しました。")
    return None
```

## 💡 作成者コメント (Insight)

この「自己修復（Self-Healing）」パターンは、AIの活用方法を根本から変える力を持っています。

以前、Webスクレイピングの自動化ツールを作成した際、AIが存在しないCSSセレクタをハルシネーション（幻覚）として出力し、コードが機能しなくなる問題がありました。その時、単なる構文エラーを修正するだけでなく、「抽出したリストが空（Empty）であればエラーと見なす」という**検証ループ（Validation Loop）**を追加しました。

すると驚くべきことが起こりました。エージェントが正常にデータを抽出できるまで、**自ら別のCSSセレクタを試し、コードを修正**し始めたのです。これは単なるバグ修正の枠を超え、AIが与えられた環境（Webページの構造）に**適応（Adaptation）**していく過程でした。コードを生成すること以上に重要なのは、生成されたコードを「検証」し、「責任」を持たせるループを設計することなのです。

## 🙋 よくある質問 (FAQ)

- **Q: コードを上書きしてスクリプトが完全に壊れてしまったらどうしますか？**
  - A: 実務では、`script_path` を直接上書きするのではなく、`script_v1.py`、`script_v2.py` のようにバージョン管理を行って保存するか、Gitのコミットを自動化していつでもロールバックできる安全装置を設けることを推奨します。

- **Q: 無限ループに陥る危険性はありませんか？**
  - A: だからこそ `max_retries`（最大再試行回数）の設定が不可欠です。通常、3〜5回以内で解決しないエラーは、プロンプト自体の限界であるか、人間の介入が必要なアーキテクチャ上の欠陥である可能性が高いです。

- **Q: どのLLMモデルを使用するのが最適ですか？**
  - A: コードの修正と論理的な推論が必要なため、GPT-4o、Claude 3.5 Sonnet、またはコーディングに特化した最新のローカルモデルを強くお勧めします。

## 🧬 プロンプトの解剖 (Why it works?)

1. **エラー原因の分析を強制（Chain-of-Thought）:** いきなりコードを修正させるのではなく、先にエラーログを分析するというタスクを与えることで、AIに問題の本質をまず理解させるよう誘導しました。
2. **出力の制御（Systematic Output）:** 「説明なしでコードのみを出力せよ」という強力な制約を設けることで、Pythonコードをパースする段階において、不要なテキストが原因で発生する二次エラーを防ぎました。
