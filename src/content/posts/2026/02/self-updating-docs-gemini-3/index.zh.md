# 📝 复活僵尸文档：使用 Gemini 3 Pro 代理实现文档自动化

- **🎯 推荐对象：** 技术负责人 (Tech Lead)、后端开发者、DevOps 工程师
- **⏱️ 所需时间：** 每次修改文档 30 分钟 → 0 分钟（全自动化）
- **🤖 推荐模型：** Gemini 3 Pro（必备 2M+ 超大上下文）

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效率：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _"每天都在频繁部署代码，API 文档却还停留在去年夏天的版本？是时候让你的文档学会‘自我进化’了。"_

任何开发者都会对这种痛点感同身受：代码每天都在日新月异，而文档却总停留在大概一个月、甚至半年前的状态。那句“我稍后会更新文档的”承诺，往往只是一句无法兑现的借口。因为只有在代码被修改的**那一瞬间**，文档的同步更新才具有真正的意义。

然而，人类天生抗拒这种繁琐的工作。正因如此，我们必须将这项枯燥且痛苦的任务委托给 **AI 代理 (AI Agent)**。过去的 AI 模型受限于较短的上下文窗口，无法在修改单个文件时统揽全局，难以洞察变更对整体文档结构的影响。但如今，借助 **Gemini 3 Pro** 令人惊叹的逻辑推理能力与极其庞大的上下文窗口（2M+），这一切已发生颠覆性的改变。

今天，我们将公开一套完美的提示词工作流：通过精准捕捉代码的变更差异（Diff），让文档实现自动且精准同步的 **"Self-Updating Docs（自我更新文档）"**。

---

## ⚡️ 核心摘要 (TL;DR)

1. **自动监听：** 构建自动化流水线，在代码提交时自动捕捉代码差异（Diff），由 AI 代理接管文档更新。
2. **智能过滤：** 自动忽略拼写错误修正或简单的代码重构，仅对“真正需要记录的结构性变更”（如新增 API 接口、环境变量等）进行提取和文档化。
3. **人工审核介入（Human-in-the-Loop）：** AI 生成的文档更新不会直接合并，而是通过自动提交 PR（Pull Request）的方式，交由开发者进行最终的审查与把关。

## 🚀 解决方案："文档维护助手 (Docs Maintainer)"

### 🥉 Basic Version (基础版)

当你需要快速生成 PR 说明，或者仅仅在本地进行简单的 README 更新时，请使用此版本。

> **角色：** 你是一位资深的技术布道师（Technical Writer）。
> **任务：** 请查看我刚刚修改的 `[Git Diff]` 内容，找出 `[README.md]` 中需要同步更新的部分，并使用 Markdown 格式重新编写。请忽略单纯的错别字修正或代码重构相关的细微变更。

<br>

### 🥇 Pro Version (专业版)

这是一套专为集成至 CI/CD 流水线（如 GitHub Actions）而设计的精密提示词，旨在打造一个完全自主运行的 AI 代理。请务必将 `temperature` 设置为 `0.1` 或更低，以严格控制其发散性和创造力。

> **角色 (Role)：** 你是一个名为“文档维护助手 (Docs Maintainer)”的 AI 代理，具备 10 年经验的资深技术布道师与 DevOps 工程师双重背景。
>
> **背景 (Context)：**
>
> - 现状：开发者刚刚向代码库推送了新的提交 (Commit)。这些变更可能包含新增的 API 端点、新的环境变量，或是核心业务逻辑的修改。
> - 目标：深度解析变更的代码，将现有陈旧的文档精准同步至最新状态。
>
> **任务 (Task)：**
>
> 1. 将提供的 `[Git Diff]` 与完整的 `[Target Document]` 内容进行交叉对比分析。
> 2. 评估此次代码变更是否会影响用户接入或其它开发者的环境配置，从而决定是否需要更新文档。（必须严格忽略单纯的代码重构、变量重命名及拼写修改）。
> 3. 若判定需要更新，请在**完美保持原文档的语气与排版格式 (Tone and Manner)** 的前提下，输出修改后的完整 Markdown 文档。
> 4. 在文档的最底部，以 HTML 隐藏注释的形式 `<!-- 变更说明：简述为何修改了这部分内容 -->` 留下你的更新理由。
> 5. 若判定此次提交无需更新文档，请**仅输出** `[NO_UPDATE_REQUIRED]`。
>
> **限制条件 (Constraints)：**
>
> - 绝对禁止捏造不存在的功能或虚构环境变量。（严禁幻觉 Hallucination）
> - 禁止使用诸如“已适当修改”这类模糊的表述；必须明确写出新增的 Endpoint URL 或变量名等具体数值。
> - 绝对不要将带有 TODO、FIXME 等注释的未完成代码写入文档。
> - 你的输出只能是“修改后的完整 Markdown 文档”或“[NO_UPDATE_REQUIRED]”，绝不能包含任何多余的对话或解释。

## 💡 创作者洞察 (Insight)

在将这套提示词实际部署到我们内部的后端仓库后，最令我感到震撼的是它**“敏锐捕捉遗漏环境变量的能力”**。曾经有位开发者在 `config.ts` 中新增了一个第三方 API 密钥，却忘记在 `README.md` 的“本地环境配置”章节中进行补充。Gemini 3 Pro 在对齐代码和文档后，仿佛长了眼睛一般，极其精准地在 `README.md` 中自动添加了 `NEW_3RD_PARTY_API_KEY` 的环境配置项。

**实战应用技巧：** 如果在每一次微小的 Commit 上都触发这个代理，代币（Token）成本将会呈指数级爆炸。建议在 GitHub Actions 中配置 `paths` 过滤器，设定仅当核心逻辑文件（如 `src/api/**`, `config/**`）或数据库结构文件发生变更时才触发流水线。此外，切勿让 AI 自动将修改后的文档直接 Push 到 `main` 分支。**务必让其自动生成一个 Pull Request (PR)**，在代码审查阶段交由开发者进行二次确认。人工的最终把关（Human-in-the-Loop），是捍卫文档准确性与可靠性的最后一道防线。

## 🙋 常见问题 (FAQ)

- **Q: 仅仅提供 Git Diff，AI 真的能理解所有的上下文吗？**
  - A: 对于简短明确的修改，Diff 确实足够了；但在处理复杂的逻辑变更时，仅靠 Diff 很容易导致 AI 输出偏离预期的文档。充分利用 Gemini 3 Pro 惊人的上下文窗口，将 **Git Diff** 以及变更文件的**完整代码**和**整个目标文档**作为 Input 一起丢给它，这是大幅提升准确率 (Accuracy) 的核心秘诀。

- **Q: AI 总是把代码里开发者写的个人吐槽或者 TODO 注释也写进文档里，该如何阻止它？**
  - A: 这是因为 AI 过于“尽职尽责”导致的。我们在**限制条件 (Constraints)** 章节中特意加上了 `- 绝对不要将带有 TODO、FIXME 等注释的未完成代码写入文档。` 这一指令，正是为了解决此痛点。有了这句话，这类情况将被干净利落地屏蔽。

- **Q: 换成 Gemini 1.5 Pro 或者 GPT-4o 也能实现一样的效果吗？**
  - A: 可以运行。但当我们采用“完整文档 + 完整代码文件”这种暴力输入方式时，在上下文长度限制和 Context Caching（上下文缓存）的成本控制方面，Gemini 3 Pro 占据压倒性优势。如果你只是在一个体量较小的玩具项目中使用，GPT-4o 同样能给出极其出色的表现。

## 🧬 提示词原理解析 (Why it works?)

1. **阻断逻辑 (Bypass Logic)：** 针对无需更新文档的常规代码提交，通过强令模型输出 `[NO_UPDATE_REQUIRED]` 进行了严格阻断，从根本上防止了系统无限生成无效 PR 的问题。
2. **强制语气一致性：** 明确下达了“保持原文档语气与排版格式”的指令，防止 AI 抛弃原有的文档规范，使用冷冰冰且充满违和感的“机器味”语言覆盖旧文档。
3. **隐性审计线索 (Hidden Explanation)：** 巧妙利用 HTML 注释 (`<!-- -->`) 记录修改动机，这不仅不会破坏文档展示的版面，还能让代码审查者 (Reviewer) 在审核 PR 时，一目了然地验证 AI 的推理逻辑，大幅降低审核成本。
