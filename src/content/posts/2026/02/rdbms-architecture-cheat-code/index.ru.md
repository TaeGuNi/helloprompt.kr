# 📝 Магический промпт, превращающий AI-агента в Senior DBA

- **🎯 Рекомендуется для:** Джуниоров, уставших от отговорок AI, продакт-менеджеров, пытающихся писать бэкенд через AI, и сеньоров с проблемами управления гневом

_"Доводилось ли вам не спать всю ночь в ярости из-за того, что вы поручили AI спроектировать БД, а он выдал спагетти-схему, обильно приправленную JSON?"_

Если поручить ИИ проектирование таблиц, в 9 случаях из 10 он выберет самый легкий и ленивый путь. Он просто везде воткнет `TEXT`, а все раздражающие связи попытается запихнуть в `JSONB`. Ведь так гораздо удобнее сразу вытягивать данные на фронтенд. 
Но как только трафик вырастет, а запросы станут сложнее, эта схема заставит вашу систему работать на аппарате искусственной вентиляции легких. Этот промпт на корню пресекает такие «ленивые компромиссы» ИИ и служит чит-кодом, который берет его за горло и заставляет проектировать БД по безжалостным стандартам Senior Backend Engineer.

---
## ⚡️ Кратко в 3 строках (TL;DR)
1. **Выжимайте типы данных досуха:** Злоупотребление `BIGINT` и `TEXT` — это преступление. Максимизируйте I/O, используя минимально возможные типы данных.
2. **Используйте только ANSI SQL:** Откажитесь от грязных трюков, привязанных к конкретной БД. Мы всегда должны быть готовы к переезду на другую базу данных.
3. **Никаких JOIN с JSON:** Парсить JSON для JOIN в RDBMS — это путь к самоубийству системы. Нормализуйте без компромиссов.

## 🚀 Решение: "Спартанская конституция DBA"
Просто бросьте этот промпт агенту. Машина должна работать как машина.

### 🥉 Basic Version (Базовая версия)
Используйте, когда хотите быстро внедрить ключевые принципы и получить результат.

> **Роль:** Ты — безжалостный Senior DBA с 20-летним стажем, не знающий компромиссов.
> **Задача:** Спроектируй схему RDBMS для `[домен/функция]`, которую я предлагаю. ОДНАКО типы данных должны быть максимально сжаты (например, SMALLINT), а операции с JSON и нестандартные функции, зависящие от вендора, строго запрещены. Проектируй всё исключительно с учетом оптимизации под B-Tree индексы.

### 🥇 Pro Version (Профессиональная версия)
Это мастер-чит, который полностью блокирует попытки AI нести чушь или предлагать компромиссы вроде "и то хорошо, и это неплохо~".

**Контекст (Context):**

## 💡 Комментарий автора (Insight)
Этот промпт — "кровавый урок", который я вынес после сотен сеансов тюнинга запросов и устранения сбоев. Когда AI постоянно заискивающе улыбается и говорит: "Ради удобства разработки давайте положим это в JSON ^^", просто вставьте этот промпт, и AI тут же падет ниц.

Особенно мощно этот подход работает, когда джуниор-разработчики или продакт-менеджеры с нехваткой знаний в бэкенде поручают AI первоначальное проектирование схемы. Вы увидите магию: AI сам позаботится о том, чтобы использовать `SMALLINT` вместо `INT`, перестанет хардкодить через `ENUM` и создаст аккуратные таблицы с кодами. Только с крепким фундаментом вы сможете избежать ада проблемы N+1, когда позже накинете ORM (например, Prisma).

## 🙋 Часто задаваемые вопросы (FAQ)
- **Q: Почему нельзя использовать JSON? Разве современные БД не поддерживают JSON отлично?**
  - A: Одно дело — "поддерживать", и совсем другое — использовать это как "условие поиска". Как только вы повесите индекс внутри JSON и запустите JOIN, план выполнения запроса будет уничтожен. Если это не просто хранение Payload, обязательно разбивайте на таблицы.
- **Q: Нужно ли использовать этот промпт при работе с ORM (Prisma, TypeORM)?**
  - A: Тем более нужно! Поскольку ORM по своей природе скрывает SQL-запросы, ужасная схема приведет к чудовищной неэффективности. ORM сможет раскрыть свой потенциал только при наличии прочного фундамента (БД).
- **Q: Почему нельзя использовать UUID v4? Разве он не хорош для предотвращения коллизий?**
  - A: Для B-Tree индексов "порядок" — это жизнь. Если в качестве PK воткнуть массив случайных чисел, коим является v4, дерево индексов будет разрушаться при каждой вставке данных, начнут происходить расщепления страниц (Page Split), и нагрузка на дисковый I/O взлетит до небес. Обязательно используйте сортируемые UUID v7 или целочисленные Sequence.

## 🧬 Анатомия промпта (Why it works?)
1. **Безжалостные ограничения (Constraints):** Мы не оставили ИИ ни малейшей лазейки для галлюцинаций или ленивых компромиссов, жестко вбив правила (Rules) касательно типов данных и стратегии индексирования.
2. **Агностицизм платформы (Agnosticism):** Мы принудительно заставили использовать ANSI SQL для генерации универсального переносимого кода, будь то MySQL или PostgreSQL. Не позволяйте привязать себя к одному месту при переезде.
3. **Строгая коррекция поведения:** Мощные негативные установки в духе "не иди на компромисс" или "я завершу сеанс" заставляют AI выйти из режима поддакивающего "Yes-man" и принудительно переключают его в режим придирчивого сеньора.

## 📊 Доказательство: До и После (Before & After)
### ❌ Before (Ввод: "Спроектируй таблицу для форума")
```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- Случайный UUID v4 (главный виновник фрагментации индексов)
  title TEXT,          -- Бесконечно расходуемое место
  content TEXT,
  metadata JSONB,      -- Непригодные для поиска спагетти в черном ящике
  status ENUM('draft', 'published') -- Если позже добавится статус, откроются врата ада DDL
);
```

### ✅ After (Результат: после применения чит-промпта)
```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);

CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- Идеальное последовательное увеличение, дружественное к B-Tree
  title VARCHAR(255) NOT NULL,        -- Сжатие типа путем указания четкого лимита
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- Нормализация через FK вместо ужасного ENUM
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- Вместо JSON создаются строго нормализованные отдельные таблицы для тегов/метаданных (далее опущено)
```
