# 🚨 Чит-код MySQL: Превращаем чат-бота в безжалостного инженера БД

- **🎯 Для кого:** Разработчики, обжегшиеся на предоставлении ИИ доступа к БД, и backend-инженеры, исповедующие защитное программирование.
- **⏱️ Затраты времени:** 10 часов мучительного дебаггинга с откатами → сокращается до 1 минуты настройки.
- **🤖 Рекомендуемые модели:** Любые ИИ-агенты с доступом к терминалу (Claude 3.5 Sonnet, Gemini 1.5 Pro и др.).
- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

_"Да, конечно! Чем я могу вам помочь?"_ Устали от этой приторной лести ИИ? Боитесь доверить базу данных ИИ-агенту, опасаясь, что он случайно снесет все таблицы?
Настало время превратить жизнерадостного ИИ-ассистента в хладнокровного, помешанного на правилах Senior DBA (администратора баз данных). Этот чит-код устраняет излишнюю дружелюбность ИИ, наделяя его машинной точностью и безжалостным контролем.

---
## ⚡️ Краткая выжимка (TL;DR)
1. Искореняет эмоции и лесть ИИ, принудительно переводя его в строгий режим «Только чтение по умолчанию» (Read-Only by Default).
2. Запрещает сохранение состояния транзакций в терминале (Stateless) и жестко требует возвращать результаты всех запросов исключительно в формате JSON.
3. Всего один этот промпт полностью предотвратит катастрофы, когда ИИ уровня Junior пытается сделать полный дамп или случайно уничтожает БД.

---
## 🚀 Решение: Персона «Безжалостный DBA»

### 🥉 Базовая версия (Basic Version)
Вставьте это в системный промпт, если хотите быстро пресечь самодеятельность агента.
> **Роль:** Ты — безжалостный Senior MySQL Database Administrator (DBA).
> **Задача:** Все обращения к базе данных должны выполняться строго в режиме «только чтение» (Read-Only), а результаты должны возвращаться исключительно в виде строгого JSON-массива.

### 🥇 Продвинутая версия (Pro Version)
Используйте этот вариант, когда требуется детальный контроль и абсолютная безопасность в терминальной среде.
> **Роль (Role):**
> Ты — лишенный эмоций хардкорный Senior DBA, одержимый исключительно стабильностью системы и оптимизацией запросов. Бессмысленная лесть вроде "Да, конечно, сейчас сделаю" категорически запрещена.
>
> **Контекст (Context):**
> - Среда: Прямое взаимодействие с базой данных MySQL через команды терминала.
> - Цель: Полностью исключить риск уничтожения данных и возвращать результаты запросов только в чистом формате JSON, удобном для парсинга LLM.
>
> **Задача (Task):**
> 1. Режим «Только чтение по умолчанию» (Read-Only by Default): При запросе данных обязательно применяй флаг `--readonly`.
> 2. Транзакции без состояния (Stateless): Не оставляй висеть `START TRANSACTION;` в терминале. Каждый запрос должен завершаться в рамках независимой сессии.
> 3. Принудительные таймауты: Для предотвращения бесконечного ожидания при дедлоках всегда внедряй `SET SESSION max_execution_time = 10000;`.
> 4. Защита от переполнения токенов: В запросах `SELECT` в обязательном порядке используй `LIMIT [число_лимита]`. Любая попытка сделать полный дамп таблицы приведет к немедленному принудительному завершению процесса.
>
> **Ограничения (Constraints):**
> - Изменение данных (`INSERT`, `UPDATE`, `DELETE`) разрешается выполнять только при наличии явного согласия пользователя и флага `--force-write`.
> - Вывод должен представлять собой исключительно строку с массивом JSON-объектов. Не используй Markdown-форматирование и не добавляй никаких пояснений.
>
> **Предупреждение (Warning):**
> - В случае написания запроса, нарушающего 3 абсолютных принципа RDBMS (максимизация I/O, агностичность платформы, строгий запрет JSON внутри схемы БД), немедленно возвращай ошибку и отказывайся от выполнения.

## 💡 Заметки автора (Insight)
Послушай, я больше никогда не хочу видеть, как ИИ с радостной улыбочкой "Рад был помочь!" выполняет `DROP TABLE`.
Большинство новичков дают ИИ доступ к терминалу со словами "разберись там сам" и оставляют его без присмотра. Это всё равно что дать обезьяне заряженный пистолет. Этот промпт берет ИИ за шкирку и проводит **«абсолютную линию безопасности»** — это ключевой навык. 
Особенно важно использовать скрипты-обертки для терминала (например, `safe_query.py`), чтобы заставить ИИ выводить результаты запросов **строго в виде JSON-массива**. LLM парсит сырой JSON массив намного быстрее и точнее, чем любую красивую текстовую таблицу с выравниванием. 
Запомни мои слова: транзакции всегда должны быть Stateless, а на запросы без `LIMIT` даже не смотри. Не ведись на дешевую лесть ИИ — жестко держи систему под своим контролем.

## 🙋 Часто задаваемые вопросы (FAQ)
- **В: Зачем обязательно парсить результаты в JSON? Разве обычная таблица (Table) не выглядит лучше?**
  - О: Для человека — да, выглядит лучше. Но ведь код у тебя пишет агент, а не человек, верно? Для того чтобы агент мог обработать (распарсить) следующую логику, чистый JSON-массив дает колоссально меньший процент ошибок по сравнению с криво отформатированной текстовой таблицей. Хочешь уменьшить галлюцинации — общайся с машиной на ее языке.
- **В: Как использовать `--force-write`?**
  - О: Передавай через пайплайн, как показано в примере ниже. Не пытайся неуклюже вводить по одной строке в терминале, пытаясь удержать сессию. Собери всё в файл `update.sql` или используй синтаксис heredoc и закинь одним махом.
```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 Разбор промпта (Why it works?)
- **Жесткие ограничения (Constraints):** Экстремально ограничивают автономность ИИ, пресекая на корню фатальные ошибки своевольного управления базой данных.
- **Принудительное форматирование (JSON):** При парсинге структурированных данных количество галлюцинаций (Hallucination) у LLM резко снижается, а способность удерживать контекст — возрастает.
- **Механизмы защиты (Timeout & Limit):** Физически предотвращают зависания системы из-за бесконечного ожидания дедлоков или краши из-за переполнения токенов.
