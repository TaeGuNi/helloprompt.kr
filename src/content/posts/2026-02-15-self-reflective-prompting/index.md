# Self-Reflective Prompting: AI 정확도 향상하기

## 서론

대규모 언어 모델(LLM)이 급격히 발전하는 환경에서 정확도는 여전히 가장 중요한 과제입니다. GPT-4나 Gemini와 같은 모델은 매우 뛰어나지만, 여전히 "환각(hallucinations)"과 논리적 불일치 현상을 보일 수 있습니다. 차세대 AI 애플리케이션을 구축하는 개발자로서 우리는 종종 모델의 첫 번째 응답을 최종 진실로 취급하곤 합니다. 하지만 인간의 인지 과정을 모방하여 신뢰성을 크게 높일 수 있는 강력한 기법이 있습니다. 바로 **Self-Reflective Prompting(자가 성찰 프롬프팅)**입니다.

이 글에서는 AI가 자신의 작업을 평가하도록 강제함으로써 "적당히 괜찮은" 응답을 견고하고 프로덕션 환경에 적합한 솔루션으로 변환하는 방법을 살펴봅니다.

## 분석

Self-reflective prompting은 간단한 전제에서 작동합니다. LLM은 종종 단 한 번의 시도로 처음부터 솔루션을 생성하는 것보다 솔루션을 검증하는 데 더 능숙하다는 것입니다. 검토 단계를 도입함으로써 우리는 모델이 초기 생성 과정에서 범했을 수 있는 오류를 잡아내도록 할 수 있습니다.

### 핵심 메커니즘 (The Core Mechanism)

워크플로우는 일반적으로 두 단계로 구성됩니다.
1.  **생성(Generation):** 모델이 초기 답변을 생성합니다.
2.  **성찰(Reflection):** 모델에게 자신의 답변을 다시 제공하고 잠재적인 결함이나 개선 사항을 식별하도록 요청합니다.

여기서 핵심 통찰은 간단하지만 혁신적입니다. **최종 확정 전에 모델에게 자신의 결과물을 비평하도록 요청하는 것입니다.**

### 실제 구현 (Practical Implementation)

자연어에서 SQL 쿼리를 생성하는 시나리오를 고려해 보겠습니다. 표준 프롬프트는 다음과 같을 수 있습니다.

> "지난달 지출액 기준 상위 5명의 사용자를 찾는 SQL 쿼리를 작성하세요."

Self-reflective 접근 방식은 정밀 검사 단계를 추가합니다.

> **1단계 (생성):** "지난달 지출액 기준 상위 5명의 사용자를 찾는 SQL 쿼리를 작성하세요."
>
> **2단계 (비평):** "위의 SQL 쿼리를 검토하세요. 시간대(timezone)를 올바르게 처리합니까? 취소된 주문은 제외합니까? 오류가 있다면 설명하세요."
>
> **3단계 (개선):** "비평에서 식별된 오류를 수정하여 SQL 쿼리를 다시 작성하세요."

명시적으로 비평을 유도함으로써 단순히 패턴을 완성하는 것이 아니라 추론 및 논리와 관련된 모델의 특정 경로를 활성화하게 됩니다. 이러한 "내적 독백(internal monologue)"은 모델이 사용자의 의도와 기술적 제약 조건에 더 밀접하게 부합하는 결과물을 내도록 돕습니다.

### 왜 효과적인가 (Why It Works)

1.  **오류 수정:** 일반적인 구문 오류나 논리적 허점(예: 코드의 off-by-one 오류)을 잡아냅니다.
2.  **문맥적 정렬:** 엣지 케이스(위 예시의 "취소된 주문" 같은)가 고려되도록 보장합니다.
3.  **생각의 사슬(Chain of Thought):** 모델이 자신의 추론을 명확히 설명하도록 강제하여 본질적으로 최종 결과물의 품질을 향상시킵니다.

## 결론

Self-reflective prompting은 "프롬프트 입력 후 기도하기(prompt and pray)"에서 "프롬프트 입력, 검증, 그리고 개선하기"로 패러다임을 전환합니다. LLM 파이프라인에 비평 단계를 통합함으로써 오류율을 획기적으로 줄이고 더 탄력적인 AI 시스템을 구축할 수 있습니다.

개발자에게 주는 교훈은 명확합니다. 초안에 안주하지 마십시오. AI 에이전트가 스스로 편집자가 되도록 권한을 부여하고, 애플리케이션의 결과물 품질이 급상승하는 것을 지켜보십시오.
