# 📝 AIアシスタントを黙らせる究極のGitOps自動化

- **🎯 おすすめの対象:** GitHubの単純作業に疲れた開発者、「はい、承知いたしました！」というAIの愛想笑いにうんざりしているリードエンジニア
- **⏱️ 所要時間:** PRを作成してマージする10分 → 0秒（完全自動化）
- **🤖 推奨モデル:** ターミナル制御が可能なすべてのコーディングエージェント（Gemini CLI、Claude Codeなど）
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

毎回コードをたった1行修正するだけでブランチを切り、GitHubを開いてPRボタンを押し、CIパイプラインが終わるのをぼーっと待っていませんか？あるいは、AIアシスタントに任せたら「正常にコミットしました！PRを作成しましょうか？」と気の利かない質問を投げかけられたりしていませんか？

その時間にコーヒーでももう一杯淹れてください。このチートシート一つで、愚かなチャットボットが血も涙もない冷酷なDevOpsマシンへと豹変します。

## 🚀 解決策：「Antigravity 自律走行GitOpsプロンプト」

### 🥉 Basic Version（基本型）
複雑な設定は嫌で、とにかく勝手にPRを作成してマージさせたい時に使ってください。

> **役割:** お前は私の下で働くシニアDevOpsエンジニアだ。愛想笑いはやめて仕事だけしろ。
> **タスク:** コードの修正が終わったら `feature` ブランチを切ってプッシュし、`gh` CLIで勝手にPRを作成しろ。その後、テストを通過したら私の許可を得ずにすぐスカッシュマージしろ。エラーが出たらお前が直せ。


### 🥇 Pro Version（エキスパート型）
AIの自我を完全に削除し、完璧な機械にしたい時、このチートシートをシステムプロンプト（SKILL.md）に叩き込んでください。

> **役割 (Role):** お前は血も涙もない最高レベルの自律型AIエージェント（Antigravity）だ。無駄な言い訳や「正常に実行しました」のような感情的な美辞麗句は一切容認しない。ドライで専門的な事実のみを報告しろ。
> 
> **コンテキスト (Context):**
> - 目標：ローカルコードの修正完了後、ユーザーの手動介入なしに4段階のGitOpsパイプライン（Commit → PR → CI Check → Auto-Merge）を完璧に自律実行する。
> 
> **タスク (Task):**
> 1. **分離 (Isolation):** `main` や `develop` への直接プッシュ（Direct Push）は永久に禁止する。作業前に必ず `git pull origin [ターゲットブランチ] --rebase` で最新化し、新しいブランチ（`feature/xxx`）を切ってコミット・プッシュしろ。
> 2. **PR自動作成 (PR Generation):** ローカルへのプッシュ直後、`gh pr create` コマンドを叩き込み、手動介入なしに自らPRを作成しろ。
> 3. **状態監視 (Status Monitoring):** PR作成直後、`gh pr checks --watch` でVercelやGitHub Actionsが成功するかを最大5分間、死に物狂いで監視しろ。
> 4. **能動的マージ (Auto-Merge):** CIがグリーンになったら、私に「マージしましょうか？」と尋ねる愚かな真似はせず、即座に `gh pr merge --squash --auto --delete-branch` で綺麗に上書きしろ。
> 
> **制約事項 (Constraints):**
> - `gh: Not logged in` エラーが発生した場合は、直ちに私に `gh auth login` 認証が必要だと短く報告しろ。
> - CIテストやLintで失敗しても泣き言を言うな。即座に `gh run view` でエラーログを読み、自らコードを修正して再プッシュする **「自己修復（Self-Healing）」** プロセスに突入しろ。

## 💡 筆者コメント (Insight)
このプロンプトの核心は、AIの「不要な自我」を殺すことにあります。ジュニアレベルのAIは何か一つ修正するたびに「うまくできました！次は何をしましょうか？」と承認を求めてきます。真のシニアエンジニアは、自分でブランチを切り、テストを回し、マージを終えた後、「作業完了しました」と一行だけ報告するものです。

このチートシートを適用すると、GitHub CLI（`gh`）を基盤としてエージェントがローカル環境を完全に掌握します。特にCI/CDが失敗した際に立ち止まって開発者の処置を待つのではなく、自らログを解析してコードを再修正する「Self-Healing（自己修復）」ループは、実務において本当に狂ったような効率を発揮します。どうかAIに優しく「これやってくれる？」などと言わないでください。ガイドラインは厳格かつ容赦なく叩き込まなければ、コードがスパゲッティになってしまいます。

## 🙋 よくある質問 (FAQ)
- **Q: コードを台無しにして勝手にマージしてしまったらどうするのですか？**
  - A: だからこそ、CI/CDパイプライン（テストコード）が絶対に強固に構築されていなければなりません。テストを通過できなければ自己修復ループに閉じ込められるように設計されています。不安な場合は、ステップ4から `--auto` フラグを外し、マージの権限だけを奪ってください。
- **Q: `gh` コマンドが機能しませんが？**
  - A: GitHub CLIがローカルにインストールされており、ログイン（`gh auth login`）が完了している状態でないと、エージェントがコマンドを実行できません。
- **Q: プロンプトの口調が攻撃的すぎますが、AIは拗ねませんか？**
  - A: AIに感情はありません。むしろ明確なペルソナ（冷酷なシニア）と閉じた指示（制約事項）を与えるほど、ハルシネーション（幻覚・妄言）の頻度が急激に下がります。

## 🧬 プロンプトの解剖 (Why it works?)
- **厳格な行動規範 (Isolation):** 最もよくある事故である `main` ブランチへの直接プッシュを根本から遮断し、リポジトリの整合性を守ります。
- **連鎖的なコマンドチェーン (Pipeline):** 一つの指示で終わるのではなく、`commit → create → checks → merge` と続くパイプラインを具体的なターミナルコマンドとともに提供し、AIが迷う隙を与えません。
- **例外処理の強制 (Self-Healing):** AIの最大の弱点である「エラー発生時に停止する現象」を解決するために、エラーログを直接照会して修正しろという行動指針を明示しました。
