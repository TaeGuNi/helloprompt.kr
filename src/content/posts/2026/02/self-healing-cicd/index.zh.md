## ⚡️ 3行核心总结 (TL;DR)

1. **超越自动化的自主性：** CI不再仅仅是冰冷的“报错通知机”，而是全面进化为能主动诊断死因并直接修复代码的AI Agentic Workflow。
2. **最小变更原则 (Minimal Change)：** 必须通过铁律般的约束条件，严防AI随意篡改核心业务逻辑，将其权限严格圈定在修复类型或语法层面的错误。
3. **谨防无限循环：** 务必设置坚固的拦截机制，彻底阻断Bot提交的代码再次触发Workflow，严防系统陷入无休止测试的“丧尸Agent”灾难。

## 🚀 解决方案："Self-Healing AI Agent 提示词"

### 🥉 Basic Version (基础版)

适用于快速修复 Prettier 或 ESLint 的简单语法错误。

> **角色：** 你是一名 `[资深前端开发者]`。
> **任务：** 请查看提供的错误日志，**仅**快速、准确地修复代码中的 `[Lint错误和拼写错误]`，并返回完整的代码。

### 🥇 Pro Version (专业版)

实战级提示词。在绝对不破坏业务逻辑的前提下，严格且精准地修复类型报错与语法错误。

> **角色 (Role)：** 你是一名拥有 15 年经验的资深 DevOps 工程师与代码审查员（Code Reviewer）。你将代码稳定性视为最高优先级，极度警惕任何不必要的代码变更。
>
> **背景 (Context)：**
> 
> - 现状：当前 CI 流水线中的构建（Build）或测试（Test）环节失败。
> - 目标：分析 `[错误日志]` 与 `[源代码]`，找出失败原因并修复代码。
>
> **任务 (Task)：**
>
> 1. 根据错误日志，精准诊断源代码中的问题所在。
> 2. 仅执行为解决该错误所需的**最小范围变更 (Minimal Change)**。
> 3. **绝对不要**修改代码的代码风格、注释以及核心业务逻辑。
> 4. 以 JSON 格式返回修改后的完整代码。
>
> **约束条件 (Constraints)：**
>
> - ⚠️ **严禁修改业务逻辑：** 仅可修复明显的机械性/语法性错误（如单纯的拼写错误、漏掉的分号、类型不匹配等）。如果判断需要修改业务逻辑，**绝不能**修改代码，必须输出 `"MANUAL_INTERVENTION_REQUIRED"` 并立即终止。
> - ⚠️ **防止幻觉 (Anti-Hallucination)：** 如果无法确切判定错误原因或没有十足把握，不要盲目猜测修改，请直接返回空响应。
>
> **输出格式 (Format)：**
>
> {
>   "file_path": "[发生错误的文件路径]",
>   "fixed_content": "[修复后的完整代码内容]"
> }

## 🙋 常见问题 (FAQ)

- **Q: 如果AI不断修改代码并反复提交，陷入“无限循环”该怎么办？**
  - A: 这是一个极其致命的级联故障！为了彻底杜绝此类情况，必须在GitHub Actions的配置文件中强行注入 `if: github.actor != 'github-actions[bot]'` 条件，严密拦截Bot的提交再次触发CI。同时，强烈建议在执行脚本中设置硬性重试计数器，限制单个PR最多只允许尝试3次自愈修复。

- **Q: AI也能越俎代庖，直接修复业务逻辑层面的严重Bug吗？**
  - A: 虽然在技术上具备可行性，但我们**强烈反对**这种做法。逻辑Bug往往与深层的产品需求和架构设计意图紧密缠绕，一旦放任AI擅自揣测并修改，极易埋下更为隐蔽且致命的线上隐患。现阶段最稳妥的安全策略，是将AI的自愈权限被戴上镣铐，严格限定在“显而易见的语法或类型错误”范畴内。

- **Q: 如此高频的自动化处理，会不会导致Token成本直接原地爆炸？**
  - A: 确实存在这种风险，因此必须在流水线脚本层面进行极致的工程化优化。例如，仅精准提取失败文件的核心代码片段及直接关联的错误日志喂给AI，将上下文窗口 (Context Window) 压缩到极限。此外，在API控制台设置严密的每日预算上限 (Budget Limit)，是不可或缺的最后一道防线。

## 🧬 提示词解剖 (Why it works?)

1. **赋予资深工程师人设 (Role)：** 通过反复强调“极度警惕不必要的变更”这一核心性格特征，从思想根源上彻底压制了AI天生渴望“过度重构 (Over-engineering)”的危险本能。
2. **明确的逃生通道 (Constraints)：** 当面临AI难以独立决断的复杂逻辑错误时，并未强行逼迫它“必须给出一个答案”，而是巧妙设置了清晰的退出机制（即输出 `"MANUAL_INTERVENTION_REQUIRED"`），将代码的最终控制权安全、平稳地交还给人类开发者。
3. **严格的JSON输出格式 (Format)：** 采用铁腕手段强制要求以标准JSON格式输出，这为CI流水线的后续自动化操作（如精准解析数据、无缝覆盖原文件并丝滑执行 `git commit`）铺平了道路。

## 📊 效果验证：Before & After

### ❌ Before (输入)

// src/utils/calculator.ts
export function addPrice(a: number, b: string) {
  return a + b;
}

```bash

TS2365: Operator '+' cannot be applied to types 'number' and 'string'.

### ✅ After (结果)

  "fixed_content": "export function addPrice(a: number, b: number) {\n  return a + b;\n}"
}
```

_(AI敏锐地捕捉到了类型不匹配的致命报错，果断将参数 `b` 的类型精准修正为 `number`。随后，CI流水线如丝般顺滑地自动将这段代码写入目标文件并光速完成 Commit。)_
