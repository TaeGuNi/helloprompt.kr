# 📝 원인을 알 수 없는 버그, AI 탐정에게 맡기세요

- **🎯 추천 대상:** 주니어 개발자, 원인 불명의 버그로 밤새우는 실무자
- **⏱️ 소요 시간:** 1시간 이상 끙끙대던 버그 → 3분 컷
- **🤖 추천 모델:** Claude 3.5 Sonnet (코드 분석 특화), GPT-4o

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"어제 퇴근할 때까지만 해도 분명 잘 돌아갔는데... 에러 로그조차 없이 앱이 죽어버린다고?"_

![\"원인을 알 수 없는 버그, AI 탐정에게 맡기세요\"](/images/hooks/bug-hunter-debugging-guide.jpg)

개발자를 가장 좌절하게 만드는 것은 '재현 불가능한 버그'나 '에러 메시지조차 없는 침묵의 버그'입니다. 스택 오버플로우를 샅샅이 뒤져봐도 내 상황에 정확히 들어맞는 답변은 없고, 사수에게 질문하자니 방대한 전체 컨텍스트를 설명할 엄두조차 나지 않죠.
이럴 때는 AI에게 당신의 '코드 조각'과 '상황'을 던져보세요. 당신이 놓친 사소한 오타부터 프레임워크의 숨겨진 라이프사이클 이슈까지, AI 탐정이 코드를 정밀하게 해부하여 근본적인 원인을 짚어냅니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **복잡한 컨텍스트 압축:** 장황한 설명 대신, 에러 로그와 코드 스니펫만으로 핵심 상황을 전달합니다.
2. **가설 기반 접근:** 단편적인 답변이 아닌, 발생 가능한 3가지 가설을 통해 디버깅의 시야를 획기적으로 넓혀줍니다.
3. **해결책 및 방어 코드 제안:** 단순한 버그 수정을 넘어, 재발을 완벽히 차단하는 안전한 코드(Defensive Coding) 패턴까지 확보할 수 있습니다.

## 🚀 해결책: "AI 셜록 홈즈 디버거"

### 🥉 Basic Version (기본형)

빠르게 문제의 원인만 짚어내고 싶을 때 사용하세요.

> **역할:** 너는 15년 차 시니어 백엔드/프론트엔드 개발자야.
> **요청:** 아래 코드와 에러 로그를 보고 무엇이 문제인지, 어떻게 고쳐야 하는지 딱 2줄로 핵심만 짚어줘.
> **코드:** `[여기에 코드 붙여넣기]`
> **에러 로그:** `[여기에 에러 로그 붙여넣기]`

### 🥇 Pro Version (전문가형)

시스템 구조가 복잡하게 얽혀있거나, 로직 자체가 꼬여 있어 딥 다이브(Deep Dive) 분석이 필요할 때 사용하세요.

> **역할 (Role):** 너는 디버깅의 천재이자 시스템 아키텍처와 메모리 누수까지 꿰뚫어 보는 '시니어 소프트웨어 엔지니어'야.
>
> **상황 (Context):**
>
> - 프레임워크/언어: `[예: React 18, Node.js v20, Python 3.11]`
> - 문제 상황: `[예: 결제 버튼을 두 번 누르면 DB에 중복 저장됨]`
> - 내가 시도해 본 것: `[예: 버튼에 disabled 처리를 해봤으나 여전히 뚫림]`
>
> **요청 (Task):**
>
> 1. 제공된 **[코드 스니펫]**과 **[에러 로그]**를 분석하여, 이 버그의 **근본 원인(Root Cause)을 3가지 가설**로 나누어 제시해 줘. (확률이 가장 높은 순서대로)
> 2. 가장 유력한 가설을 바탕으로 수정한 **클린 코드(Clean Code)**를 작성해 줘.
> 3. 단순 수정을 넘어, 앞으로 동일한 버그가 발생하지 않도록 **방어적 프로그래밍(Defensive Programming) 관점의 조언**을 1줄 추가해 줘.
>
> **데이터 입력 (Input):**
>
> - **[코드 스니펫]:**
>   `[문제가 발생한 함수나 컴포넌트 코드 복사/붙여넣기]`
> - **[에러 로그]:**
>   `[터미널이나 브라우저 콘솔의 에러 메시지 텍스트]`
>
> **제약사항 (Constraints):**
>
> - "아마도 이럴 것입니다" 식의 모호한 추측을 엄격히 배제하고, 문제가 되는 코드 라인을 정확히 지목해서 설명해.
> - 보안상 민감한 정보(API 키, 비밀번호 등)가 코드에 포함되어 있다면 즉시 경고하고 마스킹 처리해.
> - 결과는 마크다운 포맷으로 가독성 있게 정리해 줘.

## 💡 작성자 코멘트 (Insight)

이 프롬프트의 핵심은 **"내가 시도해 본 것"**을 명시하는 부분입니다. AI에게 이 정보를 제공하지 않으면, 당신이 이미 1시간 전에 시도하고 실패했던 뻔한 해결책(예: "캐시를 삭제해 보세요", "서버를 재시작해 보세요")을 1순위로 제시할 확률이 매우 높습니다.

또한, 프론트엔드라면 브라우저 종류(Chrome, Safari 등)를, 백엔드라면 DB 종류나 ORM 버전을 `[상황]`에 추가해 주면 AI의 추론 정확도가 비약적으로 상승합니다. 보안이 엄격한 환경이라면, 코드를 붙여넣기 전에 사내 보안 규정에 위배될 수 있는 민감한 비즈니스 로직이나 DB 스키마 이름을 임의의 변수명(A, B, C 등)으로 치환한 후 프롬프트를 실행하는 것을 절대 잊지 마세요.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 코드가 너무 길어서 한 번에 다 붙여넣을 수가 없어요. 어떡하죠?**
  - A: 전체 파일을 통째로 넣으면 AI가 '할루시네이션(환각)'을 일으킬 위험이 커집니다. 에러가 발생한 지점을 중심으로 앞뒤 50줄 정도의 핵심 함수나 클래스만 발췌해서 넣는 것이 가장 정확한 결과를 얻는 비결입니다.

- **Q: 에러 로그도 남지 않고 시스템이 그냥 멈추는 버그(Deadlock 등)도 잡을 수 있나요?**
  - A: 네, 가능합니다! 이럴 때는 `[에러 로그]` 란을 비워두고, `[문제 상황]`에 "특정 API 호출 시 CPU 점유율이 100%로 치솟으며 서버가 다운된다"처럼 현상을 최대한 구체적으로 묘사하세요. AI가 무한 루프나 데드락 발생 가능성이 높은 코드 라인을 예리하게 짚어줍니다.

- **Q: 회사 코드인데 AI에게 맡겨도 안전할까요?**
  - A: 퍼블릭 AI(무료 ChatGPT 등)에 기밀 코드를 그대로 입력하는 것은 매우 위험합니다. 반드시 API를 통한 호출 방식을 사용하거나, 데이터 학습 거부(Opt-out) 설정이 완비된 엔터프라이즈 환경에서만 사용하시길 강력히 권장합니다.

## 🧬 프롬프트 해부 (Why it works?)

1.  **가설 생성 유도 (Hypothesis Generation):** 버그의 원인은 결코 하나가 아닐 수 있습니다. "3가지 가설을 제시해라"라는 명령은 AI가 단순한 문법 오류를 넘어 비동기 타이밍 이슈나 의존성 충돌 같은 복합적인 시스템 문제까지 깊이 있게 고민하도록 강제합니다. (Chain-of-Thought 기법 적용)
2.  **안전망 구축 (Defensive Coding):** 단순히 "고쳐줘"라고 요구하는 대신 방어 코드 작성을 지시함으로써, 임시방편(Hotfix)이 아닌 근본적인 코드 품질 향상을 이끌어냅니다.

## 📊 증명: Before & After

### ❌ Before (상황)

**에러:** `Uncaught TypeError: Cannot read properties of undefined (reading 'map')`
**상황:** React 컴포넌트에서 API 데이터를 받아와 목록을 렌더링하는데, 새로고침할 때마다 간헐적으로 하얀 화면(White Screen of Death)이 나타납니다.

### ✅ After (결과)

**🔍 근본 원인 분석 (3가지 가설):**

1. **유력:** 비동기 데이터 로딩 지연. `users` 데이터가 API로부터 도착하기 전(초기 렌더링 시점)에 값이 `undefined` 상태에서 `.map()`을 호출하여 앱에 크래시가 발생했습니다.
2. **가능성 있음:** API 응답 스키마 변경. 백엔드에서 반환하는 객체의 키워드가 `users`에서 다른 이름(예: `userList`)으로 변경되었을 수 있습니다.
3. **희박:** 상태 관리 초기화 오류. `useState`의 초기값이 빈 배열 `[]`이 아닌 `null`이나 `undefined`로 잘못 설정되었을 수 있습니다.

**✨ 수정된 클린 코드:**

```javascript
// 옵셔널 체이닝(?.)과 널 병합 연산자(??)를 사용한 방어적 코드
function UserList({ data }) {
  // data.users가 없거나 배열이 아닐 경우 빈 배열로 폴백
  const users = data?.users ?? [];

  if (users.length === 0) {
    return <p className="text-gray-500">조회된 사용자 데이터가 없습니다.</p>;
  }

  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

**🛡️ 방어적 프로그래밍 조언:**
데이터 렌더링 시에는 항상 옵셔널 체이닝(`?.`) 사용을 습관화하고, 배열 메서드를 활용할 때는 초기 상태값(Initial State)이 빈 배열(`[]`)로 보장되도록 설계하는 것이 안전합니다.
