# 🍝 Долой запутанный код! AI-анализатор сложности (Code Complexity Analyzer)

- **🎯 Целевая аудитория:** Junior-разработчики, специалисты по поддержке унаследованного кода (legacy), ревьюеры кода
- **⏱️ Экономия времени:** с 1 часа → до 3 минут
- **🤖 Рекомендуемые модели:** GPT-4o, Claude 3.5 Sonnet (рекомендуются модели с сильными способностями к анализу кода)

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐⭐

> _"Вложенные `if` внутри циклов `for`, внутри которых еще одни `if`... Код работает, но кажется бомбой замедленного действия, которая взорвется при любой правке. Знакомо это чувство отчаяния?"_

В процессе разработки мы неизбежно сталкиваемся с кодом высокой сложности — так называемым «спагетти-кодом». С добавлением новых функций и исправлением багов код разрастается, а его цикломатическая сложность взлетает до небес. Такой код не только трудно читать, но и опасно менять: непредсказуемые побочные эффекты превращают поддержку в настоящий кошмар.

Но что делать, если вы не знаете, с какой стороны подступиться к рефакторингу? Поручите анализ сложности кода искусственному интеллекту! AI поставит объективный диагноз, выявит проблемные участки и предложит конкретную стратегию рефакторинга, основанную на паттернах проектирования и принципах чистого кода (Clean Code).

## ⚡️ Краткое содержание (TL;DR)

1. AI структурно анализирует громоздкие функции и трудночитаемые вложенные условия.
2. Вы получаете практические рекомендации по рефакторингу: ранний возврат (Early Return), извлечение методов (Extract Method) и др.
3. В результате у вас на руках чистый код с максимальной читаемостью и удобством поддержки — без изменения бизнес-логики.

## 💡 Комментарий автора (Insight)

Это мой самый любимый промпт, когда нужно провести ревью или исправить чужой (или свой собственный старый) легаси-код. AI не просто бездумно сокращает строчки, а дает логическое обоснование: «Это условие имеет 4 уровня вложенности (Depth), поэтому его лучше вынести через Guard Clauses».

Промпт особенно полезен для самопроверки перед код-ревью. Если вы прогоните свой код через этот фильтр перед тем, как открыть Pull Request (PR), вероятность получить от коллег восхищенное «Вау, какой чистый код!» возрастает многократно.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Можно ли отправить на анализ весь файл целиком?**
  - Ответ: Не рекомендую из-за ограничений контекстного окна LLM и потери фокуса. Вместо файлов на сотни строк лучше разбивать код на логические блоки по 50–150 строк («конкретная функция» или «один класс»). Так вы получите гораздо более точные и глубокие результаты рефакторинга.

- **Вопрос: Безопасно ли копировать отрефакторенный AI код прямо в продакшен?**
  - Ответ: Ни в коем случае не копируйте его вслепую! AI иногда упускает тонкие пограничные случаи (edge cases) бизнес-логики и может допустить галлюцинацию, изменив саму суть работы кода. **Обязательно убедитесь, что новый код проходит все существующие юнит-тесты (Unit Tests)**, прежде чем коммитить изменения.

- **Вопрос: Адаптируется ли код под конкретный язык программирования или фреймворк?**
  - Ответ: Да! Если в переменной `[Язык/Фреймворк]` продвинутой (Pro) версии вы укажете, например, `Java/Spring Boot` или `Python/FastAPI`, AI улучшит код с учетом стандартов именования и паттернов проектирования, принятых именно в этой экосистеме.

## 🧬 Разбор промпта (Why it works?)

1. **Фокус на метриках:** Мы не просим AI «сделать красиво». Требование проанализировать «цикломатическую сложность» заставляет модель сосредоточиться на структурных недостатках ПО, применяя строгий инженерный подход.
2. **Жесткие ограничения (Constraints):** Явно прописав главное правило рефакторинга — «неизменность внешнего поведения», — мы минимизируем риск того, что AI самовольно изменит или удалит важную логику.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Ввод: Спагетти-код с глубокой вложенностью)

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ После (Результат: Чистый код с использованием Guard Clauses и методов массивов)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: Ранний возврат при исключениях (уменьшает вложенность)
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. Функции высшего порядка: Упрощение циклов
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. Выделение и упрощение бизнес-логики
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```
