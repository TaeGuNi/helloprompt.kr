### ❌ Before (我们曾经历的痛苦)

_“帮我写一个 Python、Django、PostgreSQL 的 Docker 文件。”_

如果像这样下达片面的指令，AI 会不加批判地使用体积超过 1GB 的低效基础镜像。而且，它会让容器以最高管理员 `root` 权限运行，生成这种将致命安全漏洞暴露无遗的危险代码。多阶段构建更是压根儿都不会去尝试。

### ✅ After (完美蜕变的结果)

![Docker 环境搭建完成](/images/hooks/docker-environment-setup.jpg)

```dockerfile
# 1. Build Stage
FROM python:3.11-slim as builder

WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 2. Production Stage
FROM python:3.11-slim

WORKDIR /app
# 仅复制在构建阶段安装的包 (大幅减少体积)
COPY --from=builder /root/.local /root/.local
COPY . .

# 环境变量设置
ENV PATH=/root/.local/bin:$PATH
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# 为了安全，创建非 root 用户并授予权限
RUN adduser --disabled-password --gecos '' myuser && \
    chown -R myuser:myuser /app
USER myuser

CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

我们可以立即确认，编译器和沉重的构建工具已被干净利落地移除，镜像瘦身至 100MB 级别，应用了防止黑客威胁的完美用户权限分离，甚至连抑制生成不必要字节码缓存的选项 (`PYTHONDONTWRITEBYTECODE`) 等实务部署必备要件都已完美落实。

## 💡 作者点评 (Insight & How to use)

此提示词真正的杀伤力和价值在于对 **“多阶段构建 (Multi-stage build)”** 模式的强制应用以及 **“非 root 用户 (Non-root user)”** 这两个核心约束条件的控制。

通常情况下，如果不加任何限制地简单指示 ChatGPT 或 Claude 等 AI “帮我的项目写个 Docker 文件”，结果会怎样呢？AI 为了输出最常见的模式，通常会不加批判地使用包含大量项目完全不需要的编译器和系统级依赖包的、数百兆大小的沉重基础镜像。甚至往往会给出让容器在安全致命的 **`root` 权限** 开启状态下运行的代码。这样生成的镜像不仅会拖慢构建速度，而且一旦部署到生产服务器，就会像一颗随时可能爆炸的炸弹，成为黑客轻易猎取的食物。

相反，如果利用上面提供的 **专家版 (Pro Version) 提示词**，情况将发生 180 度转弯。此提示词明确指示 AI 将负责安装沉重依赖包的 **构建环境 (Build Stage)** 与仅保留运行应用所需的最小文件和二进制文件的 **运行时环境 (Production Stage)** 完美分离。结果，它能实现魔术般的优化，将原本像滚雪球一样膨胀的 Docker **镜像体积压缩至原来的十分之一左右**。

除此之外，在最重要的安全方面也天衣无缝。得益于插入到 **约束条件 (Constraints)** 块中的严苛规则，为了防止容器内部的应用进程夺取宿主机的控制权，它会自动应用创建新 **普通用户 (Non-root user)** 账号并进行权限分离的设置。这种能够从源头上封锁容器逃逸攻击 (Container Breakout) 的坚固设置，其可靠性高到即使不经任何修改直接反映到你的实务 CI/CD 流水中也完全没问题。

在实际工作中可以如何改写并使用此提示词呢？`[技术栈]` 变量块是一个可以根据你的口味进行无限自定义的魔法画布。除了简单更换语言和框架，如果有特殊的基础设施需求，请积极地在这个变量区域进行添加。例如，你可以丢出复杂的架构需求变量，如 **“在前段挂载 Nginx 反向代理，并联动通过 Let's Encrypt 自动更新 HTTPS SSL 证书的机器人”**，或者 **“将 Kafka 消息队列和 Elasticsearch 日志采集器统一整合到 `docker-compose.yml` 的网络中”**。AI 会令人惊讶地配置好内部桥接网络 (Bridge Network) 以确保所有这些容器能顺畅通信，并精准调优依赖项执行顺序 (`depends_on`)，从而完整地打造出一个完美的系统。

此外，通过此提示词生成的 `docker-compose.yml` 文件将成为你团队 **消除本地开发与生产部署之间隔阂的绝佳教材**。你将学会在开发模式下通过挂载 `volumes` 将实时代码更改 (Hot-Reloading) 立即反映到容器内部，而在生产部署时则卸载挂载并仅提供静态构建的文件。如果发生特定包无法安装的问题，只需在提示词的角色 (Role) 区域下方添加一行要求：“请包含分析并解决安装时发生的依赖项错误日志的 RUN 指令分支处理”，即可获得一个具备完美自愈 (Self-healing) 能力的基础设施脚本。这正是超越简单的指令复制粘贴，通过提示词工程掌握完美控制权的真正专家之路。

## 🙋 常见问题 (FAQ)

- **Q：我正在使用仅限公司内部访问的私有包仓库 (Private Registry)，这也能反映在 Docker 构建过程中吗？**
  - A：当然可以！请在提示词的 `[技术栈]` 变量区域添加特殊的约束条件，例如 _“需要通过公司内部的 Nexus Repository 下载 npm 包，且需要身份验证令牌”_。AI 会天衣无缝地编写出 `Dockerfile` 内部安全的认证流程、环境变量注入逻辑，以及为了防止构建后秘密 (Secret) 信息留在镜像中而进行的删除安全处理。
