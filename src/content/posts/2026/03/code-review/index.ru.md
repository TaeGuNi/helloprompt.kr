## ⚡️ Краткая выжимка (TL;DR)
1. Не просите ИИ просто "исправить", назначьте ему роль "Senior код-ревьюера".
2. Четко предоставьте контекст: логи ошибок и первоначальную задумку вашего кода.
3. Не копируйте код слепо, изучайте причины и аргументы, которые приводит ИИ для каждого изменения.

## 🚀 Решение: Промпт "Безжалостный Senior-разработчик"
### 🥉 Базовая версия (Basic Version)
Используйте, когда нужно быстро найти причину ошибки.
> **Роль:** Ты — Senior `[Frontend/Backend]` разработчик с 10-летним опытом.
> **Задача:** Найди причину `[описание ошибки]`, возникающей в коде ниже, и предоставь исправленный код.
> **Код:** `[Вставьте сюда ваш код]`
### 🥇 Продвинутая версия (Pro Version)
Используйте, когда требуется не просто исправление бага, а улучшение архитектуры и рефакторинг по принципам чистого кода.
> **Роль (Role):**
> Ты — безжалостный, но невероятно компетентный Staff Software Engineer (Ведущий инженер) с 15-летним опытом работы в FAANG. Ты абсолютно бескомпромиссен в вопросах читаемости, производительности и поддерживаемости кода.
> 
> **Контекст (Context):**
> - Окружение: `[Укажите ваш стек технологий, например, React / Node.js / Python]`
> - Цель: `[Опишите бизнес-логику, которую должен выполнять этот код]`
> - Текущая проблема: `[Полный лог ошибки, падение производительности или другая проблемная ситуация]`
> 
> **Задача (Task):**
> 1. Проанализируй предоставленный код и укажи на фундаментальные проблемы (причины багов, антипаттерны и т.д.).
> 2. Проведи рефакторинг кода, основываясь на принципах чистого кода (SOLID, DRY и т.д.).
> 3. Предоставь полный исправленный код.
> 4. В формате комментариев к код-ревью объясни, **почему** ты сделал эти изменения, выделив 3 главных пункта.
> 
> **Ограничения (Constraints):**
> - Пропусти лишние вступления и приветствия, сразу переходи к ревью.
> - Используй современный синтаксис (ES6+ и т.д.) и идиоматические паттерны выбранного языка.
> - Обязательно учти обработку ошибок и пограничные случаи (edge cases).
> - Оформи вывод в виде списка.
> 
> **Предупреждение (Warning):**
> - Не выдумывай несуществующие библиотеки или методы (никаких галлюцинаций).
> 
> **Код (Code):**

## 💡 Комментарий автора (Insight)
Суть этого промпта заключается в персоне "безжалостного ведущего инженера" и требовании "объяснить причины изменений". 
Самая частая ошибка джуниоров на практике — слепое копирование (Ctrl+C, Ctrl+V) кода, сгенерированного ИИ, без понимания причин. Да, баг будет исправлен прямо сейчас, но ваши навыки не вырастут ни на миллиметр. 
Этот промпт говорит ИИ не "исправь код", а "научи меня". Обращайте внимание на указанные антипаттерны и визуально запоминайте идиоматические конструкции (Idiomatic patterns). Если вы подробно опишете в блоке `Context` вашу изначальную задумку (бизнес-логику), ИИ не только исправит синтаксические ошибки, но и укажет на структурные недостатки. Не ищите оправданий, позвольте вашему ИИ-наставнику "разнести" ваш код и учитесь на этом. Это самый быстрый путь к профессиональному росту.

## 🙋 Часто задаваемые вопросы (FAQ)
- **В: Безопасно ли отправлять внутренний код компании как есть?**
  - О: Абсолютно нет! Ключи API, внутренние IP-адреса и конфиденциальную бизнес-логику необходимо обязательно замаскировать фиктивными (Dummy) данными или обобщить имена переменных перед тем, как вставлять их в промпт. Безопасность не терпит компромиссов.
- **В: Иногда код от ИИ не запускается. Что делать?**
  - О: Это проявление галлюцинаций (Hallucination). Не паникуйте, просто скопируйте лог возникшей ошибки и задайте вопрос снова. Если вы предъявите претензию: "Твой код выдает вот такую ошибку", ИИ сам выдаст исправленную версию.
- **В: Какая ИИ-модель лучше всего подходит для код-ревью?**
  - О: По состоянию на 2026 год, в сфере программирования модели уровня Claude 3.5 Sonnet и выше демонстрируют подавляющее преимущество в понимании контекста и способностях к рефакторингу. ChatGPT Plus (GPT-4o) также является отличным выбором.

## 🧬 Разбор промпта (Why it works?)
1. **Задание безжалостной персоны:** Обычный ИИ старается не обидеть пользователя, поэтому часто игнорирует критические структурные недостатки, лишь бы код хоть как-то заработал. Назначение роли строгого ведущего инженера принуждает к бескомпромиссному ревью.
2. **Явное указание контекста (Context):** Компьютер читает код, а Senior читает "намерения". Если вы объясните, что код должен делать, ИИ сможет предложить лучший подход (другие структуры данных, более подходящие паттерны проектирования).
3. **Требование причин (Why):** Это ключевой механизм для вашего роста. Он не дает ИИ просто выдать готовый код и заставляет передавать знания.

## 📊 Доказательство: До и После (Before & After)
### ❌ До (Ввод)
```javascript
// Код для получения и обработки данных пользователя (Пример спагетти-кода)
async function getU(id) {
  let res = await fetch('/api/user/' + id);
  let d = await res.json();
  if(d) {
    if(d.status === 'active') {
      let r = await fetch('/api/roles/' + d.roleId);
      let rd = await r.json();
      return { n: d.name, r: rd.name };
    } else {
      return null;
    }
  }
}
```
### ✅ После (Результат)
```javascript
// Результат рефакторинга от ИИ Senior-инженера
async function getActiveUserRoleSummary(userId) {
  try {
    const userResponse = await fetch(`/api/user/${userId}`);
    if (!userResponse.ok) throw new Error('Failed to fetch user');
    
    const user = await userResponse.json();
    
    // Применение паттерна раннего возврата (Early Return) избавляет от вложенных if
    if (!user || user.status !== 'active') {
      return null; 
    }
    const roleResponse = await fetch(`/api/roles/${user.roleId}`);
    if (!roleResponse.ok) throw new Error('Failed to fetch role');
    
    const role = await roleResponse.json();
    // Использование понятных имен переменных (n, r -> name, roleName)
    return { 
      name: user.name, 
      roleName: role.name 
    };
  } catch (error) {
    console.error(`[getActiveUserRoleSummary] Error for userId ${userId}:`, error);
    throw error; // Предотвращение скрытия ошибок
  }
}
```
*(Комментарий ИИ: Вложенные условные операторы заменены на Early Return для повышения читаемости, имена переменных сделаны интуитивно понятными, а также добавлена отсутствовавшая обработка ошибок.)*
