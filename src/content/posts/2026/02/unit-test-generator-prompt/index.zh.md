# 📝 编写测试代码，5 分钟搞定的提示词

- **🎯 推荐对象:** 开发者、QA 工程师、所有渴望提升代码质量的职场人
- **⏱️ 所需时间:** 30 分钟 → 缩短至 5 分钟
- **🤖 推荐模型:** 所有对话型 AI (ChatGPT, Claude, Gemini 等)

- ⭐ **难度:** ⭐⭐☆☆☆
- ⚡️ **有效性:** ⭐⭐⭐⭐⭐

> _“业务代码都写不完了，哪还有时间去补那些枯燥的测试覆盖率？”_

所有人都知道单元测试（Unit Test）至关重要，但在实际开发中，编写测试用例往往比实现核心逻辑还要耗时。你不仅要绞尽脑汁地覆盖各种极端边界情况（Edge Cases），还要费时费力地处理错综复杂的模拟对象（Mocking）……光是想想就让人头疼，对吧？

现在，只需复制粘贴你的业务代码，把那些繁琐、机械的测试用例编写工作，统统交给 AI 来搞定吧！

---

## ⚡️ 3 行总结 (TL;DR)

1. **全面覆盖：** 不仅能走通正常的快乐路径（Happy Path），更能严密防范各种隐蔽的边界异常。
2. **完美兼容：** 无缝适配 Jest、PyTest、JUnit 等所有主流测试框架的语法特性。
3. **智能 Mock：** 自动识别代码中的外部依赖，精准生成隔离逻辑与 Mock 数据。

## 💡 作者评论 (Insight)

在节奏飞快的敏捷开发中，测试驱动开发（TDD）往往因为排期紧迫而沦为一句空话。这套提示词的真正杀手锏，并不在于盲目追求“100% 覆盖率”的虚荣指标，而在于**它能瞬间帮你击碎编写测试代码时的“冷启动”壁垒**。

## 🙋 常见问题 (FAQ)

- **Q: AI 生成的 Mock 代码总是报错“找不到路径”，该怎么解决？**
  - A: 当外部依赖过于复杂时，AI 很难凭空猜对你的项目目录结构。建议你在 `[请在此处粘贴你需要测试的业务代码]` 的下方，顺手贴上一两个项目中已有的正确 Mock 示例片段，或者直接告诉它依赖项的绝对路径。只要给足上下文，AI 的一次通过率将会产生质的飞跃。

- **Q: 这套提示词能直接用来写集成测试（Integration Test）或端到端测试（E2E Test）吗？**
  - A: 不能完全照搬。这套提示词是专门针对底层的“单元测试”进行深度优化的。如果你要写 E2E 测试（比如 Playwright 或 Cypress），你需要额外喂给 AI 真实的 DOM 结构（HTML 代码）以及详尽的用户交互行为链路。对于这类复杂场景，强烈建议将其拆分成更细颗粒度的任务步骤再交给 AI 处理。

## 🧬 提示词解剖 (Why it works?)

1. **确立专家角色 (Role):** 为 AI 注入“追求完美的 QA 工程师”这一硬核人设，迫使它跳出“代码能跑就行”的思维定势，转而死磕测试的严密性与边界覆盖率。
2. **强制罗列清单 (Task):** 要求 AI 在提笔写代码前，必须先输出一份“测试用例清单”。这是非常经典的思维链（Chain of Thought）技巧，强迫 AI 先在脑海中理清逻辑分支，再动手编写断言，从而大幅降低场景遗漏的概率。
3. **明确 Mock 指令 (Constraints):** 现代应用开发中最令人头疼的痛点，莫过于处理盘根错节的外部依赖。通过强制命令 AI 处理 Mock，可以直接斩断它试图去连接真实数据库或发起真实网络请求的荒谬行径，确保生成出纯粹、有效的隔离测试。

## 📊 证明: Before & After

### ❌ Before (输入的业务代码)

```javascript
// 一个简单的用户余额扣除逻辑
function deductBalance(user, amount) {
  if (amount <= 0) throw new Error("扣除金额必须大于 0");
  if (user.balance < amount) throw new Error("余额不足");
  user.balance -= amount;
  return user.balance;
}
```

### ✅ After (AI 生成的测试代码)

```javascript
describe("deductBalance 函数测试", () => {
  let mockUser;

  beforeEach(() => {
    // 每次测试前初始化干净的用户状态
    mockUser = { id: 1, balance: 100 };
  });

  // 1. 成功场景测试 (Happy Path)
  test("正常扣除金额时，应返回正确的剩余余额并更新用户对象", () => {
    const newBalance = deductBalance(mockUser, 30);
    expect(newBalance).toBe(70);
    expect(mockUser.balance).toBe(70);
  });

  // 2. 异常场景测试 - 边界值分析
  test("扣除金额为 0 时，应抛出异常", () => {
    expect(() => deductBalance(mockUser, 0)).toThrow("扣除金额必须大于 0");
  });

  test("扣除金额为负数时，应抛出异常", () => {
    expect(() => deductBalance(mockUser, -10)).toThrow("扣除金额必须大于 0");
  });

  // 3. 异常场景测试 - 业务规则限制
  test("扣除金额大于用户可用余额时，应抛出余额不足异常", () => {
    expect(() => deductBalance(mockUser, 150)).toThrow("余额不足");
  });
});
```
