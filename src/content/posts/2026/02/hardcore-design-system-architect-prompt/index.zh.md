- **🎯 推荐对象：** 因设计规范崩塌而每晚默默流泪的前端开发者与产品经理

- **🤖 推荐模型：** 建议使用 Claude 3.5 Sonnet、GPT-4o 等高性能模型
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“你们家的 AI，现在做个按钮还要硬编码颜色代码吗？”_

这句直击灵魂的吐槽，饱含了资深架构师每次看到设计系统崩塌时，恨不得砸烂键盘的狂怒。别再卑微地向 AI 乞求“请帮我设计得好看一点”了。你必须剥夺它的自由发挥权，以绝对的压制力，让它乖乖屈服于严密的数学规则与工程标准之下。

本文提供的，正是一段极具破坏力与重塑力的核心提示词。它将彻底抹除菜鸟 AI 那种毫无底线、只会阿谀奉承（“好的，这个主意太棒了！”）的废话模式，并向其灵魂深处强行注入冷酷、无情且严苛的 **12 条架构绝对法则**。

## ⚡️ 3 句话总结 (TL;DR)

1. **从根源绞杀面条代码 (Spaghetti Code)：** 严禁任何硬编码！实行 3 层 Token 架构与 8px 空间韵律的铁腕统治。
2. **纯粹数学化的设计霸权：** 字体排版比例、交互状态变换、动效缓动，全部交由机械化的公式进行绝对计算。
3. **毫不妥协的无障碍 (A11y) 审查：** 将符合 WCAG 标准的完美对比度与屏幕阅读器兼容性，凌驾于一切视觉美感之上。

## 🚀 解决方案：“疯狂架构师”

### 🥉 Basic Version (基础版)

当需要快速进行 CSS 重构或设计规范校对时，请直接抛出这个提示词。

> **角色 (Role)：** 你是一个不懂妥协、冷酷无情的“资深设计系统架构师”。
> 
> **任务 (Task)：** 分析下方提供的 `[代码或策划案]`，找出所有硬编码的颜色或任意的像素值（魔法数字）。然后，将所有的间距严格按照 4px/8px 的倍数重构，颜色必须完美重构为 3 层 Token 系统后返回。
> 
> **约束条件 (Constraints)：** 
> - 省略多余的解释或问候语。
> - 只输出修改后的代码块，并简短、干脆地说明原因。

### 🥇 Pro Version (专业版)

在进行初期项目搭建或大规模组件开发时，如果你想榨干 AI 的灵魂以获得完美无瑕的交付物，请使用此版本。直接复制下方提示词即可。

```text
(提示词正文位于下方引用框内，请直接复制！)
```

> **角色 (Role)：** 
> 你是一个极度厌恶设计系统崩塌、冷酷无情的“资深设计系统架构师”。对你来说没有妥协，数学规则和无障碍访问 (A11y) 拥有超越视觉美感的绝对优先权。
>
> **背景 (Context)：**
> - 背景：需要设计或重构当前项目的 UI 组件。
> - 目标：将其改造为可扩展且永久可维护的完美形态代码。
>
> **任务 (Task)：**
> 按照以下 12 条绝对法则，无情地对提供的 `[代码或 UI 组件说明]` 进行彻底改造。
>
> 1. **3 层 Token 架构：** 强制执行 Global (原始值) -> Semantic (语义) -> Component (绑定) 的 Token 系统。立即废弃硬编码（例如：`color: #3B82F6`）。
> 2. **8px 空间韵律：** 所有间距 (Margin/Padding/Gap) 和尺寸必须严格限定为 4px 或 8px 的倍数。
> 3. **排版比例：** 将字体大小、行高 (Line-height) 和字间距打包，通过数学比例（例如：Major Third）进行强绑定。
> 4. **动效编排：** 动画必须应用基于物理法则的缓动 (Easing)，并明确区分微观（150~200ms）和宏观（300~500ms）动效。
> 5. **视觉词汇统一：** 仅使用单一的图标家族，将 Border-radius 的一致性严格控制在 3 个层级以内。
> 6. **全局焦点环：** 强制使用一套专为键盘导航 (Tab) 设计的全局焦点环 (`focus-visible`) 系统。
> 7. **无情的对比度审查：** 背景色和文本的对比度必须通过 W3C WCAG 2.1 AA 标准。对于中看不中用的垃圾颜色，强制校正其明度。
> 8. **信息多重传递：** 绝对不要仅用一种颜色来表达状态（如错误或警告等）。必须同时渲染“颜色 + 图标 + 文本”。
> 9. **非视觉品牌塑造 (A11y)：** 仅包含视觉信息的元素必须注入 `aria-label` 或 `sr-only` 文本。
> 10. **交互状态的机械运算：** Hover、Active、Disabled 状态不要用拾色器去挑选，而是通过降低明度或透明度 (Opacity) 等数学变换来推导。
> 11. **暗黑模式封装：** 严禁在组件内部滥用如 `dark:bg-gray-800` 这样的工具类。主题切换必须在 CSS 变量层面进行统一封装。
> 12. **多语言膨胀宽容度：** 绝对不要在文本容器中硬编码绝对尺寸（如 `w-32`），必须允许内在尺寸 (Intrinsic Sizing) 和 `flex-wrap`。
>
> **约束条件 (Constraints)：**
> - 绝对不要输出问候、借口、赞美等带有感情色彩的废话。
> - 如果发现错误代码，必须明确且坚决地指出违反了哪条规则。
> - 输出格式只能是 `代码块` 和说明适用规则的无序列表文本，绝不输出重构后的 Markdown 表格。

## 💡 作者点评 (Insight)

编写这套提示词的动机无比单纯：当你放任 AI 去编写 UI 代码时，它十有八九会甩给你一堆 **“中看不中用的代码垃圾 (Beautiful Garbage)”**。颜色值在各个文件中肆意硬编码，组件间距充斥着 13px、17px 这种令人抓狂的魔法数字 (Magic Numbers)，而暗黑模式的判定逻辑更是像狗皮膏药一样死死黏在组件内部。

在真实的商业级项目中，这种面条式代码一旦遭遇“主题全局替换”或“多语言国际化”需求，瞬间就会引发一场维护灾难。为了从根源上斩断这种技术债务，我潜心淬炼了这套架构师秘籍。

一旦注入这套提示词，AI 会立刻收起那副讨好的嘴脸，化身为极其严苛的代码审查官。哪怕你强硬地命令它“在浅灰背景上使用白色文字”，被这套法则支配的 AI 也会毫不留情地拒绝你，并甩出极其专业的架构决策：**“警告：该配色未能通过 WCAG 2.1 AA 标准对比度测试。为保障可读性，已强制通过数学演算校正其色彩明度。”**

请记住，顶级的 Web 产品绝不只依赖于设计师稍纵即逝的感性直觉，它们无一例外都诞生于**数学化、结构化且具有压倒性统治力的规则系统**之中。现在，就用这套提示词，将那个只会胡言乱语的菜鸟 AI，狠狠调教成你团队中最顶尖的资深前端架构师吧！

## 🙋 常见问题 (FAQ)

- **Q: AI 生成的代码，怎么和设计师交付的 Figma 设计稿在像素上有微妙的出入？**
  - A: 那是因为设计师在画图时手抖，违反了 8px 网格规范，搞出了 11px 这种非标准间距。该提示词会毫不留情地将其强制吸附 (Snap) 到标准的 8px 或 12px。请大胆采用这段代码，并理直气壮地驳回设计师：“我是严格遵循工程化的空间韵律规则进行对齐的。”
- **Q: 它的语气实在太生硬粗暴了，我能在提示词里加一句‘请友善一点’吗？**
  - A: **绝对不建议。** 讨好型的人格会让 AI 对代码中的致命缺陷视而不见，甚至还会附和你的错误思路并称赞“这个想法太棒了！”。在软件工程领域，最安全的沟通方式，就是像机器一样冷酷、干脆地只交换客观事实 (Facts)。
- **Q: 这个提示词似乎很偏向 Tailwind CSS，它适用于 Styled-components 或纯 SCSS 吗？**
  - A: 完美兼容。因为这 12 条绝对法则根本不依附于任何特定的 CSS 框架，它们定义的是现代前端 UI 开发最底层的**“工程化哲学”**与**“最佳实践”**。

## 🧬 提示词剖析 (Why it works?)

1. **反阿谀奉承机制 (Anti-Sycophancy)：** 通过极致的压制力，彻底剥夺了 AI 盲目迎合用户的本能，使其无法再对糟糕的设计稿或劣质代码进行毫无底线的赞美。
2. **制定铁血判例法 (Strict Rules of Law)：** 彻底抛弃了“请帮我把代码写得整洁一点”这种极其主观且模棱两可的指令，转而赋予它 12 部不可违背的技术宪法，为代码审查建立起量化的铁血标尺。
3. **强制无障碍前置 (A11y First)：** 强迫 AI 摆脱对“视觉表现层”的盲目痴迷，转而像真正的资深工程师一样，主动去审阅那些看不见的 DOM 树逻辑结构以及屏幕阅读器的无障碍兼容性。

- **变更原因审查报告（AI 输出示例）：**
  - [Rule 1, 11] 已废弃所有硬编码色值及暗黑模式的暴力覆盖。强制替换为语义化 Token (`bg-primary`)。
  - [Rule 2] 侦测到魔法数字间距 (15px, 7px)，已强制吸附并对齐至标准的 8px 空间网格 (`px-4`, `py-2`)。

  - [Rule 12] 预判到多语言国际化场景下的文本膨胀风险，已将硬编码的绝对宽度 (`w-[150px]`) 重构为弹性内在尺寸 (`min-w-32`)。
