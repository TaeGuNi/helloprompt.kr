## 📝 コンテキストのコピペはやめよう：5分で作る初めてのMCPサーバー

- **🎯 おすすめの方:** プロンプトに毎回DBスキーマやAPIドキュメントをコピペすることに疲れた開発者、ローカルデータの活用者
- **⏱️ 所要時間:** 5分 → 恒久的な時間の短縮
- **🤖 最高パフォーマンス:** Claude Desktop、OllamaなどのMCP対応モデル

- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「Claudeのウィンドウを開き、DBスキーマをコピペします。10分後に新しいウィンドウを開き、同じ作業を繰り返します。この退屈な消耗戦を、今こそ終わらせる時が来ました。」_

!["Stop Pasting Context: Build Your First MCP Server in 5 Minutes"](/images/hooks/mcp-server-tutorial.jpg)

開発者やデータアナリストなら誰もが共感する、恐ろしい日常があります。それは、ChatGPTやClaudeの空の対話ウィンドウを立ち上げ、社内データベースのスキーマ（`schema.sql`）や数千行に及ぶAPI仕様書をドラッグしてコピペするという、終わりのない反復作業です。「このテーブル構造に基づいてクエリを書いて」「うちの社内API仕様書はこれなんだけど、どうやって呼び出せばいい？」と、新しい対話ウィンドウを開くたびに、テキストの塊を手動で「食べさせて」あげなければなりません。

コンテキストウィンドウが少しでもこじれたり、トークン制限に引っかかったりすると、AIは的外れなテーブルを参照したり、存在しないカラムを捏造したりします。さらに、決して外部に流出してはならない<b>機密性の高い社内コード</b>や<b>顧客データ</b>をブラウザのウィンドウにコピー＆ペーストするたびに、背筋が凍るような思いをすることもあります。この過程で、私たちはコーディングに集中するのではなく、AIの好みに合わせてプロンプトを整え、コンテキストを組み立てる単純労働者に成り下がってしまいます。

この無意味な肉体労働をいつまで繰り返すべきでしょうか？一日にプロンプトウィンドウを10回新しく開くと仮定しても、テキストを探してコピー＆ペーストする事前作業だけで、少なくとも30分が空に消えていきます。一週間なら2時間半、一ヶ月なら丸一日を完全にテキストのコピペに費やしていることになります。単に時間が浪費されるだけでなく、開発者の創造力や集中力までもが粉々に砕け散ってしまいます。

さらに致命的な問題は、まさに <span style="color:var(--color-cyber-cyan)">コンテキストの断片化（Context Fragmentation）</span>です。昨日貼り付けたスキーマが今日の午前中に更新されたのに、AIは依然として昨日の古いスキーマに基づいてコードを書きます。結局、間違ったコードによるエラーを修正するために数十分を費やし、「いっそ自分で書いたほうが早い」と腹を立てることになります。他の人々はAIで退社時間を早めているというのに、私たちはAIの機嫌を伺い、学習させるためにむしろ残業を自ら買って出ているという、逆説的な状況に陥っているのです。

しかし2026年現在、これらすべての苦痛を一瞬で終わらせるゲームチェンジャーが登場しました。それが、Anthropicが主導する <b>Model Context Protocol (MCP)</b>です。この巨大な技術的飛躍がAIエコシステムの標準として定着したことで、もはやAIに手動でデータを食べさせる必要は完全になくなりました。AIが自分のローカルデータに <b>直接ストローを差し込み、リアルタイムで勝手に読み込んでくる</b>ようにできるからです。

MCPを分かりやすく例えるなら、AIモデル専用の <b>汎用USBポート</b>のようなものです。自分のコンピュータにサーバーを一度だけ立ち上げ、ポートを接続しておけば、Claude DesktopやOllamaのようなクライアントが自分のローカルファイルシステム、内部データベース、さらには閉鎖網にある社内APIに直接アクセスし、最新情報をリアルタイムでストリーミングします。

さらに驚くべき点は、このような強力なカスタムサーバーを構築するのに、たった5分あれば十分だということです。今日紹介するプロンプトを活用すれば、開発者が直接複雑なボイラープレートコードを書く代わりに、AIに指示して固有のビジネス環境にぴったりな、完璧なTypeScriptベースのMCPサーバーコードを一瞬で <b>スキャフォールディング（Scaffolding）</b>できます。数十個の依存関係や初期設定で迷うことなく、完璧に動作するコードを一度に手に入れることができます。

さあ、このプロンプトを適用した後の驚くべき変化を想像してみてください。何もない空の対話ウィンドウを開き、<i>「今日新しく加入したユーザー数は何人？」</i>あるいは <i>「昨日発生した決済エラーログの主な原因は何？」</i>と自然に問いかけるだけで完了です。

Claudeがバックグラウンドで静かにローカルDBやログファイルに直接クエリを飛ばし、寸分の狂いもない正確な最新データに基づいて完璧な回答を導き出します。データ流出の心配をすることなく、自分のコンピュータの中で最も快適で安全、かつ強力な自分専用のAIインフラが完成します。あなたはもはや単なるコーダー（Coder）を超え、自分だけの強力なAIシステムを指揮する <b>アーキテクト（Architect）</b>へと生まれ変わる第一歩を踏み出すことになります。今すぐそのチートキーを公開します。

## 💡 筆者コメント (Insight & How to use)

このプロンプトが持つ真の価値は、単にAIにコードを書いてほしいと頼む漠然としたアプローチを超え、**MCPエコシステムが要求する厳格なプロトコル規格をAIに強制的に注入する**点にあります。

私が初めてカスタムMCPサーバーを作ろうとした時に経験した悲惨な失敗が、このプロンプトの骨組みになりました。単に「社内DBを読み込んで教えてくれるMCPサーバーを作って」と指示したところ、AIは見た目こそ立派ですが、実際には全く動作しない **「幽霊SDKメソッド」** を大量に捏造しました。MCP通信の核心的なライフサイクルである `ListTools` と `CallTool` のリクエスト処理は完全に無視され、ただのありふれた一般的なExpress.jsウェブサーバーコードを吐き出すばかりだったのです。結局、そのめちゃくちゃなコードを修正し、デバッグするのに半日という貴重な時間を無駄にする羽目になりました。

この手痛い失敗を身をもって経験して悟った唯一の真理は、<span style="color:var(--color-cyber-cyan)">「下書き作成の段階で、AIにアーキテクチャの主導権を絶対に渡してはならない」</span>ということでした。

そこで私は、プロンプトの指示の中に公式 `@modelcontextprotocol/sdk` の必須インターフェースである `ListToolsRequestSchema` と `CallToolRequestSchema` を必ず実装するように明示的に釘を刺しました。さらに、ツールの入力パラメータがこじれるのを防ぐために、ランタイム型検証ライブラリである `zod` の使用を強制する安全装置を細かく組み込みました。その結果、たった一度のプロンプト実行だけで、文法的な誤りや型エラーが全くない、即座にターミナルで実行可能な **完璧な単一ファイル(`index.ts`)サーバーコード** が誕生することになりました。

特に現場の実務において、このカスタムMCPサーバーが真価を発揮する瞬間は、まさに <b>「圧倒的なプライバシー(Privacy)」</b> と <b>「動的コンテキスト(Dynamic Context)」</b> が結合する時です。従来の方式では、社内ネットワークの奥深くに閉じ込められている機密性の高い顧客ログデータや非公開API仕様書をAIと結合するために、セキュリティチームの顔色を伺いながら、いちいち非識別化作業を行わなければなりませんでした。

しかし、徹底的にローカル環境で駆動する自分専用のカスタムMCPサーバーを立ち上げておけば、状況は完全に変わります。あなたが扱う機密データは決して外部のクラウドサーバーに送信されることはなく、ただ **ローカルの物理メモリの中だけで安全に処理** されます。Claude Desktopのようなクライアントは、単にローカルホスト(localhost)にクエリ(Query)を投げ、結果値だけを安全に受け取るだけです。

実務に即座に適用する際の核心的なコツは、**変数制御法** です。プロンプト内の `[目的]` 変数を入力する際は、曖昧に書かずに具体的なツールの形態を明示する必要があります。例えば、`[特定のデータベースの今日加入者数を照会する機能]` と書くよりは、`[PostgreSQLデータベースのusersテーブルでdateカラムを基準に今日加入したユーザー数をSELECTするクエリを実行するツール]` と明確に制御するのが得策です。ツールが受け取るパラメータと返すべきデータ形式を明確にするほど、AIは `zod` スキーマをより精巧に作り上げます。

私は実際にこのプロンプトを応用して、社内のJira課題トラッカーや内部のSlack通知とリアルタイムで連動する強力なMCPサーバーを自ら構築して使用しています。朝出勤してすぐに <i>「今自分に割り当てられている今回のスプリントチケットのうち、進捗率が50%未満のものだけを抽出して要約して」</i> と打てば、AIがリアルタイムでJira APIを叩いて最新データを取得し、コンテキストを構成して即座に完璧なブリーフィングを生成します。

皆さんも、最初は特定のフォルダのテキストファイルを読み込む単純な「読み取り専用(Read-only)」ツールから、軽やかに、そして安全に始めてみてください。サーバーの骨組みに慣れ、データが行き来する動作原理が体得されれば、次第にデータベースに直接クエリを投げたり、社内メッセンジャーに自動通知を送ったりするなど、自分だけのローカルインフラを自在に指揮する真の生産性の新世界を経験することになるでしょう。

## 🙋 よくある質問 (FAQ)

- **Q: コーディングの知識が全くなくても、このプロンプトを使用できますか？**
  - A: はい、もちろんです！ `[目的]` 欄に「社内Wikiドキュメント検索機能」のように自然言語で書くだけでも、AIが構造的に完璧なコードを代わりに作成してくれます。皆さんは生成されたコードをコピーして実行環境に貼り付け、スクリプトを起動するだけです。

- **Q: 生成されたサーバーコードはどうやって実行し、LLMクライアントに接続するのですか？**
  - A: コードを `index.ts` ファイルとして保存した後、Claude Desktopの設定ファイル(`claude_desktop_config.json`)内の `mcpServers` 項目に、該当スクリプトの実行コマンド（例：`npx tsx index.ts`）と絶対パスを追加していただければ、即座に連動してツールタブに表示されます。

- **Q: このプロンプトで作ったサーバーはClaudeでしか動作しませんか？**
  - A: いいえ。Model Context Protocolは特定のモデルに依存しない汎用的な <b>オープンスタンダード(Open Standard)</b> です。したがって、Ollama、Cursor、Zedなど、MCP規格をサポートするすべてのクライアント環境で修正なしに完璧に互換します。

## 🧬 プロンプト解剖 (Why it works?)

1. **制約条件(Constraints)の強力な統制:** `@modelcontextprotocol/sdk` と `zod` ライブラリの使用を強制し、必須スキーマ(`ListToolsRequestSchema`, `CallToolRequestSchema`)を厳格に明示しました。これにより、AIがスペックに合わない幽霊コードを捏造するハルシネーション（幻覚）現象を根本的に遮断します。
2. **単一ファイルおよびコード専用出力の統制:** 複数のファイルに断片化された複雑なアーキテクチャの代わりに単一ファイル(`index.ts`)構造を指示し、不要な補足説明を完全に排除するように命令しました。その結果、コードをコピーしてテストするのにかかる時間(Time-to-Value)が飛躍的に短縮されます。
3. **ランタイム例外処理の強制化:** ローカル環境でツール実行中に予期せぬエラーが発生しても、MCPサーバープロセス全体がダウン(Crash)しないように、`try/catch` の防御ロジックを事前に徹底的に規定しました。
