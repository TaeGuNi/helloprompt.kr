# 📝 Рефакторинг грязного кода в чистый за 1 секунду

- **🎯 Рекомендуется для:** Junior и Middle разработчиков, Tech Lead-ов, уставших от ревью
- **⏱️ Требуемое время:** 5 минут → 1 секунда
- **🤖 Рекомендуемая модель:** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _"Я даже не могу узнать код, который сам написал месяц назад... Как вообще распутать это спагетти?"_

У всех нас был опыт написания беспорядочных имен переменных в спешке перед релизом или создания «функций-монстров» на 100+ строк. Что если бы у вас был личный ассистент, который мгновенно рефакторил бы код, который стыдно показывать коллегам, приводя его к стандартам **«Чистого кода» (Clean Code)**? С помощью всего одного этого промпта вы можете одновременно улучшить читаемость, производительность и поддерживаемость вашего проекта.

---

## ⚡️ 3-строчное резюме (TL;DR)

1. Интуитивно понятное переименование переменных и функций по стандартам индустрии.
2. Устранение дублирования логики и строгое соблюдение принципа единственной ответственности (SRP).
3. Автоматическое применение современного синтаксиса языка и микро-оптимизаций производительности.

## 🚀 Решение: "Clean Code Refactoring"

### 🥉 Basic Version (Базовая версия)

Используйте это, когда вам нужен просто быстрый результат без лишних деталей.

> **Роль:** Ты старший инженер-программист с 20-летним опытом и эксперт по написанию чистого кода.
> **Запрос:** Преврати этот сложный спагетти-код в легко читаемый и поддерживаемый чистый код. Поясни основные изменения.


### 🥇 Pro Version (Профессиональная версия)

Используйте этот вариант для глубокого рефакторинга критически важных участков кода. Скопируйте шаблон ниже и вставьте его в ИИ.

> **Роль (Role):** Ты Senior Software Engineer с 20-летним опытом и абсолютный эксперт по Clean Code и архитектуре ПО.
>
> **Контекст (Context):**
>
> - Фон: Код, который я предоставляю, работает без ошибок, но он плохо читается, его трудно масштабировать и поддерживать. Имена переменных неясны, присутствует жесткое связывание (tight coupling) и много дублирующейся логики.
> - Цель: Получить идеальный, соответствующий индустриальным стандартам код, которым можно гордиться на Code Review.
>
> **Задача (Task):**
>
> Отрефактори предоставленный ниже код в соответствии со следующими критериями:
>
> 1. **Именование (Naming):** Измени имена переменных и функций так, чтобы их назначение было кристально ясным без дополнительных пояснений.
> 2. **Принцип SRP:** Разбей большие функции на более мелкие, чтобы каждая функция выполняла только одну конкретную задачу.
> 3. **DRY (Don't Repeat Yourself):** Извлеки повторяющуюся логику в отдельные функции или утилиты.
> 4. **Комментирование:** Оставляй комментарии только там, где требуется объяснить 'Почему' код написан именно так, а не 'Что' он делает. Удали очевидные комментарии.
> 5. **Современный синтаксис:** Примени самые современные и безопасные конструкции целевого языка программирования (например, ES6+ для JS, паттерн-матчинг в Python 3.10+ и т.д.).
>
> **Код:**
>
>
> [Вставьте ваш спагетти-код сюда]
>
>
> **Ограничения (Constraints):**
>
> - Ни при каких обстоятельствах не меняй исходную бизнес-логику и поведение кода.
> - Не используй сторонние библиотеки, если они не были импортированы в оригинальном коде.
>
> **Формат вывода (Format):**
>
> - Сначала выведи отрефакторенный код в блоке кода.
> - Затем предоставь маркированный список "Что изменилось и почему" для легкого понимания.

## 💡 Комментарий автора (Insight)

Этот промпт спасает меня как минимум пару раз в неделю, особенно когда я работаю с унаследованным кодом (legacy code) или прототипами, написанными в спешке. Самая большая ценность здесь — это не просто причесывание синтаксиса, а применение **SRP (Single Responsibility Principle)**. ИИ отлично видит, когда функция делает слишком много вещей одновременно, и аккуратно разделяет её на логические блоки.

_Совет из практики:_ Всегда прогоняйте полученный результат через ваши unit-тесты. ИИ иногда может слишком увлечься оптимизацией и изменить краевые случаи (edge cases) бизнес-логики.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Сработает ли это для редких или старых языков программирования?**
  - A: Да, базовые принципы чистого кода (понятные имена, короткие функции) универсальны. Однако для очень специфичных языков (например, COBOL или старых версий Delphi) лучше дополнительно указать версию и диалект языка в промпте.

- **Q: Модель разбила мой код на 10 маленьких функций, и теперь в нем сложно ориентироваться. Что делать?**
  - A: Вы можете добавить в блок ограничений правило: `- Не разбивай функции, если их длина не превышает 15 строк` или `- Сохрани баланс между абстракцией и читаемостью сверху вниз`.

- **Q: Безопасно ли отправлять проприетарный код компании в публичные ИИ?**
  - A: Если вы используете публичные бесплатные версии ChatGPT или Claude, ваш код может использоваться для обучения моделей. Всегда удаляйте API-ключи, пароли и чувствительную бизнес-информацию перед отправкой, либо используйте корпоративные (Enterprise) тарифы с гарантией конфиденциальности.

## 🧬 Анатомия промпта (Why it works?)

1. **Явное указание не ломать логику:** Жесткое ограничение "Не меняй бизнес-логику" страхует вас от того, что ИИ "оптимизирует" код, удалив важный кусок алгоритма.
2. **Ориентация на "Почему", а не "Что":** ИИ часто грешит добавлением избыточных комментариев вроде `// увеличиваем i на 1`. Запрос комментировать только неочевидные решения очищает код от визуального мусора.
3. **Требование списка изменений:** Вывод изменений списком помогает вам быстро провести мини-ревью предложенного кода, прежде чем бездумно копировать его в IDE.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод)

```javascript
function calc(d) {
  let t = 0;
  for (let i = 0; i < d.length; i++) {
    if (d[i].price > 100) {
      t += d[i].price * 0.9;
    } else {
      t += d[i].price;
    }
  }
  return t;
}
```

### ✅ After (Результат)

```javascript
/**
 * Вычисляет общую стоимость товаров в корзине.
 * Применяет скидку 10% для товаров стоимостью выше 100 условных единиц.
 */
function calculateTotalOrderValue(items) {
  const DISCOUNT_THRESHOLD = 100;
  const DISCOUNT_MULTIPLIER = 0.9;

  return items.reduce((total, item) => {
    const currentPrice =
      item.price > DISCOUNT_THRESHOLD
        ? item.price * DISCOUNT_MULTIPLIER
        : item.price;

    return total + currentPrice;
  }, 0);
}
```

_Изменения: невнятное `calc` заменено на `calculateTotalOrderValue`, `d` -> `items`, магические числа вынесены в константы с понятными именами, а цикл заменен на декларативный метод `reduce`._
