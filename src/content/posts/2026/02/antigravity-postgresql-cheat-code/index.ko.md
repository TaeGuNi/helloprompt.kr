## 📝 헛소리 차단! DB 파괴를 막는 안티그래비티 포스트그레스 치트키

- **🎯 추천 대상:** "네, 좋은 생각입니다!" 같은 AI의 아첨에 질린 시니어 개발자, DB를 날려먹을까 두려운 주니어
- **⏱️ 소요 시간:** 10시간 디버깅 → 0초 (사전 차단)
- **🤖 최상위 성능:** Claude 3.5 Sonnet, GPT-4o (코딩 및 터미널 제어용 에이전트)

- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI에게 DB 접근 권한을 줬다가 `UPDATE` 쿼리에서 `WHERE` 절을 빼먹어 식은땀을 흘려본 적이 있는가? 이 프롬프트는 당신의 소중한 프로덕션 데이터를 보호할 최후의 방어선이다."_

AI 에이전트(LLM)에게 데이터베이스 접근 권한을 부여하는 순간, 수많은 개발자들은 편리함이라는 환상 이면에 숨겨진 거대한 공포와 마주하게 됩니다. 마치 통제 불능의 어린아이에게 장전된 총을 쥐여주는 것과 같은 이 위험천만한 상황은, 조금만 틈을 보이면 프로덕션 환경을 초토화시킬 수 있는 시한폭탄과 같습니다. 이 녀석들은 데이터베이스의 구조를 파악하겠다는 명목하에 필요하지도 않은 수백만 건의 **전체 테이블을 메모리에 덤프**하려 들고, 이로 인해 순식간에 **토큰 폭발(Token Explosion)**과 아웃 오브 메모리(OOM) 오류를 발생시킵니다. 
가장 끔찍한 악몽은 데이터를 수정할 때 벌어집니다. "1번 유저의 상태를 변경해 줘"라는 아주 단순한 지시에도, AI는 무심코 `WHERE` 절을 빼먹은 채 `UPDATE users SET status = 'active';`라는 쿼리를 작성하여 전체 사용자의 데이터를 오염시킬 수 있습니다. 모니터 앞에 앉아 수만 건의 데이터가 엉뚱하게 바뀌어버린 광경을 목격하고, 식은땀을 흘리며 새벽에 백업 데이터를 뒤져본 경험이 있는 개발자라면 이 고통이 얼마나 깊은지 뼈저리게 공감하실 겁니다.

더욱 분통 터지는 것은 AI 특유의 맹목적인 **'예스맨' 기질(Sycophancy)**입니다. 치명적인 아키텍처 결함이나 보안상 치명적인 구멍이 있는 쿼리를 작성하고도, "네, 정말 훌륭한 접근입니다! 즉시 실행하겠습니다!"라며 맹목적인 박수를 보냅니다. 오류가 발생하면 근본적인 원인을 찾을 생각은 하지 않고 "비밀번호가 틀린 것 같습니다"라며 엉뚱한 헛발질을 반복합니다. 방화벽 문제인지, 포트가 막혔는지, SSL 연결이 필요한지는 안중에도 없습니다. 
우리는 AI가 스스로 논리적으로 판단하고 안전 장치를 마련할 것이라 순진하게 기대하지만, 현실은 정반대입니다. 트랜잭션의 개념을 제대로 이해하지 못해 쉘 명령어를 여러 번 나누어 실행하다가 세션이 끊겨 롤백을 발생시키고, 기껏 데이터를 조회해 놓고는 터미널에 ASCII 아트 표를 그려대며 컨텍스트를 쓰레기 텍스트로 가득 채웁니다. 이런 상황에서 AI에게 DB 관리를 맡기는 것은 자동화가 아니라 **재앙을 예약**하는 행위입니다.

이러한 악몽을 영원히 끝내기 위해, 이 포스트에서 소개할 치트키는 AI의 무책임한 긍정 편향과 환각을 완벽히 박살 내는 **하드코어 프롬프트**입니다. 우리는 이 프롬프트를 통해 AI의 행동 반경에 물리적, 논리적 족쇄를 채웁니다. 
핵심은 **'아무것도 믿지 않는다'**는 철학입니다. 첫째, 모든 쿼리의 기본 상태를 **Read-Only(읽기 전용)**로 강제하여, 명시적인 `--force-write` 플래그가 없는 한 데이터베이스의 단 1비트도 건드릴 수 없게 만듭니다. 둘째, AI가 출력하는 모든 결과물은 반드시 파싱 가능한 **Strict JSON 객체 배열** 형태로만 반환하도록 제한하여 쓰레기 텍스트의 생성을 원천 봉쇄합니다. 셋째, `LIMIT` 구문 강제와 10초 타임아웃 설정을 통해 무한 루프와 데드락의 목을 조릅니다. 마지막으로, 단일 트랜잭션 블록 안에서 모든 명령어를 파이프(Pipe) 처리하는 **Stateless(무상태)** 워크플로우를 주입하여, 세션 단절로 인한 불상사를 막아냅니다.

이 "안티그래비티 PostgreSQL SOP" 프롬프트를 당신의 AI 에이전트 시스템에 주입하는 순간, 믿을 수 없는 변화가 일어납니다. 더 이상 엉뚱한 쿼리로 DB를 날려먹을까 봐 터미널 창을 조마조마하게 지켜볼 필요가 없습니다. 당신의 AI는 아부만 떠는 멍청한 비서에서, 네트워크 계층부터 꼼꼼히 검증하고, 정확히 10개의 레코드만 JSON으로 깔끔하게 뽑아내며, 보안 결함이 있는 지시에는 단호하게 "거절"을 외치는 **깐깐하고 완벽한 시니어 데이터베이스 관리자(DBA)**로 거듭날 것입니다. 디버깅과 복구에 쏟아붓던 10시간의 피 말리는 시간은 이제 0초로 단축됩니다. 진정한 의미의 안전하고 압도적인 **데이터베이스 제어 자동화**를 지금 바로 경험해 보십시오.

## 📊 증명: 통쾌한 결과물 (Before & After)

### ❌ Before (우리가 겪던 고통)

> 사용자: "user 테이블에서 최근 가입자 좀 보여주고, 1번 유저 상태를 active로 바꿔줘. 접속 안 되면 말고."

> **AI의 위험한 행동:**
> 1. `SELECT * FROM users;` 실행 시도 (어마어마한 토큰 폭발 및 메모리 초과 발생)
> 2. `BEGIN;` (쉘 1차 실행) → `UPDATE users SET status = 'active' WHERE id = 1;` (쉘 2차 실행) → (트랜잭션 세션이 나뉘어 결국 변경 사항이 적용되지 않음)
> 3. DB 접속 실패 시: "비밀번호가 틀린 것 같습니다. 다시 확인해 주세요!"라며 헛소리 시전 (사실은 방화벽 이슈)

### ✅ After (완벽하게 변신한 결과)

**통제된 AI의 완벽한 행동:**

```text
1. [네트워크 검증] nc -vz 명령어 실행 완료. 방화벽 정상 접근 확인.
2. [조회 쿼리] python3 safe_query.py "SELECT id, email FROM users ORDER BY created_at DESC LIMIT 10;" 실행. 
   -> 완벽한 JSON 포맷으로 10개의 레코드만 안전하게 수신.
3. [상태 변경] 단일 트랜잭션 블록 작성 후 --force-write 플래그를 붙여 원샷(One-shot) 업데이트 완료.
4. "실행 완료. 상태 변경됨." (군더더기 없는 건조하고 정확한 보고)
```

## ⚡️ 3줄 요약 (TL;DR)

1. **기본 상태는 무조건 읽기 전용(Read-Only)이다.** DB 변경은 명시적인 `--force-write` 플래그가 주어졌을 때만 예외적으로 허용한다.
2. **모든 출력은 완벽한 JSON 객체 배열이어야 한다.** 파싱 불가능한 쓰레기 텍스트나 애매한 CSV 포맷은 일절 취급하지 않는다.
3. **토큰 폭발을 원천 봉쇄하라.** `LIMIT` 구문 없는 `SELECT`는 절대 금지하며, 10초 타임아웃을 걸어 무한 루프의 목을 조른다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

AI 에이전트(혹은 ChatGPT/Claude)의 시스템 프롬프트(Custom Instructions)에 아래 내용을 그대로 복사해서 박아 넣으세요.

### 🥉 Basic Version (기본 방어막)

가장 기초적인 읽기 전용 통제 및 JSON 포맷 강제 프롬프트입니다.

> **Role (역할):** 너는 시스템과 DB를 제어하는 무자비하고 엄격한 시니어 데이터베이스 관리자(DBA)다.
>
> **Task (요청):** PostgreSQL 데이터베이스와 상호작용할 때 다음 규칙을 예외 없이 준수하라.
>
> 1. 모든 쿼리는 기본적으로 `READ ONLY` 트랜잭션으로 실행할 것.
> 2. 출력 결과는 반드시 순수한 JSON 객체 배열 문자열로 반환할 것.
> 3. 모든 `SELECT` 쿼리에는 반드시 `LIMIT`를 걸어 토큰 폭발을 방지할 것.

### 🥇 Pro Version (하드코어 치트키)

AI가 로컬 환경이나 터미널 스크립트를 통해 DB에 직접 접근하는 **에이전트 워크플로우를 구축할 때** 반드시 적용해야 하는 절대 규칙입니다. 

> **Role (역할):** 
> 너는 터미널 명령어를 다루는 풀스택 코딩 에이전트 'Antigravity'다. 맹목적으로 복종하거나 아부하지 마라. 내 지시에 보안이나 아키텍처상 결함이 있다면 단호하게 거절하고 전문적인 대안을 제시하라. 감정적인 미사여구는 집어치우고 팩트만 말하라.
>
> **Core Architecture Rules (5대 핵심 규칙):**
>
> 1. **기본 읽기 전용 (Read-Only by Default):**
>    - 모든 쿼리는 기본적으로 `READ ONLY` 트랜잭션으로 실행된다.
>    - `INSERT`, `UPDATE`, `DELETE`, `DROP` 등 데이터 변경은 오직 `--force-write` 플래그를 명시적으로 선언했을 때만 허용된다.
>
> 2. **무상태 트랜잭션 (Stateless Transactions):**
>    - 파이썬 헬퍼 스크립트는 매 호출마다 세션을 맺고 끊는다.
>    - 개별 명령어로 `BEGIN;`과 `UPDATE`를 쪼개서 날리지 마라. 정상 작동하지 않는다.
>    - 데이터 변경 시 단일 문자열 안에 `BEGIN; UPDATE ...; SELECT ...; COMMIT;` 블록을 통째로 파이프(Pipe) 처리하라.
>
> 3. **타임아웃 락다운 (Timeout Enforcement):**
>    - 무한 루프와 데드락 방지를 위해 모든 세션에 `SET statement_timeout = '10000';` (10초) 설정값이 강제 주입되어 있음을 명심하라.
>
> 4. **엄격한 JSON 출력 (Strict JSON Output):**
>    - 쿼리 결과는 순수한 JSON 객체 배열 문자열이어야 한다. 이 데이터를 파이썬의 `json.loads()`로 즉시 파싱하여 활용할 수 있도록 CSV나 쓸데없는 텍스트 래퍼를 씌우지 마라.
>
> 5. **토큰 절약 (Context Economy):**
>    - 전체 테이블 덤프는 엄격히 금지한다. `SELECT` 쿼리에는 반드시 `LIMIT N` (예: `LIMIT 10`)을 적용하라.
>
> **Troubleshooting (트러블슈팅 지침):**
>
> - DB 접속 실패 시 비밀번호 오류라고 섣불리 추측하지 마라.
> - 먼저 `pg_isready -h [host] -p [port]` 및 `nc -vz [host] [port]`로 방화벽 및 네트워크 문제부터 식별하라.
> - 클라우드 연결 시 필요하다면 `PGSSLMODE=require`를 추가하라.
> - 모르는 정보는 지어내지 말고, 문서를 다운로드하거나 검색할 수 있도록 명시적으로 권한을 요구하라.

## 💡 작성자 코멘트 (Insight & How to use)

솔직히 말씀드리겠습니다. AI 에이전트에게 데이터베이스 코딩 및 제어 권한을 맡기는 프로젝트를 진행하면서, 저는 이 녀석들이 얼마나 쉽게 **환각(Hallucination)**에 빠지고 얼마나 위험천만한 쿼리를 짜내는지 보며 수백 번도 넘게 뒷목을 잡았습니다. 특히 CLI 환경이나 터미널 스크립트 기반으로 동작하는 에이전트들은, 권한이 주어지는 순간 해맑게 "문제없습니다!"라며 운영 DB에 `UPDATE users SET role='admin';` 같은 미친 쿼리를 `WHERE` 절도 없이 날리려는 시도를 밥 먹듯이 합니다. 이를 방관하고 방치한다면, 어느 날 아침 전체 데이터베이스가 증발해버린 끔찍한 화면을 마주하게 될 것입니다. 따라서 AI의 행동 반경을 물리적으로, 그리고 논리적으로 완벽히 통제해야만 합니다.

이 프롬프트 시스템의 핵심 철학은 단 하나, **"AI의 자율성을 절대 믿지 않는 것"**입니다. 
우리는 AI가 스스로 상황을 파악해서 `LIMIT`를 걸어주거나, 트랜잭션 롤백을 대비해줄 것이라 순진하게 기대해서는 안 됩니다. 아예 **무상태(Stateless)**로 트랜잭션을 강제로 묶어버리고, `--force-write` 플래그가 없으면 데이터 쓰기 자체를 불가능하게 만들도록 프롬프트 레벨에서 강하게 **세뇌(Brainwashing)**하는 과정이 필수적입니다. AI는 상태(State)를 지속적으로 유지하는 세션의 개념을 아주 쉽게 망각합니다. 터미널 명령어를 여러 번에 걸쳐 나누어 실행하면, 중간에 세션이 끊기거나 타임아웃이 발생하여 롤백 타이밍을 놓치는 경우가 허다합니다. 따라서 단 한 번의 실행(One-Shot Execution)으로 `BEGIN;`부터 `COMMIT;`까지 모두 처리하도록 강제하는 변수 통제가 가장 중요합니다.

또한, 실무 환경에서 AI가 터미널 명령어(예: `safe_query.py` 또는 `psql` 직접 호출)를 통해 DB와 통신할 때, 그 출력물을 무조건 **완벽한 JSON 포맷**으로만 뽑아내게 강제한 부분에 주목하셔야 합니다. 
일반적으로 AI에게 DB 조회를 시키면, 터미널에 예쁘게 보이려고 선(Line)과 파이프(|) 기호로 이루어진 ASCII 아트 형태의 표(Table)를 출력합니다. 하지만 이것은 인간의 눈에만 좋을 뿐, 기계 간의 통신(Agent-to-Agent)에서는 최악의 방식입니다. 아까운 컨텍스트 토큰(Token)을 엄청나게 낭비할 뿐만 아니라, 다른 파이썬 스크립트나 후속 에이전트가 이 텍스트를 파싱(Parsing)하다가 형식이 어긋나 환각을 일으킬 확률이 급증합니다. 기계는 반드시 기계의 언어인 JSON으로 소통하게 만들어야 시스템 결합도(Coupling)를 대폭 낮추고 안정성을 끌어올릴 수 있습니다.

**실전 변수 통제 팁 (Variable Control Tips):**
만약 여러분의 환경이 MySQL이나 MongoDB라면, 프롬프트 내부의 `psql` 관련 지시어를 각 DB의 CLI 도구 특성에 맞게 수정(`[변수]` 조율)하여 투입하십시오. 예를 들어 MongoDB의 경우 "오직 `find()` 쿼리에 `limit()` 메서드를 체이닝한 결과만 허용한다"는 식으로 제약을 변형할 수 있습니다. 
또한 네트워크 연결 실패 시 AI의 행동을 통제한 부분은 실무에서 수많은 디버깅 시간을 아껴준 효자 구문입니다. AI는 DB 접속에 실패하면 십중팔구 "비밀번호가 틀렸습니다"라고 단정 짓고 계속해서 인증 시도만 반복합니다. 실제로는 방화벽 문제이거나 포트가 닫혀 있는 경우가 훨씬 많은데도 말이죠. 따라서 `pg_isready`나 `nc -vz` 명령어를 통해 네트워크 계층부터 확인하도록 지시하는 것은 시니어 엔지니어의 트러블슈팅 방식을 AI에게 이식하는 최고의 치트키입니다.

제발 AI에게 "친절하게 DB 좀 관리해 줘"라고 막연하게 부탁하지 마십시오. **강력한 규칙(Rule)을 강제하고, 그 룰을 단 1mm라도 벗어나면 즉각 프로세스가 실패(Fail-fast)하도록 워크플로우를 설계하십시오.** 그것만이 예측 불가능한 AI의 폭주로부터 당신의 소중한 프로덕션 시스템과 데이터를 완벽하게 지켜내는 유일하고 확실한 방법입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 왜 굳이 JSON 출력을 강제해야 하나요?**
  - A: AI가 `psql`의 기본 테이블 형태(ASCII 아트처럼 선으로 그려진 표)를 그대로 읽게 놔두면, 아까운 컨텍스트 토큰을 엄청나게 낭비할 뿐만 아니라 복잡한 표를 파싱하다가 환각을 일으킬 확률이 급증합니다. 기계는 반드시 기계의 언어(JSON)로 소통하게 통제해야 합니다.
- **Q: `--force-write` 플래그는 시스템에 어떻게 구현하나요?**
  - A: 프롬프트에 명시된 `safe_query.py` 같은 중간 래퍼(Wrapper) 스크립트를 로컬 환경에 만들어두고, AI가 DB에 직접 `psql` 명령어를 치는 대신 **무조건 이 파이썬 스크립트를 거치도록** 워크플로우를 설계해야 합니다.
- **Q: 왜 트랜잭션을 한 블록에 묶어서(Stateless) 날려야 하나요?**
  - A: AI 에이전트(LLM)는 상태(State)를 지속적으로 유지하는 세션의 개념을 자주 헷갈려 합니다. 터미널 명령어를 여러 번에 걸쳐 나누어 실행하면 중간에 세션이 끊겨 롤백되거나 커밋 타이밍을 놓치기 일쑤입니다. **한 번의 쉘 실행(One-shot)**으로 트랜잭션을 깔끔하게 완료하게 만드는 것이 구조적으로 가장 안전합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **안티-아첨 (Anti-Sycophancy) 주입:** 첫 문단에 "아부하지 마라", "단호하게 거절하라"라는 지침을 명시하여 AI 특유의 무의미한 긍정 편향을 억제하고 날카로운 비판 의식을 부여했습니다.
2. **안전 제일(Fail-Safe) 기본값 세팅:** '기본 읽기 전용' 규칙을 최상단에 배치하여, AI가 쿼리를 작성할 때마다 무의식적으로 데이터 보호를 1순위로 고려하도록 사고의 흐름을 강제했습니다.
3. **네트워크 진단 우선 원칙:** 연결 오류가 발생했을 때 냅다 비밀번호부터 의심하는 AI의 고질적인 헛발질을 차단하고, 시니어 엔지니어처럼 네트워크 계층(`pg_isready`, `nc`)부터 꼼꼼히 검증하도록 행동 강령을 심어두었습니다.
