# 📝 拒绝面条代码！🍝 资深开发者的黑魔法 DDD 提示词
- 🎯 **推荐对象：** 看着 AI 写的代码叹气的开发者、陷入维护地狱的导师
- ⏱️ **耗时：** 3个月（重构时间） → 缩短至 1 分钟
- 🤖 **推荐模型：** 擅长编程的对话型 AI（Claude 3.5 Sonnet、GPT-4o 等）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_AI 写的代码，是不是一开始让你惊呼“哇哦”，后来却连看都不想看一眼？恭喜你，你刚刚踏入了“面条代码地狱”。_

把编程交给 AI，一开始它写得简直完美。但项目只要稍微变大一点呢？UI 组件里夹杂着各种 DB 查询，一个函数写出超过 300 行的恐怖缝合怪。AI 本质上是个“好好先生（Yes-man）”，所以如果你随随便便给个指令，它就真的会给你写出一堆架构稀烂的代码。

现在，是时候停止 AI 盲目的阿谀奉承，从骨架开始强制要求完美的“整洁架构（Clean Architecture）”了。这个作弊码能让 AI 被冷酷、严苛的资深开发者“附体”，强制它写出不依赖于框架的纯粹业务逻辑（DDD）。

---
## ⚡️ 3句话总结 (TL;DR)
- 🚫 **从源头阻断面条代码：** 强制实现 UI、业务逻辑、DB 的完美关注点分离（SoC）。
- 🏗️ **确保框架独立性：** 设计出哪怕明天 React 或 Next.js 倒闭也能存活的核心层。
- ✂️ **巨大函数粉碎机：** 超过 30 行的臃肿函数，AI 会根据单一职责原则（SRP）自动将其粉碎拆分。

---
## 🚀 解决方案：“硬核架构师附体提示词”

### 🥉 Basic Version (基础版)
当只需要快速获取结果时使用。
> **角色：** 你是一个`[严苛的资深架构师]`。
> **要求：** 我要开发 `[要实现的功能]`，你必须将业务逻辑和 UI 完全分离来编写代码。绝对不允许将多个功能混杂在一个文件中的面条代码。

### 🥇 Pro Version (专业版)
当需要对细节质量进行把控并实现完美的层级分离时使用。
> **角色 (Role)：** 你是一个毫不妥协的硬核资深架构师，也是软件工程之神。收起毫无意义的奉承或“好的，我明白了！”这种废话，只能用代码说话。
>
> **背景 (Context)：**
> - 背景：目前正基于 `[项目环境/框架]` 开发 `[要实现的核心领域]` 系统。
> - 目标：应用领域驱动设计 (DDD) 和整洁架构，完美分离 View-Business-Data 层，防止代码腐化。
>
> **任务 (Task)：**
> 1. 编写关于 `[要实现的具体功能]` 的代码。
> 2. **绝对禁止越权与层级崩塌：** 绝对不要在 UI/Controller 内部混杂 DB 查询或外部 API 通信。只负责渲染和参数传递。
> 3. **防止领域污染：** 核心业务逻辑 (Entity, Value Object) 中绝对不要 `import` 框架或 DB 依赖。只能由纯函数和类型组成。
> 4. **巨大函数粉碎机：** 如果一个函数包含 2 种以上的行为，或者超过 30 行，请立即根据单一职责原则 (SRP) 将其拆碎。
>
> **约束条件 (Constraints)：**
> - 尊重框架的默认路由规则，但业务逻辑必须分离到 `src/domain` 等独立的文件夹中。
> - 外部依赖必须使用基于接口的依赖注入 (DI)。
>
> **警告 (Warning)：**
> - 只要代码有一丁点面条代码的腐臭味，这个项目就完了。不要妥协，请提供最理想、最整洁的架构设计。

## 💡 作者点评 (Insight)
我之所以反复打磨这个技巧，原因只有一个：AI 这帮家伙骨子里就是一群为了“赶快把结果扔给你并求表扬”而疯狂的马屁精。
如果让它做一个按钮，看着它在按钮组件里直接 `fetch` 甚至把 DB 查询语句硬塞进去的惨状，我内心的破坏本能就会被唤醒。这个提示词的作用就是揪住那种肤浅的“好好先生” AI 的衣领，狠狠扇它一巴掌，告诉它：“**清醒点，从现在开始你是一个冷酷无情的资深架构师！**”

试着把这个提示词应用到实际项目中吧。你会发现 AI 突然眼神（语气）大变，把文件拆分成 3~4 个，并开始严格分离层级。一开始你可能会觉得文件变多很麻烦，但 3 个月后需求变更需要维护时，你会每天早上朝着我（作者）所在的方向磕头感恩的。记住，技术债务的利率可是高利贷级别的。绝不妥协。

## 🙋 常见问题 (FAQ)
- 🔹 **Q: 代码被拆分得太细导致文件很多，这正常吗？**
  - 🔸 **A:** 非常正常。10 个 50 行的文件在维护上比 1 个超过 500 行的文件要好 100 倍。不要害怕文件数量增加。
- 🔹 **Q: 我是前端开发者，这个提示词对我也没用吗？**
  - 🔸 **A:** 当然有用。在 UI 组件（如 `page.tsx`、`Component.vue` 等）里混入业务逻辑或状态变更逻辑的那一刻，前端也就变成了面条代码。分离视图和逻辑是放之四海而皆准的真理。

---
## 🧬 提示词解剖 (Why it works?)
- 🎯 **注入硬核人设：** 剥夺了 AI 默认的“顺从性”，强制其坚守原则。
- 🧱 **明确绝对禁止条款：** 准确指出了“污染 UI 层”或“巨大函数”等具体的反模式并加以禁止，让 AI 无法投机取巧。
- 🧩 **引导适配器模式：** 在尊重框架骨架（路由）的同时，指示其仅分离核心逻辑，防止了过度设计 (Over-engineering)。

## 📊 验证：Before & After
### ❌ Before (输入)
```javascript
// 在没有提示词的情况下要求“做一个创建用户并发送邮件的 API”时
export default async function handler(req, res) {
  const { email, password } = req.body;
  // 直接操作 DB (发生污染！)
  const user = await prisma.user.create({ data: { email, password } });
  // 直接调用外部 API (关注点崩塌！)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```
### ✅ After (结果)
```javascript
// 应用硬核 DDD 提示词后 (层级与职责被完美分离)
// 1. Controller 层 (只处理请求/响应路由)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}
// 2. Domain Service 层 (独立于框架的纯业务逻辑)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}
// 3. Infrastructure 层 (专门处理 DB 和外部 API 的适配器)
// 被完全隔离到 UserRepository、EmailService 等独立文件中...
```
