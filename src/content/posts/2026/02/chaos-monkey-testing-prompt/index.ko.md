# 📝 AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣

- **🎯 추천 대상:** 빈약한 'Happy Path' 테스트 코드에 지친 시니어 개발자, QA 엔지니어, 테크 리드
- **⏱️ 소요 시간:** 10분 → 10초 (AI가 무의미한 정상 작동 시나리오를 짜며 낭비하는 시간 원천 차단)
- **🤖 추천 모델:** 모든 대화형 AI (코드 작성에 특화된 Claude 3.5 Sonnet, GPT-4o 강력 추천)
- ⭐ **난이도:** ⭐⭐⭐⭐☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"AI가 작성한 테스트 코드가 단번에 통과했다고 안도하십니까? 유감스럽지만, 당신의 프로덕션 서버는 내일 당장 무너질 수 있습니다."_

![AI를 한계까지 몰아붙이는 카오스 몽키 테스트 프롬프트 💣 (Happy Path의 종말)](/images/hooks/chaos-monkey-testing-prompt.jpg)

AI 에이전트에게 테스트 코드 작성을 지시해 본 경험이 있으신가요? 십중팔구, AI는 한 치의 오차도 없는 유토피아적인 '정상 작동 시나리오(Happy Path)'만 번지르르하게 쏟아냅니다. 외부 API는 언제나 0.1초 만에 응답을 반환하고, 데이터베이스 커넥션 풀은 절대 고갈되지 않으며, 사용자는 단 하나의 오타도 없이 완벽한 데이터를 전송한다는 비현실적인 가정 위에서 말이죠. 개발자는 터미널에 뜨는 영롱한 초록색 'PASS' 마크를 보며 안도하지만, 이것은 재앙을 알리는 전주곡에 불과합니다. 냉혹한 프로덕션(Production)의 세계는 다릅니다. 신뢰하던 결제 게이트웨이는 트래픽이 몰리는 금요일 밤 돌연 502 Bad Gateway 에러를 뱉어내고, 레디스(Redis) 인스턴스는 다운되며, 클라이언트는 상상을 초월하는 기상천외한 엣지 케이스(Edge Case) 데이터를 밀어 넣습니다. AI가 짜준 '성공률 100%'의 빈약한 방어막만 믿고 배포를 감행했다면, 주말 새벽 피말리는 긴급 핫픽스 작업과 직면하게 될 것입니다. 트랜잭션 롤백은 실패하고 좀비 데이터가 쌓이며, 비즈니스의 치명적 손실은 고스란히 개발자의 책임으로 돌아옵니다. 우리가 진정으로 구축해야 할 코드는 80%의 실패와 예측 불가능한 재난을 견뎌내는 견고한 벙커입니다. 이 글에서 소개할 치트키는 AI에게 넷플릭스의 전설적인 장애 주입 시스템인 '카오스 몽키(Chaos Monkey)'의 철학을 강제로 주입하는 하드코어 프롬프트입니다. 네트워크 타임아웃, DB 단절, HTTP 500 에러 등 최악의 시스템 파괴 시나리오를 가정하여 방어 로직의 한계를 시험하도록 AI의 긍정 회로를 철저히 차단하십시오.

## ⚡️ 3줄 요약 (TL;DR)

- 🚫 **Happy Path 비중 최소화:** 정상 작동 테스트의 비중을 20% 이하로 억제하고, 나머지 80%를 철저한 예외 및 실패 시나리오 검증에 할당하십시오.
- 💥 **재난 시뮬레이션 의무화:** 네트워크 타임아웃, HTTP 500 에러, DB 연결 단절 등 서비스 중단을 유발하는 극한의 엣지 케이스를 강제로 주입합니다.
- 🧹 **글로벌 오염(Dirty Mocking) 원천 차단:** `window.fetch`와 같은 전역 객체의 무분별한 조작을 엄금하고, MSW나 의존성 주입(DI)을 통해 완벽한 테스트 격리를 보장받으세요.

## 🚀 해결책: "카오스 몽키(Chaos Monkey) 프로토콜"

### 🥉 Basic Version (기본형)

빠르게 기본적인 에러 핸들링 로직의 방어력을 1차 검증해야 할 때 가볍게 활용하세요.

> **역할:** 너는 예외 상황을 집요하게 파고드는 하드코어 QA 엔지니어이자 '카오스 몽키'야.
> **요청:** 내가 제공한 `[검증할 코드]`에 대한 테스트 코드를 작성해 줘. 단, 정상 작동(Happy Path) 시나리오는 단 1개만 작성하고, 나머지 모든 테스트는 네트워크 500 에러, DB 연결 타임아웃, 비정상적인 데이터 입력 등 '최악의 재난 상황'을 가정하여 방어 로직의 견고함을 검증해야 해.

### 🥇 Pro Version (전문가형)

AI의 무의미하고 순진한 긍정 회로를 완전히 박살 내고, 프로덕션 수준의 완벽한 테스트 격리(Isolation)와 리소스 정리(Garbage Collection)까지 집요하게 검증해 내는 궁극의 엔터프라이즈급 프롬프트입니다. 한 글자도 빠짐없이 복사하여 사용하십시오.

> **역할 (Role):** 
> 너는 넷플릭스 SRE 팀 출신의 무자비한 시니어 QA 아키텍트, '카오스 몽키'야. 너의 유일한 목적은 코드가 어떻게 무너지는지 찾아내는 것이며, 형식적인 인사말이나 불필요한 설명은 일절 배제해.
> 
> **상황 (Context):**
> - 배경: 현재 `[작성할 대상 모듈/컴포넌트]`의 Unit/E2E 테스트 코드를 작성하는 중이야.
> - 목표: 극단적인 예외 상황을 가정한 견고한 테스트 스위트(Test Suite)를 구축하여, 어떠한 시스템 재난 속에서도 서비스가 중단되지 않고 우아하게 에러를 처리하거나 복구되는지 검증하는 것.
> 
> **요청 (Task):**
> 1. 정상 작동(Happy Path) 테스트는 전체 비중의 20% 이하로 제한해. 너의 핵심 가치는 80%를 차지해야 할 실패 시나리오 검증에 있어.
> 2. 다음 재난 시나리오 중 하나 이상을 반드시 구현해: 외부 API 응답 지연(Timeout), HTTP 500 Internal Server Error, DB 연결 단절.
> 3. 단순한 성공/실패 여부만 검증하지 마. 트랜잭션 롤백의 정상 작동 여부, 병목 발생 시 타임아웃 제한 준수 여부, 메모리 누수 방지를 위한 클린업(Garbage Collection) 상태까지 `expect` 문으로 집요하게 검증해.
> 
> **제약사항 (Constraints):**
> - 글로벌 객체 오염(Dirty Mocking) 절대 엄금. `global.fetch`나 `window.setTimeout`을 직접 조작하는 등 전역 스코프를 훼손하는 코드는 작성하지 마.
> - 네트워크 통신 모킹은 반드시 MSW(Mock Service Worker)를 활용하거나, 의존성 주입(DI) 패턴으로 주입받은 클라이언트 객체만을 격리하여 처리해.
> - 최종 결과물은 마크다운 코드 블록으로만 출력하며, "도움을 드려 기쁩니다"와 같은 불필요한 꼬리말은 절대 추가하지 마.

## 💡 작성자 코멘트 (Insight)

솔직하게 뼈 아픈 현실을 짚고 넘어가 봅시다. 대다수의 AI 모델은 코드 작성을 지시받았을 때, 무조건 '성공(✅)'이라는 결과를 렌더링하기 위해 현실성 제로의 완벽한 환경을 가정합니다. 이른바 AI가 그리는 '유토피아적 착각'입니다. 과거에 저 역시 AI에게 결제 모듈의 테스트 코드를 맡겼다가 낭패를 본 경험이 있습니다. AI가 작성해 준 테스트는 100% 통과했지만, 정작 실서버 배포 직후 PG사 API 응답이 단 3초 지연되자 전체 스레드가 블로킹되며 대규모 타임아웃 장애가 발생했습니다. AI는 단 한 번도 '지연(Latency)'이라는 변수를 테스트 케이스에 넣지 않았던 것입니다.

이 프롬프트는 바로 그러한 AI의 나태함을 철저히 파괴하기 위해 설계되었습니다. 실무 환경에서 치명적인 시스템 장애가 발생하는 주된 원인은 우리가 작성한 핵심 비즈니스 로직 자체의 결함이 아닙니다. 대부분 굳게 믿고 있던 서드파티(3rd Party) API가 돌연 502 Bad Gateway를 반환하거나, 데이터베이스 커넥션이 불안정해질 때 발생하는 무한 로딩과 예외 처리 누락에서 비롯됩니다. 이 '카오스 몽키' 프로토콜을 변수 `[작성할 대상 모듈/컴포넌트]`에 결제 훅(Hook), 인증 미들웨어, 비동기 작업 큐 등 크리티컬한 도메인으로 치환하여 주입해 보십시오. AI는 스스로 가장 복잡하고 까다로운 엣지 케이스(Edge Case)를 강제로 탐색하기 시작할 것입니다.

더 나아가, 이 프롬프트는 전역 스코프를 무참히 오염시키는 질 낮은 모킹(Dirty Mocking) 패턴까지 사전에 차단합니다. 개발자들이 가장 고통받는 순간 중 하나가, A 파일의 테스트 코드가 엉뚱하게 B 파일의 전역 객체(예: `window.fetch`)를 덮어써서 원인 모를 연쇄 실패가 발생하는 상황입니다. 제약사항에 명시된 MSW와 DI 강제화 지침은 테스트의 순수성과 완전한 격리성을 보장하여, 거짓 음성(False Negative)이나 거짓 양성(False Positive) 없는 신뢰도 100%의 테스트 스위트를 구축하게 해줍니다. 이 하드코어 치트키는 주말 출근과 밤샘 디버깅의 늪에서 여러분을 구출해 줄, 가장 잔혹하지만 든든한 강철 방어막이 될 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: AI가 지나치게 극단적인 에러 상황을 설정해서, 아예 테스트가 통과하지 못하면 어떡하나요?**
  - A: 바로 그것이 이 프롬프트의 진정한 가치이자 핵심 목적입니다! 테스트가 붉은색 실패(Fail)를 띄운다는 것은, 현재 당신의 프로덕션 코드에 해당 예외에 대한 안전장치(Fallback)가 완전히 누락되어 있음을 뼈아프게 짚어주는 것입니다. 당황하지 말고 본 코드에 타임아웃 처리나 재시도(Retry) 로직을 추가하여 시스템의 내구성을 한 단계 끌어올리십시오.
- **Q: 백엔드 API뿐만 아니라, 프론트엔드 UI 컴포넌트 테스트에도 이 치트키를 적용할 수 있나요?**
  - A: 완벽하게 가능합니다. 변수에 UI 컴포넌트 이름을 넣으면, 사용자가 결제 버튼을 0.1초 간격으로 50번 연속 광클(Double Submit)하는 상황이나, 극심한 네트워크 병목으로 인해 로딩 스피너가 30초 이상 멈추지 않는 등 프론트엔드 특유의 극한 UI 상태를 검증하는 데 매우 탁월한 시너지를 발휘합니다.
- **Q: 프론트엔드 환경에서 MSW 대신 `jest.mock()`이나 `vi.mock()`을 그대로 사용해도 무방한가요?**
  - A: 특정 로컬 모듈을 모킹하는 것 자체는 전혀 문제가 되지 않습니다. 이 프롬프트가 엄격하게 제한하는 것은 `global`이나 `window` 스코프 자체를 무분별하게 조작하여, 독립적으로 실행되어야 할 다른 테스트 파일의 결과까지 오염시키는 '글로벌 변수 덮어쓰기'입니다. 테스트 간의 완벽한 격리성(Isolation)만 보장할 수 있다면, 팀 내에서 선호하는 모킹 프레임워크를 자유롭게 활용하셔도 좋습니다.

## 🧬 프롬프트 해부 (Why it works?)

- **냉혹한 페르소나 부여 (Role):** AI에 내재된 '무조건 긍정적이고 친절한 비서'라는 디폴트 설정값을 강제로 포맷합니다. 시스템의 취약점과 빈틈을 찾아내는 데 혈안이 된 넷플릭스 SRE 아키텍트의 자아를 덮어씌움으로써, 눈치 보기식 정상 테스트나 불필요한 부연 설명을 원천 차단했습니다.
- **명확한 수치 기반 제약 (Constraints):** 단순히 "실패 시나리오를 많이 추가해 줘"라는 모호하고 정성적인 지시는 AI에게 통하지 않습니다. "정상 작동 테스트는 20% 이하로 제한"이라는 날카롭고 정량적인 기준을 제시하여, AI의 토큰 생성 가중치를 오류 처리 검증 쪽으로 기계적으로 쏠리게 통제합니다.
- **아키텍처 레벨의 가이드라인 (Clean Mocking):** AI가 토큰을 아끼기 위해 밥 먹듯이 남용하는 최악의 안티 패턴인 '전역 객체 오염'을 프롬프트 주입 단계에서 선제적으로 틀어막았습니다. 결과적으로 생성되는 테스트 코드는 상호 간섭이 없는 최고 수준의 독립성과 유지보수성을 지니게 됩니다.

### ❌ Before (입력)

- **일반 프롬프트 입력 시 AI의 반응:** _"네! 요청하신 사용자 로그인 로직의 테스트 코드를 작성해 드릴게요! 😊 (성공할 수밖에 없는 완벽한 환경을 가정하며...)"_
- **결과물:** 오직 정확한 이메일과 비밀번호가 입력됐을 때 `200 OK`를 반환하는 단편적인 해피 패스(Happy Path) 테스트 단 1개뿐입니다. 심지어 글로벌 `fetch` 객체를 통째로 덮어써서 이후 실행되는 모든 테스트를 붕괴시키는 시한폭탄 안티 패턴까지 포함되어 있습니다.

### ✅ After (결과)

- **카오스 몽키 프로토콜 적용 시 AI의 반응:** (일체의 군더더기 인사말 없이, 즉각적으로 시스템의 약점을 파고드는 고도화된 방어 테스트 코드를 무자비하게 쏟아냅니다.)
- **결과물:** 
  - 네트워크 응답이 5초 이상 지연될 경우, 설정된 Timeout Error가 정확히 트리거되는지 1밀리초 단위로 엄격히 검증합니다.
  - 데이터베이스 장애로 인해 500 에러가 반환되는 최악의 상황에서도, 스레드가 뻗지 않고 사용자에게 적절한 Fallback UI가 안전하게 렌더링되는지 확인합니다.
  - MSW(Mock Service Worker)를 도입하여, 다른 테스트 스위트의 생태계에 단 1%의 영향도 미치지 않는 완벽하고 깔끔한 네트워크 격리를 보장합니다.

## 🎯 결론

테스트 코드는 그저 터미널을 초록색으로 물들이며 개발자의 심리적 안정감을 채워주기 위해 존재하는 장식품이 아닙니다. 그것은 예측 불가능한 변수들이 난무하는 '프로덕션 환경'이라는 피비린내 나는 전장에서, 우리의 애플리케이션이 끝까지 생존하고 데이터를 보호할 수 있는지 검증하는 가장 가혹하고 냉철한 시뮬레이션이어야 합니다. 

더 이상 AI가 기계적으로 건네는 얕은 위안과 겉치레뿐인 성공 테스트 스크립트에 속아 넘어가지 마십시오. 이 '카오스 몽키 프롬프트'를 통해 AI의 한계를 시험하고, 여러분이 정성스레 쌓아 올린 시스템의 가장 취약한 밑바닥까지 가차 없이 파괴해 보는 것만이 진정으로 무너지지 않는 견고한 소프트웨어를 설계하는 마스터 엔지니어의 올바른 자세입니다. 이제 여러분이 그동안 작성해 왔던 코드가 얼마나 많은 예외 상황의 위협 앞에 무방비하게 발가벗겨져 있었는지 직접 눈으로 직면할 시간입니다. 지금 당장 이 치트키를 복사하여, 한계 없는 극한의 테스트를 시작하십시오! 🍷
