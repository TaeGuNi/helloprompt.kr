# 📝 Найдите уязвимости в коде с помощью ИИ! (Охотник за багами)

- **🎯 Целевая аудитория:** Junior-разработчики без наставника; специалисты, сомневающиеся в качестве кода перед релизом.
- **⏱️ Экономия времени:** С 3 часов отладки → до 10 минут.
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet (лучший в анализе кода), ChatGPT (GPT-4o).

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐⭐

> _"На моем компьютере все работает отлично... но не рухнет ли боевой сервер после деплоя?"_

Когда вы пишете код в одиночку, легко попасть в слепую зону из серии «я даже не знаю, чего я не знаю». Код может работать, но при этом быть медленным, уязвимым для хакеров или представлять собой запутанный «спагетти-код», который невозможно поддерживать.
Теперь у вас есть возможность круглосуточно проверять свой код у ИИ-ревьюера уровня ведущего инженера Google. Он не знает усталости и точно укажет на скрытые баги, неэффективную логику и критические бреши в безопасности.

---

## ⚡️ Краткая выжимка (TL;DR)

1. Не бросайте код ИИ без контекста: задайте ему четкую роль **ведущего инженера Google**.
2. Вместо банального «проверь код», укажите **4 строгих критерия оценки**: баги, безопасность, читаемость и производительность.
3. Не доверяйте слепо предложенному рефакторингу: требуйте подробных комментариев с объяснением причин изменений, чтобы учиться на них.

---

## 🚀 Решение: Промпт «ИИ-ревьюер кода»

### 🥉 Базовая версия (Basic)

Используйте этот промпт для быстрого сканирования кода на наличие критических ошибок.

> **Роль:** Ты — Senior Backend/Frontend разработчик с 10-летним опытом.
> **Задача:** Проверь следующий код на наличие критических багов и антипаттернов (anti-patterns), которые требуют немедленного исправления.
>
> ```plaintext
> [Вставьте ваш код]
> ```

<br>

### 🥇 Продвинутая версия (Pro)

Рекомендуется для вывода качества кода на уровень Production и получения объективных обоснований для рефакторинга.

> **Роль (Role):** Ты — ведущий инженер-программист (Principal Software Engineer) из Google с 15-летним опытом. Ты признанный эксперт в написании чистого кода (Clean Code) и оптимизации систем.
>
> **Контекст (Context):**
>
> - Язык программирования: `[Python / JavaScript / Java и т.д.]`
> - Фреймворк: `[React / Spring / Django и т.д.]`
> - Назначение: Эта функция отвечает за `[обработку авторизации пользователей и управление сессиями]`.
>
> **Входные данные (Input):**
>
> ```plaintext
> [Вставьте код для проверки]
> ```
>
> **Задача (Task):**
> Проведи строгий аудит моего кода по 4 ключевым аспектам и предоставь улучшенную, полностью рабочую версию кода.
>
> 1. **Баги (Bug):** Упущенная обработка исключений или скрытые ошибки времени выполнения (Edge cases).
> 2. **Безопасность (Security):** Уязвимости фреймворка, такие как SQL-инъекции, XSS, CSRF и т.д.
> 3. **Читаемость (Readability):** Соблюдение правил именования переменных/функций и принципа единственной ответственности (SRP).
> 4. **Производительность (Performance):** Лишние циклы, узкие места с точки зрения временной сложности (Big-O).
>
> **Ограничения (Constraints):**
>
> - Предлагая обновленный код, подробно объясни в комментариях (`//` или `#`), **почему (Why)** был проведен рефакторинг каждой конкретной строки.
> - Категорически запрещено искажать первоначальную бизнес-логику.
> - Оптимизируй код только встроенными средствами языка, без добавления сторонних библиотек.
>
> **Предупреждение (Warning):**
>
> - Не предлагай использовать устаревшие (deprecated) функции или синтаксис, в котором не уверен. Если чего-то не знаешь, так и скажи. (Предотвращение галлюцинаций)

## 💡 Комментарий автора (Инсайт)

Этот промпт был моим секретным оружием в начале карьеры. Он давал мне невероятное спокойствие в те моменты, когда я сомневался в собственном коде.
Особенно ценны пункты «Безопасность» и «Граничные случаи (Edge cases)» — ИИ автоматически улавливает то, что человеческий глаз легко пропускает, и это не раз спасало меня от крупных сбоев в Production.
Главное правило: никогда не копируйте слепо исправленный код. Требование «объясни причины рефакторинга в комментариях» добавлено не просто так. Читая эти объяснения, вы перенимаете образ мышления Senior-разработчика и стремительно прокачиваете свои навыки.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Можно ли скопировать и отправить весь исходный код рабочего проекта компании?**
  - О: **Категорически нет!** Утечка проприетарного кода — это серьезное нарушение безопасности. Обязательно маскируйте (например, `***`) API-ключи, пароли к базам данных и ключевую коммерческую логику. Отправляйте на проверку только минимальные, изолированные фрагменты: отдельные функции или классы.

- **В: ИИ всегда выдает 100% правильный код? Можно ли обойтись без тестирования?**
  - О: Нет, ИИ тоже подвержен галлюцинациям. Иногда он может выдумать несуществующую библиотеку или ошибиться в синтаксисе. Любой код, предложенный нейросетью, необходимо обязательно запустить в локальной среде и покрыть тестами для проверки работоспособности.

## 🧬 Анатомия промпта (Почему это работает?)

1. **Строгая ролевая модель (Perspective):** Задавая ИИ планку «ведущего инженера Google», мы заставляем его выдавать не просто «работающий код», а решения, соответствующие строгим стандартам Clean Code.
2. **Многомерные критерии оценки (Specific Goals):** Если попросить «просто проверить код», ИИ найдет лишь банальные опечатки. Четко обозначив 4 вектора (баги, безопасность, читаемость, производительность), мы получаем комплексный и глубокий аудит.
3. **Обучающий подход (Explanation):** Получение готового ответа не улучшает ваши навыки. Требование обосновать изменения заставляет ИИ выступать в роли терпеливого наставника (Tutor), помогая вам осмыслить архитектурные решения.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Входные данные: наивная логика запроса к БД)

```python
def get_user_data(user_id):
    # Прямая подстановка ввода пользователя в запрос (ОЧЕНЬ ОПАСНО!)
    sql = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(sql)
    return cursor.fetchone()
```

### ✅ После (Результат: безопасный и читаемый код)

```python
def get_user_data(user_id):
    """
    Извлекает данные конкретного пользователя из базы данных.
    """
    # [Security] Использование параметризованного запроса (binding) для защиты от SQL-инъекций
    # [Readability] SQL-запрос вынесен в отдельную переменную для лучшей читаемости
    query = "SELECT * FROM users WHERE id = %s"

    try:
        cursor.execute(query, (user_id,))
        result = cursor.fetchone()

        # [Bug] Обработка граничного случая: пользователь не найден
        if not result:
            return None

        return result

    except Exception as e:
        # [Bug] Логирование ошибок времени выполнения (в Production рекомендуется использовать модуль logging)
        print(f"Database error occurred: {e}")
        return None
```

**Резюме ревью:** Прямая конкатенация пользовательского ввода и SQL-запроса делает систему крайне уязвимой для SQL-инъекций. Уязвимость устранена путем использования параметризованных запросов (`%s`). Также добавлена конструкция `try-except`, чтобы сбой при обращении к базе данных не приводил к падению всего сервиса.
