## ⚡️ 3行核心摘要 (TL;DR)

1. **Look-aside (旁路缓存)：** 最经典的模式。先查缓存，没有再查数据库，随后将数据写入缓存。
2. **Write-back (回写模式)：** 应对极端写入负载的利器。先在内存中快速记录，再通过异步批处理（Batch）持久化到数据库。
3. **防御缓存击穿 (Cache Stampede)：** 核心在于应对缓存失效瞬间爆发的数据库连接风暴，需引入“互斥锁（Mutex Lock）”与“概率性提前重算（PER）”机制。

## 💡 作者深度解析 (Insight)

在引入 Redis 时，开发者最容易犯的致命错误就是“盲目且无底线地信任缓存基础设施”。请牢记，Redis 本质上是一个**“易失性”**的内存数据库。如果你把支付流水、用户密码等绝对不能丢失的“数据源（Source of Truth）”只存在 Redis 里，一旦宕机就是灾难性的事故。

Redis 中只能存放那些“随时可以丢弃，且能以极低成本从主数据库中恢复”的**副本数据**。此外，真正能体现“资深（Senior）”水平的代码，必须包含完善的后备降级（Fallback）逻辑：当 Redis 服务器彻底宕机时，系统不应该直接白屏报错，而是通过 `try-catch` 捕获异常，哪怕响应变慢，也要让请求穿透去读取底层数据库，保证核心业务的可用性。

## 🙋 常见问题解答 (FAQ)

- **Q: 为什么不能直接把数据缓存在服务器的本地内存（如全局变量或 Map）里？**
  - A: 如果只有单台服务器，这当然没问题。但在应对高并发而进行横向扩展（Scale-out）部署多台服务器时，用户每次请求落到不同的节点，就会读到不同的缓存状态，导致严重的“数据不一致”。这也是我们必须剥离出外部 Redis 作为全局分布式缓存池的根本原因。

- **Q: 数据的过期时间（TTL）该如何设定？有标准吗？**
  - A: 这完全取决于数据的“更新频率”和“业务敏感度”。对于时效性要求不高的系统公告，可以设置 1 小时甚至 1 天；用户个人主页信息通常设定为 5~10 分钟；而对于秒级变化的排行榜或金融行情，TTL 设置为 10 秒左右才是合理的。

- **Q: 为什么行业内通常推荐 Redis 而不是 Memcached？**
  - A: Memcached 的设计非常极简，只支持单纯的字符串键值对存储。而 Redis 则提供了 Hash、List、Set、Sorted Set 等极其丰富的内存数据结构。特别是利用 `Sorted Set`（有序集合），你可以直接在 Redis 内存中完成复杂的实时排行榜计算，速度快到令人发指，完全不需要去消耗主数据库的算力。
