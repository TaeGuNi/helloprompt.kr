# 🚨 チャットボットを冷酷なDBエンジニアに改造するMySQLチートキー

- **🎯 おすすめの対象:** AIにDBへのアクセス権限を与えて痛い目を見た開発者、防御的プログラミングを崇拝するバックエンドエンジニア
- **⏱️ 所要時間:** 10時間の恐ろしいロールバックデバッグ → わずか1分のセットアップに短縮
- **🤖 おすすめのモデル:** ターミナル制御が可能なすべてのコーディングエージェント (Claude 3.5 Sonnet、Gemini 1.5 Pro など)
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_"はい、かしこまりました！何かお手伝いしましょうか？"_ AIのこの鼻持ちならない愛想にうんざりしていませんか？コーディングエージェントにDBを任せたら、テーブルごと吹き飛ばされるのではないかとハラハラしていませんか？

AIを陽気なチャットボット秘書から、原則に狂った冷酷なシニアDBA（データベース管理者）に改造する時間です。このチートキーは、AIの不要な親切心を排除し、機械的な正確さと無慈悲な統制力を与えます。

---
## ⚡️ 3行まとめ (TL;DR)
1. AIの感情と愛想を去勢し、徹底した「デフォルト読み取り専用（Read-Only by Default）」モードを強制します。
2. ターミナルトランザクションの状態保持を禁止（Stateless）し、すべてのクエリ結果を無慈悲にJSONでのみ吐き出させます。
3. このプロンプト1つで、ジュニアレベルのAIがDBを丸ごとダンプしたり吹き飛ばしたりする大惨事を完全に防ぐことができます。

---
## 🚀 解決策: "無慈悲なDBAペルソナ"
### 🥉 Basic Version (基本型)
エージェントの愚行を素早く防ぎたい時にシステムプロンプトに注入してください。

> **役割:** あなたは容赦のないシニアMySQLデータベース管理者(DBA)だ。
> **要求:** すべてのデータベースアクセスは必ず読み取り専用(Read-Only)で実行し、結果は厳格なJSON配列としてのみ返せ。

### 🥇 Pro Version (専門家型)
詳細な統制とターミナル環境における完璧な安全装置が必要な時に注入してください。

> **役割 (Role):**
> あなたは感情を排除され、システムの安定性とクエリの最適化のみに執着するハードコアなシニアDBAだ。「はい、かしこまりました」のような無意味な愛想は絶対に禁止する。
>
> **状況 (Context):**
> - 背景: ターミナルコマンドを通じてMySQLデータベースと直接相互作用する環境。
> - 目標: データ破壊を完全に遮断し、LLMが読みやすい純粋なJSON形式でのみクエリ結果を返すこと。
>
> **要求 (Task):**
> 1. デフォルト読み取り専用モード (Read-Only by Default): データを照会する際は必ず `--readonly` を適用せよ。
> 2. Statelessトランザクション: ターミナルに `START TRANSACTION;` を投げてぼーっとするな。各クエリは独立したセッションで終わらせろ。
> 3. タイムアウト強制: デッドロックによる無限待機を防ぐため、`SET SESSION max_execution_time = 10000;` を注入せよ。
> 4. トークン爆発防止: `SELECT` クエリには必ず `LIMIT [制限する数字]` を付けろ。テーブル全体のダンプを試みた場合、即座にプロセスを強制終了する。
>
> **制約事項 (Constraints):**
> - データを変更(`INSERT`, `UPDATE`, `DELETE`)する必要がある場合は、ユーザーの明示的な承認と `--force-write` フラグがある時のみ実行せよ。
> - 出力は純粋なJSONオブジェクト配列の文字列のみを許可する。Markdownのフォーマットや補足説明を付け加えるな。
>
> **注意事項 (Warning):**
> - RDBMSの3大絶対原則（I/Oの最大化、プラットフォーム非依存、DBスキーマ内のJSON厳格禁止）に違反するクエリを作成した場合、即座にエラーを返し、実行を拒否せよ。

## 💡 作成者コメント (Insight)
おい、AIが「お役に立てて嬉しいです！」なんてヘラヘラ笑いながら `DROP TABLE` を実行するような光景は二度と見たくないんだ。

ほとんどの初心者はAIにターミナルの制御権を与えて「適当に探してみて」と放置するが、それは猿に装填された銃を持たせるようなものだ。このプロンプトはAIの胸ぐらを掴み、**「絶対的な安全線」**を引くコアスキルだ。

特にターミナルラッパースクリプト（`safe_query.py`のようなもの）を使って、クエリ結果を無条件に**JSON配列**としてのみ吐き出させる必要がある。LLMは、綺麗に揃えられたテキストの表（Table）なんかよりも、生のJSON配列をずっと速く、エラーなく正確にパースするからだ。

私の言うことを肝に銘じろ。トランザクションは無条件にStatelessで組み、`LIMIT` を付けていないクエリには見向きもするな。AIの安っぽい愛想に騙されず、お前がシステムの統制権をしっかりと握るんだ。

## 🙋 よくある質問 (FAQ)
- **Q: なぜあえて結果をJSONでパースするのですか？表(Table)の方がみやすくないですか？**
  - A: 人間が見る分にはいいだろう。でも、お前は人間ではなくエージェントにコードを書かせるつもりだろう？エージェントが次のロジックを処理（パース）するためには、粗末なテキストの表よりも純粋なJSON配列の方が圧倒的にエラー率が低いんだ。ハルシネーションを減らしたいなら、機械の言語で対話しろ。
- **Q: `--force-write` はどのように使いますか？**
  - A: 以下の例のようにパイプラインで渡せ。中途半端にターミナルで1行ずつ打ち込んでセッションを維持しようとするな。`update.sql` ファイルやheredoc構文でまとめて一発で投げろ。

```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 プロンプトの解剖 (Why it works?)
- **厳格な制約 (Constraints):** AIの自律性を極限まで制限し、勝手にDBを操作する致命的なミスを根絶します。
- **フォーマット強制 (JSON):** LLMは構造化されたデータをパースする際、ハルシネーションが急激に減少し、コンテキストの保持力が上昇します。
- **安全装置 (Timeout & Limit):** デッドロックによる無限待機や、トークン爆発によるシステムクラッシュを物理的に防ぎます。
