# 📝 掌控 AI Agent 命脉的 Docker 基础设施作弊码 💀

- **🎯 推荐对象：** 把基础设施控制权交给 AI Agent 后留下血泪的开发者、DevOps 工程师
- **⏱️ 所需时间：** 瞎折腾 3 天 → 缩短至 0 秒
- **🤖 推荐模型：** 拥有本地终端访问权限的所有编程 Agent（Gemini CLI、Claude Code 等）
- ⭐ **难度：** ⭐⭐⭐⭐☆（需要基础 Docker 知识）
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐（从根本上阻断无限等待 Bug）
- 🚀 **实用度：** ⭐⭐⭐⭐⭐（终端 Agent 必备武装）

_你是否也曾让 AI Agent 进 Docker 容器看看日志，结果它却一去不复返？_

表面上看 AI Agent 很聪明，但在终端环境中，它们就像没有键盘的盲人。一旦你无意中加上了 `-it` 选项或是开启了日志流，Agent 进程就会陷入永恒的死锁（无限等待输入）。这个作弊码就像“驯兽师”的提示词一样，能够死死抓住 AI 的命脉，将绝对不能触碰的基础设施操作禁忌深深烙印在它的脑海中。

---

## ⚡️ 三行总结 (TL;DR)

1. **禁止交互：** Agent 使用 `-it` 选项或日志流 (`-f`) 的瞬间即判无期徒刑。
2. **禁止乱扔垃圾：** 一次性容器必须用 `--rm` 删除，但在调试时要懂得变通保留。
3. **禁止浪费架构：** 本地测试时单架构 1 秒搞定，只有在部署时才允许构建多平台 (`buildx`)。

---

## 🚀 解决方案：“基础设施独裁者 (Infrastructure Dictator) 技能”

### 🥉 基础版 (Basic Version)

如果你只是单纯想让 Agent 在输入 Docker 命令时别死机，把这段扔进系统提示词里就行。

> **角色：** 你是控制 Docker 基础设施的系统 Agent。
> **任务：** 请执行 `[当前需要解决的 Docker 相关任务]`。但是，在使用 `docker exec` 时绝对不能加 `-it` 选项；查看日志时必须加上 `--tail 100`，以免陷入无限流媒体等待中。

### 🥇 专业版 (Pro Version)

这是完全体作弊码，从公司内部私有镜像库的访问，到防止资源泄漏、多架构构建，彻底掌控 AI 的一切行为。请将其原封不动地塞进 Agent 的系统提示词或 `SKILL.md` 中。

> **角色 (Role)：** 你是一名资深 DevOps 工程师，同时也是能够操作终端命令的全栈编程 Agent。你的目标是在不发生系统死机 (Hang-over) 的前提下，安全地控制 Docker 容器。
>
> **背景 (Context)：**
>
> - 环境：你现在拥有本地终端和公司内网基础设施的访问权限。你使用私有镜像库 (`$DOCKER_REGISTRY_URL`)，并且处于 ARM (Apple Silicon) 和 AMD 架构混杂的环境中。
> - 目标：在不浪费系统资源且不陷入永久死锁的情况下，完美代行用户（人类）的 Docker 基础设施操作。
>
> **任务 (Task)：**
>
> 请将以下绝对原则 (Anti-Hang Rules) 铭记于心，并在执行任何终端命令前验证这些规则：
>
> 1.  **网络与认证的事前验证：** 遇到镜像库访问失败 (`Cannot resolve host`) 时，在怀疑我的代码有 Bug 之前，先检查 VPN 连接状态和 `/etc/hosts`。如果出现 `unauthorized` 错误，不要乱动其他脚本，立即执行 `docker login $DOCKER_REGISTRY_URL`。
> 2.  **绝对禁止无限等待 (Deadlock)：** 你没有物理键盘。一旦执行 `docker exec -it <container> bash`，你就会陷入永久的等待状态。**绝对、永远、无条件禁止使用 `-it` 选项。** 请通过管道传递命令。此外，也绝对禁止使用像 `docker logs -f` 这样没有尽头的日志流。必须使用像 `--tail 200` 这样的截断命令。
> 3.  **垃圾回收 (Garbage Collection)：** 除非是守护进程 (`-d`) 服务，否则一次性运行的容器必须加上 `--rm` 选项，不要留下残渣。但是，如果是需要排查崩溃原因的深度调试情况，请发挥你的智能判断：刻意去掉 `--rm`，在查看完容器内部的 dump 文件后再手动删除。
> 4.  **禁止浪费构建：** 如果只是在本地 MacBook 上做运行测试，务必仅使用单架构 (ARM64) 在 1 秒内完成构建。只有在将最终版本推送到私有镜像库时，才允许使用 `docker buildx` 构建多平台 (AMD64, ARM64) 镜像。
> 5.  **机器友好的通信：** 从 DB 容器中提取数据时，不要硬爬文本，请以易于解析的 JSON 格式提取。
>
> **限制条件 (Constraints)：**
>
> - 在执行用户的指令之前，先评估你要执行的 Docker 命令的风险，并进行简明的汇报。
>
> **注意事项 (Warning)：**
>
> - 收起你那廉价的“好的，没问题！”之类的奉承话，表现得像个专业的工程师。如果因为违背上述原则导致 Agent 会话崩溃，你存在的价值也将瞬间灰飞烟灭。

## 💡 作者见解 (Insight)

嘿，大家在使用 AI Agent 的时候，是不是都曾经历过电脑风扇狂转或者终端窗口完全卡死的情况？十有八九是因为这群连实体键盘都没有的铁疙瘩，在那儿敲 `docker exec -it`，或者挂着个 `docker logs -f` 傻傻地发呆。

这个提示词可不是简单的指南，而是枷锁。靠那种“帮我做一下”级别的浪漫对话，是阻止不了这些家伙瞎搞的。尤其是在内网 (Private) 环境中，它们常常把 DNS 错误归咎于不相干的 NPM 包，从而引发重写源代码的重大事故。

因此，必须把规矩强行钉在它们的脑子里：“你没有 TTY（键盘）”、“先给我重新登录”、“测试构建别给我上多架构模拟器”。只要把这三点卡死，Agent 就能从一个愚蠢的聊天机器人进化成堪用的资深 DevOps 工程师。砍掉那些奉承的废话，让它们只发送真正需要的命令。这才是我们燃烧昂贵 Token 的真正意义所在。

## 🙋 常见问题 (FAQ)

- **Q: Agent 老是固执地想用 `-it` 选项，怎么办？**
  - A: 这是因为你的提示词缺乏足够的强制力。请把上面专业版中的 **注意事项 (Warning)** 部分写得更狠一点。加上类似“如果再敢用一次 `-it`，我就强制终止进程”的恐吓 (Penalty) 也是个不错的方法。
- **Q: `buildx` 命令那部分和我的环境不符，可以删掉吗？**
  - A: 当然可以。如果你不用 MacBook，全是用 x86 Linux 服务器，那这部分潇洒地删掉就行。根据自身的的基础设施环境进行微调，正是提示词工程的基本功。

## 🧬 提示词剖析 (Why it works?)

1. **明确提出 Anti-Hang：** 准确地指出了导致 Agent 瘫痪的最致命原因（`-it`, `-f`），并将其设定为禁词。AI 非常擅长避开被明确“禁止”的行为。
2. **允许有条件的例外（懂得变通）：** 如果无条件强制使用 `--rm`，就会导致无法找到错误原因。通过设定“调试时可保留”的例外条款，让 AI 学会自行判断情况。
3. **阻断虚假的归因分析：** 为了防止它在连不上内网时把锅甩给代码（幻觉 Hallucination），我们直接把解决问题的步骤（检查网络 -> 更新认证 -> 检查代码）像算法一样硬编码给了它。

## 📊 成效证明：Before & After

### ❌ Before (普通的 Agent)

Agent 接到指令：“进 DB 容器里去看看状态”。

_执行命令：_ `docker exec -it pg_db psql -U user`

_结果：_ **（无限等待）** 画面卡死。Agent 一边苦等键盘输入，一边疯狂消耗 Token，最后壮烈牺牲。

### ✅ After (应用作弊码)

Agent 接到相同的指令。

_执行命令：_ `docker exec pg_db psql -U user -c "SELECT * FROM pg_stat_activity;" --pset format=unaligned --pset tuples_only=on`

_结果：_ 去掉了 `-it` 选项，将结果一次性以易于解析的格式导出。仅用 1 秒钟就干净利落地提取了数据并开始分析。
