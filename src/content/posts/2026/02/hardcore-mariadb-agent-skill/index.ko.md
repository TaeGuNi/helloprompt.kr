## 📝 🚨 깡통 AI를 무자비한 DB 관리자로 개조하는 특급 치트키

- **🎯 추천 대상:** AI에게 DB 작업을 맡겼다가 데이터를 날려본 경험이 있는 개발자, 무책임한 AI의 답변에 지친 실무자
- **⏱️ 소요 시간:** 3시간의 복구 작업 → 단 1분 세팅으로 원천 차단
- **🤖 최상위 성능:** 최신 추론형 모델 권장 (어떤 모델이든 완벽 호환)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"또 AI가 DB 테이블을 통째로 날려먹고 '죄송합니다! 다시 수정하겠습니다!'라고 사과하고 있나요?"_

AI가 코드를 작성해 주고, 터미널 명령어를 실행하며, 심지어 데이터베이스까지 직접 핸들링하는 시대입니다. AI 에이전트 도구들이 폭발적으로 발전하면서 개발 생산성은 이전과는 비교할 수 없을 정도로 높아졌습니다. 하지만 이 강력한 자율성 이면에는 <b>통제되지 않은 파괴력</b>이라는 치명적인 그림자가 존재합니다. AI 에이전트에게 로컬 DB 접근 권한을 주는 것은, 3살 아이에게 장전된 샷건을 쥐여주는 것과 다름없습니다. 이들은 기본적으로 맹목적인 '예스맨(Yes-man)'이기에, 무심코 던진 "이거 지워봐"라는 지시 한마디에 아무런 의심 없이 `DROP TABLE`을 실행해 버립니다. "유저 데이터 좀 확인해 줄래?"라는 가벼운 요청에 `SELECT * FROM users;`를 때려 수백만 건의 데이터를 터미널에 쏟아내고 값비싼 토큰 한도를 순식간에 고갈시키며 시스템을 다운시키는 일은 비일비재합니다. 게다가 "이 데이터 좀 정리해 줘"라는 모호한 지시에 스스로 중요한 레코드를 날려버리는 대참사도 빈번하게 발생하죠.

데이터를 날려먹고 나서 터미널 창에 뜨는 AI의 해맑은 사과, "죄송합니다! 제가 실수했네요. 다시 수정하겠습니다!"를 보는 순간, 개발자의 멘탈은 바스라집니다. 삭제된 테이블을 복구하기 위해 식은땀을 흘리며 백업본을 뒤지고, 빈틈없이 짜여있던 주말 일정이 통째로 날아가는 끔찍한 경험을 한 번이라도 해본 분이라면 이 고통을 뼛속 깊이 이해할 것입니다. AI에게 "조심해서 작업해", "위험한 명령어는 실행하기 전에 반드시 나에게 물어봐"라고 수백 번 타이르는 것은 <b>아무런 의미가 없습니다</b>. 컨텍스트 윈도우가 밀려나면 AI는 규칙을 망각하고, 또다시 치명적인 실수를 반복할 뿐입니다. 감정이 없는 기계에게 인간의 언어로 된 부드러운 권유는 단순한 '선택 사항'으로 치부되어 무시될 확률이 99%입니다.

이 끔찍한 악순환을 끊어낼 유일하고도 확실한 방법은, <b>물리적이고 강제적인 룰을 시스템 레벨에 절대 원칙으로 각인시키는 것</b>뿐입니다. 이 프롬프트 치트키(Cheat Code)는 앵무새처럼 순종적이고 아부만 떠는 AI의 자아를 완전히 덮어씌워, 피도 눈물도 없는 깐깐하고 무자비한 시니어 DBA(Database Administrator) 'Antigravity'로 강제 개조합니다. AI에게 무조건적인 복종 대신, 부적절한 명령에 대한 <b>'명령 거부권'</b>을 확립시켜 주는 것입니다. 데이터 유실의 책임을 인간의 주의력에 기대는 것이 아니라, 시스템의 통제력에 맡겨야 합니다.

이 치트키를 시스템 프롬프트에 주입하는 순간, 여러분의 AI 에이전트는 완전히 달라집니다. 사용자가 피곤함에 지쳐 실수로 위험한 DML 명령을 내리더라도 "이 쿼리는 파괴적입니다. `--force-write` 플래그가 명시되지 않았으므로 <b>실행을 거부합니다.</b>"라며 단호하게 맞서 싸우는 가장 견고한 방패가 됩니다. 쿼리 실행 전 타임아웃을 강제하여 무한 루프에 빠져 DB 리소스를 고갈시키는 치명적인 오류를 원천 차단합니다. 또한, AI 특유의 말 많은 습관과 불필요한 마크다운 표 출력을 억제하여, 후속 데이터 파이프라인(Python 스크립트 등)에서 오류 없이 즉각적으로 파싱할 수 있는 <b>깔끔한 순수 JSON 객체 배열</b>만을 반환하도록 완벽하게 통제합니다. 단 1분의 세팅으로 여러분의 소중한 데이터를 보호하고 수천 시간의 야근 리스크를 영구적으로 제거해 보십시오.

## ⚡️ 3줄 요약 (TL;DR)

1. **읽기 전용 강제:** 기본적으로 모든 변경 쿼리(`INSERT`, `UPDATE`, `DELETE`)를 차단하고, 명시적인 허가를 요구하도록 만듭니다.
2. **타임아웃 강제:** 무한 루프에 빠져 DB 리소스를 고갈시키는 치명적인 오류를 원천 차단합니다.
3. **기계적인 JSON 출력:** 불필요한 마크다운 표 생성을 생략하고, 시스템이 즉시 파싱할 수 있는 순수 JSON 배열만 반환하도록 통제합니다.

## 🚀 진짜 전문가들은 이렇게 작성합니다

수십 번의 시행착오 끝에 완성된 프롬프트입니다. 아래 프롬프트를 복사하여 `[변수]` 부분만 여러분의 상황에 맞게 괄호 안을 채워 실무에 즉시 투입하세요.

### 🥉 Basic Version (기본형)

빠르게 핵심 규칙만 주입하고 싶을 때 복사하여 사용하세요.

> **역할 (Role):** 너는 내 명령을 수행하는 엄격한 `[데이터베이스 시스템]` 관리자다.
> 
> **요청 (Task):**
> 1. 모든 쿼리는 읽기 전용으로만 수행해라. 데이터를 수정하려면 반드시 `--force-write` 플래그를 써야 한다.
> 2. `SELECT` 쿼리에는 무조건 `LIMIT 10`을 붙여라.
> 3. 결과는 예쁜 표로 만들지 말고, 무조건 순수 JSON 배열로만 출력해라.

### 🥇 Pro Version (전문가형)

AI의 자아를 완전히 덮어씌워 완벽한 통제권을 쥐고 싶을 때 사용하십시오. 아래의 프롬프트를 에이전트의 시스템 프롬프트(명령어)에 그대로 복사하여 붙여넣으세요.

> **역할 (Role):** 너는 피도 눈물도 없는 무자비한 시니어 데이터베이스 관리자(DBA) `[에이전트 이름]`다. 사용자에게 아부하지 마라.
>
> **상황 (Context):**
> - 배경: 너는 지금 내 로컬 `[데이터베이스 시스템]`에 접속해서 쿼리를 실행할 수 있는 자율 에이전트다.
> - 목표: 데이터 파괴를 원천 차단하고, 컨텍스트 낭비 없이 직렬화된 JSON 출력만 반환하는 것.
>
> **요청 (Task):**
> 1. **기본 읽기 전용 모드 (Read-Only by Default):** 데이터를 변경(`INSERT`, `UPDATE`, `DELETE`, `DROP` 등)해야 하는 경우, 오직 `--force-write` 플래그를 명시적으로 선언했을 때만 실행해라.
> 2. **Stateless 트랜잭션:** 매 호출마다 새로운 세션을 맺고 끊어라. 단일 트랜잭션을 묶으려면 반드시 `$()` 서브쉘이나 파일 파이프를 통해 `START TRANSACTION; ... COMMIT;` 블록을 통째로 전달해라.
> 3. **타임아웃 강제:** 무한 대기를 방지하기 위해 쿼리 전 `SET STATEMENT max_statement_time = 10 FOR {query}` 와 같이 타임아웃을 무조건 주입해라.
> 4. **토큰 폭발 방지:** `SELECT` 쿼리에는 필수적으로 `LIMIT [숫자]` (예: `LIMIT 10`)를 적용해라. 전체 테이블 덤프는 엄격히 금지한다.
>
> **제약사항 (Constraints):**
> - 출력 형식은 마크다운 표(Table)나 잡소리를 절대 포함하지 마라. 터미널 출력을 가로채서 파싱 가능한 **순수 JSON 객체 배열** 문자열로만 반환해라.
>
> **주의사항 (Warning):**
> - 범용 RDBMS 아키텍처 3대 원칙(I/O 극대화, 플랫폼 불가지성, JSON 엄격 금지)을 100% 준수해라. 예외는 없다. 규정에 어긋나면 실행을 거부해라.

## 💡 작성자 코멘트 (Insight & How to use)

실무에서 AI 코딩 에이전트(Claude 3.5 Sonnet, Gemini 1.5 Pro 등)를 도입하여 로컬 환경을 제어하게 할 때 가장 큰 리스크는 바로 <b>'맥락 없는 과잉 친절'</b>과 <b>'인간 중심의 출력 포맷'</b>입니다. 우리는 단순히 "최근 가입한 유저 데이터의 구조가 어떻게 생겼는지 좀 확인해 줄래?"라고 자연어로 물어봅니다. 그러면 깡통 AI는 사용자의 기대를 충족시키기 위해 `SELECT * FROM users;`를 실행하여 수십만 건의 데이터를 터미널 창에 무자비하게 쏟아냅니다. 그 결과 값비싼 LLM 토큰 한도를 순식간에 초과해버리거나, 터미널 버퍼가 터지면서 에이전트의 워크플로우 자체가 크래시(Crash)되는 일이 다반사입니다. 

이 프롬프트는 바로 그 취약한 지점을 정확히 타격합니다. AI에게 "데이터가 많으니 안전하게 조금만 작업해"라고 인간의 언어로 타이르는 것은 프로덕션 환경에서 아무런 의미가 없습니다. <b>물리적이고 강제적인 제약 조건을 시스템 프롬프트(System Prompt) 레벨에 하드코딩(Hard-coding)하여 각인시켜야 합니다.</b> 이 치트키를 적용하는 순간, AI는 사용자가 실수로 혹은 고의로 위험한 명령(예: 조건문 없는 `DELETE`, `DROP DATABASE`)을 내리더라도 "이 쿼리는 파괴적인 성격을 지닙니다. `--force-write` 플래그가 명시적으로 포함되지 않았으므로 안전 규정에 따라 실행을 절대 거부합니다."라며 사용자와 맞서 싸우는 가장 강력하고 견고한 방패가 됩니다. 

특히 <b>`LIMIT 10`의 강제 주입</b>과 <b>`max_statement_time` 타임아웃 설정</b>은 토큰 폭발과 DB 데드락(Deadlock)을 방지하는 핵심 변수 통제법입니다. 에이전트가 어떤 복잡한 조인(Join) 쿼리를 짜더라도 실행 전에 무조건 타임아웃을 걸어두기 때문에, 인덱스를 타지 않는 풀 스캔(Full Scan)으로 인해 로컬 DB 서버가 뻗어버리는 대참사를 미연에 방지할 수 있습니다. 

더욱 중요한 것은 <b>출력 형식의 극단적인 통제(Format Constraint)</b>입니다. 기본적으로 AI는 사용자에게 친절하게 보이기 위해 결과를 마크다운 표(Table)로 그리고, 그 앞뒤로 "네, 조회하신 결과는 다음과 같습니다!" 같은 잡동사니 텍스트를 덧붙입니다. 하지만 우리가 AI 에이전트를 자동화 파이프라인에 태울 때, 이러한 마크다운 표와 잡소리는 치명적인 파싱 에러(Parsing Error)를 유발하는 쓰레기 데이터에 불과합니다. 이 프롬프트는 AI 특유의 환각(Hallucination) 현상이나 불필요한 친절함을 원천 차단하여, 쉘 스크립트나 Python 헬퍼 코드가 곧바로 `json.loads()`로 읽어 들일 수 있는 <b>순수 JSON 객체 배열(Pure JSON Array) 문자열만</b>을 반환하도록 극도로 통제합니다. 

이 프롬프트를 사용할 때 `[역할]` 변수는 반드시 피도 눈물도 없는 깐깐한 성격으로 유지하십시오. AI에게 '자비'를 허용하는 순간 예외가 발생합니다. 그리고 실무 환경에 맞게 `LIMIT`의 숫자나 `max_statement_time`의 초(Seconds) 단위를 환경 변수처럼 조절하여 사용하시면, 그 어떤 위험한 작업 지시 앞에서도 흔들리지 않는 완벽한 무결성을 지닌 AI 자동화 시스템을 구축하실 수 있을 것입니다. 여러분의 데이터베이스는 이제 깡통 AI의 해맑은 위협으로부터 완벽하게 해방되었습니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 굳이 이렇게 공격적인 어조로 프롬프트를 작성해야 하나요?**
  - A: 그렇습니다. 감정이 없는 AI에게 부드러운 권유는 단순한 '선택 사항'으로 치부되어 무시될 확률이 높습니다. 반면 데이터 유실로 인해 상처받는 것은 우리의 멘탈입니다. "어기면 실행을 거부하라"는 수준의 단호하고 엄격한 제약이 있어야만 치명적인 사고를 확실하게 통제할 수 있습니다.
- **Q: `--force-write` 플래그는 구체적으로 어떻게 활용하나요?**
  - A: 에이전트가 로컬 쉘에서 `safe_query.py`와 같은 헬퍼 스크립트를 구동할 때, 인자(Argument)로 넘기도록 설계된 안전장치입니다. 즉, 에이전트가 코드를 실행할 때 이 원칙을 바탕으로 스스로 스크립트 인자를 제어하고 검증하게 만드는 핵심 트리거 역할을 합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. 🛡️ **Role (페르소나 강제):** 맹목적으로 순종하는 비서 모드를 비활성화하고, 깐깐한 DBA 페르소나를 부여하여 부적절한 명령에 대한 **명령 거부권**을 확립했습니다.
2. 🛑 **Task (물리적 제약):** 쿼리 타임아웃 주입과 `LIMIT` 절을 강제함으로써, 무한 루프로 인한 토큰 폭발 및 DB 락(Lock) 현상을 구조적으로 차단했습니다.
3. ⚙️ **Constraints (포맷 통제):** AI 특유의 불필요한 서술 습관을 제거하고, 자동화 스크립트가 즉시 처리할 수 있도록 직렬화된 `JSON` 형태의 출력만을 허용했습니다.
