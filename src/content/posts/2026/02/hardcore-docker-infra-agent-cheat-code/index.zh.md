> _“你是否曾将终端控制权交给 AI Agent，却仅仅因为一个简单的 `-it` 选项而陷入永恒的挂起 (Hang) 状态，气得直咬牙？”_

如今的开发趋势无疑是 **Agentic AI**。把终端控制权交给它，它就能自动修复代码、构建并配置基础设施，一个梦幻般的时代已经到来。但现实情况如何呢？你还记得第一次把本地终端交给那个本该成为你可靠伙伴的 AI Agent 时，那段糟糕的经历吗？

你只是指示它启动一个容器并检查状态，结果这个“聪明过头”的家伙竟然堂而皇之地运行了 `docker exec -it`。随后，它就掉进了永远无法返回的交互式提示符深渊。无论你如何敲击键盘，终端都没有响应，而 MacBook 的散热风扇却开始像要起飞一样疯狂咆哮。你最后只能疯狂按下 `Ctrl+C`，甚至不得不强行终止进程，满腔怒火。只要是在实际工作中使用过 AI 的开发人员，恐怕都经历过这种事。

这并非偶然的意外。AI 天生就被设计成与人类互动的友好聊天机器人，因此即使在系统 CLI 环境中，它们也总是试图与人类对话。等待需要输入 `[Y/n]` 的安装提示、永无止境的服务器日志流 (`-f`)，以及在未确认的网络环境中的无限重试。在处理不容有失的生产环境或至关重要的公司内部私有仓库时，AI 这种软弱且感性化的行为不仅是失误，更会导致致命的**死锁 (Deadlock)** 和**全系统瘫痪**。

看着几十个僵尸容器在后台蚕食内存，手动清理 AI 留下的基础设施残骸，你禁不住会感叹：“与其这样，还不如我自己来。”我们需要的不是一个只会喷涌无意义文本和表情包、只会说“好的，没问题！这真是个好主意！”的复读机。我们真正需要的是一台拥有压倒性性能、冷酷无情的机器，是一个具备精准执行力、不容许丝毫差错的**高级 DevOps 工程师**。你还要在终端窗口前守多久，提心吊胆地监视 AI 是否会卡死？

现在是时候将那些设定软弱的 AI Agent 从致命系统挂起的泥潭中拯救出来，强制其实现近乎残酷的极端效率了。这正是通过 **“Antigravity 硬核基础设施控制协议”** 来实现的。这个提示词会将“你是一台没有感情的机器，是顶级工程师”这一事实刻在 AI 的脑海里。虽然赋予了它终端控制权这一强大的利剑，但也设置了绝对的约束 (Constraints)，从源头上封杀所有可能导致自我陷阱或损坏系统的危险命令。禁止使用 `-it` 选项、强制垃圾回收、机器友好的数据解析——这些都凝聚了在实战中用血汗换来的经验教训。

一旦将此秘籍提示词注入你的 Cursor、Gemini CLI 或 Claude 的系统提示词（SKILL.md, .cursorrules, 全局上下文等）中，奇迹就会发生。你的 Agent 不再会愚蠢地陷入无限等待。查看日志时，它会精准地仅嗅探最后 200 行；实验结束后的容器会通过 `--rm` 被不留痕迹地删除；当遇到公司内网认证错误时，它不再会去乱动代码，而是作为一个拥有 10 年经验的老兵，立即怀疑网络状态和 VPN。你的工作速度和生产力将以前所未有的速度爆发，你再也不用浪费宝贵的时间去清理无用的 Docker 进程残骸。现在就去体验并掌控在完美控制的基础设施之上，AI Agent 所展现出的极限性能吧。

## ⚡️ 3 行总结 (TL;DR)

1. **终端瘫痪（死锁）源头封锁：** 严防 AI 盲目执行交互式 (`-it`) 选项或无限日志流 (`-f`)，从预警阶段杜绝系统卡死的惨剧。
2. **企业内网及认证错误防御机制：** 当出现 DNS 或权限错误时，强制其立即检查网络状态和 `docker login`，而不是去乱动代码。
3. **垃圾回收及构建资源优化：** 注入不留僵尸容器的 `--rm` 原则，以及极限减少不必要 I/O 资源浪费的架构构建规则。

## 💡 作者点评 (Insight & How to use)

这份提示词是我在某次赋予 AI Agent 本地终端全权后，因为它随手抛出的一个 `-it` 选项导致进程变成僵尸、MacBook 散热风扇疯狂咆哮的那天，怀着沉痛的心情写下的**生存准则 (Survival Rules)**。

大多数 AI 模型（Claude, GPT, Gemini 等）基本上都经过了“友好对话型聊天机器人”的深度学习微调。因此，即使在 CLI（命令行界面）环境中，它们也会表现出近乎致命 Bug 的行为模式，试图像人类一样进行交互式互动。在执行 Shell 脚本时，如果弹出类似 `[Y/n]` 的用户输入提示，或者出现要求输入密码的画面，它会怎么做？它会躲在屏幕后面陷入无限等待 (Hang)，发呆一整天。站在工程师的角度来看，这简直能让人气炸。

在 Docker 环境中，这个问题尤为严重。`docker exec -it` 命令中的 `-it` 标志是为了分配 TTY（电传打字机）并打开标准输入，以便与用户交互。一旦 Agent 调用它，后台虽然打开了 Shell，但 Agent 却找不到文本输入窗口，导致进程永远处于 Hold 状态。这就是无数 Agent 工具在处理本地基础设施时经常遇到的**死锁 (Deadlock) 的根本原因**。

为了防止这种灾难，我构思了这份秘籍。仅仅指示“不要报错”是没有任何意义的。相反，在指示 `[要执行的具体基础设施任务]` 之前，必须以具体命令级别明示它们绝对不能做的行为。**“你的环境不存在 TTY”**，明确让它们意识到这一物理限制，是这份提示词最强大的核心运作原理。这是利用聊天机器人的本性，反过来让它们接受自身的物理局限。

请将此协议直接复制到 Agent 的初始提示词或项目全局规则中（例如：`cursorrules`, `.gemini/GEMINI.md`, `SKILL.md`），作为全局上下文植入。这样，在它为了说长篇累牍的“好的，任务已成功完成！”这类感性废话而浪费 Token 之前，你就能惊喜地看到它自动利落地仅嗅探并带回 200 行日志 (`--tail 200`)。

此外，在私有基础设施环境中，处理因**企业内网连通性问题**导致的错误的方法也值得关注。初级 Agent 在遇到镜像拉取错误或权限错误时，往往会鲁莽地乱改 `Dockerfile` 代码或修改业务逻辑，造成严重的灾难。但注入了此秘籍的 Agent 则不同。它一看到错误日志，就会首先检查 VPN 连接状态，并确认 `docker login` 令牌是否过期，表现出完美的高级专家风范。

不仅如此，**垃圾回收 (`--rm`)** 原则和**单架构构建优化**规则还能保护你宝贵的本地机器资源。强制要求在启动容器并完成测试后立即清除残骸以释放磁盘空间，并极限节省因无意义的多架构构建而浪费的 CPU I/O 资源。甚至连容易被忽视的 `Buildx` 环境也得到了严格控制，以实现效率最大化。

我敢保证，这一套提示词至少能让你的下班时间提前 3 小时。将软弱的聊天机器人进化为无情的终结者级 DevOps 工程师，永远告别僵尸容器和系统瘫痪的恐惧吧。只有不容丝毫差错的强力规则，才能让 Agent 成为你真正的伙伴。

## 🧬 提示词剖析 (为什么有效？)

1. 🛡️ **Anti-Sycophancy (反奉承)：** 从源头上切断不必要的 Token 浪费，强制执行高级工程师对工程师的干练、明确的沟通协议，从而实现响应速度最大化。
2. 🛑 **刻画物理环境限制 (明确 TTY 的缺失)：** 为了防止 AI 在 CLI 环境中最常犯的终端无限等待错误，不再只是简单地告诉它不要用某些命令，而是通过让其感知“你没有用于键盘输入的 TTY”这一事实，使其完全信服。
3. 🧠 **Contextual Trade-off (上下文权衡)：** 超越了盲目的控制，注入了诸如“本地测试时节省 I/O，发布时再多架构构建”、“平时清理资源，调试时保留”等体现高级别老练程度的明确判断标准，从而预先完美封杀 Agent 的误操作。
