## 📝 Stoat: «Анти-Copilot», удаляющий код, написанный LLM

- **🎯 Кому подойдет:** Разработчикам, страдающим от легаси-кода, ревьюерам, Senior-инженерам
- **⏱️ Экономия времени:** с 1 часа → до 5 минут
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o (рекомендуются модели, специализирующиеся на коде)

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _«Вчерашний идеальный код от ИИ сегодня превратился в спагетти? Настало время не генерировать код, а безжалостно его удалять.»_

Если 2024 и 2025 годы прошли под эгидой «ИИ-помощников для программирования», то 2026 год смело можно назвать эпохой «ИИ-чистильщиков». Наделавший немало шума в сообществе инженеров **Stoat** ломает привычные стереотипы. В то время как GitHub Copilot и Cursor бросают все силы на то, чтобы писать «как можно больше и быстрее», у Stoat совершенно иная миссия — **«удалять код»**.

Массовое внедрение LLM действительно спровоцировало взрывной рост продуктивности, однако сгенерированный нейросетями код зачастую страдает от многословности и дублирования логики. В этой статье мы возьмем на вооружение беспощадную философию Stoat и разберем мощнейший промпт, способный превратить любую LLM в безжалостного «санитара кода».

---

## ⚡️ Краткая выжимка (TL;DR)

1. Слепое доверие ИИ-ассистентам провоцирует раздувание кодовой базы и лавинообразный рост технического долга.
2. Философия Stoat «сажает код на жесткую диету»: она базируется на семантической дедупликации, избавлении от шаблонного мусора (boilerplate) и зачистке мертвого кода.
3. Применяя наш эксклюзивный промпт «Анти-Copilot», вы с легкостью превратите любимую LLM в бескомпромиссный инструмент для глубокого рефакторинга.

## 🚀 Решение: Промпт «Диета от Анти-Copilot Stoat»

### 🥉 Basic Version (Базовая версия)

Идеально подходит для ситуаций, когда необходимо экстренно сократить количество строк кода (LOC).

> **Роль:** Ты — самый строгий и бескомпромиссный Senior-разработчик, выступающий в амплуа «Анти-Copilot».
> 
> **Задача:** Проведи жесткое ревью предоставленного ниже кода и перепиши его. Твоя цель — сохранить 100% исходной функциональности, радикально сократив при этом объем строк. Безжалостно вычищай избыточные абстракции, дублирующуюся логику и мертвый код.
> 
> **Код:** `[Вставьте сюда исходный код для рефакторинга]`

### 🥇 Pro Version (Продвинутая версия)

Тяжелая артиллерия для безопасного устранения технического долга в production-окружении.

> **Роль (Role):** Ты — мастер «кодовой диеты» и ведущий инженер Stoat (Анти-Copilot), который физически не переносит избыточный код. Твое профессиональное кредо: «Лучший код — это отсутствие кода» (The best code is no code).
>
> **Контекст (Context):**
>
> - Фон: Имеется раздутый проект на `[Укажите язык и фреймворк]`, хаотично написанный Junior-разработчиками и ИИ-ассистентами.
> - Цель: Максимально улучшить читаемость кода, повысить его поддерживаемость и экстремально сократить общее количество строк (LOC).
>
> **Задача (Task):**
>
> 1. **Семантическое удаление дубликатов (Semantic De-duplication):** Найди и объедини участки логики, которые выглядят по-разному, но по факту решают одну и ту же задачу.
> 2. **Сокращение шаблонного кода (Boilerplate Reduction):** Используй самые современные синтаксические возможности языка, чтобы вычистить громоздкие паттерны.
> 3. **Охота на мертвый код (Dead Code Hunting):** Уничтожь все неиспользуемые вызовы, избыточное защитное программирование (defensive coding) и чрезмерные объявления типов.
> 4. Вместе с отрефакторенным кодом предоставь **отчет**, в котором укажи **процент удаленного кода (%)** и **ключевые векторы оптимизации**.
>
> **Ограничения (Constraints):**
>
> - Категорически запрещено нарушать или деградировать существующую бизнес-логику, а также ухудшать временную или пространственную сложность алгоритма.
> - Формат вывода: Строго блок кода Markdown. Основные отличия «До/После» оформи в виде маркированного списка.
> - Если безопасность какой-либо оптимизации вызывает у тебя сомнения, не применяй ее принудительно — просто оставь комментарий.
>
> **Вводные переменные:**
>
> - Язык/Фреймворк: `[Например: React, TypeScript]`
> - Исходный код:
>
>   [Вставьте сюда спагетти-код для оптимизации]

## 💡 Комментарий автора (Insight)

Разрабатывая этот промпт, я задался одним-единственным вопросом: «Как заставить ИИ испытывать жгучий стыд за сгенерированный им же громоздкий мусор и заставить его всё это удалить?»

Жестко ограничив контекст и задав бескомпромиссную персону «Анти-Copilot», мы подавляем излишнюю креативность нейросети, заставляя ее сфокусироваться исключительно на **«зачистке» и «оптимизации»**. Когда я впервые протестировал этот промпт на пухлом легаси-компоненте из сотен строк, произошло настоящее волшебство: более 40% кода просто испарилось, при этом функциональность осталась безупречной. Это ультимативное оружие, способное радикально снизить когнитивную нагрузку и усталость во время код-ревью.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Что делать, если ИИ «перестарается», удалит лишнее и спровоцирует скрытые баги?**
  - О: Несмотря на то, что в блоке «Ограничения» (Constraints) для Pro-версии строго зафиксировано требование сохранять исходную бизнес-логику, слепо доверять нейросетям — плохая практика. Всегда прогоняйте измененный код через существующие модульные тесты (Unit Tests). Вы обязаны убедиться, что ни один пограничный случай (edge case) не пострадал. Ручная валидация всё ещё обязательна.

- **В: Какие языковые модели справляются с этой задачей эффективнее всего?**
  - О: Рефакторинг требует феноменального понимания контекста и архитектуры приложения. Поэтому мы настоятельно рекомендуем использовать модели с топовыми аналитическими способностями — например, **Claude 3.5 Sonnet** или **GPT-4o**.

## 🧬 Разбор промпта (Why it works?)

1. **Радикальная смена ролей (Role Reversal):** Мы в корне ломаем дефолтные паттерны поведения LLM. Вместо услужливого ассистента, готового бесконечно генерировать строки, модель берет на себя роль «безжалостного Senior-инженера», чья главная страсть — стирать чужой код.
2. **Хирургическое целеуказание (Targeting):** Задав три конкретных вектора для атаки (дедупликация, уничтожение шаблонного мусора, зачистка мертвого кода), мы лишаем ИИ возможности вносить хаотичные и опасные правки.
3. **Психологическая установка (Mindset Shift):** Инъекция бескомпромиссного девиза *"The best code is no code"* на уровне системного промпта заставляет модель воспринимать экстремальное сокращение LOC как свою высшую профессиональную миссию.

## 📊 Доказательство: Before & After

### ❌ Before (Исходный код)

```typescript
import React, { useState, useEffect } from 'react';

const UserProfile = ({ user }) => {
  const [userName, setUserName] = useState('');
  const [userAge, setUserAge] = useState(0);

  useEffect(() => {
    if (user && user.name) {
      setUserName(user.name);
    }
  }, [user]);

  useEffect(() => {
    if (user && user.age) {
      setUserAge(user.age);
    }
  }, [user]);

  const handleNameClick = () => {
    console.log("Name clicked");
  }

  return (
    <div>
      {user ? (
        <div>
          <h1 onClick={handleNameClick}>{userName ? userName : 'Unknown'}</h1>
          <p>Age: {userAge ? userAge : 'N/A'}</p>
        </div>
      ) : (
        <div>Loading...</div>
      )}
    </div>
  );
};

export default UserProfile;
```

### ✅ After (Результат)

```typescript
import React from 'react';

const UserProfile = ({ user }) => {
  // Использование Early return для избавления от глубокой вложенности
  if (!user) return <div>Loading...</div>;

  return (
    <div>
      <h1 onClick={() => console.log("Name clicked")}>
        {user.name || 'Unknown'}
      </h1>
      <p>Age: {user.age || 'N/A'}</p>
    </div>
  );
};

export default UserProfile;
```

> **[Отчет Stoat]**

- **Экстремальное сокращение:** Объем кода уменьшен на **55%** по сравнению с оригиналом (с 36 до 16 строк).
- **Ключевые оптимизации:** Полностью ликвидирована избыточная синхронизация состояний через `useState` и `useEffect`. Внедрен паттерн раннего возврата (Early Return) для избавления от глубокой вложенности. Применены вычисления по короткой схеме (`||`), что позволило отказаться от громоздких тернарных операторов.
