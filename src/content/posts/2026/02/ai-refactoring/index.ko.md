## ⚡️ 3줄 요약 (TL;DR)

1. **철통같은 테스트 안전망 구축:** 본격적인 코드 수정에 앞서, AI가 빈틈없는 회귀 테스트(Regression Test)를 자동으로 생성해 완벽한 안전을 보장합니다.
2. **구조적 안티 패턴의 뿌리를 뽑다:** '신 객체(God Class)'나 얽히고설킨 복잡한 의존성을 날카롭게 식별하고, 즉각적인 의미론적 분리를 실행합니다.
3. **아키텍처 중심의 지능적 변환:** 인간이 마이크로서비스 전환 등 큰 그림(목표)만 지시하면, AI가 전술적이고 세밀한 코드 리팩토링을 완벽하게 수행합니다.

## 🚀 해결책: "무결점 AI 리팩토링 프롬프트"

### 🥉 Basic Version (기본형)

복잡한 설정 없이 코드의 가독성만 빠르게 끌어올리고 싶을 때 활용해 보세요.

> **역할:** 너는 `[20년 차 시니어 소프트웨어 아키텍트]`야.
> **요청:** 아래의 `[레거시 코드]`를 가독성 좋고 유지보수하기 쉬운 클린 코드로 리팩토링해줘. 수정된 이유도 주석으로 상세히 달아줘.


### 🥇 Pro Version (전문가형)

테스트 코드가 전무한 위태로운 환경에서, 시스템의 뼈대부터 근본적으로 바로잡고 싶을 때 강력히 추천합니다.

> **역할 (Role):** 너는 글로벌 빅테크 기업의 `[수석 소프트웨어 엔지니어이자 리팩토링 전문가]`야. SOLID 원칙과 디자인 패턴에 대한 깊은 통찰력을 가지고 있어.
>
> **상황 (Context):**
>
> - 배경: 담당자가 퇴사하여 테스트 코드가 전혀 없는 낡은 `[스파게티 코드]`를 유지보수해야 하는 상황이야.
> - 목표: 기존의 비즈니스 로직(동작)은 100% 동일하게 유지하면서, 코드를 모듈화하고 테스트 가능한 아키텍처로 변경하는 것.
>
> **요청 (Task):**
>
> 1. **테스트 코드 작성:** 리팩토링을 시작하기 전, 기존 코드의 동작을 검증할 수 있는 `[Jest/JUnit 등 테스트 프레임워크]` 기반의 포괄적인 단위 테스트(Unit Test)를 먼저 작성해.
> 2. **구조 분석 및 리팩토링:** 단일 책임 원칙(SRP)이 지켜지도록 함수와 클래스를 분리하고, 의미를 알 수 없는 변수명과 매직 넘버를 직관적으로 수정해.
> 3. **변경 리포트:** 변경된 부분과 그 이유(어떤 디자인 패턴을 적용했는지 등)를 마크다운 리스트 형태로 요약해서 알려줘.
>
> **데이터 (Data):**
> `[여기에 리팩토링할 코드를 붙여넣으세요]`
>
> **제약사항 (Constraints):**
>
> - 시간 복잡도와 공간 복잡도가 기존보다 악화되어서는 안 돼.
> - 불필요한 외부 라이브러리 도입은 철저히 배제해.
>
> **주의사항 (Warning):**
>
> - 비즈니스 로직의 결과값이 달라질 위험이 있는 변환은 절대 하지 마. 확신이 없다면 수정하지 말고 경고 메시지를 남겨. (할루시네이션 방지)

## 💡 작성자 코멘트 (Insight)

이 프롬프트에 숨겨진 진정한 가치는 **'테스트 코드 선행 작성'**을 AI에게 엄격하게 강제한다는 점에 있습니다. 2026년 현재, AI는 얽힌 레거시 로직의 맥락을 짚어내고 촘촘한 테스트 케이스를 직조해 내는 데 탁월한 능력을 발휘합니다. 안전망 없이 무턱대고 코드부터 뜯어고치려다간 걷잡을 수 없는 사이드 이펙트에 직면하기 십상입니다. AI가 먼저 작성해 준 단위 테스트를 돌려 초록색 불(Pass)을 두 눈으로 확인하세요. 그다음 리팩토링된 코드를 조심스럽게 입히는 겁니다. 단지 이 순서를 지키는 것만으로도, 개발팀 전체의 심리적 안정감과 생산성은 상상 그 이상으로 치솟을 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 방대한 규모의 프로젝트 전체를 한 번에 입력해도 모두 분석해 주나요?**
  - A: AI의 컨텍스트 윈도우 한계를 고려하고 예기치 않은 환각(Hallucination) 현상을 방지하기 위해, 한 번에 단일 클래스나 핵심 모듈 단위(약 500~1,000줄 내외)로 쪼개어 프롬프트를 실행하는 방식을 강력히 권장합니다.

- **Q: AI가 코드를 고치다 핵심 비즈니스 로직까지 조용히 망가뜨릴까 봐 우려됩니다.**
  - A: 바로 그 점을 방지하고자 Pro 버전 프롬프트의 주의사항(Warning) 섹션에 '로직 변경 엄격 금지'라는 강력한 제약을 걸어두었습니다. 무엇보다 선행 작성된 촘촘한 테스트 코드가 가장 확실하고 단단한 안전망이 되어줄 것입니다.

- **Q: 이 프롬프트는 특정 프레임워크나 프로그래밍 언어에 종속되나요?**
  - A: 전혀 그렇지 않습니다. `[테스트 프레임워크]` 변수 란에 Python의 PyTest, Java의 JUnit, 혹은 React의 React Testing Library 등 현재 팀에서 실무에 사용 중인 기술 스택을 명시하기만 하면 어떤 환경에서도 완벽하게 호환됩니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **철저한 방어적 프롬프팅 (Defensive Prompting):** 리팩토링의 제1원칙인 '동작 변경 절대 불가'를 제약사항에 명시함으로써, AI의 통제되지 않은 창의성이 귀중한 코드를 훼손하는 사태를 원천적으로 차단했습니다.
2. **안전망을 최우선으로 (Test-Driven Approach):** 코드를 건드리기 전에 무조건 테스트부터 작성하도록 순서를 강제했습니다. 이로써 레거시 시스템을 다룰 때 필연적으로 따르는 치명적인 리스크를 말끔히 거둬냈습니다.
3. **명확하고 권위 있는 페르소나 부여 (Role-Playing):** '수석 아키텍트'라는 고도로 전문화된 역할을 부여해, 단순한 코드 컨벤션(Lint) 수정을 넘어 아키텍처의 큰 그림을 그리는 깊이 있는 개선을 이끌어냅니다.

### ❌ Before (입력)

```javascript
// 수년 전 누군가 작성한 의도를 알 수 없는 코드
function process(d, x) {
  let r = 0;
  if (x == 1) {
    for (let i = 0; i < d.length; i++) {
      if (d[i].s == "A") r += d[i].v * 0.9;
    }
  } else if (x == 2) {
    for (let i = 0; i < d.length; i++) {
      if (d[i].s == "B") r += d[i].v * 0.8;
    }
  }
  return r;
}
```

### ✅ After (결과)

```javascript
// AI가 의미를 파악하고 분리한 테스트 가능한 클린 코드
const DISCOUNT_RATES = {
  PREMIUM: 0.9,
  STANDARD: 0.8,
};

const USER_TYPES = {
  VIP: 1,
  REGULAR: 2,
};

function calculateTotalDiscount(userData, userType) {
  if (!Array.isArray(userData)) throw new Error("Invalid data format");

  switch (userType) {
    case USER_TYPES.VIP:
      return _sumDiscount(userData, "A", DISCOUNT_RATES.PREMIUM);
    case USER_TYPES.REGULAR:
      return _sumDiscount(userData, "B", DISCOUNT_RATES.STANDARD);
    default:
      return 0;
  }
}

function _sumDiscount(data, statusTarget, discountRate) {
  return data
    .filter((item) => item.status === statusTarget)
    .reduce((sum, item) => sum + item.value * discountRate, 0);
}
```
