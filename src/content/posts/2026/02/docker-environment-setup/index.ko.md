![도커(Docker) 환경 구축, 설정 파일 하나로 3분 컷](/images/hooks/docker-environment-setup.jpg)

개발 환경과 운영 환경의 불일치는 모든 개발자가 공통으로 겪는 끔찍한 악몽입니다.
이 고질적인 문제를 해결하고자 도커(Docker)를 도입하려 해도, 낯선 `Dockerfile` 문법과 복잡한 `docker-compose.yml` 설정, 나아가 멀티 스테이지 빌드(Multi-stage Build)를 활용한 이미지 최적화라는 높은 진입장벽 앞에서 좌절하기 일쑤입니다.

하지만 이제 벅찬 도커 문법과 씨름할 필요가 없습니다. 현재 사용 중인 **프로젝트의 기술 스택**만 제시해 주세요. AI가 실무에 즉시 투입 가능한, 완벽하게 최적화된 프로덕션 레벨의 도커 설정 파일을 단 3분 만에 완성해 드립니다.

---

## ⚡️ 3줄 요약 (TL;DR) {#tl-dr}

1. **프로덕션 레벨 도커 세팅:** 단순한 로컬 개발용을 넘어, 실제 서비스 배포에 즉각 적용할 수 있는 견고한 설정 파일을 생성합니다.
2. **멀티 스테이지 빌드 적용:** 불필요한 빌드 의존성을 제거하고 이미지 용량을 획기적으로 경량화하여 배포 속도를 극대화합니다.
3. **보안 및 의존성 완벽 해결:** Non-root 유저 권한 설정부터 PostgreSQL, Redis 등 연동 데이터베이스 컨테이너 구성까지 한 번에 해결합니다.

## 🚀 해결책: "도커라이징 마스터 엔지니어"

### 🥉 Basic Version (기본형)

복잡한 설정 없이, 가장 기본적인 도커 파일의 뼈대만 빠르게 구축하고 싶을 때 활용하세요.

> **역할:** 너는 10년 차 시니어 DevOps 엔지니어야.
> **요청:** 내가 제시하는 기술 스택을 기반으로 애플리케이션 컨테이너화를 위한 `Dockerfile`과 `docker-compose.yml`을 작성해 줘.
> **기술 스택:** `[Node.js 20, NestJS, PostgreSQL]`

### 🥇 Pro Version (전문가형)

멀티 스테이지 빌드 및 철저한 보안 최적화가 요구되는 실제 프로덕션 환경에 투입할 때 활용하세요. 아래 프롬프트를 복사한 뒤, 대괄호 `[ ]` 안의 내용만 여러분의 프로젝트 환경에 맞게 수정하면 됩니다.

> **역할 (Role):** 너는 대규모 트래픽을 안정적으로 처리하는 글로벌 서비스의 시니어 DevOps 엔지니어야.
>
> **상황 (Context):**
>
> - 배경: 로컬 개발 환경과 프로덕션 배포 환경의 파편화를 막기 위해 애플리케이션을 도커라이징(Dockerizing)하고자 해.
> - 목표: 보안이 뛰어나고 경량화된 프로덕션용 컨테이너 이미지를 빌드하고, 애플리케이션과 연동되는 필수 인프라 환경까지 한 번에 실행하는 것.
>
> **요청 (Task):**
>
> 1. 하단에 제시된 기술 스택에 맞춰 프로덕션 레벨로 최적화된 **`Dockerfile`**을 작성해 줘. (단, 반드시 멀티 스테이지 빌드(Multi-stage build) 패턴을 적용할 것)
> 2. 애플리케이션 컨테이너는 물론, 연동되는 데이터베이스와 캐시 시스템을 모두 포함한 **`docker-compose.yml`**을 작성해 줘.
> 3. 도커 이미지 용량을 극한으로 최적화하기 위해 **`.dockerignore`** 파일에 등록해야 할 디렉터리 및 파일 목록을 함께 제시해 줘.
>
> **기술 스택 (Variables):**
>
> - 주 언어 및 프레임워크: `[Python 3.11, FastAPI]`
> - 패키지 매니저: `[Poetry]`
> - 데이터베이스: `[PostgreSQL 15]`
> - 인메모리 캐시: `[Redis 7]`
>
> **제약사항 (Constraints):**
>
> - 이미지 크기 최소화: 극도로 경량화된 베이스 이미지(예: `alpine` 또는 `slim`)를 채택할 것.
> - 보안 강화: 컨테이너 내부 애플리케이션 프로세스가 `root` 권한이 아닌 제한된 일반 사용자(Non-root user) 계정으로 실행되도록 엄격히 설정할 것.
> - 출력 형식: 불필요한 부연 설명은 일절 생략하고 `Dockerfile`, `docker-compose.yml`, `.dockerignore`의 완성된 코드블록만 마크다운 포맷으로 깔끔하게 출력할 것.

## 💡 작성자 코멘트 (Insight) {#insight}

이 프롬프트가 지닌 진정한 파괴력은 바로 **"멀티 스테이지 빌드(Multi-stage build)"**와 **"Non-root user"**라는 두 가지 핵심 제약 조건에서 나옵니다.
일반적으로 AI에게 "도커 파일 만들어줘"라고 단순하게 지시하면, 불필요한 패키지가 잔뜩 포함된 무거운 베이스 이미지를 사용하거나 보안에 치명적인 `root` 권한이 열려 있는 코드를 내놓기 십상입니다.

반면, 위 프롬프트를 활용하면 빌드 환경(무거운 의존성 패키지 포함)과 런타임 환경(실행에 필요한 최소한의 파일만 포함)을 완벽하게 분리하여 **이미지 용량을 기존 대비 1/10 수준으로 과감하게 압축해 줍니다.** 더불어 컨테이너 탈취를 원천 봉쇄하는 권한 분리 설정까지 자동으로 적용되므로, 생성된 결과물을 실무 CI/CD 파이프라인에 즉시 반영해도 무방할 만큼 견고합니다. 실제로 제가 새로운 프로젝트 인프라를 세팅할 때마다 항상 0순위로 꺼내 드는 '치트키' 프롬프트입니다.

## 🙋 자주 묻는 질문 (FAQ) {#faq}

- **Q: 회사 내부에서만 접근 가능한 프라이빗 패키지 저장소(Registry)를 사용 중인데, 이것도 반영할 수 있나요?**
  - A: 물론입니다! `[기술 스택]` 영역에 "사내 Nexus Repository를 통해 npm 패키지를 다운로드해야 함"과 같은 특수한 제약 조건을 추가해 보세요. AI가 `Dockerfile` 내부의 인증 프로세스와 환경 변수 주입 로직까지 빈틈없이 작성해 줍니다.

- **Q: 윈도우(Windows) 기반의 개발 환경에서도 문제없이 작동할까요?**
  - A: 그것이 바로 도커가 지닌 최고의 무기입니다. AI가 생성해 준 `docker-compose.yml` 파일을 실행하기만 하면 운영체제(Windows, macOS, Linux)와 무관하게 완전히 동일하고 격리된 컨테이너 환경이 완벽하게 구축됩니다.

- **Q: AI가 출력한 코드를 수정 없이 바로 실행해도 안전한가요?**
  - A: 95% 이상은 수정 없이 즉시 실행할 수 있을 만큼 완성도가 높습니다. 다만, `docker-compose.yml`에 명시된 데이터베이스 비밀번호나 포트 매핑 등의 민감한 환경 변수(Environment Variables)는 로컬 보안 환경에 맞춰 `.env` 파일로 분리하여 안전하게 관리하는 것을 강력히 권장합니다.

1. **역할 부여 (Role):** AI에게 "시니어 DevOps 엔지니어"라는 무거운 페르소나를 장착시킴으로써, 얕은 수준의 코드가 아닌 '시스템 보안과 리소스 최적화'까지 깊이 있게 고려한 전문가급 결과물을 도출하도록 강제했습니다.

## 📊 증명: Before & After

### ❌ Before (단순 요청 시)

> **"파이썬, 장고(Django), 포스트그레스큐엘(PostgreSQL) 도커 파일 만들어줘."**

이처럼 단편적으로 지시할 경우, AI는 용량이 1GB를 훌쩍 넘기는 비효율적인 베이스 이미지를 무비판적으로 사용합니다. 게다가 최고 관리자인 `root` 권한으로 컨테이너를 실행하게 만들어, 치명적인 보안 취약점을 고스란히 노출하는 위험한 코드를 생성하게 됩니다.

### ✅ After (프롬프트 사용 결과) {#dockerfile}

```dockerfile
# 1. Build Stage
FROM python:3.11-slim as builder

WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 2. Production Stage
FROM python:3.11-slim

WORKDIR /app
# 빌드 스테이지에서 설치한 패키지만 복사 (용량 획기적 감소)
COPY --from=builder /root/.local /root/.local
COPY . .

# 환경 변수 설정
ENV PATH=/root/.local/bin:$PATH
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# 보안을 위해 non-root 유저 생성 및 권한 부여
RUN adduser --disabled-password --gecos '' myuser && \
    chown -R myuser:myuser /app
USER myuser

CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

_결과 증명: 컴파일러와 무거운 빌드 툴이 깔끔하게 제거되어 100MB대로 다이어트 된 슬림 이미지, 해킹 위협을 방지하는 완벽한 유저 권한 분리, 불필요한 바이트코드 캐시 생성을 억제하는 옵션(`PYTHONDONTWRITEBYTECODE`)까지 프로덕션 필수 요건이 완벽하게 적용된 것을 확인할 수 있습니다._
