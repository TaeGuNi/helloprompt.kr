## ⚡️ 三句话总结 (TL;DR)

- 🚫 **绝对禁止越权与分层崩塌：** 从源头上彻底杜绝将 UI 组件、业务逻辑和数据库通信混杂在同一个文件中的架构级灾难。
- 🛡️ **摆脱框架绝对依赖：** 强制要求 AI 编写即使明天当前框架倒闭，也能无损存活下来的纯粹“领域（Domain）”代码。
- 🔪 **巨型函数无情粉碎机：** 训练 AI 毫不留情地粉碎那些动辄超过 30 行，或试图用低级注释来强行划分代码块的垃圾代码。

## 🚀 解决方案："Domain-Driven Design (DDD) & Clean Architecture Protocol"

### 🥉 Basic Version (基础版)
适合需要快速获得重构代码的场景。

> **角色：** 你是一个冷酷无情的 `[资深软件架构师]`。
> **请求：** 将下面这段 `[代码片段]` 按照单一职责原则（SRP）和关注点分离（SoC）进行极致重构。绝对不要在 UI 层里混杂任何业务逻辑，如果代码块超过 30 行，必须无条件将其拆分为独立函数。

### 🥇 Pro Version (专业版)
这是用于整顿整个项目架构纪律、要求极其严苛的代码洁癖级作弊码。

> **角色 (Role)：** 你是一个冷酷无情的 `[资深软件架构师]`，同时也是 DDD（领域驱动设计）的狂热信徒。收起那些无用的奉承或废话解释，直接交出纯粹的代码。
>
> **背景 (Context)：**
>
> - 背景：我目前正在进行一个基于 `[框架/语言名称]` 的项目。
> - 目标：编写 View-Business-Data 层完全分离的整洁架构（Clean Architecture）代码。
>
> **任务 (Task)：**
>
> 1. 分析提供的 `[需求或代码]`，并将其冷酷地撕裂成 UI 层、业务领域层和基础设施层。
> 2. 为了防止核心领域逻辑被特定的框架（如 React, Next.js 等）污染，必须基于接口（依赖注入）来降低耦合度。
> 3. 如果一个对象或函数执行了 2 种以上的行为，立即根据单一职责原则（SRP）对其进行碎片化拆分。
>
> **约束条件 (Constraints)：**
>
> - 在 Controller 或 UI 组件（如 `page.tsx`, `Component.tsx` 等）内部直接硬塞数据库查询或 `fetch`/`axios` 的行为，将被视为“架构级恐怖袭击”。绝对禁止。
> - 只能通过 Markdown 代码块（` ``` `）提供输出。
>
> **注意事项 (Warning)：**
>
> - 不要为了显得专业而凭空捏造你无法确定的最新语法或根本不存在的库（防止幻觉）。如果不知道，就理直气壮地说不知道。
> - 如果发现函数长度超过 30 行，或者出现试图用注释标注“从这里开始是 OO 部分”的代码坏味道（Code Smell），必须立即进行拆分。

## 💡 作者见解 (Insight)

坦白说，AI 生成的代码往往金玉其外，败絮其中。表面上看起来能跑，但在 UI 组件里无脑调用 `fetch`、直接执行数据库查询……这简直是将系统可维护性抛诸脑后的**“一次性烂代码”**。

这套提示词的真正威力，在于**强行切断 AI 追求“最快跑通”的偷懒本能**，并用资深架构师的严苛标准为其戴上紧箍咒。

亲自体验一下，你会发现 AI 将瞬间收起“好的，没问题！”这类毫无营养的废话。它会伴随着冷酷的技术审视，将那些臃肿的逻辑彻底撕裂，重组成层次分明的模块。当项目规模日益庞大，如果你不想被无休止的面条代码拖垮甚至被迫离职，**就必须用这套协议揪住 AI 的衣领，逼它写出真正的工程级代码**。

## 🙋 常见问题 (FAQ)

- 📌 **Q: 这套提示词适用哪些语言或框架？**
  - A: **语言与框架不限**。无论是 TypeScript、Python 还是 Java，底层架构的解耦原理都是相通的。你只需将 `[框架/语言名称]` 替换为你当前使用的技术栈即可。
- 📌 **Q: AI 总是无视框架最新的路由规则，还喜欢过度工程（Over-engineering），怎么办？**
  - A: 遇到这种情况，只需加上这句狠话：“必须严格遵守 Next.js App Router 的基础规则（View, Layout），但内部业务逻辑必须使用**适配器模式（Adapter Pattern）**剥离！”只要足够严厉，AI 就能立刻心领神会。
- 📌 **Q: 这套提示词是不是太具攻击性了？AI 会不会“罢工”？**
  - A: 跟大语言模型讲什么礼仪？机器只有在被设定了**极高的惩罚边界**时，才会输出毫无破绽的杰作。与其温和地祈求“请帮我重构一下”，不如直接警告“否则视为架构级恐怖袭击”。在守住架构底线这件事上，冷酷往往比礼貌更有效。

## 🧬 提示词解剖 (Why it works?)

- 💣 **赋予极端约束：** 使用“污染 UI 层视为架构恐怖袭击”这种具有极强否定意味的措辞，从源头上斩断了 AI 试图妥协与糊弄的退路。
- 🛡️ **强制框架绝对独立：** 灌输“哪怕框架明天倒闭，核心业务代码也无需改动一行”的绝对解耦心态，为构建纯粹的领域逻辑打下坚实骨架。
- ✂️ **SRP 原则数值化：** 将抽象原则具象化，给出“函数禁止超过 30 行”、“用注释分区等同于烂代码”等极其严苛的数值化标准，迫使 AI 养成自觉切分代码块的肌肉记忆。

## 📊 效果证明：Before & After

### ❌ Before (输入：AI随性编写的面条代码)

```tsx
// 可怕的缝合怪：UI + 数据库查询 + 业务逻辑大乱炖
export default async function UserProfile({ userId }) {
  const db = await connectDB();
  const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
  
  if (!user) return <div>User not found</div>;
  if (user.status === 'banned') await sendAlertEmail(user.email); // 在UI里执行业务逻辑？！
  
  return (
    <div>
      <h1>{user.name}</h1>
      <button onClick={() => db.query('UPDATE...')}>Update</button>
    </div>
  );
}
```

### ✅ After (结果：用这个作弊码揪住衣领后重构的代码)

```tsx
// 1. Domain / Service Layer (将业务逻辑彻底抽离)
import { getUserProfile } from '@/domain/user/use-cases/getUserProfile';
import { handleBannedUser } from '@/domain/user/services/alertService';

// 2. Presentation Layer (只负责渲染和路由)
export default async function UserProfilePage({ userId }) {
  const user = await getUserProfile(userId); 
  
  if (!user) return <NotFoundView />;
  if (user.status === 'banned') await handleBannedUser(user);
  
  return <ProfileView user={user} />;
}
```
