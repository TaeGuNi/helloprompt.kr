## ⚡️ Краткая выжимка (TL;DR)

1. Не просите ИИ просто «починить» код — назначьте ему роль «строгого Senior-ревьюера».
2. Дайте четкий контекст: приложите логи ошибок и объясните изначальную бизнес-логику.
3. Не копипастите вслепую — вникайте в причины и аргументы, которые ИИ приводит для каждого изменения.

## 🚀 Решение: Промпт "Безжалостный Senior-разработчик"

### 🥉 Базовая версия (Basic Version)

Используйте, когда нужно быстро найти причину ошибки.

> **Роль:** Ты — Senior `[Frontend/Backend]` разработчик с 10-летним опытом.
> **Задача:** Найди причину `[описание ошибки]`, возникающей в коде ниже, и предоставь исправленный код.
> **Код:** `[Вставьте сюда ваш код]`

### 🥇 Продвинутая версия (Pro Version)

Используйте, когда требуется не просто исправление бага, а улучшение архитектуры и рефакторинг по принципам чистого кода.

> **Роль (Role):**
> Ты — безжалостный, но невероятно компетентный Staff Software Engineer (Ведущий инженер) с 15-летним опытом работы в FAANG. Ты абсолютно бескомпромиссен в вопросах читаемости, производительности и поддерживаемости кода.
> 
> **Контекст (Context):**
> - Окружение: `[Укажите ваш стек технологий, например, React / Node.js / Python]`
> - Цель: `[Опишите бизнес-логику, которую должен выполнять этот код]`
> - Текущая проблема: `[Полный лог ошибки, падение производительности или другая проблемная ситуация]`
> 
> **Задача (Task):**
> 1. Проанализируй предоставленный код и укажи на фундаментальные проблемы (причины багов, антипаттерны и т.д.).
> 2. Проведи рефакторинг кода, основываясь на принципах чистого кода (SOLID, DRY и т.д.).
> 3. Предоставь полный исправленный код.
> 4. В формате комментариев к код-ревью объясни, **почему** ты сделал эти изменения, выделив 3 главных пункта.

> **Ограничения (Constraints):**

> - Обязательно учти обработку ошибок и пограничные случаи (edge cases).
> - Оформи вывод в виде списка.

> **Предупреждение (Warning):**
> - Не выдумывай несуществующие библиотеки или методы (никаких галлюцинаций).

## 💡 Комментарий автора (Insight)

Главная фишка этого промпта — суровая персона «безжалостного техлида» и жесткое требование обосновывать все изменения. 

Самая частая ошибка джунов — бездумный копипаст (Ctrl+C, Ctrl+V) сгенерированного кода. Да, баг исчезнет здесь и сейчас, но ваш скилл не вырастет ни на миллиметр. 

Этот промпт транслирует ИИ команду: не просто «исправь», а «научи меня». Внимательно изучайте найденные антипаттерны и визуально запоминайте правильные идиоматические конструкции (Idiomatic patterns). Если в блоке `Context` подробно расписать изначальную задумку, ИИ не только пофиксит синтаксис, но и укажет на архитектурные дыры. Не ищите оправданий своему коду — позвольте ИИ-наставнику разнести его в пух и прах и учитесь на этих ошибках. Это самый быстрый путь к уровню Middle и Senior.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Безопасно ли скармливать ИИ внутренний код компании как есть?**
  - A: Категорически нет! API-ключи, внутренние IP-адреса и конфиденциальную бизнес-логику нужно обязательно маскировать фиктивными (Dummy) данными, а имена переменных — обобщать. В вопросах безопасности компромиссов не бывает.

- **Q: Иногда сгенерированный код вообще не запускается. Что делать?**
  - A: Вы столкнулись с галлюцинацией (Hallucination). Без паники: просто скопируйте лог ошибки и верните его ИИ. Достаточно написать: «Твой код падает с такой ошибкой», и нейросеть сама всё исправит.

- **Q: Какая ИИ-модель лучше всего справляется с код-ревью?**
  - A: На 2026 год в задачах программирования безоговорочно лидирует Claude 3.5 Sonnet (и выше) — она глубже понимает контекст и делает более элегантный рефакторинг. ChatGPT Plus (GPT-4o) — тоже отличная альтернатива.

## 🧬 Разбор промпта (Why it works?)

1. **Задание «безжалостной» персоны:** Обычный ИИ слишком вежлив — он часто закрывает глаза на структурные костыли, лишь бы код просто запустился. Роль строгого Staff-инженера заставляет его проводить ревью безжалостно и по стандартам FAANG.
2. **Явный фокус на контексте (Context):** Машина читает символы, а Senior читает «намерения». Если объяснить ИИ изначальную задумку, он предложит принципиально иной, более грамотный подход (другие структуры данных, оптимальные паттерны).
3. **Требование объяснений (Why):** Главный драйвер вашего профессионального роста. ИИ лишается возможности просто швырнуть в вас готовым кодом — он вынужден аргументировать решения и передавать вам свой «опыт».

## 📊 Доказательство: До и После (Before & After)

### ❌ Before (Ввод)

```javascript
// Код для получения и обработки данных пользователя (Пример спагетти-кода)
async function getU(id) {
  let res = await fetch('/api/user/' + id);
  let d = await res.json();
  if(d) {
    if(d.status === 'active') {
      let r = await fetch('/api/roles/' + d.roleId);
      let rd = await r.json();
      return { n: d.name, r: rd.name };
    } else {
      return null;
    }
  }
}
```

### ✅ After (Результат)

```javascript
// Результат рефакторинга от ИИ Senior-инженера
async function getActiveUserRoleSummary(userId) {
  try {
    const userResponse = await fetch(`/api/user/${userId}`);
    if (!userResponse.ok) throw new Error('Failed to fetch user');
    
    const user = await userResponse.json();
    
    // Применение паттерна раннего возврата (Early Return) избавляет от вложенных if
    if (!user || user.status !== 'active') {
      return null; 
    }
    const roleResponse = await fetch(`/api/roles/${user.roleId}`);
    if (!roleResponse.ok) throw new Error('Failed to fetch role');
    
    const role = await roleResponse.json();
    // Использование понятных имен переменных (n, r -> name, roleName)
    return { 
      name: user.name, 
      roleName: role.name 
    };
  } catch (error) {
    console.error(`[getActiveUserRoleSummary] Error for userId ${userId}:`, error);
    throw error; // Предотвращение скрытия ошибок
  }
}
```

*(Комментарий ИИ: Вложенные условные операторы заменены на Early Return для повышения читаемости, имена переменных сделаны интуитивно понятными, а также добавлена отсутствовавшая обработка ошибок.)*
