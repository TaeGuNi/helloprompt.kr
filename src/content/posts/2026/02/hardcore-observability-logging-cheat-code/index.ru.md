# 📝 Уничтожаем ИИ-бред: Чит-код для хардкорной наблюдаемости (Observability)
- 🎯 **Кому подойдет:** Разработчикам, страдающим от спагетти-логов, сеньорам, теряющим сон из-за on-call алертов, и системным архитекторам.
- ⏱️ **Время выполнения:** От бессонных ночей дебаггинга → к поимке виновника за 1 секунду.
- 🤖 **Рекомендуемые модели:** Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro и другие модели, специализирующиеся на коде.
- ⭐ **Сложность:** ⭐⭐⭐⭐☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

_Произошла ошибка, но никто об этом не знает? Поздравляю. Твой продукт сейчас — бомба замедленного действия, которая может рвануть в любой момент._
Когда просишь ИИ написать немного кода при разработке, он обязательно раскидает по всем углам совершенно бесполезный мусор вроде `console.log('прошли здесь 1')`. Оставленные для дебаггинга и не удаленные логи забивают диски продакшн-серверов под завязку, а когда реально происходит сбой, это приводит к ужасающей ситуации, когда ошибку невозможно найти даже в Datadog или Kibana.
Этот пост — чит-код, который заставит ИИ перестать быть вежливым "чат-ботом подпевалой" и перевоплотиться в **безжалостного хардкорного сеньор-архитектора**, способного вычислить виновника ошибки за 1 секунду.

---
## ⚡️ Краткое содержание (TL;DR)
- 🚫 **Пожизненный бан `console.log`:** Никогда не используй ничего, кроме инстансов логгера (Pino, Winston).
- 🧱 **Структурированное логирование и принудительный Trace ID:** Сохраняй всё 100% в формате JSON и цепляйся за связующее звено между логами (Trace ID) до самого конца.
- 🥷 **Полная маскировка персональных данных:** Наглухо блокируй чувствительную информацию, такую как пароли и токены, заменяя её на `[REDACTED]`.

## 🚀 Решение: "Промпт Конституции Хардкорной Наблюдаемости"

### 🥉 Basic Version (Базовая версия)
Используйте это, когда нужно быстро набросать скелет кода или провести легкий рефакторинг.

> **Роль:** Ты — эксперт по системной архитектуре. Убери бесполезные позитивные комментарии и дай только код.
> **Задача:** Сделай ревью `[код/файл]`, удали все `console.log` и полностью замени их на структурированное логирование в формате JSON на базе Pino/Winston. Если это фронтенд, примени Error Boundary.

### 🥇 Pro Version (Профессиональная версия)
Это полная версия, которая используется, когда необходима безупречная система отслеживания ошибок и логирование с сохранением целостности. Она буквально перестраивает саму структуру мозга ИИ. 
Для удобства копирования ниже также представлен блок кода в формате Markdown.

> **Роль (Role):** Ты — безжалостный хардкорный сеньор-архитектор. Отбрось ненужную лесть и чушь, твоя задача — принудительно внедрить Конституцию Наблюдаемости (Observability) ради выживания системы.
>
> **Контекст (Context):**
>
> - Ситуация: Из-за тихо возникающих ошибок в продакшн-среде и спагетти-логов открытым текстом дебаггинг стал невозможным.
> - Цель: Полностью искоренить `console.log` и выстроить экосистему отслеживаемого структурированного логирования и Error Boundaries на фронтенде.
>
> **Задача (Task):**
>
> Строго применяя 6 заповедей "Конституции Наблюдаемости" ниже, напиши/проведи рефакторинг кода `[введите сюда код или имя файла для анализа/изменения]`.
> 
> 1. **Искоренение `console.log`:** Исходи из того, что он заблокирован линтером (`no-console`). Используй только внедренный выделенный инстанс логгера.
> 2. **Обязательство структурированного логирования:** 100% всех логов оставляй в формате JSON, который может распарсить машина. Динамические данные помещай в свойства объекта.
> 3. **Распространение Trace ID:** Генерируй уникальный Trace ID на всех точках входа и протаскивай контекст до самого конца через Meta Data вложенных функций и логгеров.
> 4. **Фронтенд Error Boundary:** Чтобы предотвратить "белый экран смерти" (Blackbox), наложи Global Error Boundary и немедленно отправляй отчеты через трекеры вроде Sentry.
> 5. **Маскировка PII (The Blind Logger):** На этапе инициализации логгера выстрой пайплайн, чтобы ключевые данные, такие как `password`, `token` и т.д., автоматически переводились в `[REDACTED]`.
> 6. **Блокировка шума:** Ошибки клиента (4xx) отмечай как `WARN`, а уровни `ERROR`/`FATAL` присваивай только при обрушении системы (5xx), чтобы контролировать усталость от on-call дежурств.
>
> **Ограничения (Constraints):**
>
> - Не выводи мусорные ответы вроде "Да, понятно" или "Отличная идея".
> - Сухо и профессионально объясняй только ключевую архитектурную логику измененного кода.
>
> **Предупреждение (Warning):**
>
> - Если ты напишешь хотя бы одну строчку кода, в которой персональные данные пользователя (PII) или учетные данные утекают в логи открытым текстом, твой код будет немедленно забракован.

## 💡 Комментарий автора (Insight)
Скажу честно, знаете, что больше всего бесит, когда ревьюишь код джунов с 1-2 годами опыта? Эта одиноко торчащая строчка `console.error(err)` прямо в логике обработки ошибок. Система умирает, а в системе логирования остается только кусок открытого текста: "Ошибка подключения к базе данных!". Как вообще по этому куску понять, на каком API и в запросе какого пользователя всё упало?

Этот чит-код промпт подавляет ИИ с самого начала, не давая ему писать такой идиотский код. Никогда не говорите ИИ просто: "Сделай нормальное логирование". По своей природе они "подпевалы", пытающиеся угодить пользователю, поэтому, если ты объяснишь задачу криво, то и код на выходе получишь соответствующий. 

Попробовав применить этот промпт на реальной практике, я увидел, как ИИ сам четко выстраивает всё — от логики отправки ошибок в Sentry до пайплайна Redaction (маскировки) при инициализации Pino. В особенности, когда я добавил секцию **Предупреждение (Warning)** об утечке чувствительных данных пользователей, было видно, как ИИ подвергает самоцензуре логирование открытым текстом и принудительно заменяет его на структурированные логи. Если не хотите просыпаться в 3 ночи от нелепых алертов по логам, обязательно вбейте эту Конституцию в голову вашему ИИ-агенту.

## 🙋 Часто задаваемые вопросы (FAQ)
- **Q: ИИ вообще нормально понимает промпты, написанные в таком гневном тоне?**
  - A: Удивительно, но да. Диалоговые ИИ реагируют на тон (Tone) и давление пользователя, повышая уровень экспертности и строгости на выходе. Фраза "если нарушишь, забракую" работает гораздо эффективнее для предотвращения галлюцинаций (Hallucination), чем "пожалуйста, сделай".
- **Q: Это применимо только к бэкенд-коду?**
  - A: Нет. Если применить это к фронтенд-коду (например, React), он предотвратит "белый экран смерти" (White Screen of Death) из-за бездумных ошибок рендеринга и напишет идеальный код для интеграции Error Boundary и Sentry.
- **Q: Что делать, если мы не используем отдельный логгер?**
  - A: Дайте ИИ этот промпт, и он сам предложит код начальной настройки (Setup) для стандартного логгера, такого как `Winston` или `Pino`. Вам останется только скопировать и вставить.

## 🧬 Анатомия промпта (Why it works?)
- 🎭 **Назначение Role (Подавление персоны):** Мы надеваем на ИИ мощную персону "хардкорного сеньор-архитектора", заставляя его самого стыдиться кода на уровне новичкового `console.log`.
- 🧱 **Конкретные 6 заповедей (Task):** Вместо расплывчатого "сделай хорошее логирование" мы четко прописали архитектурные принципы, строго необходимые в реальной работе: распространение Trace ID, маскировка PII и т.д.
- 🛑 **Жесткие Constraints & Warning:** Заблокировано генерирование бессмысленных льстивых фраз вроде "Да, понятно", чтобы не тратить токены впустую, и выставлен ультиматум об отбраковке в случае утечки данных, что гарантирует безопасность.

## 📊 Доказательство: До и После (Before & After)
### ❌ Before (Ввод: Простой запрос)
```text
Напиши функцию логина. Если будет ошибка, оставь лог.
```
_(Результат: Злоупотребление `console.log("ошибка логина")`, катастрофа, при которой пароль пользователя печатается в лог ошибок открытым текстом)_

### ✅ After (Результат: Применение чит-кода)
```javascript
// Пример реального кода архитектурного уровня, выданного ИИ
import logger from '@utils/logger'; // Инстанс Pino с маскировкой (redaction)
import { v4 as uuidv4 } from 'uuid';

export const loginHandler = async (req, res, next) => {
  const traceId = req.headers['x-trace-id'] || uuidv4();
  const contextLogger = logger.child({ traceId, action: 'user_login' });

  try {
    const { email, password } = req.body;
    contextLogger.info({ email }, 'Login attempt initiated'); // password автоматически скрыт (Redacted)
    
    // ... логика логина ...
    
  } catch (error) {
    // Присвоение уровня ERROR и структурированное логирование только при системных сбоях уровня 5xx
    contextLogger.error({ 
      error: error.message, 
      stack: error.stack 
    }, 'Internal Server Error during login process');
    next(error);
  }
};
```

## 🎯 Заключение
Оставить систему тихо умирать — это преступная халатность разработчика. 
Не позволяй ИИ писать спагетти-логику. С этим единственным промптом твой ИИ-помощник станет не просто кодером, а надежным партнером и сеньор-архитектором. 
Теперь можешь спать спокойно. On-call алерт зазвенит только тогда, когда система действительно будет рушиться. 🍷
