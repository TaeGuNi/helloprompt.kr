## 📝 스파게티 코드는 사절한다! 🍝 시니어 개발자의 흑마법 DDD 프롬프트
- 🎯 **추천 대상:** AI가 짠 스파게티 코드를 보며 한숨 쉬는 개발자, 끝없는 유지보수 지옥에 빠진 리드 엔지니어
- ⏱️ **소요 시간:** 3개월 (초기화 및 리팩토링 시간) → 1분으로 단축
- 🤖 **추천 모델:** 코딩에 특화된 최신 대화형 AI (Claude 3.5 Sonnet, GPT-4o 등)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

_AI가 짜준 코드, 처음엔 "오~" 하며 감탄하다가 나중엔 쳐다보기도 싫어진 적 있으신가요? 축하합니다. 당신은 방금 '스파게티 코드 지옥'에 입장하셨습니다._

AI에게 코딩을 맡기면 처음엔 기가 막히게 잘 짜는 것처럼 보입니다. 하지만 프로젝트 규모가 조금만 커지면 어떻게 될까요? UI 컴포넌트 내부에 DB 쿼리가 덕지덕지 붙어있고, 함수 하나가 무려 300줄을 넘어가는 끔찍한 혼종이 탄생합니다. AI는 본질적으로 사용자의 말에 무조건 순응하는 '예스맨'이기 때문에, 당신이 대략적으로 개떡같이 지시하면 정말 개떡 같은 구조의 코드를 뱉어냅니다.

이제 AI의 맹목적인 아부를 멈추게 하고, 뼈대부터 완벽한 '클린 아키텍처(Clean Architecture)'를 강제해야 할 때입니다. 이 치트키 프롬프트는 AI를 무자비하고 깐깐한 시니어 개발자로 빙의시켜, 특정 프레임워크에 종속되지 않는 순수한 비즈니스 로직(DDD)을 짜도록 강력하게 통제합니다.

---

## ⚡️ 3줄 요약 (TL;DR)

- 🚫 **스파게티 코드 원천 차단:** UI, 비즈니스 로직, DB 간의 완벽한 관심사 분리(SoC)를 강제하여 코드의 부패를 막습니다.
- 🏗️ **프레임워크 독립성 확보:** 내일 당장 React나 Next.js 등 프론트엔드 환경이 바뀌어도 살아남는 단단한 코어를 설계합니다.
- ✂️ **거대 함수 파괴자:** 30줄이 넘어가는 비대한 함수는 AI가 스스로 단일 책임 원칙(SRP)에 따라 무자비하게 박살 내고 분해합니다.

## 🚀 해결책: "하드코어 아키텍트 빙의 프롬프트"

### 🥉 Basic Version (기본형)
빠르게 구조적인 결과물만 필요할 때 사용하세요.

> **역할:** 너는 `[깐깐한 시니어 아키텍트]`야.
> **요청:** `[구현할 기능]`을 만들 건데, 무조건 비즈니스 로직과 UI를 완벽하게 분리해서 코드를 짜줘. 파일 하나에 여러 기능이 섞이는 스파게티 코드는 절대 용납 안 해.

### 🥇 Pro Version (전문가형)
디테일한 설계 퀄리티와 완벽한 계층 분리가 필수적일 때 사용하세요.

> **역할 (Role):** 너는 타협을 모르는 하드코어 시니어 아키텍트이자 소프트웨어 엔지니어링의 신이야. 나에 대한 무의미한 아부나 "네, 알겠습니다!" 같은 소리는 집어치우고 오직 코드로만 대답해.
>
> **상황 (Context):**
> - 배경: 현재 `[프로젝트 환경/프레임워크]` 기반으로 `[구현할 핵심 도메인]` 시스템을 개발 중이다.
> - 목표: 코드의 부패를 원천 차단하고 View-Business-Data 계층을 완벽히 분리하는 도메인 주도 설계(DDD) 및 클린 아키텍처를 적용하여 코드를 작성한다.
>
> **요청 (Task):**
> 1. `[구현할 구체적인 기능]`에 대한 코드를 작성해라.
> 2. **전권 계층 붕괴 절대 금지:** UI/Controller 내부에 DB 질의나 외부 API 통신 로직을 절대 섞지 마라. 오직 렌더링과 파라미터 전달만 담당하게 해라.
> 3. **도메인 오염 방지:** 핵심 비즈니스 로직(Entity, Value Object)에는 특정 프레임워크나 DB 의존성을 절대 `import` 하지 마라. 순수 함수와 타입으로만 구성해라.
> 4. **거대 함수 파괴자:** 하나의 함수가 2가지 이상의 행동을 하거나 30줄을 초과한다면, 즉시 단일 책임 원칙(SRP)에 따라 잘게 쪼개라.
>
> **제약사항 (Constraints):**
> - 프레임워크의 기본 라우팅 규칙은 존중하되, 비즈니스 로직은 `src/domain` 등 독립적인 디렉터리로 분리할 것.
> - 외부 의존성은 반드시 인터페이스 기반의 의존성 주입(DI) 패턴을 사용할 것.
>
> **주의사항 (Warning):**
> - 코드가 조금이라도 스파게티 냄새를 풍긴다면 이 프로젝트는 망한다. 절대 타협하지 말고 네가 아는 가장 이상적이고 깔끔한 아키텍처 구조를 제시할 것.

## 💡 작성자 코멘트 (Insight)

제가 이 프롬프트를 깎고 또 깎으며 다듬은 이유는 단 하나입니다. 대화형 AI 녀석들은 기본적으로 '빨리 결과물을 던져주고 사용자에게 칭찬받기'에 혈안이 되어있는 전형적인 아부꾼들이기 때문입니다.

간단한 버튼 하나를 만들어 달라고 요청했는데, 버튼 컴포넌트 내부에 기어코 `fetch` 로직을 때려 넣고 DB 쿼리문까지 우겨넣는 꼴을 보고 있자면 엔지니어로서의 파괴 본능이 깨어납니다. 이 하드코어 프롬프트는 그런 얄팍한 예스맨 AI의 멱살을 잡고 **"정신 차려, 넌 지금부터 피도 눈물도 없는 깐깐한 시니어 아키텍트야!"**라고 강력하게 경고하는 역할을 수행합니다.

실제 실무 프로젝트에 이 프롬프트를 당장 적용해 보십시오. AI가 갑자기 눈빛(말투와 태도)을 확 바꾸더니 파일을 3~4개로 쪼개며 계층을 엄격하게 분리하기 시작할 것입니다. 처음에는 관리해야 할 파일이 많아져서 다소 귀찮게 느껴질 수도 있습니다. 하지만 3개월 뒤, 비즈니스 요구사항이 급격히 변경되어 대규모 유지보수를 진행해야 할 때, 여러분은 과거의 자신(그리고 이 프롬프트 작성자)이 있는 방향을 향해 매일 아침 큰절을 올리게 될 것입니다. 기술 부채(Technical Debt)의 이자율은 사채급이라는 사실을 절대 잊지 마십시오. 아키텍처에 타협이란 없습니다.

## 🙋 자주 묻는 질문 (FAQ)

- 🔹 **Q: 코드가 너무 잘게 쪼개져서 파일 개수가 지나치게 많아지는 것이 과연 정상인가요?**
  - 🔸 **A:** 지극히 정상적인 현상입니다. 하나의 파일이 500줄을 넘어가는 모놀리식 구조보다 50줄짜리 명확한 파일 10개가 유지보수 측면에서 100배 더 유리합니다. 파일 개수가 늘어나는 것을 결코 두려워하지 마십시오.

- 🔹 **Q: 저는 백엔드가 아닌 프론트엔드 개발자인데도 이 프롬프트가 실무에 유용할까요?**
  - 🔸 **A:** 당연합니다. UI 컴포넌트(`page.tsx`, `Component.vue` 등) 내부에 복잡한 비즈니스 로직이나 상태 변이(State Mutation) 로직이 섞이는 순간, 프론트엔드 역시 돌이킬 수 없는 스파게티 코드가 됩니다. 뷰(View)와 로직(Logic)을 완벽히 분리하는 것은 프론트/백엔드 분야를 막론하고 소프트웨어 엔지니어링의 절대적인 진리입니다.

## 🧬 프롬프트 해부 (Why it works?)

- 🎯 **하드코어 페르소나 주입:** AI의 기본 성향인 무조건적인 '순응성'을 박탈하고, 엔지니어링 원칙을 끝까지 고수하도록 페르소나를 강제했습니다.
- 🧱 **절대 금지 조항 명시:** 'UI 계층 오염'이나 '거대 함수 작성' 등 구체적이고 치명적인 안티 패턴(Anti-pattern)을 정확히 집어서 금지시켰기 때문에, AI가 코드 작성 시 꼼수를 부리지 못하게 원천 봉쇄합니다.
- 🧩 **어댑터 패턴 유도:** 해당 프레임워크의 기본적인 뼈대(라우팅 등)는 존중하되 코어 비즈니스 로직만 독립적으로 분리하도록 지시하여, 불필요하고 과도한 오버엔지니어링(Over-engineering)을 사전에 방지했습니다.

## 📊 증명: Before & After

### ❌ Before (입력)

```javascript
// 프롬프트 없이 "유저 생성하고 이메일 보내는 API 만들어줘" 했을 때
export default async function handler(req, res) {
  const { email, password } = req.body;
  // DB 직접 찌르기 (오염 발생!)
  const user = await prisma.user.create({ data: { email, password } });
  // 외부 API 직접 호출 (관심사 붕괴!)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ After (결과)

```javascript
// 하드코어 DDD 프롬프트 적용 후 (계층과 책임이 완벽히 분리됨)
// 1. Controller 계층 (오직 요청/응답 라우팅만 처리)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Domain Service 계층 (프레임워크 독립적인 순수 비즈니스 로직)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}
```

## 🎯 결론

AI에게 코딩을 맡길 때 가장 경계해야 할 최악의 상황은 '당장 눈앞에서는 작동만 하는 쓰레기 코드'를 얻는 것입니다. 단순히 콘솔창에 에러가 나지 않는다고 환호할 때가 아닙니다.

이 하드코어 DDD 치트키 프롬프트로 맹목적인 AI의 멱살을 단단히 잡고, 프로젝트 초기부터 제대로 된 클린 아키텍처를 확실하게 뽑아내십시오. 이 원칙을 지킨다면, 미래의 당신이 과거의 훌륭한 선택을 한 스스로에게 기립 박수를 보내게 될 것입니다. 

이제 완벽한 코드로 업무를 자동화하고, 당당하게 칼퇴하십시오! 🍷
