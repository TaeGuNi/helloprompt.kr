## 📝 Хватит запутанного кода! ИИ-анализатор сложности (Code Complexity Analyzer)

- **🎯 Рекомендовано для:** Джуниор-разработчиков, специалистов по поддержке legacy-систем, строгих код-ревьюеров
- **⏱️ Затраты времени:** с 1 часа до 3 минут
- **🤖 Рекомендуемая модель:** GPT-4o, Claude 3.5 Sonnet (рекомендуются модели с развитым логическим мышлением)

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _"Цикл внутри условия `if`, внутри еще условие... Код работает, но вы боитесь его трогать, потому что он похож на бомбу замедленного действия? Знакомо ли вам это чувство отчаяния при виде спагетти-кода?"_

В процессе разработки каждый рано или поздно сталкивается с ужасающим **«спагетти-кодом»**. Бесконечные новые функции, изменения требований и спешные исправления багов (hotfixes) из-за жестких дедлайнов превращают код в раздутого и неуправляемого монстра. **Цикломатическая сложность (Cyclomatic Complexity)** растет в геометрической прогрессии, и на свет рождается чудовище, где одна функция занимает сотни строк.

Такой legacy-код не просто портит читаемость. Даже простая правка цвета кнопки или текста может вызвать **непредсказуемые побочные эффекты (Side Effects)** в самых неожиданных частях системы, заставляя разработчика дрожать от страха. «Зачем здесь это условие?», «Где изменяется (mutation) эта глобальная переменная?» — на чтение и понимание намерений автора уходит весь рабочий день, и вместо созидания вы тратите энергию на раскопки руин прошлого. Если вы хоть раз проводили бессонную ночь перед релизом, барахтаясь в болоте запутанного кода, вы понимаете, о чем речь.

Особенно опасен код с аномально высокой **когнитивной сложностью (Cognitive Complexity)**: он убивает продуктивность всей команды. Новые сотрудники тратят вечность на онбординг, а даже опытные синьоры на код-ревью не знают, с чего начать критику. В итоге в команде воцаряется негласное правило «работает — не трогай», и система медленно гибнет под тяжестью неподъемного технического долга. Попытки написать тесты для рефакторинга оборачиваются провалом из-за слишком сильной связанности (Tightly Coupled) модулей, где даже мокирование становится невозможным.

Мы часто тешим себя надеждой: «Вот закончим этот спринт, и на следующей неделе обязательно займемся рефакторингом». Но «следующая неделя» никогда не наступает, а код запутывается все сильнее. Рефакторинг **Legacy-кода** похож на замену колес у автомобиля, несущегося на скорости 100 км/ч. Живая система не должна останавливаться, бизнес требует новых функций, и, конечно, нельзя плодить новые баги. В условиях такого давления мозг человека достигает предела: как только вложенность `if` превышает 3-4 уровня, способность к логическому анализу резко падает.

Но теперь у вас есть совершенный инструмент, который избавит вас от этого страха. Это промпт **«ИИ-анализатор сложности кода (Code Complexity Analyzer)»**. Вам больше не нужно полагаться на интуицию. С этим промптом ИИ станет вашим строгим, но гениальным напарником (Senior Pair Programmer), который на основе объективных метрик программной инженерии поставит точный диагноз вашему коду.

Самое поразительное, что ИИ не просто сокращает код как обычный форматер. Опираясь на **паттерны проектирования (Design Patterns)** и принципы **чистого кода (Clean Code)**, он за минуту предложит конкретную и элегантную стратегию улучшения архитектуры. Сохраняя целостность бизнес-логики, он распутает ад вложенных `if-else` с помощью **ранних возвратов (Early Return)** и разделит гигантские функции на модули согласно **принципу единой ответственности (SRP)**. С этим промптом любой legacy-код станет холстом для демонстрации вашего мастерства. Пишите чистые Pull Request'ы, вызывающие восхищение коллег, и верните себе радость от разработки.

## 📊 Доказательство: Ошеломительный результат (Before & After)

### ❌ Before (Боль, с которой мы сталкивались)

Типичный пример запутанного legacy-кода: вложенные условия и циклы, в которых невозможно быстро разобраться. Понимание логики занимает десятки минут.

```javascript
function processOrder(order) {
  if (order != null) {
    if (order.status === "PENDING") {
      if (order.items && order.items.length > 0) {
        let total = 0;
        for (let i = 0; i < order.items.length; i++) {
          if (order.items[i].price > 0) {
            total += order.items[i].price * order.items[i].quantity;
          }
        }
        if (total > 100) {
          order.discount = total * 0.1;
        } else {
          order.discount = 0;
        }
        return total - order.discount;
      } else {
        return 0;
      }
    } else {
      throw new Error("Order is not pending");
    }
  } else {
    throw new Error("Invalid order");
  }
}
```

### ✅ After (Идеальное преображение)

![Хватит запутанного кода! ИИ-анализатор сложности (Code Complexity Analyzer)](/images/hooks/code-complexity-analyzer.jpg)

```javascript
function processOrder(order) {
  // 1. Guard Clauses: ранний возврат при исключениях (уменьшение вложенности)
  if (!order) throw new Error("Invalid order");
  if (order.status !== "PENDING") throw new Error("Order is not pending");
  if (!order.items || order.items.length === 0) return 0;

  // 2. Использование функций высшего порядка: упрощение циклов
  const total = order.items
    .filter((item) => item.price > 0)
    .reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 3. Разделение и упрощение бизнес-логики
  const discount = total > 100 ? total * 0.1 : 0;
  order.discount = discount;

  return total - discount;
}
```

## ⚡️ Краткое резюме (TL;DR)

1. ИИ проводит структурный анализ вложенных условий и раздутых legacy-функций.
2. Вы получаете проверенные рекомендации по рефакторингу: ранние возвраты (Early Return), выделение методов (Extract Method) и др.
3. Сохраняя логику в целости, вы мгновенно получаете элегантный код с максимальной читаемостью и удобством поддержки.

## 💡 Комментарий автора (Insight)

Этот промпт — мое главное оружие, когда я сталкиваюсь с legacy-кодом, написанным кем-то другим (или мной в прошлом). Читать чужой код всегда больно, особенно если это спагетти с невнятными именами переменных. Но ИИ не испытывает раздражения. Без лишних эмоций он препарирует структурные дефекты кода с хирургической точностью.

Сила этого промпта в том, что он дает вам логическое обоснование для коллег. Вместо «мне так больше нравится», вы сможете сказать: «Здесь вложенность достигает 4 уровней, поэтому применение паттерна **Guard Clauses** архитектурно оправдано». Рефакторинг — это не вкусовщина, а следование принципам компьютерных наук, и этот промпт дает вам эти принципы в руки.

Использование **переменных в скобках `[ ]`** — это ключ к успеху. Указание стека, например `[TypeScript/NestJS]`, дает ИИ огромный контекст. Он предложит не просто правки синтаксиса, а использование внедрения зависимостей (DI) или кастомных декораторов для разделения сквозной функциональности (Cross-cutting Concerns). Если вы укажете `[Python/FastAPI]`, он сделает упор на Pydantic или оптимизацию асинхронности.

Особенно эффективно использовать этот промпт для **самопроверки (Self-Review)** перед отправкой Pull Request. Сразу после написания кода мы часто находимся в плену предвзятости подтверждения («мой код идеален»). Но прогон через ИИ часто выявляет краевые случаи (Edge Cases) или возможности для снижения связанности (Coupling), о которых вы не подумали. Результат — восторженные комментарии коллег: «Архитектура — просто искусство!».

Наконец, обратите внимание на **строгие ограничения (Constraints)**. Фраза про неизменность бизнес-логики — это защита от «галлюцинаций» ИИ. LLM иногда так увлекаются упрощением, что выбрасывают важные проверки. Жесткий контроль инвариантности позволяет получать надежный код, который можно смело отправлять в продакшн. Используйте это оружие в войне с техническим долгом и побеждайте!

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Можно ли вставить весь файл целиком?**
  - Ответ: Не рекомендуется. Из-за ограничений контекстного окна ИИ может потерять фокус или начать галлюцинировать. Лучше подавать код частями по 50–150 строк (конкретные функции или классы). Так вы получите гораздо более глубокий и точный анализ.

- **Вопрос: Безопасно ли копировать код ИИ сразу в продакшн?**
  - Ответ: Нет! ИИ — это помощник, а не замена разработчику. Он может упустить нюансы логики. Код ИИ — это отличный черновик. **Обязательно убедитесь, что все unit-тесты проходят**, и проверьте целостность логики вручную перед мерджем.

- **Вопрос: Понимает ли ИИ специфику моего фреймворка?**
  - Ответ: Да. Если вы укажете стек в профессиональном промпте (например, Java/Spring Boot или Go/Gin), ИИ применит соответствующие конвенции именования и паттерны, принятые в данном сообществе.

## 🧬 Анатомия промпта (Почему это работает?)

1. **Фокус на метриках:** Вместо абстрактного «сделай красиво», мы требуем снижения «цикломатической сложности». Это направляет вычислительную мощность ИИ на решение структурных проблем, а не на косметический ремонт.
2. **Строгий контроль поведения:** Принцип инвариантности внешнего поведения стоит во главе угла. Это предотвращает удаление критически важных проверок ради «красоты» кода.
