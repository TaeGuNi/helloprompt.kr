## 🚀 La Soluzione: Il Prompting "Everything Claude Code"

### 🥉 Versione Basic (Installazione Rapida e Pianificazione)

Assumi un Senior Agent direttamente nel tuo terminale in meno di 2 minuti.

> **1. Installazione del Plugin (per Claude Code):**
> `/plugin marketplace add affaan-m/everything-claude-code`
> `/plugin install everything-claude-code@everything-claude-code`
>
> **2. Esecuzione del Prompt di Pianificazione:**
> `/plan "Crea una pagina responsive per l'elenco dei post del blog utilizzando Next.js 14 App Router. Usa Tailwind CSS e shadcn/ui."`

### 🥇 Versione Pro (Forzatura TDD e Regole Personalizzate)

Utilizza questa configurazione quando hai bisogno di imporre un rigoroso workflow da Senior Developer, cucito su misura per il tuo progetto.

> **Ruolo (Role):** Sei un `[Sviluppatore TypeScript Senior]` che applica in modo intransigente i principi del TDD (Test-Driven Development).
>
> **Contesto (Context):**
>
> - Stiamo operando secondo le linee guida di `[everything-claude-code]`.
> - Obiettivo: Dobbiamo implementare la `[funzionalità di autenticazione utente]`.
>
> **Attività (Task):**
>
> 1. Esegui il comando `/tdd` nel terminale.
> 2. Prima di procedere con l'implementazione del codice, devi obbligatoriamente scrivere i test (coprendo anche gli edge case).
> 3. È vietato passare alla fase successiva se la test coverage è inferiore all'80%.
> 4. Qualora vi siano `[requisiti aggiuntivi]`, integrali immediatamente nel codice.
>
> **Vincoli (Constraints):**
>
> - Definisci in modo esplicito i Type Hints per tutte le variabili e le funzioni.
> - L'utilizzo del tipo `any` è assolutamente proibito.
>
> **Avvertenze (Warning):**
>
> - Non generare codice che contraddica l'architettura o le convenzioni del progetto. In caso di dubbio, chiedimi sempre conferma prima di procedere (Prevenzione delle allucinazioni).

## 💡 Il Commento dell'Autore (Insight)

L'aspetto più sbalorditivo che ho riscontrato applicando questa configurazione in scenari reali è che **"l'IA si rifiuta categoricamente di accettare prompt pigri o approssimativi"**.

Prima, limitandosi a un banale "aggiungi una pagina di login", l'IA avrebbe sfornato codice disordinato e privo di qualsiasi gestione degli errori. Oggi lo scenario è completamente ribaltato. Invocando il comando `/plan`, l'IA inizierà a interrogarti in modo proattivo: _"Quale metodo di autenticazione (OAuth, JWT) intendiamo adottare? Come vogliamo gestire l'interfaccia utente in caso di errore?"_ L'esperienza è identica a quella di avere al proprio fianco un **Lead Developer meticoloso, che si rifiuta di toccare la tastiera senza aver prima redatto un documento di specifiche dettagliato**.

Tuttavia, il vero punto di svolta è il comando **`/evolve`**. Man mano che il progetto prende forma, l'IA assimila in totale autonomia le convenzioni del tuo team (ad esempio, "Non utilizziamo il prefisso 'I' per le interfacce TypeScript") e le converte in regole ferree e inviolabili. Per chiunque sia stanco di fare i conti con gli "spaghetti code" o per gli sviluppatori solisti che sentono la mancanza di una code review d'autore, questo workflow non rappresenta una semplice alternativa: è un imperativo assoluto.

## 🙋 Domande Frequenti (FAQ)

- **D: Questa configurazione è limitata esclusivamente alla CLI di Claude Code?**
  - R: Assolutamente no! È perfettamente compatibile anche con l'editor Cursor (impostandola all'interno del file `.cursorrules`) o negli ambienti basati su OpenCode.

- **D: È sicuro adottare questa pratica in progetti aziendali interni o altamente riservati?**
  - R: Il file delle regole è open-source e risiede unicamente in locale, garantendo la massima sicurezza strutturale. Ciononostante, è sempre fondamentale rispettare le policy aziendali (ad esempio, assicurandosi di utilizzare un piano Enterprise) prima di trasmettere porzioni di codice proprietario a qualsiasi modello LLM esterno.

- **D: Questo livello di rigidità non rischia di risultare eccessivo per uno sviluppatore alle prime armi?**
  - R: Inizialmente potrebbe apparire frustrante, poiché l'IA continuerà a fare domande mirate e a forzare il TDD. Tuttavia, è un passaggio formativo imprescindibile: la disciplina di oggi ti eviterà le notti insonni spese a ripagare il debito tecnico di domani.

## 🧬 Anatomia del Prompt (Perché funziona?)

1. **Flusso di Lavoro Forzato (Forced Workflow):** Abbandonando le istruzioni generiche, l'uso dei comandi slash come `/plan` e `/tdd` obbliga l'IA ad aderire a un ciclo di sviluppo rigoroso e strutturato (Analisi dei Requisiti → Scrittura dei Test → Implementazione del Codice Principale).
2. **Separazione dei Ruoli (Persona Separation):** Segmentando l'agente in figure distinte (il Planner che architetta, il Developer che scrive, il Reviewer che valida), la configurazione innesca un sistema di controlli incrociati che assicura un'oggettività e una solidità del codice senza precedenti.
3. **Apprendimento Continuo (Instincts):** A differenza dei classici chatbot dove il contesto viene perso a ogni sessione, questa IA interiorizza i tuoi pattern di correzione e le tue preferenze, salvandole dinamicamente nel file `.rules` per i task futuri.

## 📊 Dimostrazione: Prima & Dopo

### ❌ Prima (Coding IA Standard)

```text
Utente: "Creami una lista di post per il blog"
IA: (Genera istantaneamente una lunga sfilza di codice React, senza paginazione, senza gestione degli errori e ignorando il design system del progetto)
```

### ✅ Dopo (Con everything-claude-code)

```text
Utente: "> /plan Creami una lista di post per il blog"
IA: L'agente Planner è stato attivato. Per un'implementazione perfetta, dobbiamo prima confermare questi 3 punti:
1. Quanti post mostriamo per pagina? (Infinite scroll vs Paginazione)
2. Il data fetching avverrà lato server (SSR) o lato client (CSR)?
3. Hai bisogno di un design Skeleton UI da mostrare durante il caricamento dei dati?

Non appena risponderai, darò istruzioni all'agente TDD di procedere con la stesura dei test.
```
