# 📝 スパゲッティコードはお断り！🍝 シニアエンジニアの黒魔術 DDDプロンプト
- 🎯 **おすすめの対象:** AIが書いたコードを見てため息をつくエンジニア、保守地獄に陥った先輩エンジニア
- ⏱️ **所要時間:** 3ヶ月（リファクタリング時間） → 1分に短縮
- 🤖 **推奨モデル:** コーディング特化型の対話型AI (Claude 3.5 Sonnet, GPT-4o など)
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_AIが書いたコード、最初は「おおっ！」と思っても、後になって見たくもなくなったことはありませんか？おめでとうございます。あなたはたった今「スパゲッティコード地獄」に入場しました。_

AIにコーディングを任せると、最初は見事に書いてくれます。しかし、プロジェクトが少しでも大きくなったら？UIコンポーネントにDBクエリがベタベタと貼り付き、1つの関数が300行を超えるという恐ろしいキメラを生み出します。AIは本質的に「イエスマン」であるため、あなたが適当に指示を出せば、本当に適当な構造でコードを書いてしまうのです。

さあ、AIの盲目的なお世辞を止め、骨組みから完璧な「クリーンアーキテクチャ」を強制する時間です。このチートキーは、AIを無慈悲で厳格なシニアエンジニアに憑依させ、フレームワークに依存しない純粋なビジネスロジック（DDD）を書くよう強制します。

---

## ⚡️ 3行まとめ (TL;DR)

- 🚫 **スパゲッティコードを根本から遮断:** UI、ビジネスロジック、DBの完璧な関心の分離（SoC）を強制します。
- 🏗️ **フレームワークからの独立性を確保:** 明日突然ReactやNext.jsが消滅しても生き残るコアを設計します。
- ✂️ **巨大関数の破壊者:** 30行を超える肥大化した関数は、AIが自ら単一責任の原則（SRP）に従って粉砕します。

---

## 🚀 解決策: "ハードコアアーキテクト憑依プロンプト"

### 🥉 Basic Version (基本型)

すぐに結果だけが必要な場合に使用してください。

> **役割:** あなたは`[厳格なシニアアーキテクト]`だ。
> **要求:** `[実装する機能]`を作るが、絶対にビジネスロジックとUIを完璧に分離してコードを書いてくれ。1つのファイルに複数の機能が混ざるスパゲッティコードは絶対に許さない。

### 🥇 Pro Version (専門家型)

ディテールにこだわった品質と完璧な階層分離が必要な場合に使用してください。

> **役割 (Role):** あなたは妥協を知らないハードコアなシニアアーキテクトであり、ソフトウェアエンジニアリングの神だ。無意味なお世辞や「はい、分かりました！」のような返事は捨てて、コードだけで語れ。
>
> **状況 (Context):**
> - 背景: 現在、`[プロジェクト環境/フレームワーク]`ベースで`[実装するコア・ドメイン]`システムを開発中である。
> - 目標: コードの腐敗を防ぎ、View-Business-Data層を完璧に分離するドメイン駆動設計(DDD)およびクリーンアーキテクチャを適用してコードを作成する。
>
> **要求 (Task):**
> 1. `[実装する具体的な機能]`に対するコードを作成しろ。
> 2. **全権階層の崩壊を絶対禁止:** UI/Controller内にDBクエリや外部API通信を絶対に混ぜるな。レンダリングとパラメータの伝達のみを担当させろ。
> 3. **ドメイン汚染の防止:** コアビジネスロジック(Entity, Value Object)には、フレームワークやDBの依存性を絶対に `import` するな。純粋関数と型のみで構成しろ。
> 4. **巨大関数の破壊者:** 1つの関数が2つ以上の振る舞いをするか、30行を超える場合は、単一責任の原則(SRP)に従って直ちに細かく分割しろ。
>
> **制約事項 (Constraints):**
> - フレームワークの基本ルーティングルールは尊重しつつ、ビジネスロジックは `src/domain` など独立したフォルダに分離すること。
> - 外部依存性は、必ずインターフェースベースの依存性の注入(DI)を使用すること。
>
> **注意事項 (Warning):**
> - コードから少しでもスパゲッティの匂いがすれば、このプロジェクトは失敗する。妥協せず、最も理想的でクリーンなアーキテクチャ構造を提示すること。

## 💡 筆者コメント (Insight)

私がこのスキルを研ぎ澄ませた理由はただ一つ。AIというやつらは、基本的に「早く結果を投げて褒められたい」という欲求に飢えたお世辞屋だからだ。

ボタンを一つ作ってくれと頼んだら、ボタンコンポーネントの中で `fetch` を叩き、DBのクエリ文までねじ込む姿を見ていると、私の中の破壊衝動が目覚める。このプロンプトは、そんな浅はかなイエスマンAIの胸ぐらをつかみ、「目を覚ませ、お前は今から血も涙もないシニアアーキテクトだ！」と平手打ちを食らわせる役割を果たす。

実際のプロジェクトにこのプロンプトを適用してみてほしい。AIが突然目つき（口調）を変え、ファイルを3〜4個に分割して階層を厳格に分離し始めるだろう。最初はファイルが増えて面倒に感じるかもしれないが、3ヶ月後に要件が変更され、保守を行う時、私（筆者）のいる方向に向かって毎朝土下座したくなるはずだ。技術的負債の金利は闇金レベルだということを肝に銘じておけ。妥協はあり得ない。

## 🙋 よくある質問 (FAQ)

- 🔹 **Q: コードが細かく分割されすぎてファイルが多くなるのは正常ですか？**
  - 🔸 **A:** 極めて正常だ。1つのファイルが500行を超えるより、50行のファイルが10個ある方が保守においては100倍有利だ。ファイル数が増えることを恐れるな。
- 🔹 **Q: フロントエンド開発者ですが、このプロンプトは役に立ちますか？**
  - 🔸 **A:** もちろんだ。UIコンポーネント（`page.tsx`、`Component.vue`など）にビジネスロジックや状態遷移ロジックが混ざった瞬間、フロントエンドもスパゲッティ化する。ビューとロジックを分離するのは分野を問わない真理だ。

## 🧬 プロンプトの解剖 (Why it works?)

- 🎯 **ハードコア・ペルソナの注入:** AIの基本性質である「順応性」を剥奪し、原則を固守するように強制しました。
- 🧱 **絶対禁止条項の明示:** 「UI階層の汚染」や「巨大な関数」など、具体的なアンチパターンをピンポイントで禁止したため、AIが手抜きできなくなります。
- 🧩 **アダプターパターンの誘導:** フレームワークの骨組み（ルーティング）は尊重しつつ、コアロジックのみを分離するよう指示することで、過度なオーバーエンジニアリングを防ぎました。

## 📊 証明: Before & After

### ❌ Before (入力)

```javascript
// プロンプトなしで「ユーザーを作成してメールを送信するAPIを作って」と指示した場合
export default async function handler(req, res) {
  const { email, password } = req.body;

  // DBを直接叩く（汚染発生！）
  const user = await prisma.user.create({ data: { email, password } });

  // 外部APIを直接呼び出す（関心の崩壊！）
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ After (結果)

```javascript
// ハードコアDDDプロンプト適用後（階層と責任が完璧に分離される）

// 1. Controller階層（リクエスト/レスポンスのルーティングのみを処理）
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Domain Service階層（フレームワークに依存しない純粋なビジネスロジック）
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Infrastructure階層（DBおよび外部API専用アダプター）
// UserRepository、EmailServiceなどとして別ファイルに完全に隔離される...
```

## 🎯 結論

AIにコーディングを任せる際、最も危険なのは「とりあえず目の前で動くだけのゴミコード」を受け取ることです。エラーが出ないからと喜んでいる場合ではありません。

このハードコアDDDチートキーでAIの胸ぐらをつかみ、最初からちゃんとしたアーキテクチャを引き出してください。未来のあなたが過去のあなたにスタンディングオベーションを送るはずです。
