## 📝 Слишком ленивы писать код тестов? ИИ сделает это за вас

- **🎯 Рекомендуется для:** Backend- и Frontend-разработчиков
- **⏱️ Требуемое время:** 5 минут → 30 секунд
- **🤖 Рекомендуемая модель:** Любая (ChatGPT, Claude, Gemini)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _«Писать код — весело, а вот писать тесты — невыносимо скучно. Но без них каждый деплой превращается в русскую рулетку»._

Модульное тестирование (Unit Testing) — абсолютная необходимость для создания отказоустойчивых сервисов. Однако ручное написание тестов отнимает колоссальное количество времени и ментальной энергии. Особенно тяжело предусмотреть все коварные граничные случаи (Edge cases), о которых в момент написания фичи даже не задумываешься. Делегируйте эту монотонную рутину искусственному интеллекту, а сами сфокусируйтесь на том, что действительно важно — проектировании архитектуры и сложной бизнес-логике.

---

## ⚡️ 3 кратких вывода (TL;DR)

1. ИИ способен за секунды генерировать пуленепробиваемые модульные тесты, учитывающие как идеальные условия, так и скрытые уязвимости.
2. Назначение ИИ на роль Senior QA-инженера заставляет его мыслить критически и находить нестандартные сценарии сбоев.
3. Вы избавляетесь от рутины и отправляете код на ревью с абсолютной уверенностью в его надежности.

## 🚀 Решение: «Страж покрытия тестами»

### 🥉 Базовая версия (Basic Version)

Используйте этот вариант, когда нужно быстро набросать тесты без глубокого погружения в детали.

> **Роль:** Ты `[Ведущий QA-инженер]`.
> **Запрос:** Напиши модульные тесты для моего кода, обязательно учитывая граничные случаи и возможные ошибки.

### 🥇 Профессиональная версия (Pro Version)

Используйте этот промпт, когда требуется максимальное покрытие и готовый к продакшену код тестов.

> **Роль (Role):** Ты `[Ведущий QA-инженер]`.
>
> **Ситуация (Context):**
>
> - Я написал функцию/класс: `[Название функции или класса]`.
> - Фреймворк для тестирования: `[Jest / JUnit / PyTest / и т.д.]`.
>
> **Запрос (Task):**
>
> 1. Напиши тест для успешного сценария (Happy path).
> 2. Определи как минимум 3 исключительные ситуации и граничные случаи (Edge cases), после чего напиши для них негативные тесты (тесты на отказ).
> 3. Добавь комментарии к каждому тесту, четко объясняющие, что именно и почему он проверяет.
>
> **Код для тестирования:**
>
>
> [Вставьте ваш код сюда]
>
>
> **Ограничения (Constraints):**
>
> - Убедись, что все тесты полностью изолированы и независимы друг от друга.
> - Выведи только готовый код тестов в формате Markdown, без лишних вступлений и объяснений.

## 💡 Комментарий автора (Insight)

Написание тестов — это та самая задача, которую разработчики вечно откладывают «на потом» из-за её монотонности. Однако с помощью ИИ вы можете превратить эту скучную обязанность в быстрый и высокоэффективный процесс. Данный промпт особенно хорош для выявления **граничных случаев (Edge cases)**, о которых замыленный глаз разработчика может даже не подумать при написании бизнес-логики. ИИ в данном контексте выступает в роли **«адвоката дьявола»**, целенаправленно пытаясь сломать ваш код. Именно такой строгий подход делает итоговый продукт в разы стабильнее.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Можно ли использовать этот промпт для интеграционных или E2E (End-to-End) тестов?**
  - A: Да, но потребуется небольшая адаптация. Для интеграционных тестов обязательно опишите ИИ структуру вашей **базы данных** или **API**, чтобы он мог корректно смоделировать взаимодействие между компонентами системы.

- **Q: Что делать, если ИИ генерирует тесты, которые не проходят (падают)?**
  - A: Это отличный сигнал! Либо ИИ нашел реальный баг в вашем коде, либо он неправильно понял контекст. Внимательно проверьте свою логику или добавьте больше поясняющих деталей в блок **«Ситуация (Context)»**.

## 🧬 Анатомия промпта (Why it works?)

1. **Смена перспективы (Role):** Роль бескомпромиссного QA-инженера заставляет ИИ мыслить не как разработчик (который просто хочет, чтобы код запустился), а как тестировщик (чья цель — найти уязвимость).
2. **Фокус на уязвимостях (Task):** Прямое требование найти как минимум 3 исключительные ситуации не позволяет ИИ ограничиться одними лишь базовыми «счастливыми путями» (Happy paths).
3. **Требование к комментариям:** Заставляет ИИ четко аргументировать выбор проверок. Это помогает вам глубже понять логику каждого сгенерированного теста.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод: простая функция деления)

```javascript
function divide(a, b) {
  if (b === 0) throw new Error("Нельзя делить на 0.");
  return a / b;
}
```

### ✅ After (Результат)

```javascript
// Пример с использованием Jest

describe("Тестирование функции divide", () => {
  // 1. Успешный сценарий (Happy path)
  test("Деление 10 на 2 должно возвращать 5", () => {
    expect(divide(10, 2)).toBe(5);
  });

  // 2. Граничный случай: отрицательные числа
  test("Деление с отрицательными числами должно давать корректный результат", () => {
    expect(divide(-10, 2)).toBe(-5);
    expect(divide(-10, -2)).toBe(5);
  });

  // 3. Исключительная ситуация: деление на ноль
  test("Деление на 0 должно выбрасывать ожидаемую ошибку", () => {
    expect(() => divide(10, 0)).toThrow("Нельзя делить на 0.");
  });

  // 4. Граничный случай: дробные результаты
  test("Деление, приводящее к дробному результату, должно обрабатываться точно", () => {
    expect(divide(5, 2)).toBe(2.5);
  });
});
```
