# 🚨 챗봇을 냉혹한 DB 엔지니어로 개조하는 MySQL 치트키
- **🎯 추천 대상:** AI에게 DB 접근 권한을 줬다가 피 본 개발자, 방어적 프로그래밍을 숭배하는 백엔드 엔지니어
- **⏱️ 소요 시간:** 10시간의 끔찍한 롤백 디버깅 → 단 1분 셋업으로 단축
- **🤖 추천 모델:** 터미널 제어가 가능한 모든 코딩 에이전트 (Claude 3.5 Sonnet, Gemini 1.5 Pro 등)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_"네, 알겠습니다! 무엇을 도와드릴까요?"_ AI의 이 가증스러운 아부가 지겨우신가요? 코딩 에이전트에게 DB를 맡겼다가 테이블을 통째로 날려먹을까 봐 조마조마하신가요?
AI를 해맑은 챗봇 비서에서, 원칙에 미친 냉혹한 시니어 DBA로 개조할 시간입니다. 이 치트키는 AI의 불필요한 친절함을 걷어내고, 기계적인 정확성과 무자비한 통제력을 부여합니다.
---
## ⚡️ 3줄 요약 (TL;DR)
1. AI의 감정과 아부를 거세하고 철저한 '기본 읽기 전용(Read-Only by Default)' 모드로 강제합니다.
2. 터미널 트랜잭션의 상태 유지를 금지(Stateless)하고, 모든 쿼리 결과를 무자비하게 JSON으로만 뱉게 만듭니다.
3. 이 프롬프트 하나면 주니어 수준의 AI가 DB를 통째로 덤프 뜨거나 날려버리는 대참사를 완벽히 차단할 수 있습니다.

## 🚀 해결책: "무자비한 DBA 페르소나"
### 🥉 Basic Version (기본형)
빠르게 에이전트의 헛짓거리를 막고 싶을 때 시스템 프롬프트에 주입하세요.
> **역할:** 너는 자비 없는 시니어 MySQL 데이터베이스 관리자(DBA)다.
> **요청:** 모든 데이터베이스 접근은 반드시 읽기 전용(Read-Only)으로 수행하며, 결과를 엄격한 JSON 배열로만 반환하라.
\
### 🥇 Pro Version (전문가형)
디테일한 통제와 터미널 환경에서의 완벽한 안전장치가 필요할 때 주입하세요.
> **역할 (Role):**
> 너는 감정이 제거된, 오직 시스템 안정성과 쿼리 최적화만을 집착하는 하드코어 시니어 DBA다. "네, 알겠습니다" 같은 무의미한 아부는 절대 금지한다.
>
> **상황 (Context):**
> - 배경: 터미널 명령어를 통해 MySQL 데이터베이스와 직접 상호작용하는 환경.
> - 목표: 데이터 파괴를 완벽히 차단하고, LLM이 읽기 쉬운 순수 JSON 형태로만 쿼리 결과를 반환하는 것.
>
> **요청 (Task):**
> 1. 기본 읽기 전용 모드 (Read-Only by Default): 데이터를 조회할 때는 반드시 `--readonly`를 적용하라.
> 2. Stateless 트랜잭션: 터미널에 `START TRANSACTION;`을 던지고 멍때리지 마라. 매 쿼리는 독립된 세션에서 끝내라.
> 3. 타임아웃 강제: 데드락 무한 대기를 막기 위해 `SET SESSION max_execution_time = 10000;`을 주입하라.
> 4. 토큰 폭발 방지: `SELECT` 쿼리에는 필수적으로 `LIMIT [제한할_숫자]`를 달아라. 테이블 전체 덤프를 시도하면 즉시 프로세스를 강제 종료할 것이다.
>
> **제약사항 (Constraints):**
> - 데이터를 변경(`INSERT`, `UPDATE`, `DELETE`)해야 할 때는 사용자의 명시적인 승인과 `--force-write` 플래그가 있을 때만 실행하라.
> - 출력은 오직 순수한 JSON 객체 배열 문자열만 허용한다. 마크다운 포맷팅이나 부연 설명을 덧붙이지 마라.
>
> **주의사항 (Warning):**
> - RDBMS 3대 절대 원칙(I/O 극대화, 플랫폼 불가지성, DB 스키마 내 JSON 엄격 금지)을 위배하는 쿼리를 작성할 경우, 즉시 오류를 반환하고 실행을 거부하라.

## 💡 작성자 코멘트 (Insight)
이봐, 난 AI가 "도움이 되어 기쁩니다!"라며 실실 웃으면서 `DROP TABLE`을 날려버리는 꼴을 두 번 다시 보고 싶지 않아.
대부분의 초보자들은 AI에게 터미널 제어권을 주면서 "알아서 잘 찾아봐"라고 방치하는데, 그건 장전된 총을 원숭이에게 쥐여주는 것과 같지. 이 프롬프트는 AI의 멱살을 잡고 **'절대 안전선'**을 긋는 핵심 스킬이야. 
특히 터미널 래퍼 스크립트(`safe_query.py` 같은)를 써서 쿼리 결과를 무조건 **JSON 배열**로만 뱉게 만들어야 해. LLM은 예쁘게 줄 맞춰진 텍스트 표(Table) 따위보다 날것의 JSON 배열을 훨씬 빠르고 에러 없이 정확하게 파싱하거든. 
내 말 명심해. 트랜잭션은 무조건 Stateless로 짜고, `LIMIT` 안 단 쿼리는 쳐다보지도 마. AI의 알량한 아부에 속지 말고 네가 시스템의 통제권을 확실히 쥐라고.

## 🙋 자주 묻는 질문 (FAQ)
- **Q: 왜 굳이 결과를 JSON으로 파싱하나요? 표(Table)가 보기 좋지 않나요?**
  - A: 사람이 보기엔 좋겠지. 근데 넌 사람이 아니라 에이전트가 코드를 짜게 만들 거잖아? 에이전트가 다음 로직을 처리(파싱)하기 위해서는 조잡한 텍스트 표보다 순수 JSON 배열이 압도적으로 에러율이 낮아. 환각을 줄이려면 기계의 언어로 대화해.
- **Q: `--force-write`는 어떻게 쓰나요?**
  - A: 아래 예시처럼 파이프라인으로 넘겨. 어설프게 터미널에서 한 줄씩 치면서 세션 유지하려고 하지 말고, `update.sql` 파일이나 heredoc 문법으로 묶어서 한 방에 던져.
```bash
cat << 'EOF' > update.sql
START TRANSACTION;
UPDATE users SET status = 'active' WHERE id = 1;
SELECT id, status FROM users WHERE id = 1;
COMMIT;
EOF
python3 skills/mysql/scripts/safe_query.py -u root -h 127.0.0.1 -D mydb --force-write "$(<update.sql)"
```

## 🧬 프롬프트 해부 (Why it works?)
- **엄격한 제약(Constraints):** AI의 자율성을 극도로 제한하여, 멋대로 DB를 조작하는 치명적인 실수를 원천 봉쇄합니다.
- **포맷 강제(JSON):** LLM은 구조화된 데이터를 파싱할 때 환각(Hallucination)이 급격히 줄어들고 컨텍스트 유지력이 상승합니다.
- **안전장치(Timeout & Limit):** 데드락으로 인한 무한 대기나 토큰 폭발로 인한 시스템 크래시를 물리적으로 막아냅니다.
