## 📝 커밋 전 자동 점검! AI Git Hook 생성기 (Git Hook Generator)

- **🎯 추천 대상:** 프론트엔드/백엔드 개발자, DevOps 엔지니어, 테크 리드
- **⏱️ 소요 시간:** 1시간 → 1분 단축
- **🤖 최상위 성능:** Claude 3.5 Sonnet, GPT-4o (코드 작성에 특화된 모델)

- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"아차, 린트(Lint) 돌리는 거 깜빡했다! 🚨 CI/CD 파이프라인을 붉게 물들이는 실패 로그, 이제 Git Hook 자동화로 영원히 안녕을 고하세요."_

개발자로 일하면서 가장 등골이 서늘해지는 순간은 언제일까요? 아마도 퇴근 직전, 무심코 올린 단 하나의 커밋이 **CI/CD 파이프라인을 통째로 새빨갛게 물들였을 때**일 것입니다. "아차, 린트(Lint) 돌리는 거 깜빡했다!" 혹은 "테스트 코드 실행을 까먹었네!"라는 탄식이 조용한 사무실에 울려 퍼집니다. 슬랙(Slack)의 #deploy 채널에는 봇이 보내는 요란한 빌드 실패 알림이 도배되고, 동료들의 따가운 시선 속에서 부랴부랴 `fix: 린트 에러 수정`, `fix: 진짜 마지막 린트 에러 수정` 같은 부끄러운 커밋을 연달아 올려야만 합니다.

이러한 사소한 실수는 단순히 개발자 개인의 부끄러움으로 끝나지 않습니다. 무거운 빌드 과정을 처음부터 다시 돌려야 하므로 **클라우드 컴퓨팅 리소스의 낭비**를 초래하고, 다른 팀원들의 배포 순서를 끝없이 뒤로 밀리게 만들며, 전체적인 애자일(Agile) 스프린트의 생산성을 심각하게 저하시킵니다. 우리는 모두 "다음부터는 커밋하기 전에 무조건 `npm run lint`와 `npm run test`를 꼼꼼히 돌려야지"라고 굳게 다짐하지만, 인간의 기억력과 의지력은 바쁜 마감 일정 앞에서는 한없이 나약해지기 마련입니다. 실수는 반드시 반복됩니다.

그렇다면 개인의 의지에 기대지 않고 시스템으로 강제하면 어떨까요? 맞습니다. **Git Hook**을 도입하면 이 모든 문제가 해결됩니다. Git Hook은 커밋(Commit)이나 푸시(Push) 같은 특정 이벤트가 발생하기 직전에 가로채어 우리가 원하는 스크립트를 강제로 실행해 주는 강력한 기능입니다. 하지만 여기서 우리는 또 다른 거대한 기술적 장벽에 부딪히게 됩니다. 바로 **쉘 스크립트(Shell Script) 작성의 고통**입니다.

대부분의 프론트엔드 웹 개발자나 백엔드 개발자에게 Bash 쉘 문법은 외계어처럼 낯설게 다가옵니다. 띄어쓰기 하나에 원인 모를 에러가 터지고, 수많은 파일 중 정규식을 활용해 스테이징된(Staged) 파일만 골라내는 최적화 로직을 짜려면 Stack Overflow를 수십 번 뒤져야 합니다. 게다가 맥(macOS)에서는 완벽하게 돌아가던 스크립트가 윈도우(Windows) 환경을 쓰는 팀원의 PC에서는 권한 문제나 파일 경로 이슈로 실행조차 되지 않는 **크로스 플랫폼(Cross-Platform) 호환성 에러**까지 심심치 않게 발생합니다. 결국 "이거 세팅하고 디버깅하는 시간에 그냥 손으로 검사하고 말지"라며 며칠 만에 포기하게 되는 것이 현업의 냉혹한 현실입니다.

이제 더 이상 복잡하고 난해한 쉘 스크립트 문법과 씨름하며 여러분의 귀중한 시간을 낭비하지 마세요. 우리에게는 코딩과 인프라 설정에 특화된 초거대 AI가 있습니다. 여러분의 현재 프로젝트 환경(예: Node.js, Spring Boot)과 팀의 깐깐한 컨벤션만 자연어로 툭 던져주면, AI가 단 **1초 만에 완벽하게 동작하는 Git Hook 스크립트를 생성**해 냅니다.

이 프롬프트는 단순히 스크립트 한 줄만 짜주고 끝나는 수준이 아닙니다. `Husky`나 `lint-staged` 같은 최신 프론트엔드 생태계 도구들의 설정 방법부터, 윈도우와 맥을 모두 아우르는 완벽한 크로스 플랫폼 호환성, 그리고 전체 파일이 수천 개에 달하는 대규모 레포지토리에서도 단 1초 만에 검사를 끝내는 **성능 최적화(Performance Optimization) 로직**까지 한 번에 알아서 구성해 줍니다. 우리는 그저 AI가 정갈하게 만들어준 코드를 복사해서 프로젝트 설정 파일에 붙여넣기만 하면 됩니다.

이 프롬프트를 여러분의 실무에 적용하는 순간, 개발 워크플로우와 팀의 분위기는 180도 달라집니다. 코드를 커밋하는 순간 터미널에서 자동으로 린트 검사와 단위 테스트가 백그라운드로 빠르게 실행됩니다. 만약 팀의 컨벤션에 어긋나는 코드가 단 한 줄이라도 있다면 커밋 자체가 생성되지 않고 그 자리에서 단호하게 튕겨 나가며, **어느 파일의 몇 번째 줄을 어떻게 수정해야 하는지** 아주 친절하고 명확한 에러 메시지를 띄워줍니다.

이제 더 이상 빌드 파이프라인이 터질까 봐 조마조마하며 마음 졸일 필요가 없습니다. 동료들의 코드 리뷰(Code Review) 시간도 획기적으로 줄어듭니다. 사소한 띄어쓰기나 네이밍 오타를 지적하며 감정을 소모하는 대신, 비즈니스 로직의 결함과 아키텍처 개선에만 오롯이 집중할 수 있는 진정한 의미의 **코드 품질 사수(Code Quality Assurance)**가 실현됩니다. 지금 당장 아래의 전문가용 프롬프트를 복사하여 여러분의 프로젝트에 한 치의 오차도 허용하지 않는 무적의 수문장을 세워보세요!

1. **자동화의 끝판왕:** 커밋 메시지 컨벤션 검사부터 코드 포맷팅까지, 커밋 직전에 필요한 모든 검증 과정을 자동으로 강제합니다.
2. **AI의 정밀한 스크립팅:** 복잡한 정규식이나 크로스 플랫폼 호환성을 고민할 필요 없이, AI가 현재 환경에 최적화된 쉘 스크립트를 즉시 생성합니다.
3. **CI 리소스 절약:** 파이프라인에 도달하기 전 로컬 환경에서 에러를 100% 차단하여 클라우드 컴퓨팅 비용과 불필요한 대기 시간을 아낍니다.

> 4. **적용 가이드:** 팀원들이 레포지토리를 클론한 후 즉시 환경을 셋업할 수 있도록, `[package.json]` 설정과 초기 설치 명령어를 마크다운 코드 블럭으로 정리해 줘.

> - 모바일 가독성을 위해 표(Table)는 절대 사용하지 말고, 가독성 높은 글머리 기호(List) 형태로 정리해 줘.

## 💡 작성자 코멘트 (Insight & How to use)

이 프롬프트의 진정한 가치는 단순히 **'사소한 실수 방지'** 수준을 넘어, 개발자와 팀 전체에게 흔들리지 않는 **'심리적 안정감(Psychological Safety)'**을 제공한다는 데 있습니다. 로컬 환경에서 모든 1차적 결함을 걸러낸다는 확신이 생기면, 개발자는 배포 프로세스에 대한 막연한 두려움을 떨쳐내고 오직 문제 해결과 코드 작성 그 자체에만 완벽하게 몰입할 수 있게 됩니다. 저는 현업의 대규모 프로젝트는 물론이고, 아주 작은 토이 프로젝트를 시작할 때조차 이 프롬프트를 사용하여 초기 보일러플레이트와 함께 Git Hook을 가장 먼저 세팅하는 것을 철칙으로 삼고 있습니다.

**1. `[변수]` 통제를 통한 프롬프트 무한 확장법**

이 프롬프트가 강력한 이유는 구조가 고도로 모듈화되어 있어, 여러분의 프로젝트 특성에 맞게 `[변수]`만 살짝 비틀어도 완전히 다른 차원의 수문장을 만들어낼 수 있기 때문입니다.

*   **언어 및 프레임워크 변경:** 만약 여러분이 파이썬(Python) 기반의 데이터 엔지니어링 팀이라면 환경 변수를 `[Python, Poetry, pre-commit 라이브러리 사용 중]`으로 변경하고, Task를 `[black 포매팅과 ruff 린팅 적용]`으로 수정해 보세요. AI가 Python 생태계의 표준인 `.pre-commit-config.yaml` 파일을 기가 막히게 작성해 줄 것입니다.
*   **보안 검사 추가:** 최근 화두가 되고 있는 시큐어 코딩을 강제하고 싶다면, 요청(Task) 섹션에 `[trufflehog나 git-secrets를 활용해 하드코딩된 API Key나 비밀번호가 커밋되는 것을 원천 차단해 줘]`라는 지시를 한 줄만 추가해 보세요. 보안 사고를 미연에 방지하는 완벽한 방패가 탄생합니다.
*   **브랜치 보호 훅 생성:** 저는 이 프롬프트를 응용하여 `[pre-push]` 훅을 자주 생성합니다. 목표(Goal)를 `[메인 브랜치(main)에 직접 푸시(Direct Push)하는 것을 막는 것]`으로 설정하면, 새벽에 졸린 눈으로 실수로 프로덕션 브랜치에 강제 푸시를 하려다 터미널에서 **"🚨 잠깐! main 브랜치는 반드시 PR을 통해서만 병합할 수 있습니다."**라며 튕겨내는 짜릿한 안도감을 경험할 수 있습니다.

**2. 성능과 최적화의 핵심: `lint-staged`의 마법**

과거에 Git Hook을 도입했다가 실패했던 팀들의 공통적인 패인은 바로 **'실행 속도'**입니다. 커밋을 할 때마다 수천 개의 파일이 있는 전체 코드베이스를 대상으로 린트를 돌리게 되면, 커밋 하나에 수십 초에서 몇 분까지 걸리게 됩니다. 개발자는 결국 답답함을 견디지 못하고 `--no-verify` 플래그를 남용하게 되며, 시스템은 유명무실해집니다.

그렇기 때문에 이 프롬프트의 제약사항(Constraints) 파트에 명시된 **"반드시 '변경이 발생한 파일(staged files)'만 추려내서 검사해 줘"**라는 문장이 핵심입니다. AI는 이 지시를 받고 단순한 쉘 명령어 대신 프론트엔드 생태계의 치트키인 `lint-staged` 패키지를 도입하는 아키텍처를 제안하게 됩니다. 이를 통해 내가 방금 수정한 2~3개의 파일만 핀셋처럼 집어내어 0.1초 만에 검사를 끝내므로, 개발 흐름을 전혀 방해하지 않고도 완벽한 코드 품질을 사수할 수 있습니다.

**3. 트러블슈팅: 권한 문제와 윈도우 환경 대응**

AI가 완벽한 코드를 짜주더라도 실행 환경에 따라 가끔 훅이 동작하지 않을 때가 있습니다. 가장 흔한 원인은 **'실행 권한(Executable Permission)'**의 누락입니다. 맥이나 리눅스 환경에서 `.husky/pre-commit` 파일이 그저 텍스트 파일로 인식되어 무시되는 현상입니다. 이럴 때는 당황하지 말고 터미널에 `chmod +x .husky/pre-commit` 명령어를 입력하여 실행 권한을 부여해 주면 즉시 마법처럼 작동하기 시작합니다.

또한 윈도우 환경을 혼용하는 팀이라면 프롬프트에 명시된 **'크로스 플랫폼'** 조건 덕분에 AI가 알아서 크로스 플랫폼 스크립트를 짜주지만, 간혹 줄바꿈 문자(`CRLF` vs `LF`) 충돌로 에러가 날 수 있습니다. 이럴 때는 AI에게 "윈도우 사용자를 위해 git의 `core.autocrlf` 설정 가이드도 포함해 줘"라고 후속 질문을 던지면 완벽한 온보딩 가이드를 얻을 수 있습니다. 직접 해보시면 이 프롬프트가 여러분의 퇴근 시간을 지켜주는 가장 든든한 동료가 될 것임을 확신합니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 윈도우(Windows)를 쓰는 팀원도 있는데, 쉘 스크립트가 잘 작동할까요?**
  - A: 프롬프트에 '크로스 플랫폼 제약사항'을 걸어두었기 때문에 AI가 운영체제 차이를 알아서 처리해 줍니다. 윈도우에서는 주로 Git Bash 환경에서 실행되도록 작성되며, Node.js 환경이라면 순수 쉘 스크립트보다는 `Husky`와 `lint-staged` 조합을 추천해 달라고 명시하는 것이 호환성 면에서 가장 안전합니다.

- **Q: 스크립트가 너무 무거워서 커밋할 때마다 몇 분씩 걸립니다.**
  - A: `npm run lint`나 `npm run test`를 전체 코드베이스에 대해 돌리고 있기 때문입니다. 프롬프트의 주의사항 섹션에 있는 것처럼 **"반드시 스테이징된 파일(변경된 파일)만 검사해달라"**고 명시하세요. AI가 `git diff --cached --name-only`를 활용한 가볍고 빠른 스크립트로 다시 짜줄 것입니다.

- **Q: 이미 터진 대형 에러가 있는데, 긴급하게 커밋을 해야 할 때는 Hook이 방해됩니다.**
  - A: 정말 급하게 Hook을 무시하고 커밋이나 푸시를 해야 할 때는 명령어 뒤에 `--no-verify` (또는 `-n`) 플래그를 붙이면 됩니다. (예: `git commit -m "fix: 핫픽스" --no-verify`) 단, 보안과 코드 품질을 위해 남용은 절대 금물입니다!

## 🧬 프롬프트 해부 (Why it works?)

1.  **배경과 환경(Context)의 구체화:** 무작정 "훅 만들어줘"라고 지시하는 대신, 현재 사용 중인 기술 스택(`Husky`, `Node.js`)과 겪고 있는 페인포인트(잦은 빌드 실패, 리소스 낭비)를 명시하여 AI가 정확한 타겟팅(예: `lint-staged` 제안)을 할 수 있도록 유도했습니다.
2.  **성능 최적화 제약(Warning):** 쉘 스크립트 초보자가 가장 많이 하는 실수인 '전체 파일 검사'를 프롬프트 단에서 선제적으로 차단하여, 현업에서 즉시 쓸 수 있는 빠르고 가벼운 코드를 얻어냅니다.
3.  **크로스 플랫폼 제약(Constraints):** 다양한 OS를 사용하는 팀 환경을 고려하여 윈도우, macOS, 리눅스 모두에서 에러 없이 동작하도록 호환성을 강제했습니다.
