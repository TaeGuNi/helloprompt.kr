## 1. Intro

개발하다 보면 어느새 `if` 안에 `for` 안에 `if`가 중첩된 "괴물"을 마주하게 됩니다. 코드가 돌아가긴 하는데, 건드리기 무서운 스파게티 코드가 되어버리죠. 🍝
이런 코드는 가독성도 떨어지고 버그가 숨기 딱 좋습니다. "순환 복잡도(Cyclomatic Complexity)"를 낮춰야 한다고 머리로는 알지만, 어디서부터 손대야 할지 막막할 때가 많습니다.
AI에게 복잡한 코드를 던져보세요. 객관적인 수치로 복잡도를 분석해주고, 어떻게 분리하면 좋을지 리팩토링 전략까지 제안해줍니다.

## 2. TL;DR

- **핵심:** 중첩된 조건문과 반복문으로 꼬인 코드를 AI가 분석하여 단순화 방안을 제시합니다.
- **준비물:** 리팩토링이 필요한 복잡한 함수나 클래스 코드.
- **효과:** 코드 가독성 향상, 유지보수 비용 절감, 버그 발생 가능성 감소.

## 3. Solution (The Prompt)

복잡한 로직을 가진 코드를 아래 프롬프트에 넣어 분석해 보세요.

**🧶 코드 복잡도 분석 및 단순화 프롬프트**

```markdown
아래 코드는 로직이 꽤 복잡해 보입니다. 이 코드의 복잡도를 분석하고 단순화할 수 있는 방법을 제안해 주세요.

**코드:**
```

[여기에 복잡한 코드 붙여넣기]

```

**요청 사항:**
1. **복잡도 진단:** 이 코드의 순환 복잡도(Cyclomatic Complexity)가 높은지, 가독성을 해치는 주요 원인이 무엇인지(예: 깊은 중첩, 긴 함수 등) 설명해 주세요.
2. **리팩토링 전략:** 함수 분리(Extract Method), 조기 리턴(Early Return), 조건문 단순화(Guard Clauses) 등 구체적인 리팩토링 기법을 제안해 주세요.
3. **개선된 코드:** 위 전략을 적용하여 개선된 코드 버전을 작성해 주세요. 기존 로직과 동일하게 동작해야 합니다.

개발자가 이해하기 쉽게 단계별로 설명해 주세요.
```

## 💡 작성자 코멘트 (Insight)

저는 레거시 코드를 분석할 때 이 프롬프트를 먼저 돌려봅니다. AI가 "이 부분은 깊이가 5단계나 됩니다"라고 지적해주면, 리팩토링할 명분이 확실해지거든요. 특히 `Guard Clauses`(조건이 맞지 않으면 빨리 리턴해버리기) 패턴만 적용해도 코드가 훨씬 깔끔해지는 것을 볼 수 있습니다.

## 🙋 자주 묻는 질문 (FAQ)

**Q. 전체 프로젝트를 한꺼번에 분석할 수 있나요?**
A. LLM은 토큰 제한이 있으므로, 파일 단위나 핵심 함수 단위로 나누어 분석하는 것이 정확합니다.

**Q. AI가 로직을 바꿀 수도 있나요?**
A. 리팩토링은 동작을 유지하는 것이 원칙이지만, AI가 실수할 수 있습니다. 개선된 코드는 반드시 테스트 케이스(Unit Test)를 통해 검증하세요.
