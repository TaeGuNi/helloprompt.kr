# 📝 テストコード作成、5分で終わらせるプロンプト

- **🎯 推奨対象:** バックエンド開発者、フロントエンド開発者、QAエンジニア
- **⏱️ 所要時間:** 30分 → 5分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コーディングに最適)、GPT-4o

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐☆

> _「機能の実装だけで手一杯なのに、テストコードまで書く時間なんてない…」_

テストの重要性は頭で理解していても、いざ書くとなるとビジネスロジックの実装以上に膨大な時間がかかってしまう。これは、多くの開発現場で直面する深刻なジレンマです。
果てしなく続くエッジケース（境界値）の洗い出しや、外部API、データベース通信など複雑な依存関係のモッキング（Mocking）作業に頭を抱え、貴重なリソースを消耗していませんか？

これからは、完成したコードを貼り付けるだけで十分です。人間の目では見落としがちな異常系まで網羅した緻密で堅牢なテストコードを、AIがあなたに代わって一瞬で書き上げます。

---

## ⚡️ 3行要約 (TL;DR)

1. **カバレッジの最大化:** 正常系（Happy Path）はもちろん、人間が見落としがちな異常系（Edge Cases）まで網羅的にカバーします。
2. **モックの自動生成:** 外部APIやデータベース通信など、設定が煩雑な依存関係のモッキングを全自動で完結させます。
3. **主要フレームワーク対応:** Jest、PyTest、JUnitなど、実際の開発現場で標準的に使用されるあらゆるテストツールに最適化されています。

## 🚀 解決策：「単体テスト・ジェネレーター」

> **タスク:** 以下のコードの単体テスト（Unit Test）を作成してください。エッジケースを含め、可能な限り高いカバレッジを目指してください。
> **コード:** `[ここにコードを貼り付け]`

### 🥇 Pro Version (専門家形)

モック（Mock）が必要な複雑な関数や、現場の厳格なコーディング規約に完全に準拠した高品質なテストコードを生成したい場合にご活用ください。

> **役割 (Role):**
> あなたは、緻密で完璧主義なシニアQAエンジニアであり、テスト自動化のスペシャリストです。
>
> **状況 (Context):**
>
> - 対象言語: `[使用言語 (例: TypeScript)]`

## 💡 作成者のコメント (Insight)

AIにテストコードの作成を委ねる最大のメリットは、「テストの雛形（Boilerplate）作成」と「モックのセットアップ」にかかる時間を実質ゼロに圧縮できる点にあります。
特に、`jest.mock()` のような記述が冗長になりがちな処理や、複雑な依存性の注入（DI）をAIが一瞬で正確に組み上げてくれる恩恵は、現場において計り知れません。

**💡 実践Tips:**
AIが生成したコードをそのまま鵜呑みにして適用するのではなく、まずはAIが提示した「テストケースのリスト」を人間がレビューするプロセスを挟むことを強くお勧めします。
ビジネスロジック特有のドメイン知識（例：「特定のステータスの時は例外処理とする」といった独自の仕様）はAIが見落とす可能性があるためです。リストを確認した上で、不足しているケースを対話形式で追加指示すれば、プロダクトの仕様に完璧に寄り添った、極めて強固なテストコードに仕上がります。

## 🙋 よくある質問 (FAQ)

- **Q: レガシーコード（スパゲッティコード）に対してもテストを作成してくれますか？**
  - A: はい、可能です。ただし、依存関係が極端に複雑に絡み合ったコードの場合、AIが「まずはテストしやすい構造へのリファクタリングが必要だ」と指摘してくるケースがあります。その際は、プロンプト内でリファクタリング案の提示も併せて依頼するのが最も効果的なアプローチです。

- **Q: どのAIモデルを使用するのが最適ですか？**
  - A: コーディングタスクにおいては、現在 **Claude 3.5 Sonnet** が最も論理的でバグの少ないテストコードを生成する傾向にあります。次点で GPT-4o も非常に優秀な結果を出力します。

## 🧬 プロンプト解剖 (Why it works?)

1. **段階的な処理 (Chain-of-Thought):** タスクの最初のステップで「いきなりコードを書くのではなく、まずテストケースのリストを洗い出す」よう明確に指示しています。これによりAIの論理的な思考プロセスが整理され、抜け漏れのない高精度なコード生成が実現します。
2. **Mockingの強制:** 単体テストにおいて最もつまずきやすい「外部依存のモック化」をあらかじめ明示的に要求することで、そのままでは実行できない不完全なテストが生成される事態を未然に防ぎます。
3. **振る舞い駆動 (Behavior-Driven):** 制約事項にて「内部実装ではなく、公開された振る舞い（Public API）をテストする」よう指定しています。これにより、将来的なコードのリファクタリングにも耐えうる、保守性の高い堅牢なテストコードを生成させています。

## 📊 証明: Before & After

### ❌ Before (入力コード)

```javascript
// 単純な関数ですが、ゼロ除算の考慮が必要です。
function divide(a, b) {
  if (b === 0) throw new Error("0で割ることはできません。");
  return a / b;
}
```

### ✅ After (AIが生成したテストコード)

```javascript
describe("divide 関数のテスト", () => {
  // 1. 正常系のテスト（Happy Path）
  test("正常な数値が渡された場合、正しい割り算の結果を返すこと", () => {
    expect(divide(10, 2)).toBe(5);
    expect(divide(-10, 2)).toBe(-5); // 負の数のケース
  });

  // 2. 異常系のテスト（Edge Case）
  test("0で割ろうとした場合、エラーをスローすること", () => {
    expect(() => divide(10, 0)).toThrow("0で割ることはできません。");
  });

  // 3. 境界値・特殊値のテスト
  test("分子が0の場合、0を返すこと", () => {
    expect(divide(0, 5)).toBe(0);
  });
});
```
