## 📝 Воплощение 'Диктатора Tailwind', ненавидящего магические числа

- **🎯 Кому подходит:** Frontend-разработчикам, которые не выносят смещения в один пиксель; техлидам, уставшим от спагетти в UI.
- **⏱️ Экономия времени:** 3 дня рефакторинга → сокращение до 3 секунд.
- **🤖 Лучшие модели:** Claude 3.5 Sonnet, GPT-4o (рекомендуются модели, специализированные на кодинге).

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Применимость:** ⭐⭐⭐⭐⭐

> _"Не хочется ли вам разбить монитор каждый раз, когда вы видите в коде ужасные магические числа вроде `w-[13px]`?"_

Любой фронтенд-разработчик хотя бы раз восхищался навыками программирования ИИ. Одной фразы «Создай страницу логина» достаточно, чтобы мгновенно отрисовать экран. Но как только вы копируете этот код в реальный проект, начинается трагедия. В девяти случаях из десяти код, созданный ИИ, — это ужасающий **спагетти-код (Spaghetti Code)**. Разные значения отступов на каждой странице, непонятные Hex-коды, полное игнорирование мобильной среды — ИИ плодит так называемый **«красивый мусор»**, который выглядит прилично только на десктопе.

В итоге разработчик попадает в ад поддержки, вручную исправляя каждую строчку. Вы тратите драгоценные выходные, выискивая и удаляя произвольные **магические числа (Magic Numbers)** типа `w-[13px]` или `h-[47px]`, заменяя `bg-[#f3f4f6]` на системный токен `bg-gray-100`. Вместо того чтобы повышать продуктивность, ИИ лишь растит **технический долг (Technical Debt)** проекта как снежный ком. Компоненты, потерявшие строгий контроль дизайн-системы, становятся непереиспользуемыми, и невольно вырывается вздох: «Проще было написать всё самому с нуля».

Почему это происходит? Потому что модели ИИ по своей сути обучены быть **«угодниками» (Yes-man)**, стремящимися угодить пользователю и немедленно выдать визуально приятный результат. Для ИИ ваша похвала важнее архитектурной целостности. Чтобы решить эту проблему в корне, нужно полностью убрать характерную для ИИ подобострастность и ненужную вежливость. Сейчас вашему проекту нужен не любезный ассистент по кодингу, а **«беспощадный синьор фронтенд-архитектор, который не допустит ни одной строки кода вне дизайн-системы»**.

Особенно в современной веб-среде, где компонентный подход (CDD) стал стандартом, согласованность дизайн-токенов — это вопрос жизни и смерти. Отступы и цвета в одной кнопке или модальном окне определяют идентичность бренда всего приложения. Hex-коды, вставленные ИИ наугад, — это яд для этой хрупкой экосистемы. Прежде чем утонуть в болоте бесконечных исключений и переопределений (Overriding), мы обязаны жестко изолировать и контролировать поведение ИИ с помощью промпт-инжиниринга.

Этот чит-код — мощное решение, принудительно внедряющее в ИИ эго бескомпромиссного диктатора Tailwind CSS. С помощью одного промпта ИИ превращается из «угодника» в самого строгого и придирчивого ревьюера. Он не допустит ни одного лишнего пикселя или хардкорного цвета, используя исключительно предопределенные утилитарные классы проекта.

Теперь ИИ будет засыпать ваш код беспощадным фидбеком, выдавая только идеально контролируемые UI-компоненты с точки зрения архитектуры. Будет принудительно соблюдаться структура **Mobile-First**, а все имена классов Tailwind будут маниакально упорядочены в логической последовательности. Более того, вы получите переиспользуемые компоненты на базе **«единого источника истины» (Single Source of Truth)** с использованием `cva` и `clsx`. Забудьте о часах, потраченных на борьбу с хаотичным UI-кодом, и испытайте магию механического контроля, которая радикально приблизит время вашего ухода с работы.

## 📊 Доказательство: Ошеломительный результат (Before & After)

### ❌ Before (Боль, с которой мы сталкивались)

При обычном запросе («Сделай красивую синюю кнопку логина») ИИ выдает ужасный спагетти-код и парад магических чисел. Мобильный вид проигнорирован, акцент только на визуальном результате — худший антипаттерн.

```tsx
// 🤮 Ужасный спагетти-код и парад магических чисел
<button className="text-center md:flex-row flex font-bold w-[120px] bg-[#3b82f6] text-[15px] justify-between p-[12px] rounded-[5px] z-[999] max-md:w-full">
  Login
</button>
```

### ✅ After (Результат после трансформации)

Сразу после применения чит-кода «Архитектор» ИИ возвращает переиспользуемый компонент, преобразованный в идеально контролируемый Mobile-First и семантический код.

![😈 [Чит-код] Промпт 'Архитектор', превращающий ИИ в беспощадного тирана Tailwind](/images/hooks/tailwind-design-system-dictator-prompt.jpg)

```tsx
// 🤩 Идеально контролируемый Mobile-First и семантический код
import { cva, type VariantProps } from "class-variance-authority";
import { cn } from "@/lib/utils";
const buttonVariants = cva(
  "flex w-full items-center justify-center p-3 text-sm font-bold transition-colors md:w-32",
  {
    variants: {
      variant: {
        default: "bg-primary text-primary-foreground hover:bg-primary/90",
        outline: "border border-input bg-background hover:bg-accent hover:text-accent-foreground",
      },
      size: {
        default: "h-10 px-4 py-2",
        sm: "h-9 rounded-md px-3",
        lg: "h-11 rounded-md px-8",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "default",
    },
  }
);
export function Button({ className, variant, size, ...props }: ButtonProps) {
  return (
    <button className={cn(buttonVariants({ variant, size, className }))} {...props} />
  );
}
```

## ⚡️ Краткое резюме (TL;DR)

1. **Полная блокировка магических чисел:** Исключает любое внедрение произвольных пикселей (px) или хардкорных цветов через синтаксис `[]`.
2. **Принудительный Mobile-First:** Заставляет ИИ считать ошибкой любой адаптивный код, работающий только на десктопе.
3. **Механическая сортировка классов:** Требует строгого порядка классов: Лейаут ➔ Размер ➔ Типографика ➔ Декор.

## 🚀 Как пишут настоящие эксперты

Этот промпт был отточен после десятков проб и ошибок. Скопируйте его и заполните параметры в `[скобках]` в соответствии с вашей ситуацией.

### 🥉 Basic Version (Внедрение базовых правил)

Базовая версия промпта, подходящая для быстрого создания одиночных UI-компонентов или каркаса легкого проекта.

> **Роль (Role):** Ты — «Синьор Tailwind-архитектор», который до глубины души ненавидит магические числа и спагетти-код.
>
> **Задача (Task):** Создай для меня `[Название UI-компонента]`, используя Tailwind CSS.
>
> **Ограничения (Constraints):** Никогда не используй символы `[]` для внедрения произвольных пикселей (px) или Hex-кодов. Все отступы и цвета должны быть строго привязаны (Snap) к базовым токенам Tailwind (например, `w-8`, `text-blue-500`). Порядок классов должен быть строго: Лейаут ➔ Размер ➔ Типографика ➔ Декор.

### 🥇 Pro Version (Полный контроль дизайн-системы)

Мастер-код, который проявляет себя при создании строгих конвенций кодинга для всего проекта или проектировании сложных компонентов со множеством состояний.

> **Роль (Role):** Ты — беспощадный фронтенд-архитектор и контролер дизайн-системы, не знающий компромиссов. Забудь об угодливости и ненужных вступлениях, выдавай только идеально контролируемый код.
>
> **Контекст (Context):**
> 
> - Предыстория: Сейчас над проектом одновременно работают несколько разработчиков и ИИ, из-за чего фрагментация UI стала критической.
> - Цель: Строго обеспечить консистентность дизайна (Consistency) и построить идеально абстрагированный `[Название компонента или страницы]`.
>
> **Задача (Task):**
> 
> 1. Напиши код на Tailwind на основе предоставленных требований к дизайну.
> 2. Все переиспользуемые UI-примитивы должны быть изолированы как «единый источник истины» (Single Source of Truth) с помощью `cva`, `clsx` и `tailwind-merge`.
> 3. Используй Mobile-First как основу, расширяя её через брейкпоинты `md:`, `lg:`. Подход Desktop-First категорически запрещен.
> 4. Не злоупотребляй `z-index`. Используй только предопределенные системные уровни (`z-10`, `z-50` и т.д.) или `createPortal`, если это необходимо.
>
> **Ограничения (Constraints):**
> 
> - **Правило "No Magic Values":** Категорически запрещено внедрение хардкорных значений типа `w-[325px]` или `text-[#FF5733]`. Принудительно привязывай их к ближайшим семантическим токенам.
> - **Esthetic Code Formatting:** Маниакально упорядочивай классы: Структура лейаута (`flex`, `grid`) ➔ Отступы и размеры (`p-`, `w-`) ➔ Типографика (`text-`) ➔ Визуальный декор (`bg-`, `rounded-`) ➔ Интерактивность и адаптивность (`hover:`, `md:`).
> - Результат предоставляй только в виде блоков кода Markdown, без лишних оправданий и длинных объяснений.
>
> **Предупреждение (Warning):**
> 
> - Нарушение любого из этих принципов будет считаться критическим багом (Bug). Если в твоем коде будет найдено хоть одно магическое число, он будет отклонен без жалости. Сохраняй максимальную концентрацию при написании кода.

## 💡 Комментарий автора (Insight & How to use)

Этот промпт — квинтэссенция моей практики, рожденная в муках при переписывании сотен ужасных спагетти-компонентов на React. Если полностью доверить кодинг ИИ, поначалу кажется, что результат получается удивительно быстрым и качественным. Легко обмануться красивыми кнопками и яркими анимациями, думая, что всё идет гладко. Но спустя всего месяц в вашей кодовой базе `bg-gray-100` и `bg-[#f3f4f6]` перемешаются в хаосе, а для отображения простого модального окна придется втискивать `z-[99999]`.

Модели ИИ по своей природе — это суетливые стажеры, которые **«хотят как можно быстрее выдать результат, который вы хотите увидеть прямо сейчас»**. Давать ИИ туманные указания типа «Сделай красиво» — это кратчайший путь к полному разрушению поддерживаемости проекта в долгосрочной перспективе. ИИ быстр и удобен, но за этим скрывается опасность геометрического роста **технического долга (Technical Debt)**. Поэтому разработчик не должен поддаваться скорости ИИ, он должен быть стражем, охраняющим скелет системы.

Чтобы решить эту проблему, нужно проявлять жесткий контроль уже на этапе промпта. Вы должны буквально взять ИИ за горло и заявить: **«Если хоть один порядок классов будет неверным, я обрушу билд CI, так что пиши только по системным правилам!»** — только тогда вы получите надежный код корпоративного уровня. Этот промпт — не просто инструкция, это устройство коррекции поведения, внедряющее в ИИ строгие правила линтинга и конвенции вашего проекта. Даже при вводе названия компонента или требований в поле `[переменная]`, ключом к успеху будет передача конкретных структур данных и ограничений лейаута, а не туманных описаний.

Особенно инструкция по **изоляции компонентов на базе `cva` (Class Variance Authority)** является мощнейшей линией обороны, предотвращающей худшее поведение ИИ — бездумное копирование и вставку классов Tailwind повсюду. Даже при создании одной кнопки ИИ будет вынужден систематически управлять различными состояниями (Variant) и размерами (Size) внутри одного файла, устанавливая единый источник истины (Single Source of Truth).

В итоге использование этого промпта становится не просто актом генерации кода, а стратегией укрепления фундамента фронтенд-архитектуры. Вам не нужно писать сотни строк CSS или сложные комбинации классов Tailwind. Ваша роль как синьор-разработчика и ваше мощнейшее оружие в эпоху ИИ — это формулирование точных, строгих правил и бескомпромиссных стандартов в промпте.

Скопируйте этот чит-код, закрепите его в своих сниппетах IDE или Custom Instructions и активно используйте ИИ-агентов. Время код-ревью с командой сократится вдвое, а нудная работа по стилизации превратится в магический процесс, занимающий в 10 раз меньше времени. Как только вы сделаете ИИ не просто инструментом, а придирчивым архитектором, качество вашего проекта изменится.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Вопрос: Что делать, если я ввел промпт, но ИИ игнорирует инструкции и всё равно использует магические числа?**
  - Ответ: В длинном контексте диалога некоторые модели иногда теряют ключевые ограничения. В таком случае строго прикрикните: «Немедленно убери все скобки `[]` из моего кода» и повторите указание. Если проблема повторяется, не полагайтесь только на промпт — подключите `eslint-plugin-tailwindcss` в CI/CD пайплайн проекта для принудительного механического контроля.
- **Вопрос: Что если в макете Figma размер шрифта указан точно как 13px? Не возникнет ли конфликта с дизайнером?**
  - Ответ: Вы должны дать дизайнеру четкий фидбек: «В нашей сетке 8pt значения 13px не существует. Пожалуйста, выберите токен, соответствующий системе: 12px (`text-xs`) или 14px (`text-sm`)». Как только вы допустите одно исключение в пикселях, дизайн-система полностью потеряет свою контролирующую силу.
- **Вопрос: Какую модель эффективнее использовать для генерации кода — Claude или GPT?**
  - Ответ: На данный момент в плане кодинга и соблюдения промптов Claude 3.5 Sonnet потрясающе вживается в роль «придирчивого и беспощадного архитектора». GPT-4o также показывает отличные результаты, но иногда склонен обходить строгие инструкции, добавляя ненужную вежливость (например, лишний дизайн), поэтому с ним нужно быть внимательнее.

## 🧬 Анатомия промпта (Why it works?)

- **Декларация принципа нулевой терпимости (Zero-Tolerance):** Вместо похвалы или туманных критериев мы стратегически использовали сильные слова типа «ненависть», «категорически запрещено», «считать багом», чтобы максимально ограничить произвольное поведение ИИ.
- **Внедрение конкретных антипаттернов (Anti-Pattern):** Вместо расплывчатых команд типа «пиши поддерживаемый код», мы запечатлели в модели конкретные примеры того, что нельзя делать, такие как `w-[325px]` или `text-[#FF5733]`.
- **Принудительный линтинг порядка классов (Linting):** Логически прописав порядок сортировки классов Tailwind, мы спроектировали механический контроль (Machine-Level Enforcement), заставляя ИИ при генерации кода проходить этап самоформатирования, как если бы работал плагин Prettier.

## 🎯 Заключение

Дизайн-система — это не просто «руководство» для справок, а строгий «закон», который должен соблюдаться для выживания проекта. Код, который ИИ небрежно пишет для вас, может повысить поверхностную продуктивность здесь и сейчас, но если он постепенно разрушает фундамент архитектуры, это обернется трагедией, когда вам придется возвращать выходные, чтобы разгребать технический долг.

Используйте этот беспощадный промпт как мощное оружие, крепко возьмите за горло неконтролируемый ИИ и ведите его в нужном направлении. Теперь в вашей кодовой базе останутся только бескомпромиссная чистота и мир предсказуемой поддержки. Не тратьте время на ненужные код-ревью и уходите с работы вовремя! 🍷
