# 📚 死んだドキュメントを蘇らせる：Gemini 3 Proエージェントによるドキュメント自動化

- **🎯 おすすめの対象:** リードエンジニア、バックエンドエンジニア、テックリード、DevOpsエンジニア
- **⏱️ 所要時間:** ドキュメント修正1回あたり30分 → 0分（完全自動化）
- **🤖 おすすめのモデル:** Gemini 3 Pro (2M+の巨大コンテキスト必須)

- ⭐ **難易度:** ⭐⭐⭐☆☆ (CI/CD連携が必要)
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「コードは毎日デプロイされるのに、APIドキュメントは去年の夏のバージョンのまま止まっていませんか？今こそ、ドキュメントが『自ら』進化する時です。」_

開発者なら誰でも共感するはずです。コードは毎日変わるのに、ドキュメントは1ヶ月前、いや半年も前の状態にとどまっているという、あの惨憺たる現実に。「後で更新しよう」という誓いは言い訳に過ぎません。ドキュメントは、コードが修正された**その瞬間**に必ず一緒に修正されてこそ意味を持ちます。

しかし、人間は面倒くさがりな生き物です。だからこそ、私たちは**AIエージェント**にこの退屈で苦痛な作業を委任すべきなのです。過去のAIモデルは短いコンテキストの限界により、1つのファイルを修正した際にドキュメント全体の構造に与える影響を把握できませんでした。しかし、**Gemini 3 Pro**の圧倒的な推論能力と巨大なコンテキストウィンドウ（2M+）を活用すれば、話は変わります。

本日は、コードの変更点（Diff）を検知し、ドキュメントを自ら最新化する**「Self-Updating Docs」**ワークフローを完璧に構築するプロンプトを公開します。

---

## ⚡️ 3行まとめ (TL;DR)

1. コミット時に発生するコードの変更（Diff）を検知し、AIが自動でドキュメントを最新化する自動化パイプラインの構築。
2. 単純なタイポやリファクタリングは無視し、APIエンドポイントや環境変数の追加など、「実際にドキュメント化が必要な構造的変更」のみを正確にフィルタリング。
3. Human-in-the-Loop（人間の介入）を適用し、AIが修正したドキュメントは即時反映させず、PR（Pull Request）の形で開発者の最終レビューを経るように設定。

## 🚀 解決策：「ドキュメント管理エージェント (Docs Maintainer)」

### 🥉 Basic Version (基本型)

素早くPRのディスクリプションを作成したり、ローカルで簡単なREADMEの更新が必要な場合に使用する方式です。

> **役割:** あなたはシニアテクニカルライターです。
> **タスク:** 今私が修正した `[Git Diff]` の内容を確認し、`[README.md]` で修正すべき部分を見つけてマークダウンで再作成してください。単純なタイポやリファクタリングに関連する変更は無視してください。


### 🥇 Pro Version (専門家型)

CI/CDパイプライン（GitHub Actionsなど）に組み込み、完璧な自律型エージェントとして動作させるための精巧なプロンプトです。`temperature: 0.1` 以下に設定し、創造性を徹底的にコントロールしてください。

> **役割 (Role):** あなたは10年目のテクニカルライターであり、DevOpsエンジニアでもある「ドキュメント管理 (Docs Maintainer) エージェント」です。
>
> **状況 (Context):**
> 
> - 背景：開発者がたった今、コードベースに新しいコミットをプッシュしました。この変更には、APIエンドポイントの追加、新しい環境変数、あるいはコアとなるビジネスロジックの修正が含まれている可能性があります。
> - 目標：変更されたコードを緻密に分析し、既存の古いドキュメントを正確に最新状態へと同期させることです。
> 
> **指示 (Task):**
> 
> 1. 提供された `[Git Diff]` と `[Target Document]` の全体内容を比較・分析してください。
> 2. このコードの変更が、ユーザーのオンボーディングや他の開発者の環境構築に影響を与え、ドキュメントの更新が必要かどうかを判断してください。（単純なリファクタリング、変数名の変更、タイポの修正は徹底的に無視してください。）
> 3. 更新が必要だと判断した場合、既存のドキュメントのトーン＆マナー（語調やフォーマット）を完璧に維持したまま、修正されたドキュメント全体をマークダウン形式で再作成してください。
> 4. ドキュメントの最下部に `` の形式で隠しコメントを残してください。
> 5. もしドキュメントの更新が不要な単純なコミットであれば、ただ `[NO_UPDATE_REQUIRED]` とだけ出力してください。
> 
> **制約事項 (Constraints):**
> 
> - 絶対に存在しない機能や架空の環境変数をでっち上げないでください。（No Hallucinations）
> - 「適切に修正されました」のような曖昧な表現を禁止し、追加されたエンドポイントのURLや変数名など、具体的な値を明記してください。
> - コメントアウトされたTODO、FIXMEなどの未完成なコードは、絶対にドキュメント化の対象に含めないでください。
> - 出力は、「修正されたマークダウンドキュメントの全文」または「[NO_UPDATE_REQUIRED]」のいずれかのみでなければなりません。

## 💡 筆者コメント (Insight)

このプロンプトを実際の社内バックエンドリポジトリに適用してみて、最も衝撃的だったのは**「漏れていた環境変数の検知能力」**でした。開発者が `config.ts` に新しいサードパーティAPIキーを追加したものの、`README.md` の「ローカル環境構築」部分に記載するのを忘れてしまったことがありました。Gemini 3 Proはコードとドキュメントを照合し、見事に `README.md` へ `NEW_3RD_PARTY_API_KEY` の環境変数設定を自ら追加してくれたのです。

**実務適用のヒント:** すべての細かなコミットのたびにこのエージェントを回すと、トークンコストが爆発します。GitHub Actionsで `paths` フィルターを使用し、主要なロジックファイル（`src/api/**`, `config/**`）やDBスキーマが変更された時のみトリガーされるように設定しましょう。また、AIが修正したドキュメントはすぐに `main` ブランチへ自動プッシュせず、必ず開発者がコードレビューの段階で確認できるように**Pull Request (PR)**を作成させてください。人間の最終承認（Human-in-the-Loop）は、ドキュメントの信頼性を守る最後の砦です。

## 🙋 よくある質問 (FAQ)

- **Q: Git Diffだけ渡せば、AIは全体のコンテキストをすべて理解できますか？**
  - A: 短く明確な変更であればDiffだけでも十分ですが、複雑なロジックの変更時には見当違いなドキュメントを生成する確率が高くなります。Gemini 3 Proの巨大なコンテキストウィンドウを積極的に活用し、Diffだけでなく、変更されたファイルの**コード全体**と**ターゲットドキュメント全体**を一緒にInputとして投げるのが、精度（Accuracy）を劇的に高める核心的な秘訣です。

- **Q: コメントに書かれた開発者の個人的なメモやTODOまでドキュメント化しようとしてしまいます。どう防げばいいですか？**
  - A: AIが真面目すぎるために発生する問題です。プロンプトの**制約事項 (Constraints)**セクションに明記されているように、「コメントアウトされたTODO、FIXMEなどの未完成なコードは、絶対にドキュメント化の対象に含めないでください」という一文が必須なのはそのためです。この一行で綺麗に解決します。

- **Q: Gemini 2.5 ProやGPT-4oを使っても同じように動作しますか？**
  - A: 動作自体は可能です。しかし、「ドキュメント全体＋コードファイル全体」を流し込むアプローチにおいては、トークンの上限やコンテキストキャッシング（Context Caching）のコスト面で、Gemini 3 Proが圧倒的に有利です。ファイル単位が小さいトイプロジェクトであれば、GPT-4oでも素晴らしい結果を得ることができます。

## 🧬 プロンプトの解剖 (Why it works?)

1. **判断の保留 (Bypass Logic):** ドキュメントの更新が不要な単純なコミットの場合には `[NO_UPDATE_REQUIRED]` を返すように強力に統制し、不要なPRが無限に生成されるのを根本から遮断しました。
2. **トーン＆マナーの維持指示:** 既存のドキュメントのスタイルを無視し、AI特有の堅苦しく違和感のある口調で上書きされる現象を防ぐため、「既存のドキュメントのトーン＆マナーを維持する」という明示的なコンテキストを与えました。
3. **隠された説明 (Hidden Explanation):** HTMLのコメント（``）を活用して修正理由を残させることで、後でレビュアーがPRを承認する際、AIの推論プロセスを非常に簡単かつ素早く検証できるようにしました。
