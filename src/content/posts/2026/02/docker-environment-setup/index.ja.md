> **技術スタック:** `[Node.js 20、NestJS、PostgreSQL]`

### 🥇 Pro Version (専門家形)

本番環境への安全なデプロイを見据え、極めて軽量かつセキュアなコンテナを構築したい場合に最適な、プロ仕様のプロンプトです。

> **役割 (Role):** あなたはコンテナオーケストレーションの専門家であり、セキュリティとパフォーマンス最適化に精通したシニアDevOpsエンジニアです。
>
> **状況 (Context):**
>
> - 背景: 自身のアプリケーションをDockerコンテナとしてパッケージングし、デプロイする予定です。ローカル開発環境と本番環境の差異を完全に排除することを目的としています。
> - 目標: 高度に最適化され、実運用に耐えうる堅牢なDocker設定ファイルを生成すること。

> **タスク (Task):**
>
> 1. 指定された技術スタックに最適化された **Dockerfile** を作成してください。

> **変数 (Variables):**
>

> - **キャッシュ最適化:** パッケージのインストールとソースコードのコピー工程を分離し、Dockerのレイヤーキャッシュを最大限に活用できる順序で記述してください。

## 💡 作成者コメント (Insight)

Dockerは「とりあえず動かす」だけであれば比較的簡単ですが、「本番環境で安全かつ軽量に稼働させる」ためには、高度な専門知識と細やかなチューニングが求められます。
特に、マルチステージビルドの導入やnon-rootユーザー（一般ユーザー）での実行設定は、システムのセキュリティ担保やインフラコスト（ストレージ容量や通信量）削減の観点から必要不可欠な要素です。
このプロンプトの最大の核心は、AIに単なるファイル作成を丸投げするのではなく、「マルチステージビルド」や「non-rootユーザー」、「レイヤーキャッシュの最適化」といった**必須の要件（制約事項）を、あえて専門用語を用いて明示的に指定している点**にあります。
AIは曖昧な指示を受けると、一番簡単で無難な（しかし最適化されていない）コードを出力しがちです。しかし、このようにDevOpsのプロフェッショナルが使う用語で制約を設けることで、AIが安易に妥協することを防ぎます。結果として、経験豊富なシニアDevOpsエンジニアが手作業で書き上げたかのような、高品質な設定ファイルを引き出すことが可能になるのです。
実務で新しい技術スタックを検証する際や、既存プロジェクトのインフラを刷新する際など、このプロンプトは煩わしい環境構築のストレスからあなたを解放する「最強の武器」となるでしょう。

## 📊 証明: Before & After

### ❌ Before (雑なプロンプト)

```text
Python 3.9, Django, PostgreSQLのDockerファイルを作って。
```

結果：ビルドツールから不要なキャッシュまで、あらゆる要素が一つの巨大なコンテナに詰め込まれてしまいます。結果として、イメージサイズが平気で1GBを超えるような、全く最適化されていない無骨な設定ファイルが生成されます。

### ✅ After (本プロンプトを使用)

```dockerfile
# 1. Builder Stage
FROM python:3.9-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 2. Production Stage
FROM python:3.9-slim
WORKDIR /app

# セキュリティのためのnon-rootユーザー作成
RUN useradd -m myuser
USER myuser

# ビルドステージから必要なパッケージのみをコピー
COPY --from=builder /root/.local /home/myuser/.local
COPY . .

# 環境変数の設定
ENV PATH=/home/myuser/.local/bin:$PATH

CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

結果：不要なビルドツールが完全に排除されるため、イメージサイズが劇的に削減されます。さらに最小権限（一般ユーザー）で安全に動作するよう設計されており、そのまま実運用に持ち込めるレベルのセキュアなコンテナ環境が瞬時に完成します。
