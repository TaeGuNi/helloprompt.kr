## 📝 面倒なAPI文書化、コードさえ渡せばSwaggerがあっという間に

- **🎯 おすすめの対象:** バックエンド開発者、フロントエンド開発者、QAエンジニア
- **⏱️ 所要時間:** 30分 → 1分に短縮
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro (コーディングに強いモデル推奨)

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「APIの実装は終わったのに、仕様書作成のせいで今日も帰れない…」_

APIの開発そのものより、ドキュメント作成に苦痛を感じているのは、決してあなただけではありません。
パラメータのデータ型から必須項目の有無、さらにはレスポンスのユースケースまでを一つ一つ手作業で記述していく作業は、もはや現実逃避したくなるほどの退屈な苦行です。

しかし、もう悩む必要はありません。これからは、完成したコントローラーのコードをそのままコピー＆ペーストするだけで十分です。AIが、フロントエンド開発者も大絶賛するほど美しく、そして正確なSwagger（OpenAPI）仕様書へと一瞬で変換してくれます。

---

## ⚡️ 3行要約 (TL;DR)

1. **コードベースの正確な抽出:** 既存のコードから、パラメータやレスポンスの仕様を1つの漏れもなく自動で抽出します。
2. **フォーマットの瞬時生成:** OpenAPI (Swagger) 準拠のYAML/JSON、あるいはMarkdown形式へと一瞬で変換を完了させます。
3. **リッチなモックデータ:** 誰もが直感的に理解できる説明に加え、フロントエンドの実装ですぐに活用できる実践的なレスポンス例を自動生成します。

## 🚀 解決策：「APIドキュメント自動生成プロンプト」

### 🥉 Basic Version (基本形)

とにかく手軽に、素早く結果だけが欲しい場合はこちらをご活用ください。

> **役割:** あなたは細部までこだわる几帳面なテクニカルライター兼シニアバックエンド開発者です。
> **タスク:** 以下のAPIコードを読み解き、フロントエンド開発者が即座に理解できるMarkdown形式のAPIドキュメントを作成してください。

### 🥇 Pro Version (専門家形)

実務レベルの緻密なディテールや、そのままSwaggerに組み込めるほどの高品質なアウトプットが求められる場合は、こちらを使用してください。

> **役割 (Role):** あなたは細部までこだわる几帳面なテクニカルライターであり、経験豊富なシニアバックエンド開発者です。
>
> **状況 (Context):**
>
> - 背景: 新しいAPIエンドポイントの開発が完了しましたが、フロントエンドチームへ共有するためのドキュメントが未整備の状況です。
> - 目標: 開発済みのAPIコードを基に、正確で誰もが理解しやすいAPIドキュメントを生成すること。
>
> **タスク (Task):**
>
> 以下のコードを分析し、**[希望するフォーマット: OpenAPI 3.0 YAML / Markdownなど]** の形式でドキュメントを作成してください。
> 成果物には、必ず以下の内容を含める必要があります：
>
> 1. **Endpoint & Method:** URLパスおよびHTTPメソッドを明記すること。
> 2. **Summary:** このAPIが果たす役割について、簡潔な要約を記載すること。
> 3. **Request Parameters:** Body、Query、Pathパラメータ（それぞれのデータ型、必須の有無、詳細な説明を網羅すること）。
> 4. **Response:** 成功時 (200/201) および失敗時 (400, 401, 404, 500など) の網羅的なレスポンス構造と、実業務に即したJSONの例示。
>
> **コード (Code):**
>
> `[ここにコントローラーやルーターの対象コードを貼り付けてください]`
>
> **制約事項 (Constraints):**
>
> - データ型（String、Integer、Booleanなど）を極めて明確に記述すること。
> - レスポンスの例（Example Value）には、`"string"` や `0` のような無意味なダミー値ではなく、実際のユースケースに沿った現実的なデータを使用すること。
> - コードから読み取れない独自のビジネスロジックや仕様を、勝手に推測して捏造しないこと（不明な場合はプレースホルダーを残すこと）。

## 💡 筆者のインサイト (Insight)

このプロンプトがもたらす真の価値は、何よりも**「フロントエンド開発者とのコミュニケーションコストを劇的に削減できる」**という点に尽きます。

実際の開発現場において、ドキュメントの些細な不備は「このパラメータは必須ですか？」「エラー発生時のレスポンス構造はどうなりますか？」といった、生産性のない無駄なやり取り（ピンポン）を無限に発生させます。特に、`Example Value`（レスポンス例）をAIに指示して**現実的なデータ**で埋めさせるアプローチは絶大な効果を発揮します。これにより、フロントエンド開発者はバックエンドの完全な実装を待つことなく、即座にモックサーバーを立ち上げてUI開発を前進させることができるのです。

さらに応用テクニックとして、複雑なNestJSやSpring Bootのコントローラー単体だけでなく、ZodやJoiなどの**バリデーションスキーマ**も一緒にコードブロックへ貼り付けてみてください。AIがそれらの制約を読み取り、驚くほど正確で堅牢なドキュメントを生成してくれるはずです。ぜひ日々の業務で試してみてください。

## 🙋 よくある質問 (FAQ)

- **Q: どのプログラミング言語やフレームワークでも使用可能ですか？**
  - A: はい、全く問題ありません。Node.js (Express, NestJS) をはじめ、Python (FastAPI, Django)、Java (Spring Boot) など、言語やフレームワークの種類を問わず、AIがコードの背後にある意図を的確に汲み取り、汎用的なドキュメントへと昇華してくれます。

- **Q: 生成されたYAMLをそのままSwagger UIに組み込んでもエラーは起きませんか？**
  - A: 基本的にはそのまま使用可能ですが、AIはしばしば全体の構造（`components/schemas` など）を省略し、特定のエンドポイント部分のみを出力する傾向があります。そのため、既存の `swagger.yaml` にマージする際は、インデントのズレや参照構造（`$ref`）に破綻がないか、事前に軽く目視で確認することを強く推奨します。

- **Q: データベースのスキーマ（SQL）を渡すだけでもドキュメント化は可能ですか？**
  - A: データベースのスキーマ単体では、エンドポイントのURLやHTTPメソッドといったAPI固有の情報が欠落しているため、不完全なドキュメントになってしまいます。実用的な仕様書を生成するためには、必ずコントローラーやルーティングを定義したコードとセットで渡すようにしてください。

## 🧬 プロンプト解剖 (Why it works?)

1. **Role（役割）の付与:** 「テクニカルライター」と「シニアバックエンド開発者」という2つの強力なペルソナを同時に与えることで、エンジニアに求められる技術的な正確さと、ドキュメントとしての読みやすさの完璧な両立を実現しています。
2. **現実的なデータの要求:** 「実際のユースケースに基づいたデータを使用すること」という厳格な制約を設けることで、実務では全く役に立たない無意味なダミーデータが生成されるのを防いでいます。
3. **幻覚（Hallucination）の抑止:** 「コードから読み取れない仕様を勝手に作り出さない」というルールを明記することで、AIの想像による誤ったドキュメント生成を防ぎ、開発現場での致命的な混乱を未然にブロックします。

## 📊 証明: Before & After

### ❌ Before (入力したコード - Node.js Express)

```javascript
app.post("/users", (req, res) => {
  const { username, email } = req.body;
  if (!email) return res.status(400).send({ error: "Email is required" });
  // DB 保存ロジック...
  res.status(201).send({ id: 1042, username, email, createdAt: new Date() });
});
```

### ✅ After (結果 - Markdown形式を選択した場合)

### **POST /users**

新規ユーザー会員登録API

**Request Body**

- `username` (String, Optional): ユーザーの表示名
- `email` (String, **Required**): ログインおよび連絡用のメールアドレス（ユニーク）

**Responses**

```json
{
  "id": 1042,

}
```


## 🎯 結論

ドキュメント作成は、決して開発プロセスの「終わり」ではありません。むしろ、チーム全体とのスムーズな協業に向けた「始まり」に過ぎないのです。

しかし、そのスタートラインに立つためだけに、あなたの貴重なエンジニアリングのエネルギーを浪費する必要はもうありません。誰もが敬遠する退屈で面倒な作業はすべてAIに一任し、あなたは複雑なビジネスロジックの設計や限界を突破するパフォーマンスチューニングといった、エンジニア本来の最も価値あるコア業務に全力を注いでください！ 🍷
