## 📝 Рефакторинг грязного кода в чистый за 1 секунду

- **🎯 Рекомендуется для:** Junior- и Middle-разработчиков, а также Tech Lead-ов, уставших от бесконечных код-ревью
- **⏱️ Сокращение времени:** от 5 минут → до 1 секунды
- **🤖 Рекомендуемая модель:** Claude 3.5 Sonnet, GPT-4o, Gemini 2.5 Pro

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _«Я даже не могу узнать код, который сам написал месяц назад... Как вообще распутать это спагетти?»_

Каждый из нас хоть раз грешил непонятными названиями переменных в спешке перед дедлайном или создавал «функции-монстры» на сотни строк. А что, если бы у вас был личный ассистент, готовый мгновенно переписать код, за который стыдно перед коллегами, приведя его к строгим стандартам **«Чистого кода» (Clean Code)**? Всего один этот промпт позволит вам одним махом улучшить читабельность, производительность и легкость поддержки любого проекта.

---

## ⚡️ 3-строчное резюме (TL;DR)

1. Интуитивное переименование переменных и функций согласно индустриальным стандартам.
2. Устранение дублирующейся логики и строгое соблюдение принципа единственной ответственности (SRP).
3. Автоматическое применение современного синтаксиса и микрооптимизация производительности.

## 🚀 Решение: "Clean Code Refactoring"

### 🥉 Basic Version (Базовая версия)

Используйте этот вариант, когда нужен быстрый результат без лишних деталей.

> **Роль:** Ты Senior-разработчик с 20-летним стажем и признанный эксперт в написании чистого кода.
> **Запрос:** Преврати этот запутанный спагетти-код в легко читаемый, поддерживаемый и чистый код. Кратко объясни ключевые изменения.

### 🥇 Pro Version (Профессиональная версия)

Используйте этот вариант для глубокого рефакторинга критически важных участков кода. Скопируйте шаблон ниже, заполните переменные и отправьте ИИ.

> **Роль (Role):** Ты Senior Software Engineer с 20-летним опытом и абсолютный эксперт по Clean Code и архитектуре ПО.
>
> **Контекст (Context):**
>
> - Фон: Предоставленный мной код работает без ошибок, но он плохо читается, его трудно масштабировать и поддерживать. Имена переменных неясны, присутствует сильная связность (tight coupling) и много дублирующейся логики.
> - Цель: Получить идеальный, соответствующий индустриальным стандартам код, которым можно гордиться на Code Review.
>
> **Задача (Task):**
>
> Проведи рефакторинг предоставленного ниже кода в соответствии со следующими критериями:
>
> 1. **Именование (Naming):** Измени названия переменных и функций так, чтобы их назначение было кристально ясным без дополнительных пояснений.
> 2. **Принцип SRP:** Разбей большие функции на более мелкие, чтобы каждая из них выполняла только одну конкретную задачу.
> 3. **DRY (Don't Repeat Yourself):** Вынеси повторяющуюся логику в отдельные функции или утилиты.
> 4. **Комментирование:** Оставляй комментарии только там, где требуется объяснить «Почему» код написан именно так, а не «Что» он делает. Удали очевидные комментарии.
> 5. **Современный синтаксис:** Примени самые современные и безопасные конструкции целевого языка программирования (например, ES6+ для JS, паттерн-матчинг в Python 3.10+ и т.д.).
>

>

>
>
> **Ограничения (Constraints):**
>
> - Ни при каких обстоятельствах не меняй исходную бизнес-логику и поведение кода.
> - Не используй сторонние библиотеки, если они не были импортированы в оригинальном коде.

## 💡 Комментарий автора (Insight)

Этот промпт спасает меня как минимум пару раз в неделю, особенно когда я работаю с унаследованным (legacy) кодом или прототипами, написанными на скорую руку. Самая большая ценность здесь — это не просто визуальное причесывание синтаксиса, а строгое применение **SRP (Single Responsibility Principle)**. ИИ безошибочно определяет, когда функция берет на себя слишком много, и ювелирно разделяет её на независимые логические блоки.

_Практический совет:_ Всегда прогоняйте полученный результат через unit-тесты. Иногда нейросети слишком увлекаются оптимизацией и могут случайно зацепить пограничные случаи (edge cases) бизнес-логики.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Сработает ли это для редких или устаревших языков программирования?**
  - A: Да, базовые принципы чистого кода (понятные имена, короткие функции) абсолютно универсальны. Однако для очень специфичных языков (например, COBOL или старых версий Delphi) лучше дополнительно указать версию и диалект в самом промпте.

- **Q: Модель разбила мой код на 10 крошечных функций, и теперь в нем сложно ориентироваться. Что делать?**
  - A: Вы можете добавить в блок ограничений дополнительное правило: `- Не разбивай функции, если их длина не превышает 15 строк` или `- Сохраняй баланс между абстракцией и читабельностью сверху вниз`.

- **Q: Безопасно ли отправлять проприетарный код компании в публичные ИИ?**
  - A: Если вы используете бесплатные версии ChatGPT или Claude, ваш код может применяться для обучения моделей. Всегда удаляйте API-ключи, пароли и чувствительную коммерческую информацию перед отправкой, либо используйте корпоративные (Enterprise) тарифы с гарантией конфиденциальности данных.

## 🧬 Анатомия промпта (Why it works?)

1. **Явный запрет на изменение логики:** Жесткое ограничение «Не меняй бизнес-логику» страхует вас от неприятных сюрпризов, когда ИИ в попытках «оптимизировать» код случайно удаляет критически важный кусок алгоритма.
2. **Ориентация на «Почему», а не «Что»:** ИИ часто грешит добавлением мусорных комментариев вроде `// увеличиваем i на 1`. Прямой запрет на очевидные пояснения мгновенно очищает код от визуального шума.
3. **Требование списка изменений:** Вывод всех правок в виде удобного списка позволяет вам быстро провести мини-ревью предложенного кода, прежде чем бездумно копировать его в свою IDE.

## 📊 Доказательство: Before & After

### ❌ Before (Ввод)

```javascript
function calc(d) {
  let t = 0;
  for (let i = 0; i < d.length; i++) {
    if (d[i].price > 100) {
      t += d[i].price * 0.9;
    } else {
      t += d[i].price;
    }
  }
  return t;
}
```

### ✅ After (Результат)

```javascript
/**
 * Вычисляет общую стоимость товаров в корзине.
 * Применяет скидку 10% для товаров стоимостью выше 100 условных единиц.
 */
function calculateTotalOrderValue(items) {
  const DISCOUNT_THRESHOLD = 100;
  const DISCOUNT_MULTIPLIER = 0.9;

  return items.reduce((total, item) => {
    const currentPrice =
      item.price > DISCOUNT_THRESHOLD
        ? item.price * DISCOUNT_MULTIPLIER
        : item.price;

    return total + currentPrice;
  }, 0);
}
```

_Изменения: невнятное `calc` заменено на `calculateTotalOrderValue`, `d` превратилось в `items`, магические числа вынесены в константы с говорящими именами, а классический цикл заменен на декларативный метод `reduce`._
