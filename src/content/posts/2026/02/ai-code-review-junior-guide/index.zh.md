# 📝 像资深开发者一样进行代码审查：写给初级开发者的 AI 提示词

- **🎯 推荐对象：** 害怕代码审查的初级开发者（1~3年经验）、独自负责项目的独立开发者
- **⏱️ 预计耗时：** 代码分析 30 分钟 → 缩短至 1 分钟
- **🤖 推荐模型：** Claude 3.5 Sonnet（代码分析能力最强）, GPT-4o

- ⭐ **难度：** ⭐⭐☆☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用性：** ⭐⭐⭐⭐⭐

> _"我的代码里会不会隐藏着致命的 Bug？在点击 PR（Pull Request）按钮之前，你是否也曾惊出一身冷汗？"_

对于初级开发者来说，最紧张的时刻莫过于**代码审查（Code Review）**了。当被前辈问及“这个变量名为什么这样起？”或者“这里似乎存在内存泄漏的风险，你是故意这样设计的吗？”时，大脑往往会一片空白。反过来，当需要审查同事的代码时，常常会因为“我真的有资格指出问题吗？”的顾虑，最终只留下了一句“LGTM (Looks Good To Me)”便匆匆逃避。

但是，如果你把 **AI** 变成你的“私人导师”，情况就会完全不同。这不仅仅是让它帮你写代码，而是要求它：**“请用 10 年经验资深开发者的犀利眼光，提前审查我的代码。”** 本文将为你介绍 2026 年最高效的**“AI 资深代码审查员提示词”**，帮助你捍卫代码质量，并获得堪比技术面试级别的专业反馈。

---

## ⚡️ 3 句话总结 (TL;DR)

1. **通过预审建立自信：** 在提交 PR 之前，先让 AI 为你“把关”，提前修复缺陷，让你成为团队中基础扎实、值得信赖的开发者。
2. **多维度深度分析：** 指示 AI 从 Bug、安全漏洞、代码可读性、性能优化 4 个核心维度对代码进行全面解剖。
3. **移动端友好的反馈：** 强制要求 AI 以结构化的列表（List）而非表格（Table）形式输出，确保你在手机上也能轻松、清晰地阅读审查报告。

---

## 🚀 解决方案："召唤 10 年经验的严苛资深审查员"

### 🥉 Basic Version (基础版)

适用于只想快速检查整体代码质量的场景。

> **角色：** 你是一位拥有 10 年经验的资深后端开发者。
> **任务：** 请检查以下代码是否存在逻辑错误或可读性问题，并指出 3 个需要改进的核心点。
> **代码：** `[在此处粘贴你的代码]`

\

### 🥇 Pro Version (专业版)

适用于提交 PR 前夕，或需要分析晦涩难懂的历史遗留代码（Legacy Code）时，进行深度质量验证。

> **角色 (Role)：**
> 你是一位拥有 10 年以上经验、"严苛但耐心"的资深软件工程师（Senior Software Engineer）。你尤其擅长整洁代码（Clean Code）架构、安全漏洞分析以及系统性能优化。
> 
> **背景 (Context)：**
> - 现状：我是一名初级开发者，正在准备向团队提交一份 PR（Pull Request）。
> - 目标：我希望找出代码中隐藏的致命缺陷，并学习如何从资深开发者的视角，优雅地重构这段代码。
> 
> **任务 (Task)：**
> 请对以下 `[Input Code]` 进行彻底的代码审查（Code Review）。请从以下 4 个维度分析潜在问题，并提供具体的改进方案：
> 1. Bug 与逻辑错误 (Bugs & Logic)：运行时错误或边缘情况（Edge Case）处理遗漏。
> 2. 安全漏洞 (Security)：SQL 注入、XSS 攻击、越权漏洞等安全隐患。
> 3. 可读性与可维护性 (Readability)：命名规范、是否违反单一职责原则（SRP）。
> 4. 性能优化 (Performance)：不必要的计算、N+1 查询问题、内存泄漏。
> 
> **约束条件 (Constraints)：**
> - 为了保证移动端的阅读体验，输出格式必须使用**结构化的无序列表（Bullet List）**，严禁使用 Markdown 表格（Table）。
> - 提供“建设性反馈（Constructive Feedback）”，而非单纯的批评。
> - 必须耐心且详细地解释“为什么（Reasoning）”需要这样修改，确保初级开发者能够理解背后的原理。
> - 如果代码已经非常完美，请给予赞美并输出 "LGTM 🚀"。
> 
> **输入数据 (Input Code)：**
> ```python
> [在此处粘贴你的代码]
> ```

## 💡 作者见解 (Insight)

在实际工作中应用这套提示词后，我准备代码审查的时间比过去**缩短了 40% 以上**。特别是对于那些导师很忙，或者独自负责项目的初级开发者来说，这个提示词完美地扮演了“虚拟资深导师”的角色。

它最大的价值不仅在于帮你修改代码，更在于让你学到了**“为什么要这样改（Why）”**。基于审查报告中给出的“修改理由（Reasoning）”，你能够在团队面前逻辑清晰地阐述自己的设计意图，这极大地提升了团队对你的信任度。不过需要提醒的是：AI 建议的架构调整或第三方库引入，可能会与团队现有的开发规范发生冲突。因此，请不要盲目复制粘贴，而是将其作为有价值的“参考意见”。

## 🙋 常见问题 (FAQ)

- **Q: AI 总是建议我使用团队规定之外的最新语法或外部第三方库，怎么办？**
  - A: 你可以在提示词的背景（Context）部分追加约束条件。例如：`我们团队规定只能使用 Python 3.8 语法，且严禁引入外部库（如 pandas）。请仅使用内置模块进行优化。`

- **Q: 可以直接把公司的业务代码复制进去吗？我有点担心安全问题。**
  - A: **绝对不要**直接输入敏感信息（如 API 密钥、数据库密码、公司核心业务逻辑等）。建议将变量名进行泛化处理，或者仅截取剥离了业务属性的核心逻辑片段进行提问。如果是企业环境，强烈建议使用不会将数据用于模型训练的企业版（Team/Enterprise）订阅服务。

## 🧬 提示词解剖 (Why it works?)

1. **角色具象化 (Role)：** 不仅仅是赋予“开发者”的身份，而是增加了“10 年经验”、“严苛但耐心”、“整洁代码专家”等细节，从而完美掌控了反馈的深度与语气。
2. **强制多维度分析 (Task)：** 明确给出了 Bug、安全、可读性、性能这 4 个评判标准，有效防止 AI 仅仅停留在指出表面语法错误的浅层审查。
3. **移动端优化 (Constraints)：** 包含代码块的表格（Table）在手机屏幕上极易引发横向滚动，阅读体验极差。强制使用列表形式输出，充分考虑了 UX，让你随时随地都能舒适地浏览审查结果。

## 📊 效果对比：Before & After

### ❌ Before (输入)

```python
def get_user_data(user_id):
    db = connect_to_db()
    query = "SELECT * FROM users WHERE id = " + str(user_id)
    result = db.execute(query)
    if result:
        return result[0]
    else:
        return None
```

### ✅ After (结果)

```markdown
初级同学，编写这段代码辛苦了！整体逻辑走向是清晰的，但我发现了一些在生产环境中可能引发致命问题的地方，希望能帮你进一步完善代码：

- **🚨 安全漏洞 (Security): SQL 注入风险**
  - **问题所在：** 你使用了字符串拼接（`+ str(user_id)`）来生成 SQL 语句。如果恶意用户在 `user_id` 中输入 `1 OR 1=1` 等恶意载荷，整个数据库可能面临被拖库或篡改的风险。
  - **改进方案：** 必须使用参数化查询（Parameterized Query）。
  - **重构建议：** `query = "SELECT * FROM users WHERE id = %s"; db.execute(query, (user_id,))`

- **⚙️ Bug 与逻辑错误 (Bugs & Logic): 资源泄漏 (Resource Leak)**
  - **问题所在：** 代码中打开了数据库连接（`connect_to_db()`），但却没有在操作结束后关闭它。如果这段代码被高频调用，会导致数据库连接池耗尽，最终引发服务器宕机。
  - **改进方案：** 必须使用 `finally` 块或上下文管理器（`with` 语句）来确保数据库连接被正确释放。

- **📖 可读性 (Readability): 冗余的条件判断**
  - **问题所在：** `if result:` 的判断逻辑没有问题，但我们可以用更符合 Pythonic 风格的代码让它变得更简洁优雅。
  - **改进方案：** 可以简写为 `return result[0] if result else None`，或者直接利用很多框架自带的 `.fetchone()` 方法。
```
