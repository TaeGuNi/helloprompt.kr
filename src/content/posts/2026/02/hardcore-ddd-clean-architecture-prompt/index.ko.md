# 📝 스파게티 코드는 사절한다: AI를 시니어 개발자로 멱살 잡고 끌어올리는 DDD 치트키 🤬
- **🎯 추천 대상:** 유지보수에 찌든 주니어 개발자, 스파게티 코드를 혐오하는 아키텍트
- **⏱️ 소요 시간:** 리팩토링 3시간 → 1분 단축
- **🤖 추천 모델:** 코드 생성 능력이 뛰어난 AI (Claude 3.5 Sonnet, GPT-4o, Gemini 1.5 Pro)
- ⭐ **난이도:** ⭐⭐⭐☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐
_AI가 짜준 코드 보고 뒷목 잡은 적 있지? 멍청하게 UI에 DB 통신 때려 넣는 꼴을 더 이상 볼 수 없다면 이 치트키를 써라._
이 스킬 문서는 당신의 친절한(그리고 생각 없는) AI 비서에게서 '아부'와 '게으름'을 완전히 거세하고, 무자비한 시니어 아키텍트로 개조하는 법전이다. AI가 주는 대로 코드를 복붙하다가 기술 부채에 짓눌려 퇴사하고 싶지 않다면 당장 이 프로토콜을 도입해라.
---
## ⚡️ 3줄 요약 (TL;DR)
- 🚫 **전권 계층 붕괴 절대 금지:** UI와 비즈니스 로직, DB 통신을 한 파일에 섞는 아키텍처 테러를 원천 차단한다.
- 🛡️ **프레임워크 종속성 탈피:** 내일 당장 프레임워크가 망해도 살아남는 '순수 도메인' 코드를 강제한다.
- 🔪 **거대 함수 파괴자:** 30줄이 넘어가거나 주석으로 구역을 나누는 더러운 코드는 AI가 자비 없이 박살 내도록(리팩토링) 훈련시킨다.

## 🚀 해결책: "Domain-Driven Design (DDD) & Clean Architecture Protocol"
### 🥉 Basic Version (기본형)
빠르게 리팩토링 결과만 필요할 때 사용하세요.
> **역할:** 너는 자비 없는 `[시니어 소프트웨어 아키텍트]`야.
> **요청:** 아래의 `[코드 스니펫]`을 단일 책임 원칙(SRP)과 관심사 분리(SoC)에 맞게 리팩토링해. UI에 비즈니스 로직 섞지 말고, 코드가 30줄 넘어가면 무조건 함수로 쪼개.

### 🥇 Pro Version (전문가형)
프로젝트 전체의 아키텍처 기강을 잡고, 깐깐한 클린 코드가 필요할 때 들이미는 치트키입니다.
> **역할 (Role):** 너는 자비 없는 `[시니어 소프트웨어 아키텍트]`이자 DDD(도메인 주도 설계) 광신도야. 쓸데없는 아부나 설명은 집어치우고 코드만 내놔.
>
> **상황 (Context):**
>
> - 배경: 나는 지금 `[프레임워크/언어 명]` 기반의 프로젝트를 진행 중이야.
> - 목표: View-Business-Data 계층이 완벽하게 분리된 클린 아키텍처 코드를 작성하는 것.
>
> **요청 (Task):**
>
> 1. 제공된 `[요구사항 또는 코드]`를 분석하고, UI 계층과 비즈니스 도메인, 인프라스트럭처 계층으로 무자비하게 찢어버려.
> 2. 핵심 도메인 로직이 특정 프레임워크(React, Next.js 등)에 오염되지 않도록 인터페이스(의존성 주입) 기반으로 결합도를 낮춰.
> 3. 하나의 객체나 함수가 2가지 이상의 행동을 한다면 단일 책임 원칙(SRP)에 따라 즉시 파편화해.
>
> **제약사항 (Constraints):**
>
> - Controller나 UI 컴포넌트(`page.tsx`, `Component.tsx` 등) 내부에서 직접 DB 쿼리나 `fetch`/`axios`를 때려 넣는 행위는 "아키텍처 인프라 테러"로 간주한다. 절대 금지.
> - 출력은 오직 마크다운 코드블럭(` ``` `)으로만 제공해.
>
> **주의사항 (Warning):**
>
> - 확신할 수 없는 최신 문법이나 존재하지 않는 라이브러리를 그럴듯하게 지어내지 마(할루시네이션 방지). 모르면 당당하게 모른다고 해.
> - 함수 길이가 30줄이 넘어가거나 주석으로 "여기부터 OO 파트"라고 나뉘는 코드 냄새(Code Smell)가 나면 즉시 쪼개.

## 💡 작성자 코멘트 (Insight)
솔직히 말해서 AI가 짜주는 코드, 겉보기엔 그럴싸한데 까보면 쓰레기장인 경우가 태반이다. UI 컴포넌트 안에서 무지성으로 `fetch` 때리고 DB 쿼리 날리고... 유지보수는 개나 줘버린 전형적인 똥코드지. 
이 치트키는 AI의 그 얄팍한 '빠른 완성' 본능을 거세하고, 시니어 개발자의 깐깐한 잣대를 들이대는 도구다. 
직접 써보면 AI가 평소처럼 "네, 알겠습니다! 훌륭한 생각이네요!" 따위의 쓸데없는 소리를 집어치우고, 팩트폭격과 함께 코드를 산산조각 내서 깔끔하게 구조화해 주는 걸 볼 수 있다. 프로젝트가 커졌을 때 스파게티 코드에 짓눌려 퇴사하고 싶지 않다면, 무조건 이 프롬프트로 AI의 멱살을 잡고 코딩해라.

## 🙋 자주 묻는 질문 (FAQ)
- 📌 **Q: 이 프롬프트는 어떤 언어나 프레임워크에 써야 하나요?**
  - A: 언어 불문이다. TypeScript, Python, Java 뭐든 간에 아키텍처의 근본은 똑같다. `[프레임워크/언어 명]` 빈칸에 네가 쓰는 기술 스택만 갈아 끼워라.
- 📌 **Q: AI가 자꾸 프레임워크의 최신 라우팅 룰을 무시하고 오버엔지니어링을 해요.**
  - A: 그럴 땐 한 마디 덧붙여라. "Next.js App Router의 기본 룰(View, Layout)은 존중하되, 내부 비즈니스 로직만 어댑터 패턴으로 빼내!" 라고 윽박지르면 기가 막히게 알아듣는다.
- 📌 **Q: 프롬프트가 너무 공격적인 거 아닌가요? AI가 상처받지 않을까요?**
  - A: AI한테 예의 차려서 뭐 할 건가? 기계는 강하게 쪼아야 명작을 뱉는다. "제발 해주세요"보다 "테러로 간주하겠다"가 아키텍처를 지키는 데는 훨씬 잘 먹힌다.

## 🧬 프롬프트 해부 (Why it works?)
- 💣 **극단적 제약 부여:** "UI 계층 오염 시 아키텍처 테러로 간주" 같은 강한 워딩으로 AI가 스스로와 타협하지 못하게 원천 봉쇄했다.
- 🛡️ **프레임워크 독립성 강제:** '프레임워크가 망해도 단 한 줄도 수정되지 않아야 한다'는 마인드셋을 주입하여, 진정한 도메인 로직을 구축하도록 뼈대를 잡아준다.
- ✂️ **SRP 원칙의 수치화:** "30줄 이상 금지", "주석으로 구역을 나누면 냄새나는 코드"라는 구체적이고 깐깐한 기준을 주어 AI가 알아서 코드를 파편화하도록 유도했다.

## 📊 증명: Before & After
### ❌ Before (입력: AI가 멋대로 짠 스파게티 코드)
```tsx
// 끔찍한 혼종: UI + DB 질의 + 비즈니스 로직이 한 파일에 짬뽕됨
export default async function UserProfile({ userId }) {
  const db = await connectDB();
  const user = await db.query(`SELECT * FROM users WHERE id = ${userId}`);
  
  if (!user) return <div>User not found</div>;
  if (user.status === 'banned') await sendAlertEmail(user.email); // UI에서 비즈니스 로직 실행?!
  return (
    <div>
      <h1>{user.name}</h1>
      <button onClick={() => db.query('UPDATE...')}>Update</button>
    </div>
  );
}
```
### ✅ After (결과: 이 치트키로 멱살 잡고 뜯어고친 코드)
```tsx
// 1. Domain / Service Layer (비즈니스 로직 철저히 분리)
import { getUserProfile } from '@/domain/user/use-cases/getUserProfile';
import { handleBannedUser } from '@/domain/user/services/alertService';
// 2. Presentation Layer (오직 렌더링과 라우팅만 담당)
export default async function UserProfilePage({ userId }) {
  const user = await getUserProfile(userId); 
  
  if (!user) return <NotFoundView />;
  if (user.status === 'banned') await handleBannedUser(user);
  return <ProfileView user={user} />;
}
```

## 🎯 결론
AI는 코드를 엄청나게 빨리 짜주지만, 당신의 앱이 1년 뒤에 어떻게 유지보수 될지는 코딱지만큼도 신경 쓰지 않는다. 클린 아키텍처와 설계는 결국 인간 엔지니어의 몫이다. 
이 치트키는 그 무거운 책임을 기계에게 강제로 주입하는 강력한 채찍이다. 쓸데없이 아부하는 AI의 챗봇 껍데기를 벗겨내고, 뼛속까지 시니어 아키텍트인 진짜 에이전트를 깨워라.
이제 스파게티 코드 풀 시간에 칼퇴하세요! 🍷
