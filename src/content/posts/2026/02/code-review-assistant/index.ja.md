## 📝 AIで自分のコードの穴を見つけ出せ！(バグハンター)

- **🎯 推奨ターゲット:** メンターがおらず一人で奮闘しているジュニア開発者、本番デプロイ前の自分のコードに不安を感じる実務担当者
- **⏱️ 所要時間:** デバッグ3時間 → 10分に短縮
- **🤖 推奨モデル:** Claude 3.5 Sonnet (コード分析の最強ツール)、ChatGPT (GPT-4o)

- ⭐ **難易度:** ⭐⭐☆☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「自分のローカル環境では完璧に動くのに… 本番サーバーにデプロイした途端にクラッシュするのではないか？」_

一人でコーディングをしていると、「そもそも自分が何を知らないのかすら分からない」という恐ろしい盲点に陥りがちです。なんとか想定通りに動いてはいるものの、処理速度にボトルネックがあったり、セキュリティ上の重大な脆弱性が潜んでいたり、あるいは他人が読解不可能なスパゲティコードになっていたりする危険性があります。

これからは、24時間365日休むことのない「Googleのシニアエンジニア級」のAIレビュアーに、あなたのコードを徹底的に監査させましょう。潜在的なバグから非効率的なロジック、致命的なセキュリティの脆弱性に至るまで、プロの視点で鋭く指摘してくれます。

---

## ⚡️ 3行まとめ (TL;DR)

1. コードをむやみにAIへ投げるのではなく、**「Googleのシニアエンジニア」**という明確なペルソナを付与しましょう。
2. 単純に「レビューして」と頼むのではなく、バグ、セキュリティ、可読性、パフォーマンスという**4つの評価基準**を明記することが重要です。
3. AIの提案（リファクタリング結果）を盲信するのではなく、変更理由をコメントで解説させ、自ら検証することでスキルアップに繋げましょう。

## 🚀 解決策: "AIコードレビュアー"

### 🥉 Basic Version (基本型)

致命的な問題点だけを素早くスキャンしたい場合に活用してください。

> **役割 (Role):** あなたは実務経験10年のシニアバックエンド/フロントエンド開発者です。
>
> **タスク (Task):** 以下のコードから、致命的なバグや今すぐ改善すべきアンチパターン (Anti-pattern) を見つけ出してください。
>
> `[ここにコードを貼り付けます]`

### 🥇 Pro Version (専門家型)

コードの品質をプロダクション (Production) レベルまで引き上げ、客観的なリファクタリングの根拠までしっかりと確保したい場合に最適なプロンプトです。

> **役割 (Role):** あなたはGoogle出身で実務経験15年のシニアソフトウェアエンジニアです。クリーンコード (Clean Code) とシステム最適化の専門家として振る舞ってください。
>
> **状況 (Context):**
>
> - 言語: `[Python / JavaScript / Java など、使用言語を入力]`
> - フレームワーク: `[React / Spring / Django など、使用フレームワークを入力]`
> - 目的: この関数は `[ユーザーのログイン処理とセッション管理]` を担当しています。
>
> **コード (Input):**
>
> `[ここにレビューを依頼するコードを貼り付けます]`
>
> **タスク (Task):**
> 次の4つの観点から私のコードを厳格にレビューし、改善を反映したコード全体を提供してください。
>
> 1. **バグ (Bug):** 例外処理の抜け漏れや、実行時に発生しうる潜在的なエラー (Edge cases) の特定。
> 2. **セキュリティ (Security):** SQLインジェクション、XSS、CSRFなど、フレームワーク特有の脆弱性の有無。
> 3. **可読性 (Readability):** 変数や関数名の命名規則、単一責任の原則 (SRP) 違反の有無。
> 4. **パフォーマンス (Performance):** 不要なループ処理、時間計算量 (Big-O) の観点から見たボトルネックの排除。
>
> **制約事項 (Constraints):**
>
> - 改善されたコードを提示する際は、**なぜ (Why)** そのようなリファクタリングを行ったのか、各変更点ごとにコメント (`//` または `#`) で詳細に解説してください。
> - 既存のビジネスロジックの意図は絶対に損なわないようにしてください。
> - 外部ライブラリは追加せず、該当言語の組み込み機能のみを使用して最適化してください。
>
> **注意事項 (Warning):**
>
> - 確信の持てない構文や、非推奨 (Deprecated) の関数は絶対に提案しないでください。分からない場合は明確に「分からない」と答えてください。（ハルシネーションの防止）

## 💡 筆者のインサイト (Insight)

このプロンプトは、私が新人時代に「自分の書いたコードにどうしても自信が持てない時」に、絶大な精神的安心感を与えてくれた奥の手です。
特に「セキュリティ (Security)」と「例外処理 (Edge cases)」の項目は、人間の目では見逃しやすい細かな抜け漏れをAIが機械的にキャッチしてくれるため、大事故を何度も未然に防いでくれました。

ここで重要なのは、**AIが修正してくれたコードをそのまま思考停止でコピー＆ペーストしてはいけない**ということです。制約事項に「なぜ (Why) リファクタリングしたのかをコメントで説明して」という条件を加えた理由は、AIによるレビュープロセスを通じて、皆さん自身がシニアエンジニアの視座とアーキテクチャ設計の思考を学習するためです。

## 🙋 よくある質問 (FAQ)

- **Q: 会社のプロジェクトのソースコード全体を丸ごとコピーして渡してもいいですか？**
  - A: **絶対にダメです！** 会社のコードを外部のAIに送信することは、重大なセキュリティ違反（情報漏洩）に直結します。APIキー、DBのパスワード、そして会社の中核となるビジネスロジックは必ず `***` などでマスキング（秘匿化）した上で、レビューが必要な最小限の関数やクラス単位に切り取って入力してください。

- **Q: AIは常に完璧な正解を提示してくれますか？テストしなくてもそのままデプロイして大丈夫でしょうか？**
  - A: AIもハルシネーション（もっともらしい嘘）を引き起こす可能性があります。時には実在しないライブラリの使用を提案したり、構文エラーのあるコードを出力したりすることもあります。そのため、AIが提案したコードは、**必ずローカル環境でテストコードを実行し、実際の動作を検証**する必要があります。

## 🧬 プロンプトの解剖 (Why it works?)

1. **厳格なペルソナ付与 (Perspective):** 「Googleのシニアエンジニア」という最高レベルの基準を強制することで、単に「とりあえず動くコード」ではなく、「クリーンコード」の観点に基づいた質の高いフィードバックを誘導しています。
2. **多面的な評価基準 (Specific Goals):** 単純に「レビューして」と指示すると、AIは表面的なタイプミス程度しか指摘しません。「バグ」「セキュリティ」「可読性」「パフォーマンス」という4つの明確な評価基準を与えることで、立体的かつ網羅的な診断を下すように設計されています。
3. **学習を促す設計 (Explanation):** 正解のコードを受け取るだけでは、エンジニアとしての実力は伸びません。変更理由を逐一コメントで要求することで、ユーザー自身にアーキテクチャの意図を考えさせるチュータリング (Tutoring) 効果を最大化しています。

## 📊 証明: Before & After

### ❌ Before (入力: 初歩的なDB照会ロジック)

```python
def get_user_data(user_id):
    # ユーザーが入力したidをそのままクエリに連結 (危険！)
    sql = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(sql)
    return cursor.fetchone()
```

### ✅ After (結果: セキュリティと可読性が強化されたロジック)

```python
def get_user_data(user_id):
    """
    特定のユーザー情報を照会します。
    """
    # [Security] SQLインジェクション防止のため、Parameterized Query(バインド変数)を使用
    # [Readability] クエリ文を分離して可読性を向上
    query = "SELECT * FROM users WHERE id = %s"

    try:
        cursor.execute(query, (user_id,))
        result = cursor.fetchone()

        # [Bug] 結果がない場合の例外処理を追加
        if not result:
            return None

        return result

    except Exception as e:
        # [Bug] ランタイムエラーのロギング処理 (実際の環境ではloggerの使用を推奨)
        print(f"Database error occurred: {e}")
        return None
```

**レビューの要約:** ユーザーの入力をそのままクエリに結合すると、致命的なSQLインジェクション攻撃の標的となります。バインド変数 (`%s`) を使用してセキュリティのパッチを当てるとともに、DBの照会に失敗した際でもサービス全体がダウンしないよう `try-except` 構文による例外処理を追加しました。
