## 📝 JSON만 던지면 TypeScript 타입 정의가 짠! (feat. Zod)

- **🎯 추천 대상:** 프론트엔드 개발자, Node.js 백엔드 개발자, 타입스크립트 입문자
- **⏱️ 소요 시간:** 10분 → 10초 단축
- **🤖 추천 모델:** Claude 3.5 Sonnet, GPT-4o (코드 작성에 특화된 모델)

- ⭐ **난이도:** ⭐☆☆☆☆
- ⚡️ **효과성:** ⭐⭐⭐⭐⭐
- 🚀 **활용도:** ⭐⭐⭐⭐⭐

> _"수십 개의 필드가 중첩된 API 응답 JSON, 아직도 한땀 한땀 `interface` 치고 계신가요?"_

개발자라면 `interface User { ... }` 구조를 일일이 타이핑하며 오타를 찾느라 눈이 빠질 뻔한 경험이 한 번쯤은 있으실 겁니다. 특히 객체 안에 배열이 있고, 그 배열 안에 다시 객체가 존재하는 복잡한 중첩(Nested) 구조나 간헐적으로 반환되는 옵셔널(`null`, `undefined`) 값들은 예측 불가능한 런타임 에러의 주범이 되곤 합니다.

이제 백엔드 API 명세서에 있는 JSON 응답 데이터를 그대로 복사해 오기만 하세요. AI가 완벽한 TypeScript 타입(Type) 정의뿐만 아니라, 실시간 데이터 유효성 검증까지 책임지는 강력한 **Zod 스키마(Schema)**를 단 10초 만에 완벽하게 작성해 줍니다.

---

## ⚡️ 3줄 요약 (TL;DR)

1. **단 한 번의 복붙:** 가공되지 않은 JSON 데이터를 TypeScript 인터페이스와 타입으로 즉각 변환합니다.
2. **강력한 런타임 안정성:** 단순한 정적 타입 선언을 넘어, 실제 데이터 검증을 수행하는 Zod 스키마까지 동시에 생성합니다.
3. **섬세한 예외 처리:** 복잡한 중첩 객체의 분리, `null` 값에 대한 옵셔널(`?`) 처리, 명확한 JSDoc 주석까지 완벽하게 자동화합니다.

## 💡 작성자 코멘트 (Insight)

이 프롬프트는 결코 단순한 '코드 생성기' 수준에 머물지 않습니다. 프론트엔드 실무에서 가장 빈번하고 뼈아픈 이슈인 **'API 명세서와 실제 응답 데이터 간의 불일치'**를 완벽하게 방어해 내는 강력한 방패 역할을 합니다.

여기서 가장 주목해야 할 핵심은 2 Depth 이상 깊어지는 중첩 객체들을 각각 독립적인 `interface`로 분리하도록 강제한 대목입니다. 이렇게 구조를 분리해 두면, 특정 하위 객체만 React 컴포넌트의 Props로 전달해야 할 때 타입 재사용성이 폭발적으로 증가합니다. 

더 나아가, 백엔드 API가 `snake_case` 형태의 필드명을 반환하여 프론트엔드의 `camelCase` 컨벤션과 충돌하는 상황이라면 어떻게 해야 할까요? 프롬프트의 **제약사항(Constraints)** 영역에 *"모든 키 값을 camelCase로 자동 변환하는 Zod의 `.transform()` 로직을 추가해 줘"*라는 마법의 한 문장만 덧붙여 보세요. 반복적이고 지루한 데이터 매핑 작업이 즉각 사라지며, 여러분의 퇴근 시간은 최소 30분 이상 앞당겨질 것입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 프로젝트에서 Zod 대신 Yup이나 Joi를 사용 중인데 적용할 수 있나요?**
  - A: 물론입니다. 완벽하게 호환됩니다. 프롬프트 본문에 명시된 'Zod 스키마'라는 키워드를 사용하시는 라이브러리(예: 'Yup 스키마' 또는 'Joi 스키마')로 수정하여 실행해 보세요. AI가 즉시 해당 라이브러리의 공식 문법과 컨벤션에 맞춰 코드를 재작성해 줍니다.

- **Q: API 응답 JSON 데이터의 용량이 상당히 큰 편인데 전부 복사해서 넣어도 괜찮은가요?**
  - A: 모델의 텍스트 토큰 제한(일반적으로 수만에서 수십만 자) 범위 내라면 작동에는 문제가 없습니다. 하지만 배열 내부에 동일한 구조의 객체가 수백 개씩 반복되는 형태라면, 대표적인 샘플 객체 1~2개만 남기고 나머지를 삭제한 뒤 프롬프트를 실행하는 것을 권장합니다. 이렇게 하면 AI의 추론 속도가 빨라질 뿐만 아니라 결과물의 정확도 역시 비약적으로 상승합니다.

- **Q: AI가 자동으로 달아준 JSDoc 주석이 실제 비즈니스 도메인 로직과 어긋나면 어떻게 하나요?**
  - A: AI는 기본적으로 JSON의 '키 이름(Key)'을 기반으로 그 역할을 유추합니다. 따라서 `empNo`와 같은 사내 고유 축약어의 경우 해석이 다를 수 있습니다. 이를 방지하려면 프롬프트를 실행할 때 JSON 데이터 바로 아래에 `* 참고: empNo는 사원 번호를 의미합니다.`와 같이 간략한 도메인 힌트를 추가해 보세요. 훨씬 더 정교하고 의미 있는 문서화가 이루어집니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **Top-down 설계의 전략적 유도:** TypeScript `interface`를 먼저 선언한 후 Zod 스키마를 끼워 맞추는 방식이 아닙니다. **Zod 스키마를 최우선으로 설계하고, 이를 통해 `z.infer`로 타입을 자동 추출하도록(Task 1, 2)** 작업 순서를 엄격하게 강제했습니다. 이는 코드의 중복을 원천 차단하고 '진실의 단일 원천(Single Source of Truth)'을 유지하는 프론트엔드 생태계의 핵심 베스트 프랙티스입니다.
2. **명시적이고 견고한 예외 처리 기준 설정:** 실제 서비스 운영 중 가장 빈번하게 크래시를 유발하는 `null` 및 `undefined` 값에 대해 매우 명확한 제약 조건(Constraints)을 프롬프트에 부여했습니다. 이를 통해 AI가 어떠한 상황에서도 런타임 에러를 방어할 수 있는 견고한 코드를 작성하도록 유도합니다.

## 📊 증명: Before & After

### ❌ Before (입력 JSON)

```json
{
  "id": 1,
  "name": "Jay",
  "contact": {
    "email": "jay@test.com",
    "phone": null
  },
  "tags": ["developer", "admin"]
}
```

### ✅ After (결과)

```typescript
import { z } from "zod";

/**
 * 사용자 연락처 정보 스키마
 */
export const ContactSchema = z.object({
  /** 이메일 주소 */
  email: z.string().email(),
  /** 전화번호 (미입력 가능) */
  phone: z.string().nullable().optional(),
});

/**
 * 시스템 사용자 정보 스키마
 */
export const UserSchema = z.object({
  /** 사용자 고유 식별자 */
  id: z.number(),
  /** 사용자 이름 */
  name: z.string(),
  /** 연락처 정보 객체 */
  contact: ContactSchema,
  /** 사용자에게 부여된 태그 목록 */
  tags: z.array(z.string()),
});

// Zod 스키마를 통한 TypeScript 타입 추론 (Single Source of Truth)
export type Contact = z.infer<typeof ContactSchema>;
export type User = z.infer<typeof UserSchema>;
```
