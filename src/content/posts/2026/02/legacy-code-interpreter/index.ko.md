단순히 AI에게 "이 코드 좀 설명해 줘"라고 지시하면, 영혼 없는 교과서적 문법 해설만 돌아올 뿐입니다. 핵심은 AI에게 **'30년 경력의 산전수전 다 겪은 시니어 장인'이라는 강력한 페르소나를 부여하고, 직관적인 변수명 재정의와 치명적인 사이드 이펙트 검출을 명시적으로 강제하는 것**입니다.

## 🙋 자주 묻는 질문 (FAQ)

- **Q: 사내 보안 정책이 엄격한데, 회사 코드를 AI에 그대로 올려도 안전할까요?**
  - A: 절대 실데이터나 식별 가능한 정보를 입력해서는 안 됩니다! 실명, DB 테이블명, 내부 API 엔드포인트, 인증 키 등 민감한 정보는 `Table_A`, `User_Data`, `API_KEY` 등으로 철저하게 마스킹(Masking) 처리한 후, 순수한 로직 구조만 추려내어 분석을 요청하는 것을 강력히 권장합니다.

- **Q: 레거시 코드가 너무 방대해서 AI가 한 번에 분석하지 못하고 끊깁니다.**
  - A: LLM(대형 언어 모델)이 소화할 수 있는 컨텍스트 윈도우(Context Window)에는 명확한 한계가 존재합니다. 거대한 클래스나 수천 줄의 함수는 논리적 흐름에 따라 적절히 분리한 뒤, "이 블록은 주로 DB에서 유저 데이터를 조회하는 역할이야. 이 부분의 핵심 로직만 먼저 분석해 줘"와 같이 맥락을 쪼개어 단계별로 질문하세요.

- **Q: AI가 분석해 준 코드의 비즈니스 로직을 100% 신뢰해도 괜찮을까요?**
  - A: 절대 맹신해서는 안 됩니다. AI는 코드의 흐름을 짚어주는 압도적으로 뛰어난 '페어 프로그래머'일 뿐, 최종 책임자는 아닙니다. AI가 도출해 낸 로직의 뼈대를 바탕으로, 반드시 디버거를 연결해 직접 흐름을 추적하거나 단위 테스트(Unit Test)를 작성하여 스스로 교차 검증하는 과정을 거쳐야만 완벽한 내재화가 가능합니다.

## 🧬 프롬프트 해부 (Why it works?)

1. **압도적인 페르소나 부여 (Role):** AI에게 "30년 경력의 소프트웨어 장인"이라는 구체적인 정체성을 입히면, 단순한 신택스(Syntax) 번역기를 넘어 '엔터프라이즈 유지보수 관점'에서 아키텍처의 결함과 설계의 맹점을 꿰뚫어 보는 통찰력을 발휘하게 됩니다.
2. **인지 부하를 낮추는 구조화 (Task):** [핵심 요약 ➡️ 단계별 해설 ➡️ 지뢰 탐지]의 3단계 포맷으로 출력을 엄격하게 통제했습니다. 이를 통해 바쁜 개발자가 가장 시급하게 알아야 할 비즈니스 임팩트부터 빠르게 흡수할 수 있도록 가독성을 극대화했습니다.
3. **선제적 지뢰 탐지 (Constraints):** 레거시 코드를 건드릴 때 가장 두려운 존재는 예상치 못한 '사이드 이펙트'입니다. 수정 시 시스템을 셧다운 시킬 수 있는 잠재적 폭탄(타입 캐스팅 오류, 엣지 케이스 누락 등)을 프롬프트 단에서 미리 경고하도록 강제하여 실무 적용 시의 안전망을 겹겹이 구축했습니다.

## 📊 증명: Before & After

밋밋한 질문과 정교하게 엔지니어링 된 프롬프트의 결과물 차이는 하늘과 땅 차이입니다. 본 포스트에서 제공하는 최적화된 프롬프트를 활용하면 흩어져 있던 코드의 파편들이 구조화된 통찰로 변모하며, 원하는 답변을 얻기 위해 끝없이 질문을 수정해야 하는(Re-prompting) 불필요한 리소스를 극적으로 절약할 수 있습니다.

### ❌ Before (입력 코드 - 의도를 알 수 없는 C언어)

```c
int proc(int v) {
  int r = 1;
  while (v > 1) {
    r *= v--;
  }
  return r;
}
```

### ✅ After (AI의 완벽한 해설 결과)

**📝 핵심 요약:**
이 코드는 입력받은 특정 숫자의 **팩토리얼(Factorial)** 값을 계산하여 반환하는 수학 연산 함수입니다. (예: 5를 입력할 경우 5 × 4 × 3 × 2 × 1의 결과 도출)

**🔍 단계별 로직 해설:**

1. **상태 초기화:** 최종 연산 결과를 안전하게 담아둘 변수 `r`을 1로 초기화합니다.
2. **조건부 반복 곱셈:** 입력받은 매개변수 `v`의 값이 1보다 클 때까지 `while` 루프를 순회합니다.
   - 루프 내부에서 현재의 `v` 값을 `r`에 지속적으로 곱하여 누적 계산합니다 (`r *= v`).
   - 곱셈 연산이 완료된 직후, 후위 연산자를 통해 `v`의 값을 1씩 순차적으로 감소시킵니다 (`v--`).
3. **최종 산출물 반환:** 루프 조건이 종료되면 누적 계산이 완료된 변수 `r`을 반환하고 함수를 종료합니다.

💡 **직관적인 변수명 제안:** 불분명한 `proc` ➡️ `calculateFactorial`, `v` ➡️ `targetNumber`, `r` ➡️ `result`

**⚠️ 사이드 이펙트 경고 (크리티컬 지뢰 탐지):**

- **치명적인 메모리 오버플로우 위험:** 입력값 `v`가 13을 넘어가기만 해도 표준 정수형(`int`)의 최대 표현 범위를 초과하여, 완전히 왜곡된 음수 쓰레기 값을 반환하게 됩니다. 엔터프라이즈 환경에서 더 큰 스케일의 숫자를 다뤄야 한다면 반환 타입을 `long long`이나 `BigInt` 등으로 반드시 마이그레이션 해야 합니다.
- **엣지 케이스(음수/0) 방어 로직 누락:** 만약 매개변수 `v`에 0이나 음수가 주입될 경우, `while` 루프를 아예 타지 않고 묻지도 따지지도 않고 1을 반환해 버립니다. 이는 비즈니스 로직상 심각한 결함이 될 수 있으므로, 함수 최상단에 예외를 던지거나(Throw Exception) 유효성을 검증하는 방어 코드가 시급합니다.
