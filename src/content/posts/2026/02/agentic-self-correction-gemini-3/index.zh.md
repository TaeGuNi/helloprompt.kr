## 📝 AI Agent 为何失败：Gemini 3 Pro 的“自我修正循环”提示词模式

- **🎯 推荐对象：** 开发 AI Agent 的后端工程师、提示词工程师、产品经理
- **⏱️ 所需时间：** 代码调试从 15 分钟缩短至 1 分钟
- **🤖 顶级性能：** Gemini 3 Pro, GPT-4o, Claude 3.5 Sonnet（建议使用推理能力强的模型）

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐☆

> _“看似完美的 AI Agent 是否在重复同样的错误，白白消耗 API Token？现在是时候下达‘编写、批判、修正’的指令，而非仅仅是‘编写代码’了。”_

任何尝试使用 Gemini 3 Pro、GPT-4o 或 Claude 3.5 Sonnet 等顶级推理型 AI 模型构建 Agent 的开发者，可能都经历过令人抓狂的时刻。这就是被称为 **“死亡螺旋 (Death Spiral)”** 的无限错误循环现象。

起初看起来完美的 Agent 在遇到特定的边界情况或处理复杂逻辑时，一旦发生错误，就会像复读机一样陷入死循环，不断吐出同样的代码。盯着终端窗口里滚动的重复错误日志，看着宝贵的 API Token 预算白白浪费，那种挫败感刻骨铭心。

“是这个模型本来就这么笨吗？”，“是提示词太短了吗？”，“还是说必须引入像 LangChain 或 AutoGen 这样笨重复杂的框架才能解决问题？”

你可能会不断自责，修改了几十次提示词和代码，但 Agent 那固执的“愚蠢行为”却很难停止。持续到凌晨三点的无意义调试会消磨开发者的意志，生产环境的发布计划也会被无限期推迟。如果只是简单的文本生成倒也罢了，但在可能直接导致**系统致命故障**的代码生成或数据管道中，这种无限循环无异于一场伴随着巨大经济损失和服务中断的噩梦。我们等于是把基础设施的命运交给了 AI 的幻觉。

这些惨痛失败的原因绝非模型本身的智力限制或上下文窗口不足。问题的核心在于我们操作模型的方式，即提示词中缺失了一个关键要素：<span style="color:var(--color-cyber-cyan)">**“自我修正循环 (Self-Correction Loop)”**</span>。

大多数工程师和产品经理在下达指令时，都习惯采用“**发射后不管 (Fire and Forget)**”的方式，例如“帮我解决这个问题”或“编写这段代码”。模型按照指示尝试一次并得出结果后，无论结果对错，都会不负责任地将其丢给用户或下一个管道。

人类开发者在编写代码后会尝试编译、运行测试，并进行自审以检查逻辑漏洞，而我们却甚至不给 AI 一秒钟的“**验证时间**”，就强求它给出完美的答案。系统越复杂，初始草案 (Draft) 就越容易包含缺陷。如果不给它发现并纠正这些缺陷的机会，Agent 将永远停留在第一次失败中。

为了在生产级别稳定实现真正意义上的自主 Agent 工作流 (Agentic Workflow)，必须在提示词结构中设计一个模型在输出最终结果前，先进行自我批判和严苛验证的“**编辑器 (Editor)**”阶段。

通过同时赋予它“创造者 (Creator)”和“批判者 (Critic)”的双重人格，引导它找出自己代码中的逻辑漏洞。这种提示词架构的微小变化会产生惊人的蝴蝶效应。实际上，在引入这种模式后，曾因频繁的幻觉 (Hallucination) 和逻辑错误而头疼的复杂数据管道 Agent，其致命错误率大幅降低了 60% 以上。

今天，我们将公开这个在实际生产环境中经过无数次测试验证、最强大且实用的“**批判者-执行者 (Critic-Actor) 模式**”，它将终结吞噬你 Token 的无限循环噩梦。应用这项提示词工程技术后，你的 Agent 将不再是只会找借口的实习生，而是进化为能够独立承担责任并修复 Bug 的资深工程师。那么，让我们深入了解本质的解决方案吧。

## 📊 证明：立竿见影的效果 (Before & After)

### ❌ Before（我们曾经历的痛苦）

当使用标准提示词指示时，模型生成了只是简单地将 `list1 + list2` 后调用 `sorted()` 的低效代码。这会产生 O(N log N) 的时间复杂度，在大规模数据处理时可能会导致严重的性能下降。

```python
def merge_lists(list1, list2):
    # 产生 O(N log N) 的时间复杂度。大规模数据下存在严重性能下降的隐患。
    return sorted(list1 + list2)
```

### ✅ After（完美蜕变后的结果）

应用自我修正提示词后，模型在自我批判阶段敏锐地指出：“对已经排序的列表使用 `sorted()` 是低效的，为了实现 O(N) 复杂度，应该使用双指针 (Two-pointer) 方法。”随后，它立即给出了甚至完美防御了空列表边界情况的最优代码。15 分钟的代码审查和重构时间缩短到了 1 秒钟。

```python
def merge_lists(list1, list2):
    # 优化至 O(N) 时间复杂度，并完美防御空列表边界情况
    merged = []
    i, j = 0, 0

    while i < len(list1) and j < len(list2):
        if list1[i] < list2[j]:
            merged.append(list1[i])
            i += 1
        else:
            merged.append(list2[j])
            j += 1

    # 合并剩余元素
    merged.extend(list1[i:])
    merged.extend(list2[j:])

    return merged
```

## ⚡️ 3 行总结 (TL;DR)

1. AI Agent 的无限错误循环并非由于模型智力限制，而是因为**缺乏验证阶段 (Self-Correction)**。
2. 同时赋予 AI“创造者 (Creator)”和“批判者 (Critic)”的角色，让它自发地寻找并批判逻辑错误。
3. 将撰写草案、自我批判、最终修正这三个步骤融入一个提示词中，可以极大地减少**幻觉 (Hallucination)**。

将草案编写 (Drafting) 阶段和批判 (Critique) 阶段明确分离是该提示词的核心。请复制以下提示词，并将 `[变量]` 部分根据你的实际情况进行填充，立即投入实战。

### 🥉 基础版 (Basic Version)

当你只想快速查看自我修正结果时，可以轻量化使用。

> **角色 (Role)：** 你是一名 `[资深后端工程师]`。
>
> **任务 (Task)：** 请编写一个 `[合并两个已排序列表的函数]`。编写代码后，请自行寻找潜在的 Bug 或低效部分进行批判，并根据批判结果给出修正后的最终代码。

### 🥇 专业版 (Pro Version)

当你需要完美的细节代码质量验证和边界情况 (Edge Case) 防御时使用。

> **角色 (Role)：** 你是一名拥有 10 年经验的资深 Python 后端工程师，同时也是一名严谨的代码审查员 (Code Reviewer)。你将代码的整洁、高效和无 Bug 视为最高优先级。
>
> **背景 (Context)：**
> 
> - 背景：目前需要在生产环境中使用一段高可靠性的代码来处理大规模数据。
> - 目标：编写一个能够自行发现逻辑漏洞并进行完善的完美函数。
>
> **任务 (Task)：**
> 
> 1. **草案 (Draft)：** 编写一个 `[合并两个已排序列表的 Python 函数]`。
> 2. **批判 (Critique)：** 对编写的草案代码进行自审。重点找出以下项：
>    - 边界情况（例如：空列表输入、包含负数等）
>    - 性能瓶颈（时间/空间复杂度）
>    - 潜在的运行时错误
> 3. **修正 (Refine)：** 基于批判阶段发现的问题，重新编写完美的最终代码。
>
> **约束条件 (Constraints)：**
> 
> - 输出格式必须严格遵守下面提供的 Markdown 结构。
> - 在“3. 最终完成代码”部分，绝对不要对代码进行额外说明。仅提供代码块。
> - 最终完成代码必须包含针对批判阶段识别出的边界情况的异常处理逻辑。
>
> **输出格式 (Format)：**
> 
> ## 1. 初始草案 (Initial Draft)
> 
> \`\`\`python
> (草案代码)
> \`\`\`
> 
> ## 2. 自我批判 (Self-Critique)
> 
> - **批判 1：** (发现的问题及原因)
> - **批判 2：** (发现的问题及原因)
> 
> ## 3. 最终完成代码 (Final Polished Code)
> 
> \`\`\`python
> (最终修正的代码)
> \`\`\`

## 💡 作者点评 (Insight & How to use)

仅仅说“再检查一下代码”与使用具有明确结构的“**批判者-执行者 (Critic-Actor) 模式**”有着天壤之别。这个提示词之所以拥有如此惊人的威力，根本原因在于它强制提升了 LLM 的“**元认知 (Meta-cognition)**”能力。

在实务中使用此提示词时，最应关注的是对 `[变量]` 的精准控制和约束条件 (Constraints) 的细节设置。除了创建一个 Python 函数，该模式在所有视“**准确度 (Accuracy)**”为生命的知识工作中都能大显身手，包括编写涉及复杂业务逻辑的 **SQL 查询**、设计大规模**数据管道**，甚至是需要严密逻辑的法律文件或商务邮件文案撰写。

特别是“自我批判 (Self-Critique)”阶段，它扮演着最坚固、最安全的防波堤角色，预先拦截了 AI 常犯的无意义幻觉 (Hallucination) 和逻辑跳跃。正如人类很难立即发现自己文章中的错别字一样，语言模型也很难一次性察觉到自己刚生成的文本中内含的错误。但是，通过分离角色并指示它“**站在另一位资深工程师的角度彻底拆解这段代码**”，模型就会探索完全不同的权重路径，开始尖锐的批判。

当处理拥有巨大上下文窗口和压倒性推理能力的模型（如 **Gemini 3 Pro** 或 GPT-4o）时，这种模式的破坏力会成倍增加。分享一个实战技巧：在批判阶段，可以要求它直接引用现有代码库的特定行，或者在系统提示词中注入**公司内部编码规范指南**，然后指示它“批判是否有违反指南第 N 条的部分”。这样不仅能提高代码的准确性，还能极限提升与现有遗留系统的上下文一致率。

此外，通过强制要求“批判”部分的格式，可以从源头上封锁 AI 试图含糊其辞地说出“整体看起来不错”的倾向。在实务中，我无一例外地会在约束条件中明确要求**强制找出至少 3 个以上的潜在问题**。一旦创造了必须找茬的环境，AI 甚至能捕捉到我们未曾想到的罕见边界情况或内存泄漏点。这就是高级提示词工程的变量控制技巧。

如果应用了提示词但结果仍不尽如人意，或者 Agent 无法准确指出问题，该怎么办？这时应该检查“批判阶段”的指南是否过于抽象。与其简单地指示“进行批判”，不如在提示词中硬编码具体的检查清单，如“**内存泄漏可能性、死锁发生情况、SQL 注入漏洞、并发处理限制**”。这样，模型就会逐一对比清单，给出更犀利、更严苛的批判。

然而，作为一线工程师，必须指出一个致命的权衡 (Trade-off)。那就是**成本 (Cost) 和延迟 (Latency)** 的问题。由于该模式必然经历生成“错误的初始草案”和“自我批判分析内容”的过程，因此与直接要求答案的传统方式相比，它会消耗约 2 倍甚至近 3 倍的输出 Token (Output Tokens)。这意味着 API 调用成本的上升和响应速度的下降。因此，在简单的文本摘要、简单翻译、一次性正则转换等一维任务中滥用这种笨重的模式，无异于杀鸡用牛刀。

总而言之，该模式应战略性地引入到“**准确度高于成本 (Accuracy > Cost)**”的核心业务逻辑中，以及一旦出错可能导致致命故障的生产环境复杂推理管道中。我自己在实际运营的自动化 Agent 系统中选择性应用该模式的结果是：分析 Bug 报告和发布修正代码所浪费的大量加班时间和压力成本得到了极大节省，节省的金额远超 API Token 的费用。如果你想将宝贵的工程资源从调试地狱中解救出来，即使多付一点费用，也请抱着雇佣一名靠谱的“资深审查员”的心态，积极利用这一模式。

## 🙋 常见问题 (FAQ)

- **Q: 担心 Token 消耗量。有什么降低成本的方法吗？**
  - A: 请构建**多 Agent 路由 (Multi-Agent Routing)** 方式：在处理内部逻辑时使用相对廉价且快速的模型（如 Gemini 3 Flash）编写草案，仅将“批判与修正”阶段的严格监修角色交给 Gemini 3 Pro。这样可以在保持质量的同时大幅降低成本。

- **Q: 如果 AI 在批判阶段也找不到问题怎么办？**
  - A: 在提示词的任务部分注入具体的检查清单。例如明确指出“内存泄漏可能性”、“异步处理死锁”、“SQL 注入漏洞”等 AI 应重点审查的要点，批判的分辨率就会变得更加锐利。

- **Q: 除了编码，也能应用于策划或博客写作吗？**
  - A: 当然可以。你可以尝试应用“撰写草案 → 批判逻辑矛盾及读者画像不匹配 → 修正最终文案”的结构。即使没有人工干预，也能获得像专家仔细推敲过的高水平结果。

## 🧬 提示词解析 (Why it works?)

1. **Role & Context 分离：** 通过强制赋予 AI“开发者”和“审查员”两个相反的人格，激活了它自行发现代码缺陷的**元认知 (Meta-cognition)** 能力。
2. **引导步骤式推理 (Chain-of-Thought)：** 不让它立即给出完美答案，而是通过 Markdown 结构将“草案 → 批判 → 修正”的思考过程显式化，防止了逻辑跳跃和胡言乱语。
3. **Constraints (约束) 控制：** 通过强制在最终输出中移除所有不必要的说明，确保了后续自动化管道（如 CI/CD 等）解析 (Parsing) 代码并立即进行测试的便利性。
