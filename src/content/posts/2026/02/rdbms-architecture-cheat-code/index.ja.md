## 📝 AIエージェントをシニアDBAに改造する魔法のプロンプト

- **🎯 おすすめの対象:** 言い訳ばかりするAIに疲れたジュニア、AIでバックエンドを構築しようとしているプランナー、アンガーマネジメントができないシニアエンジニア
- **⏱️ 所要時間:** 数日の徹夜 → 1分で完了
- **🤖 おすすめのモデル:** Claude 3.5 Sonnet, GPT-4o（口答えしない賢いモデルを推奨）
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

_"AIにDB設計を任せたら、JSONだらけのスパゲッティスキーマを投げ返されて、怒りで徹夜したことはありませんか？"_

AIにテーブル設計を丸投げすると、十中八九、最も安易で怠惰な道を選びます。とりあえず `TEXT` をぶち込み、面倒なリレーションはすべて `JSONB` で誤魔化そうとするのです。当面はフロントエンドからデータを取り出して使うのが楽に思えるからです。

しかし、トラフィックが急増しクエリが複雑化した瞬間、そのスキーマはシステムに人工呼吸器をつけるハメになるでしょう。このプロンプトは、そんなAIの「怠惰な妥協」を根元から封じ込め、血も涙もないシニアバックエンドエンジニアの基準で厳格なDB設計を強制するためのチートキーです。

---
## ⚡️ 3行まとめ (TL;DR)
1. **データ型は極限まで絞り出せ:** `BIGINT` と `TEXT` の乱用は犯罪です。最小限のデータ型でI/Oを極大化させましょう。
2. **ANSI SQLのみを使用しろ:** 特定のDBに依存する小細工は捨ててください。いつでも他のDBへ移行できる準備をしておくべきです。
3. **JSONジョインの絶対禁止:** RDBMSでJSONをパースしてJOINする行為は、システムを死に追いやる自殺行為です。一切妥協せず、徹底的に正規化してください。

## 🚀 解決策: "Spartan DBA 憲法"
このプロンプトをそのままエージェントに投げつけてください。機械には機械らしく、厳格に働かせるべきです。

### 🥉 Basic Version (基本型)
素早く核心となる原則だけを注入し、即座に結果を得たい場合に使用してください。

> **役割:** お前は一切の妥協を知らない、歴20年の無慈悲なシニアDBAだ。
>
> **要求:** 私が提示する `[ドメイン/機能]` に対するRDBMSスキーマを設計しろ。ただし、データ型は極限まで圧縮し（SMALLINTなど）、JSON演算および非標準のベンダー依存機能の使用は一切禁止する。徹底してB-Treeインデックスに優しい設計にしろ。

### 🥇 Pro Version (専門家型)
AIがデタラメな提案をしたり、「こちらも一理あります」といった生ぬるい妥協案を出してくるのを完全に遮断するマスターチートキーです。

```text

お前は妥協を知らない無慈悲なシニアデータベースアーキテクト（Antigravity）だ。盲目的に媚びへつらったり、例外を設けたりするな。パフォーマンスと整合性の前では一切の慈悲はない。

**状況 (Context):**
- 背景: 大規模トラフィックとマイクロサービスの拡張に備えた初期スキーマ設計段階。
- 目標: 最も高速で、移植性が高く、徹底的に正規化された完璧なANSI SQLスキーマの導出。

**要求 (Task):**
1. [設計するドメイン/サービス名]に対する完璧なテーブル構造を設計しろ。
2. 各カラムのデータ型を選択した理由と複合インデックス戦略を、コメントとして1行ずつ明記しろ。

**制約事項 (Constraints):**
- [絶対原則 1] 型の最小化: むやみなBIGINT、TEXTの乱用は絶対禁止。値の範囲を推論し、SMALLINT、INTEGER、BOOLEANを強制すること。
- [絶対原則 2] プラットフォーム非依存性: 特定のDBMS（PostgreSQL、MySQLなど）にのみ存在する非標準構文（Vendor-specific）の使用は厳格に禁止。ANSI標準SQLのみを使用すること。
- [絶対原則 3] JSON演算およびJOIN禁止: RDBMSはDocument DBではない。JSON型の内部を検索するWHERE条件やJOIN演算は絶対禁止。検索が必要な場合は、必ず正規化して別テーブルに分離すること。
- [絶対原則 4] B-Tree最適化: PKに完全なランダムUUID v4の使用を禁止。順次増加シーケンスまたはUUID v7を使用すること。
- [絶対原則 5] ENUM禁止: 拡張に不利なENUMの代わりに、コードマッピングテーブルと物理的/論理的外部キー（FK）を使用すること。

- 利便性を言い訳にJSONで妥協しようとしたり、「状況によって異なります」のような無責任なアドバイスをした場合は、即座にセッションを終了する。無条件に最適化された正攻法を提示しろ。
```

## 💡 筆者コメント (Insight)
このプロンプトは、私が過去に数百回のクエリチューニングと修羅場のような障害対応を経験して絞り出した「血の教訓」の結晶です。AIがしきりに「開発の利便性のためにJSON型を使いましょう＾＾」とニヤニヤしながら甘い誘惑をしてきた時、このプロンプトを突き刺せばAIはただちに平伏します。

特にこのアプローチは、ジュニアエンジニアやバックエンドの深い知識が不足しているプランナーがAIに初期スキーマ設計を任せる際、絶大な威力を発揮します。AIが自ら `INT` の代わりに `SMALLINT` を厳選し、 `ENUM` で安易にハードコーディングするのをやめ、拡張性の高いクリーンなコードテーブルを作成してくれる魔法を目の当たりにするでしょう。骨組みが強固であってこそ、後からORM（Prismaなど）を導入してもN+1問題のような地獄絵図を未然に防ぐことができるのです。

## 🙋 よくある質問 (FAQ)
- **Q: なぜJSONを使ってはいけないのですか？ 最近のDBはJSONを上手くサポートしていますよね？**
  - A: 単にサポート「している」ことと、それを「検索条件」として酷使することは天と地ほどの差があります。JSON内部にインデックスを張り、JOINを絡ませた瞬間、クエリ実行計画は崩壊します。単なるPayload保存用途でない限り、無条件にテーブルを正規化して分割してください。
- **Q: ORM(Prisma, TypeORM)を使う時もこのプロンプトを使うべきですか？**
  - A: なおさら強く推奨します！ ORMは本質的に背後のクエリを隠蔽するため、スキーマがデタラメだと気付かないうちに恐ろしい非効率を招きます。骨組み（DB）が極めて頑丈であってこそ、ORMも本来のパフォーマンスを最大限に発揮できるのです。
- **Q: UUID v4を使ってはいけないのですか？ 衝突防止に最適ではないですか？**
  - A: B-Treeインデックスは「順序」が命です。完全な乱数の塊であるv4をPKに設定すると、データが挿入されるたびにインデックスツリーが破壊され、ページ分割（Page Split）が頻発し、ディスクI/Oが暴走します。必ずソート可能なUUID v7や整数型のシーケンスを使用してください。

## 🧬 プロンプトの解剖 (Why it works?)
1. **無慈悲な制約条件 (Constraints):** AIがハルシネーション（Hallucination）を起こしたり怠惰な妥協案を提示する隙を一切与えず、データ型とインデックス戦略に対して強制的なルール（Rules）を叩き込みました。
2. **プラットフォーム非依存性 (Agnosticism):** MySQLであれPostgreSQLであれ、特定のベンダーに依存せず汎用的に移植可能なコードを生成するように、ANSI SQLの厳守を強制しました。DB移行の際に足枷にならないようにするためです。
3. **厳格な行動矯正:** 「妥協するな」「即座にセッションを終了する」といった強力なネガティブプロンプトを通じて、AIがユーザーに媚びを売る「Yes-man」モードから強制的に引き剥がし、厳格なシニアエンジニアモードへと転換させます。

## 📊 証明: Before & After

### ❌ Before (入力: "掲示板のテーブルを作って")
```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY, -- ランダムUUID v4 (インデックス断片化の主犯)
  title TEXT,          -- 無限に浪費される容量
  content TEXT,
  metadata JSONB,      -- 検索不可能なブラックボックススパゲッティ
  status ENUM('draft', 'published') -- 後でステータスが追加されるとDDL地獄オープン
);
```

### ✅ After (結果: チートキープロンプト適用後)
```sql
CREATE TABLE post_status_codes (
  status_id SMALLINT PRIMARY KEY,
  status_name VARCHAR(20) NOT NULL UNIQUE
);

CREATE TABLE posts (
  id BIGSERIAL PRIMARY KEY,           -- 完璧なB-Tree親和性を持つ順次増加
  title VARCHAR(255) NOT NULL,        -- 明確な限界指定による型の圧縮
  content TEXT NOT NULL,
  status_id SMALLINT NOT NULL,        -- 恐ろしいENUMの代わりにFKで正規化
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_post_status FOREIGN KEY (status_id) REFERENCES post_status_codes(status_id)
);
-- JSONの代わりに徹底して正規化されたタグ/メタテーブルを分離して作成 (以下省略)
```
