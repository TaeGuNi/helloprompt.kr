## 📝 Найдите уязвимости в коде с помощью ИИ! (Охотник за багами)

- **🎯 Целевая аудитория:** Junior-разработчики без наставника, а также специалисты, сомневающиеся в качестве своего кода перед релизом.
- **⏱️ Экономия времени:** 3 часа отладки → 10 минут.
- **🤖 Рекомендуемые модели:** Claude 3.5 Sonnet (идеален для анализа кода), ChatGPT (GPT-4o).

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐⭐

> _«На моем локальном сервере всё работает идеально... но не рухнет ли боевой сервер после деплоя?»_

Когда вы пишете код в одиночку, легко оказаться в слепой зоне: вы просто не замечаете собственных ошибок. Ваш код может успешно компилироваться и даже работать, но при этом оставаться медленным, зиять дырами в безопасности или представлять собой хрупкий «спагетти-код», который невозможно масштабировать. Самое страшное — отправить такой код в продакшен и с ужасом ждать первых баг-репортов от реальных пользователей.

Но теперь у вас есть круглосуточный доступ к ИИ-ревьюеру уровня **Senior-инженера из Google**. Он не знает усталости, не уходит на обед и мгновенно подсвечивает скрытые уязвимости, неэффективную логику и критические бреши в безопасности, спасая вас от провальных релизов.

---

## ⚡️ Краткая выжимка (TL;DR)

1. Не скармливайте код ИИ без контекста: сразу задайте ему строгую роль **Senior-инженера из Google**.
2. Вместо банального «проверь код», требуйте анализа по **4 ключевым векторам**: баги, безопасность, читаемость и производительность.
3. Не копируйте слепо результаты рефакторинга: запрашивайте подробные комментарии с обоснованием каждого изменения, чтобы прокачивать собственные навыки.

## 🚀 Решение: Промпт «ИИ-ревьюер кода»

### 🥉 Базовая версия (Basic)

Используйте этот базовый промпт для молниеносного сканирования вашего кода на предмет критических ошибок.

> **Роль:** Ты — Senior Backend/Frontend разработчик с 10-летним опытом.
> **Задача:** Проверь следующий код на наличие критических багов и антипаттернов (anti-patterns), которые требуют немедленного исправления.
>
>
> [Вставьте ваш код]
>

### 🥇 Продвинутая версия (Pro)

Идеально подходит для доведения качества кода до стандартов Production и получения аргументированного рефакторинга.

> **Роль (Role):** Ты — ведущий инженер-программист (Principal Software Engineer) в Google с 15-летним опытом. Ты признанный эксперт в написании чистого кода (Clean Code) и проектировании высоконагруженных систем.
>
> **Контекст (Context):**
>
> - Язык программирования: `[Python / JavaScript / Java и др.]`
> - Фреймворк: `[React / Spring / Django и др.]`
> - Назначение: Эта функция отвечает за `[обработку авторизации пользователей и управление сессиями]`.
>
> **Входные данные (Input):**
>
>
> [Вставьте исходный код для проверки]
>
>
> **Задача (Task):**
> Проведи жесткий аудит моего кода по 4 ключевым аспектам и предоставь отрефакторенную, готовую к Production версию.
>
> 1. **Баги (Bug):** Упущенная обработка исключений или скрытые ошибки времени выполнения (Edge cases).
> 2. **Безопасность (Security):** Уязвимости фреймворка, такие как SQL-инъекции, XSS, CSRF и т.д.
> 3. **Читаемость (Readability):** Соблюдение правил именования переменных/функций и принципа единственной ответственности (SRP).
> 4. **Производительность (Performance):** Лишние циклы, узкие места и проблемы с временной сложностью алгоритма (Big-O).
>
> **Ограничения (Constraints):**
>
> - В улучшенной версии кода **обязательно** подробно объясни в комментариях (`//` или `#`), **почему (Why)** ты изменил каждую конкретную строку.
> - Категорически запрещено искажать или удалять первоначальную бизнес-логику.
> - Оптимизируй код исключительно встроенными средствами языка (нативными API), без использования сторонних библиотек.
>
> **Предупреждение (Warning):**
>
> - Не используй устаревшие (deprecated) методы и синтаксис. Если не уверен в решении — честно признайся в этом. Не выдумывай несуществующие функции (галлюцинации недопустимы).

## 💡 Комментарий автора (Инсайт)

Этот промпт стал моим **секретным оружием** в начале карьеры. Он дарил невероятное спокойствие в те моменты, когда я смотрел на свой свеженаписанный код и сомневался: «А точно ли здесь всё учтено?».

Особенно ценными оказались векторы **«Безопасность»** и **«Граничные случаи (Edge cases)»**. ИИ-ревьюер автоматически выхватывает неочевидные уязвимости, которые замылившийся человеческий глаз легко пропускает. Использование этого промпта не раз спасало меня от критических сбоев прямо на боевых серверах (Production).

**Главное правило:** никогда не применяйте предложенный рефакторинг вслепую (Copy-Paste). Требование _«объясни причины рефакторинга в комментариях»_ добавлено в промпт не ради красоты. Вдумчиво читая эти пояснения, вы буквально перенимаете образ мышления Senior-инженера. Это самый быстрый способ прокачать свою архитектурную насмотренность и перейти на новый уровень качества разработки.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Можно ли скопировать и отправить весь исходный код рабочего проекта компании?**
  - О: **Категорически нет!** Утечка проприетарного коммерческого кода — это грубейшее нарушение NDA и корпоративной безопасности. Перед отправкой обязательно маскируйте (например, `***`) любые API-ключи, токены, пароли к базам данных и уникальную бизнес-логику. Отправляйте на проверку исключительно небольшие, изолированные фрагменты: отдельные функции, классы или компоненты.

- **В: ИИ всегда выдает 100% правильный код? Можно ли обойтись без локального тестирования?**
  - О: Нет, современные нейросети по-прежнему подвержены галлюцинациям. ИИ может выдумать несуществующий метод библиотеки, ошибиться в логике синтаксиса или применить метод от другой версии фреймворка. Любой код, полученный от нейросети, вы **обязаны** запустить в локальной среде разработки и покрыть unit-тестами для окончательной верификации.

## 🧬 Анатомия промпта (Почему это работает?)

1. **Строгая ролевая модель (Perspective):** Назначая ИИ на должность «Senior-инженера из Google», мы отсекаем посредственные ответы. Это заставляет модель выдавать не просто «код, который компилируется», а элегантные решения, соответствующие жестким стандартам Clean Code.
2. **Многомерные векторы оценки (Specific Goals):** Если попросить нейросеть «просто проверить код», она исправит лишь пару банальных опечаток. Четко обозначив 4 строгих вектора (баги, безопасность, читаемость, производительность), мы инициируем глубокий, комплексный архитектурный аудит.
3. **Встроенный механизм наставничества (Explanation):** Бездумное копирование чужого кода тормозит ваш рост. Требование детально обосновать каждую строчку изменений заставляет ИИ выступать в роли вашего личного ментора, помогая осознать логику принятия инженерных решений.

## 📊 Доказательство: До и После (Before & After)

### ❌ До (Входные данные: наивная логика запроса к БД)

```python
def get_user_data(user_id):
    # Прямая подстановка ввода пользователя в запрос (ОЧЕНЬ ОПАСНО!)
    sql = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(sql)
    return cursor.fetchone()
```

### ✅ После (Результат: безопасный и читаемый код)

```python
def get_user_data(user_id):
    """
    Извлекает данные конкретного пользователя из базы данных.
    """
    # [Security] Использование параметризованного запроса (binding) для защиты от SQL-инъекций
    # [Readability] SQL-запрос вынесен в отдельную переменную для лучшей читаемости
    query = "SELECT * FROM users WHERE id = %s"

    try:
        cursor.execute(query, (user_id,))
        result = cursor.fetchone()

        # [Bug] Обработка граничного случая: пользователь не найден
        if not result:
            return None

        return result

    except Exception as e:
        # [Bug] Логирование ошибок времени выполнения (в Production рекомендуется использовать модуль logging)
        print(f"Database error occurred: {e}")
        return None
```

**Резюме ревью:** Прямая конкатенация пользовательского ввода со строкой запроса делала систему беззащитной перед SQL-инъекциями. Эта критическая уязвимость была устранена за счет использования параметризованных запросов (binding через `%s`). Кроме того, была внедрена конструкция `try-except` для перехвата исключений: теперь сбой при подключении к базе данных не приведет к падению всего микросервиса.
