# 📝 Redisキャッシング戦略：クエリの応答速度を100倍にするプロンプト

- **🎯 おすすめの対象:** トラフィックスパイクでDBのCPU使用率が100%に張り付くアラートを経験したバックエンドエンジニア、大規模なアクセス集中が予想されるイベントを控えるインフラ担当者

- **🤖 推奨モデル:** Claude 3.5 Sonnet（複雑なアーキテクチャや並行性制御のコード生成に最適）、GPT-4o

> _「DBスケールアップの稟議書を提出する前に、少し立ち止まってみてください。それは本当にDBの限界でしょうか？それとも、全く同じデータを何万回も無駄に読み込んでいるだけではありませんか？」_

ディスクベースのリレーショナルデータベース（RDBMS）にすべての読み取りリクエストを集中させるのは、例えるなら、本を読むたびに広大な図書館の奥底にある書架までわざわざ足を運ぶようなものです。一方、インメモリデータストアであるRedisを導入すれば、よく読む本を常に自分のデスクの上に置いてすぐ手に取れるように、劇的なパフォーマンスの向上を実感できるはずです。

これまで数十から数百ミリ秒（ms）かかっていたクエリの応答時間を1ms以下へと短縮し、サーバーの負荷を劇的に下げる**「魔法のキャッシングアーキテクチャ設計プロンプト」**をご紹介します。AIの力を最大限に活用して、堅牢かつ障害に強いRedis連携コードを瞬時に生成しましょう。

## ⚡️ 3行まとめ (TL;DR)

1. **Look-aside（遅延読み込み）:** 最も王道なパターン。まずはキャッシュを確認し、データが存在しない場合のみDBへクエリを発行して、その結果をキャッシュに格納します。
2. **Write-back（後書き込み）:** 書き込みトラフィックが極端に多い環境で有効。まずは高速なメモリ上に書き込みを行い、その後、非同期でDBへ一括保存（バッチ処理）します。
3. **Cache Stampede（キャッシュ暴走）対策:** キャッシュの有効期限が切れた瞬間にDBへのアクセスが殺到するのを防ぐため、「Mutex Lock」や「確率的早期再計算（PER）」のロジックをAIに実装させます。

## 🚀 解決策: "Cache Strategy Prompt"

### 🥉 Basic Version (基本型)

単純な参照APIへ素早くキャッシュを適用したい場合に使用します。ORMのクエリをラップする、簡潔なラッパー（Wrapper）コードを生成します。

> **役割:** あなたはシニアバックエンドエンジニア（Node.js / NestJSの専門家）です。
> 
> **リクエスト:** 以下の関数にRedisの `Look-aside` パターンキャッシングを適用してください。
> 
> async function getUserProfile(userId) {
>   return await db.users.findOne({ id: userId });
> }
> 
> **条件:**
> 
> - Redisのキーフォーマットは `user:profile:{userId}` に指定してください。
> - TTL（有効期限）は10分（600秒）に設定してください。
> - キャッシュミス（Cache Miss）時にDBから参照し、Redisに保存するロジックを明確に実装してください。
> - 例外処理（Redisの接続に失敗した場合でもDBの参照は正常に動作すること）を必ず含めてください。

### 🥇 Pro Version (専門家型)

毎秒数万件ものアクセスが殺到するグローバルサービスや、先着順のチケット販売システムなどを設計する際に必須となるプロンプトです。単なるキャッシングの域を超え、**システムダウンを未然に防ぐための防御的プログラミング**をAIに要求します。

> **役割 (Role):** あなたは毎秒10万件以上のトラフィックを処理する、グローバルサービスの大規模分散システムアーキテクトです。
> 
> **状況 (Context):**
> 
> - **ドメイン:** 人気アイドルのコンサートの先着順チケット販売サービス。
> - **問題状況:** 特定のコンサート情報のキャッシュ（TTL）が期限切れになるまさにその瞬間、待機していた数万人のリクエストがキャッシュミスを引き起こし、同時にDBへ殺到する**Cache Stampede（キャッシュの暴走）**現象が懸念されています。
> 
> **リクエスト (Task):**
> 
> 1. **Mutex Lockの実装:** Redisの `SETNX`（またはRedlockアルゴリズム）を活用し、キャッシュが期限切れになった際、ただ1つのスレッド/プロセスのみがDBにアクセスしてデータを更新するように、分散ロックのロジックを作成してください。
> 2. **PERアルゴリズム (Probabilistic Early Recomputation):** TTLが完全に終了する前に、一定の確率でバックグラウンドにてキャッシュを事前に更新し、遅延時間のスパイクをなくすコードを実装してください。
> 3. **Circuit Breakerの設計:** Redisクラスターに障害が発生した場合でもシステム全体の障害に波及させず、迂回するかGraceful Degradation（機能低下状態でのサービス維持）処理を行う手法を提案してください。
> 
> **制約事項 (Constraints):**
> 
> - 使用言語/フレームワーク: `[TypeScript / NestJSなど、ご自身のプロジェクト環境を入力してください]`
> - 成果物は単なるコードスニペットではなく、サービスレイヤー（Service Layer）に直接適用可能な構造化されたクラス形式で作成してください。
> - コメントを通じて、なぜこのような防御ロジックが組み込まれたのかを詳細に説明してください。

## 💡 筆者コメント (Insight)

Redis導入時、ジュニアエンジニアが最も陥りやすい罠が「キャッシュインフラへの無条件の信頼」です。Redisはあくまで**「揮発性」**のインメモリデータストアにすぎません。決済履歴やユーザーのパスワードといった、絶対に消失してはならない「信頼できる唯一の情報源（Single Source of Truth）」をRedisのみに依存するのは、致命的な障害の引き金となります。

実務において、Redisにキャッシュすべきなのは**「いつ消失してもDBから再構築できるデータ」のみ**です。さらに、Pro版プロンプトでも強調したように、Redisサーバーのダウンがシステム全体の停止（単一障害点：SPOF）に直結するのを防ぐため、`try-catch`を用いたDBへの直接参照（フォールバック）ロジックを実装することは、シニアエンジニアとして譲れない必須要件と言えます。AIにコードを生成させる際にも、こうした「防御的な視点」を指示に必ず盛り込むよう心がけましょう。

## 🙋 よくある質問 (FAQ)

- **Q: サーバーのローカルメモリ（グローバル変数やMapなど）に直接キャッシュしてはいけませんか？**
  - A: サーバーが1台のみ稼働する小規模環境であれば問題ありません。しかし、トラフィック増加に伴いサーバーを複数台にスケールアウトさせた途端、ユーザーが接続したサーバーごとにキャッシュ内容が異なる「データ不整合」の温床となります。分散環境全体で状態を一元管理するグローバルキャッシュストアとして、Redisのような外部システムを導入する最大の理由はここにあります。

- **Q: データの有効期限（TTL）の設定基準はどのように決めるべきですか？**
  - A: データの「変更頻度」と「ビジネス上の重要度」のバランスを見て決定します。リアルタイム性がそこまで求められないお知らせやマスターデータは1時間〜1日、ユーザーのプロフィール情報は5〜10分、秒刻みで変動するランキングや株価などのデータは10秒前後と極端に短く設定するのが、実務におけるベストプラクティスです。

- **Q: キャッシュ戦略を導入したのに、DBの負荷が一向に下がらない場合はどうすればいいですか？**
  - A: キャッシュのヒット率（Cache Hit Ratio）が極端に低いか、あるいは書き込み（Write）トラフィックが圧倒的に多いケースが考えられます。このような場合は、読み取り中心の「Look-aside」ではなく、メモリ上で変更内容を一時的にプールし、定期的にDBへ一括反映させる「Write-back」パターンの導入を検討すべきです。AIに「Write-backパターンの具体的な実装方法」を追加で質問して掘り下げてみてください。

## 🧬 プロンプトの解剖 (Why it works?)

1. **正確なアーキテクチャパターンの明示:** プロンプト内に `Look-aside`、`Mutex Lock`、`Circuit Breaker` といったバックエンド特有の専門用語（デザインパターン）を意図的に組み込んでいます。LLMはこれらのキーワードをトリガーとして認識し、学術的かつ実務で十分に検証された、堅牢な構造のコードを優先的に出力します。
2. **最悪のエッジケースの想定:** システムが平常稼働している前提ではなく、「障害発生時」や「アクセス暴走（Cache Stampede）」の真っ只中にあるという厳しい状況（Context）を明確に与えました。これにより、単なるキャッシュのサンプルコードではなく、AIに**実務レベルの防弾（Bulletproof）ロジック**まで深く考慮させることに成功しています。
