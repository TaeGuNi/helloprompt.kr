# 📝 Написание кода тестов, промпт, который заканчивает за 5 минут

- **🎯 Рекомендуется для:** Разработчиков, QA-инженеров, тимлидов
- **⏱️ Требуемое время:** 5 минут (вместо часов рутины)
- **🤖 Рекомендуемая модель:** Любая современная LLM (GPT-4o, Claude 3.5 Sonnet, Gemini 2.5 Pro)

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _«Опять дедлайн горит, фича готова, а тесты писать еще на полдня?»_

Все мы знаем, насколько важны юнит-тесты, но давайте будем честны: их написание часто занимает больше времени, чем создание самой бизнес-логики.
Продумывание пограничных случаев (edge cases), настройка моков (mocking), создание фикстур... Звучит утомительно?
Теперь вам достаточно просто передать вашу логику ИИ, и он сгенерирует надежные, покрывающие все сценарии тесты за считанные секунды.

---

## ⚡️ Краткое содержание в 3 строках (TL;DR)

1. **Глубокий анализ:** ИИ покрывает не только успешные сценарии (happy path), но и сложные пограничные случаи.
2. **Любой фреймворк:** Идеальная поддержка синтаксиса популярных библиотек тестирования (Jest, PyTest, JUnit и др.).
3. **Автоматизация рутины:** Мгновенная генерация mock-объектов и фикстур без лишней головной боли.

---

## 🚀 Решение: "Генератор модульных тестов"

### 🥉 Basic Version (Базовая версия)

Используйте этот вариант, когда нужен быстрый результат без сложных настроек.

> **Роль:** Ты дотошный QA-автоматизатор и Senior-разработчик.
> **Запрос:** Напиши юнит-тесты для следующего кода. Постарайся достичь 100% покрытия. `[Вставьте ваш код]`

<br>

### 🥇 Pro Version (Профессиональная версия)

Используйте этот промпт для сложных функций, требующих глубокого анализа и мокирования внешних зависимостей.

> **Роль (Role):** Ты дотошный QA-инженер и эксперт по автоматизации тестирования с многолетним опытом.
>
> **Ситуация (Context):**
>
> - Фон: У меня есть код, написанный на `[Язык программирования, например: TypeScript]`.
> - Цель: Мне нужны надежные и исчерпывающие юнит-тесты для повышения отказоустойчивости продакшен-кода.
>
> **Задача (Task):**
>
> 1. Внимательно проанализируй предоставленный код и составь список тестовых сценариев (включая успешные пути, обработку ошибок и анализ пограничных значений).
> 2. Напиши готовый к запуску код тестов, используя фреймворк `[Библиотека тестирования, например: Jest]`.
> 3. Если в коде есть внешние зависимости (API, базы данных), реализуй для них корректный Mocking.
> 4. Добавь понятные комментарии к каждому тестовому блоку, объясняющие, что именно проверяется.
>
> **Код:**
>
> ```
> [Вставьте ваш исходный код сюда]
> ```
>
> **Ограничения (Constraints):**
>
> - Обеспечь максимально возможное покрытие кода (Code Coverage).
> - Выведи результат в виде готового блока кода, который можно сразу скопировать и запустить.
>
> **Предупреждение (Warning):**
>
> - Описания тестов должны быть исключительно на русском языке. Убедись, что моки не усложняют тесты сверх меры (избегай овер-инжиниринга).

## 💡 Комментарий автора (Insight)

Этот промпт — настоящее спасение при работе с легаси-кодом или сложной бизнес-логикой. Главный секрет успеха здесь заключается в первом шаге задачи: мы просим ИИ **сначала составить список сценариев**, и только потом писать код. Это заставляет модель использовать подход Chain-of-Thought (цепочка рассуждений), что кардинально снижает риск упустить неочевидные баги или пограничные значения. Особенно хорошо это работает, когда вы просите мокировать вызовы к внешним API — ИИ часто генерирует заглушки, о которых вы сами могли бы забыть!

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Справится ли бесплатный ChatGPT (GPT-3.5/GPT-4o-mini) с этой задачей?**
  - A: Базовая логика будет покрыта, но для сложных тестов с большим количеством моков и нестандартных пограничных случаев настоятельно рекомендуется использовать продвинутые модели вроде GPT-4o или Claude 3.5 Sonnet.

- **Q: Что делать, если сгенерированный тест падает с ошибкой?**
  - A: Просто скопируйте сообщение об ошибке из консоли и отправьте его ИИ с текстом: «Тест упал с этой ошибкой. Исправь код теста или укажи, где ошибка в моей бизнес-логике». Часто ИИ находит реальный баг в вашем изначальном коде!

## 🧬 Анатомия промпта (Why it works?)

1. **Двухэтапная генерация (Chain-of-Thought):** Требование сначала выписать сценарии заставляет ИИ анализировать код глубже, прежде чем выдавать готовые функции.
2. **Управление зависимостями:** Явное указание обрабатывать Mocking спасает от написания хрупких (flaky) тестов, которые ломаются из-за внешних факторов.

## 📊 Доказательство: Before & After

### ❌ Before (Ваш исходный код)

```javascript
function divide(a, b) {
  if (b === 0) throw new Error("Нельзя делить на 0.");
  return a / b;
}
```

### ✅ After (Результат ИИ)

```javascript
describe("Тестирование функции divide", () => {
  // 1. Тест нормальной работы (Happy path)
  test("Деление 10 на 2 должно возвращать 5", () => {
    expect(divide(10, 2)).toBe(5);
  });

  // 2. Тест обработки исключений (Error handling)
  test("Деление на 0 должно выбрасывать ошибку", () => {
    expect(() => divide(10, 0)).toThrow("Нельзя делить на 0.");
  });

  // 3. Тест пограничных значений (Edge case)
  test("Деление отрицательного числа на положительное дает отрицательный результат", () => {
    expect(divide(-10, 2)).toBe(-5);
  });
});
```
