# 🏗️ Инфраструктура без кликов! Генератор кода Terraform

- **🎯 Кому подойдет:** Разработчикам, уставшим от консоли AWS, и бэкендерам, которым лень вручную управлять инфраструктурой.
- **⏱️ Экономия времени:** 1 час на чтение документации → сокращается до 3 минут.
- **🤖 Рекомендуемая модель:** Claude 3.5 Sonnet (идеально для написания кода и проектирования архитектуры).

- ⭐ **Сложность:** ⭐⭐⭐☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐☆

> _"Как долго вы еще будете держаться за консоль, рискуя удалить базу данных одним неверным кликом?"_

К инфраструктуре нужно относиться не как к домашнему питомцу, а как к скоту (Pet vs. Cattle). Если сервер падает, вы не должны чинить его всю ночь — его нужно безжалостно уничтожить и поднять точную копию за 1 секунду. Именно это позволяет сделать **IaC (Infrastructure as Code)**. Однако запоминать сложный синтаксис HCL (HashiCorp Configuration Language) и бесконечные опции провайдеров каждый раз — задача не из легких.

Этот промпт превращает расплывчатые требования вроде "подними мне в AWS один EC2 и подключи к нему базу" в **идеальный, модульный код Terraform, соответствующий строгим правилам безопасности**, который можно сразу использовать в production.

---

## ⚡️ 3 главных преимущества (TL;DR)

1. Мгновенная генерация кода, **строго соблюдающего стандарты HCL**, без необходимости часами изучать документацию.
2. Принудительное применение **настроек безопасности с наименьшими привилегиями (SG, IAM)**, чтобы избежать фатальных ошибок вроде открытия портов `0.0.0.0/0`.
3. Создание **модулей с высокой степенью переиспользования** за счет строгого разделения на `variables.tf` и `outputs.tf` без хардкода.

---

## 🚀 Решение: "Генератор идеальной инфраструктуры"

### 🥉 Базовая версия (Basic Version)

Используйте, когда нужно быстро поднять один ресурс для тестирования.

> **Роль (Role):** Ты Senior DevOps инженер.
> **Задача (Task):** Напиши код Terraform для создания ресурса AWS `[Название сервиса]`.
> **Ограничения (Constraints):** Используй последнюю версию AWS Provider. Избегай хардкода, все параметры выноси в переменные (variables).

<br>

### 🥇 Продвинутая версия (Pro Version)

Используйте для построения реальной production-среды, где безопасность и высокая доступность (HA) критически важны.

> **Роль (Role):** Ты облачный архитектор с 10-летним стажем и эксперт по Terraform, специализирующийся на облачной безопасности и высокой доступности (HA).
>
> **Контекст (Context):**
>
> - Ситуация: Я проектирую инфраструктуру в AWS для `[Описание проекта, например: глобального бэкенда e-commerce, обрабатывающего 10 000 запросов в секунду]`.
> - Цель: Получить готовый к production код IaC (Infrastructure as Code) с соблюдением требований безопасности, высокой доступности и строгой модульности.
>
> **Задача (Task):**
>
> 1. **Архитектура:** Напиши определение основных ресурсов (`main.tf`) для построения запрошенной инфраструктуры.
> 2. **Структурирование и модульность:** Строго исключи хардкод. Четко раздели конфигурацию на `variables.tf` и `outputs.tf`.
> 3. **Усиление безопасности:** В Security Group обязательно примени принцип наименьших привилегий (Least Privilege). Категорически запрещено использовать `0.0.0.0/0` без абсолютной необходимости.
> 4. **Управление доступом:** При необходимости напиши код для IAM Role и Policy, выдавая только минимально требуемые права.
>
> **Ограничения (Constraints):**
>
> - Используй новейший синтаксис HashiCorp Configuration Language (HCL) и AWS Provider версии v5.0 или выше.
> - Имена всех ресурсов и теги (Tags) должны содержать префикс `[Название проекта]`.
> - Идеально соблюдай отступы и стайл-гайд в соответствии со стандартом `terraform fmt`.
>
> **Предупреждение (Warning):**
>
> - Никогда не используй устаревшие (Deprecated) опции или небезопасные настройки по умолчанию, которые неприменимы в реальной практике. (Предотвращение галлюцинаций ИИ и инцидентов ИБ).

## 💡 Комментарий автора (Insight)

Самое страшное при полном доверии написания Terraform-кода искусственному интеллекту — это **безопасность (Security)**. Код, сгенерированный обычным промптом, часто ради "удобства" открывает все порты фаервола (`0.0.0.0/0`) или выдает IAM-ролям избыточные права.

Суть продвинутой (Pro) версии промпта заключается в том, что она **заставляет ИИ соблюдать "принцип наименьших привилегий" и строгую "модульность"**. На практике я настоятельно рекомендую использовать этот промпт для создания надежного каркаса, а затем обязательно прогонять код через инструменты статического анализа безопасности, такие как `tfsec` или `checkov`, внедренные в ваш CI/CD пайплайн. Вы почувствуете настоящую магию, когда первоначальная настройка инфраструктуры, занимавшая дни, сократится до 3 минут.

## 🙋 Часто задаваемые вопросы (FAQ)

- **В: Можно ли использовать этот промпт для GCP или Azure, а не только для AWS?**
  - О: Да, конечно. Просто измените облачного провайдера на 'Google Cloud' или 'AzureRM' в разделах `[Контекст]` и `[Ограничения]`, а затем укажите правильные названия сервисов для выбранного вендора. Промпт отработает безупречно.

- **В: ИИ постоянно выдает ошибки, потому что генерирует код с устаревшим синтаксисом (например, v0.11).**
  - О: Это частая проблема при использовании старых моделей (например, GPT-3.5). Хотя в промпте и указано "использовать новейший синтаксис Terraform", для генерации инфраструктурного кода я рекомендую использовать **Claude 3.5 Sonnet** (отлично справляется с кодом) или последнюю версию **GPT-4o**. Они выдают гораздо более стабильный и современный HCL код.

## 🧬 Разбор промпта (Why it works?)

1. **Задание роли (Role):** Использование персоны "облачного архитектора с 10-летним стажем" заставляет ИИ не просто создавать ресурсы, а проектировать архитектуру уровня Enterprise с учетом безопасности и отказоустойчивости.
2. **Ограничения (Constraints):** Требование использовать новейший синтаксис HCL и соблюдать `terraform fmt` гарантирует получение чистого, стандартизированного кода, который можно сразу запускать (`terraform init && terraform apply`).
3. **Контроль безопасности:** Прямой запрет на использование "небезопасных настроек по умолчанию" и акцент на "принципе наименьших привилегий" предотвращают критические уязвимости в конфигурациях фаерволов (SG) и правах доступа (IAM).

## 📊 Доказательство: До и После (Before & After)

Разница в качестве HCL-кода, сгенерированного по простому запросу "сделай мне" и с помощью структурированного промпта, колоссальна.

### ❌ До (Простой запрос)

```text
Создай VPC в AWS, сделай 2 публичных сабнета и 2 приватных.
```

**Результат (Хардкод и уязвимости):**

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
# Параметры захардкожены без переменных, отсутствуют теги (Tags), монолитный файл без возможности расширения.
```

### ✅ После (Результат Pro-промпта)

**Результат (Модульность и стандарты production):**

```hcl
# main.tf
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"

  name = "${var.project_name}-vpc"
  cidr = var.vpc_cidr

  azs             = var.availability_zones
  private_subnets = var.private_subnets
  public_subnets  = var.public_subnets

  enable_nat_gateway     = true
  single_nat_gateway     = false # NAT для каждой AZ ради высокой доступности
  one_nat_gateway_per_az = true

  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "Terraform"
  }
}
```

_(※ Кроме этого, ИИ также генерирует идеально структурированные файлы variables.tf и outputs.tf.)_
