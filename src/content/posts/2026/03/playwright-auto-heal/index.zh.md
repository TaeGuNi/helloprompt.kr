- **🎯 推荐对象：** QA 工程师、前端开发者、自动化测试负责人
- **⏱️ 节省时间：** 2 小时 → 1 分钟
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o

- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **效果：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“昨天还跑得好好的 E2E 测试，今天早上就因为改了一个按钮的 CSS 类名全挂了。难道又要去重新找选择器吗？”_

在前端业务的高速迭代中，UI 的微调往往是引发 E2E（端到端）测试“雪崩”的罪魁祸首。无论是 Playwright 还是 Cypress，只要 DOM 结构或 CSS 类名出现一丝偏差，原本顺畅的自动化测试就会瞬间被满屏飘红的报错日志淹没。为了修复这些失效的选择器，你不得不痛苦地翻阅终端输出，重新打开开发者工具排查 DOM 树，再小心翼翼地替换成新的定位代码。这种机械式的修补不仅毫无技术含量，更是对宝贵研发时间的极大浪费。

如今，是时候转变思路了：与其手动死磕，不如将报错日志与最新的 HTML 结构直接喂给 AI，让它为你瞬间完成选择器的“自愈（Auto-healing）”。

1. 利用专属 AI 提示词，精准锁定并自动修复因 UI 变动导致的 Playwright 选择器失效问题。
2. 不仅能直接输出修复后的代码，还会提供具备高鲁棒性（Resilient）与抗干扰能力的测试编写策略。
3. 仅需输入报错日志与最新的 DOM 快照，1 分钟内即可获得完美“自愈”的测试脚本。

## 🚀 解决方案："Playwright 自愈 (Auto-healing) 提示词"

### 🥉 Basic Version (基础版)

追求极致效率，只想最快拿到修复方案时，请直接复制此版本。

> **角色：** 你是一名 `[资深 QA 工程师]`。
> 
> **任务：** 请根据我提供的 Playwright 报错日志和最新的 HTML 结构，修复 `[这段崩溃的测试代码]`。

### 🥇 Pro Version (专业版)

当你不仅需要修复报错，更渴望从根本上重构出易于维护、坚如磐石的测试架构时，请使用此版本。

> **角色 (Role)：** 你是一名自动化测试专家、`[资深 SDET (测试开发工程师)]`。你对 Playwright 的最佳实践（Best Practices）了如指掌。
> 
> **背景 (Context)：**
> 
> - **现状：** 近期的 UI 更新导致现存的 Playwright E2E 测试发生大面积崩溃，核心肇因是元素选择器（Selector）的失效。
> - **目标：** 精准修复报错代码，并将其彻底重构为对未来 UI 变动具有极强抗性（Resilient）的稳健定位策略。
> 
> **任务 (Task)：**
> 
> 1. 深度剖析下方提供的 `[报错日志]` 与 `[变更后的 HTML 结构]`，精准锁定报错的根本原因。
> 2. 重写 `[现有测试代码]`，确保测试链路顺畅通行。
> 3. 在重写过程中，必须优先采用面向用户（User-Facing）的属性（如文本内容、ARIA 属性、data-testid 等），严禁过度依赖脆弱的 CSS 类名或 XPath。
> 4. 为修改的代码行添加简明扼要的注释，阐述“采用此种定位策略的技术依据”。
> 
> **输入数据：**
> 
> - **报错日志：** `[请在此完整粘贴报错日志]`
> - **变更后的 HTML 结构：** `[请在此粘贴从开发者工具中提取的相关 HTML 片段]`
> - **现有测试代码：** `[请在此粘贴运行失败的 Playwright 代码段]`
> 
> **约束条件 (Constraints)：**
> 
> - 最终输出必须是可供直接复制/粘贴的完整代码块。
> - 必须优先使用 Playwright 的内置定位器（Locators）（例如：`getByRole`、`getByText`、`getByTestId`）。
> 
> **注意事项 (Warning)：**
> 
> - 若在提供的 HTML 片段中无法找到目标元素，绝不可凭空捏造。你必须明确回复：“由于提供的 HTML 信息不足，无法推导准确的选择器，请补充更完整的 DOM 信息。”（严禁任何形式的幻觉）

## 💡 作者点评 (Insight)

在真实的业务迭代场景中，当几十个测试用例因一次普通的 UI 改版而全线飘红时，这套提示词的真正威力便会显露无疑。曾几何时，哪怕只是某个全局按钮的类名微调，我也必须硬着头皮打开数十个测试文件，逐一使用 `Ctrl + F` 苦哈哈地进行全局替换。

而现在，只需将这套提示词作为预设模板内置于 IDE 的 AI 助手或 Claude 中，面对报错时，你只需将崩溃日志与浏览器 Elements 面板中的最新代码片段“喂”给它，即可将繁杂的选择器重构工作完美外包。

其中最核心的精髓在于**“强制使用 User-Facing 属性”**这一严苛约束。如果不加限制，AI 往往会“偷懒”去抓取最显眼的 CSS 类名；但戴上这道“紧箍咒”后，它便能写出诸如 `getByRole('button', { name: 'Submit' })` 这样优雅且具备**极强抗脆弱性**的代码。这绝对是一件能让你每天至少提前一小时打卡下班的自动化神器。只要体验过一次这种行云流水般的修复过程，你就再也回不去了！

## 🙋 常见问题 (FAQ)

- **Q: 页面的 HTML 结构极其庞大，全部丢给 AI 会导致上下文溢出或解析失败吗？**
  - A: 完全无需复制整个页面的 DOM 树。你只需精准截取报错元素（例如目标按钮或输入框）及其直接父级容器的 HTML 片段，这些核心信息便足以让 AI 做出准确判断。

- **Q: 如果我的团队主力框架是 Cypress 或 Selenium，这套方法论还适用吗？**
  - A: 绝对适用。底层的重构逻辑是共通的，你只需将提示词中的“Playwright”一词替换为你当前使用的自动化框架名称，它同样能够完美执行选择器的自愈任务。

- **Q: 在缺乏规范 `data-testid` 的祖传代码（Legacy Code）中，这套提示词还能发挥作用吗？**
  - A: 毫无压力。在缺失专属测试属性的恶劣环境下，AI 会极其聪明地退而求其次，自动寻觅基于无障碍访问（Accessibility）的最佳替代方案，例如巧妙运用 `getByRole` 或 `getByText` 来精准锚定目标元素。

## 🧬 提示词解剖 (Why it works?)

1. **赋予 SDET 专家人设：** 拔高 AI 的工程视角，迫使其以资深测试架构师的严苛标准来审视上下文代码，从而产出完美契合 Playwright 最佳实践的工业级脚本。
2. **强制推行 User-Facing 定位器：** 在提示词的底层逻辑中直接注入 Playwright 的核心测试哲学，从根本上根除测试用例的脆弱性（Flakiness），大幅延长测试套件的生命周期。
3. **熔断幻觉的安全阀门：** 为防止 AI 在上下文信息缺失时“强行推演”从而引发更严重的调试灾难，明确设定了“信息不足时拒绝捏造”的强约束，确保每一次输出都绝对安全可靠。

## 📊 效果验证：Before & After

### ❌ Before (输入)

```text
报错日志：
Error: locator.click: Target closed
=========================== logs ===========================
waiting for locator('.btn-primary.submit-btn')
============================================================

变更后的 HTML 结构：
<div class="actions">
  <button class="button--blue" aria-label="Submit Order">Order Now</button>
</div>

现有测试代码：
await page.locator('.btn-primary.submit-btn').click();
```

### ✅ After (结果)

```typescript
// 修改原因：原有的 CSS 类名 ('.btn-primary.submit-btn') 发生变更，导致元素定位失效。
// 改进方案：摒弃脆弱的 CSS 选择器，改用对 UI 变更具有强抗性的无障碍属性 (ARIA label) 和 Role 进行定位。
await page.getByRole('button', { name: 'Submit Order' }).click();
```
