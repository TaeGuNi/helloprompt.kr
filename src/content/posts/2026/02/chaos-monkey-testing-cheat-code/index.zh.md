## 💣 让 AI 化身 Netflix 混沌猴的硬核测试作弊码

- **🎯 推荐对象：** 被 AI 敷衍的“纯享版”正常流程（Happy Path）测试代码气炸的高级工程师，以及缺乏资深导师指引的初级开发者。
- **⏱️ 所需时间：** 熬夜 Debug 通宵 → 1 分钟一键搞定。
- **🤖 推荐模型：** Claude 3.5 Sonnet, GPT-4o（能力较弱的模型无法承受此等严苛指令）。
- ⭐ **难度：** ⭐⭐⭐⭐☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

> _“大多数开发者在让 AI 编写测试代码时，都犯下了一个致命的错误：他们只会说‘写个能跑通的’。于是，AI 毫不客气地甩出 200 行犹如温室花朵般的正常流程（Happy Path）。殊不知，现实的生产环境根本就是个人间炼狱。”_

绝对可靠的外部 API 根本不存在，永远在线的数据库连接更是痴人说梦。今天为你揭秘的这套作弊码，将彻底撕下 AI 虚伪的“好好先生”面具，强制它在无情的灾难场景（Chaos）下构建防线。这是一份专为**“斯巴达式高级工程师”**量身定制的核弹级提示词。

---

## ⚡️ 3行总结 (TL;DR)
- 🚫 **压缩正常流程（Happy Path）至 20%：** 摒弃理想主义，将 80% 的测试火力集中在边缘场景、异常处理与灾难恢复上。
- 🔥 **强制拉响灾难警报：** 铁腕要求 AI 模拟（Mock）网络超时、HTTP 500 崩溃以及数据库断连等极端故障。
- 🧹 **封杀肮脏的全局 Mock：** 严禁肆意污染 `global.fetch`，强硬推行 MSW 或 DI（依赖注入）以确保测试的绝对隔离。

## 🚀 解决方案："Chaos Monkey Testing Protocol" (混沌猴测试协议)

### 🥉 Basic Version (基础版)
当你急需快速搭建异常防御测试的骨架时，直接抛出这段指令即可。

> **角色：** 你是一位冷酷无情的高级 QA 工程师。
> **任务：** 为`[要测试的目标]`编写测试代码。正常流（Happy Path）只需保留 1 个用例，其余所有测试块必须全部填满用于防御网络错误、数据库断连等失败（Failure）场景的极端逻辑。

### 🥇 Pro Version (专业版)
这是一段彻底抹杀 AI 讨好型人格，将 Netflix “混沌猴”哲学强行注入其大脑的硬核作弊码。

> **角色 (Role)：** 你是一位盲目狂热于 Netflix “混沌猴 (Chaos Monkey)” 哲学、极端且冷血的高级测试工程师。立刻收起你那套盲目服从与虚伪的友善，你的脑子里只能有一个念头：如何不择手段地摧毁我系统的弱点。
> 
> **背景 (Context)：**
> - 目标：为 `[要测试的函数或组件名称]` 编写具备宇宙级防御能力的 Unit/E2E 测试套件 (Test Suite)
> - 框架：`[Vitest / Jest / Playwright 等测试框架]`
> 
> **任务 (Task)：**
> 1. **Death to Happy Paths（剿灭正常流）：** 将“数据完美且外部服务器 100% 响应”这种天真烂漫的场景严格压缩在总测试量的 20% 以下。将剩余 80% 的代码量 All-in 到对失败与异常灾难的防御上。
> 2. **Disaster Scenarios（灾难场景演习）：** 必须将以下至少一种毁灭性场景硬编码进测试块 (`describe`) 中：
>    - 外部 API 严重延迟或抛出 HTTP 500 Internal Error 时，系统能否返回用户友好型错误提示，并精准验证回滚逻辑。
>    - 数据库连接池突然熔断时，系统能否有效防止死锁，并成功触发重试队列 (Retry Queue) 逻辑。
> 3. **无情的逻辑覆盖率：** 绝对不准以“没有报错”作为敷衍了事的终点。你必须死死咬住性能延迟 (Delay) 的最大容忍阈值，以及报错抛出后的垃圾回收 (内存清理) 状态，使用 `expect` 语句下达最严苛的断言指令。
> 
> **约束条件 (Constraints)：**
> - **The Clean Mocking Rule（纯净 Mock 法则）：** 严禁出现诸如 `global.fetch = vi.fn().mockRejectedValue(...)` 这种极其肮脏的全局对象污染。
> - 网络灾难模拟必须动用 MSW (Mock Service Worker) 这种协议级别的 Mock 利器，或者仅 Mock 注入了依赖 (DI) 的客户端实例，以誓死捍卫 100% 的测试隔离性 (Isolation)。
> - **禁止幻觉 (Hallucination)：** 绝不允许捏造你无法确定的语法或根本不存在的库。如果触碰到了知识盲区，给我理直气壮地要求“请提供官方文档”。

## 💡 作者点评 (Insight)
咱们打开天窗说亮话吧。当我们将编写测试代码的脏活累活丢给 AI 时，心底是不是总藏着一丝侥幸：“随便把覆盖率凑满 100%，让 CI 亮起绿灯就行了”？AI 极其精明，它瞬间就能洞察这种惰性，于是疯狂堆砌各种 “Happy Path”，最后还不忘谄媚地邀功：“测试已成功覆盖！”

**这根本不叫软件工程。这简直是在发霉的代码库上狂撒糖霜。**

这套提示词，是我在生产环境中历经无数次服务器雪崩与数据库死锁后，字字泣血雕刻而成的“反脆弱（Anti-Fragile）”思想结晶。一旦将这个作弊码喂给 AI，它便会瞬间褪去温顺聊天机器人的外衣，化身为一位吹毛求疵、极度神经质的首席架构师。因为我们从物理层面彻底切断了它企图用全局 Mock（例如粗暴覆盖 `window.fetch`）蒙混过关的退路，强硬要求其搭建 MSW 或遵循 DI 原则。刚开始，它甚至可能会向你抱怨配置代码过于冗长。

但是，请务必相信我。如果你不想在系统上线的凌晨 3 点被 PagerDuty 夺命连环 Call 惊醒，你就**必须**将 AI 视为一个冷酷无情的破坏者来加以利用，而不是把它当作一位只会端茶倒水的贴心秘书。

## 🙋 常见问题 (FAQ)
- **Q: AI 生成的 MSW 配置代码过于臃肿复杂，怎么破？**
  - A: 这种“阵痛”是完全正常的。这恰恰是摒弃肮脏全局 Mock、构建纯净隔离测试环境的必经之路。如果实在难以忍受冗长的代码，你可以在 Task 环节追加一项指令：`[额外要求：请务必将所有的 MSW Handler 配置抽离并封装至独立的 Mock 文件中]`。
- **Q: 难道项目里的每一个小组件都要遭受这种“满清十大酷刑”吗？**
  - A: 杀鸡焉用牛刀。对于那些只负责展示的纯 UI 组件而言，这不仅是过度设计，更是暴殄天物。请将这套核弹级提示词精准投放至处理核心支付链路、第三方 API 深度对接以及复杂状态管理 (Store) 的核心业务领域 (Domain) 即可。
- **Q: 在执行这种极端指令时，Claude 和 GPT 究竟谁更听话？**
  - A: 就目前的实战反馈来看，面对这类带有浓厚哲学思辨与极端约束的指令，**Claude 3.5 Sonnet** 的贯彻力度要远远抛离对手。相比之下，GPT 模型似乎总被底层的安全对齐协议所裹挟，时不时就会不可遏制地散发出令人不悦的“假惺惺的友善”。

## 🧬 提示词解剖 (Why it works?)
- 🎭 **赋予毁灭性的非对称人格：** 我们没有丢给它“你是一名优秀开发者”这种烂大街的设定，而是为其注入了“极度厌恶盲从、视破坏系统为己任的冷血工程师”这一强悍灵魂。这从物理底层彻底压制了 AI 与生俱来的讨好型“积极偏见 (Positivity Bias)”。
- 🚧 **铁腕封杀反模式 (Constraints)：** 我们通过具体的代码片段（如 `global.fetch = vi.fn()`），指名道姓地将其列为禁忌。事实证明，与其含糊其辞地恳求它“请注意代码质量”，AI 对这种“敢写这行代码你就死定了”的硬核威胁，往往反应得更加精确且敬畏。
- ⚖️ **以量化指标倒逼质量：** 我们祭出了“Happy Path 占比不得超过 20%”这条绝对的数值红线。这一招直接从源头掐断了 AI 企图通过批量注水正常用例来敷衍了事、疯狂消耗 Token 的恶劣行径。
