## 📝 E2Eテストがエラーで止まる？PlaywrightのAuto-healingプロンプトを使えば1分で解決 🛠️

- **🎯 推奨ターゲット:** QAエンジニア、フロントエンドエンジニア、テスト自動化担当者
- **⏱️ 所要時間:** 2時間 → 1分に短縮
- **🤖 推奨AIモデル:** Claude 3.5 Sonnet, GPT-4o
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果性:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

> _「昨日まで完璧に動いていたE2Eテストが、ボタンのCSSクラスが一つ変わっただけで全滅…。またDevToolsを開いてセレクタを探し直す苦行が始まるのでしょうか？」_

フロントエンド開発やQAの現場において、UIのほんのわずかな変更でE2E（End-to-End）テストが壊れてしまう（Flakyになる）現象は、もはや日常茶飯事です。PlaywrightやCypressといった強力な自動化ツールを導入していても、DOM構造やクラス名が少し変わっただけで、テストコードは容赦なく真っ赤なエラーを吐き出します。

エラー箇所を特定し、ブラウザのDevToolsを開いて新しいセレクタを血眼で探し、コードを書き直して再びテストを回す……。この終わりの見えない不毛なモグラ叩きは、エンジニアの貴重な開発時間を容赦なく奪い去っていきます。しかし、もうそんな消耗戦に付き合う必要はありません。AIに「エラーログ」と「変更されたHTML構造」を投げるだけで、自律的にテストコードを「Auto-healing（自動復旧）」させる最新のアプローチを取り入れ、この苦行から今すぐ抜け出しましょう。

---

## ⚡️ 3行まとめ (TL;DR)

1. UI変更によって壊れたPlaywrightのセレクタをAIに分析させ、即座に復旧させる強力なプロンプトです。
2. 表面的な応急処置にとどまらず、将来的なUI変更にも耐えうる堅牢（Resilient）なセレクタ戦略まで提案させることができます。
3. エラーログと最新のDOM要素をコピー＆ペーストして渡すだけで、テストを確実に通過する修正コードがわずか1分で手に入ります。

## 🚀 解決策：「Playwright Auto-healing プロンプト」

### 🥉 Basic Version (基本型)

スピードを最優先とし、壊れたテストコードとエラーメッセージから一刻も早く修正版を受け取りたい場合に使用してください。

> **役割:** あなたは `[シニアQAエンジニア]` です。
> **タスク:** 提供されたPlaywrightのエラーログと新しいHTML構造を基に、以下の `[壊れたテストコード]` を修正してください。

### 🥇 Pro Version (専門家型)

単なる場当たり的な対処ではなく、今後のメンテナンスを容易にし、UI変更に強い堅牢なテストコードへと根本からリファクタリングしたい場合に使用してください。

> **役割 (Role):** あなたはテスト自動化を専門とする `[シニアSDET (Software Development Engineer in Test)]` です。Playwrightのベストプラクティスを完全に熟知しています。
>
> **状況 (Context):**
>
> - 背景: 最近のUIアップデートの影響で、既存のPlaywrightのE2Eテストが失敗しています。主な原因はセレクタの変更です。
> - 目標: 失敗しているテストコードを正常に復旧させると同時に、今後のUI変更にも壊れにくい（Resilientな）堅牢なセレクタ構造へと改善すること。
>
> **タスク (Task):**
>
> 1. 以下に提供された `[エラーログ]` と `[変更されたHTML構造]` を詳細に分析し、エラーの根本原因を正確に特定してください。
> 2. `[既存のテストコード]` を修正し、テストが確実にパスするように最適化して書き直してください。
> 3. 修正の際は、壊れやすいCSSクラスやXPathへの過度な依存を避け、ユーザー向け属性（表示テキスト、ARIA属性、data-testidなど）を最優先で使用してください。
> 4. 修正箇所には、「なぜこのアプローチで修正したのか」という簡潔かつ論理的な理由をコードコメントとして必ず付与してください。
>
> **入力データ:**
> - エラーログ: `[エラーログ全体をここにペースト]`
> - 変更されたHTML構造: `[DevToolsからコピーした該当要素のHTMLスニペットをここにペースト]`
> - 既存のテストコード: `[失敗しているPlaywrightのテストコードをここにペースト]`
>
> **制約事項 (Constraints):**
>
> - 出力結果は、そのままコピー＆ペーストして即座に使える完全なコードブロック形式（TypeScript/JavaScript）で提供してください。
> - Playwrightが公式に推奨する組み込みLocators（例: `getByRole`, `getByText`, `getByTestId`）の利用を絶対的な最優先事項としてください。
>
> **注意事項 (Warning):**
>
> - 万が一、提供されたHTMLスニペットの中に適切な要素が見当たらない場合は、勝手な推測でコードを生成しないでください。その場合は「HTML情報が不足しているため、堅牢なセレクタを特定できません。より広い範囲のDOM情報を提供してください」と正直に回答し、AIの幻覚（Hallucination）を徹底的に防いでください。

このアプローチの真価は、大規模なUIリニューアルなどによって数十個のテストが一斉に真っ赤に染まった、あの絶望的な瞬間にこそ発揮されます。

私自身、かつてはコンポーネントの構造がたった一つ変わるたびに、関連する30個以上のテストファイルをエディタで開き、`Ctrl + F` を連打してひたすらセレクタを置換し続けるという、不毛な作業に追われていました。しかし、このプロンプトをGitHub CopilotやClaudeにスニペットとして登録してからは、開発体験が劇的に変わりました。エラーログと、ブラウザのElementsタブからコピーしたわずか数行のHTMLを渡すだけで、あの面倒極まりないセレクタの特定と置換作業を、丸ごとAIに完全アウトソーシングできるようになったのです。

ここで絶対に外せない最も重要なポイントは、プロンプト内に設けた**「ユーザー向け属性（User-Facing Locators）を使用しろ」**という制約事項です。AIに自由に書かせると、一番手っ取り早いCSSクラス（例：`.btn-primary`）を安易にターゲットにしがちです。しかし、この制約を明確に指示することで、`getByRole('button', { name: 'Submit Order' })` のような、UI変更に対して極めて堅牢であり、かつアクセシビリティにも配慮された美しいコードを確実に返してくれます。皆さんの退社時間を間違いなく1時間は早めてくれる、圧倒的に強力な武器になるはずです。次回のテスト修正の際に、ぜひ騙されたと思って試してみてください！

## 🙋 よくある質問 (FAQ)

- **Q: ページ全体のHTMLをコピーして渡すと長すぎてエラーになるのですが、どうすればいいですか？**
  - A: ページ全体のソースコードを丸ごと渡す必要はありません。エラーが発生したターゲット要素（例：クリックできなかったボタンや入力フォーム）と、その直近の親タグ（ラッパー要素）周辺の数行〜十数行だけをコピーして渡せば、AIは必要な文脈を十分に理解してくれます。

- **Q: Playwright以外のテストフレームワーク（CypressやSeleniumなど）でも応用可能ですか？**
  - A: もちろん可能です。プロンプト内の「Playwright」という単語を、現在お使いのフレームワーク名（Cypress, Selenium, Puppeteerなど）に書き換えるだけで、AIはそれぞれのツールのベストプラクティスに最適化されたコードを生成してくれます。

- **Q: 現場のコーディング規約上 `data-testid` が付与されていないプロジェクトなのですが、それでも機能するのでしょうか？**
  - A: はい、全く問題ありません。指定されたHTML構造の中に `data-testid` が見つからない場合でも、AIは `getByRole` や `getByText` 、あるいはARIA属性といったアクセシビリティベースの堅牢な代替セレクタを賢く見つけ出し、適切に適用してくれます。

## 🧬 プロンプトの解剖 (Why it works?)

1. **専門的なSDETペルソナの付与:** 単なる「プログラマー」ではなく、テスト自動化のスペシャリストである「SDET」というペルソナを明確に与えることで、「とりあえず動くその場しのぎのコード」ではなく、「Playwrightのベストプラクティスに完全に準拠した高品質なコード」を生成するようAIを強力に方向付けています。
2. **User-Facing Locatorsの強制:** 「テストの不安定性（Flakiness）を根本から排除する」というPlaywright本来の設計哲学を、プロンプト内で絶対的な制約として強制しています。これにより、些細なUI変更ではビクともしない、極めて強靭なテストコードが生み出されます。
3. **ハルシネーション（幻覚）防止の安全装置:** HTML情報が不十分な場合、AIが適当なセレクタをでっち上げてデバッグ作業をかえって混乱させる事態を防ぐため、「情報不足の時は素直に分からないと答えろ」というフェイルセーフ機構を意図的に組み込んでいます。

## 📊 証明: Before & After

### ❌ Before (入力)

```text
エラーログ:

============================================================
変更されたHTML構造:
<div class="actions">
  <button class="button--blue" aria-label="Submit Order">Order Now</button>
</div>
既存のテストコード:
await page.locator('.btn-primary.submit-btn').click();
```

### ✅ After (結果)

```typescript
// 変更理由: CSSクラス（'.btn-primary.submit-btn'）が変更され、テストが失敗しました。
// 改善事項: 変更されやすいCSSクラスの代わりに、アクセシビリティ属性（ARIA label）とRoleを使用し、UIの変更に強くなるようにしました。
await page.getByRole('button', { name: 'Submit Order' }).click();
```

## 🎯 結論

E2Eテストは、本番プロダクトの品質を担保し、ユーザー体験を守るための強固な「盾」です。しかし、その盾のメンテナンス（セレクタの修正作業）ばかりにエンジニアの貴重な開発時間が奪われてしまっては、本末転倒です。

AIの力を最大限に引き出すこの「Auto-healingプロンプト」をプロジェクトに導入し、終わりの見えないテストメンテナンス地獄から今すぐ抜け出しましょう。これからの時代、テストを直す煩わしい単純作業はすべてAIに任せてしまえばいいのです。浮いた時間で美味しいコーヒーでも淹れて、颯爽と定時退社をキメましょう！ 🍷
