## 📝 Meu Código é Lixo! Treinador de Refatoração Clean Code

- **🎯 Recomendado para:** Desenvolvedores de todos os níveis, Analistas de Sistemas e Engenheiros de Software
- **⏱️ Tempo necessário:** 5 minutos → Reduzido para menos de 1 minuto
- **🤖 Modelo recomendado:** Qualquer modelo de IA conversacional de ponta (ChatGPT, Claude, Gemini, etc.)

- ⭐ **Dificuldade:** ⭐⭐☆☆☆
- ⚡️ **Eficácia:** ⭐⭐⭐⭐⭐
- 🚀 **Utilidade:** ⭐⭐⭐⭐☆

> _"Eu disse que consertaria depois, mas já se passou um ano. Agora, tenho pavor de encostar nesse código."_

Código espaguete é, sem dúvida, o maior pesadelo de qualquer desenvolvedor. Quando a legibilidade vai para o espaço, encontrar bugs e adicionar novas funcionalidades se transforma em uma verdadeira tortura. Este prompt atua como o seu próprio Robert C. Martin (o lendário autor de _Clean Code_), limpando e polindo seu código até que ele atinja a excelência absoluta.

---

## ⚡️ Resumo em 3 linhas (TL;DR)

1. **Eliminação de Redundâncias:** Erradica códigos duplicados e separa responsabilidades de forma cirúrgica (Extract Method).
2. **Nomenclatura Semântica:** Transforma nomes obscuros de variáveis e funções, adotando convenções claras, descritivas e altamente intuitivas.
3. **Otimização e Modernização:** Refatora a lógica para maximizar o desempenho, aplicando as sintaxes mais modernas e eficientes da sua linguagem.

## 🚀 A Solução: "Lavanderia Clean Code"

### 🥉 Versão Básica (Basic Version)

Use esta versão quando precisar de um resultado rápido, direto e sem enrolação.

> **Papel:** Você é um `[Evangelista de Clean Code e Arquiteto de Software Sênior]`.
>
> **Solicitação:** Refatore o `[código fornecido]` que "apenas funciona" transformando-o em um código limpo, de fácil leitura, manutenção e altamente escalável.

### 🥇 Versão Pro (Expert Version)

Use esta versão quando a qualidade dos detalhes e a perfeição arquitetural forem inegociáveis.

> **Papel (Role):** Você é um `[Evangelista de Clean Code e Arquiteto de Software Sênior]`.
>
> **Contexto (Context):**
>
> - Cenário: Meu código atual até funciona, mas está extremamente confuso, desorganizado e difícil de decifrar (o clássico Código Espaguete).
> - Objetivo: Refatorá-lo seguindo as melhores práticas da engenharia de software, garantindo um código altamente legível e à prova de balas para manutenção a longo prazo.
>
> **Tarefa (Task):**
>
> 1. **[Nomenclatura Semântica]**: Renomeie variáveis e funções para que sejam perfeitamente intuitivas e revelem seus reais propósitos (ex: de `a` para `userAge`).
> 2. **[Melhoria Estrutural]**: Desmembre funções longas e complexas aplicando o Princípio de Responsabilidade Única (SRP) e consolide lógicas duplicadas (DRY).
> 3. **[Modernização]**: Sempre que possível, aplique a sintaxe e os recursos mais modernos da linguagem (ES6+, Python 3.10+, Java 21+, etc.).
> 4. **[Análise de Impacto Antes & Depois]**: Apresente as mudanças de forma detalhada e justifique as razões arquiteturais e lógicas por trás de cada alteração.
>
> **Restrições (Constraints):**
>
> - O comportamento final do código (sua lógica de negócios) NUNCA deve ser alterado. Nenhuma funcionalidade pode ser perdida.
> - Adicione comentários apenas quando for estritamente necessário para explicar o "porquê" de uma decisão complexa. Remova comentários inúteis que explicam "o que" o código faz (o próprio código deve ser sua melhor documentação).
>
> **Código Alvo:**
>
> `[Cole o seu código desorganizado aqui]`

## 💡 Comentário do Autor (Insight)

A principal barreira para aplicar o _Clean Code_ no dia a dia não é a falta de conhecimento técnico, mas sim a **esmagadora pressão do tempo**. Em ambientes de entrega contínua, muitas vezes a regra do "feito é melhor que perfeito" acaba imperando. Este prompt atua exatamente como o seu par de programação sênior, assumindo toda a carga cognitiva de reorganizar a estrutura sem jamais quebrar a lógica de negócios subjacente.

Uma estratégia de ouro é utilizá-lo durante a preparação para um _Code Review_ (Pull Request). Antes de submeter suas alterações, passe os arquivos modificados por este prompt. Ele não apenas elevará drasticamente a qualidade da sua entrega, como também lhe ensinará padrões e sintaxes mais eficientes da linguagem que talvez você ainda não dominasse, acelerando exponencialmente o seu crescimento profissional.

## 🙋 Perguntas Frequentes (FAQ)

- **Q: O prompt pode acidentalmente quebrar a lógica do meu programa?**
  - **A:** Teoricamente, não. Nós definimos explicitamente nas restrições que o comportamento original deve permanecer intacto. No entanto, é **fundamental** que você tenha testes unitários ou realize validações manuais rigorosas após a refatoração. Ocasionalmente, a IA pode introduzir bugs sutis ao interpretar de forma equivocada uma lógica que não estava devidamente documentada.

- **Q: Funciona bem com qualquer linguagem de programação?**
  - **A:** Absolutamente! O prompt é totalmente agnóstico em relação à linguagem. Quer você esteja programando em JavaScript, Python, Go, Rust ou C#, princípios fundamentais de _Clean Code_ (como SRP, DRY e nomenclaturas claras) são universais. A IA se encarregará de adaptar as sugestões sintáticas às particularidades da linguagem do seu código original.

- **Q: Devo colar o arquivo inteiro ou apenas a função problemática?**
  - **A:** Para extrair os melhores resultados, cole apenas a função, a classe ou o bloco de código específico que você deseja lapidar (o ideal é manter entre 200 e 300 linhas). Se você despejar um arquivo gigantesco, a IA pode acabar cortando a resposta pela metade ou perdendo o foco nos detalhes mais críticos.

## 🧬 Anatomia do Prompt (Why it works?)

1. **Atribuição de Papel e Autoridade:** Ao definir a IA como um "Evangelista de Clean Code e Arquiteto de Software Sênior", nós calibramos seu padrão de respostas para fornecer análises profundas, estritamente baseadas nos princípios mais sólidos da engenharia de software (como os ensinamentos sagrados do _Uncle Bob_).
2. **Restrições Protetoras:** A regra inegociável de "nunca alterar a operação do código" atua como um escudo contra alucinações. O modelo é forçado a focar 100% na estrutura, evitando a perigosa reinvenção da lógica de negócios.
3. **Foco na Modernização:** A instrução explícita para aplicar as sintaxes mais recentes obriga o modelo a entregar uma refatoração que não só melhora a legibilidade, mas também otimiza o desempenho e garante a idiomaticidade do código, varrendo do mapa os padrões obsoletos.

## 📊 Prova: Antes e Depois

### ❌ Antes (Entrada)

```python
def calc(x, y):
    if x > 19:
        print("Adulto")
        p = y * 0.9
        return p
    else:
        print("Estudante")
        return y
```

(O nome da função `calc` não reflete absolutamente nada sobre o que ela faz. Os parâmetros `x` e `y`, juntamente com a variável local `p`, são completamente obscuros, exigindo um esforço mental desnecessário para decifrá-los.)

### ✅ Depois (Resultado)

```python
def calculate_discounted_price(age, price):
    ADULT_AGE_THRESHOLD = 20
    DISCOUNT_RATE = 0.9

    if age >= ADULT_AGE_THRESHOLD:
        print("Você é um adulto.")
        return price * DISCOUNT_RATE

    print("Você é um estudante.")
    return price
```

**[Pontos de Melhoria Observados]**

1. **Nomenclatura Clara:** A genérica função `calc` foi renomeada para `calculate_discounted_price`, revelando seu propósito logo de cara. Os parâmetros também ganharam clareza absoluta (`age` e `price`).
2. **Fim dos Números Mágicos (Magic Numbers):** Os valores numéricos perdidos no código (`19` e `0.9`) foram isolados em constantes nomeadas (`ADULT_AGE_THRESHOLD` e `DISCOUNT_RATE`), trazendo contexto imediato e facilitando manutenções futuras.
3. **Remoção do `Else` Desnecessário:** A profundidade de indentação foi reduzida e a complexidade ciclomática caiu drasticamente através da elegante aplicação do padrão de retorno antecipado (_Early Return_).
