## 📝 Надоедливая документация API? Swagger появляется из одного только кода

- **🎯 Рекомендуется для:** Бэкенд-разработчиков, Fullstack-инженеров, технических писателей
- **⏱️ Требуемое время:** 30 минут → 1 минута
- **🤖 Рекомендуемая модель:** ChatGPT (GPT-4o), Claude 3.5 Sonnet, Gemini 2.5 Pro

- ⭐ **Сложность:** ⭐⭐☆☆☆
- ⚡️ **Эффективность:** ⭐⭐⭐⭐⭐
- 🚀 **Полезность:** ⭐⭐⭐⭐⭐

> _«Вы ведь уже закончили API? А где документация?» — этот вопрос от фронтендеров способен испортить любой день._

Написание документации к API часто раздражает больше, чем сама разработка. Типы параметров, обязательные поля, примеры ответов для каждого статус-кода... Ручное создание Swagger- или Markdown-файлов убивает кучу времени и мотивации.

Но что, если можно просто скопировать код контроллера, а ИИ сам напишет идеальную спецификацию? Давайте автоматизируем эту рутину.

---

## ⚡️ Краткое содержание в 3 строках (TL;DR)

1. **Точность:** Автоматическое извлечение спецификаций параметров и структуры ответов прямо из кода.
2. **Универсальность:** Мгновенная генерация OpenAPI (Swagger) в формате YAML/JSON или чистого Markdown.
3. **Готовность к использованию:** Понятные описания полей и реалистичные mock-данные для фронтенда.

---

## 🚀 Решение: «Генератор документации API»

### 🥉 Basic Version (Базовая версия)

Используйте этот вариант, когда нужен быстрый черновик без сложных настроек.

> **Роль (Role):** Ты опытный бэкенд-разработчик и технический писатель.
>
> **Запрос (Task):** Я предоставлю код API. Проанализируй его и сгенерируй документацию в формате `[Markdown / OpenAPI YAML]`. Укажи эндпоинты, методы, параметры и примеры ответов.

### 🥇 Pro Version (Профессиональная версия)

Идеально подходит для создания production-ready спецификаций с учетом всех пограничных случаев (edge cases).

> **Роль (Role):** Ты Senior Backend Engineer и педантичный технический писатель. Твоя задача — создавать безупречную, стандартизированную документацию API.
>
> **Ситуация (Context):**
>
> - Фон: Я закончил разработку нового эндпоинта, и мне нужно передать его фронтенд-команде.
> - Цель: Получить кристально чистую и точную документацию, которая исключит любые дополнительные вопросы от клиентских разработчиков.
>
> **Запрос (Task):**
> Проанализируй предоставленный ниже код и создай спецификацию в формате **`[OpenAPI 3.0 YAML / Таблица Markdown]`**.
>
> Документация должна строго включать:
>
> 1. **Endpoint & Method:** Точный URL-путь и HTTP-метод.
> 2. **Summary & Description:** Краткое резюме и детальное описание бизнес-логики API.
> 3. **Request Parameters:** Разбор параметров Path, Query и Body (тип данных, обязательность, ограничения).
> 4. **Responses:** Структура ответа для успешного выполнения (200/201) и обработки ошибок (400/401/404/500).
>
> **Код API:**
>
> `[Вставьте сюда код вашего контроллера / маршрутизатора]`
>
> **Ограничения (Constraints):**
>
> - Явно указывай типы данных (String, Integer, Boolean, Array).
> - Значения в примерах (Example Values) должны быть максимально реалистичными (например, используй реальные форматы email или UUID, а не просто "string").
> - Если логика кода возвращает ошибку, обязательно задокументируй формат ответа с ошибкой.
>
> **Предупреждение (Warning):**
>
> - Не придумывай параметры или эндпоинты, которых нет в предоставленном коде. Опирайся исключительно на логику исходника.

## 💡 Комментарий автора (Insight)

Этот промпт — настоящее спасение при работе с legacy-кодом или в условиях жестких дедлайнов. Самое главное здесь — требование генерировать **«реалистичные примеры данных (mock data)»**.

Часто ИИ ленится и пишет в примерах `{"name": "string"}`. Заставив его использовать реалистичные данные (например, `{"name": "Иван Иванов"}`), вы сильно облегчаете жизнь фронтенд-разработчикам. Они смогут сразу использовать эти JSON-ответы для мокирования на своей стороне. Если вы используете TypeScript или Pydantic, передача интерфейсов и моделей вместе с контроллером обеспечит практически 100% идеальный результат.

## 🙋 Часто задаваемые вопросы (FAQ)

- **Q: Поймет ли ИИ сложную архитектуру, если логика размазана по сервисам и репозиториям?**
  - A: Если вы отправите только контроллер, ИИ опишет лишь то, что видит (параметры запроса и структуру ответа контроллера). Для сложной бизнес-логики лучше скопировать в промпт и контроллер, и соответствующий класс DTO/Service.

- **Q: Можно ли сразу получить аннотации для кода (например, Swagger-декораторы для NestJS или Spring)?**
  - A: Да! Просто измените **Запрос (Task)** в Pro-версии на: _«Добавь к моему коду соответствующие Swagger-декораторы (например, @ApiOperation, @ApiResponse) и верни обновленный код»_.

## 🧬 Анатомия промпта (Why it works?)

1. **Role (Роль):** Назначение ИИ роли «Senior Backend Engineer» заставляет его обращать внимание на технические детали (коды статусов HTTP, точные типы данных).
2. **Context & Goal (Контекст):** Указание конечного потребителя («фронтенд-команда») помогает ИИ сфокусироваться на удобочитаемости и практичности документации.
3. **Constraints (Ограничения):** Жесткое требование реалистичных примеров избавляет от типичной проблемы сгенерированной ИИ документации — абстрактных и бесполезных `dummy`-данных.

## 📊 Доказательство: Before & After

### ❌ Before (Исходный код — Node.js Express)

```javascript
app.post("/api/users", (req, res) => {
  const { username, email, age } = req.body;
  if (!email || !username) {
    return res.status(400).send({ error: "Missing required fields" });
  }
  if (age && typeof age !== "number") {
    return res.status(400).send({ error: "Age must be a number" });
  }
  // Логика сохранения в БД...
  res.status(201).send({
    id: "usr_12345",
    username,
    email,
    createdAt: new Date().toISOString(),
  });
});
```

### ✅ After (Результат ИИ — Markdown)

### **POST /api/users**

Создание нового пользователя в системе.

**Request Body (`application/json`)**

| Поле | Тип | Обязательно | Описание | Пример |
|---|---|:---:|---|---|
| `username` | String | ✅ | Уникальный никнейм пользователя | `"alex_dev"` |
| `email` | String | ✅ | Контактный email-адрес | `"alex@example.com"` |
| `age` | Integer | ❌ | Возраст пользователя | `28` |

**Responses**

**`201 Created`** - Пользователь успешно создан

```json
{
  "id": "usr_12345",
  "username": "alex_dev",
  "email": "alex@example.com",
  "createdAt": "2026-02-04T12:00:00.000Z"
}
```

**`400 Bad Request`** - Ошибка валидации

```json
{
  "error": "Missing required fields"
}
```
