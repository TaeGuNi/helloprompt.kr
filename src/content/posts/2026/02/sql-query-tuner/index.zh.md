## 📊 证明: Before & After

### ❌ Before (输入查询)

```sql
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders WHERE price > 10000);
```

_问题点剖析：在部分较旧的数据库版本或特定存储引擎下，这种看似直白的 `IN` 子查询极易被优化器错误解析为非相关的依赖子查询 (Dependent Subquery)。这将导致外部 `users` 表中的每一行数据，都会触发一次内部 `orders` 表的全表扫描查询，当数据量达到百万级时，性能将呈现指数级崩溃。_

### ✅ After (结果)

**优化重构后的极致查询 (引入 JOIN 机制):**

```sql
SELECT DISTINCT u.*
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.price > 10000;
```

**一针见血的最佳索引推荐:**

```sql
CREATE INDEX idx_orders_price_userid ON orders(price, user_id);
```

_优化原理解析：面对低效困境，AI 极为敏锐地将原本臃肿的子查询重写为高效的 `INNER JOIN` 结构，从而大幅释放了连接效率。更令人拍案叫绝的是，它精准推送了基于 `(price, user_id)` 字段的复合索引方案。这一神来之笔使得数据库引擎能够完美利用“覆盖索引 (Covering Index)”或“索引条件下推 (ICP)”等高级特性，在存储引擎层直接完成数据过滤与字段关联，将高昂的回表 (Lookup) 成本降至最低。_
