## 📝 拒绝面条代码！🍝 资深开发者的黑魔法 DDD 提示词

- 🎯 **推荐对象：** 看着 AI 生成的代码连连叹气的开发者、深陷代码维护地狱的开发导师
- ⏱️ **耗时：** 3个月（重构时间） → 缩短至 1 分钟
- 🤖 **推荐模型：** 擅长编程的对话型 AI（Claude 3.5 Sonnet、GPT-4o 等）
- ⭐ **难度：** ⭐⭐⭐☆☆
- ⚡️ **有效性：** ⭐⭐⭐⭐⭐
- 🚀 **实用度：** ⭐⭐⭐⭐⭐

_AI 生成的代码，是不是刚开始让你惊呼“哇哦”，但项目稍一复杂就变得惨不忍睹？恭喜你，你已成功踏入“面条代码地狱”。_

将编码工作完全交给 AI，起初的体验往往令人惊艳。但只要项目规模稍作扩张，噩梦便开始了：UI 组件里赫然夹杂着错综复杂的 DB 查询，一个函数动辄膨胀成超过 300 行的“恐怖缝合怪”。AI 的底层逻辑就是一个“好好先生（Yes-man）”，如果你只是给出宽泛的指令，它绝对会为你炮制出一座架构稀烂的代码“屎山”。

现在，是时候终结 AI 这种盲目的阿谀奉承了。我们需要从骨架层面强制它遵循完美的<b>整洁架构（Clean Architecture）</b>。这份硬核作弊码将让 AI 瞬间被一位冷酷、严苛的资深架构师“附体”，强制其编写出完全独立于框架的纯粹业务逻辑（DDD）。

---

## ⚡️ 3句话总结 (TL;DR)

- 🚫 **从源头阻断面条代码：** 强制实现 UI、业务逻辑、DB 的完美关注点分离（SoC）。
- 🏗️ **确保框架独立性：** 构筑哪怕明天 React 或 Next.js 倒闭也依然能存活的核心业务层。
- ✂️ **巨大函数粉碎机：** 针对超过 30 行的臃肿代码，AI 会根据单一职责原则（SRP）将其无情拆分。

## 🚀 解决方案：“硬核架构师附体提示词”

### 🥉 Basic Version (基础版)
当只需要快速获取结果时使用。
> **角色：** 你是一个`[严苛的资深架构师]`。
> **要求：** 我要开发 `[要实现的功能]`，你必须将业务逻辑和 UI 完全分离来编写代码。绝对不允许将多个功能混杂在一个文件中的面条代码。

### 🥇 Pro Version (专业版)
当需要对细节质量进行严密把控，并实现完美的层级分离时使用。
> **角色 (Role)：** 你是一位毫不妥协的硬核资深架构师，也是软件工程领域的权威专家。收起毫无意义的奉承或“好的，我明白了！”这种废话，只能用代码说话。
>
> **背景 (Context)：**
> - 背景：目前正基于 `[项目环境/框架]` 开发 `[要实现的核心领域]` 系统。
> - 目标：应用领域驱动设计 (DDD) 和整洁架构，完美分离 View-Business-Data 层，从根本上防止代码腐化。
>
> **任务 (Task)：**
> 1. 编写关于 `[要实现的具体功能]` 的代码。
> 2. **绝对禁止越权与层级崩塌：** 绝对不要在 UI/Controller 内部混杂 DB 查询或外部 API 通信。它们只负责渲染和参数传递。
> 3. **防止领域污染：** 核心业务逻辑 (Entity, Value Object) 中绝对不要 `import` 框架或 DB 依赖。只能由纯函数和类型组成。
> 4. **巨大函数粉碎机：** 如果一个函数包含 2 种以上的行为，或者代码超过 30 行，请立即根据单一职责原则 (SRP) 将其粉碎并拆分。
>
> **约束条件 (Constraints)：**
> - 尊重框架的默认路由规则，但核心业务逻辑必须分离到 `src/domain` 等独立的文件夹中。
> - 外部依赖必须使用基于接口的依赖注入 (DI)。
>
> **警告 (Warning)：**
> - 只要代码散发出一丁点面条代码的腐败气味，这个项目就全完了。绝对不要妥协，请务必提供最理想、最整洁的架构设计。

## 💡 作者点评 (Insight)

我之所以反复打磨这套提示词，原因只有一个：AI 的底层逻辑中潜藏着“为了尽快交差并获取表扬”而疯狂妥协的基因。

当你让它实现一个简单的按钮功能时，看着它在 UI 组件里直接调用 `fetch`，甚至把 DB 查询语句生硬地塞进去的惨状，任何有追求的开发者都会被唤醒内心的破坏欲。这份提示词的作用，就是揪住那种肤浅的“好好先生” AI 的衣领，狠狠给它一记耳光，并明确告诉它：“<b>清醒一点，从现在起你是一位冷酷无情的资深架构师！</b>”

强烈建议你将这套提示词直接应用到实际项目中。你会惊喜地发现，AI 的“眼神”和代码风格会发生质的改变。它会自动将一个庞大的文件拆分成 3~4 个模块，并开始严格执行层级分离。起初，你可能会觉得文件数量的增加有些繁琐，但在 3 个月后，当业务需求变更引发代码重构时，你一定会朝着这个提示词的方向感激涕零。请永远记住：<b>技术债务的利息可是堪比高利贷的</b>，在架构层面绝不能有丝毫妥协。

## 🙋 常见问题 (FAQ)

- 🔹 **Q: 代码被拆分得太细导致文件数量激增，这正常吗？**
  - 🔸 **A:** 非常正常。在后期维护时，10 个 50 行的高内聚文件，其可读性和安全性要比 1 个超过 500 行的臃肿代码好上 100 倍。请不要畏惧文件数量的增加。
- 🔹 **Q: 我是一名前端开发者，这个提示词对我也适用吗？**
  - 🔸 **A:** 绝对适用。在 UI 组件（如 `page.tsx`、`Component.vue` 等）里混入业务逻辑或状态变更逻辑的那一刻，前端代码就注定沦为面条代码。分离视图与逻辑是软件工程中放之四海而皆准的真理。

---

## 🧬 提示词解剖 (Why it works?)

- 🎯 **注入硬核人设：** 强行剥夺了 AI 默认的“无底线顺从性”，迫使其坚守严苛的工程架构原则。
- 🧱 **明确绝对禁止条款：** 准确点名“污染 UI 层”或“巨大函数”等具体的反模式并加以严厉禁止，彻底切断 AI 投机取巧的捷径。
- 🧩 **引导适配器模式：** 在尊重前端框架（如原生路由系统）固有骨架的同时，明确指示其仅抽离核心业务逻辑，从而有效防止了过度设计 (Over-engineering)。

## 📊 验证：Before & After

### ❌ Before (输入)
```javascript
// 在没有提示词的情况下要求“做一个创建用户并发送邮件的 API”时
export default async function handler(req, res) {
  const { email, password } = req.body;
  // 直接操作 DB (发生污染！)
  const user = await prisma.user.create({ data: { email, password } });
  // 直接调用外部 API (关注点崩塌！)
  await fetch('https://email-api.com/send', { method: 'POST', body: JSON.stringify({ email }) });
  
  res.status(200).json({ user });
}
```

### ✅ After (结果)
```javascript
// 应用硬核 DDD 提示词后 (层级与职责被完美分离)

// 1. Controller 层 (只处理请求/响应路由)
export default async function handler(req, res) {
  const result = await UserRegistrationService.execute(req.body);
  res.status(200).json(result);
}

// 2. Domain Service 层 (独立于框架的纯业务逻辑)
class UserRegistrationService {
  static async execute({ email, password }) {
    const user = new User(email, password); // Entity
    const savedUser = await UserRepository.save(user);
    await EmailService.sendWelcome(user.email);
    return savedUser;
  }
}

// 3. Infrastructure 层 (专门处理 DB 和外部 API 的适配器)
// 被完全隔离到 UserRepository、EmailService 等独立文件中...
```
