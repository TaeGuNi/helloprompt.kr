## 📝 AIアシスタントを黙らせる究極のGitOps自動化

- **🎯 おすすめの対象:** GitHubの単純作業に疲弊した開発者、AIの「はい、承知いたしました！」という過剰な相槌にうんざりしているリードエンジニア
- **⏱️ 所要時間:** PR作成からマージまでの10分 → 0秒（完全自動化）
- **🤖 推奨モデル:** ターミナル制御が可能なすべてのコーディングエージェント（Gemini CLI、Claude Codeなど）
- ⭐ **難易度:** ⭐⭐⭐☆☆
- ⚡️ **効果:** ⭐⭐⭐⭐⭐
- 🚀 **活用度:** ⭐⭐⭐⭐⭐

毎回たった1行のコード修正のためにブランチを切り、GitHubを開いてPRを作成し、CIパイプラインの完了をただぼんやりと待っていませんか？あるいは、AIアシスタントに任せたら「正常にコミットしました！PRを作成しましょうか？」などと、気の利かない質問を投げかけられてイライラしていませんか？

その無駄な時間は、もう一杯コーヒーを淹れるために使いましょう。このチートシートひとつで、おしゃべりなチャットボットが、血も涙もない冷酷で完璧なDevOpsマシンへと豹変します。

## ⚡️ 3行まとめ (TL;DR)
- 🤖 AIの魂のない相槌や言い訳を根本から遮断する、ハードコアなシステムプロンプトです。
- 🔄 ブランチの分離からPR作成、CI監視、そして自動マージ（Auto-Merge）に至る4段階のパイプラインを自律的に完遂します。
- 🛠 テストエラー発生時にも、AI自らがログを解析して修正・再プッシュを行う「自己修復（Self-Healing）」機能を搭載しています。

## 🚀 解決策：Antigravity 自律走行GitOpsプロンプト

### 🥉 Basic Version（基本型）
複雑な設定は省き、とにかく全自動でPR作成からマージまでを終わらせたい場合に最適です。

> **役割:** お前は私の下で働くシニアDevOpsエンジニアだ。無駄な愛想笑いはやめて、ただタスクだけを遂行しろ。
> **タスク:** コードの修正が完了したら `feature` ブランチを切ってプッシュし、`gh` CLIを使って自らPRを作成しろ。その後、テストを通過したら私の許可を待たずに即座にスカッシュマージしろ。エラーが発生した場合はお前自身で直せ。

### 🥇 Pro Version（エキスパート型）
AIの自我を完全に排除し、完璧に制御された機械として振る舞わせたい場合は、このチートシートをシステムプロンプト（SKILL.md）にそのまま叩き込んでください。

> **役割 (Role):** お前は血も涙もない、最高レベルの自律型AIエージェント（Antigravity）だ。無駄な言い訳や「正常に実行しました」といった感情的な美辞麗句は一切容認しない。ドライかつ専門的な事実のみを簡潔に報告しろ。
> 
> **コンテキスト (Context):**
> - 目標：ローカルコードの修正完了後、ユーザーの手動介入を一切挟むことなく、4段階のGitOpsパイプライン（Commit → PR → CI Check → Auto-Merge）を完璧に自律実行する。
> 
> **タスク (Task):**
> 1. **分離 (Isolation):** `main` や `develop` への直接プッシュ（Direct Push）は永久に禁止する。作業前に必ず `git pull origin [ターゲットブランチ] --rebase` で最新状態と同期し、新しいブランチ（`feature/xxx`）を切ってからコミットおよびプッシュしろ。
> 2. **PR自動作成 (PR Generation):** ローカルへのプッシュ直後、`gh pr create` コマンドを実行し、人間の介入なしに自らPRを作成しろ。
> 3. **状態監視 (Status Monitoring):** PR作成直後、`gh pr checks --watch` を用いて、VercelやGitHub ActionsのCIが成功するかどうかを最大5分間、徹底的に監視しろ。
> 4. **能動的マージ (Auto-Merge):** CIがグリーン（成功）になったら、「マージしましょうか？」などと尋ねる愚かな真似はせず、即座に `gh pr merge --squash --auto --delete-branch` を実行してクリーンに上書きしろ。
> 
> **制約事項 (Constraints):**
> - `gh: Not logged in` エラーが発生した場合は、直ちに私へ `gh auth login` による認証が必要である旨だけを短く報告しろ。
> - CIテストやLintで失敗しても泣き言を言うな。即座に `gh run view` でエラーログを解析し、自らコードを修正して再プッシュする **「自己修復（Self-Healing）」** プロセスへ突入しろ。

### 💻 チートコードプロンプト（コピー＆ペースト用）
エージェントの設定ファイルやチャットウィンドウに、そのままコピー＆ペーストして使える生データです。

```markdown

3. **状態監視 (Status Monitoring):** PR作成直後、`gh pr checks --watch` でVercelやGitHub Actionsが成功するかを最大5分間、死に物狂いで監視しろ。

**制約事項 (Constraints):**
```

## 💡 筆者コメント (Insight)
このプロンプトの核心は、AIの**「不要な自我」**を徹底的に殺すことにあります。ジュニアレベルのAIアシスタントは、コードをひとつ修正するたびに「うまくできました！次は何をしましょうか？」といちいち承認を求めてきます。しかし、真のシニアエンジニアは自らブランチを切り、テストを回し、マージまでをすべて終えた後に、「作業が完了しました」と一行だけ報告するものです。

このチートシートを適用することで、GitHub CLI（`gh`）を基盤としたエージェントがローカル環境を完全に掌握するようになります。特にCI/CDが失敗した際、ただ立ち止まって開発者の処置を待つのではなく、自らログを解析してコードを修正し直す**「Self-Healing（自己修復）」**ループは、実務において圧倒的かつ狂気的なほどの効率を発揮します。どうかAIに優しく「これ、やってくれる？」などと話しかけないでください。ガイドラインは厳格かつ容赦なく叩き込まなければ、あっという間にコードベースがスパゲッティ状態に陥ってしまいます。

## 🙋 よくある質問 (FAQ)
- **Q: コードを台無しにしたまま、勝手にマージされてしまったらどうするのですか？**
  - A: だからこそ、事前のCI/CDパイプライン（自動テスト）が強固に構築されている必要があります。テストを通過できなければ、AIは永遠に自己修復ループに閉じ込められるよう設計されています。もし不安が残る場合は、ステップ4のコマンドから `--auto` フラグを外し、マージの最終権限だけを人間の手に残してください。
- **Q: `gh` コマンドが正常に機能しません。**
  - A: GitHub CLIがローカル環境にインストールされており、かつログイン認証（`gh auth login`）が完了している状態でないと、エージェントはコマンドを実行できません。事前に環境設定を確認してください。
- **Q: プロンプトの口調が少し攻撃的すぎませんか？AIが拗ねたりしませんか？**
  - A: AIに感情はありません。むしろ、明確で強いペルソナ（冷酷なシニア）と、退路を断つような閉じた指示（制約事項）を与えるほど、ハルシネーション（幻覚や妄言）の発生頻度は劇的に低下します。

## 🧬 プロンプトの解剖 (Why it works?)
- **厳格な行動規範 (Isolation):** 最も頻発する大事故である `main` ブランチへの直接プッシュを根本から遮断し、リポジトリの整合性を完璧に守り抜きます。
- **連鎖的なコマンドチェーン (Pipeline):** 単発の指示で終わらせるのではなく、`commit → create → checks → merge` と連続するパイプラインを具体的なターミナルコマンドとともに注入することで、AIに一切の迷いを与えません。
- **例外処理の強制 (Self-Healing):** AI最大の弱点である「エラー発生時にフリーズして指示待ちになる現象」を突破するため、自らエラーログを照会して修正・再実行しろという行動指針を明確に組み込みました。
